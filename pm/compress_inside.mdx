# LargeFileBridge ‚Äî Compress Videos & Images Inside

The user navigates the [File System](./file_system.mdx) column browser to some mid-level directory,
opens that directory's **`‚ãÆ`** menu, and picks **"Compress videos/images inside‚Ä¶"**
([menus.mdx ¬ß5.2](./menus.mdx)). Today that fires a bare browser `window.confirm` ‚Äî a jarring hard
dialog with no options. **This spec replaces it with a proper pop-over configuration dialog** and wires
**Continue** to a background, **per-file-safe** batch compression of that directory.

The same dialog also backs the page-level **"Compress all videos‚Ä¶ / Compress all images‚Ä¶"** action-links
([page_actions.mdx ¬ß4](./page_actions.mdx)) on any file-list page ‚Äî those links open this dialog scoped to
the page's directory, with the matching kind pre-checked.

Siblings this builds on and does not repeat:

* [compression.mdx](./compression.mdx) ‚Äî the **engine** (`compressFile`): the quality-controllable brew
  tools, the resolution-preserving + alpha-safety invariants, the temp-file ‚Üí verify ‚Üí replace flow. This
  file drives that engine over many files; it does not restate how one file is compressed.
* [menus.mdx ¬ß5.2](./menus.mdx) ‚Äî the **directory action catalog** whose "Compress videos/images inside‚Ä¶"
  item opens this dialog (was a `window.confirm`; now this pop-over).
* [page_actions.mdx](./page_actions.mdx) ‚Äî the **page-level** compress links that also open this dialog.
* [job_queue.mdx](./job_queue.mdx) ‚Äî the **background queue** the eligible files are enqueued onto (a new
  `compress` op) and the **batch** that groups them.
* [processing.mdx](./processing.mdx) ‚Äî the **surfaces** that show the batch running (the conditional
  Processing nav item, the bottom-left progress window) and the **final error list** when it finishes.

## 1. Why a pop-over, not a browser confirm (LOCKED)

The user's words: *"It does a browser hard message, which is not great. We definitely want a pop-up
window, pop-over window ‚Äî behind it is kind of see-through but a little bit darkens what's behind it, and
then we have the pop-up window on top."* So:

* The trigger opens a **modal pop-over**: a fixed full-screen **darkened, semi-transparent backdrop**
  (`bg-black/40`, what's behind stays faintly visible) with a **centered white card** on top ‚Äî the app's
  standard hand-rolled modal ([ConfirmDialog](./page_actions.mdx), the wizard pattern), never
  `window.confirm` / `window.prompt`.
* Backdrop-click cancels, inner click does not bubble, **Esc** cancels, and the **Cancel** control is
  focused by default so the batch is never one stray **Enter** away.

The dialog is a **web modal, not a native prompt**, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/compress/CompressInsideDialog.tsx` ‚Äî `CompressInsideDialog` (the fixed `bg-black/40` overlay + centered card, `stopPropagation`, Esc-to-cancel).

## 4. The per-file transactional safety rule (LOCKED ‚Äî the load-bearing invariant)

This is the heart of the spec and it is **LOCKED**. The batch treats **each file independently** and it
**never deletes an original that did not successfully compress**:

* **Per file, in order:** compress the file to a **temp** output, **verify** it (same resolution, real
  size gain ‚Äî [compression.mdx ¬ß5](./compression.mdx)), and **only then** dispose of the original per the
  Originals radio (hard-delete or trash) and move the temp into place.
* **A file that errors part-way is left completely untouched.** If a file's transcode fails, or its output
  fails verification, or the alpha check blocks it, **its original is not deleted** ‚Äî neither hard-deleted
  nor trashed ‚Äî and the temp is discarded. The batch **records the error and continues to the next file**.
  One bad file never halts the run and never endangers a different file's original.
* **Half-done is safe.** If the run compresses the first ten files fine (and, with Hard delete chosen,
  those ten originals are gone) and the eleventh errors, the eleventh's original **remains on disk**; files
  twelve onward still process. Deletion is strictly downstream of that one file's verified success.
* **Final error list.** When the whole run finishes, the files that errored are **shown to the user as a
  list** (with each file's reason). A clean run shows a success summary and no list.

This safety is structural: disposal lives at the very end of the single-file path in the engine, reached
only after verify + gain. It is enforced in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `compressFile()` (the disposition block runs only after `verifyOutput` + the size-gain check; on any earlier failure it returns a `blocked`/`failed` result **without** touching the original). The per-op override that the radio drives is the `deleteOriginal` field of `CompressFileOpts` (`"hard"` ‚Üí `fs.unlinkSync`, `"trash"` ‚Üí `trashOriginal`).

The batch's per-file isolation (catch a task's throw, record it against the batch, move on) is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` ‚Äî `runTask()` (the `compress` branch: `track("compress", ‚Ä¶)` around `compressFile`, then `recordBatchResult()`; `compressed`/`skipped` count as OK, `blocked`/`failed` are recorded as errors; a thrown task is still counted so the batch always finishes) and `recordBatchResult()` (appends `{ path, reason }` to the batch's `errors[]`).

### 4.1 The default-hard-delete contradiction, reconciled (LOCKED)

[compression.mdx ¬ß1.5 / ¬ß8](./compression.mdx) and the charter say a compressed file's original always goes
to the **recoverable LFBridge trash** ‚Äî never hard-deleted ‚Äî and that stays the default for a **one-off**
Compress (the file/viewer menu, [menus.mdx ¬ß5.3](./menus.mdx)). **This bulk dialog deliberately differs:**
it makes disposition an **explicit per-run choice** and **defaults it to Hard delete**, because the user
asked for exactly that ("It will default to delete"). The reconciliation:

* The **global** `compression.replace_original_to_trash` setting still governs one-off compresses (trash by
  default).
* This **dialog's radio** is a **per-run override** passed to the engine (`deleteOriginal`), so a bulk run
  does what the user picked **for that run** regardless of the global default.
* The dialog's **default is Hard delete**; the user can switch to the recoverable trash for any run.

The per-run override winning over the global default is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `compressFile()` (`const disposition = o.deleteOriginal ?? (settings.replaceOriginalToTrash ? "trash" : "hard")`).

## 5. Continue ‚Üí background batch, return immediately (LOCKED)

**Continue** does **not** run the work in the request. It calls the enqueue endpoint, which plans the set
and hands it to the [background queue](./job_queue.mdx), then **returns immediately** so the user can keep
navigating other tabs while the batch drains ([processing.mdx](./processing.mdx)).

* **The walk (server-side).** The endpoint walks the directory for the **selected kinds**, honoring the
  **recursive** flag and skipping the scan's `HARD_SKIP` / hidden / tracking dirs. It includes only files
  whose extension heuristic says they **should** compress (`compressInfo`), so already-compressed media is
  cheaply skipped rather than needlessly re-encoded.
* **A batch is created** grouping those files, and each file is enqueued as a `compress` task carrying the
  chosen `deleteOriginal` mode and the `batchId`.
* **The plan returns immediately** ‚Äî `{ batchId, considered, eligible, queued, images, videos }` ‚Äî and a
  single toast states what will happen (¬ß6). The queue drains the tasks with a bounded per-op concurrency
  (`compress` cap 2, [job_queue.mdx ¬ß3](./job_queue.mdx)); the heavy transcode runs via an **async**
  `spawn` so it **never blocks the event loop** (the web app stays responsive during the batch).

`POST /api/compress/inside` with body `{ root, images, videos, recursive, deleteOriginal }` returns the
`CompressInsidePlan`, implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.router.ts` ‚Äî `POST /inside` (rejects a no-kind selection with `400`) ‚Üí `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `enqueueCompressInside()` / `walkCompressible()` (the kind-filtered, recursive-or-shallow, HARD_SKIP-skipping, should-compress-only walk), which creates the batch via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` ‚Äî `createBatch()` and enqueues through `enqueue()`.

The async transcode that keeps the batch from freezing the app is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `runAsync()` (the `child_process.spawn` runner `compressFile` awaits for the heavy tool), while quick probes stay synchronous.

The client method + the API surface are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts` ‚Äî `compressInside()`.

## 6. The notification (LOCKED wording)

On Continue, one toast states what was queued (never a silent start ‚Äî [sync_process.mdx ¬ß6](./sync_process.mdx)):

* **Something eligible** ‚Üí **"{N} files will be compressed"** (with the breakdown implied by the plan ‚Äî
  e.g. images + videos). `N` is the `queued` figure from the plan.
* **Nothing eligible** (no files of the selected kinds under the directory, or all already compressed) ‚Üí
  **"Nothing to compress here ‚Äî no {images/videos} that need it"** ‚Äî an honest no-op, no batch card.

When the batch **finishes**, the user is told the outcome and, if any errored, pointed at the list
([processing.mdx ¬ß4](./processing.mdx)):

* **Clean** ‚Üí **"Compressed {done} files"**.
* **With errors** ‚Üí **"Compressed {done} files ‚Äî {failed} had errors"**, and the failed files are shown as
  a list on the [Processing page](./processing.mdx) (each with its reason).

The Continue toast is emitted from the plan in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/compress/CompressInsideDialog.tsx` ‚Äî `CompressInsideDialog` (the `onContinue` handler toasts the returned `queued`). The finish summary + the error list live in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingPage.tsx` (processing.mdx ¬ß4).

1. **Pop-over, not a browser confirm (¬ß1):** the directory `‚ãÆ` "Compress videos/images inside‚Ä¶" item opens

4. **Per-file transactional safety (¬ß4, LOCKED):** each file is compressed independently; an original is
   disposed (hard-deleted or trashed per the radio) **only after that file's output verified**. A file that
   errors part-way is **left untouched**, its error is recorded, and the batch **continues**. A run that
   compresses some and errors on one still deletes only the succeeded originals.
5. **Final error list (¬ß4/¬ß6):** when the run finishes, the files that errored are shown to the user as a
   list with reasons; a clean run shows only a success summary.
6. **Default-hard-delete reconciliation (¬ß4.1):** this bulk dialog defaults to Hard delete and passes a

7. **Background + responsive (¬ß5):** Continue plans + enqueues and **returns immediately**; the batch
   drains in the background (per-op cap 2) with the heavy transcode on an **async** child process, so the
   app stays responsive and the user can navigate away. Already-compressed media is skipped, not
   re-encoded.
8. **Notification (¬ß6):** one honest toast on Continue ("{N} files will be compressed" / the nothing-to-do
   line) and a finish summary that names the error count when non-zero.
9. **Page-action reuse (¬ß2):** the page-level "Compress all videos‚Ä¶ / Compress all images‚Ä¶" links open the
   same dialog scoped to the page's directory with the matching kind pre-checked.
