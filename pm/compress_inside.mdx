# LargeFileBridge — Compress Videos & Images Inside

The user navigates the [File System](./file_system.mdx) column browser to some mid-level directory,
opens that directory's **`⋮`** menu, and picks **"Compress videos/images inside…"**
([menus.mdx §5.2](./menus.mdx)). Today that fires a bare browser `window.confirm` — a jarring hard
dialog with no options. **This spec replaces it with a proper pop-over configuration dialog** and wires
**Continue** to a background, **per-file-safe** batch compression of that directory.

The same dialog also backs the page-level **"Compress all videos… / Compress all images…"** action-links
([page_actions.mdx §4](./page_actions.mdx)) on any file-list page — those links open this dialog scoped to
the page's directory, with the matching kind pre-checked.

Siblings this builds on and does not repeat:

* [compression.mdx](./compression.mdx) — the **engine** (`compressFile`): the quality-controllable brew
  tools, the resolution-preserving + alpha-safety invariants, the temp-file → verify → replace flow. This
  file drives that engine over many files; it does not restate how one file is compressed.
* [menus.mdx §5.2](./menus.mdx) — the **directory action catalog** whose "Compress videos/images inside…"
  item opens this dialog (was a `window.confirm`; now this pop-over).
* [page_actions.mdx](./page_actions.mdx) — the **page-level** compress links that also open this dialog.
* [job_queue.mdx](./job_queue.mdx) — the **background queue** the eligible files are enqueued onto (a new
  `compress` op) and the **batch** that groups them.
* [processing.mdx](./processing.mdx) — the **surfaces** that show the batch running (the conditional
  Processing nav item, the bottom-left progress window) and the **final error list** when it finishes.

## 1. Why a pop-over, not a browser confirm (LOCKED)

The user's words: *"It does a browser hard message, which is not great. We definitely want a pop-up
window, pop-over window — behind it is kind of see-through but a little bit darkens what's behind it, and
then we have the pop-up window on top."* So:

* The trigger opens a **modal pop-over**: a fixed full-screen **darkened, semi-transparent backdrop**
  (`bg-black/40`, what's behind stays faintly visible) with a **centered white card** on top — the app's
  standard hand-rolled modal ([ConfirmDialog](./page_actions.mdx), the wizard pattern), never
  `window.confirm` / `window.prompt`.
* Backdrop-click cancels, inner click does not bubble, **Esc** cancels, and the **Cancel** control is
  focused by default so the batch is never one stray **Enter** away.

The dialog is a **web modal, not a native prompt**, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/compress/CompressInsideDialog.tsx` — `CompressInsideDialog` (the fixed `bg-black/40` overlay + centered card, `stopPropagation`, Esc-to-cancel).

```
│                                   [ Cancel ]   [ Continue ]     │
└───────────────────────────────────────────────────────────────┘
```

The controls, in order, all LOCKED:

1. **Images** — a checkbox. When **on**, image files are included.
2. **PNG→JPEG warning** — the line **directly under the Images checkbox**, shown **only while Images is
   checked**, tinted **red** (`var(--lfb-bad)`): **"Warning: PNG files will be turned into JPEG files."**
   (The engine's alpha-safety check still protects genuinely-transparent PNGs — [compression.mdx §6](./compression.mdx)
   — but the warning states the common outcome plainly up front.)
3. **Videos** — a checkbox. When **on**, video files are included.
4. **Also compress recursively in subdirectories** — a checkbox, **default ON**. When off, only the
   chosen directory's **immediate** files are compressed; when on, the whole subtree is (skipping the same
   hidden / tracking / build-output dirs the scan skips, [scan.mdx §4](./scan.mdx) `HARD_SKIP`).
5. **Originals** — a **radio group** (exactly one selected) deciding what happens to each original **after**
   its file compresses successfully:
   * **Hard delete** — **the DEFAULT**. The original is `unlink`-ed (permanently removed).
   * **LargeFileBridge trash (recoverable)** — the original is moved to the LFBridge trash under the state
     dir (recoverable), the same trash a per-file Compress uses.
6. **Continue** / **Cancel** — **Continue** starts the batch (§5); **Cancel** closes with no effect.
   **Continue is disabled** while **neither Images nor Videos** is checked (nothing to do).

> **At least one kind (LOCKED).** With both Images and Videos unchecked there is nothing to compress, so
> Continue is disabled and the enqueue endpoint also rejects the empty selection with a `400` — the UI
> never sends a no-kind request.

The four inputs, the conditional red PNG warning, the default-on recursive box, and the delete-defaulted
radio are rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/compress/CompressInsideDialog.tsx` — `CompressInsideDialog` (state: `images`/`videos`/`recursive`/`deleteOriginal`; the PNG warning gated on `images`; Continue gated on `images || videos`).

## 4. The per-file transactional safety rule (LOCKED — the load-bearing invariant)

This is the heart of the spec and it is **LOCKED**. The batch treats **each file independently** and it
**never deletes an original that did not successfully compress**:

* **Per file, in order:** compress the file to a **temp** output, **verify** it (same resolution, real
  size gain — [compression.mdx §5](./compression.mdx)), and **only then** dispose of the original per the
  Originals radio (hard-delete or trash) and move the temp into place.
* **A file that errors part-way is left completely untouched.** If a file's transcode fails, or its output
  fails verification, or the alpha check blocks it, **its original is not deleted** — neither hard-deleted
  nor trashed — and the temp is discarded. The batch **records the error and continues to the next file**.
  One bad file never halts the run and never endangers a different file's original.
* **Half-done is safe.** If the run compresses the first ten files fine (and, with Hard delete chosen,
  those ten originals are gone) and the eleventh errors, the eleventh's original **remains on disk**; files
  twelve onward still process. Deletion is strictly downstream of that one file's verified success.
* **Final error list.** When the whole run finishes, the files that errored are **shown to the user as a
  list** (with each file's reason). A clean run shows a success summary and no list.

This safety is structural: disposal lives at the very end of the single-file path in the engine, reached
only after verify + gain. It is enforced in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile()` (the disposition block runs only after `verifyOutput` + the size-gain check; on any earlier failure it returns a `blocked`/`failed` result **without** touching the original). The per-op override that the radio drives is the `deleteOriginal` field of `CompressFileOpts` (`"hard"` → `fs.unlinkSync`, `"trash"` → `trashOriginal`).

The batch's per-file isolation (catch a task's throw, record it against the batch, move on) is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `runTask()` (the `compress` branch: `track("compress", …)` around `compressFile`, then `recordBatchResult()`; `compressed`/`skipped` count as OK, `blocked`/`failed` are recorded as errors; a thrown task is still counted so the batch always finishes) and `recordBatchResult()` (appends `{ path, reason }` to the batch's `errors[]`).

### 4.1 The default-hard-delete contradiction, reconciled (LOCKED)

[compression.mdx §1.5 / §8](./compression.mdx) and the charter say a compressed file's original always goes
to the **recoverable LFBridge trash** — never hard-deleted — and that stays the default for a **one-off**
Compress (the file/viewer menu, [menus.mdx §5.3](./menus.mdx)). **This bulk dialog deliberately differs:**
it makes disposition an **explicit per-run choice** and **defaults it to Hard delete**, because the user
asked for exactly that ("It will default to delete"). The reconciliation:

* The **global** `compression.replace_original_to_trash` setting still governs one-off compresses (trash by
  default).
* This **dialog's radio** is a **per-run override** passed to the engine (`deleteOriginal`), so a bulk run
  does what the user picked **for that run** regardless of the global default.
* The dialog's **default is Hard delete**; the user can switch to the recoverable trash for any run.

The per-run override winning over the global default is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile()` (`const disposition = o.deleteOriginal ?? (settings.replaceOriginalToTrash ? "trash" : "hard")`).

## 5. Continue → background batch, return immediately (LOCKED)

**Continue** does **not** run the work in the request. It calls the enqueue endpoint, which plans the set
and hands it to the [background queue](./job_queue.mdx), then **returns immediately** so the user can keep
navigating other tabs while the batch drains ([processing.mdx](./processing.mdx)).

* **The walk (server-side).** The endpoint walks the directory for the **selected kinds**, honoring the
  **recursive** flag and skipping the scan's `HARD_SKIP` / hidden / tracking dirs. It includes only files
  whose **extension heuristic** (`compressInfo`) says they should compress, so already-lossy extensions are
  dropped with zero I/O. The stronger, cross-machine skip — the authoritative **in-file marker**
  ([compression.mdx §8.4](./compression.mdx)) — is enforced **per file when it reaches the front of the
  queue**, inside `compressFile()`: a file that already carries our `comment` marker (`LFBcompressed;…`)
  returns `skipped ("already compressed (marker)")` **before any transcode**. That is why a **re-run** (a
  second pass after a reboot) — and a run on **another computer in the mesh** that already has the
  compressed files — skips them **instantly instead of re-encoding**: the marker rides inside the file's
  bytes over IPFS, so it is present even on a peer with an empty local index. (Pruning known-compressed
  files from the batch **up-front** via the local `.lfbridge` record is a tracked speed follow-up.) As it
  goes, the walk **stamps each eligible file with its media kind** — `image` or `video` — from that same
  extension heuristic, because the two kinds draw **different** concurrency budgets downstream (below,
  [job_queue.mdx §3](./job_queue.mdx)).
* **A batch is created** grouping those files, and each file is enqueued as a `compress` task carrying the
  chosen `deleteOriginal` mode, the stamped **`mediaKind`** (image vs video), and the `batchId`.
* **The plan returns immediately** — `{ batchId, considered, eligible, queued, images, videos }` — and a
  single toast states what will happen (§6). The queue drains the tasks against the **media-aware
  mass-compute core budget** ([job_queue.mdx §3](./job_queue.mdx), [parallelization.mdx](./parallelization.mdx)):
  **image** tasks fan **wide** — ≈90% of cores, one thread each, since oxipng/cwebp/mozjpeg/magick are
  single-threaded per file — while **video** tasks fan **narrower** — `floor(budget / videoThreads)` jobs,
  each ffmpeg `-threads`-capped — so `concurrentJobs × threadsPerJob ≈ budget`, never `cores²`. A folder of
  300 files therefore **pins the machine near 90%** until it's done, instead of crawling two-at-a-time. The
  heavy transcode still runs via an **async** `spawn` so it **never blocks the event loop** (the web app
  stays responsive during the batch), and the per-job thread cap prevents oversubscription while it runs
  hot.

> **Parallelism does not weaken the safety rule (§4, LOCKED).** Draining many files at once changes
> **how many** files are in flight, never **how** each one is handled: every file is still processed
> **independently** — temp → verify → dispose — so the transactional guarantee of §4 is **unchanged**. A
> file that errors part-way is still left completely untouched while its siblings keep going, and the
> per-job thread cap (image = 1 thread, video = capped ffmpeg `-threads`) keeps the wide fan-out from
> oversubscribing the machine. More parallel, exactly as safe.

`POST /api/compress/inside` with body `{ root, images, videos, recursive, deleteOriginal }` returns the
`CompressInsidePlan`, implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.router.ts` — `POST /inside` (rejects a no-kind selection with `400`) → `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `enqueueCompressInside()` / `walkCompressible()` (the kind-filtered, recursive-or-shallow, HARD_SKIP-skipping, should-compress-only walk), which creates the batch via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `createBatch()` and enqueues through `enqueue()`.

The async transcode that keeps the batch from freezing the app is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `runAsync()` (the `child_process.spawn` runner `compressFile` awaits for the heavy tool), while quick probes stay synchronous.

The client method + the API surface are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts` — `compressInside()`.

## 6. The notification (LOCKED wording)

On Continue, one toast states what was queued (never a silent start — [sync_process.mdx §6](./sync_process.mdx)):

* **Something eligible** → **"{N} files will be compressed"** (with the breakdown implied by the plan —
  e.g. images + videos). `N` is the `queued` figure from the plan.
* **Nothing eligible** (no files of the selected kinds under the directory, or all already compressed) →
  **"Nothing to compress here — no {images/videos} that need it"** — an honest no-op, no batch card.

When the batch **finishes**, the user is told the outcome and, if any errored, pointed at the list
([processing.mdx §4](./processing.mdx)):

* **Clean** → **"Compressed {done} files"**.
* **With errors** → **"Compressed {done} files — {failed} had errors"**, and the failed files are shown as
  a list on the [Processing page](./processing.mdx) (each with its reason).

The Continue toast is emitted from the plan in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/compress/CompressInsideDialog.tsx` — `CompressInsideDialog` (the `onContinue` handler toasts the returned `queued`). The finish summary + the error list live in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingPage.tsx` (processing.mdx §4).

## 7. Acceptance criteria

1. **Pop-over, not a browser confirm (§1):** the directory `⋮` "Compress videos/images inside…" item opens
   a darkened-backdrop, centered-card modal — never `window.confirm`. Backdrop/Esc cancels; Cancel is the
   default-focused control.
2. **Four inputs (§3):** an **Images** checkbox, a **Videos** checkbox, an **also-recursive** checkbox
   defaulting **ON**, and an **Originals** radio group of **Hard delete (DEFAULT)** / **LargeFileBridge
   trash (recoverable)**. **Continue** is disabled unless at least one of Images/Videos is checked.
3. **PNG→JPEG red warning (§3):** the line **"Warning: PNG files will be turned into JPEG files."** shows —
   red — **only while Images is checked**, directly under the Images checkbox.
4. **Per-file transactional safety (§4, LOCKED):** each file is compressed independently; an original is
   disposed (hard-deleted or trashed per the radio) **only after that file's output verified**. A file that
   errors part-way is **left untouched**, its error is recorded, and the batch **continues**. A run that
   compresses some and errors on one still deletes only the succeeded originals.
5. **Final error list (§4/§6):** when the run finishes, the files that errored are shown to the user as a
   list with reasons; a clean run shows only a success summary.
6. **Default-hard-delete reconciliation (§4.1):** this bulk dialog defaults to Hard delete and passes a
   per-run override to the engine; the global recoverable-by-default still governs one-off compresses.
7. **Background + media-aware core budget (§5):** Continue plans + enqueues and **returns immediately**;
   the walk stamps each file's **`mediaKind`** and the batch drains against the **media-aware mass-compute
   core budget** ([job_queue.mdx §3](./job_queue.mdx), [parallelization.mdx](./parallelization.mdx)) —
   **image** tasks fan **wide** (≈90% of cores, 1 thread each), **video** tasks fan **narrower**
   (thread-capped ffmpeg), so `concurrentJobs × threadsPerJob ≈ budget` (never `cores²`) and a 300-file
   folder pins the machine near 90% until done instead of crawling two-at-a-time. The heavy transcode stays
   on an **async** child process so the app remains responsive and the user can navigate away; the per-file
   safety of §4 is unchanged. Already-compressed media is skipped, not re-encoded.
8. **Notification (§6):** one honest toast on Continue ("{N} files will be compressed" / the nothing-to-do
   line) and a finish summary that names the error count when non-zero.
9. **Page-action reuse (§2):** the page-level "Compress all videos… / Compress all images…" links open the
   same dialog scoped to the page's directory with the matching kind pre-checked.
10. **Durable skip across passes & mesh (§5, [compression.mdx §8.4](./compression.mdx)):** a **second run**
   over the same tree (e.g. after a reboot mid-run) and a **first run on another computer** that already
   holds the compressed files **skip the already-compressed files without re-encoding**. The **in-file
   marker** in the `comment` tag (`LFBcompressed;…`, written inline by ffmpeg/ImageMagick, read by
   ffprobe/`magick identify`) is checked per file in `compressFile()`, so an 18,000-file re-run becomes
   *"probe-and-skip per file"* rather than *"re-transcode everything"* — the marker rides inside the file's
   bytes over IPFS, so peers honor it even with an empty local record. (Verified end-to-end.)
