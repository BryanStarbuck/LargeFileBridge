## 1. The rule ‚Äî Large File Bridge never opens a native browser dialog (LOCKED)

`window.confirm`, `window.alert`, and `window.prompt` are **forbidden** anywhere in the frontend. They are
the operating-system "hard" dialogs the product owner rejected: they look nothing like the app, block the
whole tab, can't be styled or themed, can't carry a file preview or a checklist, and on some platforms are
suppressed entirely. **Every** decision the app asks the user to make is expressed in one of **our** HTML
surfaces instead:

| The user is asked to‚Ä¶ | Use this in-app surface | Owner |
| --- | --- | --- |
| **Confirm a yes/no action** (Compress this? Move to trash? Stop pinning? Remove repo/peer/pin?) | **`ConfirmDialog`** ‚Äî a fixed-overlay modal, backdrop + `Esc` cancel, **Cancel focused by default** so a destructive confirm is never one stray `Enter` away, red button for destructive. | ¬ß2.1 |
| **Type one value** (the move-file destination path) | **`PromptDialog`** ‚Äî the same modal shell with a single labelled text input, **Confirm disabled until non-empty**, `Enter` submits, `Esc` cancels. | ¬ß2.2 |
| **Configure + confirm a batch** (compress videos/images inside; git-ignore) | the **pop-over configuration dialog** ‚Äî [Compress Inside](./compression.mdx) / [Git Ignore](./git_ignore.mdx); the dialog **is** the confirm. | those specs |
| **Confirm + adjust a producing batch** (Create Transcriptions / Create AI descriptions) | the **unified batch-confirm popup** (¬ß5) ‚Äî the checked-by-default candidate list. | ¬ß5 |
| **Be passively told an outcome** ("28 files will have their Transcriptions created") | a **toast** ([sonner](./page_actions.mdx#2-the-notification-locked-wording)) ‚Äî never `window.alert`. | page_actions.mdx ¬ß2 |

There is **no** exception. A quick "are you sure?" is a `ConfirmDialog`; a one-line text ask is a
`PromptDialog`; a batch is a pop-over. The bar is: **if the user reads it as a dialog, it is one of ours,
rendered in the app's own chrome and theme.**

## 2. The two primitive modals (LOCKED)

### 2.1 `ConfirmDialog` ‚Äî the yes/no confirm

Already exists and is the house confirm modal (used by the page action-links row for its red-tinted
offers). It is a fixed overlay + centered card: a title, an optional body (React node ‚Äî so it can list what
will happen, show counts, or warn about data loss), a **Cancel** (focused) and a **Confirm** button, with
the Confirm button **red** (`--lfb-bad`) when `danger` and **blue** (`--lfb-primary`) otherwise. Backdrop
click, `Esc`, and Cancel all dismiss without acting.

Rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/ConfirmDialog.tsx` ‚Äî `ConfirmDialog`.

### 2.2 `PromptDialog` ‚Äî the single-text-input ask (NEW)

The **one** thing `ConfirmDialog` can't do is collect a value, which is why the codebase still had a
`window.prompt` for "Move file to (absolute path):". Add a **`PromptDialog`** ‚Äî the same modal shell with a
single labelled `<input>` (pre-filled with an optional default, e.g. the file's current path), **Confirm
disabled while the field is empty**, `Enter` submits, `Esc`/backdrop/Cancel returns `null`. It resolves the
typed string or `null`.

Rendered in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/PromptDialog.tsx` ‚Äî `PromptDialog`.

### 2.3 The imperative bridge ‚Äî `confirmModal()` / `promptModal()` + one `ModalHost` (NEW, LOCKED)

Almost every native-dialog site is **imperative** ‚Äî it lives inside a click handler and reads like
`if (!window.confirm(msg)) return;` or `const dest = window.prompt(msg, def);`. Rewriting each into a piece
of render-time React state would be invasive and error-prone across ~15 files. Instead we provide **two
promise-returning helpers** that any handler can **`await`**, backed by **one** `ModalHost` mounted once at
the app root (the exact single-slot bus pattern the app already uses for the setup wizard and the
transcribe-model consent dialog ‚Äî [transcribe.ts](./transcribe_engine.mdx) `requestModelConsent`,
`setupWizard.ts` `requestStorageSetup`):

```ts
// lib/modals.ts
confirmModal(opts: {
  title: string; body?: ReactNode; confirmLabel?: string; cancelLabel?: string; danger?: boolean;
}): Promise<boolean>;                     // resolves true = confirmed, false = cancelled/Esc/backdrop

promptModal(opts: {
  title: string; label?: string; defaultValue?: string; placeholder?: string;
  confirmLabel?: string; validate?: (v: string) => string | null;   // return an error string to block Confirm
}): Promise<string | null>;               // resolves the typed value, or null on cancel
```

* A call **pushes a request onto the single-slot bus**; the mounted `ModalHost` renders the matching
  `ConfirmDialog` / `PromptDialog` and resolves the promise on the user's choice, then clears the slot.
* Handlers convert **mechanically**: `if (!window.confirm(m)) return;` ‚Üí `if (!(await confirmModal({ title: m }))) return;`
  and `const d = window.prompt(m, def);` ‚Üí `const d = await promptModal({ title: m, defaultValue: def });`.
* The host lives at the app root so a modal shows regardless of which page fired it; only **one** modal is
  ever on screen (a second request replaces/queues ‚Äî single-slot).

Implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/modals.ts` ‚Äî `confirmModal()` / `promptModal()` / `onModalRequested()` (the bus) ¬∑ the provider `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/ModalHost.tsx` ‚Äî `ModalHost` (mounted once in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/App.tsx`).

## 4. Why a modal, not a toast, for a decision (rationale, non-normative)

A **toast** is a passive receipt ‚Äî it tells the user what already happened and vanishes. A **modal** is an
active gate ‚Äî it stops and asks. A destructive or irreversible action (compress-with-trash, move, unpin,
remove) needs the **gate**, so it is a `ConfirmDialog`; the *result* of that action, once taken, is the
**receipt**, so it is a toast. Never invert them: never use a toast to ask a question the user must answer,
and never use a modal to announce a routine success.

## 5. The unified batch-confirm popup ‚Äî the "great pop-up" (LOCKED)

The product owner's model: on the **View-one-repo** page, clicking the **Transcribable** metric tile opens
a large, well-made popup ‚Äî the candidate audio/video list, checked by default, a **Continue** into the
checklist, uncheck anything, then a solid-blue **"Transcribe N files"** confirm that runs the work in the
background and shows the **Processing** surface ([task_tabs.mdx ¬ß5.1](./task_tabs.mdx)). *"That works
great."* The rule of this section: **that exact popup is the ONE surface every producing entry point
opens** ‚Äî nothing produces transcripts or AI descriptions any other way.

### 5.1 What the popup is (LOCKED)

It is the **`WarningPopup`** ([warnings.mdx ¬ß4](./warnings.mdx)) fed a **transcribe** or **describe**
`WarningDef` whose `targets` are the candidate files:

* **Two-pane, ~80% of screen** ([warnings.mdx ¬ß4.0](./warnings.mdx), [transcribe_calc_engine.mdx ¬ß4](./transcribe_calc_engine.mdx)):
  left = the "what this is / why it matters" education + the media preview on row hover; right = the
  **candidate checklist, every row checked at open**.
* **Live count in the confirm button.** The solid-blue button reads **"Transcribe N files"** /
  **"Describe N files"** where **N is the live checked count** (`actionLabel` "Transcribe" + the popup's
  appended `‚Äî N files`); unchecking a row decrements it. Confirm is **disabled at zero checked**.
* **The "Filter:" kind row** ([warnings.mdx ¬ß4.5.4](./warnings.mdx)). When the action spans two media kinds,
  the right pane leads with a **"Filter:"** row of per-kind checkboxes ‚Äî **Videos** and **Images** for AI
  descriptions ([ai_description.mdx ¬ß12.4.1](./ai_description.mdx)) ‚Äî **both checked by default**.
  Unchecking a kind **hides those rows and drops them from the batch**: the popup applies **visible ‚à©
  checked**, so a file that is not in the list on the right is **never** processed, and the live count falls
  with it.
* **Apply = model/provider gate, then background enqueue.** Confirm runs the transcribe/describe **consent
  gate** ([transcribe_engine.mdx ¬ß3](./transcribe_engine.mdx) `withModelReady` for transcribe; the AI
  provider-key check for describe) ‚Äî which may interpose the one-time model-download or the credentials
  dialog and then **resume** ‚Äî and then **enqueues the checked paths as background jobs**
  (`api.transcribeEnqueue({ paths })` / `api.describeEnqueue({ paths })`), never an inline blocking run.
* **Processing surface + toast.** The popup closes; the [Processing](./processing.mdx) surface shows the
  jobs draining; the LOCKED toast fires with the queued count ([page_actions.mdx ¬ß2](./page_actions.mdx) ‚Äî
  *"N files will have their Transcriptions created"*).

The transcribe / describe `WarningDef` builders (checked-by-default targets, "Transcribe/Describe N files"
button, gated background enqueue) already exist for the repo-scoped tiles in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/metricWarningDefs.tsx` ‚Äî `buildTranscribeWarning()` / `buildDescribeWarning()`.

### 5.2 The candidate list comes from a plan/preview, not the enqueue count (LOCKED)

The repo tiles build their `targets` from `RepoDetail.files` (already loaded, filtered to
`transcribe === "could"` / `describe === "could"`). The **other** entry points ‚Äî a **directory**, a **File
System column**, a **storage** ‚Äî have no `RepoDetail`; they have a **scope** (a root path, or a checked
subset). For those, the frontend fetches a **preview plan** that returns the eligible candidate **file
list** (paths + sizes) under the scope ‚Äî Rules 1 (scope) + 2 (skip-already-done) applied **on the server**,
exactly as enqueue does, but **without queuing anything** ‚Äî and builds the same `WarningDef` from it.

* **New endpoints** (plan-only, no enqueue), mirroring the enqueue routes:
  * `POST /api/transcribe/plan` ‚Äî body `{ paths?: string[]; root?: string }` ‚Üí `{ files: {path, sizeBytes}[]; considered; alreadyDone; unsupported }`.
  * `POST /api/describe/plan` ‚Äî same shape.
* Both walk-or-take the set with the **same `walkMedia()` / skip-already-done** logic the enqueue planners
  use ([page_actions.mdx ¬ß1.1/¬ß1.2](./page_actions.mdx)), so the preview list and a later enqueue never
  disagree.
* **Empty preview** ‚Üí the popup opens in its **all-clear** state ("Nothing to transcribe here ‚Äî all N files
  already have transcripts") rather than a silent no-op, honoring the honest-outcome rule
  ([page_actions.mdx ¬ß2](./page_actions.mdx)).

The plan endpoints (walk the scope, drop already-done + unsupported, return the candidate list without
queuing) are implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` ‚Äî `previewTranscribe()` (reuses `resolveCandidates()` + the already-transcribed filter, returns the list) ¬∑ `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` ‚Äî `previewDescribe()`; routes `POST /plan` on `transcribe.router.ts` / `describe.router.ts`; client methods `api.transcribePlan` / `api.describePlan` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts`.

### 5.2.1 The excluded already-done count is STATED, not just applied (NEW, LOCKED)

Rule 2 (skip-already-done) is applied on the server (¬ß5.2) ‚Äî a file that already carries a finished
transcript / AI description is **never** in `files`, so the popup **cannot** offer to redo it, and Confirm
**cannot** re-enqueue it. That much has always held. What the popup did **not** do was **say so**: it showed
only the eligible remainder and never mentioned the excluded ones.

That silence is a real product defect, because **the user cannot tell a correct list from a broken one**.
Re-open a 2,000-file tree you have run this action on before and the popup says "1,900 files are ready for
an AI description" ‚Äî a number the user reads as *"it is redoing everything I already did"*. The list was
right; the popup just never showed its work. The user's only recourse is to count sidecars on disk by hand.

**The rule.** Whenever the plan excluded anything (`alreadyDone > 0`), the popup's **sub line states the
excluded count** next to the candidate count ‚Äî e.g. *"‚Ä¶nothing runs until you apply. **91 files already
have an AI description and are excluded from this list.**"* When `alreadyDone` is 0 the clause is omitted
(nothing was skipped, so there is nothing to explain). This makes the skip **visible and checkable at a
glance**, and it turns "is this thing redoing my work?" into a question the popup answers itself.

* The count is the plan's own `alreadyDone` (¬ß5.2) ‚Äî the popup **never** recomputes it, so the number shown
  and the filter applied can never disagree.
* This is a **transparency** requirement, not a filtering one ‚Äî the filter is upstream and unchanged.
* Both producing actions obey it: transcriptions and AI descriptions.

Stating the excluded count in the popup sub line is implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/batchPopup.ts` ‚Äî `excludedClause(alreadyDone, noun)`, appended to the `sub` built by `openTranscribeBatch()` / `openDescribeBatch()`.

The server-side skip this clause reports is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` ‚Äî `previewDescribe()` (the `readDescription(abs)` ‚Üí `alreadyDone++; continue;` branch) ¬∑ `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` ‚Äî `previewTranscribe()`.

### 5.4 The "Opening window‚Ä¶" spinner while the plan walks (NEW, LOCKED)

The plan preview (¬ß5.2) has to **walk the scope's directory tree** to find the candidates ‚Äî and on a large
subtree that is **slow**: the product owner opened a directory whose subtree held **~2,000 files** and the
popup took roughly **two minutes** to appear. During that walk, **nothing was on screen**, so the app
looked **hung**. That is unacceptable ‚Äî a long-running open must always show it is doing something.

**The rule:** the instant a producing entry point is invoked (¬ß6 ‚Äî the metric tile builds its list from
already-loaded `RepoDetail.files`, but **every scope-walking entry point** ‚Äî the page action-link, the
`‚ãÆ` / right-click menu on a directory, a File-System column) the app **immediately** shows an **"Opening
window‚Ä¶" loading modal with an animated spinner**, *before* awaiting the plan. When the plan resolves, that
loading modal is **replaced in place** by the real ¬ß5 batch popup. So the sequence the user sees is:

1. **Click** "Create AI descriptions" (or Create Transcriptions) on a directory.
2. **At once** ‚Äî a small centered modal appears: an **animated spinner** + **"Opening window‚Ä¶"** headline
   + a one-line sub explaining the wait (e.g. *"Scanning this folder for files ‚Äî large folders can take a
   moment."*). This appears **synchronously**, before the `POST /api/{describe,transcribe}/plan` request is
   even awaited, so there is **never** a blank, hung-looking gap.
3. **When the walk finishes** ‚Äî the spinner modal is swapped for the full batch popup (the candidate
   checklist, the "Describe/Transcribe N files" confirm), or, if the scope had no candidates, the popup's
   all-clear state (¬ß5.2).
4. **On error** ‚Äî the spinner modal closes and a toast explains the failure (¬ß5.2's catch).

Additional rules:

* **Cancellable.** The spinner modal has a **Cancel** (and `Esc` / backdrop click) that closes it. Because
  the walk is a fire-and-forget request the user abandoned, a **generation guard** in the launcher makes the
  eventual plan result a **no-op** ‚Äî cancelling the spinner must **not** later pop the real window open.
* **Only for scope-walking opens.** The metric tile / header primary build their list from data already in
  memory (no walk), so they need no spinner ‚Äî they open the popup directly. The spinner is specifically for
  the **plan-fetch** opens (`openTranscribeBatch` / `openDescribeBatch`), which is exactly where the
  multi-minute walk lives.
* **One host, one slot.** The spinner and the popup share the **same single global host** (¬ß5.3): the
  loading modal and the batch popup are two states of the one `BatchPopupHost` bus, so there is never a
  spinner *and* a popup on screen at once, and the swap is seamless.
