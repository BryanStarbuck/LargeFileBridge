## 1. The rule — Large File Bridge never opens a native browser dialog (LOCKED)

`window.confirm`, `window.alert`, and `window.prompt` are **forbidden** anywhere in the frontend. They are
the operating-system "hard" dialogs the product owner rejected: they look nothing like the app, block the
whole tab, can't be styled or themed, can't carry a file preview or a checklist, and on some platforms are
suppressed entirely. **Every** decision the app asks the user to make is expressed in one of **our** HTML
surfaces instead:

| The user is asked to… | Use this in-app surface | Owner |
| --- | --- | --- |
| **Confirm a yes/no action** (Compress this? Move to trash? Stop pinning? Remove repo/peer/pin?) | **`ConfirmDialog`** — a fixed-overlay modal, backdrop + `Esc` cancel, **Cancel focused by default** so a destructive confirm is never one stray `Enter` away, red button for destructive. | §2.1 |
| **Type one value** (the move-file destination path) | **`PromptDialog`** — the same modal shell with a single labelled text input, **Confirm disabled until non-empty**, `Enter` submits, `Esc` cancels. | §2.2 |
| **Configure + confirm a batch** (compress videos/images inside; git-ignore) | the **pop-over configuration dialog** — [Compress Inside](./compression.mdx) / [Git Ignore](./git_ignore.mdx); the dialog **is** the confirm. | those specs |
| **Confirm + adjust a producing batch** (Create Transcriptions / Create AI descriptions) | the **unified batch-confirm popup** (§5) — the checked-by-default candidate list. | §5 |
| **Be passively told an outcome** ("28 files will have their Transcriptions created") | a **toast** ([sonner](./page_actions.mdx#2-the-notification-locked-wording)) — never `window.alert`. | page_actions.mdx §2 |

There is **no** exception. A quick "are you sure?" is a `ConfirmDialog`; a one-line text ask is a
`PromptDialog`; a batch is a pop-over. The bar is: **if the user reads it as a dialog, it is one of ours,
rendered in the app's own chrome and theme.**

### 2.2 `PromptDialog` — the single-text-input ask (NEW)

The **one** thing `ConfirmDialog` can't do is collect a value, which is why the codebase still had a
`window.prompt` for "Move file to (absolute path):". Add a **`PromptDialog`** — the same modal shell with a
single labelled `<input>` (pre-filled with an optional default, e.g. the file's current path), **Confirm

typed string or `null`.

Rendered in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/PromptDialog.tsx` — `PromptDialog`.

### 2.3 The imperative bridge — `confirmModal()` / `promptModal()` + one `ModalHost` (NEW, LOCKED)

Almost every native-dialog site is **imperative** — it lives inside a click handler and reads like
`if (!window.confirm(msg)) return;` or `const dest = window.prompt(msg, def);`. Rewriting each into a piece
of render-time React state would be invasive and error-prone across ~15 files. Instead we provide **two
promise-returning helpers** that any handler can **`await`**, backed by **one** `ModalHost` mounted once at
the app root (the exact single-slot bus pattern the app already uses for the setup wizard and the
transcribe-model consent dialog — [transcribe.ts](./transcribe_engine.mdx) `requestModelConsent`,
`setupWizard.ts` `requestStorageSetup`):

```ts
// lib/modals.ts
confirmModal(opts: {
  title: string; body?: ReactNode; confirmLabel?: string; cancelLabel?: string; danger?: boolean;
}): Promise<boolean>;                     // resolves true = confirmed, false = cancelled/Esc/backdrop

promptModal(opts: {
  title: string; label?: string; defaultValue?: string; placeholder?: string;
  confirmLabel?: string; validate?: (v: string) => string | null;   // return an error string to block Confirm
}): Promise<string | null>;               // resolves the typed value, or null on cancel
```

* A call **pushes a request onto the single-slot bus**; the mounted `ModalHost` renders the matching
  `ConfirmDialog` / `PromptDialog` and resolves the promise on the user's choice, then clears the slot.
* Handlers convert **mechanically**: `if (!window.confirm(m)) return;` → `if (!(await confirmModal({ title: m }))) return;`
  and `const d = window.prompt(m, def);` → `const d = await promptModal({ title: m, defaultValue: def });`.
* The host lives at the app root so a modal shows regardless of which page fired it; only **one** modal is
  ever on screen (a second request replaces/queues — single-slot).

Implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/modals.ts` — `confirmModal()` / `promptModal()` / `onModalRequested()` (the bus) · the provider `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/ModalHost.tsx` — `ModalHost` (mounted once in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/App.tsx`).

## 4. Why a modal, not a toast, for a decision (rationale, non-normative)

A **toast** is a passive receipt — it tells the user what already happened and vanishes. A **modal** is an
active gate — it stops and asks. A destructive or irreversible action (compress-with-trash, move, unpin,
remove) needs the **gate**, so it is a `ConfirmDialog`; the *result* of that action, once taken, is the
**receipt**, so it is a toast. Never invert them: never use a toast to ask a question the user must answer,
and never use a modal to announce a routine success.

## 5. The unified batch-confirm popup — the "great pop-up" (LOCKED)

The product owner's model: on the **View-one-repo** page, clicking the **Transcribable** metric tile opens
a large, well-made popup — the candidate audio/video list, checked by default, a **Continue** into the
checklist, uncheck anything, then a solid-blue **"Transcribe N files"** confirm that runs the work in the
background and shows the **Processing** surface ([task_tabs.mdx §5.1](./task_tabs.mdx)). *"That works
great."* The rule of this section: **that exact popup is the ONE surface every producing entry point
opens** — nothing produces transcripts or AI descriptions any other way.

### 5.1 What the popup is (LOCKED)

It is the **`WarningPopup`** ([warnings.mdx §4](./warnings.mdx)) fed a **transcribe** or **describe**
`WarningDef` whose `targets` are the candidate files:

* **Two-pane, ~80% of screen** ([warnings.mdx §4.0](./warnings.mdx), [transcribe_calc_engine.mdx §4](./transcribe_calc_engine.mdx)):
  left = the "what this is / why it matters" education + the media preview on row hover; right = the
  **candidate checklist, every row checked at open**.
* **Live count in the confirm button.** The solid-blue button reads **"Transcribe N files"** /
  **"Describe N files"** where **N is the live checked count** (`actionLabel` "Transcribe" + the popup's
  appended `— N files`); unchecking a row decrements it. Confirm is **disabled at zero checked**.
* **Apply = model/provider gate, then background enqueue.** Confirm runs the transcribe/describe **consent
  gate** ([transcribe_engine.mdx §3](./transcribe_engine.mdx) `withModelReady` for transcribe; the AI
  provider-key check for describe) — which may interpose the one-time model-download or the credentials
  dialog and then **resume** — and then **enqueues the checked paths as background jobs**
  (`api.transcribeEnqueue({ paths })` / `api.describeEnqueue({ paths })`), never an inline blocking run.
* **Processing surface + toast.** The popup closes; the [Processing](./processing.mdx) surface shows the
  jobs draining; the LOCKED toast fires with the queued count ([page_actions.mdx §2](./page_actions.mdx) —
  *"N files will have their Transcriptions created"*).

The transcribe / describe `WarningDef` builders (checked-by-default targets, "Transcribe/Describe N files"
button, gated background enqueue) already exist for the repo-scoped tiles in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/metricWarningDefs.tsx` — `buildTranscribeWarning()` / `buildDescribeWarning()`.

### 5.2 The candidate list comes from a plan/preview, not the enqueue count (LOCKED)

The repo tiles build their `targets` from `RepoDetail.files` (already loaded, filtered to
`transcribe === "could"` / `describe === "could"`). The **other** entry points — a **directory**, a **File
System column**, a **storage** — have no `RepoDetail`; they have a **scope** (a root path, or a checked
subset). For those, the frontend fetches a **preview plan** that returns the eligible candidate **file
list** (paths + sizes) under the scope — Rules 1 (scope) + 2 (skip-already-done) applied **on the server**,
exactly as enqueue does, but **without queuing anything** — and builds the same `WarningDef` from it.

* **New endpoints** (plan-only, no enqueue), mirroring the enqueue routes:
  * `POST /api/transcribe/plan` — body `{ paths?: string[]; root?: string }` → `{ files: {path, sizeBytes}[]; considered; alreadyDone; unsupported }`.
  * `POST /api/describe/plan` — same shape.
* Both walk-or-take the set with the **same `walkMedia()` / skip-already-done** logic the enqueue planners
  use ([page_actions.mdx §1.1/§1.2](./page_actions.mdx)), so the preview list and a later enqueue never
  disagree.
* **Empty preview** → the popup opens in its **all-clear** state ("Nothing to transcribe here — all N files
  already have transcripts") rather than a silent no-op, honoring the honest-outcome rule
  ([page_actions.mdx §2](./page_actions.mdx)).

The plan endpoints (walk the scope, drop already-done + unsupported, return the candidate list without
queuing) are implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` — `previewTranscribe()` (reuses `resolveCandidates()` + the already-transcribed filter, returns the list) · `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `previewDescribe()`; routes `POST /plan` on `transcribe.router.ts` / `describe.router.ts`; client methods `api.transcribePlan` / `api.describePlan` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts`.

### 5.4 The "Opening window…" spinner while the plan walks (NEW, LOCKED)

The plan preview (§5.2) has to **walk the scope's directory tree** to find the candidates — and on a large
subtree that is **slow**: the product owner opened a directory whose subtree held **~2,000 files** and the
popup took roughly **two minutes** to appear. During that walk, **nothing was on screen**, so the app
looked **hung**. That is unacceptable — a long-running open must always show it is doing something.

**The rule:** the instant a producing entry point is invoked (§6 — the metric tile builds its list from
already-loaded `RepoDetail.files`, but **every scope-walking entry point** — the page action-link, the
`⋮` / right-click menu on a directory, a File-System column) the app **immediately** shows an **"Opening
window…" loading modal with an animated spinner**, *before* awaiting the plan. When the plan resolves, that
loading modal is **replaced in place** by the real §5 batch popup. So the sequence the user sees is:

1. **Click** "Create AI descriptions" (or Create Transcriptions) on a directory.
2. **At once** — a small centered modal appears: an **animated spinner** + **"Opening window…"** headline
   + a one-line sub explaining the wait (e.g. *"Scanning this folder for files — large folders can take a
   moment."*). This appears **synchronously**, before the `POST /api/{describe,transcribe}/plan` request is
   even awaited, so there is **never** a blank, hung-looking gap.
3. **When the walk finishes** — the spinner modal is swapped for the full batch popup (the candidate
   checklist, the "Describe/Transcribe N files" confirm), or, if the scope had no candidates, the popup's
   all-clear state (§5.2).
4. **On error** — the spinner modal closes and a toast explains the failure (§5.2's catch).

Additional rules:

* **Cancellable.** The spinner modal has a **Cancel** (and `Esc` / backdrop click) that closes it. Because
  the walk is a fire-and-forget request the user abandoned, a **generation guard** in the launcher makes the
  eventual plan result a **no-op** — cancelling the spinner must **not** later pop the real window open.
* **Only for scope-walking opens.** The metric tile / header primary build their list from data already in
  memory (no walk), so they need no spinner — they open the popup directly. The spinner is specifically for
  the **plan-fetch** opens (`openTranscribeBatch` / `openDescribeBatch`), which is exactly where the
  multi-minute walk lives.
* **One host, one slot.** The spinner and the popup share the **same single global host** (§5.3): the
  loading modal and the batch popup are two states of the one `BatchPopupHost` bus, so there is never a
  spinner *and* a popup on screen at once, and the swap is seamless.

Implemented in code (NEW):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/batchPopup.ts` — the bus carries a
`loading | popup | null` state; `openTranscribeBatch()` / `openDescribeBatch()` emit the **loading** state
(with a generation token) **before** awaiting `api.{transcribe,describe}Plan`, then emit the **popup** state
(guarded by the token) when it resolves · the loading modal + swap render in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ui/BatchPopupHost.tsx` — `BatchPopupHost`
(renders a spinner "Opening window…" card for the loading state, the `WarningPopup` for the popup state).

## 7. The File-System hub — where a sub-directory is selected (LOCKED reference)

On the **File System** column browser ([file_system.mdx](./file_system.mdx)), the popup's scope follows the
**deepest (rightmost) open column** — the same [scope-containment rule (§5.1)](./file_system.mdx) the
action-links row already obeys: **downward-only**, never a parent or sibling. So:

* Clicking **Create Transcriptions** in the action-links row with a sub-directory open as the deepest column
  opens the popup over **that directory's subtree only**.
* Right-clicking (or ⋮-ing) a **selected sub-directory row** in a column opens the popup over **that
  row's** subtree (§6.4).
* Selecting a **file** row in the deepest column and choosing **Create Transcriptions** opens the popup with
  that one file checked (§6.4).

Nothing about the popup differs by entry point — same list, same confirm, same gate, same Processing. Only
the **scope** the plan preview walks changes. [file_system.mdx](./file_system.mdx) owns the column-browser
mechanics; this file owns what the popup does once opened.

## 8. Acceptance criteria

1. **No native dialogs (§1–§3).** A grep of `packages/frontend/src` for `window.confirm`, `window.alert`,
   and `window.prompt` returns **zero** matches. Every former site opens a `ConfirmDialog` / `PromptDialog`
   / pop-over / batch popup per the §3 catalog.
2. **`PromptDialog` exists (§2.2)** and replaces the move-file `window.prompt` in both `EntityMenu` and
   `MediaViewer` — a themed modal with one text input, Confirm disabled while empty, `Enter` submits,
   `Esc`/backdrop cancels (returns `null`).
3. **`confirmModal()` / `promptModal()` + `ModalHost` (§2.3)** back the imperative conversions: a click
   handler `await`s a real HTML modal; exactly one modal shows at a time; the host is mounted once at the
   app root.
4. **The unified batch popup (§5)** is `WarningPopup` fed a transcribe/describe `WarningDef` — candidates
   checked by default, a live **"Transcribe N files"** / **"Describe N files"** solid-blue confirm (disabled
   at zero), the model/provider consent gate on Apply, background enqueue, the Processing surface, and the
   LOCKED toast.
5. **The plan/preview endpoints (§5.2)** `POST /api/transcribe/plan` and `POST /api/describe/plan` return
   the eligible candidate file list under a scope (Rules 1+2 applied, nothing queued); the scope→popup
   builders in `lib/batchPopup.ts` build the popup from that list, so preview and later enqueue never
   disagree.
6. **A global popup host (§5.3)** lets the action-links row and the ⋮ / right-click menu open the same
   popup the metric tile opens.
7. **All four entry points open the one popup (§6):** the metric tile, the header primary (already), and —
   NEW — the **Create Transcriptions** / **Create AI descriptions** page action-link and ⋮ / right-click
   menu item. No producing action is a fire-and-forget toast or a `window.confirm`.
8. **The page action-link change is scope-preserving (§6.1):** the only change is that the enqueue+toast
   move from the link's click to the popup's Confirm; Rules 1–3 and the first-time-storage gate are
   unchanged.
9. **File-System scope containment (§7):** the popup opened from the FS action-links row / a selected
   sub-directory / a selected file is scoped to the deepest open column / the selected row — downward only,
   never a parent or sibling.
10. **The "Opening window…" spinner (§5.4):** every scope-walking open (`openTranscribeBatch` /
    `openDescribeBatch`) shows an animated-spinner **"Opening window…"** modal **synchronously**, before the
    `/plan` request is awaited, so a slow multi-minute tree walk never looks hung; the spinner is **replaced
    in place** by the batch popup when the walk resolves (or closed + toasted on error); a **Cancel** on the
    spinner makes the eventual plan result a **no-op** via a generation guard. The metric-tile / header-primary
    opens (no walk) skip the spinner.
