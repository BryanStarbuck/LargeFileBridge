# LargeFileBridge — Company Storage

**Company storage** is a **shared pile of a company's large files** — the big media/assets a team needs
in common. There is **one company storage per company**, and a user may belong to **more than one
company**, so several can coexist on the same machine.

Read the shared model first: the storage family, the `storage.yaml` schema, the tracking
layout, cloud clones, and media analysis are defined once in **[storages.mdx](./storages.mdx)**. This spec
covers only what is **specific to company storage**.

Siblings this builds on and does not repeat:

* [storages.mdx](./storages.mdx) — the umbrella: `storage.yaml` (§3), the tracking layout (§4), clones (§5),
  media analysis (§6).
* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — **owns the rule** that decides
  whether a root uses a `.lfbridge/` directory at all. A company storage is a **dedicated LFB file repo**
  (an **SDL**), so it has **no `.lfbridge/`**: `.lfbridge/` is a **working-repo-only** concept, and the
  **SDL root itself *is* the `.lfbridge` area**. Everything below places files at the company root
  accordingly.
* [syncable_data_location.mdx](./syncable_data_location.mdx) — the **SDL** model this spec now rests on:
  the company root is the storage's **Syncable Data Location** (small text that travels), and the big
  files live in a **list of mapped source directories** (§1.1), not under the root.
* [devices.mdx](./devices.mdx) — the per-computer **graft** that resolves each mapped directory to this
  machine's local path (§1.1).
* [storage_repo.mdx](./storage_repo.mdx) — the canonical directory-based storage; company storage is the
  same shape with a `company` block and a shared (multi-user) audience.
* [storage_personal.mdx](./storage_personal.mdx) — the single-user private counterpart.
* [repo_company_mapping.mdx](./repo_company_mapping.mdx) — the owner mapping whose §5 reassign the company
  settings' Owned-repos list (§7.1) reuses, and [repo_owner_propagation.mdx](./repo_owner_propagation.mdx) —
  how a reassignment **into** this company travels (an `owner_map.yaml` assertion) and is consent-gated on
  other members' computers (§7.1).
* [storage_settings.mdx §4c](./storage_settings.mdx#4c-owned-repos--the-repos-mapped-to-this-storage) —
  where the company's Owned-repos list + per-row reassign actually render (§7.1 points there).

## 1. Anatomy of a company storage

The company root is the storage's **[Syncable Data Location](./syncable_data_location.mdx)** — the small,
travelling text that describes the storage. The company's **big files are not stored under it**; they
stay in their natural directories and are brought into scope by a **list of mapped source directories**
(§1.1).

```
~/BGit/Bryan_git/act3_large_files_bridge/     ← ONE company's storage root = its Syncable Data Location ("ACT3")
│                                              ← this root IS the .lfbridge area — an SDL has NO .lfbridge/
├── LargeFilesBridge_SyncList.yaml            ← the portable list (sync_list.mdx), committed to git here
├── storage.yaml            ← name + type: company + company block (storages.mdx §3)
├── mapped_dirs.yaml        ← §1.1 the company's mapped source directories (syncable_data_location.mdx §3)
├── files.yaml              ← per-file fingerprint index (storages.mdx §4.1)
├── manifest.yaml           ← large-file → IPFS CID + pin state
├── devices/                ← the device registry + per-device graft (devices.mdx)
├── bookmarks.yaml          ← bookmarked files, travels (syncable_data_location.mdx §4.4)
├── analysis/               ← per-file media analysis + compression records (storages.mdx §6, SDL §4)
└── …NO big files here — they live in the MAPPED directories (§1.1)…
```

* **Root:** one **SDL** directory per company, e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` — a small
  Git repo of metadata. A user in three companies has three such roots, each its own storage. **The company's
  metadata sits at the root — there is no `.lfbridge/` here.** `.lfbridge/` is a **working-repo-only**
  concept (where LFB is a guest and quarantines its files); a company storage is a **dedicated LFB file
  repo**, so the **root itself *is* the `.lfbridge` area** and hiding its files one level down would buy no
  separation. The rule, its rationale, and the reserved root names are owned by
  [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — this spec only applies it. Because this
  root is a **dedicated tracking-data Git repo the user made for LFB**, when it is itself a git working
  tree LFB **auto-adopts it as the storage's Git backbone** and **commits + pushes** this computer's
  `devices/<self>.yaml` there on first write and every update — no manual dedicated-repo setting needed
  ([devices.mdx §11.2.1](./devices.mdx#1121-a-company--personal-tracking-repo-auto-adopts-its-own-git-root--commit--push-first-write-and-every-update-locked-learned-from-a-defect)).
* Because it is **shared across members**, both the SDL (over Git/backbones) and the mapped big files
  (over IPFS) are carried across **the members' computers** — a company storage is the multi-user case (peers
  here include teammates' machines, within the company's allow-list).

### 1.1 A company maps a LIST of source directories (LOCKED)

The defining trait of a company storage: in its settings the user lists **the directories of their hard
drive that belong to this company**, and **everything recursive under each is in scope**. A company with
seven directory hierarchies adds all seven; LFB scans each subtree for big files, tracks them, pins the
opted-in ones over IPFS, and stores their derived artifacts in the SDL — **without relocating a single
byte**.

* The **logical** list travels in `<root>/mapped_dirs.yaml`
  ([syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked));
  each mapped directory has a stable **key**.
* The **absolute local path** of each mapped directory is **per-computer** — resolved by that machine's
  **graft** ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)).
  So the same company storage grafts into `~/Movies/act3` on a laptop and `/Volumes/Media/act3` on a
  tower, and each computer pulls files to its own location.
* This is the generalization of a **repo** storage (one hierarchy, path-resolved by Git) to **many**
  hierarchies path-resolved by the device graft. A user adds/removes mapped directories from the
  company's [storage settings](./storage_settings.mdx).

## 2. `storage.yaml` for a company

```yaml
# ~/BGit/Bryan_git/act3_large_files_bridge/storage.yaml   (at the SDL root — no .lfbridge/ segment)
name: "ACT3 Company Files"
type: company
created: 2026-05-01T10:12:00Z
company:
  company_name: "ACT3 AI"     # which company — the label shown in the Storages tab
  owner_slugs: ["ACT3ai"]     # §8.4.4 the forge orgs this company claims — binds remotes to this storage
  # company-scoped child properties:
  division: "Film"            # optional
  allow_list: ["bryan@whitehatengineering.com", "…"]   # who may carry this storage
  drive_folder: "Shared drives/ACT3"                    # optional shared-drive anchor
clones:
  google_drive: "Shared drives/ACT3/LFB"   # a company storage often mirrors to a SHARED drive
  dropbox: null
```

* **`type: company`** with a **`company`** block. **`company_name`** names which company; child properties
  scope the storage (division, allow-list of members, shared-drive anchor, …).
* A user in multiple companies has **multiple** company storages, each with its own `company_name`.

## 3. Media analysis + cloud clones

Company storage produces the [storages.mdx §6](./storages.mdx) **media-analysis** outputs into
`<root>/analysis/` — and because these are shared, a teammate who pulls the storage gets the
transcript / description / visuals-by-time without re-running them. A company storage **often clones to a
shared Google Drive / Dropbox** ([storages.mdx §5](./storages.mdx)) so members without LFB can still reach
the files; recorded in the `clones` block.

## 5. Repos auto-derived from an org appear as this company

A company storage is created two ways: **explicitly** (the user sets one up for a shared media pile, §1), or
**automatically** — when a repo's git remote names a forge **organization**, LFB derives a company from that org
and **auto-creates this storage** to hold the grouping ([repo_company_mapping.mdx §3/§6](./repo_company_mapping.mdx#6-auto-creating-the-company--and-its-friendly-name)).

* Every repo whose remote owner maps to this company (e.g. `git@github.com:ACT3ai/…`) **groups under this company**
  in the Storages disclosure and the Repos list ([repo_company_mapping.mdx §7.1](./repo_company_mapping.mdx#71-grouping-in-the-left-bar-and-repos-list-locked)).
  The mapping is derived offline from the remote URL — **no** forge network call.
* An **auto-created** company is a **lightweight** entry at first: it exists to group repos and to be **renamed**
  (§6). It has no mapped source directories (§1.1), no allow-list, and no sync repo (§7) until the user adds
  them. **A repo-group company and a shared-media-pile company are the same storage kind** at different levels of
  configuration — filling in §1.1 / the allow-list / the sync repo (§7) promotes the same storage from "just a
  group" to a full shared pile.
* The company's **`id`** is the stable id `resolveRepoOwner` stamps on each owned repo's `owner.companyId`
  ([repo_company_mapping.mdx §1.1](./repo_company_mapping.mdx#11-the-repo-owner-field-locked-shape)), so grouping and the
  company row agree.

Creating-if-absent the company storage for a derived org owner is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureCompanyForOwner(slug)` (creates via `ensureStorage(root, "company", …)` when missing), surfaced by `listStoragesPage()`.

## 6. The friendly display name

An org slug from a remote is often **cryptic** — `ACT3ai`, `acme-eng-2`, `~sircmpwn`. So a company storage carries
a user-settable **friendly display name**, and the product shows **that** name **everywhere** — the Storages
left-bar child, the Repos grouping header, and each owned repo's Owner control
([repo_company_mapping.mdx §5.1](./repo_company_mapping.mdx#51-the-reassign-control)).

* **Where it is stored.** The friendly name **is** the company descriptor's **`company_name`** in
  the root `storage.yaml` (§2) — the same field that labels the Storages child. On auto-creation `company_name`
  is seeded to the **owner slug** (e.g. `ACT3ai`); the rename simply **overwrites** it with the human name
  (e.g. `ACT3 AI`). There is no second field: the descriptor's `company_name` is the single source of truth for
  the label.
* **Fallback, never blank.** Until renamed, the displayed name **is** the slug — never blank and never the raw
  remote URL. `buildRow()` already falls back to `prettyName(base)` when `company_name` is absent, so a company is
  always labeled.
* **The rename flow.** From the company's [storage settings](./storage_settings.mdx) the user edits a **Display
  name** field; saving writes `company_name` and the label refreshes everywhere on the next read (it is derived,
  not frozen onto repos — [repo_company_mapping.mdx §9](./repo_company_mapping.mdx#9-error--empty-cases)).
* **Travels with the storage.** Because `company_name` lives in the committed `storage.yaml`, the friendly name
  **travels to teammates** who carry this company storage — everyone sees the same human name, not the slug.

Reading and writing the company's friendly `company_name` (the display name) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `readDescriptor` / `writeDescriptor` (the `company.company_name` field), edited from the settings page and rendered by the Storages tab.

## 7. The company sync repo

A company can optionally configure a **dedicated sync repo** — a git repo the company sets up *"only for the
Large File Bridge webapp to sync changes"* (not for the big media bytes) — and, when it is set, a member's
repos owned by this company **additionally mirror** their **LFB tracking state** into it so teammates share it.

* **What it is.** A git repo (the company's own — e.g. `git@github.com:ACT3ai/act3_lfb_state.git`) whose only job
  is to carry LFB's small tracking text between the company's members. It is the **company** analogue of the
  Personal sync repo ([storage_personal.mdx §14](./storage_personal.mdx#14-personals-optional-sync-repo)) and
  of this company storage's own SDL Git backbone (§1) — but scoped to **member repos' tracking**, not this
  storage's media pile.
* **What travels there.** Per the placement policy ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
  [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)), an owned repo's tracking state — `repo_storage.yaml`,
  `files.yaml`, the per-file sidecars, `history/<device>.txt`, `decisions.yaml`, and `manifest.yaml` — lives in
  **Local Storage always** at `~/T/_large_files_bridge/repos/<repoKey>/`. **When the repo's owner is this company,
  this company has a sync repo configured, AND the repo's per-repo toggle is on**
  ([repo_settings.mdx §2.9](./repo_settings.mdx)), that same state is **also** written under
  **`<syncRepo>/repos/<repoKey>/`** — **in addition to** Local Storage, not instead of it — so it commits + pushes
  to the company repo and every teammate on the same company sees it. Path-mirrored by `<repoKey>` so two members
  never collide. Logs never travel here ([artifact_placement_policy.mdx §8](./artifact_placement_policy.mdx)).
* **How the user configures it.** From the company's [storage settings §4b](./storage_settings.mdx#4b-the-sync-repo--an-additive-mirror-of-owned-repos-tracking-state)
  the user picks a **sync repo** and supplies it as a **local file path** or an
  **HTTP(S) URL** — the same two remote shapes and OS-credential-helper rules as the Personal Git backbone
  ([storage_personal.mdx §5](./storage_personal.mdx#5-turning-the-git-backbone-on--local-path-vs-https-url),
  [git_backbone.mdx](./git_backbone.mdx)). LFB runs the same fetch → auto-merge → commit → push cycle against it.
* **Default: OFF.** Absent a configured sync repo (or with the per-repo toggle off), member repos keep their
  tracking in **Local Storage only** — real and honored locally, simply not yet shared. Turning it on is an
  explicit, per-repo opt-in.
* **Precedence.** A repo manually reassigned to this company ([repo_company_mapping.mdx §5](./repo_company_mapping.mdx#5-manual-reassignment--the-user-always-wins-locked))
  mirrors to this sync repo just like an auto-owned one — ownership (not how it was derived) decides which sync
  repo a repo can mirror to.

Resolving the company's sync repo and the additive mirror root for an owned repo (`<syncRepo>/repos/<repoKey>/` or null) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `readStorageSettings()` (the company's `sync_repo` block) consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)`, whose mirror is written by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)`.

### 7.1 The company's owned-repos list & reassign

This company's [settings page](./storage_settings.mdx) surfaces the **repos owned by this company** with a
per-row **reassign** control, so a member manages the whole company's repo membership from one place rather
than one repo at a time. The list and the control are defined in full in
[storage_settings.mdx §4c](./storage_settings.mdx#4c-owned-repos--the-repos-mapped-to-this-storage); this
section only ties it to company storage:

* **What it lists.** Every repo whose resolved owner is **this company** (§5,
  [repo_company_mapping.mdx](./repo_company_mapping.mdx)) — auto-derived from the forge org **or** manually
  reassigned — appears as a row in the company settings' Owned-repos table.
* **Reassigning INTO this company records the travelling assertion.** When a member reassigns a repo **to
  this company** (and this company has a **sync repo**, §7), LFB commits an **ownership assertion** into that
  sync repo's `owner_map.yaml` ([repo_owner_propagation.mdx §2](./repo_owner_propagation.mdx#2-where-the-assertion-lives--owner_mapyaml-in-the-company-sync-repo-locked)),
  which **travels** to the other members and is applied on their machines **only through the explicit
  review-and-consent page** ([repo_owner_propagation.mdx §4](./repo_owner_propagation.mdx#4-the-review-page--member-2-reviews-and-consents-locked))
  — never silently. Reassigning a repo **out** of this company tombstones its assertion.
* **No bytes move.** Reassignment is grouping/tracking-target metadata only; it changes **which sync repo**
  an owned repo's Category-B tracking mirrors to (§7), never the files themselves
  ([repo_company_mapping.mdx §1](./repo_company_mapping.mdx#1-what-an-owner-is-locked)).

The Owned-repos read (`getOwnedRepos(storageId)`) and the reassign endpoint (`POST /api/repos/:repoId/owner`, which asserts into this company's sync repo via `assertCompanyOwnership()`) are the same as cited in [storage_settings.mdx §4c](./storage_settings.mdx#4c-owned-repos--the-repos-mapped-to-this-storage) and [repo_owner_propagation.mdx §2](./repo_owner_propagation.mdx#2-where-the-assertion-lives--owner_mapyaml-in-the-company-sync-repo-locked).

## 8. The load-bearing use case — one file, two computers (LOCKED)

This is **the** use case company storage exists to serve, written as one concrete file's journey. Everything
above (§1 the SDL, §5 the auto-derived company, §7 the sync repo) is machinery; **this section is the
product promise those parts must add up to.** If this walkthrough does not work end to end, the company
storage is decorative.

### 8.1 The scenario

One user, two of their own computers, one company (`ACT3ai`), one working repo, one big file:

| | |
| --- | --- |
| **Working repo** | `~/BGit/Bryan_git/charlie-kirk` — remote `https://github.com/ACT3ai/charlie-kirk.git` |
| **The file** | `videos/2079054276320440416.mp4` — 5.3 MB, **git-ignored** (`.gitignore:27` → `videos/`) |
| **Company storage (SDL)** | `~/BGit/Bryan_git/act3_large_files_bridge` — remote `github.com/ACT3ai/act3_large_files_bridge` |
| **Computer A ("the Tower")** | device `bryan-mac-pro` — has the file, pinned it, CID `bafybeihf5o…k7tmi` |
| **Computer B ("the laptop")** | device `bryanstarbuck-macbook-pro` — has the repo cloned, does **not** have the file |

Both computers already appear in `<companySDL>/devices/`, so the **transport is proven** — the SDL's git
backbone fetches, merges, commits and pushes between them today. What fails is that **nothing but device
state rides it.**

### 8.2 What the user must see (the acceptance narrative)

On the **laptop**, with the file still absent from its disk, opening
[View One Repo](./one_repo.mdx) for `charlie-kirk`:

1. **The file is a row in the file list** — by name, size, and CID — even though **no local file exists**.
   It renders in the **red / "not on this computer"** state ([one_repo.mdx §4.9](./one_repo.mdx),
   §8.5 below), because the user chose to sync it and this machine does not have it yet.
2. **It is counted in the `Pull down` metric tile** — the tile whose whole job is "a computer of yours has
   bytes you don't."
3. **It is counted in the `Add to IPFS` metric** when this computer has not decided it — the row is a
   recommendation to bring the file here and pin it.
4. **It appears in the [To Do](./to_do.mdx) tab** as a `pull_down` recommendation in this company's batch.
5. **Acting on any of those pulls the bytes down over IPFS** and the row turns from red to a normal local
   row.

None of the five works today. §8.3 names why, link by link.

### 8.3 The chain, and where each link was broken

The journey has **six links**. Five of the six were already built; the chain still delivered nothing
because **every link has to hold**, and three did not.

| # | Link | Status found | The break |
| --- | --- | --- | --- |
| 1 | **Pin + record.** Tower pins the file and records `path → cid + pinned_by` in its manifest | ✅ works | — |
| 2 | **Identity.** Both computers agree on *which repo* a mirrored manifest belongs to | ❌ **broken** | keyed by `sha1(absolute path)` — a different value on each computer (§8.4) |
| 3 | **Ownership + adoption.** The repo resolves to *this company's* SDL as its sync repo | ❌ **broken** | ownership resolved by **path containment** (→ Personal), and the mirror is **default OFF** (§8.4) |
| 4 | **Send.** The manifest is copied into the SDL and **committed + pushed** | ✅ works *once fed* | `mirrorToSyncRepo` copies, and `commitAndPush` stages pathspec-lessly (`git add -A`), so anything written to the SDL **does** ship — it just needs a **trigger** after a mirror write instead of waiting for the 10-minute device worker |
| 5 | **Receive.** The laptop merges the pulled manifest into its own tracking state | ❌ **broken** (since fixed) | `reconcileFromSyncRepo()` had **zero callers**, and it *last-writer-copied* rather than merged — it is now a real per-entry union merge (`mergeManifests`) with callers on both pin passes |
| 6 | **Surface.** The absent-but-known file becomes a row, a metric, a To-Do item | ⚠️ partial | `missingPinnedFromPeers()` + the `Pull down` tile + the To-Do batch all exist — they were simply always fed an empty manifest; the **file list itself** has no way to show a non-local file (§8.5) |

**The diagnosis in one line:** the product had a working road and a working destination, and never put the
cargo on the truck.

Evidence on the Tower at the time of writing: the file's CID is recorded locally in
`~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml`, the repo's config says `sync_repo: enabled: false`,
and `<companySDL>/manifest.yaml` reads `files: []` with a git log containing nothing but `LFB: device state`.

### 8.4 The four rules this use case LOCKS

#### 8.4.1 A repo's shared identity is its REMOTE, never its path (LOCKED)

The subtree a repo's tracking mirrors to is **`<syncRepo>/repos/<repoUid>/`**, where **`repoUid` is derived
from the repo's normalized git remote URL** — not from its absolute path.

* **Normalization.** Lowercase; drop scheme, credentials, and any `.git` suffix or trailing slash; fold
  `git@host:owner/name` and `https://host/owner/name` to the same **`host/owner/name`**. So
  `https://github.com/ACT3ai/charlie-kirk.git` → `github.com/act3ai/charlie-kirk`. `repoUid` is a short hash
  of that string.
* **Why this is load-bearing.** The old key was `sha1(path.resolve(root))`. The same repo lives at
  `/Users/bryan/BGit/…` on the Tower and `/Users/bryanstarbuck/BGit/…` on the laptop, so the two computers
  wrote and looked for **different directories** in the same sync repo. The manifest could have travelled
  perfectly and still never been found. **A machine-local value must never key machine-shared data.**
* **No remote → no shared identity.** A repo with no git remote **cannot** mirror. That is honest, not a
  failure: LFB says so in the repo's settings ("no remote — tracking stays on this computer") rather than
  silently writing to a subtree no other computer will ever read.
* **The Local-Storage path is unchanged.** `~/T/_large_files_bridge/repos/<repoKey>/` stays keyed by path —
  it is machine-local state and *should* be. Only the **mirror** subtree changes key. The two keys coexist
  deliberately: one names "this repo on this computer", the other names "this repo, anywhere".

Deriving the shared identity and the mirror subtree is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `normalizeRemoteKey()` / `repoUidFor()` (re-exported from the leaf `repo-identity.ts`) (the remote-derived key) and `resolveStateSyncRepo()` (which now joins `repos/<repoUid>/`).

#### 8.4.2 The company SDL IS the sync repo, and the mirror is ON by default (LOCKED)

This **supersedes §7's "Default: OFF"** for the case where a sync repo can be resolved at all.

* **Auto-adoption.** A company storage whose root is a git working tree (the normal case — an SDL *is* a
  git repo, §1) **is** the company's sync repo. The user does not configure a second repo to make the
  product's core promise work. An explicitly configured `sync_repo` in the company's
  [storage settings](./storage_settings.mdx) still **wins** when present; auto-adoption is the fallback,
  not an override.
* **This is the rule the Git backbone already follows.** `getGitBackboneRemote()` **already** auto-adopts an
  SDL's own git root when no dedicated repo path is configured — which is precisely why device state has
  been travelling between the two computers all along ([devices.mdx §11.2.1](./devices.mdx)). The sync repo
  simply stops disagreeing with the backbone about what the company's repo *is*. **One SDL, one git root,
  one answer.**
* **Ownership comes from the remote, not from path containment.** `charlie-kirk` is **not** inside
  `act3_large_files_bridge`, so a containment test resolves it to **Personal** — the wrong storage. The
  owning storage is the one the repo's **remote org** maps to (§5,
  [repo_company_mapping.mdx](./repo_company_mapping.mdx)): remote org `ACT3ai` → the `ACT3ai` company
  storage. Path containment remains the fallback for repos with no remote org.
* **Default ON.** When an owning storage with a git-backed SDL resolves, the per-repo mirror is **on**. The
  per-repo toggle ([repo_settings.mdx §2.9](./repo_settings.mdx)) still exists and still wins — it is now an
  **opt-out**, not an opt-in.
* **Why the default flipped.** Default-OFF meant the shipped configuration of the product did not do the
  thing the product is for. The mirrored payload is **small text** (a manifest, a decisions ledger,
  sidecars) whose entire purpose is to travel; there is no bandwidth, privacy, or safety argument for
  withholding it from the user's *own* SDL — the same SDL that is already committing and pushing this
  computer's device fingerprint. A default that has to be found and switched on before anything works is a
  bug wearing a setting's clothes.
* **Flipping the default is not enough on its own — the old default was PERSISTED.** The field used to carry
  a schema-level `false`, and because a unit's config is rewritten whole on every update, that `false` was
  written to disk for **every** repo (178 of them on the machine this was traced on). Simply making the field
  optional would have left every existing repo reading as an explicit opt-out, so the feature would have
  worked **only on a fresh install** — the most deceptive possible outcome. A **one-time migration clears a
  persisted `false`**, because a value the user was never shown and never chose is a default, not a decision.
  An explicit `true` is left untouched, and a user who really wants the mirror off can turn it off after the
  migration has run once.
* **Nothing about bytes changes.** This decides where *tracking text* travels. Media bytes still move only
  over IPFS, only for files the user decided to sync ([decisions.mdx](./decisions.mdx)).

Resolving the owning storage from the repo's remote and auto-adopting its SDL as the sync repo is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveOwnerDedicatedRepo()` (now remote-owner first, containment second), consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `ensureSyncRepoMarker()`.

#### 8.4.3 Receiving is a MERGE, never a copy (LOCKED)

An incoming mirrored manifest is a set of **claims from another computer**
([sync_list.mdx §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)). Reconciling it into
Local Storage therefore **merges per file entry** and never overwrites the local manifest wholesale:

* **Union by `path`.** An entry present on either side survives. An entry **absent** from the incoming copy
  is **never** treated as a delete — absence is untrusted.
* **Union `pinned_by`.** The two computers' pin claims are **added together**, never replaced. A
  last-writer copy would erase the Tower's claim on the laptop and with it the entire "a peer has this"
  signal — the exact signal this use case is built on.
* **Conflicting CID for the same path** resolves by the newer `modified_at`; ties keep the local value.
* **This computer's own claim is re-derived from its real pinset**, never from what arrived
  ([ipfs.mdx](./ipfs.mdx)): a peer's manifest cannot make this machine believe it holds bytes it does not.
* **Reconcile runs on every backbone pull** for the SDL, not on demand. A merge that only happens when
  someone clicks something is a merge that does not happen.

Merging a pulled sync-repo subtree into Local Storage is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `reconcileFromSyncRepo()` (a per-entry union merge over `mergeManifests()`), called from the repo pin pass in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinRepoFolder()`, and from the SDL git cycle (`syncStorageTextInner()` / `pinStorageUnitInner()`) via `reconcileMirroredRepos()` after fetch + merge.

#### 8.4.4 Binding a forge org to a company storage — `owner_slugs` (LOCKED)

§8.4.2 needs to turn the remote's org slug (`ACT3ai`) into **a specific company storage row**. Today nothing
does: `deriveOwnerForRemote()` returns `companyId: null`, and the only join is a best-effort normalized
name comparison that fails the moment the directory is named `act3_large_files_bridge` while the org is
`ACT3ai` — an off-by-two-letters mismatch silently costing the user the entire feature.

* **The binding is explicit and it travels.** The company descriptor (§2) carries
  **`company.owner_slugs: ["ACT3ai", …]`** — the forge orgs this company storage claims. It lives in the
  committed `storage.yaml`, so once one member binds an org, **every** member's computer resolves it the
  same way. A company may claim several orgs; an org belongs to at most one company.
* **How a slug resolves, in order.** (1) An **explicit `owner_slugs` entry** (case-insensitive) — always
  wins. (2) A **normalized name match** against the company's `company_name`, row name, or root slug (the
  existing best-effort rule). (3) **A single company storage** on this computer takes an unclaimed org — the
  overwhelmingly common case is one company, and refusing to guess there means the product does nothing for
  the user who has exactly one. More than one company and no match ⇒ **unresolved**, and LFB says which orgs
  are unclaimed rather than guessing.
* **A resolution by rule 2 or 3 is written back as an `owner_slugs` entry.** The heuristic runs **once**;
  from then on the binding is a recorded fact the user can see, edit, and share — not a guess re-made on
  every pass. This is how a fuzzy match becomes an auditable one.
* **The user can always correct it** from the company's [storage settings](./storage_settings.mdx), and a
  **manual owner override** on a repo ([repo_company_mapping.mdx §5](./repo_company_mapping.mdx)) still
  beats every rule above — the user always wins.

Resolving a forge-org slug to a company storage row (and recording the binding) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureCompanyForOwner(slug)` (the function §5 cites; it exists and implements all three rules above, with heuristic write-back in `recordOwnerSlug()`), backed by the descriptor's `company.owner_slugs`.

### 8.5 Remote-only rows — a file in the list that is not on this disk (LOCKED)

Today **every** row in the One-Repo file list comes from the scanner's local disk walk, so a file this
computer does not have **cannot be shown**. That makes the most important row in this use case invisible.

* **A `FileRow` may be `remote-only`.** The row carries **`presence: "local" | "remote-only"`** (absent ⇒
  `"local"`, so every existing row and reader is unchanged). A `remote-only` row is composed from the
  reconciled **manifest** entry, not from a `statSync`: name, `path`, `sizeBytes`, `cid`, and `peers` all
  come from the manifest, and `addedByDevice` names the peer that has it.
* **Four conditions, all required** — a manifest entry becomes a remote-only row only when it has a **CID**,
  is claimed by **at least one device that is not this one**, has **no scanned candidate**, and has **no
  file on this disk**. The peer-claim condition is what keeps a stale self-only entry — a file *this*
  computer deleted on purpose — from resurrecting as a row that offers to pull bytes nobody has. Absence
  still deletes nothing (§8.4.3); it simply stops being offered.
* **It renders red**, consistent with the existing pin-icon rule ([one_repo.mdx §4.9](./one_repo.mdx)):
  decided-to-sync but not pinned here **is** the red state; a remote-only row is the purest case of it.
  Its row tooltip reads *"On {device} — not on this computer yet."*
* **What it counts toward.** `Pull down` **always**. `Add to IPFS` when this computer has not decided it.
  It counts toward **no space, compression, or git-ignore metric** — there are no local bytes to reclaim,
  compress, or ignore — and toward **no analysis metric** (`transcribe` / `describe` / `ocr` are `na`):
  offering to transcribe a file whose bytes are absent would queue work that cannot run.
  This mirrors how `analysisOnly` rows are excluded from the large-file metrics
  ([one_repo.mdx §4.1](./one_repo.mdx)).
* **Its actions are exactly one:** *pull it down*. Compress / transcribe / describe / OCR / git-ignore are
  **disabled** on the row and say why. Once pulled, the next scan makes it an ordinary local row.
* **It is never a "missing file" alarm.** A remote-only row is a **normal, healthy** state — it is how a
  second computer learns what exists. The red says *"available, not here yet"*, not *"lost"*.
* **Deletion is still not inferred.** A file that vanishes from a peer's manifest does not delete anything
  here (§8.4.3); it simply stops being offered.

Composing remote-only rows by joining the reconciled manifest against the scanned candidates is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `composeFileRows()` (appends manifest entries with no local candidate and no local file), with the type in `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` — `FileRow.presence`.

### 8.6 Two manifests must not disagree

A repo's CID/pin facts are written to **two** files today — the unit manifest at
`~/T/_large_files_bridge/pin/r/<folder>/manifest.yaml` (which the file rows read) and the Local-Storage
tracking manifest at `~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml` (which the pull-down reader
and the mirror read). **Reconcile must land in both**, or the laptop counts a `Pull down` it cannot show a
row for — or shows a row it cannot pull.

Likewise the pull action must read the **same** manifest the pull-down list was built from. Reading the
SDL-only committed manifest for a *working repo* finds nothing and fails every file in the batch — a
defect this section closes by naming the single reader.

Reading the one manifest both the pull-down list and the pull action agree on is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `missingPinnedFromPeers()` and `pullMissing()` (both `readRepoTrackingManifest`).

Found while tracing the chain; each one alone was enough to keep the file invisible on the laptop. The

* **The SDL's own manifest goes through the storage-kind base directory.** `committedManifestPath()` is

## 9. Acceptance criteria

1. There is **one company storage per company**, rooted in its own directory (e.g.
   `~/BGit/Bryan_git/act3_large_files_bridge/`), which is the storage's **Syncable Data Location** —
   small travelling text tracked **at the root itself**. It is a **dedicated LFB file repo (an SDL)**, so it
   has **no `.lfbridge/` directory at all** — `.lfbridge/` is working-repo-only and the SDL root **is** the
   `.lfbridge` area ([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)); **no big files
   live under the root**.
2. Its root `storage.yaml` has **`name`** + **`type: company`** and a **`company`** block carrying
   **`company_name`** plus company-scoped child properties (division, allow-list, drive anchor, …).
3. A company maps a **list** of source directory hierarchies (`<root>/mapped_dirs.yaml`), each
   **recursive**; LFB scans each subtree for big files without relocating them, and each computer
   resolves a mapped directory to its own absolute path via the device **graft**
   ([devices.mdx §4](./devices.mdx)).
4. A user may belong to **more than one** company; each is a separate storage and gets its **own child**
   (labeled by `company_name`) in the Storages tab.
5. It is carried across the **members'** computers — the SDL over Git/backbones, the mapped big files over
   IPFS (within the company allow-list) — and often clones to a **shared** Google Drive / Dropbox;
   the **root** holds the mapped-dir list, fingerprints, manifest, devices, bookmarks, and `analysis/`
   per [storages.mdx §4](./storages.mdx) + [syncable_data_location.mdx §2](./syncable_data_location.mdx),
   placed by the SDL rule in [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx).
6. A company is created **explicitly** or **automatically** from a repo whose remote names a forge **org**; every
   repo mapped to it **groups under this company** ([repo_company_mapping.mdx §7.1](./repo_company_mapping.mdx#71-grouping-in-the-left-bar-and-repos-list-locked)),
   and an auto-created company is a **lightweight** entry (same storage kind, less configuration) that is promoted
   to a full shared pile only when the user fills in mapped dirs / allow-list / sync repo (§5).
7. The company's **friendly display name** is its descriptor **`company_name`** (§6, §2) — seeded to the cryptic
   org slug on auto-creation, overwritten by the user's rename, shown **everywhere**, never blank (falls back to
   the slug), and **travels** to teammates via the committed `storage.yaml`.
8. A company may configure a **sync repo** (local path or HTTP(S) URL, same shapes/credentials as the Git
   backbone); when set, member repos **owned by this company** with the per-repo toggle on **additionally mirror**
   their LFB tracking state to **`<syncRepo>/repos/<repoKey>/`** — in addition to Local Storage
   (`~/T/_large_files_bridge/repos/<repoKey>/`), never instead of it (§7,
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)); default **OFF**, and **ownership** (auto or
   manual) — not how it was derived — decides which sync repo a repo can mirror to.
9. The company's [settings page](./storage_settings.mdx#4c-owned-repos--the-repos-mapped-to-this-storage)
   lists the **repos owned by this company** with a per-row **reassign** control (§7.1); reassigning a repo
   **into** this company (when it has a sync repo) records the travelling **ownership assertion** in
   `owner_map.yaml` ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx)) — applied on other members'
   machines **only via review-and-consent**, never silently — and **moves no bytes**, only re-targeting which
   sync repo the repo's tracking mirrors to.

**The §8 use case — a big file reaching the user's second computer:**

10. **The whole narrative passes.** With the Tower having pinned a git-ignored big file in a company-owned
    working repo, the laptop's **View One Repo** page for that repo shows the file as a **red remote-only
    row**, counts it in **`Pull down`** (and in **`Add to IPFS`** while undecided here), lists it in the
    **To Do** tab, and **pulls the bytes down over IPFS** when the user acts — after which it becomes an
    ordinary local row (§8.2).
11. **The mirror subtree is keyed by the repo's normalized git REMOTE** (`repos/<repoUid>/`), never by an
    absolute-path hash, so both computers read and write the same directory; a repo with **no remote** does
    not mirror and **says so** (§8.4.1). Local Storage stays path-keyed.
12. **The owning company storage's git-backed SDL is auto-adopted as the sync repo, and the per-repo mirror
    defaults ON** — an explicit `sync_repo` still wins, the per-repo toggle is now an **opt-out**, and
    ownership is resolved from the repo's **remote org** before any path-containment fallback (§8.4.2).
13. **Receiving is a per-entry MERGE, never a copy** — union by `path`, union `pinned_by`, newer

    re-derived from its real pinset. It runs on **every** SDL backbone pull, not on demand (§8.4.3).
14. **A `FileRow` may be `remote-only`** — composed from the reconciled manifest with no local file behind

15. **Both per-repo manifests agree**, and the pull-down list and the pull action read the **same** one

16. **A mirror write nudges its SDL's backbone** (debounced) rather than waiting for the next timer tick,

17. **A forge org binds to a company storage through `owner_slugs`** in the committed descriptor —
    explicit entry, then normalized name match, then a lone company storage claiming an unclaimed org; a
    match won by heuristic is **written back** as an explicit entry so it is auditable and travels, and a
    manual owner override still beats all of it (§8.4.4).

## 10. A company IS a forge organization (LOCKED)

A company storage is **one-to-one with a forge organization**. The org is a level in every repo's remote
URL — `https://github.com/**ACT3ai**/charlie-kirk.git` — so **the repos already say which company they
belong to**. LFB does not ask the user to draw that map; it reads it off the disk.

```
        the disk                       the URL level              the storage
  ┌────────────────────┐        ┌─────────────────────┐     ┌────────────────────────────┐
  │ ~/BGit/…/charlie-  │  git   │ github.com/ ACT3ai  │     │ act3_large_files_bridge/   │
  │   kirk/.git/config ├───────►│              /repo  ├────►│   storage.yaml             │
  └────────────────────┘ remote └─────────────────────┘     │   company.owner_slugs:     │
                                        ▲                   │     ["ACT3ai"]             │
        …102 more repos ────────────────┘                   └────────────────────────────┘
```

This is what makes **more than one company** work without configuration: three orgs across the user's repos
mean three companies, each with its own storage, each knowing exactly which repos are its own.

### 10.1 Finding the organizations

LFB already walks the scanner roots for git working trees and records each one's `origin`. Grouping them by
org needs no new scan:

* **Parse the remote** into `host / owner / repo`; **`owner` IS the organization**
  ([repo_company_mapping.mdx §2](./repo_company_mapping.mdx)). SSH and HTTPS forms fold together.
* **Group every repo by org.** The result is the candidate list: each org, its display casing, and the repos
  that belong to it.
* **No network, ever.** The org is read from a string in `.git/config`. LFB never calls a forge API to ask
  who owns what ([external_tools_policy.mdx](./external_tools_policy.mdx)) — the URL is sufficient and
  offline.
* **A repo whose remote does not parse** (a bare local path, a self-hosted host we do not recognize, or no
  remote at all) belongs to **no org**, and therefore to Personal. It is not a company candidate.

### 10.2 Which organizations are actually YOURS (LOCKED)

**Not every org on the disk is a company you belong to** — and this is the difference between a useful
feature and an unusable one. A real developer machine is full of clones of other people's projects. Measured
on the machine this spec was written against: **32 distinct orgs**, of which **28 were clones** — `KDE`,
`OpenShot`, `xai-org`, `twitter`, `mltframework`, `NatronGitHub`, … Creating a company storage per org would
have made **31 directories and 31 Storages entries** for organizations the user has no membership in.

The test is **local, cheap, and decisive**:

> **An org becomes a company only when the user's own git identity has authored at least one commit in one
> of that org's repos on this computer.**

* **Why this test.** Cloning is passive; committing is membership. `git log --author=<your email>` answers it
  from the local object store with no network and no credentials. On the measured machine it cut 32 orgs to
  **4**, and every one of the 4 was correct.
* **Your own account is Personal, not a company.** An org listed in `personal_accounts`
  ([settings.mdx](./settings.mdx)) resolves to **Personal** and is never proposed as a company — that is what
  keeps `github.com/<you>/<your-repo>` out of the company list.
* **The test is a PROPOSAL filter, never a lock.** The user can add any org as a company by hand, and can
  dismiss a proposed one. A heuristic decides what to *offer*; the user decides what *exists*.
* **Not-yet-committed is not never.** An org that fails the test today passes the first time the user commits
  to one of its repos. Nothing needs re-scanning by hand.
* **Say the number.** The Storages page reports what was skipped — *"28 organizations you've only cloned from
  were ignored"* — with a way to see them. A silent filter and a bug look identical
  ([warnings.mdx](./warnings.mdx)).

### 10.3 Proposing and creating the company storages (LOCKED)

Creating a company storage **makes a real directory on the user's disk and runs `git init` in it**. That is
not something to do behind the user's back, and the charter is explicit that LFB surfaces and offers rather
than acting on files on its own. So:

* **LFB proposes; one click creates.** Every qualifying org (§10.2) with no company storage yet appears as a
  proposed row — org name, repo count, the directory that would be created — with a checkbox, and a single
  **Create company storages** button makes all the checked ones at once. Unchecking one before creating means
  no directory is ever made for it.
* **A dismissed org stays dismissed.** Dismissal is remembered per org so the same proposal does not return
  on the next scan. It can be undone from the company settings.
* **Adopt before you create.** If a company storage already **claims** the org — an explicit
  `company.owner_slugs` entry, or a normalized name match (§8.4.4) — it is **adopted**, never duplicated.
  This is exactly how `act3_large_files_bridge` (named `Act3`) serves the org `ACT3ai`: it is adopted and the
  binding is written back, rather than a second `act3ai_large_files_bridge` appearing beside it.
* **What creation does.** Make `<storages parent>/<org-slug>_large_files_bridge/`, `git init` it, and write
  a `storage.yaml` carrying `type: company`, `company_name` (the org's display casing, renameable per §6),
  and **`company.owner_slugs: ["<org>"]`** — the binding, recorded up front so it never has to be guessed
  again. The storage then appears in the Storages tab by the ordinary discovery rules.
* **A brand-new company storage has NO git remote yet, and LFB must say so.** It can commit locally, but
  nothing travels until the user supplies a remote (a local path or an HTTP(S) URL,
  [git_backbone.mdx §3](./git_backbone.mdx)). A repo that commits forever and never pushes, silently, is the
  worst failure this product has — see §11.2.
* **The parent directory is a setting**, defaulting to where the user's existing `*_large_files_bridge`
  storages already live. It is not hardcoded to one person's layout.

### 10.4 What the linkage buys (LOCKED)

Once org → company storage is a recorded fact, three things follow with no further configuration:

1. **A repo knows its company.** Its remote's org resolves to exactly one company storage
   (§8.4.4), which is the storage whose settings list it under Owned repos
   ([storage_settings.mdx §4c](./storage_settings.mdx)).
2. **A repo knows where its tracking state travels.** That company's SDL is its sync repo (§8.4.2), so its
   `repos/<repoUid>/` subtree mirrors there and reaches the user's other computers and their teammates.
3. **Companies stay separate.** Two orgs never share a storage, so one company's tracking text is never
   pushed into another company's repo. **This is a confidentiality boundary, not just tidiness** — the org
   in the URL is what enforces it.

Deriving the org from a repo's remote, grouping repos by org, applying the authored-commit membership test, and creating a company storage per org is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-discovery.service.ts` — `discoverOrganizations()` (group + membership test) and `createCompanyStorage(org)` (mkdir + `git init` + descriptor with `owner_slugs`), surfaced by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.router.ts` — `GET /api/storages/organizations` and `POST /api/storages/companies`.

## 11. The company tracking repo governs itself (LOCKED)

The company's dedicated tracking repo is **Large File Bridge's to drive**. LFB has permission to — and
therefore **must** — `fetch`, `merge`, `commit`, and `push` it, and **resolve merge conflicts
automatically**, without ever waiting on the user.

> **The load-bearing sentence.** *Waiting for the customer to resolve a conflict means it never gets
> resolved.* A tracking repo that is stuck pending a human is a tracking repo that has stopped syncing —
> and a sync product that has stopped syncing is indistinguishable from one that was never installed. The
> user did not ask for a git workflow; they asked for their files to be on both computers.

This is not a new engine. It is the **same** cycle Personal already runs
([storage_personal.mdx §16.1](./storage_personal.mdx) puts a company sync repo explicitly in scope, and
[git_backbone.mdx](./git_backbone.mdx) is written against "a storage", never against Personal). What §11
does is close the gaps where the company case is exposed to failures Personal rarely hit — because a
company repo now carries the whole `repos/<repoUid>/` mirror, not just a device file.

### 11.1 A conflict is ALWAYS resolved, never surfaced as homework (LOCKED)

Today one unresolvable conflict **freezes the storage's entire backbone**: the merge aborts, and the commit
and push are skipped on every subsequent cycle. Nothing travels again until a human runs git by hand. Two
specific defects make that likely now:

* Only `repo_storage.yaml` has an auto-resolution, and it applies **only when every conflicted path is one**
  — a single sidecar in the same merge aborts the resolvable ones too.
* The union-merge attribute list covers `manifest.yaml`, `decisions.yaml` and `owner_map.yaml`, but **not**
  `files.yaml`, the `files/<rel>.yaml` sidecars, or `repo_storage.yaml` — all of which now ride in every
  company repo inside `repos/<repoUid>/`.

**The rule: every file LFB owns in this repo has a defined automatic resolution, and a conflict in one file
never blocks another.** Resolution is per file, by what the file *is*:

| Data shape | Files | Automatic resolution |
| --- | --- | --- |
| **Append-only lists** | `manifest.yaml`, `decisions.yaml`, `owner_map.yaml`, `history/<device>.txt`, `files/<rel>.yaml` events | **Union** both sides, then fold on read (§8.4.3). Nothing is lost, because nothing was ever meant to be overwritten. |
| **Regenerable caches** | `repo_storage.yaml`, `files.yaml` | **Drop the file from the merge** and let the next pass rebuild it from Local Storage. Neither side's copy is worth preserving, so there is nothing to reconcile. |
| **Self-owned** | `devices/<self>.yaml` | This computer's copy wins; other devices' files are never touched. |
| **Shared mutable** | `storage.yaml`, `mapped_dirs.yaml`, `bookmarks.yaml` | Field-level merge where the shape allows (set-valued fields **union**; `owner_slugs` in particular is a set); otherwise newest `updated_at` wins, and the losing value is **recorded in the log**, never discarded silently. |

* **Per-file, not per-merge.** Each conflicted path is resolved by its own rule. A mixed conflict resolves
  the files it can and only escalates the ones it genuinely cannot.
* **Escalation is the last resort, and it is still not homework.** If a file has no rule, LFB quarantines
  that one file (the same way a half-merged manifest is quarantined today), completes the merge, and
  **keeps the backbone moving** — then surfaces the quarantined file. One unresolvable file must never stop
  the other thousand from syncing.
* **Never destructive.** Resolution only ever unions, regenerates, or picks a side and records it. LFB does
  not `reset --hard`, `clean`, or force-push ([git_backbone.mdx §1](./git_backbone.mdx)).

### 11.2 A repo that cannot push must SAY so (LOCKED)

The highest-value fault found in the company path: when the tracking repo has **no `origin`**, the cycle
still "succeeds". It commits, returns without pushing, sets no problem, and logs nothing — **indistinguishable
from a healthy repo with nothing to do.** A `git init`ed company storage can therefore accumulate local
commits forever while the user believes their computers are in sync.

* **No remote ⇒ a loud, standing fault.** Say it once per storage, in the log and on the storage's row:
  *"Large File Bridge is tracking this company locally, but it has no remote — nothing is reaching your
  other computers."* With the fix in one click: add a remote.
* **Committed-but-unpushed is a first-class state**, not an absence. The pass reports it, and the
  always-push-when-ahead rule keeps retrying it once a remote exists
  ([backbone_resilience.mdx §6](./backbone_resilience.mdx)).
* **Every silent null becomes loud.** A backing read that fails, a backbone that will not resolve, a push
  that never happened — each gets a log line naming the storage and the reason. Silence must mean success.

### 11.3 One repo, one lock (LOCKED)

The company path has **two** independent locks over the same working tree: the pin/device cycle's
per-storage lock, and a separate one used when writing `owner_map.yaml`. Two locks are not a lock. Every
git operation against a storage's working copy — pin pass, device write, mirror nudge, ownership assertion —
takes **the same per-storage lock**, so the tree is never touched by two writers at once.

### 11.4 Push retries

A push rejected as non-fast-forward is re-pulled and retried. The retry budget is **three attempts with
backoff**, not one — a single retry loses to any peer that pushes twice while we are working. After the last
attempt the commit stays local, the fault is reported, and the ahead-count rule sends it on the next cycle.

Driving the company tracking repo's autonomous cycle and resolving its conflicts is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone.pull()` (per-file conflict resolution) and `commitAndPush()` (ahead-count push + retry budget), driven per storage by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `syncStorageTextInner()` under `withStorageGitLock()`.

## 12. Use case — three organizations, three companies, one linkage

The acceptance narrative for §10 and §11. It is deliberately the *multi*-company case, because one company
hides every mapping bug.

### 12.1 The setup

A developer's laptop holds repos from several places at once:

| Org in the remote | Repos on disk | Do they commit there? | Outcome |
| --- | --- | --- | --- |
| `ACT3ai` | 102 | yes | **Company** — adopts the existing `act3_large_files_bridge` |
| `stoke-gh` | 2 | yes | **Company** — new storage created |
| `trykimu` | 2 | yes | **Company** — new storage created |
| `BryanStarbuck` | 28 | yes | **Personal** — it is the user's own account |
| `KDE`, `OpenShot`, `xai-org`, `twitter`, … (28 orgs) | ~40 | no — clones only | **Ignored**, and the count is reported |

### 12.2 What must happen

1. **Discovery reads the orgs off the remotes** — no forge API call, no user mapping step.
2. **The membership test cuts 32 orgs to 3 companies** (§10.2); the user's own account goes to Personal, and
   the 28 cloned orgs are skipped with a visible count.
3. **`ACT3ai` adopts the existing storage** rather than creating `act3ai_large_files_bridge` beside it, and
   the adoption is written into `owner_slugs` so it never has to be re-guessed (§10.3).
4. **`stoke-gh` and `trykimu` are PROPOSED**, not created. Nothing touches the disk until the user clicks
   **Create company storages**.
5. **On confirmation**, two directories exist, each `git init`ed, each with a `storage.yaml` naming its org.
   Both appear in the Storages tab. Both report **"no remote yet — nothing is reaching your other
   computers"** (§11.2) until the user supplies one.
6. **Every repo lands under exactly one company.** All 102 `ACT3ai` repos list under ACT3, the two `stoke-gh`
   repos under stoke-gh, and **no repo appears under two companies**.
7. **Each company's tracking travels in its own repo.** An `ACT3ai` repo's `repos/<repoUid>/` subtree mirrors
   into the ACT3 storage and **never** into stoke-gh's (§10.4.3).
8. **The cycle runs itself.** Once a remote is set, each company storage pulls, merges, commits, and pushes
   on its own schedule, resolving conflicts per §11.1, with **no git command ever asked of the user**.
9. **A new org appears the moment it is earned.** The first commit the user authors in a repo from a new org
   makes that org a proposal on the next scan.

### 12.3 What this proves that §8 does not

§8 proves one file reaches one other computer. §12 proves the **mapping** underneath it: that the product can
tell three companies apart on a disk that mentions thirty-two organizations, put each repo under the right
one, keep their tracking repos separate, and do it without the user drawing a single line between a repo and
a company.

## 13. Acceptance criteria — §10–§12

1. **A company storage is 1:1 with a forge organization**, derived from the org level of each repo's git
   remote, offline, with no forge API call (§10, §10.1).
2. **Repos are grouped by org**, and an org qualifies as a company **only** when the user's git identity has
   authored a commit in one of its repos on this computer (§10.2). An org in `personal_accounts` resolves to
   **Personal** and is never proposed. A repo with no parseable remote belongs to no org.
3. **Skipped orgs are counted and visible** — the user can see what the membership filter excluded and add
   any of them by hand (§10.2).
4. **Company storages are PROPOSED, never silently created** — one click creates the checked ones; an
   unchecked or dismissed org produces no directory, and dismissal is remembered (§10.3).
5. **An org already claimed by a company storage is ADOPTED, never duplicated**, and the binding is written
   back to `company.owner_slugs` (§10.3, §8.4.4). `ACT3ai` adopts `act3_large_files_bridge`.
6. **Creation makes `<parent>/<org-slug>_large_files_bridge/`**, `git init`s it, and writes a `storage.yaml`
   with `type: company`, `company_name`, and `owner_slugs: ["<org>"]`. The parent directory is a **setting**,
   not a hardcoded path (§10.3).
7. **Each company's repos and tracking are separate** — a repo resolves to exactly one company, and one
   company's `repos/<repoUid>/` mirror never enters another's repo (§10.4).
8. **The company tracking repo runs itself**: pull → merge → commit → push, on schedule, with **no git
   command ever required of the user** (§11).
9. **Every conflict has an automatic resolution, applied PER FILE** — union for append-only lists, regenerate
   for caches, self-owned wins for device files, field/set merge or newest-wins-and-log for shared mutable
   state. A conflict in one file **never blocks** another, and an unresolvable file is quarantined while the
   backbone keeps moving (§11.1). Resolution is never destructive: no `reset --hard`, no `clean`, no
   force-push.
10. **A repo with no remote reports a standing, loud fault** and is never mistaken for healthy;
    committed-but-unpushed is a reported state, and every silent null becomes a log line naming the storage
    and the reason (§11.2).
11. **One lock per storage** covers every git operation against its working tree, including ownership
    assertions (§11.3); a rejected push is retried **three times with backoff** before the commit is left for
    the next cycle's ahead-count push (§11.4).
12. **The §12 narrative passes end to end** on a machine with 32 orgs: 3 companies, correct adoption, nothing
    created without consent, every repo under exactly one company, and no user-run git (§12.2).
