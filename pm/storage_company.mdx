# LargeFileBridge — Company Storage

**Company storage** is a **shared pile of a company's large files** — the big media/assets a team needs
in common. There is **one company storage per company**, and a user may belong to **more than one
company**, so several can coexist on the same machine.

Read the shared model first: the storage family, the `storage.yaml` schema, the `.lfbridge/` tracking
layout, cloud clones, and media analysis are defined once in **[storages.mdx](./storages.mdx)**. This spec
covers only what is **specific to company storage**.

Siblings this builds on and does not repeat:

* [storages.mdx](./storages.mdx) — the umbrella: `storage.yaml` (§3), `.lfbridge/` (§4), clones (§5),
  media analysis (§6).
* [syncable_data_location.mdx](./syncable_data_location.mdx) — the **SDL** model this spec now rests on:
  the company root is the storage's **Syncable Data Location** (small text that travels), and the big
  files live in a **list of mapped source directories** (§1.1), not under the root.
* [devices.mdx](./devices.mdx) — the per-computer **graft** that resolves each mapped directory to this
  machine's local path (§1.1).
* [storage_repo.mdx](./storage_repo.mdx) — the canonical directory-based storage; company storage is the
  same shape with a `company` block and a shared (multi-user) audience.
* [storage_personal.mdx](./storage_personal.mdx) — the single-user private counterpart.

## 1. Anatomy of a company storage

The company root is the storage's **[Syncable Data Location](./syncable_data_location.mdx)** — the small,
travelling text that describes the storage. The company's **big files are not stored under it**; they
stay in their natural directories and are brought into scope by a **list of mapped source directories**
(§1.1).

```
~/BGit/Bryan_git/act3_large_files_bridge/     ← ONE company's storage root = its Syncable Data Location ("ACT3")
├── LargeFilesBridge_SyncList.yaml            ← the portable list (sync_list.mdx), committed to git here
├── .lfbridge/
│   ├── storage.yaml        ← name + type: company + company block (storages.mdx §3)
│   ├── mapped_dirs.yaml     ← §1.1 the company's mapped source directories (syncable_data_location.mdx §3)
│   ├── files.yaml          ← per-file fingerprint index (storages.mdx §4.1)
│   ├── manifest.yaml       ← large-file → IPFS CID + pin state
│   ├── devices/            ← the device registry + per-device graft (devices.mdx)
│   ├── bookmarks.yaml      ← bookmarked files, travels (syncable_data_location.mdx §4.4)
│   └── analysis/           ← per-file media analysis + compression records (storages.mdx §6, SDL §4)
└── …NO big files here — they live in the MAPPED directories (§1.1)…
```

* **Root:** one **SDL** directory per company, e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` — a small
  Git repo of metadata. A user in three companies has three such roots, each its own storage. Because this
  root is a **dedicated tracking-data Git repo the user made for LFB**, when it is itself a git working
  tree LFB **auto-adopts it as the storage's Git backbone** and **commits + pushes** this computer's
  `devices/<self>.yaml` there on first write and every update — no manual dedicated-repo setting needed
  ([devices.mdx §11.2.1](./devices.mdx#1121-a-company--personal-tracking-repo-auto-adopts-its-own-git-root--commit--push-first-write-and-every-update-locked-learned-from-a-defect)).
* Because it is **shared across members**, both the SDL (over Git/backbones) and the mapped big files
  (over IPFS) are carried across **the members' computers** — a company storage is the multi-user case (peers
  here include teammates' machines, within the company's allow-list).

### 1.1 A company maps a LIST of source directories (LOCKED)

The defining trait of a company storage: in its settings the user lists **the directories of their hard
drive that belong to this company**, and **everything recursive under each is in scope**. A company with
seven directory hierarchies adds all seven; LFB scans each subtree for big files, tracks them, pins the
opted-in ones over IPFS, and stores their derived artifacts in the SDL — **without relocating a single
byte**.

* The **logical** list travels in `.lfbridge/mapped_dirs.yaml`
  ([syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked));
  each mapped directory has a stable **key**.
* The **absolute local path** of each mapped directory is **per-computer** — resolved by that machine's
  **graft** ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)).
  So the same company storage grafts into `~/Movies/act3` on a laptop and `/Volumes/Media/act3` on a
  tower, and each computer pulls files to its own location.
* This is the generalization of a **repo** storage (one hierarchy, path-resolved by Git) to **many**
  hierarchies path-resolved by the device graft. A user adds/removes mapped directories from the
  company's [storage settings](./storage_settings.mdx).

## 2. `storage.yaml` for a company

```yaml
# ~/BGit/Bryan_git/act3_large_files_bridge/.lfbridge/storage.yaml
name: "ACT3 Company Files"
type: company
created: 2026-05-01T10:12:00Z
company:
  company_name: "ACT3 AI"     # which company — the label shown in the Storages tab
  # company-scoped child properties:
  division: "Film"            # optional
  allow_list: ["bryan@whitehatengineering.com", "…"]   # who may carry this storage
  drive_folder: "Shared drives/ACT3"                    # optional shared-drive anchor
clones:
  google_drive: "Shared drives/ACT3/LFB"   # a company storage often mirrors to a SHARED drive
  dropbox: null
```

* **`type: company`** with a **`company`** block. **`company_name`** names which company; child properties
  scope the storage (division, allow-list of members, shared-drive anchor, …).
* A user in multiple companies has **multiple** company storages, each with its own `company_name`.

## 3. Media analysis + cloud clones

Company storage produces the [storages.mdx §6](./storages.mdx) **media-analysis** outputs into
`.lfbridge/analysis/` — and because these are shared, a teammate who pulls the storage gets the
transcript / description / visuals-by-time without re-running them. A company storage **often clones to a
shared Google Drive / Dropbox** ([storages.mdx §5](./storages.mdx)) so members without LFB can still reach
the files; recorded in the `clones` block.

---

## 4. Where it shows in the UI

In the **[Storages](./storages.mdx#2-the-storages-left-bar-tab)** left-bar tab, **each company gets its
own child**, labeled by **`company_name`**. A user in ACT3 and two other companies sees three company
children under Storages, alongside Personal, the Repos link, and their communities.

---

## 5. Repos auto-derived from an org appear as this company

A company storage is created two ways: **explicitly** (the user sets one up for a shared media pile, §1), or
**automatically** — when a repo's git remote names a forge **organization**, LFB derives a company from that org
and **auto-creates this storage** to hold the grouping ([repo_company_mapping.mdx §3/§6](./repo_company_mapping.mdx#6-auto-creating-the-company--and-its-friendly-name)).

* Every repo whose remote owner maps to this company (e.g. `git@github.com:ACT3ai/…`) **groups under this company**
  in the Storages disclosure and the Repos list ([repo_company_mapping.mdx §7.1](./repo_company_mapping.mdx#71-grouping-in-the-left-bar-and-repos-list-locked)).
  The mapping is derived offline from the remote URL — **no** forge network call.
* An **auto-created** company is a **lightweight** entry at first: it exists to group repos and to be **renamed**
  (§6). It has no mapped source directories (§1.1), no allow-list, and no state-sync repo (§7) until the user adds
  them. **A repo-group company and a shared-media-pile company are the same storage kind** at different levels of
  configuration — filling in §1.1 / the allow-list / §7 promotes the same storage from "just a group" to a full
  shared pile.
* The company's **`id`** is the stable id `resolveRepoOwner` stamps on each owned repo's `owner.companyId`
  ([repo_company_mapping.mdx §1.1](./repo_company_mapping.mdx#11-the-repo-owner-field-locked-shape)), so grouping and the
  company row agree.

Creating-if-absent the company storage for a derived org owner is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureCompanyForOwner(slug)` (creates via `ensureStorage(root, "company", …)` when missing), surfaced by `listStoragesPage()`.

## 6. The friendly display name

An org slug from a remote is often **cryptic** — `ACT3ai`, `acme-eng-2`, `~sircmpwn`. So a company storage carries
a user-settable **friendly display name**, and the product shows **that** name **everywhere** — the Storages
left-bar child, the Repos grouping header, and each owned repo's Owner control
([repo_company_mapping.mdx §5.1](./repo_company_mapping.mdx#51-the-reassign-control)).

* **Where it is stored.** The friendly name **is** the company descriptor's **`company_name`** in
  `.lfbridge/storage.yaml` (§2) — the same field that labels the Storages child. On auto-creation `company_name`
  is seeded to the **owner slug** (e.g. `ACT3ai`); the rename simply **overwrites** it with the human name
  (e.g. `ACT3 AI`). There is no second field: the descriptor's `company_name` is the single source of truth for
  the label.
* **Fallback, never blank.** Until renamed, the displayed name **is** the slug — never blank and never the raw
  remote URL. `buildRow()` already falls back to `prettyName(base)` when `company_name` is absent, so a company is
  always labeled.
* **The rename flow.** From the company's [storage settings](./storage_settings.mdx) the user edits a **Display
  name** field; saving writes `company_name` and the label refreshes everywhere on the next read (it is derived,
  not frozen onto repos — [repo_company_mapping.mdx §9](./repo_company_mapping.mdx#9-error--empty-cases)).
* **Travels with the storage.** Because `company_name` lives in the committed `storage.yaml`, the friendly name
  **travels to teammates** who carry this company storage — everyone sees the same human name, not the slug.

Reading and writing the company's friendly `company_name` (the display name) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `readDescriptor` / `writeDescriptor` (the `company.company_name` field), edited from the settings page and rendered by the Storages tab.

## 7. The company LFB state-sync repo

A company can optionally configure a **dedicated LFB state-sync repo** — a git repo the company sets up **for LFB
state** (not for the big media bytes) — and, when it is set, a member's repos owned by this company route their
**noisy per-repo tracking** into it so teammates share it.

* **What it is.** A git repo (the company's own — e.g. `git@github.com:ACT3ai/act3_lfb_state.git`) whose only job
  is to carry LFB's small tracking text between the company's members. It is the **company** analogue of the
  Personal state-sync repo ([storage_personal.mdx §14](./storage_personal.mdx#14-personals-optional-state-sync-repo)) and
  of this company storage's own SDL Git backbone (§1) — but scoped to **member repos' tracking**, not this
  storage's media pile.
* **What travels there.** Per the churn policy ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)),
  a repo's `repo_storage.yaml` and its **noisy** per-file sidecars normally stay in the machine-local state root
  `~/T/_large_files_bridge/repos/<repoKey>/` until a transcript / AI-description exists. **When the repo's owner is
  this company and this company has a state-sync repo configured, that override kicks in**: the tracking is written
  under **`<syncRepo>/repos/<repoKey>/`** instead, so it commits + pushes to the company repo and every teammate on
  the same company sees it. Path-mirrored by `<repoKey>` so two members never collide.
* **How the user configures it.** From the company's [storage settings](./storage_settings.mdx#4-backing-locations--dedicated-repo--google-drive--dropbox)
  the user enables a **Company state-sync repo** backing location and supplies it as a **local file path** or an
  **HTTP(S) URL** — the same two remote shapes and OS-credential-helper rules as the Personal Git backbone
  ([storage_personal.mdx §5](./storage_personal.mdx#5-turning-the-git-backbone-on--local-path-vs-https-url),
  [git_backbone.mdx](./git_backbone.mdx)). LFB runs the same fetch → auto-merge → commit → push cycle against it.
* **Default: OFF.** Absent a configured state-sync repo, member repos keep their tracking in the state root per the
  default policy — nothing changes. Turning it on is an explicit opt-in that the company makes once.
* **Precedence.** A repo manually reassigned to this company ([repo_company_mapping.mdx §5](./repo_company_mapping.mdx#5-manual-reassignment--the-user-always-wins-locked))
  routes to this state-sync repo just like an auto-owned one — ownership (not how it was derived) decides.

Resolving the company's state-sync repo and routing an owned repo's tracking to `<syncRepo>/repos/<repoKey>/` is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `readStorageSettings()` (the company's `backing.state_sync_repo` block) consumed by the placement resolver in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` — `resolveTrackingRoot(repoKey, owner)`.

## 8. Acceptance criteria

1. There is **one company storage per company**, rooted in its own directory (e.g.
   `~/BGit/Bryan_git/act3_large_files_bridge/`), which is the storage's **Syncable Data Location** —
   small travelling text, tracked via a hidden **`.lfbridge/`** at its root; **no big files live under
   the root**.
2. Its `.lfbridge/storage.yaml` has **`name`** + **`type: company`** and a **`company`** block carrying
   **`company_name`** plus company-scoped child properties (division, allow-list, drive anchor, …).
3. A company maps a **list** of source directory hierarchies (`.lfbridge/mapped_dirs.yaml`), each
   **recursive**; LFB scans each subtree for big files without relocating them, and each computer
   resolves a mapped directory to its own absolute path via the device **graft**
   ([devices.mdx §4](./devices.mdx)).
4. A user may belong to **more than one** company; each is a separate storage and gets its **own child**
   (labeled by `company_name`) in the Storages tab.
5. It is carried across the **members'** computers — the SDL over Git/backbones, the mapped big files over
   IPFS (within the company allow-list) — and often clones to a **shared** Google Drive / Dropbox;
   `.lfbridge/` holds the mapped-dir list, fingerprints, manifest, devices, bookmarks, and `analysis/`
   per [storages.mdx §4](./storages.mdx) + [syncable_data_location.mdx §2](./syncable_data_location.mdx).
6. A company is created **explicitly** or **automatically** from a repo whose remote names a forge **org**; every
   repo mapped to it **groups under this company** ([repo_company_mapping.mdx §7.1](./repo_company_mapping.mdx#71-grouping-in-the-left-bar-and-repos-list-locked)),
   and an auto-created company is a **lightweight** entry (same storage kind, less configuration) that is promoted
   to a full shared pile only when the user fills in mapped dirs / allow-list / state-sync repo (§5).
7. The company's **friendly display name** is its descriptor **`company_name`** (§6, §2) — seeded to the cryptic
   org slug on auto-creation, overwritten by the user's rename, shown **everywhere**, never blank (falls back to
   the slug), and **travels** to teammates via the committed `storage.yaml`.
8. A company may configure a **state-sync repo** (local path or HTTP(S) URL, same shapes/credentials as the Git
   backbone); when set, member repos **owned by this company** route their noisy per-repo tracking to
   **`<syncRepo>/repos/<repoKey>/`** instead of the machine-local state root (§7,
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)); default **OFF**, and **ownership** (auto or
   manual) — not how it was derived — decides.
