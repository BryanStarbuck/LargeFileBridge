# LargeFileBridge — Use Cases & UI Redesign

This document is written from the chair of a **usability designer** doing a heuristic review of the
whole app. It has three jobs:

1. State the **one question** every screen must help a non-expert answer.
2. Define the shared **health model** and **progressive-disclosure** rules that make the answer
   obvious, plus the reusable **components** that implement them.
3. Give a **per-page redesign** — exactly what goes where — for every screen **except the
   Home/Repos dashboard**, which is already in good shape and is treated as the fixed reference the
   rest of the app should feel like a sibling of.

It builds on the existing specs and does not repeat them: [Repos](./repos.mdx),
[One repo](./one_repo.mdx), [IPFS pinset](./ipfs.mdx), [IPFS dashboard](./ipfs_ui.mdx),
[Settings](./settings.mdx), [Security](./security.mdx), [Scan](./scan.mdx), [Files](./files.mdx),
[Directories](./directories.mdx), [Full paths](./full_paths.mdx), [Left bar](./left_bar.mdx).

## 0. Who is the user, and what do they actually want?

The person using LargeFileBridge is **not** an IPFS expert. They are a creator, a filmmaker, an
engineer, or a family member who has a laptop, a desktop, maybe a studio machine or a server, and a
pile of **big files** — videos, renders, image libraries, model weights, datasets — that git can't
carry. They installed LFBridge so those files are **safe and up to date on all their computers**.

They come to the app in one of a few moods:

* **"Is everything OK?"** — a reassurance check. They want a green light and to leave.
* **"Something's wrong."** — a file didn't show up on the other machine; a status looks red. They
  want to know *what* is broken and *what to click* to fix it — not to read a manual on Kubo.
* **"Can this be better?"** — they have 40 GB of uncompressed video, or a big file that isn't
  git-ignored, and they half-know they should clean it up. They want the app to point it out and
  offer a one-click fix.

Every screen is judged against this: **does it answer "am I OK / what's broken / how do I fix it"
without making a non-expert learn our internals first?**

> The internals still matter — CIDs, peer IDs, multiaddrs, reprovide strategy, GC. But they are the
> *second* thing the user sees, never the first. See the disclosure rule (§2).

## 1. The health model (Broken / Improvable / Fixed) — LOCKED

The whole app speaks in **three states**. This is the single most important idea in the redesign:
one vocabulary, one color language, one set of icons, used identically on every page, for a repo, a
file, the IPFS node, a background job, or the whole computer.

| State | Meaning to the user | Color | Icon | Dot |
| --- | --- | --- | --- | --- |
| **Fixed** | Working as intended. Nothing to do. | green | `CheckCircle2` | 🟢 |
| **Improvable** | It works, but it could be safer / smaller / cleaner. An *offer*, never an alarm. | amber | `AlertTriangle` | 🟡 |
| **Broken** | Something is actually not working. Files are at risk or not moving. Needs action. | red | `AlertCircle` | 🔴 |
| *(Neutral)* | Not applicable / nothing set up yet / informational. | slate | `Circle` | ⚪ |

Rules:

* **Broken beats Improvable beats Fixed.** A page's overall state is the worst state among its
  parts. One broken thing turns the page header red even if everything else is green.
* **Improvable is an offer, not a warning-that-nags.** Amber means "here's something you *could* do,"
  aligned with the charter: *never compress, git-ignore, or alter a file unless the user asks.* We
  detect and surface; we never act on our own. Amber must never feel like an error.
* **Every non-Fixed state carries a plain-English "what this means" line and, wherever we can, a
  primary Fix action.** "The IPFS engine isn't running — your files can't move between computers
  until it starts. **[Start it]**." Never a bare red pill with no next step.
* **Fixed states are quiet.** Green gets a small dot and a short line; it does not shout. The screen's
  visual weight goes to whatever is Broken or Improvable.

Color tokens (add to `styles.css`, reuse everywhere — see §3):

```
--lfb-ok:      #16a34a   /* green-600  — Fixed   */
--lfb-ok-bg:   #f0fdf4   /* green-50            */
--lfb-warn:    #d97706   /* amber-600  — Improvable */
--lfb-warn-bg: #fffbeb   /* amber-50            */
--lfb-bad:     #dc2626   /* red-600    — Broken  */
--lfb-bad-bg:  #fef2f2   /* red-50              */
```

## 2. Progressive disclosure — surface the answer, hide the mechanism

The complaint the redesign fixes: today several pages dump *everything* at one altitude — a flat row
of PeerID, reprovide strategy, gateway, GC, pinned bytes, tracked/untracked counts all in one
sentence. A non-expert can't tell what matters.

The rule going forward, on **every** page:

1. **Row 1 — the verdict.** One line, in plain English, colored by the health model: *"Everything's
   pinned across your 3 computers."* / *"1 thing needs attention."* / *"Pinning is broken."*
2. **Row 2 — the one next action.** When not Fixed, a single primary button that does the obvious
   fix. At most one primary per screen.
3. **Then — the working surface.** The table or the cards the user came to use.
4. **Deeper — the mechanism.** CIDs, peer IDs, multiaddrs, raw metrics, reprovide/GC posture, logs.
   These live **lower on the page** or **behind a chevron** (`Disclosure`). The user can always get
   to every detail — they just **click to go deeper**. We never hide information; we **rank** it.

**Chevrons are the primary "there's more here" affordance.** Use them consistently:

* A **right chevron `›`** that rotates to **down `⌄`** on expand = a collapsible section
  (`Disclosure` / `DiagnosticCard`). Label it with what's inside ("Technical details", "How to fix
  it manually", "Node details", "Per-file breakdown").
* A **trailing `›`** on a table row / list item = "click to open the deeper page for this thing"
  (already the house table convention — keep it).
* A **down chevron on a nav item** = that item expands into children (the IPFS → pinning-repos
  disclosure in the left bar — keep it).

Default collapsed state:

* **Broken** detail starts **expanded** (they need it now).
* **Improvable** and **Fixed** detail starts **collapsed** (offer it, don't shove it).

## 4. Customer use-case flows (the non-expert's point of view)

These are the journeys the redesign must make effortless. Each is written as the *user's* question,
then **where the app answers it** and **what they see**. The through-line is always
**broken → what → fix**.

### UC-1 — "Is everything OK?" (the reassurance check)
* **Enters:** Home / Repos dashboard.
* **Sees:** The dashboard already answers this well (kept as-is). The redesign's contribution: every
  other page they might wander to also opens with a one-line green verdict, so the reassurance holds
  everywhere, not just home.
* **Fixed state:** a quiet green StatusBanner: *"All 6 repos are up to date. Last pin 8 minutes
  ago."* Done — they leave.

### UC-2 — "A file didn't show up on my other computer."
This is the flagship "why is it broken" flow — LFBridge's equivalent of *"why is my email in spam?"*
The user doesn't know if the cause is: the IPFS engine is off, the file was never set to pin, the
other computer isn't a peer yet, or the transfer is still pending. **The app must diagnose it for
them**, in order of likelihood, and hand them the fix.

* **Enters:** the repo (or the file's properties page).
* **The One-repo StatusBanner runs the diagnosis** and states the *first* real problem in plain
  English, worst-first:
  1. **IPFS engine down (Broken):** *"Pinning is paused — the IPFS engine on this computer isn't
     running."* Primary: **[Start IPFS]** (→ links to the IPFS page). Everything below still loads,
     decisions still save, but transfers are paused — say so.
  2. **File set to Ignore / Undecided (Improvable):** *"3 files here aren't set to pin yet."*
     Primary: **[Review files]** scrolls to / filters the table to Undecided.
  3. **No peers backing it up (Improvable→Broken):** *"5 files are pinned from this computer but
     aren't on any other computer yet."* Sub: *"Open LFBridge on your other machine so it can pull
     them, or add it as a peer."* Primary: **[See peers]**.
  4. **Transfer still pending (Improvable):** *"2 files are queued to transfer."* Sub: *"They'll move
     on the next pin, or pin now."* Primary: **[Pin now]**.
* **The file's row** carries the same verdict at file altitude via the transfer pill + a per-file
  StatusBanner on the properties page (*"Set to pin, but on 0 other computers — not backed up
  yet."*).
* **Outcome:** the user never has to know *which* of the four causes it was — the page told them and
  gave them the button.

### UC-3 — "The app says something's red. What do I do?"
* **Any red** the user sees is a `DiagnosticCard` or red `StatusBanner` — so it **always** has: a
  plain-English title, a "what this means," and either a **Fix** button or, if we can't fix it for
  them, the **exact manual step** behind the chevron (e.g. the `brew install ipfs` / daemon command,
  copyable). No dead-end red states.

### UC-4 — "Why is my disk full / this is a lot of data." (compression offer)
* **Enters:** a directory or the Full-paths table, or the per-directory rollup.
* **Sees (Improvable, amber, never automatic):** *"12 videos and 40 images here look uncompressed —
  about 34 GB could shrink."* Each row shows the **C** (should-compress) badge; the rollup table row
  says `12 · Compress…`. Clicking **Compress…** always confirms first and only ever acts on explicit
  request (charter). The "how we decided it's uncompressed" (duration × resolution baseline) is a
  chevron detail, not front-and-center.
* **PNG → compressible:** the convert offer runs the alpha-channel / functionality safety checks
  first and **warns** ("this PNG uses transparency — converting to JPEG loses it") before offering.

### UC-5 — "Is this big file safe to just leave, or should I git-ignore it?"
* **Enters:** Full-paths / directory rollup.
* **Sees:** category rows — *"Big files that aren't git-ignored (a bad idea to check in): 3 ·
  [Git-ignore]"* and *"Big git-ignored files we're not pinning: 7 · [Track]"*. One-click, always an
  offer, never automatic (charter: we never edit `.gitignore` for anyone).

### UC-6 — "Are my background pins even running?"
* **Enters:** Scans page.
* **Sees:** a StatusBanner verdict: *"Background pinning is on — runs every 15 minutes."* /
  *"Background pinning is installed but turned off."* / *"Background pinning isn't installed on this
  computer yet."* Then one `DiagnosticCard` per job (discovery 4h, transfer 15m) with a plain
  purpose line, Installed / On pills, last-run health, and the Install / Turn on / off / Uninstall
  actions. The launchd/plist mechanics are chevron detail.

### UC-7 — "Did I set this up right? What's missing?" (configuration confidence)
* **Enters:** Settings.
* **Sees:** each configurable area reports its own health: **Authentication** — green *"Google sign-in
  configured"* or red *"Not configured — sign-in won't work"* with the exact credentials file path
  and placeholder schema behind the chevron. **IPFS node** — green/red with the only-our-content
  posture summarized as one plain line (*"Serving only your own content ✓"*), the raw
  reprovide/gateway/GC values in the chevron. **Access** — how many domains / accounts can sign in,
  link to manage. The security-critical **only-our-content** default is stated in plain words with
  its compliance state, per the charter.

### UC-8 — "Which of my computers has these files?"
* **Enters:** Peers.
* **Sees:** if empty, not a blank table but a guided empty state: *"No other computers yet."* + a
  chevron *"How to add one"* (install LFBridge on the other machine, sign in with the same Google
  account, it appears here). If populated, the table, with each peer's last-seen recency colored by
  the health model (seen recently = green, stale = amber).

## 5. Per-page redesign

Ordering rule for every page: **PageHeader → StatusBanner (verdict + fix) → working surface →
disclosed detail.** Tables stay TanStack tables with the standard control row (search left; filter +
sort icons right; pagination default 500) — unchanged (charter).

### 5.0 Home / Repos dashboard — KEEP AS-IS
The landing dashboard is the reference the rest of the app is being brought in line with. **No
structural change.** It already answers UC-1 at a glance. The only thing the redesign asks of it: it
is the *tuning fork* — the shared components below are styled so every other page feels like it
belongs next to this one.

### 5.1 IPFS — dashboard (`/ipfs`) + Shared files (`/ipfs/pins`)
The IPFS surface is split in two, and the redesign treats them differently:

* **IPFS dashboard (`/ipfs`)** — the node control panel (install / start / on-off, metric tiles that
  drill in, gateway summary, security card). This page **already embodies the model**: mutually
  exclusive states (Not installed → **Install IPFS**; Installed-but-stopped → **Turn on**;
  Running → tiles + security), each with a plain-English headline and the one primary action, and a
  live progress/error view for install/start jobs. It is **kept** as the IPFS landing and, like the
  Home dashboard, is the reference the Shared-files page is brought in line with. (Its metric tiles
  drill into `/ipfs/pins` and `/devices`; its security card fixes non-compliance in place.)
* **Shared files (`/ipfs/pins`)** — the pinset table, reached from the dashboard's "Shared files"
  tile. This is where the StatusBanner + tiles + node-details redesign below applies.

Shared-files page, top to bottom:

1. **PageHeader** — back-link "‹ IPFS" to the dashboard; title "Shared files" (+ `· {repoName}` when
   filtered by a pinning repo). Actions: **Import all untracked (N)** (when N>0), **Rescan**.
2. **StatusBanner (the verdict).** Worst-first:
   * Node unreachable → **Broken**, red: *"The IPFS engine isn't running — your files can't move
     between computers."* Primary **[Start IPFS]** (or, if the CLI is missing, **[Install IPFS]**),
     with the exact command behind a chevron. This replaces today's separate red box.
   * Node OK but **not compliant** (serving more than our own content) → **Broken** (or **Improvable**
     if it's the acknowledged public-gateway opt-out): *"This computer is set to serve other people's
     content."* Primary **[Fix — serve only my content]** (→ the existing enforce action).
   * Node OK + compliant → **Fixed**, quiet green: *"IPFS is running and serving only your own
     content."*
3. **StatTile row** — the metrics, each a tile (not a run-on sentence):
   `Pinned files` (count) · `Pinned size` (bytes) · `Tracked` · `Untracked` (amber tile when >0,
   click → filters the table to import candidates). Peers/bandwidth tiles when available.
4. **Untracked backlog** — when `untrackedCount > 0`, a `DiagnosticCard` (Improvable): *"N pinned
   files aren't tracked by LFBridge yet."* Fix: **Import all**. Chevron: what "tracked" means.
5. **"Pinned files" section** — the existing pinset TanStack table, unchanged in behavior
   (per-row Import, kebab, `?repo` filter, "Untracked only" toggle, selection + bulk import).
   Sits under an `h2` so it reads as the working surface, not the whole page.
6. **"Node details" `Disclosure`** (collapsed) — PeerID (copyable), reprovide strategy, gateway
   binding, GC. The mechanism, one click away. This is where today's strip fields go to live.

The IPFS worst-first verdict banner (down / not-compliant / running-and-compliant) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/IpfsStatusBanner.tsx` — `IpfsStatusBanner`.

## 6. What we are deliberately NOT changing

* **The Home/Repos dashboard** (§5.0) — kept.
* **All tables stay TanStack tables** with the standard control row and 500-default pagination
  (charter). Every row of every table carries the trailing vertical **`⋮`** row kebab
  ([menus.mdx §3](./menus.mdx#3-the-row--kebab--every-table-has-one)) — three dots up-and-down, never
  the horizontal `⋯`.
* **The left bar** — data source of truth stays `pm/left_bar.yaml`; the IPFS disclosure children
  stay.
* **The charter guardrails** — we never compress, git-ignore, convert, or alter a file unless the
  user explicitly asks; the IPFS node defaults to serving only our own content; nothing phones home.
  The redesign only makes these **more legible**, never looser.
* **The badge vocabulary and colors** — unchanged; only made self-explanatory via the legend
  disclosure (§5.7).

## 7. Acceptance criteria

* **AC-1** Every page (except Home) opens with a one-line health verdict in the Broken/Improvable/
  Fixed language before any table or dense detail.
* **AC-2** No red state anywhere is a dead end: each has a plain-English cause + either a Fix button
  or the exact manual step (copyable) behind a chevron.
* **AC-3** Technical detail (CIDs, peer IDs, multiaddrs, reprovide/gateway/GC, raw metrics, launchd
  labels) is never at row 1 — it is lower on the page or behind a `Disclosure`, and always reachable
  in ≤1 click.
* **AC-4** The three-state model is implemented in exactly one place (`ui/health.ts`) and every
  page's colors/icons come from it.
* **AC-5** UC-2 ("a file didn't pin") is answered by the One-repo StatusBanner without the user
  needing to know which of the four causes applied.
* **AC-6** Improvable (amber) offers never read as errors and never trigger an action without an
  explicit click + confirm (charter compliance).
* **AC-7** The shared components (`StatusBanner`, `DiagnosticCard`, `Disclosure`, `StatTile`,
  `PageHeader`, `Section`) are used by IPFS, Scans, Settings, One-repo, Peers, and the entity pages,
  so the app feels like one product with the Home dashboard.

## 8. Media-viewer use-case flows (OS hand-off & AI analysis)

These journeys live on the **View one file** media viewer ([media_viewer.mdx](./media_viewer.mdx)) and
its companion features: opening a file in its **real desktop app** ([os_open.mdx](./os_open.mdx)), and
turning a media file into **searchable text** — a **transcript** ([Transcribe.mdx](./Transcribe.mdx)) or
an **AI description** ([ai_description.mdx](./ai_description.mdx)). Same UC format as §4: the user's
question, where the app answers it, and what they see.

### UC-9 — "I want to watch this video in QuickTime / VLC, not in the browser tab."
* **Enters:** the video viewer (`/video?path=…`) on their Mac.
* **Sees:** the action bar's **`Open: in browser · on Mac`** compound control. Clicking **on Mac** hands
  the file to the OS default handler — QuickTime/VLC opens the clip natively — and a toast confirms
  *"Opened file on Mac."* Clicking **in browser** instead opens the raw streamed bytes in a new tab.
* **Note:** the **on Mac** half only appears on a **localhost** app (local mode + loopback); on a hosted
  instance it is hidden and only **in browser** shows (os_open.mdx §1, §6).

### UC-10 — "Show me this file in Finder so I can work with it alongside other files."
* **Enters:** any media viewer.
* **Sees:** the **`Open folder: in browser · on Mac`** compound. **on Mac** opens the file's **parent
  directory in Finder/Explorer**; **in browser** instead navigates the **in-app File System tab**
  (`/fs?path=<dir>`) so they stay inside LFBridge. Two clearly different destinations, one control
  (os_open.mdx §5).

### UC-11 — "My window is narrow and I can't see all the buttons."
* **Enters:** a media viewer in a small or split window.
* **Sees:** the action bar is an **overflow bar** — the lowest-priority buttons **fold into a trailing
  "More" (⋯) menu** instead of wrapping or clipping. **Move**, **View properties**, then the **Open** /
  **Open folder** compounds overflow first; inside **More** their in-browser and on-{label} halves appear
  as separate items (the OS items still gated to localhost). Nothing is lost — the row just changes shape
  (os_open.mdx §5.1).

### UC-12 — "What is said in this recording? I need it as text."
* **Enters:** an audio or video viewer.
* **Sees:** the bottom **Transcription** tab. If no transcript exists yet, a **Generate transcript**
  pane explains it runs **locally with Whisper — nothing leaves this computer**. Clicking it transcribes
  and fills the tab with the text (with an **open file** link to the `.txt`). If Whisper isn't installed,
  the button is disabled with the exact `pipx install openai-whisper` / `brew install ffmpeg` hint
  (Transcribe.mdx §2.1).

### UC-13 — "Describe what's in this video so I can find it later without re-watching it."
* **Enters:** a video viewer.
* **Sees:** the **AI description** tab with a **Generate description** pane that states plainly the
  **video is uploaded to the provider (Gemini)** — the one deliberate network upload, on an explicit
  click only. Generating returns a hyper-detailed description (overview, shot-by-shot timeline,
  cinematography shot sizes, per-person head-to-toe, location guess) stored beside the file so the
  Storage table's **description** indicator lights up (ai_description.mdx §1–§3, §7). **Only Gemini**
  describes video, so a Gemini key must be present.

### UC-14 — "Describe this photo — I don't have a Gemini key, just OpenAI."
* **Enters:** an image viewer.
* **Sees:** the **AI description** tab. An **image** can be described by **any** provider —
  **OpenAI (gpt-4o)** or **Grok** as well as Gemini — so with an OpenAI key configured, **Generate
  description** works; the pane notes the image is uploaded to the provider. Over the **18MB** inline cap,
  it fails with a *"compress it first"* message rather than uploading (ai_description.mdx §3.1, §3.3).

### UC-15 — "This video already has a transcript AND a description — let me flip between them."
* **Enters:** a described-and-transcribed video viewer.
* **Sees:** both tabs carry a **✓** (content exists). The panel **defaults to Transcription** (the
  preferred tab when it has content), and a click switches to **AI description**; the chosen tab sticks
  for the session. An image shows only the AI-description tab; an audio file only Transcription
  (ai_description.mdx §1).

### UC-16 — "I want the AI descriptions to always call out the license plates and street signs."
* **Enters:** **Settings → Tools**, the **AI description** section.
* **Sees:** a **video / image** prompt toggle and the in-use prompt in an editable textarea. Editing and
  **Save (customize)** writes a **per-computer override** (the marker flips to *"customized on this
  computer"*); that text is then sent to the provider in place of the shipped default. **Reset to
  default** reverts it. The change is local — it only chooses which prompt text is sent
  (ai_description.mdx §4.2, §6).

### UC-17 — "I clicked Generate description and it's greyed out. Now what?"
* **Enters:** the AI-description tab with **no provider configured**.
* **Sees (disabled state, not a dead end):** the **Generate** button is disabled with a plain hint —
  *"No AI provider is configured. Add a Gemini, Grok, or OpenAI API key in Settings → Tools (or export
  GEMINI_API_KEY / XAI_API_KEY / OPENAI_API_KEY), then reload."* **Settings → Tools** shows each provider
  as **● configured** or **✗ no key** with the exact env var / config key to add, and reminds them **only
  Gemini describes video**. Adding a key and reloading enables the button (ai_description.mdx §6, §8).
