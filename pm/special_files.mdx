## 1. What makes a file special (LOCKED)

A file is **special** when **ANY** of the following categories applies. The categories are **not
mutually exclusive** â€” a 4K `trees.mov` that is git-ignored and pinned is `large` **and** `video`
**and** `ipfs_pinned` at once.

| Category | A file is in it whenâ€¦ | Reuses |
| --- | --- | --- |
| **`large`** | It satisfies the scan **payload rule**: git-ignored media of **any** size, OR any file at/above the effective big-file threshold (default 100 MB), OR a forced `include_globs` match. | [scan.mdx Â§4.1](./scan.mdx) payload rule; `isMediaFile`, effective threshold. |
| **`ipfs_pinned`** | It is **pinned on this computer's IPFS node** â€” whether LFB pinned it, or it was **pinned outside LFB** (by hand on the CLI or another tool) and a scan **observed** it ("Untracked (pinned)"). | [ipfs.mdx Â§1/Â§4](./ipfs.mdx); the 15-min pinset read ([scan.mdx Â§2.1](./scan.mdx)). |
| **`video`** | Its type is a **video** ([Transcribe.mdx](./Transcribe.mdx) `mediaKindForName` = video; `VIDEO_EXT`). | `mediaKindForName`, `VIDEO_EXT`. |
| **`image`** | Its type is an **image** (`IMAGE_UNCOMPRESSED_EXT` and the broader image set). | `isMediaFile` image branch. |

* **Media is special regardless of size.** A 5 MB clip is still `video` and still earns a sidecar â€” this
  matches the payload rule that deliberately does **not** gate media on the 100 MB threshold
  ([scan.mdx Â§4.1](./scan.mdx); [repos.mdx Â§4.1](./repos.mdx) "counts include git-ignored media of *any*
  size"). This is why `charlie-kirk` (dozens of 5â€“100 MB clips) is full of special files, not "nearly
  empty."
* **Already-pinned is special even if small.** A file someone pinned on the CLI is `ipfs_pinned` and
  therefore special â€” LFB must track it, whether or not it is otherwise large or media. The **provenance**
  (LFB pinned it vs. observed pinned outside us) is recorded in the sidecar as an `observed` /
  `not-lfbridge` event ([repo_tracking_scheme.mdx Â§3.3](./repo_tracking_scheme.mdx)).

## 2. The `compressible` sub-flag â€” video/image/audio that look uncompressed

For **media** categories (`video`, `image`, `audio`) LFB additionally asks: **is it compressed or not?**
This is the offer-driving question the charter cares about ("many large files people sync are
uncompressed when they don't need to be").

* **Verdict source.** `compressible` is the [compression.mdx](./compression.mdx) verdict: the file's
  **duration Ă— pixel-resolution** puts its size **above** the learned "typical uncompressed" band of the
  baseline (mean + 1Ïƒ/2Ïƒ), or it is a losslessly-stored format (PNG, etc.) that a safety-checked
  conversion could shrink. A file already stamped with the in-file `LFBcompressed;v1;<codec>` marker, or
  whose size sits in the "compressed" band, is **not** compressible.
* **Scope.** Primarily **video**, secondarily **image**; **audio** may be compressible too. Nothing else
  is offered for compression ([charter](../CLAUDE.md)).
* **It is an offer, never an action.** `compressible: true` only lights the **C** indicator and feeds the
  "N videos/images can be compressed" warnings ([warnings.mdx Â§10.5](./warnings.mdx)); LFB never
  compresses without an explicit click ([compression.mdx Â§1](./compression.mdx)).

## 3. The `transcribable` sub-flag â€” video (has audio) and audio

For **video** and **audio** LFB asks: **could this be turned into text?**

* **Video is transcribable** because it carries an **audio track** â€” LFB demuxes the audio and runs it
  through Whisper ([Transcribe.mdx Â§1](./Transcribe.mdx)). A video with **no** audio stream is still
  classified transcribable, but transcription writes a short "no audio to transcribe" placeholder rather
  than failing ([Transcribe.mdx Â§1](./Transcribe.mdx)).
* **Audio is transcribable**, optionally â€” the user may or may not transcribe it.
* **Images are not transcribable** (they get an AI *description* instead â€” [ai_description.mdx](./ai_description.mdx)).
* **It is an offer, never automatic.** `transcribable: true` enables the **Create Transcriptions** action
  and the viewer's Transcribe button; nothing is transcribed until the user asks. Whether a transcript
  **exists yet** is tracked separately (the `transcribe` event in the sidecar, and the
  `repo_storage.yaml â†’ counts.transcribed`).

## 4. What "special" unlocks

Being special is the gate for the rest of the product's per-file machinery:

* **A per-file sidecar.** Only special files get a `.lfbridge/files/<path>/<name>.yaml` sidecar
  ([repo_tracking_scheme.mdx Â§3](./repo_tracking_scheme.mdx)) with identity, the perceptual fingerprint
  (media only â€” [perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx)), and the event history.
* **The repo rollup.** Every category and sub-flag increments the `repo_storage.yaml â†’ counts:` block
  ([repo_tracking_scheme.mdx Â§2.2](./repo_tracking_scheme.mdx)) and the [Repos](./repos.mdx) table's
  per-repo counts, and drives the four category-rollup rows ([storage_repo.mdx](./storage_repo.mdx), the
  charter category table).
* **The warnings/offers.** `compressible` â†’ the "can be compressed" offers; `large` + not-git-ignored â†’
  the "big files aren't git-ignored" offer; `ipfs_pinned` but untracked â†’ the Import offer;
  `transcribable` â†’ the transcription action ([warnings.mdx Â§10](./warnings.mdx)).

Filling `repo_storage.yaml â†’ counts:` and the Repos-table counts from `classifySpecial` over the repo's files is handled in code:

## 5. Acceptance criteria

1. A single classifier decides **special**: a file is special when it is **`large`** (payload rule),
   **`ipfs_pinned`** (by us or observed pinned outside us), or a **`video` / `image` / `audio`** file.
   Categories are **not exclusive** â€” a file may carry several.
2. Media is special **regardless of size**; an already-pinned file is special **even if small**; the
   classifier **reuses** the existing `isMediaFile` / `mediaKindForName` / payload-rule / pinset signals
   rather than re-deriving them.
3. The classifier also returns two sub-flags: **`compressible`** (media that looks uncompressed vs. the
   [compression.mdx](./compression.mdx) baseline; primarily video, secondarily image, sometimes audio)
   and **`transcribable`** (video â€” because it has an audio track â€” and audio).
4. Both sub-flags are **offers, never actions** â€” they light indicators and feed warnings, but LFB never
   compresses or transcribes without an explicit user click.
5. Only special files earn a per-file **sidecar** and count into **`repo_storage.yaml â†’ counts:`** and the
   Repos-table counts; the categories/sub-flags drive the compression, git-ignore, import, and transcribe
   surfaces.
