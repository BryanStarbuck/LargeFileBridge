## 1. What "the engine" is (LOCKED)

The engine is the copied-in transcription layer ([Transcribe.mdx ┬¦5](./Transcribe.mdx#5-the-engine--copied-in-called-in-process-locked))
ŌĆö in-process, async, non-blocking. This spec extends it with **engine selection** (three engines in a fixed
preference order), **model provisioning** (for the heavyweight Qwen third choice), **full-duration
guarantees**, and **parallel calibration**. Nothing here changes where the transcript is written (that is
[Transcribe.mdx ┬¦3](./Transcribe.mdx#3-where-transcripts-are-stored--inside-the-owning-roots-tracking-area-path-mirrored-extension-appended-locked--supersedes-the-earlier-beside-media-layout)) or that transcription is always an explicit user action.

The engine abstraction (a common interface each concrete engine implements ŌĆö `available()` /
`transcribeToFile(onProgress)` ŌĆö so the caller is engine-agnostic) and the ordered selector are implemented
in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/engine.ts` ŌĆö `pickEngine()` (returns the best engine for this machine) + `transcribeWithEngine()` (the speech ŌåÆ mac ŌåÆ qwen runtime fallback chain).

## 2. Three engines ŌĆö Apple SpeechAnalyzer (preferred) ŌåÆ Whisper Small ŌåÆ Qwen3-ASR (LOCKED)

There are **three** transcription engines, tried in a fixed **preference order** (learned from the canonical
tool at `~/BGit/all/tools/Transcription/Transcribe.js`, which LFBridge's copied engine mirrors). All three
produce the **same** `.transcription` output ([Transcribe.mdx ┬¦3ŌĆō┬¦4](./Transcribe.mdx)); they differ only in
how the speech-to-text is computed. **We never use Apple's legacy `SFSpeechRecognizer`** ŌĆö only the new
SpeechAnalyzer.

| Pref | Engine id | What it is | When it's used |
| --- | --- | --- | --- |
| **1 ŌĆö PREFERRED** | **`speech`** (Apple SpeechAnalyzer) | **Apple SpeechAnalyzer + SpeechTranscriber** (label `apple-speechanalyzer`), Apple's **NEW** on-device speech engine (macOS 26+ / "Tahoe"). The model **ships inside the OS** ŌĆö no multi-GB download; the per-locale asset auto-installs on first use via `AssetInventory`. Driven by a tiny **Swift helper** (`speech/SpeechAnalyzerCLI.swift`) compiled once with `swiftc` and cached. On the first published benchmark it beat every Whisper model (including Small) on English, at ~3├Ś Whisper Small's speed, fully on-device. | **Auto-preferred on a Mac** running **macOS 26+** with the Xcode command-line tools (`swiftc`) present. |
| **2** | **`mac`** (Whisper Small) | **OpenAI Whisper** CLI at model **`small`** (`whisper-small`, ~460 MB), preferring the Apple-Silicon **MPS** GPU with an automatic **CPU fallback** and an English sanity-check. Reliable, cross-platform, English-tuned; installs via `pipx install openai-whisper`. | When SpeechAnalyzer is unavailable (non-Mac, macOS < 26, or no `swiftc`), or **automatically when a `speech` run errors** (┬¦2.1). This is the **second choice** across all hardware. |
| **3** | **`qwen`** (heavyweight LLM ŌĆö "another LLM") | **Qwen3-ASR via Apple MLX** ŌĆö the `mlx-qwen3-asr` CLI at **`Qwen/Qwen3-ASR-1.7B`** (label `qwen3-asr-1.7b-mlx`). Apple-Silicon-native. The **multi-GB download** engine: `pipx install mlx-qwen3-asr` + **~3.4 GB weights** from Hugging Face on first run into `~/.cache/huggingface/hub/`. **Apple Silicon (M1+) only.** | The **third choice** ŌĆö used only when **explicitly selected** (`transcribe.engine: qwen`) or reached as the **last runtime fallback** after both `speech` and `mac`. It is **not** auto-selected. |

* **Selection order.** `pickEngine()` returns **`speech`** when the machine is a Mac on macOS 26+ with
  `swiftc` present; otherwise **`mac`** (Whisper Small). `qwen` is returned only when the
  `transcribe.engine` setting pins it (and it is available), else that pin degrades to `mac`. The
  `transcribe.engine` setting (`auto` / `speech` / `mac` / `qwen`) overrides the default (┬¦5.4).
* **Auto-fallback (┬¦2.1).** A run that errors retries **down the order** (speech ŌåÆ mac ŌåÆ qwen) so a file is
  never lost ŌĆö it just costs a retry.
* **Quality + honesty.** SpeechAnalyzer Ōēź Whisper Small Ōēź whisper-base on English; Qwen3-ASR-1.7B is a strong
  LLM alternative. The transcript **header records which engine actually ran**
  ([Transcribe.mdx ┬¦4](./Transcribe.mdx#4-what-a-transcript-file-contains)) so the user always knows ŌĆö
  including when a fallback produced it.
* **All three must satisfy ┬¦4** (full-duration, no truncation). SpeechAnalyzer reads the whole `AVAudioFile`
  to end-of-input (`analyzeSequence(from:)` ŌåÆ `finalizeAndFinishThroughEndOfInput()`), so a successful run
  inherently covers the full duration; Whisper and Qwen are verified per ┬¦4.2.

The three concrete engines + the selector are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/speech-analyzer.ts` ŌĆö class `SpeechAnalyzerTranscriber` (compiles/drives the Swift helper) ┬Ę `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/Transcribe.ts` ŌĆö class `Transcriber` (Whisper **Small**, MPSŌåÆCPU) ┬Ę `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/qwen-asr.ts` ŌĆö class `Qwen3AsrTranscriber` (drives `mlx-qwen3-asr`) ┬Ę `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/engine.ts` ŌĆö `pickEngine()` + the speech ŌåÆ mac ŌåÆ qwen fallback wrapper. The Swift helper is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/speech/SpeechAnalyzerCLI.swift`.

### 2.1 Auto-fallback `speech ŌåÆ mac ŌåÆ qwen` (LOCKED)

The selected engine **transparently falls back** to the next engine in the preference order on any error,
exactly as the reference tool does: a `swiftc` build failure or a macOS-too-old exit from `speech`, an MPS
hiccup from `mac`, a missing-weights error from `qwen` ŌĆö none loses the file; it retries on the next engine
(installing that engine's tool first if needed), and the header reflects **whichever engine actually produced
the text**. A user who explicitly pins one engine (`transcribe.engine: mac`, say) gets that engine with no
fallback layer, so a pinned failure is reported honestly rather than silently switched.

### 2.2 Hardware supports SpeechAnalyzer but the OS is too old ŌĆö warn + fall back (LOCKED)

A common real case: an **Apple-Silicon Mac whose hardware could run SpeechAnalyzer, but the OS is older than
macOS 26**, so the SpeechAnalyzer/SpeechTranscriber APIs are not present. LFBridge must **not** fail silently
and must **not** pretend it used the best engine. Instead it:

* **Detects** the situation ŌĆö a Mac (`darwin`) whose `sw_vers -productVersion` **major < 26** (note: macOS 26
  is Darwin kernel 25, so we read the **product** version, never the kernel version).
* **Warns** ŌĆö the engine-status payload sets `speech.needsOsUpdate = true`, and the UI shows a notice:
  *"This Mac supports Apple's new on-device SpeechAnalyzer, but it needs a macOS update (macOS 26+). Install
  the latest macOS update for the highest-quality transcription. Falling back to Whisper Small for now."*
  The message **names the engine it fell back to** (`whisper-small`), so the user knows exactly what produced
  their transcripts and what upgrading would buy them.
* **Falls back** ŌĆö transcription proceeds on **Whisper Small** (`mac`), never blocked. Updating macOS to 26+
  (and having `swiftc`) is all it takes to flip the machine to the preferred `speech` engine automatically on
  the next run ŌĆö no other configuration.

Detection of the macOS product-version gate + the `needsOsUpdate` warning are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/audio-prep.ts` ŌĆö `macOSMajorVersion()` / `speechAnalyzerSupported()` / `speechAnalyzerNeedsOsUpdate()`, surfaced on `GET /api/transcribe/engine` (`speech.needsOsUpdate`) and rendered by the Settings Transcription panel (┬¦6).

## 3. Provisioning the heavyweight model ŌĆö download ŌåÆ install ŌåÆ configure, with permission (LOCKED)

> **Scope ŌĆö this section is about the `qwen` THIRD-choice engine only.** The **preferred** engine, Apple
> **SpeechAnalyzer**, needs **no** heavyweight provisioning: its model **ships inside macOS 26+**, and the
> per-locale asset auto-installs on first use via `AssetInventory` inside the Swift helper. SpeechAnalyzer's
> only "readiness" is the OS/toolchain gate of ┬¦2.2 (macOS 26+ and `swiftc`) ŌĆö there is no multi-GB download,
> no disk-space popup, and no consent flow for it. The **Whisper Small** second choice likewise just
> downloads its ~460 MB model on first `whisper` run (no popup). The permission-gated
> download ŌåÆ install ŌåÆ configure flow below exists **only** for the heavyweight Qwen3-ASR weights, which a
> user opts into **only** if they deliberately choose the `qwen` third engine.

The heavyweight `qwen` engine is a **multi-GB download**: the `mlx-qwen3-asr` CLI installs via
`pipx install mlx-qwen3-asr` (an isolated venv with `mlx`, `numpy`, `huggingface-hub`, ŌĆ”), and the
**`Qwen/Qwen3-ASR-1.7B` weights download from Hugging Face on first run** (~**3.4 GB**) into
`~/.cache/huggingface/hub/models--Qwen--Qwen3-ASR-1.7B/`. **Most users will not have it, and that is
normal** ŌĆö many people install LargeFileBridge and never use transcription at all. So we do **not** ship it
and we do **not** download it eagerly. We provision it **the first time the user actually tries to
transcribe** and it is absent ŌĆö **only with their explicit permission**. (Apple Silicon only; a non-Apple
machine simply uses the `mac`/Whisper engine, which needs no big download.)

### 3.1 Detecting readiness + estimating disk (LOCKED)

Before any transcribe runs, the engine reports a **readiness** for the heavyweight model, one of:

| Readiness | Meaning | What happens |
| --- | --- | --- |
| **`installed`** | The `mlx-qwen3-asr` CLI **and** the `Qwen/Qwen3-ASR-1.7B` weights are present and pass the health check (┬¦3.4). | Transcribe proceeds ŌĆö **no popup** (┬¦3.5). |
| **`missing`** | Not downloaded at all. | First-time **permission popup** (┬¦3.2) with the disk estimate. |
| **`partial`** | Half-installed ŌĆö weights present but runtime broken, or an interrupted prior download. | **Health-check/repair** (┬¦3.4): finish/redo only what's missing, then proceed. |
| **`outdated`** | Installed but an older model/runtime than we now target. | Offer to **update** (same download/configure flow, framed as an upgrade). |

* **Disk estimate ŌĆö measured, then stored.** We **find out** how much the install actually needs rather
  than guessing: probe the target model's published weight size and the runtime footprint, sum them with
  headroom, and **warn the user with that number** before they approve. After a real install we **store the
  measured actual size** so future estimates (and the Settings display) are exact, not ballparked. The
  estimate is a **range** ("about 3ŌĆō6 GB") until we have a stored measurement, then a firm number.
* We **also** check there is **enough free disk** for the estimate and warn if not ŌĆö never start a download
  that cannot finish.

Readiness detection + the (ballpark ŌåÆ stored) disk estimate are computed in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/model-provision.service.ts` ŌĆö `describeReadiness()` (returns `{ state, installedBytes, estimateBytes, freeDiskBytes }`; the stored measurement lives in the app config under `transcribe.model`).

> **Detection must survive a thin `PATH` (hardened).** `describeReadiness()` checks the `mlx-qwen3-asr` CLI
> with `commandExists()` and the weights with a direct filesystem read of the Hugging Face cache
> (`~/.cache/huggingface/hub/models--Qwen--Qwen3-ASR-1.7B/`, verified on a real machine to hold the full
> ~4.4 GB snapshot once installed) ŌĆö so the **weights** check never depends on `PATH` at all. The **CLI**
> check does, though, and pipx (which installs `mlx-qwen3-asr`) shims land in `~/.local/bin`, a directory
> that is on an interactive shell's `PATH` but is **not** guaranteed to be on a background/launchd process's
> ŌĆö see [Transcribe.mdx ┬¦5.2.1](./Transcribe.mdx#521-detection-detail--why-a-genuinely-installed-tool-must-never-be-reported-missing-locked)
> for the full detection-robustness writeup (measured proof that a minimal `PATH` finds none of `whisper`,
> `ffmpeg`, or `mlx-qwen3-asr`, while a Homebrew-only `PATH` still misses the two pipx tools). `commandExists()`
> therefore searches a **widened** `PATH` (current `PATH` plus `~/.local/bin` and both Homebrew prefixes) so a
> `partial`/`missing` readiness is never misreported for a machine that actually has the CLI installed ŌĆö this
> is the same `toolSearchPath()`/`toolEnv()` helper the tool-missing binary check (┬¦Transcribe.mdx ┬¦5.2) uses,
> shared so the two never disagree.

> **Every failure path here logs ŌĆö never a silent catch (LOCKED).** A `describeReadiness()` /
> `provision()` / `healthCheck()` / `repair()` / `removeModel()` / `recordConsent()` failure always writes a
> reason to `error.err` via the shared `log` (`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/logging.ts`)
> before returning/throwing ŌĆö e.g. a broken `statfsSync` (free-disk check silently disabled for that request),
> a `pipx`/`brew` bootstrap failure during `installCli()`, or a config-write failure in `recordConsent()` that
> would otherwise make the popup nag again with no trail of why. This applies equally to the
> `transcribeConcurrency()` admission path (┬¦5 below): a transient config read failure there logs a warning
> and keeps the last-known engine/RAM key rather than failing the whole batch silently.

### 3.2 The first-time permission popup (LOCKED)

When readiness is `missing` and the user takes a transcribe action, we do **not** silently download. We
show a **one-time permission popup**:

* **What it says.** That transcription uses a powerful local model that must be **downloaded and installed
  once**, that it will use **about {estimate}** of disk (the ┬¦3.1 number), that it runs **entirely on this
  computer afterwards** (no per-use network, no content leaves the machine), and that they can instead use
  the **Whisper-on-Mac** engine (┬¦2) without downloading anything.
* **The choice.** **Download & install** (primary), **Use the Whisper-on-Mac engine instead** (secondary, on
  macOS), or **Cancel**. Approval is the charter's explicit consent ŌĆö nothing large is fetched without it.
* **Shown once.** The decision (approved / declined / use-fallback) is **remembered** so the popup does not
  nag on every file; the user can change it later in **Settings ŌåÆ Transcription** (┬¦6).

The permission popup + the remembered decision are rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/TranscribeModelConsentDialog.tsx` ŌĆö `TranscribeModelConsentDialog` (opened via the same global request bus the storage wizard uses, `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/setupWizard.ts`).

### 3.3 Two progress UIs ŌĆö download, then install/configure (LOCKED)

On approval, provisioning runs as **two sequential progress operations**, each its own dock/Processing job
([processing.mdx](./processing.mdx), [webapp.mdx ┬¦11](./webapp.mdx)):

1. **Download** ŌĆö a `download` op with a **determinate byte bar** ("Downloading transcription model ŌĆö 1.8
   GB / 3.0 GB"). Resumable where the source supports it; a failed/partial download leaves readiness
   `partial` (┬¦3.4), never a corrupt "installed".
2. **Install & configure** ŌĆö an `install` op ("Installing & configuring the transcription model") that puts
   the weights in their cache dir, ensures the `whisper` runtime + its Python deps, and runs the **health
   check** (┬¦3.4). Determinate where step counts are known, else a labeled spinner with streamed detail.

Both surface on the **Processing page** and the dock like any other background work, and the triggering UI
shows an inline state ("Setting up the transcription modelŌĆ”") so the user knows why their transcribe is
waiting.

The two-phase provisioning job (download then install/configure, each a registry op) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/model-provision.service.ts` ŌĆö `provision()` (registers `track("download", ŌĆ”)` then `track("install", ŌĆ”)` from `modules/progress/progress.registry.ts`), served by `POST /api/transcribe/engine/provision` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.router.ts`.

### 3.4 Health check & repair ŌĆö half-installed or outdated (LOCKED)

An install can be interrupted (a killed download, a machine sleep, a partial pip). So provisioning is
**idempotent and self-healing** ŌĆö we never assume "the folder exists ŌćÆ it works":

* **Verify, don't trust.** The health check confirms: the `mlx-qwen3-asr` CLI is on `PATH`; the
  `Qwen/Qwen3-ASR-1.7B` snapshot exists in the Hugging Face cache with a **non-trivial size**; and a **tiny
  smoke transcription** of a bundled ~2-second clip returns sane text. Only all-pass ŌćÆ `installed`.
* **Repair only what's broken.** A `partial` state re-does **just** the failed step (re-`pipx install` the
  CLI, or re-trigger the weights download) rather than wiping and restarting ŌĆö fast, and safe to run
  repeatedly. On non-Apple-Silicon, "repair" is a no-op and selection falls to `mac`.
* **Outdated ŌćÆ update.** When the target model/runtime version moves ahead of what's installed, readiness is
  `outdated` and the same flow runs as an **upgrade** (with the user's ok).

The verify-then-repair health check (checksum + import + smoke test; per-step repair) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/model-provision.service.ts` ŌĆö `healthCheck()` / `repair()`.

### 3.5 Skip the popup when already installed (LOCKED)

When readiness is `installed`, the whole permission/provisioning UI is **skipped entirely** ŌĆö the transcribe
runs immediately. The popup, the two progress UIs, and the consent prompt exist **only** for the
first-time / broken / outdated cases. A user who provisioned the model months ago never sees any of it again.

### 3.6 Flow-resume after provisioning ŌĆö return to exactly where they were (LOCKED)

Provisioning interrupts something the user was doing; when it finishes we **resume that exact thing**. Two
cases, both required:

* **They had already CONFIRMED a transcribe.** (e.g. they left all 58 files checked in the transcribe popup
  and hit the blue **Confirm** ŌĆö [transcribe_calc_engine.mdx ┬¦4](./transcribe_calc_engine.mdx),
  [task_tabs.mdx ┬¦5.1](./task_tabs.mdx) ŌĆö and only *then* did we discover the model was missing.) After the
  install completes, we **proceed with the transcription** they confirmed: the `transcribe` jobs enqueue and
  the **Processing** page shows them running. The UI keeps them informed (a bottom toast ŌåÆ the Processing
  tab powering the run).
* **They had only OPENED the offer UI.** (e.g. they clicked the Transcribe metric/icon which *opened* the
  choose-what-to-transcribe window, but had **not** confirmed ŌĆö and we prompted to install first.) After the
  install completes, we **return to the right page and re-open the window they left off on**, now showing the
  **"continue to transcribe"** option ŌĆö so they finish the choice they were making, nothing lost.

The distinction is a **pending intent** captured before provisioning starts (`confirmed` vs `offered`) and
replayed after. The pending-intent capture + replay is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/transcribe.ts` ŌĆö `withModelReady(intent, run)` (records the intent, awaits `provision()`, then either enqueues the confirmed set or re-opens the offer popup on the originating route).

> **`withModelReady` gates every transcribe launcher ŌĆö no exceptions (hardened).** Both frontend entry
> points that start a single-file transcription ŌĆö the Media Viewer button/tab
> (`useTranscribeFile()`) and the Ōŗ« kebab / row action (`runTranscribeFile()`) ŌĆö call `withModelReady()`
> before ever invoking `api.transcribeFile()`. A prior defect had the Media Viewer hook bypass this
> entirely (calling the transcribe API directly), which meant the first-time consent popup **never** showed
> for that button ŌĆö the model would silently resolve to whichever fallback was available with no chance for

## 4. CRITICAL ŌĆö a 1ŌĆō2-hour-plus file MUST transcribe in FULL, never truncated (LOCKED)

**This is a hard, load-bearing requirement, not a nicety.** We routinely transcribe media that is **one or
two hours long (and sometimes longer)**. Whichever engine runs ŌĆö the heavyweight Qwen3-ASR **or** the
Whisper-on-Mac fallback ŌĆö the settings and the pipeline are **triple-checked** so the transcript covers the
**entire** source. A run that transcribes the first 20 minutes and stops, or silently drops the tail, is a
**failure** ŌĆö it must **never** be reported or stored as a success.

### 4.1 The failure modes we design against (LOCKED)

Long-audio truncation has specific, known causes; each is closed off:

* **No time/segment limit is ever passed.** We never pass a flag that caps duration (no `--clip_timestamps`
  bound, no max-segment cutoff). Whisper's default is to slide its 30-second window across the **whole**
  file; we keep it that way.
* **No process/HTTP timeout kills a long run.** The engine is async ([Transcribe.mdx ┬¦5.1](./Transcribe.mdx#51-async--non-blocking--the-generate-transcript-didnt-work-fix-locked));
  the spawn has **no wall-clock timeout**, and the request may stay open for the full run without being
  reaped. A 2-hour file at RTFŌēł2ŌĆō3 on CPU can take hours ŌĆö that is allowed.
* **Enough scratch + memory.** The videoŌåÆMP3 demux and the model both need headroom. We ensure **temp-disk
  space** for the demuxed audio of a long file and pick settings/chunking (┬¦4.3) that fit the machine's RAM
  so the OS never OOM-kills the run mid-file.
* **The MPSŌåÆCPU fallback must not accept a partial.** The existing Apple-Silicon MPS path re-runs on CPU when
  it emits garbage; we additionally require the **English/sanity check to run against a FULL-length output**,
  not a truncated one ŌĆö a short MPS result is rejected and CPU re-runs the whole file (┬¦4.2).
* **Both CLIs transcribe full-length audio natively.** Neither `mlx-qwen3-asr` nor `whisper` caps duration ŌĆö
  Whisper slides its 30-second window over the **whole** file and Qwen3-ASR ingests the full audio ŌĆö so a
  long file is transcribed end-to-end by default. We never pass a limiting flag, and for the very longest /
  memory-tight files we **chunk-and-stitch** (┬¦4.3) to bound peak memory rather than risk an OOM kill.

### 4.2 Verify coverage end-to-end (LOCKED)

We do not merely hope it finished ŌĆö we **verify**:

* The engine reads the **source duration** (`ffprobe`) and, on completion, checks the **last decoded segment
  end timestamp** (whisper) or the **last recognized time** (Apple) against it.
* If coverage is **Ōēź (duration ŌłÆ tolerance)** (a few seconds), the run is **complete**: the header records
  `Transcript covers: HH:MM:SS Ō£ō full` ([Transcribe.mdx ┬¦4](./Transcribe.mdx#4-what-a-transcript-file-contains)).
* If coverage is **short**, the run is **incomplete ŌåÆ it is a failure**: we do **not** write a "done"
  transcript. We retry per ┬¦4.3 (chunked / CPU), and if still short, report an honest **failed** with the
  covered-vs-total figures ŌĆö surfaced on [Processing](./processing.mdx). A previously-stored short transcript
  makes the file a **re-transcribe candidate** ([transcribe_calc_engine.mdx ┬¦2](./transcribe_calc_engine.mdx)).

### 4.3 Chunk-and-stitch for very long / memory-tight runs (LOCKED)

For files beyond a length the machine can hold in one pass ŌĆö or after a first attempt comes back
short ŌĆö the engine **splits the audio into overlapping chunks** (e.g. ~10-minute windows with a few seconds
of overlap), transcribes each, and **stitches** them into one transcript (de-duplicating the overlap). This:

* bounds peak memory (a 2-hour file never needs the whole thing decoded at once),
* makes a failure **recoverable per chunk** (retry the one chunk, not the whole 2 hours), and
* still yields **one** `.transcription` covering the full duration, verified by ┬¦4.2.

Chunking is **transparent** ŌĆö the output is a single stitched transcript; the header still shows full
coverage. It is used when needed, not always (a short file runs in one pass).

The duration-coverage verification, the no-timeout long spawn, and the chunk-and-stitch path are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/Transcribe.ts` ŌĆö `transcribeToFile()` (Whisper; coverage check via `parseSegmentEndSec()` vs `probeDurationSec()`, MPSŌåÆCPU on a short/garbled result), and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/qwen-asr.ts` ŌĆö `Qwen3AsrTranscriber.transcribeToFile()` (drives `mlx-qwen3-asr`; coverage verified against `probeDurationSec()`). The `engine.ts` wrapper applies the shared chunk-and-stitch and the `qwenŌåÆmac` auto-fallback.

## 5. Per-machine parallel calibration (LOCKED)

Transcription is **resource-heavy**, and how many files we run **at once** must be **calibrated to the
machine**, not fixed. The product owner: *"A lot of lower-power computers can only handle one in parallel,
but my computer [~24 cores] should be able to handle several more in parallel."* This section is the
transcribe-specific application of the shared **Core Budget** ([parallelization.mdx](./parallelization.mdx)).

### 5.1 The transcribe concurrency formula (LOCKED)

Batch transcription draws the **mass-compute budget** ([parallelization.mdx ┬¦1](./parallelization.mdx)) but,
because Qwen3-ASR is **internally multi-threaded and memory-heavy**, it fans **narrow** and is
additionally **RAM-clamped**:

```
transcribeConcurrency = clamp(
  1,                                             // never below 1 ŌĆö every machine can do one
  floor(coreBudget() / whisperThreadsPerJob),    // CPU: don't exceed the ~90%-of-cores budget
  floor(usableRAM / modelRamPerJob),             // RAM: each Qwen3-ASR job needs several GB
  gpuStreams                                     // GPU: how many streams the MPS/VRAM allows (Ōēź1)
)
```

* **`whisperThreadsPerJob`** ŌĆö a heavy ASR job uses multiple threads/GPU per file; default ~**4** so
  `jobs ├Ś threads Ōēł budget`, never `cores┬▓` ([parallelization.mdx ┬¦2](./parallelization.mdx)).
* **`modelRamPerJob`** ŌĆö the measured resident footprint of one job: **`qwen` (Qwen3-ASR-1.7B via MLX)**
  holds the model in **unified memory** (several GB per instance); **`mac` (whisper-small)** is much lighter
  (~1 GB). This is the clamp that keeps a **16 GB laptop at 1** `qwen` job while a **64 GB / 24-core
  workstation runs several**.
* **The `mac` fallback** is lighter, so it resolves to a **higher** concurrency than `qwen` on the same box.
* **Result:** low-power boxes converge on **1 at a time** (correct ŌĆö one heavy run is all they can hold);
  the product owner's ~24-core / large-RAM machine runs **several** jobs concurrently, filling the budget
  without thrashing.

### 5.2 The minimum bar to run Qwen3-ASR (LOCKED)

A machine that cannot hold even **one** Qwen3-ASR instance (too little RAM/VRAM) is **not** forced to ŌĆö
`pickEngine()` (┬¦2) routes it to the **Whisper Small** fallback (`whisper-small`) instead of OOM-crashing.
The bar is a simple RAM/VRAM floor checked at selection time.

### 5.3 Calibrate once, cache it (LOCKED)

The core count, usable RAM, and the model's measured footprint are read **once** and cached (refreshed when
the model or the parallelism knob changes), so every batch doesn't re-probe the machine. The resolved number
is **shown to the user** on the Processing page's worker-utilization read
([processing.mdx ┬¦3a](./processing.mdx#3a-worker-utilization--the-parallelism-read-locked)) ŌĆö e.g. *"3 / 3
transcription workers busy."*

### 5.4 The knobs (LOCKED)

* **`performance.max_core_fraction`** (default 0.9) ŌĆö the shared Core-Budget fraction
  ([parallelization.mdx ┬¦4](./parallelization.mdx)); tunes the CPU term above.
* **`transcribe.max_parallel`** (optional override) ŌĆö a hard cap the user can set in **Settings ŌåÆ
  Transcription** for machines where the auto-calibration is too aggressive/conservative; blank = auto (┬¦5.1).
* **`transcribe.engine`** ŌĆö `auto` (default; ┬¦2 selection order ŌĆö `speech` when the Mac qualifies, else `mac`)
  ┬Ę `speech` ┬Ę `mac` ┬Ę `qwen`.

> **Concurrency note.** The `speech` engine is **light** (the model lives in the OS; the Swift helper streams
> the file) and clamps like Whisper Small rather than the heavyweight Qwen3-ASR ŌĆö so on the preferred path a
> machine runs **more** files in parallel than it would on the `qwen` third engine, whose `modelRamPerJob`
> (┬¦5.1) is several GB. `pickEngine()` still routes a machine that cannot hold one `qwen` instance away from
> it (┬¦5.2); that clamp never applies to `speech`/`mac`.

The transcribe concurrency calibration (the clamp, the RAM/GPU terms, the min-bar routing) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe-concurrency.ts` ŌĆö `transcribeConcurrency()` (draws `coreBudget()`/`logicalCores()` from `shared/concurrency.ts`, clamps by measured `modelRamPerJob` + `usableRAM` + `gpuStreams`), read by the queue's `transcribe` admission bucket in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts`.

## 7. Acceptance criteria

1. **Three engines in a fixed preference order (┬¦2):** **Apple SpeechAnalyzer** (`speech`, preferred) ŌåÆ
   **Whisper Small** (`mac`, second) ŌåÆ **Qwen3-ASR** (`qwen`, third). `pickEngine()` selects `speech` on a Mac
   running macOS 26+ with `swiftc`, else `mac` (Whisper **Small**, not base); `qwen` is used only when pinned

   The transcript header records which engine actually ran.
1a. **Apple SpeechAnalyzer path (┬¦2, ┬¦2.2):** the `speech` engine drives Apple's new on-device
