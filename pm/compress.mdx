# Large File Bridge — Compress

**Compression is a core value** ([charter — Compression](../CLAUDE.md#compression-a-core-value)): people
sync huge files that are **uncompressed when they don't need to be**. Large File Bridge finds them and —
**only when the user asks** — shrinks them without visible quality loss. This spec is about **the Compress
action itself**: the one thing it does, the one page that runs it, and the **action links scattered across
the product** that a user clicks to "go off and compress." It covers the two headline transforms the user
cares about — **PNG → JPEG** and **video re-encode to a smaller codec** — the **medium-quality** policy, the
**"may we change the file extension?"** setting, and the **hard data-safety guarantee** (a compress must
never leave a broken file behind).

This is the **user-and-surfaces** view. The **engine internals** (tool detection, plan building, the
already-compressed marker, the parallel job queue) live in the sibling engine spec and are **not repeated
here**:

  the temp-file → verify → recoverable-replace flow, before/after fingerprints, the in-file marker.
* [compress_inside.mdx](./compress_inside.mdx) — the **"Compress videos & images inside a directory"**
  dialog + per-file-safe background batch.

## 1. The one action, in one sentence

**Compress** takes a single image or video file, re-encodes it to a smaller, quality-controlled form —
optionally in a **better file format** (PNG → JPEG, an old video codec → H.264) — while keeping the **exact
same pixel resolution**, and swaps the new file in **only after proving the new file is valid**; the
original is moved to a **recoverable trash**. It is **always explicit** — never automatic.

Everything the action does — from the click to the replaced file — is orchestrated by one backend function.

* The whole per-file compress (plan → alpha check → transcode → verify → recoverable replace → record) runs in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile()` (line 552).

## 4. PNG → JPEG (the headline image transform)

The most common win: a **PNG that is really just a photo** (no transparency) is **much** larger than the
same picture as a **JPEG**. Compress converts it. This is a **format change** — governed by the setting in
§7 — and it is guarded by an **alpha-channel safety check** so we never silently throw away transparency.

How it works, in order:

1. **Is the alpha actually used?** Having an alpha channel is not the same as using it. We ask ImageMagick
   `identify -format '%[opaque]'`: `True` = fully opaque (alpha unused, safe to drop) → PNG→JPEG is allowed.
   `False` = real transparency present → **JPEG is refused**; we keep PNG (lossless recompress) or, if
   allowed, target **lossless WebP** (which preserves alpha). Undeterminable → **fail safe** (refuse).
   * The alpha probe + the safe/steer decision run in code:
     `~/…/backend/src/modules/compress/compression.service.ts` — `imageAlphaUsed()` (line 224), consumed by `pickImageTarget()` (line 301) and surfaced by `checkFile()` (line 390).
2. **Pick the target.** With the format-change setting on and no transparency in the way, the preferred,
   allowed, available image codec wins — **JPEG** by default (**JPEG 2000** only if the user prefers it and
   has not denied it).
   * Target/codec selection for images runs in code:
     `~/…/backend/src/modules/compress/compression.service.ts` — `pickImageTarget()` (line 301), against the `IMAGE_TARGETS` table (line 250).
3. **Re-encode at medium quality**, keeping the exact resolution (no resize).
   * The ImageMagick / oxipng / cwebp command is built in code:
     `~/…/backend/src/modules/compress/compression.service.ts` — `imageCommand()` (line 749).

### 4.1 Lossless-source images that stay put

A PNG the user wants to **keep** as PNG (or the quality is set to **lossless**) is recompressed **losslessly**
with `oxipng` — same pixels, smaller bytes, no quality lost.

* The PNG lossless path is chosen and built in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `pickImageTarget()` (line 301, the `LOSSLESS_IMAGE_EXT` branch line 261) → `imageCommand()` oxipng branch (line 773).

### 4.2 HEIC / HEIF / AVIF → JPEG (Apple photos)

Apple photos are byte-efficient but **poorly supported** by the sites people post to, so the default is a
**compatibility conversion to JPEG**. It decodes the **primary still only** (never a thumbnail, depth-aux,
or Live-Photo motion frame) and is **exempt from the "must shrink" rule** (JPEG can be larger than HEVC).

* HEIC-family handling (primary-still `[0]` decode, JPEG/WebP target, size-gain exemption) runs in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `pickImageTarget()` HEIC branch (line 323) + `imageCommand()` HEIC branch (line 764); the `HEIC_FAMILY_EXT` set is at line 81. Full policy in [images.mdx §4](./images.mdx).

## 5. Video → a smaller codec (the headline video transform)

For video, a better codec at the same resolution is a big win. Compress runs **ffmpeg** to re-encode to the
user's preferred codec — **H.264 by default** (the social-media-safe choice) — at a **medium CRF**,
**preserving resolution** (no scale), copying the audio stream through untouched.

* Video target/codec + CRF selection runs in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `pickVideoTarget()` (line 373) against `VIDEO_TARGETS` (line 256), with `videoCrf()` (line 269).
* The ffmpeg command (encoder, `-crf`, `-pix_fmt yuv420p`, audio copy, in-file marker) is assembled inside
  `compressFile()` in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `compressFile()` (line 552, the video `cmd` at line 617).

### 5.1 Video alpha safety

Most delivery codecs (H.264, HEVC, MP4) have **no** alpha. If the **source** carries a used alpha channel
and the target codec can't keep it, we **refuse** — unless the user's prefs include an alpha-preserving
target.

* The video alpha check (`pix_fmt` vs. target capability) runs in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `videoInfo()` (line 233) + `pixFmtHasAlpha()` (line 244), decided in `checkFile()` (line 390).

### 5.2 "Convert to H.264" for compatibility

When a video's codec won't play in the browser, the viewer offers **Convert to H.264 (MP4)** — the same
engine with a **forced** codec that overrides the user's preferred list and pins the output to the
universal-safe H.264. Like HEIC→JPEG, it is exempt from the "must shrink" rule.

* The forced-codec path runs in code:

## 6. Medium quality — give up a little size, keep the quality

We deliberately target the **middle** of every tool's quality↔size range — not the tiniest-possible file
that visibly degrades, and not maximum quality that barely shrinks. We only use tools that expose an
**explicit numeric quality knob** so the middle can be pinned:

* **Video** — ffmpeg `-crf` (0 = lossless … 51 = worst): **CRF 21** (H.264/AV1), **CRF 24** (HEVC).
* **Lossy image** — `-quality` / cwebp `-q` (0…100): **quality 85**.
* **Lossless image** — `oxipng` optimization level; pixels never change.

The four steps `low | medium | high | lossless` map to concrete numbers, with `medium` always the middle.

* The quality→number mappings run in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `videoCrf()` (line 269) and `jpegQuality()` (line 266).
* The `medium` default that anchors the middle lives in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — the `compression.images.quality` / `compression.video.quality` defaults (near line 141).

Deeper rationale (the "no middle = disqualified" rule) is in [compression.mdx §2.1](./compression.mdx#21-quality-control--the-middle-setting-hard-requirement-locked).

## 7. The "change the file extension?" setting (default YES, switchable to NO)

Some of the best wins **change the file's type** — PNG → JPEG, HEIC → JPEG. Whether Compress is *allowed*
to change the extension is a **setting**, and by **default it is YES** (we may convert). The user can flip
it to **NO** in Settings, and then Compress becomes **format-preserving**: it will still shrink a file
in place (a PNG stays a PNG via lossless oxipng; a JPEG re-encodes to a JPEG), but it will **never** rename
`.png` to `.jpg`.

* Master toggle: **`convert_types`** — default **ON** (may change format). Off = format-preserving only.
* Scope list: **`skip_exts`** — a per-extension **opt-OUT** (empty = every recognized type is in scope).

How the toggle changes behavior:

* When **`convert_types` is OFF**, target selection stays inside the source format in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `pickImageTarget()` format-preserving branch (line 309) and `pickVideoTarget()`'s `keepContainer` (line 384, keeps the source container instead of forcing `.mp4`).
* The per-extension opt-out is applied **before any probe** in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `checkFile()` `skipExts` check (line 406), normalized by `normExt()` (line 84).
* The settings are read/written in code:
  `~/…/backend/src/modules/compress/compression.service.ts` — `getCompressionSettings()` (line 38) / `setCompressionSettings()` (line 57); persisted via `compression.router.ts` `GET/PATCH /api/compress/settings` (lines 26 / 31); the settings UI is `~/…/frontend/src/pages/settings/SettingsPage.tsx`.

Full conversion policy (the two settings, per-extension scope, HEIC handling) is in

## 8. The data-safety guarantee — a compress must NEVER leave a broken file (LOCKED)

This is the **most important correctness rule** of the whole feature. Compressing a file means **replacing**
it, and a replace that ships a **broken** file is **data loss**. The engine is **transactional and
fail-closed**: it works on a **temp file**, and it only disposes of the original **after** the new file has
**passed every check**. A file that fails any check keeps its original **untouched**.

The checks, in the exact order they run inside `compressFile()`
(`~/…/backend/src/modules/compress/compression.service.ts`, line 552):

1. **Tool exit + non-empty.** The transcode must exit `0` and produce a file with **bytes > 0**. Else fail,
   delete temp, keep original.
   * `runAsync()` result check + `safeSize()` in code — `compressFile()` (line 611) / `safeSize()` (line 796).
2. **Output integrity — the output must actually DECODE (§8.1, the fix).** We re-probe the output's pixel
   dimensions with the **same** probe that read the input. If the input's dimensions were readable but the
   **output's are not**, the encoder produced a **corrupt/truncated** file → **refuse, keep the original**.
   * The fail-closed integrity gate runs in code — `compressFile()` §5.1 block (`if (inDims && !outDims)`, line 627).
3. **Resolution unchanged.** Output width×height must equal input; a size change **aborts** (we never
   downscale).
   * The resolution guard runs in code — `compressFile()` (line 631), dims via `imageDims()` (216) / `videoInfo()` (233).
4. **Real size gain.** The output must be **smaller** than the input, or we keep the original and report
   **"no gain."** The only exception: a **compatibility conversion** (HEIC→JPEG, forced H.264) carries a
   `formatConvert` flag and is exempt from this one check (never from 2 or 3).
   * The size-gain guard + `formatConvert` exemption run in code — `compressFile()` (line 639).
5. **Recoverable replace.** Only now do we dispose of the original (to **trash** by default — recoverable)
   and move the temp into place (with the new extension if the format changed).
   * The disposition + rename run in code — `compressFile()` (line 649); `trashOriginal()` (line 445).

### 8.1 The bug we fixed — big file replaced by a ~134-byte broken JPEG

**Symptom (reported 2026-07):** JPEG files that were several KB were found reduced to **~134 bytes of
broken garbage** after a compress run; they had to be restored from a bigger prior version.

**Root cause — a missing output-integrity gate.** Before the fix, the only output validation between the
transcode and the irreversible replace was **`safeSize()` = "bytes > 0."** A **134-byte truncated/corrupt
JPEG passes that check** (134 > 0). The one deeper guard — the **resolution check** — was written as
`if (preserveResolution && inDims && outDims && dims differ)`, so when the output was **too broken to
probe** (`magick identify` exits non-zero → `imageDims()` returns `null`), that guard was **silently
skipped**. The corrupt 134-byte file is **smaller** than the original, so the **"no gain"** check *passed*
too — and the engine went on to **replace the original with garbage**. Verified reproduction: a 134-byte
truncated JPEG returns non-zero from `magick identify` (so `imageDims → null`) yet passes `safeSize`.

**The fix — fail closed on unreadable output.** We added **§8.1 above (step 2)**: if the input's dimensions
were readable but the **output's are not**, the output is corrupt → **block and keep the original**. This is
**independent of `preserveResolution`** (a user turning that off must not disable corruption detection). A
134-byte file can no longer decode to `640×480`, so it is now caught and the original is preserved.

* The fix is in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile()` §5.1 output-integrity gate (the `if (inDims && !outDims)` block, line 627, added 2026-07-14).

**Two facts that keep this honest:**

* The default originals disposition is **trash** (`replace_original_to_trash: true`,
  `~/…/shared/src/schemas.ts` line ~142) — which is *why the restore was possible*. The bulk "Compress
  inside" flow can opt into **hard delete** ([compress_inside.mdx §4](./compress_inside.mdx)), which would
  have made the same corruption **unrecoverable** — so the integrity gate matters most there.
* The specific example file (`austin_food_truck_tacos.jpg` in the `jfk/code` repo) currently shows **129-byte
  blobs in git** — but those are **Git LFS pointer files** (that repo's `.gitattributes` LFS-tracks
  `*.jpg/*.png/…`, per its "Renormalize test feed images into Git LFS" commit), **not** compressor output.
  That is a separate git-LFS smudge artifact. The **latent compressor defect above is real and independent**,
  fits the reported symptom exactly, and is now closed regardless of that repo's LFS state.

### 8.2 Residual / follow-up

If ImageMagick is **absent** but `oxipng` is present (a PNG-only lossless path), `imageDims()` can't read
input dims, so the integrity gate can't compare — the pre-existing tool-gap behavior. In practice `magick`
is installed (it's needed for almost every image plan and for the alpha check), so the gate is active on the
real path. A future hardening could add a format-agnostic decode probe for the oxipng-without-magick case.

## 12. Acceptance criteria

1. **Explicit only.** Compress runs on a user click — one file (View-one-file / `⋮`), several checked rows,
   the whole current filter, or a whole directory ("Compress inside") — **never automatically**.
2. **PNG → JPEG works and is alpha-safe.** A no-transparency PNG converts to JPEG; a PNG with **used** alpha
   is **never** silently converted (keep PNG / lossless WebP), and undeterminable alpha fails safe.
3. **Video re-encodes to a smaller codec** (default H.264, social-media-safe) at medium CRF, **same
   resolution**, audio copied; a used-alpha source blocks a no-alpha codec.
4. **Medium quality by default** — CRF 21 (H.264/AV1) / 24 (HEVC); image quality 85 — with `low`/`high`/
   `lossless` available; the middle is the config default and never an extreme.
5. **Extension-change setting.** `convert_types` defaults **ON** (may change `.png`→`.jpg`); switching it
   **OFF** makes every compress **format-preserving**; `skip_exts` is a per-extension opt-out.
6. **Never leaves a broken file (the §8 guarantee).** Before the irreversible replace, the output must (a)
   exit 0 with bytes > 0, (b) **decode** — unreadable output dimensions **block** the replace and keep the
   original (the 2026-07 fix), (c) keep the exact resolution, and (d) be smaller (except compatibility
   conversions). The original goes to **recoverable trash** by default.
7. **Records + skip.** Before/after hash+size+fingerprint, a compression record, and an in-file

   transcode.
8. **Same engine everywhere.** Every trigger surface (§3) ends in the one `compressFile()` path, so the
   alpha check, the integrity/resolution/gain guards, and the recoverable replace apply identically no matter
   which link the user clicked.
