# LargeFileBridge — Company Auto-Detect Tour (clone an org repo → a company appears in the left bar)

This is a **scenario walkthrough**. It answers *"what happens, step by step, when I add a repo that clearly
belongs to a company — how does Large File Bridge figure that out from the git remote, show it, let me
rename it, and let me correct it when the guess is wrong."* In doing so it exercises **every** branch of
[repo_company_mapping.mdx](./repo_company_mapping.mdx): the auto-detect from the remote owner, the
Personal default, the ambiguous-owner fallback, the rename, manual reassignment, and the company
sync-repo mirror of tracking state.

The through-line: **a repo defaults to Personal; a forge ORG in its remote auto-creates a company you can
rename and re-file — you are always one click from correcting the guess.**

One computer, one user, several repos:

* **`studio-tower`** — the user's Mac tower; the user is signed in as `bryan@act3ai.com`, whose own GitHub
  handle is `BryanStarbuck` (this handle is what tells "my own account" apart from "an org").

This spec builds on the siblings and does not repeat them:

* [repo_company_mapping.mdx](./repo_company_mapping.mdx) — **the mapping rules this tour proves**: parse the
  remote owner, org → company, personal account → Personal, ambiguous → Personal + offer, manual override.
* [storage_company.mdx](./storage_company.mdx) — the **company storage** a detected company materializes
  (its `storage.yaml` `company_name`, mapped dirs, the Storages child labeled by company name).
* [storage_personal.mdx](./storage_personal.mdx) — the **default** home for any repo with no company.
* [storages.mdx §2](./storages.mdx) / [left_bar.mdx §3](./left_bar.mdx) — the **Storages** left-bar tab
  whose **children** are Personal · one per Company (labeled by `company_name`, derived at runtime) · Repos
  · Communities. A detected company is inserted here.
* [repo_settings.mdx](./repo_settings.mdx) — the per-repo settings page where **manual reassignment** lives.
* [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) — the data-category policy whose
  always-in-Local-Storage tracking a company **sync repo** **additionally** mirrors to
  `<syncRepo>/repos/<repoKey>/`.
* [devices.mdx §4](./devices.mdx) — the **graft** that resolves a company's mapped directories to each
  computer's local path.

## 0. The setup — how a repo gets a company, and where that lives

Every repo maps to **exactly one** owner in the Storages tab: **a company**, or **Personal** (the default).
The mapping is computed once, at add time, from the repo's **git remote**, then stored so the user can
override it:

* **Parse the remote.** From `https://github.com/AcmeCorp/marketing-videos.git` (or the `git@` form) LFB
  extracts the **host** (`github.com`) and the **owner** segment (`AcmeCorp`) — the org-or-account slug that
  precedes the repo name.
* **Classify the owner.** Is that owner **the user's own personal account**, or an **organization**? LFB
  answers **locally, from configuration** (the user's known personal handles per host) — it does **not**
  phone a forge API (charter: LFB does not reach the network except the one AI-description path). Owner ∈
  known-personal-handles → **Personal**; owner is a plausible org (not a known personal handle) →
  **company**; can't decide → **ambiguous → Personal, with an offer** (§4).
* **Store two things.** A **company registry** in the state root
  (`~/T/_large_files_bridge/companies.yaml`) listing each known company (id, display name, source org, host,
  optional sync repo), and a **per-repo company tag** so the repo remembers its assignment across re-scans.

Companies default to **Personal**; the whole feature exists so a repo that obviously belongs to a company
lands in the right bucket without manual filing, while staying **correctable**.

Parsing the remote owner/host, classifying org-vs-personal from local config, and reading/writing the
company registry are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-mapping.service.ts` — `parseRemoteOwner(remote)` / `classifyOwner(owner, host)` / `readCompanies()` / `upsertCompany()` / `assignRepoCompany(repoKey, companyId)`.

## 2. STEP 1 — add an org repo → a company auto-appears in the left bar

**What the user does.** On `studio-tower` they `git clone https://github.com/AcmeCorp/marketing-videos.git`
into `~/BGit/act3/marketing-videos` and **+ Add repo** ([repos.mdx §6](./repos.mdx)). The repo holds a
`campaigns/` folder of big promo videos.

**What LFB does automatically — parse, classify, auto-create.** LFB reads the repo's `origin` remote,
extracts host `github.com` and owner `AcmeCorp`, and classifies the owner: `AcmeCorp` is **not** in the
user's known-personal-handles (`BryanStarbuck`), and it matches the org shape, so it is an **organization**.
LFB **auto-creates a company** from it:

```yaml
# ~/T/_large_files_bridge/companies.yaml
companies:
  - id: "co_acmecorp"
    display_name: "AcmeCorp"        # ← seeded from the org slug (cryptic on purpose until the user renames it)
    source:
      host: github.com
      org: AcmeCorp                 # ← the immutable slug parsed from the remote (rename never touches this)
    created: 2026-03-01T10:00:00Z
    auto_created: true              # ← LFB made this from a remote, not the user
    sync_repo: null                 # ← none configured yet (Step 6)
```

The repo is **tagged** to that company (so a re-scan never re-asks), and — the visible payoff — a **new
child labeled "AcmeCorp" appears in the left-bar Storages disclosure**, between Personal and the Repos
child ([storages.mdx §2](./storages.mdx) / [left_bar.mdx §3](./left_bar.mdx)):

```
Storages
├── Personal
├── AcmeCorp                 ← NEW — auto-created from the org in the remote
├── Repos  →                ← (links to the Repos tab)
└── (communities…)
```

Opening the **AcmeCorp** child shows a company view listing its repos — currently just `marketing-videos`.

> **No directory is relocated.** Auto-detect creates a **company record** and a **grouping**, not a moved
> file. A full [company storage](./storage_company.mdx) (a dedicated SDL directory with mapped dirs) is
> **materialized on demand** — when the user gives the company a sync repo (Step 6) or maps
> directories to it. Until then the company is a lightweight left-bar grouping of repos.

Deriving a company from a repo's remote on add (auto-create when the owner classifies as an org, tag the
repo, insert the Storages child) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.service.ts` — `onRepoAdded(repo)` (calls `company-mapping.service.ts` `classifyOwner()` then `upsertCompany()` + `assignRepoCompany()`); the runtime Storages children are assembled in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/layout/Sidebar.tsx` from the discovered companies.

**What we double-check (Step 1):**
* The remote owner `AcmeCorp` is parsed and classified **org** from **local config only** (no forge API
  call).
* `companies.yaml` gains a `co_acmecorp` record with `auto_created: true`, `source.org: AcmeCorp`, and a
  seeded `display_name: "AcmeCorp"`.
* A **new Storages child "AcmeCorp"** renders in the left bar and lists `marketing-videos` under it.

## 3. STEP 2 — rename the cryptic org to a friendly company name

**What the user does.** "AcmeCorp" is the raw org slug; the user wants it to read nicely. They open the
**AcmeCorp** company view, click its **settings gear**, and rename it to **"Acme Marketing"** ([storage_company.mdx §2](./storage_company.mdx),
`company_name` is the label shown in the Storages tab).

**What LFB does automatically — rename the label, keep the identity.** LFB updates **`display_name`** only.
The **`source.org` slug is immutable** — it is the identity key the auto-detect matched on, so a future
re-add of another `AcmeCorp` repo still recognizes the company and files it under the renamed entry:

```yaml
# ~/T/_large_files_bridge/companies.yaml  (after rename)
companies:
  - id: "co_acmecorp"
    display_name: "Acme Marketing"   # ← renamed by the user
    source: { host: github.com, org: AcmeCorp }   # ← UNCHANGED — the match key survives the rename
    auto_created: true
```

The Storages child **re-labels to "Acme Marketing"** immediately; `marketing-videos` stays grouped under
it (the grouping is by company **id**, not by label, so a rename never regroups anything). If a company
storage directory has been materialized, its root `storage.yaml` `company.company_name` is updated to
match ([storage_company.mdx §2](./storage_company.mdx)).

Renaming a company (updating `display_name`, never `source.org`) and re-labeling the Storages child are
handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-mapping.service.ts` — `renameCompany(companyId, displayName)`.

**What we double-check (Step 2):**
* Only `display_name` changes; `source.org` is untouched.
* The Storages child re-labels to "Acme Marketing" and `marketing-videos` remains grouped under the same
  company id.
* Re-adding a second `github.com/AcmeCorp/...` repo files under the **renamed** company (matched by org
  slug), not a fresh "AcmeCorp".

## 4. STEP 3 & 4 — a personal-account repo stays Personal; an ambiguous owner defaults to Personal with an offer

### 4.1 Step 3 — the owner is the user's own account → Personal (the default)

**What the user does.** They add `git@github.com:BryanStarbuck/dotfiles.git`.

**What LFB does automatically.** The owner `BryanStarbuck` **is** a known personal handle for the signed-in
user, so it classifies as **Personal**. No company is created, no new Storages child appears; the repo shows
under **Personal** ([storage_personal.mdx](./storage_personal.mdx)). This is the default path for the common
case — most of a solo user's repos are their own.

```
Storages
├── Personal          ← dotfiles lands here (owner = the user's own account)
├── Acme Marketing
├── Repos  →
└── (communities…)
```

The user's known personal handles are configuration the user can edit (add a second GitHub username, a
GitLab handle, a work account) so "my own account" is recognized on every host. Where an org is *also* one
the user personally controls, they can add it to the personal-handles list to force it Personal.

### 4.2 Step 4 — an ambiguous owner → Personal, plus a one-click "This is a company"

**What the user does.** They add `https://github.com/renderlab/promo-cuts.git`. `renderlab` is neither a
known personal handle nor confidently an org — it is **ambiguous** (a lowercase handle that could be a
solo dev's account or a small studio's org).

**What LFB does automatically — default safe, offer the promotion.** LFB **does not guess wrong loudly**: an
ambiguous owner **defaults to Personal** (the conservative choice — a repo under Personal is never
mis-shared into a company), and surfaces a **non-intrusive offer** on the repo row and the Personal view:

> **"`renderlab/promo-cuts` looks like it could belong to a company. File it under a company?"** —
> **[Create company "renderlab"]** · **[Assign to existing…]** · **[Keep in Personal]**

Clicking **Create company "renderlab"** runs the same auto-create as Step 1 (a `companies.yaml` entry +
a Storages child + the repo re-tagged), which the user can then rename. **Keep in Personal** dismisses the
offer and **remembers the dismissal** (per-repo) so it never nags again for that repo.

```yaml
# the per-repo tag while ambiguous-and-defaulted:
repo_company:
  repo_key: promo-cuts-9f8e7d
  company_id: null                 # ← null = Personal (the default)
  owner_ambiguous: true            # ← drives the offer
  offer_dismissed: false           # ← flips true on "Keep in Personal"
```

Classifying an owner as ambiguous, defaulting to Personal, and surfacing/dismissing the promote offer are
handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-mapping.service.ts` — `classifyOwner()` returns `{ kind: "ambiguous" }`; the offer state (`owner_ambiguous`, `offer_dismissed`) rides the per-repo tag written by `assignRepoCompany()`.

**What we double-check (Steps 3 & 4):**
* A repo whose owner is a **known personal handle** classifies **Personal** with **no** company created.
* An **ambiguous** owner **defaults to Personal** and shows a **one-click promote offer**; "Keep in
  Personal" dismisses it stickily.
* No ambiguous repo is ever silently filed into a company.

## 5. STEP 5 — manual reassignment (move between Personal and any company, or a new one)

**What the user does.** They realize `dotfiles` (Step 3) actually holds a client's brand assets and belongs
to Acme Marketing; and separately they want to file a fourth repo under a **brand-new** company they'll type
by hand. Both are done from the repo's **[Settings](./repo_settings.mdx)** page (or the repo-row kebab's
"Move to company…"), which carries a **Company** control:

* A dropdown: **Personal** (default) · **Acme Marketing** · *(each known company)* · **+ New company…**.
* Choosing **Acme Marketing** re-tags `dotfiles` to `co_acmecorp`; it leaves Personal and appears under Acme
  Marketing on the next render. The move is a **tag change only** — no bytes move, no directory relocates.
* Choosing **+ New company…** opens an inline name field; typing "Skunkworks" and confirming creates a
  **hand-authored** company (`auto_created: false`, `source: null` since it came from no remote) and files
  the repo under it.

```yaml
# companies.yaml after the hand-created company:
companies:
  - id: "co_acmecorp"    { display_name: "Acme Marketing", source: {host: github.com, org: AcmeCorp} }
  - id: "co_skunkworks"  { display_name: "Skunkworks", source: null, auto_created: false }
```

Manual reassignment is the universal escape hatch: **any** repo can be moved to Personal, any existing
company, or a new one, regardless of what auto-detect guessed — the guess is only a starting point.

Reassigning a repo's company (including creating a hand-named company with `source: null`) is handled in
code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `PATCH /api/repos/:repoId/company` (via `company-mapping.service.ts` `assignRepoCompany()` / `upsertCompany()`); the Company control renders in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/RepoSettingsPage.tsx`.

**What we double-check (Step 5):**
* Reassigning `dotfiles` to Acme Marketing re-tags it (tag change only, no relocation) and it regroups in
  the Storages tab.
* **+ New company…** creates a hand-authored company (`auto_created: false`, `source: null`) and files the
  repo under it.
* A manual assignment **wins** over auto-detect and survives re-scans (the tag is the source of truth once
  the user has touched it).

## 6. STEP 6 — configure the company's sync repo → tracking is additionally mirrored there

**What the user does.** Acme Marketing has several member computers, so the user wants the company's LFB
tracking to travel. In the Acme Marketing company settings they set a **sync repo** — a dedicated
tracking-data git repo they made for the company, e.g. `~/BGit/Bryan_git/acme_large_files_bridge/`
([storage_company.mdx §1/§7](./storage_company.mdx), [devices.mdx §11.2](./devices.mdx)) — and turn on the
per-repo sync toggle for its repos ([repo_settings.mdx §2](./repo_settings.mdx), default OFF).

**What LFB does automatically — materialize the company storage and add a tracking mirror.** Setting the
sync repo does two things:

1. **Materializes the company storage.** A [company storage](./storage_company.mdx) is created rooted at the
   sync repo, with a root `storage.yaml` carrying `type: company` + `company.company_name: "Acme
   Marketing"`. (It sits at the root, not under a `.lfbridge/` — a company storage is an **SDL**, and an SDL
   has no `.lfbridge/` at all: its root *is* the `.lfbridge` area,
   [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx).) The company is now a real storage,
   not just a grouping.

   ```yaml
   # companies.yaml — the company now points at its sync repo:
   - id: "co_acmecorp"
     display_name: "Acme Marketing"
     source: { host: github.com, org: AcmeCorp }
     sync_repo: ~/BGit/Bryan_git/acme_large_files_bridge   # ← configured in Step 6
   ```

2. **Adds a tracking mirror** for **every repo mapped to Acme Marketing** with the sync toggle on. This is the
   direct hand-off to the [data-category placement policy](./artifact_placement_policy.mdx): a company repo's
   Category-B tracking **always** lives in Local Storage at `~/T/_large_files_bridge/repos/<repoKey>/`, and is
   now **additionally** mirrored to **`<syncRepo>/repos/<repoKey>/`** — **in addition to** Local Storage, not
   instead of it. So `marketing-videos`' `repo_storage.yaml` and sidecars are written to Local Storage **and**
   the sync repo, and travel to the other member computers — while **`marketing-videos`' own working tree
   stays untouched** (the tracking-never-enters-the-repo invariant holds regardless):

```
~/T/_large_files_bridge/repos/marketing-videos-77aa/  ← ALWAYS written here (authoritative Local-Storage copy)
├── repo_storage.yaml
└── files/campaigns/…

~/BGit/Bryan_git/acme_large_files_bridge/          ← the company sync repo (committed, travels to members)
├── storage.yaml                                   ← type: company, company_name: "Acme Marketing" (SDL root — no .lfbridge/)
└── repos/
    └── marketing-videos-77aa/                      ← <repoKey> — the ADDITIVE mirror of the tracking above
        ├── repo_storage.yaml
        └── files/campaigns/…

~/BGit/act3/marketing-videos/                        ← the media repo itself — STILL untouched by scan churn
```

When someone later transcribes or describes a clip in `marketing-videos`, that **content artifact**
(Category A) lands in the **media repo's** committed `.lfbridge/` (that is where a transcript belongs) and
travels via the media repo's own git — while the tracking state stays in Local Storage + the sync repo. The
sync repo governs only the **tracking** mirror, never content placement. (See
[use_case_churn_free_repo.mdx §9.2](./use_case_churn_free_repo.mdx).)

Materializing a company storage on sync-repo configuration and mirroring its repos' Local-Storage tracking
are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-mapping.service.ts` — `setCompanySyncRepo(companyId, path)` (writes the company `storage.yaml` via `storage.service.ts` and updates `companies.yaml`); the mirror target is resolved by `tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)` (returns the company's `<syncRepo>/repos/<repoKey>/` for a company-mapped repo) and written by `tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)`.

**What we double-check (Step 6):**
* Setting the sync repo materializes a real company storage (`storage.yaml`, `type: company`,
  `company_name: "Acme Marketing"`).
* `marketing-videos`' `repo_storage.yaml` + sidecars are in **both** `~/T/_large_files_bridge/repos/
  marketing-videos-77aa/` (authoritative) **and** `<syncRepo>/repos/marketing-videos-77aa/` (mirror).
* The **media repo's** working tree is **still untouched** — the sync repo adds a mirror, it does not move
  tracking out of Local Storage or into the media repo.

### 6.1 A teammate sees the mapping — it travels, consent-gated

The tour so far is **one computer, one user**. The moment Acme Marketing has a **sync repo** (Step 6), the
repo→company assignments made on `studio-tower` become a **team fact** and travel to other members — because
the assignment now decides that `marketing-videos`' tracking mirrors into the **shared** company repo, not
just where it groups in one user's left bar.

* **Member #1 (`bryan@act3ai.com`) asserts.** When Step 5/6 assigns `marketing-videos` (and `dotfiles`) to
  Acme Marketing, LFB commits an **ownership assertion** into the company sync repo's root
  `owner_map.yaml`, **keyed by the git remote** (`github.com/AcmeCorp/marketing-videos`) — the only identity
  portable across machines. It rides the same git backbone as the tracking state.
* **Member #2 pulls and gets a review page.** On another member's Mac, pulling the Acme Marketing sync repo
  brings that `owner_map.yaml`. LFB matches the asserted remotes against that member's local repos and, for
  any match **not** already company-owned there, routes them to **`/company-mappings/review`** before
  applying anything — a per-repo checkbox + **Move to company** (default) / **Keep personal** choice + a
  **Continue & apply ›** button. Nothing is re-mapped on member #2's machine silently, because moving the
  repo redirects **their** tracking into the shared company repo.

This two-member travel + consent flow is owned by
[repo_owner_propagation.mdx](./repo_owner_propagation.mdx); this tour only shows where it begins (the Step 6
sync repo turns the single-computer assignment into a travelling, consent-gated assertion).

## 7. The cross-cutting invariants this tour proves

1. **A repo defaults to Personal.** No company, no assignment, ambiguous owner — all resolve to Personal; a
   company assignment is always something LFB detected from a remote org or the user chose.
2. **Detection is local, never a forge API call.** The org-vs-personal decision is made from the parsed
   remote owner against the user's known-personal-handles config — no network (charter).
3. **A forge org auto-creates a renamable company.** The company seeds its label from the org slug, appears
   as a Storages child, and the user renames the label while the immutable `source.org` stays the match key.
4. **The guess is always correctable.** Ambiguous → Personal + a one-click promote offer; any repo →
   manual reassignment to Personal, an existing company, or a new one; a manual choice wins and sticks.
5. **Assignment is a tag, not a move.** Changing a repo's company re-groups it in the left bar; **no bytes
   relocate** and **no working tree is mutated**.
6. **The sync repo is an additive tracking mirror.** A company's sync repo **additionally** mirrors its
   repos' always-in-Local-Storage tracking to `<syncRepo>/repos/<repoKey>/` (in addition to Local Storage,
   not instead of it) while the media repo's working tree stays untouched and content artifacts still travel
   via the media repo's own git.
7. **A company assignment travels to teammates, consent-gated.** Once the company has a sync repo, a
   repo→company assignment is committed as an assertion in `<syncRepo>/owner_map.yaml` (keyed by the git
   remote) and travels to other members; it is applied on their computers only through a review-and-consent
   page, never silently ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx), §6.1).

## 8. Acceptance criteria

1. Adding `github.com/AcmeCorp/marketing-videos` parses owner `AcmeCorp`, classifies it an **org** from
   **local config only**, **auto-creates** a company (`companies.yaml` entry, `auto_created: true`,
   `source.org: AcmeCorp`, seeded `display_name`), tags the repo to it, and inserts a **"AcmeCorp" child**
   in the left-bar **Storages** disclosure listing that repo.
2. Renaming the company changes **`display_name`** only (never `source.org`); the Storages child re-labels
   to "Acme Marketing" and the repo stays grouped by company **id**; a later same-org repo files under the
   renamed company.
3. A repo whose owner is a **known personal handle** classifies **Personal** with **no** company created and
   appears under Personal.
4. An **ambiguous** owner **defaults to Personal** and shows a **one-click "This is a company"** offer;
   "Keep in Personal" dismisses it stickily; promoting runs the same auto-create.
5. **Manual reassignment** (repo Settings / kebab) moves a repo to Personal, an existing company, or a
   **+ New company…** typed in place (creating an `auto_created: false`, `source: null` company); a manual
   choice **wins over auto-detect** and survives re-scans; assignment relocates **no bytes**.
6. Configuring a company's **sync repo** materializes a real [company storage](./storage_company.mdx) and
   **additionally** mirrors its repos' always-in-Local-Storage tracking to **`<syncRepo>/repos/<repoKey>/`**
   ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)) — in addition to Local Storage, not
   instead of it — while the media repo's working tree stays untouched; content artifacts still travel via
   the media repo's own git.
7. Once the company has a **sync repo**, a repo→company assignment is committed as an assertion into
   `<syncRepo>/owner_map.yaml` (keyed by the git remote) and **travels** to other members, who apply it only
   through the review-and-consent page — never silently ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx),
   §6.1).
8. Every write matches [repo_company_mapping.mdx](./repo_company_mapping.mdx),
   [storage_company.mdx](./storage_company.mdx), [storage_personal.mdx](./storage_personal.mdx),
