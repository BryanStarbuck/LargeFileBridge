# LargeFileBridge — Repos

The **Repos** page is the landing view of LargeFileBridge and the screen users live in. It answers
one question at a glance: *for every repo on this computer, are my large files pinned across my
other machines?* It is a header over **one TanStack Table** — **one row per repo, for every git repo on
this computer** (LFB discovers and knows them all automatically — there is no "managed" opt-in;
[pin_process.mdx §1](./pin_process.mdx#1-lfb-knows-every-repo--there-is-no-lfb-managed-gate-locked)) —
with the standard search / filter / sort control row above it.

The left bar's **Repos** item (order 1, icon `FolderGit2`, route `/` — source of truth
`pm/left_bar.yaml`) lands here. Clicking any repo row opens that repo's per-file screen
([One repo](./one_repo.mdx)).

The Repos page is loaded and rendered as one TanStack table in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/ReposPage.tsx` — `ReposPage` component

This spec builds on the siblings and does not repeat them:

* [README](../README.md) — the product charter: Git tracks the small files; LFBridge moves the big,
  git-ignored files peer-to-peer over IPFS.
* [Left bar](./left_bar.mdx) — the left navigation (data source of truth `pm/left_bar.yaml`); this
  document does not redefine it and only shows where it sits.
* [One repo](./one_repo.mdx) — the per-file screen a row click opens.

## 1. Vocabulary — Repo → Large file → Decision (LOCKED)

Three terms, used consistently across every spec and every screen:

| Term | Definition |
| --- | --- |
| **Repo** | **Any** git repository on this computer. LFBridge **discovers and knows every repo automatically** — a repo is never "unmanaged" ([pin_process.mdx §1](./pin_process.mdx#1-lfb-knows-every-repo--there-is-no-lfb-managed-gate-locked)); IPFS pinning of its bytes is the only opt-in. It has a local **path**, a **name** (the folder name), a git-ignore that excludes the big files, and a **manifest** (checked into git) listing the large files by path + IPFS CID. |
| **Large file** | A big, git-ignored file inside a repo (video, image, model weights, dataset, render output). LFBridge — not git — is responsible for moving its bytes between your computers. |
| **Decision** | Per large file, whether LFBridge should move it: **Add to IPFS (pin)** (stored decision value `sync`), **Ignore** (deliberately not pinned), or **Undecided** (LFBridge found it but the user has not yet chosen). Defined in full on the [One repo](./one_repo.mdx) screen; the Repos table only shows the **counts** per decision. |

```ts
// The per-repo summary the backend returns for each Repos-table row.
export interface RepoRow {
  repoId: string;        // stable id (hash of the absolute path)
  bookmarked: boolean;   // user favorite — persisted per-repo in config.yaml (§8); drives the leading ribbon toggle
  name: string;          // folder name
  path: string;          // absolute local path
  counts: {
    pinned: number;      // large files pinned (CID present + pinned locally) — INCLUDES files
                         //   that were already IPFS-pinned and got imported (ipfs.mdx §4)
    pending: number;     // decided "Add to IPFS (pin)" but not yet transferred (fetch/push queued)
    undecided: number;   // found by LFBridge, no pin/Ignore decision yet
    ignored: number;     // decided "Ignore" — deliberately not pinned
  };
  peerCount: number;     // how many of your computers currently pin this repo
  lastPinAt: string | null;  // ISO date-time of the last completed pin pass, or null
  status: RepoStatus;    // rolled-up pin status (see §4.2)
}

export type RepoStatus =
  | "up_to_date"   // every "Add to IPFS (pin)" file present locally and on >=1 peer
  | "pinning"      // a transfer is in progress
  | "behind"       // pending > 0 (files owed in or out)
  | "needs_review" // undecided > 0 (user has decisions to make)
  | "error"        // last pin failed / IPFS node unreachable
  | "never";       // repo added, never pinned
```

The per-repo summary row (`RepoRow`) each table row shows is assembled by the backend in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `computeRepoRow`

### 3.1 Control row (above the table) — LOCKED

A single control row sits above the table, matching the standard toolbar:

* **Search box** (left) — a free-text input; substring match, case-insensitive, across **Repo name
  + path**. Filters the current data set live.
* **Filter icon** (right, icon-only `⛛` funnel — **no text label, no button box**) — opens a
  dropdown listing the filterable columns; each filter row is an **operator select + value input**;
  multiple filters are **ANDed**; Clear / Apply, `Enter` applies. The icon is **colored** when any
  filter is active (with an optional count). Applying a filter resets to page 1.
* **Sort icon** (right, icon-only sort glyph — **no text label, no button box**) — opens a dropdown
  listing the sortable columns, each with asc/desc; the icon is **colored** when a sort is active.
* **Header-click sort (LOCKED).** In addition to the sort dropdown, **clicking a column header**
  sorts the table by that column. Clicking the **same** header again **toggles ascending ↔
  descending** (it does not cycle to "unsorted" — the header always leaves a sort applied; clear it
  from the dropdown's **Clear sort**). This adds **no new UI** — the header text *is* the control; the
  active sort column shows a small **caret** (`↑` asc / `↓` desc) beside its label, plus a pointer
  cursor / hover on sortable headers. Header click and the dropdown share **one** sort state, so a
  header-driven sort shows as active in the dropdown and vice-versa. **Only sortable columns respond**
  — every column the sort dropdown lists (including the leading **Bookmark**, whose header sorts
  bookmarked-first exactly like the dropdown) is clickable; a column with `sortable: false` and the
  **non-data** trailing chevron / kebab cells do nothing on header click.
`DataTable` toolbar (search left; icon-only sort/filter popovers right; rounded fills reserved for
the floating popovers, never the data surface).

### 3.2.1 Responsive column priority — drop the least-important column as the table narrows (LOCKED)

The table must **never wrap a cell onto a second line** to make everything fit. Instead, **every column
carries a priority**, and as the available width shrinks, the table **hides the lowest-priority visible
column first**, then the next, and so on — so what remains always fits on one line. As the window widens
again, hidden columns **reappear** in reverse order (highest priority first). This runs on the shared
`DataTable`, so every table inherits it.

**The trigger (plain-language).** If a cell would **wrap to two rows** because the row is too narrow,
that is the signal to **remove a column** — and if one isn't enough, remove **two or three** — until
every row is a single line again. In code we achieve this with a **per-column minimum width budget**:
the sum of the visible columns' minimum widths (plus the leading select + trailing chevron/kebab) must
fit the container; while it doesn't, the **lowest-priority** visible column is dropped and the budget
recomputed. Implemented in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` —
`useResponsiveColumns()` (measures the container with a `ResizeObserver`, drops lowest-priority columns
until the min-width budget fits), driven by the per-column `priority` + `minWidth` fields on
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/types.ts` — `LfbColumn`.

**Priority numbers — LOWER number = MORE important (kept longest); HIGHER number = dropped first.** The
master stack-rank below covers **every column that can appear** across LFB's repo-style/file tables (the
Repos list uses the subset it has; a files table that adds **Size** slots it in at the same rank). Two
special rules the user set: **Peers is the single lowest priority (dropped first)**, and **Size is
second-lowest (dropped next)**.

| Priority | Column | Pinned? | Drop behavior | Rationale |
| --- | --- | --- | --- | --- |
| **1** | **Repo / File (name)** | **Pinned — never drops** | never | The row's identity. Without it the row is meaningless. |
| **2** | **Status** | **Pinned — never drops** | never | The "am I OK?" verdict (§4.2) — the whole point of the screen. |
| **0** | **`▮` Bookmark** (leading toggle) | **Pinned — never drops** | never | A tiny fixed-width control; always fits, always useful. Rendered leftmost regardless. |
| **3** | **Undecided** | no | dropped 8th | Actionable — files that need a decision now. |
| **4** | **Last pin** | no | dropped 7th | "When did this last move?" — high-value reassurance/alarm. |
| **5** | **Pending** | no | dropped 6th | In-flight transfers. |
| **6** | **Pinned** | no | dropped 5th | Reassurance count of backed-up files. |
| **7** | **Path** | no | dropped 4th | Useful but long and already middle-truncated with a tooltip; the name carries identity. |
| **8** | **Ignored** | no | dropped 3rd | Least-actionable count. |
| **9** | **Size** | no | **dropped 2nd** | *(File tables only — not on the Repos list.)* Second-lowest per the rule. |
| **10** | **Peers** | no | **dropped 1st** | **Lowest priority — the first to go.** Redundancy detail; the Status verdict already flags "not backed up." |

**Drop order as the Repos list narrows** (it has no Size column, so the sequence skips priority 9):
`Peers` (10) → `Ignored` (8) → `Path` (7) → `Pinned` (6) → `Pending` (5) → `Last pin` (4) → `Undecided`
(3). It **stops** before ever dropping the three pinned columns (**Bookmark · Repo · Status**), which
always remain so the table is never useless. The trailing **› chevron** and **`⋮` kebab** control cell
also always remain (they are not priority columns).

**Rules.**
* A column with **no `priority`** set is **treated as pinned** (never auto-hidden) — hiding is
  **opt-in** per table by assigning a priority; the Repos list assigns one to every column per the table
  above.
* **Pinned columns** (Bookmark, Repo, Status) never drop even at the narrowest width; if even those
  don't fit, the pinned set may scroll horizontally rather than wrap — pinned identity/verdict beats
  everything.
* **Sort & filter still reach hidden columns.** A column being responsively hidden from the body does
  **not** remove it from the **Sort** and **Filter** dropdowns (§3.1) — the user can still sort/filter by
  Peers even when the Peers column is hidden at the current width. (Hiding is presentation-only.)
* **Persistence.** Responsive hiding is derived from **live width**, not a saved preference — it is not
  written to `config.yaml`. The per-user sort/filter persistence ([tables.mdx §4](./tables.mdx)) is
  unaffected.
* **`minWidth`.** Each column declares a sensible `minWidth` (px) used for the budget; sensible defaults
  by kind — text ≈ 140, timestamp ≈ 96, enum ≈ 100, int ≈ 72, toggle ≈ 40 — overridable per column.

### 3.3 Pagination — LOCKED

* **Default page size 500.** Do **not** default to a small page size. Show the **item count**
  (`N repos`) to the left of the pager. Page-size options: `500 / 1000 / All`.

### 3.3.1 Height — fill the page (LOCKED)

The Repos table has **nothing beneath it** (the table is the last thing on the page), so it takes the
**full-page-height** house behavior:

* The table's **body scroll region grows to the bottom of the viewport** — no fixed short height, no
  dead white space below it when the browser window is tall. Bryan runs a **tall** window; a fixed
  `max-h` table would leave a big empty block at the bottom, which this removes.
* **The page itself does not scroll.** The **control row** (§3.1) and the **count/pagination footer**
  (§3.3) stay **pinned**; only the **rows** scroll, inside the table (the `thead` is already
  `sticky`). The body is the single scroll container the windowing (`performance.mdx` P-01) already
  reads.
* **Mechanism.** The scroll container is `flex-1 min-h-0` (fills its flex-column parent) instead of a
  fixed `max-h`; the page's outer element is a full-height flex column (`flex flex-1 flex-col
  min-h-0`) with the page header `shrink-0`, so the table is the one element that flexes to fill. The
  app frame's content column is already `min-h-full` so this height is available on every route.
* **This is the site-wide default** ([charter Tables rule](../CLAUDE.md)). The **only exception** is a
  page with content **below** the table — see [one_repo.mdx §3.5](./one_repo.mdx) and
  [directories.mdx](./directories.mdx) — where the table keeps a **bounded** height so what sits under
  it stays visible. Content **above** a table (headers, banners, stat tiles) never triggers the
  exception.

## 4. Status

### 4.1 The counts, at a glance

Each row's four counts are the whole point of the page — they let the user decide, per repo, what
needs attention: **Undecided > 0** means there are files to triage; **Pending > 0** means bytes are
owed in or out; **Peers = 0** means nothing is backing this repo up.

> **These counts include git-ignored media of *any* size.** The scan's payload rule
> ([scan.mdx §4.1](./scan.mdx#41-what-counts-as-a-candidate--the-payload-rule-locked)) treats every
> git-ignored video/audio/image as a large file **regardless of the 100 MB threshold** — so a repo of
> 5–100 MB clips (e.g. `charlie-kirk`: 46 files, `Intel_Murder_Docus`: 27) shows its *real* Undecided
> count, not just the two or three files over 100 MB. A previous scanner bug gated everything on the
> size threshold and made these repos look nearly empty; that is fixed.

### 4.2 Status pill colors (LOCKED)

| `status` | Meaning | Pill |
| --- | --- | --- |
| `up_to_date` | Every "Add to IPFS (pin)" file present locally and on ≥1 peer. | green |
| `pinning` | A transfer is in progress right now. | blue (accent), animated |
| `behind` | `pending > 0` — files owed in or out. | amber |
| `needs_review` | `undecided > 0` — the user has decisions to make. | accent outline |
| `error` | Last pin failed, or the IPFS node is unreachable. | red |
| `never` | Repo added but never pinned. | grey |

Precedence when several could apply: `error` > `pinning` > `behind` > `needs_review` > `up_to_date`
> `never`.

## 6. Add repo (top-right)

A **+ Add repo** button, top-right of the header. It opens a small dialog to register a git repo on
this computer with LFBridge:

* **Pick a folder** — a path field (the repo's git root). LFBridge validates it is a git working
  tree.
* On confirm: LFBridge reads the repo's `.gitignore` + manifest, scans for large git-ignored files,
  and adds a row. New large files it finds start as **Undecided** (the user triages them on the
  [One repo](./one_repo.mdx) screen).
* A repo already registered is rejected with an inline message (no duplicate rows).

Registering a repo by folder path (and kicking a background rescan) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `POST /api/repos` (calls `registerRepo`)

### 6.1 Page actions — the action-links row

Directly under the page title — with the **+ Add repo** primary button (§6) staying in the header — the
page carries the page-level **action-links row** ([page_actions.mdx](./page_actions.mdx)): the visible
horizontal row of blue hyperlinks (each with a leading icon, **not** the old `Actions ▾` dropdown), the
third action surface, distinct from the per-row `⋮` repo kebab ([menus.mdx §5.1](./menus.mdx#51-repo)).
Because **Repos rows are repos, not files**, this page does **not** carry the file-producing actions
(Create Transcriptions / Create AI descriptions live on the file-list pages). Instead its links are the
"Repos list" row of the per-page catalog ([page_actions.mdx §4](./page_actions.mdx)) — the **whole-list**
operations:

* **Rescan all** — re-run the scan across **every** repo on this computer (the per-repo Rescan
  [menus.mdx §5.1](./menus.mdx#51-repo), lifted to all rows).
* **Pin all** — kick a pin pass on every repo that has pending work (the per-repo **Pin now**, lifted to all
  rows).

## 8. Bookmark (favorite) — the leading ribbon toggle

Each row's **leading cell** is a **bookmark** (favorite) toggle: a ribbon-shaped icon the user clicks
to mark repos they care about so they float to the top of a long list.

### 8.1 The icon and its two states (LOCKED)

Use the **ribbon bookmark** glyph — lucide **`Bookmark`** (the classic banner/ribbon shape), sized to
the row (≈16px), vertically centered in the leading cell.

| State | `bookmarked` | Rendering |
| --- | --- | --- |
| **On** (favorite) | `true` | **Solid yellow** — filled *and* stroked in the warning/yellow token (e.g. `#EAB308`). It reads as a fully-colored ribbon. |
| **Off** (default) | `false` | A **thin gray outline** with a **white** (empty) fill — just the 1px ribbon stroke in a muted gray, no fill. |

Implementation note: one `<Bookmark>` with `fill` toggled — `fill="currentColor"` + yellow color when
on; `fill="none"` + `stroke` muted-gray when off. On hover of an **off** star, preview a faint yellow
so it's discoverable. No layout shift between states (same box).

### 8.2 Interaction (LOCKED)

* **Click toggles** `bookmarked` for that repo. It is a **control cell** — exactly like the `⋮` kebab
  (§5.2): the click **stops propagation** and **never opens the repo** or triggers cell navigation.
* **Optimistic**: the icon flips immediately; the write (§8.3) happens in the background; on failure it
  reverts and shows a toast.
* **Accessible**: render as a `<button aria-pressed={bookmarked} aria-label="Bookmark this repo">`;
  it is keyboard-focusable and toggles on `Enter`/`Space`. Tooltip: *"Bookmark"* / *"Bookmarked"*.
* The same toggle is also offered in the repo **`⋮` menu** as *"Bookmark / Remove bookmark"*
  ([menus.mdx §5.1](./menus.mdx#51-repo)) so it is reachable without hunting for the tiny icon.

The leading ribbon toggle and its optimistic flip are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/ReposPage.tsx` — `BookmarkToggle` (mutated via `ReposPage` `toggleBookmark`)

### 8.3 Persistence — it lives in the repo's `config.yaml` (LOCKED)

The bookmark is a **local, per-computer** preference (a favorite *this machine's* user set), so it
persists in that repo's **pin-unit `config.yaml`** — the same file that holds `pinned`,
`big_file_override`, etc. — **not** in the shared manifest and **not** replicated to peers.

```yaml
# pin/r/<repo>/config.yaml  (excerpt)
schema_version: 1
repo: { name: charlie-kirk, path: /Users/…/charlie-kirk, remote: … }
pinned: false
bookmarked: true          # ← the favorite flag (default false). Written by the bookmark toggle.
big_file_override: { … }
```

* **Default `false`.** A newly discovered/added repo is not bookmarked.
* **The scan never touches it.** Discovery writes `status.yaml` and reconciles `config.yaml` **without
  clobbering** `bookmarked` (or any other user field) — re-registering an existing repo preserves it.
* **API:** `POST /api/repos/:repoId/bookmark` with `{ bookmarked: boolean }` writes the flag to
  `config.yaml` and returns the updated `RepoRow`. Idempotent; unknown `repoId` → 404.

Persisting the bookmark flag to the repo's `config.yaml` is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `POST /api/repos/:repoId/bookmark`

### 8.4 Sort & filter — the shared model ([tables.mdx](./tables.mdx))

The bookmark's sort/filter behavior is the **cross-cutting** one defined once in
[tables.mdx](./tables.mdx) (bookmarks, promoted filter, multi-level sort, per-user persistence) — Repos
is its **reference implementation**. In short:

* **Sort is multi-level** (primary / secondary / tertiary — [tables.mdx §3](./tables.mdx#3-multi-level-sort--primary--secondary--tertiary-locked)).
  The **default** here is **Bookmarked descending first, then repo name** — favorites float to the top
  of a 100-repo list. The sort dropdown's **1st/2nd/3rd** priority slots let the user re-rank; the
  choice is **saved per user for this page** and restored next visit.
* **Filter** — the filter dropdown **promotes a "Bookmarked only" checkbox** to the top
  ([tables.mdx §2](./tables.mdx#2-the-promoted-bookmarked-only-filter-locked)); the per-column
  *Bookmarked: yes / no* filter still exists in the standard list (§3.1).
* Bookmark state is **independent** of `status` — a bookmarked repo still shows its true status pill.
* **Persistence** — this page's sort + filter are saved in `users/<email>/config.yaml` `tables.views`
  ([tables.mdx §4](./tables.mdx#4-persistence--sort--filter-saved-per-user-per-page-locked)).
