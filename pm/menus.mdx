# LargeFileBridge тАФ Cell Navigation, Row "тЛп" Menus & Entity "More" Menus

Every screen in LargeFileBridge (LFB) is built from the same two shapes: **tables** (the TanStack
`DataTable` used on [Repos](./repos.mdx), [One repo](./one_repo.mdx), [Peers](./left_bar.mdx),
[IPFS](./ipfs.mdx), тАж) and **single-entity pages** (view *one* repo, *one* file, *one* directory).
This file defines the three interactions those shapes share, **once**, so no other spec restates them:

1. **Cell navigation** тАФ a click on a *cell* goes somewhere specific (┬з2).
2. **The row `тЛп` kebab** тАФ every table row's trailing per-row action menu (┬з3). **Right-clicking the
   row (or a file/dir entry anywhere) opens the same menu** (┬з3.1).
3. **The entity `тЛп` "more" menu** тАФ the same actions, top-right, on a single-entity page (┬з4). The
   two single-entity pages are [files.mdx](./files.mdx) (**View one file**) and
   [directories.mdx](./directories.mdx) (**View one directory**).

The rule that ties them together: **a table cell is a deep link, and the row's `тЛп` (or a right-click,
or the page's top-right `тЛп`) is that same entity's action menu.** Clicking *data* navigates; clicking
`тЛп` (or right-clicking) *acts*.

Siblings that reference this file (and must not redefine these conventions):

* [Repos](./repos.mdx) тАФ the repos table (repo rows).
* [One repo](./one_repo.mdx) тАФ a single repo page (its top-right `тЛп`) whose files table has file rows.
* [File System](./directories.mdx) тАФ the column browser; its selected file/dir gets the entity menu.
* [IPFS](./ipfs.mdx) тАФ the pinned-content table (file/CID rows).
* [Repo settings](./repo_settings.mdx) / [Settings](./settings.mdx) тАФ where the "SettingsтАж" items land.
* [Web app](./webapp.mdx#11-what-shows-up-in-the-dock--the-lfbridge-operation-taxonomy) тАФ every action
  that does background work reports through the progress dock's **operation taxonomy**.

## 1. Vocabulary тАФ Cell link / Row menu / Entity menu / Action (LOCKED)

| Term | Definition |
| --- | --- |
| **Cell link** | A table cell that, when clicked, **navigates** to the most specific screen for that cell's value (┬з2). Not every cell links; the ones that do are listed per table. |
| **Row `тЛп` kebab** | The trailing, right-aligned control on **every** table row. It opens the **entity action menu** (┬з3) for the entity that row represents. It is a control column тАФ it **never** triggers cell navigation or row navigation. |
| **Entity `тЛп` "more" menu** | On a **single-entity page**, the top-right `тЛп` button that opens the **same** action menu as that entity's row kebab (┬з4). "The page is one entity, so the page has that entity's menu." |
| **Entity** | The thing a row or page is *about*: a **repo**, a **directory**, a **file**, or a **peer**. Each entity has exactly **one** LOCKED action catalog (┬з5), used by both its row kebab and its page's more-menu. |
| **Action** | One menu item. Either **navigates** (opens a screen), **mutates** (does work тАФ always through the progress dock + charter guardrails, ┬з6), or **copies** (path / CID to clipboard). |

## 2. Cell navigation тАФ clicking a cell is a deep link

Whole-row navigation still exists where a spec defines it (a Repos row opens that repo). **On top of
that**, individual cells navigate to the *most specific* view for their value, so the user can jump
straight to the slice they care about instead of opening the parent and re-filtering by hand.

### 2.1 The rule (LOCKED)

* **A count/metric cell** тЖТ the child list, **pre-filtered** to exactly that slice. (Clicking a repo's
  `Undecided 5` opens that repo's files table already filtered to the 5 undecided files.)
* **A path/name cell** тЖТ the place that path lives: a **repo name** тЖТ that repo's page; a **file path**
  тЖТ the [File System](./directories.mdx) browser revealing that file; a **directory path** тЖТ the browser
  at that directory.
* **A relationship cell** (Peers, pinned-by) тЖТ the related list **filtered to this entity**
  (`Peers 3` тЖТ [Peers](./left_bar.mdx) filtered to this repo/file).
* **A status/enum cell** тЖТ the same target as the row (the entity page), scrolled/aware of that status.
* **An identifier cell** (CID) тЖТ does **not** navigate away; it **copies** (monospace, click-to-copy) тАФ
  and optionally opens the [IPFS](./ipfs.mdx) page filtered to that CID via a small inline affordance.
* **The `тЛп` kebab column never navigates** тАФ it only opens the menu (┬з3).

A linking cell shows a pointer cursor + hover underline/emphasis so it reads as clickable; a
non-linking cell (e.g. a raw size) does not. Middle-click / тМШ-click opens the target in context where
the platform supports it. Cell navigation **never mutates** тАФ it only routes.

### 2.2 The per-table cell-link maps referenced elsewhere

Each table spec owns its own columnтЖТtarget map and links back here for the rule:

* **Repos table** тАФ [repos.mdx ┬з5.1](./repos.mdx#5-row--cell-navigation).
* **One-repo files table** тАФ [one_repo.mdx ┬з4.7](./one_repo.mdx#4-the-files-table-tanstack-house-style).
* **IPFS pinned table** тАФ [ipfs.mdx](./ipfs.mdx).
* **Peers table** тАФ its own spec, following this rule.

## 3. The row `тЛп` kebab тАФ every table has one

Every LFB TanStack table gets a **trailing control column** whose cell is a right-aligned **`тЛп`
kebab** button. It is the standard, discoverable home for *acting on one row* тАФ distinct from the
existing header **`тЛо` bulk menu** (which acts on the *selected set*, [one_repo.mdx ┬з4.4](./one_repo.mdx)).

* **Placement.** Last column, fixed narrow width, right-aligned, no header label. It stays pinned as
  the row's other cells navigate тАФ clicking `тЛп` opens a menu, it does **not** open the row.
* **Contents.** The **entity action catalog** for whatever that row is (┬з5) тАФ a repo row shows the
  **repo** catalog, a file row the **file** catalog, and so on.
* **Popover.** A rounded floating popover (the house popover style used by the sort/filter dropdowns,
  [repos.mdx ┬з3.1](./repos.mdx)), grouped with thin dividers, destructive items last and tinted red,
  gated items hidden without the claim (┬з6.3). Opens on click, closes on select / outside-click / `Esc`.
* **Keyboard + a11y.** Focusable button (`aria-haspopup="menu"`), arrow-key navigation, `Enter`
  activates, `Esc` closes and returns focus. Each item has a clear label; icons are decorative.

### 3.1 Right-click opens the same menu (LOCKED)

Wherever a **file or directory entry** is rendered тАФ a table row, a [File System](./directories.mdx)
column-browser row, a tile тАФ **right-clicking (context menu) opens the exact same entity action
catalog** (┬з5) as that entity's `тЛп` kebab. The kebab is the discoverable, always-visible affordance;
the right-click is the power-user shortcut. They are two triggers for **one** menu component fed by
**one** catalog тАФ never a divergent list.

* The context menu opens at the pointer, is flipped to stay on-screen, and closes on
  select / outside-click / `Esc` (same popover as the kebab).
* Right-click **suppresses the browser's native context menu** only over LFB entity rows; elsewhere the
  native menu is untouched.
* Right-click never *navigates* and never *selects* тАФ it only opens the menu (the kebab discipline, ┬з3).

## 6. Guardrails the menus inherit (LOCKED)

These are charter rules the menus must honor тАФ they are **why** several actions are confirm-gated.

### 6.1 Never act on files without an explicit click
**Compress, convert (PNGтЖТcompressible), and git-ignore are OFFERS, not automatic.** They exist in the
menu, but selecting one **opens a confirm step** (a review list for the ignore/compress set; the
alpha-channel/functionality warning for a convert) тАФ LFB never compresses, converts, or writes a
`.gitignore` entry on its own (charter: "Never compress or alter a file unless the user explicitly
asks"; "Never add a `.gitignore` entry automatically"). Once confirmed, the work runs through the
progress dock as its `compress` / `ignore` op ([webapp.mdx ┬з11](./webapp.mdx#11-what-shows-up-in-the-dock--the-lfbridge-operation-taxonomy)).

### 6.2 Local bytes are never deleted by LFB
**Remove repo**, **Stop syncing (untrack/unpin)**, and **Remove peer** change only *LFB's tracking* тАФ
they unregister, unpin, or forget. They **never delete the user's folder or files** on disk (matching
the decision model: "changing a decision never deletes the user's local file", [one_repo.mdx ┬з4.5](./one_repo.mdx)).
Every destructive item is visually separated (last, red-tinted) and requires a confirm.

### 6.3 Permission gates
Menu items honor OpenAuthFederated claims exactly as the nav does ([left_bar.mdx ┬з5](./left_bar.mdx)):
a gated item (e.g. an admin-only action) is **hidden**, not disabled, when the claim is absent.

### 6.4 Local-only, never phones home
**Find duplicates (fingerprint)** and any hashing run **entirely locally** (charter: PhotoDNA-style
perceptual match with **no external networking**). No menu action performs a network call to a third
party; the only network LFB does is IPFS between the user's own machines.

### 6.5 The IPFS-only-our-content rule still holds
No menu action ever turns this machine into a public gateway/relay
([knowledge/ipfs.mdx ┬з6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)); "View on
IPFS" and "View pinned" are read-only views of **our own** pinned content.

### 6.6 The two sticky flags тАФ Never IPFS & Do not compress (LOCKED)

Both are **persistent per-entity flags** (not one-shot actions): they are stored, survive rescans, and
apply to **either a file or a directory**. They are the user's standing "leave this alone" wishes.

* **Never IPFS** тАФ the entity (and, for a directory, everything under it) is **never** added to IPFS or
  synced. While it is on: no `sync` decision may be set, the **Add to IPFS** item is **hidden**, the
  sync `S` badge never appears, and background sync skips it. Turning it **on** demotes any existing
  `sync` decision (a file тЖТ `ignore`; a directory тЖТ clears its files' `sync` decisions). It is a
  *policy*, stronger than a per-file `ignore` decision, and it is scoped by path so a directory's flag
  covers new files that appear later. It **never deletes local bytes** (┬з6.2) тАФ it only forbids syncing.
* **Do not compress** тАФ suppress the `C` "should be compressed" offer and badge for this file (or all
  media under this directory). Compression is already an explicit-click-only offer (┬з6.1); this flag
  makes the *offer itself* go away for entities the user has judged fine as-is. It does **not** alter
  the file.
* Both flags are **toggles** in the menu with a check when on, are reflected on the entity's page
  ([files.mdx](./files.mdx) / [directories.mdx](./directories.mdx)) as a labeled switch, and тАФ like every
  mutating action тАФ write immediately and can be flipped back at any time.

## 7. Acceptance criteria

1. In every TanStack table, **data cells are deep links**: a count cell opens the child list
   pre-filtered to that slice; a path/name cell opens the entity or the File System at that path; a
   Peers/pinned cell opens Peers filtered to that entity; a CID cell copies (does not navigate away);
   per ┬з2. Linking cells show a clickable affordance; the `тЛп` column never navigates.
2. Every table row (and every File System entry) has a trailing **`тЛп` kebab** that opens the **entity
   action catalog** (┬з5) for that row's entity, as a grouped popover with destructive items separated
   last; it is a control column and never opens the row. **Right-clicking the same entity opens the same
   menu** (┬з3.1).
3. Every **single-entity page** (view one repo / file / directory; the File System selection) has a
   **top-right `тЛп` "more" menu** rendering the **same catalog** as that entity's row kebab (┬з4); the one
