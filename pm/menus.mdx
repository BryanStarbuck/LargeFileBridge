# LargeFileBridge — Cell Navigation, Row "⋮" Menus & Entity "More" Menus

Every screen in LargeFileBridge (LFB) is built from the same two shapes: **tables** (the TanStack
`DataTable` used on [Repos](./repos.mdx), [One repo](./one_repo.mdx), [Peers](./left_bar.mdx),
[IPFS](./ipfs.mdx), …) and **single-entity pages** (view *one* repo, *one* file, *one* directory).
This file defines the three interactions those shapes share, **once**, so no other spec restates them:

1. **Cell navigation** — a click on a *cell* goes somewhere specific (§2).
2. **The row `⋮` kebab** — every table row's trailing per-row action menu (§3). **Right-clicking the
   row (or a file/dir entry anywhere) opens the same menu** (§3.1).
3. **The entity `⋮` "more" menu** — the same actions, top-right, on a single-entity page (§4). The
   two single-entity pages are [files.mdx](./files.mdx) (**View one file**) and
   [directories.mdx](./directories.mdx) (**View one directory**).

The rule that ties them together: **a table cell is a deep link, and the row's `⋮` (or a right-click,
or the page's top-right `⋮`) is that same entity's action menu.** Clicking *data* navigates; clicking
`⋮` (or right-clicking) *acts*.

Siblings that reference this file (and must not redefine these conventions):

* [Repos](./repos.mdx) — the repos table (repo rows).
* [One repo](./one_repo.mdx) — a single repo page (its top-right `⋮`) whose files table has file rows.
* [File System](./directories.mdx) — the column browser; its selected file/dir gets the entity menu.
* [IPFS](./ipfs.mdx) — the pinned-content table (file/CID rows).
* [Repo settings](./repo_settings.mdx) / [Settings](./settings.mdx) — where the "Settings…" items land.
* [Web app](./webapp.mdx#11-what-shows-up-in-the-dock--the-lfbridge-operation-taxonomy) — every action
  that does background work reports through the progress dock's **operation taxonomy**.

## 1. Vocabulary — Cell link / Row menu / Entity menu / Action (LOCKED)

| Term | Definition |
| --- | --- |
| **Cell link** | A table cell that, when clicked, **navigates** to the most specific screen for that cell's value (§2). Not every cell links; the ones that do are listed per table. |
| **Row `⋮` kebab** | The trailing, right-aligned control on **every** table row. It opens the **entity action menu** (§3) for the entity that row represents. It is a control column — it **never** triggers cell navigation or row navigation. |
| **Entity `⋮` "more" menu** | On a **single-entity page**, the top-right `⋮` button that opens the **same** action menu as that entity's row kebab (§4). "The page is one entity, so the page has that entity's menu." |
| **Entity** | The thing a row or page is *about*: a **repo**, a **directory**, a **file**, or a **peer**. Each entity has exactly **one** LOCKED action catalog (§5), used by both its row kebab and its page's more-menu. |
| **Action** | One menu item. Either **navigates** (opens a screen), **mutates** (does work — always through the progress dock + charter guardrails, §6), or **copies** (path / CID to clipboard). |

## 2. Cell navigation — clicking a cell is a deep link

Whole-row navigation still exists where a spec defines it (a Repos row opens that repo). **On top of
that**, individual cells navigate to the *most specific* view for their value, so the user can jump
straight to the slice they care about instead of opening the parent and re-filtering by hand.

### 2.1 The rule (LOCKED)

* **A count/metric cell** → the child list, **pre-filtered** to exactly that slice. (Clicking a repo's
  `Undecided 5` opens that repo's files table already filtered to the 5 undecided files.)
* **A path/name cell** → the place that path lives: a **repo name** → that repo's page; a **file path**
  → the [File System](./directories.mdx) browser revealing that file; a **directory path** → the browser
  at that directory.
* **A relationship cell** (Peers, pinned-by) → the related list **filtered to this entity**
  (`Peers 3` → [Peers](./left_bar.mdx) filtered to this repo/file).
* **A status/enum cell** → the same target as the row (the entity page), scrolled/aware of that status.
* **An identifier cell** (CID) → does **not** navigate away; it **copies** (monospace, click-to-copy) —
  and optionally opens the [IPFS](./ipfs.mdx) page filtered to that CID via a small inline affordance.
* **The `⋮` kebab column never navigates** — it only opens the menu (§3).

A linking cell shows a pointer cursor + hover underline/emphasis so it reads as clickable; a
non-linking cell (e.g. a raw size) does not. Middle-click / ⌘-click opens the target in context where
the platform supports it. Cell navigation **never mutates** — it only routes.

### 2.2 The per-table cell-link maps referenced elsewhere

Each table spec owns its own column→target map and links back here for the rule:

* **Repos table** — [repos.mdx §5.1](./repos.mdx#5-row--cell-navigation).
* **One-repo files table** — [one_repo.mdx §4.7](./one_repo.mdx#4-the-files-table-tanstack-house-style).
* **IPFS pinned table** — [ipfs.mdx](./ipfs.mdx).
* **Peers table** — its own spec, following this rule.

## 3. The row `⋮` kebab — every table has one

Every LFB TanStack table gets a **trailing control column** whose cell is a right-aligned **`⋮`
kebab** button. It is the standard, discoverable home for *acting on one row* — distinct from the
header **`⋮` bulk menu** (which acts on the *selected set*, [one_repo.mdx §4.4](./one_repo.mdx)): the
row kebab is the **last cell of each row**, the bulk menu is in the **control row above the table**.

The trailing control column that carries every row's ⋮ kebab is rendered by the shared table in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (the `rowMenu` trailing cell).

> **Orientation (LOCKED).** The kebab is always the **vertical** three-dot glyph **`⋮`** — three dots
> stacked **up-and-down**, never the horizontal **`⋯`**. This holds for every trigger: the row kebab,
> the right-click menu's affordance, and the single-entity page's "more" button (§4). One glyph,
> everywhere.

* **Placement.** Last column, fixed narrow width, right-aligned, no header label. It stays pinned as
  the row's other cells navigate — clicking `⋮` opens a menu, it does **not** open the row.
* **Every row, no exceptions.** *Every* row of *every* table carries the kebab — including rows that
  don't resolve to a file/dir/repo/peer (e.g. an untracked IPFS pin, §5.5, or a category-rollup row).
  A row that has no meaningful action still shows the kebab with at least a **Copy** and any
  navigate/inspect item; the column is never blank.
* **Contents.** The **entity action catalog** for whatever that row is (§5) — a repo row shows the
  **repo** catalog, a file row the **file** catalog, a pinned-CID row the **pin** catalog (§5.5), and
  so on. For a table whose rows are **not** one of the catalog entities (the directory category
  rollup, [directories.mdx §8](./directories.mdx)), the **owning spec** defines the row's menu
  contents; the vertical `⋮` column itself is still standard.
* **Popover.** A rounded floating popover (the house popover style used by the sort/filter dropdowns,
  [repos.mdx §3.1](./repos.mdx)), grouped with thin dividers, destructive items last and tinted red,
  gated items hidden without the claim (§6.3). Opens on click, closes on select / outside-click / `Esc`.
* **Keyboard + a11y.** Focusable button (`aria-haspopup="menu"`), arrow-key navigation, `Enter`
  activates, `Esc` closes and returns focus. Each item has a clear label; icons are decorative.
* **Optional row gear (shortcut to settings).** A row whose entity has a **settings page** may show a
  **⚙ gear** in the same trailing action cell, **immediately to the left of the `⋮`**, as a one-click
  shortcut to that settings page — the **Storages** rows ([storage_settings.mdx §1](./storage_settings.mdx#1-how-you-get-here--gear-kebab-right-click-more-menu))
  and **Repos** rows ([repo_settings.mdx §1](./repo_settings.mdx#1-how-you-get-here--the-gear-icon)) do
  this. The gear is a **redundant shortcut**, never a replacement: it lands on the exact same page as the
  kebab's **… settings…** item, and (like the kebab) stops row-click propagation so it opens settings
  without navigating the row.

The vertical ⋮ kebab button (stops row-click propagation, opens a portaled popover) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `KebabButton` component.

### 3.1 Right-click opens the same menu (LOCKED)

Wherever a **file or directory entry** is rendered — a table row, a [File System](./directories.mdx)
column-browser row, a tile — **right-clicking (context menu) opens the exact same entity action
catalog** (§5) as that entity's `⋮` kebab. The kebab is the discoverable, always-visible affordance;
the right-click is the power-user shortcut. They are two triggers for **one** menu component fed by
**one** catalog — never a divergent list.

* The context menu opens at the pointer, is flipped to stay on-screen, and closes on
  select / outside-click / `Esc` (same popover as the kebab).
* Right-click **suppresses the browser's native context menu** only over LFB entity rows; elsewhere the
  native menu is untouched.
* Right-click never *navigates* and never *selects* — it only opens the menu (the kebab discipline, §3).

The pointer-positioned menu (same catalog, clamped on-screen, closes on Esc / outside-click) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `EntityMenuAt` component.

### 3.2 The menu is ISOLATED from the row underneath it (LOCKED)

**Choosing a menu item does exactly what that item says — and nothing the row would have done.** A
click inside the popover **never** also fires the row's click. Selecting **Copy path** copies the path
and leaves you where you are; it does **not** open the file's viewer, and neither does any other item
whose own job isn't to navigate.

This is a **locked, product-wide rule** because a menu that also triggers its row is *wrong twice*: the
user lands somewhere they didn't ask for, **and** the action they did ask for is obscured by the
navigation.

* **Isolation is the popover's job, not each item's.** Every trigger (§3 kebab, §3.1 right-click, §4
  more-menu) shares **one** popover component, and **that** component stops the click. An item author
  never has to remember to stop propagation — a rule enforced per-item is a rule that breaks the first
  time someone adds an item.
* **Why the trigger button's `stopPropagation` is NOT enough (the trap).** The popover is **portaled to
  `<body>`**, so it looks isolated in the DOM — but **React re-dispatches events through the React
  tree, not the DOM tree**. The kebab is rendered *inside* the row, so a click on a portaled menu
  **item** still bubbles to the row's `onClick` handler. Stopping the click on the trigger button
  covers only the button itself; the items are a separate dispatch and stay exposed.
* **This is a fixed bug, kept documented so it cannot return.** Right-clicking a media file in a repo's
  file table and choosing **Copy path** navigated to that file's viewer (`/video?path=…`) instead of
  simply copying — the row's `onClick` fired via the portal's React-tree bubbling. Fixed by stopping
  `click` / `dblclick` at the popover root.

Menu isolation (the popover root stops `click`/`dblclick` so a portaled item can never fire the row's `onClick`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `MenuPortal` component.

The row `onClick` that this isolates the menu from is applied in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (the body `<tr>`, which also hosts the `rowMenu` kebab cell).

### 3.3 A copy action copies — and says what really happened (LOCKED)

A **copy** action (§5: **Copy path**, **Copy CID**, **Copy Peer ID**, and every click-to-copy identifier
cell) has exactly **one** effect: the text lands on the clipboard. It never navigates, never mutates,
never touches a file.

**The feedback must report the real outcome — never an optimistic guess.**

* **Confirm only a write that landed.** The clipboard write is **asynchronous** and can genuinely fail
  (permissions, an insecure context, an unfocused document). The toast/✓ waits for the write to settle:
  success **only** on a resolved write.
* **A failed copy says so.** On failure the user gets an explicit error — *"Couldn't copy the path to the
  clipboard"* — and the failure is logged to the [fault trail](../CLAUDE.md). A copy that silently fails
  while claiming *"Path copied"* is the worst case: the user pastes stale content and trusts it.
* **Fall back before failing.** When the async Clipboard API is unavailable or rejects, retry via the
  legacy `execCommand("copy")` path so the copy still lands; only report failure when **both** fail.
* **One clipboard write, everywhere.** Every copy surface calls the shared helper — no call site
  hand-rolls `navigator.clipboard.writeText`, because that is exactly how the optimistic-toast bug got
  copy-pasted across a dozen surfaces.

The one clipboard write (awaits the write, `execCommand` fallback, toasts the true outcome) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/clipboard.ts` — `writeClipboard()` (no toast; for surfaces with their own inline ✓) and `copyText()` (writes + toasts the real result).

The menu's copy actions (Copy path / Copy CID), which only copy and never navigate, are built in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `buildActions()`, the local `copy()` helper.

## 4. The entity `⋮` "more" menu — single-entity pages

A page that shows **one** entity — the [One repo](./one_repo.mdx) page, the **View one file** page
([files.mdx](./files.mdx)), the **View one directory** page ([directories.mdx](./directories.mdx)), or
a selected entry in the [File System](./directories.mdx) browser — carries a **top-right `⋮` "more"
button**. Opening it shows the **same catalog** as that entity's row kebab — because the page *is* that
entity.

```
┌───────────────────────────────────────────────────────────────────────┐
│  ‹ Repos  /  Our_Movies                          ┌────────────┐  ┌───┐ │
│  ~/BGit/…/Our_Movies                             │ ⟳ Pin now  │  │ ⋮ │ │ ← entity "more" menu
│                                                  └────────────┘  └───┘ │
└───────────────────────────────────────────────────────────────────────┘
                                                                    │
                              ┌─────────────────────────────────────▼──┐
                              │  Open in File System                    │
                              │  Rescan                                 │
                              │  Repo settings…                         │
                              │  Toggle Pinned                          │
                              │  Git ignore                             │
                              │  View pinned on IPFS                    │
                              │  Copy path                              │
                              │  ───────────────────────────────────── │
                              │  Remove repo (unregister)          ⚠    │  ← destructive, last
                              └─────────────────────────────────────────┘
```

* **Primary vs. more.** The single **most-common** action stays a visible button (e.g. *Pin now* on a
  repo page); everything else lives under `⋮`. The most-common action also appears **inside** the menu
  so the menu is a complete catalog.
* **Same catalog, one source.** The page-level menu and the row kebab render from the **same** per-entity
  definition (§5) — they never drift. If an action makes no sense at the page altitude (e.g. "Open" when
  you are already on the page), it is simply **omitted/disabled**, not a different list.

The single-entity page's top-right ⋮ "more" button, rendering the same catalog as the row kebab, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `EntityMore` component.

## 5. The per-entity action catalogs (LOCKED)

Four entities, four catalogs (§5.1–§5.4), plus the **pin catalog** (§5.5) for pinned-CID rows that
don't resolve to a local entity. Each is used verbatim by that entity's **row kebab** (§3) **and** its
**page more-menu** (§4). Order and grouping are as shown; **destructive items are last and separated**.

### 5.1 Repo

| Group | Action | Kind | Notes |
| --- | --- | --- | --- |
| Open | **Open repo** | navigate | → [`/repos/$repoId`](./one_repo.mdx). (Omitted on the repo page itself.) |
| Open | **Open in File System** | navigate | → [File System](./directories.mdx) at the repo's path. |
| Open | **View pinned on IPFS** | navigate | → [IPFS](./ipfs.mdx) filtered to this repo. |
| Work | **Rescan** | mutate | Runs the discovery scan for this repo now (`scan` op, [webapp.mdx §11](./webapp.mdx#11-what-shows-up-in-the-dock--the-lfbridge-operation-taxonomy)). |
| Work | **Pin now** | mutate | Pin/fetch this repo's Add-to-IPFS (pin) files — those with the `sync` decision (`pin` op). Never changes a decision. |
| Work | **Git ignore** | mutate | Opens the **Git Ignore pop-over dialog** ([git_ignore.mdx](./git_ignore.mdx)) scoped to this repo's working tree (Recursive checkbox shown; off = the immediate big files, on = the whole tree). Never edits `.gitignore` silently — the dialog is the confirm (§6.1). |
| Config | **Toggle Pinned** | mutate | Flip this repo's master `pinned` on/off ([repo_settings §2.2](./repo_settings.mdx)). |

| Copy | **Copy path** | copy | Absolute repo path to clipboard. |

### 5.6 The decision toggles are the FAST PATH; the menu is the same actions (LOCKED)

Two of the catalog's most-used mutations — **Add to IPFS / Remove from IPFS** ([§5.3](#53-file)) and
**Git ignore** ([§5.1](#51-repo) / [§5.2](#52-directory-non-repo) / [§5.3](#53-file)) — also have an
**inline, one-click** form: the two per-row **[decision toggles](./decision_toggles.mdx)** (the
**Add-to-IPFS pin** toggle and the **Add-to-git-ignore ⊘** toggle) rendered directly on file/dir rows
wherever files are listed. The relationship is deliberate and **additive**:

* **Pin actions and git-ignore actions are NEVER in one menu group — the menu mirrors the two-facet rule
  (LOCKED, [tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).**
  Add-to-IPFS and Add-to-git-ignore are **two independent booleans** with **two sources of truth** — the
  decision ledger's IPFS axis vs. live `git check-ignore`
  ([decisions.mdx §1.1](./decisions.mdx#11-this-ledger-records-intent--git-records-the-git-ignore-state-locked)) —
  and a file can be either, both, or neither. So in **every** catalog (§5.1–§5.3): the pin actions live in
  the **IPFS** group (**Add to IPFS** · **Remove from IPFS** · **Set pin ▸**), **Git ignore** lives in the
  **Work** group, and **no group is named "Decision"** — a heading that spans both axes is the same
  conflation §2.4 retires from the facet rail, and it does the same damage here: two adjacent items that
  read as one choice and write different state. This is the **same split**, expressed three ways — two
  facets on the rail, two sort keys, two menu groups.
* **The toggles are the fast path.** On a table row the user flips a decision **in place** — no menu to
  open — and each click writes **one axis** of the team-shared decision immediately
  ([decision_toggles.mdx §2](./decision_toggles.mdx#2-clicking-a-toggle--one-decision-event-locked)). The
  toggle **stops click propagation** and never navigates, exactly like the `⋮` kebab column (§3).
* **The menu is the complete, discoverable list.** The **`⋮` kebab / right-click / more-menu** still offer
  the **same actions** in words — **Add to IPFS** / **Remove from IPFS**, the three-state **Set pin**
  submenu, and **Git ignore** — for users who prefer the labelled catalog, for keyboard/a11y navigation, and
  for the cases the toggle **defers to the menu's dialog**: a **directory** ignore and a **multi-row** flip
  open the **[Git-ignore pop-over](./git_ignore.mdx)** / bulk menu rather than writing silently
  ([decision_toggles.mdx §2.1](./decision_toggles.mdx#21-bulk-clicks-and-directories-still-open-the-pop-over-locked),
  §6.1 here). The two surfaces write the **same** single-axis decision, so a decision set by a toggle shows
  in the menu and vice-versa — they never diverge.
* **The git-ignore toggle is repo-scoped, matching the menu.** The **Add-to-git-ignore ⊘** toggle renders
  **only** for files/dirs under a repo (it is a blank **N/A** slot otherwise), mirroring the menu's
  **Git ignore** item, which is likewise meaningful only inside a repo
  ([decision_toggles.mdx §4](./decision_toggles.mdx#4-the-git-ignore-toggle-is-repo-scoped-locked)). The
  **Add-to-IPFS pin** toggle has no such restriction — any large file can be pinned, just as the menu's
  **Add to IPFS** appears for any eligible file.
* **A To-Do batch's recommendations mirror these actions.** The **[batch popup](./to_do.mdx#6-the-batch-popup-the-evolved-warnings-dialog-locked)**
  (the evolved warnings dialog) renders the **same** decision toggles per subject row, **pre-set to
  RECOMMENDED** (dashed) from the [TO DO Batch Calc Engine](./to_do_batch_calc_engine.mdx); confirming a
  recommendation there writes the identical single-axis decision the row toggle and the menu item write.
  So a file's decision can be set from **four** consistent places — the inline toggle, the entity menu, the
  bulk menu, or a To-Do batch — all routing to the one decision-write path.
* **The two sticky flags stay menu-only.** **Never IPFS** and **Do not compress** (§6.6) are **not**
  toggles-on-the-row — they are per-entity *policies* set from the menu. A file with **Never IPFS** on shows
  **no** Add-to-IPFS toggle (the axis is forbidden, so it renders as OFF/absent per
  [decision_toggles.mdx §4](./decision_toggles.mdx#4-the-git-ignore-toggle-is-repo-scoped-locked) / §6.6
  here), keeping the fast path and the policy consistent.

The inline toggles and their menu twins share the one decision-write path in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/decision/DecisionToggles.tsx` — `DecisionToggle`
(the row/popup toggle) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `buildActions()`

## 6. Guardrails the menus inherit (LOCKED)

These are charter rules the menus must honor — they are **why** several actions are confirm-gated.

### 6.1 Never act on files without an explicit click — and NEVER via a native browser dialog (LOCKED)
**Compress, convert (PNG→compressible), and git-ignore are OFFERS, not automatic.** They exist in the
menu, but selecting one **opens a pop-over configuration dialog** that *is* the confirm step — the
[Compress Inside dialog](./compression.mdx) for compress, the [Git Ignore dialog](./git_ignore.mdx)
for git-ignore, and the alpha-channel/functionality warning for a convert — LFB never compresses,
converts, or writes a `.gitignore` entry on its own (charter: "Never compress or alter a file unless the
user explicitly asks"; "Never add a `.gitignore` entry automatically"). Once applied, the work runs
through the progress dock as its `compress` / `ignore` op ([webapp.mdx §11](./webapp.mdx#11-what-shows-up-in-the-dock--the-lfbridge-operation-taxonomy)).

> **The confirm is ALWAYS one of our HTML modals — never `window.confirm` / `window.alert` / `window.prompt`
> (LOCKED — [dialogs.mdx](./dialogs.mdx)).** Every menu action that asks "are you sure?" opens a
> **`ConfirmDialog`** (or, for a value the user must type — the **Move…** destination path — a
> **`PromptDialog`**), never an operating-system browser dialog. The imperative handlers `await
> confirmModal(...)` / `await promptModal(...)` ([dialogs.mdx §2.3](./dialogs.mdx)). This applies to **every**
> destructive item in every catalog (§5): **Remove repo**, **Stop pinning / Unpin**, **Remove peer**,
> **Delete… (to trash)**, **Move…**, single-file **Compress… / Convert…** — all HTML modals, zero native
> dialogs ([dialogs.mdx §3](./dialogs.mdx) catalogs every converted site).

The explicit-click Compress OFFER with its `confirmModal` step (never compresses a file on its own, never a native `window.confirm`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `buildActions()` `compress` handler (uses `confirmModal()` from `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/modals.ts`); the **Move…** destination uses `promptModal()` from the same module.

> **Directory compress is no longer a native confirm (§5.2).** The old bare browser confirm for
> **Compress videos/images inside…** is **superseded** by the pop-over configuration dialog in
> [compression.mdx §12](./compression.mdx) (four inputs + a per-file-safe background batch). The
> single-file **Compress…** offer (§5.3) is unchanged in intent — it stays a confirm-gated per-file offer,
> but its confirm is now a **`ConfirmDialog`** ([dialogs.mdx](./dialogs.mdx)), not a `window.confirm`.

> **Git ignore is no longer a bare confirm (§5.1–§5.3).** The old **Git-ignore…** items fired a generic
> ConfirmDialog (or a not-yet-wired stub). They are **superseded** by the pop-over **Git Ignore dialog**
> ([git_ignore.mdx](./git_ignore.mdx)): it resolves the target from the **rightmost column with a selection**,
> adapts to the selection shape, shows a **Recursive** checkbox for directories, **previews the exact
> `.gitignore` lines**, and its **Apply Git ignore ›** / **Recursive apply ›** button writes anchored,
> repo-root-relative lines into each owning repo's `.gitignore` (skip-already-ignored, multi-repo aware,
> never a `.lfbridge/` path).

### 6.2 Local bytes are never deleted by LFB
**Remove repo**, **Stop pinning (untrack/unpin)**, and **Remove peer** change only *LFB's tracking* —
they unregister, unpin, or forget. They **never delete the user's folder or files** on disk (matching
the decision model: "changing a decision never deletes the user's local file", [one_repo.mdx §4.5](./one_repo.mdx)).
Every destructive item is visually separated (last, red-tinted) and requires a confirm.

### 6.3 Permission gates
Menu items honor OpenAuthFederated claims exactly as the nav does ([left_bar.mdx §5](./left_bar.mdx)):
a gated item (e.g. an admin-only action) is **hidden**, not disabled, when the claim is absent.

### 6.4 Local-only, never phones home
**Find duplicates (fingerprint)** and any hashing run **entirely locally** (charter: PhotoDNA-style
perceptual match with **no external networking**). No menu action performs a network call to a third
party; the only network LFB does is IPFS between the user's own machines.

### 6.5 The IPFS-only-our-content rule still holds
No menu action ever turns this machine into a public gateway/relay
([knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)); "View on
IPFS" and "View pinned" are read-only views of **our own** pinned content.

### 6.6 The two sticky flags — Never IPFS & Do not compress (LOCKED)

Both are **persistent per-entity flags** (not one-shot actions): they are stored, survive rescans, and
apply to **either a file or a directory**. They are the user's standing "leave this alone" wishes.

* **Never IPFS** — the entity (and, for a directory, everything under it) is **never** added to IPFS or
  pinned. While it is on: no `sync` decision may be set, the **Add to IPFS** item is **hidden**, the
  pin `P` badge never appears, and background pinning skips it. Turning it **on** demotes any existing
  `sync` decision (a file → `ignore`; a directory → clears its files' `sync` decisions). It is a
  *policy*, stronger than a per-file `ignore` decision, and it is scoped by path so a directory's flag
  covers new files that appear later. It **never deletes local bytes** (§6.2) — it only forbids pinning.
* **Do not compress** — suppress the `C` "should be compressed" offer and badge for this file (or all
  media under this directory). Compression is already an explicit-click-only offer (§6.1); this flag
  makes the *offer itself* go away for entities the user has judged fine as-is. It does **not** alter
  the file.
* Both flags are **toggles** in the menu with a check when on, are reflected on the entity's page
  ([files.mdx](./files.mdx) / [directories.mdx](./directories.mdx)) as a labeled switch, and — like every
  mutating action — write immediately and can be flipped back at any time.

The two sticky per-entity toggles — Never IPFS and Do not compress — are built and written in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/EntityMenu.tsx` — `buildActions()` (the `flags` mutation via `api.setEntityFlags`).

Whether a file earns the pin **`P`** badge (and Never IPFS suppressing it) is computed in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `fileIsPinned()`.
