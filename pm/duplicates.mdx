# LargeFileBridge — Duplicates

The **Duplicates** page answers: *which of my videos and images exist more than once on this
computer — and are the copies truly the same, so I can safely delete the extras?* It shows
duplicate **groups** in a table and lets the user review each group's members **side-by-side** in a
right review column with real previews, so "same or not" is judged with eyes, not faith.

Two copies of the same content are often **not byte-identical**: they may differ in **codec**, in
**file size**, and in **pixel depth / resolution** — a ProRes master and its H.264 export, a PNG
and its heavily-compressed JPEG. That is why grouping runs on **two identities**: the exact
**SHA-256** hash (byte-identical copies) and the **perceptual fingerprint**
([perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx) — PhotoDNA-style capability, PDQ/vPDQ,
**not** PhotoDNA, entirely local).

**LFB never deletes on its own.** The page surfaces groups and gives the user the evidence; any
delete is an explicit user action per file (charter: detect, surface, offer — never act).

> name the **actual** homes. Implementation deltas from the planned pointers: the router is the one

## 3. User interface — the group table (left 60%) (LOCKED — revised 2026-07-22)

The page uses the shared **60/40 review layout** ([videos.mdx §3](./videos.mdx)): the table owns
the left **60%** of the content width; the right **40%** is the review column (§4). The table is a
house TanStack `DataTable` with **500-per-page** pagination. Nothing renders beneath it, so it is a
**full-page-height** table — the body scrolls inside, header and count/pagination footer stay pinned
(charter table rule).

**The control row spans the FULL page width, above the split** ([tables.mdx
§2.13](./tables.mdx)). Search, `⇅`, `⛛` and `⚏` govern the whole page, so they get the whole page;
**the right review column starts underneath that bar, never beside it.**

**ONE ROW PER GROUP.** A group — not a file — is the unit the user reviews: they pick a group, then
compare its files in the right column. The table therefore lists **groups, one line each**, labelled
with **one of the group's file names**.

> **WHAT THIS REPLACED, AND WHY.** The table used to interleave a slim group-header row with one row
> per member. A nine-file group ate ten rows, a tall screen showed three groups, and the page's whole
> job — *scan the list of duplicate sets, biggest win first* — needed constant scrolling. Worse, the
> per-file rows implied per-file review in a surface that has no room for it: the files sit side by
> side in the right column, which is the only place they can actually be compared. Collapsing to one
> row per group is not a density tweak; it makes the left side a list of DECISIONS and the right side
> the evidence for the one in focus.


### 3.1 The column set — deliberately small (LOCKED)

This table has **fewer columns** than the file tables elsewhere. Exactly:

| Column | Content | Rules |
| --- | --- | --- |
| **File** | **One of the group's file names**, basename only — no path. Any member's name is truthful (they are the same content), so it is simply the **first member in display order**; on Subsets that is deliberately the **superset**. | The full path is NOT a column — it lives in the right column's file blocks and in the left-bar hover-info panel ([non_intrusive_tooltip.mdx](./non_intrusive_tooltip.mdx)). Truncate with ellipsis. |
| **Files** | How many files are in the group. | Right-aligned. The one group-level fact the old header row carried that the single row must not lose. First column to drop on a narrow split ([tables.mdx §4a](./tables.mdx)). |
| **Size** | The group's **reclaimable bytes** — what deleting the extra copies would free (sum of member sizes minus the largest; Subsets: sum of the subsets' sizes). | Right-aligned, house size formatting. This is the number that ranks the list. |

**Explicitly absent (LOCKED):** there is **no leading checkbox column** (no multi-select on this
table), **no bookmark column**, no Path, no per-member rows — and **no icon control columns**. The
five icons (Pin · Ignore · Transcribe · AI description · OCR) describe a **file**, and the table's
rows are **groups**, so they moved to the right review column and sit beside the files themselves
(§4.3). File · Files · Size — that is the whole table.

### 3.2 Ordering, search, and filtering (LOCKED)

* Groups sort by **reclaimable bytes descending** by default — the most disk-winning groups first;
  the `⇅` dropdown and header-clicks re-sort per house rules.
* **Search matches on EVERY member's name and path**, even though the row shows only one name.
  Anything else would be a trap: typing the name of the duplicate you are hunting for would silently
  find nothing because that file was not the one the row happened to display. Search is client-side
  over the served rows (the whole CSV ships to the page — house table pattern).
* The `⛛` filter carries the File-type facet (Videos · Images — [tables.mdx
  §2.10](./tables.mdx) subset) and a **match-basis** facet (`sha256` · `fingerprint`),
  value-checkbox model, all checked by default. Both are **group-level** values, so filtering keeps
  whole groups.

### 3.3 Selection and hover — what the right column is showing (LOCKED — added 2026-07-22)

The right column always shows **one** group. Two inputs decide which, and they are deliberately
different in character:

| Input | Character | Row tint |
| --- | --- | --- |
| **Selected** — the group the user **clicked** | **Sticky.** Survives the pointer leaving the table, a scroll, a re-render | **light yellow** |
| **Hovered** — the group the pointer is **over** | **Transient.** Lives only while the pointer is on that row | **light green** |

Rules:

* **Hover wins while it lasts; selection is the fallback.** Move the pointer down the list and the
  right column follows it — that is how a user surveys twenty groups in a few seconds without
  clicking anything. Move the pointer **off the table** and the column falls back to the **selected**
  group, so the thing they deliberately chose is still there when they reach for it.
* **Clicking a row selects it.** That is what makes a group stick while the user works in the right
  column (scrolling it, playing a preview, reading paths).
* **The first group is selected on arrival**, so the column is never empty while real results sit on
  screen. It is seeded **once**; after that only a rescan that removes the group clears it — never a
  re-render, or **Done** (§4.1) would be undone the instant it was pressed.
* Keyboard **↑/↓** moves the **selection** between groups; it is inert while typing in the search box
  or interacting with a media element.
* The hover CLEAR is wired to the table **body**, never to each row — see [tables.mdx
  §2.14](./tables.mdx) for why a per-row clear flickers.
* **The hover is debounced (~120 ms).** A hover is a *sweep*: dragging the pointer down two hundred
  rows would otherwise build two hundred review columns and request a grant and a poster for every
  file in every group brushed past. A short pause is also what the user means by "show me this one".

The group row model (one row per group, the representative file name, reclaimable bytes, the search text spanning every member) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/videos/videoGroups.tsx` — `buildDuplicateGroups()` / `buildVideoColumns()`.
Selection + hover precedence and the full-width control-row slot are handled in code:

### 4.3a Previews are POSTER-FIRST, and the player only mounts on a click (LOCKED — added 2026-07-22)

> **THE BUG THIS SECTION EXISTS TO PREVENT.** Every block mounted a live
> `<video src="/api/media/raw">`, and **the previews simply never appeared** — no error, no broken-image
> glyph, nothing. Nothing had failed: the requests were never sent. A browser allows about **six
> connections per origin** over HTTP/1.1, and a media element **holds its connection open** while it
> buffers. A nine-member group therefore starved itself — the first few players took every socket, the
> rest never started, and **the page's own `/api` calls queued behind them**, including the media grants
> for those very members. The group with the biggest win was, inevitably, the group with the most
> files and the largest ones (nine 4 GB 4K masters), so the worst case was the first thing the user saw.
> A preview surface that shows N files at once cannot open N streams; that is not a heavy version of
> the right design, it is the wrong design.

The rules:

1. **Each block loads ONE small cached JPEG poster**, not the media file. It finishes in milliseconds
   and releases its connection, so a group of any size loads.
2. **The backend renders the poster** — `GET /api/media/poster`, gated by the **same signed grant**
   as `/raw` and `/stream` (a plain `<img>` cannot send a Bearer header). Images go through `sharp`,
   video frames through the bundled **ffmpeg** (fast input seek, one frame). This is 100% local —
   no network, ever (charter).
3. **It fixes the second failure class for free:** formats the browser cannot decode *at all* —
   **HEIC/HEIF, TIFF, ProRes, HEVC** — still show a picture, because **we** decode them. Previously
   these were silently blank even when the bytes streamed perfectly.
4. **Posters are cached, keyed on the file's content** (path + size + mtime + width), so re-hovering a
   group is instant and a replaced file can never show a stale frame. Generation is
   **concurrency-capped** so a twelve-member group cannot spawn twelve simultaneous 4K decodes.
5. **Videos do not autoplay. The user clicks the ▶ to play** — mounting a dozen autoplaying streams is
   exactly the problem the poster solves. The click swaps in the real `<video autoPlay controls>` for
   **that one file**; the poster stays as its `poster` attribute so the frame never blinks away.
6. **A codec the browser refuses falls back to the transcode pipe** (`/api/media/stream`,
   [codecs.mdx §6](./codecs.mdx)) exactly as the media viewer does — and only if **that** also fails
   does the block say **"Preview unavailable"**, with a **try again** link. A blank rectangle is never
   an acceptable answer.
7. **Grants are refreshed inside their 10-minute TTL**, so a review column left open on screen never
   starts serving 403s from expired URLs.

### 4.4 The resolution print rule (LOCKED)

Resolution prints as **`{width}x{height}`**, and **only when the dimensions snap to a known common
class** is the class appended in parentheses: **`1280x720 (720p)`**. The known classes:
**480p, 720p, 1080p, 2K, 4K** (extendable — the list is a named constant, not scattered literals).
Dimensions that match no class print bare: `1442x1080`. Snapping tolerates the usual encode
variance (±8 px per axis; portrait matches by the transposed dimensions).

The class-snapping helper is one shared function in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/videos.ts` — `resolutionLabel(width, height)` (the `RESOLUTION_CLASSES` named constant; `codecLabel()` sits beside it for §4.5).

### 4.5 The codec print rule (LOCKED)

The codec — the compression-algorithm format of the file (h264, hevc, prores, av1, vp9; png, jpeg,
heic, webp for images) — prints as the word **`codec`** followed by the value **in parentheses**:

## 5. Starting a scan — the action-links row and the entry pop-up (LOCKED)

A duplicate scan can start **two** ways, and the two are deliberately different in character:

| Surface | Character | When the user sees it |
| --- | --- | --- |
| **The page action-links row** (§5.1) | **Always available, non-modal** — the user's own initiative, one click, no dialog | **Every** visit, directly under the page title |
| **The entry pop-up** (§5.3) | **A recommendation of ours** — a one-time nudge that interrupts | Only when the data is old enough to be worth interrupting for (§5.2) |

> **The design principle (LOCKED): a modal is an interruption, and an interruption has to earn its
> place.** The **row** is the durable, discoverable home of the control — it is always there, so the
> pop-up never has to be. The **pop-up** exists only to catch the case where the user does not yet
> know the page's data is old. The moment the data is fresh, the pop-up must get out of the way
> completely; the row alone is enough.

(The Subsets page has its **own, separate** row link, pop-up, scan, and clocks —
[subsets.mdx §5](./subsets.mdx); running one scan never satisfies the other.)

### 5.1 The page action-links row (LOCKED)

**The row holds exactly ONE link (revised 2026-07-22).** Two others were removed, and the reasons are
worth keeping, because both looked useful:

* **"Refresh results" is gone.** The page already refetches on the `videos` live-refresh topic (which
  every publish bumps, §8.3) and on the running-status poll. A manual Refresh could therefore only
  ever repeat what had already happened — and its presence implied the opposite, that the numbers on
  screen might be stale unless you press something. A page that refreshes itself must not ship a
  button suggesting it does not.
* **"View progress" is gone.** It competed for attention with the one control that matters on this
  page, and it was never the only route: the start toast carries its own **View progress** action

  left-bar item appears on its own while a batch runs (§6). Three honest routes already existed.

Rules:

* **The row is never hidden and never moves.** It does not appear only when a scan is recommended,
  and it does not disappear while one runs — a control that relocates itself is a control the user
  has to re-find.
* **The row takes the highest z-order of the page content.** It is one line of links directly under
  the title with nothing above it, so anything painted after it — the control row that follows, a
  sticky table head — was winning the stacking order and **clipping the links' bottom edge**. The fix
  is z-order only: the row **never pushes anything down**, it just stops being covered.
* **While a scan is running, "Scan for duplicates" is disabled with a stated reason** in its
  `title` — *"A duplicate scan is already running — see the Processing page"*. A greyed link with
  no reason is a dead end ([menus.mdx](./menus.mdx)); the reason plus the neighbouring **View
  progress** link turns the dead end into the next step.
* **The row is the answer to a suppressed pop-up.** Because the pop-up can be quiet for days
  (§5.2), the row is what guarantees the user can *always* rescan on demand — that is precisely why
  the quiet window is safe.
* The Subsets page carries the identical row with **Scan for subsets**
  ([subsets.mdx §5](./subsets.mdx)).

### 5.2 When the pop-up opens — the 2-day quiet window (LOCKED)

> **THE ANNOYANCE THIS SECTION EXISTS TO PREVENT.** The first rule opened the pop-up whenever the
> last run was 4+ days old **or had not completed**. A long scan interrupted by an ordinary backend
> restart leaves `complete: false` **forever** — nothing ever clears it but a full successful run.
> So a user who had scanned **an hour ago**, and whose fresh results were **already on screen behind
> the modal**, was asked to scan again on **every single visit** to the page. A recommendation that
> fires when the answer is already fresh is not a recommendation; it is a dialog the user learns to
> dismiss without reading — which is exactly how a genuinely stale-data warning later gets ignored.

The evaluation, in order — **the first matching rule wins**:

| # | Condition | Pop-up |
| --- | --- | --- |
| 1 | A duplicate scan is **running right now** | **No** — the page shows the running state (§5.5) instead |
| 2 | The last run was **within the last 2 days** (`VIDEOS_SCAN_PROMPT_QUIET_DAYS = 2`) — **complete or partial, it does not matter** | **No** — the quiet window. Two days is fresh; we do not interrupt over it. |
| 3 | The scan has **never run** | **Yes** |
| 4 | The last run is **2+ days old and did not complete** (§8.5) | **Yes** — partial data, and the job is worth finishing |
| 5 | The last run is **4+ days old** (`VIDEOS_SCAN_STALE_DAYS = 4`) | **Yes** — stale |
| 6 | otherwise (a complete run between 2 and 4 days old) | **No** |

* **Two clocks, deliberately.** The **quiet window (2 days)** is a *suppression* rule — it can veto
  a prompt no matter what else is true. The **staleness window (4 days)** is a *recommendation*
  rule. Between them (2–4 days, run complete) neither fires: the data is neither fresh enough to
  brag about nor old enough to interrupt over, so we say nothing and leave the row (§5.1).
* **`complete: false` no longer overrides freshness.** It still *recommends* a rescan (§8.5), and
  it still opens the pop-up once the run has aged past the quiet window (rule 4) — but it can no
  longer make a two-hour-old scan nag on every visit.
* **At most once per page entry.** The pop-up auto-opens **once**; skipping it does not re-arm it
  for that visit — no re-open on a live-refresh, a status poll, or a scan completion.
* **The status route carries both verdicts as separate fields**, because they mean different
  things: **`promptOnEntry`** (may we interrupt?) drives the auto-open, and **`recommend`** (would a
  scan be useful?) is available for wording and affordances. The page auto-opens on
  `promptOnEntry` alone.

The two clocks are evaluated in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/dedupe.service.ts` — `dedupeStatus()` (`VIDEOS_SCAN_PROMPT_QUIET_DAYS` suppression + `VIDEOS_SCAN_STALE_DAYS` recommendation, both from `@lfb/shared` `videos.ts`); the auto-open-once latch is `VideosReviewPage.tsx` (the `autoOpened` ref, keyed on `status.promptOnEntry`).

### 5.4 Start is fire-and-forget — instant dismissal, a toast, background work (LOCKED)

> **THE BUG THIS SECTION EXISTS TO PREVENT.** Pressing **Start scan** left the modal sitting on
> screen over a dimmed page while the browser waited for the POST to resolve — and the POST did not
> resolve promptly, because the request handler entered the engine's **synchronous prologue**
> (config reads, unit enumeration) **before the response was flushed**. From the user's chair the
> app simply **hung**: no progress, no dismissal, no way to tell "starting" from "frozen". A start
> button must never be able to look like a crash.

The rules — they apply identically to the pop-up's **Start scan ›** and to the row's **Scan for
duplicates** (§5.1), because both call one shared start path:

1. **The modal closes on the click, not on the response.** Dismissal is optimistic and immediate;
   the HTTP request is already in flight behind an unblocked page. There is **no** blocking
   spinner, no dimmed overlay, no disabled page.
2. **A bottom toast confirms the start** — house toast, bottom of the viewport
   ([dialogs.mdx](./dialogs.mdx)): **"Duplicate scan started — it runs in the background."** with a
   **"View progress"** action that navigates to `/processing`. The toast, not the modal, is what
   tells the user the click landed.
3. **The page flips to its running state immediately** (§5.5) — optimistically, without waiting for
   the next status poll. Two confirmations of one action (toast + banner) is correct here: the
   toast is transient, the banner is the durable state.
4. **The server flushes the HTTP response before the engine does any work.** The POST handler marks
   the single-flight state **synchronously** — so an immediate second click coalesces and
   `GET …/status` already reads `running: true` — and then **defers the run to the next tick**. No
   enumeration, no config read, no filesystem walk may happen on the request's stack.
5. **A coalesced start is a normal outcome, not an error.** `started: false` shows the info toast
   **"A duplicate scan is already running"** with the same **View progress** action.
6. **A failed start is a toast, never a stuck modal.** The modal is already gone; the error surfaces
   as an error toast and the row's link stays clickable for a retry.

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/videos/VideosReviewPage.tsx` — the `start` mutation (`onMutate` closes the modal + flips the optimistic running status; `onSuccess` raises the toast with its View-progress action); the server side is

## 7. The fingerprint libraries — evaluation & choices (web-researched, LOCKED)

Duplicate grouping beyond byte-identity runs on perceptual fingerprints. The platform-level
algorithm family is owned by [perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx); this
section pins **which library detects duplicates on this page, per media type** — one choice for
**images**, one for **videos** (they turn out to share one primitive) — and the operational
contract around them. ([subsets.mdx §7](./subsets.mdx) makes its own, different choice for
detecting **subsets** — segment containment is a different problem.)

### 7.1 The four hard requirements (LOCKED)

Any library used here must satisfy **all four**; failing any one disqualifies it:

1. **Open source and embeddable.** A library we include and ship inside our TypeScript/Node app
   (npm package, WASM, or a local native binding we build) — never an external service, and under
   a permissive license (MIT/BSD/Apache; GPL is avoided — [perceptual_fingerprint.mdx §2.1](./perceptual_fingerprint.mdx)).
2. **A storable, text-encodable fingerprint.** The computed value must serialize to a compact
   **text string** (hex or hex lines) so it can sit **in the `fingerprint` column of
   `duplicates.csv`** (§9) and in the per-file YAML sidecar.
3. **A compare function that takes stored values back.** Later comparisons must be a pure function
   over two **previously stored** fingerprint strings (Hamming distance / match score) — grouping
   must **never** require re-decoding the media bytes.
4. **No phone-home (charter).** No network call of any kind — not to do the work, not to report
   anything. Enforced by the no-network guard test
   ([perceptual_fingerprint.mdx §6](./perceptual_fingerprint.mdx)).

### 7.2 Image libraries evaluated (for duplicate detection)

| Library | Fingerprint (text-encodable?) | Compare function (stored values in?) | Local / no phone-home | License | Verdict |
| --- | --- | --- | --- | --- | --- |
| **PDQ** (Meta; Rust `pdqhash` crate via napi-rs or WASM) | ✅ 256-bit → **64-hex string** + a quality score | ✅ in-library **Hamming distance** over two hashes | ✅ pure computation | BSD / Apache-2.0 | **CHOSEN (primary)** |
| **blockhash-core** (npm) | ✅ 256-bit → hex string (`bits: 16`) | ✅ Hamming over the two hex strings | ✅ pure JS | MIT | **CHOSEN (fallback #1)** |
| **sharp-phash** (npm) | ✅ 64-bit DCT pHash string | ✅ ships its own **`sharp-phash/distance`** compare | ✅ pure JS on `sharp` | MIT | **CHOSEN (fallback cross-check)** |
| **imghash** (npm) | ✅ hex output | ⚠️ no bundled distance helper (external Hamming) | ✅ | MIT | viable, redundant with the above |
| **node-phash** (pHash.org binding) | ✅ 64-bit | ✅ `hammingDistance` | ✅ | **GPLv3 ⚠ + stale binding** | rejected (license + maintenance) |
| **PhotoDNA** (Microsoft) | ⛔ proprietary block | ⛔ service-side | **⛔ cloud service + reporting model** | proprietary, gated | **disqualified** (fails #1, #4) |

### 7.3 Video libraries evaluated (for duplicate detection)

| --- | --- | --- | --- | --- | --- |

| **TMK+PDQF** (Meta) | ⚠️ ~256 KB **binary** signature — not text-friendly | ✅ cosine/level-2 score | ✅ | BSD | rejected (fails #2; no Node path) |

### 7.4 The choices, per media type (LOCKED)

* **Images — to detect duplicates: PDQ (256-bit).** Transform-robust (survives resize,
  re-compression, PNG→JPEG conversion, pixel-depth change), a compact **64-hex** CSV-friendly
  value, an in-library Hamming compare, and a quality score to gate junk images. Run via the Rust
  `pdqhash` crate wrapped napi-rs/WASM. **Until that binding lands:** **blockhash-core** (256-bit
  hex, MIT, pure JS) cross-checked by **sharp-phash** (64-bit, MIT, own `distance` helper) — both
  values can be stored side-by-side during the transition
  ([perceptual_fingerprint.mdx §3](./perceptual_fingerprint.mdx)).
* **Videos — to detect duplicates: vPDQ (per-frame PDQ).** The same PDQ primitive over frames
  sampled by the **bundled ffmpeg**; two videos are duplicates when a high fraction of frames
  match. Its fingerprint is **natively text lines**, and matching feeds two **stored** frame lists
  back in — requirements #2 and #3 exactly. One primitive covers both media types.
* These choices are **for duplicate detection on this page**. Subset detection needs
  offset-reporting containment and picks differently — MPEG-7 Video Signature primary
  ([subsets.mdx §7](./subsets.mdx)).

### 7.5 How to use the algorithm

* **Images:** decode with `sharp` → internal bounded downscale (`resize(512, 512, { fit: "inside"
  })` on a throwaway buffer — [perceptual_fingerprint.mdx §1](./perceptual_fingerprint.mdx)) → RGB
  pixel buffer → `pdq.hash(buffer)` → `{ hash: <64-hex>, quality: 0–100 }`. Fallback path:
  `blockhash-core` at `bits: 16` → 64-hex; `sharp-phash` → 64-bit string.
* **Videos:** bundled ffmpeg samples frames (~1 fps) → each frame through the **same** image
  pipeline → an ordered list of `frame_number,hex,quality,timestamp` lines (the vPDQ format).
* Everything is a pure local computation; the only external process is the bundled ffmpeg.

### 7.6 When to run it

* **During the dedicated duplicate scan only** (§8, phase 2 of the engine) — never on page load,
  never on a request path, never in the 15-minute/4-hour scheduleTasks.
* **Compute only what is missing or stale:** a file is (re)fingerprinted only when it has no
  stored fingerprint or its `sha256` changed since the fingerprint was computed. A warm re-scan
  hashes almost nothing.
* Freshly computed fingerprints are **also written back to the per-file sidecar** so every other
  consumer (compression before/after verify, decisions matching) shares them
  ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).

### 7.7 Where the fingerprint values are stored

* **Canonical home:** the per-file `<name>.yaml` sidecar in Local Storage
  (`~/T/_large_files_bridge/repos/<repoKey>/files/…` — [perceptual_fingerprint.mdx §5](./perceptual_fingerprint.mdx)).
* **In `duplicates.csv` (§9), the `fingerprint` column:** for **images**, the 64-hex value
  **inline**; for **videos**, the frame list is too large for one cell, so the column carries the
  **relative reference to the `.vpdq` sidecar file** — itself plain text lines
  (`frame_number,hex,quality,timestamp`), so the whole chain stays text-encodable.
* Fingerprints are Category-B computed state — state root only, never a working repo
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).

### 7.8 How comparisons run later — stored values back into the library (LOCKED)

Grouping (§8, pass 2) and any later re-grouping read **only stored fingerprints** and feed them
back into the library's compare functions — media bytes are never re-read to compare:

* **Images:** `hammingDistance(hexA, hexB)` over the two stored 64-hex strings; same-content at
  **≤ 32/256**, with the stricter **≤ 24** used for automatic de-dup grouping, plus the PDQ
  **quality gate** (fallback: Hamming over blockhash hexes, cross-checked by
  `sharp-phash/distance ≤ ~10`) — thresholds owned by
  [perceptual_fingerprint.mdx §4](./perceptual_fingerprint.mdx).
* **Videos:** the vPDQ matcher takes **two stored frame lists** and returns the fraction of
  frames matching within the image threshold; the pair is a duplicate when the shared-frame
  fraction clears the high bar **in both directions** (symmetric — this is what distinguishes a
  duplicate from a mere subset).
* The compare boundary is one module so the fallback→PDQ swap never touches the engine.

The hash + compare boundary is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/perceptual.service.ts` — `fingerprintImage()` / `hammingDistance()` / `sameContent()` ([perceptual_fingerprint.mdx §3](./perceptual_fingerprint.mdx)); the video side is
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/vpdq.service.ts` — `ensureVpdqFrames()` (sha256-keyed `.vpdq` cache) / `symmetricSharedFraction()` (the stored-value duplicate matcher) / `anySampledFrameMatch()` (the cheap prefilter before the full compare).

## 8. The calc engine — the dedicated duplicate-detection scan (LOCKED)

A **dedicated scan** — its own pass, not a rider on the 15-minute or 4-hour scheduleTasks — run
**on demand from the pop-up** (§5). It computes **duplicate groups only**; subset relations have
their **own** scan and engine ([subsets.mdx §8](./subsets.mdx)).

### 8.1 Candidate set

All **video and image files** LFB knows on this computer — every scanned unit (repos and the
computer pin unit, [scan.mdx §4](./scan.mdx)), videos first, images second
([videos.mdx §2](./videos.mdx)). The engine reuses hashes and fingerprints already recorded in the
per-file sidecars ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)) and computes only what
is missing — a warm re-scan is cheap. **"Knows" must mean "as of now", not "as of the last repo
scan"** — see §8.4, which is a hard requirement, not an optimization.

### 8.2 Grouping — exact first, then perceptual (LOCKED)

1. **Pass 1 — SHA-256.** Bucket by exact content hash. Any bucket with ≥ 2 files is a duplicate
   group with `match_basis: sha256`. Byte-identical copies never need the fingerprint.
2. **Pass 2 — perceptual fingerprint.** For files not already grouped byte-identically: compute
   any missing/stale fingerprints (§7.5–§7.6), then compare **stored values only** (§7.8) —
   images by PDQ Hamming distance within the strict de-dup threshold + quality gate, videos by
   the symmetric vPDQ shared-frame fraction. Connected components of matches become groups with
   `match_basis: fingerprint`. **This is the pass that catches copies differing in codec, file
   size, and pixel depth / resolution** — the transform-robust identity is exactly what the
   fingerprint exists for.
3. **Attribute probe.** For every grouped member, probe the display attributes (size, duration,
   width×height, codec) via the bundled ffprobe/sharp — all local, no network (charter).
4. **Write** `duplicates.csv` atomically (temp + rename), stamp the run timestamp for the 4-day
   staleness check, bump the live-refresh topic.

**Signal only.** The engine groups and records; it never deletes, replaces, or alters a file
(charter + [perceptual_fingerprint.mdx §4](./perceptual_fingerprint.mdx)).

The engine is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/dedupe.service.ts` — `runDedupeScan()` (the phases of §8.3) with the pure, unit-tested grouping split into `groupByExactHash()` (pass 1) and `groupByFingerprint()` (pass 2), composed by `computeDuplicateGroups()`; candidate enumeration in `known-media.ts` — `collectKnownMedia()`; warm-re-scan hash/fingerprint caches in `hash-cache.ts` — `VideosCaches`.

### 8.3 The run is PROGRESSIVE — hash first, publish, then fingerprint (LOCKED)

The two passes of §8.2 have **wildly different costs**. Pass 1 is a streamed read per file. Pass 2
shells out to ffmpeg for **every video** to sample frames. On a real machine (~26,500 candidates)
pass 1 is minutes and pass 2 is hours.

**The engine must therefore publish at every phase boundary, never once at the end.**

> **THE BUG THIS SECTION EXISTS TO PREVENT.** The first implementation hashed *and* fingerprinted
> every candidate before writing a single row. Three separate scans ran on the author's machine;
> every one was killed mid-run by an ordinary backend restart; **`duplicates.csv` was never created
> at all**. A user who copied one image onto another and pressed **Start scan** got nothing back —
> not because hash matching failed, but because the instant answer was held hostage by a pass that
> never finished. A fast, correct result that is never published is indistinguishable from a bug.

The phases:

| Phase | Work | Publishes |
| --- | --- | --- |
| **0 — candidates** | Enumerate + freshen the candidate set (§8.4). | nothing |
| **1 — hash** | Full-content SHA-256 for **every** candidate (streamed, cached, checkpointed), bucketed by exact hash. Display attributes probed for **grouped members only**. | **`duplicates.csv` with every byte-identical group**, a **partial** run stamp, live-refresh bump |
| **2 — fingerprint** | Perceptual fingerprints for everything pass 1 did not group, then the stored-value compare (§7.8). | republished **periodically** during the phase, then finally, with the run stamp **closed as complete** |

Rules:

* **Phase 1 output is never delayed by phase 2.** Byte-identical duplicates — the overwhelmingly
  common case, and the only one a plain file copy produces — are on screen while phase 2 runs.
* **Attributes are probed only for files that actually made a group.** Probing all candidates would
  spawn an ffprobe per video for rows nobody will ever see; the probe is a display concern, not a
  grouping input (the one exception is video duration, which phase 2 needs for its own gate).
* **Fingerprints for byte-identical groups cost one computation per group, not one per file.** The
  fingerprint caches are keyed by **`sha256`**, and byte-identical files share a `sha256` **by
  definition** — so phase 2 fingerprints **one representative** per pass-1 group and the whole group
  reads the same stored value. This is what keeps the `fingerprint` column populated for
  `match_basis: sha256` rows without paying for it N times.
* **The UI reflects the phase.** `GET /api/videos/duplicates/status` carries `phase`
  (`candidates` · `hash` · `fingerprint` · `idle`) and a `phaseDone`/`phaseTotal` counter; the page's
  running banner names the phase and says results appear **as they are found**, never "when it
  completes".
* **Batch items settle in phase 1**, because the fast pass is the one pass **every** candidate goes
  through — so the Processing counter shows real, monotonic progress instead of sitting at `0/26547`
  for hours. Phase-2 work reports through the progress registry and the `phase` field.

The phased run is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/dedupe.service.ts` — `runDedupeScan()` (phase 0/1/2 in sequence), `hashOneFile()` (phase 1 per file), `fingerprintOneFile()` (phase 2 per file), `publish()` (the CSV write + live-refresh bump used at every boundary), `probeAttrsFor()` (grouped members only).
The phase is surfaced to the page in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/videos/VideosReviewPage.tsx` — `phaseLabel()` (the running banner's phase clause).

### 8.4 Candidate freshness — the scan must see files created since the last repo scan (LOCKED)

The persisted per-unit rows are the **last repo scan's census**. A file created after that scan is
**absent from them**. Reusing them alone means the Start-scan button promises a fresh answer and
silently computes over stale inputs.

> **THE BUG THIS SECTION EXISTS TO PREVENT.** The author copied `2078856527541362697_3.jpg` to
> `mr_dup.jpg` in a tracked repo and pressed **Start scan**. The engine enumerated **26,547**
> candidates — and `mr_dup.jpg` was **not one of them**, because no repo scan had run in the two
> minutes since the copy. The engine only ever held **one** of the two files, so no amount of correct
> hashing could have produced a group. Duplicate detection is a **pairwise** property: a candidate
> set missing one member of a pair cannot report that pair, and reports nothing rather than an error.

Therefore, **phase 0 unions the persisted census with a live media sweep of the same unit roots**:

* The sweep is **media-extension-only** — no `git check-ignore`, no sidecar reads (the two dominant
  per-file costs of a real scan), so it is a small fraction of one. Measured on the author's machine:
  **+8 s across 184 roots** on top of an 11 s enumeration.
* It is **async and cooperatively yielding** (the shared walk collector), so it never freezes the
  event loop, and it honours the same skip rules as every other walk — `HARD_SKIP`, dot-directories,
  macOS bundles-are-opaque, plus our own `.lfbridge` / `.transcribe` artifact homes.
* Files the persisted rows already know **keep their richer row** (decision, git-ignore, task icon
  state). Newly discovered files get **default icon state** — they are real candidates that have not
  been scanned yet, and a blank icon is honest.
* The duplicate scan **does not trigger a repo scan** and does not wait on one. It is a *reader* that
  refuses to read a stale list; ownership of the census stays with [scan.mdx](./scan.mdx).

Candidate freshening is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/known-media.ts` — `collectKnownMedia(kinds, { freshen: true })` → `freshenFromDisk()` (the media-only sweep, unioned with the persisted rows).

### 8.5 Restart resilience — checkpointing (LOCKED)

A duplicate scan is a **long** job on a **developer machine**: `tsx watch` restarts, `just run`,
reboots, and crashes are ordinary events, not exotic ones. **A run that is interrupted must keep the
work it already paid for.**

> **THE BUG THIS SECTION EXISTS TO PREVENT.** The caches were saved **once, after every candidate had
> been hashed and fingerprinted**. Every interrupted run therefore discarded **100%** of its hashing.
> After three full scans on the author's machine, neither `hash_cache.yaml` nor `image_fp.yaml` had
> ever been written — so no run was ever warm and each one restarted from zero, guaranteeing that a
> run could never finish on a machine that restarts every few minutes.

Rules:

* **Caches checkpoint on a timer** (30 s) throughout both phases, not only at the end. An interrupted
  run is then a partial **win**: the next run reuses every hash it already computed.
* **The run stamp records completeness.** A phase-1 publish writes `complete: false`; only the final
  publish writes `complete: true`. Staleness (§5) recommends a rescan when the last run is stale
  **or did not complete** — so partial results render, and the pop-up still asks the user to finish
  the job. A stamp with no `complete` field reads as complete (it predates progressive publishing,
  and the old writer only ever stamped at the very end).
* **The CSV is still rewritten whole** at every publish (§9) — progressive publishing changes *how
  often* we write, never the atomic whole-file contract.

Checkpointing is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/hash-cache.ts` — `VideosCaches.maybeSave()` (throttled checkpoint, called per file in both phases) beside the end-of-phase `save()`; the completeness flag is `dedupe-store.ts` — `VideosRunStamp.complete` written by `writeDedupeRunStamp()` and read by `dedupeStatus()`.

### 8.6 Hash comparison — the hash alone, deliberately (LOCKED)

Pass 1 is a **single O(N) hash-map build**: bucket by `sha256`, emit buckets of ≥ 2. It is already
optimal, and the following tempting "optimizations" are **explicitly rejected**:

* **Do NOT pre-bucket or pre-filter by file size.** The intuition ("hashes can't match if sizes
  differ, so compare sizes first") is a **pairwise-comparison** optimization, and we never compare
  hashes pairwise — we bucket them. A size index could only ever add a second index to reach the same
  answer more slowly. Worse, it introduces a **failure mode**: size is reported by the filesystem and
  can differ from content length across mounts, sparse files, cloud-provider placeholders, and
  compressed/deduplicating filesystems. **A size mismatch must never be allowed to suppress a hash
  match.** The hash is the identity; nothing gates it.
* **Do NOT short-circuit on a partial/head-tail hash.** The tracking scheme's cheap
  size+mtime+head/tail fingerprint is a *change detector*, not a content identity: two byte-identical
  copies have different mtimes. Pass 1 needs **full-content** SHA-256, streamed.
* **Do NOT skip hashing a file because it has no fingerprint.** Hash grouping is independent of, and
  runs before, fingerprinting.
* **An empty/absent hash is never an identity.** Files whose hash failed must not collapse into one
  giant "same hash" group — they are excluded from bucketing entirely.

The legitimate hash optimizations are the ones we do use: **cache by path + size + mtime** so an
unchanged file is never re-streamed (§7.6), **checkpoint** that cache (§8.5), and **stream** rather
than read whole files into the heap.

Pass 1 is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/dedupe.service.ts` — `groupByExactHash()` (the O(N) bucket; no size gate anywhere); the streamed hash with its size+mtime cache is `exec.ts` — `sha256File()` and `hash-cache.ts` — `VideosCaches.sha256()`.

### 8.7 Scaling the fingerprint compare — banding, not brute force (LOCKED)

Pass 2 compares **stored fingerprints pairwise**, which is O(N²) if written naively. At ~26,500
candidates that is **~340 million** synchronous comparisons, each re-parsing two 64-character hex
strings — **minutes with the Node event loop fully blocked**, i.e. the whole app unresponsive.

The compare must scale, and it must scale **without ever missing a real match**:

* **LSH banding (images) — a pre-filter with NO false negatives.** Split each 256-bit (64-hex)
  fingerprint into **32 bands of 8 bits**. Two fingerprints within the strict de-dup threshold differ
  in at most **24** bits, so at most 24 bands can contain a differing bit and **at least 8 bands must
  be byte-identical**. Comparing only pairs that share ≥ 1 identical band therefore **cannot miss a
  match** — it only skips pairs that could not possibly have matched. This is exact, not approximate.
  * The guarantee requires `bandCount > threshold`. A fingerprint too short for that (a 64-bit
    fallback hash) falls back to **full pairwise within its own length bucket** rather than risk a miss.
  * A pair is compared in the **lowest band it collides in**, so a pair sharing all 32 bands costs
    **one** comparison, not 32.
* **Duration windowing (videos).** Equal-ish length is already a *requirement* of being a duplicate
  (a much shorter same-content file is a **subset**, not a duplicate). Sorting by duration and
  sweeping a window prunes exactly the pairs the pairwise loop would have rejected on its first test.
  Videos with **unknown** duration keep full pairwise treatment — precisely as the per-pair guard did.
* **Early-abort Hamming on pre-parsed nibbles.** Fingerprints are parsed to a byte array **once**;
  the pair compare walks it and returns the moment the running distance passes the threshold. Most
  pairs exit after a handful of nibbles.
* **Union-find is consulted first.** A pair already in the same component is skipped without any
  comparison.
* **The thresholds and the quality gate stay owned by
  [perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx).** Everything above is a *pre-filter*;
  `sameContent()` still makes every final, authoritative call.

Measured effect on the unit corpus (4,000 unrelated images + one planted matching pair): **115 s →
under 1 s**, same answer.

The scaled compare is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/dedupe.service.ts` — `unionImagesByFingerprint()` (banding + lowest-band rule + union-find skip), `withinThreshold()` / `toNibbles()` (the early-abort Hamming pre-filter), `videoCandidatePairs()` (duration windowing); the authoritative verdict stays `media/perceptual.service.ts` — `sameContent()` with the exported `IMAGE_THRESHOLD_STRICT` the banding math depends on.

### 8.8 Engine logging (LOCKED)

The engine is a long background job with no synchronous caller, so **the log is the only way to know
what it did**. It must log, at INFO, on every run:

1. **`duplicate calc engine STARTED`** — before any work.
2. The **candidate count**, and the **freshen result** (how many files the sweep found that the last
   repo scan did not know) — the two numbers that tell you whether the scan could possibly have seen
   your file.
3. **Each phase start and finish**, with elapsed seconds, the groups/rows **published**, and how many
   files carry into the next phase.
4. **Periodic phase-2 progress** at each interim republish.
5. **`duplicate calc engine FINISHED`** — with the group split (`sha256` vs `fingerprint`), member row
   count, candidate count, and total elapsed.

A run that ends **without** a FINISHED line was killed — which is itself the diagnosis, and is why the
partial run stamp (§8.5) exists. Per-file fingerprint failures stay **WARN** and never fail the run
(§7.6: a fingerprint failure loses a signal, never the item).

Logging goes to the state root's `log.log` / `error.err`
([storage_local.mdx](./storage_local.mdx)); nothing is written to `/tmp`.

## 10. Acceptance criteria

1. **Left bar:** Duplicates is the first level-2 child under Videos, route `/videos/duplicates`,
   yaml-driven ([videos.mdx §1](./videos.mdx)); it is where selecting Videos lands (§1).
2. **Table — ONE ROW PER GROUP:** left 60% of the content width; TanStack, 500-page,
   full-page-height. Every group is **exactly one row**, labelled with **one of its file names**;
   there are **no group-header rows and no per-member rows**. Columns are exactly **File (name
   without path)** + **Files (count)** + **Size (reclaimable bytes)** — **no leading checkbox**, no
   bookmark, no path column, and **no icon control columns** (they moved to the right column, §4.3).
   The **control row spans the full page width above the split**, and the right review column starts
   **underneath** it (§3, [tables.mdx §2.13](./tables.mdx)).
   * *Acceptance test:* a nine-file group occupies **one** table line, not ten.
3. **Search reaches every member (§3.2):** typing the name of ANY file in a group finds that group,
   including the members whose name the row does not display.
4. **Selection & hover (§3.3):** clicking a row **selects** its group — light-yellow row, sticky.
   Hovering a row shows **that** group in the right column — light-green row — and hover **outranks**
   the selection while the pointer is on the table. Moving the pointer **off the table** falls back to
   the selected group. The **first** group is selected on arrival and the seed fires **once**, so
   **Done** actually empties the column instead of being instantly undone. ↑/↓ moves the selection.
5. **Right column:** the reserved right 40%, scrolling **independently**; top action-links row
   holding exactly **"Done"** for now; then the **"All files are the same:"** block listing the
   attributes common to every member (file size, SHA-256 hash, fingerprint, duration for videos,
   resolution, codec) comma-space separated; then **one file block per member** — the **five icon
   controls (Pin · Ignore · Transcribe · AI description · OCR) to the LEFT of the file name**, the
   full path beneath, the preview at **full column width with aspect ratio kept**, and that member's
   **differing** attributes comma-separated beneath it. An attribute appears in the common block or
   the differing lines, never both (§4).
6. **Previews actually load, at any group size (§4.3a):** each block loads ONE cached JPEG from
   `GET /api/media/poster` (same signed grant as `/raw`) — **never** N live media streams. Videos show
   the poster with a ▶ and **only mount a player on the click**; an undecodable codec retries through
   `/api/media/stream`; only after that does the block say **"Preview unavailable"** with a **try
   again** link. HEIC/TIFF/ProRes/HEVC render, because the backend decodes them.
   * *Acceptance test:* select the largest group (nine 4 GB 4K masters) — **all nine** previews
     appear, and the page's other requests are not blocked.
7. **Print rules:** resolution as `{w}x{h}` with a parenthesized class **only** for the known
   classes 480p/720p/1080p/2K/4K (§4.4); codec as `codec (h264)` — the value in parentheses (§4.5).
8. **Action-links row (§5.1):** every visit shows the house action row under the page title with
   **exactly one link — Scan for duplicates**. Refresh results and View progress are **removed**
   (the page self-refreshes; the toast, the banner, and the conditional nav item are the routes to
   progress). Scan starts with **no** confirmation dialog; while a scan runs it is **disabled with a
   stated reason**; the row never hides, never moves, and takes the **highest z-order** so nothing
   below it can clip the links.
9. **Pop-up (§5.2/§5.3):** the in-app **"Start Duplicate Scan"** modal auto-opens **at most once per
   page entry**, and **only** when never-scanned, or 2+ days old and incomplete, or 4+ days stale —
   **never** when the last run was **within 2 days** (complete *or* partial), and never while a scan
   runs. Body = status line ("Never scanned." / "Last scanned {timestamp} ({relative}).") +
   recommendation + the background-work line; **Skip** hyperlink to the **left** of the dominant
   **"Start scan ›"** button. Never a native dialog.
   * *Acceptance test:* scan, then reload the page an hour later — **no pop-up**, even when the run
     was killed mid-phase-2 and stamped `complete: false`.
10. **Fire-and-forget start (§5.4):** clicking **Start scan ›** (or the row's **Scan for
    duplicates**) closes the modal **on the click**, raises the bottom toast *"Duplicate scan
    started — it runs in the background."* with a **View progress** action into `/processing`, and
    flips the page to its running state immediately. The page never blocks, dims, or spins. The
    server marks single-flight **synchronously** and **defers the run past the response flush**, so
    no enumeration or walk happens on the request's stack. A coalesced start (`started: false`) is an
    info toast, not an error.
11. **Progress:** Start scan enqueues an async `dedupe_scan` batch tracked by the standard
    Processing surfaces (page, dock, conditional nav item); detached and single-flight (§6).
12. **Fingerprint libraries (LOCKED):** every library used satisfies the four requirements —
   open-source & embeddable, **text-encodable fingerprint** storable in the CSV column, a compare
   function that takes **stored values** back in, **no phone-home**. For duplicate detection the
   choices are: **images → PDQ 256-bit / 64-hex** (fallback blockhash-core + sharp-phash, both
   MIT pure-JS), **videos → vPDQ** per-frame text lines; PhotoDNA, TMK+PDQF, GPL pHash bindings,
   and whole-video hashes are rejected for the reasons in §7.2–§7.3. Fingerprints are computed
   only during the dedicated scan for missing/stale files (§7.6), stored per §7.7, and all later
   comparisons are pure functions over stored values (§7.8).
13. **Engine:** the dedicated scan groups by **SHA-256 first, perceptual fingerprint second**, so
    copies differing in codec, file size, or pixel depth still group; probes display attributes;
    never mutates a file; computes **duplicates only** — subsets have their own scan
    ([subsets.mdx](./subsets.mdx)) (§8).
14. **Progressive publish (§8.3):** phase 1 writes `duplicates.csv` with every byte-identical group
    **before** phase 2 begins, and phase 2 republishes periodically. A scan killed during phase 2
    still leaves a usable CSV and a run stamp marked `complete: false`. The status route reports
    `phase` + `phaseDone`/`phaseTotal`, and the page's running banner names the phase and promises
    results **as they are found**.
    * *Acceptance test:* copy a file onto a new name in a tracked repo, press **Start scan**, and the
      pair appears as a `match_basis: sha256` group **without waiting for any fingerprinting** —
      verified end-to-end on 26,548 candidates: phase 1 published 3,013 groups / 9,057 rows in 90 s,
      including the planted pair.
15. **Candidate freshness (§8.4):** the candidate set is the persisted census **unioned with a live
    media sweep**, so a file created since the last repo scan is a candidate. A scan whose candidate
    set can only see one member of a duplicate pair is a defect, not a "no duplicates" answer.
16. **Restart resilience (§8.5):** hash/fingerprint caches checkpoint on a timer during both phases,
    so an interrupted run keeps the work it paid for; the run stamp distinguishes a **partial** run
    from a **complete** one and `recommend` stays true until one completes — **without** overriding
    the §5.2 quiet window.
17. **Hash comparison (§8.6):** pass 1 buckets on the hash **alone** — no size pre-filter, no partial
    hash, no fingerprint precondition — and an empty hash is never an identity.
18. **Compare scaling (§8.7):** the fingerprint compare uses **exact** LSH banding (no false
    negatives, proven by the band-count > threshold argument), lowest-band pair selection, duration
    windowing for videos, and an early-abort Hamming over pre-parsed nibbles. Thresholds and the
    quality gate remain owned by `sameContent()`.
19. **Engine logging (§8.8):** every run logs STARTED, the candidate + freshen counts, each phase's
    start/finish with what it published, interim phase-2 progress, and FINISHED with the group split.
    A run with no FINISHED line was killed — and that is the diagnosis.
20. **CSV:** `~/T/_large_files_bridge/videos/duplicates.csv`, one row per member, groups keyed by
    a shared unique `duplicate_group`, with `full_path`, `sha256`, `fingerprint` (text-encoded per
    §7.7), and the display attributes; rewritten atomically per run; the UI reads only this file
    (§9).
