# LargeFileBridge — Decision Toggles

Every large file the user can act on carries **two independent decisions** — **Add to IPFS (pin)** and
**Add to git ignore** — each on/off separately ([decisions.mdx §1](./decisions.mdx#1-two-independent-axes--not-one-mutually-exclusive-choice-locked)).
This spec defines the **one small inline control that renders each axis directly on the row**, so the
user can flip either decision in place — in the Repos file list, in the File System columns, and inside
the warning / To-Do batch popups — without opening a dialog.

There are **two toggles**, one per axis, sitting **side by side** on the row:

| # | Toggle | Axis it writes | Glyph | Shown when |
| --- | --- | --- | --- | --- |
| 1 | **Add-to-IPFS checkbox** (the *pin* toggle) | Add to IPFS — the `sync` decision value ([one_repo.mdx §1](./one_repo.mdx#1-the-decision-model-locked)) | the **pin** mark (the same pin used by the `P` badge, [directories.mdx §3.2](./directories.mdx#32-the-pin-badge-p--bright-pink-files-only)) | always (any large file) |
| 2 | **Add-to-git-ignore checkbox** (the *ignore* toggle) | Add to git ignore ([git_ignore.mdx](./git_ignore.mdx)) | a **circle with a cross through it** — **⊘** | **only when the file/dir is under a Git repo** (§4) |

> **Naming.** In dictation these were called "the IPFS checkbox" and "the git-ignore toggle." The
> product truth is simpler: they are the **two axes of one decision**, drawn as two checkboxes. This
> spec calls them the **Add-to-IPFS checkbox** and the **Add-to-git-ignore checkbox**; together they are
> the **decision toggles**.

They are **not** the same thing as the row's leading **☐ select** checkbox (which picks the row for a
bulk action, [one_repo.mdx §4.3](./one_repo.mdx)) — the decision toggles **write a decision** the moment
you click them.

Siblings this builds on and does not repeat:

* [decisions.mdx](./decisions.mdx) — the **two-axis, team-shared, attributed** ledger these toggles
  write (who / when / SID). A click here is one **decision event** appended there.
* [git_ignore.mdx](./git_ignore.mdx) — the `.gitignore`-writing engine the **ignore** axis drives, and
  the pop-over the *bulk* / directory case still opens.
* [one_repo.mdx](./one_repo.mdx) — the per-repo file table these toggles are added to (§5 here).
* [directories.mdx](./directories.mdx) — the File System column browser + **code badges**; the toggles
  are the *editable* counterpart of the read-only `P` (pin) and `I` (git-ignored) badges (§6 here).
* [warnings.mdx](./warnings.mdx) / [to_do.mdx](./to_do.mdx) — the popups whose subject list renders
  these toggles per row (§7 here).
* [Task tabs](./task_tabs.mdx) — the per-repo screen's Transcribe/Compress **status icons**, which reuse
  this toggle's box grammar (§9 here) but are *status* indicators, not two-axis decisions.

## 1. One visual grammar, FOUR render states (LOCKED)

Both toggles share **exactly one** look, so the user learns it once. A toggle is a small (16×16)
rounded square holding its glyph, in one of **four** states:

| State | Fill | Edge | Glyph | Means |
| --- | --- | --- | --- | --- |
| **OFF** | **white** | **thin (1px) medium-grey** line (`--lfb-toggle-off-edge` `#9ca3af`) | glyph in medium-grey | Decision is **off** — this axis is *not* applied (not pinned / not git-ignored). |
| **ON** | **medium-dark orange** (`--lfb-decision-on` `#c2410c`), **no edge** | *none* | glyph in **white** | Decision is **on** — this axis **is** applied (pinned / git-ignored). |
| **RECOMMENDED** | white | **dashed** orange (`--lfb-decision-on`, 1px dashed) | glyph in orange | A **calc engine suggests turning this on**, but the user has **not decided** yet ([to_do_batch_calc_engine.mdx](./to_do_batch_calc_engine.mdx)). This is the visual for **Undecided-but-recommended**. |
| **N/A** | *(not rendered — no box at all)* | — | — | The axis does not apply here (e.g. the git-ignore toggle on a file that is **not** under a repo, §4). |

The **RECOMMENDED** state is what makes the toggles teach: after a scan, a file the To-Do Batch Calc
Engine thinks should be git-ignored shows a **dashed-orange** ⊘ before the user has touched it. One click
fills it solid (ON) — confirming the suggestion. Clicking a solid (ON) toggle returns it to **OFF**
(white/grey), an explicit *no*.

> **Two axes → four combinations.** Because each toggle is independently OFF/ON, a single file shows one
> of **four** combinations at a glance: pin-off/ignore-off · pin-on/ignore-off · pin-off/ignore-on ·
> pin-on/ignore-on. The single most common LFBridge decision — *"pin it AND git-ignore it"* — is both
> toggles filled ON.

> **Color reconciliation with the icon-column block (LOCKED).** When these two toggles render as the first
> two of the **five leading icon control columns** ([tables.mdx §4c](./tables.mdx)) — which is now the
> default on every file table — they render through the **shared `StatusActionIcon`**, whose **ON/"done"
> fill is the column's UNIQUE color** so the five columns are distinguishable side by side
> ([tables.mdx §4c.5](./tables.mdx)): the **Pin** toggle's ON fill is **blue** (`--lfb-pin` `#1d4ed8`) and
> the **Ignore** toggle's ON fill is **orange** (`--lfb-decision-on` `#c2410c`). Historically *both*
> toggles used the one orange (`--lfb-decision-on`); Pin moved to blue only so it no longer reads the same
> as Ignore now that the two sit adjacent in the fixed icon block (and Transcribe moved blue→indigo for the
> same reason). The **RECOMMENDED** (dashed "could") edge and the **N/A** hairline are unchanged and
> **identical across all five** columns; only the filled ON swatch differs per axis.

The token set and the `<DecisionToggle>` render states are defined in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/decision/DecisionToggles.tsx` —
`DecisionToggle` (renders one axis) and `DecisionToggles` (renders the pair); tokens in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/index.css` (`--lfb-toggle-off-edge`, `--lfb-decision-on`).

### 1.0 Each toggle shows REALITY, not the recorded intent (LOCKED)

A toggle reports **the state of the world**, not what our ledger remembers deciding. The two axes
therefore read from **different** sources, and the ignore axis in particular must **never** be sourced
from the decision ledger:

| Axis | ON is read from | Why not the ledger |
| --- | --- | --- |
| **Add-to-git-ignore (⊘)** | **Git itself** — `git check-ignore`, one batched call per repo | The ledger only holds files **we** ignored via our own toggle. A rule the user wrote by hand, or any **pattern** rule (e.g. `**/videos/**`), produces **no ledger event** — so a ledger-sourced toggle renders **OFF for a file git genuinely ignores**, which is a lie the user can see. |
| **Add-to-IPFS** | the manifest / pin state | (unchanged) |

The ledger keeps its role as **provenance** — `decided_by` / `decided_at`, i.e. *who chose this and
when* — and as the travelling team record. It is **not** the display source for the ignore toggle.

> **The invariant:** if `git check-ignore` says a file is ignored, the ⊘ toggle is **solid** — whatever
> the ledger does or does not contain. Reality wins over intent, every time.

Deriving the git-ignore axis from git for every file row is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` —
`composeFileRows()` (one batched `checkIgnore()` per repo), using
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `checkIgnore()`,
the same helper (and therefore the same definition of *ignored*) the File-System badges use.

### 1.1 Sizing, hit-target, and pairing

* Each toggle is **16×16** with an invisible **≥24×24** hit target so it is clickable on a dense table
  row and on touch.
* The pair renders **pin then ignore**, left to right, with a 6px gap, in a single non-wrapping cell.
* When the ignore axis is **N/A** (§4) the pair still reserves the ignore slot's width so columns stay
  aligned across rows (repo rows and non-repo rows line up); the slot is simply **blank**.

### 1.2 It is a control cell — it never navigates (LOCKED)

Wherever a row is also a deep link (the Repos file table, the File System columns), the toggle **stops
click propagation**: clicking a toggle flips that decision and does **nothing else** — it never opens the
file, never selects the row. This mirrors the existing rule for the inline **Decision** select and the
**CID** copy cell ([one_repo.mdx §4.7](./one_repo.mdx#47-row--cell-navigation-per-file--locked)).

### 1.2.1 LOCKED is not DISABLED (LOCKED)

A toggle can be **ON and true, yet not ours to change from here**. The case that matters today: git
ignores a file via a rule LFBridge must not rewrite — a pattern like `**/videos/**`, or a rule in
`.git/info/exclude` ([git_ignore.mdx §5.5](./git_ignore.mdx)). Such a toggle renders **LOCKED**:

| | Fill / edge | Opacity | Click | Tooltip |
| --- | --- | --- | --- | --- |
| **LOCKED** | **exactly the ON look** — solid orange, no edge | **full** — *never dimmed* | none (no hover affordance) | **names the rule**: *"Git-ignored by `.gitignore:2` — `**/videos/**`. That rule covers more than this file, so Large File Bridge will not rewrite it. Edit `.gitignore` to change this."* |
| **DISABLED** | its normal state look | dimmed (40%) | none | why the control is unavailable |

> **Why full opacity:** the file **really is** git-ignored. Dimming it would read as *"not really on"* and
> undo the whole point of showing real state ([§1.0](#10-each-toggle-shows-reality-not-the-recorded-intent-locked)).
> LOCKED changes only **what a click can do**, never **what is true**.

The rule is: **never render a click that would silently do nothing.** If OFF cannot be performed, the
toggle must say why. `FileRow.gitignoreLocked` + `FileRow.gitignoreRule` carry this per row.

The LOCKED-vs-DISABLED render split is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/decision/DecisionToggles.tsx` —
`DecisionToggle` (the `locked` prop: full opacity, `cursor-default`, `aria-disabled`, explanatory `title`).

## 2. Clicking a toggle = one decision event (LOCKED)

A click is **optimistic** and **immediate**:

1. The toggle flips to the new state instantly (ON⇄OFF; a RECOMMENDED toggle flips straight to ON).
2. LFBridge appends **one decision event** for **that axis only** to the repo's team-shared decision ledger
   in Local Storage — `~/T/_large_files_bridge/repos/<repoKey>/decisions.yaml`
   ([decisions.mdx §4](./decisions.mdx#4-where-it-lives--local-storage-always-shared-via-the-sync-repo-locked)) —
   with `decided_by` (the signed-in email), `decided_at`, the Storage ID, the path, and the fingerprint
   ([decisions.mdx §3](./decisions.mdx#3-the-decision-record--every-field-and-why-locked)). It never touches
   the working repo, and travels to teammates via the sync repo when enabled. The other axis is untouched.
3. The axis's **effect** is carried out by that axis's engine:
   * **Add-to-IPFS ON** → the file is added to the manifest, its CID recorded, pinned, and queued to
     peers ([one_repo.mdx §4.5](./one_repo.mdx), [pin_process.mdx](./pin_process.mdx)). OFF → removed
     from the manifest; local bytes are **never** deleted.
   * **Add-to-git-ignore ON** → the `.gitignore` engine writes an anchored, repo-root-relative rule
     ([git_ignore.mdx §5](./git_ignore.mdx#5-the-gitignore-content-rules-locked-the-engine)); it is
     idempotent and append-only. **OFF** → the engine removes **the exact anchored single-file line**
     `/<rel>` and verifies with `git check-ignore` that the file really is un-ignored, rolling the line
     back if another rule still covers it ([git_ignore.mdx §5.5](./git_ignore.mdx)). If a **broad/pattern**
     rule ignores the file, OFF is **refused** and the toggle is **locked** (§1.2.1) — it is never a
     silent no-op.

     > **Superseded wording:** this used to read *"the corresponding rule is removed (only if LFBridge
     > added it)"*. That was never buildable — LFB appends **bare** lines with no marker or attribution,
     > so "did we add it?" is unknowable. The exact-anchored-line test replaces it and is *stronger*:
     > such a line matches exactly one file, so removing it is provably scoped to the clicked file no
     > matter who typed it.
4. On error the toggle **rolls back** to its prior state and a non-intrusive error note appears
   ([warnings.mdx §5.5](./warnings.mdx)).

A single click therefore **records intent** (the ledger) **and** performs the effect. Because the ledger
is team-shared and union-merged, a teammate who already decided this file's axis means this toggle
arrives **already ON/OFF**, not RECOMMENDED — nobody is re-asked ([decisions.mdx §5](./decisions.mdx#5-team-dynamics--the-merge-model-locked-think-team-of-ten)).

Writing the single-axis decision event from a toggle click reuses the one decision write path in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/decisions.service.ts` — `recordDecision(axis, …)`
(exposed by `PATCH /api/repos/:repoId/files`, [decisions.mdx §19](./decisions.mdx#19-the-write-api--one-endpoint-two-accepted-bodies-locked)).

### 2.1 Bulk clicks and directories still open the pop-over (LOCKED)

A toggle writes **one file, one axis**. Two cases route to a confirm dialog instead of an instant write:

* **A directory's ignore toggle** — turning ⊘ **on** for a *folder* under a repo can ignore many files at
  once, so it opens the **Git-ignore pop-over** ([git_ignore.mdx §3](./git_ignore.mdx#3-the-dialog-anatomy--adaptive-to-the-selection-shape-locked)) with the Recursive choice, rather
  than writing silently.
* **Selected rows** — flipping a decision across a multi-row selection uses the table's **bulk menu**
  ([one_repo.mdx §4.4](./one_repo.mdx#44-bulk-actions)), which confirms the count.

This preserves the charter rule: **never git-ignore or move files without an explicit, scoped click**
([menus.mdx §6.1](./menus.mdx)).

## 3. Where the toggles appear (LOCKED inventory)

The pair renders in **every place a file is listed for a decision**:

1. **The One-repo file table** — a **Pin** column and, to its right, a **Git-ignore** column, one toggle
   each (§5).
2. **Next to an existing pin** — anywhere a pinned file already shows its pin state, the ignore toggle
   sits beside it so both axes read together.
3. **The File System column browser** — the toggles are the editable form of the `P` / `I` badges on a
   file row, and appear on directory rows under a repo (§6).
4. **The File System flat Full-paths table** — same two columns as the repo table.
5. **Warning popups** — the subjects list of a file-scoped warning renders the toggles per row
   ([warnings.mdx §4.5](./warnings.mdx#45-the-subjects-list--the-actual-filesdirectories-with-checkboxes-locked)).
6. **To-Do batch popups** — the batch's file list renders the toggles per row, pre-set to the calc
   engine's **RECOMMENDED** target so the user confirms or overrides in place ([to_do.mdx §6](./to_do.mdx)).

The **read-only** badges (`P`, `I`) still exist for dense, non-interactive contexts; the toggles are for
**interactive rows** where the user is expected to decide.

## 4. The git-ignore toggle is REPO-SCOPED (LOCKED)

Git-ignore only means something **inside a Git repo**. Therefore the **Add-to-git-ignore** toggle:

* **Renders only when the file (or directory) is under a Git repo** — i.e. the row resolves to a repo
  root with a `.gitignore` LFBridge can write ([git_ignore.mdx §5.3](./git_ignore.mdx#53-which-gitignore-and-multiple-repos-locked)).
* Is **N/A (blank slot)** for files outside any repo — loose personal files, files inside a Dropbox or
  Google Drive area that is not itself a repo, etc. Those files still show the **Add-to-IPFS** toggle
  (pinning is not repo-scoped).

The **Add-to-IPFS** toggle has **no** such restriction — any large file can be pinned. So a loose personal
video shows **one** toggle (pin); a repo video shows **two** (pin + ignore).

Deciding whether a given path is under a repo (and which repo's `.gitignore` an ignore-toggle would write) is computed in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `repoRootFor(path)`
(null ⇒ the ignore toggle is N/A for that row).

## 5. In the One-repo file table (LOCKED)

The [One-repo](./one_repo.mdx) file table renders the two toggles as the **first two of the five leading
icon control columns** ([tables.mdx §4c](./tables.mdx)) — **Pin** then **Ignore**, immediately right of the
row-select checkbox, ahead of Transcribe · AI Description · OCR — each a narrow (`tight`/`px-1`) icon column
with an **icon-only header** and a **hover explanation** in the docked hover-info region
([tables.mdx §4c.2–§4c.3](./tables.mdx)). They render through the **shared `StatusActionIcon`** (Pin's ON
fill blue, Ignore's orange, §1) instead of any per-page toggle rendering, so Pin/Ignore speak the same
visual language as the three analysis icons beside them.

> **The labelled Decision select and the provenance columns are removed from file tables (LOCKED).** The
> icon block **fully replaces** the old spread-out decision UI: the inline `[Decision ▾]` / `[Pin ▾]` select
> **column**, the **Decided by** column, and the **Decided (when)** column are **gone from the file table**,
> as is the transfer **Status** column ([tables.mdx §4c](./tables.mdx)). The **Pin** and **Ignore** icon
> cells now carry both decision axes in full — a click sets that axis exactly as before (§2). **The two
> axes still exist as data** (the team-shared ledger, [decisions.mdx](./decisions.mdx)) and provenance still
> exists — *who/when* now surfaces **on hover** ([one_repo.mdx §4.8](./one_repo.mdx), the docked hover-info
> region and the file page) rather than as its own column. Only the *columns* are retired; no data or axis
> is lost.

The row still expresses the **two axes directly** as toggles:

```
☐  File                     Size    Pin   Ignore   Status    Peers  Changed
☐  render/final_4k.mov      8.2GB   ◉     ⊘●       Pinned    3      2h ago   ⋮
☐  render/take_07.mov       6.1GB   ◌     ⊘◌       —         0      5m ago ⚠ ⋮
☐  scratch/proxy.mp4        220MB   ◌     ⊘◌·      —         0      1d ago   ⋮   ← ⊘◌· = RECOMMENDED ignore
```

* **Pin** column = the **Add-to-IPFS** toggle (solid orange = pinned).
* **Ignore** column = the **Add-to-git-ignore** toggle (⊘; solid orange = git-ignored; dashed = the calc
  engine recommends ignoring it).
* **Each column carries its own axis's facet — two facets, never one** ([tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).
  Both columns are **sortable and filterable**, but they filter through **two separate facets** on the
  rail, each a set of **value checkboxes, all checked by default** (all-checked = inactive; no `any`
  pseudo-value), not one shared `on / off / recommended` list
  ([tables.mdx §2.1 / §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked)):

  | Column | Its facet | Values (each a checkbox, all checked by default) | Reads ON from |
  | --- | --- | --- | --- |
  | **Pin** | **Pin (IPFS)** | ☑ **Pinned** (`sync`) · ☑ **Not pinned** (`ignore` — deliberately not pinned) · ☑ **Recommended** · ☑ **Undecided** | the manifest / pin state (§1.0) |
  | **Ignore** | **Git-ignore** | ☑ **Git-ignored** · ☑ **Not git-ignored** · ☑ **Recommended** | **`git check-ignore`**, live (§1.0, LOCKED) |

  **The four-state toggle grammar (§1) is what the facet values are.** OFF / ON / RECOMMENDED map onto
  each facet's values one-for-one — the Pin facet splits ON/OFF into the axis's own words (`Pinned` /
  `Not pinned`, and its OFF-with-no-record case is `Undecided`), the Git-ignore facet into
  `Git-ignored` / `Not git-ignored`. So a chip on the rail and a box on a row say the same word. **N/A is
  not a facet value** ([tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked)): a row
  where the axis does not apply (§4) renders no box and is not a value anyone can filter for.
* **The two columns can never share a facet or a sort key**, because they read from **two different
  sources of truth** (§1.0): a merged facet would have to pick one source and lie about the other. Sort
  obeys the same split — a **Pin** key and an **Ignore** key, no combined "Decision" key, ordered
  **On → Recommended → Off → N/A** ([tables.mdx §3.6](./tables.mdx#36-sort-obeys-the-axis-split-too-locked)).
* **The labelled select column is removed** (see the boxed rule above): the Pin/Ignore icon cells are the
  only per-row decision surface in the file table now. The IPFS axis's **one facet and one sort key** still
  live on the **Pin** column, and the git-ignore axis's on the **Ignore** column — the retired select never
  carried a facet or sort key anyway ([tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).
  The stored `sync` / `ignore` / `undecided` values are untouched — **frozen wire values**
  ([decisions.mdx §7](./decisions.mdx#7-reconciliation--folding-the-shared-ledger-onto-this-computer-locked)) —
  the icons write them exactly as the select did. (Where a labelled one-of-N editor is still wanted — e.g.
  the file kebab's *Set pin ▸* submenu, [menus.mdx §5.3](./menus.mdx) — it stays available off-table; it is
  only the *column* that is gone.)

The Pin/Ignore icon columns (the first two of the five leading icon control columns, [tables.mdx §4c](./tables.mdx)), wired to the same `setDecision` mutation, are rendered in code through the shared `StatusActionIcon`:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage`
(the `pin` and `ignore` `LfbColumn`s, `tight` + icon-only `headerCell`, mounting `StatusActionIcon` via the `TaskStatusCell` kit).

Splitting these into **two** rail facets with the §2.4.1 vocabulary (the removed col-3c `<select>` used to carry the IPFS-axis filter) is a **new** surface; it is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` (`filterOptions` on the `pin` and `ignore` `LfbColumn`s, removed from the `decision` column), rendered by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` (the facet rail).

## 6. In the File System (LOCKED)

In the [File System](./directories.mdx) column browser and the Full-paths table:

* On a **file row under a repo**, the toggles render both axes; on a **file row outside any repo**, only
  the pin toggle renders (ignore is N/A, §4).
* On a **directory row whose subtree is under a repo**, the **ignore** toggle renders and — because a
  folder ignore is a many-file action — turning it on opens the Git-ignore pop-over (§2.1). A directory's
  **pin** toggle pins the directory's eligible large files (the same "pin all under here" scope rule as
  the page action, [directories.mdx §1.2](./directories.mdx#12-page-actions--the-action-links-row)).
* The toggles are the **editable** counterpart of the read-only `P` (pin) and `I` (git-ignored) code
  badges ([directories.mdx §3.2 / §3.4a](./directories.mdx#34a-the-git-ignored-badge-i--near-white-files-and-directories)); a badge shows *state*, a toggle *changes* it. When a
  toggle flips, the corresponding badge updates on the next render.

Adding the decision-toggle cell to File System file/directory rows (repo-scoped ignore, pop-over on directory ignore) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/` — the column-browser and Full-paths row renderers (mount `DecisionToggles`, gate the ignore axis on `repoRootFor`).

## 7. In the warning & To-Do batch popups — THREE axes per row (LOCKED)

The popup **subjects list** — the checklist of the actual files a warning or batch is about — renders the
decision toggles per row so the user decides **inside the popup**
([warnings.mdx §4.5.1](./warnings.mdx#451-the-per-row-action-toggles--up-to-three-checkboxes-per-file-locked)):

* **Up to THREE toggles per row here — not two.** The popup is the one place the **Compress** axis joins
  the pin + ignore pair, giving a row up to three action toggles left→right: **Add to IPFS (pin/get)**,
  **Add to git-ignore (add eventually)**, and **Compress**. It is the **same** `DecisionToggle`
  component and the same four-state grammar (§1) — the compress axis is a third axis id (`"compress"`,
  glyph **K**), rendered only when the file looks compressible & uncompressed. This third toggle is
  popup-only; the table rows (§5/§6) still show just the two decision axes (pin + ignore), because
  Compress is an *action*, not a stored two-axis decision.
* Each subject row shows: the file, its size, and its **action toggles** — **pre-checked (solid ON) where
  LFBridge recommends the action**, so the popup opens with the right boxes already ticked and the user
  only unchecks; a To-Do batch may instead open axes it is less sure of as **RECOMMENDED/dashed**.
* **Columns render only when relevant** (§1.1 alignment rule extends here): a column shows only if some
  row in the set has that axis applicable — a pull-down-only batch shows two columns, a compress-only
  batch shows one ([warnings.mdx §4.5.1](./warnings.mdx#451-the-per-row-action-toggles--up-to-three-checkboxes-per-file-locked)).
* The popup's live **count** ([warnings.mdx §4.6](./warnings.mdx#46-the-live-count-the-6-that-updates)) reflects how many rows still have **≥1 axis ON**;
  unchecking a row's last axis drops it out of the count.
* The popup's **Apply** button spells out the checked axes across the list with a **per-axis count** — its
  composed label follows [warnings.mdx §4.4.1](./warnings.mdx#441-composed-action-verb-labels--the-button-spells-out-the-checked-axes-locked)
  (e.g. *"Compress 12 · git-ignore 41 · Continue: IPFS Add — 41"*). Apply then writes each row's
  decision(s) — and queues each row's compress job — exactly as a single-axis toggle click does (§2).
* **Preview, not just a checklist.** Hovering or keyboard-focusing a media row previews that image/video
  full-size in the popup's left pane ([warnings.mdx §4.5.2](./warnings.mdx#452-the-left-pane-doubles-as-a-full-size-media-preview-locked)),
  so the user sees what they're deciding on before flipping a toggle.

This makes the popup and the row toggles **one model**: the same `DecisionToggle` component, the same
single-axis write, whether the user flips it on a table row or inside the popup — extended in the popup
with the third **Compress** action axis.

## 8. Acceptance criteria

1. Two inline toggles render per actionable file row — an **Add-to-IPFS (pin)** toggle and an
   **Add-to-git-ignore (⊘)** toggle — side by side, pin then ignore.
2. Both share **one** visual grammar with **four** states: **OFF** (white fill, thin medium-grey edge),
   **ON** (medium-dark-orange fill, no edge, white glyph), **RECOMMENDED** (white fill, **dashed** orange
   edge), and **N/A** (not rendered).
3. Clicking a toggle writes **one axis** of the **team-shared** decision immediately (optimistic, with
   rollback on error), appending a `decided_by`/`decided_at`/SID event to the Local-Storage ledger
   `~/T/_large_files_bridge/repos/<repoKey>/decisions.yaml` (never the working repo; travels via the sync
   repo when enabled), and carries out that axis's effect (pin/unpin or `.gitignore` write) — never
   deleting local bytes.
4. The **git-ignore** toggle renders **only** for files/dirs **under a repo**; it is a blank (aligned)
   N/A slot otherwise. The **pin** toggle renders for any large file.
5. A **directory** ignore toggle and a **multi-row** flip open the confirm surface (Git-ignore pop-over /
   bulk menu) instead of writing silently.
6. The toggles appear in the One-repo file table (Pin + Ignore columns), the File System columns and
   Full-paths table, and the warning / To-Do batch popup subject lists — everywhere a file is listed for
   a decision — and stop click propagation so they never navigate.
7. A **RECOMMENDED** (dashed) toggle reflects a calc-engine suggestion for an **Undecided** file; a
   teammate's prior decision arrives as **ON/OFF**, never RECOMMENDED (nobody is re-asked).
8. In the **warning / To-Do batch popups only**, a row shows **up to three** action toggles — pin,
   git-ignore, **and Compress** (`"compress"`, glyph **K**) — using the same component and four-state
   grammar; each column renders only when relevant to the set, axes are **pre-checked where recommended**,
   and the Apply label carries a **per-axis count** (§7). Table rows keep just the two decision axes.
9. **Each toggle column filters and sorts through its OWN axis** (§5): the **Pin** column carries the
   **Pin (IPFS)** facet and the **Ignore** column the **Git-ignore** facet — **two facets, never one**,
   each a value-checkbox set all-checked by default, with the LOCKED
   [tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked) vocabulary and **no facet
   named "Decision"**. The four-state grammar maps onto the facet values; **N/A is not a facet value**.
   Sort splits the same way (a **Pin** key and an **Ignore** key, never a combined one), ordered
   **On → Recommended → Off → N/A**.
10. **Pin + Ignore render as the first two of the five leading icon control columns** ([tables.mdx §4c](./tables.mdx),
    §5): narrow (`tight`/`px-1`), **icon-only headers**, hover explanation in the docked hover-info region,
    all through the shared **`StatusActionIcon`** with **unique ON colors — Pin blue (`--lfb-pin`), Ignore
    orange (`--lfb-decision-on`)** (§1 color reconciliation). The old **Decision** select column, the
    **Decided by** and **Decided** provenance columns, and the transfer **Status** column are **removed from
    the file table** — the icons carry both axes; the two axes still exist as data and provenance surfaces on
    hover.

* **The "done" fill is not orange.** A finished action reads in its own color — **blue** for a completed

  normal-thickness edge, 99%-white/1%-orange fill — the actionable state, mirroring **RECOMMENDED**'s
