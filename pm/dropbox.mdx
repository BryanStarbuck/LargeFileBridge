# LargeFileBridge — Dropbox Backbone

**Dropbox** is one of the three **bridge backbones** LargeFileBridge (LFB) uses to move the
**Sync List** between a user's own computers. The Sync List is the small human-readable YAML file
`LargeFilesBridge_SyncList.yaml`; Dropbox is a transport channel that already syncs a folder across
the user's machines, so LFB drops the list into that folder and lets Dropbox carry it. **A backbone
is "the road"; the Sync List is "the cargo."** This spec covers the Dropbox road only — it does not
redefine the cargo.

This file is a **sibling** of the other two backbone specs and a child of the hub:

* [Sync List (hub)](./sync_list.mdx) — THE hub. The `large_files` YAML hierarchy, the two scopes,
  the `origin`/`revision` provenance, and the [§5 trust/reconciliation model](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)
  are all defined there and are **not** restated here.
* [Google Drive](./google_drive.mdx) — the sibling whole-computer file backbone (same shape as this
  one, different vendor).
* [Repo (git)](./repo__list_syns.mdx) — the sibling backbone that carries each **per-repo** list across
  the user's computers over git (via the company sync repo, not the working repo —
  [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).
* [storage_local.mdx](./storage_local.mdx) — the on-disk state model (state root, per-unit `config.yaml` +
  `manifest.yaml`, `peers.yaml`). Dropbox is the **file-based alternative to
  [storage_local.mdx §9.3](./storage_local.mdx#93-computer-manifests-travel-via-ipns)'s IPNS transport** for the
  whole-computer list.
* [knowledge/ipfs.mdx](../knowledge/ipfs.mdx) — the only-our-content rule ([§6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule))
  the list's IPFS settings must honor.

> **The one Dropbox rule that shapes everything here.** **Only the small YAML Sync List goes into
> Dropbox — NEVER the large files themselves.** IPFS carries the bytes; putting big files in Dropbox
> would defeat the product's entire reason to exist (the whole point is that Git ignores the big
> files and IPFS moves them peer-to-peer). Dropbox carries **the list, and only the list.** This is
> **LOCKED**.

## 1. Scope — Dropbox carries the COMPUTER list

Dropbox is a **whole-computer backbone**: it syncs a folder that belongs to the user's Dropbox
account, not to any one repo. So it carries the **computer-scope** Sync List
([hub §2](./sync_list.mdx#2-two-scopes--one-per-computer-one-per-repo)) — the file whose
`large_files.scope` is `computer`, covering the global `settings`, every managed `repos:` entry, and
every non-repo `directories:` entry.

Walking the whole-computer roots — every managed repo unit plus the non-repo computer unit that the computer-scope list covers — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()`

It does **not** carry per-repo lists — those ride the **company sync repo** over git (not the working
repo — [artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
[repo__list_syns.mdx](./repo__list_syns.mdx)). When a repo appears both in the Dropbox-carried
computer list and in its own sync-repo list, the **sync-repo (git) list is authoritative for that repo's
files** ([hub §5.4](./sync_list.mdx#54-when-the-same-repo-arrives-on-two-backbones)); the Dropbox
copy is a convenience mirror.

| Question | Answer |
| --- | --- |
| Which scope? | `computer` only ([hub §2](./sync_list.mdx#2-two-scopes--one-per-computer-one-per-repo)). |
| What travels? | The YAML Sync List. **Never** the large files (LOCKED, above). |
| Alternative to? | [storage_local.mdx §9.3](./storage_local.mdx#93-computer-manifests-travel-via-ipns) IPNS transport. Both may be enabled; they reconcile through the same [hub §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule) rules. |

## 2. Path convention — one subfolder per computer (LOCKED)

Dropbox syncs a **single shared folder** into every machine, so if two computers wrote to one shared
file they would forever clobber each other and spawn conflict copies. LFB avoids that by giving
**each computer its own subfolder**, keyed by that computer's label:

```
~/Dropbox/LargeFileBridge/<computer_label>/LargeFilesBridge_SyncList.yaml
```

* A computer **WRITES only its own subfolder** (`<computer_label>` = `config.yaml → computer.label`,
  [storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)).
* A computer **READS every peer subfolder** and **reconciles each one** against its own reality
  ([hub §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)) — never applies any of
  them blindly.

```
~/Dropbox/                                    ← Dropbox root (located per §3)
└── LargeFileBridge/                          ← backbone root (settings.backbones.dropbox.root)
    ├── bryan-macbook/                         ← THIS machine writes ONLY here
    │   └── LargeFilesBridge_SyncList.yaml     ← our computer-scope list (scope: computer)
    ├── bryan-studio/                          ← a peer — READ ONLY, reconcile
    │   └── LargeFilesBridge_SyncList.yaml
    ├── work-mini/                             ← a peer — READ ONLY, reconcile
    │   └── LargeFilesBridge_SyncList.yaml
    └── family-imac/                           ← a peer — READ ONLY, reconcile
        └── LargeFilesBridge_SyncList.yaml
```

* The filename is **exactly** `LargeFilesBridge_SyncList.yaml` — "Files" is plural even though the
  product is "LargeFileBridge" ([hub §1](./sync_list.mdx#1-vocabulary--sync-list--backbone--scope-locked)).
  Honor it verbatim; the scanner matches on this name.
* The subfolder key is the peer's **label**, matching `peers.yaml`
  ([storage_local.mdx §11](./storage_local.mdx#11-peersyaml--the-users-other-computers)) — that is how a
  discovered file resolves to a known peer.
* Sanitize `<computer_label>` as a single path segment (same rule as the repo/email keys,
  [storage_local.mdx §6.1](./storage_local.mdx#61-keying--folder-name)): lowercase, any char outside
  `[A-Za-z0-9._-]` → `_`, reject `/`, `\`, `..`, NUL. A label collision across two machines is a
  configuration error surfaced in the web app — never silently merged.

## 3. Locating the Dropbox root

The backbone root is **not** hard-coded to `~/Dropbox`. Dropbox's folder location varies by account
type and OS setup: it may be `~/Dropbox`, `~/Dropbox (Personal)`, `~/Dropbox (Company)`, a
Business-account path, or a user-chosen location. LFB resolves it in order:

1. **Explicit override** — `large_files.settings.backbones.dropbox.root` (from the hub schema). If
   set, use it verbatim; do not probe further.
2. **`~/.dropbox/info.json`** — Dropbox writes this file with the **actual root path(s)** of each
   linked account. Parse it and read the `path` field of the relevant account (`personal` /
   `business`). This is the authoritative discovery source.
3. **Give up gracefully** — if neither yields a real directory, mark the Dropbox backbone
   `unavailable` in the web app (do not error the whole app), and skip it until it resolves.

```jsonc
// ~/.dropbox/info.json — Dropbox's own record of where each account's root lives
{
  "personal": { "path": "/Users/bryan/Dropbox (Personal)", "host": 123, "is_team": false },
  "business": { "path": "/Users/bryan/Dropbox (Company)",  "host": 456, "is_team": true  }
}
```

The resolved backbone root is then `<account root>/LargeFileBridge/` (or whatever
`settings.backbones.dropbox.root` overrides it to). Enable/disable is
`settings.backbones.dropbox.enabled` — when `false`, LFB neither reads nor writes Dropbox at all.

Resolving and validating a configured root path (tilde expansion, absolute-path resolution, and the existing-directory check that lets us mark a root unavailable rather than error) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/paths.ts` — `resolveDir()`

## 4. Dropbox-specific hazards (map each to hub §5.5)

Dropbox mangles files in vendor-specific ways. Each hazard below is a concrete instance of the
backbone-delivered hazards the reader must tolerate
([hub §5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect)) — quarantine or
reject, **never trust blindly**.

| Hazard | What it looks like | Rule |
| --- | --- | --- |
| **Conflicted copies** | `LargeFilesBridge (bryan's conflicted copy 2026-07-02).yaml`, `... (1).yaml` | Detect by filename pattern, treat as an **additional untrusted input**, reconcile it like any peer copy ([hub §5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect)) — never load blindly, never overwrite the canonical file with it. |
| **Online-only / Smart Sync placeholders** | A zero-content placeholder / `.dropbox` cloud-only stub; the bytes live in the cloud, not on disk | **Materialize (download) the file before reading it.** A placeholder is not "absent" and not "empty" — force a local download, then parse. Never treat a placeholder as a missing or zero-byte list. |
| **Selective-sync exclusions** | A peer subfolder the user excluded from this machine simply isn't present | This is a **spurious absence**, not a delete. Per [hub §5.2](./sync_list.mdx#52-absence-is-not-deletion-default), **absence ≠ delete** (`settings.conflict.absence_is_delete: false`). Record the peer as *not-currently-visible*, keep the last reconciled truth for it, do **not** drop its files. |
| **Dropbox metadata** | `.dropbox`, `.dropbox.cache/`, `.dropbox.attr` inside the tree | **Ignore entirely** — never scan, read, reconcile, or write these. They are Dropbox's own bookkeeping. |
| **Partial / zero-byte mid-sync files** | A file caught mid-download: truncated, zero bytes, invalid YAML | **Reject loudly and keep the previous good copy** ([storage_local.mdx §15](./storage_local.mdx#15-the-store-layer-how-code-touches-disk)). A file that fails to parse as the expected `large_files` schema is never trusted. |
| **Foreign owner** | A peer list whose `origin.owner` is not an allow-listed user | **Never adopt it** ([hub §5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect)). It is the user's own Dropbox, but only allow-listed owners' lists are reconciled. |

Skipping metadata / junk directories while walking a backbone subtree (the shared hard-skip set that keeps `.dropbox` / `.cache` and similar bookkeeping out of the walk) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/scan-filters.ts` — `HARD_SKIP`

## 5. Change detection — watch, then reconcile

LFB learns that a peer's list changed two ways, both feeding the same reconcile step:

* **fsevents watch (macOS), debounced.** Watch the Dropbox backbone subtree
  (`<root>/LargeFileBridge/`) with an fsevents recursive watcher. Dropbox writes files in bursts as a
  sync lands, so **debounce** (settle for a few seconds of quiet) before acting, and coalesce a burst
  into one reconcile pass. Ignore events under `.dropbox` / `.dropbox.cache` (§4).
* **Periodic poll fallback.** fsevents can miss events (placeholders materializing, a machine waking
  from sleep, a missed distributed notification), so the **background pin process**
  ([storage_local.mdx §13](./storage_local.mdx#13-the-background-pin-process--the-transparency-contract), the
  launchd job every 15 min) also **polls** the subtree each run: list peer subfolders, stat each
  `LargeFilesBridge_SyncList.yaml`, and reconcile any whose mtime/size changed since last seen.

Detecting which entries changed since the last poll by comparing each file against the previously recorded size/path set is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `diffStatus()`

When a peer's file changes, LFB **reconciles it** — verifies local claims against the real filesystem
+ IPFS pinset ([hub §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)), treats absences
as untrusted ([hub §5.2](./sync_list.mdx#52-absence-is-not-deletion-default)), and resolves conflicts
by revision → content → time ([hub §5.3](./sync_list.mdx#53-conflicts-resolve-by-revision-then-content-then-time)).
The reconciled result is written through the [storage_local.mdx §15](./storage_local.mdx#15-the-store-layer-how-code-touches-disk)
store layer. **Never apply a Dropbox-delivered list directly.**

## 6. Publishing — writing OUR list to Dropbox

The inverse direction. After a scan/pin, this computer writes its **own** current computer-scope
list into **its own subfolder only** (§2), following the hub's publish rules
([hub §6](./sync_list.mdx#6-publishing--writing-our-list-out-to-the-backbones)):

* Bump `origin.revision`, stamp `origin.written_at`, set `origin.computer_id/label/owner` to **this**
  machine.
* Write **only what this computer can vouch for** — `pinned_by` includes this computer only for CIDs
  it actually pins right now ([hub §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed));
  claims about peers are copied forward from the last reconciled truth, never invented.
* **Write atomically** — serialize to a unique `*.tmp` in the same subfolder, then `rename` over the
  real file, so Dropbox never uploads a half-written list (which would reach peers as a partial/
  zero-byte hazard, §4).
* Only when `settings.backbones.dropbox.enabled` is `true`. Honor the IPFS only-our-content defaults
  ([knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)):
  publishing the list never turns this machine into a public gateway/relay, and
  `settings.ipfs.reprovide_strategy` stays `pinned`.

## 7. Privacy & security

Dropbox is a **third party**. What LFB puts there is deliberately low-risk, but the posture is
explicit:

* **Metadata only, never bytes.** The list carries file **paths** and **IPFS CIDs** — metadata about
  what should exist and where it is content-addressed. The **actual file bytes never touch Dropbox**
  (LOCKED, top of file); IPFS moves them.
* **No secrets, ever.** The list holds no auth sessions, no OAuth credentials, no signing secret —
  those live under the state root and never leave it
  ([storage_local.mdx §10](./storage_local.mdx#10-auth-session--signing-secret-runtime-not-settings)).
* **Owner allow-list.** It is the user's **own** Dropbox account, and only **allow-listed owners'**
  lists are adopted — a peer file with a non-allow-listed `origin.owner` is rejected
  ([hub §5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect)).
* **Only-our-content honored.** Nothing about the Dropbox backbone loosens the IPFS defaults
  ([knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)); the
  computer stays a private, gateway-off node.

## 8. YAML — the backbone block, and a reminder of the file it lives in

The Dropbox backbone is configured by the hub's own schema block — no new keys beyond
backbone-local `enabled` / `root`:

```yaml
# inside large_files.settings.backbones — the Dropbox knobs (defined by the hub, §3)
large_files:
  settings:
    backbones:
      dropbox:
        enabled: true                          # read from / publish to Dropbox at all?
        root: ~/Dropbox (Personal)/LargeFileBridge   # override; else resolved via ~/.dropbox/info.json (§3)
      google_drive: { enabled: false, root: null }
      repo_git:     { enabled: true }
```

The file at `~/Dropbox/…/<computer_label>/LargeFilesBridge_SyncList.yaml` is a **normal
computer-scope Sync List** — do not restate its schema here; the hub owns it
([hub §3](./sync_list.mdx#3-the-file--largefilesbridge_synclistyaml)). Its first three lines are the
reminder of what it is:

```yaml
large_files:
  scope: computer          # Dropbox carries the COMPUTER list only (§1)
  origin: { computer_label: bryan-macbook, revision: 42, written_at: 2026-07-02T14:32:15Z, owner: bryan@act3ai.com }
  # ...settings / repos / directories exactly per sync_list.mdx §3 — NOT repeated here.
```

## 9. The other role — Dropbox as a browseable File System root

Everything above is Dropbox's **first** role: a **Sync-List backbone** (a road that carries the small YAML
list between the user's computers). Dropbox has a **second, completely separate role** in LFBridge, and this
section records it so the two are never confused.

| | Role 1 — **Sync-List backbone** (§1–§8) | Role 2 — **Browseable File System root** (this §9) |
| --- | --- | --- |
| What it is | A transport for `LargeFilesBridge_SyncList.yaml` between the user's own machines. | A place on disk the user browses and compresses inside the **File System** page. |
| Owned by | This spec. | [file_system.mdx §4](./file_system.mdx) — this section only supplies the Dropbox-specific discovery. |
| Gated by | `large_files.settings.backbones.dropbox.enabled`. | **Nothing** — visible whenever the Dropbox mount exists on this machine. |
| Writes to Dropbox? | Yes — the YAML list only, into our own subfolder (§6). | **No.** Read-only directory metadata to draw and walk rows. Never writes. |

> **The separation is LOCKED.** Surfacing Dropbox in the File System browser (role 2) **never** enables the
> Dropbox backbone (role 1), **never** writes a Sync List, and **never** places bytes into Dropbox. A user with
> `backbones.dropbox.enabled: false` still sees Dropbox at the top of the home column and can compress the
> videos inside it. The only thing the two roles share is the **discovery path** below.

### 9.1 Why Dropbox is surfaced (the product reason)

The multi-gigabyte files LFBridge exists to compress and pin increasingly live **in Dropbox**, not in the
user's plain home tree. On macOS the Dropbox client mounts under `~/Library/CloudStorage/Dropbox`, buried one
level deep — so [file_system.mdx §4](./file_system.mdx) lifts it to the **top of the home column**, presented
as if it sat directly in `~`. The user opens **File System** and Dropbox is simply *there*, one click from the
big files, without having to know the `Library/CloudStorage` path exists.

### 9.2 Dropbox-specific discovery for the surfaced root (reuses §3)

The browseable root reuses the **same location logic** that §3 already defines for the backbone — there is one
source of truth for "where is Dropbox on this machine":

* **Primary (macOS): the CloudStorage mount.** `~/Library/CloudStorage/Dropbox` (and `Dropbox (Personal)` /
  `Dropbox (Company)` when several accounts are linked). This is what [file_system.mdx §4.3](./file_system.mdx)
  enumerates and classifies as `provider: "dropbox"`.
* **Authoritative path check: `~/.dropbox/info.json`.** The same file §3 parses for the backbone gives the
  real root path(s) per linked account — the tie-breaker when the mount name is ambiguous or relocated.
* **Legacy `~/Dropbox` symlink de-dup (LOCKED).** Older installers leave `~/Dropbox → ~/Library/CloudStorage/
  Dropbox`. The home column would otherwise show Dropbox twice; [file_system.mdx §4.4](./file_system.mdx)
  de-duplicates by realpath so **exactly one** Dropbox row appears (the cloud shortcut), while an unrelated
  `~/Dropbox alias` document is left untouched.

### 9.3 Hazards carry over, read-only

Dropbox's browsing role hits the **same vendor hazards** as the backbone (§4), but only in the gentle,
read-only direction:

* **Online-only / Smart Sync placeholders** are `stat`-ed for metadata **without materializing** them — we
  never trigger a download just to draw a row or compute the interest tint
  ([file_system.mdx §4.7](./file_system.mdx)). A placeholder video still counts toward "this folder is
  interesting"; its bytes are only fetched if the user explicitly opens or compresses it.
* **`.dropbox` / `.dropbox.cache` metadata** is skipped by the shared `HARD_SKIP` set (§4) during any walk.
* **Conflicted-copy / partial files** are irrelevant to browsing — they are just more rows; the reconcile-or-
  reject rules of §4 apply only to the *backbone* reading the YAML list, not to a human browsing folders.

The Dropbox mount is discovered and classified for the File System browser (the `provider: "dropbox"` row) in code, reusing the §3 location logic:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/cloud-roots.ts` — `detectCloudRoots()` (the `Dropbox`-prefixed branch) · surfaced on the home column by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDirectory()` (the `isHomeDir` block + symlink de-dup).

## 10. TO DO Batch for this storage (LOCKED)

Dropbox participates in the product-wide **TO DO Batch** system
([to_do_batches.mdx](./to_do_batches.mdx)) — but through its **second role** (the browseable File System
root, §9), not its backbone role. A **TO DO Batch is a per-storage bundle of recommended file actions** —
there is exactly **one** Dropbox batch, scoped to the big files **under the Dropbox root**, persisted as
hierarchical YAML at **`~/T/_large_files_bridge/_do_batches/dropbox_2_do.yaml`**. Like every batch it is
**local, disposable, and uncommitted** — machine-local state only, **never** written into any `.lfbridge/`
and **never** team-shared. (This is entirely separate from the Sync List that rides the Dropbox backbone;
the batch never enters Dropbox — only the YAML list does, per the LOCKED rule at the top of this file.)

* **Which categories apply to Dropbox.** The Dropbox root is a plain filesystem tree, not a repo, so
  **`git_ignore`** ("big-but-not-git-ignored") applies **only to a Dropbox subfolder that is itself a Git
  repo** — otherwise the batch carries **`compress_video`**, **`compress_image`**, **`pin`**, and
  **`pull_down`**, plus the transcription discovery below.
* **Dropbox-specific hazard — network / placeholder files.** The calc engine
  ([to_do_batch_calc_engine.mdx](./to_do_batch_calc_engine.mdx)) **inherits the scan's network-mount
  caution** (§4, §9.3): it judges candidates from `stat` metadata and **will not stall on unmaterialized
  online-only / Smart Sync placeholders** — it never forces a download just to score a file for a
  `pin`/`compress`/`pull_down` recommendation. Placeholders are recommended-on-metadata, materialized only
  if the user later Applies.
* **Recalculate-and-replace each scan.** The batch is **recomputed from scratch and replaced** on every
  scan pass; it **recommends only** — never writes a decision, never moves a byte — and **skips any
  file/axis the team already decided**.
* **It surfaces on the To Do page as a slug.** Any batch **with work** renders on the To Do page
  ([to_do.mdx](./to_do.mdx)) as a **slug**; opening the slug's chevron shows the recommendations as
  two-axis **decision toggles** pre-set to the recommended state ([decision_toggles.mdx](./decision_toggles.mdx)),
  which the user confirms and **Applies**.
* **Transcription discovery for Dropbox.** The on-demand **"Show what could be transcribed"** link runs
  the **What Can Be Transcribed Calc Engine** ([transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx)),
  which walks the Dropbox root for **video/audio that has no `.transcription` artifact at its resolved
  placement target** ([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — a `.lfbridge/`
  only for a Dropbox subfolder that is itself a working repo; otherwise the owning SDL's root) (again from
  metadata, no forced materialization) and emits a per-storage **transcribe batch** → a **transcribe
  slug** (a ~80%-width, default-checked popup: *"Transcribe N files"* → background jobs on
  [processing.mdx](./processing.mdx)).

Producing the Dropbox batch during scan recalc, and naming/replacing its YAML, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batch.engine.ts` —
`TodoBatchCalcEngine.recalcForStorage(storage)` writing via
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batches.store.ts` —
`batchFileName(storage)` (→ `dropbox_2_do.yaml`); the transcribe discovery walk is
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/transcribe-scan.engine.ts`.

## 11. Acceptance criteria

Dropbox as a backbone is correct when **all** of the following hold:

1. **Only the YAML Sync List is written to Dropbox — never a large file.** No code path ever places a
   bridged large file (or its bytes) into the Dropbox tree (LOCKED).
2. Dropbox carries the **computer-scope** list only (`scope: computer`); per-repo lists ride the
   company sync repo's git ([repo__list_syns.mdx](./repo__list_syns.mdx)), not Dropbox and not the working repo.
3. Each computer writes to **its own subfolder only** —
   `~/Dropbox/LargeFileBridge/<computer_label>/LargeFilesBridge_SyncList.yaml` (label sanitized) —
   and **reads every peer subfolder**, so two machines never clobber one shared file.
4. Every peer file is **reconciled**, never applied blindly ([hub §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)):
   local claims verified against the filesystem + IPFS pinset, absence ≠ delete, conflicts by
   revision → content → time.
5. The Dropbox root is resolved via `settings.backbones.dropbox.root` first, else
   **`~/.dropbox/info.json`** (Personal/Business, `~/Dropbox` / `~/Dropbox (Personal)` /
   `~/Dropbox (Company)`); enable/disable via `settings.backbones.dropbox.enabled`.
6. All Dropbox hazards are tolerated per §4 → [hub §5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect):
   conflicted copies (reconciled as extra untrusted input), online-only placeholders (materialized
   before reading), selective-sync exclusions (spurious absence, not delete), `.dropbox` /
   `.dropbox.cache` metadata (ignored), partial/zero-byte files (rejected, previous kept).
7. Change detection uses an **fsevents watch with debounce** plus a **periodic poll fallback** via
   the background pin process ([storage_local.mdx §13](./storage_local.mdx#13-the-background-pin-process--the-transparency-contract));
   a changed peer file triggers reconciliation, never a blind apply.
8. Publishing bumps `revision`, stamps `written_at`, writes **only self-verifiable `pinned_by`**
   claims, writes **atomically** (temp-then-rename in our own subfolder), only when
   `dropbox.enabled` is true, honoring the IPFS only-our-content defaults
   ([knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)).
9. The list contains **paths + CIDs only** — never bytes, never secrets — and only **allow-listed
   owners'** lists are adopted (§7).
10. **Second role, kept separate (§9).** Dropbox is **also** surfaced as a browseable File System root at the
    top of the home column ([file_system.mdx §4](./file_system.mdx)) — visible whenever the mount exists,
    **regardless of** `backbones.dropbox.enabled`, reusing the §3 discovery + the legacy-`~/Dropbox`-symlink
    de-dup. This browsing role is **read-only metadata** (placeholders are never materialized to draw a row)
    and **never** writes to Dropbox or enables the backbone.
11. Dropbox owns exactly **one TO DO Batch** (§10), persisted local/disposable/uncommitted at
    `~/T/_large_files_bridge/_do_batches/dropbox_2_do.yaml`, scoped to big files under the Dropbox root and
    recalculated-and-replaced each scan; its applicable categories are **`compress_video`,
    `compress_image`, `pin`, `pull_down`** — **`git_ignore` only** for a Dropbox subfolder that is itself a
    Git repo — plus the on-demand transcribe batch; the calc engine inherits the network-mount caution and
    **never materializes placeholders** to score a file; batches with work surface as slugs on the To Do
    page.
