# LargeFileBridge — Transcribe (audio & video → text)

LFBridge can **transcribe** the speech in a media file to a **plain-text transcript**. It works on
**audio** (MP3, WAV, FLAC, M4A, AAC, OGG, …) and on **video** (MP4, MOV, MKV, WEBM, … — the audio
track is demuxed to MP3 first), so any file whose "what was said" matters becomes searchable, quotable
text.

Two things make this spec worth pinning down: **where transcripts are stored** (a parallel hidden
hierarchy under the storage's own root — §3), and **how the engine runs** (a copy of our `Transcribe`
tool called **in-process** as a class, never shelled out — §5).

## 1. What it does (the loop)

For one media file, transcription is:

1. **Classify** the file as audio or video by name ([media.ts `mediaKindForName`](../code/packages/shared/src/media.ts)).
   Anything else is skipped with a clear reason.
2. **Demux if video** — a video/container is not directly digestible by Whisper, so its audio track is
   extracted to a temporary MP3 with **ffmpeg** (`-vn -acodec libmp3lame`). A video with **no audio
   stream** writes a short "no audio to transcribe" placeholder instead of failing.
3. **Transcribe** the audio with **Whisper** (`whisper --model base --language en`), preferring the
   Apple-Silicon **MPS** GPU with an automatic **CPU fallback** when MPS emits garbage (the sanity check
   the copied engine already carries). Whisper is spawned **asynchronously** and its stdout is streamed
   to derive a **determinate percentage** (segment end-timestamp ÷ media duration) — see §5.1.
4. **Write the transcript** — a plain-text file with a small metadata header (source name, timestamp,
   engine/model, source duration) — **inside the owning root's `.lfbridge/`**, path-mirrored, under the
   media's full filename with `.transcription` **appended** (§3): `<repo>/a/b/talk.mp3` →
   `<repo>/.lfbridge/a/b/talk.mp3.transcription`.
5. **Report truthfully** — word count on success; an honest skip/error otherwise (never a silent
   "done"; the [pin_process.mdx §6](./pin_process.mdx#6-never-a-silent-no-op--report-truthfully-locked-learned-from-a-defect)
   rule applies here too).

Transcription is **always an explicit user action** — like compression, LFBridge **offers**, it never
transcribes a file on its own. It is **local-only**: no account, API key, or Google Cloud credential is
involved (§5.2).

The per-file loop (classify → ffmpeg demux → whisper → write transcript → report), wrapped in a
progress-registry job so the dock shows a live card, is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` — `transcribeOne()` (async; wraps the run in `track("transcribe", …)` from `modules/progress/progress.registry.ts`).

## 3. Where transcripts are stored — inside the repo/storage `.lfbridge/`, path-mirrored, extension APPENDED (LOCKED — supersedes the earlier beside-media layout)

Transcripts do **not** go in the Local state dir, and — as of this revision — they do **not** sit loose
next to the media either. They live **inside the owning repo/storage's own `.lfbridge/` tracking
directory**, in a **path-mirrored hierarchy**, under the media's **full filename (extension kept)** with
**`.transcription` appended** as a **second** extension.

> **This supersedes the previous LOCKED "beside the media, extension replaced" layout.** An earlier
> revision wrote `render/talk.mp3` → `render/talk.transcription` next to the media. The product owner's
> current, authoritative instruction replaces that: *"because I'm in the repo and what I'm transcribing is
> in the repo, it'll have that `.lfbridge` directory in the repo. Say a file is two directories deep in the
> repo — it creates those two directories inside the `.lfbridge`, then creates the same file name including
> that extension, and we add a second dot extension `.transcription`, and puts the transcription output
> there."* So the transcript is a **tracking artifact under `.lfbridge/`**, not a sibling of the media, and
> its name **appends** `.transcription` rather than **replacing** the original extension.

### 3.1 The layout — under `.lfbridge/`, mirror the path, append `.transcription` (LOCKED)

For a repo file at `<repo>/a/b/talk.mp3`, the transcript is written to
`<repo>/.lfbridge/a/b/talk.mp3.transcription`:

```
<repo>/
├── a/
│   └── b/
│       └── talk.mp3                       ← the real media file (untouched, still in the working tree)
└── .lfbridge/
    └── a/
        └── b/
            └── talk.mp3.transcription     ← the transcript: same relative path under .lfbridge/,
                                              full filename (talk.mp3) with .transcription APPENDED
```

Three rules, together, define the path — **all LOCKED**:

1. **Root = `<owning root>/.lfbridge/`.** The transcript lands under the owning repo/storage's hidden
   `.lfbridge/` directory — **not** beside the media and **not** in `~/T/_large_files_bridge/`. Note that
   the working repo's committed `.lfbridge/` now holds **only** the two content artifacts (transcripts and
   AI descriptions); LFB's tracking state — `repo_storage.yaml`, the per-file sidecars, and `history/` —
   lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/`, never the working repo
   ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx),
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).
2. **Mirror the media's relative directory hierarchy.** A file two directories deep (`a/b/talk.mp3`)
   **creates `a/b/` inside `.lfbridge/`** and writes there. The media's path relative to the root is
   reproduced verbatim under `.lfbridge/`.
3. **Keep the full filename, APPEND `.transcription`.** The transcript filename is the media's **entire
   filename including its original extension**, with **`.transcription` appended** — `talk.mp3` →
   `talk.mp3.transcription` (**not** `talk.transcription`). The `.transcription` token is a single constant
   — `TRANSCRIPTION_EXT` — **appended**, never substituted.

> **Why `.lfbridge/` (and not beside the media).** The working repo's `.lfbridge/` is **committed and
> travels with the repo** ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx); the working tree's own
> `git status` shows the committed `.lfbridge/…/talk.mp3.transcription` artifact tracked). Putting the
> transcript there means it is **version-controlled and
> synced across the user's computers by git** — exactly the *"the transcript is produced on one computer and
> travels with the repo"* behavior of [use_case_two_computers.mdx](./use_case_two_computers.mdx) — while the
> media working tree stays **clean** (no loose `.transcription` files scattered next to videos), and every
> derived tracking artifact lives in **one** hidden place. Transcripts are small UTF-8 text, so committing
> them is cheap and desirable (unlike the multi-GB media, which stays git-ignored and rides IPFS).

### 3.2 Append-not-replace resolves the old collision caveat (LOCKED)

Because the original extension is **kept** and `.transcription` is **appended** (not substituted), two
media files that differ only by extension in the same folder **no longer collide**: `talk.mp3` →
`.lfbridge/…/talk.mp3.transcription` and `talk.wav` → `.lfbridge/…/talk.wav.transcription` are **distinct
files**. The earlier layout's accepted "`talk.mp3` and `talk.wav` overwrite each other" caveat is therefore
**retired** — appending the extension is precisely what fixes it.

The `.transcription` body is plain text and opens in the ordinary `/file` viewer like any other text file.
A tool that wants a media file's transcript computes the path deterministically:
`join(root, ".lfbridge", relDir, mediaFilename + ".transcription")`.

### 3.3 Relationship to `.lfbridge/analysis/`

[storages.mdx §6](./storages.mdx#6-per-file-media-analysis-transcription--description--visuals-by-time)
sketched transcripts as `transcript.yaml` under `.lfbridge/analysis/<relpath>/`. **This spec is
authoritative for the transcript files.** Transcripts live **inside `.lfbridge/`** (§3.1) — the same hidden
tracking area storages.mdx §6 pointed at — but as a **path-mirrored, human-openable plain-text
`<name.ext>.transcription` file placed directly under `.lfbridge/`** (mirroring the media's relative
directory), **not** as a `transcript.yaml` and **not** under an `analysis/` sub-directory. When the
storage/repo file table needs to note that a transcript exists (the `analysis: ["transcript"]` indicator),
it detects the presence of `.lfbridge/<rel-dir>/<name.ext>.transcription`, computed deterministically from
the media path.

### 3.4 Resolving where the parallel hierarchy lives — the ordered rule (LOCKED)

The transcript (and the `.ai_description` of ai_description.mdx §2) is written into a **placement root**
that does **not** always match the media file's nearest storage-carrying ancestor. A media file very often
lives **out in the user's wider filesystem** (e.g.
`~/_Mirror/Politics/Charlie_Kirk_Mi/CIA/FINAL_CIA_Drones.mp4`) — not inside a Git repo and not inside a
storage root. For those files we want the transcript to travel with the storage that owns them (often a
dedicated Git repo), mirroring the media's relative hierarchy under that root's `.lfbridge/`. Placement is
therefore resolved by an **ordered rule** — the first case that matches wins. This is the **authoritative**
resolution for *every* derived-artifact producer (transcription and AI description share it). In every case
the transcript lands at **`<root>/.lfbridge/<rel-dir>/<name.ext>.transcription`** — the resolved `root`, the
`.lfbridge/` tracking directory, the media's mirrored relative directory, and the full filename with
`.transcription` **appended** (§3.1):

**A. Containing storage root (walk-up).** Walk **up** from the media file's directory to the filesystem
root. The **nearest ancestor** carrying a `storage.yaml`, a `.lfbridge/`, or a `.git/` (a repo *is* a
storage — [storages.mdx](./storages.mdx)) is the containing root. This is the "repo root" walk the
product owner described: we are usually in some subdirectory *under* a repo, so we climb until we find the
`.git/`. When found:
* `root` = that ancestor · `rel` = the media file's path **relative to it** · the transcript lands at
  **`<root>/.lfbridge/<rel-dir>/<name.ext>.transcription`**. This is the **common, primary case** the
  product owner described: *"I'm in the repo and what I'm transcribing is in the repo,"* so the transcript
  is written **inside the repo's `.lfbridge/`**, path-mirrored, extension appended (§3.1).
* **No `*.transcription` git-ignore nudge is needed** anymore. Because the transcript lives **inside
  `.lfbridge/`** — which is **committed and travels with the repo** ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)) —
  it is version-controlled and synced by git across the user's computers on purpose (small text; the
  multi-GB media stays git-ignored and rides IPFS). This **removes** the earlier revision's need to inject a
  `*.transcription` line into a plain repo's `.gitignore`. A repo's own per-repo config UI is
  [repo_settings.mdx](./repo_settings.mdx) — reached *because* we detected we are inside a repo.

**B. Owning storage by mapping — routed to the dedicated repo (the fix, LOCKED).** When walk-up finds
nothing (the file is out in the wider filesystem), the file is still **owned** by a storage:
* **Which storage owns it.** The **most specific** company/personal storage whose **mapped source
  directory** ([syncable_data_location.mdx §3](./syncable_data_location.mdx), storage_settings.mdx §4a)
  contains the file — matched against this computer's **graft** local paths ([devices.mdx §4](./devices.mdx)).
  If no explicit mapped dir contains it, the **Personal** storage is the **default catch-all** owner for
  anything under the user's home tree ([storage_personal.mdx](./storage_personal.mdx)).
* **Where the artifact goes.**
  * If the owning storage has its **dedicated Git repo** backing **enabled** (storage_settings.mdx §4.1),
    the artifact is written **under that dedicated repo's `.lfbridge/`**, mirroring the file's hierarchy —
    `<dedicatedRepo>/.lfbridge/<rel-dir>/<name.ext>.transcription`. `rel` is the file's path relative to its
    **mapped-dir root** (prefixed by the mapped-dir key), or relative to **`~`** for the personal default
    catch-all. The transcript is **committed** (it lives in `.lfbridge/`) — the dedicated repo **exists to
    hold and back up these artifacts** (the product owner: *"it won't need to add the `.gitignore` if you
    have a dedicated git repo — it'll just put it in there"*).
  * If the owning storage has **no** dedicated repo enabled, the artifact goes under the storage's own
    **root** — `<storageRoot>/.lfbridge/<rel-dir>/<name.ext>.transcription`, where its `.lfbridge/` already
    lives.

**C. First-time experience (no storage yet).** If the file is owned by **no** storage **and** the user has
**no Personal storage configured at all**, we do **not** silently write beside the media. The action is
gated by the **first-time setup wizard** (§3.5).

**D. Last resort.** Personal exists but the file is genuinely outside every mapped hierarchy and no
dedicated repo is enabled → the media file's **own directory** is the placement root, so a `.lfbridge/` is
created **in that directory** and the transcript lands at `<mediaDir>/.lfbridge/<name.ext>.transcription`.
This is the narrow fallback, not the common path — the `.lfbridge/` + append-extension invariant (§3.1)
still holds, so a transcript is **never** written loose beside the media.

> **Worked example (the reported case).** With the Personal storage's **dedicated Git repo** turned ON at
> `~/BGit/Bryan_git/personal_large_files_bridge/`, transcribing
> `~/_Mirror/Politics/Charlie_Kirk_Mi/CIA/FINAL_CIA_Drones.mp4` writes to
> `~/BGit/Bryan_git/personal_large_files_bridge/.lfbridge/_Mirror/Politics/Charlie_Kirk_Mi/CIA/FINAL_CIA_Drones.mp4.transcription`
> — mirrored under the dedicated repo's **`.lfbridge/`**, the full filename with **`.transcription`
> appended**, **committed** (travels with the repo), and **no** `.transcribe/` directory. In-repo, the same
> file two directories deep at `<repo>/CIA/FINAL_CIA_Drones.mp4` writes to
> `<repo>/.lfbridge/CIA/FINAL_CIA_Drones.mp4.transcription`.

The ordered resolution (walk-up → owning-storage/dedicated-repo → first-time signal → last-resort), the
`.lfbridge/`-rooted path-mirrored placement, the **append**-`.transcription` naming, and the home-`~`
expansion of pasted paths are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveArtifactPlacement()` (transcript path via `lfbridgeArtifactPath(root, relPath)` = `join(root, ".lfbridge", relPath) + TRANSCRIPTION_EXT`, where `TRANSCRIPTION_EXT` is **appended**, not substituted).
The transcript path (`<root>/.lfbridge/<rel-dir>/<name.ext>.transcription`) and the owning-root resolution wrap it in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` — `resolveTranscriptPath()` / `resolveStorageRoot()`. The per-case `gitIgnore` flag is **retired** — transcripts live in the committed `.lfbridge/` area, so no `*.transcription` ignore line is ever injected.

### 3.5 The first-time setup wizard — actions that trigger it (LOCKED)

A derived-artifact action must know **where its output will land before it runs**. When placement rule C
fires — the file is owned by no storage and **no Personal storage exists yet** — the action is redirected
to a **first-time setup wizard** instead of producing a file in a surprising place.

* **Triggering actions.** Any derived-artifact action taken on a file/directory that resolves to
  `needsSetup`:
  * **Transcribe** (this spec).
  * **Get AI details** / **AI description** ([ai_description.mdx §2](./ai_description.mdx)).
  (The list grows as new per-file analysis actions are added; every one consults placement first.)
* **What the wizard does.** It walks the user through the minimum configuration for the situation:
  1. **In a repo?** If the file *is* under a repo (rule A would have matched) we never reach the wizard —
     repo configuration is [repo_settings.mdx](./repo_settings.mdx). The wizard is only for the
     no-storage case.
  2. **Company vs personal.** If a **company** storage already covers the path, use it. Otherwise
     **default to Personal** — create the one Personal storage.
  3. **Where the Personal `.lfbridge/` root lives.** Default to creating it under the **home (`~`)
     directory** ([storage_personal.mdx §1](./storage_personal.mdx)); or, if the user prefers, point it at
     a **dedicated Git repo** (`~/BGit/Bryan_git/personal_large_files_bridge/`) — in which case artifacts
     store **under that repo** per rule B.
* **After setup.** The originally-clicked action re-resolves placement (now a storage exists) and runs —
  the user lands back where they were, no reload.

The backend reports `needsSetup` from the same placement resolver; every derived-artifact launcher checks it
and, when set, opens the wizard (via a global request bus, so no entry point needs bespoke wiring) and
re-runs the original action once a Personal storage exists. The wizard offers exactly the two choices above
— **Dedicated Git repo** (recommended; artifacts tracked + backed up, placement rule B) or **Plain folder** —

## 4. What a transcript file contains

A transcript is plain UTF-8 text: a short metadata header, a divider, then the transcript body. The header
records the **actual engine + model** used ([transcribe_engine.mdx](./transcribe_engine.mdx)) — the
heavyweight `qwen` engine (`qwen3-asr-1.7b-mlx`) or the Whisper-on-Mac fallback (`whisper-base`) — and the
**source duration**, so a reader can confirm the transcript covers the whole file (§critical: the full
1–2h+ duration, [transcribe_engine.mdx §4](./transcribe_engine.mdx)):

```
Transcription of: talk.mp3
Generated on: 2026-07-05 09:41:00
Engine: qwen3-asr-1.7b-mlx (device: mlx, language: en)
Source duration: 01:47:12   ·   Transcript covers: 01:47:12  ✓ full
============================================================

Welcome back to the show. Today we're talking about …
```

A video with **no audio stream** yields a one-line placeholder (`[No audio stream — nothing to
transcribe]`) so a re-run over a tree can skip it cheaply. Results are **idempotent by path**: an existing
transcript at `<root>/.lfbridge/<rel-dir>/<name.ext>.transcription` (§3) is reported as "already
transcribed" unless the caller asks to overwrite (Re-transcribe). A transcript whose header shows
**`Transcript covers` < `Source duration`** (a truncated run) is treated as **incomplete** and is a
candidate to re-transcribe, never a silent success ([transcribe_engine.mdx §4](./transcribe_engine.mdx)).

## 5. The engine — copied in, called in-process (LOCKED)

> **Deep engine spec: [transcribe_engine.mdx](./transcribe_engine.mdx).** This section defines the
> *in-process, non-blocking* discipline (the load-bearing §5.1 rule). The **two engines** (the heavyweight
> local **Qwen3-ASR** LLM and the **Whisper-on-Mac** fallback), the **on-demand model
> download/install/configure with the user's permission** (disk estimate, health-check/repair, the two
> progress UIs, and the flow-resume after install), the **critical full-duration / no-truncation
> requirement for 1–2h+ media**, and the **per-machine parallel calibration** all live in
> [transcribe_engine.mdx](./transcribe_engine.mdx). This file references it and does not restate it.

The transcription engine is our existing **`Transcribe.js`** tool (canonically
`~/BGit/all/tools/Transcription/Transcribe.js`, and a copy in the ACT3-internal web app). LFBridge does
**not** reuse the ACT3-internal copy and does **not** bounce through the command line or spawn a separate
`node` process to run it. Instead:

* The engine is **copied into this repo's own code `tools/` subdirectory** and adapted to TypeScript as a
  **class** with a callable method (no `process.argv`, no `process.exit`, no top-level auto-run). We own
  our copy; upstream changes are pulled in by re-copying, not by importing across repos.
* The web-app backend **imports the class and calls it directly, in-process** — one function call, a
  structured result back. This is the same discipline as
  [compression.mdx](./compression.mdx): drive the local media binary from inside the Node process.

* **It still drives the local `whisper` and `ffmpeg` binaries** — those are the actual
  transcription/demux tools (Whisper is a Python CLI). "In-process" means *we don't shell out to our own
  Node script*; invoking the underlying `whisper`/`ffmpeg` executables is expected and mirrors how
  `compression.service.ts` invokes `ffmpeg`/`magick`.
* **Entirely local at run time — no phone-home.** Transcription runs **on-machine**; nothing about the
  **media or the transcript** ever leaves the computer (charter alignment: only-our-content, no phone-home).
  The **one** networked step is the **one-time provisioning download of open-source model weights** (the
  heavyweight Qwen3-ASR weights) — this is analogous to `brew install` fetching a formula, sends **none
  of the user's content**, happens **only with explicit permission**, and never recurs once installed. See
  [transcribe_engine.mdx §3](./transcribe_engine.mdx).
* **Binaries vs. model weights are two different "missing" cases.**
  * A **missing CLI binary** (`whisper`, `ffmpeg`) is handled by the **Tools preflight**
    ([tools.mdx](./tools.mdx)) / the **setup card** (§5.2): a clear `tool_missing` result with the exact
    install command and a **Re-check** — an **explicit-click** install, never silent.
  * A **missing / half-installed / outdated heavyweight model** is a **separate, larger provisioning step**
    that LFBridge **can download, install, and configure for the user — with their explicit permission and
    a disk-space warning** — showing a **download progress UI** then an **install/configure progress UI**
    ([transcribe_engine.mdx §3](./transcribe_engine.mdx)). This is the piece the earlier "does not
    auto-install behind a web request" wording did **not** cover; it is now allowed **because the user
    approves it** (charter consent posture preserved).

The copied in-process engine (ffmpeg demux + Whisper with the MPS→CPU fallback and English sanity check, returning a structured result instead of logging/exiting) lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/Transcribe.ts` — class `Transcriber`, methods `toolStatus()` / `transcribeToFile()`.

### 5.1 Async & non-blocking — the "Generate transcript didn't work" fix (LOCKED)

The standalone `Transcribe.js` uses `spawnSync`/`execSync`. Copied verbatim that would be a **defect in a
web server**: `spawnSync` **blocks the Node event loop for the entire Whisper run**. On a 6-minute video
that froze the whole API — `GET /api/health` and `GET /api/progress` stopped responding, the open
`POST /api/transcribe/file` request looked dead, and the button appeared to "do nothing." This was the
reported bug.

The in-process engine therefore spawns **every** child process (`ffprobe`, `ffmpeg`, `whisper`)
**asynchronously** (`child_process.spawn`, awaited) so the event loop stays free to serve other requests
while a transcription runs. Two behaviours ride on that:

* **Determinate progress.** Whisper streams each decoded segment to stdout as `[start --> end]`; the
  engine parses the **end timestamp** and divides by the media **duration** (from `ffprobe`) to emit a
  0–1 fraction. `transcribeOne()` maps that to a `{ done, total: 100, unit: "%" }` update on its
  progress-registry job, so the dock card fills a real bar. When duration is unreadable the card stays a
  spinner rather than lying.
* **Cross-tab visibility.** Because the run registers in the server-side job registry, the dock card
  shows even in a browser tab that did **not** start it (a launchd tree run, another window).

The async spawn (with the streamed stdout progress parse and the duration probe) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/Transcribe.ts` — class `Transcriber`, private `spawnAsync()` / `runWhisper()` / `probeDurationSec()` / `parseSegmentEndSec()` (the public `transcribeToFile()` takes an `onProgress` sink).

#### 5.2.1 Detection detail — why a genuinely-installed tool must never be reported missing (LOCKED)

The setup card's entire premise — "install what's missing, then Re-check" — only works if the **detection**
is trustworthy. A false "missing" for a tool that is actually on the machine sends the user chasing a
problem that doesn't exist (the exact complaint this hardening pass answers: *"I clicked Transcribe and it
failed — I suspect it didn't install what it needs,"* when in fact `whisper`, `ffmpeg`, and the Qwen3-ASR
model were all present).

* **How detection works.** `commandExists(cmd)` shells out to `which` (`where` on Windows) — never a shell
  alias, never `command -v` (which can see builtins/aliases a spawned child can't run). A tool is "installed"
  exactly when `which` finds it **on the searched `PATH`**.
* **The PATH gap this closes.** Both `whisper` and `mlx-qwen3-asr` are installed by **pipx**, which places
  its shims in **`~/.local/bin`** — a directory that is on an **interactive shell's** `PATH` (via `.zshrc` /
  `.bash_profile`) but is **not** guaranteed to be on the `PATH` of every process that might run this code:
  a `launchd` agent with no `EnvironmentVariables` block gets macOS's bare minimum
  (`/usr/bin:/bin:/usr/sbin:/sbin`), and even a hand-rolled `EnvironmentVariables` PATH (see the IPFS
  daemon's autostart plist, which hard-codes `/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin`) still omits
  `~/.local/bin`. Measured on a real machine with all three tools genuinely installed, a minimal/launchd-style
  `PATH` finds **none** of `whisper`, `ffmpeg`, or `mlx-qwen3-asr`; a Homebrew-only `PATH` finds `ffmpeg` but
  still misses the two pipx tools. So the risk is real, not hypothetical: a background-launched or
  restricted-environment process can see the **same machine** as "missing everything" while an interactive
  terminal sees it as fully set up.
* **The fix — search a WIDENED path, not the bare inherited one.** Tool detection (`commandExists()`) and
  tool **execution** (`spawnAsync()`, so a bare command name like `"whisper"` resolves the same way it was
  detected) both search `process.env.PATH` **plus** the well-known install locations pipx and Homebrew use:
  `~/.local/bin`, `/opt/homebrew/bin`, `/opt/homebrew/sbin`, `/usr/local/bin`, `/usr/local/sbin`, in addition
  to the standard system dirs. This is **additive only** — an already-correct interactive `PATH` is untouched
  — so it only ever makes a genuinely-installed tool **more** likely to be found, never less.
* **Detection and execution must agree.** It would be a subtler version of the same bug to detect a tool
  successfully (widened `PATH`) but then fail to actually run it (bare inherited `PATH`) — the button would
  look enabled and then fail on click. Both `commandExists()` and `spawnAsync()` search the identical widened
  `PATH` for exactly this reason.

The widened tool-search `PATH` (shared by detection and execution, so the two never disagree) is implemented
in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/audio-prep.ts` — `toolSearchPath()` / `toolEnv()` / `commandExists()` / `spawnAsync()`, reused by
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/qwen-asr.ts` and
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/model-provision.service.ts`
(pipx/brew/ffmpeg detection during provisioning, §3 below). The Whisper engine
(`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/tools/transcribe/Transcribe.ts` — private
`commandExists()` / `spawnAsync()`) keeps its own copy of the engine logic (deliberately not merged into
`audio-prep.ts`, to avoid churn on the working engine) but draws the **same** `toolEnv()` so the two engines
never disagree on what's installed.

backend: the **Media Viewer** button/tab (`useTranscribeFile()`, §2.1) and the **⋮ kebab / row actions**
(`runTranscribeFile()` in `lib/transcribe.ts`, §2.2). Both funnel through `withModelReady()`
([transcribe_engine.mdx §3.2](./transcribe_engine.mdx#32-the-first-time-permission-popup-locked)) before

## 7. Acceptance criteria

1. LFBridge transcribes **audio** (MP3, WAV, FLAC, M4A, AAC, OGG, …) and **video** (MP4, MOV, MKV,
   WEBM, … — audio demuxed via ffmpeg first) to a **plain-text transcript**; non-media files are skipped
   with a reason; a video with no audio yields a placeholder.
2. **Entry points at every scale:** the **Media Viewer** Transcribe button on an open audio/video; a
   **Transcribe…** item in the file ⋮ kebab / right-click / more-menu; a selected-set batch; and
   **Transcribe all files…** on a **directory**, a **repo**, and a **storage**.
3. Transcripts are stored **inside the owning repo/storage's `.lfbridge/` directory** (not the Local state
   dir, and no `.transcribe/` directory), **mirroring the media file's relative directory path** under
   `.lfbridge/`, keeping the media's **full filename including its extension** with **`.transcription`
   appended** (`<repo>/a/b/talk.mp3` → `<repo>/.lfbridge/a/b/talk.mp3.transcription`). Because `.lfbridge/`
   is **committed**, transcripts are version-controlled and travel with the repo across the user's
   computers; **no `*.transcription` git-ignore line is injected**. Appending (not replacing) the extension
   means `talk.mp3` and `talk.wav` no longer collide.
4. The **owning root** is the nearest ancestor with `storage.yaml` / `.lfbridge/` / `.git/`; when none
   exists, a `.lfbridge/` is created in the file's own directory as the base (§3.4 rules A–D).
5. The engine is **copied into this repo's `tools/` directory** (the ACT3-internal copy is **not**
   reused) and is **called in-process as a class** — never by shelling out to `node Transcribe.js`; it
   still invokes the local `whisper` + `ffmpeg` binaries and runs **entirely on-machine** at run time (the
   only networked step is the one-time, permissioned model-weights download, §5 / [transcribe_engine.mdx](./transcribe_engine.mdx)).
6. Missing `whisper`/`ffmpeg` **binaries** produce a clear **tool-missing** result with an install hint
   (explicit-click install, no silent no-op); a missing / half-installed / **outdated heavyweight model**
   triggers the **permission-gated provisioning** flow ([transcribe_engine.mdx §3](./transcribe_engine.mdx))
   — a disk-space warning, then download → install → configure with progress UIs — not a bare error. Every
   run **reports truthfully** (word count on success, honest skip/error otherwise).
7. Transcription is **always an explicit user action** — LFBridge offers, never transcribes on its own; and
   the heavyweight model is **never downloaded/installed without explicit permission** (charter consent).
8. The engine is **async / non-blocking**: a running transcription never freezes the API (health &
   progress keep responding), the button flips to a **Transcribing…** pending state the instant it's
   clicked, and a **live progress card** (determinate % for one file) shows in the dock — including for a
   run this tab did not start.
9. Transcription needs **no credentials and no cloud** (local engine); the repo ships **no transcription
   credentials file**. When the local **binaries** are missing, a **setup card** shows the install commands
   and a **Re-check** button that re-enables the run once fixed, **without a page reload**; when the
   **model** is missing/partial/outdated, the permission popup + provisioning flow of
   [transcribe_engine.mdx](./transcribe_engine.mdx) handles it and then **resumes the original action**.
9a. **Two engines + long-media robustness** ([transcribe_engine.mdx](./transcribe_engine.mdx)): a primary
    **heavyweight Qwen3-ASR** engine and a **Whisper-on-Mac** fallback; whichever runs, the settings are
    **triple-checked to transcribe the FULL duration of 1–2h+ media without truncating** — the run is
    verified to cover the source duration end-to-end (a short/truncated result is a failure, never a silent
    success), and the machine's **parallel-job count is calibrated** to its cores/RAM/model (low-power
    boxes = 1 at a time; a ~24-core box runs several).
10. **Bulk discovery via the To-Do transcribe slugs:** the on-demand **What Can Be Transcribed Calc
    Engine** ([transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx)) — triggered only by the
    **"Show what could be transcribed"** link on the To Do page, never by the scheduled scan — walks every
    storage for transcribable audio/video **lacking a `.transcription` sidecar** and writes one per-storage
    **transcribe batch**; each renders as a **transcribe slug** whose ~80%-of-screen popup lists candidates
    **checked by default**, and **Apply** ("Transcribe N files") **enqueues background jobs** (this spec's
    §5 engine, unchanged) watchable on [processing.mdx](./processing.mdx). Completed files gain a sidecar
    and drop out of the next run.
11. **Transcribe-tab metric tile + header primary (§9.1):** clicking the **`Transcribable`** metric tile
    opens the transcribe popup (candidates checked by default → **"Transcribe N files"** confirm → model
    gate may interpose → async parallel → Processing), instead of the old no-op that re-selected the tab.
    When transcription is the repo's top pending recommendation, the header primary is a **blue**
    (`--lfb-primary`) **"Transcribe ›"** button (trailing `›`) that opens the same popup; it sits at the
    **transcribe** slot of the worst-first order **ipfs-down → pull-down → undecided → transcribe → describe
    → compress**, is **model-gated on Apply not on display**, and disappears (green-`0` tile) once every
    audio/video file has a complete transcript. **Where** the transcript lands is not decided here — it is
    the placement radio owned by [placement_radios.mdx](./placement_radios.mdx) / [repo_settings.mdx](./repo_settings.mdx)
    (§9.2, cross-link only).

## 8. Bulk discovery — "Show what could be transcribed" (the To-Do transcribe slugs) (LOCKED)

Every entry point in §2 acts on media the user has **already found** — a file they opened, a directory
they right-clicked, a page they scoped. This section adds the **discovery** direction: a single, on-demand
sweep that answers *"across all my storages, what could still be transcribed?"* and lands the answer on the
**To Do** page as clickable **transcribe slugs**. It **produces nothing** on its own — it only lists
candidates. The actual transcription is **this spec's §5 engine, wholesale** — nothing about the engine,
the sidecar layout (§3), or the truthful reporting (§1) changes; discovery is purely a **fan-out that
feeds Apply**.

### 8.1 The engine — on-demand, never the scheduled scan (LOCKED)

The **What Can Be Transcribed Calc Engine** ([transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx))
is **explicitly on-demand**: it runs **only** when the user clicks the **"Show what could be
transcribed"** action link on the To Do page — it is **never** part of the periodic background scan
([scan_process.mdx](./scan_process.mdx) / the launchd tree run). Discovery walks are expensive over cloud
mounts and whole-home trees, so they happen when the user asks, not on a timer.

When invoked it walks **each storage** — Personal, Dropbox, every Google Drive, and every repo
([storages.mdx](./storages.mdx)) — for **VIDEO and AUDIO** files that are:

* **Transcribable** — audio, or **video that carries an audio track**. This is the same
  `transcribable` flag defined in [special_files.mdx §3](./special_files.mdx#3-the-transcribable-flag)
  (a silent video is **not** a candidate — it would only ever yield the "no audio" placeholder of §4), and
* **Not yet transcribed** — they have **no `.transcription` transcript** at the resolved
  `.lfbridge/<rel-dir>/<name.ext>.transcription` path (§3), **or** an existing transcript whose header shows
  it **did not cover the full source duration** (a truncated run, §4 / [transcribe_engine.mdx §4](./transcribe_engine.mdx)).
  This is exactly the **skip-already-done** test the page action uses
  (§2.5 / [page_actions.mdx §1.2](./page_actions.mdx)): *absence of a complete transcript* is the candidate test, so a

### 8.2 The transcribe slug + the ~80% popup (Apply → background jobs → Processing) (LOCKED)

transcribe slugs are self-clearing as the user works through them.
