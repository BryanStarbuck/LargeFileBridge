# LargeFileBridge ‚Äî OCR (read the text out of images & videos)

`LargeFileBridge` can **read the text that is visible in the pixels** of an **image** or a **video** and
store it so those words become **searchable, greppable, and re-findable without opening the file**. A
screenshot of a terminal, a slide deck exported to MP4, a photo of a whiteboard, a receipt, a news chyron,
a street sign, burned-in subtitles ‚Äî all of it is text the user can *see* but that no `grep` can reach,
because it is pixels, not characters. OCR turns those pixels back into characters.

This is the **third transaction type**, and the reason it exists as its own transaction rather than a corner
of an existing one is that it answers a **different question** from either sibling:

| Transaction | The question it answers | Input kinds | Output |
| --- | --- | --- | --- |
| **Transcription** ([Transcribe.mdx](./Transcribe.mdx)) | *"What was **said**?"* | audio + video | Speech ‚Üí text, verbatim |
| **AI description** ([ai_description.mdx](./ai_description.mdx)) | *"What is **seen**?"* | image + video | A vision model's **prose** about the frame |
| **OCR** (this spec) | *"What does it **say on screen**?"* | image + video + **PDF** | The **literal glyphs**, verbatim, with positions |

OCR has the **widest** input axis of the three, and PDF is the reason (¬ß1.7.1). A document is a stack of pages,
each of which is just pixels once rendered ‚Äî so a scanned contract, a signed agreement, or a slide export all
become searchable text the same way a screenshot does. This is where OCR stops being the mirror of AI
description (image + video) and becomes a superset: **image + video + PDF.**

Implemented in code: `ocrKindFor()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/ocr.service.ts` returns `image` / `video` / `pdf` (else null); the PDF page-render path is `renderPdfPages()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/pdf.ts`.

The distinction is not academic. AI description **paraphrases** and may *mention* that a sign exists; OCR
**quotes** the sign, exactly, and can be searched for. A slide deck video described by Gemini yields "a
slide listing quarterly figures"; the same video OCR'd yields the actual figures. They compose ‚Äî they do not
substitute.

Two things make this feature worth pinning down: **the two-speed rule** (¬ß2 ‚Äî image = slow/accurate, video =
fast at a 15-second stride, which is the whole performance story; a **PDF page reads accurate**, like an
image, because a page is a document and there are far fewer pages than a video has frames), and **the engine
being a library rather than a `brew` tool** (¬ß3 ‚Äî the explicit product directive, and the thing that keeps
OCR installable, offline, and charter-clean). One deliberate exception rides alongside the engine directive:
**PDF and video both lean on an external command** (poppler's `pdftoppm` to rasterize pages; ffmpeg to sample
frames) ‚Äî the recognizer is still a library, but a page/frame has to be produced first, and that is a
subprocess we detect on PATH and state the absence of honestly (¬ß6).

Siblings this builds on and does not repeat:

* [Transcribe.mdx](./Transcribe.mdx) ‚Äî the **template** for every media-analysis surface: the entry points at
  every scale (¬ß2), the tracking-area artifact placement (¬ß3), the readiness gate, the calc engine. OCR
  mirrors its shape wholesale.
* [ai_description.mdx](./ai_description.mdx) ‚Äî the **closest** sibling (same input kinds: image + video). Its
  **¬ß0 symmetry contract** is the rule this spec extends from a **pair** to a **triple** (¬ß0).
* [artifact_placement_policy.mdx ¬ß0](./artifact_placement_policy.mdx) ‚Äî the **owner** of the storage-KIND
  rule (`.lfbridge/` only in a working git repo; an SDL's root **is** the tracking area). This spec applies
  it identically to `.ocr` (¬ß5).
* [parallelization.mdx](./parallelization.mdx) ‚Äî the **owner** of the Core Budget. OCR is a **local-CPU**
  consumer and draws the **mass-compute** budget, which is the deliberate asymmetry against describe (¬ß10).
* [dialogs.mdx ¬ß5](./dialogs.mdx) ‚Äî the **owner** of the unified batch-confirm popup. Every OCR producer
  opens **that one popup** (¬ß9).

**The two deliberate asymmetries** ‚Äî and they are the whole personality of this feature:

1. **OCR is LOCAL.** Transcription is local (an ASR model); AI description is the app's *one* deliberate
   network upload; **OCR is local like transcription and, unlike description, needs no key, no provider
   matrix, and no credentials popup** (¬ß4). The whole `ai_credentials.mdx` axis is simply **absent** here ‚Äî
   there is nothing to configure and nothing to leak.
2. **OCR is TWO-SPEED, keyed on media kind** (¬ß2). Neither sibling has this: transcription runs one engine at
   one quality, description sends one prompt per kind. OCR deliberately runs a **different recognition
   level** for an image than for a video frame, because the workloads differ by three orders of magnitude.

The symmetry is asserted in code by the parallel module trees `modules/transcribe/` ‚Üî `modules/describe/` ‚Üî
**`modules/ocr/`**, `lib/transcribe.ts` ‚Üî `lib/describe.ts` ‚Üî **`lib/ocr.ts`**, `TranscribeStatusIcon.tsx` ‚Üî
`DescribeStatusIcon.tsx` ‚Üî **`OcrStatusIcon.tsx`**, and the paired fields on
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` (`TaskMetrics`, `FileRow`,
`OcrResult`/`OcrBatchResult` mirroring the describe/transcribe pairs).

## 1. Use cases

The feature earns its place through concrete, recurring jobs on the product owner's own tree. Each of these
is a case where the words are **visible but unsearchable** today.

### 1.1 "Which screenshot had that error message?"

`~/screenshots/` holds hundreds of PNGs. The user remembers a stack trace mentioning `ECONNREFUSED 8787`
but not which image it was in. Today: open them one by one. With OCR: every screenshot carries a `.ocr`
artifact, and one search across the tree lands on the file. This is the **highest-frequency** case, and it is
pure **image / slow-accurate** work (¬ß2.1, ¬ß16).

### 1.2 "Find the slide that showed the pricing tier"

A 40-minute recorded deck (`.mp4`). The words live on the slides, not in the narration ‚Äî so the
**transcript can't find them** (the presenter said "as you can see here"). OCR at a 15-second stride catches
every slide (a slide is on screen far longer than 15s) and timecodes it, so the user gets *"03:15 ‚Äî Creator
$29/mo‚Ä¶"* and can seek straight there. This is the case the **15-second stride was chosen for** (¬ß2.2.1).

### 1.3 The investigation tree ‚Äî chyrons, exhibits, and documents-as-images

The Charlie Kirk / Intel / UAP investigation repos hold thousands of clips and screenshots whose evidentiary
content is **on-screen text**: a news lower-third naming a person, a court exhibit photographed rather than
scanned, a document screenshotted from a PDF. OCR makes that corpus **greppable**, which is the difference
between an archive and a research tool. It composes with the sibling artifacts: the `.transcription` gives
what was said, the `.ocr` gives what the chyron claimed, and the `.ai_description` gives the scene.

### 1.4 Receipts, statements, and the Medical LLC folder

Photographed receipts and statements are images of text. OCR gives a searchable text layer beside each one
without any upload ‚Äî which matters, because these are **financial documents** and the charter's local-only
posture (¬ß4) is the reason this use case is permissible at all. AI-describing a receipt would upload it to a
provider; **OCR never does**.

### 1.5 Burned-in subtitles on a downloaded clip

A clip with hardcoded subtitles and no audio track (or non-English audio). Transcription yields nothing
useful; OCR at the 15-second stride recovers the subtitle text. The **dedupe rule** (¬ß2.2.3) matters here ‚Äî
a subtitle persists across many frames and must not be recorded a dozen times.

### 1.6 "Is this file worth keeping?"

The rollup case. Before compressing or pinning a directory of 300 clips, the user wants to know **what is in
them**. OCR is the cheapest of the three transactions (no network, no model download in the primary path),
so it is often the **first** one run over a new tree ‚Äî it costs nothing but CPU and answers "what is this?"
for anything screen-recorded.

### 1.7.1 The third kind ‚Äî PDF (a stack of pages we rasterize)

A PDF is not media (no player, no IPFS payload) but it is a **first-class OCR target**, and for the common
case it is the **most valuable** one: legal contracts, signed agreements, invoices, and scanned documents are
overwhelmingly PDFs, and "read the text out of these PDFs" is exactly what a user in a folder of them wants.

* **How it works.** A PDF is a stack of **pages**. Each page is rendered to a PNG (poppler's `pdftoppm`, at
  150 DPI ‚Äî crisp enough for body text without paying 4√ó the pixels of a 300-DPI render), and each page image
  is read at the **accurate** level ‚Äî the same pass a standalone image gets (¬ß2.1), because a page IS a
  document and accuracy is the whole point. The per-page texts are concatenated (blank-line separated) into
  the searchable `text`, and every block is tagged with its **1-based page** so a hit can name its page (¬ß5.1).
* **Bounded + honest.** `max_pages` (default **500**) caps a pathological document exactly as `max_frames`
  caps a pathological video; when it bites, the artifact records `truncated: true` and the UI **says** so
  (¬ß15.2 rule 7). The artifact also records `page_count` (from `pdfinfo`) and `pages_read`.
* **Concurrency.** A 200-page PDF is itself a batch; its page recognitions draw the **one shared recognition
  limiter** (¬ß10.2), never a fresh per-document budget ‚Äî the same rule the video frame path obeys.
* **The tool.** Rasterization needs poppler's `pdftoppm` on PATH ‚Äî the **same posture** as the video path's
  ffmpeg (¬ß6). Without it, every image still OCRs fine and every PDF reports `needs_pdf_tools` with a
  `brew install poppler` hint. Empty-text is still a SUCCESS (a blank page contributes nothing, ¬ß2.3).

Implemented in code: `ocrPdf()` and the `needs_pdf_tools` gate in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/ocr.service.ts`; page rendering +
`pdfToolsPresent()` / `pdfPageCount()` in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/pdf.ts`.

### 1.7 The negative case ‚Äî what OCR is NOT for

* **Not audio.** An `.mp3` has no pixels. The OCR tab does not appear for audio (¬ß7) and audio files are
  never candidates (¬ß14).
* **Not other documents.** Only **PDF** is the document OCR kind today (¬ß1.7.1). `.docx` / `.pptx` / `.txt`
  are not rasterized ‚Äî a `.txt` already IS text, and Office formats are out of scope until asked for.
* **Not a description.** OCR on a photo of a beach returns nothing (there is no text) ‚Äî and that is a
  **correct, successful** result, not a failure (¬ß2.3). A file with `text: ""` is **OCR'd**; it simply has no
  text. It must never be re-offered as "not yet OCR'd" (¬ß12.5).
* **Not a reporting service.** Like the perceptual-fingerprint feature, OCR reads the user's own files
  locally and tells **only the user** (¬ß4).

## 2. The two-speed rule (LOCKED ‚Äî the core of this spec)

**The recognition speed is keyed on the media kind, and it is not negotiable:**

| Kind | Pass | Why |
| --- | --- | --- |
| **Image** | **SLOW / ACCURATE** ‚Äî `.accurate` recognition level, language correction **on** | One file, one pass. The user is asking about *this* image; accuracy is worth every millisecond. A 4K screenshot costs a few hundred ms even at the slow setting ‚Äî there is nothing to save. |
| **Video** | **FAST** ‚Äî `.fast` recognition level, language correction **off**, one frame every **15 s** | A 40-minute video at a 15s stride is **160 frames**. At the slow level that is minutes per file; at the fast level it is seconds. Multiply by a 2,000-file tree and the choice is the difference between a feature and an unusable one. |
| **PDF page** | **SLOW / ACCURATE** ‚Äî `.accurate`, language correction **on** ‚Äî like an image (¬ß1.7.1) | A page is a document, and the reader wants every clause right ‚Äî accuracy is the whole point. A PDF also has **far fewer pages** than a video has frames (tens, not 160), so the accurate pass stays affordable, bounded by `max_pages`. |

This mapping is **exactly** what Apple's Vision framework exposes as `VNRequestTextRecognitionLevel.accurate`
vs `.fast` (¬ß3.2), which is a large part of why that engine was chosen: our product rule and the engine's
native knob are the **same knob**.

### 2.1 The image path ‚Äî slow, accurate, whole-file

An image is decoded once and handed to the engine at the **accurate** level with **language correction on**
(the engine may consult a lexicon to fix `rn` ‚Üí `m` and similar confusions). No striding, no sampling, no
frame extraction, no ffmpeg. The result is a **single** text block plus per-observation bounding boxes and
confidences (¬ß5.1). Detail: ¬ß16.

### 2.2 The video path ‚Äî advance 15 seconds, grab a frame, fast-OCR it

A video is **sampled**, never fully decoded frame-by-frame. The pipeline:

1. **Probe** the clip for duration (`ffprobe`).
2. **Extract** one frame every **15 seconds** in **one** ffmpeg pass ‚Äî `-vf fps=1/15` ‚Äî writing numbered
   JPEGs into a **temporary** directory under the state dir. One pass, not one `-ss` seek per frame: a
   single sequential decode is dramatically cheaper than N seeks, and the frames come out already strided.
3. **Fast-OCR each frame**, in parallel (¬ß10.2), tagging each result with its **timecode** (frame index √ó 15s).
4. **Dedupe** consecutive identical text (¬ß2.2.3).
5. **Delete the temp frames** in a `finally`. **The user's original video is never modified or moved** ‚Äî
   this is a read-only sampling pass. (Same posture as describe's compress-to-fit temp copy,
   [ai_description.mdx ¬ß3.3](./ai_description.mdx).)

Detail and the throughput numbers: ¬ß15.

#### 2.2.1 Why 15 seconds (LOCKED)

The stride is the product decision, and it is chosen against **how long on-screen text stays on screen**:

* A **slide** lives 30 s‚Äì5 min. A **chyron** lives 10‚Äì30 s. A **title card** lives 3‚Äì8 s. A **subtitle** lives
  2‚Äì5 s.
* At 15 s we catch **every slide and effectively every chyron** ‚Äî the ¬ß1.2 and ¬ß1.3 cases, which are the ones
  the feature exists for ‚Äî while sampling **0.07%** of a 30fps stream. Halving it to 7.5 s **doubles the cost**
  to buy mostly title cards; going to 30 s starts dropping chyrons.
* We accept, explicitly, that a **fast-moving subtitle track may be sampled rather than captured whole**
  (¬ß1.5). The user who needs every subtitle wants the `subtitle_stride` override, not a different default.

The stride is a **setting** (`ocr.video_stride_seconds`, default **15**) so a user with a subtitle-heavy tree
can tighten it (¬ß17), but **15 is the default and the number every UI states**.

#### 2.2.2 The first frame is never frame 0 (LOCKED)

`fps=1/15` emits its first frame at **t=0**, which on a huge share of real videos is **black, a fade-in, or a
platform splash** ‚Äî i.e. reliably the **worst** frame in the clip. The extraction therefore offsets the
sampling grid by **half a stride** (`-ss 7.5` equivalent, via the filter's phase), so frames land at
**7.5 s, 22.5 s, 37.5 s‚Ä¶** ‚Äî the **middle** of each 15-second window, where the content of that window
actually is. The count of frames is unchanged; only the phase moves.

**A clip shorter than one stride has exactly ONE window ‚Äî itself.** The half-stride phase must therefore be
**clamped to the clip's own midpoint** (`duration/2`) and the `fps` grid dropped for a single frame. Applying
the fixed `-ss 7.5` to a sub-stride clip seeks at or past its end, **zero frames** reach the encoder, and
ffmpeg fails opening mjpeg at EOF with `-22 (Invalid argument)` ‚Äî a message that names neither the length nor
the cause. This is not hypothetical: it failed **36 of 1,779** files in the 2026-07-16 batch ‚Äî every video
under 15 s in the tree, and *only* those. The rule above is unchanged for anything ‚â• one stride; this is the
same intent ("sample the middle of the window, never t=0") applied honestly to a clip that has one window.

Phases the grid, clamps sub-stride clips, and rejects a zero-frame extraction rather than laundering it into
an empty-but-`done` artifact, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/frames.ts` ‚Üí `extractFrames()`.

#### 2.2.3 Consecutive-duplicate frames collapse (LOCKED)

A slide on screen for 3 minutes produces **12 identical frames** ‚Üí 12 identical text blocks. Recording all 12
would bloat the artifact, wreck the timecode index, and make the text tab unreadable. So consecutive
observations whose **normalized text is identical** (whitespace-collapsed, case-sensitive) **collapse into one
entry with a time RANGE**:

```yaml
- start: 180.0
  end: 375.0        # the same slide across 13 samples ‚Üí ONE entry, 3m15s wide
  text: "Q3 Revenue ‚Äî $4.2M ‚Ä¶"
```

Non-adjacent repeats do **not** collapse (a slide returned to later is a genuinely separate appearance, and
the user searching for it wants both timecodes). The comparison is on **normalized** text so that a
one-character OCR wobble between two samples of the same static slide does not defeat the collapse ‚Äî a
**similarity threshold** (‚â•95% by Levenshtein ratio) is used rather than strict equality, because two
fast-level passes over the *same* pixels can and do differ by a character.

### 2.3 Empty is a SUCCESS, not a failure (LOCKED)

Most images have **no text**. A beach photo OCRs to `""`. This is the **correct** and **successful** outcome
and it must be recorded as one: `status: done`, `text: ""`, `blocks: []`. Getting this wrong is the bug that
would make the feature unusable at scale, because:

* If empty read back as "no artifact yet", **every text-free file in the tree would be re-offered forever** ‚Äî
  the popup would say "2,000 files can be OCR'd" the morning after OCR'ing 2,000 files (the exact class of
  defect [ai_description.mdx ¬ß12.5](./ai_description.mdx) locks closed for descriptions).
* If empty read back as **`failed`**, the Processing page would show a wall of red for a run in which nothing
  went wrong.

So: **the artifact's existence with `status: done` is the truth**, and `text` may legitimately be empty.
Only a real fault (engine crash, unreadable file, ffmpeg missing on a video) is `failed`. The presence
indicator (¬ß5.2) and the `ocred` metric (¬ß12) both key on the **artifact**, never on the text being non-empty.

## 3. The engine ‚Äî a LIBRARY, not a `brew install` (LOCKED, with the research)

**The product directive is explicit:** *"We actually want to have it where it's a library our web app uses,
not a `brew install` tool."* This is a real constraint with real consequences, and it is what ¬ß3.1's research
was conducted against.

**Why it matters.** Transcription's engines (`qwen3-asr` via pipx, `whisper` via brew) taught us the cost of
a **provisioned tool**: a consent dialog, a `withModelReady` gate, a tools-preflight page, a "Re-check"
button, a detection routine that must never report an installed tool missing
([Transcribe.mdx ¬ß5.2.1](./Transcribe.mdx)), and a support surface where a user's broken PATH becomes our
bug. **A library has none of that.** It arrives with `pnpm install`, it is version-pinned in the lockfile, it
cannot be half-installed, and it works on a fresh clone with zero user action. For a feature we want run over
**every image in the tree**, that difference decides adoption.

### 3.1 The research ‚Äî candidates evaluated

### 3.2 The decision (LOCKED) ‚Äî Vision primary, Tesseract fallback

**Primary: `mac-ocr` / Apple Vision.** Every developer machine and every deployment target of this app today
is a Mac (the product owner's tower, the laptop, the Mac Studio "CLAW" server). Vision is **state-of-the-art
accuracy, free, on-device, requires no model download, is maintained by the OS vendor, and is
hardware-accelerated on Apple Silicon** ‚Äî and, decisively, its **`.fast` / `.accurate` recognition levels are
precisely the two speeds ¬ß2 requires**. We do not have to build a speed abstraction; we have to *expose* one.

* **Image ‚Üí accurate** (`usesLanguageCorrection` on) ¬∑ **Video frame ‚Üí `--fast`** (correction off).
* Results carry **bounding boxes + confidence** per observation, which is what fills `blocks[]` (¬ß5.1).
* Languages are BCP-47 (`-l en-US -l ja-JP`), queryable per macOS version; the default follows the app locale
  and is overridable (¬ß17).

**Fallback: `tesseract.js`.** Keeps the feature alive on Linux/Windows (the charter's *"server-side later"*
posture) and on a Mac where the Vision binary won't run. Selection is **automatic**, mirroring transcription's
qwen‚Üíwhisper auto-fallback ([transcribe_engine.mdx](./transcribe_engine.mdx)); the resolved engine is
**recorded in the artifact** (¬ß5.1) so a file OCR'd by the weaker engine can be identified and re-run later.

**The engine set is a pluggable adapter table**, exactly like describe's provider adapters ‚Äî so adding
RapidOCR later is a new row, not a refactor.

The engine adapters, per-kind speed selection, availability probe, and auto-fallback are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/engines.ts` ‚Äî `ENGINES` (`vision` / `tesseract`), `selectEngine(kind, requested)`, `engineStatus()`, `recognize(imagePath, { level })`.

### 3.3 The one provisioning caveat ‚Äî Tesseract's language data must be VENDORED (LOCKED)

`mac-ocr` needs **nothing**: the binary ships in the package and Vision's models are part of macOS. That is
the whole point.

**`tesseract.js` is the exception, and it is a charter problem if ignored.** Out of the box it **fetches
`<lang>.traineddata` from a CDN on first use** ‚Äî meaning the *fallback* engine would make LFBridge **phone
home**, silently, on a feature whose entire premise is that it does not (¬ß4). Therefore:

* The language data for the **default language set** is **vendored into the package** and `langPath` /
  `corePath` are pointed at those **local files** (the `tesseract.js-offline` pattern). The fallback engine
  **must work with the network unplugged**, and that is an acceptance criterion (¬ß18.4).
* If a user requests a language whose data is **not vendored**, that is a **permissioned provisioning
  download** ‚Äî it reuses transcription's consent flow (`withModelReady` / the model-provision service,
  [transcribe_engine.mdx ¬ß3](./transcribe_engine.mdx)), surfacing a **consent dialog** that names the download.
  It is **never** silent, and never a side effect of clicking "OCR‚Ä¶".

Vendoring the traineddata and pinning `langPath`/`corePath` to local paths (so the fallback engine never reaches the network) are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ocr/engines.ts` ‚Äî the `tesseract` adapter's `createWorker(lang, oem, { langPath: VENDORED_LANG_DIR, corePath: VENDORED_CORE_DIR, gzip: false })`; data files under `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/assets/tessdata/`.

## 4. The charter posture ‚Äî OCR NEVER uploads (LOCKED)

[ai_description.mdx ¬ß7](./ai_description.mdx) carries a prominent warning: describing a file is *"the ONE
deliberate external-network path in LFBridge."* **OCR is not a second one, and must never become one.**

* **No bytes leave the machine.** Vision recognizes on-device; the vendored Tesseract runs in-process on WASM
  (¬ß3.3). There is **no** provider, **no** endpoint, **no** API key, **no** `ai_credentials` surface, and
  **no** credentials popup anywhere in this feature.
* **Therefore OCR needs no "explicit user action for privacy reasons."** It still only runs when clicked
  (menus.mdx ¬ß6.1's never-act-without-a-click guardrail is universal), but the *reason* is different: describe
  is click-gated because it **uploads**; OCR is click-gated because it **costs CPU**. This distinction is why
  OCR is a candidate for the To-Do batch flow with a far lower cost/safety bar than describe
  ([describe_calc_engine.mdx ¬ß5](./describe_calc_engine.mdx) vs ¬ß14.3).
* **It aligns with the perceptual-fingerprint rule.** The charter demands the fingerprint feature *"run
  entirely locally‚Ä¶ must NOT phone home‚Ä¶ we are not a reporting service."* OCR sits on **that** side of the
  line, not description's. A user's receipts (¬ß1.4) and investigation exhibits (¬ß1.3) are OCR'd with **no
  disclosure of any kind to anyone**.
* **Any change that introduces a network call to this module is a charter violation** and must be caught in
  review. This spec is the place that says so.

The `.ocr` token is a single constant ‚Äî **`OCR_EXT`** ‚Äî **appended**, never substituting the original


## 6. Engine readiness ‚Äî `withOcrReady` (LOCKED)

The third column of ¬ß0's readiness row. Transcription gates on a **model download**; description gates on a
**provider key**; **OCR gates on almost nothing** ‚Äî which is the feature's best property, and the gate exists
only to make the rare failure honest:

* **The common case is ALWAYS ready.** On a Mac, `mac-ocr` is an npm dependency and Vision is in the OS.
  There is **no** setup card, **no** consent dialog, and **no** "Re-check" button in the path the user will
  actually hit. **A ready check that is always true must not cost a dialog.**
* **The two not-ready cases:**
  1. **No engine at all** ‚Äî the Vision binary won't run (non-Mac, or a broken package) **and** the vendored
     Tesseract fallback is unavailable. Surfaces as `no_engine` with an honest reason.
  2. **A non-vendored language was requested** ‚Äî the one permissioned download (¬ß3.3), which reuses
     transcription's consent dialog and **never** proceeds silently.
* **Video adds one tool dependency: ffmpeg/ffprobe** (frame extraction, ¬ß2.2). When a **video** is OCR'd on a
  machine without ffmpeg, the run **`failed`**s with the install hint (`brew install ffmpeg`) ‚Äî the same
  honest-outcome posture describe takes for its compress-to-fit ([ai_description.mdx ¬ß3.3](./ai_description.mdx)).
  **Images need no external tool at all.** Note the asymmetry this creates and do not paper over it: an
  ffmpeg-less machine OCRs every image fine and every video not at all.
* **PDF adds one tool dependency: poppler's `pdftoppm`** (page rasterization, ¬ß1.7.1) ‚Äî the **exact same
  shape** as the video/ffmpeg dependency. When a **PDF** is OCR'd on a machine without poppler, the run
  reports **`needs_pdf_tools`** with the install hint (`brew install poppler`). Same stated asymmetry: a
  poppler-less machine OCRs every image fine and every PDF not at all. `pdfToolsPresent()` is surfaced on
  `GET /api/ocr/engines` (`OcrEnginesStatus.pdfToolsPresent`) beside `videoToolsPresent`.
* **Every entry point gates identically ‚Äî no bypass** (the rule [Transcribe.mdx ¬ß5.2.2](./Transcribe.mdx)
  locks): the viewer button, the ‚ãÆ item, the page action, the popup Apply, and the queue worker all route
  through the same gate. There is no path that reaches the engine without it.

## 10. Parallelization (LOCKED)

**The headline: OCR is LOCAL CPU work, so it draws the MASS-COMPUTE core budget** ‚Äî the opposite of describe.
[parallelization.mdx ¬ß3](./parallelization.mdx)'s map gains a row, and getting it right matters because OCR
is the transaction most likely to be run over an entire tree (¬ß1.6).

### 10.1 Which budget, and why (LOCKED)

[ai_description.mdx ¬ß12.6](./ai_description.mdx) fans describe **~24-wide** on one premise: *"the machine
sits idle while the provider thinks"* ‚Äî parallelism is free because the work is a **network round-trip**.
**That premise is false for OCR.** Vision and Tesseract burn **local cores**; nothing waits on a network
(¬ß4). Fanning OCR out 24-wide would be exactly the `cores¬≤` oversubscription
[parallelization.mdx ¬ß2](./parallelization.mdx) exists to prevent.

So OCR draws the **mass-compute budget** ‚Äî `round(cores √ó max_core_fraction)`, default **90%** ‚Äî the same
budget compression and batch transcription use. It is *"pure batch CPU work the user explicitly kicked off and
is watching drain in the background"*: the textbook mass-compute consumer.

| Operation | Budget | Fan-out shape | Why |
| --- | --- | --- | --- |
| **OCR ‚Äî images** | **mass-compute** | **wide**: `budget` jobs, **1 thread each** | One recognition ‚âà one core. Textbook single-threaded-tool ‚Üí fan wide (¬ß2 of parallelization.mdx). |
| **OCR ‚Äî video frames** | **mass-compute** | **wide over frames**, inside a **core-budgeted** extraction gate (¬ß10.3) | The recognitions are wide; the **ffmpeg extraction** is not (¬ß10.3). |
| **OCR ‚Äî frame extraction (ffmpeg)** | **mass-compute** | **narrow**: `floor(budget / videoThreads)`, threads **capped** | ffmpeg is internally multi-threaded ‚Äî the same narrow shape compress-video uses. |

**Anti-oversubscription holds:** `concurrentJobs √ó threadsPerJob ‚âà budget`, never `cores¬≤`.

### 10.2 Two levels of fan-out, one shared limiter

A video is **itself** a batch (160 frames). Naively that is two nested fan-outs ‚Äî `budget` videos √ó `budget`
frames = `cores¬≤`, the classic mistake. So the **frame recognitions and the file-level jobs draw from ONE
shared limiter**, exactly as the pin pass does ([pin_process.mdx ¬ß4](./pin_process.mdx), the shipped
reference for two-level-one-limiter). A single video being OCR'd alone fans its 160 frames across the whole
budget (fast); 20 videos at once share that same budget between them. **The box is never oversubscribed and
never idle.**

### 10.3 The extraction gate ‚Äî the describe ¬ß12.6.1 lesson, applied UP FRONT (LOCKED)

[ai_description.mdx ¬ß12.6.1](./ai_description.mdx) records a defect worth not repeating: a bucket documented
as *"the one bucket where more parallelism is safe even on a small laptop"* turned out to run **up to ~24
concurrent ffmpeg encodes**, because one **conditional CPU branch** (compress-to-fit) hid inside an
"I/O bucket". The rule it locked: **a queue cap may only be called "network-bound" if EVERY path in that job
is network-bound.**

**OCR takes that lesson as a starting condition.** Its video path **always** shells out to ffmpeg ‚Äî the CPU
branch is not conditional, it is the norm. Therefore:

* **Frame extraction acquires the SHARED transcode slot** ‚Äî the same semaphore describe's `fitMediaUnderLimit()`
  holds, sized from `coreBudget() / VIDEO_THREADS_PER_ENCODE`. So a describe transcode and an OCR extraction
  **compete for one budget** rather than each believing it owns the machine.
* **Images never touch the gate.** They have no ffmpeg step, so they cost nothing and wait for nothing ‚Äî the
  common, high-volume case keeps the full wide fan-out.
* **The gate is released before recognition.** The slot covers **only** the ffmpeg pass; the 160 recognitions
  that follow draw the ordinary limiter (¬ß10.2). Holding a transcode slot across recognition would serialize
  the cheap part behind the expensive part's budget.

### 10.4 Async end-to-end ‚Äî never `spawnSync` (LOCKED)

**The single most important implementation rule in this spec.** Memory: *"'Processing page won't load during
~2000 AI descriptions' = event-loop FREEZE: fit-media.ts used spawnSync on the ~24-way describe hot path."*
That is [performance.mdx P-27](./performance.mdx), theme **T3 ‚Äî the backend blocks the Node event loop**, and
[ai_description.mdx ¬ß3.3.1](./ai_description.mdx) locks it closed.

**OCR must be born with it closed:**

* **Every** child process ‚Äî `ffprobe`, `ffmpeg` ‚Äî runs through an **async** runner (`child_process.spawn`,
  awaited, with a hard-kill timeout), **never** `spawnSync`. `ocrOne()` / `extractFrames()` / `probeVideo()`
  are **all `async`** and awaited.
* **Every** file read is `fs.promises` ‚Äî **never** `readFileSync`. A frame JPEG read on a wide fan-out blocks
  the loop exactly as a base64 payload read did.
* **The Tesseract fallback runs in a WORKER, never on the main thread.** `tesseract.js` recognition is a
  multi-second **CPU-bound WASM** call; run in-process it freezes the loop just as thoroughly as `spawnSync`
  ‚Äî with none of the visibility, since there is no child process to blame. It uses the library's
  **scheduler + worker pool** ([tesseract.js workers vs schedulers](https://github.com/naptha/tesseract.js/blob/master/docs/workers_vs_schedulers.md)),
  pool size = the core budget, workers **reused** across files (worker startup dominates a small image's
  cost, and *"each worker uses a high amount of memory, so code should never be able to create an arbitrary
  number of workers"*).

## 15. Performance ‚Äî OCR for VIDEO (LOCKED)

The expensive half, and the reason ¬ß2's fast level exists.

### 15.1 The cost model

For a clip of duration `D` seconds at stride `S` (default 15):

```
frames        = ‚åàD / S‚åâ
cost          = extract(D)                +  frames √ó recognize_fast
                ‚îî ONE sequential ffmpeg      ‚îî ~30‚Äì60 ms/frame (Vision .fast, 1080p, Apple Silicon)
                  decode+scale pass
```

Worked, on a ~10-core Apple Silicon machine:

| Clip | Frames @15s | Extract | Recognize | **Total** |
| --- | --- | --- | --- | --- |
| 5 min | 20 | ~3 s | ~1 s | **~4 s** |
| 40 min (the ¬ß1.2 deck) | 160 | ~20 s | ~6 s | **~26 s** |
| 139 min (the longest real file) | 556 | ~70 s | ~22 s | **~90 s** |

**Extraction dominates ‚Äî by roughly 3√ó.** That single fact drives every rule below. The naive expectation
("OCR is the slow part") is wrong, and optimizing recognition while ignoring the decode would be optimizing
the cheap half.

### 15.2 The rules (all LOCKED)

1. **One sequential pass, not N seeks.** `-vf fps=1/15` in **one** ffmpeg invocation. `N` separate `-ss`
   seeks pay container-open + keyframe-seek + teardown **per frame** and, on a long clip, cost **multiples**
   of a straight decode. **Never loop `-ss`.**
2. **Downscale in the extraction, not after.** Emit frames at **max 1920px wide** (`-vf fps=1/15,scale=‚Ä¶`)
   ‚Äî ffmpeg's scaler is far faster than the recognizer's, and text legible at 4K is legible at 1080p. A 4K
   clip that skips this pays ~4√ó the recognition cost for **zero** accuracy.
3. **`-an` ‚Äî drop audio.** The audio track is decoded for nothing otherwise. Speech is the transcript's job.
4. **JPEG frames at q‚âà3, not PNG.** Frames are ephemeral (¬ß2.2) and PNG encoding is a needless CPU + disk
   cost at 160 frames per file.
5. **The extraction holds the shared transcode slot; the recognitions do not** (¬ß10.3).
6. **`fps=1/15` is a filter, not a sample-and-discard.** ffmpeg decodes every frame regardless (it must ‚Äî
   inter-frame compression), so the win is in **not encoding** 71,999 of 72,000 frames. Do not expect a 15√ó
   speedup from a 15√ó stride; expect the encode side to vanish and the decode side to remain. This is why
   ¬ß2.2.1 does **not** treat stride as a free performance knob: **halving the stride does not double the
   cost** (the decode is already paid) ‚Äî it roughly doubles only the recognition third.
7. **Cap the frame count.** A pathological input (a 10-hour stream) would emit 2,400 frames. `max_frames`
   (default **1,000**) bounds it; when it bites, the artifact records `truncated: true` and the UI **says
   so** ‚Äî a silent cap reads as "we covered it all" when we didn't ([parallelization.mdx](./parallelization.mdx)'s
   no-silent-caps rule).
8. **Delete the temp frames in a `finally`.** 160 JPEGs √ó 2,000 videos is real disk. They live under the
   state dir, never beside the media.

### 15.3 The 2,000-video scenario

The [use_case_ai_description_at_scale.mdx](./use_case_ai_description_at_scale.mdx) tour, applied: 2,000
videos √ó ~26 s ‚âà **14.4 core-hours**, drained at 90% of 10 cores ‚âà **1.6 hours** wall-clock ‚Äî and, critically,
**with the app fully responsive throughout** (¬ß10.4), every file visible on `/processing`, and the poll payload
bounded ([processing.mdx ¬ß5](./processing.mdx)). Compare **describe**'s same-tree run, which was
**provider-rate-limited into an overnight failure cascade** ([ai_description.mdx ¬ß3.5](./ai_description.mdx)):
OCR has **no provider**, so it has **no rate limit, no 429s, no timeouts, and no transient-vs-permanent
problem** (¬ß4). Its only ceiling is the box, and the box is ours.

## 16. Performance ‚Äî OCR for IMAGES (LOCKED)

The cheap, high-volume half ‚Äî and the one that must never be made slow in the name of the video path.

### 16.1 The cost model

```
cost = decode(image) + recognize_accurate     ‚âà 150‚Äì400 ms   (Vision .accurate, a 4K screenshot)
```

**No ffmpeg. No frames. No temp files. No gate.** An image is the simplest path in the app: read, recognize,
write. A directory of 1,204 screenshots (¬ß1.1) at 90% of 10 cores ‚âà **45 seconds** for the whole folder.

### 16.2 The rules (all LOCKED)

1. **Accurate, always** (¬ß2.1). At ~250 ms there is nothing worth saving ‚Äî and the `.fast` level's errors are
   precisely the ones that ruin a *search index*, which is the artifact's entire purpose. **Never** apply the
   video path's fast level to a standalone image.
2. **Language correction ON.** It is what turns `rn`‚Üí`m` and `0`‚Üí`O` confusions into words a user will
   actually search for. On the video path it is **off** (the wrong trade at 160 frames).
3. **Fan WIDE ‚Äî `budget` jobs, 1 thread each** (¬ß10.1). A single recognition is effectively single-threaded;
   this is the textbook wide fan-out.
4. **Do NOT pre-scale images.** Unlike the video path (¬ß15.2 rule 2), a still is a **one-shot** cost and the
   full resolution is exactly what makes small text (a form label, a stack trace) legible. Rule 2 there and
   this rule here **look contradictory and are not**: one is about 160 frames of redundant pixels, the other
   about the single frame the user actually asked about.
5. **A HEIC is an image.** It routes through the same conversion the media pipeline already uses
   (`magick "file.heic[0]"` ‚Äî the primary still only; [images.mdx](./images.mdx)), then OCRs normally.
6. **Reuse the Tesseract worker pool across files** (¬ß10.4). At ~250 ms of work, a per-file worker **spawn**
   would dominate the run.
7. **Empty is the common case and must be CHEAP** (¬ß2.3). A text-free photo does a full recognition and
   writes an artifact with `text: ""`. Do not add a "does this look like it has text?" pre-filter to avoid
   it ‚Äî that heuristic costs about what the recognition costs and would be wrong in the direction that
   matters (skipping a real hit).

## 19. Acceptance criteria

1. **The two speeds (¬ß2):** an **image** OCRs at the **accurate** level with language correction **on**; a
   **video** samples **one frame every 15 s** and OCRs each at the **fast** level with correction **off**. The
   stride is `ocr.video_stride_seconds` (default 15), frames are phased **half a stride** in (never t=0,
   ¬ß2.2.2), and consecutive duplicate text collapses into one **time-ranged** entry at ‚â•95% similarity
   (¬ß2.2.3).
2. **Empty is success (¬ß2.3):** a text-free file yields `status: done`, `text: ""` ‚Äî the ‚úì shows, the status
   icon reads **Done**, the `ocred` metric counts it, and it is **never re-offered**. It is not `failed` and
   not "no artifact".
3. **The engine is a library (¬ß3):** OCR works after `pnpm install` with **no `brew install`**, no Xcode, and
   no user provisioning ‚Äî `mac-ocr`/Vision primary (its `.fast`/`.accurate` levels **are** the two speeds),
   `tesseract.js` fallback, auto-selected, the resolved engine recorded in the artifact.
4. **Local, offline, forever (¬ß4, ¬ß3.3):** OCR makes **zero** network calls. With the network unplugged, both
   engines OCR normally (the Tesseract `traineddata` is **vendored**, `langPath` local). There is no API key,
   no provider matrix, and no credentials popup anywhere in the feature. A non-vendored language is a
   **consented** download, never silent.
5. **Placement (¬ß5):** the artifact is `<trackingBase(root)>/<rel-dir>/<name.ext>.ocr` ‚Äî extension
   **appended**, path-mirrored, inside the **committed** tracking area, resolved through `trackingBaseDir()`
   (a working repo keeps `.lfbridge/`; an **SDL has none** and an SDL's own `.git/` never demotes it). No
   `*.ocr` git-ignore line is injected. The 3-way placement radios govern it identically to `.transcription`.
6. **A third analysis COLUMN, not a tab (¬ß7):** the viewer's band 6 shows **AI description ¬∑ Transcription ¬∑
   Text (OCR)** **side by side, all at once** ‚Äî **no tab strip, no radio buttons, no default-tab rule**
   ([media_viewer.mdx ¬ß6](./media_viewer.mdx) is authoritative; ai_description.mdx ¬ß1's two-tab description is
   **stale**). **audio** ‚Üí Transcription only; **image** ‚Üí AI description + Text; **video** ‚Üí all three, widths
   dividing evenly among the columns shown. Each ‚úì reflects artifact presence **including empty text**. A
   video's OCR rows **seek** on click.
7. **Every entry point exists, with the RIGHT names (¬ß8):** the four producing entry points of
   [dialogs.mdx ¬ß6](./dialogs.mdx) ‚Äî metric tile, header primary, page action-**link**, and ‚ãÆ/right-click ‚Äî
   each carry OCR. The menu item is **Create OCR text** in the **`Create`** group (file **and** directory),
   directly after `Create Transcription(s)` / `Create AI description(s)`. There is **no `Actions ‚ñæ` dropdown**
   (page_actions.mdx ¬ß0) and **no repo-kebab item** (menus.mdx ¬ß5.1 has no producing actions ‚Äî ¬ß8.4). The
   File System (¬ß8.9) and One-Repo (¬ß8.10) inventories are complete and each row resolves as tabulated.
8. **One popup (¬ß9):** every entry point ‚Äî **including the single-file menu item** ‚Äî opens **the** unified
   `WarningPopup`-based batch popup, never a native dialog (grep `window.confirm` = 0), always fed by
   `POST /api/ocr/plan` for scope-walking opens, always **stating** the excluded already-done count via
   `excludedClause()` (omitted at 0), with the **"Filter: Videos / Images"** row and the honest per-kind cost

### 20.4 ¬ß10.2 caught a real bug in this implementation (worth keeping)

The nested fan-out rule is not theoretical ‚Äî **the first cut of this module violated it**, and the spec is
what found it. The queue caps *files* (`ocr:video` = `floor(budget / VIDEO_THREADS)` = 5 on a 22-core box),
and the video path then fanned its *frames* out across `coreBudget()` ‚Äî a fresh 22 **per video**:

```
5 concurrent videos √ó 22 frames each = 110 recognitions on a 22-core box
```

Exactly the `cores¬≤` over-subscription [parallelization.mdx ¬ß2](./parallelization.mdx) calls the load-bearing
invariant. **A per-video `mapLimit(frames, coreBudget())` looks correct in isolation and is wrong in
composition** ‚Äî a per-call budget cannot see the other calls; only shared state can. That is *why* ¬ß10.2 says
"one shared limiter" and not "bound the frames".

Fixed with a module-level semaphore both levels draw from (`withRecognitionSlot()`), the pin pass's
two-level/one-limiter shape ([pin_process.mdx ¬ß4](./pin_process.mdx)). **Verified:** 6 videos √ó 8 frames ‚Äî 48
recognitions wanting to run at once ‚Äî peaked at **exactly 22 = the budget**, never above.
