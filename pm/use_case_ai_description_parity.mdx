# LargeFileBridge — AI-Description Parity Tour (everything transcription does, description does the same)

This is a **scenario walkthrough** organized as a **mirror**. Its single claim: **wherever transcription
shows up in Large File Bridge, AI description shows up the same way** — same tab pattern, same metric tile,
same popup, same blue header primary, same bulk page action and background jobs, same placement radio, same
status icon, and it is the same first-class Category-A content. Each step **names the transcription surface it
mirrors** and proves the description surface behaves identically. Together they prove the parity principle
stated in [ai_description.mdx §1](./ai_description.mdx) and [task_tabs.mdx](./task_tabs.mdx): *"everywhere
this spec says Transcribe, read 'and Describe' too."*

The through-line: **AI description is a first-class sibling of transcription, not a bolt-on.**

One computer, one repo:

* **`studio-tower`** — the user's Mac tower.
* **`photo-lib`** — a repo whose `stills/` folder holds **dozens of big `.png` / `.jpg` images** and a few
  `.mp4` clips. It is the ideal repo to prove description parity because it is **image-heavy** — where
  transcription mostly has nothing to do, description has everything to do.

This spec builds on the siblings and does not repeat them:

* [ai_description.mdx](./ai_description.mdx) — **the feature** (adapters, prompts, placement, the one
  deliberate network path) whose surfaces this tour walks.
* [describe_calc_engine.mdx](./describe_calc_engine.mdx) — the **bulk-discovery engine** for "what could be
  described," the point-for-point mirror of [transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx).
* [Transcribe.mdx](./Transcribe.mdx) — **the mirror feature**; every step below cites the transcription
  surface it parallels.
* [task_tabs.mdx](./task_tabs.mdx) — the per-repo **AI descriptions tab** (§11), the **describable/described
  metric tiles** (§2, §4.5), the **three-state Describe status icon** (§5.2), and the blue **header primary**
  (§1.1).
* [warnings.mdx](./warnings.mdx) — the **educate-and-fix popup** a metric tile opens.
* [placement_radios.mdx](./placement_radios.mdx) / [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)
  — where the `.ai_description` **content** lands (the working repo's committed `.lfbridge/`); LFB's tracking
  state always stays in Local Storage, never moved by producing content.
* [page_actions.mdx](./page_actions.mdx) / [processing.mdx](./processing.mdx) / [job_queue.mdx](./job_queue.mdx)
  — the **Create AI descriptions** folder action and its background jobs.

## 2. STEP — the media viewer's two analysis tabs (mirror: the Transcription tab)

**What the user does.** They open a still `stills/skyline.png` in the media viewer, then a clip
`clips/interview.mp4` ([media_viewer.mdx §6](./media_viewer.mdx)).

**What LFB does — the tab set mirrors transcription by media kind.** The bottom analysis panel shows up to
two tabs, and **which** show is the exact mirror rule ([ai_description.mdx §1](./ai_description.mdx)):

* **`skyline.png` (image)** → **only the AI description tab** (an image has nothing to transcribe — the
  mirror of "audio shows only the Transcription tab").
* **`interview.mp4` (video)** → **both** Transcription and AI description tabs (a video is both audible and
  visual).
* Each tab carries a **✓** when its content already exists — the same presence marker transcription uses.
* The **default-tab rule** is the transcription-authored one: prefer Transcription when it has content, else
  the first tab with content, else the first tab — so a described-but-not-transcribed video still opens on
  real content (its AI-description tab).

**What we double-check (Step 2):**
* An image shows **only** the AI-description tab; a video shows **both**; the ✓ marker lights from an
  existing `.ai_description` exactly as it does from an existing transcript.

**What we double-check (Step 3):**
* The **AI descriptions** tab exists beside Transcribe and re-tunes strip/columns/sort/filter to
  description work (image+video only), mirroring the Transcribe tab's tuning (audio+video).

## 4. STEP — the "Describable" metric tile opens the educate-and-fix popup (mirror: the Transcribable tile)

**What the user does.** On the AI descriptions tab, the metrics strip shows two tiles — **Describable** and
**Described** — mirroring Transcribe's **Transcribable** / **Transcribed**. `photo-lib` has 214 un-described
images, so the **Describable** tile reads a health-tinted **`214`**; the user clicks it.

**What LFB does — the tile IS the warning surface, and it opens the shared popup.** On the One-repo page the
metric tiles **replace the warning banner** ([task_tabs.mdx §2.6](./task_tabs.mdx)) — clicking the
**Describable** tile opens that metric's **educate-and-fix popup** ([warnings.mdx](./warnings.mdx)), the same
educate-and-fix dialog the **Transcribable** tile opens. The popup educates (*"AI descriptions make these
images searchable and re-findable without opening them; generating one uploads the file to your configured
vision provider"* — honest about the one network path, [ai_description.mdx §7](./ai_description.mdx)) and
offers the fix (**Create AI descriptions**, §6). A **Described** count of 0 with nothing describable renders
the reassuring **green-zero** tile, exactly as Transcribe's tiles do.

**What LFB does on Confirm — enqueues, never inline.** Pressing the popup's solid-blue **"Describe N files"**
**enqueues** the checked files onto the background queue (op `describe`) and returns immediately, so each
file appears on the [Processing](./processing.mdx) page — the exact mirror of the Transcribable popup's
Confirm. It must **not** describe the set inline in one HTTP request; an inline Apply surfaces **nothing** on
Processing and dies on a long list or a network blip ([ai_description.mdx §12.4](./ai_description.mdx)).

**What we double-check (Step 4):**
* The **Describable** / **Described** tiles mirror **Transcribable** / **Transcribed**; clicking Describable
  opens the **same** educate-and-fix popup surface (no separate banner), health-tinted when > 0 and
  green-zero when 0.
* The popup's **Confirm enqueues** (op `describe`, background queue) and the files then appear on the
  **Processing** page — never an inline one-request run ([ai_description.mdx §12.4](./ai_description.mdx)).
* **Already-described files are never re-offered or re-processed.** Both the plan preview and the enqueue
  drop any file that already has a `done` description, so a big directory's popup (e.g. the **1,953-file**
  case) lists **only** the not-yet-described files, and confirming can never re-run an existing description
  ([ai_description.mdx §12.5](./ai_description.mdx)).
* **Describe fans wide.** Because describe waits on a network round-trip (no hardware bottleneck), the queue
  runs it at a **generous** concurrency (default ~24, not core-bound) — far wider than transcribe/compress
  ([ai_description.mdx §12.6](./ai_description.mdx), [parallelization.mdx §3](./parallelization.mdx)).
* **A slow open never looks hung.** Opening the popup over a large directory shows an **"Opening window…"**
  spinner while the tree is walked, then swaps to the candidate list ([dialogs.mdx §5.4](./dialogs.mdx)).

## 5. STEP — the blue "Describe ›" header primary (mirror: "Transcribe ›")

**What the user does.** They glance at the One-repo header's **smart primary** button
([task_tabs.mdx §1.1](./task_tabs.mdx), [one_repo.mdx §3.1](./one_repo.mdx)).

**What LFB does — the header primary recommends the top task, and "Describe ›" is one of its faces.** The
header primary is the single smart button that names the **most valuable next action** for the repo. When
the top recommendation is description work, it reads a blue **"Describe ›"** — the exact sibling of
**"Transcribe ›"** and **"Compress ›"**. For an image-heavy repo like `photo-lib` (214 describable stills,
nothing to transcribe), the primary surfaces **"Describe ›"**; clicking it drives the bulk describe (§6).
The three blue primaries share one grammar — same shape, same placement, same click-to-fix — differing only
in which task they name.

**What we double-check (Step 5):**
* The header primary can read blue **"Describe ›"**, mirroring **"Transcribe ›"** / **"Compress ›"**, and
  surfaces it when description is the repo's top task.

## 6. STEP — "Create AI descriptions" on a folder of images (mirror: "Create Transcriptions")

**What the user does.** They select the `stills/` folder (or check a set of rows, or leave nothing checked
to mean "everything on the page") and choose **Create AI descriptions** from the header **`Actions ▾`** menu
([ai_description.mdx §5.1](./ai_description.mdx)) — the mirror of **Create Transcriptions**
([Transcribe.mdx §2.5](./Transcribe.mdx)).

**What LFB does — the bulk-discovery engine, then background jobs (mirror of the transcribe engine).** The
[describe calc engine](./describe_calc_engine.mdx) — the point-for-point mirror of the
[transcribe calc engine](./transcribe_calc_engine.mdx) — resolves the eligible set: every **image/video**
under the scope that has **no `done` `.ai_description` yet**, dropping already-described and non-image/-video
files ([page_actions.mdx §1.2](./page_actions.mdx)). It obeys the three LOCKED page-action rules exactly as
Create Transcriptions does:

* **Scope** — checked rows, else the whole page recursively.
* **Skip-already-done** — already-described + audio/other files dropped before queuing.
* **Background queue** — the eligible remainder is pushed onto the [job queue](./job_queue.mdx) (op
  `describe`, cap 3) and the call **returns immediately**; each file surfaces a `describe` dock card as it
  runs on the [Processing](./processing.mdx) page.

One toast states the count — **"198 files will have their AI descriptions created"** (or *"All 214 already
have descriptions — nothing to do"*) — the mirror of the transcription toast. Because generating a
description **uploads bytes to a provider** ([ai_description.mdx §7](./ai_description.mdx), the one
deliberate network path), the batch is still **always an explicit click** — never automatic — exactly like
transcription's explicit-action posture.

Planning the eligible describe set, skipping already-described, and background-queuing the remainder are
handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `planDescribe()` / `enqueueDescribe()` (mirror of `transcribe.service.ts` `planTranscribe()` / `enqueueTranscribe()`); the page action in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/pageActions.ts` — `createDescriptions()` (mirror of `createTranscriptions()`).

**What we double-check (Step 6):**
* **Create AI descriptions** mirrors **Create Transcriptions**: scope (checked-else-all), skip-already-done,
  background-queued with a count toast, and each file a `describe` dock card on Processing.
* The eligible-set computation comes from the **describe calc engine**, the mirror of the transcribe calc
  engine.

## 7. STEP — the `.ai_description` artifacts land per the placement radio (mirror: transcript placement)

**What LFB does — placement mirrors transcription exactly.** As each `describe` job completes, its
`.ai_description` artifact is written to **wherever the AI-descriptions placement radio points**
([placement_radios.mdx](./placement_radios.mdx)) — the same three targets, resolved by the **same shared
ordered placement rule** as transcripts ([ai_description.mdx §2](./ai_description.mdx),
[Transcribe.mdx §3](./Transcribe.mdx)). With the repo's default (hidden `.lfbridge/`):

```
~/BGit/act3/photo-lib/
├── .lfbridge/
│   └── stills/
│       ├── skyline.png.ai_description       ← path-mirrored, filename + .ai_description APPENDED
│       ├── crowd.jpg.ai_description
│       └── … (198 of them as jobs complete)
└── stills/{skyline.png, crowd.jpg, …}
```

Each artifact is a self-contained `done` YAML record (`source`/`status: done`/`engine`/`provider`/`generated`/`kind`/`description`),
and the Storage table's **`description` indicator** lights up from its presence — the mirror of the
transcript indicator. Had the user set the AI-descriptions radio to **next to the file** or the **company
sync repo**, the artifacts would land there instead, exactly as transcripts do under their own radio —
the two radios are independent ([use_case_placement_switching.mdx §6](./use_case_placement_switching.mdx)).

**What we double-check (Step 7):**
* `.ai_description` artifacts land at the AI-descriptions radio's target (default `.lfbridge/`,
  path-mirrored, ext appended), each a self-contained `done` record lighting the Storage `description`
  indicator — the placement mirror of transcripts.

## 8. STEP — the three-state Describe status icon (mirror: the Transcribe status icon)

**What the user does.** They scan the AI descriptions tab's file table, whose leading status column shows a
**Describe** status icon per row ([task_tabs.mdx §5.2](./task_tabs.mdx)).

**What LFB does — three states, mirroring the Transcribe icon exactly.** The icon echoes the toggle grammar
but reads as a **status**, not a decision ([decision_toggles.mdx](./decision_toggles.mdx)):

| State | Look | Meaning | Transcribe mirror |
| --- | --- | --- | --- |
| **Described** | **filled blue** | a `done` `.ai_description` exists | filled-blue "transcribed" |
| **Describable** | **orange edge / 99%-white** | image/video with no description yet | orange-edge "could be transcribed" |
| **N/A** | **thin light-grey edge** | audio or a non-describable file | grey "not applicable" |

So an image row that has been described shows filled blue; an un-described still shows the orange edge; an
**audio** file (never describable) shows the light-grey N/A — the precise mirror of the Transcribe column,
where a **video/audio** with a transcript is blue, one without is orange-edge, and an **image** (never
transcribable) is grey. The two status columns are the same component parameterized by kind.

**What we double-check (Step 8):**
* The **Describe** status icon renders **blue described · orange-edge describable · grey N/A**, mirroring the
  **Transcribe** status icon; an audio file reads N/A for Describe just as an image reads N/A for Transcribe.

## 9. STEP — an AI description is first-class durable CONTENT (mirror: a transcript is content)

**What the user does.** `photo-lib` is a **pure image repo** — there is **nothing to transcribe**, and the
user never runs a transcription. They simply describe the images (§6).

**What LFB does — a description is Category-A content, exactly like a transcript.** Under the
[data-category placement policy](./artifact_placement_policy.mdx), a `.ai_description` is **Category-A
content** — the same class as a `.transcription`. An image repo can never produce a transcript, so the **AI
description is the only Category-A content it can ever produce**. The moment the **first** `.ai_description`
is written into the repo's committed `.lfbridge/`, the repo behaves identically to a video repo that got its
first transcript ([use_case_churn_free_repo.mdx §9.1](./use_case_churn_free_repo.mdx)):

* The `.ai_description` lands in the repo's committed `.lfbridge/` and travels with the repo via its own git.
* **No tracking state migrates.** `repo_storage.yaml` and the per-file sidecars **stay in Local Storage** at
  `~/T/_large_files_bridge/repos/<repoKey>/` — producing content never moves Category-B tracking into the
  repo. `counts.described` climbs in the **Local-Storage** `repo_storage.yaml` as jobs complete.
* Scans keep re-stamping `repo_storage.yaml` **in Local Storage**, never in the repo.

This is the load-bearing parity claim: **description is a first-class durable content artifact**, not a
lesser one — it is committed into the repo's `.lfbridge/` and travels with the repo exactly as a transcript
does, and it is the *only* Category-A content an image-only repo ever produces. Tracking state stays in Local
Storage regardless, identical to the transcript case.

Writing the first durable **content** artifact — a **transcript or a description** — into the repo's
committed `.lfbridge/` (while leaving Category-B tracking untouched in Local Storage) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `resolveDescriptionPath()` (the Category-A placement radio, mirror of `transcribe.service.ts` — `resolveTranscriptPath()`); the Category-B writers continue to route through `tracking-root.service.ts` — `resolveTrackingRoot()` (Local Storage), unaffected by the content write.

**What we double-check (Step 9):**
* Describing the first image in an image-only repo (no transcript anywhere) writes the `.ai_description` into
  the repo's committed `.lfbridge/` — the only Category-A content the repo can produce — while
  `repo_storage.yaml` and sidecars **stay in Local Storage** (no migration), and `counts.described` ticks in
  the **Local-Storage** `repo_storage.yaml` — identical to a transcript in a video repo.

## 10. The cross-cutting invariants this tour proves

1. **Every transcription surface has a description twin.** Viewer tab, task tab, metric tile, popup, header
   primary, bulk page action, calc engine, placement, status icon, and threshold — each exists for
   description in the same shape it exists for transcription.
2. **Kind decides which twin applies.** Transcription is for the *audible* (audio+video); description is for
   the *visual* (image+video); a video gets both, and each "N/A" state is the mirror of the other's.
3. **Description is a first-class durable content artifact.** It background-queues like transcription, lands
   per its own placement radio like transcription, and is **Category-A content committed into the repo's
   `.lfbridge/` on its own** — the only Category-A content an image-only repo produces — while LFB's tracking
   state always stays in Local Storage.
4. **The one network path is honestly surfaced everywhere.** Every description entry point (tab, tile popup,
   header primary, page action) states that generating uploads the file to a provider — the explicit,
   user-initiated posture transcription shares, and separate from the local-only perceptual fingerprint.
5. **The two are independent where they must be.** Placement radios, status icons, and metric tiles for
   description are set and read separately from transcription — parity of *shape*, independence of *state*.
