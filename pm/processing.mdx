# LargeFileBridge — Background Processing

When the user kicks off heavy background work — a **"Compress inside"** run
([compression.mdx §12](./compression.mdx)), a batch of transcriptions/descriptions
([page_actions.mdx](./page_actions.mdx)) — the work drains **in the background**
([job_queue.mdx](./job_queue.mdx)) while they navigate elsewhere. This spec owns the **three surfaces**
that make that work visible and reachable:

1. **§2 — a conditional "Processing" left-bar item** that appears only while work is running.
2. **§3 — the bottom-left progress window** in the main area (the existing dock), now clickable.
3. **§4 — the Processing page** (`/processing`): **two stacked tables** — the **batches** the user started,
   over **every item** of every batch together.

All three are driven by **one** polled endpoint, `GET /api/progress` (§5), so they never disagree.

Siblings this builds on and does not repeat:

* [processing_batches.mdx](./processing_batches.mdx) — the **batch record**: what a batch is (one popup
  Confirm), its label, its **five-way outcome taxonomy**, and the **rejection rate** math (`rejected /
  answered`) the top table renders. **This file owns the page; that file owns the counting.**
* [webapp.mdx §10–§14](./webapp.mdx) — the **progress dock**, the in-process **job registry**, the
  operation **taxonomy** (verbs per `ProgressKind`), and the poll. §3 here is that dock, enhanced.
* [job_queue.mdx](./job_queue.mdx) — the **background queue** (the `compress` / `transcribe` / `describe` /
  `ocr` ops), the pending backlog count, and the **batch** grouping (`createBatch` / `recordBatchResult`).
* [dialogs.mdx §5–§6](./dialogs.mdx) — the **unified batch-confirm popup** whose blue **"Describe N files"**
  Confirm is the moment a batch is born.
* [left_bar.mdx](./left_bar.mdx) / [left_bar.yaml](./left_bar.yaml) — the nav the **Processing** item joins
  as a runtime-conditional entry.
* [compression.mdx §12](./compression.mdx) — one of the batch **producers**; its per-file error list is shown
  on the Processing page (§4).

## 1. What "processing" means here (LOCKED)

**Background work is "processing"** when either of these is true:

* at least one **job is running** (a registry card — a compress/transcribe/describe/pin/scan in flight,
  [webapp.mdx §12](./webapp.mdx)), **or**
* the **queue has a pending backlog** (tasks waiting to start, [job_queue.mdx §4](./job_queue.mdx)), **or**
* a **batch is still active** (a "Compress inside" run not yet finished, [job_queue.mdx](./job_queue.mdx)).

When **none** of those hold, there is **no processing** — the conditional nav item and the progress window
are both **absent from the DOM** (no empty box, no dead nav row). This is the single predicate every
surface keys off:

```ts
const processing = jobs.length > 0 || queued > 0 || activeBatches > 0;
```

The shared predicate is read from the poll in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/progress/ProgressContext.tsx` — `useProgress()` (exposes `jobs`, `queued`, and the `batches`; a derived `processing`/`isProcessing` boolean is computed from them).

## 2. The conditional "Processing" left-bar item (LOCKED)

The user's words: *"Whenever something is processing in the background … the last item in that top list
will be Processing. It only appears when there's background work doing processing, and then it disappears
when the processing is done."*

* **Position.** It is the **last item of the top nav list** — rendered **after** the static yaml items
  (…Scans, Settings) and **before** the large empty space that runs down to the pinned account slot at the
  bottom. It is **not** in the account slot; it sits at the tail of the scrolling nav list.
* **Conditional.** It renders **only while `processing` is true** (§1) and **vanishes** the moment work
  finishes. It is never shown disabled or greyed — it is simply present or absent.
* **Look.** Same nav-row treatment as every other item ([left_bar.mdx §3](./left_bar.mdx)): black label,
  a lucide icon (a spinning `Loader2` / `RefreshCw` to read as "active"), the active accent tint when the
  user is on `/processing`. It may show a small **count badge** (running + queued) on the right, like the
  IPFS children's pinned counts.
* **Click.** Routes to **`/processing`** (§4).
* **Source-of-truth note.** The static nav items still come from `pm/left_bar.yaml`
  ([left_bar.mdx](./left_bar.mdx)); **Processing** is a **runtime-injected** item (like the dynamic company
  children under Storages), not a static yaml row — it depends on live state, so it is appended by the
  sidebar when `processing` is true. It is **documented** in `left_bar.yaml` as a commented, runtime-only
  entry so the nav's full shape is discoverable in the source of truth.

The conditional Processing row (appended after the yaml nav items, only while work runs, routing to
`/processing`) is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/layout/Sidebar.tsx` — `Sidebar` (reads `useProgress()`; renders the Processing `<Link>` after the `leftBar.navItems` map when `processing` is true, with the active-route tint and the running+queued count).

## 3. The bottom-left progress window (the dock, enhanced)

The user's words: *"It will show a little progress window at the very bottom left of the main area … not
in the left window."* That surface already exists — the **progress dock** ([webapp.mdx §10](./webapp.mdx)):
a fixed region at the **bottom-left of the main area**, offset past the 256px left bar, showing one small
**card per running job** (spinner + "Verb target" + an optional determinate bar) and a **"+ N queued"**
backlog footer, **absent from the DOM while idle**.

This spec adds **two** things: the window is now a **way in** to the Processing page, and it is **capped**
so a mass run can never wall the screen.

* **Clickable.** Clicking anywhere on the dock (a card or the queued footer) **navigates to `/processing`**
  so the small window is the shortcut to the full view. The cards keep their live per-item detail; the
  click is a whole-dock affordance (cursor pointer, subtle hover).
* Everything else about the dock is unchanged and owned by [webapp.mdx §10](./webapp.mdx) — this file does
  not restate the card layout, the offset math, or the poll.

The dock's click-through to the Processing page is wired in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ProgressDock.tsx` — `ProgressDock` (an `onClick` → `navigate({ to: "/processing" })` wrapper around the card stack; `role="button"`, keyboard-activatable).

### 3.1 The card cap — at most 3 live cards + a summary line (LOCKED)

The user's words: *"it shoots all the way up screens … we only wanted to show at least three max of those
progresses on the bottom, not all of them … we like the idea of one of them saying how many there are."*

Now that the queue runs the **mass-compute Core Budget** in parallel ([§3a](#3a-worker-utilization--the-parallelism-read-locked), [job_queue.mdx §3](./job_queue.mdx)), a
16-core box transcodes **~14 files at once**, so a card-per-running-job stacks a **wall of cards** up the
screen. The dock is therefore **capped**:

* **At most `MAX_DOCK_CARDS = 3` live cards** are shown at once — the most recently started running jobs.
* **The overflow collapses into one summary line** — **"+ N more running"** — rendered directly **under**
  the 3 cards (and above the existing **"+ N queued"** backlog footer). So during an 18,000-file run the
  whole dock is **five short rows at most**: 3 job cards, `+ 11 more running`, `+ 17,982 queued` — never
  the full stack.
* The cap is a **display** limit only — nothing about the queue, the batch, or the `workers` read
  ([§3a](#3a-worker-utilization--the-parallelism-read-locked)) changes; every running job is still counted,
  and the **full** running list is one click away on the Processing page (§4.3, the items table).
* This mirrors the two backlog/summary rows the dock already uses (the "+ N queued" footer and sonner's
  default 3-toast cap), so it reads as one consistent house pattern, not a special case.

The 3-card cap + the "+ N more running" summary line is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ProgressDock.tsx` — `ProgressDock` (`MAX_DOCK_CARDS = 3`; `jobs.slice(0, MAX_DOCK_CARDS)` for the cards, `jobs.length − visible.length` → the "+ N more running" line, ordered before the visible cards so the column-reverse stack places it under them and above the "+ N queued" footer).

## 3a. Worker utilization — the parallelism read (LOCKED)

Now that the queue drains **many** jobs at once against the **Core Budget**
([parallelization.mdx](./parallelization.mdx), [job_queue.mdx §3](./job_queue.mdx)) — the mass-compute budget
is `round(cores × max_core_fraction)`, default **0.9** (*use up to ~90% of cores*) — the user should be able
to **see the mass parallelization working**. A single running card no longer implies a single-file crawl; a
16-core box may be chewing through **14 files at once**. This subsection adds a small **utilization read** so
the busyness of the machine is visible, not inferred.

* **The surface.** A compact line **at the top of the Processing page — above the batches table** (§4.0)
  reads how many of the
  budget's slots are **busy** vs. the **budget total**, with the resolved fraction — e.g.
  **"12 / 14 workers busy (~90% of cores)"**. It is a plain read of live state, not a control (the knob that
  sets the budget lives in **Settings** → *Parallelism*, [parallelization.mdx §4](./parallelization.mdx)); it
  is **absent** when nothing is running (`budget` is meaningless with an idle queue).
* **The dock/nav echo (optional).** The same busy/budget figure may ride along as a tiny **`12/14`** badge on
  the bottom-left dock ([§3](#3-the-bottom-left-progress-window-the-dock-enhanced)) or beside the conditional
  **Processing** nav item ([§2](#2-the-conditional-processing-left-bar-item-locked)), so the parallelism read
  is glanceable without opening the page. It reuses the running+queued badge slot already specified in §2 —
  it does not add a second poll.
* **Derived from the one source.** Like every other surface here it comes from the **same** polled
  `GET /api/progress` (§5) — never a second endpoint. `busy` is how many admission slots are occupied across
  all buckets right now; `budget` is the live mass-compute budget the queue admits against
  ([job_queue.mdx §3](./job_queue.mdx)). The **`~90% of cores`** gloss is the resolved
  `performance.max_core_fraction`. When the field is absent (idle), the read simply does not render.

The worker-utilization read (busy/budget of the core budget, with the resolved fraction) is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingPage.tsx` — `ProcessingPage` (reads `workers` from `useProgress()`; renders the "{busy} / {budget} workers busy (~{pct}% of cores)" line atop the page — above the batches table (§4.1) — and passes the same figure to the dock/nav badge). The figure is assembled server-side from the queue's live in-flight slot count and `coreBudget()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/progress/progress.router.ts` — `GET /` (§5), sourcing `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — the in-flight slot count + `coreBudget()`.

## 4. The Processing page (`/processing`) — two stacked tables (LOCKED)

Clicking the nav item or the dock opens the **Processing page** — the full, correctly-described view of
what is being processed. The user's words: *"go to the correctly described screen to see what's being
processed. As a [batch] queue is built up, we want to show the right appropriate UI there."*

The page answers **two different questions**, and the redesign is built on the observation that they are
different:

* **"What did I set running, and how is it going?"** → the **batches table** (§4.1) — **short**, on top,
  **often one row**.
* **"What is happening to my files right now?"** → the **items table** (§4.3) — **long**, beneath,
  **full-height**, holding **every batch's items together**.

The user's words: *"a short table at the top, which is the list of processing batches, often only one, and
then the main list underneath that, which is the items that are being downloaded currently or pending. That
bottom list is all the processing batches together."*

**The AI-descriptions row, worked through** (every figure follows from
[processing_batches.mdx §4](./processing_batches.mdx), and they must always reconcile like this):

* **Settled** = `ok 1,163 + rejected 41 + failed 3` = **1,207** of **1,440** → the bar and **84%**. The
  refusals and the failure **advance** the bar: those files are finished, whatever the verdict.
* **Done** = **1,163** — `ok` alone. The 41 refusals are **not** in it (§4.1's *"the plain success count"*).
* **Rejected** = `41 / (1,163 + 41 answered)` = **3.4% → 3%**. The 3 failures and the 233 not-yet-asked are
  **excluded from both sides** — over `total` this same row would read **2.8%** and would have read ~0% an
  hour ago ([processing_batches.mdx §4.4](./processing_batches.mdx#44-the-rejection-rate--the-denominator-is-answered-never-total-locked)).
* **Problems** = **3** (red). The 41 are **not** problems — nothing is broken; Gemini looked and declined.

### 4.0.1 Why two tables and not the old three regions (LOCKED)

The previous design stacked **Running now** · **Waiting** · **Batches** as three regions, with batches at
the **bottom** as cards. That is now replaced, for three reasons worth recording so it is not re-litigated:

* **The batch is the thing the user started; it belongs on top.** The user clicked a button that said
  *"Describe 1,440 files"*. The first thing they want on this page is **that** run's row. Burying it under
  the item churn put the summary furthest from the eye.
* **Running / Waiting were never separate things — they are one column.** Splitting the same file list
  across two regions by *state* meant a file **moved between regions** as it started, and the user could not
  sort or filter across the whole set. §4.3's table already carried Status as a column and made the split
  redundant; the **Waiting** count survives as the Status facet's `Queued` value and the batch bar's
  remainder.
* **Cards don't compare; rows do.** With three batches running, cards force a vertical scan of prose. The
  user's actual question — *"which one is furthest along, and which one is in trouble?"* — is a
  **column comparison**, which is what a table is for.

### 4.1 The batches table (top, short) (LOCKED)

**One row per batch** ([processing_batches.mdx §1](./processing_batches.mdx) — one popup Confirm = one
batch). It is a [TanStack table](./tables.mdx) like every other table in the product.

* **No facet rail, no search.** This table is **short by construction**; a filter rail over three rows would

  **sort** icon alone is offered.

  the one they came to look at. A **real `defaultSort`** the [chip run](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked)

### 4.1.1 Height — bounded, and it shrinks to fit (LOCKED)

The batches table has **content below it** (the items table), so it takes the **content-below exception** to
the house full-page-height rule ([repos.mdx §3.3.1](./repos.mdx) — *"the table keeps a **bounded** height so
what sits under it stays visible"*): **`fillHeight={false}`**.

But bounded is not enough, because **"often only one"** is the common case:

* The table is **sized to its content up to 6 rows**, then scrolls internally. **One batch = a one-row
  table**, not a six-row box with five rows of dead space. Reserving height for batches that do not exist
  would steal it from the items table — the thing the user is actually watching — in the exact case (one
  batch) that happens most.
* **Zero batches → the table is absent from the DOM entirely** — header and all. Ad-hoc work
  ([processing_batches.mdx §1](./processing_batches.mdx)) still renders in the items table below. This is the
  same **no-empty-chrome** rule the nav item and dock already follow (§6 AC 6): an empty table that says
  *"no batches"* above a table full of running items is a contradiction the user has to resolve.
* The **items table below keeps `fillHeight` and still fills to the viewport bottom** — content **above** a
  table never triggers the exception ([repos.mdx §3.3.1](./repos.mdx)). The page itself **never scrolls**;
  only the two bodies do.

### 4.1.2 The ETA is honest or it is absent (LOCKED)

**Elapsed** is a fact and always shows. **ETA** is a projection, and shows **only** when it can be one:

* It appears once **≥ 10 items have settled** and **≥ 30 s** have elapsed, derived from the **trailing rate**
  (settled per minute over the last 2 minutes), and reads **approximate** — *"~14 min left"*.
* Below those thresholds it is **absent**, not a guess. A batch's first two files say nothing about the next
  1,438 — a 4-second image and a 40-minute video share a queue, and an ETA extrapolated from the image is a
  confident lie. The product would rather show nothing than a number that halves and doubles while the user
  watches it.
* It is **absent** for a finished batch (Elapsed becomes the total runtime) and for a **stopped/halted** one.

### 4.1.3 Clicking a batch sets a VISIBLE facet (LOCKED)

Clicking a batch row — or its ⋮ → *Show only this batch's items* — **sets the items table's Batch facet**
(§4.3.1b) to that batch and nothing more.

This is master/detail done the way this product already does it — exactly the pattern
[one_repo.mdx §4.1](./one_repo.mdx)'s summary segments use ("*clicking a segment **sets the Pin (IPFS)
facet** to that one value*"), and it is required by the LOCKED rule §4.3.1a already states for the
**Failed only** preset:

> *"filtering correctly and **showing** that you filtered are two separate requirements"* — a preset that
> quietly applied its own hidden predicate would leave a **traceless narrowing**.

So the narrowing lands **on the rail as the facet's own chips**, where the user can **see why 1,300 rows
vanished** and undo it by re-checking a box or hitting the facet's **All** link. Clicking the **already-selected**
batch row again returns the facet to all-checked. The facet stays the single source of truth; the click is a
shortcut to a state the user could have set themselves.

A hidden `selectedBatchId` filter would be faster to build and would produce the identical row set — and the
first time a user clicked a batch, walked away, came back, and found the items table "missing" three
quarters of their work with no visible reason, it would cost more than it saved.

### 4.1.4 The toast points here (LOCKED)

The Confirm's LOCKED toast ([page_actions.mdx §2](./page_actions.mdx)) — *"1,440 files will have their AI
descriptions created"* — gains a **"View progress"** action that opens `/processing` **with that batch's

([processing_batches.mdx §7](./processing_batches.mdx)) precisely so the toast can name its own row rather

prose.

### 4.2 The items table (bottom, full-height) — every batch together (LOCKED)

The user's words: *"That bottom list is all the processing batches together."*

The bottom table is **not** scoped to a batch by default. It is **every item of every batch**, plus ad-hoc
work, in one list — because the machine has **one** queue and the user's real question is *"what is my
computer doing?"*, not *"what is batch #2 doing?"*. The **Batch** column tells each row apart; the **Batch
facet** narrows when the user wants that (§4.1.3).

Its states, columns, facets, and responsive behavior are §4.3 — unchanged in structure, extended with the

### 4.3 The per-item processing table — every state, with responsive columns (LOCKED)

This is the page's **bottom table** (§4.2) — **all the batches together**. The top table (§4.1) answers
*"how are my runs going?"*; this one answers, **per file**, *"which ones are running in parallel, which are
pending, which failed, which are underway."* It is a [tables.mdx](./tables.mdx) house table that makes
**every item's state explicit** and, crucially, shows the **parallelism** directly.

### 4.3.1 The six item states (LOCKED)

Every item is in exactly one state — the same
[five-way outcome taxonomy](./processing_batches.mdx#4-the-outcome-taxonomy--five-ways-a-file-ends-locked)
the batch counters aggregate, plus `Running`. **The item rows and the batch counters are two views of one
truth and can never disagree**: each row's state is exactly the bucket its batch counted it into.

| State | Meaning | How it reads |
| --- | --- | --- |
| **Running** (underway, **in parallel**) | Actively being worked **right now**. **Multiple `Running` rows at once = running in parallel** ([transcribe_engine.mdx §5](./transcribe_engine.mdx) — calibrated to the machine). | A determinate **% bar** per row + a spinner; the **worker-utilization read** (§3a) above says how many of the budget's slots are busy (e.g. *"12 / 14 workers busy (~90% of cores)"* — one global read across all buckets, never a per-op one). |
| **Pending** (queued, **not started**) | Admitted to the queue but **waiting for a free worker slot**. | A muted **"Queued"** chip; the backlog shown as **rows** so the user sees exactly what's in line. |
| **Rejected** (**the provider said no**) | The provider **considered the file and refused** — a `RECITATION`/`SAFETY` verdict that **survived every retry**, or a blocked input ([ai_description.mdx §2.3](./ai_description.mdx)). A `.ai_description_rejected` record was written. **It is settled, and it is not a fault.** | A **slate/violet "Rejected"** chip + the provider's reason. **Never red** — nothing is broken; the provider answered. **No retry affordance**: it is settled after four attempts, and *overwrite on the file* is the only way to ask again ([processing_batches.mdx §6](./processing_batches.mdx)). |
| **Halted** (**never attempted**) | The provider's circuit **opened** ([to_fix.mdx §2.4](./to_fix.mdx)), or the user **stopped the batch** ([processing_batches.mdx §6.2](./processing_batches.mdx)). It did **not** error — it was **never tried**. | An **amber "Not attempted"** chip + the reason. **Not red, and never worded as a failure** ([warnings.mdx §2](./warnings.mdx) — nothing is broken). One **banner per circuit** groups them with a one-click **re-queue**; the rows keep their own per-item truth. |
| **Failed** | The item **errored / could not complete** — including a transcription that came back **short of the full duration** ([transcribe_engine.mdx §4](./transcribe_engine.mdx)). | A **red "Failed"** chip + the **reason** (and, for a truncated transcript, the covered-vs-total figure). Failures **persist** through the batch's [retention](./processing_batches.mdx#5-retention--the-row-outlives-the-run-locked) so they stay readable after the run. |
| **Done** (recent) | Completed successfully in this run. | A **green "Done"** chip; recent completions linger briefly so the user sees progress, then age out. |

* **Three non-green outcomes, three colors, three different meanings.** `Rejected` (slate) · `Halted`
  (amber) · `Failed` (red) are the states users and code most often collapse into one — and the whole
  design turns on keeping them apart. **Red means we are broken. Amber means we never tried. Slate means
  the provider looked and declined.** Each implies a different action (fix it · re-queue it free · accept
  it), so one color for all three would tell the user to do the wrong thing two times out of three.

* **Nothing is hidden.** A file that has not started yet is a **Pending row**, not an invisible integer — the
  user can see the whole queue that is pending but not started, the ones underway, and the ones that failed
  out, all at once.
* **`Halted` is never a flavour of `Failed`.** They are **different facts**: a failed item **was attempted**
  and the work is genuinely lost; a halted item **was never sent** and costs nothing to run now. Rendering a
  halted item as "failed" makes the user **re-run work believing it was attempted** — and hides that the
  cheap fix is to re-queue the batch once the provider is healthy. It must therefore be **re-queueable in
  one click** (§4.3.1's banner), not buried in a retry-each-row flow.

  Draining the doomed tasks and marking them `halted` — never `failed` — is in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `haltDoomedTasks()` (splices `pending`, `recordFailure({ state: "halted" })`, `appendTerminal(id, "halted")`), carried on `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` — `FailedItemView.state?: "failed" | "halted"`.

  The amber **"Not attempted"** rendering and the one-banner-per-circuit grouping are in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingItemsTable.tsx` — `ItemState` / `STATE_ORDER` / the state labels; `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/haltedWarning.tsx` — `groupHalted()` / `haltedWarningDef()` (the re-queue action).
* **Sort.** Default **Running → Pending → Failed → Halted → Rejected → Done**, so in-flight work is on top,
  the **actionable** problems (`Failed`, then the free-to-fix `Halted`) sit next where they are found without
  hunting, and the **settled** outcomes (`Rejected`, `Done`) — the ones with nothing to do about them — sit
  below. **This must be a real default sort, not an invisible
  JS pre-sort.** Today the rows are ordered by a `STATE_ORDER` constant before they ever reach the table,
  with **no `defaultSort`** — so the state ordering is a fact the user can neither **see** nor **change**:
  the sort chip run ([tables.mdx §3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked))
  renders nothing, the Status header shows no caret, and clicking a header silently discards the grouping.
  The order above is therefore declared as the table's **default sort key (Status, in the state order)**
  ([tables.mdx §3.4](./tables.mdx#34-the-default-sort--bookmarks-first-locked-default-user-changeable)), so
  it displays honestly as a chip, is user-changeable, and persists per user like every other sort (§4 there).
* **Filter.** Per [tables.mdx §2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked) —
  see §4.3.1a.

#### "Failed only" is a preset that sets the Status facet — not a second filter (LOCKED)

**"Failed only" is fully expressible in the Status facet**: it is exactly *Failed ✓, every other value ✗*.
So it does **not** become a promoted checkbox. **Bookmarked only** earns that exception because it is a
promotion of a **non-column** flag ([tables.mdx §2.7](./tables.mdx#27-the-promoted-bookmarked-only-filter-locked));
a second control that duplicates a facet's own values would give the same predicate two sources of truth
and let the rail contradict the checkbox.

Instead it is a **one-click preset** — a small **`Failed only`** link beside the Status facet, offered
**only when any item failed** — that **sets the Status facet** (leaves only `Failed` checked) and nothing
more. **It does not touch the Batch facet**, so a batch the user already isolated (§4.1.3) stays isolated
and the preset reads as *"the failures **of what I am looking at**"* — the two facets AND together
([tables.mdx §2.1](./tables.mdx)) exactly as the rail shows. The resulting state is then **visible on the
rail** as the facets' own chips, so the user can see
*why* rows vanished and undo it by re-checking a box or hitting the facet's **All** link. This applies
[page_actions.mdx §1.2](./page_actions.mdx#12-rule-2--skip-already-done-for-producing-actions-locked)'s
LOCKED rule to filtering: *"a user … cannot see a filter that left no trace"* — **filtering correctly and
*showing* that you filtered are two separate requirements**. A preset that quietly applied its own hidden
predicate would leave exactly that traceless narrowing; one that sets the facet leaves the trace on the
rail. The facet stays the single source of truth; the preset is just a shortcut to a state the user could
have clicked themselves.

### 4.3.1b The Batch facet — a runtime vocabulary, and why it breaks two house rules on purpose (LOCKED)

The Batch facet is what makes §4.1.3's *"click a batch, see its items"* a **visible** narrowing rather than a
hidden predicate. It is the product's **first facet whose values are not knowable at design time** — they
are the live batches — and that forces **two deliberate, documented departures** from
[tables.mdx §2](./tables.mdx). Both are recorded here so neither reads as an oversight:

* **Departure 1 — `Ad hoc` IS a declared value, though the N/A rule would say otherwise.**
  [tables.mdx §2.4.1](./tables.mdx) says an item holding no value on an axis contributes **no facet value**
  (which is why a `pin` job adds no `Kind` value). By that reading, un-batched work would simply be
  unfilterable on this axis — and would therefore be **stuck on screen forever**, because a row that holds
  no value can never be excluded. That would defeat the facet's entire purpose: clicking a batch would show
  *"that batch's 1,440 items **and** every stray right-click compress"*, which is not what the user asked
  for and not what the row promises. So **"has no batch" is treated as a real, checkable value**, because —
  unlike "this pin has no media kind" — it is a fact the user actively reasons about and needs to both
  isolate and exclude. **The N/A rule holds where N/A is meaningless; here it is meaningful.**
* **Departure 2 — this facet does NOT persist.** Every other facet persists per user as `hidden:`
  ([tables.mdx §4.1](./tables.mdx)). This one is **ephemeral session state**. Persisting it would be
  incoherent: a batch id is **gone** by the next session (§5's retention, and the queue is in-memory), so a
  saved `hidden: [a1b2c3d4]` would be a permanent instruction about a batch that no longer exists, silently
  accumulating dead ids in `config.yaml` forever. The precedent already exists —
  [tables.mdx §6](./tables.mdx) LOCKs that *"tab selection is presentation, not preference"* and is never
  written to config. **A facet over ephemeral values is presentation, not preference.**
* **The vocabulary is never empty.** [tables.mdx §2.8](./tables.mdx) makes an empty facet a bug. With **zero
  batches** the facet has no values to offer, so — like the batches table itself (§4.1.1) — it is **absent
  from the rail entirely**, not rendered empty. With batches present it always has ≥ 1 value.
* **Overflow.** Many concurrent batches are rare, but the rail's standard **`+N more` → popover**
  ([tables.mdx §2.2](./tables.mdx)) handles it with no special case. Values sort **newest batch first**,
  matching the batches table's default sort (§4.1) so the two orders agree.
* **The last-value rule still holds** ([tables.mdx §2.1](./tables.mdx)): unchecking the final checked box is
  disabled — *"At least one value must be shown."*

The Batch facet's runtime vocabulary, its non-persistence, and the `Ad hoc` value are to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingItemsTable.tsx` — `ProcessingItemsTable` (the `id: "batch"` column, its vocabulary derived from `useProgress().batches` + the `Ad hoc` sentinel for `batchId == null`, held in component state and excluded from the `tables.views.<tableId>` persistence write).

The Status facet ships its vocabulary today (`filterOptions: ["Running", "Queued", "Not attempted", "Failed"]` — it still lacks `Rejected`) but as a `<select>`; the Kind facet's values, the rail rendering, the `Failed only` preset, and the real `defaultSort` (replacing the invisible `STATE_ORDER` pre-sort) are to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/processing/ProcessingItemsTable.tsx` — `ProcessingItemsTable` (the `id: "status"` / `id: "kind"` columns and the `STATE_ORDER` row sort), against the new facet-rail region of `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable`.

### 4.4 Performance at scale — thousands of items without a slow page (LOCKED)

A single batch can be **2,000+ files** (the "2,000 AI descriptions" scenario,
[ai_description.mdx §12.7](./ai_description.mdx)). The Processing page must stay **fast to load and smooth to
watch** at that size — it is a monitoring surface, and a monitoring surface that itself janks under load is
useless. Four properties, each LOCKED, guarantee it:

1. **The backend never blocks — so the poll always answers.** The single API server must keep serving
   `GET /api/progress` on its normal cadence even while thousands of jobs drain. That holds only because
   **every producer's heavy work is asynchronous** — the `describe` compress-to-fit transcode + base64 read
   ([ai_description.mdx §3.3.1](./ai_description.mdx)) and the `compress` transcode
   ([job_queue.mdx §3](./job_queue.mdx)) all run via `child_process.spawn` / `fs.promises`, never a
   synchronous `spawnSync`/`readFileSync` that would freeze the event loop
   ([performance.mdx P-27](./performance.mdx)). **This is the property whose absence produced the reported
   "the Processing page would not load" bug** — with a synchronous describe transcode, the poll was stuck
   behind a blocked loop.
2. **The poll payload is bounded, the counts stay exact.** The pending rows (`queuedItems`) and the failure
   rows (`recentFailures`) are **capped** (500 each, §5) so a 2,000- or 20,000-item queue is still a **small,
   fixed-size** JSON body — the rows are the **head of the line**, not the whole backlog. The **exact**
   totals still travel as integers (`queued` / `queuedByOp`), so the per-op backlog read says *"2,000 describing"*
   even though only the first 500 ride as rows. The poll cost is **O(cap)**, not O(N).
3. **The table renders a fixed number of rows.** The per-item table (§4.3) is the house `DataTable`, which
   **windows** its rows — only the ~30 rows intersecting the viewport are in the DOM, regardless of whether
   the page has 50 rows or the 500-row cap ([performance.mdx P-01](./performance.mdx)) — and paginates at
   500. So the render is **fixed-cost**; a big backlog never becomes a 500/2,000-`<tr>` DOM dump.
4. **The batches table costs O(batches), not O(files) — and that is why the counters live on the batch.**
   A 2,000-file batch is **one row of integers**. Every figure the top table shows — progress, the
   [rejection rate](./processing_batches.mdx#44-the-rejection-rate--the-denominator-is-answered-never-total-locked),
   Problems — is read from **counters the queue already incremented** as each task returned
   (`recordBatchResult()`), **never** by scanning item rows. This is load-bearing, not an optimization: the
   item rows are **capped** (500, above), so a rate derived by counting them would be computed over the
   **head of the line** and would be **flatly wrong** for any batch bigger than the cap — reporting the
   rejection rate of the first 500 files as if it were the run's. **The counters are exact at any size
   precisely because they are never derived from the truncated view.**

Together these mean the page **loads instantly and scrolls smoothly** at any batch size, the numbers are
always truthful, and the only thing that grows with the backlog is the integer counts — never the work the
browser or the poll has to do. The poll also **self-accelerates** while busy (§4.5) so the picture stays
live without a heavier idle poll.

The bounded-payload caps are enforced server-side (`listQueuedItems()` / `listRecentFailures()` cap at 500)
in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts`; the
row windowing + 500-default pagination live in the shared
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` (the
`useWindowedRows` windowing, [performance.mdx P-01](./performance.mdx)); the non-blocking backend is the
async producers cited in §5 and [ai_description.mdx §3.3.1](./ai_description.mdx).

### 4.5 An idle page starts moving when work appears elsewhere (LOCKED)

The user's words: *"the page might show that there's nothing progressing because there may be nothing
progressing. Another tab might actually add things … and then that page should be notified by the backend
and then start to move to show progress is happening."*

This is the required behavior, and it already falls out of the **one-source** design (§5) — it must never
regress:

* **The backend queue/registry/batch state is process-global**, held in the single API server
  ([job_queue.mdx](./job_queue.mdx), [webapp.mdx §12](./webapp.mdx)). Work started **anywhere** — another
  **browser tab**, a **"Compress inside"** run enqueued server-side, the **launchd** scan/pin worker
  ([storage_local.mdx §13](./storage_local.mdx)) — lands in that **same** shared state.
* **Every surface polls it** (`GET /api/progress`), so an **idle** Processing page (or the conditional nav
  item, or the dock) **starts moving on its own** within **one poll interval** of work appearing — even
  though *this* tab did nothing. That is the "the backend notifies the page" behavior the user described,
  delivered by the shared state + poll rather than a per-tab push.
* **The poll self-accelerates.** While anything is in flight the interval tightens to **~1.2 s** (from
  ~3 s idle), so a freshly-queued batch shows up in **≤ ~1.2 s** — fast enough to read as immediate, with
  **no** second source of truth to disagree.
* **Latency note (and the seam if instant is ever needed).** Because it is polling, an idle tab lags by up
  to **one interval** (≤ ~3 s) and there is **no** server push to wake a truly-backgrounded tab sooner. If
  instant cross-tab wake is ever required, the seam is a single **SSE** stream on the same payload — but the
  poll is the current, deliberately-simple design and is sufficient for this feature.

The shared-state + polling merge (so another tab's / launchd's work surfaces here) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/progress/ProgressContext.tsx` — `ProgressProvider` (the `useQuery(["progress"], api.progress, { refetchInterval: dockBusy ? 1200 : 3000 })` poll, merged de-duped by job id with the tab's own optimistic `run()` set) against the process-global `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/progress/progress.registry.ts` and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` that every producer writes into.

## 5. The one polled source — `GET /api/progress` (extended)

All three surfaces read the **same** poll ([webapp.mdx §12](./webapp.mdx)); this spec extends its payload
with the two things the Processing page needs beyond the raw running jobs and the single backlog integer:

```ts
GET /api/progress → {
  jobs: ProgressJob[],                                   // running now (registry + active scan) — + batchId?
  queued?: number,                                       // total pending backlog (dock footer)
  queuedByOp?: Partial<Record<ProgressKind, number>>,    // pending split per op (the Status facet's Queued read)
  queuedItems?: QueuedItem[],                            // PENDING items as ROWS (§4.3.1) — { op, path, batchId?, kind?, sizeBytes?, durationSec? }, capped (e.g. 500)
  recentFailures?: FailedItem[],                         // SETTLED-NOT-OK items (§4.3.1) — { op, path, batchId?, reason, state: "failed" | "halted" | "rejected", coveredSec?, durationSec?, at }, capped
  batches?: ProcessingBatch[],                           // the §4.1 batches table (processing_batches.mdx §2)
  workers?: { busy: number; budget: number },            // Core Budget utilization (§3a) — busy/total slots
}
```

* `queuedByOp` gives the per-op backlog read without the page re-counting.
* `queuedItems` are the **pending-but-not-started** items as **rows** for the items table (§4.3): the files
  in line, each with enough to render (op, path, **batchId**, kind, size, duration). **Capped** (e.g. 500) so
  a huge queue stays a light poll — the count (`queued`/`queuedByOp`) is still exact; the rows are the head
  of the line. Absent when nothing is queued.
* `recentFailures` carries **every settled-but-not-`ok`** item + its **reason** — discriminated by
  **`state`** into the three that must never be confused (§4.3.1): **`failed`** (attempted, errored — incl.
  a transcript short of full duration, [transcribe_engine.mdx §4](./transcribe_engine.mdx)), **`halted`**
  (never attempted), and **`rejected`** (the provider refused). **The field name is a legacy wire literal
  and is now a misnomer** — a `rejected` item is not a failure ([processing_batches.mdx §4.2](./processing_batches.mdx))
  — but `state` is the discriminator every surface reads, and renaming a shipped wire field to fix a comment
  would be churn for nothing. **Never infer state from the field name or the reason prose** — that is
  [ai_description.mdx §3.5](./ai_description.mdx)'s LOCKED rule (*"classify on the evidence object, never on
  the message prose"*) applied one layer out: a reason string reading *"…(finishReason: PROHIBITED_CONTENT)"*
  must not make a **timeout** render slate. **Structure decides; text is for reading.**
* `batches` carries the [full batch record](./processing_batches.mdx#2-the-batch-record-locked-shape) —
  `{ batchId, kind, label, scope, provider?, engine?, total, ok, rejected, failed, halted, running, errors[],
  startedAt, finishedAt, stoppedBy?, manifestPath }`. Active ones (`finishedAt: null`) drive the §4.1 bars;
  finished ones persist per [processing_batches.mdx §5](./processing_batches.mdx)'s retention. All
  optional/absent when empty, so an idle poll is tiny.
* **`batchId` rides every item** (`jobs`, `queuedItems`, `recentFailures`) — it is what joins the two tables.
  Absent on ad-hoc work, which is exactly the **Ad hoc** facet value (§4.3.1b). **The join is done client-side
  off the one payload; there is no per-batch item endpoint** — narrowing to a batch (§4.1.3) must never cost
  a fetch, or a click on a row would stall behind the network and the two tables could momentarily disagree.
* `workers` is the **worker-utilization read** (§3a): `busy` = admission slots occupied across all buckets
  right now, `budget` = the live mass-compute Core Budget the queue admits against
  ([job_queue.mdx §3](./job_queue.mdx), [parallelization.mdx](./parallelization.mdx)). Absent when the queue
  is idle, so an idle poll carries no worker figure. The `~90% of cores` gloss is the resolved
  `performance.max_core_fraction`.

The extended payload is assembled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/progress/progress.router.ts` — `GET /` (folds `queueDepthByOp()` → `queuedByOp`, `listBatches()` → `batches`, and the in-flight slot count + `coreBudget()` → `workers` into `ProgressListResult`), reading `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `queueDepthByOp()` / `listBatches()` / the live in-flight slot count (which prunes finished batches past the retention window), against `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts` — `coreBudget()`. The client + provider consume it via `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts` — `progress()` and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/progress/ProgressContext.tsx` — `ProgressProvider` (exposes `batches` and `workers` alongside `jobs`/`queued`).

## 6. Acceptance criteria

1. **Conditional nav item (§2):** a **Processing** item appears as the **last item of the top nav list**
   (after Settings, above the account slot) **only while background work is running** — a running job, a
   pending backlog, or an active batch — and **disappears** when work finishes. Clicking it opens
   `/processing`. It is runtime-injected (not a static yaml row) and documented in `left_bar.yaml`.
2. **Progress window (§3):** the existing bottom-left dock still shows a card per running item + a
   "+ N queued" footer, is absent while idle, and is now **clickable to open `/processing`**.
2a. **Dock card cap (§3.1):** the dock shows **at most 3 live cards**; any further running jobs collapse
   into a single **"+ N more running"** summary line (under the cards, above the "+ N queued" footer), so a
   mass compress-inside run **never walls the screen**. The cap is display-only — the full running list is
   on the Processing page.
3. **Processing page = two stacked tables (§4):** `/processing` shows a **short batches table on top**
   (§4.1) over a **full-height items table** (§4.3). The old three regions (Running now · Waiting ·
   Batches-as-cards) are **gone** (§4.0.1): Running/Waiting are the items table's Status column, and batches
   are **rows on top, not cards at the bottom**. An empty page shows a calm "Nothing is processing right
   now." — not two empty tables.
3a. **The batches table (§4.1):** **one row per batch** = one popup Confirm
   ([processing_batches.mdx §1](./processing_batches.mdx)), minted **at Confirm** (before the first worker,
   §1.1 there) and reading **Batch · Progress · Rejected · Problems · Started · Done · Elapsed/ETA** with a
   **⋮** menu. **Rejected is priority 3** and survives almost any width. Default sort **Started ↓** (a real,
   visible, user-changeable `defaultSort`). Height is **bounded** (`fillHeight={false}` — the content-below
   exception) and **content-sized up to 6 rows**, so **one batch renders a one-row table**, never a six-row
   box of dead space; **zero batches → absent from the DOM** (§4.1.1). **ETA is absent until ≥ 10 settled and
   ≥ 30 s** — never a guess (§4.1.2).
3b. **The rejection read (§4.1, [processing_batches.mdx §4.2/§4.4](./processing_batches.mdx)):** the
   **Rejected** column shows the product owner's number — refusals that **survived every retry** — as
   **count + rate over `answered` (`rejected / (ok + rejected)`)**, **never over `total`** (which would read
   ~2% during a 50%-refusal run) and never over `settled`. **Count only below 20 answered**; **`—`** (not
   `0`) for any op no provider judges (§4.5 there). It is **slate, never red**, it **advances** the progress
   bar (it is settled), and above **20%** it turns **amber with a warning but does not pause the batch** —
   the deliberate opposite of [to_fix.mdx §2.7](./to_fix.mdx)'s transient ceiling.
3c. **Master/detail is a visible facet (§4.1.3):** clicking a batch row (or ⋮ → *Show only this batch's
   items*) **sets the items table's Batch facet** — landing on the **rail** as chips the user can see and
   undo — **never** a hidden `selectedBatchId` predicate. Clicking the selected row again restores
   all-checked. This is [one_repo.mdx §4.1](./one_repo.mdx)'s segment pattern and §4.3.1a's LOCKED rule:
   *"filtering correctly and showing that you filtered are two separate requirements."*
3d. **Per-item table (§4.3):** **every batch's items together** (§4.2), with **every item's state explicit**
   — **Running** (**multiple Running rows = running in parallel**), **Pending** (rows, not just a count),
   **Rejected** (slate — the provider refused), **Not attempted** (amber — halted/stopped), and **Failed**
   (red, with the reason, incl. a transcript **short of the full duration**) — default-sorted
   **Running → Pending → Failed → Halted → Rejected → Done** (actionable above settled). **`Done` is spec'd
   but has no wire slice yet** (§4.3.2) — the batch row's Done counter carries that truth, and when the slice
   lands it must be **capped and age out**. Columns (Status,
   File, Progress, **Batch**, Kind, Size, Duration, Engine, Elapsed/Started, Reason) follow the **responsive
   priority-drop** rule ([repos.mdx §3.2.1](./repos.mdx#321-responsive-column-priority--drop-the-least-important-column-as-the-table-narrows-locked)):
   lowest drops first so the row **never wraps**; a **Reason is never hidden**, and **Batch drops when the
   Batch facet is narrowed to one batch**. It **fills the page height** — the batches table above it does
   **not** trigger the content-below exception. Fed by `jobs` + `queuedItems` + `recentFailures` (split on
   `state`) from the one `GET /api/progress` poll (§5).
3e. **Facets + an honest default sort (§4.3.1a/b):** the table's categorical filters are **value-checkbox
   facets on the rail, all checked by default** ([tables.mdx §2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked)),
   never a `<select>` — **Status** (☑ Running ☑ Queued ☑ Rejected ☑ Not attempted ☑ Failed), **Kind**
   (☑ Audio ☑ Video ☑ Image), and **Batch** (one value per live batch · ☑ **Ad hoc**). The **Batch** facet is
   the product's only **runtime vocabulary** and takes **two documented departures** (§4.3.1b): **`Ad hoc` is
   a real value** (the N/A rule would make un-batched rows permanently un-excludable, defeating §4.1.3), and
   it **does not persist** (batch ids are ephemeral — persisting `hidden:` ids would accumulate dead ids
   forever; *presentation, not preference*, per [tables.mdx §6](./tables.mdx)). It is **absent** at zero
   batches rather than empty. File/Reason are `contains`; Size/Duration are thresholds in the `⛛` popover.
   **"Failed only" is a one-click preset that sets the Status facet** (leaves only `Failed`), **not** a
   second promoted checkbox, and **does not touch the Batch facet**. The state order is a **real default
   sort** the chip run displays and the user can change — not the invisible `STATE_ORDER` JS pre-sort.
4. **Worker utilization (§3a):** while jobs run, the Processing page shows a **busy / budget** read **at the
   top of the page, above the batches table** — e.g. **"12 / 14 workers busy (~90% of cores)"** — derived from `workers` on the same
   `GET /api/progress` poll (`busy` = occupied admission slots, `budget` = the live Core Budget,
   [job_queue.mdx §3](./job_queue.mdx), [parallelization.mdx](./parallelization.mdx)); it is **absent when the
   queue is idle**. The same busy/budget figure may echo as a compact badge on the dock/nav item, reusing the
   §2 badge slot (no second poll).
5. **One source (§5):** all three surfaces read `GET /api/progress`, extended with `queuedByOp`, `batches`,
   and `workers` (all optional); they never disagree because there is no second source of truth. **`batchId`
   rides every item** and is what joins the two tables — **client-side, off the one payload**; there is **no
   per-batch item endpoint**, so narrowing to a batch never costs a fetch. **`recentFailures` is
   discriminated by `state`** (`failed` | `halted` | `rejected`) and **state is never inferred from the field
   name or the reason prose** ([ai_description.mdx §3.5](./ai_description.mdx) — *structure decides*).
6. **No empty chrome:** when nothing is processing, the nav item and the dock are both absent from the DOM;
   only the directly-navigated `/processing` page renders (its empty state). With **zero batches** the
   batches table and the Batch facet are **absent**, not empty (§4.1.1, §4.3.1b).
7. **Cross-tab wake (§4.5):** an **idle** Processing page (showing "Nothing is processing right now.") that
   is open when **another tab**, a server-side batch, or the **launchd** worker queues work **starts moving
   on its own** within one poll interval (**≤ ~1.2 s** while busy) — driven by the shared process-global
   queue/registry state + the `GET /api/progress` poll, with no second source of truth. **A batch started in
   another tab appears as a row in the batches table**, same interval, same reason.
8. **TO DO batch Apply (§4.6):** **Applying a [TO DO Batch](./to_do_batches.mdx)** enqueues its effects as
   **background jobs** that surface across all three surfaces — **pull-down fetch**, **compress**, and
   **transcribe** kinds appear as Running / dock cards and count into the Status facet's `Queued`; a
   **transcribe slug's** Apply is the **main producer of transcribe jobs**; a **mixed** batch fans out all
   three kinds into the same queue and the same `GET /api/progress` poll, with no new surface — and lands as
   **ONE** batch row of kind `mixed` ([processing_batches.mdx §1.2](./processing_batches.mdx)), never three.
9. **Scale — 2,000+ items (§4.4):** enqueuing **2,000 AI descriptions** (or any large batch) and opening
   `/processing` — including in a **second browser that did nothing** — **loads promptly and stays smooth**:
   the poll keeps answering because **no producer blocks the event loop** (async transcode + async base64
   read, [ai_description.mdx §3.3.1](./ai_description.mdx), [performance.mdx P-27](./performance.mdx)); the
   payload is **bounded** (pending/failure rows capped at 500) while the counts stay **exact**; and the
   per-item table **windows** its rows so the render is fixed-cost. The reported "the Processing page would
   not load during a mass describe run" defect stays closed.
9a. **The batches table is O(batches) (§4.4 item 4):** a 2,000-file batch is **one row of integers**. Every top-
   table figure — progress, the rejection rate, Problems — is read from the **counters
   `recordBatchResult()` already incremented**, **never** by scanning the item rows. Deriving a rate from
   rows would compute it over the **capped head of the line** (500) and be **wrong for any batch larger than
   the cap**.
