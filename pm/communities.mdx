# LargeFileBridge — Communities

A **community** is a publisher of large public files — **videos first**, images second — that a
user can **subscribe** to. Where [Peers](./storage_local.mdx) are *your own computers* pinning *your own*
files, **Communities are other people's content you choose to help carry.** The two are siblings:
same IPFS-pinning machinery, different owner of the bytes.

Subscribing is an act with an **intent** and a **budget**. This page is where the user:

1. Browses / searches communities and **subscribes**.
2. Picks, per community, **why** they subscribe — **Get** the videos (consume) and/or **Support**
   the community (rebroadcast to keep it secure).
3. Sets, per community, **how much** to keep — **Block**, **Recommended**, or **Full backup** —
   against a **storage budget** LFB derives from this computer's real free disk space.

> **This is public content, by definition.** Communities only ever carry files a publisher has made
> **public** (per the charter, LFBridge is *not* for private files). A user rebroadcasting a
> community is helping keep **already-public** content alive across more machines.

[Storage](./storage_local.mdx) (the state root + pin units this reuses) ·
[IPFS](./ipfs.mdx) (the local pinset these subscriptions add to) ·
[Security](./security.mdx) (the only-our-content policy §7 keeps us inside).

## 1. Charter compliance — why rebroadcasting is still "only our content" (LOCKED)

The charter's **IPFS gateway / no-relay policy** is strict: by **default** this computer must **not**
act as a public gateway or relay that bounces or caches **other people's** content or traffic; we
store, pin, and serve **our own content — nothing else**. Communities touches that line directly, so
the design is drawn to stay on the safe side of it:

* **Subscribing is an explicit, per-community opt-in.** Nothing is rebroadcast until the user
  deliberately subscribes to a **named** community and chooses **Support**. This is exactly the
  "user has gone out of their way" case the charter carves out — and it stays **narrow**: we pin the
  **specific files of the specific publisher the user chose**, not arbitrary third-party traffic.
* **We never become a general-purpose gateway/relay.** Pinning a chosen publisher's chosen files is
  **content storage**, not a relay. We do **not** open the node to forward, cache, or serve content
  we were not explicitly subscribed to. The default gateway-off posture from
  [security.mdx](./security.mdx) still holds; Communities does not flip it.
* **Off by default; blockable always.** Every community defaults to **Block** (§4) until the user
  acts. The user can return any community to **Block** at any time and its content stops being
  served.
* **Flag out-of-compliance.** Per the charter, if the app ever finds itself serving community content
  the user did **not** subscribe to (e.g. a stale pin after unsubscribing, or a relay setting drifting
  on), surface it and offer the fix. Rebroadcast never happens silently.

Bottom line: **Support = "pin this publisher's public files to add redundancy," not "run a public
relay."** That distinction is the thing that keeps Communities compliant, and it is LOCKED.

## 2. What a community is

A community record is small, human-readable, and lives in the state tree as YAML (§8). It carries:

| Field | Meaning |
| --- | --- |
| `id` | Stable slug (primary key across the app and the URL, e.g. `holon-citizen`). |
| `name` | Display name. |
| `publisher` | Who runs it (handle / identity the files are published under). |
| `description` | One line of what they publish. |
| `library` | The set of large files the community publishes → each with its IPFS **CID**, size, duration/resolution (for videos), and published date. This is the manifest we pin against. |
| `totals` | Rolled-up counts and bytes: total items, total size, video count, image count. |
| `subscription` | The user's choices for this community — **intent** (§3) and **backup mode** (§4). Absent = not subscribed. |

The `library` is the same shape as the large-file manifests in [storage_local.mdx](./storage_local.mdx) §9 —
a list of files mapped to CIDs and pin state — so a community is, mechanically, **just another pin unit** whose bytes happen to be owned by someone else.

---

## 3. Subscription intent — Get vs. Support

When a user subscribes, they pick **why** (one or both):

* **Get** — *consume for yourself.* Download/pin the community's videos so **you** can watch them,
  offline and fast, on your own machine. The value is to *you*.
* **Support** — *rebroadcast to keep it secure.* Pin the community's content to add **redundancy**
  across more machines so a publisher losing their copy doesn't lose the work. The value is to the
  *community's resilience*. (This is the intent that touches §1 — it is an explicit opt-in.)

Intent and backup mode (§4) are **independent axes**: intent says *whose benefit*, backup mode says
*how much disk*. A user can **Get** on **Recommended**, **Support** on **Full**, etc. Both intents
draw from the same per-community pin and the same storage budget (§5) — choosing both does not double
the bytes.

---

## 4. Backup mode — the three states (per community)

Every community sits in exactly **one** of three backup modes. This is the core control on the page:

1. **Block** — *pin nothing.* Do not download or pin any of this community's files. The default for
   every community until the user acts. A previously-pinned community set back to **Block**
   **unpins** its content (stops rebroadcasting). This is the charter-safe resting state (§1).
2. **Recommended** — *let LFB choose a subset that fits the budget.* LFB pins the **recommended
   amount to keep secure** for this community (§5) — a subset of the library selected within the
   storage budget. **This is the default we steer users toward** once they subscribe.
3. **Full backup** — *pin the entire library.* Download/pin **every** file the community publishes,
   ignoring the recommended cap (still bounded by real free disk — §5 headroom is never violated).

The mode is a single control per row (a 3-way segmented control: **Block · Recommended · Full**).
Changing it re-plans what gets pinned on the next pin and updates the live budget meter (§6).

## 5. The storage math — disk space, budget, and the recommended amount

Communities decides *how much* to keep by measuring **this computer's real storage**, not a guess.

### 5.1 What we measure

* **Total disk space** — the capacity of the volume the LFB state root lives on.
* **Free disk space OUTSIDE IPFS** — the free space that is genuinely available, i.e. **excluding**
  what the local IPFS datastore already occupies. We report free space as *"room we can still grow
  into,"* so bytes already committed to our own pins are not double-counted as free.
* **Reserved headroom** — a floor we never cross. We do **not** fill the disk; a configurable safety
  margin (default a percentage of total, with an absolute minimum) is always kept free. Every plan is
  bounded so headroom is preserved — **Full backup included**.

> These are measured with Node's own filesystem APIs (`statfs`/`statvfs`) on the state-root volume —
> local only, no shell, consistent with the storage-layer rules in [storage_local.mdx](./storage_local.mdx).

### 5.2 The community storage budget

The **community storage budget** is the total disk LFB may devote to **all community content
combined** — the amount the **user is willing to give** to communities.

* LFB proposes a **recommended budget** = `free_outside_ipfs − reserved_headroom`, then a sensible
  fraction of that (we do not assume the user wants to hand the whole disk to other people's videos).
* The user can **accept the recommendation or set their own number** (a slider/field bounded by
  `[0 … free_outside_ipfs − headroom]`). This one number answers the charter question *"how much
  storage is this user willing to do for community videos."*
* The budget is **computer-wide** (a property of this machine's disk), stored on the app-level
  config, not per-user — consistent with pin state being computer-owned in
  [storage_local.mdx](./storage_local.mdx) §2.

### 5.3 The recommended amount to keep secure (per community)

Within the budget, each **Recommended** community gets a **recommended amount to keep secure** — the
subset LFB would pin for it. The subset is chosen to maximize the value of the bytes:

* Prefer content **most at risk** (fewest other pinners / lowest redundancy) and **most wanted**
  (newest, most-watched), trimming oldest/most-redundant first when space is tight.
* Communities in **Full backup** are funded first from the budget; **Recommended** communities then
  share the remainder, each capped at its recommended amount.
* If the budget cannot cover everything, LFB **degrades gracefully and says so** — it never silently
  drops content. The page shows what is pinned vs. deferred and why (over budget / lower priority).

No file is ever compressed or altered to fit — that is a separate, explicit user action (charter).

## 7. The communities table (TanStack)

**All tables are TanStack tables** (charter). This one lists every community — subscribed or
discoverable — one row each. It uses the **shared table behavior** from
[tables.mdx](./tables.mdx): a leading **bookmark** toggle, the **value-checkbox facet rail** with the
**promoted "Bookmarked only"** checkbox at its head, **multi-level** (primary/secondary/tertiary) sort,
and **per-user, per-page persistence** of sort + filter.

Control row above the table (charter pattern, matching **SysRaps**):

* A **search** box (by name / publisher / description).
* A **facet rail** directly beneath it — always visible, never a dropdown (§7.1).
* One **filter** icon → popover. It no longer holds the categorical filters; it keeps only the
  free-text `contains` filters, the numeric thresholds (`Library ≥`, `Redundancy ≥`), and rail
  overflow at narrow widths
  ([tables.mdx §2.3](./tables.mdx#23-what-survives-in-the-filter--popover-locked)). It stays **colored
  whenever any filter is active — rail facets included**.
* One **sort** icon → **multi-level** dropdown with **1st/2nd/3rd** priority slots
  ([tables.mdx §3](./tables.mdx#3-multi-level-sort--primary--secondary--tertiary-locked)), driving the
  **same** state as header clicks and the **active-sort chip run** on the control row (shift-click a
  header appends a key — [tables.mdx §3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked)).
  **Default sort: Bookmarked first, then community name.**

Columns:

| Column | Shows |
| --- | --- |
| **`▮` Bookmark** | **Leading** ribbon favorite toggle ([tables.mdx §1](./tables.mdx#1-the-leading-bookmark-favorite-toggle-locked)) — a control cell; flips `bookmarked` (stored in `pin/c/<community_id>/config.yaml`, §8) and never navigates. Its filter is the promoted **`☐ Bookmarked only`** checkbox on the rail (§7.1), **not** a `yes / no` facet. |
| **Community** | Name + publisher + one-line description. |
| **Library** | Item count and total size (e.g. `142 videos · 88 GB`). A number → a `≥` threshold in the `⛛` popover, not a facet. |
| **Intent** | **Get** / **Support** toggles (both allowed) — §3. Two independent booleans → **two** rail facets (§7.1). |
| **Backup mode** | The 3-way **Block · Recommended · Full** segmented **editor** — §4. Its **filter** is the Mode **facet** on the rail; editor ≠ filter (§7.2). |
| **Keeping secure** | For this community: bytes currently pinned vs. its target (recommended amount, or full-library size). A small meter. |
| **Redundancy** | How many known pinners already carry it (informs the "most at risk" recommendation). A number → a `≥` threshold in the `⛛` popover, not a facet. |

Table behavior (all charter defaults):

* **Pagination default 500 / page**; item count shown above pagination.
* **Click-a-header to sort**; click again toggles asc/desc; a caret `↑`/`↓` marks the active column;
  the header, the **active-sort chip run**, and the sort dropdown share one sort state, and
  **shift-clicking a header appends** a key
  ([tables.mdx §3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked)).
* **Full-page-height:** with nothing rendered beneath it, the table body fills to the bottom of the
  viewport and the **rows** scroll inside it (header/control row + facet rail + count/pagination footer
  pinned); the page itself does not scroll. (The storage header in §6 is *above* the table, so it does
  not trigger the "content below" exception.)
* **Persistence:** this page's sort + filter (including **Bookmarked only**, **Subscribed only**, and
  the sort priority order) are saved per user in `users/<email>/config.yaml` `tables.views` and restored
  on load ([tables.mdx §4](./tables.mdx#4-persistence--sort--filter-saved-per-user-per-page-locked)).
  Each facet persists as its **unchecked** values (`facets: { mode: { hidden: [block] } }`), never the
  shown ones, so a mode/intent value added later stays visible to existing users
  ([tables.mdx §4.1](./tables.mdx#41-facets-persist-as-the-hidden-values-never-the-shown-ones-locked)).

### 7.1 The facet rail — this page's facets (LOCKED)

Filtering follows the value-visibility checkbox model owned by
[tables.mdx §2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked) — **no facet is a
`<select>`**, a facet is its **values, each a checkbox, all checked by default** (all-checked =
inactive, no `any` option), checkboxes within a facet OR and facets AND. This page does not restate the
model; it declares only which facets it carries and their vocabularies (per
[tables.mdx §2.8](./tables.mdx#28-a-facets-values-must-be-declared--an-empty-facet-is-a-bug-locked), an
enum facet with no declared vocabulary is a bug):

```
 ☐ Subscribed only  ·  ☐ Bookmarked only  │  Mode  ☑ Block ☑ Recommended ☑ Full   ·   Get  ☑ Getting ☑ Not getting   ·   Support  ☑ Supporting ☑ Not supporting
```

| Rail item | Kind | Values / behavior |
| --- | --- | --- |
| **`☐ Bookmarked only`** | promoted single checkbox | **Head of the rail** ([tables.mdx §2.7](./tables.mdx#27-the-promoted-bookmarked-only-filter-locked)) — on = only bookmarked communities. The one documented exception to the facet model; **not** a `Bookmarked: yes / no` facet. |
| **`☐ Subscribed only`** | promoted single checkbox | Same promoted shape and same reasoning: *"show me only the communities I've subscribed to"* is one high-frequency intent with an obvious off state, and it is **not a column vocabulary** (subscription is the presence/absence of the `subscription` record, §2). Sits beside Bookmarked-only at the head of the rail. |
| **Mode** | facet | ☑ **Block** (`block`) · ☑ **Recommended** (`recommended`) · ☑ **Full backup** (`full`) — the three backup states of §4, each a checkbox, all checked by default. Unchecking **Recommended** + **Full** is the one-gesture *"what am I not carrying?"* view; the retired one-of-N select could not express *"Recommended **and** Full"* at all. |
| **Get** | facet | ☑ **Getting** · ☑ **Not getting** — the `get` intent boolean (§3). |
| **Support** | facet | ☑ **Supporting** · ☑ **Not supporting** — the `support` intent boolean (§3). |

**Why intent is TWO facets, not one (LOCKED).** `get` and `support` are **two independent booleans** —
a subscription can be **both**, **either**, or **neither** (§3: *"one or both"*). That is structurally
the same shape as Pin (IPFS) vs. Git-ignore, and it takes the same ruling
([tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)):
**independent booleans never collapse into one one-of-N facet.** A single "Intent" facet offering
`Get · Support` would have to invent a `Both` member and a `Neither` member — a fake vocabulary that no
field on the row holds — and it would silently make the two axes look mutually exclusive when they are
not. Two two-value facets express every combination natively: *(Get: Getting ✓, Not getting ✗)* AND
*(Support: all ✓)* = *"everything I'm getting, whether or not I support it."* One facet = one axis;
intent is two axes, so it is two facets.

### 7.2 A one-of-N **editor** is not a one-of-N **filter** (LOCKED)

The **Backup mode** cell renders a **3-way segmented control** (`Block · Recommended · Full`) and it
**stays segmented**. The rail's **Mode facet** is checkboxes. This is not an inconsistency — they answer
**two different questions** ([tables.mdx §2.5](./tables.mdx#25-checkbox-vs-radio-vs-segmented--pick-by-the-question-locked)):

| Surface | Question | Control | Why |
| --- | --- | --- | --- |
| **The cell** (§4) | *"What mode should **this** community be in?"* | **Segmented — correct, keep it.** | This is an **editor** writing one field. A community sits in **exactly one** of the three modes (§4) — "Block **and** Full" is meaningless. A genuine one-of-N **mode**, which is precisely what §2.5 reserves radio/segmented for. |
| **The rail** (§7.1) | *"Which modes do I want to **see**?"* | **Checkbox facet.** | This is a **predicate over a value set**. *"Show Recommended **and** Full"* is an obvious, common request, so by §2.5's rule of thumb — *if the user could ever plausibly want two of the options at once, it is a checkbox set* — it cannot be one-of-N. |

The **real** inconsistency this resolves is the old **filter**: `mode` shipped as a `<select>` in the
filter popover while its own cell was segmented — two different controls for the same field, and
**neither** could express "Recommended and Full." The filter becomes a facet; the editor does not move.

The Backup-mode **cell** is a segmented one-of-N editor (correct — it stays) in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/communities/CommunitiesPage.tsx` — `CommunitiesPage` (the `mode` column's `cell`, mapping `MODES` to segment buttons that `patch` `backupMode`).

The Mode **filter** is today a `<select>` of `MODES` and the Bookmark filter a `["yes","no"]` select; both are to be rebuilt as rail items per §7.1, and the intent facets do not exist yet (`intent` is `filterable: false`) — **to be implemented in code**:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/communities/CommunitiesPage.tsx` — `CommunitiesPage` (the `mode` / `bookmark` / `intent` column definitions), rendered by the shared facet rail in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable`.

## 8. State & storage

A community subscription is a **pin unit** like any other (see [storage_local.mdx](./storage_local.mdx) §5–§9),
stored **computer-wide** (owned by the machine, not the logged-in user):

```
~/T/_large_files_bridge/
├── config.yaml                 # + community_budget: the machine's community storage budget (§5.2)
└── pin/
    └── c/                      # one folder PER SUBSCRIBED COMMUNITY (mirrors pin/r/<repo>/)
        └── <community_id>/
            ├── config.yaml     #   intent (get/support) + backup_mode (block|recommended|full) + bookmarked
            ├── manifest.yaml   #   the community library → CID + local pin state
            └── status.yaml     #   machine-written last-scan / last-pin cache
```

* `pin/c/<community_id>/` mirrors `pin/r/<repo_name>/` — same three files, same store layer, same
  atomic writes. A community is mechanically a repo-shaped pin unit whose bytes are someone else's.
* `config.yaml` (app-level) gains **`community_budget`** — the single computer-wide budget number.
* **Block** communities may keep a record (so the choice sticks) but hold **no pins**.
