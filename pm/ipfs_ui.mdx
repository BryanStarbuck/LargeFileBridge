# LargeFileBridge — The IPFS Page (Node Control Panel)

The **IPFS** left-bar item lands here. This is the **dashboard for the user's local IPFS (Kubo)
node** — the one screen that makes the invisible visible: *is IPFS installed on this computer, is the
daemon running this second, and how much is it doing for me?* Everything a non-expert needs to
understand and control their node lives here; anything deeper is **one click away**.

Think of it as the node's **home page**: a status hero at the top with the two controls that matter
most (**Install** and an **on/off toggle**), a grid of **live metric tiles** below, and a **gateway**
and **security** summary. Tiles are **drill-in surfaces** — clicking *Shared files* opens the full
pinset table; clicking *Peers* opens the Peers page. The heavy, row-per-CID **pinset table** is a
distinct page ([pm/ipfs.mdx](./ipfs.mdx)) that now lives at **`/ipfs/pins`** and is reached *from*
this dashboard — so this landing page stays clean and readable.

> **Page split (LOCKED).** Three URLs, one section:
> * `/ipfs` = **this dashboard for a RUNNING node** (metrics, gateway, security, on/off, reboot
>   auto-start). When the node is **not running**, `/ipfs` **redirects to `/ipfs/off`**.
> * `/ipfs/off` = the **IPFS-off page** ([§12](#12-the-ipfs-off-page-turn-it-on-and-keep-it-on)) — a
>   deliberately *different* page whose whole job is turning IPFS on (install / start) and, ideally,
>   keeping it on across reboots. It **redirects to `/ipfs`** once the node is healthy. This is where
>   you land after a reboot.
> * `/ipfs/pins` = the **pinset table** ([pm/ipfs.mdx](./ipfs.mdx)) — the "what is actually pinned /
>   import untracked" workhorse, reached from the *Shared files* tile here (and from the left-bar
>   disclosure children).

Siblings this spec builds on and does not repeat:

* [pm/ipfs.mdx](./ipfs.mdx) — the **pinset table** (`/ipfs/pins`): one row per pinned CID, tracked vs.
  untracked, import. The *Shared files* tile is the door to it.
* [pm/tools.mdx](./tools.mdx) — the **install-tools preflight**. The IPFS install here is the same
  package-manager install that preflight defines, surfaced on the node's own page.
* [knowledge/ipfs.mdx](../knowledge/ipfs.mdx) — what a **node / daemon / gateway / pin / reprovide
  strategy** is, and the [only-our-content rule](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule).
* [pm/settings.mdx](./settings.mdx) — where the reprovide strategy and public-gateway opt-out live.
* [pm/left_bar.mdx](./left_bar.mdx) — the nav item; [pm/menus.mdx](./menus.mdx) — drill-in navigation.

## 1. Use case

A user runs LargeFileBridge to sync their large, git-ignored files across their own computers over
IPFS. IPFS is the engine underneath — but most users have **never run an IPFS node** and have no
mental model of it. When something isn't syncing, the first question is almost always *"is IPFS even
running?"* Today the answer is buried in a red/green word on the Scans page.

This page exists so that a user can, without leaving the app or opening a terminal:

* **See at a glance** whether IPFS is **installed** and **running** on this computer.
* **Install it** with one click when it's missing — on whatever OS they're on — and **watch the
  progress**; and if the automated install can't run, get the **exact command to paste** into a
  terminal instead.
* **Turn the node on or off** — because sometimes they just want it off on this machine for a while
  (battery, bandwidth, privacy, they're travelling) and back on later.
* **Understand what it's doing** — how many files it's sharing, how much disk it's using, how many of
  their other computers it's connected to — and **click through** to the detail behind any number.
* **Trust it** — confirm the node is only serving *their own* content (the only-our-content rule),
  and fix it in one click if it drifts.

Concrete moments this serves:

* First run on a fresh laptop → *"IPFS isn't installed"* → **Install** → progress → running. Done.
* "My files aren't showing up on my desktop" → open IPFS → the toggle says **Off** → **Turn on**.
* "Is my stuff safe / private?" → the security card says **Only your content ✓**, gateway **local-only**.
* Curiosity → *"812 GB across 1,204 files, 3 of my computers connected"* → click *Shared files* → the
  pinset.

## 2. Goals

1. **Answer "installed? running? should it be?" in under two seconds** — the hero card is legible at a
   glance, no scrolling, no jargon.
2. **Make install a one-click, cross-platform, watchable action** — detect macOS / Windows / Linux,
   drive the right installer, show live progress, and **always** offer a copyable manual command as
   the fallback that never fails.
3. **Make on/off a single obvious toggle** — start or stop the daemon from the UI; reflect the real OS
   state, not a wish.
4. **Give the right level of dashboard** — a small set of **high-signal metric tiles**, each a door to
   a deeper page. Clean first, deep on demand.
5. **Guard privacy by default** — surface the only-our-content posture and fix drift, per the charter.
6. **Be honest about errors** — every failure state names what happened and the next concrete step.
7. **Nudge from everywhere** — when the node is missing or off, a small app-wide banner points back
   here; when it's healthy, that banner is **absent** (no nagging).

Non-goals: this page is **not** the pinset browser (that's [/ipfs/pins](./ipfs.mdx)); it does **not**
manage individual files; it does **not** turn the machine into a public gateway (charter forbids it).

## 3. Audience

* **Primary — the non-technical owner of a few computers.** Wants their videos on all their machines,
  has never heard of "Kubo" or "a DHT." For them the page is: a light that's green or red, an Install
  button, an on/off switch, and a couple of reassuring numbers.
* **Secondary — the tinkerer / power user.** Wants the PeerID, the version, the repo size vs.
  StorageMax, the reprovide strategy, the gateway URL, and a way to copy the manual install command.
  They get all of it — in the tiles and the "details" drill-ins — without it cluttering the primary view.
* **Tertiary — the future server operator.** Same page, `server` mode; install/toggle may be
  read-only (managed by ops), metrics still shown. Design must not assume localhost-only.

Design implication: **progressive disclosure.** The surface is calm and non-technical; every technical
fact is present but tucked into a tile subtitle, a tooltip, or a details page.

push the rest into drill-ins. The full inventory, and where each lives:

| # | Thing IPFS exposes | Where it comes from (Kubo RPC / CLI) | Shown on dashboard? |
| --- | --- | --- | --- |
| 1 | **Installed?** (CLI/binary present) | `which ipfs` / `where ipfs` / RPC reachability | **Hero** — primary |
| 2 | **Running?** (daemon up) | RPC `id` reachable (health) | **Hero** — the on/off state |
| 3 | **Version** | `ipfs version` / RPC `version` | Hero subtitle |

| 6 | **Blocks / objects** in the repo | RPC `repo/stat` → `NumObjects` | **Tile** |
| 7 | **Storage used** vs. **StorageMax** | RPC `repo/stat` → `RepoSize` / `StorageMax` | **Tile** (used / max) |
| 8 | **Connected peers** (swarm) | RPC `swarm/peers` count | **Tile → /devices** |
| 9 | **Bandwidth** (total in/out, rate) | RPC `stats/bw` | **Tile** (best-effort) |
| 10 | **Gateway** — on? local-only? URL? kind? | config `Addresses.Gateway` + posture | **Gateway panel → details** |
| 11 | **Reprovide strategy** (what we announce) | config `Reprovider/Provide.Strategy` | **Security card** |

| 13 | **Compliant?** (only-our-content) | derived from 10–12 (knowledge/ipfs.mdx §6) | **Security card** (Fix) |
| 14 | **Repo path / IPFS_PATH** | `repo/stat` → `RepoPath` | Gateway/details drill-in |
| 15 | **Directories / MFS roots shared** | MFS (`files/stat /`) | Details drill-in (advanced) |
| 16 | **Addresses** (API addr, swarm addrs) | config `Addresses.*` | Details drill-in (advanced) |

Tiles chosen for the dashboard: **Shared files (5)**, **Storage (7)**, **Blocks (6)**, **Peers (8)**,
**Bandwidth (9)**. Everything else is hero, security card, gateway panel, or a details drill-in.

## 6. Node states (what renders)

The page is a small state machine over two booleans — **installed** and **running** — plus in-flight
**install** / **starting** jobs.

Deriving the installed/running booleans, live metrics, gateway summary, and compliance for the dashboard payload is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `nodeStatus()`

Which URL owns each state (§12 page split): a **running** node renders on **`/ipfs`**; every
**not-running** state renders on **`/ipfs/off`**. `/ipfs` redirects to `/ipfs/off` the moment the node
is down (and `/ipfs/off` redirects back to `/ipfs` once it's healthy). A start/stop/install **job in
flight** shows its live progress on whichever page you're on, then the redirect settles you on the
right one.

| State | URL | Hero shows | Body shows |
| --- | --- | --- | --- |
| **Not installed** (no CLI, no daemon) | `/ipfs/off` | red dot · *"IPFS isn't installed on this computer"* | the **Install card** (§7) — big primary **Install IPFS** button + what it is / why. No tiles. |
| **Installing** (job running) | `/ipfs/off` | spinner · *"Installing IPFS…"* | the **progress view** (§7.2) — live log lines + phase. |
| **Installed, stopped** | `/ipfs/off` | amber dot · *"IPFS is installed but not running"* | the **two-button turn-on choice** (§12) — turn on + auto-start on reboot (recommended), or turn on only. |
| **Starting** (job running) | `/ipfs/off` | spinner · *"Starting the IPFS daemon…"* | progress view (§7.2), phase = `starting` → `autostart` (when the reboot setup runs). |
| **Running** (healthy) | `/ipfs` | green dot · *"IPFS is running"* · On/Off toggle | full dashboard (§5.1) + the **reboot auto-start** control (§13). |
| **Unreachable / error** | `/ipfs/off` | red dot · *"Can't reach the IPFS node"* | the **error panel** (§9) with cause + retry + manual command. |

The **on/off toggle** is only meaningful once installed. **On** = daemon running; flipping **Off**
stops the daemon (RPC `shutdown`); flipping **On** starts it (spawn `ipfs daemon`, then poll health).
Both are **jobs with progress** (§7.2) because starting a daemon isn't instant.

Starting or stopping the daemon behind the on/off toggle (and, on start, optionally installing reboot auto-start) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `controlDaemon()`

Choosing which URL renders (running dashboard vs. off page) and redirecting between them is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/ipfs/IpfsDashboardPage.tsx` — `IpfsDashboardPage` (running; redirects to `/ipfs/off` when down)

## 7. Install & progress (cross-platform, watchable, always-has-a-fallback)

### 7.1 The Install card (state = not installed)

Shown when the `ipfs` CLI/daemon is absent. Contents:

* A one-paragraph, plain-language *"IPFS is the engine that moves your big files between your
  computers. It runs locally on this machine — it isn't a cloud service."*
* The detected **platform** (macOS / Windows / Linux) and the **method** we'll use:
  * **macOS** → Homebrew (`brew install ipfs`) when `brew` is present.
  * **Windows** → winget (`winget install --id IPFS.Kubo -e`) when `winget` is present.
  * **Linux** → the available package manager (snap, or the official tarball instructions) — best-effort.
* A primary **Install IPFS** button → starts the install **job** (§7.2).
* A secondary, always-present **"Install it yourself"** disclosure with the **exact command**
  (click-to-copy) for the detected platform — so even if we can't run it, the user is never stuck.
* If the **package manager itself is missing** (no brew/winget), we **don't** try to install a package
  manager; we show its official install page + the manual command (mirrors [pm/tools.mdx](./tools.mdx)).

### 7.2 The progress view (install OR start)

Any long action — installing, or starting the daemon — routes into a **progress view** on this page
(not a modal): a spinner, a **phase label**, and a **live, append-only log** of what's happening
(`detecting → installing → initializing → starting → done`). Modeled on the app's existing job/scan
progress pattern (server-side job, polled — so navigating away and back **re-attaches** to the same
run; closing the tab does not cancel it).

Detecting the platform, choosing the installer command, and running the single-flight, re-attachable install job is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `startInstall()` (platform/command via `installPlan()`)

* Phases: **detecting** (find package manager / platform) → **installing** (run the installer,
  streaming its output) → **initializing** (`ipfs init` if the repo is new) → **starting** (spawn the
  daemon, wait for health) → **done** (dashboard renders) — or **error** at any step.
* On **done**, the page flips to the running dashboard automatically.
* On **error**, the progress view becomes the **error panel** (§9) carrying the failed phase, the
  captured output tail, and the **manual command** to copy — the guaranteed escape hatch.

### 7.3 Where install can also be triggered

* From **this page's** Install card / Turn-on button.
* From the **app-wide banner** (§10) on any other page.
* From the **install-tools preflight** ([pm/tools.mdx](./tools.mdx)) when a scan needs `ipfs` and it's
  missing. All three drive the **same** install job — never two at once (single-flight).

the user **deliberately** flipped the public-gateway [setting](./settings.mdx) on this machine — then

## 9. Errors — every failure is a designed state

The page never dead-ends. Each failure names the cause and the next step.

| Error | What we show | Primary action |
| --- | --- | --- |
| **CLI missing** | Install card (§7.1) | Install / copy command |
| **Package manager missing** | "Homebrew/winget isn't installed" + official link + manual command | Copy command / open link |
| **Install failed** | error panel: failed phase + output tail | Retry · Copy manual command |
| **`ipfs init` failed** | error panel: init output | Retry · manual `ipfs init` |
| **Daemon won't start** | error panel: start output (e.g. port 5001 busy, lock held) | Retry · copy log · troubleshooting hint |
| **Daemon unreachable** (was running, now not) | red hero + "Can't reach the node" | Turn on · Retry · manual `ipfs daemon` |
| **RPC timeout on a metric** | that tile shows `—` + tooltip; page stays up | Refresh |
| **Compliance drift** | security red banner | Fix (§8) |

Design rules: an error **replaces only the region it belongs to** (a dead metric tile doesn't blank
the hero); the **manual command is always reachable** from any install/start failure; nothing is
auto-retried silently — the user clicks **Retry** (charter consent posture).

## 11. Where the data comes from

* A single **GET `/api/ipfs/node`** returns the dashboard payload: installed, running, version,
  peerId, platform, install hint (the manual command), the metric block (pins, repo stat, swarm,
  bandwidth), the gateway summary, the posture (reprovide / gc / compliant / publicGateway), and the
  **`autostart`** block (`{ supported, installed, enabled }` — the reboot auto-start state, §13).
  Metric reads are **best-effort and bounded**: a slow or unset value comes back null → the tile shows
  `—`. Reading is **metadata-only** (like the scan) — it opens no file and moves no bytes.
* **POST `/api/ipfs/daemon`** `{action:"start"|"stop", autostart?:boolean}` — drives the on/off toggle;
  returns the fresh node payload (or kicks a job, below). On `start`, `autostart:true` **also** sets up
  reboot auto-start once the daemon is healthy (the off-page's primary button, §12).
* **POST `/api/ipfs/autostart`** `{action:"install"|"remove"}` — set up or remove reboot auto-start
  directly (the running dashboard's toggle, §13); returns the fresh node payload so the UI settles on
  the real launchd state.
* **POST `/api/ipfs/install`** — starts the install **job** (single-flight); **GET
  `/api/ipfs/install/status`** — polls the job (phase + log) for the progress view and the banner.
* The **cheap** summary the banner polls is the same `/api/ipfs/node` narrowed to `{installed,
  running}` (frontend `select`), so there's no second source of truth.
* Install/start jobs are **server-side** and detached from the request (re-attachable, cancel-safe on
  navigation) — same discipline as the scan job.

## 12. The IPFS-off page — turn it on, and keep it on

`/ipfs/off` is a **separate page** from the running dashboard, with a **different job**: you are here
because IPFS is **not running** (most often right after a reboot), and everything on the page is about
getting it back on. `/ipfs` redirects here whenever the node is down; this page redirects back to
`/ipfs` the moment the node is healthy. (Rationale: an *off* node and a *running* node want genuinely
different UIs — one is "let's turn this on," the other is "here's what it's doing." Two URLs keep each
clean.)

The off-page hosts three of the state-machine states (§6): **not installed** (Install card, §7.1),
**installed-but-stopped** (the two-button choice below), and the shared **progress / error** views
(§7.2, §9) while a job runs.

### 12.1 Installed-but-stopped: the two-button choice (LOCKED)

This is the common case — IPFS is installed, the daemon just isn't up (a reboot dropped it). We present
**two ways to turn it on**, and we deliberately **steer toward the first** (keeping IPFS on across
reboots), because a node that silently stays off after every restart is the core problem this feature
fixes.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  ●(amber)  IPFS is installed but not running                                  │
│  Kubo v0.29. Turn it on so your big files can sync across your computers.      │
│                                                                               │
│  ↻ Automatically restart when you reboot                                      │  ← label ABOVE button 1
│  [ ⏻ Turn On IPFS ]  Turns IPFS on now AND sets it to start automatically      │  ← PRIMARY (filled)
│                      every time you reboot or log in.  Recommended.           │
│                                                                               │
│  ─────────────────────────────  Or  ─────────────────────────────            │
│                                                                               │
│  [ ⏻ Turn On IPFS ]  But don't modify whether or how IPFS starts up when       │  ← SECONDARY (outline)
│                      you restart the computer.                                │
│                                                                               │
│  ▸ Or start it yourself:  ipfs daemon --enable-gc                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

* **Both buttons read “Turn On IPFS.”** Both start the daemon immediately. What differs is the effect
  on **reboot auto-start**:
  * **Button 1 (primary, recommended)** — `POST /api/ipfs/daemon {action:"start", autostart:true}`.
    Starts IPFS **and** installs the reboot auto-start LaunchAgent (§13) once the daemon is healthy.
    The label **above** it is *"Automatically restart when you reboot."*
  * **Button 2 (secondary)** — `POST /api/ipfs/daemon {action:"start"}` (no `autostart`). Starts IPFS
    now and **leaves the reboot setup exactly as it is** — it does not add, remove, or change
    auto-start. The text **after** it says so explicitly.
* **We only show two buttons when there's a real choice.** If reboot auto-start is **already enabled**,
  there's nothing to opt into → a **single** "Turn On IPFS" button, with a note *"IPFS is already set to
  start automatically when you reboot ✓."* If the OS **can't** auto-start (not macOS today) → a single
  button + a note that auto-start-on-reboot isn't available on this OS yet.
* **Consent, not surprise.** Pressing button 1 **is** the permission to install the LaunchAgent — we
  never set up reboot auto-start on our own; the user chooses it.

## 13. Auto-start on reboot (keep IPFS on)

The problem: after a machine restart, the IPFS daemon is **not** running, and nothing brings it back —
so files silently stop syncing until the user notices and turns it on. Auto-start fixes that.

* **Mechanism (macOS).** A per-user **launchd LaunchAgent** — `com.largefilebridge.ipfs` — written to
  `~/Library/LaunchAgents/`, running the **absolute** `ipfs daemon --enable-gc` at **login/boot**
  (`RunAtLoad`). It is a **sibling** of the scan/sync worker agents ([pm/scan.mdx](./scan.mdx)) but a
  distinct, purpose-built plist.
* **`RunAtLoad` yes, `KeepAlive` no (deliberate).** We want *"start once at reboot/login,"* **not**
  *"relaunch whenever it exits."* With `KeepAlive` off, the app's own **On/Off toggle** still works — a
  deliberate **Off** stays off (launchd won't fight it and immediately restart the daemon). It comes
  back at the **next** login/reboot, which is exactly the promise.
* **Absolute binary + pinned PATH.** launchd has no login shell PATH, so we resolve the absolute `ipfs`
  path (`command -v ipfs`) and pin a conservative PATH (`/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin`)
  so both Apple-silicon and Intel Homebrew installs work.
* **State is the OS, not a wish.** `autostart.installed` = the plist exists; `autostart.enabled` = it's
  loaded in launchd (`launchctl print` succeeds) → it *will* run at the next boot. The app-config flag
  `ipfs.auto_start_daemon` records the **intent**, but the node payload reports the **real** launchd
  state. On non-macOS, `autostart.supported` is `false` and the UI hides the option.
* **Where the user controls it.**
  * **Off page (§12.1)** — button 1 turns it on as part of starting IPFS.
  * **Running dashboard (§5.1)** — a compact *"Start on reboot: on ✓ / off"* row with a one-click
    **Turn on / Turn off auto-start** toggle, so a user whose node is already up can set it up (or
    remove it) preemptively.
* **Security posture unchanged.** The auto-started daemon runs the same `--enable-gc` and is brought to
  only-our-content compliance on start (§8); auto-start changes *when* the daemon runs, not *what* it
  serves.

The reboot auto-start LaunchAgent — status, install (write plist + `launchctl bootstrap`/`enable`), and remove — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-autostart.service.ts` — `autostartStatus()`, `installAutostart()`, `removeAutostart()`

## 14. Acceptance criteria

1. The **IPFS** left-bar item lands on **`/ipfs`** — this **dashboard** — with a hero (installed /
   running / version / PeerID + on/off toggle), a metric-tile grid, a gateway panel, and the security
   card. The **pinset table** is a separate page at **`/ipfs/pins`** ([pm/ipfs.mdx](./ipfs.mdx)),
   reached from the **Shared files** tile and the left-bar disclosure children.
2. The page renders the correct **state** (§6): not-installed → Install card; installing/starting →
   progress view; installed-stopped → Turn on; running → full dashboard; unreachable → error panel.
3. **Install is one click, cross-platform, and watchable**: the page detects macOS/Windows/Linux,
   drives the platform installer, shows **live progress** (phases + log), and **always** offers a
   copyable **manual command** fallback (and never tries to install a missing package manager) (§7).
4. The **on/off toggle** starts/stops the daemon and reflects the **real** node state; both start and
   stop run as **progress jobs** (§6, §7.2).
5. **Metric tiles** show shared files, storage (used/max), blocks, connected peers, and bandwidth; each
   drillable tile navigates (Shared files → `/ipfs/pins`, Peers → `/devices`); an unknown value shows `—`
   without crashing the page (§5.2).
6. The **security card** shows reprovide / gateway / GC / compliant and offers a **Fix** that restores
   only-our-content defaults, with the deliberate public-gateway opt-out rendered amber not red (§8,
   [knowledge/ipfs.mdx §6/§10](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)).
7. **Every failure is a designed state** (§9): CLI missing, package-manager missing, install failed,
   init failed, daemon won't start, unreachable, metric timeout, compliance drift — each names the
   cause and a concrete next step, and the manual command is always reachable from install/start
   failures.
8. An **app-wide banner** appears on every OTHER page **only when** IPFS is not installed or not
   running, with an **Install / Start** action, and is **absent** when the node is healthy and
   **suppressed** on the IPFS page itself (§10); it reads a cheap summary, never the pinset.
9. Data comes from **`/api/ipfs/node`** (+ `/api/ipfs/daemon`, `/api/ipfs/install`,
   `/api/ipfs/install/status`); reads are metadata-only and bounded; install/start are server-side,
   single-flight, re-attachable jobs (§11).
10. The left-bar **disclosure children** (pinning repos) route to **`/ipfs/pins?repo=<repoId>`** and the
    disclosure stays open across `/ipfs` and `/ipfs/pins`; the **IPFS parent** lands on this dashboard
    (§5.3).
11. **Off ↔ running is a two-page split** (§12): when the node is **not running**, `/ipfs` redirects to
    **`/ipfs/off`**; when it becomes **healthy**, `/ipfs/off` redirects back to `/ipfs`. A job in flight
    shows live progress on the current page before the redirect settles.
12. **The installed-but-stopped state shows the two-button choice** (§12.1): both buttons read **"Turn
    On IPFS"** and both start the daemon; **button 1 (primary, recommended)** — labelled *"Automatically
    restart when you reboot"* — also sets up reboot auto-start (`daemon {autostart:true}`), while
    **button 2** (after an **"Or"** divider) turns it on **without** modifying the reboot setup
    (`daemon` with no `autostart`). When auto-start is already enabled, or the OS can't do it, a
    **single** Turn-On button is shown instead with the appropriate note.
13. **Auto-start on reboot actually persists across restarts** (§13): choosing it installs a per-user
    launchd LaunchAgent (`com.largefilebridge.ipfs`, `RunAtLoad` yes / `KeepAlive` no) running the
    absolute `ipfs daemon --enable-gc`; `GET /api/ipfs/node` reports the real `autostart {supported,
    installed, enabled}` state; the running dashboard has a **Start-on-reboot** toggle; and the app
    **never** installs auto-start without the user's click. Turning IPFS **Off** from the toggle stays
    off (KeepAlive is off) and simply comes back at the next reboot/login.
14. Data adds **`POST /api/ipfs/autostart {install|remove}`** and an **`autostart`** flag on **`POST
    /api/ipfs/daemon`** (start only); both return the fresh node payload (§11).
