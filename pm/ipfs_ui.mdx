# LargeFileBridge — The IPFS Page (Node Control Panel)

The **IPFS** left-bar item lands here. This is the **dashboard for the user's local IPFS (Kubo)
node** — the one screen that makes the invisible visible: *is IPFS installed on this computer, is the
daemon running this second, and how much is it doing for me?* Everything a non-expert needs to
understand and control their node lives here; anything deeper is **one click away**.

Think of it as the node's **home page**: a status hero at the top with the two controls that matter
most (**Install** and an **on/off toggle**), a grid of **live metric tiles** below, and a **gateway**
and **security** summary. Tiles are **drill-in surfaces** — clicking *Shared files* opens the full
pinset table; clicking *Peers* opens the Peers page. The heavy, row-per-CID **pinset table** is a
distinct page ([pm/ipfs.mdx](./ipfs.mdx)) that now lives at **`/ipfs/pins`** and is reached *from*
this dashboard — so this landing page stays clean and readable.

> **Page split (LOCKED).** Three URLs, one section:
> * `/ipfs` = **this dashboard for a RUNNING node** (metrics, gateway, security, on/off, reboot
>   auto-start). When the node is **not running**, `/ipfs` **redirects to `/ipfs/off`**.
> * `/ipfs/off` = the **IPFS-off page** ([§12](#12-the-ipfs-off-page-turn-it-on-and-keep-it-on)) — a
>   deliberately *different* page whose whole job is turning IPFS on (install / start) and, ideally,
>   keeping it on across reboots. It **redirects to `/ipfs`** once the node is healthy. This is where
>   you land after a reboot.
> * `/ipfs/pins` = the **pinset table** ([pm/ipfs.mdx](./ipfs.mdx)) — the "what is actually pinned /
>   import untracked" workhorse, reached from the *Shared files* tile here (and from the left-bar
>   disclosure children).

Siblings this spec builds on and does not repeat:

* [pm/ipfs.mdx](./ipfs.mdx) — the **pinset table** (`/ipfs/pins`): one row per pinned CID, tracked vs.
  untracked, import. The *Shared files* tile is the door to it.
* [pm/tools.mdx](./tools.mdx) — the **install-tools preflight**. The IPFS install here is the same
  package-manager install that preflight defines, surfaced on the node's own page.
* [Task tabs](./task_tabs.mdx) — the per-repo screen's **IPFS tab**, one of the four task tabs; it filters
  the file table to large/media files and sorts by transfer urgency (worst first).
* [knowledge/ipfs.mdx](../knowledge/ipfs.mdx) — what a **node / daemon / gateway / pin / reprovide
  strategy** is, and the [only-our-content rule](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule).
* [pm/settings.mdx](./settings.mdx) — where the reprovide strategy and public-gateway opt-out live.
* [pm/left_bar.mdx](./left_bar.mdx) — the nav item; [pm/menus.mdx](./menus.mdx) — drill-in navigation.

## 1. Use case

A user runs LargeFileBridge to pin their large, git-ignored files across their own computers over
IPFS. IPFS is the engine underneath — but most users have **never run an IPFS node** and have no
mental model of it. When something isn't pinning, the first question is almost always *"is IPFS even
running?"* Today the answer is buried in a red/green word on the Scans page.

This page exists so that a user can, without leaving the app or opening a terminal:

* **See at a glance** whether IPFS is **installed** and **running** on this computer.
* **Install it** with one click when it's missing — on whatever OS they're on — and **watch the
  progress**; and if the automated install can't run, get the **exact command to paste** into a
  terminal instead.
* **Turn the node on or off** — because sometimes they just want it off on this machine for a while
  (battery, bandwidth, privacy, they're travelling) and back on later.
* **Understand what it's doing** — how many files it's sharing, how much disk it's using, how many of
  their other computers it's connected to — and **click through** to the detail behind any number.
* **Trust it** — confirm the node is only serving *their own* content (the only-our-content rule),
  and fix it in one click if it drifts.

Concrete moments this serves:

* First run on a fresh laptop → *"IPFS isn't installed"* → **Install** → progress → running. Done.
* "My files aren't showing up on my desktop" → open IPFS → the toggle says **Off** → **Turn on**.
* "Is my stuff safe / private?" → the security card says **Only your content ✓**, gateway **local-only**.
* Curiosity → *"812 GB across 1,204 files, 3 of my computers connected"* → click *Shared files* → the
  pinset.

## 2. Goals

1. **Answer "installed? running? should it be?" in under two seconds** — the hero card is legible at a
   glance, no scrolling, no jargon.
2. **Make install a one-click, cross-platform, watchable action** — detect macOS / Windows / Linux,
   drive the right installer, show live progress, and **always** offer a copyable manual command as
   the fallback that never fails.
3. **Make on/off a single obvious toggle** — start or stop the daemon from the UI; reflect the real OS
   state, not a wish.
4. **Give the right level of dashboard** — a small set of **high-signal metric tiles**, each a door to
   a deeper page. Clean first, deep on demand.
5. **Guard privacy by default** — surface the only-our-content posture and fix drift, per the charter.
6. **Be honest about errors** — every failure state names what happened and the next concrete step.
7. **Nudge from everywhere** — when the node is missing or off, a small app-wide banner points back
   here; when it's healthy, that banner is **absent** (no nagging).

Non-goals: this page is **not** the pinset browser (that's [/ipfs/pins](./ipfs.mdx)); it does **not**
manage individual files; it does **not** turn the machine into a public gateway (charter forbids it).

## 3. Audience

* **Primary — the non-technical owner of a few computers.** Wants their videos on all their machines,
  has never heard of "Kubo" or "a DHT." For them the page is: a light that's green or red, an Install
  button, an on/off switch, and a couple of reassuring numbers.
* **Secondary — the tinkerer / power user.** Wants the PeerID, the version, the repo size vs.
  StorageMax, the reprovide strategy, the gateway URL, and a way to copy the manual install command.
  They get all of it — in the tiles and the "details" drill-ins — without it cluttering the primary view.
* **Tertiary — the future server operator.** Same page, `server` mode; install/toggle may be
  read-only (managed by ops), metrics still shown. Design must not assume localhost-only.

Design implication: **progressive disclosure.** The surface is calm and non-technical; every technical
fact is present but tucked into a tile subtitle, a tooltip, or a details page.

## 6. Node states (what renders)

The page is a small state machine over two booleans — **installed** and **running** — plus in-flight
**install** / **starting** jobs.

Deriving the installed/running booleans, live metrics, gateway summary, and compliance for the dashboard payload is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `nodeStatus()`

Which URL owns each state (§12 page split): a **running** node renders on **`/ipfs`**; every
**not-running** state renders on **`/ipfs/off`**. `/ipfs` redirects to `/ipfs/off` the moment the node
is down (and `/ipfs/off` redirects back to `/ipfs` once it's healthy). A start/stop/install **job in
flight** shows its live progress on whichever page you're on, then the redirect settles you on the
right one.

| State | URL | Hero shows | Body shows |
| --- | --- | --- | --- |
| **Not installed** (no CLI, no daemon) | `/ipfs/off` | red dot · *"IPFS isn't installed on this computer"* | the **Install card** (§7) — big primary **Install IPFS** button + what it is / why. No tiles. |
| **Installing** (job running) | `/ipfs/off` | spinner · *"Installing IPFS…"* | the **progress view** (§7.2) — live log lines + phase. |
| **Installed, stopped** | `/ipfs/off` | amber dot · *"IPFS is installed but not running"* | the **two-button turn-on choice** (§12) — turn on + auto-start on reboot (recommended), or turn on only. |
| **Starting** (job running) | `/ipfs/off` | spinner · *"Starting the IPFS daemon…"* | progress view (§7.2), phase = `starting` → `autostart` (when the reboot setup runs). |
| **Running** (healthy) | `/ipfs` | green dot · *"IPFS is running"* · On/Off toggle | full dashboard (§5.1) + the **reboot auto-start** control (§13). |
| **Unreachable / error** | `/ipfs/off` | red dot · *"Can't reach the IPFS node"* | the **error panel** (§9) with cause + retry + manual command. |

The **on/off toggle** is only meaningful once installed. **On** = daemon running; flipping **Off**
stops the daemon (RPC `shutdown`); flipping **On** starts it (spawn `ipfs daemon`, then poll health).
Both are **jobs with progress** (§7.2) because starting a daemon isn't instant.

Starting or stopping the daemon behind the on/off toggle (and, on start, optionally installing reboot auto-start) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `controlDaemon()`

Choosing which URL renders (running dashboard vs. off page) and redirecting between them is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/ipfs/IpfsDashboardPage.tsx` — `IpfsDashboardPage` (running; redirects to `/ipfs/off` when down)

## 7. Install & progress (cross-platform, watchable, always-has-a-fallback)

### 7.1 The Install card (state = not installed)

Shown when the `ipfs` CLI/daemon is absent. Contents:

* A one-paragraph, plain-language *"IPFS is the engine that moves your big files between your
  computers. It runs locally on this machine — it isn't a cloud service."*
* The detected **platform** (macOS / Windows / Linux) and the **method** we'll use:
  * **macOS** → Homebrew (`brew install ipfs`) when `brew` is present.
  * **Windows** → winget (`winget install --id IPFS.Kubo -e`) when `winget` is present.
  * **Linux** → the available package manager (snap, or the official tarball instructions) — best-effort.
* A primary **Install IPFS** button → starts the install **job** (§7.2).
* A secondary, always-present **"Install it yourself"** disclosure with the **exact command**
  (click-to-copy) for the detected platform — so even if we can't run it, the user is never stuck.
* If the **package manager itself is missing** (no brew/winget), we **don't** try to install a package
  manager; we show its official install page + the manual command (mirrors [pm/tools.mdx](./tools.mdx)).

### 7.2 The progress view (install OR start)

Any long action — installing, or starting the daemon — routes into a **progress view** on this page
(not a modal): a spinner, a **phase label**, and a **live, append-only log** of what's happening
(`detecting → installing → initializing → starting → done`). Modeled on the app's existing job/scan
progress pattern (server-side job, polled — so navigating away and back **re-attaches** to the same
run; closing the tab does not cancel it).

Detecting the platform, choosing the installer command, and running the single-flight, re-attachable install job is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `startInstall()` (platform/command via `installPlan()`)

* Phases: **detecting** (find package manager / platform) → **installing** (run the installer,
  streaming its output) → **initializing** (`ipfs init` if the repo is new) → **starting** (spawn the
  daemon, wait for health) → **done** (dashboard renders) — or **error** at any step.
* On **done**, the page flips to the running dashboard automatically.
* On **error**, the progress view becomes the **error panel** (§9) carrying the failed phase, the
  captured output tail, and the **manual command** to copy — the guaranteed escape hatch.

### 7.3 Where install can also be triggered

* From **this page's** Install card / Turn-on button.
* From the **app-wide banner** (§10) on any other page.
* From the **install-tools preflight** ([pm/tools.mdx](./tools.mdx)) when a scan needs `ipfs` and it's
  missing. All three drive the **same** install job — never two at once (single-flight).

## 8. Security posture card

Unchanged in model from what ships on the pinset page — reproduced here because the dashboard is the
node's home. It reflects and **guards** the [only-our-content rule](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule):

| Field | Compliant value |
| --- | --- |
| **Reprovide** | `pinned` (or `roots`) ✓ — never `all`. |
| **Gateway** | **local-only** (127.0.0.1) ✓ — never public/recursive. |
| **GC** | **on** ✓ — incidental third-party cache stays transient. |

If the node announces `all`, or exposes a public/recursive gateway, the card shows a **red banner**
*"This node is configured to serve more than your own content"* with a **Fix** that restores the
[defaults](../knowledge/ipfs.mdx#10-concrete-defaults-lfb-should-adopt). The one accepted exception:
the user **deliberately** flipped the public-gateway [setting](./settings.mdx) on this machine — then
it's an amber *acknowledged* note, not a red error.

Restoring the only-our-content defaults behind the security card's **Fix** is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs.service.ts` — `enforceCompliance()`

> **LOCKED — enforce writes the MODERN key.** Enforcing compliance must set **`Provide.Strategy`**, never
> `Reprovider.Strategy`. On Kubo 0.42+ writing the deprecated `Reprovider.Strategy` **re-creates** the
> `Reprovider` block, which makes the daemon FATAL on the next start — i.e. "enforcing compliance" would
> silently re-arm the exact crash this feature exists to fix (§14). We fall back to the legacy key only
> on an older build that lacks `Provide` entirely. Read compliance from `Provide.Strategy` first, too.

## 9. Errors — the exhaustive taxonomy (every failure is a designed state)

The page never dead-ends. Every way IPFS can fail on this computer has a **named state**, a **plain
cause**, and a **concrete next step** — and, whenever the failure is repairable by us, a **one-click
fix** (config problems route into the guided repair, §14). We deliberately **do not** collapse
distinct failures into one generic "something went wrong": the reason the user hit a misleading *"the
daemon didn't come up in time"* is exactly that the real cause (a **deprecated config key** that made
Kubo exit immediately) was hidden behind a timeout. The rule is: **surface the real cause, never a
symptom.**

* An error **replaces only the region it belongs to** — a dead metric tile (row 13) never blanks the

* The **manual command is always reachable** from any install / init / start / migrate / upgrade

* **Nothing is auto-retried silently** and **nothing that edits config is applied without an explicit
  click** — the user clicks **Retry** / **Fix** (charter consent posture, §14).
* **Every failure is logged to the fault trail** (`error.err`) via the app's logging system, with
  enough context (phase, captured output) to diagnose after the fact (§19).

## 11. Where the data comes from

* A single **GET `/api/ipfs/node`** returns the dashboard payload: installed, running, version,
  peerId, platform, install hint (the manual command), the metric block (pins, repo stat, swarm,
  bandwidth), the gateway summary, the posture (reprovide / gc / compliant / publicGateway), and the
  **`autostart`** block (`{ supported, installed, enabled, lastExitCode, lastRunFailed, failureReason,
  conflict }` — the reboot auto-start state, §13). Note `enabled` means launchd will **try**;
  `lastRunFailed` says whether it actually **worked** (§13.1) — the UI must read both.
  Metric reads are **best-effort and bounded**: a slow or unset value comes back null → the tile shows
  `—`. Reading is **metadata-only** (like the scan) — it opens no file and moves no bytes.
* **POST `/api/ipfs/daemon`** `{action:"start"|"stop", autostart?:boolean}` — drives the on/off toggle;
  returns the fresh node payload (or kicks a job, below). On `start`, `autostart:true` **also** sets up
  reboot auto-start once the daemon is healthy (the off-page's primary button, §12).
* **POST `/api/ipfs/autostart`** `{action:"install"|"remove"}` — set up or remove reboot auto-start
  directly (the running dashboard's toggle, §13); returns the fresh node payload so the UI settles on
  the real launchd state.
* **POST `/api/ipfs/install`** — starts the install **job** (single-flight); **GET
  `/api/ipfs/install/status`** — polls the job (phase + log) for the progress view and the banner.
* The **cheap** summary the banner polls is the same `/api/ipfs/node` narrowed to `{installed,
  running}` (frontend `select`), so there's no second source of truth.
* Install/start jobs are **server-side** and detached from the request (re-attachable, cancel-safe on
  navigation) — same discipline as the scan job.

The dashboard's REST surface (`GET /node`, `POST /daemon`, `POST /install`, `GET /install/status`, `POST /autostart`) is defined in code:

## 13. Auto-start on reboot (keep IPFS on)

The problem: after a machine restart, the IPFS daemon is **not** running, and nothing brings it back —
so files silently stop pinning until the user notices and turns it on. Auto-start fixes that.

* **Mechanism (macOS).** A per-user **launchd LaunchAgent** — `com.largefilebridge.ipfs` — written to
  `~/Library/LaunchAgents/`, running the **absolute** `ipfs daemon --enable-gc` at **login/boot**
  (`RunAtLoad`). It is a **sibling** of the scan/pin worker agents ([pm/scan.mdx](./scan.mdx)) but a
  distinct, purpose-built plist.
* **`RunAtLoad` yes; `KeepAlive` retries a FAILED start only.** We want *"start once at reboot/login,"*
  **not** *"relaunch whenever it exits"* — otherwise the app's own **Off** toggle would be fought by
  launchd instantly restarting the daemon. But plain `KeepAlive = false` had a hole that cost the user
  this feature entirely (§13.2): a start that **failed** also stayed dead, forever, until the next
  reboot. So `KeepAlive = { SuccessfulExit: false }` (+ `ThrottleInterval 30`): a deliberate **Off**
  (`ipfs shutdown`) exits **0** → launchd leaves it alone, so the toggle still wins; a **failed** start
  exits **non-zero** → launchd retries. This is the documented intent, finally implemented.
* **Absolute binary + pinned PATH.** launchd has no login shell PATH, so we resolve the absolute `ipfs`
  path (`command -v ipfs`) and pin a conservative PATH (`/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin`)
  so both Apple-silicon and Intel Homebrew installs work.

### 13.1 "Registered" is NOT "working" — the status must not lie

**This was a real bug, and it is the one the user reported.** The IPFS page said *"Start on reboot:
on ✓"* while IPFS was off after **every** reboot. Both readings came from the same machine at the same
moment, and the page was simply wrong.

The cause: `autostart.enabled` was defined — **in this spec** — as *"`launchctl print` succeeds."* But
`launchctl print` succeeding only proves the job is **REGISTERED with launchd**. It says **nothing**
about whether the job ran, or died. The agent was registered *and dead*:

```
state = not running   ·   runs = 1   ·   last exit code = 1
```

**The rule now:**

* **`enabled`** = registered **AND** not disabled → launchd will **try** at the next boot.
* **`lastRunFailed`** = it ran at the last boot and **exited non-zero** → it **did not work**.
* **`failureReason`** = the daemon's **own last error line** (from `ipfs-autostart.err`), so a failure
  **names its cause** instead of shrugging.
* **The UI only claims "on ✓" when `enabled && !lastRunFailed`.** A failed agent renders **amber** —
  *"set up, but it failed at the last reboot (exit 1) — IPFS did not come back"* — plus the real reason.
* **Never a cheerful default.** When launchd can't be read at all, every field reports **not working**.

> **Deriving "it works" from "it is registered" is the bug.** Any future state added here must be read
> from what launchd actually reports, not inferred from the presence of a plist.

### 13.2 One daemon, one owner — the repo-lock race

**The reason auto-start actually failed.** The user had run `brew services start kubo` long ago,
**outside this app**. That installs `homebrew.mxcl.kubo` with `RunAtLoad`. So at every login **two**
LaunchAgents raced to start `ipfs daemon` and take `~/.ipfs/repo.lock`:

```
Error: lock /Users/bryan/.ipfs/repo.lock: someone else has the lock
```

The loser exited 1 and — with the old `KeepAlive = false` — **was never retried**. Whether IPFS came
back after a reboot was a **coin flip we did not control**, and on this machine we lost it.

* **We detect a foreign auto-start agent** — any non-LFB launchd job whose `ProgramArguments` run an
  `ipfs` `daemon` — in `~/Library/LaunchAgents`, `/Library/LaunchAgents`, `/Library/LaunchDaemons`.
* **We refuse to become the second owner.** If one exists, `installAutostart()` does **not** write a
  competing plist. Something already starts IPFS at login; the user's intent (*keep IPFS on*) is
  **satisfied**, just not by our agent. Adding ours would **re-create the bug**, not fix it.
* **We say so, plainly**, naming the other owner: *"Homebrew (brew services) also starts IPFS at login
  (homebrew.mxcl.kubo) and is running it now. Large File Bridge is letting it do that rather than
  starting a second daemon that would fight it for the repository lock."*
* **We never touch the user's brew setup on our own.** Stopping or disabling a service the user
  configured outside this app is an explicit, **confirmed** action ([dialogs.mdx](./dialogs.mdx)) —
  never a silent side effect of pressing a button in our UI. *(Trade-off worth surfacing: Homebrew's
  plist runs plain `ipfs daemon`, without our `--enable-gc`.)*
* **A foreign owner still means "on ✓".** The row answers *"will IPFS come back after I reboot?"* — and
  when Homebrew starts it, the honest answer is **yes**, credited to its real owner (*"on ✓ — Homebrew
  (brew services) starts IPFS when you log in"*). Reporting **off** merely because **our** agent isn't
  the one doing it would be a **new lie in the other direction**, and the §13.1 rule cuts both ways.
* **The control acts only on OUR agent, and disappears when it has nothing to do.** With a foreign owner
  and no agent of ours: **no button** — "Turn off" would be a no-op that reads like it stops Homebrew's,
  and "Turn on" would promise the competing agent we deliberately refuse to create. When a **stale** LFB
  agent is still losing the race, the button becomes **"Clear the conflict"** (removes ours only).

### 13.3 This whole section is scheduled for deletion

**The honest framing:** §13 exists only because IPFS is a **daemon we do not own**, provisioned by
**brew**, whose lifecycle we can only influence through launchd — where anyone else may also be an
owner. Every fix above is **damage control around that fact**.

An **in-process node ([Helia](./external_tools_policy.mdx))** has **no daemon, no `repo.lock`, no
LaunchAgent, and no second owner** — this bug becomes **unreachable** rather than fixed, and §13, §14,
§15 and the whole IPFS-off page are deleted with it. "Keep IPFS on across reboots" becomes "keep Large
File Bridge on across reboots": **one agent we fully own** instead of two we half-own.

See [external_tools_policy.mdx](./external_tools_policy.mdx) §4.1 for the migration and its gates —
including the DHT provide/reprovide risk that must be proven on a real two-computer pull **before** the
Kubo path is deleted. Until that lands, §13.1–13.2 are the contract.
* **Where the user controls it.**
  * **Off page (§12.1)** — button 1 turns it on as part of starting IPFS.
  * **Running dashboard (§5.1)** — a compact *"Start on reboot: on ✓ / off"* row with a one-click
    **Turn on / Turn off auto-start** toggle, so a user whose node is already up can set it up (or
    remove it) preemptively.
* **Security posture unchanged.** The auto-started daemon runs the same `--enable-gc` and is brought to
  only-our-content compliance on start (§8); auto-start changes *when* the daemon runs, not *what* it
  serves.

## 14. Config health & guided self-repair (the crash the user actually hit)

This section exists because of a **real incident**: the user pressed **Turn on IPFS** and the app said
*"The daemon didn't come up in time."* It hadn't timed out at all — Kubo **0.42** exits **immediately**
with `FATAL … Deprecated configuration detected. Manually migrate 'Reprovider' … Remove 'Reprovider'
from your config.` The daemon crash-looped, health never came up, and the app mislabeled a **config
crash** as a **timeout**. A human had to open a terminal, hand-edit `~/.ipfs/config`, and restart it.
**The web app must do that work for the user.** This section is that capability.

### 14.1 What "config health" checks

We read the node's config file (`$IPFS_PATH/config`, default `~/.ipfs/config`) and classify it — a
**metadata-only** read that opens nothing else. The health report is a list of **issues**, each with a
severity, a plain-language title, the exact keys involved, and whether we can **auto-fix** it.

Reading `$IPFS_PATH/config`, parsing it, and classifying every issue (missing / unreadable / deprecated / non-compliant) into a health report is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-config-health.service.ts` — `configHealth()`

| Issue class | Severity | Example / trigger | Auto-fixable? |
| --- | --- | --- | --- |
| **Missing** | blocker | no `config` file (repo never `ipfs init`-ed) | yes — offer `ipfs init` (turn-on path §7.2) |
| **Unreadable / corrupt** | blocker | file exists but isn't valid JSON | partial — offer to back up + recreate defaults |
| **Deprecated key present** | blocker (crashes daemon) | `Reprovider.*` on Kubo ≥ 0.42 → migrate to `Provide.*` | **yes** — the one-click migration (§14.3) |
| **Repo needs migration** | blocker | repo version behind the binary (`ipfs daemon --migrate`) | yes — run the migration as a job |
| **Non-compliant posture** | warn | reprovide `all` / public gateway | yes — the only-our-content Fix (§8) |
| **Suspicious value** | info | e.g. `StorageMax` unset/huge, GC off | offer a sane default, never forced |

Severity drives urgency: a **blocker** means IPFS **cannot run** until it's resolved, so it takes over
the off-page (§17.2) with the repair front-and-center; a **warn/info** shows as a quiet card on the
running dashboard and never blocks.

### 14.2 Surfacing the REAL start-failure reason (no more fake timeouts)

The turn-on flow (§7.2, §16) no longer reports a bare *"didn't come up in time."* When the daemon
doesn't answer health within the window, we **read the tail of the daemon log** (`ipfs-daemon.log`) and
**classify** it before we fail the job (the taxonomy in §9): deprecated config, repo-needs-migrate,
port busy, lock held, or genuinely-slow/unknown. Only an **unclassified** stall is reported as a
timeout — and even then we show the captured log tail.

Classifying the daemon's own output (deprecated-config / migrate / port-in-use / lock-held / unknown) into a cause + suggested fix is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-config-health.service.ts` — `diagnoseStartFailure()`

Wiring that diagnosis into the start/install jobs — so the job's `error` names the real cause and, when it's a config crash, hands off to the repair — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-node.service.ts` — `startDaemon()` / `startInstall()` / `controlDaemon()`

### 14.3 The Config-health card — review, then fix (confirm-then-apply, LOCKED)

We **never rewrite the user's config on our own.** The flow is always **surface → explain → confirm →
apply → verify**:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  ⚠  IPFS can't start — its configuration needs a small fix                    │
│                                                                               │
│  Kubo v0.42 removed an old setting your config still has. Until it's          │
│  migrated, the IPFS service exits immediately (this is why "Turn on"          │

│  What we'll change:                                                           │
│   • Remove the deprecated  Reprovider  block                                  │
│   • Keep  Provide.Strategy = "pinned"  (your announce-only-our-content        │
│     setting — unchanged in effect)                                            │
│   • Back up your current config to  ~/.ipfs/config.bak.<timestamp>  first     │

```

* The card **lists each change in plain language** before the user commits — what is removed, what is

* **Confirm-then-apply.** Pressing **Review & Fix** is the consent. On click we: (1) **back up** the

* **Always a manual escape hatch.** A disclosure shows the exact commands (`cp config config.bak…`,

## 15. Keeping IPFS current — the upgrade flow

IPFS moves fast, and an **out-of-date** Kubo is exactly what caused the incident (the config-key
removal landed in a release). So the dashboard **notices when the installed version is behind** and
offers a guided upgrade — with the same **confirm-then-do** posture as everything else.

### 15.1 How we decide "old"

* We compare the installed `ipfs version` against a **recommended-minimum baseline baked into the app**

## 16. The redesigned "Turn on IPFS" experience (great UI above, terminal below)

The old turn-on was a plain button that dropped straight into a black terminal log — and when the
daemon crashed, that log was the *only* thing the user saw, with a misleading title. The redesign
leads with a **clear, friendly status hero** and treats the terminal as **secondary, collapsible
detail**.

### 16.1 Layout — hero first, terminal second

```
┌─────────────────────────────────────────────────────────────────────────────┐
│   ◐  Starting IPFS…                                              phase: starting│  ← BIG status hero
│      Bringing your node online so your big files can be pinned.                │
│      ● Initialize repository   ● Start service   ○ Verify health   ○ Keep-on   │  ← step chips (lit as they pass)
│                                                                               │
│   ▸ Show technical details (terminal)                                          │  ← collapsed BY DEFAULT
│   ┌─────────────────────────────────────────────────────────────────────┐    │
│   │ $ ipfs daemon --enable-gc                                           │    │  ← the existing log, now tucked away
│   │ Initializing daemon…                                               │    │
│   └─────────────────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────────────────┘
```

* **The hero is the story:** a status glyph (spinner / green check / amber warning), a one-line
  human sentence, the current **phase**, and a row of **step chips** (*Initialize · Start · Verify ·
  Keep-on-reboot*) that light up as each phase passes. A non-technical user watches the chips, not the
  log.
* **The terminal is demoted** to a **"Show technical details"** disclosure, **collapsed by default**.
  It's the same append-only log (unchanged data), kept for the power user and for support — but it no
  longer *is* the UI.
* **On success**, the hero flips to a green *"IPFS is running"* and the page settles onto the running
  dashboard (or, on the primary path, confirms *"…and it'll start automatically on reboot ✓"*).
* **On failure**, the hero flips to the **diagnosed** state (§9/§14): a config crash becomes the
  Config-health card with the real reason, **not** a scary red log. The terminal detail is still one
  click away.

## 17. The three start-up scenarios & first-run routing

Per the prompt: **when a user starts the web app**, if IPFS is (1) **not installed**, (2) **not
running**, or (3) **not set to restart on reboot**, the app should **guide them to the UI** that fixes
that specific scenario. These are three distinct destinations with three distinct designs.

How the app picks the scenario on entry — checking installed → running → autostart, in that priority order — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/IpfsStatusBanner.tsx` — `IpfsStatusBanner` (chooses the nudge + destination) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/ipfs/IpfsOffPage.tsx` (the destination pages)

Priority order (the first true one wins): **not installed → not running → running but no reboot
auto-start**. A machine that is installed + running + auto-starting is fully healthy → **no nudge at
all** (the charter's no-nagging rule holds).

### 17.1 Scenario A — IPFS is not installed

* **Trigger:** no `ipfs` CLI and no reachable daemon.
* **App-start behavior:** the app-wide banner reads *"IPFS isn't installed — your files can't pin,"*
  action **Install IPFS**, routing to **`/ipfs/off`**, which renders the **Install card** (§7.1).
* **Agree-button (Install) behavior:** starts the cross-platform install **job** (§7.2/§16) —
  detect → install → init → start → compliance → (offer keep-on-reboot). On done, redirect to `/ipfs`.
* This is a **blocker**: nothing pins without a node.

### 17.2 Scenario B — IPFS is installed but not running

* **Trigger:** CLI present, daemon not answering (the classic **post-reboot** state, or a crash).
* **App-start behavior:** banner reads *"IPFS isn't running — your files can't pin,"* action **Start
  IPFS**, routing to **`/ipfs/off`**, which renders the **two-button turn-on choice** (§12.1) — unless
  **config health** finds a blocker (§14), in which case the **Config-health card takes over** (you
  can't turn it on until the config is fixed; this is the crash-recovery path).
* **Agree-button behavior:**
  * **Turn On + keep on (primary)** → `POST /daemon {action:"start", autostart:true}` — start now
    **and** install reboot auto-start (§13). The redesigned progress hero (§16) shows all four step
    chips including *Keep-on-reboot*.
  * **Turn On only (secondary, after "Or")** → `POST /daemon {action:"start"}` — start now, **don't**
    touch the reboot setup.
  * **Review & Fix (when a config blocker exists)** → the guided repair (§14.3), then auto-retry the
    start.

### 17.3 Scenario C — running, but won't come back after a reboot

* **Trigger:** installed **and** running, but reboot **auto-start is off** (`autostart.supported &&
  !autostart.enabled`). Pinning works *now*, but the next restart silently drops the node — the exact
  failure this whole feature exists to end.
* **App-start behavior — deliberately gentle (LOCKED).** Because the node is **healthy right now**, we
  must **not** show the alarming red/amber "can't pin" banner (that would violate no-nagging). Instead
  we show a **low-prominence, dismissible, one-time** encouragement — an **info** strip (or the blue
  card on the dashboard, §18) reading *"IPFS is on, but it won't restart automatically after you
  reboot. Turn on auto-start so your files keep pinning."* with a single **Turn on auto-start** button.
  * It is **dismissible** and, once dismissed (or once auto-start is enabled), it does not return —
    this is encouragement, not a nag. It never blocks a page.
* **Agree-button (Turn on auto-start) behavior:** `POST /autostart {action:"install"}` (§13) — writes
  and loads the launchd LaunchAgent. On success the strip disappears and the dashboard's control flips
  to its quiet "on ✓" status (§18). No daemon restart, no interruption.

The dismissible, one-time auto-start encouragement (distinct from the blocking "can't pin" banner, shown only when running-but-not-auto-starting) is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/IpfsStatusBanner.tsx` — `IpfsStatusBanner`

## 18. Auto-start on reboot — the "encourage it" control & prominence rules

Auto-start is something we **want** the user to turn on (it's what keeps files pinning after a
restart), so when it's **off** we make the offer **prominent**; once it's **on**, we get out of the way
and reduce it to a **quiet status line**.

| Auto-start state | Prominence | UI |
| --- | --- | --- |
| **Off** (supported, not enabled) | **High — encourage** | A **filled blue** primary button *"Turn on auto-start"* with a one-line "why" (*so IPFS restarts itself after a reboot*). It reads as a call-to-action. |
| **On** (enabled) | **Low — inform** | **No filled button.** A muted status line *"Starts on reboot ✓"* with a small, secondary **Turn off** affordance. Not a CTA — just a confirmation. |
| **Unsupported OS** | **Low — inform** | A muted *"not available on this operating system yet"* line; no button. |

* The **blue fill is reserved for the off→on call-to-action** — the thing we're encouraging. When
  auto-start is already on, using a filled button would wrongly compete for attention; it becomes plain
  status text with only a understated way to turn it back off.
* This rule holds in **both** places auto-start appears: the running dashboard's **auto-start row**
  (§5.1/§13) and the off-page's two-button choice (§12.1, where "keep on reboot" is the **primary,
  filled** button precisely because it's the encouraged path).

The prominence switch — filled blue CTA when off vs. muted status line when on — is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/ipfs/ipfsShared.tsx` — `AutostartRow` component

## 19. Logging every IPFS fault to the error trail

Everything that can go wrong with IPFS — an install exit code, a daemon that won't start, a config
crash, a failed migration, an upgrade error, a launchd hiccup, a compliance-fix failure — is written
to the app's **existing logging system**, so there is always a durable fault trail to diagnose from
(the user pressed a button, it failed; the *why* must be recoverable after the fact).

* **One logging system, two files.** The app already writes an append-only `log.log` (everything) and
  `error.err` (the complete WARN/ERROR/FATAL fault trail) at the state-root log dir. IPFS code logs
  through it — never `console.log`, never a bespoke file.
* **Severity discipline (so `error.err` stays meaningful):**
  * **`log.error`** — a genuine fault the user cares about: a failed start we couldn't recover, a
    failed config repair, a failed install/upgrade, a failed pin/add. These land in **`error.err`**.
  * **`log.warn`** — a soft/expected-ish problem: a launchd non-zero on a benign case, a best-effort
    metric read that failed, a compliance enforce that couldn't apply. Also in `error.err` but marked
    WARN.
  * **`log.debug`** — routine "node is just off" probes (health/peerId/version when unreachable) —
    **kept out of `error.err`** so a normally-off node doesn't spam the fault trail.
* **Context tag `ipfs`.** Every line carries the `ipfs` context so the trail is filterable.
* **The crash we hit is now logged loudly.** A `FATAL … Deprecated configuration` from the daemon is
  captured, classified (§14.2), and logged at **ERROR** with the diagnosed cause — not swallowed behind
  a timeout.

The two-file logging system (`log.log` = everything, `error.err` = the WARN/ERROR/FATAL fault trail) IPFS code writes through is defined in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/logging.ts` — the exported `log`

The IPFS fault sites that must record to the trail (start-failure diagnosis, config repair, upgrade, autostart, compliance) log via `log.error` / `log.warn` in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs-config-health.service.ts`, `ipfs-node.service.ts`, `ipfs-autostart.service.ts`, and `ipfs.service.ts`
