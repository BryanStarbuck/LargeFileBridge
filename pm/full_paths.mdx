# LargeFileBridge — Full Paths

The **Full paths** tab is the **flat, whole-tree** companion to the [File System](./directories.mdx)
column browser. Both live under the **File System** left-bar item ([left_bar.mdx](./left_bar.mdx));
a small **tab strip** at the top of the File System area switches between them:

* **Column view** (`/fs`) — the Finder/Miller column browser ([directories.mdx](./directories.mdx)):
  one directory level per column, names only, drill down to the right.
* **Full paths** (`/fs/paths`) — *this page*: a single **TanStack table** of the **large files
  under a chosen root, recursively**, one row per file, whose two lead columns are the **file name**
  and its **full absolute path**.

The column browser is for *walking*; Full paths is for *acting on many files at once*. It looks like
a file system but it is not one — it is a **Repos-style table** ([repos.mdx](./repos.mdx)) whose rows
happen to be files gathered from every depth below the root. That flatness is the point: you can sort
every large file on the machine by size, filter to the ones that aren't in IPFS yet, check a batch,
and pin them in one click.

Siblings this spec builds on and does **not** repeat:

* [Directory / File System](./directories.mdx) — the **code badges** (R/r · P · C/c · i, §3) painted on
  every row here, and the recursive, **no-shell**, metadata-only walk discipline (§2/§AC9).
* [One repo](./one_repo.mdx) — the **decision model** Pin / Ignore / Undecided (§1) that **IPFS pin /
  Unpin** drive, and the standard **control row** + **pagination default 500** conventions.
* [IPFS](./ipfs.mdx) — what "in IPFS / pinned" means; pinning here is the same **decision → sync**,
  and the only-our-content posture is unchanged (nothing here turns this machine into a gateway).
* [Menus](./menus.mdx) — the per-row **`⋮`** kebab / right-click **file** catalog (§5.3) and the two
  sticky flags (§6.6); this page renders that catalog, it never invents a different one.
* [Repos](./repos.mdx) — the house flat/chromeless **DataTable**, the control-row and pagination rules.
* [scan.mdx](./scan.mdx) / [settings.mdx](./settings.mdx) — the **big-file threshold** that scopes
  which files appear (§2) and the hard-skip set the walk honors.

## 1. Scope — which files are rows (LOCKED)

Full paths is a **large-file** table, not an every-file table. A flat recursive list of *every* file
under home would be unusable (hundreds of thousands of rows) and pointless for the pin/compress
workflow. So the rows are exactly the files the rest of LFB already cares about:

* **A row is a file at or above the big-file threshold** ([settings.mdx §1](./settings.mdx)) found
  anywhere below the chosen root, at **any depth**.
* The walk is **recursive, metadata-only, Node `fs` only** — never the `find`/`du` shell
  ([directories.mdx §2](./directories.mdx), charter + macOS-indexing rule). It stats sizes; it opens no
  file bytes (the only capped content peek is the badge module's IPFS-list detection, [directories.mdx
  §3.4](./directories.mdx)).
* It honors the same **hard-skip set** (`.git`, `node_modules`, `.Trash`, `.cache`, `Caches`) and
  hides dotfiles by default ([scan.mdx §4](./scan.mdx)); symlinks are not followed.
* The walk is **bounded** (a file cap and an iteration budget). If the cap is hit, the page shows a
  **`truncated`** note ("showing the first N — narrow the root") so a capped list never reads as
  complete (charter: no silent truncation).

The recursive, metadata-only, Node-`fs`-only large-file walk (cap, iteration budget, hard-skip, threshold) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fsindex/fsindex.service.ts` — `walkFilesFlatStreaming()` (buffered via `fs.service.ts` → `listFilesFlat()`).

Directories are **never** rows here (this is a file table; the column browser is where directories
live). The chosen **root** defaults to the OS **home directory**, with a **Home** button and an
absolute-path input to re-root (and a `?path=` deep link, same as [directories.mdx §1](./directories.mdx)).

## 3. The control row — the facet rail (LOCKED)

**This section was rewritten; its LOCKED status is intact but its control changed.** Full paths
originally specified this row as **three segmented controls** (`Both · Compressed · Uncompressed`,
`Size · Name · Path`, `In IPFS · Not · Both`) because filters that live inside a dropdown are filters
nobody uses. That instinct was right and is now **product-wide law**: [tables.mdx §2.2](./tables.mdx#22-the-facet-rail--filtering-without-opening-a-dropdown-locked)
makes an always-visible **facet rail** the shared filter surface for every table, and it explicitly
**credits this page** for pioneering it. Only the *control* was wrong — a segmented control is a
one-of-N radio in different clothes and cannot say *"show me X and Y but not Z"*, so
[tables.mdx §2.6](./tables.mdx#26-retiring-the-both--x--y-segmented-filters-locked) retires every
`Both · X · Y` **filter** here into a two-value checkbox facet whose all-checked resting state **is**
the old **Both**. This is a promotion of the idea, not a repudiation of it.

So the control row is now: the house **search box**, the **`⛛`** and **`⇅`** icons, the shared
**active-sort chip run**, and — directly beneath — the **facet rail** (§3.1/§3.3), plus the **action
row** (§3.5). Nothing on this page is bespoke any more; the whole row is the shared `DataTable`
surface. **The model is not restated here** — see [tables.mdx §2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked)
for the grammar (a facet is its values; every box checked by default; all-checked = inactive;
checkboxes OR within a facet, facets AND; no `any`/`Both` pseudo-value; the last checked box cannot be
unchecked). Only what is **page-specific** is below.

```
 ┌──────────────────────────────────────────────────────────────────────────────────┐
 │ [ 🔍 search name + path…    ]        Sort: [1 Bookmarked ↓] [2 Size ↓]    ⇅   ⛛  │ ← control row
 │ ☐ Bookmarked only  ·  Compressed ☑ Compressed ☑ Uncompressed  ·                  │
 │ Pin (IPFS) ☑ Pinned ☑ Not pinned ☑ Recommended ☑ Undecided  ·                     │ ← facet rail
 │ Git-ignore ☑ Git-ignored ☑ Not git-ignored ☑ Recommended                          │
 └──────────────────────────────────────────────────────────────────────────────────┘
```

### 3.1 The **Compressed** facet — ☑ Compressed ☑ Uncompressed (both checked by default)

The old `Both · Compressed · Uncompressed` segmented control is now a **two-value facet**. Its
derivation is unchanged — the values read the **C/c** badge
([directories.mdx §3.3](./directories.mdx)):

| Value (checkbox) | Shows |
| --- | --- |
| ☑ **Compressed** | Files already compressed — the light-yellow **`c`** state. |
| ☑ **Uncompressed** | Files that **could be compressed but are not** — the bright-yellow **`C`** state (a video/image that looks uncompressed). This is the **"offer to compress" backlog**, and unchecking *Compressed* is how the user isolates it. |

* **Both checked is the resting state** and is exactly what the old **Both** segment meant: every large
  file — compressed, uncompressed, and non-media alike — with no constraint applied
  ([tables.mdx §2.6](./tables.mdx#26-retiring-the-both--x--y-segmented-filters-locked)).
* **Non-media files carry neither badge**, so the compress axis does not apply to them; per
  [tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked) a row whose axis does not
  exist is **never hidden by that facet** — an `.iso` is not secretly filtered out because the user
  unchecked *Compressed*. (This is the same rule §3.3 leans on for git-ignore.)

### 3.2 Sort — the shared chip run, not a segmented control (LOCKED)

**Sort is a genuine one-of-N-per-slot control, not a filter facet**, so §2.6 does not retire it — but
[tables.mdx §3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked)
now gives **every** table a live **active-sort chip run** on the control row, which does everything the
`Size · Name · Path` segmented control did and more. **The two are redundant, and the chip run wins:**

* **Recommendation (LOCKED direction): retire the segmented sort control and adopt the shared chip
  run.** Keeping both would mean two controls driving one state, and it would leave this page as the
  only table in the product with a bespoke sort widget — the exact inconsistency §2.6 removed on the
  filter side. The chip run is strictly more capable: the segmented control could only set the
  **primary** key from a fixed list of three, while the chip run shows and edits **all** active keys.
* **It drives the same multi-level state** the segmented control drove
  ([tables.mdx §3](./tables.mdx#3-multi-level-sort--primary--secondary--tertiary-locked)): chips,
  headers, and the `⇅` dropdown are **one** state. Click a chip's caret → asc ⇄ desc (what toggling an
  active segment did). Click a header → promote to primary (what picking a segment did).
  **Shift**-click a header → append the next key (what the segmented control could never do).
* **Size · Name · Path do not disappear as sort keys** — they are simply columns, sortable by header
  click and listed in the `⇅` dropdown like every other. Sorting on **Size** is still on **raw bytes**;
  **Path** still groups the table by folder because paths sort lexicographically.
* **The page default is unchanged: Bookmarked ↓ then Size ↓** — bookmarked files float to the top,
  biggest-first within ([tables.mdx §3.4](./tables.mdx#34-the-default-sort--bookmarks-first-locked-default-user-changeable)).
  It renders at rest as `Sort: [1 Bookmarked ↓] [2 Size ↓]` and applies until the user re-ranks; their
  choice then persists (§3.4).

### 3.3 The **Pin (IPFS)** facet — and why this page also gains a **Git-ignore** facet

The old `In IPFS · Not · Both` segmented control becomes the shared **Pin (IPFS)** facet, with the
fixed vocabulary of [tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked) —
**no page-local wording**:

| Value (checkbox) | Shows |
| --- | --- |
| ☑ **Pinned** (`sync`) | Files we already pin — the pink **`P`** badge ([directories.mdx §3.2](./directories.mdx), [one_repo.mdx §1](./one_repo.mdx)). |
| ☑ **Not pinned** (`ignore`) | Files **deliberately not pinned** — a real, decided answer, **not** "git-ignored". |
| ☑ **Recommended** | The calc engine suggests pinning ([to_do_batch_calc_engine.mdx](./to_do_batch_calc_engine.mdx)). |
| ☑ **Undecided** | Never asked. |

* The old **Not** segment conflated *"deliberately not pinned"* with *"never asked"*; the four-value
  facet separates them, which is the whole point of the page's workflow — **Undecided ✓ + Recommended
  ✓, everything else unchecked** is the precise "big files that aren't backed up yet" list the old
  three-segment control could only approximate.
* All four checked = the old **Both** (every large file regardless of IPFS membership).

**This page must also carry a Git-ignore facet (NEW).** [tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)
is unambiguous: a table that filters the pin axis must not imply anything about the git-ignore axis, and
the two are **always two facets, never one**. Full paths today offers **no** git-ignore filter at all,
which leaves the user with a pin facet whose `Not pinned` value is the historical source of exactly the
confusion §2.4 outlaws. So the **Git-ignore** facet (☑ Git-ignored · ☑ Not git-ignored · ☑ Recommended,
read live from `git check-ignore`) joins the rail.

**How it handles files outside any repo — always present, N/A rows exempt (LOCKED).**
[decision_toggles.mdx §6](./decision_toggles.mdx) scopes the git-ignore axis to rows **under a repo**;
Full paths walks an arbitrary root (home by default) and therefore lists files **inside** repos and
**outside** every repo in one table. The answer is **not** to show/hide the facet with the root — that
would make the vocabulary flicker and re-hide it, the very failure §2.2 exists to prevent. Instead:

* **The Git-ignore facet is always on the rail**, with its full §2.4.1 vocabulary.
* **A row outside any repo is N/A on that axis and is never hidden by that facet** — the rule already
  LOCKED in [tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked) (*"N/A is not a
  facet value… a row where the axis does not apply is filtered by the axis's own facet only when that
  axis exists on the row"*). It is the same exemption §3.1 gives non-media files on the Compressed
  facet, so the page has one rule, not two.
* **One line of justification:** a permanent facet whose N/A rows opt out keeps the vocabulary readable
  at every root and matches the row cells — the ⊘ toggle is already rendered N/A-but-width-reserved for
  those same rows ([directories.mdx §3.6](./directories.mdx)) — whereas a facet that appears and
  disappears with the root would teach the user that the axis does not exist.
* **Affordance:** when the current root is **outside** every repo, the facet's label carries a muted
  **`(repo files only)`** hint so an N/A-heavy result never reads as a broken filter.

### 3.4 House search + the `⛛` / `⇅` icons — the new division of labour (charter)

With the facets on the rail, the two icons keep only what a rail chip cannot express
([tables.mdx §2.3](./tables.mdx#23-what-survives-in-the-filter--popover-locked)):

* a **search box** (left) — case-insensitive substring over **name + path**. Unchanged.
* **`☐ Bookmarked only`** — **promoted out of the `⛛` dropdown to the head of the facet rail**
  (leftmost, visually separated). It stays the one **single checkbox** rather than a two-value facet —
  the single documented exception ([tables.mdx §2.7](./tables.mdx#27-the-promoted-bookmarked-only-filter-locked)).
* the **`⛛` filter icon** — now **only**: the **min size** threshold, the **name/path contains**
  free-text, and narrow-width **facet overflow**. No enum facet lives inside it. It stays **colored
  whenever any filter is active — rail facets included** — so it remains the single honest "something is
  filtered" indicator.
* the **`⇅` sort icon** — the **1st/2nd/3rd** priority-slot editor and **Clear sort**
  ([tables.mdx §3.1](./tables.mdx#31-the-sort-dropdown-ui-locked)); it is the **discovery/overflow**
  surface for a sortable column that is not currently in the chip run, and shares one state with the
  chips and the headers (§3.2).

This page's **sort + filter persist per user** in `users/<email>/config.yaml` `tables.views`
([tables.mdx §4](./tables.mdx#4-persistence--sort--filter-saved-per-user-per-page-locked)); its facets
persist as `hidden:` — the **unchecked** values — never `shown:`
([tables.mdx §4.1](./tables.mdx#41-facets-persist-as-the-hidden-values-never-the-shown-ones-locked)).
The worked `files:` example in that section **is** this page.

#### 3.4a This page must stop hand-rolling its table (NEW — engineering note)

Full paths does **not** use the shared `DataTable`: `FullPathsPage` builds its **own** `useReactTable`
instance with a bespoke control row, and its sort state is **single-level only** (one column, one
direction) — which is why the segmented control and the multi-level model of
[tables.mdx §3](./tables.mdx#3-multi-level-sort--primary--secondary--tertiary-locked) never reconciled
here. **The fix is adoption, not another bespoke control:** this page should render the shared
`DataTable` and inherit the facet rail, the chip run, the priority slots, the `hidden:` persistence, and
the responsive column priority, rather than growing a second, page-local implementation of each. Every
behavior §3.1–§3.4 specifies is a *shared* behavior configured with this page's facet definitions
(Compressed, Pin (IPFS), Git-ignore) — none of it is a reason for a custom row.

to be implemented in code (the migration):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FullPathsPage.tsx` — `FullPathsPage` (replace its local `useReactTable` + segmented control row with the shared `DataTable` from `code/packages/frontend/src/components/table/DataTable.tsx`, passing its facet definitions).

### 3.5 Action row — `Select all · IPFS pin · Unpin`

Three action buttons operate on the **checked** rows (the header checkbox selects the current page;
**Select all** selects **every row in the current filtered set**, not just the page):

| Action | Effect |
| --- | --- |
| **Select all** | Checks every row that currently passes the filters (§3.1/§3.3/§3.4). |
| **IPFS pin** | For each checked file, sets its decision to `sync` — pins it and starts pinning it across peers ([one_repo.mdx §4.5](./one_repo.mdx), [ipfs.mdx §4](./ipfs.mdx)). Never moves or deletes bytes. |
| **Unpin** | For each checked file, sets its decision to **ignore** — stops pinning. The file **stays on disk**; only LFB's tracking/pin is removed ([menus.mdx §6.2](./menus.mdx)). |

A **"N selected"** counter shows beside the actions; the pin/unpin buttons are disabled with nothing
checked. Pin/Unpin reuse the per-entity decision endpoint, so — like everywhere else — a file that is
**not inside a registered repo** cannot be pinned yet; those are reported as skipped (a toast shows
`pinned N, skipped M`), never silently dropped. Both flags apply: a file with **Never IPFS** on
([menus.mdx §6.6](./menus.mdx)) is refused by pin and reported as skipped.

Batch pin/unpin over the checked rows (per-entity decision → sync/ignore, skip-and-report) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FullPathsPage.tsx` — `FullPathsPage` (the `pin` mutation), backed by `entity.service.ts` → `setEntityDecision()`.

**Header-click sort shares the chip run's state.** Clicking a column header **promotes it to the
primary** key, click-again toggles asc ↔ descending, and **shift**-click **appends** it as the next key
([tables.mdx §3.3/§3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked)).
The headers, the **active-sort chip run** (§3.2), and the `⇅` dropdown (§3.4) are **one** sort state —
clicking the **Size** header updates its chip in lock-step. **Every** column here is sortable by header,
not just the three the retired segmented control exposed.

**Height — fills the page.** This page's table is the **last** element on it — the control row, the
**facet rail**, and the action row (§3) all sit **above**, and **nothing** sits below the table. So it takes the site-wide
**full-page-height** default ([repos.mdx §3.3.1](./repos.mdx)): the body scroll region grows to the
bottom of the viewport (no dead white space when the window is tall), the page doesn't scroll, only the
rows do. In code: `fillHeight` (the default) on the `DataTable`, inside a full-height flex-column page.

* **Name** / **Path** → **View one file** ([files.mdx](./files.mdx)) for that file.
* **Size** / **Changed** → View one file (details).
* The row **`⋮`** and its checkbox never navigate.

## 5. Data model

The backend returns one flat listing the page renders directly (`@lfb/shared`,
`GET /api/fs/flat?path=<abs>&hidden=1`):

```ts
interface FlatFileListing {
  root: string;          // absolute directory the walk was rooted at (server-normalized)
  home: string;          // the OS home dir (the default root)
  thresholdBytes: number;// the big-file threshold that scoped this list (settings.mdx §1)
  files: FsEntry[];      // FILES only, gathered recursively; each carries badges (directories.mdx §5)
  truncated: boolean;    // true if the file-cap / iteration budget was hit (show the "narrow root" note)
}
```

`files[]` are the **same `FsEntry`** the column browser returns ([directories.mdx §5](./directories.mdx))
— `name`, absolute `path`, `sizeBytes`, `modifiedAt`, and the ordered `badges[]`. The frontend derives
the **Compressed** facet from the badges (**Compressed** = badge `compressed`, **Uncompressed** = badge
`compress`) and the **Pin (IPFS)** facet's `Pinned` value from badge `pin`. No new per-row shape is
needed for those — Full paths is a different *view* of the same file knowledge, not new knowledge.

**The two four-state axes do need the row's decision state.** The `Pinned / Not pinned / Recommended /
Undecided` and `Git-ignored / Not git-ignored / Recommended` facets (§3.3) cannot be derived from the
read-only badges — a badge only says "pinned or not". They read the **same** per-row decision state the
column browser's rows already carry ([directories.mdx §5](./directories.mdx), the note under `FsEntry`):
each axis's `on` / `off` / `recommended` / `n/a` value, plus **whether the entry is under a repo** (so
the git-ignore axis is N/A outside one, §3.3). Full paths inherits that shape; it does not invent one.

Streaming the flat listing into the page as rows arrive (NDJSON from `GET /api/fs/flat`) and windowing the large row set for scroll performance is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/useStreamedFlatListing.ts` — `useStreamedFlatListing` (rows windowed by `useWindowedRows.ts` — `useWindowedRows`).

## 6. Empty, truncated & loading states

* **No large files under this root:** a card — "No files at or above the big-file threshold under
  `<root>`. Lower the threshold in [Settings](./settings.mdx) or pick another folder."
* **A filter emptied the list:** the table body shows "No files match these filters" with the rail's
  **Clear filters** link ([tables.mdx §2.2](./tables.mdx#22-the-facet-rail--filtering-without-opening-a-dropdown-locked)),
  which re-checks every box on every facet (the all-checked resting state) and clears the `⛛`
  thresholds/contains. (This state is now **rarer by construction**: the last checked box in a facet
  cannot be unchecked, so a facet alone can no longer blank the table — [tables.mdx §2.1](./tables.mdx#21-a-facet-is-its-values-each-with-a-checkbox-all-checked-by-default-locked).)
* **Truncated:** a note above the table — "Showing the first N files (walk capped) — narrow the root
  to see everything." (`truncated: true`, §1.)
* **Loading:** skeleton rows sized to the columns; no layout shift.
* **Bad root** (missing / not a directory): the root bar shows the error inline and the table stays
  empty; the Home button re-roots.

## 7. Acceptance criteria

1. A **tab strip** at the top of the File System area offers **Column view** (`/fs`,
   [directories.mdx](./directories.mdx)) and **Full paths** (`/fs/paths`); both keep the **File System**
   left-bar item active.
2. **Full paths** is **one TanStack table** (house flat/chromeless) whose rows are the **large files
   under a chosen root, gathered recursively** — file **Name** and full absolute **Path** as the two
   lead columns, plus Size, Changed, right-aligned **code badges**, a **select** checkbox, and a
   trailing **`⋮`** kebab. The root defaults to **home**, with a Home button, a path input, and a
   `?path=` deep link.
3. The walk is **recursive, metadata-only, Node `fs` only** (no `find`/`du`), honors the **big-file
   threshold** (§1) and the **hard-skip** set, hides dotfiles by default, does not follow symlinks, and
   is **bounded** — a hit cap surfaces a **truncated** note (never silent).
4. **No segmented filter survives on this page** ([tables.mdx §2.6](./tables.mdx#26-retiring-the-both--x--y-segmented-filters-locked)).
   The **Compressed** facet is **☑ Compressed ☑ Uncompressed**, both checked by default (all-checked =
   the old **Both**), derived from the **C/c** badge; unchecking **Compressed** isolates the
   bright-yellow **C** backlog ("could compress but isn't"). Non-media rows have no compress axis and
   are never hidden by that facet.
5. **Sort is the shared multi-level model, not a bespoke control** (§3.2). The retired **Size · Name ·
   Path** segmented control is replaced by the **active-sort chip run** on the control row
   ([tables.mdx §3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked));
   chips, header clicks (shift-click appends), and the `⇅` dropdown's **1st/2nd/3rd** priority slots are
   **one** state. The **page default is unchanged: Bookmarked ↓, then Size ↓**.
6. The table carries the **shared table behavior** ([tables.mdx](./tables.mdx)): a leading **bookmark**
   toggle (col 0, stored per file by path + fingerprint in the owning pin unit's `config.yaml`), the
   **"Bookmarked only"** checkbox **promoted to the head of the facet rail**
   ([tables.mdx §2.7](./tables.mdx#27-the-promoted-bookmarked-only-filter-locked)), and **per-user,
   per-page persistence** of the sort + filter state in `users/<email>/config.yaml` — facets stored as
   `hidden:` (the **unchecked** values), never `shown:`
   ([tables.mdx §4.1](./tables.mdx#41-facets-persist-as-the-hidden-values-never-the-shown-ones-locked)).
7. **Pin (IPFS) and Git-ignore are two separate facets on the rail** — never merged, never nested, and
   **neither is named "Decision"** ([tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).
   **Pin (IPFS)** (☑ Pinned · ☑ Not pinned · ☑ Recommended · ☑ Undecided, all checked = the old **Both**)
   replaces the **In IPFS · Not · Both** control; unchecking **Pinned** reveals the un-pinned backup
   candidates. **Git-ignore** (☑ Git-ignored · ☑ Not git-ignored · ☑ Recommended, read live from
   `git check-ignore`) is **always present**; a row **outside any repo is N/A** on that axis and is
   **never hidden** by that facet (§3.3), and the facet's label carries a muted **`(repo files only)`**
   hint when the root is outside every repo.
7a. **The `⛛` popover holds only** the **min size** threshold, the **name/path contains** free-text, and
   narrow-width facet **overflow** — **no enum facet lives inside it** — and it stays **colored whenever
   any filter is active, rail facets included** (§3.4).
7b. **The page renders the shared `DataTable`** rather than its own `useReactTable` + control row
   (§3.4a), so the facet rail, the chip run, the priority slots, `hidden:` persistence, and responsive
   column priority are all **inherited**, not re-implemented here.
8. The **action row** — **Select all** (selects the whole filtered set), **IPFS pin** (decision →
   sync), **Unpin** (decision → ignore) — acts on checked rows, shows an **N selected** counter, never
   moves or deletes local bytes, and **reports skipped** files (not in a registered repo, or Never
   IPFS) rather than dropping them silently.
9. Each row's **Name/Path cells deep-link** to [View one file](./files.mdx); the **`⋮`** kebab and
   right-click open the **file** catalog ([menus.mdx §5.3](./menus.mdx)) and never navigate.
10. **Pagination defaults to 500** with the file count shown; empty / filtered-empty / truncated /
   loading / bad-root states render per §6; the house **search box** (name + path) and **filter icon**
   are present per the [Tables charter](../CLAUDE.md).
11. Nothing on this page ever turns this machine into a public IPFS gateway or alters local file bytes —
    pinning is the same only-our-content **decision → sync** as the rest of LFB
    ([ipfs.mdx §4](./ipfs.mdx), [knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx)).
