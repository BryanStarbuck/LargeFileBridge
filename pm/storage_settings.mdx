# LargeFileBridge — Storage Settings (Per-Storage)

Every storage can be configured on its own. This is the **per-storage settings page** — the screen you
land on from a **single storage** ([storages.mdx](./storages.mdx)). It is the storage counterpart of
[repo_settings.mdx](./repo_settings.mdx): a repo's gear opens the per-repo settings; a storage's gear
opens **this** page, scoped to **one storage**.

It answers two questions the charter leaves per-computer:

1. **Where does this computer keep this storage's tracking area?** For a **`repo`** storage that is
   "does this computer keep the hidden `.lfbridge/` directory, and where does it go?"; for a
   **personal / company / community SDL** there **is** no `.lfbridge/` — the storage **root** *is* the
   tracking area — so the question does not arise (§3).
2. **Which backing locations is this storage materialized into on this computer** — a **dedicated Git
   repo**, a **Google Drive** location, and/or a **Dropbox** directory — each independently **on/off**
   with its own directory?

LFB **picks good default directories**; the user can change any of them. The choices persist to a
**machine-local** per-storage config file (§5), which the **pin pass** then reads to **create and
update** the enabled directories (§6).

Siblings this builds on and does not repeat:

* [storages.mdx](./storages.mdx) — the storage family: `storage.yaml` (§3), the hidden (working-repo)
  `.lfbridge/` layout (§4), and the `clones` block that records Drive/Dropbox mirrors (§5).
* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — the **storage-KIND rule** that
  owns *whether `.lfbridge/` is used at all*: only a working git repo (kind `repo`) has one; an SDL's
  root **is** the `.lfbridge` area (§0.1), keyed on kind and never on a path or name (§0.2), with the
  SDL-root **reserved names** (§0.3) this page's mapped-dir validator enforces (§4a).
* [repo_settings.mdx](./repo_settings.mdx) — the per-repo settings page; a **repo is one kind of
  storage**, so a repo's gear may land on either its repo-settings or (for its storage facet) here — the
  two are kept aligned, never duplicated (§2).
* [google_drive.mdx](./google_drive.mdx) / [dropbox.mdx](./dropbox.mdx) — the cloud-drive integrations a
  backing location reuses (§4).
* [storage_local.mdx](./storage_local.mdx) — where the machine-local per-storage config lives under the
  state root (§5). · [pin_process.mdx](./pin_process.mdx) — the pass that materializes enabled
  locations (§6). · [menus.mdx](./menus.mdx) — the kebab / right-click / more-menu that opens this page
  (§1).
* [repo_company_mapping.mdx](./repo_company_mapping.mdx) — the owner model whose mapping feeds the
  **Owned repos** list (§4c) and whose §5 / §5.2 manual reassign this page's per-row reassign control reuses.
* [repo_owner_propagation.mdx](./repo_owner_propagation.mdx) — the **travel + consent** layer: reassigning
  a repo **into** a company with a sync repo records the travelling **assertion** (§4c); this page is its
  companion surface.

## 1. How you get here — gear, kebab, right-click, more-menu

## 2. Machine-local vs. shared — what lives where (LOCKED)

The one distinction this page turns on. A storage has **two** kinds of configuration, and they live in
**two** files:

* **Shared / travels with the storage** — the storage's **identity** and its **canonical** mirror paths.
  This is [`storage.yaml`](./storages.mdx#3-storageyaml--the-per-storage-descriptor-locked-shape): `name`,
  `type`, the type block, and the **`clones`** block (the canonical relative Drive/Dropbox paths). It
  travels between computers via the git backbone, so every machine agrees on *what* the storage is.
* **Machine-local / per-computer** — **this computer's** choices for the storage: for a **`repo`**
  storage whether `.lfbridge/` is kept and **where** it goes on this box (§3 — an SDL has no such
  choice), and whether each backing location is **enabled** and its **resolved local path** on this box.
  This is the **`pin/s/<storage_id>/config.yaml`** file under the
  Local state root (§5) — the local "settings file" (the `.ini`-style local config) this page writes.

Why split them: a Google Drive folder is at a different absolute path on a laptop than on a work tower;
a machine may have the dedicated repo cloned but no Dropbox; a server may keep a working repo's
`.lfbridge/` on a faster volume. Those are **per-computer** facts and must **not** be forced into the shared descriptor.
The descriptor says *where the Drive mirror lives relative to the drive*; the machine-local config says
*is it turned on here, and what is its absolute path here*.

For a **repo** storage, the per-repo settings in [repo_settings.mdx](./repo_settings.mdx) remain the
home of repo-specific behavior (threshold override, selection rules, pin policy, sharing); this page adds
only the **storage-facet** controls (the `.lfbridge/` placement — a **`repo`**-only control, §3 — plus
backing locations). The two never restate each other.

## 3. The tracking area — and the `.lfbridge/` control, which is `repo`-only (LOCKED)

The first control. This storage's **tracking area** holds everything LFB knows about the storage — the
per-file fingerprint index (`files.yaml`), the IPFS manifest, and the media-analysis outputs.

**What that area *is* depends on the storage's KIND**, and this page is reached from **every** kind, so
the section renders differently for each. The rule is not this page's to make — it is owned by
[artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx), keyed on the **resolved storage
kind** and never on a path or a name (§0.2):

| Storage kind | Tracking area | Does this section show the `.lfbridge/` control? |
| --- | --- | --- |
| **`repo`** — a user's working git repo (Git Storage) | **`<root>/.lfbridge/`** — LFB is a **guest**, so it quarantines its files in one hidden corner ([storages.mdx §4](./storages.mdx#4-where-a-storages-state-lives--the-kind-split-then-category-a-vs-category-b-locked)) | **YES** — checkbox + relocate picker below |
| **`personal` / `company` / `community`** — a dedicated LFB file repo (**SDL**) | **the storage ROOT itself** — an SDL has **no `.lfbridge/` directory at all** | **NO** — read-only root line instead |

**For a `repo` storage — keep the hidden `.lfbridge/`, and where.**

* **Checkbox: "Keep a hidden `.lfbridge/` tracking directory for this repo on this computer."**
  **Default: ON.** ON means this computer tracks the repo's files locally; OFF means this computer
  carries the repo's bytes but keeps **no** local tracking area (rare — e.g. a read-only mirror
  machine). This is a **machine-local** choice (§2).
* **Location.** Default **`<repo root>/.lfbridge/`** — the hidden folder at the repo's own root
  (e.g. `~/BGit/xyz/.lfbridge/`). The user may **relocate** it — a **directory picker** pre-filled
  with the default — for cases where the repo root should stay pristine or is read-only. When
  relocated, LFB writes tracking to the chosen directory and maps it back to the storage; the
  descriptor is still found via the root → `.lfbridge/` fallback already in code.
* Whenever `.lfbridge/` sits **inside a Git repo**, LFB keeps it git-ignored (appends `.lfbridge/` to
  `.gitignore`) so tracking never bloats commits — charter behavior, already enforced. (This is
  about a **working repo**'s hidden dir only; an SDL has no `.lfbridge/` to ignore, and its root
  content is exactly what its git is **for**.)

**For an SDL (personal / company / community) — there is nothing to toggle or relocate.**

A **dedicated LFB file repo** is a repo whose entire reason to exist is to hold LFB's files, so hiding
them under a `.lfbridge/` inside a repo that contains nothing else buys no separation and only adds a
meaningless path segment. **The SDL root *is* the `.lfbridge` area**
([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)). Concretely, for an SDL this
section:

* **Does not offer the "keep a hidden `.lfbridge/`" checkbox** — the storage's tracking text is not
  optional in the same way: the SDL exists *to hold it*, and turning it off would empty the storage of
  its only reason to exist.
* **Does not offer the relocate picker** — there is no `<root>/.lfbridge/` to move. The SDL's metadata
  (`storage.yaml`, `mapped_dirs.yaml`, `files.yaml`, `manifest.yaml`, `bookmarks.yaml`, `devices/`,
  `analysis/`) and its path-mirrored artifact tree all sit **at the root**
  ([artifact_placement_policy.mdx §0.1](./artifact_placement_policy.mdx)). Relocating them would mean
  relocating the storage, which is what the storage **root** already says.
* **Shows instead a read-only line** naming the root as the tracking area — *"This storage's root **is**
  its tracking area — a dedicated Large File Bridge file repo has no hidden `.lfbridge/` directory."* —
  with the root path and a link to the placement policy. It is informative, not editable.

An SDL root **also** carries a `.git/` (every SDL is a git repo). That never demotes it to a working
repo: the **kind wins** over the presence of `.git/`
([artifact_placement_policy.mdx §0.2](./artifact_placement_policy.mdx)).

The kind→base-directory decision this section renders lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `trackingBaseDir(root, type?)` (returns `<root>/.lfbridge` for kind `repo`, and `<root>` itself for an SDL) and its predicate `usesLfbridgeDir(type)`; every LFB writer resolves its base through it and none joins `LFBRIDGE_DIR` directly. The ensure-directory + git-ignore-on-init behavior the `repo`-only checkbox drives is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureStorage()` / `ensureGitignore()`. Honoring a **relocated** `repo` `.lfbridge/` path (resolve against the machine-local config, not always `<root>/.lfbridge/`) is a change to `descriptorPath()` / the tracking-path resolvers in the same module, layered **over** `trackingBaseDir()`.

### 3.1 The relocated-`.lfbridge/` control vs. the tracking-state placement policy

The relocate control (§3) answers *where does this **working repo** keep its own `.lfbridge/` on this
computer* — it exists for kind `repo` only; an SDL has no `.lfbridge/` to relocate (§3). It
is **complemented — not replaced —** by the **data-category placement policy** that governs a **repo**'s
tracking state, owned by [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) and detailed in
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx). Three rules stack here and must not be confused:
[§0 of the placement policy](./artifact_placement_policy.mdx) decides **whether `.lfbridge/` exists at
all** (kind `repo` → yes; SDL → never); §3 then places that hidden folder on **this** machine; the
data-category rule below decides **which root** each file may live in.

* **§3 relocate** moves a **working repo's own** `.lfbridge/` — its committed Category-A content area
  (transcripts / AI descriptions) — to a chosen directory on **this machine**. It is a per-computer
  placement of the hidden folder, and it has **no SDL counterpart**: for a company/Personal SDL the
  equivalent area is the storage **root** itself, which the storage's root path already names.
* **The placement policy** decides where a **repo**'s noisy LFB tracking state (`repo_storage.yaml`,
  `files.yaml`, per-file sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`) lives — and
  the answer is fixed: **Local Storage always** at `~/T/_large_files_bridge/repos/<repoKey>/`, and
  **additionally** the owning storage's **sync repo** at `<syncRepo>/repos/<repoKey>/` when one is
  configured and the per-repo toggle is on (default off). That tracking state **never** enters the working
  repo at all, so relocating a working repo's `.lfbridge/` here never touches it.

So a repo's tracking state has **two homes** — Local Storage (always) and, when the user opts in, the
storage's **sync repo** (§4b) as the additive travelling mirror. The old content-threshold **"three homes"**
model (state root / the repo's own `.lfbridge/` / a state-sync repo) is **retired**: no tier ever writes
tracking state into a working repo, and no content threshold gates it. The state root `repos/<repoKey>/`
location itself is owned by [storage_local.mdx](./storage_local.mdx); this page only configures the sync
repo that state **additionally** mirrors to (§4b).

## 4. Backing locations — dedicated repo · Google Drive · Dropbox

The heart of the page. Beyond its **IPFS** copy (the primary cross-computer pin path,
[pin_process.mdx](./pin_process.mdx)), a storage can be **materialized into one or more backing
locations** — real directories that hold the same hierarchy as a second, human-reachable copy. Each is
**independently ON/OFF** and carries **its own directory**. LFB proposes a good **default directory**;
the user can change it.

### 4.1 Dedicated Git repo (preferred when available)

* **What.** A **dedicated Git repo for this storage** — the ideal home, because the small committed
  manifest travels via Git to the user's other computers while IPFS carries the bytes (the charter's
  git-for-small-files / IPFS-for-big-files split). *We would love a dedicated repo per storage; when one
  isn't available we fall back to the cloud mirrors below.*
* **Toggle + path.** ON reveals a **remote field** — the repo the SDL text travels through — accepting
  **either** shape ([git_backbone.mdx §3](./git_backbone.mdx#3-the-two-remote-shapes--a-local-path-vs-an-https-url-locked)):
  * a **local file path** to a checkout already on this computer, defaulting to the convention
    **`~/BGit/Bryan_git/<slug>_large_files_bridge/`** (e.g. `act3_large_files_bridge/`), matching how
    storages are discovered — LFB uses that directory **in place** as the working copy; **or**
  * an **HTTP(S) URL** to a Git server **not** mapped to the local filesystem — LFB **clones** it into a
    machine-local working-copy cache (`pin/s/<storage_id>/git/`) and manages Git there, authenticating
    via the OS git credential helper ([git_backbone.mdx §5](./git_backbone.mdx#5-credentials-for-a-url-remote--the-os-git-credential-helper-never-a-prompt-locked)).

  For a **repo** storage the dedicated repo **is** the repo itself, so this is pre-satisfied and shown
  read-only.
* **Turning this ON is the permission grant.** It authorizes LFB to run **`git fetch`, `git merge`
  (automatic), `git add`/`commit`, and `git push`** on that repo on **every pin pass** — the
  [git_backbone.mdx](./git_backbone.mdx) engine (`simple-git` over the system git binary) — so the SDL's small
  text is pulled, merged, and pushed between the user's computers while IPFS carries the bytes. No Git
  command runs against a storage whose backbone is off.
* When ON and a **local path** does not exist, the **pin pass creates it** (and `git init`s it if it is
  meant to be a repo) and keeps it updated (§6); a **URL** remote is cloned on first use instead.
* **It also becomes the home for this storage's derived artifacts (LOCKED).** Once the dedicated repo is
  ON, transcripts and AI descriptions for any file this storage **owns but that lives out in the wider
  filesystem** (a mapped source directory, or — for Personal — anywhere under `~`) are written **inside the
  dedicated repo**, path-mirrored **off its root**, with the media's **full filename and the extension
  APPENDED** (`<dedicatedRepo>/<rel-dir>/<name.ext>.transcription`,
  `<dedicatedRepo>/<rel-dir>/<name.ext>.ai_description`). **There is no `.lfbridge/` segment here:** a
  company/Personal dedicated repo **is an SDL**, and an SDL's root **is** the `.lfbridge` area
  ([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)) — so the mirror hangs off the
  root, e.g.
  `~/BGit/Bryan_git/personal_large_files_bridge/_Mirror/Politics/…/Teacher_Lounge_Shooter.mp4.transcription`.
  Because the SDL repo's contents are **committed and travel with it**, these artifacts are committed and
  carried by git — the dedicated repo exists to **hold and carry** them. This is the authoritative
  placement rule B in [Transcribe.mdx §3.4](./Transcribe.mdx); turning the dedicated repo ON is precisely
  what routes those artifacts into this repo (mirrored). Turning it OFF returns them to the owning
  storage root's own tracking area — which for a **working repo** is its `.lfbridge/`, and for an SDL is
  the root itself (the last resort being a `.lfbridge/` in the media's own directory, only ever inside a
  working repo).

#### 4.1.1 "No remote yet" is a fault this page shows, and fixes (LOCKED)

A brand-new company storage is created by `git init`
([storage_company.mdx §10.3](./storage_company.mdx#103-proposing-and-creating-the-company-storages-locked)),
so **it has no `origin`**. Large File Bridge can commit into it locally, but **nothing travels** — and the
backbone's cycle used to look identical to a healthy repo with nothing to do
([storage_company.mdx §11.2](./storage_company.mdx#112-a-repo-that-cannot-push-must-say-so-locked)). This
page is where the user sees it and where the one-click fix lives.

* **The remote field carries a standing fault, not a placeholder.** When this storage's backbone repo has
  no `origin`, the field shows an inline warning rather than an empty box:
  *"Large File Bridge is tracking this company locally, but it has no remote — nothing is reaching your
  other computers."* The same fault appears on the storage's row in the
  [Storages tab](./storages.mdx#2-the-storages-left-bar-tab) and in the log
  ([backbone_resilience.mdx §6.2](./backbone_resilience.mdx#62-a-repo-with-no-origin--committing-forever-reaching-nobody-locked)).
* **The fix is this field.** Supplying either remote shape (§4.1 — a local path or an HTTP(S) URL) clears
  it, and the **always-push-when-ahead** rule ([git_backbone.mdx §6.1](./git_backbone.mdx#61-always-push-when-the-branch-is-ahead--deliver-stranded--foreign-commits-locked-learned-from-a-defect))
  delivers everything that accumulated while there was nowhere to push. Nothing that piled up is lost.
* **It is standing, not one-shot.** It persists until a remote exists — a toast that scrolls away would
  reproduce exactly the silence this rule exists to end.
* **It is not an error state for the storage.** Local tracking, scanning, analysis, and IPFS pinning all
  keep working; only **travel** is blocked. The wording says what is not happening, not that something
  broke.

### 4.2 Google Drive location

* **What.** A clone of the storage inside a connected **Google Drive** ([google_drive.mdx](./google_drive.mdx)) —
  so teammates or your other machines can reach the files without LFB/IPFS.
* **Toggle + path.** ON reveals a **path field** (a path *inside* the connected Drive) defaulting to
  **`<Drive>/LFB/<Storage Name>/`** (e.g. `My Drive/LFB/ACT3` or, for a company, a `Shared drives/…`
  anchor). The **canonical** relative path is recorded in `storage.yaml → clones.google_drive` (§2, §5);
  the **enable flag + resolved absolute path on this computer** are machine-local.

### 4.3 Dropbox directory

* **What.** A clone of the storage inside **Dropbox** ([dropbox.mdx](./dropbox.mdx)).
* **Toggle + path.** ON reveals a **path field** (a path inside Dropbox) defaulting to
  **`Apps/LFB/<Storage Name>/`**. As with Drive: canonical relative path in `storage.yaml → clones.dropbox`,
  enable flag + absolute local path machine-local.

### 4.4 Rules across all three

* Each backing location is **independent** — any combination (repo only, repo + Drive, all three, none)
  is valid. With **none** enabled, the storage still pins over IPFS; the backing locations are
  **additional** human-reachable mirrors, never the source of truth
  ([storages.mdx §5](./storages.mdx#5-cloud-clones-google-drive--dropbox)).
* **Defaults are proposed, never forced.** LFB pre-fills a sensible directory for each; nothing is
  created until the user enables it **and** the pin pass runs (§6). This honors the charter rule that
  LFB **surfaces and offers** but does not act on the filesystem on its own.
* A location whose connected drive is **not present** on this computer (no Dropbox, Drive not linked)
  shows disabled with a "connect Drive/Dropbox first" hint rather than erroring.

Resolving the effective backing locations for a storage (default directory proposal per type + the machine-local enable flags) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `resolveBackingLocations()` (new module; reuses the Drive/Dropbox path helpers from `google_drive` / `dropbox`).

## 4a. Mapped source directories — the directories that belong to this storage

For a **company** or **personal** storage, this page also carries the **list of source directories** that
belong to the storage. This is the control the charter's company use case describes: *"in the settings
for that company storage they can have a list of brief directories … everything recursive is part of that
company storage."* It is defined in full in
[syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked)
and [storage_company.mdx §1.1](./storage_company.mdx#11-a-company-maps-a-list-of-source-directories-locked);
this page is where the user **edits** it.

* **A list, add/remove rows.** The user adds each directory hierarchy that belongs to this storage (a
  company with seven hierarchies adds seven). Each row is a mapped directory; **everything recursive under
  it is in scope**. A **[ + Add directory ]** button opens a directory picker; each row has a remove
  affordance.
* **Shared list vs. per-computer path (LOCKED, mirrors §2).** The **logical** list of mapped directories
  is **shared** — it travels in the SDL's **`mapped_dirs.yaml` at the storage root** so every computer
  agrees on the set (each row has a stable **key** + label). The **absolute local path** of each mapped
  directory on **this** computer is the **graft**, stored per-device in **`devices/<device>.yaml`, also
  at the storage root**
  ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)). Both
  sit at the **root**, not under a `.lfbridge/`: this section shows only for a company/personal storage,
  and such a storage **is an SDL**, which has no `.lfbridge/` directory at all — its root **is** the
  `.lfbridge` area ([artifact_placement_policy.mdx §0.1](./artifact_placement_policy.mdx)). So on
  this page the path field for each row edits **this computer's graft**; adding/removing a row edits the
  **shared** list.
* **Repo storages have exactly one, implicit mapped directory** — the repo working tree — so this section
  is shown read-only (or omitted) for a `repo` storage; the mapped-directory model is the generalization
  of the repo case to a list.
* Editing the list is what tells the **scanner** which subtrees to walk for this storage's big files
  ([scan.mdx](./scan.mdx)); nothing is moved — LFB tracks files **in place** and pins the opted-in ones
  over IPFS.

### 4a.1 Reserved mapped-dir keys — the validator this page owns (LOCKED)

Because an SDL's mirror hangs off the **root** (§4a, [artifact_placement_policy.mdx §0.1](./artifact_placement_policy.mdx)),
a mapped-dir **key** sits as a directory name **beside** the SDL's own metadata — so a key named after
that metadata would **shadow** it. These names are **reserved at the SDL root**; the rule is declared by
[artifact_placement_policy.mdx §0.3](./artifact_placement_policy.mdx), which names **this page** as the
home of the validator that enforces it:

```
RESERVED at the SDL root — a mapped-dir key may not be any of:
  devices        analysis        files.yaml      storage.yaml
  manifest.yaml  bookmarks.yaml  mapped_dirs.yaml
```

* **Add / rename rejects a colliding key.** The mapped-dir **add** flow and the **rename** of an existing
  row validate the key against the reserved list and **refuse** a match, with an inline error that
  **names the reserved word** — *"`devices` is reserved: this storage's root holds Large File Bridge's
  own `devices/` metadata. Pick a different key."* Save stays disabled until the key changes, exactly
  like the backing-location and sync-repo validation elsewhere on this page (§4.4, §4b).
* **Existing mapped dirs are checked on load, never silently honored.** A list written by an older
  version (or hand-edited, or arriving from a teammate through the shared `mapped_dirs.yaml`) may already
  carry a colliding key. On load LFB surfaces it as a **storage warning**
  ([warnings.mdx](./warnings.mdx)) naming the reserved word and the offending row — it does **not**
  silently overwrite the SDL's metadata, and it does **not** silently drop the row. The user resolves it
  by renaming the key.
* **This is a `repo`-storage non-issue.** A working repo's mapped directory is implicit (its working
  tree, above) and its metadata lives under `.lfbridge/`, one level away from any user directory — the
  collision only exists because an SDL deliberately has no such segment.

Reading/writing the shared mapped-directory list (add/remove rows → `mapped_dirs.yaml` at the SDL root) and this computer's per-row graft path (→ `devices/<device>.yaml` at the SDL root, both resolved via `trackingBaseDir()` in `storage-type.service.ts`, never by joining `LFBRIDGE_DIR`) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getMappedDirs()` / `setMappedDirs()`, with the per-device graft written via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `writeSelfDevice()`.

Rejecting a mapped-dir key that collides with a **reserved** SDL-root name (§4a.1) on add/rename, and surfacing an already-colliding key from an existing list as a storage warning on load, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `assertMappedDirKeyAllowed(key)` (the add/rename gate called from `setMappedDirs()`, throwing an error naming the reserved word) and `checkReservedMappedDirs(storageId)` (the load-time audit that emits the warning), reading the reserved list from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `RESERVED_SDL_ROOT_NAMES` (the single source the SDL's own writers and this validator share).

## 4b. The sync repo — an additive mirror of owned repos' tracking state

For a **company** or **Personal** storage, this page also configures the storage's optional **sync repo** —
the dedicated Git repo the user set up *"only for the Large File Bridge webapp to sync changes."* Every repo
this storage **owns** ([repo_company_mapping.mdx](./repo_company_mapping.mdx)) keeps its LFB tracking state
in **Local Storage always**; when a sync repo is configured **and** that repo's per-repo toggle is on
([repo_settings.mdx §2.9](./repo_settings.mdx)), the same tracking state is **additionally** mirrored to
`<syncRepo>/repos/<repoKey>/` so it travels across the user's computers **without** polluting each working
repo's own history. The concept and the full data-category placement policy are defined once and only
cross-linked here:
[artifact_placement_policy.mdx](./artifact_placement_policy.mdx) ·
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx). This page is where the user **picks** or **clears**
it.

* **This section shows only for company / Personal storages.** A **repo** storage does not own other
  repos, `local` is settings/config, and a **community** storage consumes rather than owns — so the
  control is hidden (or read-only "not applicable") for those three types.
* **Pick control.** A **remote field**, exactly the two shapes a backing location's dedicated repo accepts
  ([git_backbone.mdx §3](./git_backbone.mdx#3-the-two-remote-shapes--a-local-path-vs-an-https-url-locked)):
  * a **local file path** to a git checkout already on this computer — defaulting to the convention
    **`~/BGit/Bryan_git/<slug>_state_sync/`** (e.g. `act3_state_sync/`), used **in place** as the working
    copy; **or**
  * an **HTTP(S) URL** to a git server — LFB **clones** it into a machine-local working-copy cache
    (`pin/s/<storage_id>/state_sync/`) and manages Git there via the OS git credential helper
    ([git_backbone.mdx §5](./git_backbone.mdx#5-credentials-for-a-url-remote--the-os-git-credential-helper-never-a-prompt-locked)).
* **Clear control.** A **[ Clear ]** affordance unsets the sync repo. Clearing does **not** delete the repo
  on disk (charter: surface and offer, never destroy); it stops the additive mirror. Repos owned by this
  storage then keep their tracking state in **Local Storage only** — no working repo is ever touched, and
  the state is real and honored locally, simply not yet shared
  ([artifact_placement_policy.mdx §7.2](./artifact_placement_policy.mdx)).
* **Validation before saving.** LFB checks that the pick is a **git working tree** (or a reachable URL
  remote), is **writable**, and is **not one of this storage's mapped source directories** (§4a) nor a
  tracked big-file location — the sync repo carries **small text only**, never the big bytes. An
  invalid pick shows an inline error and the Save stays disabled, matching the backing-location hints
  (§4.4). A storage may reuse its **dedicated repo** (§4.1) as its sync repo, or point at a
  separate one; both are valid.
* **What mirrors there.** When configured, for **every repo this storage owns** (owner mapping,
  [repo_company_mapping.mdx](./repo_company_mapping.mdx)) whose per-repo sync toggle is **on**: the repo-wide
  `repo_storage.yaml`, the `files.yaml` fingerprint index, the per-file sidecars, `history/<device>.txt`,
  `decisions.yaml`, `manifest.yaml`, and — when a repo's **placement radio**
  ([repo_settings.mdx](./repo_settings.mdx), [placement_radios.mdx](./placement_radios.mdx)) is set to
  *sync repo* — its transcripts / AI descriptions, all path-mirrored under `<syncRepo>/repos/<repoKey>/`.
  This is **additive** — Local Storage stays the authoritative working copy on each machine; the mirror
  never replaces it ([artifact_placement_policy.mdx §2](./artifact_placement_policy.mdx)). ~~Default **OFF**,
  per-repo.~~ **SUPERSEDED** — the per-repo mirror now defaults **ON** (§4b.1,
  [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked));
  it is still per-repo, and the toggle is now an **opt-out**. The set of owned repos and the
  **`ownedRepoCount`** surfaced on the Storages page both derive from the same owner mapping.
* **Machine-local, like the rest of this page (§2).** The chosen remote + its enable state persist to the
  machine-local per-storage config (§5); the *fact that repos route here* is derived from the shared owner
  mapping plus each repo's per-repo toggle, not stored per-computer on this storage.

Reading/writing the sync-repo pick (validate git working tree / URL, reject mapped-dir & big-file overlaps, clone a URL remote into the machine-local cache) and resolving the effective mirror root for an owned repo is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getStateSyncRepo()` / `setStateSyncRepo()` / `clearStateSyncRepo()`, feeding `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null`), whose additive mirror is written by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)`.

### 4b.1 Auto-adoption — the field shows the storage's own SDL as the effective value (LOCKED)

The rule is owned by
[storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)
(and, for Personal, [storage_personal.mdx §14](./storage_personal.mdx#14-personals-optional-sync-repo)). This
section is only how it **presents on this page**. A company or Personal storage whose root is a **git working
tree** — the normal case, since an SDL *is* a git repo — **is** its own sync repo, adopted without the user
configuring anything.

* **The field is never empty when an SDL can be adopted.** Instead of a blank input and an implied *"set this
  up first"*, the remote field shows the **auto-adopted SDL root** as the **effective value**, rendered as a
  distinct **derived** state — the path in the field, with an **`(auto — this storage's own repo)`** hint
  beneath it. It is a real, working sync repo, not a placeholder.
* **Override is one click.** An **[ Override… ]** affordance turns the derived value into an editable pick,
  which then accepts exactly the two remote shapes and passes exactly the validation the section above
  defines. An **explicitly configured `sync_repo` wins** over auto-adoption whenever it is present — the
  precedence is *configured, else adopted*, never the reverse.
* **[ Clear ] reverts to the adopted value, it does not blank it.** Clearing an override drops the explicit
  pick and the field falls **back** to the auto-adopted SDL, with the derived hint returning. Clearing still
  deletes nothing on disk. Blank — "no sync repo at all" — is now reachable only for a storage whose root is
  **not** a git working tree.
* **This is the answer the Git backbone already gives.** The backbone auto-adopts the SDL's own git root when
  no dedicated repo path is set, which is why device state has been travelling between the user's computers
  all along ([devices.mdx §11.2.1](./devices.mdx), [git_backbone.mdx](./git_backbone.mdx)). Making this field
  agree with it means **one SDL, one git root, one answer** — the sync repo and the backbone stop disagreeing
  about what the storage's repo *is*.
* **Not a git working tree → the old behavior, stated honestly.** A storage rooted in a plain directory (a
  Drive/Dropbox-only pile with no `.git/`) has nothing to adopt: the field stays empty with the hint *"This
  storage's root is not a git repo — pick a sync repo to share tracking state."* Owned repos then keep their
  tracking in **Local Storage only** — real and honored locally, simply not shared.
* **What it changes for owned repos.** Because a sync repo now normally resolves, the per-repo toggle
  ([repo_settings.mdx §2.9.1](./repo_settings.mdx#291-the-default-is-on--this-toggle-is-an-opt-out-locked))
  is normally **enabled and ON** rather than disabled-with-a-setup-hint. Only **small text** travels either
  way — media bytes still move over IPFS alone ([decisions.mdx](./decisions.mdx)).

Resolving the effective sync repo — the explicitly configured pick first, the storage's own git root auto-adopted second — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getStateSyncRepo(storageId)` (the configured pick) with the SDL-root fallback the backbone already performs in `getGitBackboneRemote(storageId)` (same file), surfaced by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/storages/StorageSettingsPage.tsx` — `StorageSettingsPage` (the derived-value field + Override/Clear) and consumed per repo by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveOwnerDedicatedRepo(repoRoot)`.

### 4c.2 Reassigning records the travelling assertion — and re-targets tracking (LOCKED)

Reassigning here is not only a local grouping change; it has the same two consequences as any owner change
([repo_company_mapping.mdx §5.2](./repo_company_mapping.mdx#52-persistence--the-override-lives-in-the-repos-configyaml-locked),

* **Into a company with a sync repo → the assertion travels.** When the user reassigns a repo **to a company
  that has a sync repo** (§4b) configured, LFB also commits an **ownership assertion** (*"this repo, by its
  git remote, belongs to company Y"*) into that sync repo's `owner_map.yaml`
  ([repo_owner_propagation.mdx §2](./repo_owner_propagation.mdx#2-where-the-assertion-lives--owner_mapyaml-in-the-company-sync-repo-locked)).

* **No bytes move.** Reassigning is **grouping/tracking-target** metadata only — it never relocates,
  re-roots, or copies a single file ([repo_company_mapping.mdx §1](./repo_company_mapping.mdx#1-what-an-owner-is-locked)).

Per [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)

## 6. The pin pass creates & updates the enabled directories

The settings on this page are **inert until a pin pass runs**. During the **full pass**
([pin_process.mdx §2](./pin_process.mdx)), for each
storage LFB reads its machine-local config (§5) and, **for every enabled backing location**:

1. **Ensure the directory exists** — create it (and `git init` a dedicated repo) if missing; a location
   that is enabled but absent is created on the next pass, never before.
2. **Ensure the tracking area** — resolve it through `trackingBaseDir()` (§3) and create/refresh it:
   for a **`repo`** storage that is the hidden `.lfbridge/` at its configured location, kept
   git-ignored; for an **SDL** it is the storage **root** itself, which needs no hidden dir and no
   `.gitignore` entry.
3. **Update the mirror** — reconcile the storage's files + manifest into the location so it holds the
   current hierarchy (the human-reachable copy). Drive/Dropbox mirrors reuse
   [google_drive.mdx](./google_drive.mdx) / [dropbox.mdx](./dropbox.mdx).

A **disabled** location is **left untouched** — LFB neither creates nor deletes it (charter: surface and
offer, never act on files unasked). Turning a location **off** stops updating it; it does **not** delete
the directory that is already there.

Materializing each enabled backing location during the pass (create-if-missing + ensure the tracking area resolved via `trackingBaseDir()` — `.lfbridge/` for a `repo`, the root for an SDL + update the mirror) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureBackingLocations(id)` (new; called from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll()` per known storage, through the same bounded `cores − 2` limiter as repo/computer units).

## 8. Acceptance criteria

1. The per-storage settings page is opened from a **single storage** — the **per-row gear** in the
   Storages table (just left of the row `⋮`), the **gear** on the storage detail header, the storage
   row's **`⋮` kebab / right-click**, and the detail page's **`⋮` more-menu** — all at
   **`/storages/$storageId/settings`**, scoped to that storage and naming it + its root.
2. The **Tracking area** section is **keyed on storage kind** (§3, rule owned by
   [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)). For a **`repo`** storage it
   exposes a **"keep `.lfbridge/`" checkbox (default ON)** and a **location** control (default
   `<root>/.lfbridge/`, relocatable) — a **machine-local** choice. For a **`personal` / `company` /
   `community` SDL** it exposes **neither**: an SDL has **no `.lfbridge/` at all** — its **root** *is*
   the tracking area — so the section shows only a **read-only** line naming the root. An SDL's own
   `.git/` never demotes it to a working repo, and a stray `lfbridge.path` on an SDL is **ignored**, not
   obeyed. Resolved once via `trackingBaseDir()` / `usesLfbridgeDir()` in `storage-type.service.ts`; no
   writer joins `LFBRIDGE_DIR` directly.
3. It exposes **three independent backing locations**, each **ON/OFF with its own directory**: a
   **dedicated Git repo** (preferred; for a repo storage it is the repo itself), a **Google Drive**
   location, and a **Dropbox** directory. LFB pre-fills a **good default directory** for each, which the
   user can change.
4. **Machine-local vs. shared is respected:** the enable flags + absolute local paths persist to
   **`pin/s/<storage_id>/config.yaml`** (the local settings file); the **canonical** Drive/Dropbox
   relative paths persist to the **shared** `storage.yaml → clones`. Neither restates repo-specific
   settings owned by [repo_settings.mdx](./repo_settings.mdx).
5. Settings are **inert until a pin pass**: on each pass, for **every enabled** backing location LFB
   **creates the directory if missing** (and `git init`s a dedicated repo), ensures its **tracking area**
   as resolved by `trackingBaseDir()` (a `.lfbridge/` for a `repo`; the **root** for an SDL — no hidden
   dir, no `.gitignore` entry), and **updates** the mirror; a **disabled** location is left untouched
   (never auto-created, never deleted). Nothing is created merely by saving the form.
6. Storage **identity** (`storage:` id/name/type/root) is written by discovery and shown **read-only**;
   the user configures behavior, not identity.
7. For a **company/personal** storage the page edits the **list of mapped source directories** (§4a):
   add/remove rows edit the **shared** `mapped_dirs.yaml` **at the SDL root**; each row's path field
   edits **this computer's graft** (`devices/<device>.yaml`, **also at the SDL root**) — both at the
   root because a company/personal storage **is an SDL** and has no `.lfbridge/` segment. Everything
   recursive under a mapped directory is in scope; files are tracked **in place** (never relocated). A
   **repo** storage's single mapped directory (its working tree) is shown read-only.
8. The **reserved-name validator lives here** (§4a.1), enforcing the rule declared by
   [artifact_placement_policy.mdx §0.3](./artifact_placement_policy.mdx): because the SDL mirror hangs
   off the **root**, a mapped-dir key of `devices`, `analysis`, `files.yaml`, `storage.yaml`,
   `manifest.yaml`, `bookmarks.yaml`, or `mapped_dirs.yaml` would **shadow** the SDL's own metadata.
   **Add/rename REJECTS** such a key with an inline error **naming the reserved word**, and Save stays
   disabled; **existing** mapped dirs are **checked on load** and a collision is surfaced as a **storage
   warning** naming the reserved word and the offending row — never silently overwriting the metadata
   and never silently dropping the row. Enforced by `assertMappedDirKeyAllowed()` /
   `checkReservedMappedDirs()` in `storage-settings.service.ts` against `RESERVED_SDL_ROOT_NAMES` in
   `storage-type.service.ts`.
9. For a **company/Personal** storage the page **configures the optional sync repo** (§4b): a
   **pick** control (local git tree **or** URL remote, default `~/BGit/Bryan_git/<slug>_state_sync/`) and a
   **Clear** control that unsets it **without deleting** the repo. The pick is **validated** (a writable
   git working tree / reachable URL, not overlapping a mapped source directory or a big-file location) and
   persists to the machine-local config (§5) as `sync_repo`. It is an **additive mirror** of owned repos'
   Local-Storage tracking state, written to `<syncRepo>/repos/<repoKey>/` only for repos whose per-repo
   toggle is on (~~default OFF~~ — **superseded by AC 9b**). The section is **hidden** for `repo`, `local`, and
   `community` storages. Concept + placement policy are cross-linked, not redefined
   ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).
9b. The sync-repo field **auto-adopts this storage's own SDL** (§4b.1, authority
   [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)):
   for a company/Personal storage whose root is a **git working tree**, the field shows that root as the
   **effective value** with an **`(auto — this storage's own repo)`** hint rather than sitting empty; an
   **[ Override… ]** affordance replaces it with an explicit pick (which **wins**), and **[ Clear ]** reverts to
   the adopted value instead of blanking it. A storage root that is **not** a git working tree keeps the empty
   field and the honest hint. This is the same answer the Git backbone already gives (`getGitBackboneRemote`) —
   one SDL, one git root, one answer — so the per-repo mirror is normally **enabled and ON**
   ([repo_settings.mdx §2.9.1](./repo_settings.mdx#291-the-default-is-on--this-toggle-is-an-opt-out-locked)),
   an **opt-out** rather than an opt-in. Only tracking **text** is affected; media bytes still move over IPFS
   alone.
9c. The **Owned-repos list is resolved remote-org first** (§4c.3): a repo whose remote org binds to this storage
   is listed **even when it lives outside this storage's directory** (e.g. `~/BGit/Bryan_git/charlie-kirk` with
   remote `github.com/ACT3ai/charlie-kirk` under the **ACT3ai** company, not Personal), with **path containment
   only as the fallback** for repos with no forge org. The rows in this table and the repos whose tracking
   mirrors into this storage's sync repo are, by construction, the **same set**; a manual reassign (§4c.1)
   still overrules the derivation.
10. The relocated-`.lfbridge/` control (§3, kind `repo` only) is **complemented, not replaced,** by the
    data-category placement policy (§3.1): §3 places **a working repo's own** `.lfbridge/` on this
    computer, while the
    placement policy fixes a **repo's** tracking-state home to **two homes** — Local Storage always
    (`~/T/_large_files_bridge/repos/<repoKey>/`) and, when the per-repo toggle is on, this storage's **sync
    repo** as the additive mirror (§4b). Tracking state **never** enters a working repo, and the old
    content-threshold "three homes" model is **retired**.
11. For a **company/Personal** storage the page shows an **Owned repos** list (§4c) — a TanStack table of the
    repos whose owner is this storage (Personal includes the auto-defaulted no-org repos) — each row carrying
    a **reassign dropdown** (Personal · each company · **+ New company…**) that re-files the repo via the same
    **`POST /api/repos/:repoId/owner`** as the per-repo Owner control, writing the local `owner_override`
    (`source: "manual"`). Reassigning **into** a company with a sync repo records the travelling **assertion**
    ([repo_owner_propagation.mdx §2](./repo_owner_propagation.mdx)); it **moves no bytes** and only changes
    **which sync repo** the repo's Category-B tracking mirrors to (§4c.2). The section is **hidden** for
    `repo`, `local`, and `community` storages. Read via `getOwnedRepos(storageId)` in
    `storage-settings.service.ts`, rendered by `StorageSettingsPage.tsx`.
12. **"No remote yet" is surfaced here as a standing fault, and fixed here** (§4.1.1): a `git init`ed company
    storage has **no `origin`**, so it commits locally and **nothing travels** — a state the backbone must
    never report as healthy
    ([storage_company.mdx §11.2](./storage_company.mdx#112-a-repo-that-cannot-push-must-say-so-locked)). The
    remote field shows the warning until a remote is supplied; supplying either remote shape clears it and
    the **always-push-when-ahead** rule delivers the accumulated backlog. Local tracking, analysis, and IPFS
    pinning are unaffected — only **travel** is blocked, and the wording says so.
13. **The storages parent directory is a setting, with a derived default** (§4d): a new company storage is
    created at **`<parent>/<org-slug>_large_files_bridge/`**, and `<parent>` is **configurable**, never
    hardcoded to one person's layout. Unset means *derive it* — where the user's existing
    `*_large_files_bridge` storages already live — and the UI shows the **effective** value labeled
    **configured** or **derived**, never blank. Changing it affects only **where the next** storage is
    created. The **proposal list itself lives on the Storages page**
    ([storages.mdx §2.2](./storages.mdx#22-organizations-found-in-your-repos--proposed-company-storages-locked)),
    because choosing which companies exist is a cross-storage decision; **adoption precedes creation**, so an
    org already claimed by an existing storage never produces a second directory.
