# LargeFileBridge — Storage Settings (Per-Storage)

Every storage can be configured on its own. This is the **per-storage settings page** — the screen you
land on from a **single storage** ([storages.mdx](./storages.mdx)). It is the storage counterpart of
[repo_settings.mdx](./repo_settings.mdx): a repo's gear opens the per-repo settings; a storage's gear
opens **this** page, scoped to **one storage**.

It answers two questions the charter leaves per-computer:

1. **Does this computer keep the hidden `.lfbridge/` tracking directory for this storage, and where does
   it go?**
2. **Which backing locations is this storage materialized into on this computer** — a **dedicated Git
   repo**, a **Google Drive** location, and/or a **Dropbox** directory — each independently **on/off**
   with its own directory?

LFB **picks good default directories**; the user can change any of them. The choices persist to a
**machine-local** per-storage config file (§5), which the **pin pass** then reads to **create and
update** the enabled directories (§6).

Siblings this builds on and does not repeat:

* [storages.mdx](./storages.mdx) — the storage family: `storage.yaml` (§3), the hidden `.lfbridge/`
  layout (§4), and the `clones` block that records Drive/Dropbox mirrors (§5).
* [repo_settings.mdx](./repo_settings.mdx) — the per-repo settings page; a **repo is one kind of
  storage**, so a repo's gear may land on either its repo-settings or (for its storage facet) here — the
  two are kept aligned, never duplicated (§2).
* [google_drive.mdx](./google_drive.mdx) / [dropbox.mdx](./dropbox.mdx) — the cloud-drive integrations a
  backing location reuses (§4).
* [storage_local.mdx](./storage_local.mdx) — where the machine-local per-storage config lives under the
  state root (§5). · [pin_process.mdx](./pin_process.mdx) — the pass that materializes enabled
  locations (§6). · [menus.mdx](./menus.mdx) — the kebab / right-click / more-menu that opens this page
  (§1).

## 1. How you get here — gear, kebab, right-click, more-menu

  [menus.mdx §3](./menus.mdx#3-the-row--kebab--every-table-has-one)) — a **Storage settings…** item in

## 2. Machine-local vs. shared — what lives where (LOCKED)

The one distinction this page turns on. A storage has **two** kinds of configuration, and they live in
**two** files:

* **Shared / travels with the storage** — the storage's **identity** and its **canonical** mirror paths.
  This is [`storage.yaml`](./storages.mdx#3-storageyaml--the-per-storage-descriptor-locked-shape): `name`,
  `type`, the type block, and the **`clones`** block (the canonical relative Drive/Dropbox paths). It
  travels between computers via the git backbone, so every machine agrees on *what* the storage is.
* **Machine-local / per-computer** — **this computer's** choices for the storage: whether `.lfbridge/`
  is kept and **where** it goes on this box, and whether each backing location is **enabled** and its
  **resolved local path** on this box. This is the **`pin/s/<storage_id>/config.yaml`** file under the
  Local state root (§5) — the local "settings file" (the `.ini`-style local config) this page writes.

Why split them: a Google Drive folder is at a different absolute path on a laptop than on a work tower;
a machine may have the dedicated repo cloned but no Dropbox; a server may keep `.lfbridge/` on a
faster volume. Those are **per-computer** facts and must **not** be forced into the shared descriptor.
The descriptor says *where the Drive mirror lives relative to the drive*; the machine-local config says
*is it turned on here, and what is its absolute path here*.

For a **repo** storage, the per-repo settings in [repo_settings.mdx](./repo_settings.mdx) remain the
home of repo-specific behavior (threshold override, selection rules, pin policy, sharing); this page adds
only the **storage-facet** controls (`.lfbridge/` placement + backing locations). The two never restate
each other.

## 3. The hidden `.lfbridge/` tracking directory — keep it, and where

The first control. The hidden **`.lfbridge/`** area holds everything LFB knows about this storage — the
per-file fingerprint index (`files.yaml`), the IPFS manifest, and the media-analysis outputs — kept out
of Git ([storages.mdx §4](./storages.mdx#4-lfbridge--the-hidden-per-storage-tracking-area)).

* **Checkbox: "Keep a hidden `.lfbridge/` tracking directory for this storage on this computer."**
  **Default: ON.** ON means this computer tracks the storage's files locally; OFF means this computer
  carries the storage's bytes but keeps **no** local tracking area (rare — e.g. a read-only mirror
  machine). This is a **machine-local** choice (§2).
* **Location.** Default **`<storage root>/.lfbridge/`** — the hidden folder at the storage's own root
  (for a repo, `~/BGit/xyz/.lfbridge/`). The user may **relocate** it — a **directory picker** pre-filled
  with the default — for cases where the storage root should stay pristine or is read-only (a Drive
  folder). When relocated, LFB writes tracking to the chosen directory and maps it back to the storage;
  the descriptor is still found via the root → `.lfbridge/` fallback already in code.
* Whenever `.lfbridge/` sits **inside a Git repo**, LFB keeps it git-ignored (appends `.lfbridge/` to
  `.gitignore`) so tracking never bloats commits — charter behavior, already enforced.

The `.lfbridge/` constant, the ensure-directory, and the git-ignore-on-init behavior this control drives live in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureStorage()` / `ensureGitignore()` (dir constant `LFBRIDGE_DIR` in `tracking.service.ts`). Honoring a **relocated** `.lfbridge/` path (resolve against the machine-local config, not always `<root>/.lfbridge/`) is a change to `descriptorPath()` / the tracking-path resolvers in the same module.

### 3.1 The relocated-`.lfbridge/` control vs. the tracking-state placement policy

The relocate control (§3) answers *where does this storage keep its own `.lfbridge/` on this computer*. It
is **complemented — not replaced —** by the **data-category placement policy** that governs a **repo**'s
tracking state, owned by [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) and detailed in
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx). The two operate on different objects and must not be
confused:

* **§3 relocate** moves **this storage's own** `.lfbridge/` (a company/Personal SDL, or a repo's committed
  Category-A content area — transcripts / AI descriptions) to a chosen directory on **this machine**. It is
  a per-computer placement of the hidden folder.
* **The placement policy** decides where a **repo**'s noisy LFB tracking state (`repo_storage.yaml`,
  `files.yaml`, per-file sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`) lives — and
  the answer is fixed: **Local Storage always** at `~/T/_large_files_bridge/repos/<repoKey>/`, and
  **additionally** the owning storage's **sync repo** at `<syncRepo>/repos/<repoKey>/` when one is
  configured and the per-repo toggle is on (default off). That tracking state **never** enters the working
  repo at all, so relocating a working repo's `.lfbridge/` here never touches it.

So a repo's tracking state has **two homes** — Local Storage (always) and, when the user opts in, the
storage's **sync repo** (§4b) as the additive travelling mirror. The old content-threshold **"three homes"**
model (state root / the repo's own `.lfbridge/` / a state-sync repo) is **retired**: no tier ever writes
tracking state into a working repo, and no content threshold gates it. The state root `repos/<repoKey>/`
location itself is owned by [storage_local.mdx](./storage_local.mdx); this page only configures the sync
repo that state **additionally** mirrors to (§4b).

## 4. Backing locations — dedicated repo · Google Drive · Dropbox

The heart of the page. Beyond its **IPFS** copy (the primary cross-computer pin path,
[pin_process.mdx](./pin_process.mdx)), a storage can be **materialized into one or more backing
locations** — real directories that hold the same hierarchy as a second, human-reachable copy. Each is
**independently ON/OFF** and carries **its own directory**. LFB proposes a good **default directory**;
the user can change it.

### 4.1 Dedicated Git repo (preferred when available)

* **What.** A **dedicated Git repo for this storage** — the ideal home, because the small committed
  manifest travels via Git to the user's other computers while IPFS carries the bytes (the charter's
  git-for-small-files / IPFS-for-big-files split). *We would love a dedicated repo per storage; when one
  isn't available we fall back to the cloud mirrors below.*
* **Toggle + path.** ON reveals a **remote field** — the repo the SDL text travels through — accepting
  **either** shape ([git_backbone.mdx §3](./git_backbone.mdx#3-the-two-remote-shapes--a-local-path-vs-an-https-url-locked)):
  * a **local file path** to a checkout already on this computer, defaulting to the convention
    **`~/BGit/Bryan_git/<slug>_large_files_bridge/`** (e.g. `act3_large_files_bridge/`), matching how
    storages are discovered — LFB uses that directory **in place** as the working copy; **or**
  * an **HTTP(S) URL** to a Git server **not** mapped to the local filesystem — LFB **clones** it into a
    machine-local working-copy cache (`pin/s/<storage_id>/git/`) and manages Git there, authenticating
    via the OS git credential helper ([git_backbone.mdx §5](./git_backbone.mdx#5-credentials-for-a-url-remote--the-os-git-credential-helper-never-a-prompt-locked)).

  For a **repo** storage the dedicated repo **is** the repo itself, so this is pre-satisfied and shown
  read-only.
* **Turning this ON is the permission grant.** It authorizes LFB to run **`git fetch`, `git merge`
  (automatic), `git add`/`commit`, and `git push`** on that repo on **every pin pass** — the
  [git_backbone.mdx](./git_backbone.mdx) engine (`simple-git` over the system git binary) — so the SDL's small
  text is pulled, merged, and pushed between the user's computers while IPFS carries the bytes. No Git
  command runs against a storage whose backbone is off.
* When ON and a **local path** does not exist, the **pin pass creates it** (and `git init`s it if it is
  meant to be a repo) and keeps it updated (§6); a **URL** remote is cloned on first use instead.
* **It also becomes the home for this storage's derived artifacts (LOCKED).** Once the dedicated repo is
  ON, transcripts and AI descriptions for any file this storage **owns but that lives out in the wider
  filesystem** (a mapped source directory, or — for Personal — anywhere under `~`) are written **inside the
  dedicated repo's `.lfbridge/`**, path-mirrored, with the media's **full filename and the extension
  APPENDED** (`<dedicatedRepo>/.lfbridge/<rel-dir>/<name.ext>.transcription`,
  `<dedicatedRepo>/.lfbridge/<rel-dir>/<name.ext>.ai_description`). Because `.lfbridge/` is **committed and
  travels with the repo**, they are committed and carried by git — the dedicated repo exists to **hold and
  carry** these artifacts. This is the authoritative placement rule B in
  [Transcribe.mdx §3.4](./Transcribe.mdx); turning the dedicated repo ON is precisely what routes those
  artifacts into this repo's `.lfbridge/` (mirrored). Turning it OFF returns them to the owning storage
  root's own `.lfbridge/` (the last-resort a `.lfbridge/` in the media's own directory).

### 4.2 Google Drive location

* **What.** A clone of the storage inside a connected **Google Drive** ([google_drive.mdx](./google_drive.mdx)) —
  so teammates or your other machines can reach the files without LFB/IPFS.
* **Toggle + path.** ON reveals a **path field** (a path *inside* the connected Drive) defaulting to
  **`<Drive>/LFB/<Storage Name>/`** (e.g. `My Drive/LFB/ACT3` or, for a company, a `Shared drives/…`
  anchor). The **canonical** relative path is recorded in `storage.yaml → clones.google_drive` (§2, §5);
  the **enable flag + resolved absolute path on this computer** are machine-local.

### 4.3 Dropbox directory

* **What.** A clone of the storage inside **Dropbox** ([dropbox.mdx](./dropbox.mdx)).
* **Toggle + path.** ON reveals a **path field** (a path inside Dropbox) defaulting to
  **`Apps/LFB/<Storage Name>/`**. As with Drive: canonical relative path in `storage.yaml → clones.dropbox`,
  enable flag + absolute local path machine-local.

### 4.4 Rules across all three

* Each backing location is **independent** — any combination (repo only, repo + Drive, all three, none)
  is valid. With **none** enabled, the storage still pins over IPFS; the backing locations are
  **additional** human-reachable mirrors, never the source of truth
  ([storages.mdx §5](./storages.mdx#5-cloud-clones-google-drive--dropbox)).
* **Defaults are proposed, never forced.** LFB pre-fills a sensible directory for each; nothing is
  created until the user enables it **and** the pin pass runs (§6). This honors the charter rule that
  LFB **surfaces and offers** but does not act on the filesystem on its own.
* A location whose connected drive is **not present** on this computer (no Dropbox, Drive not linked)
  shows disabled with a "connect Drive/Dropbox first" hint rather than erroring.

Resolving the effective backing locations for a storage (default directory proposal per type + the machine-local enable flags) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `resolveBackingLocations()` (new module; reuses the Drive/Dropbox path helpers from `google_drive` / `dropbox`).

## 4a. Mapped source directories — the directories that belong to this storage

For a **company** or **personal** storage, this page also carries the **list of source directories** that
belong to the storage. This is the control the charter's company use case describes: *"in the settings
for that company storage they can have a list of brief directories … everything recursive is part of that
company storage."* It is defined in full in
[syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked)
and [storage_company.mdx §1.1](./storage_company.mdx#11-a-company-maps-a-list-of-source-directories-locked);
this page is where the user **edits** it.

* **A list, add/remove rows.** The user adds each directory hierarchy that belongs to this storage (a
  company with seven hierarchies adds seven). Each row is a mapped directory; **everything recursive under
  it is in scope**. A **[ + Add directory ]** button opens a directory picker; each row has a remove
  affordance.
* **Shared list vs. per-computer path (LOCKED, mirrors §2).** The **logical** list of mapped directories
  is **shared** — it travels in the SDL's `.lfbridge/mapped_dirs.yaml` so every computer agrees on the
  set (each row has a stable **key** + label). The **absolute local path** of each mapped directory on
  **this** computer is the **graft**, stored per-device in `.lfbridge/devices/<device>.yaml`
  ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)). So on
  this page the path field for each row edits **this computer's graft**; adding/removing a row edits the
  **shared** list.
* **Repo storages have exactly one, implicit mapped directory** — the repo working tree — so this section
  is shown read-only (or omitted) for a `repo` storage; the mapped-directory model is the generalization
  of the repo case to a list.
* Editing the list is what tells the **scanner** which subtrees to walk for this storage's big files
  ([scan.mdx](./scan.mdx)); nothing is moved — LFB tracks files **in place** and pins the opted-in ones
  over IPFS.

Reading/writing the shared mapped-directory list (add/remove rows → `.lfbridge/mapped_dirs.yaml`) and this computer's per-row graft path (→ `.lfbridge/devices/<device>.yaml`) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getMappedDirs()` / `setMappedDirs()`, with the per-device graft written via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `writeSelfDevice()`.

## 4b. The sync repo — an additive mirror of owned repos' tracking state

For a **company** or **Personal** storage, this page also configures the storage's optional **sync repo** —
the dedicated Git repo the user set up *"only for the Large File Bridge webapp to sync changes."* Every repo
this storage **owns** ([repo_company_mapping.mdx](./repo_company_mapping.mdx)) keeps its LFB tracking state
in **Local Storage always**; when a sync repo is configured **and** that repo's per-repo toggle is on
([repo_settings.mdx §2.9](./repo_settings.mdx)), the same tracking state is **additionally** mirrored to
`<syncRepo>/repos/<repoKey>/` so it travels across the user's computers **without** polluting each working
repo's own history. The concept and the full data-category placement policy are defined once and only
cross-linked here:
[artifact_placement_policy.mdx](./artifact_placement_policy.mdx) ·
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx). This page is where the user **picks** or **clears**
it.

* **This section shows only for company / Personal storages.** A **repo** storage does not own other
  repos, `local` is settings/config, and a **community** storage consumes rather than owns — so the
  control is hidden (or read-only "not applicable") for those three types.
* **Pick control.** A **remote field**, exactly the two shapes a backing location's dedicated repo accepts
  ([git_backbone.mdx §3](./git_backbone.mdx#3-the-two-remote-shapes--a-local-path-vs-an-https-url-locked)):
  * a **local file path** to a git checkout already on this computer — defaulting to the convention
    **`~/BGit/Bryan_git/<slug>_state_sync/`** (e.g. `act3_state_sync/`), used **in place** as the working
    copy; **or**
  * an **HTTP(S) URL** to a git server — LFB **clones** it into a machine-local working-copy cache
    (`pin/s/<storage_id>/state_sync/`) and manages Git there via the OS git credential helper
    ([git_backbone.mdx §5](./git_backbone.mdx#5-credentials-for-a-url-remote--the-os-git-credential-helper-never-a-prompt-locked)).
* **Clear control.** A **[ Clear ]** affordance unsets the sync repo. Clearing does **not** delete the repo
  on disk (charter: surface and offer, never destroy); it stops the additive mirror. Repos owned by this
  storage then keep their tracking state in **Local Storage only** — no working repo is ever touched, and
  the state is real and honored locally, simply not yet shared
  ([artifact_placement_policy.mdx §7.2](./artifact_placement_policy.mdx)).
* **Validation before saving.** LFB checks that the pick is a **git working tree** (or a reachable URL
  remote), is **writable**, and is **not one of this storage's mapped source directories** (§4a) nor a
  tracked big-file location — the sync repo carries **small text only**, never the big bytes. An
  invalid pick shows an inline error and the Save stays disabled, matching the backing-location hints
  (§4.4). A storage may reuse its **dedicated repo** (§4.1) as its sync repo, or point at a
  separate one; both are valid.
* **What mirrors there.** When configured, for **every repo this storage owns** (owner mapping,
  [repo_company_mapping.mdx](./repo_company_mapping.mdx)) whose per-repo sync toggle is **on**: the repo-wide
  `repo_storage.yaml`, the `files.yaml` fingerprint index, the per-file sidecars, `history/<device>.txt`,
  `decisions.yaml`, `manifest.yaml`, and — when a repo's **placement radio**
  ([repo_settings.mdx](./repo_settings.mdx), [placement_radios.mdx](./placement_radios.mdx)) is set to
  *sync repo* — its transcripts / AI descriptions, all path-mirrored under `<syncRepo>/repos/<repoKey>/`.
  This is **additive** — Local Storage stays the authoritative working copy on each machine; the mirror
  never replaces it ([artifact_placement_policy.mdx §2](./artifact_placement_policy.mdx)). Default **OFF**,
  per-repo. The set of owned repos and the **`ownedRepoCount`** surfaced on the Storages page both derive
  from the same owner mapping.
* **Machine-local, like the rest of this page (§2).** The chosen remote + its enable state persist to the
  machine-local per-storage config (§5); the *fact that repos route here* is derived from the shared owner
  mapping plus each repo's per-repo toggle, not stored per-computer on this storage.

Reading/writing the sync-repo pick (validate git working tree / URL, reject mapped-dir & big-file overlaps, clone a URL remote into the machine-local cache) and resolving the effective mirror root for an owned repo is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getStateSyncRepo()` / `setStateSyncRepo()` / `clearStateSyncRepo()`, feeding `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null`), whose additive mirror is written by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)`.

## 5. Persistence — the machine-local per-storage config

Everything on this page (except the canonical `clones` paths, which go to `storage.yaml`) writes to a
**machine-local** per-storage config under the Local state root — the direct parallel to the per-repo
`pin/r/<repo_name>/config.yaml` ([storage_local.mdx §6](./storage_local.mdx#6-pinrrepo_name--the-per-repo-pin-unit-primary)):

```yaml
# ~/T/_large_files_bridge/pin/s/<storage_id>/config.yaml   (machine-local — the "settings file")
schema_version: 1
updated_at: 2026-07-05T09:41:00.000Z

storage:                         # identity mirror (written by discovery; read-only on this page)
  id: act3                       # the Storages row id (personal | communityId | short hash of root)
  name: "ACT3 Company Files"
  type: company                  # local | repo | personal | company | community
  root: ~/BGit/Bryan_git/act3_large_files_bridge

lfbridge:                        # §3 the hidden tracking directory
  enabled: true                  # keep .lfbridge/ on THIS computer
  path: null                     # null = default <root>/.lfbridge/ ; else an absolute override

backing:                         # §4 backing locations — each ON/OFF + this machine's absolute path
  dedicated_repo:
    enabled: true
    path: ~/BGit/Bryan_git/act3_large_files_bridge   # the storage's own repo when type: repo/company
  google_drive:
    enabled: false
    path: null                   # absolute local mount path; canonical relative path in storage.yaml
  dropbox:
    enabled: false
    path: null

sync_repo:                       # §4b company/Personal only — additive mirror of OWNED repos' tracking state
  configured: true               # false / null = this storage has no sync repo
  remote: ~/BGit/Bryan_git/act3_state_sync   # a local git working tree, OR an https:// URL (cloned to cache)
```

* **`storage:`** identity is written by discovery and shown **read-only** here — the user configures
  behavior, not identity.
* The **canonical** Drive/Dropbox relative paths live in the **shared** descriptor
  (`storage.yaml → clones`, [storages.mdx §3/§5](./storages.mdx#3-storageyaml--the-per-storage-descriptor-locked-shape));
  the **enable flags + absolute local paths** live here (§2). Saving a Drive/Dropbox path updates
  **both**: the relative path into `clones`, the enable flag + absolute path into this file.
* Keyed by **`<storage_id>`** exactly as the Storages row id (`personal`, a community id, or the short
  hash of the root), so the local settings for a storage are found the same way the row is.

Reading and writing this machine-local per-storage config (and mapping it to/from the UI) is handled in code:

## 6. The pin pass creates & updates the enabled directories

The settings on this page are **inert until a pin pass runs**. During the **full pass**
([pin_process.mdx §2](./pin_process.mdx)), for each
storage LFB reads its machine-local config (§5) and, **for every enabled backing location**:

1. **Ensure the directory exists** — create it (and `git init` a dedicated repo) if missing; a location
   that is enabled but absent is created on the next pass, never before.
2. **Ensure `.lfbridge/`** — create/refresh the hidden tracking area at its configured location (§3) and
   keep it git-ignored inside any repo.
3. **Update the mirror** — reconcile the storage's files + manifest into the location so it holds the
   current hierarchy (the human-reachable copy). Drive/Dropbox mirrors reuse
   [google_drive.mdx](./google_drive.mdx) / [dropbox.mdx](./dropbox.mdx).

A **disabled** location is **left untouched** — LFB neither creates nor deletes it (charter: surface and
offer, never act on files unasked). Turning a location **off** stops updating it; it does **not** delete
the directory that is already there.

Materializing each enabled backing location during the pass (create-if-missing + ensure `.lfbridge/` + update the mirror) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureBackingLocations(id)` (new; called from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll()` per known storage, through the same bounded `cores − 2` limiter as repo/computer units).

## 8. Acceptance criteria

1. The per-storage settings page is opened from a **single storage** — the **per-row gear** in the
   Storages table (just left of the row `⋮`), the **gear** on the storage detail header, the storage
   row's **`⋮` kebab / right-click**, and the detail page's **`⋮` more-menu** — all at
   **`/storages/$storageId/settings`**, scoped to that storage and naming it + its root.
2. It exposes a **"keep `.lfbridge/`" checkbox (default ON)** and a **location** control (default
   `<root>/.lfbridge/`, relocatable) — a **machine-local** choice.
3. It exposes **three independent backing locations**, each **ON/OFF with its own directory**: a
   **dedicated Git repo** (preferred; for a repo storage it is the repo itself), a **Google Drive**
   location, and a **Dropbox** directory. LFB pre-fills a **good default directory** for each, which the
   user can change.
4. **Machine-local vs. shared is respected:** the enable flags + absolute local paths persist to
   **`pin/s/<storage_id>/config.yaml`** (the local settings file); the **canonical** Drive/Dropbox
   relative paths persist to the **shared** `storage.yaml → clones`. Neither restates repo-specific
   settings owned by [repo_settings.mdx](./repo_settings.mdx).
5. Settings are **inert until a pin pass**: on each pass, for **every enabled** backing location LFB
   **creates the directory if missing** (and `git init`s a dedicated repo), ensures its `.lfbridge/`,
   and **updates** the mirror; a **disabled** location is left untouched (never auto-created, never
   deleted). Nothing is created merely by saving the form.
6. Storage **identity** (`storage:` id/name/type/root) is written by discovery and shown **read-only**;
   the user configures behavior, not identity.
7. For a **company/personal** storage the page edits the **list of mapped source directories** (§4a):
   add/remove rows edit the **shared** `.lfbridge/mapped_dirs.yaml`; each row's path field edits **this
   computer's graft** (`.lfbridge/devices/<device>.yaml`). Everything recursive under a mapped directory
   is in scope; files are tracked **in place** (never relocated). A **repo** storage's single mapped
   directory (its working tree) is shown read-only.
8. For a **company/Personal** storage the page **configures the optional sync repo** (§4b): a
   **pick** control (local git tree **or** URL remote, default `~/BGit/Bryan_git/<slug>_state_sync/`) and a
   **Clear** control that unsets it **without deleting** the repo. The pick is **validated** (a writable
   git working tree / reachable URL, not overlapping a mapped source directory or a big-file location) and
   persists to the machine-local config (§5) as `sync_repo`. It is an **additive mirror** of owned repos'
   Local-Storage tracking state, written to `<syncRepo>/repos/<repoKey>/` only for repos whose per-repo
   toggle is on (default OFF). The section is **hidden** for `repo`, `local`, and `community` storages.
   Concept + placement policy are cross-linked, not redefined
   ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).
9. The relocated-`.lfbridge/` control (§3) is **complemented, not replaced,** by the data-category
   placement policy (§3.1): §3 places **this storage's own** `.lfbridge/` on this computer, while the
   placement policy fixes a **repo's** tracking-state home to **two homes** — Local Storage always
   (`~/T/_large_files_bridge/repos/<repoKey>/`) and, when the per-repo toggle is on, this storage's **sync
   repo** as the additive mirror (§4b). Tracking state **never** enters a working repo, and the old
   content-threshold "three homes" model is **retired**.
