# LargeFileBridge — Repo Settings (Per-Repo)

Every repo can be configured on its own. This is the **per-repo settings page** — the screen you land
on when you click the **gear icon** on a single repo's page. It is scoped to **one repo** and holds
exactly the settings that are allowed to differ between repos; anything a repo does not override falls
back to the global values on [settings.mdx](./settings.mdx).

Everything here writes to that repo's own config,
`pin/r/<repo_name>/config.yaml` ([storage_local.mdx §6.2](./storage_local.mdx#62-pinrrepo_nameconfigyaml)) —
never to the global `config.yaml`.

Siblings: [settings.mdx](./settings.mdx) (the global defaults this page can override) ·
[scan.mdx](./scan.mdx) (applies the effective per-repo threshold) ·
[storage_local.mdx §6](./storage_local.mdx#6-pinrrepo_name--the-per-repo-pin-unit-primary) (the per-repo pin
unit and its `config.yaml`).

## 1. How you get here — the gear icon

* On a **Repos-table row**, a **⚙ gear** is shown in the row's trailing action cell **just to the left of
  the row `⋮` kebab**, so per-repo settings are **one click** from the list without opening the kebab
  first. It is the same destination as the kebab's **Repo settings…** item — the gear is the discoverable
  shortcut, the kebab is the full catalog. Present on **every** repo row.
* On a single repo's page (the [one repo](./one_repo.mdx) view — the screen a Repos-table row opens,
  route `/repos/$repoId`), a **gear icon** is shown in the header for that repo (top-right, alongside
  **Pin now**). *This gear is a new addition the one-repo header must add;
  [one_repo.mdx §3.1](./one_repo.mdx#31-header--two-rows-and-neither-of-them-wraps-locked) should gain it.*
* Clicking the gear **navigates to this per-repo settings page**, scoped to that repo — route
  `/repos/$repoId/settings`, matching the one-repo route convention (or a slide-over panel over the
  repo view; final form settled with the one-repo spec).
* The page header names the repo and its working-tree path so it is unambiguous which repo is being
  configured. A **back** affordance returns to the repo's page.

The gear is the **only** entry to per-repo settings — global settings never expose per-repo controls;
they only show the global defaults this page can override.

The gear icons that open this page (both navigating to `/repos/$repoId/settings`) live in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` settings button (the one-repo header gear) · `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/RowKebabs.tsx` — `RepoGear` (the per-row gear), placed left of `RepoKebab` in the Repos table's `rowMenu` by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/ReposPage.tsx` — `ReposPage`.

## 2. The settings on this page

The page presents these per-repo settings, each of which either overrides a global default or is
inherently repo-specific.

The per-repo settings form is rendered and loaded in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/RepoSettingsPage.tsx` — `RepoSettingsPage` component (data via `GET /api/repos/:repoId/settings`)

### 2.1 Big-file threshold override

The headline per-repo control, and the reason the checkbox model exists (rule defined in
[settings.mdx §3](./settings.mdx#3-the-per-repo-override--checkbox-off-by-default)):

* A **checkbox: "Override the global big-file threshold for this repo."**
* **Default: OFF** → the repo **inherits the global threshold** (default 100 MB). While off, the page
  shows the current global value **greyed out / read-only** so the user sees what will apply.
* **ON** → reveals a **value field + unit dropdown (MB / GB / TB)** — the same control as
  [settings.mdx §1.1](./settings.mdx#11-the-control--a-value-field-with-a-unit-dropdown) — pre-filled
  with the global value, which the user then changes. This overrides the threshold **for this repo
  only**.
* Turning it back OFF discards the override; the repo reverts to inheriting the global value.

```yaml
# pin/r/<repo_name>/config.yaml  (excerpt)
big_file_override:
  enabled: false      # the checkbox — OFF = inherit global; ON = use value+unit below
  value: 100
  unit: MB            # MB | GB | TB
```

The [scan](./scan.mdx) resolves the effective threshold as *override if enabled, else global* and
records `effective_threshold_bytes` in this repo's `status.yaml` each run.

Resolving this per-repo effective threshold (the override when enabled, else the global default) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `resolveThreshold()`

### 2.2 Pinned on / off (the master toggle)

* A single toggle: **is this repo pinned?** (`config.yaml → pinned`,
  [storage_local.mdx §6.2](./storage_local.mdx#62-pinrrepo_nameconfigyaml)).
* **Default: OFF** — a newly discovered repo is registered but **not** bridged until the user opts in,
  so a scan never bridges data without consent. `true` = LFB bridges this repo's big files; `false` =
  discovered-but-ignored (a re-scan won't keep re-proposing it).

### 2.3 Large-file selection rules

Which big files in this repo get bridged (`config.yaml → large_files`):

* **Follow `.gitignore`** (default **on**) — treat the repo's git-ignored big files as the bridge set,
  so LFB bridges exactly the big files Git is already ignoring.
* **Include globs** — force-bridge these paths even if Git tracks them.
* **Exclude globs** — never bridge these even if git-ignored (e.g. build caches).

### 2.4 Pinning / fetch policy

How this computer participates for this repo (`config.yaml → pin`):

* **Pin locally** (default on) — this computer pins the repo's big files.
* **Fetch missing** (default on) — this computer fetches from peers any manifest CID it lacks.
* **Publish manifest** (default on) — include this repo's `manifest.yaml` (large-file → IPFS CID + pin
  state) in the tracking state mirrored to the owner's sync repo when the §2.9 sync toggle is on, so your
  other computers learn its CIDs. The manifest lives in **Local Storage**
  (`~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml`), **never** the working repo — it travels via the
  sync repo, not a committed in-repo file
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).

### 2.5 Sharing — "some repos need users, some don't"

Who participates in this repo (`config.yaml → access`,
[storage_local.mdx §6.2](./storage_local.mdx#62-pinrrepo_nameconfigyaml)):

* **Private** (default) — `shared: false`. Just *your* repo across *your own* computers; no user list.
* **Shared** — `shared: true` + a list of allow-listed **participant** emails whose computers also pin
  this repo (e.g. a family member or teammate). Only those allow-listed emails' computers participate.

### 2.6 Per-file decisions are team-shared (not configured here, but noted)

The per-file **decisions** users make (Add to IPFS / Add to git ignore) are **not** stored in this
machine-local repo config, and they are **no longer committed into the working repo**. They are written to
the repo's `decisions.yaml` ledger which — like the rest of this repo's LFB tracking state — lives in
**Local Storage always** at `~/T/_large_files_bridge/repos/<repoKey>/decisions.yaml`, machine-local and
never in the working tree, and **travels to the team via the owning storage's sync repo** when this repo's
sync toggle (§2.9) is on ([decisions.mdx](./decisions.mdx),
[artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)). What governs whether decisions **travel** from this
machine is that per-repo sync toggle plus a configured sync repo; absent them, decisions are honored locally
but not yet shared ([decisions.mdx §6](./decisions.mdx)). The machine-local `config.yaml → decisions:` map
below is a **reconciled cache** of that ledger's IPFS axis, not an independently editable setting.

### 2.7 Default-decision policy for new files (shared)

This is where the team sets what happens to **newly-discovered** big files so nobody triages the same
kind of file forever ([decisions.mdx §9](./decisions.mdx)). It is a **shared** policy (travels in the
SDL), not a machine-local one:

* Per file **kind** — **videos/images** and **other big files** — choose **Auto-decide** (apply a default
  IPFS/git-ignore outcome on discovery and record it as a policy-authored decision) or **Ask** (leave
  Undecided and surface the warning). **Default: Ask** for everything (the charter's "surface, don't act"
  posture; auto-decide is a deliberate opt-in).
* When Auto-decide is on, pick the two default axes for that kind (e.g. media → **IPFS on + git-ignore
  on**). Auto-decisions respect the **Never IPFS** flag ([decisions.mdx §17](./decisions.mdx)) and are
  reported in each pass summary.

The shared default-decision policy is written to the SDL and applied on scan in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/decisions.service.ts` — `readDecisionPolicy()` / `setDecisionPolicy()` / `applyDefaultPolicy()`.

### 2.8 Decision attribution mode (privacy — shared)

Because the decision ledger records **who decided** and is **committed** to the repo, this setting governs
what identity is written ([decisions.mdx §14](./decisions.mdx)):

* **`email`** (default for a **private** repo) — the raw allow-listed email (full attribution).
* **`handle`** (default when the remote is a **public** host) — a stable opaque participant id; the
  email↔handle map stays machine-local and is never committed, so a public repo does not leak emails.
* **`anonymous`** — record only *that* a decision was made, not by whom.

LFB **warns before the first commit** if raw emails would be published to a public remote. The mode is
resolved in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/decisions.service.ts` — `attributionFor(folder, email)` (public-remote heuristic via `classifyRemote()`).

### 2.9 Sync this repo's tracking state to the company sync repo

The one control that governs whether this repo's LFB **tracking state** travels between your computers. By
default that state — `repo_storage.yaml`, `files.yaml`, the per-file sidecars, `history/<device>.txt`,
`decisions.yaml`, and `manifest.yaml` — lives in **Local Storage only**
(`~/T/_large_files_bridge/repos/<repoKey>/`), machine-local and **never** in the working repo
([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).

* A **toggle: "Sync this repo's tracking state to the company sync repo."**
* **Default: OFF** — tracking state stays in Local Storage only. **SUPERSEDED** by
  [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)
  and by §2.9.1 below: when an owning storage with a **git-backed SDL** resolves, the default is **ON** and this
  toggle is an **opt-out**. The sentence above survives only as the description of the *residual* case — no sync
  repo can be resolved at all — where there is nothing to mirror to.
* **Enabled only when the owning company/Personal storage has a sync repo configured**
  ([storage_settings.mdx §4b](./storage_settings.mdx#4b-the-sync-repo--an-additive-mirror-of-owned-repos-tracking-state),
  owner resolved per §6); otherwise the toggle is **disabled with a hint** — *"No sync repo is configured for
  &lt;Owner&gt;. Set one up in Storage settings → Sync repo."* — matching the placement radio's disabled
  state ([placement_radios.mdx §5.1](./placement_radios.mdx#51-sync_repo-disabled-when-no-sync-repo-exists)).
* **When ON**, this repo's Category-B tracking state is **additionally mirrored** to
  `<syncRepo>/repos/<repoKey>/` — in addition to Local Storage, never instead of it — so it commits + pushes
  over the owning storage's git backbone to your other computers. Logs never travel there.

It persists on the machine-local `config.yaml` as `sync_repo.enabled` (§3); the sync repo it targets is
derived from the repo's **owner** (§6), not stored here — so the same repo resolves the right sync repo on
another computer that has the owner's sync repo configured. This toggle governs only the **tracking-state**
mirror; it is distinct from the §4/§5 placement radios, which the transcript/AI-description **content**
artifacts use (those default to `lfbridge` regardless).

Reading whether a sync repo exists for the owner, and persisting this per-repo toggle, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null` when none configured or the toggle is off) · `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `PATCH /api/repos/:repoId/settings` (writes `sync_repo.enabled`).

### 2.9.1 The default is ON — this toggle is an OPT-OUT (LOCKED)

The authority for this rule is
[storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked);
this section says only what it means **on this page**. It supersedes the "Default: OFF" bullet above.

* **An owning storage's SDL that is a git working tree IS its sync repo.** No second repo has to be
  configured for this toggle to have somewhere to point: a company / Personal SDL is normally a git repo, and
  LFB **auto-adopts** it. An **explicitly configured `sync_repo`** on the owning storage
  ([storage_settings.mdx §4b](./storage_settings.mdx#4b-the-sync-repo--an-additive-mirror-of-owned-repos-tracking-state))
  still **wins**; auto-adoption is the **fallback**, never an override.
* **So the toggle ships ON.** When an owning storage with a git-backed SDL resolves, this repo's Category-B
  tracking state mirrors by default. The toggle **still exists and still wins** — the user can switch it off
  for a repo whose tracking they do not want travelling — it is simply an **opt-out** now, not an opt-in.
  The reason for the flip is in §8.4.2 and is not restated here: the shipped configuration of the product
  must already do the thing the product is for.
* **Absent means default, not off.** A repo whose `config.yaml` has **no** `sync_repo` block resolves to the
  **default (ON)**. Only an explicit `sync_repo.enabled: false` — written because the user moved this toggle —
  turns the mirror off. A repo that predates the flip and never had the block therefore starts mirroring; that
  is intended.
* **No git remote → cannot mirror, and the UI says so.** A repo's shared identity is its **remote**, never its
  path ([storage_company.mdx §8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked)),
  so a repo with **no remote** has no subtree any other computer would ever read. The toggle renders
  **disabled and OFF** with the honest hint — *"No git remote — this repo's tracking stays on this
  computer."* This is a **statement of fact, not a warning**: nothing is broken, there is simply nothing to
  address the mirror to. It is not an error state, it raises no warning banner
  ([warnings.mdx](./warnings.mdx)), and the repo works normally in every other respect.
* **No owning storage resolves → also disabled, with a different hint.** If ownership resolves to a company
  that has no discovered company storage row, or to Personal before Personal exists, there is no SDL to adopt.
  The toggle renders **disabled and OFF** with the setup hint — *"No storage owns this repo yet. Set up
  &lt;Owner&gt; in Storage settings."* — matching the placement radio's disabled state
  ([placement_radios.mdx §5.1](./placement_radios.mdx#51-sync_repo-disabled-when-no-sync-repo-exists)). Both
  disabled cases keep the tracking state in **Local Storage only**, which is real and honored locally
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)); nothing is lost, it is simply not shared
  yet.
* **Ownership decides WHICH sync repo, and it now resolves from the REMOTE first.** The target is the owning
  storage's sync repo, and that owner is derived from the repo's **remote org** before any path-containment
  test ([repo_company_mapping.mdx §7.3](./repo_company_mapping.mdx#73-ownership-decides-the-sync-repo-target-remote-first-locked)).
  A repo like `~/BGit/Bryan_git/charlie-kirk` with remote `github.com/ACT3ai/charlie-kirk` therefore mirrors to
  the **ACT3ai** company SDL even though it sits nowhere near that company's directory.
* **Still only text, still additive.** ON changes nothing about **bytes**. Media moves over IPFS, for decided
  files only ([decisions.mdx](./decisions.mdx)); Local Storage remains the authoritative copy and the mirror
  is written **in addition to** it, never instead of it.

Resolving the owner's sync repo — the explicitly configured one first, the SDL's own git root auto-adopted second — and writing this toggle's marker is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveOwnerDedicatedRepo(repoRoot)` (remote-owner first, path containment second), with the SDL-root fallback the Git backbone already performs in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getGitBackboneRemote(storageId)`, persisted by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `setSyncRepoMarker(repoRoot, syncRepoRoot)` from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `PATCH /api/repos/:repoId/settings`.

## 3. Persistence — one repo config file

Everything above writes to the single per-repo config
([storage_local.mdx §6.2](./storage_local.mdx#62-pinrrepo_nameconfigyaml)):

```yaml
# ~/T/_large_files_bridge/pin/r/<repo_name>/config.yaml
schema_version: 1
updated_at: 2026-07-02T14:32:15.123Z

repo:                          # identity — written by the scanner, read-only on this page
  name: largefilebridge
  path: ~/BGit/Bryan_git/LargeFileBridge
  remote: https://github.com/BryanStarbuck/LargeFileBridge.git

pinned: false                  # §2.2 master toggle

big_file_override:             # §2.1 threshold override (OFF = inherit settings.mdx global)
  enabled: false
  value: 100
  unit: MB

large_files:                   # §2.3 selection rules
  follow_gitignore: true
  include_globs: []
  exclude_globs: []

pin:                           # §2.4 pinning / fetch policy
  pin_locally: true
  fetch_missing: true
  publish_manifest: true

access:                        # §2.5 sharing
  shared: false
  participants: []

sync_repo:                     # §2.9 mirror this repo's tracking state to the owner's sync repo
  enabled: false               # OFF = Local Storage only; ON = ALSO mirror to <syncRepo>/repos/<repoKey>/
```

* Repo **identity** (`repo:`) is written by the [scanner](./scan.mdx) and shown read-only here — the
  user configures behavior, not identity.
* Any field a repo does **not** override falls back to the global default: the threshold falls back to
  [settings.mdx](./settings.mdx) when `big_file_override.enabled` is false; `large_files.min_bytes`
  semantics are subsumed by the threshold-override model.

Writing every field on this page to the repo's `config.yaml` (and mapping it back for the UI) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `PATCH /api/repos/:repoId/settings` (via `updateRepoConfig` / `toRepoSettings`)

### 3.1 Two repo files, two scopes — both machine-local (Local Storage)

There are **two** per-repo files and they must not be confused — but note **both are machine-local; neither
lives in the working repo**:

* **`pin/r/<repo_name>/config.yaml`** (this page) holds **this computer's participation** in the repo — its
  `pinned` master toggle, threshold override, pin/fetch policy, sharing, and the §2.9 sync toggle. It **never
  travels**; each computer has its own.
* **`repo_storage.yaml`** ([repo_tracking_scheme.mdx §2](./repo_tracking_scheme.mdx)) holds the repo-**wide**
  tracking facts that belong *with the repo* — its LFB display name, enlist provenance, the rolled-up
  **special-file counts** ([special_files.mdx](./special_files.mdx)), repo-wide recommendation policy, and
  the last-scan stamp. Its **only** level-one key is `repo_storage:`. It is **not** in the working repo
  either: it lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/repo_storage.yaml`
  (`repoKey = shortHash(resolve(repoPath))`), is auto-written on enlist, and is **not** edited from this
  settings page. It travels only when this repo's §2.9 sync toggle mirrors it to the owning storage's sync
  repo.

Rule of thumb: **`config.yaml` is about *this computer's* participation; `repo_storage.yaml` is about *the
repo* — and both are machine-local (Local Storage), never committed into the working repo.** They never
duplicate a field. (Per-file **decisions** are a third thing again — the `decisions.yaml` ledger, also in
Local Storage and travelling via the sync repo, §2.6.)

## 4. Transcription — where transcripts are stored

This section chooses **where this repo's transcripts** (`<rel>.transcription`,
[Transcribe.mdx](./Transcribe.mdx)) are written on disk, using the **shared 3-way placement radio**
defined in [placement_radios.mdx](./placement_radios.mdx). The AI descriptions section (§5) uses the
**identical** control for `<rel>.ai_description` artifacts.

### 4.1 The radio

The section header reads **"Transcription — where transcripts are stored"** and mounts one instance of
the placement radio with the transcript-flavored copy:

* **(a) In this repo's hidden `.lfbridge/` folder** *(recommended)* — `<repo>/.lfbridge/<relDir>/<name.ext>.transcription`,
  path-mirrored, extension **appended**, committed, travels with the repo by git
  ([Transcribe.mdx §3](./Transcribe.mdx#3-where-transcripts-are-stored--inside-the-owning-roots-tracking-area-path-mirrored-extension-appended-locked--supersedes-the-earlier-beside-media-layout)).
* **(b) Next to the media file** — `<mediaDir>/<name.ext>.transcription`, a sibling of the media. This
  **deliberately reintroduces** the beside-media layout Transcribe.mdx marked "superseded", now offered as a
  supported user choice with **appended** (not replaced) extensions — see the reconciliation in
  [placement_radios.mdx §4](./placement_radios.mdx#4-the-beside-media-reintroduction-reconciliation--locked).
* **(c) In this storage's LFB sync repo** — `<syncRepo>/repos/<repoKey>/…`, **only offered (enabled)** when
  the owning company/Personal storage has a sync repo configured
  ([storage_company.mdx](./storage_company.mdx)); otherwise **disabled with a hint** on how to set one up
  ([placement_radios.mdx §5.1](./placement_radios.mdx#51-sync_repo-disabled-when-no-sync-repo-exists)).

All radio behavior — labels, helper text, disabled/hint states, and the resolution to an on-disk path — is
owned by [placement_radios.mdx](./placement_radios.mdx); this section only says **which artifact** it governs
(the `.transcription` transcript) and **its default** (§4.4).

### 4.2 Switching affects only FUTURE transcripts

Changing this radio **does not move** existing transcripts. Transcripts already written stay where they are;
only the next transcript uses the new tier
([placement_radios.mdx §7](./placement_radios.mdx#7-switching-the-radio-only-affects-future-artifacts)).

### 4.3 Move existing transcripts

When transcripts exist in a tier other than the selected one, the section shows
**"Move N existing transcripts here"** — the explicit, confirmed, non-destructive migration defined in
[placement_radios.mdx §8](./placement_radios.mdx#8-the-move-existing-artifacts-action). Hidden when N = 0.

### 4.4 Default

* **Regardless of whether the owning storage has a sync repo**, transcripts default to **`lfbridge`** (the
  repo's hidden `.lfbridge/`), because a transcript is content *about the media in this repo* and users
  expect it to travel *with* that repo.
* The sync repo carries this repo's **tracking state** — a **separate data category** (Category B: Local
  Storage always, plus the sync-repo mirror when the §2.9 toggle is on), never these content artifacts. So
  turning the §2.9 sync toggle on never changes this transcript default
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
  [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).

### 4.5 Persistence

Persists as `RepoSettings.transcription.placement` (enum `lfbridge` | `beside` | `sync_repo`) — see §7.

## 6. Ownership — which company (or Personal) owns this repo

This section surfaces and lets the user change the repo's **owner** — the company or **Personal** storage the
repo maps to, derived per [repo_company_mapping.mdx](./repo_company_mapping.mdx). It is **read-mostly** (the
mapping is normally automatic) with one control to override it.

### 6.1 What it shows

* **Owner** — the resolved **company name** or **Personal**. This is the same owner that decides whether the
  §4/§5 `sync_repo` option — and the §2.9 tracking-state sync toggle — is enabled (a company with a sync repo
  makes them available).
* **Derivation source** — how the owner was determined:
  * **Auto (from git remote)** — inferred from the git remote's org/host
    ([repo_company_mapping.mdx](./repo_company_mapping.mdx)); shows the remote it matched on.
  * **Manual** — the user reassigned it (§6.2); shows *"set by you"* and when.
* **Sync repo** — whether the owner has one configured (drives §4/§5 option (c) and the §2.9 toggle); if not,
  a link to set one up ([storage_company.mdx](./storage_company.mdx) /
  [storage_settings.mdx §4b](./storage_settings.mdx#4b-the-sync-repo--an-additive-mirror-of-owned-repos-tracking-state)).

### 6.2 Reassign control

A **"Reassign owner…"** control lets the user override the derived mapping:

* Opens a picker of known companies + **Personal**; choosing one sets a **manual** mapping that **wins over**
  the auto-derivation.
* A **"Reset to auto"** affordance clears the override so the owner reverts to the git-remote derivation.
* Reassigning **re-evaluates** the §4/§5 `sync_repo` availability immediately (a repo moved to a company that
  *has* a sync repo enables option (c); moved to one that doesn't disables it with the hint).
* Reassigning **does not move** any existing artifacts or tracking data — it changes *ownership/derivation*
  only; use the §4.3/§5.3 move actions to relocate artifacts if desired.

The owner, its derivation source, and the manual override are read/written in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` reads the derived/overridden owner into `RepoSettings` and accepts the reassign via the same `PATCH /api/repos/:repoId/settings` (owner override field); the derivation itself belongs to [repo_company_mapping.mdx](./repo_company_mapping.mdx).

## 7. Persistence additions — the new RepoSettings fields

§4, §5, and §6 add fields to the per-repo config and to the shared `RepoSettings` type. They save through the
**existing** PATCH — no new settings endpoint:

```yaml
# pin/r/<repo_name>/config.yaml  (excerpt — the fields §4/§5/§6 add)
transcription:
  placement: lfbridge      # §4 — lfbridge | beside | sync_repo  (default lfbridge)
ai_description:
  placement: lfbridge      # §5 — lfbridge | beside | sync_repo  (default lfbridge)
ownership:
  owner: null              # §6 — company id or "personal"; null = use derived
  source: auto             # auto | manual
sync_repo:
  enabled: false           # §2.9 — mirror this repo's tracking state to the owner's sync repo.
                           # The literal "(default off)" this line used to carry is SUPERSEDED (§2.9.1,
                           # storage_company.mdx §8.4.2): an ABSENT sync_repo block resolves to the
                           # default, which is now ON whenever an owning storage with a git-backed SDL
                           # resolves. Only an explicit `false` — written by the user moving the toggle —
                           # opts this repo out.
```

* `transcription.placement` and `ai_description.placement` are **machine-local intents**; the `sync_repo`
  placement-radio *availability* — and the §2.9 tracking-state toggle's availability — is derived from the
  owner, not stored, so the setting resolves correctly on another computer that has the sync repo configured
  (§6.1, [placement_radios.mdx §6](./placement_radios.mdx#6-persistence)).
* `ownership.owner` is the **manual override**; when null, the owner is the git-remote derivation
  ([repo_company_mapping.mdx](./repo_company_mapping.mdx)). `source` records which of the two applied.
* `sync_repo.enabled` is the **§2.9 per-repo tracking-state mirror toggle** (~~default off~~ — **default ON**
  per §2.9.1 / [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked));
  when on, this repo's Category-B tracking state is additionally mirrored to `<syncRepo>/repos/<repoKey>/`
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). **Absent ⇒ default**, so the field is a
  tri-state on disk (`true` / `false` / absent) and only an explicit `false` opts a repo out.

The two placement enums, the ownership override, and the sync-repo toggle are added to the shared type and
mapped by the settings router in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` — `RepoSettings` gains `transcription: { placement }`, `aiDescription: { placement }`, `ownership: { owner, source }`, and `syncRepo: { enabled }` · `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `toRepoSettings()` maps `config.yaml` → these fields and `PATCH /api/repos/:repoId/settings` writes them back; the mirror itself is driven by `tracking-sync.service.ts` — `mirrorToSyncRepo()`.

The placement enums are consumed when an artifact is written by
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveArtifactPlacement()` (honors the per-repo `placement` before its own fallback order).

## 8. Acceptance criteria

1. The per-repo settings page is opened by the **gear icon** on a single repo's page, is scoped to
   that one repo, and names the repo + its path.
2. The page shows a **big-file threshold override checkbox, OFF by default** (= inherit the global
   100 MB). Turning it ON reveals a **value + MB/GB/TB** control that overrides only this repo;
   turning it OFF reverts to inheriting the global value.
3. The page exposes the **pinned on/off** master toggle (default OFF — discovered repos are not
   bridged until opted in).
4. The page exposes the **large-file selection rules** (follow `.gitignore` default on, include/exclude
   globs), the **pinning/fetch policy** (pin locally / fetch missing / publish manifest, all default
   on), and **sharing** (private by default; shared adds allow-listed participants).
5. Everything on this page writes to that repo's `pin/r/<repo_name>/config.yaml` — never the global
   config; unoverridden settings fall back to the global defaults in [settings.mdx](./settings.mdx).
6. Repo identity (`repo:` name/path/remote) is written by the scanner and shown read-only here.
7. A **Transcription** section (§4) offers the shared 3-way placement radio
   ([placement_radios.mdx](./placement_radios.mdx)) for `.transcription` artifacts: `lfbridge` (default,
   recommended), `beside`, and `sync_repo` (enabled only when a sync repo exists, else disabled with a
   setup hint). Switching affects only future transcripts; a **"Move N existing transcripts here"** action
   migrates the rest.
8. An **AI descriptions** section (§5) offers the **identical** radio for `.ai_description` artifacts, with
   the same defaults, disabled states, and move action; the two radios are independent.
9. In both §4 and §5, the default is **`lfbridge`** whether or not the owning storage has a sync repo — the
   transcript/AI-description **content** (Category A) always defaults to the repo's `.lfbridge/`, while the
   separate **tracking-state** data category (Category B) lives in Local Storage always and mirrors to the
   sync repo only when the §2.9 toggle is on
   ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).

    beside-media layout (appended, not replaced, extension), reconciled with Transcribe.mdx / ai_description.mdx.

    (auto from git remote / manual), and whether a sync repo exists, with a **"Reassign owner…"** control
    (and reset-to-auto) that re-evaluates `sync_repo` availability without moving artifacts.

    **AC 13's "default OFF" is superseded by AC 14.**

    Concretely: an owning storage whose SDL is a **git working tree** is **auto-adopted** as the sync repo (an

    the default **ON**, and only an explicit `sync_repo.enabled: false` opts a repo out; a repo with **no git
    remote** renders the toggle **disabled + OFF** with *"No git remote — this repo's tracking stays on this
    computer"* (a fact, not a warning); a repo whose owner resolves to **no storage** renders it **disabled +
    OFF** with the *"Set up &lt;Owner&gt; in Storage settings"* hint; and the mirror target follows ownership
    derived **remote-org first** ([repo_company_mapping.mdx §7.3](./repo_company_mapping.mdx#73-ownership-decides-the-sync-repo-target-remote-first-locked)).
    No **bytes** behavior changes in any of these cases.
