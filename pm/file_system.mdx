## 1. Interesting directories — the idea (LOCKED)

When a user opens the File System, most folders are noise — config, source, dot-directories. The few that
matter are the ones holding **big files, videos, or uncompressed images** — the files LFBridge exists to
sync and to help compress. So we **tint the folder icon** of those directories to make them stand out at a
glance, and we tint them **more strongly** the more important their contents.

A directory is **interesting** if — considering the directory itself **and everything under it,
recursively** — **any** of the following is true:

1. It contains at least one **video** file.
2. It contains at least one **uncompressed image** file that is **≥ 3 MB**.
3. It contains at least one file **at or over the big-file threshold** (the global
   `big_file.threshold_bytes`, default **100 MB** — [settings.mdx §1](./settings.mdx); a per-repo override
   may raise/lower it, [repo_settings.mdx](./repo_settings.mdx)).
4. It contains at least one **sub-directory (recursively) that is itself interesting** — i.e. interest
   **bubbles up**: a parent of an interesting folder is interesting.

Criterion 4 is automatic once interest is computed recursively: a directory is interesting iff **its whole
subtree** contains any of 1–3. Only **one** of the criteria has to hold — a folder needs just one video,
or just one big file, to qualify.

> **Why 3 MB for images and not the compress floor.** The compress-eligibility floor for images is 1 MiB
> ([compression.mdx](./compression.mdx)); interest uses a **higher 3 MB floor** so a folder of small
> uncompressed PNG icons does **not** light up. Videos have **no** size floor for interest — any video
> makes a folder interesting (a small already-compressed clip still counts). Big files use the threshold as-is.

## 2. The three colors of interesting (LOCKED)

A folder is **one** color, chosen by the **highest-priority** category present **anywhere in its subtree**.
Priority, strongest first: **big file → video → uncompressed image**.

| Level | Condition (anywhere in the subtree) | Folder glyph — outline | Folder glyph — fill | Meaning |
| --- | --- | --- | --- | --- |
| **1. Brown (strongest)** | ≥ 1 file **≥ big-file threshold** (default 100 MB) | **dark brown** | **light brown** | The important one — real large files live in here (incl. deep). |
| **2. Blue fill (medium)** | no big file, but ≥ 1 **video** | **medium-dark blue** | **very light blue** (near white) | Video content worth syncing/compressing, nothing over the threshold yet. |
| **3. Blue outline only (weakest)** | no big file and no video, but ≥ 1 **uncompressed image ≥ 3 MB** | **blue** | **none** (transparent) | Some heavy uncompressed images; the gentlest nudge. |
| **— Not interesting** | none of the above | default slate (unchanged) | none | Plain folder — no tint. |

Rules that make this unambiguous (all LOCKED):

* **Highest wins.** A folder that contains both a 4 GB `.mov` **and** loose PNGs is **brown**, not blue —
  the big file dominates. A folder with a small compressed `.mp4` and a 5 MB `.png` (no big file) is **blue
  fill** (video beats image). The big-file (brown) level explicitly requires a file **≥ threshold**; having
  only videos or only images — however many — is **not** enough for brown.
* **Recursive.** The color reflects the **whole subtree**, so a top folder shows **brown** when the only
  big files are several levels deep. The user sees the important branch from the root.
* **The base "interesting" look is the blue-fill treatment** (medium-dark-blue outline + very-light-blue
  fill) the user described — brown is the escalation for big files, blue-outline-only is the de-escalation
  for images-only.
* **Not interesting = no change.** A directory with none of 1–3 keeps today's plain folder glyph
  (`text-slate-500`, no fill). We do **not** color every folder — only the ones worth acting on.

The exact color tokens (added to the `:root` block in `styles.css`; the brown reuses the existing repo
badge ramp so the palette stays coherent):

```css
/* Interesting-directory folder glyph — outline (stroke) + fill */
--lfb-fold-big-line:    #5b3a1a;  /* dark brown  (= --lfb-badge-repo-root) */
--lfb-fold-big-fill:    #e7d3b1;  /* light brown */
--lfb-fold-video-line:  #2f6fed;  /* medium-dark blue (= --lfb-badge-ipfs) */
--lfb-fold-video-fill:  #eef4ff;  /* very light blue, near white */
--lfb-fold-image-line:  #2f6fed;  /* blue */
--lfb-fold-image-fill:  transparent; /* outline only, no fill */
```

The folder glyph is a single `lucide-react` `<Folder>`; **outline = its `color`/stroke, fill = its `fill`**.
So brown = `{ color: var(--lfb-fold-big-line), fill: var(--lfb-fold-big-fill) }`, video = the blue pair,
image = `{ color: var(--lfb-fold-image-line), fill: transparent }`, not-interesting = the current
`text-slate-500` with no fill.

## 3. Where the coloring applies + how it's computed

### 3.1 Everywhere a folder glyph is drawn (LOCKED)

This is **not** a File-System-page-only flourish. The user's rule: *"We'll do that pattern anywhere in the
file system where we show directories or even repos also."* So the interesting-folder coloring is a
**shared helper** applied to **every folder glyph across the app**:

* the **File System column browser** rows (`/fs`) — the primary place;
* **View one directory** child listings ([directories.mdx](./directories.mdx) Part B);
* **Storage detail** directory rows ([storages.mdx](./storages.mdx));
* **repo rows** where a repo is drawn as a folder — a repo is colored by the **same rule over its working
  tree** (a repo full of movies shows **brown**), layered under its existing R/r repo badge, so repos and
  plain folders read the same way.

The Full-paths tab (`/fs/paths`, [full_paths.mdx](./full_paths.mdx)) lists **files, never directory rows**,
so it has no folder glyph to color — it is unaffected.

### 3.2 The interest signal (LOCKED shape)

Each directory entry the UI renders carries an **`interest`** value:

```ts
type FolderInterest = "big" | "video" | "image" | null;
// "big"   → brown         (≥1 file ≥ threshold in the subtree)
// "video" → blue fill      (no big file, but ≥1 video)
// "image" → blue outline   (no big file/video, but ≥1 uncompressed image ≥ 3 MB)
// null    → not interesting (plain glyph)
```

It is computed by the **same bounded recursive walk** the charter rollup already does — extend
`buildDirRollup()` (which today emits `videosToCompress` / `imagesToCompress` / `bigNotIgnored` /
`bigIgnoredNotTracked`) to also emit `interest`, or compute it from those plus a broadened "any video /
any image ≥3MB" pass. The detection reuses the existing classifiers so the vocabulary never drifts:

* **video** = extension in `VIDEO_EXT` (`fs/badges.ts`) — for interest, **any** video counts (do not
  require `compressState === "should"`).
* **uncompressed image ≥ 3 MB** = extension in `IMAGE_UNCOMPRESSED_EXT` (`fs/badges.ts`) **and**
  `size >= IMAGE_INTEREST_FLOOR_BYTES` (3 MB = `3 * 1024 * 1024`).
* **big file** = `size >= getAppConfig().big_file.threshold_bytes` (default 100 MB; per-repo override
  honored), matching the size filter in [directories.mdx §1.4](./directories.mdx).

Determining each **column-browser** directory row's interest level (the primary place — the `/fs` tree) is computed in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `computeDirInterest()` (the bounded subtree walk, mtime+TTL cache, early-exit on the first big file, and the truncation floor of §3.3), called per directory child by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDir()` (which shares **one** `interestBudget` across every child in the column so a huge sibling can't pin the thread).
The **entity-detail / View-one-directory** rollup computes the same value as a by-product of its category walk in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` — `buildDirRollup()` (emits `interest`) · both paths share the classifiers `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `compressInfo()` / `VIDEO_EXT` / `IMAGE_UNCOMPRESSED_EXT`. The value rides on each directory entry in the shared type `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` — `FsEntry` (`interest?: FolderInterest`).

### 3.3 Performance — computed on the walk, cached, never blocking (LOCKED)

Interest is a **recursive** property, so a naive per-row recompute is expensive on a deep tree. Follow the
model already in [performance.mdx](./performance.mdx):

* Reuse the **bounded** walk (depth ≤ 8, `HARD_SKIP` dirs excluded). Interest is a by-product of that same
  pass — no extra traversal.
* **Fair per-child budget, not one greedy shared pool (LOCKED).** When a column lists its children, each
  directory child gets its **own** budget capped at **`INTEREST_PER_CHILD` (3,000 entries)**, drawn from a
  larger per-listing ceiling **`INTEREST_TOTAL` (120,000)**; only what a child actually consumes is drawn
  down, so cheap dirs leave budget for the rest. This is the fix for the **`_Mirror` starvation bug**: the
  old design shared **one** 40k budget consumed first-come, so huge early siblings (Dropbox/Library) drained
  it to **0 before `_Mirror` was even walked** — `_Mirror` got `undefined` → a **plain glyph** — and because
  `undefined` is never cached, every refetch re-drained the pool and `_Mirror` stayed plain **forever**. A
  per-child cap makes the walk fair and position-independent: every child, `_Mirror` included, gets its own
  slice and resolves (`_Mirror` finds its big file in ~35 entries).
* **The walk must never block the event loop (async policy, LOCKED).** `computeDirInterest` is a synchronous
  stat loop; a single heavy child can do thousands of stats. The listing loop counts those consumed entries
  toward its yield budget and hands the event loop back (`setImmediate`) so a cold multi-second walk stays
  **non-blocking** — concurrent requests interleave instead of hanging.
* **Cache** the result by directory + mtime/`scannedAt` (20 s TTL); serve the cached `interest` instantly.
* The column browser **renders immediately** with the plain glyph and **upgrades** the tint when the
  interest value arrives — the tree never blocks on the walk.

**Truncation returns the best-known FLOOR, it never discards a positive hit (LOCKED).** When the bounded
walk hits the budget cap, we do **not** throw away what the walk already saw. The rule is a **floor**:

* If the walk **already found a video** (or an uncompressed image ≥ 3 MB) before the cap, the directory is
  **definitely at least that** — return **`"video"`** (or `"image"`). It shows **blue**. The only thing
  truncation can hide is a **deeper big file** that would have **upgraded** the color to brown — so a
  truncated floor is always **safe** (it under-states at worst, never a false "not interesting"). Leave a
  truncated floor **uncached** so a later full-budget walk can still upgrade it to `"big"`.
* Only when the walk hits the cap **having found nothing yet** is the result **unknown** (`undefined` →
  plain glyph) — a big file could still be deeper, so we don't claim "not interesting".
* A **fully drained** walk that found nothing is a definite **`null`** (not interesting), which is cached.

So a huge directory like **`~/_Mirror/`** (≈20 k entries, a ≥100 MB file and videos several levels deep)
shows **brown** (it finds its big file in ~35 of its 3,000-entry slice); before both fixes it showed a
plain glyph. Even a `_Mirror` with only videos (no big file) is guaranteed **at least blue** by the floor
rule the moment a video is seen.

Sharing the per-listing budget as a per-child cap plus a total ceiling (so no single sibling can starve the rest), and yielding the event loop as heavy children consume it, is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDirectory()` (`INTEREST_PER_CHILD` / `INTEREST_TOTAL`; per-child `childBudget`, `interestPool` draw-down, and `sinceYield += consumed`).

Returning the best-known floor on budget exhaustion (yield the already-found video/image rather than discarding it, and leave that truncated floor uncached so a later full walk can upgrade it to `"big"`) is enforced in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `computeDirInterest()` (the `truncatedFloor()` helper at each `budget.left <= 0` return) · mirrored for the entity-detail rollup in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` — `buildDirRollup()` (the `interestVideo`/`interestImage` branches taken before the `walkTruncated → undefined` branch).

The filesystem discovery scan that keeps the big-file / candidate data fresh runs **out-of-process and cooperatively async** (it yields to the event loop every few hundred entries so it never freezes the web app) on a schedule of **at least every 2 hours** (`scan_process.interval_hours`, default **2**), implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()` (the `yieldToLoop()` cadence) · scheduled via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/schedule.service.ts` — `intervalFor()` → the launchd `StartInterval`. The default lives in `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — `scan_process.interval_hours` (default **2**).

Existing installs are migrated forward so the new cadence takes effect without user action. The old stored default (4h) is healed to 2h when config loads, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig()` (rewrites `scan_process.interval_hours === 4 → 2`; a value the user deliberately set is left alone). The already-installed launchd plist still fires on its baked-in `StartInterval` until re-rendered, so at boot any drift between the plist's interval and the configured interval is reconciled (the plist is rewritten and reloaded, and only for workers the user already installed) in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/schedule.service.ts` — `reconcileWorkerSchedules()` (reads the installed interval via `SchedulerInstaller.installedIntervalSeconds()`, implemented for macOS in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/os/launchd.ts`), called once from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/main.ts` — `bootstrapState()`.

Mapping the `interest` value to the folder glyph's `color`/`fill` is done in code:

### 4.1 Scope containment — the deepest selected column, downward only (LOCKED)

This is the **safety guarantee** the whole column browser hangs on, and it is **LOCKED**. When the user
walks left → right through the columns (e.g. `Home → _Mirror → Politics → Charlie_Kirk_Mi`) and clicks a
producing action such as **Create Transcriptions** with **nothing checked**, the action's scope is the
**single deepest (rightmost) directory the user has opened** — and the recursive walk is **strictly
contained inside that directory**. It **descends only**; it **never ascends** to a parent column and
**never touches sibling directories**.

Concretely, with `Charlie_Kirk_Mi` as the deepest open column:

* ✅ The walk covers **`~/_Mirror/Politics/Charlie_Kirk_Mi/` and everything under it, recursively** — and
  nothing else.
* ❌ It does **not** walk `~/_Mirror/Politics/` (the parent column), `~/_Mirror/` (its parent), or the home
  root — even though each of those is an **open column to the left**. An open left column is **context the
  user drilled through**, not scope.
* ❌ It does **not** touch **siblings** of `Charlie_Kirk_Mi` (e.g. `~/_Mirror/Politics/Charlie_Kirk/`,
  `~/_Mirror/Politics/COVID/`) — only descendants of the deepest column.

The rules that make this unambiguous (all LOCKED):

1. **"Deepest" = the last opened column, not the last highlighted row.** The scope root is the directory
   whose **contents fill the rightmost column** — i.e. the last directory the user clicked open. Selecting
   (highlighting) a **file** inside that column does **not** change the scope root: it stays the containing
   directory. Selecting a file never widens or narrows the directory scope.
2. **A left column is never the scope.** Clicking a directory in an earlier column **replaces** every
   column to its right ([directories.mdx §1](./directories.mdx)), so at any moment the open-column chain is
   exactly the path from the root to the deepest directory. Only the **tail** of that chain is the scope;
   the ancestors are breadcrumbs.
3. **The walk is downward-only.** The recursive collection follows child directories out of the scope root
   and **never** calls `..`/parent — there is no code path by which a parent or sibling can enter the set.
4. **Checked subset overrides, still contained.** If the user has checked specific rows, the scope is
   exactly those checked paths (each of which lives at or below the deepest column) — never their parents.
5. **No-scope is a safe no-op, never a wide walk.** If somehow no column is open yet (root not resolved),
   the scope resolves to *nothing* and the enqueue endpoint returns a `400` — it must **never** fall back
   to walking the home directory or the filesystem root.

Determining the deepest-column scope root (the tail of the open-column stack) is computed in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx` — `FileSystemPage` (`currentDir = stack[stack.length - 1]`, passed as `{ root: currentDir }`; an unopened stack yields `{}`, which the server rejects rather than widening).

Enforcing downward-only containment of the recursive walk (descend into children only, skip hidden/tracking/heavy dirs, never ascend to a parent) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` — `enqueueTranscribe()` → `resolveCandidates()` → `walkMedia()` (recurses into child directories of `root` only; the mirror walk for descriptions is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `enqueueDescribe()`).

## 5. Acceptance criteria

1. **Interesting definition (§1):** a directory is interesting iff its subtree (recursively) contains a
   video, an uncompressed image ≥ 3 MB, or a file ≥ the big-file threshold (default 100 MB). Interest
   bubbles up — a parent of an interesting folder is interesting.
2. **Three colors, highest wins (§2):** the folder glyph is **brown** (dark-brown outline, light-brown
   fill) when the subtree has a **big file**; else **blue fill** (medium-dark-blue outline, very-light-blue
   fill) when it has a **video**; else **blue outline only** (blue outline, no fill) when it has an
   **uncompressed image ≥ 3 MB**; else the plain uncolored glyph. Big beats video beats image.
3. **Recursive:** a top-level folder shows brown when its only big files are several levels deep; the color
   reflects the whole subtree, subject to the walk budget.
4. **Applied everywhere (§3.1):** the same coloring is used for every folder glyph — the `/fs` column
   browser, View-one-directory child lists, storage directory rows, and repo rows (a movie-filled repo
   reads brown) — via one shared `folderGlyphStyle()` helper. Full paths (`/fs/paths`) has no directory
   rows and is unaffected.
5. **Performance + truncation floor (§3.3):** interest is a by-product of the existing bounded walk (per-
   child in the column browser, sharing one budget), cached by directory + mtime/`scannedAt`; the tree
   renders immediately with plain glyphs and upgrades the tint when the value arrives — never blocking. A
   budget-capped walk that **already found a video/image returns that as a definite floor (at least blue),
   uncached so it can still upgrade to brown**; only a budget-capped walk that found **nothing** shows the
   plain glyph (unknown), never a false "not interesting". Concretely, `~/_Mirror/` (≈20 k entries, videos
   several levels deep) shows **at least blue** even though its walk truncates.
6. **Tokens:** the six `--lfb-fold-*` CSS variables from §2 exist in `styles.css` and drive the glyph
   `color`/`fill`; the brown reuses the existing repo-badge brown and the blue reuses the IPFS-badge blue.
7. **Action-links row (§4):** the File System page shows the standard action-links row under its title with
   the catalog items above, following [page_actions.mdx](./page_actions.mdx) (visible links, not a
   dropdown); compress/git-ignore are red-tinted confirm-gated offers.
8. **Scope containment (§4.1):** with nothing checked, a producing action's scope is the **deepest
   (rightmost) open column's directory**, walked **recursively downward only**. Walking
   `Home → _Mirror → Politics → Charlie_Kirk_Mi` and clicking **Create Transcriptions** processes **only
   `~/_Mirror/Politics/Charlie_Kirk_Mi/` and its descendants** — never a parent column (`Politics`,
   `_Mirror`, home), never a sibling directory. Selecting a file inside the deepest column does not change
   the directory scope; an unopened stack is a `400`, never a fall-back walk of home or `/`.
9. **No overlap:** this file does not re-specify the column browser, badges, filters, or the Full-paths
   table — those stay owned by [directories.mdx](./directories.mdx) and [full_paths.mdx](./full_paths.mdx).
