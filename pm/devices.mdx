# LargeFileBridge — Devices / Peers

A storage is carried by **computers** — a laptop, a home desktop, a studio machine, a server, maybe a
family member's machine. LFB calls each one a **device**; because these devices **pin one another's
files over IPFS** they are also **peers**, and the left-bar page is labelled **"Devices / Peers"**
([left_bar.mdx §3](./left_bar.mdx#3-the-app-bar-main-navigation)). This spec defines how a device
**identifies itself** (a minted id + a **hardware fingerprint**), how the app **auto-names** a computer
and **disambiguates** similar machines, how the set of devices **travels with the storage** and is
shown in the **Devices / Peers table** (which always seeds *this* computer so it is never empty), each
device's **sync schedule**, and — the piece everything else depends on — the **graft**: how a storage's
shared, machine-independent directory hierarchy lands on *this* computer's local paths.

The graft is what makes the [seven-step walkthrough](./syncable_data_location.mdx#5-end-to-end-walkthrough--four-computers-one-company-storage)
work: computer 2 adds the storage's Git repo, reads the YAML, and pulls the right files down to the
**right local locations** — because each device declared, in the Syncable Data Location, exactly where
those locations are on it.

Read the shared model first; this spec does not restate it:

* [syncable_data_location.mdx](./syncable_data_location.mdx) — the **SDL** that holds
  `.lfbridge/devices/` and `.lfbridge/mapped_dirs.yaml`. Devices are one part of the SDL's inventory.
* [storage_local.mdx](./storage_local.mdx) — this machine's **local** identity (`config.yaml →
  computer`, §3) and its **`peers.yaml`** (§11). Those are **machine-local and never travel**. The
  device registry here is the opposite: it is **in the SDL and travels**. §4 reconciles the two.
* [sync_list.mdx §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule) — the trust model.
  A device file is **self-owned**: a computer writes only its own and treats the others as claims.

## 1. Device identity — a minted id + a nice name (LOCKED)

Every computer that carries a storage has a **device identity** with two parts:

* **A stable unique device id.** On first use a computer **mints a UUID** — a real id, **distinct from
  the IPFS PeerID** (the PeerID can change if the IPFS keypair is reset; the device id must not). It is
  the durable key by which this computer is known across every storage and in every `pinned_by` /
  `devices/` entry.
  * **The user can change it in settings.** The id is editable from the web app for the rare case of a
    collision, a re-imaged machine, or a deliberate rename of identity. Changing it rewrites this
    device's own `devices/<device>.yaml` (and migrates its `pinned_by` claims) — it never edits another
    device's file.
* **A nice human name.** The user names the current computer from the web app — `bryan-macbook`,
  `studio-tower`, `family-imac`. This is the label shown everywhere (the Devices / Peers UI, `pinned_by`,
  the `devices/` filename). It is **auto-seeded from the hardware fingerprint** (§7 — e.g.
  `bryan-macbook-pro` from the logged-in user + the Mac model) rather than the bare OS hostname, and the
  user can rename it any time.
* **A hardware fingerprint.** Alongside the id and name, each device records a small set of **hardware
  facts** that identify the *physical machine* — Mac model + year, screen size (14″/16″), disk size,
  RAM, chip, and the **logged-in username / home directory**. The fingerprint is what lets the app
  **auto-name** a new computer (§7) and, more importantly, **disambiguate** two similar machines by
  appending only the attributes that differ (§8) — two MacBook Pros become `bryan-macbook-pro (14-inch)`
  and `bryan-macbook-pro (16-inch)`. It is collected locally with **no networking** (§7).

The id is the **random code** the user asked for: it is minted once and then stored in **two places at
once** — this computer's **`~/T/_large_files_bridge/config.yaml`** (the machine-local source of truth,
which never travels) **and** copied into **every synced storage's** `.lfbridge/devices/<device>.yaml`
(which travels through the storage's Git repo — the Act3 company storage, the Personal storage, etc.).
So the same device id is present both locally under `~/T` and in the Dropbox/Google-Drive/Git-synced
storages, and any computer reading a storage can match the array of `devices/*.yaml` entries against its
own local `config.yaml → computer.id` to answer **"which of these devices is *me*?"** (§9).

The id lives in this computer's machine-local `config.yaml → computer`
([storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical))
as the **source of truth for who this machine is**, and is **copied into** each storage's
`devices/<device>.yaml` (§2) so the travelling registry can attribute files to it. The one change this
spec asks of storage_local: `config.yaml → computer` must carry a **minted `id` (UUID) separate from
`ipfs_peer_id`**, and the web app must let the user edit the **id** and the **label**.

Minting the stable device UUID on first boot (separate from the IPFS PeerID) and letting the user rename the id + label from the web app is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig()` / `updateAppConfig()` (mint `computer.id` when absent), exposed through the global Settings routes.

## 2. The device registry lives in the SDL and travels (LOCKED)

Each device records **itself** as one file in the storage's Syncable Data Location:

```
<SDL>/.lfbridge/devices/
├── bryan-macbook.yaml     ← written & owned by the macbook
├── studio-tower.yaml      ← written & owned by the studio tower
└── family-imac.yaml       ← written & owned by the family imac
```

Because the registry lives **in the SDL**, it **travels with the storage** — so every computer that
pulls the storage sees the full set of devices, their schedules, and their grafts. This is how a
machine knows which *other* computers carry a file (redundancy) and how the UI can show "this storage
is on your 4 computers."

The filename is the device's **nice name**, sanitized the same way repo/user folder keys are
([storage_local.mdx §6.1](./storage_local.mdx#61-keying--folder-name)). The authoritative identity
inside the file is the **`id`**, not the filename (a rename moves the file but keeps the id).

### 2.1 A device file is self-owned (trust)

A computer **writes only its own** `devices/<device>.yaml` and treats every other device file as a
**claim**, exactly like a `pinned_by` claim in
[sync_list.mdx §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed):

* This computer's graft, schedule, and pin claims are **authoritative** for itself and written by it.
* Another device's file tells us **its** graft and **its** claims — useful for showing where files are
  and for fetching from that peer, but never something this computer edits or blindly trusts as local
  truth.
* A conflict (two files claiming the same id) resolves by the same revision/provenance rules as any
  other SDL content (sync_list §5.3); the id is the key, the filename is cosmetic.

## 3. What a device file carries

```yaml
# <SDL>/.lfbridge/devices/bryan-macbook.yaml   (written & owned by this device)
schema_version: 1
updated_at: 2026-07-05T10:31:00Z

device:
  id: 4f9c-…                       # the minted UUID (§1) — the durable key; matches config.yaml→computer.id
  name: bryan-macbook              # the nice name the user set from the web app
  owner: bryan@act3ai.com          # the allow-listed user this computer belongs to
  ipfs_peer_id: 12D3KooW…          # for peer dialing (may change; the id above does not)
  # The HARDWARE FINGERPRINT (§7) — travels so OTHER computers can identify & disambiguate this one
  hardware:
    platform: darwin               # os.platform()
    kind: laptop                   # laptop | desktop | server (derived from the model + server mode)
    hostname: Bryans-MacBook-Pro   # os.hostname()
    username: bryan                # os.userInfo().username — the logged-in OS user
    home_dir: /Users/bryan         # os.homedir() — the ~ dir; which user this machine belongs to
    model_identifier: Mac14,7      # `sysctl -n hw.model` — e.g. Mac14,7
    model_name: MacBook Pro        # system_profiler SPHardwareDataType → Model Name
    marketing_name: "MacBook Pro (14-inch, 2023)"   # resolved from the model table (§7); "" if unknown
    year: 2023                     # from the model table
    chip: Apple M2 Pro             # system_profiler → Chip / Processor Name
    arch: arm64                    # os.arch()
    cpu_cores: 12                  # os.cpus().length
    ram_gb: 32                     # round(os.totalmem() / 1e9)
    disk_total_gb: 1000            # fs.statfsSync('/') total, in GB
    screen_inches: 14              # built-in display size (model table; null on a headless server)
    screen_count: 1                # SPDisplaysDataType resolution count

# ── This device's sync schedule FOR THIS STORAGE (the times/cadence it syncs) ──
schedule:
  enabled: true                    # does this device sync this storage at all
  interval_minutes: 15             # cadence (default matches the 15-min background pass)
  windows:                         # optional specific times/windows this device prefers to sync
    - { days: [mon,tue,wed,thu,fri], from: "02:00", to: "04:00" }   # e.g. overnight on a metered link
  # empty windows = sync on the normal cadence with no time restriction

# ── The GRAFT: how the storage's shared hierarchy maps onto THIS computer's local paths (§4) ──
graft:
  # one entry per mapped source directory keyed in mapped_dirs.yaml (syncable_data_location.mdx §3)
  film_renders:
    local_path: ~/Movies/act3/renders    # where THIS computer keeps this mapped directory
    wanted: true                         # does this device want this hierarchy at all
  photo_library:
    local_path: /Volumes/Media/act3/pics # a different absolute path than other computers — the point of the graft
    wanted: true
  model_weights:
    local_path: null                     # not grafted here → known-but-absent on this device
    wanted: false                        # this laptop deliberately does not pull the weights
```

* **`schedule`** is per-device **and** per-storage: the studio tower might sync a company storage every
  15 minutes; a metered laptop might restrict it to an overnight window. It layers on top of the global
  15-minute background pass ([storage_local.mdx §13](./storage_local.mdx#13-the-background-sync-process--the-transparency-contract),
  [sync_process.mdx](./sync_process.mdx)) — the pass still runs, but a storage whose device `schedule`
  says "not now / not wanted" is visited and reconciled without byte work, honoring the same opt-in
  discipline as `synced`.
* **`graft`** has one entry per mapped directory (keyed by `mapped_dirs.yaml`'s `key`, §4).

Writing this device's own file (id, name, schedule, graft) into every storage's `.lfbridge/devices/` and reading the full registry, is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `writeSelfDevice(storageRoot)` / `readDevices(storageRoot)` (new module; self-owned write, whole-registry read).

## 4. The graft — machine-independent hierarchy → local paths (LOCKED)

The graft is the answer to *"the same company storage's directory hierarchy grafts differently into
each computer's local directory hierarchy."* The storage describes its files with **machine-independent**
identifiers — a mapped-directory **key** plus a **relative path** under it
([syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked)).
Each device's graft turns those into **this computer's absolute paths**.

* **Resolution.** For a tracked file `film_renders/shot_012/final_4k.mp4`, this device resolves it to
  `<graft.film_renders.local_path>/shot_012/final_4k.mp4`. On the macbook that is
  `~/Movies/act3/renders/shot_012/final_4k.mp4`; on the studio tower it might be
  `/Volumes/Media/act3/renders/shot_012/final_4k.mp4`. **Same logical file, different local home.**
* **`wanted: false` / `local_path: null`.** A device can decline a whole mapped hierarchy (a laptop
  that does not want 200 GB of model weights). Those files stay **known-but-absent** on this device —
  listed, understood, never fetched here — until the user grafts them.
* **Never trust the writer's path.** `mapped_dirs.yaml.canonical` is advisory; the graft is the only
  thing that resolves a file locally, mirroring
  [sync_list.mdx §4](./sync_list.mdx#4-what-travels-vs-what-stays-local)'s rule that the absolute
  `identity.path` is the writer's and the reader re-resolves locally.
* **Pull to the right place.** When the sync pass fetches a marked CID, it writes the bytes to the
  **grafted** local path (creating parent directories), so computer 2 lands each file where its graft
  says — the core of walkthrough step 7.

Resolving a tracked file's machine-independent (mapped-dir key + relpath) identity to this device's absolute local path via the graft, and **materializing** the fetched bytes there, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `resolveGraftedPath(storageRoot, mappedKey, relPath)`, wired into the sync pass by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` — `syncStorageUnit(id)` (via `resolveStorageAbs`, the storage unit's `resolveAbs` in `runUnitSync`; the unit runs in `syncAll` gated by the per-storage `synced` opt-in). The fetch step of `runUnitSync` now pins the CID **and** writes the bytes to the resolved local path via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/ipfs/ipfs.service.ts` — `catToFile(cid, destPath)` (atomic temp-then-rename, parent dirs created). A mapped dir this device has **not** grafted resolves to `null` and is left **known-but-absent** — never fetched or placed here.

## 5. Device registry vs. `peers.yaml` — two lists, one shared, one local (LOCKED)

These are easy to confuse; keep them distinct:

| | **Device registry** (this spec) | **`peers.yaml`** ([storage_local.mdx §11](./storage_local.mdx#11-peersyaml--the-users-other-computers)) |
| --- | --- | --- |
| Where | `<SDL>/.lfbridge/devices/<device>.yaml` | `~/T/_large_files_bridge/peers.yaml` |
| Travels? | **Yes** — in the SDL, per storage | **No** — machine-local |
| Scope | per storage (a device file per storage) | the whole install (the user's other computers) |
| Holds | id, name, owner, **schedule**, **graft** | id, label, PeerID, owner, last_seen |
| Purpose | *how this storage lands on each computer* | *which PeerIDs to dial for any fetch* |

They cross-reference by **device id**: a `devices/` entry's `id` matches a `peers.yaml` entry's `id`, so
the app can dial the peer (from `peers.yaml`) to fetch a file the device registry says it holds.
`config.yaml → computer` (storage_local §3) is the **self** row that seeds both.

## 6. The Devices / Peers page (the table)

The **Devices / Peers** left-bar item ([left_bar.mdx §3](./left_bar.mdx#3-the-app-bar-main-navigation),
route `/devices`) opens a **TanStack table** — one row per computer that carries your files — built to the
project table contract ([tables.mdx](./tables.mdx), CLAUDE.md → *Tables*): a search box + a single
filter icon + a single sort icon above it, click-a-header to sort, 500/page, full-page-height (nothing
renders beneath it). The table answers, at a glance, **"which of my computers hold my files, which one
am I on, and how do I tell the similar ones apart?"**

### 6.1 Columns (LOCKED)

The table is deliberately **lean** — five columns, no per-spec column for every hardware fact. The
machine facts that would each be their own column (model, year, screen, disk, RAM) are **rolled into
the Device cell** as a descriptor line (§6.2) instead of eating table width; the full fingerprint is a
click away on the View-one-device page (§6.5).

| Column | Source | Notes |
| --- | --- | --- |
| **Device** | disambiguated display label (§8) **+ descriptor** (§6.2) | The nice name, plus any suffix needed to tell it from a similar machine (`bryan-macbook-pro (14-inch)`), with a small descriptor line beneath it (model · screen · chip · RAM · disk). A **"This computer"** chip marks the self row. |
| **Kind** | `hardware.kind` | laptop / desktop / server icon + label. |
| **Owner** | `hardware.username` \|\| `device.owner` | the **logged-in OS username** of that computer (§6.3) — for the self row, whoever is signed in here — not an email. |
| **IPFS Peer ID** | `device.ipfs_peer_id` | truncated to **first 8 `…` last 8** chars in a narrow, fixed-width column; **click to copy** the full id (§6.4). |
| **Last seen** | recency | health-coloured (green < 24h, amber < 7d, neutral older / never). |

**Removed columns.** Model, Year, Screen, Disk, and RAM are **no longer their own columns** — they
crowded the table and forced the page narrow. Their information now lives in the **Device descriptor**
(§6.2) and, in full, on the **View-one-device page** (§6.5).

**Fixed-width columns + a wider page.** Each column declares a fixed width so it is **wide enough** and
never squeezed (Device is the widest to hold the name + descriptor; IPFS Peer ID is narrow because it
is truncated). The `/devices` route also runs **full-width** (no `max-w-6xl` gutter) so the columns
have room to breathe.

### 6.2 The Device cell — name + a rolled-up descriptor (LOCKED)

Because the hardware columns are gone (§6.1), the **Device** cell carries two lines:

* **Line 1** — the disambiguated **display label** (§8): the nice name plus only the suffix needed to
  tell it from a colliding twin, and the **"This computer"** chip on the self row.
* **Line 2** — a compact **descriptor**: the salient machine facts in order — **model · screen · chip ·
  RAM · disk** — skipping any that are unknown (so a bare peers-only row shows just the name). This is
  what "roll a few more pieces into the device name" means: the model/screen/chip/RAM/disk that used to
  be their own columns are now shown right under the name.

Building the descriptor is **pure logic** shared so the backend, the table, and the View-one-device page all agree:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/device-naming.ts` — `deviceDescriptor(hardware)` (returns the ordered, unknown-skipping fact list); rendered by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/devices/DevicesPage.tsx` — `DevicesPage` (the two-line Device cell).

### 6.3 Owner = the logged-in OS user (LOCKED)

**Owner is the OS username**, not the allow-listed email. It comes from the hardware fingerprint's
`username` (`os.userInfo().username`, §7) — for the self row, that is exactly *who is signed in to this
computer right now*. It falls back to `device.owner` (the allow-listed email) only when the username is
unknown (a bare peers-only row with no fingerprint).

Sourcing the self row's owner from the fingerprint username is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `deviceRows()` (self row `owner = hardware.username`); the column's fallback (`hardware.username || owner`) is in `DevicesPage.tsx`.

### 6.4 IPFS Peer ID — truncated + click-to-copy (LOCKED)

A full Peer ID is far too long for a column and would fix the whole page wide. It is shown as its
**first 8 characters `…` last 8 characters** — enough to recognise both ends — in a **narrow,
fixed-width** column. **Clicking the cell copies the full id to the clipboard** (with a toast); the
click is stopped from also opening the row.

The truncation helper is `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/format.ts` — `truncatePeerId(id)`; the click-to-copy cell is in `DevicesPage.tsx` — `DevicesPage` (`copyPeerId`).

### 6.5 Row click → View one device (LOCKED)

**Clicking a row opens that computer's own page** (`/device/$deviceId`) — the "View one device"
experience. It gathers everything the lean table can't fit and every action you can take on that
device:

* **Full identity** — the nice name, the **device id** (copyable), the **owner** (OS user), the **IPFS
  Peer ID** (copyable in full), the row's **source** (this computer / peers list / storage registry),
  how many **storages** carry it, and **last seen** (absolute time).
* **The complete hardware fingerprint** (§7) — kind, model, year, screen, disk, RAM, chip, CPU cores,
  architecture, platform, hostname, home directory, model identifier. A bare peers-only row (no
  fingerprint yet) explains that it fills in once that computer writes itself into a synced storage.
* **Actions** — **Copy Peer ID**, **Open in File System** (the device's home dir; enabled only for the
  self row, since another computer's paths aren't local here), and — for a `peers.yaml`-sourced device —
  **Remove peer** (forgets it from `peers.yaml`; touches no remote content and no local bytes, per
  [menus.mdx §5.4](./menus.mdx)). Two more are **proposed** for this page's header:
  * **Ping / re-announce to this peer** *(proposed)* — dial the device's PeerID and refresh `last_seen`,
    so a user can confirm reachability instead of waiting for the next sync pass. No pointer yet — not
    implemented.
  * **Show files this device carries** *(proposed)* — filter the pinset/file view to the CIDs this
    device's `pinned_by` claims cover, answering "what does this computer hold for me?". No pointer yet.

These header buttons are page-level actions (the device *is* the page's subject), not a per-row `⋮`
kebab ([menus.mdx](./menus.mdx)); this page is not a file-list page, so it carries its own header buttons
rather than the shared `Actions ▾` menu ([page_actions.mdx](./page_actions.mdx)).

The existing three actions render in the `PageHeader` `actions` slot, right-aligned, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/devices/ViewOneDevicePage.tsx` — `ViewOneDevicePage` (the `actions={…}` header slot: Copy Peer ID / Open in File System / Remove peer).

The page is `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/devices/ViewOneDevicePage.tsx` — `ViewOneDevicePage`, routed at `/device/$deviceId` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/router.tsx`. It reads one device by id from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/peers/devices.router.ts` — `GET /api/devices/:id`, backed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `deviceRow(id)` (built on `deviceRows()` so it shows the same disambiguated label + union).

### 6.6 Self always present, never an empty table

The page **auto-seeds this computer** (§9): the row set
is the union of *this device* (always injected from `config.yaml → computer` + the fingerprint) with the
machine-local `peers.yaml` and the travelling `devices/*.yaml` registries across every storage, keyed by
device **id**. If the user has only ever run LFB on one machine, the table still shows exactly one row —
**this** one — with the empty-state guidance ("add a second computer") below it. There is never a
"no devices" state.

Aggregating the device rows the page shows — inject self from config, union `peers.yaml` and each storage's `devices/` registry by id, and disambiguate — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `deviceRows()` (self-seed + union), served by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/peers/devices.router.ts` — `GET /api/devices`. The table itself is `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/devices/DevicesPage.tsx` — `DevicesPage` (the disambiguated **Device** column comes from `disambiguateDevices()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/device-naming.ts`).

### 6.7 Every table is a TanStack table (charter)

Per the charter (*"All tables are TanStack tables. No exceptions."*) every table in the repo is built on
`@tanstack/react-table`: the shared house table `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` (used by Devices, Repos, Storages, Communities, IPFS, and the directory view) and the one bespoke instance `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FullPathsPage.tsx` — `FullPathsPage` (its own `useReactTable` for the streamed flat walk). ## 7. The hardware fingerprint — identify the physical machine (no networking)

The fingerprint is the set of **hardware facts** that make one physical computer recognizable and
distinct from another. LFB collects it **entirely locally** — it must **never** phone home (the same
no-networking rule the charter puts on perceptual hashing). It is collected **once on first run**, cached
in-process, and persisted into `config.yaml → computer.hardware`; a device's own fingerprint is copied
into each storage's `devices/<device>.yaml` (§3) so *other* computers can see and disambiguate it.

**What we collect and how (macOS-first, graceful elsewhere):**

* From Node's `os`/`fs` (always available, no spawn): `platform`, `arch`, `hostname`, **`username`**
  (`os.userInfo().username`), **`home_dir`** (`os.homedir()` — the `~` directory that says *which user*),
  `cpu_cores` (`os.cpus().length`), `ram_gb` (`os.totalmem()`), `disk_total_gb` (`fs.statfsSync('/')`).
* From macOS tooling, best-effort with a short timeout (skipped/blank off darwin): `model_identifier`
  (`sysctl -n hw.model` → `Mac14,7`), `model_name` + `chip` (`system_profiler SPHardwareDataType`),
  `screen_count` (`system_profiler SPDisplaysDataType`).
* From a **built-in model table** (§8's data file): `marketing_name`, `year`, and built-in
  `screen_inches`, resolved from `model_identifier`. macOS exposes the identifier but not a clean
  marketing string, so the table supplies "MacBook Pro (14-inch, 2023)" and the 14″ screen size. An
  unknown identifier degrades gracefully (marketing_name `""`, year/screen `null`) — the raw
  `model_name` still shows.
* **`kind`** is derived: `MacBook*` → **laptop**; `Macmini`/`iMac`/`MacStudio`/`MacPro` → **desktop**;
  `server` mode (`config.server.mode`, [storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical))
  or a headless Linux host with no display → **server**.

Collecting the fingerprint locally (os/fs + guarded `sysctl`/`system_profiler`, cached, no network) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/hardware.service.ts` — `collectHardware()` / `getHardware()`, with the model→marketing/year/screen lookup in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/hardware-models.ts`. Seeding `config.yaml → computer.hardware` (and the default nice name) on first run is done in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig()`.

## 8. Auto-naming & disambiguation — combine only what differs (LOCKED)

Two problems, one algorithm. **(a) Auto-naming**: a fresh computer needs a sensible default name so the
user is never staring at `this-computer`. **(b) Disambiguation**: when the user owns two similar
machines, the UI must tell them apart *without* the user hand-labeling each.

* **Default name** = `<username>-<model-slug>` — the logged-in user plus the model, lower-kebab:
  `bryan` + `MacBook Pro` → **`bryan-macbook-pro`**. Falls back to the hostname when the model is
  unknown. This is the seed value; the user may rename it (§1).
* **Disambiguation** runs across the **whole device set** shown in the table. Group devices whose
  **base display** collides (same nice name, or same `username` + `model_name`). Within a colliding
  group, find which fingerprint attributes actually **differ** across its members and append **only
  those**, in a fixed **priority order** until each label is unique:

  1. **screen size** (`14-inch` vs `16-inch`) — the most human, checked first;
  2. **model year** (`2021` vs `2023`);
  3. **disk size** (`512 GB` vs `1 TB`);
  4. **RAM** (`16 GB` vs `32 GB`);
  5. **chip** (`M1 Pro` vs `M2 Pro`);
  6. **hostname** (last-resort tie-break, always unique enough).

  The rule the user stated verbatim: *if the hard-drive sizes are the same you don't include disk; if
  the screen sizes differ you do include screen; if the years differ but the screens match you put the
  year in the title.* Attributes that are **identical across the group are never shown** — the label
  stays as short as possible while being unique. A device with no colliding twin shows its bare nice
  name with **no** suffix.

Deriving the default name and the disambiguated display labels is pure logic in code (shared so the backend and any client agree):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/device-naming.ts` — `defaultDeviceName(hardware)` and `disambiguateDevices(devices)`. The built-in model→(marketing/year/screen) facts they lean on live in `hardware-models.ts` (§7). The human companion write-up of the attributes + the priority ladder is `~/BGit/Bryan_git/LargeFileBridge/knowledge/device_identification.md`.

## 9. Auto-seed — the registry is never empty (LOCKED)

The table must **never** be empty and the user must **never** have to add "this computer" by hand. Two
guarantees:

* **This computer is always a row.** `deviceRows()` (§6) injects the self device from
  `config.yaml → computer` + the fingerprint even before any storage exists, so the very first launch on
  a brand-new machine shows exactly one row — **this** one, tagged *This computer*.
* **The self device is written into every storage.** Whenever a storage is created, indexed, or synced,
  this computer writes its own `devices/<device>.yaml` (self-owned, §2.1). So a storage that somehow has
  a `devices/` dir with no entry for the current machine is **healed** on the next pass — the current
  device is auto-added rather than left missing.
* **"Which one is me?"** is answered by matching each `devices/*.yaml` `id` against this machine's local
  `config.yaml → computer.id` (§1). Exactly one row is the self row; the rest are peers.

Guaranteeing the self row (config-seeded) and self-writing into each storage's registry is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `deviceRows()` (config-seeded self) and `writeSelfDevice(storageRoot)` (called from `storage.service.ts` on storage create/index/sync).

## 10. Synchronization — the identity travels, the machine facts follow

The whole point is that this data **stays synchronized across the user's computers**. The mechanism is
already the SDL's Git-backed travel ([syncable_data_location.mdx](./syncable_data_location.mdx)), driven
by the [Git backbone engine](./git_sync.mdx) (`git fetch` → automatic `merge` → `commit` → `push` each
pass); [storage_personal.mdx](./storage_personal.mdx) is the canonical two-computer walkthrough of a
device registering itself and discovering the user's other machines:

* A device's **id, nice name, fingerprint, schedule, and graft** live in its self-owned
  `devices/<device>.yaml`, which is committed to the storage's Git repo and (optionally) mirrored to
  Google Drive / Dropbox. When any computer pulls the storage, it gets the **full up-to-date array of
  devices** — every other machine's identity and hardware included.
* Because writes are **self-owned** (§2.1), two computers editing the storage at once touch **different
  files** and never collide; a stale claim is reconciled by the same revision rules as the rest of the
  SDL ([sync_list.mdx §5.3](./sync_list.mdx#53-conflict-resolution)).
* The machine-local half — `config.yaml → computer` (id, label, **fingerprint**) and `peers.yaml` —
  **does not** travel (it is *this* machine's truth), but its id is the join key that stitches the local
  and travelling halves together (§5).

Net effect: rename a computer or upgrade its disk on one machine, and the next sync carries the new nice
name / fingerprint to every other computer's Devices / Peers table.

## 11. The device-registration write-back — every 10 minutes (LOCKED)

The whole registry is worthless if **this** computer's own entry never makes it into the storage's Git
repo. The user's worry, verbatim: *"I have my device information. I worry that it hasn't written it down
to my personal Git repo."* So LFB runs a **dedicated background check** whose one job is to guarantee
that write-back.

### 11.1 A dedicated 10-minute worker, separate from the 15-minute transfer

Device write-back is its **own** scheduled worker, on its **own** cadence — **every 10 minutes** — so it
is independent of the every-15-minute IPFS **Transfer** pass ([sync_process.mdx](./sync_process.mdx),
[storage_local.mdx §13](./storage_local.mdx#13-the-background-sync-process--the-transparency-contract)).
It carries the **same transparency contract** as the Transfer and Discovery workers — **installed /
on-off / last-run**, controllable from the web app — so the user can see it and **turn it on** (*"I turn
that on"*). Every 10 minutes it visits each directory storage and makes sure this computer's device info
is **present and current** in that storage's repo, writing/updating it when it is missing or stale.

### 11.2 It is NOT gated by the IPFS opt-in

Writing your **own** device identity text to your **own** configured Git repo has **no outward
footprint** — it is the "understand", not the "pin", side of [sync_process.mdx §1](./sync_process.mdx#1-lfb-knows-every-repo--there-is-no-lfb-managed-gate).
So the write-back runs **whenever the storage's Git backbone is on**, *decoupled* from the per-storage
`synced` (IPFS pinning) opt-in. A Personal storage the user has **not** opted into IPFS pinning still
gets its device registry written and pushed — exactly the case the user was worried about. Only the
**byte work** (add/pin/fetch) stays gated by `synced`.

### 11.3 The strict order — always pull before modifying, always push after (LOCKED)

Before it **ever modifies** the repo, the check runs the Git backbone cycle
([git_sync.mdx §6](./git_sync.mdx#6-the-sync-cycle--where-git-runs-inside-the-pass-locked-order)) in a
fixed order, matching the user's requirement verbatim:

1. **`git fetch` → `git merge` (automatic)** — pull the repo down **first**, so another computer's edits
   are merged in before we touch anything. **This runs every 10 minutes even when we have nothing to
   change**, in case the device files were updated on another computer.
2. **Write / update this device's own `devices/<self>.yaml`** (self-owned, §2.1) — the only modification.
3. **`git add` (self-owned) → `git commit` → `git push`** — send this device's entry up to the server
   side, with the fetch-merge-push retry on a non-fast-forward reject. An empty diff commits/pushes
   nothing (idempotent).

A storage with **no** Git backbone still gets its **local** device file refreshed each run, so it is
ready to travel the instant the user turns Git on.

### 11.4 It never races the Transfer pass

Because the 10-minute device worker and the 15-minute Transfer pass share this one backend process and
their cadences coincide, both are routed through a **per-storage Git lock**, so at most one pass ever
runs `git` on a given repo's working copy at a time (concurrent `add`/`commit`/`push` in one working tree
corrupts the index). Different storages still run concurrently.

The dedicated every-10-minute device worker — its transparency contract (installed / on-off / last-run) and its `device_process` config block — is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — `AppConfigSchema.device_process` (default `interval_minutes: 10`, `label: com.largefilebridge.device`); driven by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/schedule.service.ts` — `workerState("device")` / `control("device", …)` (install/enable/disable the launchd plist), surfaced on the Scans page by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/sync/SyncPage.tsx` — `SyncPage` (the "Device registration" `WorkerCard`).

The pass itself — for every Git-backed storage, pull → write self device → commit → push, decoupled from the IPFS opt-in and serialized per storage — is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` — `syncDeviceRegistrations()` (the whole pass, run by the `device` worker via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/internal/internal.router.ts` — `POST /api/internal/run/device`) and `ensureDeviceRegistered(id)` (one storage's pull → `writeSelfDevice(root)` → commit/push under `withStorageGitLock`). The same decoupling is applied to the regular Transfer pass so it, too, writes device info even when a storage is not IPFS-opted-in: `syncStorageUnit(id)` now runs the Git pull + `writeSelfDevice` + commit/push regardless of `synced`, and gates only the byte reconcile behind it.

## 12. Acceptance criteria

1. Every computer has a **minted, stable device id (UUID), distinct from its IPFS PeerID**, plus a
   **user-set nice name** (auto-seeded from the fingerprint) and a **hardware fingerprint**; the id and
   name are **editable from the web app** (§1). The id lives in `config.yaml → computer` **under `~/T`**
   and is copied into **each storage's** device file — present both locally and in the synced storages.
2. Each device records **itself** as `<SDL>/.lfbridge/devices/<device>.yaml`; the registry **travels
   with the storage**, so every computer sees the full set of devices, and a device file is
   **self-owned** — a computer writes only its own and treats others as claims (§2).
3. A device file carries the **id + name + owner**, this device's **sync schedule** for this storage
   (cadence + optional time windows), and the **graft** (§3).
4. The **graft** maps each mapped source directory (keyed in `mapped_dirs.yaml`) to **this computer's
   absolute local path**, with `wanted:false` / `local_path:null` meaning known-but-absent here; the
   sync pass **fetches marked CIDs to the grafted local path**, never to the writer's path (§4).
5. The travelling **device registry** and the machine-local **`peers.yaml`** are kept distinct and
   cross-reference by **device id** (§5).
6. The **Devices / Peers** page (`/devices`) renders a **full-width TanStack table** (search + filter +
   sort icons, header-click sort, 500/page, full-page-height) with **fixed-width** columns — **Device,
   Kind, Owner, IPFS Peer ID, Last seen** — and marks the self row **"This computer"** (§6.1). The
   Model/Year/Screen/Disk/RAM columns are **removed**; those facts are rolled into the **Device
   descriptor** line (§6.2). **Owner** shows the **logged-in OS username** (§6.3). The **IPFS Peer ID**
   is shown **first-8 `…` last-8** and **copies in full on click** (§6.4). **Clicking a row** opens the
   **View-one-device** page (`/device/$deviceId`) with the full identity, the complete hardware
   fingerprint, and per-device actions (§6.5). Every table in the repo is a **TanStack** table (§6.7).
7. Every device records a **hardware fingerprint** collected **entirely locally, no networking** — Mac
   model/year, screen size, disk, RAM, chip, and logged-in **username / home_dir** — persisted to
   `config.yaml → computer.hardware` and copied into each `devices/<device>.yaml` (§7).
8. A fresh computer **auto-names** itself `<username>-<model-slug>`; similar machines are
   **disambiguated** by appending **only the attributes that differ**, in the fixed priority order
   screen → year → disk → RAM → chip → hostname (§8).
9. The registry is **never empty**: `deviceRows()` always injects **this computer** (config-seeded) and
   the app auto-writes the self device into every storage; "which one is me?" resolves by matching the
   device id against `config.yaml → computer.id` (§9).
10. The device identity + fingerprint **synchronize** across the user's computers via the SDL's
    Git-backed travel; self-owned writes never collide (§10).
11. A **dedicated, every-10-minute device-registration worker** (its own installed / on-off / last-run
    transparency contract, turnable on from the web app) guarantees this computer's device info is
    **written and pushed** to each Git-backed storage's repo. It is **decoupled from the IPFS `synced`
    opt-in** (writing your own identity to your own repo has no outward footprint), runs the strict
    **fetch → auto-merge → write self device → commit → push** order — **always pulling first, even with
    nothing to change**, so another computer's edits land here — and is **serialized per storage** so it
    never races the 15-minute Transfer pass on the same repo (§11).
