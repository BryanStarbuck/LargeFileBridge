# LargeFileBridge — Local Storage, Config & Pin State

This is the spec for how **LargeFileBridge (LFB)** stores things on disk: configuration, per-user
settings, the set of things it keeps pinned across your computers, and logs. There is **no
database** — runtime state we control lives in small, hierarchical **YAML** files (config/settings)
and machine-written **manifest** files (large-file → IPFS CID lists), all under a single **state
root** outside the repo.

LFB pins *large files* across a user's own computers over **IPFS**. The idea (see the repo
`CLAUDE.md` and `.gitignore`): **Git tracks the code plus a small manifest of what should exist; the
IPFS layer moves the actual bytes.** Big binary files are git-ignored so pushes stay small; LFB pins
them on every one of your computers, so losing a laptop never loses a file.

It applies in two deployment modes, and is written so the **same code path works in both**:

* **Local mode** — you run LFB on your own machine (localhost). Usually one user. This is the first
  and primary target.
* **Server mode** — LFB runs on a shared host and several allow-listed people log in. The design does
  not assume localhost-only.

The only things that change between the two are *where* the state root resolves and *who* is logged
in. The layout, the file formats, and the read/write code are identical — a state dir made on a
laptop and one made on a server have the same shape.

> **Learned from the sister web apps.** The reference internal app (`~/BGit/all/app/`,
> `pm/local_storage.mdx`) and EmailDeliveryHero (`~/BGit/Bryan_git/EmailDeliveryHero/pm/storage.mdx`)
> both specify the same core idea: an app-level `config.yaml` plus **per-logged-in-user** config
> under `users/<email>/config.yaml`, all under one state root, fronted by a typed store layer with
> atomic writes. LFB follows that model, adapted to this app's state root
> (`~/T/_large_files_bridge/`), its OpenAuthFederated auth (with **no anonymous account** — see §4),
> and its two **computer-wide pin units** (§5–§8), which are the piece unique to LFB.

Tech stack (repo `CLAUDE.md`): **TypeScript on Node**. The store layer, scanner, and background pin
worker are all TypeScript-on-Node — never raw shell.

## 0. This is *Local* Storage — one of a family of storages

LFBridge has **several kinds of storage**, and this spec is only the first — **Local Storage**.

* **Local Storage** (this spec, `~/T/_large_files_bridge/`) is **this computer's own settings,
  configuration, and pin bookkeeping**. It is effectively **LFBridge's database replacement** — it holds
  nothing that travels between computers by itself; it is where *this machine* keeps app config, the
  per-user config, the auth session, `peers.yaml`, logs, and the local index/manifests it needs to do its
  job. If you wiped it and re-scanned, it would rebuild.

* The **other storages are directory hierarchies that hold the actual shared large files** and travel
  between your computers over IPFS (and often *also* have a clone in **Google Drive** or **Dropbox**).
  Each of those lives in **its own root directory** with a **`storage.yaml`** at the top declaring its
  **name** and **type**, plus a hidden per-directory tracking area. A person is normally involved with
  **more than one** at a time — several communities, one or more companies, and their own personal files:

  | Storage | Type | Root directory | Spec |
  | --- | --- | --- | --- |
  | **Local** | `local` | `~/T/_large_files_bridge/` (settings/config; no `storage.yaml`) | **this file** — [storage_local.mdx](./storage_local.mdx) |
  | **Repo** | `repo` | any Git repo, e.g. `~/BGit/xyz/` (tracking in `~/BGit/xyz/.lfbridge/`) | [storage_repo.mdx](./storage_repo.mdx) |
  | **Personal** | `personal` | `~/BGit/Bryan_git/personal_large_files_bridge/` | [storage_personal.mdx](./storage_personal.mdx) |
  | **Company** | `company` | e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` (one per company) | [storage_company.mdx](./storage_company.mdx) |
  | **Community** | `community` | one per community the user helps/downloads/shares | [storage_community.mdx](./storage_community.mdx) |

* The umbrella that ties them together — the **Storages** left-bar tab, the shared **`storage.yaml`**
  schema, and the per-file **media-analysis** outputs (transcription / description / visuals-by-time) — is
  [storages.mdx](./storages.mdx).

**Division of labor.** Local Storage (this file) owns *settings + this machine's bookkeeping*. The
directory-based storages own *the shared files themselves* and their per-file tracking. Where this spec
still describes per-repo pin units under the state root (§5–§9), read it as the **local index** of what
lives in each directory-based storage; the **authoritative** per-storage tracking is the `storage.yaml` +
hidden `.lfbridge/` in that storage's own root ([storage_repo.mdx](./storage_repo.mdx)).

## 1. The state root

Everything LFB persists lives under a single **state root directory**:

```
~/T/_large_files_bridge/
```

* Default `~/T/_large_files_bridge/`, overridable with the **`LFB_STATE_DIR`** environment variable
  (server mode points it at a service-owned dir, e.g. `/var/lib/large_files_bridge`).
* The leading `~/T/` matches the convention the sibling apps use (`~/T/_voice`, `~/T/_mai`,
  `~/T/_emaildeliveryhero`) — one folder per app under `~/T`.
* The directory is created on first use (`mkdirSync(dir, { recursive: true })`, best-effort) — the
  app never crashes because the folder is missing.
* A separate **`LFB_LOG_DIR`** may point the two log files (§12) elsewhere without moving the rest of
  the state (tests use this so runs never pollute the real `error.err`).

### Resolution rule

```
stateDir =
  process.env.LFB_STATE_DIR                              // explicit override (server mode)
  ?? join(os.homedir(), "T", "_large_files_bridge")      // default (local mode)
  ?? "/tmp/_large_files_bridge"                          // last resort if homedir() throws
```

Nothing under the state root is ever committed to the repo — it is the boundary between code and
runtime data. (The one thing that *is* committed lives inside each **watched repo**, not here: the
portable in-repo manifest of §9.)

Resolving the state root and the `LFB_STATE_DIR` / `LFB_LOG_DIR` overrides is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/state-dir.ts` — `resolveStateDir` function

## 2. Directory layout

```
~/T/_large_files_bridge/
├── config.yaml                 # APP-LEVEL, COMPUTER-WIDE config — install/machine settings (§3)
├── users/                      # PER-LOGGED-IN-USER tree, keyed by verified email (§4)
│   └── <email>/
│       ├── config.yaml         #   that user's personal settings (theme, prefs, last view)
│       └── sessions/           #   OpenAuthFederated FileSessionStore — one JSON per session (§10)
├── pin/                        # THE PIN STATE — computer-wide, OUTSIDE users/ (§5). Two kinds:
│   ├── computer/               #   (a) the whole-computer pin unit — everything OUTSIDE repos (§8)
│   │   ├── config.yaml         #       this unit's settings: on/off + roots + excludes + the mask
│   │   ├── manifest.yaml       #       tracked large files → CID + pin state (the "IPFS file list")
│   │   └── status.yaml         #       machine-written last-scan / last-pin operational cache
│   └── r/                      #   (b) one folder PER DISCOVERED GIT REPO — the PRIMARY unit (§6)
│       └── <repo_name>/
│           ├── config.yaml     #       this repo's settings: pinned on/off + all per-repo settings
│           ├── manifest.yaml   #       local cache of the repo's committed manifest (§9) + pin state
│           └── status.yaml     #       machine-written last-scan / last-pin operational cache
├── peers.yaml                  # the user's OTHER computers that pin alongside this one (§11)
├── .auth_session_secret        # stable HS256 signing secret (auto-created; never in .env) (§10)
├── log.log                     # INFO/DEBUG/VERBOSE — the normal event trail (§12)
└── error.err                   # WARN/ERROR/FATAL — the fault trail (§12)
```

There are three tiers, and one of them is unusual on purpose:

* **App-level, computer-wide** (`config.yaml`, `peers.yaml`) — one copy for the whole install; the
  global machine settings, the IPFS connection, the background-pin toggle, the allow-list display,
  and this computer's identity. In local mode that is "this machine"; in server mode "this server".
* **User-level** (`users/<email>/…`) — one folder per person who has ever logged in, created lazily
  on first write. Personal, cosmetic preferences + that user's auth sessions.
* **Pin state** (`pin/computer/`, `pin/r/<repo_name>/`) — **deliberately NOT under `users/`.**
  What gets pinned is a property of **the computer and its filesystem**, not of whoever happens to be
  logged in. A repo on disk is the same repo no matter who signs into the web app. So pin state is
  **computer-wide** and lives at the top level of the state root. (Which *users* may participate in a
  given repo is a field *inside* that repo's config — see §6 "some need users, some don't" — but the
  pin unit itself is owned by the machine.)

### Why YAML for config + manifests

The product rule is **"keep human-facing data in YAML unless the data type makes that a bad fit."**

| Need | Goes in | Why |
| --- | --- | --- |
| Settings, toggles, small key/values | `config.yaml` (app / per-user / per-pin-unit) | Human-readable, hot-editable, **hierarchical**. |
| The list of the user's other computers | `peers.yaml` | A short sequence an operator reads and edits. |
| The **large-file manifest** — path → IPFS CID + size + pin state, read by humans and the app alike | `manifest.yaml` per pin unit | An operator can open any unit and see exactly which big files are tracked and where they are pinned. Small text; travels well. |
| Machine-written scan/pin cache (high-churn, never hand-edited) | `status.yaml` per pin unit | App-owned; rewritten atomically after each scan/pin. |
| Secrets (Google OAuth client id/secret) | `~/.credentials/large_files_bridge.json` **or** env | Never in the state store, never in the repo (§10). |
| Append-only logs | `log.log` / `error.err` | Flat append streams; appending to YAML is wrong. |

Everything here is **default-everything**: LFB boots on sensible defaults and writes a file only when
a value is set, so a fresh install runs with no files present.

## 3. App-level `config.yaml` — computer-wide settings (hierarchical)

`~/T/_large_files_bridge/config.yaml` is the install-wide, machine-owned settings file. Every value
is nested under a top-level group so the tree stays **hierarchical** and one concern is one block.

```yaml
# ~/T/_large_files_bridge/config.yaml — app-level, computer-wide settings
schema_version: 1
updated_at: 2026-07-02T14:32:15.123Z

# Where the web app is reachable (localhost first; the same shape on a server)
server:
  frontend_port: 2222        # web app (browser-facing page server) — ALWAYS defaults to :2222.
                             # Resolved on boot: free → take it; held by our OWN stale instance →
                             # kill & reclaim; held by a FOREIGN process → increment (+1) and report.
  backend_port: 8787         # Node/TS REST API under /api
  mode: local                # local | server

# This computer's identity in the user's personal IPFS swarm (§11)
computer:
  id: 4f9c-…                 # stable UUID minted on first boot
  label: bryan-macbook       # human name shown in the UI and in peers.yaml
  ipfs_peer_id: 12D3Koo…     # the local IPFS node's PeerID (filled once IPFS is reachable)

# How LFB talks to IPFS (a local Kubo/IPFS daemon by default)
ipfs:
  api_addr: /ip4/127.0.0.1/tcp/5001     # Kubo HTTP API multiaddr
  gateway_addr: /ip4/127.0.0.1/tcp/8080 # local gateway for reads
  pin_mode: recursive                    # how added files are pinned locally
  auto_start_daemon: false               # if true, LFB tries to start the daemon when absent
  reprovide: true                        # keep announcing our CIDs to the DHT so peers can find them

# ── The background pin process (launchd) — see §13 for the transparency contract ──
pin_process:
  installed: false           # is the launchd plist installed on this machine? (separate from on/off)
  enabled: false             # is it currently turned ON? (user toggles this from the web app)
  interval_minutes: 15       # cadence — runs every 15 minutes
  label: com.largefilebridge.pin   # launchd job label
  large_file_min_bytes: 5242880     # 5 MiB — files at/above this are "large" and get bridged

# ── Access — DISPLAY of the OpenAuthFederated-enforced allow-list (see §4) ──
access:
  allowed_emails:            # only these Google emails may sign in; enforced on the OAF side
    - bryan@act3ai.com
  # NOTE: there is NO anonymous / default account. Enforcement is OAF's; this block is the mirror
  # the Settings UI displays. Editing it here does not by itself grant access.

# ── The filesystem scanner (§7) — where it looks for repos + what it ignores everywhere ──
scanner:
  roots:                     # top-level dirs the scanner walks to discover repos
    - ~/BGit
    - ~/Documents
  ignore_globs:              # never descend into these (applies to repo discovery AND computer pin)
    - "**/node_modules/**"
    - "**/.git/**"
    - "**/.Trash/**"
  follow_symlinks: false

# ── Defaults applied to every user until their own users/<email>/config.yaml overrides them ──
defaults:
  theme: system              # system | light | dark
  density: comfortable       # comfortable | compact
```

Secrets never go here — Google client credentials come from env or the out-of-repo credentials file
(§10); the auth signing secret is its own file. `config.yaml` holds only non-secret ports, flags, and
settings.

Reading and updating the app-level `config.yaml` (with first-run default scanner roots) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig` / `updateAppConfig` functions

## 4. Per-logged-in-user storage — the per-user requirement

App-level `config.yaml` holds shared, machine-owned settings; each person's **personal** preferences
live in their own file:

```
~/T/_large_files_bridge/users/<email>/config.yaml
```

* `<email>` is the user's **verified** email from the OpenAuthFederated JWT (their Google identity).
* **There is no `default` / anonymous key.** Unlike some sibling apps that fall back to a `default`
  folder when logged out, LFB (per the repo `CLAUDE.md`) has **no no-login access path** — every
  session belongs to an authenticated, allow-listed user, so **every** per-user folder key is a real
  verified email. A request that is not authenticated has no user folder because it never reaches
  authenticated routes at all.
* In **server mode** this cleanly separates every teammate's settings on one shared install. In
  **local mode** there is usually one user, but the same folder is used, so the shape is identical.
* The folder is created **lazily** on the first write for that user. Reading a user who has no folder
  yet returns the `defaults` block from the app `config.yaml` — never an error, never a first-run
  write.

### `users/<email>/config.yaml` shape

```yaml
# ~/T/_large_files_bridge/users/bryan@act3ai.com/config.yaml
schema_version: 1
updated_at: 2026-07-02T14:32:15.123Z

# Appearance (per-user, cosmetic)
ui:
  theme: dark                # overrides defaults.theme
  density: compact
  last_route: /scans         # restore the user where they left off

# Remembered per-table sort/filter, keyed by table id, saved on every change and restored on load.
# The sort is an ORDERED list (multi-level): index 0 = primary, 1 = secondary, 2 = tertiary.
# The default (Bookmarked-first) is applied in code and only WRITTEN here once the user changes it.
# Canonical shape + behavior: pm/tables.mdx §4.
tables:
  views:
    repos:
      sort:
        - { col: bookmarked, dir: desc }   # primary — favorites float to the top
        - { col: name,       dir: asc  }   # secondary
      filters:
        bookmarked_only: false             # the promoted "Bookmarked only" checkbox (tables.mdx §2)
        pinned: { op: eq, value: true }
    files:                                 # the Full-paths files table (full_paths.mdx)
      sort:
        - { col: bookmarked, dir: desc }
        - { col: size,       dir: desc }
      filters: { bookmarked_only: true }
    communities:                           # the Communities table (communities.mdx)
      sort:
        - { col: bookmarked, dir: desc }
        - { col: name,       dir: asc  }
      filters: { bookmarked_only: false, mode: recommended }

# File System page view state — restore the column browser exactly where the user left it (open
# column chain + the 1+ selected entries + the header filter checkboxes). Debounced write, restored
# on return, stale-path-tolerant. Full spec: directories.mdx §1.3 / §1.4.
file_system:
  columns:                                       # open column chain, root → deepest, left to right
    - /Users/bryan
    - /Users/bryan/BGit/Bryan_git/LargeFileBridge/pm
  selection:                                     # the 1+ entries highlighted when the user left
    - /Users/bryan/BGit/Bryan_git/LargeFileBridge/pm/directories.mdx
  filters:                                       # the §1.4 header checkboxes (all default ON)
    only_large: true                             # hide all but big files + the dirs that lead to them
    videos: true                                 # show videos even under the threshold
    images: true                                 # show images even under the threshold
    audio: true                                  # show audio even under the threshold (no C badge)

# Last 5 web sessions, newest first — usage windows measured from page renders, NOT the auth
# session (§10). Drives the "stale return" auto-pin: starting a new session on a computer that
# has not pinned in > 48h automatically kicks off a background pin. Full spec: sessions.mdx.
sessions:
  - started_at:       2026-07-05T15:00:00.000Z
    last_activity_at: 2026-07-05T15:42:10.000Z
    ended_at:         null                       # OPEN — inside the 4-hour idle window
  - started_at:       2026-07-04T09:03:00.000Z
    last_activity_at: 2026-07-04T11:20:00.000Z
    ended_at:         2026-07-04T11:20:00.000Z   # closed; end == last render, not the 4h timer's time
```

The `sessions:` block is a small usage history — **not** the auth session (§10). A "web session" is a
window of page-render activity; it starts on the first render after a quiet spell and is discontinued
once 4 hours pass with no render (back-dated to the last render). Its purpose is freshness: when a new
session starts on a machine that has not pinned in **more than 48 hours**, LFB **automatically
kicks off a (non-blocking) pin** so the user is not working against stale files. The full
model — the 4-hour idle timer, the retroactive end time, the five-session cap, and the auto-pin
trigger — lives in [sessions.mdx](./sessions.mdx).

### Email → directory policy

* **Lowercase** the email.
* Store it **verbatim as the folder name** (`bryan@act3ai.com`) so the tree is human-browsable.
* **Sanitize defensively:** reject/strip any `/`, `\`, `..`, or NUL before using the value as a path
  segment. It comes from a trusted JWT, but we never build a path from an unsanitized string.

## 5. The two kinds of pin unit — the core model

This is the piece unique to LFB. There are exactly **two kinds of thing LFB keeps pinned**, and
both live under `pin/` (computer-wide, outside `users/`):

1. **Repos** — every **Git repository** discovered on the filesystem. A repo is mapped to a
   sub-hierarchy of the filesystem (its working tree). It is the **PRIMARY** pin unit: most of a
   user's large files live inside repos, Git already carries the repo's code + committed manifest
   between computers, and IPFS carries the git-ignored bytes. One folder per repo at
   `pin/r/<repo_name>/` (§6).

2. **The computer** — **everything else on the filesystem, OUTSIDE every registered repo.** The
   whole-computer pin unit "**masks out**" the sub-hierarchies that belong to repos and worries only
   about large files that live *outside* any repo. One folder at `pin/computer/` (§8).

```
The filesystem, as LFB sees it:

  ~/BGit/act3/WebApp/  ─┐
  ~/BGit/Bryan_git/…/  ─┤  registered repos  → each is a pin/r/<repo_name>/ unit  (PRIMARY)
  ~/Documents/proj/    ─┘

  everything else on disk (photos in ~/Pictures, ~/Movies, loose files …)
                          → the pin/computer/ unit, with the repo paths above MASKED OUT
```

The **repo is primary**; the **computer is the remainder**. The masking rule (formalized in §8): a
path belongs to the computer unit **only if it is not inside any registered repo's working tree**.
Discover a new repo and its subtree instantly leaves the computer unit's scope and becomes its own
`pin/r/<repo_name>/` unit.

Every pin unit — repo or computer — has the same three files:

* `config.yaml` — **is this unit pinned on or off**, plus all of that unit's settings.
* `manifest.yaml` — the tracked large files, each mapped to its IPFS **CID**, size, and where it is
  pinned. This is the "list of IPFS files" the charter describes.
* `status.yaml` — machine-written operational cache: last scan time, last pin time, counts, errors.

## 6. `pin/r/<repo_name>/` — the per-repo pin unit (PRIMARY)

One folder per discovered Git repo. The repo is identified by its working-tree path and its Git
remote; the **folder name** is the repo name, sanitized.

### 6.1 Keying / folder name

* `<repo_name>` = the repository's directory name, lowercased, with any char outside `[A-Za-z0-9._-]`
  mapped to `_` (same sanitize rule as the email folder key, §4). Chosen because it is
  operator-facing — a human finds `pin/r/largefilebridge/` instantly.
* **Collisions** (two repos with the same directory name in different locations) are disambiguated by
  appending `-2`, `-3`, … in discovery order; the authoritative identity is the `path` + `remote`
  recorded inside `config.yaml`, not the folder name. `status.yaml` records the resolved folder name
  so re-scans stay stable.

### 6.2 `pin/r/<repo_name>/config.yaml`

```yaml
# ~/T/_large_files_bridge/pin/r/largefilebridge/config.yaml
schema_version: 1
updated_at: 2026-07-02T14:32:15.123Z

# Identity (written by the scanner; the source of truth for what this unit is)
repo:
  name: largefilebridge
  path: ~/BGit/Bryan_git/LargeFileBridge      # working-tree root on THIS computer
  remote: https://github.com/BryanStarbuck/LargeFileBridge.git

# THE IPFS-pinning opt-in for this repo (NOT a "managed vs. unmanaged" gate — LFB knows every repo
# automatically; this flag only decides whether LFB adds/pins/fetches its bytes). See pin_process.mdx §1.
pinned: true                 # false = known & visited every pass, but LFB does not pin/add/fetch its files yet

# Which large files this repo bridges. Defaults mirror the repo's own .gitignore so LFB bridges
# exactly the big files Git is ignoring; extra include/exclude globs refine it.
large_files:
  follow_gitignore: true     # treat git-ignored big files as the bridge set (the .gitignore IS the list)
  min_bytes: 5242880         # override app-level large_file_min_bytes for this repo (optional)
  include_globs: []          # force-bridge these even if Git tracks them
  exclude_globs: []          # never bridge these even if git-ignored (e.g. build caches)

# Pinning / peers policy for this repo
pin:
  pin_locally: true          # this computer pins the repo's large files
  fetch_missing: true        # this computer fetches (from peers) any manifest CID it lacks
  publish_manifest: true     # write the committed in-repo manifest (§9) so Git carries it to peers

# ── "Some repos need users, some don't." Which allow-listed people participate in THIS repo. ──
access:
  shared: false              # false = private to the owner's own computers (no extra users needed)
  participants: []           # when shared: the allow-listed emails whose computers also pin this repo
                             # e.g. [bryan@act3ai.com, family.member@example.com]
```

* `pinned` is the per-repo **IPFS-pinning opt-in**, not a visibility gate. LFB **knows and reconciles
  every repo every pass** regardless ([pin_process.mdx §1](./pin_process.mdx#1-lfb-knows-every-repo--there-is-no-lfb-managed-gate-locked));
  a repo's bytes are **bridged (added/pinned/fetched)** only when **it** is `pinned: true`. A repo the
  user never wants pinned stays known-but-not-pinned (`pinned: false`), so a re-scan keeps understanding
  it without ever pinning it. A manual **Pin now** is the explicit action that flips it on (§13,
  pin_process.mdx §5).
* **"Some need users, some won't."** `access.shared: false` is the common case — the repo is just
  *your* repo across *your* computers; no user list is required. Set `shared: true` and list
  `participants` when a repo is shared with a family member or teammate; only those allow-listed
  emails' computers pin it. This is the per-repo realization of the app-wide allow-list (§4).

Registering a discovered repo as a `pin/r/<repo_name>/` unit (validating it is a git working tree, sanitizing/disambiguating the folder name, seeding `pinned: false`) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `registerRepo` function

### 6.3 `pin/r/<repo_name>/manifest.yaml`

The local cache of the repo's large-file manifest — see §9 for the full manifest shape and how it is
**also committed into the repo** so Git carries it between computers. `status.yaml` holds last-scan /

## 7. The scanner — discovering repos and the mask

A backend scanner (TypeScript-on-Node) periodically walks the `scanner.roots` (§3), honoring
`scanner.ignore_globs` and `follow_symlinks`:

* **Repo discovery.** Any directory containing a `.git` folder (and, in server mode, that resolves to
  a real Git working tree) is registered as a repo pin unit. On first discovery LFB creates
  `pin/r/<repo_name>/` with `pinned` defaulting per policy (default **off** until the user opts in,
  so nothing is bridged without consent), writes the `repo:` identity block, and reads the repo's
  `.gitignore` to seed `large_files`.
* **Stability.** A repo already registered keeps its folder, its `pinned` flag, and all its settings
  across re-scans — the scanner reconciles, it never clobbers user choices. A repo that has vanished
  from disk is marked `missing` in its `status.yaml` (its folder and settings are retained so a
  temporarily-unmounted drive doesn't lose configuration), not deleted.
* **The mask.** After discovery, the set of registered repo working-tree paths is the **mask** for
  the computer unit (§8): the computer scan walks the same roots but **prunes every registered repo
  subtree**, so a file is attributed to exactly one unit — never both.

The scanner runs on demand (a "Rescan" action in the web app) and as part of the background pin
process (§13).

## 9. The manifest & IPFS pin model

A **manifest** is the "list of IPFS files" at the heart of LFB: for one pin unit, it maps each
tracked large file's **relative path** to its **IPFS CID** plus metadata and pin state. Git carries a
repo's manifest between computers; IPFS carries the bytes.

### 9.1 Manifest shape (per pin unit)

```yaml
# manifest.yaml — one pin unit's tracked large files
schema_version: 1
unit: repo                    # repo | computer
generated_at: 2026-07-02T14:32:15.123Z
files:
  - path: large/trailer_v3.mp4        # relative to the unit root (repo working tree, or a computer root)
    cid: bafybeigd…                   # the IPFS content id (content-addressed — changes when bytes change)
    size: 734003200                   # bytes
    modified_at: 2026-07-01T09:14:02Z # source file mtime when hashed
    sha256: 9f2c…                     # integrity check independent of the CID
    pinned_by:                        # which of the user's computers have pinned this CID
      - bryan-macbook
      - bryan-studio
  - path: assets/hero_8k.png
    cid: bafybeih2…
    size: 41231884
    modified_at: 2026-07-01T08:02:11Z
    sha256: 1a77…
    pinned_by: [bryan-studio]
```

* **CID is content-addressed.** Editing a file changes its bytes and therefore its CID; the manifest
  entry is updated and the old CID is unpinned (subject to retention). This is how "up-to-date" works
  across computers.
* **`pinned_by`** is the redundancy record — if a file is pinned by ≥1 other computer, losing this
  laptop loses nothing. The web app surfaces any file pinned by only one computer as at-risk. It is a
  **cache of claims, not truth**: only *this* computer's own entry is locally verifiable, and it is
  verified/refreshed against the real IPFS pinset on every pin (§9.5).

### 9.2 Repo manifests travel via Git (the elegant part)

For a repo unit, the manifest is **also written into the repo working tree** as a committed file:

```
<repo>/.lfbridge/manifest.yaml     # committed to Git — small text, travels with every push/pull
```

The big files stay git-ignored (`.gitignore`); the tiny manifest is committed. When another computer
pulls the repo, LFB reads `<repo>/.lfbridge/manifest.yaml`, sees which CIDs it lacks, and fetches +
pins them from IPFS. The state-root `pin/r/<repo_name>/manifest.yaml` is the **local cache** of that
committed file plus this computer's live pin state. (`publish_manifest` in §6.2 controls writing the
committed copy.)

The **per-file decision ledger** `<repo>/.lfbridge/decisions.yaml` travels the **same way** — committed,
union-merged, carried by Git to the user's other computers **and their teammates**
([decisions.mdx §4/§5](./decisions.mdx)). It is the **source of truth** for decisions; the machine-local
`pin/r/<repo_name>/config.yaml → decisions:` frozen `sync`/`ignore`/`undecided` map is a **reconciled
projection** of the ledger's IPFS axis onto this computer ([decisions.mdx §7](./decisions.mdx)), not an
independent store. The pin engine still reads that enum; only its *feed* changed.

### 9.3 Computer manifests travel via IPNS

The computer unit has no Git repo to carry its manifest, so LFB publishes
`pin/computer/manifest.yaml` under this computer's **IPNS** name (or a small shared record among the
user's peers). Other computers resolve it, diff against their own, and fetch missing CIDs. This keeps
loose (non-repo) large files pinned across your computers without any repository.

### 9.4 Write / retention rules

* Manifests are rewritten **atomically** (§11) after each scan/pin; they are the app's own files (a
  repo's committed copy is the portable source of truth, the state-root copy is the cache).
* Unpinning of superseded CIDs follows a retention window (keep the previous CID for a grace period so
  a mid-pin peer can still fetch it), recorded in `status.yaml`.

### 9.5 The IPFS pinset is the source of truth — `pinned_by` is a stale cache

Whether a file is **pinned** has exactly one authoritative answer: the **local IPFS pinset**, as the
CLI/daemon reports it (`ipfs pin ls`). That pinset is **primary storage** for pin state. LFB's own
record — the `pinned_by` list in every `manifest.yaml` — is a **secondary, stale-by-design cache**: a
fingerprint of what the pinset looked like the last time we looked, never the thing itself.

**There is only one pinset, and the CLI owns it.** There is no separate "LFB-pinned" versus
"CLI-pinned" state — a file `ipfs pin add`ed on the command line and a file pinned through our web app
land in the *same* pinset and are indistinguishable. That matters because the pinset changes **behind
our back**: while LFB is shut down, or between runs, the user (or another tool, or IPFS garbage
collection) freely runs `ipfs pin add` / `ipfs pin rm`, adds files, and drops pins. So the instant we
persist "this computer pins CID X" into a manifest, that claim may already be false. We therefore
**never** treat our stored pin state as truth — we **re-derive it from the pinset**.

Two rules fall out of this, and the code must honor both:

* **Read from the pinset, refresh the cache — at pin time.** A pin pass is exactly the moment
  LFB *learns from the filesystem which CIDs are really pinned and which are not*, then rewrites the
  cache to match. Each pin reads the pinset once (`ipfs pin ls`) and, for every tracked CID,
  reconciles **this computer's** membership in `pinned_by`:
  * really pinned → this computer stays in (or is added to) `pinned_by`;
  * **not** really pinned → this computer's label is **dropped** from `pinned_by` (the cache was
    stale); and if the file is still decided `sync`, the CID is **re-pinned** so reality catches up to
    intent.
  An "unchanged" manifest entry (same size, same CID) is **not assumed** to still be pinned — it is
  verified against the pinset like any other. This is the local half of
  [sync_list §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed): a `pinned_by` claim
  about *this* computer is believed only if `pin ls` actually holds the CID.

* **Write to the pinset, not just to the cache — on a pin action.** When the user chooses **Pin** in
  the web app, LFB must **actually pin at the IPFS level** (`ipfs pin add`); **Unpin** actually runs
  `ipfs pin rm`. The cache is updated **only after reading the state back** from the pinset, so the UI
  settles on ground truth, never on optimism. A pin control that merely flipped a stored flag would be
  a lie the next `pin ls` exposes.

Because pin state is a cache, the UI **must not show stale data**. Any surface that reports whether a
file is pinned resolves it from the **live pinset**, not from `pinned_by` — the IPFS page already
composes its rows from `pin ls` and only *labels* a pinned CID with what the manifests know about it
([ipfs.mdx §4](./ipfs.mdx#4-import--the-load-bearing-feature)). `pinned_by` survives solely as the
cross-computer redundancy record (§9.1): a list of *claims*, refreshed each pin, of which only this
computer's own entry is ever locally verifiable. Claims about **peer** computers cannot be checked
from here — they are carried forward from the last reconciled Sync List and re-confirmed only when
that peer republishes ([sync_list §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)).

Reading the live IPFS pinset once per pin and reconciling each `pinned_by` claim against it (re-pinning lost CIDs still decided `sync`) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinRepoFolder` function (via `setPinClaim`)

## 10. Auth session & signing secret (runtime, not settings)

Authentication state lives under the state root, kept **separate** from settings so it is never
hand-edited or exported with a config backup:

| File / dir | What it is | Source |
| --- | --- | --- |
| `.auth_session_secret` | The stable HS256 signing secret — a base64url string (48 random bytes), written `0600`, auto-created on first boot and reused so sessions survive restarts. **Never** read from `.env`. | auth module (`loadOrCreateSecret`) |
| `users/<email>/sessions/<sid>.json` | The OpenAuthFederated `FileSessionStore` records behind the rotating session cookie — one JSON per session, `0600`. Sign-out writes a `revoked: true` tombstone rather than deleting. | `@auth/backend` FileSessionStore |

Google OAuth client credentials are **not** here — they come from env or the out-of-repo
`~/.credentials/large_files_bridge.json` (override the path with **`LFB_CREDENTIALS_FILE`**; env
wins). The rule: the state store holds sessions and a signing secret (operational auth state),
**never** long-lived provider secrets. Because LFB has **no anonymous account** (§4), every session
record belongs to an allow-listed email.

**Session lifetime — "sign in once, stay signed in ~10 months".** A signed-in user must **not** be
forced to re-authenticate on a normal cadence — not overnight, not after a week, and **not after a
server rebuild/restart**. Authentication survives server, browser, and OS restarts because the two
pieces of durable state above both persist to the state root: the **signing secret**
(`.auth_session_secret`) is stable across restarts (`loadOrCreateSecret`), and the **session records**
(`users/<email>/sessions/<sid>.json`) are the authoritative, restart-surviving source of truth
(`FileSessionStore`). The frontend never depends on a stored access token — on load it re-hydrates
from the session cookie (`GET /api/v1/client`) and silently mints a fresh short-lived Bearer
(`POST /api/v1/client/sessions/:id/tokens`).

The lifetime is a **policy LFB sets explicitly** when building the OpenAuthFederated frontend
(`auth-frontend.ts → createFederatedFrontend`). It **must pass all three** of these — the library's
defaults (7-day maximum lifetime + **12-hour idle timeout**) are far too short and, left unset, force
re-login after an overnight gap or a week (the "re-login after restart" bug):

| Option | Value | Why |
| --- | --- | --- |
| `sessionTtlSeconds` | `TEN_MONTHS_SECONDS` (`10 * 30 * 24 * 60 * 60` = 300 days) | Absolute maximum session lifetime — the ceiling written into each session record's `expireAt` at creation. |
| `inactivityTimeoutSeconds` | `TEN_MONTHS_SECONDS` | Idle timeout matches the lifetime, so an idle session is bounded **only** by the 10-month ceiling, never aged out by inactivity. |
| `accessTokenTtlSeconds` | `15 * 60` (15 min) | Bearer tokens stay short-lived and refresh silently from the session cookie — keeping them short costs nothing and limits a captured token's window. |

This matches the sister apps exactly (EmailDeliveryHero, the_starbucks, all/app). Note the record's
`expireAt` ceiling is fixed **at session creation** and is not re-extended on token mint (the sliding
cookie rolls forward, but the durable ceiling does not), so **changing the policy takes full effect on
the next sign-in**; existing session records keep whatever ceiling they were created with.

Setting the 10-month session lifetime policy and wiring the on-disk signing secret + `FileSessionStore` into OpenAuthFederated is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/auth/auth-frontend.ts` — `constructAuthFrontend` function

**Credentials file — schema, read order, and the setup UI.** The file is read by
`config/credentials-file.ts` and passed to OpenAuthFederated (`auth-frontend.ts`). Its exact shape:

```json

      "clientId": "YOUR_GOOGLE_OAUTH_CLIENT_ID",
```

When the credentials can't be found on a computer, the app must **guide the user to create the

### 10.1 Why a restart can STILL force re-login (client-side causes) — and the fixes (LOCKED)

The server-side persistence above is necessary but **not sufficient**. On this install the two "usual
suspects" are already ruled out — the signing secret **is** stable on disk
(`~/T/_large_files_bridge/.auth_session_secret`, unchanged across restarts) and sessions **are** durable
files (`users/<email>/sessions/<sid>.json`, `revoked:false`, `expireAt` = created + 10 months). Yet the
user still reports *"when we restart the web server, we still have to re-login"* and *"a lot of timeouts."*
The on-disk evidence explains why: a **brand-new** `sess_*` record appears on nearly every visit, and its
`lastActiveAt` is only ~85 s after `createdAt` — i.e. the browser is **not presenting a reusable session
cookie on the next visit**, so the boot gate (`main.tsx`) shows the sign-in page and a full Google
round-trip mints a fresh session. The cause is **client-side / delivery**, not lost server state. This
subsection is the authoritative list of causes and the fixes, so the work is captured and not re-diagnosed.

The industry-standard remedy for "re-login after a restart" is exactly what LFB **already** does
server-side — a **persistent external session store** (here `FileSessionStore` on disk, the Redis/DB
equivalent) plus a **stable signing secret** and an **HttpOnly persistent cookie**
([the JWT-session guidance](https://ianlondon.github.io/posts/dont-use-jwts-for-sessions/),
[Spring persistent sessions](https://medium.com/@AlexanderObregon/handling-spring-boot-restarts-without-losing-user-sessions-ee634219c8ee)).
Because that half is done, the remaining fixes are all about **cookie delivery, token refresh, and running
the right build**:

| # | Cause (client-side) | The fix |
| --- | --- | --- |
| 1 | **"Lots of timeouts."** Access tokens are **15 min** (`accessTokenTtlSeconds`) and the SDK re-mints them **only reactively** — the axios interceptor catches a `401`, calls `reloadSession()`, and retries. So the first call after each 15-min expiry **fails once** before recovering, which reads as a timeout/stutter. | Add a **proactive refresh**: re-mint the Bearer on a timer **before** expiry (and roll the cookie's `lastActiveAt` forward so an active tab never ages out), turning the reactive `401→reload→retry` into a silent renewal. This is a small change to the shared **OpenAuthFederated** SDK (`RealAuthCore` caches the token until ~10 s before `exp` and exposes no scheduled/forced re-mint), so it must expose a `refresh()`/schedule hook the app can call. |
| 2 | **Cookie/origin binding in the split dev origin.** The session cookie is set during the OAuth callback and is `SameSite=Lax`, `Secure=false` in local mode; the SPA is served on one port (default **2222**) while the API/OAuth callback lives on another (**8787**), and `allowedRedirectOrigins()` scans a **16-port band** because "the web app may increment past a port collision" on restart. A cookie/redirect bound to a **drifting** origin is the most likely "restart → re-login" trigger. | **Pin the dev port** (no drift band in normal use) and serve the SPA and API through **one stable origin** so the session cookie is always presented back on the next launch. In production this is a single `Secure`+`SameSite` origin. |
| 3 | **Fail-closed with no migration.** `readSession()` returns signed-out when a valid cookie has **no** matching durable record, because LFB never sets **`sessionStoreMigrate`**. Any transient record miss = forced re-login. | Optionally enable **`sessionStoreMigrate`** with a bounded window so a valid signed cookie **re-hydrates** across a store hiccup instead of hard-failing (weigh against the deliberate fail-closed security posture, [security.mdx](./security.mdx)). |
| 4 | **A stale deployed/running build.** The persistence + inactivity code in `auth-frontend.ts` is **recently committed** (comment-marked as new security-audit fixes). A server that is running an **older** build predates the durable store and rotates state on restart. | **Verify the running backend contains today's `auth-frontend.ts`** (and that `resolveStateDir()` / `LFB_STATE_DIR` resolves to the **same writable path** every launch, so `loadOrCreateSecret`'s silent per-process-secret fallback — which *would* rotate on every restart — never fires). This is the **first** thing to confirm. |

The persistence, TTLs, and cookie flags live in `auth-frontend.ts` (§10 above); the client-side pieces to
change are in code:

* Proactive refresh + the reactive `401` fallback: `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/axios.ts` — the response interceptor (`reloadSession()` on `401`) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/authCore.ts` — `registerAuthBridge(getToken, load)`, backed by `~/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-react/src/core.ts` — `RealAuthCore.getToken()` (caches until ~10 s before `exp`; a scheduled re-mint must be added here).
* Cookie/redirect origin + the 16-port band: `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/auth/auth-frontend.ts` — `allowedRedirectOrigins()` and the `cookiePrefix`/`secure`/`SameSite` settings.
* Fail-closed vs. migrate: `~/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/frontend.ts` — `readSession()` (the `sessionStoreMigrate` branch).
* The silent-secret-rotation footgun: `~/BGit/Bryan_git/OpenAuthFederated/code/packages/auth-backend/src/session-store.ts` — `loadOrCreateSecret()` (returns a **per-process** secret if the state dir is unwritable — every restart then rotates it; guard by ensuring the state dir is writable and shared).

> **Bottom line:** the durable server-state fix the user asked for is **already in the spec and the code**
> (§10). What remains is **client-side**: proactive token refresh (kills the "timeouts"), a **stable
> cookie origin / pinned port** (kills "restart → re-login"), and confirming the **deployed build** is the
> current one. Do #4 (verify the build) first — it may be the whole story.

  scan error — lands in `error.err` with enough context (unit, path, CID) to act on it.

## 13. The background pin process — the transparency contract

Per the repo `CLAUDE.md`, on Mac LFB runs a **background process** that periodically pins the large
files over IPFS. It is scheduled as a **launchd plist** (the macOS cron equivalent), runs **every 15
minutes**, and runs LFB's own TypeScript-on-Node worker (never raw shell/`curl`).

> **Naming ([scan.mdx §2](./scan.mdx#naming--scheduletask-internally-scan-in-the-ui-locked)).** This
> 15-minute background job — and the 4-hour discovery job — are the app's two **scheduleTasks**; the UI
> calls what they do **"scans"** (the left-bar **Scans** page). The `pin_process` config key below
> drives it, but the concept is a *scheduleTask*. The per-file **Add-to-IPFS (pin)** decision (stored
> wire value `sync`) and the portable **Sync List** are different concepts and keep their names.

Two independent facts about it, both surfaced in the web app:

* **Installed** — is the launchd plist present on this machine? (`pin_process.installed`, §3.) The
  plist must be installed for the background process to exist at all.
* **On / off** — is it currently enabled? (`pin_process.enabled`, §3.) Separate from installed: the
  user can install it and still leave it off.

The web app (first priority per `CLAUDE.md`) provides transparency and control:

* It **displays** whether the process is installed and whether it is on or off (read from
  `config.yaml → pin_process`, reconciled with the actual launchd state).
* The user can **turn it on or off** from the web app — a toggle that writes
  `pin_process.enabled` and loads/unloads the launchd job accordingly. Installing / uninstalling the
  plist is a separate explicit action.

Reconciling installed vs on/off against the real launchd state and install/uninstall/enable/disable from the web app is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/schedule.service.ts` — `workerState` / `control` functions

> **The run mechanics live in [pin_process.mdx](./pin_process.mdx).** This section owns the
> *transparency contract* (installed vs. on/off, surfaced + controlled from the web app). What a run
> actually *does* — that a pin is always a **full pass** over every known unit, that it
> runs with **bounded concurrency** across units and files (many cores), that a manual **Pin now**
> only sets **priority** and never reduces scope, that LFB **knows every repo automatically** while the
> per-unit `pinned` flag gates only the **IPFS byte work** (opt-in), and that a pin is **never a silent
> no-op** — is defined and LOCKED there. Do not restate or contradict it here.

Each background run is a **full pass** ([pin_process.mdx §2](./pin_process.mdx)):
rescan (§7) → **visit and reconcile every known unit** (all repos + the computer unit) → for every unit
the user has **opted into IPFS** (`pinned: true`), add new/changed large files to IPFS, update the
manifest (§9), pin locally, fetch any missing CIDs from peers, and write `status.yaml`. A `pinned:
false` unit is still visited and reconciled — LFB knows it and refreshes its status — but no bytes are
added/pinned/fetched until the user opts in (pin_process.mdx §1/§5). Units are pinned **concurrently**
(bounded ≈ `cores − 2`), not one-at-a-time (pin_process.mdx §4).
It also **reads the local IPFS pinset (`ipfs pin ls`) and reconciles it against the manifests** — the
pinset is the source of truth and `pinned_by` is the stale cache we refresh from it (**§9.5**)
([scan.mdx §2.1](./scan.mdx#21-the-15-minute-scheduletask-reads-the-ipfs-pinset-import-already-pinned)):
manifest CIDs that are really pinned confirm this computer's `pinned_by` claim; manifest CIDs whose pin
was lost have this computer's stale claim **dropped** and, if still decided `sync`, are **re-pinned**;
**pinned CIDs absent from every manifest become "import" candidates** surfaced on
the [IPFS page](./ipfs.mdx#4-import--the-load-bearing-feature) — importing one is metadata-only and makes
that already-pinned file **count as Pinned**. Before the run, the
[tools preflight](./scan.mdx#42-preflight--required-tools-before-a-scan-brew-install) checks that `ipfs`
is present; unattended, a missing required tool is recorded as a `needs_tool` gap and the run is skipped.

## 15. The store layer (how code touches disk)

All filesystem access for state goes through a thin **`store` module** so callers never deal with
paths, parsing, atomic writes, or the local-vs-server difference.

* **Atomic writes.** Serialize → write to a **unique** `*.tmp` (per-write, so concurrent writers never
  share a temp path) → `rename` over the real file, so a crash mid-write never leaves a half-written
  `config.yaml` / `manifest.yaml`; a failed write cleans up the temp file, logs, and rethrows.
* **Per-file mutex.** Read-modify-write is serialized per file (matters when the web app and the
  background pin run at once); every write stamps `updated_at` / `generated_at`.
* **Defaults on absence.** Reading a missing file returns typed defaults and logs nothing — expected
  first-run absence is **not** an error and must not spam `error.err`.
* **Schema validation on read.** Files are validated against the record/config schema on load;
  malformed data is rejected loudly rather than silently trusted (silently falling back on a corrupt
  manifest would hide data loss).
* **Three scopes.** App-scoped (`config.yaml`, `peers.yaml`), user-scoped (`users/<email>/…`, created
  lazily), and unit-scoped (`pin/computer/…`, `pin/r/<repo_name>/…`). The unit-scoped opener is what
  makes pin state computer-wide (§5).

The store is isolated behind an interface so a later move to a real database is a single-module
change; services never touch the filesystem directly.

Atomic write-temp-then-rename, the per-file mutex, defaults-on-absence, and schema-validate-on-read are handled in code:

## 16. Acceptance criteria

Storage is correct when **all** of the following hold:

1. All persisted state lives under a single **state root** (`~/T/_large_files_bridge/`, override
   `LFB_STATE_DIR`), created on first use, never committed to the repo.
2. App-level, **computer-wide** settings live in a hierarchical `config.yaml` (one nested block per
   concern: `server`, `computer`, `ipfs`, `pin_process`, `access`, `scanner`, `defaults`), which is
   **default-everything** — a fresh install runs with no file present.
3. Per-logged-in-user settings live in `users/<email>/config.yaml`, keyed by the **verified**
   allow-listed email, created **lazily** on first write. There is **no `default`/anonymous key** —
   every session is authenticated (repo `CLAUDE.md`).
4. Pin state lives **outside `users/`** because it is computer-wide: `pin/computer/` (the whole
   machine, minus repos) and `pin/r/<repo_name>/` (one per discovered Git repo, the primary unit).
5. Every pin unit carries `config.yaml` with a **`pinned` on/off** flag plus its settings, a
   `manifest.yaml` (path → IPFS **CID** + size + `pinned_by`), and a machine-written `status.yaml`.
6. A repo is bridged only when its own `pin/r/<repo_name>/config.yaml → pinned` is true; the computer
   unit is bridged only when `pin/computer/config.yaml → pinned` is true — the two toggles are
   independent.
7. The scanner discovers Git repos under `scanner.roots`, registers each as `pin/r/<repo_name>/`
   (sanitized, collision-disambiguated), seeds `large_files` from the repo `.gitignore`, and **masks
   out** every registered repo subtree so the computer unit attributes each file to exactly one unit.
8. A repo's manifest is **committed into the repo** at `<repo>/.lfbridge/manifest.yaml` so Git carries
   it between computers; the computer unit's manifest is published over **IPNS**. The big bytes stay
   git-ignored and move over IPFS; the manifest is the small text list that travels.
9. **"Some repos need users, some don't":** a repo defaults to `access.shared: false` (private to the
   owner's own computers, no user list); a shared repo lists `access.participants` (allow-listed
   emails whose computers also pin it), consistent with the app-wide allow-list.
10. The background pin process is a **launchd plist** running **every 15 minutes**; `config.yaml →
    pin_process` records **installed** and **enabled** as two independent facts, both displayed in
    the web app, with the user able to toggle **on/off** and install/uninstall from the UI.
11. Auth session records (`users/<email>/sessions/`) and the signing secret (`.auth_session_secret`)
    live under the state root, separate from settings; **no** provider secret is stored there or in
    `config.yaml`.
12. Every read/write goes through the `store` layer: atomic write-temp-then-rename, per-file mutex,
    defaults on absence, schema validation on read; services never touch the filesystem directly.
13. Logs append to `log.log` (INFO/DEBUG/VERBOSE) and `error.err` (WARN/ERROR/FATAL) at the state
    root, size-capped with rotation; logging never throws; every failed pin/peer/scan/write lands in
    `error.err` with unit + path + CID context.
14. The layout, formats, and code are **identical** in local and server mode — only the resolved state
    root and the logged-in user differ; a state dir copies between machines unchanged.
15. Every persisted file/key appears in the §14 inventory — never a silent store.

## 17. Open questions (to resolve as `code/` is built)

Greenfield today (no `code/` yet), so there is no code-vs-spec drift ledger. The design decisions
worth revisiting once implementation starts:

* **Repo default `pinned`.** Spec'd **off** until the user opts in (§7), so a scan never bridges data
  without consent. Confirm this is the desired default (vs. on-with-easy-off).
* **Manifest transport.** Repos via committed `<repo>/.lfbridge/manifest.yaml` + Git (§9.2); the
  computer unit via IPNS (§9.3). Validate IPNS latency/reliability for the computer unit, or replace
  with a small peer-shared pubsub record.
* **CID keying vs. path.** Manifest is keyed by relative path with CID as the value (§9.1); a rename
  is a new path with the same CID. Confirm rename handling (keep pin, update path) matches intent.
* **Collision disambiguation for `<repo_name>`.** Folder-name `-2`/`-3` suffixes (§6.1) with identity
  in `config.yaml`. If repos are more stably keyed by remote URL, revisit the folder-name scheme.
* **Retention / unpin window.** The grace period before unpinning a superseded CID (§9.4) needs a
  concrete default.
