# LargeFileBridge — Community Storage

**Community storage** is the **on-disk result of a community subscription** — one directory-based storage
per community the user chooses to **help**. Where [Personal](./storage_personal.mdx) and
[Company](./storage_company.mdx) storages hold **your / your team's** files, a community storage holds
**other people's public files** the user opted into carrying — to **download** for themselves or to
**share / rebroadcast** and help keep secure.

Read the shared model first: the storage family, the `storage.yaml` schema, the tracking
layout, cloud clones, and media analysis are defined once in **[storages.mdx](./storages.mdx)**. This spec
covers only what is **specific to community storage**.

Siblings this builds on and does not repeat:

* [storages.mdx](./storages.mdx) — the umbrella: `storage.yaml` (§3), the tracking layout (§4), clones (§5),
  media analysis (§6).
* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — **owns the rule** that decides
  whether a root uses a `.lfbridge/` directory at all. A community storage is a **dedicated LFB file repo**
  (an **SDL**), so it has **no `.lfbridge/`**: `.lfbridge/` is a **working-repo-only** concept, and an SDL
  root itself *is* the `.lfbridge` area. (Most of a community's state lives in **Local Storage** anyway —
  §1.)
* [Communities](./communities.mdx) — the **discover/subscribe UI**: a community is a publisher of large
  public files; subscribing has two intents — **Get** (consume) or **Support** (rebroadcast/pin), against
  a per-community Block / Recommended / Full-backup budget. **A community storage is what that
  subscription materializes to on disk.**
* [storage_repo.mdx](./storage_repo.mdx) — the canonical directory-based storage shape this reuses.

## 1. Anatomy of a community storage

```
<community storage root>/          ← one per community the user helps
└── …the community's public files this user carries (downloaded / rebroadcast over IPFS)…

~/T/_large_files_bridge/repos/<repoKey>/   ← LOCAL STORAGE — this community's LFB tracking state (machine-local)
├── storage.yaml        ← name + type: community + community block (storages.mdx §3)
├── files.yaml          ← per-file fingerprint index (storages.mdx §4.1)
├── manifest.yaml       ← large-file → IPFS CID + pin state
└── analysis/           ← per-file media analysis (storages.mdx §6)
```

* **One root per community.** A user helping three communities has three such storages.
* **No `.lfbridge/` — anywhere.** A community storage is a **dedicated LFB file repo** (an **SDL**), and
  `.lfbridge/` is a **working-repo-only** concept (it exists so LFB, a *guest* in someone's project repo,
  stays in one hidden corner). An SDL root **is** the `.lfbridge` area, so nothing is nested under such a
  segment here — the rule is owned by
  [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx). In practice a community's state is
  in **Local Storage** regardless (next bullet); the root carries only the carried content.
* **Tracking state lives in Local Storage, not beside the downloaded files.** A community's `files.yaml`,
  `manifest.yaml`, `analysis/`, and its descriptor live in the machine-local state root
  `~/T/_large_files_bridge/repos/<repoKey>/` ([storage_local.mdx](./storage_local.mdx),
  [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)) — never committed into the content
  directory. Communities are download/share, so there is normally no sync-repo mirror; the tracking state
  simply stays machine-local.
* It carries **other people's** content — always an **explicit, per-community opt-in** that pins a chosen
  publisher's files; **never** a general public gateway/relay (charter: only-our-content default).
* How much of the library is carried is governed by the community's **budget mode** from
  [communities.mdx](./communities.mdx): **Block** (nothing), **Recommended** (a budget-fitting subset — the
  default), or **Full backup** (the whole library).

## 2. `storage.yaml` for a community

```yaml
# ~/T/_large_files_bridge/repos/<repoKey>/storage.yaml   (Local Storage — machine-local, not in the content dir)
name: "HolonCitizen — backups"
type: community
created: 2026-05-01T10:12:00Z
community:
  id: "holon-citizen"        # WHICH community — stable key, matches communities.mdx
  role: support              # download (consume) | share (rebroadcast) | support
  # community-scoped child properties:
  publisher: "npub1…"        # the publisher's identity (e.g. Nostr pubkey)
  budget_mode: recommended   # block | recommended | full  (communities.mdx)
clones:
  google_drive: null
  dropbox: null
```

* **`type: community`** with a **`community`** block. **`id`** names **which** community (its stable key,
  matching [communities.mdx](./communities.mdx)); **`role`** records **why** the user carries it —
  **download** (for themselves), **share** (rebroadcast to help), or **support**. Child properties scope it
  (publisher identity, budget mode).
* A user in multiple communities has **multiple** community storages, each with its own `community.id`.

---

## 3. Media analysis + cloud clones

Community storage can produce the [storages.mdx §6](./storages.mdx) **media-analysis** outputs (transcript
/ description / visuals-by-time) into its Local-Storage `analysis/` (§1,
`~/T/_large_files_bridge/repos/<repoKey>/analysis/`) for the videos it carries — useful for search
and re-use. Cloud **clones** are supported ([storages.mdx §5](./storages.mdx)) but usually left off, since
the point of a community storage is IPFS rebroadcast, not a private mirror.

---

## 4. Where it shows in the UI

* The separate top-level **[Communities](./communities.mdx)** tab remains where the user **discovers and
  subscribes** (Block / Recommended / Full backup).
* In the **[Storages](./storages.mdx#2-the-storages-left-bar-tab)** left-bar tab, the **opted-in
  communities** appear as children — the ones the user chose to **download** or **share** — so a community
  the user actively carries shows up in their map of storages alongside Personal, Companies, and the Repos
  link.

---

## 5. Acceptance criteria

1. A community subscription the user opts into materializes as a **community storage** — one directory per
   community — whose LFB tracking state lives in **Local Storage** at
   `~/T/_large_files_bridge/repos/<repoKey>/`, machine-local and never committed beside the downloaded files.
2. Its `storage.yaml` (in Local Storage) has **`name`** + **`type: community`** and a **`community`** block
   naming **which** community (`id`), the **`role`** (download / share / support), and community-scoped child
   properties (publisher, budget mode).
3. A user may help **more than one** community; each is a separate storage; the **opted-in** ones appear as
   children in the Storages tab. Discovery/subscription stays on the [Communities](./communities.mdx) tab.
4. It carries **other people's** public files by **explicit opt-in**, pinned/rebroadcast over IPFS within
   the community's budget mode — **never** a public gateway (charter). Its Local-Storage tracking state holds
   the fingerprint index, manifest, and `analysis/` per [storages.mdx §4](./storages.mdx),
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx).
