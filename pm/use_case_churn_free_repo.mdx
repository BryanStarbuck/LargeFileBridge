# LargeFileBridge — Churn-Free Repo Tour (a code repo full of media that is never described or transcribed)

This is a **chronological scenario tour**. It walks a repo that fills up with big media, is scanned week
after week, and is **never** given a transcript or an AI description — proving that a repo can be enlisted,
scanned, pinned, git-ignored, decided-on, and compressed and **still** never accrue a single LFB tracking
file in its working tree. It answers *"what actually happens, week by week, and exactly where each write
lands"* under the [data-category placement policy](./artifact_placement_policy.mdx): the **working repo
carries only content** (transcripts / AI descriptions), while **all of LFB's tracking state lives in Local
Storage** and never enters git.

The through-line, in one sentence: **a scan must never dirty a customer's git working tree — LFB's
bookkeeping stays out of the repo entirely.**

One computer, one repo, six weeks and one day:

* **`studio-tower`** — the developer's day-to-day Mac tower. Everything below happens on it (the
  cross-computer behavior is the two other tours' job; here the point is what one machine writes locally).
* **`render-farm`** — a plain **code repo** (a Node render-pipeline project) that also holds a `renders/`
  folder filling with **big ProRes `.mov` clips**. It is a normal repo — `.git/`, source, a `.gitignore` —
  that happens to accumulate large media. The developer treats it as code, not as a media library.

This spec builds on the siblings and does not repeat them:

* [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) — **the placement policy this tour
  proves**: the data-category split (Category A content vs. Category B tracking state), the
  `~/T/_large_files_bridge/repos/<repoKey>/` Local-Storage home, the re-stamp-in-place, and the additive
  company sync-repo mirror.
* [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx) — the Category-B tracking files (`repo_storage.yaml`,
  the per-file sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`) and their locked shapes.
  This tour shows them written **only** to Local Storage.
* [storage_local.mdx](./storage_local.mdx) — the **state root** (`~/T/_large_files_bridge/`) whose
  `repos/<repoKey>/` subtree is the always-home for per-repo tracking.
* [Transcribe.mdx](./Transcribe.mdx) / [ai_description.mdx](./ai_description.mdx) — the **two durable content
  artifacts** (Category A) that are the *only* things LFB ever writes into the working repo's `.lfbridge/`.
* [decisions.mdx](./decisions.mdx) — the pin / git-ignore decision ledger the developer **does** produce; it
  is Category-B tracking state, so it lives in Local Storage (and the sync repo, if enabled), not the repo.

## 0. The setup — the data-category split, and why enlist never writes tracking into the repo

A **repo storage** is enlisted the moment LFB knows about it, but under the
[placement policy](./artifact_placement_policy.mdx) enlist **writes no tracking state into the repo's working
tree** — not on enlist, not ever. The rule, stated once as a split by **data category**:

> **Category A — content.** Transcripts (`.transcription`) and AI descriptions (`.ai_description`) are stable,
> content-derived artifacts the user asked LFB to *produce*. These are written into the **working repo's
> committed `.lfbridge/`**, path-mirrored, and travel with the repo via its own git.
>
> **Category B — tracking state.** `repo_storage.yaml` (which **re-stamps `last_scan.at` and recomputes
> `counts:` on every scan**), `files.yaml`, the per-file `files/…` sidecars, `history/<device>.txt`,
> `decisions.yaml`, `manifest.yaml`, and compression records are LFB's private bookkeeping about compression,
> big files, and git-ignore decisions. **None of it ever enters the working repo.** It lives in **Local
> Storage** at **`~/T/_large_files_bridge/repos/<repoKey>/`** (`repoKey = shortHash(resolve(repoPath))`),
> which never touches git.

The reasoning is a direct reading of the charter: LFB's whole job is to keep a user's git **clean**, so it
must not be the thing that fills a code repo with churny bookkeeping the user never asked for. A
`repo_storage.yaml` whose `counts:` and `last_scan:` change on **every** scan would produce a git-status
change (or, if `.lfbridge/` were git-ignored, an untracked directory) on a cadence the user did not initiate.
That is exactly the merge conflict this policy exists to prevent — so **all** of Category B goes to Local
Storage, categorically, and the working repo carries **only** the content the user deliberately generated.

**What goes into the working repo (Category A content):** a **transcript** ([Transcribe.mdx](./Transcribe.mdx))
or an **AI description** ([ai_description.mdx](./ai_description.mdx)) for a file in the repo. Producing one is
the *only* thing that ever creates a `.lfbridge/` directory in the working tree.

**What NEVER goes into the working repo (Category B tracking):** scanning, classifying, pinning,
git-ignoring, deciding (Add-to-IPFS / Add-to-git-ignore), compressing, computing perceptual fingerprints.
All of these are recorded — but **in Local Storage**, never the repo, regardless of whether a transcript ever
exists. (Git-ignoring a big file does edit the repo's own **`.gitignore`** — that is the user-requested
working-tree action itself, not tracking state.)

Resolving the always-in-Local-Storage tracking root for a repo (and the optional additive company sync-repo
mirror) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveTrackingRoot(repoRoot)` (always returns `~/T/_large_files_bridge/repos/<repoKey>/`, never the working repo) and `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null`).

## 2. WEEK 1 — enlist a code repo full of renders — and write NOTHING to it

**What the developer does.** On `studio-tower` they open Large File Bridge and **+ Add repo**
([repos.mdx §6](./repos.mdx)) pointing at `~/BGit/act3/render-farm`. The repo has a `src/`, a `.gitignore`,
and a **`renders/`** folder with **8 ProRes `.mov` clips** (each ~2–4 GB). They think of it as a code repo;
the renders just happen to live there.

**What LFB does automatically — enlist, classify, and stop at the working tree.** LFB registers the repo
and derives a stable **`<repoKey>` = `shortHash(resolve(repoPath))`** for it (the 12-hex `storageSid()`
scheme — so the same repo on the same machine always maps to the same Local-Storage folder). It scans, and
the classifier ([special_files.mdx](./special_files.mdx)) marks the 8 clips `large` + `video` +
`compressible` + `transcribable`. **But — and this is the whole point — it writes none of that into the
repo.** Category B always goes to **Local Storage**:

```
~/T/_large_files_bridge/
└── repos/
    └── a1b2c3d4e5f6/                       ← <repoKey> = shortHash(resolve("~/BGit/act3/render-farm"))
        ├── repo_storage.yaml               ← the repo-WIDE state file — IN LOCAL STORAGE, not the repo
        └── origin.txt                      ← one line: the absolute repo path this <repoKey> maps to
```

```yaml
# ~/T/_large_files_bridge/repos/a1b2c3d4e5f6/repo_storage.yaml
repo_storage:                          # same single level-one key as repo_tracking_scheme.mdx §2
  schema_version: 1
  name: render-farm
  enlisted: { at: 2026-02-02T09:00:00Z, by: bryan@act3ai.com, on_device: studio-tower }
  counts: { special: 8, large: 8, videos: 8, compressible: 8, transcribable: 8, transcribed: 0, described: 0 }
  last_scan: { at: 2026-02-02T09:01:12Z, on_device: studio-tower, headless: false }
```

And the repo itself, right after enlist:

```
~/BGit/act3/render-farm/
├── .git/
├── .gitignore                          ← UNCHANGED (LFB added nothing)
├── src/…
└── renders/                            ← the 8 clips, exactly as the developer left them
    ├── shot_01.mov … shot_08.mov
#  NO .lfbridge/ HERE. `git status` is clean. Nothing untracked.
```

The **only** level-one key is `repo_storage:` ([repo_tracking_scheme.mdx §2.1](./repo_tracking_scheme.mdx)) —
the file is byte-for-byte the same shape it always has; it simply lives in Local Storage, where its per-scan
churn is free and private to the machine.

Seeding a repo's Local-Storage `repos/<repoKey>/` directory and its `repo_storage.yaml` on enlist (and never
touching the working tree) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureRepoStorage(repoRoot)` (seeds under the Local-Storage root from `tracking-root.service.ts` — `resolveTrackingRoot()`; it no longer git-ignores anything, since nothing churny is written into the working repo).

**What we double-check (Week 1):**
* `~/BGit/act3/render-farm/` has **no `.lfbridge/` directory** and `git status` is **clean** — enlist wrote
  nothing to the working tree.
* `~/T/_large_files_bridge/repos/<repoKey>/repo_storage.yaml` exists with the 8-clip `counts:`.
* The `<repoKey>` is `shortHash(resolve(repoPath))` (same repo → same key) so a re-scan reuses the same
  folder; `origin.txt` records the absolute path it was minted from.

## 3. WEEK 2 — a scheduled scan; new clips arrive; re-stamp LOCAL STORAGE in place

**What the developer does.** Nothing. Four more clips (`shot_09.mov`…`shot_12.mov`) land in `renders/`
during the week, and the **every-4h scheduled scan** ([scan.mdx](./scan.mdx)) runs headless.

**What LFB does automatically — recompute and re-stamp, but only in Local Storage.** The scan re-walks the
repo, finds 12 clips now, and updates the rolled-up `counts:` and `last_scan:`. This is exactly the write
the policy exists to contain: `repo_storage.yaml`'s `counts.special` goes `8 → 12` and `last_scan.at`
advances — a change on **every** scan. That re-stamp happens **in the Local-Storage copy**, and the repo
stays pristine:

```yaml
# ~/T/_large_files_bridge/repos/a1b2c3d4e5f6/repo_storage.yaml  (re-stamped in place)
repo_storage:
  …
  counts: { special: 12, large: 12, videos: 12, compressible: 12, transcribable: 12, transcribed: 0, described: 0 }
  last_scan: { at: 2026-02-09T04:00:07Z, on_device: studio-tower, headless: true }   # ← advanced by the headless scan
```

> **This is the churn we keep out of git.** Had `repo_storage.yaml` lived in the repo, this scheduled,
> user-un-initiated scan would have produced a git working-tree change (or untracked-dir noise) every four
> hours — and, on a second machine, a merge conflict. In Local Storage it is invisible to the customer's
> repo — no diff, no `git status` line, nothing.

Re-stamping the Local-Storage `repo_storage.yaml` on each scan (never touching the working repo) is handled
in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` — `refreshCounts(repoRoot)` (writes under `resolveTrackingRoot()`, which always returns the Local-Storage root).

**What we double-check (Week 2):**
* The Local-Storage `repo_storage.yaml` re-stamped `counts` and `last_scan` **in place** — same file, new
  values, not a new file.
* `~/BGit/act3/render-farm/` is **still** free of any `.lfbridge/`; `git status` still clean after a
  headless scan mutated tracking state.

## 4. WEEK 3 — pin & git-ignore six clips — the decision ledger stays in Local Storage; only `.gitignore` changes

**What the developer does.** The renders are big and shouldn't be checked in, so the developer opens the
repo's file table ([one_repo.mdx](./one_repo.mdx)) and flips the **Pin** and **Ignore** inline toggles ON
for **6** of the 12 clips ([decision_toggles.mdx §5](./decision_toggles.mdx)).

**What LFB does automatically — the decision ledger and manifest are Category B; only the `.gitignore` edit
touches the repo.** A pin/ignore **decision** and a **manifest CID** are tracking state — LFB's record of
*what it was asked to do with a file* — so under the data-category split they go to **Local Storage**, not
the working repo. The one thing that legitimately lands in the working tree is the repo's own **`.gitignore`**
edit, because git-ignoring a file *is* an edit to that file, requested by the user:

* `repos/<repoKey>/decisions.yaml` — one `sync` + one `git_ignore` decision event per clip (the ledger,
  [decisions.mdx](./decisions.mdx)) — **in Local Storage**.
* `repos/<repoKey>/manifest.yaml` — each pinned clip's CID — **in Local Storage**.
* `repos/<repoKey>/files/renders/…` — a sidecar per clip carrying the `decision` and `ipfs_pin` events —
  **in Local Storage**.
* `.gitignore` — an anchored ignore line per clip ([git_ignore.mdx](./git_ignore.mdx)) — **in the repo**
  (the user's requested working-tree change).

> **Doesn't writing `decisions.yaml` mean `.lfbridge/` now exists in the repo?** No. Under the data-category
> split, `decisions.yaml` and `manifest.yaml` are **Category-B tracking state** — they live in Local Storage
> and, when the user enables it, travel via the **company sync repo** (§9.2), never via the working repo.
> The working repo's `.lfbridge/` still does **not** exist — nothing has produced a transcript or a
> description. The only working-tree change is the `.gitignore` lines the developer's own git-ignore action
> created:

```
~/BGit/act3/render-farm/
├── .gitignore                          ← 6 anchored ignore lines added (the user's git-ignore action)
├── src/…
└── renders/…
#  STILL NO .lfbridge/ in the repo. `git status` shows only the .gitignore edit.

~/T/_large_files_bridge/repos/a1b2c3d4e5f6/
├── repo_storage.yaml                   ← counts.ipfs_pinned now 6
├── decisions.yaml                      ← 6 clips × 2 axes (the ledger)
├── manifest.yaml                       ← 6 CIDs
└── files/renders/{…}.mov.yaml          ← per-clip decision + ipfs_pin sidecars
```

Appending the decision/pin events to a per-file sidecar (and re-stamping the Local-Storage `repo_storage`
counts) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/file-sidecar.service.ts` — `appendFileEvent(repoRoot, relPath, event)` (path resolved via `tracking-root.service.ts` — `resolveTrackingRoot()`, always the Local-Storage root).

**What we double-check (Week 3):**
* The repo's working tree gains **only** the `.gitignore` edit — **no** `.lfbridge/`, **no**
  `repo_storage.yaml`, **no** `decisions.yaml`/`manifest.yaml`, **no** `files/` sidecars.
* `decisions.yaml`, `manifest.yaml`, and the per-clip sidecars are all in
  `~/T/_large_files_bridge/repos/<repoKey>/`.
* The Local-Storage `repo_storage.yaml`'s `counts.ipfs_pinned` updated to 6 — recorded, in Local Storage.

## 5. WEEK 4 — compress three clips — before/after fingerprints go to LOCAL-STORAGE sidecars

**What the developer does.** Three of the ProRes clips are wildly oversized; the developer selects them and
**Compress** ([compression.mdx](./compression.mdx)).

**What LFB does automatically — capture the rich per-file history, in Local Storage.** Compression is
exactly the kind of event `repo_tracking_scheme.mdx` records in a **per-file sidecar** (`before`/`after`
`{hash,size}`, `fingerprint_before`/`fingerprint_after`, `codec`). A `compress` event is Category-B tracking
state, so the sidecar it needs is written to **Local Storage**, not the repo:

```
~/T/_large_files_bridge/repos/a1b2c3d4e5f6/
├── repo_storage.yaml
├── decisions.yaml
├── manifest.yaml
└── files/                              ← per-file sidecars — in LOCAL STORAGE (path-mirrored)
    └── renders/
        ├── shot_01.mov.yaml            ← carries the compress event (before/after hash+size, perceptual fp)
        ├── shot_04.mov.yaml
        └── shot_07.mov.yaml
```

```yaml
# ~/T/_large_files_bridge/repos/a1b2c3d4e5f6/files/renders/shot_01.mov.yaml
file:                                    # same single level-one key as repo_tracking_scheme.mdx §3
  path: renders/shot_01.mov
  events:
    - kind: compress
      at: 2026-02-16T11:20:00Z
      on_device: studio-tower
      by: bryan@act3ai.com
      before: { hash: 9f2c8b…, size: 3221225472 }
      after:  { hash: 4d7e1a…, size: 402653184 }
      fingerprint_before: a1b2c3…        # perceptual fingerprint — computed LOCALLY, never phones home
      fingerprint_after:  a1b2c4…        # (charter: perceptual fingerprint is local-only)
      codec: { from: prores, to: h264 }
```

The perceptual fingerprint is computed **entirely locally** ([perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx)) —
compression involves **no network** and produces **no content artifact that belongs in the repo**, so it
correctly leaves the repo working tree alone.

**What we double-check (Week 4):**
* The three `compress` events are recorded as **Local-Storage** sidecars under
  `repos/<repoKey>/files/renders/…` — the repo's working tree still has **no** `files/` sidecars and **no**
  `.lfbridge/`.
* Compression captured `before`/`after` `{hash,size}` + perceptual fingerprint and made **no** network call.

## 7. WEEK 6 — remove the repo — the Local-Storage directory is cleaned up

**What the developer does.** They decide `render-farm` shouldn't be bridged after all and **remove it** from

keep or delete.

```
~/BGit/act3/render-farm/                              ← untouched by the removal
```

Deleting the Local-Storage `repos/<repoKey>/` tree on repo removal (and never mutating the repo working tree
on remove) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.service.ts` — `removeRepo(repoId)` (calls `rm -rf` on the `resolveTrackingRoot()` directory; a repo that merely went **missing**, e.g. an unmounted drive, retains its directory).

**What we double-check (Week 6):**
* `~/T/_large_files_bridge/repos/<repoKey>/` is **gone** after removal — no orphaned Local-Storage tracking

## 8. WEEK 6 + 1 DAY — re-add, then transcribe ONE clip — content enters the repo; tracking stays in Local Storage

**What the developer does.** A colleague needs a transcript of one interview clip, so the developer re-adds
`render-farm` (a fresh Local-Storage `repos/<repoKey>/` is created exactly as in Week 1), opens `shot_02.mov`
in the media viewer, and clicks **Generate transcription** ([Transcribe.mdx](./Transcribe.mdx)).

**What LFB does automatically — content into the repo, tracking left in Local Storage.** Producing the
transcript is a **Category-A content artifact**. Per [Transcribe.mdx §3](./Transcribe.mdx) a transcript is
written **inside the repo's committed `.lfbridge/`**, path-mirrored, filename + `.transcription` appended.
This is the *first* time a `.lfbridge/` directory appears in the working tree — and it holds **only** the
content artifact. Crucially, the tracking state **does not migrate**:

1. **The transcript lands in the repo's committed `.lfbridge/`** —
   `~/BGit/act3/render-farm/.lfbridge/renders/shot_02.mov.transcription`. This is a small text file that
   *should* be committed and travel with the repo — Category-A content.
2. **The tracking state stays in Local Storage.** `repo_storage.yaml`, the per-file sidecars under
   `files/…`, `decisions.yaml`, `manifest.yaml`, and `history/<device>.txt` remain at
   `~/T/_large_files_bridge/repos/<repoKey>/`. There is **no migration** into the repo — producing content
   never redirects tracking state. `counts.transcribed` ticks to `1` **in the Local-Storage
   `repo_storage.yaml`**.

```
# BEFORE the transcript (re-added):
~/BGit/act3/render-farm/                                        # .gitignore only; NO .lfbridge/
```

## 9. The awkward branches

### 9.1 An AI description is content too (no transcript needed)

Had the developer instead clicked **Generate AI description** on an *image* in the repo
([ai_description.mdx §2](./ai_description.mdx)), the outcome is the same shape: a `.ai_description` artifact
is a **Category-A content artifact**, so it lands in the committed `.lfbridge/` and travels with the repo —
while tracking state stays in Local Storage. A code repo of stills that gets one image described gets a
one-file `.lfbridge/` (the description) exactly as a video repo that gets one clip transcribed. Neither event
moves any tracking state. (See the parity tour, [use_case_ai_description_parity.mdx §7](./use_case_ai_description_parity.mdx).)

### 9.2 The company / personal SYNC REPO — an ADDITIONAL travelling home for tracking state

If the user has configured a **sync repo** on the owning **company** or **Personal** storage (*"a Company
repo only for the Large File Bridge webapp to sync changes"* — [storage_company.mdx §7](./storage_company.mdx),
[storage_personal.mdx §14](./storage_personal.mdx)) **and** turned on the per-repo toggle
([repo_settings.mdx §2](./repo_settings.mdx), default **OFF**), the Category-B tracking is **additionally**
mirrored to **`<syncRepo>/repos/<repoKey>/`** — **in addition to** Local Storage, not instead of it. Local
Storage stays the authoritative working copy on each machine; the sync repo is the travel vehicle so a second
computer sees the same `repo_storage.yaml`, sidecars, `decisions.yaml`, and `manifest.yaml`:

```
# with a company sync repo ~/BGit/Bryan_git/act3_large_files_bridge/ configured + the per-repo toggle ON:
~/T/_large_files_bridge/repos/a1b2c3d4e5f6/           ← ALWAYS written here (authoritative)
├── repo_storage.yaml
└── files/renders/…

~/BGit/Bryan_git/act3_large_files_bridge/repos/a1b2c3d4e5f6/   ← ALSO mirrored here (travels to other computers)
├── repo_storage.yaml
└── files/renders/…

~/BGit/act3/render-farm/                              ← the media repo is STILL untouched by scan churn
```

The invariant is unchanged — **the media repo's working tree is never dirtied by tracking state** — and the
mirror is purely additive. A sync pass reconciles the two (append-only sidecars/history merge; `decisions.yaml`
/ `manifest.yaml` union-merge). Absent a configured sync repo and the toggle, tracking is Local-Storage-only.

Mirroring the always-in-Local-Storage `repos/<repoKey>/` subtree into the company sync repo (and reconciling
a pulled one back) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)` / `reconcileFromSyncRepo(repoRoot)` (the sync-repo root comes from `tracking-root.service.ts` — `resolveStateSyncRepo()`).

### 9.3 The keep-`.lfbridge/` consent gates the CONTENT, not the tracking

Turning the keep-`.lfbridge/` consent **off** ([storage_settings.mdx §3](./storage_settings.mdx)) governs
whether **Category-A content** (transcripts / AI descriptions) may be written into the repo's `.lfbridge/` at
all. It has **no effect on tracking state** — that always goes to Local Storage regardless. When the consent
is off, even a transcript is kept out of the working repo (relocated per that setting), so the working tree
never gains a `.lfbridge/`. The two are orthogonal: the data-category split decides *tracking vs. content*;
the keep-`.lfbridge/` consent decides *whether content may sit in the repo*.

**What we double-check (branches):**
* An **AI description** alone produces a one-file `.lfbridge/` (content only) exactly like a transcript, and
  moves no tracking state.
* With a company sync repo configured and the toggle on, tracking is at **both**
  `~/T/_large_files_bridge/repos/<repoKey>/` **and** `<syncRepo>/repos/<repoKey>/`, and the **media repo's
  working tree is still untouched**.
* The keep-`.lfbridge/` consent (off) keeps even content out of the repo; tracking is unaffected (always
  Local Storage).

## 10. The cross-cutting invariants this tour proves

1. **A scan never dirties a customer's git working tree.** Every scan-driven write (`repo_storage.yaml`
   re-stamps, sidecars, history) lands in **Local Storage**, so no un-user-initiated change ever appears in
   `git status`.
2. **Tracking state never enters the working repo.** `repo_storage.yaml`, `files.yaml`, sidecars,
   `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`, and compression records are Category B and live
   only in Local Storage (and the sync repo, if enabled) — regardless of whether a transcript ever exists.
3. **Only content enters the repo.** A **transcript** or an **AI description** (Category A) is the *only*
   thing LFB ever writes into the working repo's `.lfbridge/`; the developer's git-ignore action edits
   `.gitignore` but writes no tracking files.
4. **Re-stamps happen in place.** Repeated scans overwrite the **same** Local-Storage `repo_storage.yaml`;
   they do not accrete files.
5. **Removal is clean.** Removing a repo deletes exactly one directory
   (`~/T/_large_files_bridge/repos/<repoKey>/`) and mutates nothing in the repo.
6. **Producing content does not migrate tracking.** Writing the first transcript adds a content file to the
   repo's `.lfbridge/`; tracking state **stays** in Local Storage — there is never a migration.
7. **The sync repo is an additive mirror.** When enabled, tracking is written to Local Storage **and** the
   company sync repo — never into the media repo's working tree.

## 11. Acceptance criteria

1. **Enlist writes nothing to the repo.** Adding a repo with big media creates
   `~/T/_large_files_bridge/repos/<repoKey>/repo_storage.yaml` (single `repo_storage:` key,
   `repoKey = shortHash(resolve(repoPath))`) and **no `.lfbridge/` in the repo**; `git status` is clean.
2. **Scans re-stamp Local Storage only.** Each scan (including headless) advances `counts:`/`last_scan:` in
   the **Local-Storage** `repo_storage.yaml` **in place**, and the repo working tree gains **no** scan-driven
   change — no `.lfbridge/`, no `files/` sidecars, no `history/`.
3. **Decisions and manifest are Category B.** Pinning/git-ignoring writes `decisions.yaml`, `manifest.yaml`,
   and per-file sidecars to **Local Storage** (and the sync repo when enabled); the only working-tree change
   is the `.gitignore` edit the git-ignore action itself makes.
4. **Compression records to Local-Storage sidecars.** A `compress` captures before/after `{hash,size}` +
   perceptual fingerprint (no network) into `repos/<repoKey>/files/…` and writes nothing into the repo.
5. **Removal cleans up.** Removing a repo **deletes** `~/T/_large_files_bridge/repos/<repoKey>/` and does not
   mutate the repo working tree.
6. **Content into the repo, tracking stays put.** Producing the first **transcript OR AI description** writes
   that content artifact into the repo's committed `.lfbridge/`; `counts.transcribed`/`counts.described`
   ticks in the **Local-Storage** `repo_storage.yaml`; the tracking files do **not** migrate into the repo;
   subsequent scans re-stamp **in Local Storage**.
7. **Branches.** An **AI description** alone behaves identically to a transcript; a configured **company/
   Personal sync repo** with the per-repo toggle on **additionally** mirrors tracking to
   `<syncRepo>/repos/<repoKey>/` (never the media repo); the keep-`.lfbridge/` consent gates whether content
   sits in the repo and never affects tracking.
8. Every write matches [artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx), [storage_local.mdx](./storage_local.mdx),
