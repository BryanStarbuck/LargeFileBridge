# LargeFileBridge — AI description of images & videos

`LargeFileBridge` can generate a **hyper-detailed, searchable description** of an **image** or a **video**
so the file can be **found, understood, and re-found later without opening it**. A vision model is shown
the file and a carefully written per-kind prompt, and returns structured prose — an overview, the
composition, a best-guess city/country, building/place identification, inside-vs-outside, a
cinematographer's shot classification, and a head-to-toe description of every person in frame. That text
is stored beside the file's other analysis so it powers search and a presence indicator.

Two things make this feature worth pinning down: **where the description is stored** (unified with the
storage-level media analysis — §2), and **the charter/privacy posture** (§7 — this is the *one* place
LFBridge uploads a file to the network on purpose, always by an explicit click, and it must not be
confused with the local-only perceptual-fingerprint feature).

**structurally symmetric** across the whole product, so a user who has learned one has learned the other,
and so a developer touching one knows to touch the other. This is a **hard design rule**, not a
coincidence — every surface in the left column below has the mirror in the right column, and adding a new
surface to one obliges adding it to the other.

a **provider key**. Everywhere *else* — the buttons, the tab, the icon, the metrics, the header primary, the
calc engine, the placement radios, the shared launcher, the shared types — the two are **the same shape**.

The symmetry is asserted in code by the parallel module trees `modules/transcribe/` ↔ `modules/describe/`,

`FileRow`, `DescribeResult`/`DescribeBatchResult` mirroring `TranscribeResult`/`TranscribeBatchResult`).

## 1. The two-tab analysis panel & the default-tab rule

At the bottom of the media viewer (band 6) sits an analysis panel with up to **two tabs**:

```
┌──────────────────────────────────────────────────────────────┐
│  [✓ Transcription]   [✓ AI description]                       │  ← tab strip; ✓ = content exists
├──────────────────────────────────────────────────────────────┤
│  AI description · gemini-flash-latest · 2h ago   [⟳ Regenerate]│
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ ## Overview                                              │ │
│  │ A wide establishing shot of a rain-slicked city street…  │ │
│  │ ## Composition … ## Location … ## People …               │ │
│  └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

* **Which tabs show** depends on the media kind: **Transcription** shows for **audio + video** (media that
  *has* speech); **AI description** shows for **image + video** (media that is *visual*). A **video**
  therefore shows **both** tabs; an **image** shows **only AI description**; an **audio** file shows **only
  Transcription** (audio is deliberately *not* AI-described — transcription covers "what was said").
* **The ✓ marker.** Each tab carries a small **✓** when its content already exists (an existing transcript,
  or an existing `.ai_description` artifact), so the user can see at a glance what has already been generated.
* **The default-tab rule.** On open, the panel selects, in order: (1) **Transcription if it has content**;
  else (2) the **first tab that has content**; else (3) the **first tab**. So a described-but-not-yet-
  transcribed video still lands the user on real content, and the *preference* is transcription when both
  exist. The user's explicit tab click then wins for the rest of the session.

The two-tab panel, the ✓ presence markers, and the transcription-preferring default-tab logic are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/MediaViewer.tsx` — `MediaAnalysis` component.

## 2. Where the description is stored — inside the repo/storage `.lfbridge/`, path-mirrored, extension APPENDED (LOCKED — supersedes the earlier beside-media layout)

The description does **not** live in the Local state dir, there is **no** `.lfbridge/analysis/`
directory for it, and — as of this revision — it does **not** sit loose next to the media either. Exactly
like the transcript ([Transcribe.mdx §3](./Transcribe.mdx), the authoritative template), it lives **inside
the owning repo/storage's own `.lfbridge/` tracking directory**, in a **path-mirrored hierarchy**, under
the media's **full filename (extension kept)** with **`.ai_description` appended** as a **second**
extension:

```
<placement root>/
├── render/
│   └── skyline.mp4                              ← the real media file (untouched, still in the working tree)
└── .lfbridge/
    └── render/
        └── skyline.mp4.ai_description           ← the generated description: same relative path under
                                                    .lfbridge/, full filename (skyline.mp4) with
                                                    .ai_description APPENDED
```

For a file at **`render/skyline.mp4`** the description is written to
**`<placement root>/.lfbridge/render/skyline.mp4.ai_description`** — the media's relative directory
mirrored under `.lfbridge/`, the full filename kept, `.ai_description` **appended** (**not** substituted).
(Example: `TheFunnyVideo.mp4` → `.lfbridge/…/TheFunnyVideo.mp4.ai_description`.) The `.ai_description`
token is a single constant — `AI_DESCRIPTION_EXT` — **appended**, never replacing the original extension.

The **placement root** and the media's mirrored relative hierarchy are resolved **exactly like
transcription — by the shared ordered placement rule** ([Transcribe.mdx §3.4](./Transcribe.mdx), the
authoritative spec): walk **up** to the nearest ancestor carrying `storage.yaml` / `.lfbridge/` / `.git/`
(the artifact lands at `<root>/.lfbridge/<rel-dir>/<name.ext>.ai_description`); else route to the **owning
storage's dedicated Git repo** when enabled (mirroring the file's hierarchy under that repo's `.lfbridge/`
— `<dedicatedRepo>/.lfbridge/<rel-dir>/<name.ext>.ai_description`, **committed** there); else the owning
storage's own root (`<storageRoot>/.lfbridge/<rel-dir>/<name.ext>.ai_description`); else the **first-time
setup wizard** ([Transcribe.mdx §3.5](./Transcribe.mdx)) when no Personal storage exists; else, as a last
resort, a `.lfbridge/` created in the media's **own directory**
(`<mediaDir>/.lfbridge/<name.ext>.ai_description`) — never loose beside the media. **Get AI details / AI
description is one of the actions that triggers the first-time wizard**, and it consults placement
(`needsSetup`) before it runs. Pasted `~/…` paths are home-expanded first (Transcribe.mdx §3.6).

> **Why `.lfbridge/` (and not beside the media).** `.lfbridge/` is **committed and travels with the repo**
> ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)), so the description is **version-controlled and
> synced across the user's computers by git** — generated once on one computer and reused everywhere —
> while the media working tree stays **clean** (no loose `.ai_description` files scattered next to videos).
> Descriptions are small UTF-8 YAML, so committing them is cheap and desirable (the multi-GB media stays
> git-ignored and rides IPFS). The old **`*.ai_description` git-ignore nudge is RETIRED** — no ignore line
> is injected anymore.

> **Append-not-replace resolves the old collision caveat.** Because the original extension is **kept** and
> `.ai_description` is **appended** (not substituted), two media files that differ only by extension in the
> same folder **no longer collide**: `skyline.mp4` → `.lfbridge/…/skyline.mp4.ai_description` and
> `skyline.mov` → `.lfbridge/…/skyline.mov.ai_description` are **distinct files**. The earlier layout's
> accepted "second run overwrites the first" caveat is therefore **retired** — appending the extension is
> precisely what fixes it. A tool that wants a media file's description computes the path deterministically:
> `join(root, ".lfbridge", relDir, mediaFilename + ".ai_description")`.

### 2.1 The YAML shape — a self-contained `done` record

The `.ai_description` artifact is a small YAML document. Generating a description writes it directly as a
**done** record (there is no separate pre-seeded skeleton to upgrade — the artifact either exists as a
`done` record or does not exist yet):

```yaml
source: render/skyline.mp4
status: done
engine: gemini-flash-latest       # the model id sent (alias → newest GA Flash)
provider: gemini                  # the adapter id
generated: 2026-07-05T09:41:00Z   # ISO timestamp
kind: video                       # image | video
description: |
  ## Overview
  A wide establishing shot of a rain-slicked city street at dusk…
  ## Composition … ## People …
```

A read returns a usable `DescribeView` **only** when the artifact exists with `status: done` **and** a
non-empty `description` string — otherwise it reads back as **null** (no content yet), so the tab shows the
generate pane rather than an empty box.

### 2.2 The Storage table indicator

The **Storage file table's `analysis: ["description"]` indicator lights up** the moment the
`.ai_description` artifact exists under `.lfbridge/` for the media (computed deterministically from the
media path) — one source of truth for "this file has a description," shared by the viewer tab and the
storage table (which detects the path-mirrored artifact under `.lfbridge/` rather than a YAML under
`.lfbridge/analysis/`). Because the artifact lives **inside the committed `.lfbridge/`** in every placement
case, generating it injects **no** git-ignore line — the old **`*.ai_description` nudge is retired** (the
same retirement transcription made for `*.transcription`); the small text artifact is committed and travels
with the repo, while the multi-GB media stays git-ignored and rides IPFS.

Resolving the description path via the shared ordered placement rule, reading back a done description, and writing the `.ai_description` artifact under `.lfbridge/` are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `resolveDescriptionPath()` / `readDescription()` / `describeOne()` (artifact path via `lfbridgeArtifactPath(root, relPath)` = `join(root, ".lfbridge", relPath) + AI_DESCRIPTION_EXT` in `storage/artifact-placement.service.ts`, where `AI_DESCRIPTION_EXT` is **appended**, not substituted).

The storage-table indicator that detects the `.ai_description` artifact is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` — `analysisOutputs()`.

## 3. The provider adapters — key resolution, supported kinds, the video rule, the size cap

Vision providers are a small **pluggable set of adapters**. Which one runs is chosen by (a) an explicit
request, else (b) the configured default, else (c) the **first adapter that has a key AND supports the
media kind**. Preference order is **Gemini → OpenAI → Grok**.

| Adapter | Provider | Image | Video | Endpoint / model | Notes |
| --- | --- | --- | --- | --- | --- |
| `gemini` | Google Gemini | ✓ | **✓** | `generativelanguage.googleapis.com/v1beta/models/<model>:generateContent`, default `gemini-flash-latest` (alias → newest GA Flash, currently Gemini 3.5 Flash) | The **only** adapter that describes **video** (inline `inline_data` base64). |
| `openai` | OpenAI | ✓ | — | `api.openai.com/v1/chat/completions`, default `gpt-4o` | OpenAI-compatible chat/vision; `image_url` data URI. |
| `grok` | xAI Grok | ✓ | — | `api.x.ai/v1/chat/completions`, default `grok-2-vision-1212` | OpenAI-compatible chat/vision; images only. |

### 3.1 The video = Gemini-only rule (LOCKED)

Only **Gemini** accepts a **video**. OpenAI and Grok describe **images only**. So when the file is a video
and no Gemini key is configured, generation returns **`no_provider`** with the honest reason *"no AI
provider configured for video — add an API key for Gemini (only Gemini describes video)."* An image can be
described by **any** of the three.

### 3.2 Key resolution (config → env → shared Gemini file)

Each adapter resolves its key from the **app config** (`ai.<provider>.api_key` in the local `config.yaml`)
**first**, then falls back to well-known **environment variables**, so a machine that already exports a key
"just works":

* **Gemini** — `GEMINI_API_KEY` / `GOOGLE_GENAI_API_KEY` / `GOOGLE_API_KEY`, then the **shared Gemini key
  file** (below).
* **Grok** — `XAI_API_KEY` / `GROK_API_KEY`
* **OpenAI** — `OPENAI_API_KEY`

**Gemini has a THIRD source (LOCKED):** after config and env, it reads the **shared Google/Gemini key
file** `~/.config/GoogleCloud/apikey.yaml` — the **SAME file** the `~/BGit/all/tools` Gemini / "nano
banana" image tools already use. It has two fields; we prefer **`apiKey`**, then fall back to
**`4k_apiKey`** (matching the tools' vision path). So a machine set up for those tools describes
video/images with **no extra setup**. The full credential-resolution + the missing-key UX live in the
dedicated **[ai_credentials.mdx](./ai_credentials.mdx)**.

Reading the shared Gemini key file (prefer `apiKey`, else `4k_apiKey`; absent/placeholder/invalid → no key) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/google-apikey-file.ts` — `loadGoogleApiKey()`; wired into the `gemini` adapter's `apiKey()` in `adapters.ts`.

### 3.3 The inline cap & compress-to-fit (LOCKED)

All three providers are called with the file **inline as base64**. Because base64 inflates bytes by ~4/3
and the providers bound inline requests (~20MB), the bytes we send are capped at **18MB**. Rather than
**fail** an over-size file (a 44MB phone video is completely normal), we **compress it to fit first** and
upload the compressed copy — the description feature must "just work" on real videos.

**The rule:** before upload, ensure the media is **at or under 17.5MB** (a deliberate margin under the
18MB cap). A file already under the target is uploaded **untouched**. A file over it is transcoded to a
**temporary** copy under the state dir; that copy is uploaded and then **deleted**. **The user's original
file is NEVER modified** — this is separate from the charter's explicit, user-initiated compression
feature; it is an ephemeral upload-only proxy.

**Video ladder (charter algorithm):** *keep the pixel resolution the same if high compression alone can
get it under the target; only if that fails, reduce the resolution ~25% each step and repeat.*

1. **CRF at original resolution** — H.264 (`libx264`, `-preset medium`, `-crf 28`, `yuv420p`, faststart,
   AAC 96k audio). If the result is ≤ 17.5MB, **keep full resolution**.
2. **Harder CRF at original resolution** — one more pass at `-crf 34` (still full resolution) before we
   give up any pixels.
3. **Step the resolution down** — reduce the height by **25%**, **snapping to a standard height**
   (2160/1440/1080/900/720/540/480/360/288/240) when one is **within 3%** of the reduced value, and
   re-encode at `-crf 30`. Repeat down the ladder to a **240p floor**.
4. **Two-pass guarantee** — if even the floor rung is over, a **two-pass** encode at the floor resolution
   to an **exact target bitrate** (computed from the clip duration and the byte budget) deterministically
   lands it under the cap.

**Codec choice (LOCKED):** we compress to **H.264 in mp4**, not the "better-compressing" HEVC/AV1, on
purpose — the provider's decoder must accept whatever we send, and **H.264/mp4 is the one format every
vision provider reliably decodes**. Audio is kept but re-encoded low (96k, or 64k in the two-pass
fallback) so it can't eat the byte budget.

**Image path:** an over-size image is re-encoded to a temporary **JPEG** at descending quality (85 → 72 →
60) and then descending scale until it fits. (The temp JPEG may drop a PNG's alpha — acceptable because it
exists only to show the model; the original PNG is untouched.)

**Tooling:** video compression needs **ffmpeg** (+ ffprobe for duration/resolution); image compression
needs **ImageMagick**. When the required tool is absent and the file is over the cap, generation
**`failed`**s with an install hint (`brew install ffmpeg` / `imagemagick`) instead of silently uploading
nothing.

### 3.4 Model lifecycle & deprecation self-heal (LOCKED)

Providers **retire** older model versions on a schedule. Google hard-retired `gemini-2.0-flash` on
**2026-06-01**; after that date `generateContent` returns **`404 NOT_FOUND … is no longer available`** and
a config still **pinned** to that id can NEVER succeed.

**The default is the `gemini-flash-latest` ALIAS (LOCKED), not a pinned id.** Google hot-swaps that alias
to its newest GA Flash (with a 2-week notice window), so the app **auto-tracks new releases and never
hard-breaks** on a retirement the way a pinned version does. As of **2026-07** the alias resolves to
**Gemini 3.5 Flash** (image + video). A user may still **pin a concrete id** in Settings → AI if they want
a fixed model. Two further mechanisms keep even a pinned config working across retirements:

* **Auto-heal on load.** When `config.yaml` has a Gemini model in the **retired set**, the config loader
  rewrites it to the current default and persists it — so an old machine's stale pin (`gemini-2.0-flash`)
  silently upgrades itself the next time the app reads config. A **valid custom** model the user chose is
  left untouched (only known-dead ids are rewritten).
* **Actionable error, not a raw 404.** If a call still fails on a retired/unknown model (e.g. a hand-typed
  bad id not in the retired set), the raw Google 404 is rewritten into a message that **names the bad
  model and the recommended one**: *"Gemini model "X" is not available — the provider may have retired it.
  Update the model in Settings → AI (recommended: gemini-flash-latest)."* That message is what the viewer
  shows and what lands in **`error.err`** (§8).

The single source of truth for the current default + the retired-model set (so the adapter and the config
loader agree) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/models.ts` — `DEFAULT_GEMINI_MODEL`, `RETIRED_GEMINI_MODELS`, `looksLikeModelRetired()`.

The auto-heal of a pinned retired model on config load is handled in code:

## 4. The prompts — shipped defaults + per-computer customize

Each media kind has **its own prompt**, checked into the repo beside the describe module
(`describe/prompts/image.md` and `describe/prompts/video.md`). The prompt is what makes the output
**hyper-detailed and consistent** rather than a vague caption.

### 4.1 What the prompts ask the model to produce

Both prompts open by insisting on a **HYPER-DETAILED, factual, non-editorializing** description that marks
uncertain guesses `(uncertain)`, then require the model to write **prose under a fixed set of headings in a
fixed order**:

* **Overview** — a short plain-English "at a glance" summary of what the file is / what happens (2–4
  sentences for an image; 3–6 for a video, including its genre/purpose — interview, vlog, drone footage…).
* **Hyper-Detail** — the deep-dive section: exhaustively describe everything visible (every object,
  surface, texture, small detail), not just the headline subject. This is where the most words go.
* **Composition** — explicit **FRAME-LEFT / FRAME-CENTER / FRAME-RIGHT** placement plus foreground /
  midground / background, with a **percentage-of-frame** estimate and approximate **distance/direction**
  for notable elements (e.g. "a coffee shop ~25% at frame-left; a mailbox ~9% frame-center, ~100 ft
  forward"), separating where the **people** are from where the **non-people things** are; plus balance,
  leading lines, rule-of-thirds, depth of field, and where the eye is drawn.
* **Location** — the model's best-guess **CITY, STATE/REGION, and COUNTRY** (a single best city-NAME guess
  even if `(uncertain)`), a **city-size class** (major/big, medium, small town, suburb, rural), and
  location specifics, with reasoning (signage language, architecture, plates, flora, landmarks), or
  `(indeterminate)`.
* **Setting & Place** — state clearly **INSIDE (interior)** vs **OUTSIDE (exterior)** and the kind of
  space; identify any building/structure (name if recognizable, else type + architectural style/era); for
  exteriors, describe surroundings the way a **movie location scout** would (share-of-frame + distance).
* **Lighting** — state plainly **DAYTIME vs NIGHTTIME** (or dawn/dusk/golden hour/overcast), then the
  lighting design (natural vs artificial, key/fill/back, hard vs soft, high-key/low-key, color temperature,
  practicals in frame).
* **Cinematography** — classify the shot the way a **cinematographer** would: framing / **shot size**
  (extreme close-up, close-up, medium close-up, **cowboy shot** / medium, medium-wide, wide, extreme-wide /
  establishing), camera angle (eye-level, low, high, overhead / bird's-eye, dutch), and lens character.
* **People** — for **EACH** person in frame, **hyper-describe** them: apparent age range and height/build;
  the **full outfit head-to-toe** (garments, colors, materials, footwear, accessories, headwear, eyewear);
  hair and distinctive features; and their **placement and size in frame** with posture, gaze, and
  expression. "No people in frame." when empty.
* **Objects & Text** — notable objects, brands, and any legible **text/signage quoted verbatim**.
* **Colors & Mood** — dominant palette and overall tone.

The **video** prompt adds two video-specific headings: a **Shot-by-shot / Timeline** (approximate timecode
ranges × what's on screen × camera movement × cuts/transitions/graphics) and an **Audio & Speech** section
that *paraphrases* key lines but explicitly **does not transcribe verbatim** — the Transcription tab covers
that — and it asks the People section to note **wardrobe changes across the clip**.

### 4.2 Per-computer override (customize / save / reset)

The prompt used for a kind is the **shipped default** unless the user has **customized** it. Customizing
**copies** the shipped default into a **per-computer override** under the state dir
(`<stateDir>/describe/prompts/<kind>.md`); once that copy exists, it is the prompt that is sent. **Save**
writes edited override text; **Reset** deletes the override and reverts to the shipped default. Nothing here
touches the network — it only resolves *which text* is passed to an adapter.

Resolving the in-use prompt (per-computer override if present, else shipped default) and the customize / save / reset flow are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts.ts` — `getPrompt()` / `promptView()` / `customizePrompt()` / `savePrompt()` / `resetPrompt()`; shipped defaults `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts/image.md` and `video.md`.

## 7. Charter & privacy posture — the one deliberate network path (LOCKED)

This is the **single place** in LargeFileBridge that reaches the network **on purpose**. Generating a
description **uploads the file's bytes** (inline base64) to the chosen third-party provider — Google, xAI,
or OpenAI. That is unavoidable for a hosted vision model, so it is fenced by strict rules:

* **Always explicit, always user-initiated.** A description is generated **only** on a deliberate
  **Generate** (or **Regenerate**) click. It **never** runs automatically — not on a scan, not on a
  background pin, not on viewing a file.
* **Entirely separate from perceptual fingerprinting.** The charter's **perceptual / content-fingerprint
  matching** (PhotoDNA-style) is a *different* feature that must run **entirely locally and must never
  phone home**. AI description is **not** that feature and must never be confused with it — fingerprinting
  stays local; description is the one opt-in upload.
* **Honest about the upload in the UI.** The generate pane states plainly that the file is uploaded to the
  provider (and, for video, that the provider is Gemini), so the user is never surprised about bytes
  leaving the machine.
* **No key → no upload.** With no provider configured, nothing can be sent (the action is disabled, §8).

The adapter module is annotated as the one deliberate outbound path, and the upload happens only inside an explicit generate, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — the `describe()` methods (called only from `describeOne()` in `describe.service.ts` on an explicit request).

## 8. Empty, error & disabled states

* **No description yet** — the tab shows a **generate pane**: an icon, "No AI description yet," a blurb
  that names the upload (and Gemini for video), and a **Generate description** button.
* **No provider configured** — the **Generate description** button stays **clickable**; clicking opens the
  **credentials-missing popup** (**Close** / **Instructions**), and **Instructions** opens the
  `/ai-credentials` setup page in a new tab (path + placeholder YAML for
  `~/.config/GoogleCloud/apikey.yaml`, plus the Settings/env alternatives). A **video** with only
  image-capable keys hits the same popup at runtime (`no_provider`, because video needs Gemini). See
  **[ai_credentials.mdx](./ai_credentials.mdx)**.
* **Unsupported kind** — a non-image/-video path returns **`unsupported`**; audio never offers AI
  description (its tab is Transcription).
* **Over the 18MB inline cap** — the file is **auto-compressed to a temporary copy under 17.5MB** and that
  copy is uploaded (§3.3); the original is untouched. Generation only **`failed`**s here when the
  compression tool is missing (ffmpeg/ImageMagick — the reason names the `brew install` to run) or the
  file genuinely can't be squeezed under the cap.
* **Provider error / block / timeout** — a provider HTTP error, a Gemini safety block, non-JSON, or a
  120s timeout comes back as **`failed`** with the reason, surfaced in the toast; the existing description
  (if any) is untouched. A **retired/unknown model** 404 is rewritten into an actionable reason that names
  the bad model and the recommended one (§3.4). **Every failure is written to `error.err`** — the app's
  fault trail at `<stateDir>/error.err` (WARN/ERROR/FATAL), tagged `[describe]` with the file,
  `provider=`, and `kind=` for diagnosis. Logged by `describeOne()`'s `catch` via `log.error` in
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts`;
  the two rolling logs (`log.log` all-levels, `error.err` faults) are written by
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/logging.ts`.
* **Already described** — without `overwrite`, a second generate returns **`skipped` ("already
  described")**; **Regenerate** passes `overwrite` to replace it.

## 10. The provider-readiness gate — `withDescribeReady` (LOCKED — mirror of `withModelReady`)

Transcription wraps every launch in **`withModelReady()`**, which interposes the heavyweight-model
permission/provisioning flow when the local model is absent and then **resumes the original action**
([transcribe_engine.mdx §3.6](./transcribe_engine.mdx)). AI description has the **exact structural mirror**,
**`withDescribeReady()`** — but because descriptions call a **hosted provider** rather than a local model,
the thing it gates on is a **usable provider key**, not a multi-GB download. There is **no download, no disk
estimate, and no two-phase install** — the readiness check is a fast credential probe.

### 10.1 The shared launcher (`lib/describe.ts`) (LOCKED)

Every describe entry point (§9, §5.1, the metric popup §12, the calc-engine Apply
[describe_calc_engine.mdx §4](./describe_calc_engine.mdx)) routes through **one** frontend launcher, the
mirror of `lib/transcribe.ts`:

```
withDescribeReady(intent, run):
  1. probe readiness  → GET /api/describe/providers  (anyAvailable? + which kinds each supports)
  2. if a usable provider exists for the intent's kind → run() immediately (the common case)
  3. else capture a PENDING INTENT (confirmed | offered) and open CredentialsMissingDialog
  4. on a successful key add (the user returns from /ai-credentials) → REPLAY the pending intent:
       - confirmed  → enqueue the confirmed describe set (Processing shows it)
       - offered    → re-open the describe offer popup on the originating route
```

* **The kind-aware probe.** For a **video** the gate needs **Gemini specifically** (only Gemini describes
  video, §3.1); for an **image** any of the three keys satisfies it. So the same "no provider" gate can fire
  for a video on a machine that has an OpenAI/Grok key but no Gemini key — the popup says so.
* **Pending-intent replay is identical to transcription's.** A **confirmed** describe (the user pressed the
  blue *"Describe N files"* in the metric popup, §12) **proceeds** after the key is added; an **offered**
  popup that was only *opened* **re-opens on the original page**. This is the same `confirmed` vs `offered`
  distinction `withModelReady` captures.

`withDescribeReady()` (readiness probe, pending-intent capture, credentials-dialog interpose, replay-after-fix) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/describe.ts` — `withDescribeReady(intent, run)` (mirror of `lib/transcribe.ts` `withModelReady`); the dialog is `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/describe/CredentialsMissingDialog.tsx`, opened via the shared global request bus `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/setupWizard.ts` — `requestAiCredentials()`.

### 10.2 The two "not-ready" cases, and how they differ from transcription (LOCKED)

| | Transcription | AI description |
| --- | --- | --- |
| **Missing local dependency** | `whisper`/`ffmpeg` binary absent → **setup card** with a `brew`/`pipx` install + Re-check ([Transcribe.mdx §5.2](./Transcribe.mdx)). | **n/a** — no local binary is needed to call a provider (the compress-to-fit path *does* need ffmpeg/ImageMagick for over-cap files, §3.3; that surfaces as a `failed` with a `brew` hint). |

## 13. Where descriptions land — the shared 3-way placement radios (LOCKED — cross-reference)

**Where** a `.ai_description` artifact is written is **not** decided here — §2 gives the resolved path shape

  the same `.lfbridge/` hierarchy (§2).
* The three choices, the default, and the migration behavior are specified in
  [placement_radios.mdx](./placement_radios.mdx); this spec defers to it wholesale and only asserts that
  **descriptions obey the same radio the transcripts obey**.

## 15. Acceptance criteria

1. The media viewer's bottom band shows **Transcription** and **AI description** tabs, each with a **✓**
   when its content exists; **video** shows both, **image** shows only AI description, **audio** shows only
   Transcription. The default tab **prefers Transcription** when present, else the first tab with content,
   else the first tab.
2. AI description works on **image + video only** (audio → transcription). The description is stored
   **inside the owning repo/storage's `.lfbridge/`**, path-mirrored, under the media's **full filename with
   `.ai_description` appended** (`render/skyline.mp4` → `.lfbridge/render/skyline.mp4.ai_description`) —
   resolved by the shared ordered placement rule, **no** `.lfbridge/analysis/` and **no** `.transcribe/`
   directory. It is a self-contained `done` YAML record
   `{ source, status, engine, provider, generated, kind, description }` — so the Storage table's
   **`description` indicator** lights up from that artifact's presence. Because it lives in the committed
   `.lfbridge/`, it is **committed and travels with the repo** in every placement case — **no**
   `*.ai_description` git-ignore line is injected (that nudge is retired); appending the extension (rather
   than replacing it) means two files differing only by extension no longer collide.
3. Providers are the pluggable adapters **Gemini / OpenAI / Grok**; **only Gemini describes video**;
   selection is requested → configured-default → first available that supports the kind (preference Gemini,
   OpenAI, Grok). Keys resolve **config → env → (Gemini only) the shared `~/.config/GoogleCloud/apikey.yaml`**
   (Gemini `GEMINI_API_KEY`/`GOOGLE_GENAI_API_KEY`/`GOOGLE_API_KEY` then the shared file's `apiKey`/`4k_apiKey`;
   Grok `XAI_API_KEY`/`GROK_API_KEY`; OpenAI `OPENAI_API_KEY`). Uploads are inline base64 (~18MB bound); a
   file over **17.5MB** is first **compressed to a temporary copy that fits** (H.264 video keeping
   resolution when possible, else stepping down ~25%; JPEG image) and that copy is uploaded — **the
   original is never modified**; generation only fails over-cap when ffmpeg/ImageMagick is missing.
4. The per-kind **prompts** (shipped `image.md` / `video.md`) drive a hyper-detailed description with the
   fixed headings in order: overview → **hyper-detail** → composition (**frame-left/center/right** +
   %-of-frame + distance) → location (**city/state/country + city-size class**) → setting & place
   (inside/outside, building, movie-scout exteriors) → **lighting (day/night)** → **cinematographer shot
   classification** → per-person **head-to-toe** description → objects/text → colors & mood; video adds
   shot-by-shot timeline + audio paraphrase. The user can **customize** a prompt into a per-computer
   override and **reset** it.
5. The REST surface (`GET /providers`, `GET /file`, `POST /file`, `GET /prompt`, `POST /prompt/customize`,
   `PUT /prompt`, `DELETE /prompt`) is allow-list-gated and mounted at `/api/describe`; each generation
   reports truthfully (`described` / `skipped` / `no_provider` / `unsupported` / `failed`).
6. Settings → Tools shows the **provider-key status** (configured / no-key, with the env var / config key
   hint and the "only Gemini describes video" note) and the **per-kind prompt editor** (Save customize /
   Reset).
7. Generation is the **one deliberate external-network path**: it **uploads** the file, is **always**
   explicit and user-initiated (never automatic), states the upload in the UI, and is **entirely separate**
   from the charter's local-only perceptual-fingerprint feature (which must never phone home). With no
   provider configured, nothing is uploaded.
8. **Symmetry contract (§0):** every transcription surface has an AI-description mirror — the action button
