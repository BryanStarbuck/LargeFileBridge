# LargeFileBridge — AI description of images & videos

`LargeFileBridge` can generate a **hyper-detailed, searchable description** of an **image** or a **video**
so the file can be **found, understood, and re-found later without opening it**. A vision model is shown
the file and a carefully written per-kind prompt, and returns structured prose — an overview, the
composition, a best-guess city/country, building/place identification, inside-vs-outside, a
cinematographer's shot classification, and a head-to-toe description of every person in frame. That text
is stored beside the file's other analysis so it powers search and a presence indicator.

Two things make this feature worth pinning down: **where the description is stored** (unified with the
storage-level media analysis — §2), and **the charter/privacy posture** (§7 — this is the *one* place
LFBridge uploads a file to the network on purpose, always by an explicit click, and it must not be
confused with the local-only perceptual-fingerprint feature).

## 0. The symmetry contract with Transcribe (LOCKED)

**Everything transcription has, AI description mirrors.** The two features are deliberately kept
**structurally symmetric** across the whole product, so a user who has learned one has learned the other,
and so a developer touching one knows to touch the other. This is a **hard design rule**, not a
coincidence — every surface in the left column below has the mirror in the right column, and adding a new
surface to one obliges adding it to the other.

| Transcription ([Transcribe.mdx](./Transcribe.mdx)) | AI description (this spec) |
| --- | --- |

| **Create Transcriptions** page action-link — [Transcribe.mdx §2.5](./Transcribe.mdx) | **Create AI descriptions** page action-link — §5.1 |
| **Transcribe** task tab — [task_tabs.mdx §4.4](./task_tabs.mdx) | **AI descriptions** task tab — §11 |

| **`transcribable` / `transcribed`** metric pair + educate-and-fix popup — [task_tabs.mdx §2/§5.1](./task_tabs.mdx) | **`describable` / `described`** metric pair + popup — §12 |
| Blue **"Transcribe ›"** header recommendation — [one_repo.mdx §3.1](./one_repo.mdx) | Blue **"Describe ›"** header recommendation — §12.3 |
| **"What Can Be Transcribed" calc engine** — [transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx) | **"What Can Be AI-Described" calc engine** — [describe_calc_engine.mdx](./describe_calc_engine.mdx) |
| Permission-gated **heavyweight-model provisioning** (`withModelReady`) — [transcribe_engine.mdx §3](./transcribe_engine.mdx) | Provider/credential-gated **readiness flow** (`withDescribeReady`) — §10 |

| **3-way placement radios** (where `.transcription` lands) — [placement_radios.mdx](./placement_radios.mdx) | Same 3-way radios (where `.ai_description` lands) — §13 |
| `FileRow.transcribe` · `TaskMetrics.{transcribable,transcribed}` · `TranscribeResult` | `FileRow.describe` · `TaskMetrics.{describable,described}` · `DescribeResult` |

**The one deliberate asymmetry** is the runtime backend: transcription drives a **local ASR model**
(offline, no credentials), while AI description calls a **hosted vision provider** (the one deliberate
network upload, §7). So transcription's "readiness" is a **model download**; AI description's "readiness" is
a **provider key**. Everywhere *else* — the buttons, the tab, the icon, the metrics, the header primary, the
calc engine, the placement radios, the shared launcher, the shared types — the two are **the same shape**.

The symmetry is asserted in code by the parallel module trees `modules/transcribe/` ↔ `modules/describe/`,
`lib/transcribe.ts` ↔ `lib/describe.ts`, `TranscribeStatusIcon.tsx` ↔ `DescribeStatusIcon.tsx`, and the
paired fields on `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` (`TaskMetrics`,
`FileRow`, `DescribeResult`/`DescribeBatchResult` mirroring `TranscribeResult`/`TranscribeBatchResult`).

## 1. The two-tab analysis panel & the default-tab rule

At the bottom of the media viewer (band 6) sits an analysis panel with up to **two tabs**:

```
┌──────────────────────────────────────────────────────────────┐
│  [✓ Transcription]   [✓ AI description]                       │  ← tab strip; ✓ = content exists
├──────────────────────────────────────────────────────────────┤
│  AI description · gemini-flash-latest · 2h ago   [⟳ Regenerate]│
│  ┌──────────────────────────────────────────────────────────┐ │
│  │ ## Overview                                              │ │
│  │ A wide establishing shot of a rain-slicked city street…  │ │
│  │ ## Composition … ## Location … ## People …               │ │
│  └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

* **Which tabs show** depends on the media kind: **Transcription** shows for **audio + video** (media that
  *has* speech); **AI description** shows for **image + video** (media that is *visual*). A **video**
  therefore shows **both** tabs; an **image** shows **only AI description**; an **audio** file shows **only
  Transcription** (audio is deliberately *not* AI-described — transcription covers "what was said").
* **The ✓ marker.** Each tab carries a small **✓** when its content already exists (an existing transcript,
  or an existing `.ai_description` artifact), so the user can see at a glance what has already been generated.
* **The default-tab rule.** On open, the panel selects, in order: (1) **Transcription if it has content**;
  else (2) the **first tab that has content**; else (3) the **first tab**. So a described-but-not-yet-
  transcribed video still lands the user on real content, and the *preference* is transcription when both
  exist. The user's explicit tab click then wins for the rest of the session.

The two-tab panel, the ✓ presence markers, and the transcription-preferring default-tab logic are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/MediaViewer.tsx` — `MediaAnalysis` component.

## 2. Where the description is stored — inside the owning root's tracking area, path-mirrored, extension APPENDED (LOCKED — supersedes the earlier beside-media layout)

The description does **not** live in the Local state dir, there is **no** `.lfbridge/analysis/`
directory for it, and — as of this revision — it does **not** sit loose next to the media either. Exactly
like the transcript ([Transcribe.mdx §3](./Transcribe.mdx), the authoritative template), it lives **inside
the owning root's own tracking area**, in a **path-mirrored hierarchy**, under the media's **full filename
(extension kept)** with **`.ai_description` appended** as a **second** extension.

> **Which tracking area — the storage-KIND rule (owned by [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)).**
> `.lfbridge/` exists **only inside a working git repo**, where LFB is a **guest** and quarantines its files
> under one hidden directory. A **dedicated LFB file repo** — the Personal / Company / Community **SDL
> storage** — has **no `.lfbridge/` directory at all**: the SDL repo root **is** the `.lfbridge` area. That
> policy spec **owns** the rule; this spec applies it, identically to the transcript.
>
> | Storage kind | Example root | The description lands at |
> | --- | --- | --- |
> | **`repo`** — a working git repo (Git Storage) | `~/BGit/Bryan_git/charlie-kirk/` | `<root>/.lfbridge/<rel-dir>/<name.ext>.ai_description` |
> | **`personal` / `company` / `community`** — an SDL file repo | `~/BGit/Bryan_git/personal_large_files_bridge/` | `<root>/<rel-dir>/<name.ext>.ai_description` (**no** `.lfbridge/`) |

```
WORKING GIT REPO (LFB is a guest → one hidden .lfbridge/ corner)
<repo>/
├── render/
│   └── skyline.mp4                              ← the real media file (untouched, still in the working tree)
└── .lfbridge/
    └── render/
        └── skyline.mp4.ai_description           ← the generated description: same relative path under
                                                    .lfbridge/, full filename (skyline.mp4) with
                                                    .ai_description APPENDED

SDL FILE REPO (a repo that exists ONLY for LFB → the root IS the .lfbridge area)
~/BGit/Bryan_git/personal_large_files_bridge/
└── _Mirror/
    └── Politics/
        └── x.mp4.ai_description                 ← NO .lfbridge/ segment — the mirror hangs off the root
```

For a **working-repo** file at **`render/skyline.mp4`** the description is written to
**`<repo>/.lfbridge/render/skyline.mp4.ai_description`** — the media's relative directory
mirrored under the tracking base, the full filename kept, `.ai_description` **appended** (**not**
substituted). (Example: `TheFunnyVideo.mp4` → `.lfbridge/…/TheFunnyVideo.mp4.ai_description`.) In an **SDL**
the same file resolves to **`<sdlRoot>/render/skyline.mp4.ai_description`** — same mirroring, same appended
extension, **no `.lfbridge/` segment**. The `.ai_description` token is a single constant —
`AI_DESCRIPTION_EXT` — **appended**, never replacing the original extension.

The **placement root** and the media's mirrored relative hierarchy are resolved **exactly like
transcription — by the shared ordered placement rule** ([Transcribe.mdx §3.4](./Transcribe.mdx), the
authoritative spec): walk **up** to the nearest ancestor carrying `storage.yaml` / `.lfbridge/` / `.git/`
(the artifact lands at `<trackingBase(root)>/<rel-dir>/<name.ext>.ai_description`); else route to the
**owning storage's dedicated Git repo** when enabled (mirroring the file's hierarchy under **that repo's
root** — `<dedicatedRepo>/<rel-dir>/<name.ext>.ai_description`, **committed** there, with **no `.lfbridge/`
segment**, because a dedicated LFB file repo **is** an SDL); else the owning storage's own root
(`<storageRoot>/<rel-dir>/<name.ext>.ai_description` for an SDL-kind storage; `.lfbridge/`-prefixed only when
that root is a working repo); else the **first-time setup wizard** ([Transcribe.mdx §3.5](./Transcribe.mdx))
when no Personal storage exists; else, as a last resort, a `.lfbridge/` created in the media's **own
directory** (`<mediaDir>/.lfbridge/<name.ext>.ai_description` — that directory is not an SDL, so the
working-repo branch applies) — never loose beside the media. Because **every SDL also carries its own
`.git/`**, the walk-up must classify the root by **storage kind first** and only then fall back to "has
`.git/` → working repo" — the kind **wins** over the presence of `.git/`
([artifact_placement_policy.mdx §0.2](./artifact_placement_policy.mdx)). **Get AI details / AI
description is one of the actions that triggers the first-time wizard**, and it consults placement
(`needsSetup`) before it runs. Pasted `~/…` paths are home-expanded first (Transcribe.mdx §3.6).

> **Why the tracking area (and not beside the media).** That area — the working repo's `.lfbridge/`, or an
> SDL's root — is **committed and travels with the repo**
> ([repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)), so the description is **version-controlled and
> synced across the user's computers by git** — generated once on one computer and reused everywhere —
> while the media working tree stays **clean** (no loose `.ai_description` files scattered next to videos).
> Descriptions are small UTF-8 YAML, so committing them is cheap and desirable (the multi-GB media stays
> git-ignored and rides IPFS). The old **`*.ai_description` git-ignore nudge is RETIRED** — no ignore line
> is injected anymore.

> **Append-not-replace resolves the old collision caveat.** Because the original extension is **kept** and
> `.ai_description` is **appended** (not substituted), two media files that differ only by extension in the
> same folder **no longer collide**: `skyline.mp4` → `.lfbridge/…/skyline.mp4.ai_description` and
> `skyline.mov` → `.lfbridge/…/skyline.mov.ai_description` are **distinct files**. The earlier layout's
> accepted "second run overwrites the first" caveat is therefore **retired** — appending the extension is
> precisely what fixes it. A tool that wants a media file's description computes the path deterministically:
> `join(trackingBaseDir(root), relDir, mediaFilename + ".ai_description")` — where `trackingBaseDir()` yields
> `<root>/.lfbridge` for a working repo and `<root>` for an SDL. It **never** hard-codes `".lfbridge"` into
> the join, which would put a phantom segment into every SDL path.

### 2.1 The YAML shape — a self-contained `done` record

The `.ai_description` artifact is a small YAML document. Generating a description writes it directly as a
**done** record (there is no separate pre-seeded skeleton to upgrade — the artifact either exists as a
`done` record or does not exist yet):

```yaml
source: render/skyline.mp4
status: done
engine: gemini-flash-latest       # the model id sent (alias → newest GA Flash)
provider: gemini                  # the adapter id
generated: 2026-07-05T09:41:00Z   # ISO timestamp
kind: video                       # image | video
description: |
  ## Overview
  A wide establishing shot of a rain-slicked city street at dusk…
  ## Composition … ## People …
```

A read returns a usable `DescribeView` **only** when the artifact exists with `status: done` **and** a
non-empty `description` string — otherwise it reads back as **null** (no content yet), so the tab shows the
generate pane rather than an empty box.

### 2.2 The Storage table indicator

The **Storage file table's `analysis: ["description"]` indicator lights up** the moment the
`.ai_description` artifact exists under the owning root's **tracking base** for the media (computed
deterministically from the media path via `trackingBaseDir()`, so it looks under `.lfbridge/` in a working
repo and at the **root** in an SDL) — one source of truth for "this file has a description," shared by the
viewer tab and the storage table (which detects the path-mirrored artifact directly under that base rather
than a YAML under an `analysis/` sub-directory). Because the artifact lives **inside the committed tracking
area** in every placement case, generating it injects **no** git-ignore line — the old
**`*.ai_description` nudge is retired** (the
same retirement transcription made for `*.transcription`); the small text artifact is committed and travels
with the repo, while the multi-GB media stays git-ignored and rides IPFS.

### 2.3 When the provider REFUSES — `.ai_description_rejected` (NEW, LOCKED)

Sometimes the provider considers a file and **says no**: a safety verdict, or a `RECITATION` filter on a
slide that resembles copyrighted material. **That is an answer, not a breakdown** — and the provider is the
only party that knows why. We used to throw the whole answer away: the adapter reduced it to
*"returned no description"*, so Google's `finishMessage` — the sentence naming which rule fired — never
reached the user, and the only trace was one line in `error.err` that rotates away within days.

**A refusal earns a durable record. It is written INSTEAD of the description, named
`<media>.ai_description_rejected`.**

* **Same place, same rules.** Same root, same placement radio, same path mirror, same appended-extension
  grammar as `.ai_description` (§2) — only the extension differs. `Slide23.jpeg` →
  `Slide23.jpeg.ai_description_rejected`.
* **A separate EXTENSION, not a `status: rejected` inside `.ai_description`.** So "does this file have a
  description?" stays a pure **file-exists** test for every reader (§2.2's indicator, the viewer, the
  batch preview) instead of every one of them having to parse a YAML to find out — and so a refusal is
  visible in a plain directory listing.
* **Only a real refusal writes one** — a refusal that **survived every retry** (§3.5), or a blocked input.
  A timeout, a dead account, or a retired model **must never** write one: those are a broken *config* or a
  broken *link*, not a verdict about the file, and a retired model would otherwise litter the whole tree
  with rejection records on every file in the batch.
* **A description SUPERSEDES a refusal.** When a file is later described — a new model, an overwrite retry,
  a re-encoded upload — the stale `.ai_description_rejected` is **deleted**, so the tree can never assert
  both "described" and "refused" about one file.

**What goes in it: everything the provider said.** Two layers, because the useful detail is rarely the one
you predicted:

1. The **named fields** a human reads first — `finish_reason`, the `finish_message` that actually explains
   the refusal, `block_reason`, `model_version`, and the `response_id` **to quote to Google when disputing
   a refusal**.
2. **`provider_response`** — the provider's **entire, untouched response**, so a detail we never thought to
   parse is still on disk.

**This does not conflict with [transactions_log.mdx §9](./transactions_log.mdx)** ("never a body, never a key
in the ledger): §9 governs the **ledger**, and this never goes there. The API key rides the URL's `?key=`
and never the response body, so nothing secret can reach the file. Values are written **spelled out** —
YAML anchor/alias output is disabled, because this file exists to be read and forwarded, not to make the
reader resolve a `*a1` pointer.

**A refusal is SETTLED — never re-offered, never re-billed** (the §12.5 rule, and
[ocr.mdx §2.3](./ocr.mdx)'s "existence IS the signal"). The batch preview and the enqueue both drop a file
that already has a rejection record, and `describeOne` backstops the paths that bypass the queue. Without
this, every future batch would spend a real provider call to be told the same thing — the failures that
halted 483 files on 2026-07-16. **Overwrite is the one way to ask the provider to reconsider.**

**A refusal is NOT a queue failure.** The batch counts it **done**, not failed: nothing went wrong, and the
answer is already recorded on disk. Counting it as failed would paint a tree of copyrighted slides red and
feed the batch ceiling ([job_queue.mdx §2.7](./job_queue.mdx)) that halts the remaining files — the exact
defect this record exists to end. It drains green, the same way OCR's text-free image does.

Verified end-to-end on the real refusing slide: `status=rejected`, a `.ai_description_rejected` written with
Google's copyright `finishMessage` and full `provider_response`, **no** `.ai_description`, and a second run
that does **not** re-ask the provider.

Capturing every detail of a refusal and deciding which empty response IS one are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — `ProviderRejection` / `attachRejection()` / `rejectionOf()` / `isRefusalFinishReason()` (the `gemini` adapter's `describe()` builds the evidence; `classifyProviderFault()` reads the object, never the prose).
Writing the record, superseding it on a later success, and never re-offering a settled refusal are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `writeRejection()` / `resolveDescriptionPath()` (returns `rejectedPath`) / `isRejected()`, with the drop-already-settled guard in `previewDescribe()` / `enqueueDescribe()` / `describeOne()`.

## 3. The provider adapters — key resolution, supported kinds, the video rule, the size cap

Vision providers are a small **pluggable set of adapters**. Which one runs is chosen by (a) an explicit
request, else (b) the configured default, else (c) the **first adapter that has a key AND supports the
media kind**. Preference order is **Gemini → OpenAI → Grok**.

### 3.3 The inline cap & compress-to-fit (LOCKED)

All three providers are called with the file **inline as base64**. Because base64 inflates bytes by ~4/3
and the providers bound inline requests (~20MB), the bytes we send are capped at **18MB**. Rather than
**fail** an over-size file (a 44MB phone video is completely normal), we **compress it to fit first** and
upload the compressed copy — the description feature must "just work" on real videos.

**The rule:** before upload, ensure the media is **at or under 17.5MB** (a deliberate margin under the
18MB cap). A file already under the target is uploaded **untouched**. A file over it is transcoded to a
**temporary** copy under the state dir; that copy is uploaded and then **deleted**. **The user's original
file is NEVER modified** — this is separate from the charter's explicit, user-initiated compression
feature; it is an ephemeral upload-only proxy.

**Video ladder (charter algorithm):** *keep the pixel resolution the same if high compression alone can
get it under the target; only if that fails, reduce the resolution ~25% each step and repeat.*

1. **CRF at original resolution** — H.264 (`libx264`, `-preset medium`, `-crf 28`, `yuv420p`, faststart,
   AAC 96k audio). If the result is ≤ 17.5MB, **keep full resolution**.
2. **Harder CRF at original resolution** — one more pass at `-crf 34` (still full resolution) before we
   give up any pixels.
3. **Step the resolution down** — reduce the height by **25%**, **snapping to a standard height**
   (2160/1440/1080/900/720/540/480/360/288/240) when one is **within 3%** of the reduced value, and
   re-encode at `-crf 30`. Repeat down the ladder to a **240p floor**.
4. **Two-pass guarantee** — if even the floor rung is over, a **two-pass** encode at the floor resolution
   to an **exact target bitrate** (computed from the clip duration and the byte budget) deterministically
   lands it under the cap.

**Codec choice (LOCKED):** we compress to **H.264 in mp4**, not the "better-compressing" HEVC/AV1, on
purpose — the provider's decoder must accept whatever we send, and **H.264/mp4 is the one format every
vision provider reliably decodes**. Audio is kept but re-encoded low (96k, or 64k in the two-pass
fallback) so it can't eat the byte budget.

**Image path:** an over-size image is re-encoded to a temporary **JPEG** at descending quality (85 → 72 →
60) and then descending scale until it fits. (The temp JPEG may drop a PNG's alpha — acceptable because it
exists only to show the model; the original PNG is untouched.)

**Tooling:** video compression needs **ffmpeg** (+ ffprobe for duration/resolution); image compression
needs **ImageMagick**. When the required tool is absent and the file is over the cap, generation
**`failed`**s with an install hint (`brew install ffmpeg` / `imagemagick`) instead of silently uploading
nothing.

The compress-to-fit pipeline (the 17.5MB target, the resolution ladder, the CRF passes, and the two-pass
guarantee — all to a temp file, never the original) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/fit-media.ts` — `fitMediaUnderLimit()` (with `fitVideoUnderLimit()` / `fitImageUnderLimit()` / `resolutionLadder()`); invoked from `describeOne()` in `describe.service.ts` right before the adapter call, with the temp copy cleaned up in a `finally`.

### 3.3.2 The two-pass fit is a GUARANTEE — a fixed floor must never outvote the byte budget (LOCKED)

The last rung of the ladder (§3.3) is billed as the **deterministic "must fit"** step: a two-pass encode to
an exact bitrate derived from the byte cap and the clip's duration. It is the promise that every oversize
video eventually fits. **That promise was broken**, and the break is instructive enough to lock the rule.

**The defect.** The bitrate was derived from the budget and then **clamped up** to a fixed 120kbps floor:

```
videoKbps = Math.max(120, totalKbps - audioKbps)   // ← the floor outvotes the budget
```

The cap is a fixed number of **bytes**, so the longer the clip the fewer bits per second it may spend. Past
a certain duration the budget-derived rate falls **below** the floor, `max()` raises it back **above** what
the budget allows, the encode lands **over** the cap, and the "guarantee" **throws**. The cliff was at
**12.5 minutes**, and it was exact: every video past it failed, and the evidence agreed to the minute —
of the failures on the product owner's tree the shortest was **13.9 min**, none were shorter, and each
failure printed `kb/s:120.03` — the floor's own fingerprint.

**A per-second cost cannot be fixed against a per-file cap.** The same error hid behind the audio track: a
constant 64kbps of audio was always reserved, and 64kbps alone exceeds 17.5MB past **~36 minutes** — so for
a long clip **no** video bitrate could fit, not even zero. Lifting the floor alone would not have saved
those files.

**The rules (all LOCKED):**

1. **Never clamp the bitrate UP to a floor above the budget.** Spend what the budget allows; the only floor
   is a token `MIN_VIDEO_KBPS` that exists to stop the correction loop chasing zero.
2. **Audio is a passenger, not a fixture.** It is kept when it costs a small share of the budget (≤20%),
   degraded when tight, and **dropped entirely** when the budget can't seat it. A silent description beats
   no description, and speech is the **transcript's** job ([Transcribe.mdx](./Transcribe.mdx)) — this
   prompt describes what is **seen**.
3. **Trade frame rate before picture quality.** A vision model samples video at roughly **1fps**, so frames
   beyond that rate are bits spent on detail the model never looks at. Long clips encode at 1–5fps, which
   is what makes a sub-100kbps encode legible instead of a smear.
4. **Measure, then correct.** The computed bitrate is a *prediction* (a missing duration, container
   overhead, an incompressible source can all miss). Check the **actual** output size and, on an overshoot,
   re-derive the bitrate from what was **measured** and retry (bounded), rather than failing on first miss.
5. **Don't walk a ladder that cannot land.** The CRF ladder is a *search*, and every rung is a full encode
   of the whole clip. When the budget-per-second is too small for any CRF encode at source resolution to
   fit (< ~250kbps), all ~10 rungs are doomed — that is not a slow path but a **wasted** one, hours of
   ffmpeg per file. Skip straight to the two-pass, which is where such a clip was always going to land.

Verified against the real failures: **13.9 min/182MB → 16.7MB** (was: failed), **29.2 min/166MB → 16.4MB**
(was: failed), **139.1 min/909MB → 16.0MB at 16kbps/1fps/no audio** (was: failed; audio alone would have
needed 3.8× the whole cap), and the 11-min clip that already worked still fits — now with *more* video
bitrate, because degraded audio freed budget for the picture.

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/fit-media.ts` — `fitVideoUnderLimit()` (the `ladderIsPlausible` skip + the correcting two-pass loop), `audioKbpsFor()`, `fpsFor()`, and `encodeTwoPass()` (which takes the audio bitrate and fps).

### 3.3.1 Compress-to-fit runs ASYNC — it must NEVER block the event loop (LOCKED)

The compress-to-fit transcode (§3.3) shells out to **ffmpeg / ffprobe / ImageMagick**, and a single video
re-encode can take **minutes**. Because AI description is the product's **most parallel** background op —
the queue fans out to **~24 in-flight `describe` jobs at once** (§12.6, [job_queue.mdx §3](./job_queue.mdx))
— every one of those child-process calls **MUST be asynchronous** (`child_process.spawn`, awaited), the
**exact mirror** of the compression engine's async transcode runner ([compression.mdx](./compression.mdx)).
Running them **synchronously** (`spawnSync`) is a **LOCKED-closed defect**:

* A synchronous `spawnSync("ffmpeg", …)` **freezes the entire Node event loop** for the whole transcode —
  the single API server can serve **no other request** while it blocks. With ~24 describe workers each able
  to hit a synchronous transcode, the server stalls for **seconds to minutes at a time**.
* The user-visible symptom is precisely the reported bug: firing **2,000 AI descriptions**, then opening
  **`/processing`** in another tab — and **the Processing page never loads**, because `GET /api/progress`
  ([processing.mdx §5](./processing.mdx)) is stuck behind a blocked event loop and can't answer the poll.
  Everything downstream (the dock, the conditional nav item, every other page) hangs with it.
* Therefore: `fit-media.ts` runs **all** heavy child processes through an **async** runner
  (`runAsync` = `child_process.spawn` with a hard-kill timeout, capturing stdout for the ffprobe JSON), and
  `fitMediaUnderLimit()` / `fitVideoUnderLimit()` / `fitImageUnderLimit()` / `probeVideo()` / `encodeCrf()` /
  `encodeTwoPass()` are **all `async`** and **awaited** by `describeOne()`. The one-off temp-name generator
  uses `crypto.randomUUID()` (no child process at all), not `spawnSync("uuidgen")`. Only the near-instant
  `which` tool-detection probe stays synchronous (a few ms — the same choice the compression engine makes
  for `onPath()`).
* The **inline base64 read** of the up-to-18MB upload payload is likewise **async** (`fs.promises.readFile`,
  not `fs.readFileSync`), so encoding a file for upload doesn't block the loop under the same 24-way
  concurrency.

This is catalogued as a performance defect in [performance.mdx P-27](./performance.mdx) (theme **T3 — the
backend blocks the Node event loop**) and is the describe-side sibling of the compress queue's
"heavy transcode is async" rule ([job_queue.mdx §3](./job_queue.mdx)).

### 3.5 A transient failure is NOT a verdict — bounded retry with backoff (NEW, LOCKED)

> **The diagnosis in this section is corrected by [§3.6](#36-a-provider-call-must-never-reuse-a-pooled-socket-new-locked--corrects-35).**
> The retry policy below is right and stays LOCKED. But the *cause* it names for the 1,803 aborted files —
> *"the provider's throughput could not sustain the fan-out (§12.6)"* — is *wrong*, and it was never
> measured. The provider was never the bottleneck: those calls **timed out before they were ever sent**,
> stalled in Node's own connection pool. Retry was treating a symptom.

A hosted vision call fails for two fundamentally different reasons, and **treating them alike is the single
most expensive bug this feature has had**:

* **Permanent** — a bad key, a retired model (§3.4), unsupported media, an explicit safety `blockReason`.
  It will fail identically forever. Retrying wastes the user's time and quota.
* **Transient** — a timeout under load, a `429`, a `5xx`, a dropped socket, **or an empty `200`**. It would
  have **succeeded moments later**.

**The empty `200` belongs in the transient column, and that is the counter-intuitive one.** The provider
can return a perfectly well-formed response carrying no description text. It reads like a verdict — *"the
model won't describe this file"* — so it was treated as permanent. It isn't: the same video that came back
empty described **on the very next attempt**, `finishReason: STOP`, 1,877 tokens of text. Nothing about
the file was wrong; the provider simply had a bad moment.

**An empty body is not always a blip, though — sometimes it is a REFUSAL.** This section used to claim a
genuine refusal *"arrives as an explicit `blockReason` / safety verdict, **never** as a silent empty body."*
**That is false.** Gemini also **refuses** by returning an empty candidate carrying nothing but a
`finishReason` — HTTP `200`, no error, no `blockReason`. On the product owner's corpus that is ~3% of calls:
`RECITATION` on slide and court-document images, the occasional `PROHIBITED_CONTENT`. The adapter reported
only *"returned no description"*, so the reason — and Google's `finishMessage` explaining exactly which rule
fired — was **thrown away**. §2.3 now records all of it.

**A refusing `finishReason` is STILL transient, and this is the counter-intuitive part.** The tempting fix
is to call it permanent — *the model refused, why ask twice?* **Measured, that is wrong.** Generation is
**sampled**, so the output-side filter is a coin toss, not a verdict. One slide, **10 identical calls**:

| finishReason | calls | result |
|---|---|---|
| `STOP` | **6** | described normally, 7.6k–10.2k chars |
| `RECITATION` | **4** | empty |

The same bytes, the same prompt, the same model — a **~40% coin toss**. Calling that permanent would strand
a file that describes 6 times in 10 and brand it with a rejection record it never earned; retrying 4× fails
all four only ~2.6% of the time. So it **retries**, exactly as [§3.5](#35-a-transient-failure-is-not-a-verdict--bounded-retry-with-backoff-new-locked)'s
stated bias demands: *when ambiguous, prefer transient*. Verified: the slide above now describes **3/3**
through the app.

**The one empty response that IS permanent** is `promptFeedback.blockReason` — the **input** refused
*before* generation. Nothing was sampled, so there is nothing to re-roll.

**The rules (LOCKED):**

1. **Classify on the evidence OBJECT, never on the message prose.** The error names the `finishReason` for
   humans, and `"…(finishReason: PROHIBITED_CONTENT)"` would trip a `/SAFETY|PROHIBITED/` regex **by pure
   wording**. Structure decides; text is for reading.
2. **A refusing `finishReason` retries** (`RECITATION`, `SAFETY`, `PROHIBITED_CONTENT`, `BLOCKLIST`, `SPII`).
3. **A blocked INPUT (`blockReason`) is permanent** — attempt 1, no retries.
4. **Only a refusal that SURVIVES every retry earns a `.ai_description_rejected`** (§2.3). A blank `STOP`
   never does: recording it would assert a verdict the provider never reached, and would permanently stop
   re-offering a file that merely had a bad moment.

**What went wrong.** The queue recorded a failure per file and moved on; **nothing retried**. So a
transient blip permanently marked a file "not described." In one overnight run on the product owner's tree,
**1,803 distinct files** — the overwhelming majority of the batch — failed with Node's bare
`This operation was aborted`. Successes went **49 → 5 → 0 → 0 → 0** per hour while aborts climbed into the
hundreds: the provider's throughput could not sustain the fan-out (§12.6), every in-flight call blew its
fixed **120s** deadline, and each one burned a file **permanently**. The tree looked barely-described the
next morning, and the popup — correctly — re-offered all 1,900 files, which is what made the whole batch
look like it had never run. **The files were not skipped; they were spent.**

**The rules (LOCKED):**

1. **Retry transient classes, bounded** — 4 attempts total, exponential backoff (2s base, doubling)
   **with jitter** so a wide fan-out does not retry in lockstep and re-create the stampede.
2. **Honor the provider's own hint** — a `Retry-After` header wins over the computed backoff.
3. **Never retry a permanent fault** — it fails on the first attempt, immediately.
4. **The deadline scales with the payload.** A fixed timeout is wrong when payloads span two orders of
   magnitude: 200KB and 17.5MB cannot share one deadline. Base 120s + ~20s/MB, capped at 10 min.
5. **Name the failure honestly.** `This operation was aborted` told the user nothing — a timeout must say
   it timed out, and after how long ([§8](#8-empty-error--disabled-states)'s truthful-outcome rule).
6. **Retry is per-call, not per-batch.** It lives at the provider seam, so every adapter and every entry
   point (single Generate, queue, batch popup) inherits it with no caller opt-in.
7. **Wrap the WHOLE provider call, not just the HTTP.** The retry sits around the entire
   `adapter.describe()` — round-trip *and* response interpretation — because the empty-`200` only becomes
   an error once the adapter reads a response the HTTP layer considered a total success. A retry bolted
   onto the fetch alone silently misses it. It is wrapped **outside the fit** (§3.3), so a retry re-sends
   the already-compressed copy instead of re-running a multi-minute ffmpeg transcode.

Verified: two forced transient failures → the file still describes on attempt 3 (old code: permanent
failure); two forced empty-`200`s → described on attempt 3 (old code: permanent failure); a `401` → fails
on attempt 1 in 0.1s with no wasted retries.

The transient/permanent split, the jittered backoff, the `Retry-After` honor, the size-scaled deadline, and the honest timeout message are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — `withProviderRetry()` (the retry loop) with `isTransient()`, `retryAfterMs()`, `timeoutForBytes()`, and `postJsonOnce()` (one attempt + the honest timeout rename); applied around `adapter.describe()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` — `describeOne()`. Shared by **all** adapters, since every provider call funnels through that one seam.

The refusal-vs-blip split of the empty `200` is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — the `gemini` adapter's `describe()` names the `finishReason` in the thrown error, and `classifyProviderFault()` sorts the refusing reasons to **`transient`** (§3.5 rule 2 — generation is **sampled**, so a refusing `finishReason` is a coin toss worth 4 rolls, not a verdict; only a `promptFeedback.blockReason` is `permanent`). Regression-covered in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/provider-faults.spec.ts`.

### 3.6 A provider call must never reuse a pooled socket (NEW, LOCKED — corrects §3.5)

Node's global `fetch` keeps a per-origin **undici Pool** and reuses its keep-alive sockets. Against
`generativelanguage.googleapis.com` **under concurrency**, a request handed to an already-used socket can
sit in the pool's queue **unsent** until our own `AbortController` fires. It then surfaces as
`provider call timed out after 120s` — which reads exactly like a slow provider and is not: **the socket
never carried the request.** We were timing our own queue.

**Every previous explanation for this failure was an assumption.** §3.5 blamed the provider's throughput;
§12.6 named "the provider's rate limit" as the only real ceiling; §12.6.1 correctly noted the failures were
`120s` timeouts rather than `429`s but left the cause open. Measurement settles it.

**The evidence.** 48 real files from the failing batch, 24-way, identical file list, Node and a Python
control run **simultaneously** against the same account:

| Arm | Requests sent | Queue (create → sendHeaders) p90 | Total p50 | Outcome | Wall |
|---|---|---|---|---|---|
| Pooled reuse (Node default) | **35/48** | **121.3s** | 42s | **18 timeouts** | 155s |
| Fresh connection per request | **48/48** | **0.1s** | 21.4s | **48 ok** | **51s** |
| Python control (new conn/req) | 48/48 | — | 23s | 48 ok | — |

The decisive split is **queue time, not server time**: time from *request sent* to *response* was ~22s p50
/ ~28s max in **every** arm, including the broken one. Peak **event-loop lag was 6ms**, so nothing was
CPU-blocked either. The provider was fast and consistent throughout; sustained 24-way for 3 minutes (221
calls) produced **zero** timeouts and no throttling.

**Counter-intuitive, and the reason to write this down: lowering concurrency makes it WORSE.** At 8-way the
queue p50 rose to **98.6s** — fewer sockets means *more* reuse. The instinct to "back off the fan-out" is
exactly wrong here, and §12.6's generous ~24-wide default is **vindicated**, not implicated.

**These knobs do NOT fix it** (all measured, all still ~18 timeouts): `connections: 64`, `pipelining: 0`,
`keepAliveTimeout: 1`, and a `Connection: close` header. undici reuses a socket the moment it is **free**,
and none of those govern that path. Only genuinely declining reuse works.

**The rules (LOCKED):**

1. **Every provider call gets its own dispatcher**, destroyed when the call ends. One TLS handshake
   (~0.1s) per call is **under 1%** of a ~22s describe — it buys back a 27% failure rate and ~3x the wall
   time. This is a fixed, non-negotiable cost, not a tunable.
2. **Our `AbortController` owns the one deadline** (`timeoutForBytes`, §3.5 rule 4). The dispatcher's
   `headersTimeout` / `bodyTimeout` are disabled so there is **one** clock, not three racing ones.
3. **Scoped to provider calls only.** The IPFS client talks to `127.0.0.1` at low concurrency and is
   untouched — this is a remedy for *this* origin's behavior under fan-out, not a global policy.
4. **A timeout here is infrastructure, never a verdict about the file** — it stays `transient` (§3.5).

Verified on the real app path (the same module the queue calls), 48 files, 24-way: **47 ok, 1 permanent
refusal (§3.5), zero timeouts, wall 23s, peak event-loop lag 485ms** — against 18 timeouts and 155s before.
The one failure is a genuine model refusal, not a stall.

The fresh-dispatcher-per-call rule is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` —
`freshProviderDispatcher()`, applied in `postJsonOnce()` (passed as `dispatcher`, torn down in `finally`).
Shared by **all** adapters, since every provider call funnels through that one seam.

### 3.7 The thinking budget — the alias moved and we started paying for reasoning (NEW, LOCKED)

`DEFAULT_GEMINI_MODEL` is the **`gemini-flash-latest` alias**, chosen in §3.4 so a retirement can never hard-break
the app. That choice is right and stands — but it has a **cost side nobody was watching**: Google hot-swaps
the alias forward, and it now resolves to **`gemini-3.5-flash`**, a **thinking** model that reasons before
answering unless told not to. The day Google moved the alias, this app silently began paying for reasoning
it never asked for. **No config changed; the behavior changed underneath us.**

Describing what is in a picture is **perception, not deduction**. Measured on this corpus: **~1,400 thinking
tokens per image**, p50 **27s → 16s** with thinking off, and **output length unchanged** (~1,200 tokens).
The thinking tokens buy nothing here and cost latency on every file of a 483-file batch.

**This is a speed/cost rule, not the timeout fix** — §3.6 was the failure. It is worth ~40% of the wall
time of every batch and ~676k thinking tokens on a 483-file run.

**The rules (LOCKED):**

1. **Send `thinkingConfig.thinkingBudget: 0`** on describe calls.
2. **It must self-heal, because the alias moves.** A future Flash could *require* a non-zero budget and
   reject ours with a `400` — that would break **every** describe, far worse than the tokens saved. So a
   `400` naming the thinking config latches the option **off for the process** and retries the call plain.
   Self-healing beats a pin someone has to notice and edit — the same reasoning that chose the alias in §3.4.

The zero thinking budget and its self-healing fallback are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` —
`NO_THINKING` + `looksLikeThinkingRejected()`, applied in the `gemini` adapter's `describe()`.

## 4. The prompts — shipped defaults + per-computer customize

Each media kind has **its own prompt**, checked into the repo beside the describe module
(`describe/prompts/image.md` and `describe/prompts/video.md`). The prompt is what makes the output
**hyper-detailed and consistent** rather than a vague caption.

### 4.1 What the prompts ask the model to produce

Both prompts open by insisting on a **HYPER-DETAILED, factual, non-editorializing** description that marks
uncertain guesses `(uncertain)`, then require the model to write **prose under a fixed set of headings in a
fixed order**:

* **Overview** — a short plain-English "at a glance" summary of what the file is / what happens (2–4
  sentences for an image; 3–6 for a video, including its genre/purpose — interview, vlog, drone footage…).
* **Hyper-Detail** — the deep-dive section: exhaustively describe everything visible (every object,
  surface, texture, small detail), not just the headline subject. This is where the most words go.
* **Composition** — explicit **FRAME-LEFT / FRAME-CENTER / FRAME-RIGHT** placement plus foreground /
  midground / background, with a **percentage-of-frame** estimate and approximate **distance/direction**
  for notable elements (e.g. "a coffee shop ~25% at frame-left; a mailbox ~9% frame-center, ~100 ft
  forward"), separating where the **people** are from where the **non-people things** are; plus balance,
  leading lines, rule-of-thirds, depth of field, and where the eye is drawn.
* **Location** — the model's best-guess **CITY, STATE/REGION, and COUNTRY** (a single best city-NAME guess
  even if `(uncertain)`), a **city-size class** (major/big, medium, small town, suburb, rural), and
  location specifics, with reasoning (signage language, architecture, plates, flora, landmarks), or
  `(indeterminate)`.
* **Setting & Place** — state clearly **INSIDE (interior)** vs **OUTSIDE (exterior)** and the kind of
  space; identify any building/structure (name if recognizable, else type + architectural style/era); for
  exteriors, describe surroundings the way a **movie location scout** would (share-of-frame + distance).
* **Lighting** — state plainly **DAYTIME vs NIGHTTIME** (or dawn/dusk/golden hour/overcast), then the
  lighting design (natural vs artificial, key/fill/back, hard vs soft, high-key/low-key, color temperature,
  practicals in frame).
* **Cinematography** — classify the shot the way a **cinematographer** would: framing / **shot size**
  (extreme close-up, close-up, medium close-up, **cowboy shot** / medium, medium-wide, wide, extreme-wide /
  establishing), camera angle (eye-level, low, high, overhead / bird's-eye, dutch), and lens character.
* **People** — for **EACH** person in frame, **hyper-describe** them: apparent age range and height/build;
  the **full outfit head-to-toe** (garments, colors, materials, footwear, accessories, headwear, eyewear);
  hair and distinctive features; and their **placement and size in frame** with posture, gaze, and
  expression. "No people in frame." when empty.
* **Objects & Text** — notable objects, brands, and any legible **text/signage quoted verbatim**.
* **Colors & Mood** — dominant palette and overall tone.

The **video** prompt adds two video-specific headings: a **Shot-by-shot / Timeline** (approximate timecode
ranges × what's on screen × camera movement × cuts/transitions/graphics) and an **Audio & Speech** section
that *paraphrases* key lines but explicitly **does not transcribe verbatim** — the Transcription tab covers
that — and it asks the People section to note **wardrobe changes across the clip**.

### 4.2 Per-computer override (customize / save / reset)

The prompt used for a kind is the **shipped default** unless the user has **customized** it. Customizing
**copies** the shipped default into a **per-computer override** under the state dir
(`<stateDir>/describe/prompts/<kind>.md`); once that copy exists, it is the prompt that is sent. **Save**
writes edited override text; **Reset** deletes the override and reverts to the shipped default. Nothing here
touches the network — it only resolves *which text* is passed to an adapter.

Resolving the in-use prompt (per-computer override if present, else shipped default) and the customize / save / reset flow are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts.ts` — `getPrompt()` / `promptView()` / `customizePrompt()` / `savePrompt()` / `resetPrompt()`; shipped defaults `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts/image.md` and `video.md`.

`DescribeResult.status` reports each generation truthfully: **`described`** (success, with the model),

refused — and `overwrite` not set), **`needs_setup`** (no placement root yet — the wizard, §3.5),
**`no_provider`** (no adapter for the kind — add a key), **`unsupported`** (not an image/video), or

(incl. `needs_setup`) + `failed` === `results.length`. **`rejected` is its own count, never folded into
`skipped`** — "skipped" means *we* didn't ask, and a refused file was asked and answered
([processing_batches.mdx §4.2](./processing_batches.mdx)). Locked by
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe-counts.spec.ts`.

## 7. Charter & privacy posture — the one deliberate network path (LOCKED)

This is the **single place** in LargeFileBridge that reaches the network **on purpose**. Generating a
description **uploads the file's bytes** (inline base64) to the chosen third-party provider — Google, xAI,
or OpenAI. That is unavoidable for a hosted vision model, so it is fenced by strict rules:

* **Always explicit, always user-initiated.** A description is generated **only** on a deliberate
  **Generate** (or **Regenerate**) click. It **never** runs automatically — not on a scan, not on a
  background pin, not on viewing a file.
* **Entirely separate from perceptual fingerprinting.** The charter's **perceptual / content-fingerprint
  matching** (PhotoDNA-style) is a *different* feature that must run **entirely locally and must never
  phone home**. AI description is **not** that feature and must never be confused with it — fingerprinting
  stays local; description is the one opt-in upload.
* **Honest about the upload in the UI.** The generate pane states plainly that the file is uploaded to the
  provider (and, for video, that the provider is Gemini), so the user is never surprised about bytes
  leaving the machine.
* **No key → no upload.** With no provider configured, nothing can be sent (the action is disabled, §8).

The adapter module is annotated as the one deliberate outbound path, and the upload happens only inside an explicit generate, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — the `describe()` methods (called only from `describeOne()` in `describe.service.ts` on an explicit request).

## 8. Empty, error & disabled states

* **No description yet** — the tab shows a **generate pane**: an icon, "No AI description yet," a blurb
  that names the upload (and Gemini for video), and a **Generate description** button.
* **No provider configured** — the **Generate description** button stays **clickable**; clicking opens the
  **credentials-missing popup** (**Close** / **Instructions**), and **Instructions** opens the
  `/ai-credentials` setup page in a new tab (path + placeholder YAML for
  `~/.config/GoogleCloud/apikey.yaml`, plus the Settings/env alternatives). A **video** with only
  image-capable keys hits the same popup at runtime (`no_provider`, because video needs Gemini). See
  **[ai_credentials.mdx](./ai_credentials.mdx)**.
* **Unsupported kind** — a non-image/-video path returns **`unsupported`**; audio never offers AI
  description (its tab is Transcription).
* **Over the 18MB inline cap** — the file is **auto-compressed to a temporary copy under 17.5MB** and that
  copy is uploaded (§3.3); the original is untouched. Generation only **`failed`**s here when the
  compression tool is missing (ffmpeg/ImageMagick — the reason names the `brew install` to run) or the
  file genuinely can't be squeezed under the cap.
* **Provider error / block / timeout** — a provider HTTP error, a Gemini safety block, non-JSON, or a
  120s timeout comes back as **`failed`** with the reason, surfaced in the toast; the existing description
  (if any) is untouched. A **retired/unknown model** 404 is rewritten into an actionable reason that names
  the bad model and the recommended one (§3.4). **Every failure is written to `error.err`** — the app's
  fault trail at `<stateDir>/error.err` (WARN/ERROR/FATAL), tagged `[describe]` with the file,
  `provider=`, and `kind=` for diagnosis. Logged by `describeOne()`'s `catch` via `log.error` in
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts`;
  the two rolling logs (`log.log` all-levels, `error.err` faults) are written by
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/logging.ts`.
* **Already described** — without `overwrite`, a second generate returns **`skipped` ("already
  described")**; **Regenerate** passes `overwrite` to replace it.

## 10. The provider-readiness gate — `withDescribeReady` (LOCKED — mirror of `withModelReady`)

```
       - confirmed  → enqueue the confirmed describe set (Processing shows it)

```

### 12.4.1 The popup's **Videos / Images** kind filter (NEW, LOCKED)

An AI description applies to **two** media kinds — **videos** and **images** — and a user very often wants
only **one** of them. A folder of 1,800 photos and 12 videos offers no way to say *"just do the videos"*
short of unchecking 1,800 rows by hand, so today the user either describes everything or gives up. The
popup therefore carries a **kind filter** directly above its candidate list.

**The filter.** A row labelled **"Filter:"** sits at the top of the popup's right pane (above the candidate
list) and holds exactly **two checkboxes**:

* **Videos** — **checked / on by default**.
* **Images** — **checked / on by default**.

Both start on, so the popup's default behavior is **exactly what it is today** (every candidate listed);
the filter only ever **narrows**.

**What a filter does.** Unchecking a kind **removes those files from the candidate list on the right** —
they disappear from view. **The filter is the applied set, not a view convenience**: a file that is **not
in the right-hand list is not described**. Turn **Videos** off and the videos vanish from the list and
**none of them is described** when the user confirms; the same holds for **Images**. This is the LOCKED
guarantee — *what you see in the list is exactly what Confirm will process*, so a filtered-out file can
never be silently swept into the batch.

**How the rest of the popup follows the filter.**

* **The live count and the button.** The confirm button's live count ([dialogs.mdx §5.1](./dialogs.mdx))
  counts only **visible, checked** rows — filtering the videos out decrements *"Describe N files"* by the
  video count on the spot. The header's *"N of M selected"* reads against the **visible** set the same way.
* **Confirm is disabled at zero visible-checked.** Unchecking **both** kinds empties the list; the popup
  stays open with nothing to do and the button is disabled — never a silent no-op batch.
* **The filter is not destructive to a row's checkbox.** Hiding a kind does not uncheck its rows; turning
  the kind back on restores them **checked exactly as they were**, so a user can flip between kinds while
  auditing without losing per-row decisions.
* **"Select all" / "Clear"** act on the **visible** rows only — they mean *"all of what I am looking at"*.
* The filter is a **client-side view over the plan's candidates** — it never re-fetches the plan and never
  changes the server-side narrowing (scope, drop-unsupported, drop-already-done all still happen upstream,
  §12.5). A filtered-out file is simply not in the confirmed set.

**Where it applies.** Every describe popup: the **Describable metric tile** (§12.2), the blue **"Describe ›"
header primary** (§12.3), and the unified batch popup from the page action-links row and the ⋮ / right-click
menu ([dialogs.mdx §5–§6](./dialogs.mdx)) — they are the one same popup, so they all carry the one same
filter. The transcribe popup carries the **same mechanism** with **its** kinds (audio / video) — the
generic capability lives in the popup, not in describe ([warnings.mdx §4.5.4](./warnings.mdx)).

The kind filter (the "Filter:" checkbox row, the visible-set narrowing, and the visible∩checked apply set) is
implemented in code:

### 12.5 An already-described file is NEVER re-offered and NEVER re-described (LOCKED)

This is the **"1,953 files" scenario** the product owner called out: in the File System browser the user
opens a directory's **`⋮` → Create AI descriptions**, and the popup reports a large number of candidates
(e.g. `1953`). **Every file that already has a `done` `.ai_description` must be dropped before the popup
ever lists it** — so the candidate list is *only* the not-yet-described files, every listed row is checked
by default with a clear conscience, and pressing **"Describe N files"** can **never re-run a description
that already exists**. Confirming the popup a second time on the same tree re-scans and finds **fewer** (or
zero) candidates, never re-processes the ones already done.

The skip is enforced **on the server, in both the preview and the enqueue**, so the popup's list and the
later commit can never disagree:

* **The plan/preview drops already-done** ([dialogs.mdx §5.2](./dialogs.mdx)): the walk collects the
  scope's image/video files, then **drops any file that already reads back a `done` description** (and any
  non-image/-video file) **before** returning the candidate list — so an already-described file is **never
  a checked row** in the popup.
* **The enqueue drops already-done again** (§6 skip-already-done, [page_actions.mdx §1.2](./page_actions.mdx)):
  even if a description was written **between** the preview and the Confirm (another computer's pull, a
  concurrent run), the enqueue re-checks and **queues only the still-missing files** — the `alreadyDone`
  count rises and those files are silently skipped, never re-described.
* **A REFUSED file is settled too** ([§2.3](#23-when-the-provider-refuses--ai_description_rejected-new-locked)).
  A file carrying a `.ai_description_rejected` is dropped by the same guards, for the same reason: the
  provider has already answered, and re-queuing it spends a real call to be told the same thing.
* **Overwrite is the only exception.** Re-describing an already-done file — or asking the provider to
  reconsider a refusal — happens **only** when the user explicitly asks to overwrite (`overwrite: true`),
  never as a side effect of confirming the batch popup.

Both the preview and the enqueue apply the identical drop-already-described filter in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` —
`previewDescribe()` and `enqueueDescribe()` (the
`if (!overwrite && !dp.needsSetup && (readDescription(abs) || exists(dp.rejectedPath))) { alreadyDone++; continue; }`
guard in each). **Note the `|| exists(dp.rejectedPath)` clause** — it is the whole point of the bullet above, and the
quotation here previously omitted it, showing the pre-refusal guard as if it were current.

#### 12.5.1 …and the popup SAYS how many it dropped (NEW, LOCKED)

The filter above is necessary but **not sufficient**, because it is invisible. Re-open a tree you have run
before and the popup reports a large candidate count with no mention of the skipped files — indistinguishable
from a popup that is about to redo everything. The user then has no way to audit it short of counting
`.ai_description` sidecars on disk by hand. **So the popup states the excluded count** in its sub line
whenever `alreadyDone > 0` ([dialogs.mdx §5.2.1](./dialogs.mdx)) — *"91 files already have an AI description
and are excluded from this list."*

**A large candidate count is not, by itself, evidence that the filter is broken.** The honest reading of
`1,900 candidates / 91 already done` is that **91** files have a `done` sidecar and **1,900** genuinely do
not — the tree is mostly un-described. Before treating a big number as a re-offer bug, check the **ground
truth** rather than the recollection of an earlier run:

* **Count the sidecars** the filter actually reads: `rg --files -g '*.ai_description' <placement-root>`
  (for a file out in the wider filesystem, that root is the owning storage's dedicated repo, **not** beside
  the media — §3.4 / [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)).
* **Read the preview's own log line** — `previewDescribe()` logs
  `preview [root …]: N considered → M candidates (K already done, U unsupported)` to `~/T/_large_files_bridge/log.log`.
  If `K` equals the sidecar count on disk, the filter is behaving.
* **A prior run that "did most of them" may not have.** Video describes **fail** when the file cannot be
  compressed under the provider's inline cap (§3.3) and images fail on provider errors; a failed file
  writes **no** sidecar and is **correctly** offered again. `grep '\[describe\]' error.err` shows the real
  outcome of the earlier run.
* **A third-party index is not our state.** An external YAML/CSV that happens to carry an `ai_description`
  column (a planning file, an inventory) is **not** what this filter reads and may disagree wildly with the
  sidecars. Our source of truth is the `done` sidecar at the resolved placement path — nothing else.

### 12.6 Describe is GENEROUSLY parallel — no hardware bottleneck (LOCKED — read with §12.6.1)

> **Corrected by [§12.6.1](#1261-but-no-hardware-bottleneck-is-only-true-for-a-file-that-fits-new-locked--corrects-126).**
> The "no hardware bottleneck" premise below holds **only for a file already under the size cap**. An
> oversize file transcodes first, so the wide fan-out must be gated for that path. Read both.
>
> **The ~24-wide default is VINDICATED by [§3.6](#36-a-provider-call-must-never-reuse-a-pooled-socket-new-locked--corrects-35),
> but "the provider's rate limit" below is NOT the ceiling that was actually being hit.** Measured: 24-way
> sustained for 3 minutes (221 calls) produced **zero** throttling, and **lowering** concurrency to 8 made
> throughput *worse*. The real ceiling was Node's own connection pool. Do not "fix" a describe stall by
> narrowing this number — that is the wrong direction.

Transcription and compression are **local CPU/GPU** work: Whisper and ffmpeg saturate cores and RAM, so
their queue caps are **core-budgeted** and deliberately narrow ([parallelization.mdx §3](./parallelization.mdx)).
**AI description is different.** Its heavy wait is a **vision-API network round-trip**, not local compute —
the machine sits idle while the provider thinks. So describe is **not** core-bound and **should fan much
wider** than transcribe/compress: a large batch overlaps **many** in-flight uploads at once instead of
trickling a few files through a core budget.

* The describe queue bucket runs at a **fixed, generous concurrency** — **default ~24**, tunable via the
  `LFB_DESCRIBE_CONCURRENCY` env var — **independent of the core budget**, because it is bounded by network
  latency and the provider's rate limits, not by CPU.
* This is the one bucket where "use more parallelism" is safe even on a small laptop: overlapping I/O costs
  nothing in cores. The only real ceiling is the **provider's rate limit** (a burst of `429`s surfaces as
  failing dock cards the user can retry), which is why the number is generous-but-bounded and env-tunable
  rather than unbounded.
* Transcribe and compress keep their narrow core-budgeted caps — **only** describe fans wide.

The per-bucket concurrency (describe = the generous fixed cap, transcribe/compress = core-budgeted) is set in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` —
`DESCRIBE_CONCURRENCY` (env-overridable) returned by `capFor()` for the `describe` bucket, alongside the
core-budgeted `transcribe` / `compress:image` / `compress:video` caps. Policy: [parallelization.mdx §3](./parallelization.mdx).

### 12.6.1 …but "no hardware bottleneck" is only true for a file that FITS (NEW, LOCKED — corrects §12.6)

§12.6 fans describe out ~24-wide on one premise: *"the machine sits idle while the provider thinks"*, so
parallelism is free. **That premise is true only for a file already under the cap.** An **oversize** file
makes describe run a full **ffmpeg transcode** first (§3.3) — the most core-hungry work in the app. So a
~24-wide describe bucket silently meant **up to ~24 concurrent ffmpeg encodes**, exactly the `cores²`
over-subscription every other bucket is careful to avoid ([parallelization.mdx §3](./parallelization.mdx)).
The bucket that was documented as *"the one bucket where more parallelism is safe even on a small laptop"*
was, on a video-heavy tree, the **heaviest** thing running.

§12.6's other escape hatch — *"a burst of 429s surfaces as failing dock cards the user can retry"* — also
did not hold: the failures arrived as **120s timeouts**, not `429`s, and there was **no retry** at all
(§3.5). Both halves of the safety argument were assumptions rather than observed behavior.

**The correction (LOCKED).** Keep §12.6's *intent* — the fan-out stays wide, because it is right for the
**network** — and gate the part that was never network work:

* **The network fan-out is unchanged** — ~24 describes may be **in flight** at once.
* **The transcode is core-budgeted** — only `coreBudget() / VIDEO_THREADS_PER_ENCODE` of them may be
  **running ffmpeg** at any moment; the rest wait at the gate rather than fighting for cores.
* **A file that fits never touches the gate.** It has no transcode, so it costs nothing and waits for
  nothing — the common case keeps the full generous parallelism §12.6 intended.

The rule this locks in general: **a queue cap may only be called "network-bound" if every path in that job
is network-bound.** One conditional CPU branch inside an "I/O bucket" is enough to starve the box.

The transcode gate (wide network fan-out, core-budgeted encodes) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/fit-media.ts` — `acquireTranscodeSlot()` (the semaphore) held across `fitVideoUnderLimit()` / `fitImageUnderLimit()` inside `fitMediaUnderLimit()`, sized from `coreBudget()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts`.

### 12.7 The 2,000-file run — async end-to-end, watchable, never a server freeze (LOCKED)

This is the **"2,000 AI descriptions" scenario** the product owner walked through: open a directory with
**~2,000 describable files**, hit **Confirm** in the batch popup, watch them queue up, then — in a **second
browser** — open **`/processing`** to watch progress. Every link in that chain is **asynchronous and
non-blocking**, and the whole run must stay **fully watchable at 2,000+ items with no performance problem**.
The invariants, each cross-referenced to where it is owned:

1. **Confirm returns immediately.** The popup Apply calls `POST /api/describe/enqueue`, which resolves the
   eligible set, pushes it onto the background queue, and **returns the plan** — it never runs a single
   description inline (§12.4, [job_queue.mdx §2](./job_queue.mdx)). The popup closes at once; the 2,000 jobs
   drain in the background.
2. **The workers drain async, ~24 at a time.** The queue runs the `describe` bucket at the generous fixed
   concurrency (§12.6), and **every heavy child process is async** (§3.3.1) — so 24 in-flight uploads/
   transcodes **never freeze the event loop**. The server keeps answering `GET /api/progress` throughout,
   which is exactly why the second browser's Processing page **loads and starts moving**.
3. **Every file is visible on `/processing`.** Running descriptions show as **Running** rows/cards (each
   `describeOne` registers a `track("describe", …)` job), the not-yet-started ones show as **Pending** rows
   and a per-op **Waiting** count, and any failure shows as a **Failed** row with its reason
   ([processing.mdx §4.3](./processing.mdx)). Nothing about the run is invisible.
4. **The Processing page stays light at 2,000+ items.** The poll payload is **bounded** (the pending rows
   are capped, the failures are capped — [processing.mdx §5](./processing.mdx)) while the **counts stay
   exact**, and the per-item table **windows its rows** (only the on-screen rows are in the DOM —
   [performance.mdx P-01](./performance.mdx)). So a 2,000-file backlog is a **small, fast** poll and a
   **fixed-cost** render, not a 2,000-row DOM dump ([processing.mdx §4.4](./processing.mdx)).
5. **Any tab, any browser, wakes on its own.** Because the queue/registry state is **process-global** and
   every surface polls the one `GET /api/progress`, an **idle** Processing page opened in a **different
   browser** that did nothing itself **starts moving within one poll interval**
   ([processing.mdx §4.5](./processing.mdx)) — the behavior the product owner exercised by refreshing the
   page in a second browser.

The end-to-end async path (enqueue-and-return → async ~24-way drain → registry cards → bounded, windowed
Processing view) is the composition of the code cited in §3.3.1, §12.4, §12.6, and
[processing.mdx §4–§5](./processing.mdx) / [job_queue.mdx §2–§4](./job_queue.mdx). The concrete tour is
[use_case_ai_description_at_scale.mdx](./use_case_ai_description_at_scale.mdx).

## 15. Acceptance criteria

1. The media viewer's bottom band shows **Transcription** and **AI description** tabs, each with a **✓**
   when its content exists; **video** shows both, **image** shows only AI description, **audio** shows only
   Transcription. The default tab **prefers Transcription** when present, else the first tab with content,
   else the first tab.
2. AI description works on **image + video only** (audio → transcription). The description is stored
   **inside the owning root's committed tracking area**, path-mirrored, under the media's **full filename
   with `.ai_description` appended** — resolved by the shared ordered placement rule, **no** `analysis/`
   sub-directory and **no** `.transcribe/` directory. It is a self-contained `done` YAML record
   `{ source, status, engine, provider, generated, kind, description }` — so the Storage table's
   **`description` indicator** lights up from that artifact's presence. Because it lives in that committed
   area, it is **committed and travels with the repo** in every placement case — **no**
   `*.ai_description` git-ignore line is injected (that nudge is retired); appending the extension (rather
   than replacing it) means two files differing only by extension no longer collide.
2a. **The base is keyed on storage KIND** ([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx),
   the owner of the rule), resolved through the single choke point `trackingBaseDir(root)` — **no** code path
   joins `.lfbridge` directly:
   * **Working git repo** (kind `repo`, the default) → **keeps** `.lfbridge/`:
     `render/skyline.mp4` → `<repo>/.lfbridge/render/skyline.mp4.ai_description`.
   * **Dedicated LFB file repo** (an SDL — kind `personal` / `company` / `community`) → **no `.lfbridge/`
     at all**, the mirror hangs off the **root**: `~/_Mirror/…/x.mp4` →
     `~/BGit/Bryan_git/personal_large_files_bridge/_Mirror/…/x.mp4.ai_description` (**not**
     `…/personal_large_files_bridge/.lfbridge/_Mirror/…`).
   * An SDL's own `.git/` **never** demotes it to a working repo — the walk-up classifies by **type first**,
     then falls back to "has `.git/` → working repo".
3. Providers are the pluggable adapters **Gemini / OpenAI / Grok**; **only Gemini describes video**;
   selection is requested → configured-default → first available that supports the kind (preference Gemini,
   OpenAI, Grok). Keys resolve **config → env → (Gemini only) the shared `~/.config/GoogleCloud/apikey.yaml`**
   (Gemini `GEMINI_API_KEY`/`GOOGLE_GENAI_API_KEY`/`GOOGLE_API_KEY` then the shared file's `apiKey`/`4k_apiKey`;
   Grok `XAI_API_KEY`/`GROK_API_KEY`; OpenAI `OPENAI_API_KEY`). Uploads are inline base64 (~18MB bound); a
   file over **17.5MB** is first **compressed to a temporary copy that fits** (H.264 video keeping
   resolution when possible, else stepping down ~25%; JPEG image) and that copy is uploaded — **the
   original is never modified**; generation only fails over-cap when ffmpeg/ImageMagick is missing.
4. The per-kind **prompts** (shipped `image.md` / `video.md`) drive a hyper-detailed description with the
   fixed headings in order: overview → **hyper-detail** → composition (**frame-left/center/right** +
   %-of-frame + distance) → location (**city/state/country + city-size class**) → setting & place
   (inside/outside, building, movie-scout exteriors) → **lighting (day/night)** → **cinematographer shot
   classification** → per-person **head-to-toe** description → objects/text → colors & mood; video adds
   shot-by-shot timeline + audio paraphrase. The user can **customize** a prompt into a per-computer
   override and **reset** it.
5. The REST surface (`GET /providers`, `GET /file`, `POST /file`, `GET /prompt`, `POST /prompt/customize`,
   `PUT /prompt`, `DELETE /prompt`) is allow-list-gated and mounted at `/api/describe`; each generation
