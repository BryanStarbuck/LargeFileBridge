# LargeFileBridge ‚Äî AI description of images & videos

`LargeFileBridge` can generate a **hyper-detailed, searchable description** of an **image** or a **video**
so the file can be **found, understood, and re-found later without opening it**. A vision model is shown
the file and a carefully written per-kind prompt, and returns structured prose ‚Äî an overview, the
composition, a best-guess city/country, building/place identification, inside-vs-outside, a
cinematographer's shot classification, and a head-to-toe description of every person in frame. That text
is stored beside the file's other analysis so it powers search and a presence indicator.

Two things make this feature worth pinning down: **where the description is stored** (unified with the
storage-level media analysis ‚Äî ¬ß2), and **the charter/privacy posture** (¬ß7 ‚Äî this is the *one* place
LFBridge uploads a file to the network on purpose, always by an explicit click, and it must not be
confused with the local-only perceptual-fingerprint feature).

Siblings this builds on and does not repeat:

* [Transcribe](./Transcribe.mdx) ‚Äî the sibling media-analysis feature. AI description mirrors its shape (a
  file/prompt/provider surface, an explicit-user-action generate, a presence-cached result) and shares its
  **owning-storage-root** resolution. The two are the **two tabs** of one analysis panel (¬ß1).
* [media_viewer.mdx](./media_viewer.mdx) ‚Äî the **View one file** viewer whose bottom band hosts the
  Transcription ¬∑ AI description tabs. This file is **authoritative** for the AI-description tab.
* [storages.mdx ¬ß6](./storages.mdx) ‚Äî the per-file media-analysis area. This file **supersedes** ¬ß6's
  `description.yaml` sketch: the description is now a **`.ai_description` sidecar beside the media**, not a
  YAML under `.lfbridge/analysis/` (¬ß2).
* [os_open.mdx](./os_open.mdx) ‚Äî the neighboring localhost-only viewer action; unrelated to the network,
  but shares the viewer's action surface.
* [ai_credentials.mdx](./ai_credentials.mdx) ‚Äî the **credential resolution order** (config ‚Üí env ‚Üí the
  shared `~/.config/GoogleCloud/apikey.yaml`) and the **credentials-missing popup** (Close /
  Instructions ‚Üí the `/ai-credentials` setup page). Authoritative for how a key is found and how the user
  is told to add one.

* **Which tabs show** depends on the media kind: **Transcription** shows for **audio + video** (media that
  *has* speech); **AI description** shows for **image + video** (media that is *visual*). A **video**
  therefore shows **both** tabs; an **image** shows **only AI description**; an **audio** file shows **only
  Transcription** (audio is deliberately *not* AI-described ‚Äî transcription covers "what was said").
* **The ‚úì marker.** Each tab carries a small **‚úì** when its content already exists (an existing transcript,

* **The default-tab rule.** On open, the panel selects, in order: (1) **Transcription if it has content**;

## 2. Where the description is stored ‚Äî a `.ai_description` sidecar beside the media (LOCKED)

The description does **not** live in the Local state dir and there is **no** `.lfbridge/analysis/`
directory for it. Exactly like the transcript, it is a **sidecar written beside the media** ‚Äî the media
file's own base name with its extension **replaced** by **`.ai_description`**:

```
<placement root>/
‚îî‚îÄ‚îÄ render/
    ‚îú‚îÄ‚îÄ skyline.mp4                              ‚Üê the real media file
    ‚îî‚îÄ‚îÄ skyline.ai_description                   ‚Üê the generated description, right next to it
```

For a file at **`render/skyline.mp4`** the description is written to
**`<placement root>/render/skyline.ai_description`** ‚Äî same folder, `.mp4` ‚Üí `.ai_description`, no
intermediate directory. (Example: `TheFunnyVideo.mp4` ‚Üí `TheFunnyVideo.ai_description`.) The **placement
root** and the media's mirrored relative hierarchy are resolved **exactly like transcription ‚Äî by the
shared ordered placement rule** ([Transcribe.mdx ¬ß3.4](./Transcribe.mdx), the authoritative spec): walk
**up** to the nearest ancestor carrying `storage.yaml` / `.lfbridge/` / `.git/`; else route to the **owning
storage's dedicated Git repo** when enabled (mirroring the file's hierarchy ‚Äî
`<dedicatedRepo>/<rel-without-ext>.ai_description`, **not** git-ignored there); else the owning storage's
root; else the **first-time setup wizard** ([Transcribe.mdx ¬ß3.5](./Transcribe.mdx)) when no Personal
storage exists; else, as a last resort, the file's own directory. **Get AI details / AI description is one
of the actions that triggers the first-time wizard**, and it consults placement (`needsSetup`) before it
runs. Pasted `~/‚Ä¶` paths are home-expanded first (Transcribe.mdx ¬ß3.6).

> **Collision caveat (accepted).** As with `.transcription`, replacing the extension means two files that
> differ **only** by extension in the same folder map to the same `.ai_description` and the second run
> overwrites the first ‚Äî the deliberate tradeoff of extension-based, no-extra-directory naming.

### 2.1 The YAML shape ‚Äî a self-contained `done` record

The `.ai_description` sidecar is a small YAML document. Generating a description writes it directly as a
**done** record (there is no separate pre-seeded skeleton to upgrade ‚Äî the sidecar either exists as a
`done` record or does not exist yet):

```yaml
source: render/skyline.mp4
status: done
engine: gemini-flash-latest       # the model id sent (alias ‚Üí newest GA Flash)
provider: gemini                  # the adapter id
generated: 2026-07-05T09:41:00Z   # ISO timestamp
kind: video                       # image | video
description: |
  ## Overview
  A wide establishing shot of a rain-slicked city street at dusk‚Ä¶
  ## Composition ‚Ä¶ ## People ‚Ä¶
```

A read returns a usable `DescribeView` **only** when the sidecar exists with `status: done` **and** a
non-empty `description` string ‚Äî otherwise it reads back as **null** (no content yet), so the tab shows the
generate pane rather than an empty box.

### 2.2 The Storage table indicator

The **Storage file table's `analysis: ["description"]` indicator lights up** the moment the
`.ai_description` sidecar exists beside the media ‚Äî one source of truth for "this file has a description,"
shared by the viewer tab and the storage table (which now detects the sidecar rather than a YAML under
`.lfbridge/analysis/`). When the artifact lands in a **plain** repo (placement rule A) generating also
ensures that repo git-ignores **`*.ai_description`** (the same nudge transcription makes for
`*.transcription`) so descriptions never bloat commits; when it lands in the owning storage's **dedicated
Git repo** (rule B) it is **deliberately not** git-ignored ‚Äî that repo exists to track and back up the
artifacts.

Resolving the description path via the shared ordered placement rule, reading back a done description, and writing the `.ai_description` sidecar (with the git-ignore nudge only when the placement flags it) are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts` ‚Äî `resolveDescriptionPath()` / `readDescription()` / `describeOne()` (sidecar path via `siblingArtifactPath()` + `AI_DESCRIPTION_EXT` in `storage/artifact-placement.service.ts`).

The storage-table indicator that detects the `.ai_description` sidecar is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` ‚Äî `analysisOutputs()`.

## 3. The provider adapters ‚Äî key resolution, supported kinds, the video rule, the size cap

Vision providers are a small **pluggable set of adapters**. Which one runs is chosen by (a) an explicit
request, else (b) the configured default, else (c) the **first adapter that has a key AND supports the
media kind**. Preference order is **Gemini ‚Üí OpenAI ‚Üí Grok**.

| Adapter | Provider | Image | Video | Endpoint / model | Notes |
| --- | --- | --- | --- | --- | --- |
| `gemini` | Google Gemini | ‚úì | **‚úì** | `generativelanguage.googleapis.com/v1beta/models/<model>:generateContent`, default `gemini-flash-latest` (alias ‚Üí newest GA Flash, currently Gemini 3.5 Flash) | The **only** adapter that describes **video** (inline `inline_data` base64). |
| `openai` | OpenAI | ‚úì | ‚Äî | `api.openai.com/v1/chat/completions`, default `gpt-4o` | OpenAI-compatible chat/vision; `image_url` data URI. |
| `grok` | xAI Grok | ‚úì | ‚Äî | `api.x.ai/v1/chat/completions`, default `grok-2-vision-1212` | OpenAI-compatible chat/vision; images only. |

### 3.1 The video = Gemini-only rule (LOCKED)

Only **Gemini** accepts a **video**. OpenAI and Grok describe **images only**. So when the file is a video
and no Gemini key is configured, generation returns **`no_provider`** with the honest reason *"no AI
provider configured for video ‚Äî add an API key for Gemini (only Gemini describes video)."* An image can be
described by **any** of the three.

### 3.2 Key resolution (config ‚Üí env ‚Üí shared Gemini file)

Each adapter resolves its key from the **app config** (`ai.<provider>.api_key` in the local `config.yaml`)
**first**, then falls back to well-known **environment variables**, so a machine that already exports a key
"just works":

* **Gemini** ‚Äî `GEMINI_API_KEY` / `GOOGLE_GENAI_API_KEY` / `GOOGLE_API_KEY`, then the **shared Gemini key
  file** (below).
* **Grok** ‚Äî `XAI_API_KEY` / `GROK_API_KEY`
* **OpenAI** ‚Äî `OPENAI_API_KEY`

**Gemini has a THIRD source (LOCKED):** after config and env, it reads the **shared Google/Gemini key
file** `~/.config/GoogleCloud/apikey.yaml` ‚Äî the **SAME file** the `~/BGit/all/tools` Gemini / "nano
banana" image tools already use. It has two fields; we prefer **`apiKey`**, then fall back to
**`4k_apiKey`** (matching the tools' vision path). So a machine set up for those tools describes
video/images with **no extra setup**. The full credential-resolution + the missing-key UX live in the
dedicated **[ai_credentials.mdx](./ai_credentials.mdx)**.

Reading the shared Gemini key file (prefer `apiKey`, else `4k_apiKey`; absent/placeholder/invalid ‚Üí no key) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/google-apikey-file.ts` ‚Äî `loadGoogleApiKey()`; wired into the `gemini` adapter's `apiKey()` in `adapters.ts`.

### 3.3 The inline cap & compress-to-fit (LOCKED)

All three providers are called with the file **inline as base64**. Because base64 inflates bytes by ~4/3
and the providers bound inline requests (~20MB), the bytes we send are capped at **18MB**. Rather than
**fail** an over-size file (a 44MB phone video is completely normal), we **compress it to fit first** and
upload the compressed copy ‚Äî the description feature must "just work" on real videos.

**The rule:** before upload, ensure the media is **at or under 17.5MB** (a deliberate margin under the
18MB cap). A file already under the target is uploaded **untouched**. A file over it is transcoded to a
**temporary** copy under the state dir; that copy is uploaded and then **deleted**. **The user's original
file is NEVER modified** ‚Äî this is separate from the charter's explicit, user-initiated compression
feature; it is an ephemeral upload-only proxy.

**Video ladder (charter algorithm):** *keep the pixel resolution the same if high compression alone can
get it under the target; only if that fails, reduce the resolution ~25% each step and repeat.*

1. **CRF at original resolution** ‚Äî H.264 (`libx264`, `-preset medium`, `-crf 28`, `yuv420p`, faststart,
   AAC 96k audio). If the result is ‚â§ 17.5MB, **keep full resolution**.
2. **Harder CRF at original resolution** ‚Äî one more pass at `-crf 34` (still full resolution) before we
   give up any pixels.
3. **Step the resolution down** ‚Äî reduce the height by **25%**, **snapping to a standard height**
   (2160/1440/1080/900/720/540/480/360/288/240) when one is **within 3%** of the reduced value, and
   re-encode at `-crf 30`. Repeat down the ladder to a **240p floor**.
4. **Two-pass guarantee** ‚Äî if even the floor rung is over, a **two-pass** encode at the floor resolution
   to an **exact target bitrate** (computed from the clip duration and the byte budget) deterministically
   lands it under the cap.

**Codec choice (LOCKED):** we compress to **H.264 in mp4**, not the "better-compressing" HEVC/AV1, on
purpose ‚Äî the provider's decoder must accept whatever we send, and **H.264/mp4 is the one format every
vision provider reliably decodes**. Audio is kept but re-encoded low (96k, or 64k in the two-pass
fallback) so it can't eat the byte budget.

**Image path:** an over-size image is re-encoded to a temporary **JPEG** at descending quality (85 ‚Üí 72 ‚Üí
60) and then descending scale until it fits. (The temp JPEG may drop a PNG's alpha ‚Äî acceptable because it
exists only to show the model; the original PNG is untouched.)

**Tooling:** video compression needs **ffmpeg** (+ ffprobe for duration/resolution); image compression
needs **ImageMagick**. When the required tool is absent and the file is over the cap, generation
**`failed`**s with an install hint (`brew install ffmpeg` / `imagemagick`) instead of silently uploading
nothing.

The compress-to-fit pipeline (the 17.5MB target, the resolution ladder, the CRF passes, and the two-pass
guarantee ‚Äî all to a temp file, never the original) is handled in code:

The adapter set, per-kind support, key resolution (config ‚Üí env), `selectAdapter` precedence, the inline base64 read with the 18MB cap, and `providerStatus` are handled in code:

### 3.4 Model lifecycle & deprecation self-heal (LOCKED)

Providers **retire** older model versions on a schedule. Google hard-retired `gemini-2.0-flash` on
**2026-06-01**; after that date `generateContent` returns **`404 NOT_FOUND ‚Ä¶ is no longer available`** and
a config still **pinned** to that id can NEVER succeed.

**The default is the `gemini-flash-latest` ALIAS (LOCKED), not a pinned id.** Google hot-swaps that alias
to its newest GA Flash (with a 2-week notice window), so the app **auto-tracks new releases and never
hard-breaks** on a retirement the way a pinned version does. As of **2026-07** the alias resolves to
**Gemini 3.5 Flash** (image + video). A user may still **pin a concrete id** in Settings ‚Üí AI if they want
a fixed model. Two further mechanisms keep even a pinned config working across retirements:

* **Auto-heal on load.** When `config.yaml` has a Gemini model in the **retired set**, the config loader
  rewrites it to the current default and persists it ‚Äî so an old machine's stale pin (`gemini-2.0-flash`)
  silently upgrades itself the next time the app reads config. A **valid custom** model the user chose is
  left untouched (only known-dead ids are rewritten).
* **Actionable error, not a raw 404.** If a call still fails on a retired/unknown model (e.g. a hand-typed
  bad id not in the retired set), the raw Google 404 is rewritten into a message that **names the bad
  model and the recommended one**: *"Gemini model "X" is not available ‚Äî the provider may have retired it.
  Update the model in Settings ‚Üí AI (recommended: gemini-flash-latest)."* That message is what the viewer
  shows and what lands in **`error.err`** (¬ß8).

The single source of truth for the current default + the retired-model set (so the adapter and the config
loader agree) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/models.ts` ‚Äî `DEFAULT_GEMINI_MODEL`, `RETIRED_GEMINI_MODELS`, `looksLikeModelRetired()`.

The auto-heal of a pinned retired model on config load is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` ‚Äî `getAppConfig()` (rewrites `ai.gemini.model` when it is in `RETIRED_GEMINI_MODELS`).

Rewriting a provider 404 into the actionable, model-naming error is handled in code:

## 4. The prompts ‚Äî shipped defaults + per-computer customize

Each media kind has **its own prompt**, checked into the repo beside the describe module
(`describe/prompts/image.md` and `describe/prompts/video.md`). The prompt is what makes the output
**hyper-detailed and consistent** rather than a vague caption.

### 4.1 What the prompts ask the model to produce

Both prompts open by insisting on a **HYPER-DETAILED, factual, non-editorializing** description that marks
uncertain guesses `(uncertain)`, then require the model to write **prose under a fixed set of headings in a
fixed order**:

* **Overview** ‚Äî a short plain-English "at a glance" summary of what the file is / what happens (2‚Äì4
  sentences for an image; 3‚Äì6 for a video, including its genre/purpose ‚Äî interview, vlog, drone footage‚Ä¶).
* **Hyper-Detail** ‚Äî the deep-dive section: exhaustively describe everything visible (every object,
  surface, texture, small detail), not just the headline subject. This is where the most words go.
* **Composition** ‚Äî explicit **FRAME-LEFT / FRAME-CENTER / FRAME-RIGHT** placement plus foreground /
  midground / background, with a **percentage-of-frame** estimate and approximate **distance/direction**
  for notable elements (e.g. "a coffee shop ~25% at frame-left; a mailbox ~9% frame-center, ~100 ft
  forward"), separating where the **people** are from where the **non-people things** are; plus balance,
  leading lines, rule-of-thirds, depth of field, and where the eye is drawn.
* **Location** ‚Äî the model's best-guess **CITY, STATE/REGION, and COUNTRY** (a single best city-NAME guess
  even if `(uncertain)`), a **city-size class** (major/big, medium, small town, suburb, rural), and
  location specifics, with reasoning (signage language, architecture, plates, flora, landmarks), or
  `(indeterminate)`.
* **Setting & Place** ‚Äî state clearly **INSIDE (interior)** vs **OUTSIDE (exterior)** and the kind of
  space; identify any building/structure (name if recognizable, else type + architectural style/era); for
  exteriors, describe surroundings the way a **movie location scout** would (share-of-frame + distance).
* **Lighting** ‚Äî state plainly **DAYTIME vs NIGHTTIME** (or dawn/dusk/golden hour/overcast), then the
  lighting design (natural vs artificial, key/fill/back, hard vs soft, high-key/low-key, color temperature,
  practicals in frame).
* **Cinematography** ‚Äî classify the shot the way a **cinematographer** would: framing / **shot size**
  (extreme close-up, close-up, medium close-up, **cowboy shot** / medium, medium-wide, wide, extreme-wide /
  establishing), camera angle (eye-level, low, high, overhead / bird's-eye, dutch), and lens character.
* **People** ‚Äî for **EACH** person in frame, **hyper-describe** them: apparent age range and height/build;
  the **full outfit head-to-toe** (garments, colors, materials, footwear, accessories, headwear, eyewear);
  hair and distinctive features; and their **placement and size in frame** with posture, gaze, and
  expression. "No people in frame." when empty.
* **Objects & Text** ‚Äî notable objects, brands, and any legible **text/signage quoted verbatim**.
* **Colors & Mood** ‚Äî dominant palette and overall tone.

The **video** prompt adds two video-specific headings: a **Shot-by-shot / Timeline** (approximate timecode
ranges √ó what's on screen √ó camera movement √ó cuts/transitions/graphics) and an **Audio & Speech** section
that *paraphrases* key lines but explicitly **does not transcribe verbatim** ‚Äî the Transcription tab covers
that ‚Äî and it asks the People section to note **wardrobe changes across the clip**.

### 4.2 Per-computer override (customize / save / reset)

The prompt used for a kind is the **shipped default** unless the user has **customized** it. Customizing
**copies** the shipped default into a **per-computer override** under the state dir
(`<stateDir>/describe/prompts/<kind>.md`); once that copy exists, it is the prompt that is sent. **Save**
writes edited override text; **Reset** deletes the override and reverts to the shipped default. Nothing here
touches the network ‚Äî it only resolves *which text* is passed to an adapter.

Resolving the in-use prompt (per-computer override if present, else shipped default) and the customize / save / reset flow are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts.ts` ‚Äî `getPrompt()` / `promptView()` / `customizePrompt()` / `savePrompt()` / `resetPrompt()`; shipped defaults `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/prompts/image.md` and `video.md`.

## 7. Charter & privacy posture ‚Äî the one deliberate network path (LOCKED)

This is the **single place** in LargeFileBridge that reaches the network **on purpose**. Generating a
description **uploads the file's bytes** (inline base64) to the chosen third-party provider ‚Äî Google, xAI,
or OpenAI. That is unavoidable for a hosted vision model, so it is fenced by strict rules:

* **Always explicit, always user-initiated.** A description is generated **only** on a deliberate
  **Generate** (or **Regenerate**) click. It **never** runs automatically ‚Äî not on a scan, not on a
  background pin, not on viewing a file.
* **Entirely separate from perceptual fingerprinting.** The charter's **perceptual / content-fingerprint
  matching** (PhotoDNA-style) is a *different* feature that must run **entirely locally and must never
  phone home**. AI description is **not** that feature and must never be confused with it ‚Äî fingerprinting
  stays local; description is the one opt-in upload.
* **Honest about the upload in the UI.** The generate pane states plainly that the file is uploaded to the
  provider (and, for video, that the provider is Gemini), so the user is never surprised about bytes
  leaving the machine.
* **No key ‚Üí no upload.** With no provider configured, nothing can be sent (the action is disabled, ¬ß8).

The adapter module is annotated as the one deliberate outbound path, and the upload happens only inside an explicit generate, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` ‚Äî the `describe()` methods (called only from `describeOne()` in `describe.service.ts` on an explicit request).

## 8. Empty, error & disabled states

* **No description yet** ‚Äî the tab shows a **generate pane**: an icon, "No AI description yet," a blurb
  that names the upload (and Gemini for video), and a **Generate description** button.
* **No provider configured** ‚Äî the **Generate description** button stays **clickable**; clicking opens the
  **credentials-missing popup** (**Close** / **Instructions**), and **Instructions** opens the
  `/ai-credentials` setup page in a new tab (path + placeholder YAML for
  `~/.config/GoogleCloud/apikey.yaml`, plus the Settings/env alternatives). A **video** with only
  image-capable keys hits the same popup at runtime (`no_provider`, because video needs Gemini). See
  **[ai_credentials.mdx](./ai_credentials.mdx)**.
* **Unsupported kind** ‚Äî a non-image/-video path returns **`unsupported`**; audio never offers AI
  description (its tab is Transcription).
* **Over the 18MB inline cap** ‚Äî the file is **auto-compressed to a temporary copy under 17.5MB** and that
  copy is uploaded (¬ß3.3); the original is untouched. Generation only **`failed`**s here when the
  compression tool is missing (ffmpeg/ImageMagick ‚Äî the reason names the `brew install` to run) or the
  file genuinely can't be squeezed under the cap.
* **Provider error / block / timeout** ‚Äî a provider HTTP error, a Gemini safety block, non-JSON, or a
  120s timeout comes back as **`failed`** with the reason, surfaced in the toast; the existing description
  (if any) is untouched. A **retired/unknown model** 404 is rewritten into an actionable reason that names
  the bad model and the recommended one (¬ß3.4). **Every failure is written to `error.err`** ‚Äî the app's
  fault trail at `<stateDir>/error.err` (WARN/ERROR/FATAL), tagged `[describe]` with the file,
  `provider=`, and `kind=` for diagnosis. Logged by `describeOne()`'s `catch` via `log.error` in
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/describe.service.ts`;
  the two rolling logs (`log.log` all-levels, `error.err` faults) are written by
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/logging.ts`.
* **Already described** ‚Äî without `overwrite`, a second generate returns **`skipped` ("already
  described")**; **Regenerate** passes `overwrite` to replace it.

## 9. Acceptance criteria

1. The media viewer's bottom band shows **Transcription** and **AI description** tabs, each with a **‚úì**
   when its content exists; **video** shows both, **image** shows only AI description, **audio** shows only
   Transcription. The default tab **prefers Transcription** when present, else the first tab with content,

   **`description` indicator** lights up from the sidecar's presence. In a plain repo `*.ai_description` is
   git-ignored; in a dedicated repo it is committed and backed up.
3. Providers are the pluggable adapters **Gemini / OpenAI / Grok**; **only Gemini describes video**;
   selection is requested ‚Üí configured-default ‚Üí first available that supports the kind (preference Gemini,
   OpenAI, Grok). Keys resolve **config ‚Üí env ‚Üí (Gemini only) the shared `~/.config/GoogleCloud/apikey.yaml`**
   (Gemini `GEMINI_API_KEY`/`GOOGLE_GENAI_API_KEY`/`GOOGLE_API_KEY` then the shared file's `apiKey`/`4k_apiKey`;
   Grok `XAI_API_KEY`/`GROK_API_KEY`; OpenAI `OPENAI_API_KEY`). Uploads are inline base64 (~18MB bound); a
   file over **17.5MB** is first **compressed to a temporary copy that fits** (H.264 video keeping
   resolution when possible, else stepping down ~25%; JPEG image) and that copy is uploaded ‚Äî **the
   original is never modified**; generation only fails over-cap when ffmpeg/ImageMagick is missing.
4. The per-kind **prompts** (shipped `image.md` / `video.md`) drive a hyper-detailed description with the
   fixed headings in order: overview ‚Üí **hyper-detail** ‚Üí composition (**frame-left/center/right** +
   %-of-frame + distance) ‚Üí location (**city/state/country + city-size class**) ‚Üí setting & place
   (inside/outside, building, movie-scout exteriors) ‚Üí **lighting (day/night)** ‚Üí **cinematographer shot
   classification** ‚Üí per-person **head-to-toe** description ‚Üí objects/text ‚Üí colors & mood; video adds
   shot-by-shot timeline + audio paraphrase. The user can **customize** a prompt into a per-computer
   override and **reset** it.
5. The REST surface (`GET /providers`, `GET /file`, `POST /file`, `GET /prompt`, `POST /prompt/customize`,
   `PUT /prompt`, `DELETE /prompt`) is allow-list-gated and mounted at `/api/describe`; each generation
   reports truthfully (`described` / `skipped` / `no_provider` / `unsupported` / `failed`).
6. Settings ‚Üí Tools shows the **provider-key status** (configured / no-key, with the env var / config key
   hint and the "only Gemini describes video" note) and the **per-kind prompt editor** (Save customize /
   Reset).
7. Generation is the **one deliberate external-network path**: it **uploads** the file, is **always**
   explicit and user-initiated (never automatic), states the upload in the UI, and is **entirely separate**
   from the charter's local-only perceptual-fingerprint feature (which must never phone home). With no
   provider configured, nothing is uploaded.
