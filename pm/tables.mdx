# LargeFileBridge — Tables (bookmarks, filters, multi-level sort & persistence)

Every screen in LFB is built from the same **TanStack `DataTable`** ([menus.mdx](./menus.mdx) — see its
preamble). The charter locks the shared shape: a search box, one **filter** icon, one **sort** icon
above the table; 500/page; click-a-header to sort; full-page-height. **This file owns the pieces of
that shape that are the same on every table** so the per-page specs don't restate them:

1. The leading **bookmark** toggle (§1).
2. The **filter model** — value-visibility checkboxes on an always-visible facet rail, and the
   **one-facet-per-axis** rule that keeps Pin (IPFS) and Git-ignore permanently apart (§2).
3. The **multi-level sort** model, its dropdown UI, and its chip run (§3).
4. **Per-user, per-page persistence** of sort + filter (§4).

The shared control row (search box, filter icon, sort icon), 500/page pagination, click-a-header sort, and full-page-height are all implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component.

**Reference implementation:** [Repos](./repos.mdx) §3 + §8 already implements the control row and the
bookmark toggle in full detail; this file **generalizes** that pattern to every table and adds the
multi-level sort and persistence. Where Repos and this file agree, Repos is the worked example; where
this file adds mechanics (priority slots, persistence), it is the source of truth.

Applies to (every primary/listing table):

* [Repos](./repos.mdx) — the repo list (`/`). *(Reference implementation.)*
* [Full-paths](./full_paths.mdx) — the flat **files** table of large files under a folder (`/fs/paths`).
* [Communities](./communities.mdx) — the communities list (`/communities`).
* [Task tabs](./task_tabs.mdx) — a per-repo screen where the visible column SET (not just widths) changes
  with the active task tab, on top of this spec's per-user sort/filter persistence.
* Every future listing table inherits this by default.

> The Finder-style **column** browser ([directories.mdx](./directories.mdx)) is a column view, not a
> flat listing table; the bookmark toggle and multi-level sort apply to the **flat tables** above.
> Where a column view exposes a sort, it uses the same multi-level model and persists the same way.

## 1. The leading bookmark (favorite) toggle (LOCKED)

Every primary table's **leading (column 0)** cell is a **ribbon bookmark** toggle — the exact control
[Repos §8](./repos.mdx#8-bookmark-favorite--the-leading-ribbon-toggle) defines. Generalized:

* **Glyph & states** — lucide **`Bookmark`** (the banner/ribbon shape). **Solid yellow** = bookmarked;
  thin **gray outline** (white fill) = not. One `<Bookmark>` with `fill` toggled.
* **It is a control cell** — clicking it flips `bookmarked` for that row and **never navigates**
  (exactly like the trailing `⋮` kebab). Also offered in the row `⋮` menu as *"Bookmark / Remove
  bookmark"*.
* **Accessible** — `<button aria-pressed={bookmarked} aria-label="Bookmark this <entity>">`,
  keyboard-focusable, toggles on `Enter`/`Space`, tooltip *"Bookmark" / "Bookmarked"*.
* **Local, per-computer preference.** A bookmark is a favorite *this machine's user* set — **not**
  shared to peers, **never** in the shared manifest. It survives rescans (discovery reconciles
  without clobbering user fields). Default **`false`**.

### 1.1 Where the flag is stored, per entity type (LOCKED)

The flag lives **with the entity**, in the local state tree ([storage_local.mdx](./storage_local.mdx)):

| Table | Entity | `bookmarked` stored in |
| --- | --- | --- |
| Repos | a repo | `pin/r/<repo>/config.yaml` (Repos §8.3 — already LOCKED). |
| Communities | a community | `pin/c/<community_id>/config.yaml` (communities.mdx §8). |
| Full-paths (files) | a large file | a small **bookmark set** in the **owning pin unit's** `config.yaml` (the repo's `pin/r/<repo>/config.yaml`, or `pin/computer/config.yaml` for files outside any repo), keyed by the file's **path + fingerprint** — because a file has no `config.yaml` of its own. |

* **API shape** mirrors Repos: `POST /api/<entity>/:id/bookmark` with `{ bookmarked: boolean }`,
  idempotent, writes the flag and returns the updated row; unknown id → 404. For files the id is the
  path/fingerprint key.
* The scan **never** touches bookmarks — user fields are preserved across re-registration.

Persisting a bookmark toggle to the entity's `config.yaml` (the Repos reference implementation of the `POST /:id/bookmark` endpoint) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — the `POST /:repoId/bookmark` route (via `updateRepoConfig()` in `store-model/units.service.ts`).

## 2. Filtering — the value-visibility checkbox model (LOCKED)

This section replaces the old "each filter row is an operator select + value input" dropdown model
everywhere. It is the **single filter grammar for every table in the product**.

**The old model and why it is retired.** Until now the shared filter popover offered exactly one
control for a categorical column: a **`<select>` dropdown** whose options were a hardcoded `any` plus
one option per value. That control is wrong for this product on three counts:

1. **It hides the vocabulary.** The user cannot see what values exist without opening a dropdown
   inside a dropdown. A filter you must go spelunking for is a filter nobody uses.
2. **It only expresses `any` or `exactly one`.** There is no way to say *"show pinned and undecided
   but not ignored"* — the most common real request — because a select is one-of-N.
3. **It invites conflation.** A single select labelled "Decision" is the reason pin state and
   git-ignore state got welded into one facet (§2.4).

### 2.1 A facet is its values, each with a checkbox, all checked by default (LOCKED)

Every categorical (enum / boolean / toggle) filter is a **facet**: the flat list of the values that
column can hold, **each with its own checkbox**, and **every box starts checked**.

* **All checked = the facet is inactive.** No constraint, every row shows. This is the resting state,
  and it is what a fresh table looks like.
* **Unchecking a value hides those rows.** Nothing else. The checkbox means *"show me rows whose value
  is this"* — a **visibility** switch per value, not a selection of a single value.
* **Checkboxes within a facet OR together; facets AND together.** *(Pin: Pinned ✓ Undecided ✓,
  Ignored ✗)* AND *(Git-ignore: all ✓)* → every pinned-or-undecided row regardless of git-ignore.
* **Never offer an `any` / `all` pseudo-value.** "All" is not a value a row can hold; it is the state
  of every box being checked. An `any` option in a value list is the select model leaking through.

This is the product owner's rule verbatim: *"one might be yes, one might be no. Those could be two
[boxes] because sometimes they want to show both, so they leave both checked. They won't only want to
see one, so they'll turn off the checkbox of the other one."*

```
 Pin (IPFS)   ☑ Pinned   ☑ Not pinned   ☑ Recommended   ☑ Undecided
 Git-ignore   ☑ Ignored  ☑ Not ignored  ☑ Recommended
```

**The last-value rule.** Unchecking the **final** checked box in a facet is **not allowed** — it can
only ever produce a guaranteed-empty table, which is a dead end, not a filter. The last checked box
renders **disabled** with the tooltip *"At least one value must be shown."* (Users who want an empty
result do not exist; users who blank a table by accident and cannot see why do.)

The shared filter surface — today a `<select>`-per-enum popover, to be rebuilt as the facet rail (§2.2) — is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (the `showFilter` popover and its generic `filterFn`).

### 2.2 The facet rail — filtering without opening a dropdown (LOCKED)

**A filter the user cannot see is a filter the user does not use.** So the facets do not live inside
the filter dropdown; they live **on a rail directly above the table**, always visible, always showing
their current state.

```
 ┌──────────────────────────────────────────────────────────────────────────────┐
 │ [ 🔍 search…                    ]                                    ⇅   ⛛   │  ← control row
 │ Pin (IPFS)  ☑ Pinned ☑ Not pinned ☑ Undecided   ·   Git-ignore  ☑ Ignored ☑ Not ignored │  ← facet rail
 └──────────────────────────────────────────────────────────────────────────────┘
 │ ▮ │ File              │ Size  │ Pin │ Ignore │ Status │ …
```

* **Each facet is a labelled group of checkboxes** on the rail, rendered as **toggle chips** (a chip
  whose checked state is a filled/outlined pill). Checked = filled. The label sits left of its values.
* **The rail states the vocabulary.** A user reading the rail learns what values exist and what is
  currently hidden without a single click. Discovery and control are the same surface.
* **A facet with an inactive state is quiet, not gone.** All-checked facets render in a muted style so
  the eye lands on the facets that are actually cutting rows; they never disappear (disappearing
  facets would re-hide the vocabulary).
* **Overflow.** Facets whose values do not fit collapse into a `+N more` chip that opens the popover
  (§2.3) containing only the overflowed facets. Priority order for what stays on the rail follows the
  same **responsive priority** the columns use (§4a) — the rail narrows the way the table does.
* **Reset.** Each facet's label carries a **`n of m`** counter and an **All** link when any of its
  boxes is unchecked; the rail carries one **Clear filters** link when any facet is active.

The facet rail is a **new** shared surface; it is to be implemented in code alongside the existing control row:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (a new facet-rail region above the table body, driven by the same `columnFilters` state as the popover).

### 2.3 What survives in the filter (`⛛`) popover (LOCKED)

The charter's single **filter icon** stays — its job changes from *"the place filters live"* to
**the overflow and the long tail**:

* **Free-text `contains`** filters (name, path, CID) — a text box has no value vocabulary to show, so
  a rail chip would be meaningless. These stay in the popover.
* **Numeric thresholds** (`Size ≥`, `Peers ≥`) — same reason: an open-ended number, not a value set.
* **Overflowed facets** at narrow widths (§2.2).
* The icon stays **colored when any filter is active** — *including* rail facets, so the icon remains
  the single honest "something is filtered" indicator. Applying any filter resets to page 1.

### 2.4 One facet = one axis — Pin and git-ignore are NEVER the same filter (LOCKED)

**Add-to-IPFS (pin) and Add-to-git-ignore are two independent booleans**
([decisions.mdx §1](./decisions.mdx)) — a file can be pinned *and* git-ignored, either, or neither, and
*"both-off is a real answer."* Therefore:

* **There are always two facets, never one.** A **Pin (IPFS)** facet and a **Git-ignore** facet. They
  sit as separate labelled groups on the rail, and are never merged, nested, or interleaved.
* **No facet may be named "Decision."** The name is what caused the damage: a facet labelled
  *Decision* offering `sync / ignore / undecided` reads as if it covers both axes when it only ever
  covered the **IPFS** axis — where `ignore` means *"deliberately not pinned,"* **not** *"git-ignored."*
  That facet is **retired**; its replacement is the **Pin (IPFS)** facet (§2.4.1). The `sync` string
  stays a **frozen wire value** on disk ([decisions.mdx §7](./decisions.mdx)) — this is a **label and
  facet** change, not a data change.
* **The two facets read from two different sources, and that is why they cannot merge.** The pin axis
  is the ledger's reconciled projection; the **git-ignore axis is read from `git check-ignore`, never
  folded out of the decision ledger** ([decisions.mdx §1.1](./decisions.mdx), LOCKED — a ledger-sourced
  flag *"reports 'not ignored' for files git plainly ignores… a visible falsehood about the user's own
  filesystem"*). A single facet would have to pick one source and lie about the other.
* **This applies to sort identically** (§3) and to the `⋮` menus ([menus.mdx §5.6](./menus.mdx)): the
  IPFS group and the git-ignore group are separate groups, never one "decision" group.

#### 2.4.1 The two facets, fixed vocabulary (LOCKED)

| Facet | Values (each a checkbox, all checked by default) | Source of truth |
| --- | --- | --- |
| **Pin (IPFS)** | ☑ **Pinned** (`sync`) · ☑ **Not pinned** (`ignore` — deliberately not pinned) · ☑ **Recommended** (the calc engine suggests pinning) · ☑ **Undecided** (never asked) | the decision ledger's IPFS axis ([decisions.mdx §7](./decisions.mdx)) |
| **Git-ignore** | ☑ **Git-ignored** · ☑ **Not git-ignored** · ☑ **Recommended** (the calc engine suggests ignoring) | **`git check-ignore`**, live ([decisions.mdx §1.1](./decisions.mdx), LOCKED) |

These match the **four-state toggle grammar** of the Pin and Ignore columns
([decision_toggles.mdx §1](./decision_toggles.mdx)) — a facet's values are exactly the states its
column's toggle can show, so the rail and the cells speak one vocabulary. The **N/A** toggle state is
**not** a facet value (a row where the axis does not apply is filtered by the axis's own facet only
when that axis exists on the row).

Deriving the git-ignore axis from git itself (never from the ledger), which is what forces these into two facets, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `composeFileRows()` (via the batched `checkIgnore()`).

### 2.5 Checkbox vs. radio vs. segmented — pick by the question (LOCKED)

| Control | Use it when | Examples |
| --- | --- | --- |
| **Checkbox set** (§2.1) — **the default for every filter facet** | The question is *"which values do I want to see?"* and any combination is meaningful. This is nearly always true of a filter, because a filter is a predicate over a value set. | Pin (IPFS), Git-ignore, Status, Kind, Mode, Tracked, Compressed |
| **Radio set** | The question is genuinely **one-of-N** — the options are *modes*, not row values, and "both" is meaningless. | A view mode; a sort **direction** (asc ⇄ desc) |
| **Segmented control** | A radio set of **2–3 short options** where the choice is a *mode* and horizontal space is cheap. A segmented control is a radio in different clothes — **it does not qualify as a filter facet.** | (none remaining after §2.6) |

**The rule of thumb:** if the user could ever plausibly want to see **two of the options at once**, it
is a **checkbox set** — full stop. Every existing "Both · X · Y" segmented *filter* fails this test:
"Both" is only there because the control could not express "X and Y," which a checkbox set expresses
natively and more clearly (§2.6).

### 2.6 Retiring the `Both · X · Y` segmented filters (LOCKED)

The Full-paths table ([full_paths.mdx §3](./full_paths.mdx)) pioneered visible, no-dropdown filtering
with segmented controls — **the instinct was right and this spec generalizes it**; only the control is
wrong. Each three-segment filter collapses to a two-value checkbox facet, where the old **Both**
segment *is* the all-checked resting state:

| Old segmented filter | New facet |
| --- | --- |
| Compressed: `Both · Compressed · Uncompressed` | **Compressed** — ☑ Compressed ☑ Uncompressed |
| IPFS: `In IPFS · Not · Both` | **Pin (IPFS)** — the §2.4.1 vocabulary |

This is a strict gain: the two-value facet expresses everything the three-segment control did (both
boxes = Both; one box = either segment) **and** it stops the value vocabulary from growing a fake
member. It also removes the inconsistency where one table filtered with segments and every other table
filtered with selects.

### 2.7 The promoted "Bookmarked only" filter (LOCKED)

Bookmarked-only is the one facet that stays a **single checkbox** rather than a value set, because it
is a **special-cased promotion**, not a column vocabulary:

* **`☐ Bookmarked only`** — when **on**, the table shows **only bookmarked rows**; when off, no
  bookmark constraint. It ANDs with every other facet, resets to page 1, and colors the filter icon.
* It is **promoted to the head of the facet rail** (leftmost, visually separated) — previously the head
  of the filter dropdown — because favoriting-then-filtering-to-favorites is the primary way a user
  carves a big list down to what they care about.
* **It is deliberately not a two-value facet** (☑ Bookmarked ☑ Not bookmarked). "Show me only my
  favorites" is a single, high-frequency intent with an obvious off state; the two-value form would
  cost a chip and buy nothing. It is the documented **exception** to §2.1, and the only one.
* This is the **filter** counterpart to the **sort** default (§3.4): sort floats bookmarks to the top
  of the *full* list; this filter hides everything *except* bookmarks. Independent — either or both.

The "Bookmarked only" promotion (today pinned to the top of the filter popover, to move to the head of the rail) is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (the `id === "bookmark"` special case).

### 2.8 A facet's values must be declared — an empty facet is a bug (LOCKED)

A column declared as a filterable enum **must** ship its value vocabulary. A facet with no declared
values currently renders as a select containing only `any` — a dead control that silently filters
nothing. Under the rail model it would render as a **label with no chips**, which is more visibly
broken, but it is a bug either way and **must fail review**:

* Every enum facet declares its complete value list.
* The facet's **accessor and its cell must read the same field** — a facet that filters on a different
  value than the column displays is a lie about the row.
* Known live offenders to fix when the rail lands: the one-repo **Status** facet and the
  storage-detail **Kind** facet (the latter also has an accessor/cell mismatch).

### 2.9 The promoted "Large files only" filter (LOCKED)

Every table whose rows are **files** — the one-repo task tabs ([one_repo.mdx](./one_repo.mdx),
[task_tabs.mdx](./task_tabs.mdx)), **Full paths** ([full_paths.mdx](./full_paths.mdx)), **Processing**
([processing.mdx](./processing.mdx)), and the **File System** file listings
([file_system.mdx](./file_system.mdx)) — carries a **second promoted control**, a sibling of
"Bookmarked only" (§2.7), pinned at the head of the facet rail:

* **`☑ Large files only`** — when **on**, the table hides rows the scan admitted **only** as small,
  not-git-ignored **analysis media** (the `analysisOnly` rows — [scan.mdx §4.1](./scan.mdx)); when
  **off**, every file the scan surfaced shows, at **every size**. It ANDs with every other facet,
  resets to page 1, and colors the filter icon.
* **It defaults ON.** The product's whole subject is *large* files, so a fresh file table opens on the
  large set. This is the deliberate opposite of §2.1's all-checked-is-inactive default: the promoted
  toggle is **active by default** and the user turns it **off** to widen the view — exactly like the
  File System page's existing `only_large: true` preference, which this generalizes.
* **The one documented default-OFF exception is the OCR tab** ([task_tabs.mdx §4.7](./task_tabs.mdx),
  [ocr.mdx §11.4](./ocr.mdx)). OCR's canonical targets — a screenshot, a receipt, a slide, a JPG — are
  almost always **below** the large-file threshold, so opening the OCR tab with "Large files only" ON
  would show an empty or near-empty table and hide the very files the user came to read. The OCR tab
  therefore seeds this toggle **off**; the control is still present and the user can turn it back **on**.
* **What "large" means here is NOT a raw byte compare.** A row counts as large (shown when the toggle is
  on) when it is anything **other** than an `analysisOnly` row — i.e. it is bridge payload (including a
  **git-ignored media file of any size**, which stays visible so the 5–100 MB clip case
  [repos.mdx §4.1](./repos.mdx) is never re-hidden) **or** a checked-in-nudge file (≥ the 50 MB
  checked-in threshold). Only the newly-surfaced small, not-git-ignored media hide behind the toggle.
* Like Bookmarked-only it is a **single promoted checkbox, not a two-value facet** — "show me only the
  big files" is one high-frequency intent with an obvious off state. It is the **second** documented
  exception to §2.1 (§2.7 is the first), and the last.
* **Persistence** — the toggle's state is saved per-user, per-page (and, on the task-tab page, seeded by
  the active tab's default) alongside the rest of the filter state (§4).

The backend flag that this toggle reads — `analysisOnly` on each `FileRow`, set for a small,
not-git-ignored media file admitted purely so it can be OCR'd / described / transcribed — is produced in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `walkUnit()` (the candidate predicate) and
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `composeFileRows()`.
The toggle itself is rendered by the shared table in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` (the promoted `largeOnly` control, sibling of the `bookmark` special case).

### 2.10 The File-type facet — Images · Videos · Audio · PDFs · Other (LOCKED)

Every file table carries a **File type** facet — the value-visibility checkbox model (§2.1) applied to the
file's **media class**, so a user can carve a mixed listing down to *"just the images"* (or *"just the
PDFs"*) in one click. This is what makes OCR-on-a-PDF reachable: on a tab or page showing many kinds, check
**PDFs** and the contracts stand alone.

**Where it lives: inside the Filter ⛛ dropdown, not on the rail (LOCKED).** The facet is a **File type**
section at the **top** of the filter popover (above the per-column filters, below any "Bookmarked only"). The
product owner's rule: *"when you click on that filter icon, the pop-up lists the kinds — the most common ones —
without having to expand."* The rail keeps only the promoted **Large files only** toggle (§2.9); file-type
filtering moved into the dropdown so the filter icon is the one place a user goes to narrow a table.

The section, top to bottom:

* **`☑ All kinds`** — a reset row. Checked ⇔ **nothing** is hidden. Clicking it (when some kind is hidden)
  **shows everything**. It cannot be *unchecked* directly — that would blank the table — so the user
  unchecks individual kinds below instead. This is the *"a checkbox that means all kinds"* the product owner
  asked for.
* **The four COMMON kinds, always listed without expanding: `Images · Videos · Audio · PDFs`.** These four
  render **whether or not** they are present in the data, so the user never has to expand to reach the kind
  they want (PDF included precisely because **OCR is often done on PDFs**). Unchecking a kind that is absent
  from the current rows is a harmless no-op.
* **`More types…` disclosure** → reveals the remaining classes — today just **Other** (source, docs,
  archives, data files). Collapsed by default; the label toggles to **Fewer types**. This is the *"thing to
  click to get access to more types than that"*.

Values, defaults, and rules:

* **Values (each a checkbox):** **Images · Videos · Audio · PDFs · Other.** A row's value is derived from its
  name by the shared classifier `fileTypeForName()` — video / image / audio by extension family, `pdf` for
  `.pdf`, and **Other** for everything else.
* **Default checked:** **Images, Videos, Audio, and PDFs are ON; Other is OFF.** The non-media **Other**
  bucket is the opt-in the user *"adds"*. It is the **third** documented departure from §2.1's all-checked
  default (§2.7, §2.9 are the others); Other-off is the only box that starts unchecked.
* **The last-value rule still holds** (§2.1): the final **present** checked class cannot be unchecked (a
  common kind that is absent from the data is not guarded — unchecking it changes nothing).
* **The Filter ⛛ icon lights only when a PRESENT class is hidden** (or Large-only is on) — unchecking an
  absent kind is a no-op and must not falsely mark the table "filtered".
* **It composes by AND with the tab scope and every other facet** ([task_tabs.mdx §1.3.1](./task_tabs.mdx)).
  On a task tab the tab's `rowFilter` already narrows the media axis (OCR → images + video + PDF), so the
  facet only cuts within it — it never *widens* a tab's scope.
* **Persistence** follows §4 — saved per-user, per-page, storing the **unchecked** classes (`hidden:`) so a
  future new class stays visible to existing users.

The shared classifier both the facet and the media viewer route on lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/media.ts` — `fileTypeForName()` (image | video | audio | pdf | other), beside `mediaKindForName()`; `isPdfName()` is the PDF discriminator.
The facet is rendered inside the shared table's Filter dropdown:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` (the `fileTypeFacet` prop; `FILE_TYPE_COMMON` / `FILE_TYPE_EXTRA`, the `showMoreTypes` disclosure, the `FileTypeRow` component, and the `All kinds` reset).

## 3. Multi-level sort — primary / secondary / tertiary (LOCKED)

Sort is **not** a single column. It is an **ordered list of up to three sort keys**:

* **Primary** (1st), **Secondary** (2nd), **Tertiary** (3rd). Each key = **(column, direction)**.
* The table sorts by the primary key, breaking ties with the secondary, then the tertiary.
* Fewer than three keys is fine (one or two). More than three is not offered — three levels covers
  every real case and keeps the UI legible.

### 3.1 The sort dropdown UI (LOCKED)

Opening the **sort** icon reveals a two-side dropdown:

* **Left — the sortable columns.** Every sortable column, each with an **asc/desc** control
  (caret `↑`/`↓`). This is the existing list from the charter's sort dropdown.
* **Right — the three priority slots.** Three labeled markers per column row: **`1st` `2nd` `3rd`**
  (a compact priority picker on the right edge of each column's row). The marker that is filled shows
  that column's current priority; blank means the column is not in the active sort.

```
 Sort ▾
 ┌─────────────────────────────────────────────┐
 │ Column            Dir        Priority        │
 │ ───────────────────────────────────────────  │

 │   Name            ↑ asc       1st [2nd] 3rd  │  ← secondary
 │   Size            ↑ asc       1st  2nd [3rd] │  ← tertiary
 │   Modified        ↑ asc       1st  2nd  3rd  │  ← not in the sort
 │   …                                          │
 │ ───────────────────────────────────────────  │
 │                              [ Clear sort ]  │
 └─────────────────────────────────────────────┘
```

### 3.2 Assigning a priority — insert & cascade-demote (LOCKED)

Clicking a priority marker on a column **inserts that column at that position and shifts the others
down one**, so **no two columns ever share a priority**:

* Click **`1st`** on a column → it becomes the **primary**; the old primary drops to **secondary**,
  the old secondary to **tertiary**, and any old tertiary **falls out** of the sort (list capped at
  three). This is the user's rule verbatim: *"demote everything else down one, so there's no
  duplicate first priority."*
* Click **`2nd`** / **`3rd`** → same insertion at that slot; lower slots shift down, capped at three.
* A column can hold **only one** priority — assigning it a new slot **removes its old slot first**,
  then inserts (so re-ranking a column that is already in the sort just moves it; it never duplicates).
* Each key keeps its own **asc/desc**, toggled by its caret independent of priority.
* **`Clear sort`** empties the list, then the table falls back to the **default** (§3.4).

### 3.3 Header-click stays consistent (LOCKED)

The charter's **click-a-header-to-sort** still holds and drives the **same** state:

* Clicking a column **header** promotes that column to **primary** (`1st`) — cascade-demoting the
  others exactly as §3.2 — so a header click is a one-click "make this the primary sort."
* Clicking the **same** header again toggles that key's **asc ↔ desc** (it never cycles to unsorted;
  clear from the dropdown's **Clear sort**).
* The active primary column shows the caret (`↑`/`↓`) in its header; the dropdown and the headers
  share **one** sort state (a header-driven change shows in the dropdown and vice-versa).

Click-a-header-to-sort with the asc↔desc toggle and shared active-column caret, driving the same sort state as the dropdown, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (the header `toggleSort`).

### 3.4 The default sort — Bookmarks first (LOCKED default, user-changeable)

Out of the box, every bookmark-bearing table starts with:

* **Primary = Bookmarked, descending** → bookmarked rows float to the **top**.
* **Secondary = the table's natural sort** (Repos → repo name; Full-paths → path/name; Communities →
  name). **Tertiary** unset unless the table defines one.

This is only the **default**. The user can re-rank freely (§3.2) — e.g. drop Bookmarked out entirely
— and their choice is **saved** (§4) and restored next visit. The default applies only when the user
has no saved sort for that page.

### 3.5 Sorting without opening the dropdown — the active-sort chips (LOCKED)

The same objection that retires the filter dropdown (§2.2) applies to the sort dropdown: **the active
sort is state the user should be able to read, not state they must open a panel to discover.** The
1st/2nd/3rd priority picker (§3.1) is a good *editor* and stays — it just stops being the only way to
see or change the sort.

* **The active sort renders as an ordered chip run** on the control row, right-aligned, beside the
  sort icon: `Sort: [1 Bookmarked ↓] [2 Size ↓] [3 Name ↑]`. Only the **active** keys appear (at most
  three, §3), so the run is short by construction.
* **Each chip is live.** Click its **caret** → toggles that key's asc ⇄ desc. Click its **`×`** →
  removes that key and promotes the ones below it. **Drag** a chip → re-ranks it, driving the exact
  same insert-and-cascade-demote rule as the priority slots (§3.2).
* **Header click still sets the primary** (§3.3) and now shows its result immediately in the chip run.
  **Shift-click a header appends** that column as the **next** key (2nd, then 3rd) instead of
  promoting it to primary — the one-gesture way to build a multi-level sort without any panel.
* **The chips, the headers, and the dropdown are one state** (§3.3). The dropdown becomes the
  **overflow/discovery** surface: it is where the user finds a sortable column that is *not* currently
  in the sort, and where **Clear sort** lives.
* **The sort icon stays colored when a non-default sort is active**, unchanged.

The active-sort chip run is a **new** shared surface, sharing the sort state that today only the popover and headers drive; it is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx` — `DataTable` component (a chip run on the control row, calling the existing `assignPriority()` / `promoteToPrimary()`).

### 3.6 Sort obeys the axis split too (LOCKED)

The §2.4 rule is not filter-only. **Pin and git-ignore are two sortable columns, never one.**

* Sorting "by Pin" orders on the **IPFS axis**; sorting "by Ignore" orders on the **git-ignore axis**
  read from `git check-ignore`. There is no combined "Decision" sort key, for the same reason there is
  no combined Decision facet — the two keys have two different sources of truth (§2.4).
* Both toggle columns are **sortable and filterable** ([decision_toggles.mdx §5](./decision_toggles.mdx)).
  Their sort order follows the toggle grammar: **On → Recommended → Off → N/A** (the actionable states
  first), direction-flippable like any key.

## 4. Persistence — sort + filter saved per user, per page (LOCKED)

**Sort and filter state is saved for every page**, per user, and restored on load.

* **Where:** the logged-in user's personal config —
  `users/<email>/config.yaml`, under the existing **`tables.views`** map
  ([storage_local.mdx](./storage_local.mdx) §4, the tier for per-user "prefs / last view"). **Not** computer-wide,
  **not** in any pin unit — it is *this user's* view of *this page*.
* **Keyed by table id.** `tables.views.<tableId>` (`repos`, `files`, `communities`, …). Each entry
  stores the **ordered sort keys** and the **active filters** (including **Bookmarked only**, and the
  search text if we choose to persist it).

#### 4.1 Facets persist as the HIDDEN values, never the shown ones (LOCKED)

A facet is stored as the list of values whose box the user **un**checked — `hidden: [...]` — **not**
the list of checked values.

**Why this is LOCKED.** The two encodings look interchangeable and are not. Store `shown:` and the
day we add a new value to any enum — a new `RepoStatus`, a new pin state, a new media kind — every
user who ever touched that facet has a saved list that does not contain the new value, so the new
value is **invisible to them forever**, silently, with rows vanishing from their table for no reason
they can see. Store `hidden:` and a new value is **absent from the hidden list, therefore shown** —
which is the correct default (§2.1: all checked = inactive) and degrades safely. The same argument is
why an **empty/omitted facet key means all-checked**, and why an inactive facet writes **no key at
all** rather than a full list of its values.

```yaml
# users/<email>/config.yaml  (excerpt — see storage_local.mdx §4 for the full file)
schema_version: 1
tables:
  views:
    repos:                  # Repos table
      sort:                 # ordered — index 0 = primary, 1 = secondary, 2 = tertiary
        - { col: bookmarked, dir: desc }
        - { col: name,       dir: asc  }
      filters:
        bookmarked_only: false
        # no facet keys => every facet all-checked (inactive)
    files:                  # Full-paths (files) table
      sort:
        - { col: bookmarked, dir: desc }
        - { col: size,       dir: desc }
      filters:
        bookmarked_only: true
        facets:
          # TWO facets, never one (§2.4). Values listed here are the UNCHECKED (hidden) ones.
          pin:        { hidden: [ignore] }          # showing Pinned + Recommended + Undecided
          gitignore:  { hidden: [] }                # all-checked — equivalent to omitting the key
          compressed: { hidden: [compressed] }      # the "Uncompressed" backlog (old segmented control)
    communities:            # Communities table
      sort:
        - { col: bookmarked, dir: desc }
        - { col: name,       dir: asc  }
      filters:
        bookmarked_only: false
        facets:
          mode: { hidden: [block, full] }           # showing only Recommended
```

* **Migration.** Existing saved entries use the select model's single-value form
  (`mode: recommended`). On read, a scalar value `v` for facet `f` is interpreted as
  *"hidden = every value of `f` except `v`"* — the select's one-of-N semantics — and rewritten to the
  `hidden:` form on the next write. No user loses their view.
* **Free-text and numeric filters are unchanged** — they persist as the scalar they always were
  (`min_size_bytes: 104857600`, `path_contains: "render/"`); the `hidden:` shape is for **facets**
  only (§2.3).

* **Saved on every change.** Any sort re-rank, direction toggle, or filter change writes that table's
  `tables.views` entry (atomic write, debounced) — the user never has to "save a view."
* **Restored on load.** Opening a page applies its saved entry; if none exists, the **default**
  (§3.4 + no filters) applies and is **not** written until the user actually changes something.
* **Server mode** — because it is under `users/<email>/`, each allow-listed user carries their own
  per-page sort/filter; two users on one server do not clobber each other's views.
* **Distinct from the bookmark flag.** The `bookmarked` flag itself is a per-**entity**, per-computer
  fact (§1.1); the `tables.views` entry here is the per-**user** view (which sort, which filters). Different
  owners, different files.

## 4a. Responsive column priority (LOCKED, site-wide)

Every table **prioritizes its columns** and **hides the lowest-priority ones as the table narrows** so a
cell **never wraps to a second line**. This is a shared `DataTable` behavior — all tables inherit it. The
full model (the master column stack-rank, the LOWER-number-is-more-important convention, the pinned
identity/verdict columns that never drop, the min-width budget that drives it, and the rule that hidden
columns still appear in the Sort/Filter dropdowns) is specified once in
[repos.mdx §3.2.1](./repos.mdx#321-responsive-column-priority--drop-the-least-important-column-as-the-table-narrows-locked)
and applies to **every** table. Each table assigns a `priority` (+ optional `minWidth`) per column;
columns with no `priority` are pinned. Site-wide rule the product owner set: **Peers is the single
lowest priority (hides first), Size is second-lowest (hides next)**; a table's identity column (Name /
Repo / File) and its Status verdict are pinned. Implemented in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/DataTable.tsx`
(`useResponsiveColumns()`), typed on `LfbColumn` (`priority`, `minWidth`) in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/types.ts`.

## 5. Acceptance criteria

1. **Repos, the Full-paths files table, and Communities** each have a **leading ribbon bookmark
   toggle** (§1) — a control cell that flips `bookmarked` and never navigates, solid-yellow / gray-
   outline, mirrored in the row `⋮` menu, default `false`, stored with the entity per §1.1.
2. **No filter facet anywhere in the product is a `<select>` dropdown.** Every categorical filter is a
   **set of value checkboxes, all checked by default**, where all-checked = inactive; there is no
   `any` / `all` pseudo-value; unchecking the last checked box is disabled (§2.1). A grep for a
   `<select>` in the shared filter surface returns **zero** hits.
3. **The facets are visible without opening anything** — they render on the **facet rail** above the
   table, showing their vocabulary and current state at rest; the `⛛` popover holds only free-text
   `contains`, numeric thresholds, and narrow-width overflow, and stays colored whenever **any**
   filter (rail included) is active (§2.2–§2.3).
4. **Pin (IPFS) and Git-ignore are two separate facets on every table that filters either** — never
   merged, never nested, and **no facet is named "Decision"** (§2.4). Their vocabularies are exactly
   §2.4.1, the pin axis sourced from the ledger and the git-ignore axis from `git check-ignore`. The
   same split holds for sort keys (§3.6) and for `⋮` menu groups.
5. **No `Both · X · Y` segmented filter survives** — each is a two-value checkbox facet whose
   all-checked state *is* the old **Both** (§2.6). Segmented controls remain legal only for genuine
   one-of-N **modes**, never for filter facets (§2.5).
6. **Every filterable enum declares its full value vocabulary**, and its accessor and cell read the
   same field; an enum facet with no declared values fails review (§2.8).
7. **Sort is multi-level:** an ordered primary/secondary/tertiary list; the sort dropdown shows
   sortable columns (with asc/desc) on the left and **1st/2nd/3rd priority slots** on the right (§3.1).
8. **Assigning a priority inserts and cascade-demotes** so no two columns share a priority; a column
   holds only one priority; header-click promotes to primary and shares one state with the dropdown;
   Clear sort falls back to default (§3.2–§3.3).
9. **The active sort is readable without opening the dropdown** — an ordered chip run on the control
   row whose chips toggle direction, remove, and re-rank; **shift-click a header appends** a key;
   chips, headers, and dropdown are one state (§3.5).
10. The **default sort is Bookmarked-descending first, then the table's natural secondary** — applied
    only when the user has no saved sort; fully user-changeable (§3.4).
11. **Sort and filter persist per user, per page** in `users/<email>/config.yaml` `tables.views`, saved
    on every change and restored on load; the default is not written until the user changes something
    (§4).
12. **Facets persist as `hidden:` (the unchecked values), never `shown:`** — an omitted or empty facet
    key means all-checked, so a newly-added enum value is visible to existing users by default; legacy
    scalar values migrate on read (§4.1).

## 6. Tab-scoped column sets (LOCKED)

Beyond responsive width-based hiding ([repos.mdx §3.2.1](./repos.mdx#321-responsive-column-priority--drop-the-least-important-column-as-the-table-narrows-locked)),
a table may **switch which columns exist** based on a **task tab** above it. The per-repo files table does
this ([task_tabs.mdx §7](./task_tabs.mdx#7-how-a-tab-re-tunes-the-table-mechanism-locked)): the page defines
the **union** of all columns once, and the active tab selects and orders a **subset** (`columnIds`) plus a
**default sort** for that task. Rules:

* **Tab selection is presentation, not preference.** The active tab is ephemeral view state; it is **not**
  written to `config.yaml` (unlike this spec's per-user sort/filter persistence, §4, which still applies
  *within* whatever tab is active).
* **The two hiding mechanisms compose.** The tab picks the column *set*; the responsive priority rule then
  drops the least-important of *those* as width shrinks.
* **Sort/filter still reach columns the tab hides.** As with responsive hiding, a column absent from the
  active tab is simply not shown; switching tabs re-projects the same underlying rows (no refetch).

Applied in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` (reads `TASK_TABS[activeTab].columnIds` from `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/taskTabs.config.ts`).
