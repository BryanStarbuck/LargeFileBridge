This spec covers **both** directory-viewing surfaces of LargeFileBridge (LFB), which share a data
model, the code-badge vocabulary, and the same entity action menus:

* **Part A ‚Äî The File System** (¬ß1‚Äì¬ß6): the **column browser** (route `/files`, a.k.a. `/fs`) that
  walks the *whole filesystem*, many directories at once, and paints the **code badges** on every row.

  **one** directory ‚Äî its identity, its category rollup, and its two sticky flags.

They are two views of the same subject: the browser is where you find directories; the entity page is
where you inspect and act on **one**. The badge vocabulary defined in **¬ß3** is the shared source both
this file and every sibling spec reference.

# PART A ‚Äî The File System (column browser + code badges)

The **File System** page is the direct, hands-on way to see what LargeFileBridge knows about
**every** file and directory on this computer ‚Äî not just the repos it already manages. You reach it
from the **File System** item in the [left bar](./left_bar.mdx). It is modeled on the **Mac Finder
column view** (a *Miller-column* browser): a row of narrow columns, each showing the contents of one
directory level, where clicking a directory in one column **opens the next column to its right**.

The point of this page is the **code badges**: a small, dense visual language of single letters that
tells you, at a glance and at any depth, which directories are repos, which files we sync, which files
should be compressed, and which files/directories are part of an IPFS share. It complements the
higher-level [Repos](./repos.mdx) table (which only lists *registered git repos*) by letting you see
the **raw filesystem** with the same knowledge painted on top.

---

## 1. The column browser (Finder column view)

The page is a horizontally scrolling strip of **columns**. Each column lists the entries ‚Äî files and
directories ‚Äî at **one** level of the directory hierarchy.

* **Root column.** On first open, the leftmost column shows the contents of the user's **home
  directory** (`$HOME` on Mac/Linux, `%USERPROFILE%` on Windows). The current root path is shown above
  the strip as a breadcrumb, with a control to jump to home or type/paste an absolute path.
* **Drill down.** Clicking a **directory** row highlights it and opens a **new column immediately to
  its right** showing that directory's contents. Clicking a directory in an earlier column **replaces**
  every column to its right with a single new one (exactly like Finder).
* **Columns are narrow.** Each column is a fixed, comfortably narrow width (‚âà 280px), tall, and
  scrolls vertically on its own. The strip scrolls horizontally and, on drill-down, auto-scrolls so
  the newest column is visible.
* **Selecting a file.** Clicking a **file** row selects it (for future preview/detail) but opens no
  new column ‚Äî files are leaves.
* **Sorting within a column.** Directories first, then files, each alphabetical (locale-aware,
  case-insensitive) by default. (This page is a *browser*, not one of the product's TanStack tables ‚Äî
  the [Tables](../CLAUDE.md) rules do not apply here; the rollup counts still live in the table views.)
* **Hidden entries.** Dotfiles/dot-directories are **hidden by default**, with a toggle to show them.
  A small set of noise directories (`.git`, `node_modules`, `.Trash`, `.cache`, OS caches) is always
  collapsed to a single non-expanding row so the walk never drowns in build junk ‚Äî matching the
  scanner's hard-skip set ([scan.mdx ¬ß4](./scan.mdx)).

Each row shows, left to right: an (optional) folder/file glyph, the **entry name** (middle-truncated
if long), a flexible spacer, and then the **code badges** pinned to the **far right** (¬ß3).

Listing one directory level (Node `fs` only, sorted dirs-first, with per-entry badges) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` ‚Äî `listDirectory()`, served by `fs.router.ts` ‚Üí `GET /api/fs`.

## 2. Cross-platform

The browser must work on **Mac, Windows, and Linux** ‚Äî it is the same one page everywhere.

* The **root** is the OS home directory; the **path separator** and **drive/volume roots** are
  normalized by the backend (`/` on POSIX, drive letters on Windows). Paths cross the API as absolute,
  server-normalized strings.
* Repo detection, symlink handling, and the hard-skip set are identical across platforms; only path
  normalization branches per OS (same discipline as [scan.mdx ¬ß3](./scan.mdx)).
* No shell, ever ‚Äî the backend lists directories with Node `fs` (charter + macOS-indexing rule: never
  the `find` CLI).

## 3. Code badges (the visual language) ‚Äî LOCKED

Every row can carry a stack of **code badges** on its far right. A badge is a **single letter in white
text on a solid colored square** (a small rounded chip). The **letter** names the category, the
**color** encodes the category, and the **case** (upper vs. lower) encodes a state within it.

There are four categories: **R** (repo), **S** (sync), **C** (compress), **i** (ipfs).

| Badge | Category | Applies to | Color | Meaning |
| --- | --- | --- | --- | --- |
| **R** | Repo root | directory | **dark brown** | This directory **is a git repo** (its own `.git` working tree). Uppercase = it is the repo **root**. |
| **r** | Repo ancestor | directory | **medium-light brown** | This directory is **not** a repo itself but **contains a repo** somewhere below it (a parent/ancestor of a repo). Lowercase = not a root. |
| **r** | Repo descendant | file **or** directory | **medium brown** | This entry lives **inside** a repo (any recursive child of a repo working tree). Lowercase = not a root. |
| **S** | Sync | file | **bright pink** | We **sync and track** this file ‚Äî its decision is *sync* ([one_repo.mdx ¬ß1](./one_repo.mdx)). |
| **C** | Compress (should) | file | **bright yellow** | This file **should be compressed** ‚Äî a video (primary) or image (secondary) that looks **uncompressed**. |
| **c** | Compressed (done) | file | **light yellow** | We have **already successfully compressed** this file. Lowercase = the finished state. |
| **i** | IPFS | file **or** directory | **blue** | An **IPFS list / share artifact** (or a markdown list of IPFS files); on a directory, it **publishes** such artifacts underneath it. |

### 3.1 The repo badge (R / r) ‚Äî brown, three shades

Repos are about **directories** (only a directory can *be* a repo), but the descendant "r" is painted
on **both files and directories** so you can tell, anywhere in a tree, that you are inside a repo.

Precedence for any entry, in order (a row shows **at most one** repo badge):

1. **Repo root ‚Üí `R`, dark brown.** The directory has its own `.git` (it is a git working tree root).
   This is the strongest signal ‚Äî uppercase because it is the **root**.
2. **Repo descendant ‚Üí `r`, medium brown.** The entry (file or directory) sits **inside** a repo ‚Äî some
   ancestor directory is a git working tree. Lowercase because it is **not** the root.
3. **Repo ancestor ‚Üí `r`, medium-light brown.** A directory that is **not** in a repo and is **not** a
   repo, but **contains** at least one repo below it (it is a parent of one or more repos). Lowercase,
   and the **lightest** brown because it is the furthest from "being" a repo.

So: **uppercase R only at the actual repo root; everything else that touches a repo is a lowercase r**,
darker the deeper you are (root darkest ‚Üí descendant mid ‚Üí ancestor lightest). A plain directory with
no repo above or below it gets **no** repo badge.

> A repo does not have to be *registered/managed* by LFB to earn its badge ‚Äî the browser detects
> `.git` on disk directly, so unregistered repos still show **R**. (Registration is what puts a repo in
> the [Repos](./repos.mdx) table and turns on syncing; the badge is purely "is this a repo on disk".)

### 3.2 The sync badge (S) ‚Äî bright pink, files only

`S` marks a **file we sync and track**: its decision is **sync** in the owning unit's config
([one_repo.mdx ¬ß1](./one_repo.mdx) ‚Äî the `sync | ignore | undecided` decision model). "Sync it" means
LFB will add it to the manifest, pin it, and keep it in step across the user's computers. A file that
is only a *candidate* (big, but still `undecided`) does **not** get `S` ‚Äî only a committed **sync**
decision earns it. Directories never carry `S` (syncing is per-file).

### 3.3 The compress badges (C / c) ‚Äî yellow, files only

Compression is a **core value** (charter): find large files that are uncompressed when they need not
be, and ‚Äî only when the user asks ‚Äî offer to compress them. The badge surfaces the state:

* **`C`, bright yellow ‚Äî should be compressed.** A **video** (primary) or **image** (secondary) file
  that appears **uncompressed** (a lossless/uncompressed container, or a large media file with no
  compressed variant on record). This is an **offer**, never an action ‚Äî LFB never compresses on its
  own.
* **`c`, light yellow ‚Äî already compressed.** We have a recorded **compressed** variant for this file
  (the compressed hash/size in the tracking YAML), or the format is already an efficiently-compressed
  one. Lowercase = the finished, satisfied state.

Only **video and image** files are ever eligible (charter's scope); everything else gets no compress
badge. The uncompressed/compressed judgment is a **heuristic** in the first cut (extension/format +
the big-file threshold), with a seam to the **learned size-baseline** model (duration √ó resolution ‚Üí
expected compressed/uncompressed size, charter "Detecting whether a file is compressed") that replaces
the heuristic later. See [code_plan.mdx ¬ß13](./code_plan.mdx) (deferred/phased) ‚Äî the badge is wired
now; the baseline sharpens it later.

### 3.4 The IPFS badge (i) ‚Äî blue, files and directories

`i` marks anything that is part of an **IPFS list / share** ([ipfs_share_files.mdx](./ipfs_share_files.mdx)):

* On a **file**, `i` means it is an **IPFS list artifact** ‚Äî a pin script (`IPFS.sh` / `ipfs.sh`), a
  three-command list (`ipfs.txt`), a curl-pipe installer (`get_videos.sh`), an IPFS `manifest.yaml`
  under a public dir ‚Äî **or** a **markdown file that is a list of IPFS files** (a `.md` / `.mdx` that
  references `ipfs pin add`, `/ipfs/<CID>`, or `ipfs_cid` entries).
* On a **directory**, `i` means it is **doing special IPFS broadcast underneath it** ‚Äî it contains (or
  publishes via `static/`) one of those IPFS list artifacts or IPFS-list markdown files.

This is the same knowledge [ipfs_share_files.mdx](./ipfs_share_files.mdx) tracks per repo; the badge
just paints it onto the raw filesystem so a share site is visible while browsing. The badge is
**detect-and-report only** ‚Äî showing `i` never publishes, rewrites, or turns this machine into a public
gateway (the only-our-content rule, [knowledge/ipfs.mdx ¬ß6](../knowledge/ipfs.mdx), stands).

Detecting the IPFS `i` badge ‚Äî artifact names, the capped markdown/manifest content peek, and a directory that publishes one underneath ‚Äî is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` ‚Äî `fileIsIpfsArtifact()` and `dirPublishesIpfs()`.

### 3.5 Ordering & stacking (LOCKED)

* Badges are **pinned to the far right** of the row. When a row has several, they **stack leftward**:
  the newest/rightmost slot is filled first, and each additional badge is added **to its left**.
* The **fixed right-to-left order** (rightmost first) is: **R/r ¬∑ S ¬∑ C/c ¬∑ i**. So a repo-root
  directory that also publishes IPFS shows, right to left, `R` then `i` (i.e. `i R` reading
  left-to-right). A synced, uncompressed, IPFS-listed file inside a repo shows, right to left,
  `r ¬∑ S ¬∑ C ¬∑ i` (rendered `i C S r`).
* A row shows **at most one badge per category** (never both `C` and `c`, never two repo shades).
* Zero badges is normal and common.
* Each badge has a **hover tooltip** spelling out its meaning (e.g. "Repo root", "Sync ‚Äî tracked",
  "Should be compressed", "IPFS share artifact").

Computing the ordered badge list per entry (repo precedence, sync, compress, ipfs ‚Äî rightmost-first) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` ‚Äî `computeBadges()` (with the per-listing `buildBadgeContext()`).

## 5. Data model (what a row carries)

The backend returns, per entry, a small typed record the frontend renders directly (types in
`@lfb/shared`, endpoint `GET /api/fs`, [code_plan.mdx ¬ß14](./code_plan.mdx)):

```ts
type FsBadge =
  | "repo_root"        // R  dark brown      (dir)
  | "repo_descendant"  // r  medium brown    (file|dir, inside a repo)
  | "repo_ancestor"    // r  light brown     (dir, contains a repo below)
  | "sync"             // S  bright pink     (file, decision === "sync")
  | "compress"         // C  bright yellow   (file, video/image, looks uncompressed)
  | "compressed"       // c  light yellow    (file, video/image, already compressed)
  | "ipfs";            // i  blue            (file|dir, IPFS list/share)

interface FsEntry {
  name: string;
  path: string;          // absolute, server-normalized
  kind: "dir" | "file" | "symlink" | "other";
  sizeBytes: number | null;   // files only
  modifiedAt: string | null;  // ISO
  isRepoRoot: boolean;
  badges: FsBadge[];     // ordered rightmost-first: R/r, S, C/c, i (¬ß3.5)
  hasChildren: boolean;  // dir has at least one visible child (drives the disclosure)
}

interface FsListing {
  root: string;          // the absolute path this column lists
  parent: string | null; // for "up" navigation
  home: string;          // the OS home dir (default root)
  entries: FsEntry[];
}
```

The order of `badges[]` is authoritative ‚Äî the frontend renders them in array order **right to left**,
so the array is already sorted `[repo, sync, compress, ipfs]` per ¬ß3.5.

## 6. Acceptance criteria (File System browser)

1. The **File System** left-bar item ([left_bar.yaml](./left_bar.yaml)) opens a page that shows a
   **Finder-style column browser** rooted at the user's **home directory**, and works on **Mac,
   Windows, and Linux**.
2. Clicking a **directory** opens a **new column to its right** with that directory's contents;
   clicking a directory in an earlier column **replaces** all columns to its right. Clicking a **file**
   selects it and opens no column.
3. Every row renders its **code badges** as **white-letter-on-solid-color** chips pinned to the **far
   right**, stacking **leftward** in the fixed order **R/r ¬∑ S ¬∑ C/c ¬∑ i** (¬ß3.5), at most one badge per
   category, zero badges allowed.
4. **Repo badge:** a directory with its own `.git` shows **`R` (dark brown)**; any file/directory
   inside a repo shows **`r` (medium brown)**; a directory that merely **contains** a repo below shows
   **`r` (medium-light brown)**; a directory unrelated to any repo shows no repo badge. Unregistered
   repos still get their badge.
5. **Sync badge:** a file whose decision is **sync** shows **`S` (bright pink)**; undecided/ignored
   files and all directories do not.
6. **Compress badges:** a **video or image** file that looks **uncompressed** shows **`C` (bright
   yellow)**; one with a recorded/known-compressed variant shows **`c` (light yellow)**; non-media
   files show neither.
7. **IPFS badge:** an IPFS list artifact (`IPFS.sh`/`ipfs.sh`/`ipfs.txt`/`get_videos.sh`/public
   `manifest.yaml`) or an IPFS-list markdown file shows **`i` (blue)**; a directory that contains/
   publishes such artifacts shows **`i`** too. Showing `i` performs **no** publishing action.
8. Each badge has a **hover tooltip** naming it. Dotfiles are hidden by default with a toggle;
   `.git`/`node_modules`/caches never expand.
9. No shell (`find`/`du`) is used; the listing is Node `fs`, metadata-only for the walk.
10. **Right-clicking** a file/dir row (and a hover **`‚ãÆ`** affordance) opens that entity's action menu
    ([menus.mdx ¬ß3.1/¬ß5](./menus.mdx#31-right-click-opens-the-same-menu-locked)); compress/convert/ignore
    are explicit-click confirm offers and no action deletes local bytes. Opening
    [View one file](./files.mdx) / **View one directory** (Part B, ¬ß7) gives the same catalog as a
    top-right `‚ãÆ` more-menu.

## 8. The category rollup table (charter) ‚Äî TanStack, house style

The heart of this page is the charter's **count + category** rollup for everything under this directory
(a [TanStack `DataTable`](../CLAUDE.md) like all others ‚Äî search/filter/sort control row, 500 default,
plus **header-click sort**: click a column header to sort, click again to toggle asc ‚Üî desc,
[repos.mdx ¬ß3.1](./repos.mdx)). One row per category, each with a per-row action:

| # | Category | Count | Row action | `‚ãÆ` |
| --- | --- | --- | --- | --- |
| 1 | **Videos that can be compressed** | n | **Compress‚Ä¶** (offer only, [menus.mdx ¬ß6.1](./menus.mdx)) | kebab |
| 2 | **Images that can be compressed** | n | **Compress‚Ä¶** (offer only) | kebab |
| 3 | **Big files not git-ignored** | n | **Git-ignore‚Ä¶** (confirm list, never silent) | kebab |
| 4 | **Big files git-ignored, not tracked** | n | **Track / Sync** (add to a sync unit; triage inside) | kebab |

Clicking a **count** deep-links into the File System browser (¬ß1) at this directory, pre-filtered to
that category ([menus.mdx ¬ß2](./menus.mdx)). When **Do not compress** is on, rows 1‚Äì2 still count but
their **Compress‚Ä¶** action is hidden (the offer is suppressed); when **Never IPFS** is on, row 4's
**Track / Sync** is hidden (nothing under here may be synced).

Computing the four charter category counts by a bounded recursive walk is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` ‚Äî `buildDirRollup()` (via `buildEntityView()`).

**Height ‚Äî bounded (the content-below exception).** This page renders an **inside summary** footer
**beneath** the rollup table (¬ß7 layout). Because content sits **below** the table, it is the
**exception** to the site-wide full-page-height default ([repos.mdx ¬ß3.3.1](./repos.mdx), [charter
Tables rule](../CLAUDE.md)): the rollup table keeps a **bounded** height (it never expands to fill the
viewport) so the summary below stays visible. In code, `fillHeight={false}` on the `DataTable`. (The
rollup is only four rows anyway; the fill-height behavior is for the long, scrolling tables.)

## 9. The data payload (`EntityView` + `DirRollup`)

Same `EntityView` as [files.mdx ¬ß2](./files.mdx) (with `kind: "dir"`), plus a directory rollup:

```ts
interface DirRollup {
  videosToCompress: number;      // category 1
  imagesToCompress: number;      // category 2
  bigNotIgnored: number;         // category 3
  bigIgnoredNotTracked: number;  // category 4
  entryCount: number;            // immediate children (files + dirs)
  scannedAt: string | null;
}
```

For a directory, `EntityView.sizeBytes` may be a rolled-up total or `null`; `decision`/`transfer`/`cid`
are `null` (decisions are per-file), and `repo` is populated when the directory sits inside a
registered repo. `flags` carries the same two sticky flags, applied to **everything under** the
directory ([menus.mdx ¬ß6.6](./menus.mdx)).

## 10. The two sticky-flag switches

The badge/flag strip shows both flags as **labeled switches** (as on [files.mdx ¬ß4](./files.mdx)), but
here they are **directory-scoped** ‚Äî they cover current and future contents:

* **Never IPFS** ‚Äî on ‚áí no file under this directory is ever synced; turning it on clears any `sync`
  decisions on files below; the rollup's **Track / Sync** action hides; new files that appear later are
  covered automatically (the flag is a path-scoped policy).
* **Do not compress** ‚Äî on ‚áí the **Compress‚Ä¶** offers (rows 1‚Äì2 and any per-file offer under here) are
  suppressed and the `C` badge is not painted for media below.

Neither flag deletes or alters any bytes ([menus.mdx ¬ß6.2/¬ß6.6](./menus.mdx)).

Writing the directory-scoped sticky flags ‚Äî and clearing `sync` decisions below when Never IPFS turns on ‚Äî is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` ‚Äî `setEntityFlags()`.

## 11. Empty & error states

* **Directory not found / moved** (`exists: false`): a card ‚Äî "This directory is no longer at that
  path." with **Copy path** and a link to the File System browser (¬ß1) at its parent.
* **Nothing to act on:** all four rollup counts 0 ‚Üí the table shows a friendly "Nothing here needs
  attention ‚Äî no uncompressed media, no big un-ignored files."
* **Loading:** skeleton header + table.
* **IPFS node unreachable:** the **Track / Sync** action disables with a tooltip; flag writes still save.
