# LargeFileBridge — Directories

This spec covers **both** directory-viewing surfaces of LargeFileBridge (LFB), which share a data
model, the code-badge vocabulary, and the same entity action menus:

* **Part A — The File System** (§1–§6): the **column browser** (route `/files`, a.k.a. `/fs`) that
  walks the *whole filesystem*, many directories at once, and paints the **code badges** on every row.
* **Part B — View one directory** (§7–§12): the **single-entity page** (route `/dir?path=<abs>`) for
  **one** directory — its identity, its category rollup, and its two sticky flags.

They are two views of the same subject: the browser is where you find directories; the entity page is
where you inspect and act on **one**. The badge vocabulary defined in **§3** is the shared source both
this file and every sibling spec reference.

# PART A — The File System (column browser + code badges)

The **File System** page is the direct, hands-on way to see what LargeFileBridge knows about
**every** file and directory on this computer — not just the repos it already manages. You reach it
from the **File System** item in the [left bar](./left_bar.mdx). It is modeled on the **Mac Finder
column view** (a *Miller-column* browser): a row of narrow columns, each showing the contents of one
directory level, where clicking a directory in one column **opens the next column to its right**.

The point of this page is the **code badges**: a small, dense visual language of single letters that
tells you, at a glance and at any depth, which directories are repos, which files we pin, which files
should be compressed, and which files/directories are part of an IPFS share. It complements the
higher-level [Repos](./repos.mdx) table (which only lists *registered git repos*) by letting you see
the **raw filesystem** with the same knowledge painted on top.

## 1. The column browser (Finder column view)

The page is a horizontally scrolling strip of **columns**. Each column lists the entries — files and
directories — at **one** level of the directory hierarchy.

* **Root column.** On first open, the leftmost column shows the contents of the user's **home
  directory** (`$HOME` on Mac/Linux, `%USERPROFILE%` on Windows). The current root path is shown above
  the strip as a breadcrumb, with a control to jump to home or type/paste an absolute path.
* **Drill down.** Clicking a **directory** row highlights it and opens a **new column immediately to
  its right** showing that directory's contents. Clicking a directory in an earlier column **replaces**
  every column to its right with a single new one (exactly like Finder).
* **Columns are narrow.** Each column is a fixed, comfortably narrow width (≈ 280px), tall, and
  scrolls vertically on its own. The strip scrolls horizontally and, on drill-down, auto-scrolls so
  the newest column is visible.
* **Selecting a file.** Clicking a **file** row selects it (for future preview/detail) but opens no
  new column — files are leaves.
* **Sorting within a column.** Directories first, then files, each alphabetical (locale-aware,
  case-insensitive) by default. (This page is a *browser*, not one of the product's TanStack tables —
  the [Tables](../CLAUDE.md) rules do not apply here; the rollup counts still live in the table views.)
* **Hidden entries.** Dotfiles/dot-directories are **hidden by default**, with a toggle to show them.
  A small set of noise directories (`.git`, `node_modules`, `.Trash`, `.cache`, OS caches) is always
  collapsed to a single non-expanding row so the walk never drowns in build junk — matching the
  scanner's hard-skip set ([scan.mdx §4](./scan.mdx)).

Each row shows, left to right: an (optional) folder/file glyph, the **entry name** (middle-truncated
if long), a flexible spacer, and then the **code badges** pinned to the **far right** (§3). The folder
glyph is drawn uncolored here, but its **interesting-directory coloring** (brown/blue tinting for
directories whose subtree holds big files, videos, or heavy images) is specified in
[file_system.mdx §1–§3](./file_system.mdx) and applied to every folder glyph across the app.

Listing one directory level (Node `fs` only, sorted dirs-first, with per-entry badges) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDirectory()`, served by `fs.router.ts` → `GET /api/fs`.

### 1.2 Page actions — the action-links row

This File System area is the left-bar **File System hub** ([file_system.mdx](./file_system.mdx)); above
the columns it carries the page-level **action-links row** — the visible horizontal row of blue text
hyperlinks (each with a leading icon), directly **under the page title**, that is the third action
surface ([page_actions.mdx](./page_actions.mdx)). It is **not** a dropdown — it replaces the earlier
`Actions ▾` menu — and it is distinct from the per-row `⋮` kebab / entity `⋮` more-menu owned by
[menus.mdx §3–§5](./menus.mdx). Its links run a **batch action over the browser's files**: **Create
Transcriptions · Create AI descriptions · Compress videos · Compress images · Git ignore · Track / Pin
this directory** (short labels; one measured row with a right-pinned `More ▾` overflow) — the "File System /
a directory" row of the per-page catalog
([page_actions.mdx §4](./page_actions.mdx)). The three LOCKED rules (scope =
checked-subset-else-whole-page-recursive, skip-already-done, background queue) and the toast wording
are inherited from that spec and **not** restated here.

* **The browser needs a selection affordance for "checked subset."** Rule 1 keys off "one or more rows
  checked, else the whole page recursively" ([page_actions.mdx §1.1](./page_actions.mdx)). But this
  page is a *browser*, not a TanStack table (§1), so its rows carry **no checkboxes today** — for the
  checked-subset case to work, rows need a lightweight **selection affordance** (a hover check /
  ⌘-click multi-select) whose selected paths feed the action-links row's `scope`. With **nothing**
  selected, the links act on the **deepest (rightmost) open column's directory, recursively** (Rule 1b).

* **Scope containment (LOCKED).** "Deepest open column" means the **tail** of the open-column chain — the
  last directory the user clicked open, whose contents fill the rightmost column. The recursive walk is
  **strictly contained inside that directory: it descends only and never ascends to a parent column or
  touches a sibling directory.** Because clicking a directory in an earlier column **replaces** every
  column to its right (§1), the open-column chain is always exactly the root→deepest path, and only its
  **tail** is the scope — the ancestor columns are breadcrumbs, never scope. Highlighting a **file** inside
  the deepest column does **not** change the directory scope (it stays the containing directory). If no
  column is open yet, the scope resolves to *nothing* and the enqueue endpoint returns a `400` — it must
  **never** fall back to walking the home directory or the filesystem root. This is the full guarantee the
  File System page owner locks in [file_system.mdx §4.1](./file_system.mdx); see it for the worked
  `Home → _Mirror → Politics → Charlie_Kirk_Mi` example.

implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx` — `FileSystemPage` (mounts `PageActions` from `code/packages/frontend/src/components/menu/PageActions.tsx` — `PageActions`; the scope `root` is the deepest column, `currentDir = stack[stack.length - 1]`, passed as `{ root: currentDir }`). The downward-only recursive walk that enforces containment is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/transcribe/transcribe.service.ts` — `walkMedia()` (recurses into children of `root` only, never a parent).

## 1.3 Pick up where you left off — persisted columns & selection (LOCKED)

Walking down a deep directory hierarchy one column at a time is **real work** — a customer may click
through five or six levels to reach the folder they care about. If leaving the File System page and
coming back threw that away, every visit would start over at `$HOME`. It does not. **The File System
page restores the exact column layout and selection the user last had**, so returning after two hours
in another part of the app drops them **right back where they were**.

The scenario this serves, verbatim:

> The user comes into the **File System** left tab. It can be a lot of work to click through the
> directory hierarchy. They click another left-bar tab to navigate elsewhere, spend two hours in
> another part of the product, then click **File System** to come back. We **re-select their open
> directory columns** (the last-changed selected-directory chain) so they can pick up where they left
> off. **Same with re-selecting the files that were formerly selected.**

What is persisted and restored:

* **The open column chain** — the ordered list of directory paths that were open, left to right (root
  column → each drilled-into directory). Restoring re-opens each column and auto-scrolls so the newest
  (rightmost) column is visible, exactly as if the user had clicked back down the chain.
* **The selection** — the **one or more** entries that were highlighted when the user left (the deepest
  selected directory, and/or the ⌘-click multi-selected files from the selection affordance in
  [§1.2](#12-page-actions--the-header-actions-menu)). All selected paths are re-highlighted; a selected
  directory re-opens its child column.
* **The header filter state** — the [§1.4](#14-header-filters--only-large-filesdirs--videos--images--audio)
  checkboxes (**Only Large Files/Dirs**, **Videos**, **Images**, **Audio**), so the view returns
  filtered the way they left it.

Where it persists — **per-user `config.yaml`**, the same personal-state file the web-session history
lives in ([storage_local.mdx §4](./storage_local.mdx#4-per-logged-in-user-storage--the-per-user-requirement),
[sessions.mdx §4](./sessions.mdx#4-storage--the-five-session-history-in-usersemailconfigyaml)) — under a
top-level `file_system:` key:

```yaml
# ~/T/_large_files_bridge/users/<email>/config.yaml  (excerpt — File System view state)
file_system:
  columns:                       # open column chain, root → deepest, left to right (§1.3)
    - /Users/bryan
    - /Users/bryan/BGit
    - /Users/bryan/BGit/Bryan_git/LargeFileBridge/pm
  selection:                     # the 1+ entries highlighted when the user left (§1.2 multi-select)
    - /Users/bryan/BGit/Bryan_git/LargeFileBridge/pm/directories.mdx
    - /Users/bryan/BGit/Bryan_git/LargeFileBridge/pm/files.mdx
  filters:                       # the §1.4 header checkboxes (all default ON)
    only_large: true             # [x] Only Large Files/Dirs
    videos: true                 # [x] Videos
    images: true                 # [x] Images
    audio: true                  # [x] Audio
  updated_at: 2026-07-07T15:42:10.000Z
```

Rules (LOCKED):

* The write is **debounced** — the page saves the view state a short beat after the columns or the
  selection change (not on every keystroke), through the store's per-file mutex + atomic write
  ([storage_local.mdx §15](./storage_local.mdx)), the same discipline the `sessions:` history uses.
* **It is view state, not security state** — like the web session ([sessions.mdx §1](./sessions.mdx#1-vocabulary--web-session-vs-auth-session-locked)),
  it never gates access and never derives identity.
* **Stale-path tolerance.** A persisted path that no longer exists on restore is **skipped, not fatal**:
  the page re-opens the longest still-valid prefix of the `columns` chain and drops any missing
  `selection` entries. A wholly gone chain falls back to the **home** root column (§1). Restoring must
  never error just because a folder moved between visits.
* A **fresh user with no `file_system:` key** opens at **home** with all four filter checkboxes ON
  (§1.4 defaults) — the missing key defaults to that, never to an empty/blank page.
* **Client cache coherence on the round-trip (LOCKED — this is the recurring "it never restores"
  bug).** The restore is only correct if, when the page re-mounts, it reads back **the columns the
  user actually last selected** — not a stale earlier snapshot. Two things make this fail if not
  handled, and both are required:
  1. **The save must update the client's cached copy of the view state, not just the server.** The
     browser caches the `GET /view-state` result (react-query). Selecting directories does a
     debounced `PUT /view-state`, but if that PUT does **not** also refresh the client's cached
     view-state entry, the cache still holds the value from the first mount. On return within the
     cache window the page restores that **stale** value (typically empty → falls back to home),
     silently throwing away every directory the user just clicked. **Every persist (the debounced
     save AND the flush-on-unmount save) must write the new `{columns, selection, filters}` into the
     client view-state cache at the same time** so the cache is always the last-saved state.
  2. **The one-shot restore must not lock in a stale value before the fresh one arrives.** Restore
     seeds the columns exactly once per mount (so it can't fight the user's clicks). It must seed
     from **authoritative, current** view state — never from a cached snapshot that a background
     refetch is about to replace. With rule 1 the cache is always current, so the once-only seed is
     safe; the seed must still wait for the view-state lookup to settle before deciding home-vs-restore.
  The symptom when this is broken: the backend `config.yaml` shows the correct `columns:` (the save
  worked), yet the page keeps reopening at home. The defect is always on the **client cache side**, not
  the server.

The persisted-view-state shape (`file_system:` — open columns, selection, filters) added to the
per-user config schema is defined in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — `UserConfigSchema` (the `file_system` block, a `FileSystemViewSchema`).

Loading the saved view (pruned to the longest still-valid column prefix + existing selection, the
stale-path tolerance above) and writing it back is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs-view.service.ts` — `loadFsView()` / `saveFsView()` (which persist the `file_system:` block through `getUserConfig()` / `updateUserConfig()` in `store-model/user-config.service.ts`).

Exposing that view state to the browser is handled in code (per-user, allow-list-gated):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.router.ts` — `GET /api/fs/view-state` (pruned load) and `PUT /api/fs/view-state` (debounced save).

Restoring the column chain + selection on mount and debounce-persisting them on every change (plus a
flush on unmount so clicking away to another left-bar tab still saves) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx` — `FileSystemPage` (seeds the column `stack` from `api.fsViewState()` behind a `restoredRef` gate — `?path` deep-link wins, then the saved view, then home — and debounce-saves via `api.saveFsViewState()`).

Keeping the client's cached view state coherent with each save (the LOCKED cache-coherence rule above)
is handled in code: `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx`

for BOTH the debounced-change save and the flush-on-unmount save, so a re-mount within the cache window
restores the last-selected chain instead of a stale snapshot.

## 1.4 Header filters — Only Large Files/Dirs · Videos / Images / Audio (LOCKED)

Above the columns, alongside the breadcrumb and the action-links row (§1.2), the File System page
carries a small row of **filter checkboxes** that cut the noise down to what actually matters for
bridging large files. All four default **ON**:

```
 [x] Only Large Files/Dirs      [x] Videos   [x] Images   [x] Audio
```

**`[x] Only Large Files/Dirs` — the size cut.** When **ON**, the browser hides the small stuff so only
large files (and the directories that lead to them) remain:

* A **file** is shown iff it is **big** — its size is **≥ the global big-file threshold**
  ([settings.mdx §1](./settings.mdx#1-the-big-file-threshold--the-headline-setting), default **100 MB**;
  the same authoritative `big_file.threshold_bytes` the scan uses) — **or** it belongs to a **checked**
  media category (below).
* A **directory** is shown iff it **contains at least one shown file anywhere underneath it**
  (recursively) — i.e. at least one big file, or at least one file of a checked media category. A
  directory with **nothing qualifying** below it is **hidden**, so the walk collapses down to only the
  branches that hold large content.
* When **OFF**, there is **no size filtering** — every file and directory shows (this is the raw
  Finder-style walk of §1), and the media checkboxes below have no effect (they exist only as
  exceptions to *this* cut).

**`[x] Videos  [x] Images  [x] Audio` — per-category inclusion overrides.** Each checked category is an
**exception** to the size cut: **a file of a checked category is shown even when it is under the
threshold.** This is how a user keeps *all* their media visible while still hiding small non-media junk:

* With **Only Large Files/Dirs ON** and, say, **Videos ON**, a 12 MB `.mp4` (under 100 MB) **still
  shows** because Videos is checked; a 12 MB `.log` does **not** (not big, not a checked category).
* **Unchecking** a category subjects its under-threshold files to the size cut like anything else: with
  Videos **OFF**, that same 12 MB `.mp4` is hidden (it is not big), while a 400 MB `.mp4` still shows
  because it is **big** regardless of category.
* The three categories are **visibility filters only**. **Audio is included here even though audio is
  NOT a compressible kind** (it earns **no `C` badge** — the charter's compression scope is **video
  first, image second**, [§3.3](#33-the-compress-badges--c--c--yellow-files-only)). Category membership
  is decided by extension via the shared media map (`mediaKindForName()` → `"video" | "image" | "audio"`),
  the same map the media viewers use ([media_viewer.mdx](./media_viewer.mdx)).

The filter state **persists** with the rest of the view state (§1.3), so the browser returns filtered
the way the user left it.

> **Interaction summary (LOCKED).** *Only Large Files/Dirs* is the gate; *Videos/Images/Audio* are
> holes punched in that gate. A file shows when **(Only-Large is OFF)** OR **(it is big)** OR **(it is
> in a checked media category)**. A directory shows when at least one file under it shows. The media
> checkboxes only ever *add* files back; they never hide a big file.

Applying the header filters server-side — the size cut against the settings threshold, the checked-media
overrides, and hiding directories with no qualifying descendant — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDirectory()` (taking the `onlyLarge` / `videos` / `images` / `audio` query params; the per-directory keep test via `dirHasQualifyingDescendant()`, a bounded recursive walk honoring the §1 hard-skip set; threshold from `getAppConfig().big_file.threshold_bytes`).

Rendering the four header filter checkboxes and threading their state into the `GET /api/fs` query (and into the persisted view state) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx` — `FileSystemPage` (the `FsFilters` control row).

## 2. Cross-platform

The browser must work on **Mac, Windows, and Linux** — it is the same one page everywhere.

* The **root** is the OS home directory; the **path separator** and **drive/volume roots** are
  normalized by the backend (`/` on POSIX, drive letters on Windows). Paths cross the API as absolute,
  server-normalized strings.
* Repo detection, symlink handling, and the hard-skip set are identical across platforms; only path
  normalization branches per OS (same discipline as [scan.mdx §3](./scan.mdx)).
* No shell, ever — the backend lists directories with Node `fs` (charter + macOS-indexing rule: never
  the `find` CLI).

## 3. Code badges (the visual language) — LOCKED

Every row can carry a stack of **code badges** on its far right. A badge is a **single letter in white
text on a solid colored square** (a small rounded chip). The **letter** names the category, the
**color** encodes the category, and the **case** (upper vs. lower) encodes a state within it.

There are five categories: **R** (repo), **P** (pin), **C** (compress), **i** (ipfs), **I** (git-ignored).

| Badge | Category | Applies to | Color | Meaning |
| --- | --- | --- | --- | --- |
| **R** | Repo root | directory | **dark brown** | This directory **is a git repo** (its own `.git` working tree). Uppercase = it is the repo **root**. |
| **r** | Repo ancestor | directory | **medium-light brown** | This directory is **not** a repo itself but **contains a repo** somewhere below it (a parent/ancestor of a repo). Lowercase = not a root. |
| **r** | Repo descendant | file **or** directory | **medium brown** | This entry lives **inside** a repo (any recursive child of a repo working tree). Lowercase = not a root. |
| **P** | Pin | file | **bright pink** | We **pin and track** this file — its decision is `sync`, shown as **Add to IPFS (pin)** ([one_repo.mdx §1](./one_repo.mdx)). |
| **C** | Compress (should) | file | **bright yellow** | This file **should be compressed** — a video (primary) or image (secondary) that looks **uncompressed**. |
| **c** | Compressed (done) | file | **light yellow** | We have **already successfully compressed** this file. Lowercase = the finished state. |
| **i** | IPFS | file **or** directory | **blue** | An **IPFS list / share artifact** (or a markdown list of IPFS files); on a directory, it **publishes** such artifacts underneath it. |
| **I** | Git-ignored | file **or** directory | **near-white** (light-gray fill, hairline border, white letter) | This entry is **git-ignored** — `git check-ignore` covers it (it is not committed; it is pinned via IPFS instead, [git_ignore.mdx](./git_ignore.mdx)). A deliberately faint, de-emphasized marker. |

### 3.1 The repo badge (R / r) — brown, three shades

Repos are about **directories** (only a directory can *be* a repo), but the descendant "r" is painted
on **both files and directories** so you can tell, anywhere in a tree, that you are inside a repo.

Precedence for any entry, in order (a row shows **at most one** repo badge):

1. **Repo root → `R`, dark brown.** The directory has its own `.git` (it is a git working tree root).
   This is the strongest signal — uppercase because it is the **root**.
2. **Repo descendant → `r`, medium brown.** The entry (file or directory) sits **inside** a repo — some
   ancestor directory is a git working tree. Lowercase because it is **not** the root.
3. **Repo ancestor → `r`, medium-light brown.** A directory that is **not** in a repo and is **not** a
   repo, but **contains** at least one repo below it (it is a parent of one or more repos). Lowercase,
   and the **lightest** brown because it is the furthest from "being" a repo.

So: **uppercase R only at the actual repo root; everything else that touches a repo is a lowercase r**,
darker the deeper you are (root darkest → descendant mid → ancestor lightest). A plain directory with
no repo above or below it gets **no** repo badge.

> A repo does not have to be *registered/managed* by LFB to earn its badge — the browser detects
> `.git` on disk directly, so unregistered repos still show **R**. (Registration is what puts a repo in
> the [Repos](./repos.mdx) table and turns on pinning; the badge is purely "is this a repo on disk".)

### 3.2 The pin badge (P) — bright pink, files only

`P` marks a **file we pin and track**: its decision is `sync` in the owning unit's config
([one_repo.mdx §1](./one_repo.mdx) — the `sync | ignore | undecided` decision model, whose `sync` value
renders as **Add to IPFS (pin)**). Adding it to IPFS means LFB will add it to the manifest, pin it, and
keep it in step across the user's computers. A file that is only a *candidate* (big, but still
`undecided`) does **not** get `P` — only a committed `sync` decision earns it. Directories never carry
`P` (pinning is per-file).

Deciding whether a file earns the pin badge (its decision is `sync`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `fileIsPinned()` (badge value `pin`, letter P).

### 3.3 The compress badges (C / c) — yellow, files only

Compression is a **core value** (charter): find large files that are uncompressed when they need not
be, and — only when the user asks — offer to compress them. The badge surfaces the state:

* **`C`, bright yellow — should be compressed.** A **video** (primary) or **image** (secondary) file
  that appears **uncompressed** (a lossless/uncompressed container, or a large media file with no
  compressed variant on record). This is an **offer**, never an action — LFB never compresses on its
  own.
* **`c`, light yellow — already compressed.** We have a recorded **compressed** variant for this file
  (the compressed hash/size in the tracking YAML), or the format is already an efficiently-compressed
  one. Lowercase = the finished, satisfied state.

Only **video and image** files are ever eligible (charter's scope); everything else gets no compress
badge. The uncompressed/compressed judgment is a **heuristic** in the first cut (extension/format +
the big-file threshold), with a seam to the **learned size-baseline** model (duration × resolution →
expected compressed/uncompressed size, charter "Detecting whether a file is compressed") that replaces
the heuristic later. See [code_plan.mdx §13](./code_plan.mdx) (deferred/phased) — the badge is wired
now; the baseline sharpens it later.

### 3.4 The IPFS badge (i) — blue, files and directories

`i` marks anything that is part of an **IPFS list / share** ([ipfs_share_files.mdx](./ipfs_share_files.mdx)):

* On a **file**, `i` means it is an **IPFS list artifact** — a pin script (`IPFS.sh` / `ipfs.sh`), a
  three-command list (`ipfs.txt`), a curl-pipe installer (`get_videos.sh`), an IPFS `manifest.yaml`
  under a public dir — **or** a **markdown file that is a list of IPFS files** (a `.md` / `.mdx` that
  references `ipfs pin add`, `/ipfs/<CID>`, or `ipfs_cid` entries).
* On a **directory**, `i` means it is **doing special IPFS broadcast underneath it** — it contains (or
  publishes via `static/`) one of those IPFS list artifacts or IPFS-list markdown files.

This is the same knowledge [ipfs_share_files.mdx](./ipfs_share_files.mdx) tracks per repo; the badge
just paints it onto the raw filesystem so a share site is visible while browsing. The badge is
**detect-and-report only** — showing `i` never publishes, rewrites, or turns this machine into a public
gateway (the only-our-content rule, [knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx), stands).

Detecting the IPFS `i` badge — artifact names, the capped markdown/manifest content peek, and a directory that publishes one underneath — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `fileIsIpfsArtifact()` and `dirPublishesIpfs()`.

### 3.4a The git-ignored badge (I) — near-white, files and directories

`I` marks anything **git currently ignores** — the counterpart to the [Git Ignore action](./git_ignore.mdx).
It is how the browser shows, at a glance, that a big file is *already* handled the LFBridge way: **not
committed to git, pinned over IPFS instead** (the whole product bet — [dropbox.mdx](./dropbox.mdx)).

* On a **file or directory**, `I` means **`git check-ignore` covers this path** in its owning repo — a
  `.gitignore` rule (ours or the user's) ignores it. It is **detect-and-report only**: showing `I` never
  writes anything; the [Git Ignore dialog](./git_ignore.mdx) is the only thing that *adds* an ignore.
* **Deliberately faint (LOCKED look).** Git-ignored is a **de-emphasized** state, so its chip is the
  quietest in the vocabulary: a **near-white** (very light gray) fill with a **white letter `I`** and a
  **hairline border** carrying the shape. The user's words: *"a very light gray fill with the letter I in
  white … so light it's near white."* The border is what makes a white-on-near-white letter perceptible
  without shouting.
* An entry can carry `I` **alongside** other badges — a pinned, compressed, git-ignored video inside a repo
  shows `r · c · I` (a common combination: LFB ignores the big file in git and pins it over IPFS).

> **Why `I` and `i` don't collide.** IPFS is **lowercase `i`, blue**; git-ignored is **uppercase `I`,
> near-white** — the same case-encodes-state, color-encodes-category convention as `R`/`r` and `C`/`c`.
> The color gulf (saturated blue vs. near-white) and the case difference keep them distinct; the
> non-intrusive hover panel ([non_intrusive_tooltip.mdx](./non_intrusive_tooltip.mdx)) names either on hover
> so there is never ambiguity.

Detecting the git-ignored `I` badge (ask `git check-ignore` for the entry against its owning repo, with the result cached on the listing's badge context) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `computeBadges()` (the `git_ignored` push, using `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `checkIgnore()` / `nearestGitAtOrAbove()`, batched per directory in `buildBadgeContext()` so a column costs one `git check-ignore --stdin` per repo, not one spawn per row).

### 3.5 Ordering & stacking (LOCKED)

* Badges are **pinned to the far right** of the row. When a row has several, they **stack leftward**:
  the newest/rightmost slot is filled first, and each additional badge is added **to its left**.
* The **fixed right-to-left order** (rightmost first) is: **R/r · P · C/c · i · I**. So a repo-root
  directory that also publishes IPFS shows, right to left, `R` then `i` (i.e. `i R` reading
  left-to-right). A pinned, uncompressed, IPFS-listed file inside a repo shows, right to left,
  `r · P · C · i` (rendered `i C P r`); if that file is also git-ignored the leftmost `I` is added
  (rendered `I i C P r`).
* A row shows **at most one badge per category** (never both `C` and `c`, never two repo shades).
* Zero badges is normal and common.

### 3.5.1 Hover tooltip — name what the letter stands for (LOCKED)

A single colored letter is opaque to anyone who hasn't memorized the vocabulary. So **every badge, in
every place it renders, explains itself on hover** — the user should never have to guess what `P`, `i`, or
`I` means, and should never have to open the legend to find out.

> **The explanation now surfaces in the non-intrusive panel, not a floating bubble (LOCKED).** Hovering a
> badge publishes its **name + meaning** into the **hover-info panel at the bottom of the left bar**
> ([non_intrusive_tooltip.mdx](./non_intrusive_tooltip.mdx)) — an out-of-the-way area that **does not cover**
> the row or its neighbors. An entity with several badges stacks **one code-key block per badge** there. The
> chip + name come from the **same** `BADGE_META` the legend uses (below), so panel, tooltip, and legend can
> never drift. A native `title` / `aria-label` is still set on the chip for **accessibility** and for
> surfaces with no left bar; the floating portal tooltip remains that a11y fallback.

* **Hovering (or keyboard-focusing) a badge shows a tooltip** whose **headline is the NAME the letter
  stands for** — a short **1–3 word** phrase (e.g. **Repo root**, **Inside a repo**, **Pinned**,
  **Compress**, **Compressed**, **IPFS artifact**) — followed by **one slightly-longer line** explaining
  it when a name alone isn't enough (e.g. "Tracked and bridged over IPFS across your computers."). Keep it
  to a name plus one line; this is a glance aid, not documentation.
* **The same badges appear at the top of an entity page** — the badge strip in the header of *View one
  file* / *View one directory* / the media viewer (menus.mdx §5), the "top of the web page" the user
  sees when they open one entry. Because that strip renders the **same** `Badges` component, those
  top-of-page badges carry the **same** hover tooltip — so the meaning is reachable both while browsing
  the file list and while looking at a single entity.
* **One source of truth.** The tooltip's name + line come from the **same** per-badge metadata that
  drives the **legend** ("What do these letters mean?", §5.7 of use_cases.mdx), so the hover text and the
  legend can never drift.
* The tooltip renders into a **body-level portal** so a table cell's or scroll region's `overflow` never
  clips it, and it never fires a duplicate native browser `title`.

## 5. Data model (what a row carries)

The backend returns, per entry, a small typed record the frontend renders directly (types in
`@lfb/shared`, endpoint `GET /api/fs`, [code_plan.mdx §14](./code_plan.mdx)):

```ts
type FsBadge =
  | "repo_root"        // R  dark brown      (dir)
  | "repo_descendant"  // r  medium brown    (file|dir, inside a repo)
  | "repo_ancestor"    // r  light brown     (dir, contains a repo below)
  | "pin"              // P  bright pink     (file, decision === "sync")
  | "compress"         // C  bright yellow   (file, video/image, looks uncompressed)
  | "compressed"       // c  light yellow    (file, video/image, already compressed)
  | "ipfs"             // i  blue            (file|dir, IPFS list/share)
  | "git_ignored";     // I  near-white      (file|dir, git check-ignore covers it — §3.4a)

interface FsEntry {
  name: string;
  path: string;          // absolute, server-normalized
  kind: "dir" | "file" | "symlink" | "other";
  sizeBytes: number | null;   // files only
  modifiedAt: string | null;  // ISO
  isRepoRoot: boolean;
  badges: FsBadge[];     // ordered rightmost-first: R/r, P, C/c, i, I (§3.5)
  hasChildren: boolean;  // dir has at least one visible child (drives the disclosure)
}

interface FsListing {
  root: string;          // the absolute path this column lists
  parent: string | null; // for "up" navigation
  home: string;          // the OS home dir (default root)
  entries: FsEntry[];    // already filtered by the header checkboxes (§1.4) when requested
}
```

The order of `badges[]` is authoritative — the frontend renders them in array order **right to left**,
so the array is already sorted `[repo, pin, compress, ipfs]` per §3.5.

### 5.1 Filter query params (§1.4) and persisted view state (§1.3)

`GET /api/fs?path=<abs>` also accepts the four header-filter flags; the backend applies them so the
returned `entries[]` are already filtered (the frontend renders whatever it is handed):

```
GET /api/fs?path=<abs>&onlyLarge=1&videos=1&images=1&audio=1
```

* `onlyLarge` — the size cut (§1.4). Omitted / `0` ⇒ no filtering (raw walk).
* `videos` / `images` / `audio` — the per-category inclusion overrides; each defaults **on**.

The persisted view state (§1.3) is a small block on the per-user config, restored on mount and written
debounced:

```ts
interface FileSystemView {
  columns: string[];     // open column chain, root → deepest (absolute paths)
  selection: string[];   // the 1+ highlighted entries (absolute paths), §1.2 multi-select
  filters: {
    onlyLarge: boolean;  // [x] Only Large Files/Dirs   (default true)
    videos: boolean;     // [x] Videos                  (default true)
    images: boolean;     // [x] Images                  (default true)
    audio: boolean;      // [x] Audio                   (default true)
  };
  updatedAt: string;     // ISO
}
```

On restore, a `columns` entry that no longer exists truncates the chain to its longest valid prefix, and
a missing `selection` path is dropped — restoring is **never** an error (§1.3).

## 6. Acceptance criteria (File System browser)

1. The **File System** left-bar item ([left_bar.yaml](./left_bar.yaml)) opens a page that shows a
   **Finder-style column browser** rooted at the user's **home directory**, and works on **Mac,
   Windows, and Linux**.
2. Clicking a **directory** opens a **new column to its right** with that directory's contents;
   clicking a directory in an earlier column **replaces** all columns to its right. Clicking a **file**
   selects it and opens no column.
3. Every row renders its **code badges** as **white-letter-on-solid-color** chips pinned to the **far
   right**, stacking **leftward** in the fixed order **R/r · P · C/c · i** (§3.5), at most one badge per
   category, zero badges allowed.
4. **Repo badge:** a directory with its own `.git` shows **`R` (dark brown)**; any file/directory
   inside a repo shows **`r` (medium brown)**; a directory that merely **contains** a repo below shows
   **`r` (medium-light brown)**; a directory unrelated to any repo shows no repo badge. Unregistered
   repos still get their badge.
5. **Pin badge:** a file whose decision is `sync` shows **`P` (bright pink)**; undecided/ignored
   files and all directories do not.
6. **Compress badges:** a **video or image** file that looks **uncompressed** shows **`C` (bright
   yellow)**; one with a recorded/known-compressed variant shows **`c` (light yellow)**; non-media
   files show neither.
7. **IPFS badge:** an IPFS list artifact (`IPFS.sh`/`ipfs.sh`/`ipfs.txt`/`get_videos.sh`/public
   `manifest.yaml`) or an IPFS-list markdown file shows **`i` (blue)**; a directory that contains/
   publishes such artifacts shows **`i`** too. Showing `i` performs **no** publishing action.
8. Each badge has a **hover tooltip** naming it. Dotfiles are hidden by default with a toggle;
   `.git`/`node_modules`/caches never expand.
9. No shell (`find`/`du`) is used; the listing is Node `fs`, metadata-only for the walk.
10. **Right-clicking** a file/dir row (and a hover **`⋮`** affordance) opens that entity's action menu
    ([menus.mdx §3.1/§5](./menus.mdx#31-right-click-opens-the-same-menu-locked)); compress/convert/ignore
    are explicit-click confirm offers and no action deletes local bytes. Opening
    [View one file](./files.mdx) / **View one directory** (Part B, §7) gives the same catalog as a
    top-right `⋮` more-menu.
11. **Persisted view state (§1.3).** Leaving the File System page and returning **restores the exact open
    column chain, the 1+ selected entries, and the four filter checkboxes** from the per-user
    `config.yaml` `file_system:` block. Writes are debounced; a persisted path that no longer exists is
    skipped (longest-valid-prefix for columns, drop-missing for selection) and never errors; a fresh user
    with no `file_system:` key opens at **home** with all four filters ON.
12. **Header filters (§1.4).** The header shows **`[x] Only Large Files/Dirs`** and **`[x] Videos`
    `[x] Images` `[x] Audio`**, all default ON. With *Only Large* ON, a file shows iff it is **big**
    (≥ the [settings.mdx §1](./settings.mdx#1-the-big-file-threshold--the-headline-setting) threshold,
    default 100 MB) **or** in a **checked** media category; a directory shows iff it has a qualifying
    file underneath it. A checked category shows its files **even under the threshold**; unchecking
    subjects them to the size cut. **Audio is a visibility filter only — it earns no `C` badge.** With
    *Only Large* OFF there is no size filtering and the media checkboxes have no effect.

### 8.1 The row `⋮` kebab (every row, per [menus.mdx §3](./menus.mdx#3-the-row--kebab--every-table-has-one))

A rollup row is a **category**, not a file/dir/repo/peer entity, so it uses this **page-local** menu
(the rule in [menus.mdx §3](./menus.mdx) — a non-entity table's kebab contents are defined by the
owning spec). The trailing vertical **`⋮`** is present on **every** row, even when its inline action is
hidden (so the column is never blank). Contents by row:

| Row | `⋮` contents |
| --- | --- |
| 1–2 (compressible) | **Compress…** (the same offer as the inline button; hidden only when *Do not compress* is on — then the kebab still shows the rest) · **Show matching files** (deep-link to the File System browser filtered to this category) · **Do not compress** (folder-level sticky toggle, [menus.mdx §6.6](./menus.mdx#66-the-two-sticky-flags--never-ipfs--do-not-compress-locked)) |
| 3 (big, not git-ignored) | **Git-ignore…** (confirm list) · **Show matching files** |
| 4 (big, git-ignored, not tracked) | **Track / Pin** (hidden when *Never IPFS* is on) · **Show matching files** · **Never IPFS** (folder-level sticky toggle) |

The folder-level sticky toggles in the kebab are the **same state** as the switches in the badge/flag
strip (§7) — flipping either updates both. Compress / Git-ignore / Track remain **explicit-click
offers** with their confirm step ([menus.mdx §6.1](./menus.mdx)); the kebab never acts silently.

## 9. The data payload (`EntityView` + `DirRollup`)

Same `EntityView` as [files.mdx §2](./files.mdx) (with `kind: "dir"`), plus a directory rollup:

```ts
interface DirRollup {
  videosToCompress: number;      // category 1
  imagesToCompress: number;      // category 2
  bigNotIgnored: number;         // category 3
  bigIgnoredNotTracked: number;  // category 4
  entryCount: number;            // immediate children (files + dirs)
  scannedAt: string | null;
}
```

For a directory, `EntityView.sizeBytes` may be a rolled-up total or `null`; `decision`/`transfer`/`cid`
are `null` (decisions are per-file), and `repo` is populated when the directory sits inside a
registered repo. `flags` carries the same two sticky flags, applied to **everything under** the
directory ([menus.mdx §6.6](./menus.mdx)).

---

## 10. The two sticky-flag switches

The badge/flag strip shows both flags as **labeled switches** (as on [files.mdx §4](./files.mdx)), but
here they are **directory-scoped** — they cover current and future contents:

* **Never IPFS** — on ⇒ no file under this directory is ever pinned; turning it on clears any `sync`
  decisions on files below; the rollup's **Track / Pin** action hides; new files that appear later are
  covered automatically (the flag is a path-scoped policy).
* **Do not compress** — on ⇒ the **Compress…** offers (rows 1–2 and any per-file offer under here) are
  suppressed and the `C` badge is not painted for media below.

Neither flag deletes or alters any bytes ([menus.mdx §6.2/§6.6](./menus.mdx)).

Writing the directory-scoped sticky flags — and clearing `sync` decisions below when Never IPFS turns on — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` — `setEntityFlags()`.

## 11. Empty & error states

* **Directory not found / moved** (`exists: false`): a card — "This directory is no longer at that
  path." with **Copy path** and a link to the File System browser (§1) at its parent.
* **Nothing to act on:** all four rollup counts 0 → the table shows a friendly "Nothing here needs
  attention — no uncompressed media, no big un-ignored files."
* **Loading:** skeleton header + table.
* **IPFS node unreachable:** the **Track / Pin** action disables with a tooltip; flag writes still save.

## 12. Acceptance criteria (View one directory)

1. `/dir?path=<abs>` renders **one directory's** identity (name title, absolute path muted), its **code
   badges** (§3), and the **category rollup table** (§8) with the four charter categories, each showing
   a count and a per-row action.
2. The page carries a **top-right `⋮` "more" menu** rendering the **Directory catalog**
   ([menus.mdx §5.2](./menus.mdx)) verbatim (same as the row kebab), with **Open in File System** as the
   visible primary; right-clicking the directory entity opens the same menu ([menus.mdx §3.1](./menus.mdx)).
3. The rollup is a **TanStack `DataTable`** in the house style; clicking a **count** deep-links into the
   File System browser (§1) at this directory filtered to that category ([menus.mdx §2](./menus.mdx)).
4. Both **sticky flags** (**Never IPFS**, **Do not compress**) show as labeled switches (§10), are
   **directory-scoped**, write immediately, stay consistent with the menu toggles, and honor
   [menus.mdx §6.6](./menus.mdx) — Never IPFS hides row-4 **Track / Pin** and forbids pinning below; Do not
   compress hides the **Compress…** offers; neither deletes or alters bytes.
5. Not-found / nothing-to-do / node-unreachable / loading states render per §11.
6. The per-directory **entity** page (Part B) and the **File System column browser** (Part A) are two
   views of the same subject in this one spec — the entity page inspects **one** directory; the browser
   walks **many**. They share the badge vocabulary (§3) and the data model (§5/§9).
