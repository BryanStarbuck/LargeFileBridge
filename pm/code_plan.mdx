# LargeFileBridge ‚Äî Code Plan

This is the **engineering plan** for LargeFileBridge (LFB). It reads every spec in `pm/` and the
`knowledge/ipfs.mdx` base, studies the two sister web apps, and turns all of it into a concrete map of
**what code to write, where it lives, and how the pieces run** ‚Äî locally first, server-capable later.

> **Division of labor.** The `pm/*.mdx` specs own *product behavior* (what the app does). This file
> owns *code structure* (how it is built). Where they disagree on behavior, the spec wins; where they
> disagree on layout, this file wins. Code goes **only** in the peer `code/` directory
> (charter rule); this plan lives in `pm/`.

[settings.mdx](./settings.mdx) ¬∑ [repo_settings.mdx](./repo_settings.mdx) ¬∑

## 1. What we learned from the sister apps (and adopt verbatim)

Both `~/BGit/all/app/` and `~/BGit/Bryan_git/EmailDeliveryHero/` converge on the same shape, and
[storage.mdx ¬ßintro](./storage.mdx) already says LFB follows it. We adopt it, renamed to the `lfb`
prefix:

| Convention | Sister-app precedent | LFB adoption |
| --- | --- | --- |
| **pnpm monorepo under `code/`** | `code/packages/{backend,frontend}` + `pnpm-workspace.yaml` | same, plus `packages/shared` for cross-package types/schemas |
| **Backend = TS REST API on Express** | sisters use NestJS-on-Express; LFB's first cut uses **lean Express + TypeScript run via `tsx`** (same `/api` prefix, helmet, fail-closed CORS, zod validation, global error handler) ‚Äî a pragmatic greenfield choice that boots with no build step. The OpenAuthFederated middleware is Express-shaped, so auth wiring is identical to the sisters. NestJS remains the north-star if the API grows. | Express |
| **Frontend = React 19 + Vite 7** | TanStack Query + TanStack Router (code-based) + Tailwind v4 + lucide + sonner + axios | same, **plus TanStack Table v8** (charter: all tables are TanStack) |
| **Auth = OpenAuthFederated `link:` deps** | `@auth/backend` `createFederatedFrontend` mounted at `/api/v1`; `@auth/react` `FederatedProvider`; global `JwtAuthGuard` "identify, don't gate"; `@CurrentUser()`; `FileSessionStore` under state dir; `.auth_session_secret` on disk | same (see ¬ß8) |
| **No database ‚Äî atomic YAML store** | `YamlStore<T>`: tmp‚Üífsync‚Üírename, per-file mutex, defaults-on-absence, `updated_at` stamp | same, extended with the **three scopes** (app / user / sync-unit) LFB needs (¬ß4) |
| **State root `~/T/_<app>`** | `~/T/_act3_app`, `~/T/_emaildeliveryhero`, env override | `~/T/_large_files_bridge`, env `LFB_STATE_DIR` |
| **Rolling logger** | singleton `AppLogger` ‚Üí `log.log` + `error.err`, 5 MiB √ó 5 backups, mirrors WARN+ into both | same |
| **Zod-validated `config.yaml`** | PM-editable content validated fail-fast at boot | same |
| **`left_bar.yaml` imported raw** | frontend `import ‚Ä¶/pm/left_bar.yaml?raw`, Zod-validated, no code copy | same (Vite `server.fs.allow` grants repo-root read) |
| **`justfile` runner** | `setup / build / run / stop / logs / status`, nohup + pidfile, waits on ports | same, plus `install-agents` for the two launchd plists |
| **Secrets out-of-repo** | `~/.credentials/<app>.json`, env wins | `~/.credentials/large_files_bridge.json`, `LFB_CREDENTIALS_FILE`, env wins |

**Where we deliberately differ from the sisters:** (1) LFB has **two computer-wide sync units** and a
real **IPFS** integration ‚Äî the piece neither sister has (¬ß5‚Äì¬ß7). (2) LFB **does** ship **launchd
plists** for its two scheduled workers (EDH ships one; `all/app` ships none) ‚Äî the background scan and
sync are first-class product features with an install/on-off transparency contract (¬ß6). (3) Tables
are built on **`@tanstack/react-table`** (the charter is explicit), whereas the sisters hand-rolled a
`DataTable`; we keep their toolbar/filter/sort *design* but drive it with the TanStack table model.

## 4. Storage / store layer (implements storage.mdx)

`YamlStore<T>` (sister `yaml-store.ts` extended): serialize ‚Üí unique `*.pid.seq.tmp` ‚Üí `fsync` ‚Üí
`rename`; per-file async mutex for read-modify-write; returns typed **defaults on absence** (no error,
no first-run write); **Zod-validates on read**, rejecting malformed files loudly (storage.mdx ¬ß15).

**Three scopes** (`scopes.ts`):

* **App scope** ‚Äî `config.yaml`, `peers.yaml` at the state root.
* **User scope** ‚Äî `users/<email>/config.yaml`, `users/<email>/sessions/` ‚Äî created lazily; email
  lowercased + sanitized; **no `default`/anon key** (storage.mdx ¬ß4).
* **Unit scope** ‚Äî `sync/computer/{config,manifest,status}.yaml` and
  `sync/r/<repo_name>/{config,manifest,status}.yaml`. The unit opener is what makes sync state
  computer-wide (outside `users/`).

Every persisted file matches the storage.mdx ¬ß14 inventory; the Zod schemas live in `@lfb/shared` so
BE and FE agree on shape. `big_file.threshold_bytes` (settings.mdx) supersedes the legacy
`large_file_min_bytes` / `large_files.min_bytes` fields (scan.mdx ¬ß9 reconciliation) ‚Äî the shared
`resolveThreshold(unit, appConfig)` is the single source.

## 5. The two sync units (implements storage.mdx ¬ß5‚Äì¬ß9)

* **Repo unit (PRIMARY)** ‚Äî one per discovered git repo. Identity `{name, path, remote}` written by
  the scanner; `synced` master toggle (default **off**); `large_files.follow_gitignore` seeds the
  bridge set from the repo's `.gitignore`; per-repo `big_file_override`; `sync` pin/fetch/publish
  policy; `access.shared/participants`. Manifest is **also committed** to
  `<repo>/.lfbridge/manifest.yaml` so git carries the list; IPFS carries the bytes.
* **Computer unit** ‚Äî everything **outside** every registered repo, with repo subtrees **masked
  out** (`mask.ts`). Its manifest travels over **IPNS** (no repo to carry it), or a peer-shared record.

A path belongs to exactly one unit. Discovering a repo instantly removes its subtree from the computer
unit's scope. `pinned_by` is the redundancy record; a file pinned by only one machine is surfaced
as at-risk.

## 6. The two scheduled workers + transparency contract (scan.mdx + storage.mdx ¬ß13)

| Worker | Cadence | Job | launchd label | Config block |
| --- | --- | --- | --- | --- |
| **Scan** | 4 h (`StartInterval 14400`) | metadata-only discovery: stat big files, diff, write `status.yaml`. Never opens/hashes files, never touches IPFS. | `com.largefilebridge.scan` | `config.yaml ‚Üí scan_process` |
| **Sync** | 15 min (`StartInterval 900`) | for each `synced:true` unit: add‚ÜíCID‚Üípin‚Üímanifest‚Üífetch missing‚Üípublish list‚Üí`status.yaml`. | `com.largefilebridge.sync` | `config.yaml ‚Üí sync_process` |

Both carry two **independent** facts ‚Äî **installed** (plist present) and **enabled** (bootstrapped/on)
‚Äî reconciled against real `launchctl` state and surfaced on the **Sync** page. The web app can
install/uninstall, toggle on/off, and **Rescan now / Sync now** on demand. `RunAtLoad=false` so the
plist honors the on/off toggle. `schedule.service.ts` fronts an OS-agnostic `installer` (mac launchd
primary; systemd/Windows secondary behind the same interface). The launchd `ProgramArguments` run
`node deploy/launchd/run-worker.mjs {scan|sync}`, which POSTs the loopback-only
`/api/internal/run/<worker>` route ‚Äî the actual work runs in the app's TypeScript.

## 7. IPFS integration (implements knowledge/ipfs.mdx ‚Äî the only-our-content rule)

`ipfs.service.ts` talks to a local **Kubo** node over its HTTP RPC (`config.yaml ‚Üí ipfs.api_addr`,
default `/ip4/127.0.0.1/tcp/5001`) using `fetch`. Operations: `id`, `add -r` (‚Üí root CID, recursive
pin), `pin add/rm/ls`, `name publish/resolve` (IPNS for the computer unit), `swarm connect` (direct
peer the user's own machines), `config` (read/set).

**Compliance, enforced at startup and re-checked (charter is strict):**
* Set **`Reprovider.Strategy` = `pinned`** (never `all`) ‚Äî announce only our own pinned CIDs.
* **No public/recursive gateway** ‚Äî reads use the **local** gateway on `127.0.0.1` only.
* Keep GC on so incidental cache is transient; pin only our own content.
* Direct-peer the user's machines (`peers.yaml`) instead of leaning on the public DHT alone.
* A settings flag can opt a specific machine out of the default (charter's one allowed exception);
  otherwise the app **flags out-of-compliance state** on the Settings ‚Üí IPFS panel and offers to fix.

**Graceful degradation:** if the daemon is unreachable, `ipfs.service` reports `unreachable`; the UI
shows the red IPFS pill (one_repo.mdx ¬ß3.2), disables Sync-now, and lets decisions still be edited
(they queue). The app **boots without IPFS installed** ‚Äî IPFS is required to *move bytes*, not to run.

## 8. Authentication ‚Äî OpenAuthFederated (charter + storage.mdx ¬ß4/¬ß10)

Consumed exactly as both sisters do, via pnpm `link:` to
`~/BGit/Bryan_git/OpenAuthFederated/code/packages/{auth-backend,auth-react}`. **Embedded mode** ‚Äî the
real Google Workspace OIDC round-trip runs in-process; app session + access tokens are HS256 signed
with one on-disk secret. No separate auth server, no JWKS to host.

**Backend:** `buildAuthFrontend(cfg)` ‚Üí `createFederatedFrontend({ connections:[{ strategy:
"oauth_google", clientId, clientSecret, redirectUri }], allowedDomains, sessionSecret:
loadOrCreateSecret(<stateDir>/.auth_session_secret), issuer: "large-file-bridge", cookiePrefix:
"oaf_lfb", sessionStore: new FileSessionStore(stateDir) })`, mounted `app.use("/api/v1", ‚Ä¶)`.
`auth.strategy.ts` verifies the Bearer token via `federatedClient.verifyToken` and re-checks the
identity against the allow-list (defense in depth). Google client id/secret come from
`~/.credentials/large_files_bridge.json` or env (env wins) ‚Äî never `config.yaml`.

**Allow-list ‚Äî no anonymous account.** The charter requires *allow-listed users only* and *no
no-login path*. OpenAuthFederated's shipped enforcement is a **domain** allow-list (`allowedDomains`);
LFB additionally enforces a **per-email** allow-list from `config.yaml ‚Üí access.allowed_emails` in
`auth.strategy.ts` (the library's per-email allowlist is spec-only today, so we enforce it app-side).
`JwtAuthGuard` still uses the sister "identify, don't gate" `DEFAULT_USER` so the API never 500s, but
LFB routes are wrapped so an unauthenticated/allow-list-miss principal gets **no data** and the SPA
redirects to `/sign-in`. Allow-list *editing* is gated `@RequireRole("admin")`.

**Frontend:** `<FederatedProvider frontendApi="/api" allowedDomains={‚Ä¶} signInUrl="/sign-in"
afterSignOutUrl="/">`; `AuthBridge` registers `getToken`/`reloadSession` into the axios layer;
request interceptor attaches `Authorization: Bearer`; response interceptor retries once on 401 via
`reloadSession`, toasts on 403. `SignInPage` uses `<SignIn>` (Google button, no password);
`SsoCallbackPage` uses `<AuthenticateWithRedirectCallback>`.

## 12. Build order (how the code lands)

1. **Scaffold** ‚Äî workspace, tsconfigs, `@lfb/shared` (types + schemas + units), justfile, launchd
   templates.
2. **Store + config + logging** ‚Äî `yaml-store`, `scopes`, `app-config`, `AppLogger`, `state-dir`.
3. **Auth** ‚Äî `link:` OpenAuthFederated, `auth-frontend`, strategy, guards; boots with DEFAULT_USER
   even without Google creds.
4. **IPFS service** ‚Äî Kubo RPC + compliance; degrades when daemon absent.
5. **Scanner + sync + schedule** ‚Äî the two workers, manifest read/write, launchd install; the
   internal trigger route.
6. **REST modules** ‚Äî repos/files/settings/peers/sync/health.
7. **Frontend** ‚Äî router + shell + left bar + DataTable + the five pages + auth flow.
8. **Wire + verify** ‚Äî `pnpm -r build` typechecks; `just run` boots :8080 + :8787; sign-in screen
   renders; `/api/health` + `/api/repos` respond; a scan populates the Repos table.

**Sequencing note:** steps 2‚Äì6 are backend and can proceed while 7 (frontend) is built against the
frozen `@lfb/shared` types ‚Äî the two sides only meet at the ¬ß10 REST contract.

## 13. Deferred / phased (real but not in the first cut)

To get the **web app running first** (charter's stated priority) without faking anything, these ship
as correct-but-minimal and are called out honestly rather than hidden:

* **Dropbox / Google Drive backbones** ‚Äî the Sync List **reconcile engine and the git backbone** are
  first-class; the Dropbox/Drive file-copy adapters implement the same backbone interface but are the
  second wave (sync_list.mdx ¬ß5 trust rules already apply to all three).
* **IPNS computer-unit transport** ‚Äî implemented against Kubo `name publish/resolve`; latency/tuning
  (storage.mdx ¬ß17 open question) revisited after the repo (git) path is proven.
* **Compression + perceptual-hash detection** (charter "core value" + PhotoDNA-style local matching)
  ‚Äî the **learned size-baseline YAML** and the compress/convert *offers* are a dedicated later
  milestone; nothing compresses or alters a file without explicit user action, ever. The category
  rollup table (charter) reuses the same DataTable when it lands.
* **Windows / Linux schedulers** ‚Äî behind the `installer` interface; **macOS launchd is the shipped,
  first-class path**.

Each deferred item has a seam in the code (an interface or a stubbed service that logs "not yet
enabled"), so turning it on is additive, never a rewrite.
