# LargeFileBridge ‚Äî Code Plan

This is the **engineering plan** for LargeFileBridge (LFB). It reads every spec in `pm/` and the
`knowledge/ipfs.mdx` base, studies the two sister web apps, and turns all of it into a concrete map of
**what code to write, where it lives, and how the pieces run** ‚Äî locally first, server-capable later.

> **Division of labor.** The `pm/*.mdx` specs own *product behavior* (what the app does). This file
> owns *code structure* (how it is built). Where they disagree on behavior, the spec wins; where they
> disagree on layout, this file wins. Code goes **only** in the peer `code/` directory
> (charter rule); this plan lives in `pm/`.

## 1. What we learned from the sister apps (and adopt verbatim)

Both `~/BGit/all/app/` and `~/BGit/Bryan_git/EmailDeliveryHero/` converge on the same shape, and
[storage_local.mdx ¬ßintro](./storage_local.mdx) already says LFB follows it. We adopt it, renamed to the `lfb`
prefix:

| Convention | Sister-app precedent | LFB adoption |
| --- | --- | --- |
| **pnpm monorepo under `code/`** | `code/packages/{backend,frontend}` + `pnpm-workspace.yaml` | same, plus `packages/shared` for cross-package types/schemas |
| **Backend = TS REST API on Express** | sisters use NestJS-on-Express; LFB's first cut uses **lean Express + TypeScript run via `tsx`** (same `/api` prefix, helmet, fail-closed CORS, zod validation, global error handler) ‚Äî a pragmatic greenfield choice that boots with no build step. The OpenAuthFederated middleware is Express-shaped, so auth wiring is identical to the sisters. NestJS remains the north-star if the API grows. | Express |
| **Frontend = React 19 + Vite 7** | TanStack Query + TanStack Router (code-based) + Tailwind v4 + lucide + sonner + axios | same, **plus TanStack Table v8** (charter: all tables are TanStack) |
| **Auth = OpenAuthFederated `link:` deps** | `@auth/backend` `createFederatedFrontend` mounted at `/api/v1`; `@auth/react` `FederatedProvider`; global `JwtAuthGuard` "identify, don't gate"; `@CurrentUser()`; `FileSessionStore` under state dir; `.auth_session_secret` on disk | same (see ¬ß8) |
| **No database ‚Äî atomic YAML store** | `YamlStore<T>`: tmp‚Üífsync‚Üírename, per-file mutex, defaults-on-absence, `updated_at` stamp | same, extended with the **three scopes** (app / user / pin-unit) LFB needs (¬ß4) |
| **State root `~/T/_<app>`** | `~/T/_act3_app`, `~/T/_emaildeliveryhero`, env override | `~/T/_large_files_bridge`, env `LFB_STATE_DIR` |
| **Rolling logger** | singleton `AppLogger` ‚Üí `log.log` + `error.err`, 5 MiB √ó 5 backups, mirrors WARN+ into both | same |
| **Zod-validated `config.yaml`** | PM-editable content validated fail-fast at boot | same |
| **`left_bar.yaml` imported raw** | frontend `import ‚Ä¶/pm/left_bar.yaml?raw`, Zod-validated, no code copy | same (Vite `server.fs.allow` grants repo-root read) |
| **`justfile` runner** | `setup / build / run / stop / logs / status`, nohup + pidfile, waits on ports | same, plus `install-agents` for the two launchd plists |
| **Secrets out-of-repo** | `~/.credentials/<app>.json`, env wins | `~/.credentials/large_files_bridge.json`, `LFB_CREDENTIALS_FILE`, env wins |

**Where we deliberately differ from the sisters:** (1) LFB has **two computer-wide pin units** and a
real **IPFS** integration ‚Äî the piece neither sister has (¬ß5‚Äì¬ß7). (2) LFB **does** ship **launchd
plists** for its two scheduled workers (EDH ships one; `all/app` ships none) ‚Äî the background scan and
pin are first-class product features with an install/on-off transparency contract (¬ß6). (3) Tables
are built on **`@tanstack/react-table`** (the charter is explicit), whereas the sisters hand-rolled a
`DataTable`; we keep their toolbar/filter/sort *design* but drive it with the TanStack table model.

Ports (from [storage_local.mdx ¬ß3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)):
the **web app** ‚Äî the browser-facing page server ‚Äî **always defaults to :2222** (Vite). The backend
REST API is **8787** (`/api`); Vite proxies `/api` ‚Üí `http://localhost:8787`. The IPFS *gateway*
defaults to :8080 in Kubo, which no longer collides with the UI now that the web app lives on :2222;
the gateway addr stays configurable in `config.yaml ‚Üí ipfs` (Settings ‚Üí IPFS node).

**Web-app port collision policy (the :2222 rule).** The web port is resolved on boot before Vite
binds (`packages/frontend/scripts/web-port.mjs`, called from `vite.config.ts`):

1. **:2222 is free** ‚Üí take it.
2. **:2222 is held by our OWN app** (a stale/duplicate LargeFileBridge instance, identified by the
   stable `<meta name="x-app" content="large-file-bridge">` marker served at `/`) ‚Üí **kill that
   instance and reclaim :2222**.
3. **:2222 is held by a FOREIGN process** ‚Üí do **not** disturb it; **increment by one** (2223,
   2224, ‚Ä¶) until we find a free-or-ours port, and **report the chosen port to the user**.

The resolved port is written to `/tmp/lfb.web.port` for tooling (`justfile`), and `strictPort` is set
because the port is already decided. Backend CORS accepts any localhost origin in `mode: local` so a
collision-moved web port still reaches the API; `mode: server` fails closed to `server.cors_origins`.

## 4. Storage / store layer (implements storage_local.mdx)

`YamlStore<T>` (sister `yaml-store.ts` extended): serialize ‚Üí unique `*.pid.seq.tmp` ‚Üí `fsync` ‚Üí
`rename`; per-file async mutex for read-modify-write; returns typed **defaults on absence** (no error,
no first-run write); **Zod-validates on read**, rejecting malformed files loudly (storage_local.mdx ¬ß15).

The atomic YAML store (tmp‚Üífsync‚Üírename, per-file mutex, defaults-on-absence, schema-validate-on-read) is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/store/yaml-store.ts` ‚Äî `readYaml()` / `writeYaml()` / `updateYaml()`

**Three scopes** (`scopes.ts`):

* **App scope** ‚Äî `config.yaml`, `peers.yaml` at the state root.
* **User scope** ‚Äî `users/<email>/config.yaml`, `users/<email>/sessions/` ‚Äî created lazily; email
  lowercased + sanitized; **no `default`/anon key** (storage_local.mdx ¬ß4).
* **Unit scope** ‚Äî `pin/computer/{config,manifest,status}.yaml` and
  `pin/r/<repo_name>/{config,manifest,status}.yaml`. The unit opener is what makes pin state
  computer-wide (outside `users/`).

Every persisted file matches the storage_local.mdx ¬ß14 inventory; the Zod schemas live in `@lfb/shared` so
BE and FE agree on shape. `big_file.threshold_bytes` (settings.mdx) supersedes the legacy
`large_file_min_bytes` / `large_files.min_bytes` fields (scan.mdx ¬ß9 reconciliation) ‚Äî the shared
`resolveThreshold(unit, appConfig)` is the single source.

## 5. The two pin units (implements storage_local.mdx ¬ß5‚Äì¬ß9)

* **Repo unit (PRIMARY)** ‚Äî one per discovered git repo. Identity `{name, path, remote}` written by
  the scanner; `pinned` master toggle (default **off**); `large_files.follow_gitignore` seeds the
  bridge set from the repo's `.gitignore`; per-repo `big_file_override`; `pin` pin/fetch/publish
  policy; `access.shared/participants`. The manifest lives in **Local Storage** at
  `~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml` (never the working repo) and travels across the
  user's computers via the **company sync repo** when enabled, not the working repo
  ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)); IPFS carries the bytes.
* **Computer unit** ‚Äî everything **outside** every registered repo, with repo subtrees **masked
  out** (`mask.ts`). Its manifest travels over **IPNS** (no repo to carry it), or a peer-shared record.

A path belongs to exactly one unit. Discovering a repo instantly removes its subtree from the computer
unit's scope. `pinned_by` is the redundancy record; a file pinned by only one machine is surfaced
as at-risk.

## 7. IPFS integration (implements knowledge/ipfs.mdx ‚Äî the only-our-content rule)

`ipfs.service.ts` talks to a local **Kubo** node over its HTTP RPC (`config.yaml ‚Üí ipfs.api_addr`,
default `/ip4/127.0.0.1/tcp/5001`) using `fetch`. Operations: `id`, `add -r` (‚Üí root CID, recursive
pin), `pin add/rm/ls`, `name publish/resolve` (IPNS for the computer unit), `swarm connect` (direct
peer the user's own machines), `config` (read/set).

**Compliance, enforced at startup and re-checked (charter is strict):**
* Set **`Reprovider.Strategy` = `pinned`** (never `all`) ‚Äî announce only our own pinned CIDs.
* **No public/recursive gateway** ‚Äî reads use the **local** gateway on `127.0.0.1` only.
* Keep GC on so incidental cache is transient; pin only our own content.
* Direct-peer the user's machines (`peers.yaml`) instead of leaning on the public DHT alone.
* A settings flag can opt a specific machine out of the default (charter's one allowed exception);
  otherwise the app **flags out-of-compliance state** on the Settings ‚Üí IPFS panel and offers to fix.

**Graceful degradation:** if the daemon is unreachable, `ipfs.service` reports `unreachable`; the UI
shows the red IPFS pill (one_repo.mdx ¬ß3.2), disables Pin-now, and lets decisions still be edited
(they queue). The app **boots without IPFS installed** ‚Äî IPFS is required to *move bytes*, not to run.

## 8. Authentication ‚Äî OpenAuthFederated (charter + storage_local.mdx ¬ß4/¬ß10)

Consumed exactly as both sisters do, via pnpm `link:` to
`~/BGit/Bryan_git/OpenAuthFederated/code/packages/{auth-backend,auth-react}`. **Embedded mode** ‚Äî the
real Google Workspace OIDC round-trip runs in-process; app session + access tokens are HS256 signed
with one on-disk secret. No separate auth server, no JWKS to host.

**Backend:** `buildAuthFrontend(cfg)` ‚Üí `createFederatedFrontend({ connections:[{ strategy:
"oauth_google", clientId, clientSecret, redirectUri }], allowedDomains, sessionSecret:
loadOrCreateSecret(<stateDir>/.auth_session_secret), issuer: "large-file-bridge", cookiePrefix:
"oaf_lfb", sessionStore: new FileSessionStore(stateDir) })`, mounted `app.use("/api/v1", ‚Ä¶)`.
`auth.strategy.ts` verifies the Bearer token via `federatedClient.verifyToken` and re-checks the
identity against the allow-list (defense in depth). Google client id/secret come from
`~/.credentials/large_files_bridge.json` or env (env wins) ‚Äî never `config.yaml`.

**Allow-list ‚Äî no anonymous account.** The charter requires *allow-listed users only* and *no
no-login path*. OpenAuthFederated's shipped enforcement is a **domain** allow-list (`allowedDomains`);
LFB additionally enforces a **per-email** allow-list from `config.yaml ‚Üí access.allowed_emails` in
`auth.strategy.ts` (the library's per-email allowlist is spec-only today, so we enforce it app-side).
`JwtAuthGuard` still uses the sister "identify, don't gate" `DEFAULT_USER` so the API never 500s, but
LFB routes are wrapped so an unauthenticated/allow-list-miss principal gets **no data** and the SPA
redirects to `/sign-in`. Allow-list *editing* is gated `@RequireRole("admin")`.

**Frontend:** `<FederatedProvider frontendApi="/api" allowedDomains={‚Ä¶} signInUrl="/sign-in"
afterSignOutUrl="/">`; `AuthBridge` registers `getToken`/`reloadSession` into the axios layer;
request interceptor attaches `Authorization: Bearer`; response interceptor retries once on 401 via
`reloadSession`, toasts on 403. `SignInPage` uses `<SignIn>` (Google button, no password);
`SsoCallbackPage` uses `<AuthenticateWithRedirectCallback>`.

## 14. The File System column browser (implements directories.mdx)

The **File System** page ([directories.mdx](./directories.mdx)) is a **Finder column-view (Miller-column)
browser** over the raw filesystem, with a per-entry **code-badge** overlay. It is deliberately **not**
a TanStack table (it is a navigator, not a rollup) ‚Äî the charter's table rules do not apply here.

### 14.1 Backend ‚Äî `modules/fs/`

One thin REST module that lists **one directory level per request** (the column browser fetches lazily,
column by column):

* **`fs.router.ts`** ‚Äî `GET /api/fs?path=<abs>` returns an `FsListing` (the entries of that one
  directory, each with badges); `GET /api/fs/home` returns the default root. Both are wrapped with
  `requireAllowListed` like every other data route (¬ß8). `path` is validated: must be absolute, is
  server-normalized/`realpath`-resolved, and symlink-escape and traversal (`..`) are rejected ‚Äî the
  browser reads **metadata only**, never file contents for the walk itself.
* **`fs.service.ts`** ‚Äî `readdir(withFileTypes)` for the requested directory (Node `fs`, **never** the
  `find`/`du` shell ‚Äî charter + macOS-indexing rule), `stat` for size/mtime, normalize paths per-OS,
  hide dotfiles by default, collapse the hard-skip set (`.git`, `node_modules`, `.Trash`, caches ‚Äî the
  same set as [scan.mdx ¬ß4](./scan.mdx)), sort **directories-first then alphabetical**, and attach
  `badges` from `badges.ts`.
* **`badges.ts`** ‚Äî the badge computation (directories.mdx ¬ß3), all metadata-cheap:
  * **Repo (R/r, brown).** `isRepoRoot = isGitWorkingTree(dir)` (a `.git` child ‚Äî reused from
    `units.service.ts`); **descendant** = the listed directory is itself inside a repo (walk up once
    from the column's path, cached per request); **ancestor** = a directory that contains a repo below
    it ‚Äî resolved from the **registered repo set** (`listRepoFolders()` paths) plus a **bounded,
    depth-limited** downward probe (small budget shared across the listing) so unregistered repos still
    register as ancestors. Precedence root ‚Üí descendant ‚Üí ancestor, at most one.
  * **Pin (P, pink).** The file's decision is `sync`: map the file to its owning unit (the registered
    repo whose path is a prefix, else the computer unit) and read `config.decisions[relPath]`.
  * **Compress (C/c, yellow).** Video/image by extension (charter scope). First cut: a **heuristic**
    (`compress.ts` helper) ‚Äî lossless/uncompressed image formats (png/bmp/tiff) and large videos ‚Üí
    `C`; formats/records that are already efficiently compressed ‚Üí `c`. Seam: swap the heuristic for
    the **learned size-baseline** model (¬ß13) without touching the badge contract.
  * **IPFS (i, blue).** File name ‚àà {`IPFS.sh`,`ipfs.sh`,`ipfs.txt`,`get_videos.sh`} or a public
    `manifest.yaml` with `ipfs_cid` entries, **or** a `.md`/`.mdx` (under a size cap) whose text carries
    IPFS signals (`ipfs pin add`, `/ipfs/<CID>`, `ipfs_cid`); a **directory** earns `i` when it
    directly contains/publishes such an artifact (a bounded one-level check, incl. a `static/` symlink).
    This mirrors [ipfs_share_files.mdx](./ipfs_share_files.mdx) ‚Äî **detect-and-report only**, never a
    publish action, and it never turns the node into a public gateway
    ([knowledge/ipfs.mdx ¬ß6](../knowledge/ipfs.mdx)).

  Badges are returned **ordered rightmost-first** ‚Äî `[repo?, pin?, compress?, ipfs?]` ‚Äî so the
  frontend renders array order right-to-left (directories.mdx ¬ß3.5).

The File System column-browser REST routes are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.router.ts` ‚Äî `fsRouter` (`GET /api/fs`, `GET /api/fs/home`)

The one-directory-level listing (readdir + stat + per-entry code badges) is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` ‚Äî `listDirectory()`

### 14.2 Shared types (`@lfb/shared`)

`FsBadge` (the seven-value union), `FsEntry`, and `FsListing` from directories.mdx ¬ß5 live in
`packages/shared/src/types.ts`, so backend and frontend agree on the contract.

### 14.3 Frontend ‚Äî `pages/fs/FileSystemPage.tsx` + `components/fs/*`

* **`ColumnBrowser`** holds the **path stack** (an array of selected directory paths). It renders one
  **`FsColumn`** per stack entry; each column TanStack-Query-fetches `GET /fs?path=‚Ä¶` for its level.
  Clicking a directory **truncates the stack at that column and pushes the new path** (replacing all
  columns to the right ‚Äî Finder behavior); clicking a file selects the leaf.
* **`FsColumn`** is a fixed-narrow (~280px), independently vertical-scrolling list of **`FsRow`**s; the
  strip scrolls horizontally and auto-scrolls to the newest column.
* **`FsRow`** shows glyph + middle-truncated name + spacer + **`Badges`**.
* **`Badges`** renders each `FsBadge` as a **white-letter-on-solid-color** chip, right-pinned, in array
  order; colors come from the `--lfb-badge-*` CSS tokens (directories.mdx ¬ß4); each chip has a `title`
  tooltip. The light-yellow `c` uses dark ink for contrast (the one exception to white letters).
* A breadcrumb above the strip shows the current deepest path with a **Home** jump and an absolute-path
  input; a **Show hidden** toggle flips dotfile visibility.
