# LargeFileBridge вҖ” Task Tabs

The **One-repo** screen ([one_repo.mdx](./one_repo.mdx)) is where the user does the work on a repo's large
files. It now carries a **task tab strip** in the **first row** of its header вҖ” **a little right of
center**, with the repo **name** still on the left (the biggest, boldest text on the page вҖ” it never wraps
and is never squeezed, [one_repo.mdx В§3.1](./one_repo.mdx#31-header--two-rows-and-neither-of-them-wraps-locked))
and the settings **gear** + the smart **"View recommendation" / "Scan now"** header primary (В§1.1) still on
the right. There are **six tabs**:

| Tab | The one question it answers |
| --- | --- |
| **All** *(default)* | *"Across every task, what could I do on this repo right now?"* вҖ” the overview. |
| **IPFS** | *"Which large files are pinned across my computers, and which still need a decision or a peer?"* |
| **Compress** | *"Which videos and images could be compressed, and how big are they?"* |
| **Transcribe** | *"Which audio/video files have no transcript yet?"* |
| **AI descriptions** | *"Which images and videos have no AI description yet?"* |
| **OCR** | *"Which images, videos and PDFs haven't had their on-screen text read yet?"* |

> **The three analysis tabs вҖ” Transcribe, AI descriptions, OCR вҖ” mirror each other point-for-point** (their
> own could/done metric tiles, a three-state status icon, the same metric-tile вҶ’ popup and blue
> header-primary behavior). AI descriptions is specified in В§4.5 (tuning), В§5.2 (the Describe status icon),
> В§5.2.1 (the Describe confirm flow), cross-linking to [ai_description.mdx](./ai_description.mdx) (the
> feature) and [describe_calc_engine.mdx](./describe_calc_engine.mdx) (the bulk-discovery engine); OCR is
> В§4.6 here and [ocr.mdx](./ocr.mdx) end to end. Everywhere this spec says "Transcribe / Compress," read
> "and Describe, and OCR" too вҖ” the task tabs share one grammar.

**One tab is selected at a time, and selecting it re-tunes the whole surface below it** вҖ” the
**metrics-panel strip**, the **visible table columns**, the **default sort**, and **which files the table
shows**. This spec owns that tab system end to end; the sibling specs own their own tasks and only point
here for how the tab presents them.

The tab strip and the per-tab tuning are rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` вҖ” `OneRepoPage`

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/taskTabs.config.ts` вҖ” `TASK_TABS`).

This spec builds on the siblings and does not repeat them:

### 1.2 Selecting a tab re-tunes four things (LOCKED)

Switching tabs is a **client-side view change** вҖ” it never refetches the repo and never mutates
anything. The active tab drives, in one place ([taskTabs.config.ts](./one_repo.mdx) вҖ” `TASK_TABS`):

1. **The metrics-panel strip** вҖ” which "what could be done" panels show (В§2, В§4).
2. **The visible table columns** вҖ” a tab-scoped column set (В§4, В§7).
3. **The default sort** вҖ” what is most interesting for that task floats to the top (В§4).
4. **The row filter** вҖ” which files are appropriate for that task (В§4); files outside the tab's scope are
   hidden (still reachable on **All**).

### 1.3 Active-tab state (LOCKED)

* **Default = All** on every fresh open of a repo.
* The active tab is **ephemeral view state** (React state on `OneRepoPage`), **not** written to
  `config.yaml`. It is *not* a decision and *not* a per-user table preference вҖ” the per-user **sort /
  filter** persistence ([tables.mdx В§4](./tables.mdx)) still applies *within* whatever tab is active.
* Switching tabs is **instant** (no spinner) вҖ” the file set is already loaded; the tab only re-projects it.

#### 1.3.1 A tab's `rowFilter` is scope вҖ” it is NOT a facet (LOCKED)

A tab hides rows (В§4), and so does a filter facet ([tables.mdx В§2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked)).
They are different things and must never be merged:

* **The tab's `rowFilter` is scope, chosen by the tab.** It never appears on the **facet rail**
  ([tables.mdx В§2.2](./tables.mdx#22-the-facet-rail--filtering-without-opening-a-dropdown-locked)), it has
  no checkboxes, and the user does not un-set it вҖ” they **switch tabs** (or go to **All**, which has no
  `rowFilter`). It is honest despite being invisible on the rail because **the tab itself is the visible
  control**: a lit "Compress" tab is a plain statement that you are looking at compressible files.
* **The two compose by AND.** The tab picks the row scope; the rail's facets then cut within it. A facet
  never widens a tab's scope вҖ” unchecking nothing on the rail still shows only the tab's rows.
* **The tab's `defaultSort` is a default, not a lock.** It seeds the multi-level sort for that tab; the
  user's own re-rank persists and wins ([tables.mdx В§3.4](./tables.mdx#34-the-default-sort--bookmarks-first-locked-default-user-changeable)),
  and it displays in the active-sort **chip run** ([tables.mdx В§3.5](./tables.mdx#35-sorting-without-opening-the-dropdown--the-active-sort-chips-locked))
  like any other sort вҖ” so a tab's opinion about ordering is always readable, never hidden.
* **The IPFS tab is the pin axis only.** Its `rowFilter` and its urgency sort concern **Add-to-IPFS**
  state; they say nothing about git-ignore, and no tab may filter on both axes at once
  ([tables.mdx В§2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).

### 1.4 The overflow chevron вҖ” the repo name never loses (LOCKED)

**The rule this protects.** The repo **NAME is the biggest, boldest text on the page**. It must **never
wrap** and must **never be squeezed** by the tab strip
([one_repo.mdx В§3.1](./one_repo.mdx#31-header--two-rows-and-neither-of-them-wraps-locked)). Six tabs plus a
gear plus a primary button will not fit beside a long name on a narrow window, so **something has to give вҖ”
and it is never the name.**

* **The strip is width-measured.** It keeps the tabs that **fit** and moves the rest into a trailing
  **down-chevron (`вҢ„`) overflow menu**.
* **Overflow is always taken from the RIGHT.** Tabs are dropped in reverse order (OCR first, then AI
  descriptions, вҖҰ), so the leftmost tabs вҖ” the ones the user reaches for most вҖ” stay put and the strip's
  order never reshuffles.
* **The chevron renders ONLY when something overflows.** When all six fit there is **no chevron and no
  chrome at all** вҖ” not a greyed-out button, not a reserved gap. A user on a wide window never knows the
  mechanism exists.
* **An ACTIVE tab that would land in the overflow is swapped back into the visible set** (trading places
  with the last visible tab), so **the underline is never hidden inside the menu**. The user can always see
  which tab they are on.
* **Selecting from the overflow menu is identical in every way to clicking a visible tab** вҖ” same
  `onChange`, same client-side re-tune of metrics / columns / sort / filter (В§1.2), no refetch, no mutation.
  The menu is a placement detail, never a second behavior.

**Measurement technique.** A **hidden, off-screen copy of the strip with identical markup** is rendered
alongside the real one; each measured tab's **right edge** (`offsetLeft + offsetWidth`) and the **chevron's
width** are read from it, and the fit walks leftвҶ’right keeping tabs while they fit, reserving room for the
chevron. A **`ResizeObserver`** on the wrapper refits on every width change. This is the **same technique
the page action-links row uses for its "More в–ҫ"** ([page_actions.mdx В§3](./page_actions.mdx)) вҖ” one measured
strip, one overflow menu, no layout thrash.

Implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/TaskTabs.tsx` вҖ” `TaskTabs` (the
hidden measurement layer, the `useLayoutEffect` fit + `ResizeObserver`, the `visibleCount` split, the
active-tab swap-back, and the `ChevronDown` menu).

## 2. The metrics panels вҖ” the "what could be done" boxes (LOCKED)

Above the table sits a horizontal strip of **metric panels**. Each panel is a **terse label + a number**
answering *"how many of this thing could I do here?"* вҖ” e.g. `Add to IPFS 5`, `Git Ignore 12`,
`Compressible videos 12`, `Transcribable 71`. **This is the surface the warnings live on** for this screen:
on the **All** tab the panels are the whole focus (an at-a-glance overview of everything actionable); on a
task tab only that task's panels show.

**The strip takes the FULL page width and is ONE row that never wraps** вҖ” `flex-nowrap` with `shrink-0`
tiles; on a pathologically narrow window the row **scrolls horizontally** rather than breaking a line
([one_repo.mdx В§3.2](./one_repo.mdx#32-the-metrics-strip-never-wraps--and-the-explanation-lives-in-the-left-bar-locked)).
Nothing is docked beside it and nothing reserves width from it вҖ” the explanation goes to the **left-bar
hover-info panel** (В§3).

The metrics strip is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/MetricsStrip.tsx` вҖ” `MetricsStrip`
(one `MetricPanel` per active-tab metric in a single `flex w-full flex-nowrap вҖҰ overflow-x-auto` row;
counts from `RepoDetail` per В§2.5).

### 2.1 Anatomy of one panel (LOCKED)

A panel is a small **rounded rectangle**, kept **as narrow as possible** вҖ” **only as wide as its widest
word** (CSS `width: min-content`) вҖ” holding a vertical stack, top to bottom:

* a **terse label** вҖ” *what it is*, no sentence, no description text (the explanation goes to the left-bar
  hover-info panel, В§3). One to three words: `Add to IPFS`, `Git Ignore`, `Not Backed Up`, `Pull Down`,
  `Compressible Videos`, `Transcribable`, `AI Describable`, `OCRable`. The label is **small** and **wraps**
  onto as many lines as the narrow box needs вҖ” the label wraps *inside* the tile; the **strip** never wraps.
* **Casing is EXACT, never CSS-`capitalize`d (LOCKED).** The catalog stores **the exact string to render**,
  because labels carry meaningful casing a `capitalize` rule would fight: **`Add to IPFS`**, **`AI
  Describable`**, **`OCRable`** ([one_repo.mdx В§3.2.1](./one_repo.mdx)). No presentational transform is
  applied to a metric label anywhere.
* the **number** вҖ” **big and centered**, `tabular-nums`, sitting under the wrapped label. The number is
  the visual center of the panel; the label words sit above it.

The panel is a **whole-panel button** вҖ” clicking anywhere on it opens that metric's warning popup (В§2.4);
there is **no chevron** (the narrow box has no room for one and needs none вҖ” the whole tile is the
affordance).

### 2.4.1 Every actionable metric tile opens a popup вҖ” the transcribe / compress / describe builders (the fix) (LOCKED)

**The bug this fixes.** Historically only the `undecided` (now `addToIpfs`), `pullDown`, and `ipfs-down` metrics had a
`WarningDef` builder. So when the user clicked the **`Transcribable`**, **`Compressible videos`**,
**`Compressible images`**, or **`AI Describable`** tile, `buildMetricWarning()` returned `null` and the tile
**did nothing useful** вҖ” it fell through to its `onOpen` fallback, which merely **re-selects the tab the
user is already on** (e.g. clicking `Transcribable` on the Transcribe tab re-selected Transcribe вҖ” a
visible no-op). The header, likewise, only ever offered `IPFS-down вҶ’ pull-down вҶ’ undecided`, so a repo
whose *only* pending work was "58 files could be transcribed" showed **no** blue primary at all вҖ” just the
gear. The task tiles looked clickable but led nowhere. **This is the reported defect.**

**The fix вҖ” a builder for every actionable metric.** Each task metric now has its own educate-and-fix
`WarningDef` builder, so clicking its tile opens the **same** `WarningPopup` the wired metrics already open,
scoped to exactly the files the number counts, and lets the user **start the work** from there:

| Metric tile | Builder | Popup it opens | Apply enqueues |
| --- | --- | --- | --- |
| **`Transcribable`** | `buildTranscribeWarning` | the transcribe popup вҖ” the counted candidates **checked by default** (В§5.1) | background `transcribe` jobs, model-gated (В§5.1, [transcribe_engine.mdx В§3](./transcribe_engine.mdx)) |
| **`AI Describable`** | `buildDescribeWarning` | the describe popup вҖ” the counted candidates **checked by default** (В§5.2.1) | background `describe` jobs, provider/model-gated ([ai_description.mdx В§2](./ai_description.mdx)) |
| **`Compressible Videos`** | `buildCompressWarning` (`kind: "video"`) | the compress popup over the counted videos ([compression.mdx](./compression.mdx)) | background `compress` jobs |
| **`Compressible Images`** | `buildCompressWarning` (`kind: "image"`) | the compress popup over the counted images ([compression.mdx](./compression.mdx)) | background `compress` jobs |

* **`buildMetricWarning()` gains those cases.** Its `switch` now returns the transcribe / describe /
  compress-video / compress-image / OCR `WarningDef`s in addition to `addToIpfs` / `gitIgnore` / `pullDown`;
  it still returns `null` only for the genuinely popup-less metrics (`notBackedUp`, `pending`,
  `alreadyCompressed`, `transcribed`, `described`, `ocred`) whose tiles keep their tab-switch fallback.
  (`bigNotIgnored` is not on that list any more вҖ” it no longer exists as a `MetricId`, В§2.5.1.)
* **The `>0` tint is unchanged** (В§2.3): these are amber "offer" tiles, so the popup is an *offer to start*,
  never a data-at-risk alarm.
* **A green-`0` tile still opens the all-clear popup** (В§2.2) вҖ” "Nothing to transcribe / describe / compress
  here."
* **Apply is gated exactly as the single-file path is.** The transcribe and describe applies run behind the
  same **model-readiness / consent** gate the per-row status icon uses (`withModelReady` for transcribe,
  the provider-key + model check for describe) вҖ” so a first click that needs the heavyweight model or an AI
  key **interposes the permission/provisioning flow, then resumes** the batch (В§5.1,
  [transcribe_engine.mdx В§3.6](./transcribe_engine.mdx), [ai_description.mdx В§2](./ai_description.mdx)).

### 2.7 The per-metric blue header primary вҖ” every actionable metric can be the recommendation (LOCKED)

Because every actionable metric now has a popup (В§2.4.1), every actionable metric can also be the single
**header primary** вҖ” the smart button top-right ([one_repo.mdx В§3.1](./one_repo.mdx#31-header--two-rows-and-neither-of-them-wraps-locked)) that a
non-expert reads as *"the one thing to do next."* Before the fix that button only knew three
recommendations (`IPFS-down вҶ’ pull-down вҶ’ undecided`) and wore one generic label, **"View recommendation
вҖә"**. The fix widens it to the whole task set and **labels the button per metric** so the verb is explicit.

**Worst-first order (LOCKED).** `topRecommendation()` walks the pending work in this fixed severity order
and returns the **first** one that is pending; that becomes the header primary:

```
ipfs-down  вҶ’  pull-down  вҶ’  add-to-IPFS  вҶ’  transcribe  вҶ’  describe  вҶ’  compress  вҶ’  ocr
  (bad)       (bad)          (warn)          (warn)         (warn)       (warn)      (warn)
```

* **The first three are data-integrity first**: a paused IPFS engine, then peer-pinned files not here yet,
  then files that still need a **pin** decision вҖ” the **`addToIpfs`** metric (В§2.5.1), which is the tile
  formerly called *Undecided*. **`gitIgnore` is a tile, not a recommendation link** вҖ” it is an offer at the
  user's leisure, and it never outranks a backup problem.
* **Then the producing tasks**, in the order **transcribe вҶ’ describe вҶ’ compress вҶ’ ocr** вҖ” transcription and
  description are content the user is usually chasing (searchable text, alt-text) and rank above the
  space-reclaiming compress offer; **OCR ranks last** because missing OCR text is a *search convenience*,
  the cheapest of the set to fix and the least costly to lack ([ocr.mdx В§12.3](./ocr.mdx)). It must never
  outrank a backup problem вҖ” but on a repo where it is the only outstanding item it is, correctly, the whole
  recommendation.

**Per-metric label + color (LOCKED).** The button is **blue** (`--lfb-primary`) for every `warn`-state
recommendation and **red** (`--lfb-bad`) only for a `bad` one (IPFS-down, pull-down). Its **label names the

| Top pending recommendation | Button label | Fill |

### 4.8 Remote-only rows belong on All and IPFS only (LOCKED)

A row in the file list is no longer always a local file. A **remote-only** row вҖ” a file another of the
user's computers has pinned and this one does not have yet, composed from the reconciled manifest with no
local bytes behind it вҖ” is defined and locked in
[storage_company.mdx В§8.5](./storage_company.mdx#85-remote-only-rows--a-file-in-the-list-that-is-not-on-this-disk-locked)
and rendered by [one_repo.mdx В§4.10](./one_repo.mdx#410-remote-only-rows--a-file-in-this-list-that-is-not-on-this-computer-locked).
This section says only where the **tabs** put it:

| Tab | Shows remote-only rows? | Why |
| --- | --- | --- |
| **All** | **Yes** | All is the overview вҖ” *"everything this repo has"* now honestly includes files that exist on the user's other computer. Hiding them here is what made the two-computer case invisible. |
| **IPFS** | **Yes** | The pin axis is exactly what a remote-only row is about: decided to sync, not here yet. It does **not** lead the tab, though вҖ” the IPFS tab sorts `peers` **ascending** then `size` descending, and a remote-only row has **вүҘ1 peer** by construction, so it sits **below** the zero-peer (not-backed-up) rows. See the note under this table. |
| **Compress** | **No** | Nothing to compress вҖ” there are no local bytes. |
| **Transcribe** | **No** | `transcribe` is `na`. |
| **AI descriptions** | **No** | `describe` is `na`. |
| **OCR** | **No** | `ocr` is `na`. |

* **The three analysis tabs must not be padded.** Each of those tabs' `rowFilter` selects files whose task
  status is `could` or `done`; a remote-only row is **`na`** on all three ([one_repo.mdx В§4.10](./one_repo.mdx)),
  so it is filtered out **by the existing rule, with no special case** вҖ” the tab keeps its promise that every
  row on it is work that can actually be started. Showing a row the user cannot act on would inflate the tab
  and hand them a disabled icon as an answer.
* **The metrics strip follows the same line.** A remote-only row raises the **`Pull down`** tile (always) and
  the **`Add to IPFS`** tile (while this computer has not decided it), and raises **no** `Compressible
  videos` / `Compressible images` / `Transcribable` / `AI Describable` / `OCRable` / `Git Ignore` count
  (В§2.5, В§2.5.1). So on a laptop that is simply behind, the analysis tiles stay honest green zeros while
  `Pull down` carries the number.
* **The worst-first header recommendation is unchanged** (В§2.7): `ipfs-down вҶ’ pull-down вҶ’ add-to-IPFS вҶ’
  transcribe вҶ’ describe вҶ’ compress вҶ’ ocr`. Remote-only rows feed the **pull-down** link вҖ” already the second
  and one of the two `bad`-state links вҖ” so the laptop's header primary correctly reads a **red "View
  recommendation вҖә"** that opens the existing Pull-down popup. **No new recommendation link and no new
  ordering is introduced.**
* **Where they sort on the IPFS tab вҖ” the real order.** `TASK_TABS.ipfs.defaultSort` is
  `[{ peers, asc }, { size, desc }]`: **fewest peers first, then biggest first**, so the files **no other
  computer of yours holds** lead the table. A remote-only row exists *because* a peer pinned it, so its
  peer count is вүҘ1 and it ranks **after** those rows. There is no transfer-urgency sort key on this tab.
  **Open question, not built:** whether "bytes you are missing" should outrank "bytes only you have" and
  float remote-only rows to the top. Until that is decided, the order above is the behavior вҖ” see
  [ipfs_ui.mdx В§21.1](./ipfs_ui.mdx#211-what-the-tabs-panels-count-once-a-peers-manifest-arrives-locked).

  The tab's default sort is declared in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/taskTabs.config.ts` вҖ” `TASK_TABS.ipfs.defaultSort`.
* **`Large files only` does not hide them** on any tab ([tables.mdx В§4e](./tables.mdx#4e-remote-only-rows-in-a-file-table-locked-site-wide)):
  a remote-only row is a large file, not an `analysisOnly` row. It is unaffected by the OCR tab's default-OFF
  exception (В§4.6) because the OCR tab does not show it at all.

The tab that admits or excludes a remote-only row is the tab's own `rowFilter` in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/taskTabs.config.ts` вҖ” `TASK_TABS`
(the analysis tabs' `could`/`done` predicates already exclude `na`), over the rows composed by

## 7. How a tab re-tunes the table (mechanism) (LOCKED)

The whole tab system is **data-driven** from one config, so a tab is defined once and the page reads it:

```ts
// pages/repos/taskTabs.config.ts
export type TaskTabId = "all" | "ipfs" | "compress" | "transcribe" | "ai-descriptions" | "ocr";

export interface TaskTabDef {
  id: TaskTabId;
  label: string;
  icon: LucideIcon;
  columnIds: string[];                 // which of OneRepoPage's LfbColumns to show, in order
  defaultSort: SortingState;           // the tab's "most interesting first" sort
  rowFilter: (f: FileRow) => boolean;  // which files belong to this task
  metrics: MetricId[];                 // which MetricPanels the strip shows
  defaultHint: string;                 // what the LEFT-BAR hover panel shows at rest on this tab (В§3)
  largeOnlyDefault: boolean;           // seed for the "Large files only" rail toggle вҖ” false on OCR (В§4.6)
}

// The strip's render order; TaskTabs measures against it and overflows from the RIGHT (В§1.4).
export const TASK_TAB_ORDER: TaskTabId[] =
  ["all", "ipfs", "compress", "transcribe", "ai-descriptions", "ocr"];
```

* **Columns** вҖ” `OneRepoPage` defines the union of all columns once (including the five icon control columns
  `pinned`, `gitignore`, `transcribe`, `describe`, `ocr` and the Compress-tab `compress` content column); **every**
  tab's `columnIds` **begin with all five icon columns in the fixed order** `pinned`, `gitignore`, `transcribe`,
  `describe`, `ocr` ([tables.mdx В§4c](./tables.mdx)), then the tab's own content columns after `File`. No
  tab includes a `decision`, `decidedBy`, `decided`, or transfer `status` column anymore
  ([one_repo.mdx В§4.3.1](./one_repo.mdx)). The
  responsive-priority drop rule ([repos.mdx В§3.2.1](./repos.mdx#321-responsive-column-priority--drop-the-least-important-column-as-the-table-narrows-locked))
  still runs on whatever subset is visible.
* **Sort** вҖ” the tab's `defaultSort` becomes the table's `defaultSort`; the user can still re-sort (header
  click / dropdown) within the tab, and that choice persists per user per page
  ([tables.mdx В§4](./tables.mdx)).
* **Filter** вҖ” `rowFilter` is applied to `detail.files` before the table sees them; the table's own
  search/filter still narrow further within the tab.
* **Metrics** вҖ” `metrics` selects which `MetricPanel`s the strip renders, **in the listed order**; each maps
  to a count (В§2.5) and a `WarningDef` (В§2.4). On **All** the list **starts with the two decision tiles,
  adjacent** (`addToIpfs`, `gitIgnore`); on **IPFS** it carries `addToIpfs` only (В§2.5.1).
* **Hover default** вҖ” `defaultHint` is what the **left-bar hover-info panel** shows on that tab when
  nothing is hovered (В§3).

The active-tab column/sort/filter/metric selection is applied in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` вҖ” `OneRepoPage`
(reads `TASK_TABS[activeTab]`, passes the projected `columns` + `defaultSort` to `DataTable` and the
filtered `data`, and the tab's `metrics` to `MetricsStrip`).
