# LargeFileBridge — One repo

The **One repo** screen is where the work happens. You reach it by clicking a repo on the
[Repos](./repos.mdx) page; it opens that repo at `/repos/$repoId`. It has two jobs:

1. **Show this repo's settings, progress, and status** — is per-repo pinning on? are the files pinned?
   which of your computers hold them? when did it last pin? — and let you **Pin now**.
2. **Let you decide, file by file, what LFBridge should move** — the core interaction. For every
   large git-ignored file the repo contains, you choose **Add to IPFS (pin)** (the `sync` decision
   value), **Ignore**, or leave it **Undecided**. Most of the user's time here is spent scanning
   "how many will pin, how many will not, and which ones I still need to decide on."

The One-repo screen is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` component

This spec builds on the siblings and does not repeat them:

* [Task tabs](./task_tabs.mdx) — the **All / IPFS / Compress / Transcribe / AI descriptions** tab strip this
  screen's header now carries (§3.1 here). Selecting a tab re-tunes the metrics strip, the visible table
  columns, the default sort, and the row filter; that whole system — including the metrics panels, the
  hover-info region, the per-metric popup + blue header-primary behavior, and the three-state
  Transcribe / Compress / Describe status icons — is defined there and only summarized here.
* [Repos](./repos.mdx) — the list this screen drills into; defines the LOCKED vocabulary
  **Repo → Large file → Decision** (§1 there), the `RepoStatus` enum, and the per-repo counts.
* [pin_process.mdx](./pin_process.mdx) — the **pin process** the Pin-now button triggers:
  every pin pass is a **full pass**, a repo trigger sets **priority** (not scope), the pass is parallel,
  and it is never a silent no-op. The button is one trigger of that pass.
* [Left bar](./left_bar.mdx) — the left navigation (source of truth `pm/left_bar.yaml`).
* [README](../README.md) — the product charter (git for small files; IPFS/p2p for the big ones).

## 1. The decision model (LOCKED)

Every large file in the repo is in exactly **one** of three **decision** states. This is the
vocabulary the whole app shares:

| Decision | Meaning | LFBridge behavior |
| --- | --- | --- |
| **Add to IPFS (pin)** | The user wants this file kept pinned across their computers. Stored decision value: `sync`. | LFBridge adds it to the manifest, pins it on IPFS, and transfers it to/from peers. |
| **Ignore** | The user deliberately does **not** want this file pinned. | LFBridge leaves it alone — never added to the manifest, never transferred. Stays a purely local file. |
| **Undecided** | LFBridge found the file but the user has not chosen yet. | LFBridge does **nothing** with the bytes yet; it just surfaces the file so the user can triage it. This is the queue of decisions to make. |

Independently of the decision, an **Add-to-IPFS (pin)** file (decision value `sync`) has a **transfer
status** — where its bytes are relative to the network (§4.2). A file's decision is user intent; its
transfer status is reality.

> **This table is the IPFS axis only — one of two.** The full decision the user makes has **two
> independent axes** — **Add to IPFS (pin)** *and* **Add to git ignore** — each on/off separately, and it
> is recorded (with who/when/the Storage ID) in a **repo-level, team-shared** ledger so a whole team
> shares one set of decisions. That is specified end to end in **[decisions.mdx](./decisions.mdx)**. The
> three-state enum here is the **frozen IPFS-axis projection** of that ledger (`ipfs:true → sync`,
> `ipfs:false → ignore`, no record → undecided) — this screen keeps rendering it verbatim; the wire value
> `sync` is never renamed. The git-ignore axis is applied through [git_ignore.mdx](./git_ignore.mdx).

```ts
// One row of the files table.
export interface FileRow {
  fileId: string;        // stable id (repoId + relative path)
  path: string;          // path relative to the repo root
  sizeBytes: number;     // on-disk (or expected) size
  cid: string | null;    // IPFS content id once added to the manifest; null while Undecided/Ignore
  decision: "sync" | "ignore" | "undecided";  // FROZEN wire value; "sync" renders as "Add to IPFS (pin)" in the UI
  transfer: TransferStatus;   // meaningful for decision === "sync" (see §4.2)
  peers: string[];       // ids of your computers currently pinning this file
  changedAt: string;     // ISO date-time the file last changed on disk
}

export type TransferStatus =
  | "pinned"     // present locally AND pinned on >=1 peer
  | "pending"    // queued to push out or fetch in
  | "fetching"   // transfer in progress (inbound)
  | "pushing"    // transfer in progress (outbound)
  | "missing"    // in the manifest but the bytes are not on this machine
  | "error"      // last transfer failed
  | "na";        // not applicable (decision is ignore/undecided)
```

## 3. Header + status/settings strip

### 3.1 Header

* **Breadcrumb** — `‹ Repos / <repo name>`; the repo name is the title, the absolute **path** shown
  muted beneath it.
* **Task tabs** (a little **right of center**, between the breadcrumb and the right-hand buttons) — the
  **All / IPFS / Compress / Transcribe / AI descriptions** strip ([task_tabs.mdx §1](./task_tabs.mdx#1-the-tab-strip-locked)).
  **All** is the default. Selecting a tab re-tunes the **metrics-panel strip**, the **visible table
  columns**, the **default sort**, and **which files the table shows** — all client-side, no refetch, no
  mutation. Everything below the header in §3.2 / §4 is presented *through* the active tab; the tab system,
  its metrics panels, the hover-info region, and the three-state Transcribe / Compress / **Describe** status
  icons are defined in [task_tabs.mdx](./task_tabs.mdx).
* **Header primary — "View recommendation" / "Scan now"** (top-right, the SAME button on all four tabs).
  This one smart button replaces the old always-`Pin now` primary. It surfaces the single next thing to
  do, so a non-expert never has to guess which panel to open:
  * **"Scan now"** (leading `Search` icon) — shown **only** when the repo is **scan-stale**: it has never
    been scanned, or its `lastScanAt` is **older than 2 weeks**. A stale scan means the metrics can't be
    trusted, so re-scanning takes precedence over everything else. Clicking runs the same rescan worker;
    when the scan lands, `lastScanAt` refreshes and the button flips back to "View recommendation".
    When a repo's contents are known to have changed (the live watcher, scan.mdx §2.2), that also drives a
    fresh scan, after which the button returns to a recommendation.
  * **A pending recommendation** (no leading icon; a trailing right chevron `›`) — shown when a scan is
    **not** due and there is pending work. It opens the **most important** pending metric's educate-and-fix
    popup — the SAME popup its metric panel opens ([task_tabs.mdx §2.4](./task_tabs.mdx)) — resolved
    **worst-first across the full task set**:

    ```
    ipfs-down → pull-down → undecided → transcribe → describe → compress
      (bad)      (bad)       (warn)      (warn)        (warn)     (warn)
    ```

    **The button's label names the recommendation** ([task_tabs.mdx §2.7](./task_tabs.mdx#27-the-per-metric-blue-header-primary--every-actionable-metric-can-be-the-recommendation-locked)):
    the three **integrity** conditions keep the generic **"View recommendation ›"** ("here's what's wrong —
    view it"), while the three **producing** tasks name the verb — **"Transcribe ›"**, **"Describe ›"**,
    **"Compress ›"** — so a non-expert reads the one action to take. Every one opens the identical popup its
    metric tile opens, and its Apply **starts the work** ([task_tabs.mdx §2.4.1](./task_tabs.mdx#241-every-actionable-metric-tile-opens-a-popup--the-transcribe--compress--describe-builders-the-fix-locked)).

    | Top pending recommendation | Label | Fill |
    | --- | --- | --- |
    | IPFS engine down / peer-pinned files not here | **View recommendation ›** | **red** (`--lfb-bad`) |
    | Files need a decision | **View recommendation ›** | **blue** (`--lfb-primary`) |
    | Audio/video with no transcript | **Transcribe ›** | **blue** (`--lfb-primary`) |
    | Image/video with no AI description | **Describe ›** | **blue** (`--lfb-primary`) |
    | Video/image that could be compressed | **Compress ›** | **blue** (`--lfb-primary`) |

    The button tints **red** (`--lfb-bad`) only when the top recommendation is a Broken (`bad`) state
    (IPFS-down, pull-down), brand-**blue** (`--lfb-primary`) otherwise. A **"Transcribe ›"** / **"Describe ›"**
    primary shows whenever that work is pending even if the heavyweight model or AI key isn't ready — the
    **consent/provisioning gate fires when the popup's Apply runs**, then resumes (§3.1 gating mirrors the
    per-row status-icon path, [task_tabs.mdx §2.7](./task_tabs.mdx)).
  * **Neither** — when no scan is due and nothing is pending, there is **no** header primary: the metric
    panels (all green zeros) already say "all clear".

  `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/metricWarningDefs.tsx` —
  `scanIsStale()` (the 2-week staleness gate) and `topRecommendation()` (the worst-first pick); wired into

### 3.2 Status / settings strip

A single horizontal strip of the repo's live status (read) and its one per-repo setting:

| Field | Meaning |
| --- | --- |
| **Pin** (toggle) | This repo's **IPFS-pinning opt-in** — *not* a "managed vs. unmanaged" switch. When **off**, LFBridge still **knows, visits, and reconciles** this repo every pass ([pin_process.mdx §1/§5](./pin_process.mdx#1-lfb-knows-every-repo--there-is-no-lfb-managed-gate-locked)); it just does not add/pin/fetch its bytes during **background** pin passes (a manual **Pin now** flips this on). | **Status** | The rolled-up `RepoStatus` pill ([Repos §4.2](./repos.mdx)) — up to date / pinning / behind / needs review / error / never. |
| **Peers** | Count of your computers currently pinning this repo → clicking opens [Peers](./left_bar.mdx) filtered to this repo (peers page is out of scope here). |
| **Last pin** | Relative time of the last completed pin pass (absolute on hover). |
| **IPFS** | The local IPFS node health — `ok` (green) / `unreachable` (red). If unreachable, it becomes the top recommendation: the header primary is a **red "View recommendation ›"** opening the *Start IPFS* popup (§3.1), and the action-links row's **Pin now** is disabled with a tooltip. |

## 4. The files table (TanStack, house style)

**Height — bounded (the content-below exception).** Unlike Repos, this page renders content **beneath**
the files table — the **Repo details** disclosure (§3.2 status strip, now a click-away disclosure at
the bottom of the page). Because something sits **below** the table, it is the **exception** to the
site-wide full-page-height rule ([Repos §3.3.1](./repos.mdx), [charter Tables rule](../CLAUDE.md)): the

disclosure below it stays visible. In code this is `fillHeight={false}` on the `DataTable`.

### 4.1 Summary counts (above the control row)

A one-line summary the user reads first: **`128 Pin · 5 Undecided · 40 Ignore`** — `Pin` is the compact
label for the **Add to IPFS (pin)** decision (stored value `sync`). `Undecided` is accent-highlighted
whenever it is > 0.

> **These counts exclude `analysisOnly` rows.** The small, not-git-ignored media the scan surfaces purely
> so they can be OCR'd / described / transcribed (rule 5, [scan.mdx §4.1](./scan.mdx)) are **not** large-file
> decisions the user owes — they carry no Pin/Undecided/Ignore obligation. Counting a folder of thumbnails
> as *"Undecided"* would make every repo look like it has hundreds of pending decisions. So the summary
> counts, the repos-list counts ([repos.mdx §4.1](./repos.mdx)), and the large-file scan totals all count
> only the non-`analysisOnly` candidates. The `analysisOnly` rows appear only when **Large files only** is
> turned off (§4.2), and only the analysis metrics (`OCRable`, `Describable`, `Transcribable`) count them.

**Every segment is a quick filter on the Pin (IPFS) facet — and only that facet.** All three counts read
the **IPFS axis** ([tables.mdx §2.4.1](./tables.mdx#241-the-two-facets-fixed-vocabulary-locked)); none of
them says anything about git-ignore. Clicking a segment **sets the Pin (IPFS) facet to that one value** —
it unchecks every other Pin value and leaves the **Git-ignore** facet untouched (all-checked). Clicking the
active segment again re-checks the facet's values (back to the resting all-checked state,
[tables.mdx §2.1](./tables.mdx#21-a-facet-is-its-values-each-with-a-checkbox-all-checked-by-default-locked)):

| Segment | Sets the **Pin (IPFS)** facet to | Leaves **Git-ignore** |
| --- | --- | --- |
| `128 Pin` | ☑ Pinned only (uncheck Not pinned · Recommended · Undecided) | all-checked (untouched) |
| `5 Undecided` | ☑ Undecided only — the triage queue | all-checked (untouched) |
| `40 Ignore` | ☑ Not pinned only | all-checked (untouched) |

Because the segments drive the **rail facet** and not a private filter, the result is **visible**: after a
click the user sees which Pin chips went dark and can re-check them without hunting for a reset
([tables.mdx §2.2](./tables.mdx#22-the-facet-rail--filtering-without-opening-a-dropdown-locked)).

> **Recommendation — rename the third segment `Ignore` → `Not pinned`.** This label is the product's
> single live conflation risk: on this line `Ignore` means **"deliberately not pinned"** (the `ignore`
> wire value on the **IPFS** axis), while three columns to the right the **Ignore** toggle column means
> **git-ignored** — a different axis with a different source of truth
> ([tables.mdx §2.4](./tables.mdx#24-one-facet--one-axis--pin-and-git-ignore-are-never-the-same-filter-locked)).
> One word, two meanings, one screen. Reading `128 Pin · 5 Undecided · 40 Not pinned` makes the line
> self-evidently one axis, and it matches the **Pin (IPFS)** facet's own value label verbatim (§2.4.1), so
> the segment and the chip it drives read the same word. The stored value stays `ignore` — **frozen wire
> value**, never renamed ([decisions.mdx §7](./decisions.mdx#7-reconciliation--folding-the-shared-ledger-onto-this-computer-locked)).

> **The richer, tab-scoped surface is the metrics-panel strip** ([task_tabs.mdx §2](./task_tabs.mdx#2-the-metrics-panels--the-what-could-be-done-boxes-locked)),
> which sits above the control row and shows the **active tab's** "what could be done" panels (each a terse
> label + number + right chevron → its warning popup; a light-green big-`0` when nothing is to be done),
> with the **hover-info region** to its right ([task_tabs.mdx §3](./task_tabs.mdx#3-the-hover-info-region--the-space-to-the-right-of-the-panels-locked)).
> This one-line summary is the compact IPFS-decision readout; the metrics strip is the per-task overview.

### 4.9 The Pin icon is THREE-STATE — decision × live pin reality (LOCKED)

The **Pin** icon (col 1) conveys **two facts at once**: the user's **decision** (do I want this file synced
over IPFS?) and the **reality on THIS computer right now** (is it actually pinned here?). A single on/off
toggle can only show intent; but a user syncing across three machines needs to see, per machine, whether a
file they *chose* to sync has actually **arrived and pinned here yet**. So the icon has **three states**
(the visual grammar — bare glyph, no rounded rectangle — is [tables.mdx §4c.4/§4c.4a](./tables.mdx)):

| State | Glyph | Condition | What it tells the user |
| --- | --- | --- | --- |
| **Not set to sync** | grey **outline** pin ("white background") | `decision !== "sync"` | This file is **not** being synced over IPFS. **Click to add it.** |
| **Synced here** | **blue filled** pin ("dark-blue fill") | `decision === "sync"` **and** `pinnedHere === true` | Chosen to sync **and** this computer's IPFS node actually holds the pin — in sync here. |
| **Set to sync, not here yet** | **RED filled** pin ("red fill") | `decision === "sync"` **and** `pinnedHere === false` | You chose to sync it, but **this** computer does not have it pinned yet. The pin pass will **pull it down**; the red flags the transient out-of-sync gap ([pin_process.mdx §2](./pin_process.mdx)). |
| **Pinned outside LFB** | **GREEN filled** pin | `decision !== "sync"` **and** `pinnedForeign === true` | The bytes are **already pinned on this node** under a foreign CID (a bare `ipfs add`, IPFS Desktop, another tool) — discovered by the background pass ([foreign_pin_discovery.mdx §6](./foreign_pin_discovery.mdx#6-surfacing--the-file-shows-as-pinned-in-the-web-app-locked)). Reality without intent. **Click to have Large File Bridge sync it too.** |

* **`pinnedHere` is a LIVE, CANONICAL read of this node's pinset — never a per-file hash on the read path.**
  The blue↔red split is driven by `FileRow.pinnedHere`, computed by testing the file's recorded manifest CID
  against `ipfs.canonicalPinnedSet()` (this node's `ipfs pin ls`, canonicalized to CIDv1 base32 so a `Qm…`
  pin of a `bafy…` CID for the **same block** still counts — [knowledge/ipfs.mdx §5.1](../knowledge/ipfs.mdx)).
  The pinset is fetched **once per page load** and each row is an O(1) set lookup — **no full-file hashing**,
  honoring the read-path cost boundary this app keeps ([knowledge/ipfs.mdx §5.1 honest boundary](../knowledge/ipfs.mdx)).
* **`undefined` → intent-blue, NEVER a false red.** When the pinset can't be read (IPFS daemon down, or the
  caller didn't fetch it) `pinnedHere` is `undefined`; a decided file then shows **blue** (intent) and the
  icon **never cries red** on missing information. Red is only ever a **verified** "decided but absent here."
* **This directly fixes "the CLI says it's pinned but the web app shows it isn't"** — for any file the user
  has **opted into syncing**, the icon now reflects the node's real pinset, not just the stored decision.
* **The undecided-but-pinned case is the GREEN state — LEARNED FROM A DEFECT (the 3rd fix in this saga).**
  A file **pinned but never decided** (a legacy manual `ipfs add` under a foreign CID profile) is discovered
  by the **background** foreign-pin pass — never a read-path hash, so the honest boundary holds — and the row
  carries `pinnedForeign: true` ([foreign_pin_discovery.mdx §6](./foreign_pin_discovery.mdx#6-surfacing--the-file-shows-as-pinned-in-the-web-app-locked)).
  History of the bug ("CLI says pinned, this page says not pinned", e.g. `charlie-kirk/videos/1965884012565766172.mp4`
  pinned as `QmT5j43eo…`): fix 1 canonicalized CID encodings (v0↔v1); fix 2 built foreign-pin discovery, which
  correctly **recorded** the pin in `foreign-pins.json` and shipped `pinnedForeign` on the row — but the pin
  cell here still rendered only `decision × pinnedHere`, so the recorded reality was **never drawn** and an
  undecided pinned file stayed grey. Fix 3 is this fourth state: **recorded must mean rendered**. Any new
  surface that shows pin state must consume `pinnedForeign`, not just `pinnedHere`.
* **Toggling writes the decision to the shared ledger, which travels to the owning Personal / Company
  storage.** Clicking the Pin icon — one file, or a bulk popup for hundreds — writes the IPFS axis to the
  repo's `decisions.yaml` ledger (§4.5.1). That ledger lives under the repo's tracking area and is committed
  and union-merged by the git backbone, so a decision made on one computer travels to the user's other
  computers; when the repo is owned by a **company** storage with a sync repo, it travels to the whole team
  ([decisions.mdx §3](./decisions.mdx), [repo_company_mapping.mdx](./repo_company_mapping.mdx),
  [repo_owner_propagation.mdx](./repo_owner_propagation.mdx)). Turning the axis **off** (grey) is the same
  path in reverse. On each other computer the pass then reconciles reality → the icon there shows **red**
  until that machine pulls the bytes, then **blue**.

## 5. Empty & loading states

* **No large files:** the table is replaced by a card — "No large files found in this repo. LFBridge
  looks for git-ignored files under `large/` and the ignored media extensions."
* **All decided:** the summary shows `Undecided 0`; the quick-filter for Undecided is disabled.
* **Loading:** skeleton rows sized to the columns; no layout shift.
* **IPFS node unreachable:** a banner at the top of the content; Pin now and per-row transfers are
  disabled, but decisions can still be edited (they queue for when the node returns).

## 6. Acceptance criteria

1. The screen opens at `/repos/$repoId` from a [Repos](./repos.mdx) row click; the header shows the
   `‹ Repos / <name>` breadcrumb, the absolute path, and a **Pin now** button that **prioritizes this
   repo (pinned first) and then runs the full pin pass**

   changes a decision, and is never a silent no-op (reports what it actually did).

   Peers, Last pin, and IPFS node health; when the node is unreachable, Pin now is disabled.

   and **leaves the Git-ignore facet untouched** — no segment ever touches both axes. Undecided is
   accent-highlighted when > 0.
