# LargeFileBridge — One repo

The **One repo** screen is where the work happens. You reach it by clicking a repo on the
[Repos](./repos.mdx) page; it opens that repo at `/repos/$repoId`. It has two jobs:

1. **Show this repo's settings, progress, and status** — is per-repo pinning on? are the files pinned?
   which of your computers hold them? when did it last pin? — and let you **Pin now**.
2. **Let you decide, file by file, what LFBridge should move** — the core interaction. For every
   large git-ignored file the repo contains, you choose **Add to IPFS (pin)** (the `sync` decision
   value), **Ignore**, or leave it **Undecided**. Most of the user's time here is spent scanning
   "how many will pin, how many will not, and which ones I still need to decide on."

The One-repo screen is rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` component

This spec builds on the siblings and does not repeat them:

* [Task tabs](./task_tabs.mdx) — the **All / IPFS / Compress / Transcribe** tab strip this screen's header
  now carries (§3.1 here). Selecting a tab re-tunes the metrics strip, the visible table columns, the
  default sort, and the row filter; that whole system — including the metrics panels, the hover-info
  region, and the three-state Transcribe/Compress status icons — is defined there and only summarized here.
* [Repos](./repos.mdx) — the list this screen drills into; defines the LOCKED vocabulary
  **Repo → Large file → Decision** (§1 there), the `RepoStatus` enum, and the per-repo counts.
* [pin_process.mdx](./pin_process.mdx) — the **pin process** the Pin-now button triggers:
  every pin pass is a **full pass**, a repo trigger sets **priority** (not scope), the pass is parallel,
  and it is never a silent no-op. The button is one trigger of that pass.
* [Left bar](./left_bar.mdx) — the left navigation (source of truth `pm/left_bar.yaml`).
* [README](../README.md) — the product charter (git for small files; IPFS/p2p for the big ones).

## 1. The decision model (LOCKED)

Every large file in the repo is in exactly **one** of three **decision** states. This is the
vocabulary the whole app shares:

| Decision | Meaning | LFBridge behavior |
| --- | --- | --- |
| **Add to IPFS (pin)** | The user wants this file kept pinned across their computers. Stored decision value: `sync`. | LFBridge adds it to the manifest, pins it on IPFS, and transfers it to/from peers. |
| **Ignore** | The user deliberately does **not** want this file pinned. | LFBridge leaves it alone — never added to the manifest, never transferred. Stays a purely local file. |
| **Undecided** | LFBridge found the file but the user has not chosen yet. | LFBridge does **nothing** with the bytes yet; it just surfaces the file so the user can triage it. This is the queue of decisions to make. |

Independently of the decision, an **Add-to-IPFS (pin)** file (decision value `sync`) has a **transfer
status** — where its bytes are relative to the network (§4.2). A file's decision is user intent; its
transfer status is reality.

> **This table is the IPFS axis only — one of two.** The full decision the user makes has **two
> independent axes** — **Add to IPFS (pin)** *and* **Add to git ignore** — each on/off separately, and it
> is recorded (with who/when/the Storage ID) in a **repo-level, team-shared** ledger so a whole team
> shares one set of decisions. That is specified end to end in **[decisions.mdx](./decisions.mdx)**. The
> three-state enum here is the **frozen IPFS-axis projection** of that ledger (`ipfs:true → sync`,
> `ipfs:false → ignore`, no record → undecided) — this screen keeps rendering it verbatim; the wire value
> `sync` is never renamed. The git-ignore axis is applied through [git_ignore.mdx](./git_ignore.mdx).

```ts
// One row of the files table.
export interface FileRow {
  fileId: string;        // stable id (repoId + relative path)
  path: string;          // path relative to the repo root
  sizeBytes: number;     // on-disk (or expected) size
  cid: string | null;    // IPFS content id once added to the manifest; null while Undecided/Ignore
  decision: "sync" | "ignore" | "undecided";  // FROZEN wire value; "sync" renders as "Add to IPFS (pin)" in the UI
  transfer: TransferStatus;   // meaningful for decision === "sync" (see §4.2)
  peers: string[];       // ids of your computers currently pinning this file
  changedAt: string;     // ISO date-time the file last changed on disk
}

export type TransferStatus =
  | "pinned"     // present locally AND pinned on >=1 peer
  | "pending"    // queued to push out or fetch in
  | "fetching"   // transfer in progress (inbound)
  | "pushing"    // transfer in progress (outbound)
  | "missing"    // in the manifest but the bytes are not on this machine
  | "error"      // last transfer failed
  | "na";        // not applicable (decision is ignore/undecided)
```

  real work and its toast reports what happened, not a fixed "complete"

Pin now — which pins THIS repo first as the priority unit (moving bytes even when the repo's pinning toggle was off, and flipping it on), then runs the full pass over the remaining units in the background — is handled in code:

### 4.5 The Decision select (per row) — the core interaction

Each row's **Decision** cell is an inline select with the three states. Changing it writes
immediately (optimistic, with rollback on error):

* → **Add to IPFS (pin)** (decision value `sync`): LFBridge adds the file to the manifest,
  computes/records its **CID**, pins it, and queues transfers to peers. The row's Status moves to
  `pending` → `fetching`/`pushing` → `pinned`.
* → **Ignore**: removes it from the manifest (if present); Status becomes `—`; bytes stay local only.
* → **Undecided**: parks it back in the triage queue; no bytes move.

Changing a decision never deletes the user's local file — it only changes what LFBridge *does* with
it.

Writing a per-file decision (single or bulk) to the repo's `config.yaml` is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `PATCH /api/repos/:repoId/files`

### 4.8 Decision provenance — who decided, and when (team-aware)

Because decisions are now a **team-shared, attributed** record ([decisions.mdx §3/§10](./decisions.mdx)),

better than none:

* **Hover the Decision cell** → a non-intrusive tooltip ([non_intrusive_tooltip.mdx](./non_intrusive_tooltip.mdx)):
  *"Add to IPFS + git-ignored · decided by bob@acme.com · Jul 12, 2:32 PM"*. A small person/clock glyph
  marks a file decided by **someone else on the team** (vs. you), so others' choices are visible at a
  glance.
* **Filter/sort by decider** — the Decision column filter adds *decided by me / by a teammate /
  policy-decided / undecided*; sorting by the decided-at time surfaces the most recently triaged first.
* **Overrides are visible, not silent** — when a teammate's pulled-in decision changes an axis on a file
  this machine already acted on, the change shows immediately in the provenance and a **quiet, dismissible**
  dock note summarizes it ([decisions.mdx §11](./decisions.mdx)); it never deletes local or pinned bytes.

## 5. Empty & loading states

* **No large files:** the table is replaced by a card — "No large files found in this repo. LFBridge
  looks for git-ignored files under `large/` and the ignored media extensions."
* **All decided:** the summary shows `Undecided 0`; the quick-filter for Undecided is disabled.
* **Loading:** skeleton rows sized to the columns; no layout shift.
* **IPFS node unreachable:** a banner at the top of the content; Pin now and per-row transfers are
  disabled, but decisions can still be edited (they queue for when the node returns).
