# LargeFileBridge ‚Äî Repo (Git) Backbone

The **git repo itself is a bridge backbone.** This is one of three backbones the
[Sync List hub](./sync_list.mdx) defines; its siblings are [dropbox.mdx](./dropbox.mdx) and
[google_drive.mdx](./google_drive.mdx). Where those two carry a *computer*-scope list through a
cloud folder, this backbone carries a **per-repo (scope: `repo`) Sync List inside the repo**, so that
when the user runs `git pull` on another machine the list arrives **for free**, atomically, with the
repo's own commits. It is the **most private/secure backbone ‚Äî no third party at all ‚Äî and the
DEFAULT for repos.**

> **The one rule this backbone is built on.** *The big files stay git-ignored; only the small
> committed Sync List travels via git.* **Git carries the list, IPFS carries the bytes ‚Äî never commit
> the large files.** The `.gitignore` continues to exclude every big binary; the only new thing git
> tracks is the tiny `LargeFilesBridge_SyncList.yaml`. Losing this discipline (committing the bytes)
> defeats the entire product.

This spec does **not** restate the hub. It references it:

* [sync_list.mdx](./sync_list.mdx) ‚Äî the Sync List schema, the `large_files` hierarchy, the two
  scopes, the `origin`/`revision` provenance, and the [¬ß5 trust/reconciliation model](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule).
  Read ¬ß5 before writing any code that consumes an in-repo list.
* [storage.mdx ¬ß9.2](./storage.mdx#92-repo-manifests-travel-via-git-the-elegant-part) ‚Äî the committed
  `<repo>/.lfbridge/manifest.yaml`. This backbone is that transport, **widened** to the full
  `large_files` hierarchy; [¬ß3](#3-reconciliation-with-storagemdx-92) reconciles the two explicitly.
* [repo_settings.mdx ¬ß2.4](./repo_settings.mdx#24-pinning--fetch-policy) ‚Äî the per-repo
  `publish_manifest` toggle that enables this backbone.
* [knowledge/ipfs.mdx ¬ß6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) ‚Äî the
  only-our-content rule the list's IPFS settings must still honor.

## 1. Scope is `repo` ‚Äî exactly one repo, in its own repo

A list on this backbone is a **repo-scope** Sync List (hub [¬ß2](./sync_list.mdx#2-two-scopes--one-per-computer-one-per-repo)):

* `scope: repo` ‚Äî this list covers **exactly one repo**.
* **Exactly one entry under `repos:`** ‚Äî the repo the file lives in. There is **no second repo** and
  **no `directories:` block** (non-repo directories belong to the *computer*-scope list on
  Dropbox/Drive, not here).
* **Paths are relative to the repo working tree** (hub [¬ß4](./sync_list.mdx#4-what-travels-vs-what-stays-local)).
  The writer's absolute `identity.path` is advisory only; the reader resolves the repo by `remote` +
  local discovery, never by trusting an incoming absolute path.

The repo-scope list is the same YAML file as the computer-scope one, just narrowed ‚Äî one top-level
`large_files` key, `scope: repo`, one `repos:` child. The full schema and the level-by-level table
live in hub [¬ß3](./sync_list.mdx#3-the-file--largefilesbridge_synclistyaml); this spec adds only the
git-backbone detail.

## 2. File location ‚Äî committed inside the repo

The list is committed at a fixed path inside the repo working tree:

```
<repo>/.lfbridge/LargeFilesBridge_SyncList.yaml
```

The `.lfbridge/` directory holds the committed, git-tracked artifacts for this repo:

```
<repo>/
‚îú‚îÄ‚îÄ .git/
‚îú‚îÄ‚îÄ .gitignore                          # continues to ignore the big files (LOCKED)
‚îú‚îÄ‚îÄ .lfbridge/                          # committed ‚Äî travels via git (MUST NOT be git-ignored)
‚îÇ   ‚îú‚îÄ‚îÄ LargeFilesBridge_SyncList.yaml  # the per-repo Sync List (this backbone's cargo)
‚îÇ   ‚îî‚îÄ‚îÄ README.md                       # optional: "these bytes live in IPFS, not git" note
‚îú‚îÄ‚îÄ large/                              # the big files themselves ‚Äî GIT-IGNORED, moved by IPFS
‚îÇ   ‚îî‚îÄ‚îÄ trailer_v3.mp4
‚îî‚îÄ‚îÄ src/                                # ordinary code ‚Äî tracked by git as usual
```

* **`.gitignore` must NOT ignore `.lfbridge/`.** The list is *committed*; the big files are ignored.
  If a broad ignore rule (e.g. `*.yaml`, or a dotfile glob) would swallow the list, LFB adds an
  explicit un-ignore (`!/.lfbridge/`) so the list is always tracked. LFB verifies on scan that
  `.lfbridge/LargeFilesBridge_SyncList.yaml` is not ignored and warns loudly if it is.
* The state-root copy (`sync/r/<repo_name>/manifest.yaml`,
  [storage.mdx ¬ß6.3](./storage.mdx#62-syncrrepo_nameconfigyaml)) remains the **local cache** of this
  committed file plus this computer's live pin state. The committed file is the **portable source of
  truth**; the state-root copy is derived.

The committed in-repo list file (currently `.lfbridge/manifest.yaml`) is written by code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/manifest.service.ts` ‚Äî `writeCommittedManifest` (path from `committedManifestPath`)

## 3. Reconciliation with storage.mdx ¬ß9.2

[storage.mdx ¬ß9.2](./storage.mdx#92-repo-manifests-travel-via-git-the-elegant-part) describes a bare
committed `<repo>/.lfbridge/manifest.yaml` ‚Äî path ‚Üí CID + pin state, carried by git. **This
per-repo Sync List supersedes and absorbs that file.** The mapping is exact:

| storage.mdx ¬ß9.2 (`.lfbridge/manifest.yaml`) | This spec (`.lfbridge/LargeFilesBridge_SyncList.yaml`) |
| --- | --- |
| the manifest `files:` rows (`path`/`cid`/`size`/`sha256`/`modified_at`/`pinned_by`) | `large_files.repos.<key>.files[]` ‚Äî the **same rows**, unchanged in shape |
| *(no provenance)* | the `origin` block ‚Äî `computer_id`/`label`/`owner`/`written_at`/`revision` (hub ¬ß3) |
| *(no per-unit settings)* | `large_files.repos.<key>.settings` + top-level `large_files.settings` |
| implicit "this one repo" | explicit `scope: repo` + one `repos:` entry |

In short: **the list's `files:` block equals the old manifest rows, plus settings + provenance,
widened to the full `large_files` hierarchy.** The committed file is now a proper Sync List, not a
bare manifest. `publish_manifest` ([repo_settings.mdx ¬ß2.4](./repo_settings.mdx#24-pinning--fetch-policy),
`sync.publish_manifest` in storage ¬ß6.2) is what writes this committed file ‚Äî the same toggle, now
writing the widened artifact. Existing bare `manifest.yaml` files are read and migrated in place on
the next publish.

The `publish_manifest` toggle that writes this committed file on each sync is wired in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` ‚Äî `syncRepoFolder` (the `cfg.sync.publish_manifest` publish block)

## 4. Transport is git ‚Äî the asynchronous change is a `git pull`

On the whole-computer backbones, the "asynchronous change from another computer" is a cloud folder
sync. **Here it is a `git pull` / merge.** When the user (or LFB's background sync, ¬ß7) pulls the
repo on this machine, git delivers whatever `.lfbridge/LargeFilesBridge_SyncList.yaml` the peer
committed and pushed. That incoming file is exactly the kind of asynchronous, other-computer-authored
input the hub trust model governs ‚Äî it is a set of **claims, never ground truth** (hub
[¬ß5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)).

* The backbone is enabled per-repo by two settings, which must agree:
  * `settings.backbones.repo_git.enabled: true` (the list's own backbone flag, hub ¬ß3), **and**
  * `publish_manifest: true` on that repo ([repo_settings.mdx ¬ß2.4](./repo_settings.mdx#24-pinning--fetch-policy)).
* After a pull, LFB reads the committed list, reconciles it against this computer's filesystem + IPFS
  pinset (hub ¬ß5), and fetches any CID it lacks over IPFS ‚Äî exactly the flow storage.mdx ¬ß9.2
  describes, now driven by the widened list.

Reading the committed list that a `git pull` delivered is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/manifest.service.ts` ‚Äî `readCommittedManifest`

## 5. Git-specific hazards the reader MUST handle

Git mangles files in git's own ways. Beyond the generic backbone hazards in hub
[¬ß5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect), the reader must expect:

### 5.1 Merge conflicts in the YAML (LOCKED ‚Äî refuse, quarantine, reconcile)

A `git pull` can leave the committed list with literal conflict markers:

```
large_files:
  repos:
    largefilebridge:
      files:
<<<<<<< HEAD
        - path: large/trailer_v3.mp4
          cid: bafybeigd‚Ä¶              # this computer's line
=======
        - path: large/trailer_v3.mp4
          cid: bafybeih2‚Ä¶              # the peer's line
>>>>>>> origin/main
```

The reader **MUST detect `<<<<<<<` / `=======` / `>>>>>>>` markers, REFUSE to load the conflicted
file, and quarantine it** (never parse a half-merged file as if it were valid YAML ‚Äî it is not, and
even if it parsed it would be a lie). It then **reconciles the two sides per hub
[¬ß5.3](./sync_list.mdx#53-conflicts-resolve-by-revision-then-content-then-time)**: higher
same-writer `revision` wins; different writers with different `cid`/`sha256` is a genuine content
fork ‚Äî keep both, mark the entry `conflicted`, surface it in the web app, **never auto-delete either
side's bytes**. The previous good copy is retained until reconciliation resolves the markers.

### 5.2 A pull that is behind this computer's real state

A peer may have committed a `revision` that is **older than what this computer actually holds** (they
pushed before scanning; they were offline; the branch lagged). The pull will bring that older list,
but it does **not** override reality: LFB **verifies local claims** (hub
[¬ß5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)) ‚Äî a `pinned_by` naming this
computer is trusted only if `ipfs pin ls` actually holds the CID; this computer's own filesystem is
the source of truth for what exists here. An older incoming revision never demotes a newer local
truth.

### 5.3 Absence after a merge is NOT a delete

If a merged-in list **omits** an entry this computer currently has, that is **not** a delete signal
(hub [¬ß5.2](./sync_list.mdx#52-absence-is-not-deletion-default)). A shallow clone, a peer that had
not scanned that file yet, or a merge that dropped a side can all produce spurious absences. Deletes
must be **explicit** (`state: removed` + `removed_at`/`removed_by`). A merge that loses an entry never
destroys the bytes.

### 5.4 Detached / old / shallow checkouts

A detached-HEAD checkout, a stale branch, or a shallow clone can present a list that is arbitrarily
old. Treat it like any other untrusted input: reconcile by `origin.revision` + content, never assume
the checked-out file is current just because git handed it over.

## 6. Merge-friendliness ‚Äî deterministic serialization (LOCKED)

Because git merges the committed list **line by line**, LFB writes the YAML **deterministically** so
that independent edits on two computers merge cleanly most of the time and **only genuinely-conflicting
entries collide.** The canonical serialization is a rule, not a preference:

* **File entries are sorted by `path`** ‚Äî a stable, total order, so two machines that add different
  files touch different regions of the file and merge without conflict.
* **Stable key ordering** within every mapping (e.g. `path`, `cid`, `size`, `sha256`, `modified_at`,
  `pinned_by` always in that order) ‚Äî no key churn to create spurious diffs.
* **One entry per line-group**, consistently indented, so a single changed file is a single contiguous
  hunk git can merge in isolation.
* **Consistent formatting** ‚Äî fixed indentation, no trailing whitespace, a trailing newline, stable
  quoting ‚Äî so re-serializing an unchanged list produces a byte-identical file (no noise commits).

The payoff: two computers editing *different* files never conflict; only two computers editing the
*same* file's bytes produce a real content fork, which is exactly the case hub ¬ß5.3 wants surfaced.

## 7. Publishing ‚Äî regenerate, then let git carry it

On scan/sync (hub [¬ß6](./sync_list.mdx#6-publishing--writing-our-list-out-to-the-backbones)), for a
repo with this backbone enabled, LFB regenerates the in-repo list:

* **Bump `origin.revision`**, stamp `origin.written_at`, set `origin.computer_id`/`label`/`owner` to
  **this** machine.
* Write **only what this computer can vouch for** ‚Äî a `pinned_by` entry for this computer only for
  CIDs it actually pins right now (hub ¬ß5.1); peer claims are copied forward from the last reconciled
  truth, never invented.
* Serialize **deterministically** (¬ß6) and write **atomically** (temp-then-rename,
  [storage.mdx ¬ß15](./storage.mdx#15-the-store-layer-how-code-touches-disk)) so a concurrent reader or
  a git operation never sees a half-written file.

The atomic temp-then-rename write that keeps readers and git from seeing a half-written file is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/store/yaml-store.ts` ‚Äî `writeYaml`

Then the file is **committed + pushed by the user's normal git workflow**. **LFB writes the file but
does NOT force git commits/pushes unless explicitly configured to** ‚Äî it respects the user's git
workflow, and the project `CLAUDE.md` forbids auto-branching/pushing. When LFB *is* configured to
commit, it commits only `.lfbridge/LargeFilesBridge_SyncList.yaml` on the current branch, never the
big files, and never creates or switches branches.

## 8. Authority ‚Äî the in-repo git list wins for that repo's files

Per hub [¬ß5.4](./sync_list.mdx#54-when-the-same-repo-arrives-on-two-backbones): a repo can appear both
in a *computer*-scope list (Dropbox/Drive) **and** in its own in-repo (git) list. **The in-repo (git)
list is AUTHORITATIVE for that repo's `files`**, because it travels **atomically with the repo's own
commits** ‚Äî the committed list and the code it describes move together, so it cannot drift from the
repo's real state the way a separately-synced computer-list mirror can. The computer-list copy of that
repo is a convenience mirror only. Settings still resolve on the normal inherit chain (per-repo
override ‚Üí computer `settings` ‚Üí global `config.yaml`).

## 9. Privacy & security

* **No third party at all.** This backbone uses only the user's own git remote and the user's own
  computers ‚Äî no Dropbox, no Google Drive, no cloud folder in anyone else's account. It is the
  **safest** backbone; it is the DEFAULT for repos for exactly this reason.
* **Only-our-content still holds.** The list's `settings.ipfs` keeps the
  [ipfs.mdx ¬ß6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) defaults ‚Äî
  `reprovide_strategy: pinned`, no public/recursive gateway. Publishing a list never turns this
  machine into a gateway/relay.
* **Allow-listed owners only.** LFB never adopts an in-repo list whose `origin.owner` is not an
  allow-listed user for this install (hub ¬ß5.5). A shared repo pins only for the emails in its
  `access.participants` ([repo_settings.mdx ¬ß2.5](./repo_settings.mdx#25-sharing--some-repos-need-users-some-dont)).

## 10. The file ‚Äî a `scope: repo` list (abridged)

The full schema is hub [¬ß3](./sync_list.mdx#3-the-file--largefilesbridge_synclistyaml); this shows only
what is distinctive about a git-backbone list ‚Äî `scope: repo`, one `repos:` entry, no `directories:`,
the `backbones.repo_git` block, and deterministic `files` order.

```yaml
# <repo>/.lfbridge/LargeFilesBridge_SyncList.yaml  (scope: repo ‚Äî abridged)
large_files:
  schema_version: 1
  scope: repo                             # LOCKED ‚Äî exactly one repo, no directories: block
  sync_list_id: 1b7e-‚Ä¶

  origin:                                 # WHO wrote THIS revision (hub ¬ß3 / ¬ß5)
    computer_id: 4f9c-‚Ä¶
    computer_label: bryan-macbook
    owner: bryan@act3ai.com               # must be allow-listed
    written_at: 2026-07-02T14:32:15.123Z
    revision: 17                          # monotonic per (sync_list_id, computer_id)

  backbone:
    kind: repo_git                        # set by the READER on ingest
    path: .lfbridge/LargeFilesBridge_SyncList.yaml   # relative to the repo working tree

  settings:
    backbones:
      repo_git: { enabled: true }         # this backbone's flag (agrees with publish_manifest)
    ipfs:
      reprovide_strategy: pinned          # NEVER "all" ‚Äî ipfs.mdx ¬ß6
      public_gateway: false
    conflict:
      strategy: newest_revision_per_entry
      absence_is_delete: false            # a merge that drops an entry is NOT a delete (¬ß5.3)

  repos:                                  # EXACTLY ONE entry (¬ß1)
    largefilebridge:
      identity:
        name: LargeFileBridge
        remote: https://github.com/BryanStarbuck/LargeFileBridge.git
        path: ~/BGit/Bryan_git/LargeFileBridge   # writer's path ‚Äî advisory only
      synced: true
      files:                              # sorted by `path`, stable key order (¬ß6)
        - path: large/hero_8k.png
          cid: bafybeih2‚Ä¶
          size: 41231884
          sha256: 1a77‚Ä¶
          modified_at: 2026-07-01T08:02:11Z
          pinned_by: [bryan-macbook, bryan-studio]
        - path: large/trailer_v3.mp4
          cid: bafybeigd‚Ä¶
          size: 734003200
          sha256: 9f2c‚Ä¶
          modified_at: 2026-07-01T09:14:02Z
          pinned_by: [bryan-studio]
  # NOTE: no `directories:` block ‚Äî that belongs to the computer-scope list (Dropbox/Drive).
```

## 11. Acceptance criteria

1. The per-repo Sync List is committed inside the repo at
   **`<repo>/.lfbridge/LargeFilesBridge_SyncList.yaml`**, tracked by git; the big files stay
   **git-ignored** and move over IPFS ‚Äî **git carries the list, IPFS carries the bytes, and the large
   files are never committed**.
2. `.gitignore` **must NOT ignore `.lfbridge/`**; LFB verifies the committed list is tracked (adding
   an explicit un-ignore if a broad rule would swallow it) and warns loudly if it is ignored.
3. The list is **`scope: repo`** with **exactly one entry under `repos:`**, **no `directories:`
   block**, and paths **relative to the repo working tree** (hub ¬ß2, ¬ß4).
4. This committed file **supersedes and absorbs** [storage.mdx ¬ß9.2](./storage.mdx#92-repo-manifests-travel-via-git-the-elegant-part)'s
   bare `<repo>/.lfbridge/manifest.yaml`: its `files:` block equals the old manifest rows **plus**
   `settings` + `origin`/`revision`, widened to the `large_files` hierarchy; bare manifests are
   migrated on next publish.
5. The backbone is enabled per-repo by **`publish_manifest`** ([repo_settings.mdx ¬ß2.4](./repo_settings.mdx#24-pinning--fetch-policy))
   and **`settings.backbones.repo_git.enabled`**; the transport is a **`git pull`/merge**, and an
   incoming committed list is **reconciled, never applied** (hub ¬ß5).
6. The reader **detects git merge-conflict markers (`<<<<<<<`/`=======`/`>>>>>>>`), REFUSES to load
   the file, quarantines it, and reconciles the two sides per hub ¬ß5.3** ‚Äî content forks kept +
   surfaced, never auto-deleted; the previous good copy is retained meanwhile.
7. A pull bringing a revision **behind** this computer's real state does not override it (local claims
   verified against the filesystem + IPFS pinset, hub ¬ß5.1); an entry **absent** after a merge is
   **NOT a delete** (hub ¬ß5.2); detached/old/shallow checkouts are treated as untrusted.
8. LFB writes the list **deterministically** ‚Äî file entries **sorted by `path`**, stable key ordering,
   one entry per line-group, byte-stable formatting ‚Äî so independent edits on two computers merge
   cleanly and only genuine content forks collide (¬ß6).
9. On scan/sync, LFB **regenerates** the list (bump `origin.revision`, stamp `written_at`, write only
   self-verifiable `pinned_by`, deterministic + atomic write), but **does NOT force git commits/pushes
   unless configured** ‚Äî it respects the user's git workflow and never creates/switches branches.
10. When a repo appears on both a computer list and its own in-repo list, the **in-repo (git) list is
    authoritative for that repo's `files`** (hub ¬ß5.4); this is the safest backbone (no third party),
    honors the IPFS only-our-content defaults, and adopts a list only from an **allow-listed owner**.
