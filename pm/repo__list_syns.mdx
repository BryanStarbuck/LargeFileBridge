# LargeFileBridge вЂ” Repo (Git) Backbone

The **git repo is a bridge backbone** for a repo's large-file tracking. This is one of three backbones the
[Sync List hub](./sync_list.mdx) defines; its siblings are [dropbox.mdx](./dropbox.mdx) and
[google_drive.mdx](./google_drive.mdx). Where those two carry a *computer*-scope list through a cloud
folder, this backbone carries a **per-repo (scope: `repo`)** big-file manifest вЂ” the large-file в†’ IPFS CID +
pin-state map вЂ” so a user's other computers can fetch the right bytes.

> **What changed (the data-category split).** The per-repo manifest is **Category B** tracking state
> ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). It **no longer lives committed inside
> the working repo**. It lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml`
> **always**, and **travels via the company/Personal sync repo** вЂ” the dedicated `*_large_files_bridge`
> tracking repo driven by the [git backbone engine](./git_backbone.mdx) вЂ” mirrored to
> `<syncRepo>/repos/<repoKey>/manifest.yaml` and union-merged, when that sync repo is configured and the
> per-repo toggle is on. The **working repo's own git** carries only its small committed files (code) and
> its committed **Category-A** content (transcripts + AI descriptions in `.lfbridge/`) вЂ” **never** the
> manifest. This supersedes the earlier design that committed `<repo>/.lfbridge/manifest.yaml` /
> `LargeFilesBridge_SyncList.yaml` into the working repo, whose per-scan churn produced merge conflicts.

> **The one rule this backbone is built on.** *The big files stay git-ignored; only the small manifest
> travels via git.* **Git carries the list, IPFS carries the bytes вЂ” never commit the large files.** The
> working repo's `.gitignore` continues to exclude every big binary; the small manifest that describes them
> travels through the **sync repo**, not the working repo. Losing this discipline (committing the bytes)
> defeats the entire product.

This spec does **not** restate the hub or the placement policy. It references them:

* [sync_list.mdx](./sync_list.mdx) вЂ” the Sync List schema, the `large_files` hierarchy, the two
  scopes, the `origin`/`revision` provenance, and the [В§5 trust/reconciliation model](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule).
  Read В§5 before writing any code that consumes a manifest a backbone delivered.
* [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) вЂ” the **data-category placement policy**
  that puts the manifest in Local Storage (always) + the sync repo (when enabled), never the working repo.
* [git_backbone.mdx](./git_backbone.mdx) вЂ” the **git engine** that fetches/merges/commits/pushes the sync
  repo, including the `merge=union` `.gitattributes` for the `repos/<repoKey>/manifest.yaml` +
  `decisions.yaml` this backbone rides.
* [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx) вЂ” where `repos/<repoKey>/manifest.yaml` sits among
  the rest of the repo's Category-B tracking state.
* [repo_settings.mdx В§2](./repo_settings.mdx) вЂ” the per-repo **sync-repo toggle** that turns this travel on.
* [knowledge/ipfs.mdx В§6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) вЂ” the
  only-our-content rule the manifest's IPFS settings must still honor.

## 1. Scope is `repo` вЂ” exactly one repo's manifest

A manifest on this backbone is **repo-scope** (hub [В§2](./sync_list.mdx#2-two-scopes--one-per-computer-one-per-repo)):

* `scope: repo` вЂ” this manifest covers **exactly one repo**.
* **Exactly one entry under `repos:`** вЂ” the repo the files live in. There is **no second repo** and
  **no `directories:` block** (non-repo directories belong to the *computer*-scope list on
  Dropbox/Drive, not here).
* **Paths are relative to the repo working tree** (hub [В§4](./sync_list.mdx#4-what-travels-vs-what-stays-local)).
  The writer's absolute `identity.path` is advisory only; the reader resolves the repo by `remote` +
  local discovery, never by trusting an incoming absolute path.

The full schema and the level-by-level table live in hub
[В§3](./sync_list.mdx#3-the-file--largefilesbridge_synclistyaml); this spec adds only the placement and the
git-transport detail.

## 2. File location вЂ” Local Storage always, the sync repo when enabled (LOCKED)

The per-repo manifest is **Category B**, so it lives in the two homes the placement policy defines вЂ” and
**never** in the working repo:

```
ALWAYS  в†’  ~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml    (Local Storage вЂ” authoritative working copy)
ALSO    в†’  <syncRepo>/repos/<repoKey>/manifest.yaml                 (the travelling MIRROR, when the owning
           IFF the sync repo is configured AND the per-repo          company/Personal storage has a sync repo
           "sync tracking state" toggle is ON (default OFF)          and the toggle is on вЂ” written IN ADDITION)
```

The working repo, by contrast, holds only what travels with the repo itself:

```
<repo>/
в”њв”Ђв”Ђ .git/
в”њв”Ђв”Ђ .gitignore                          # continues to ignore the big files (LOCKED)
в”њв”Ђв”Ђ .lfbridge/                          # committed вЂ” CATEGORY-A CONTENT ONLY (travels via the repo's own git)
в”‚   в”њв”Ђв”Ђ videos/trees.mov.transcription  # a transcript (Transcribe.mdx) вЂ” committed, path-mirrored
в”‚   в””в”Ђв”Ђ videos/trees.mov.ai_description # an AI description (ai_description.mdx) вЂ” committed
в”њв”Ђв”Ђ large/                              # the big files themselves вЂ” GIT-IGNORED, moved by IPFS
в”‚   в””в”Ђв”Ђ trailer_v3.mp4
в””в”Ђв”Ђ src/                                # ordinary code вЂ” tracked by git as usual
```

* **`repoKey` = `shortHash(resolve(repoPath))`** вЂ” the 12-hex `storageSid()` scheme
  ([artifact_placement_policy.mdx В§3](./artifact_placement_policy.mdx)).
* **The manifest never touches the working repo's git.** In Local Storage it is machine-local and cannot
  merge-conflict; in the sync repo it rides the purpose-built tracking repo the git backbone auto-pushes.
* **`.gitignore` still must NOT ignore `.lfbridge/`** вЂ” but now for the **Category-A** reason: the
  transcripts / AI descriptions committed there are *meant* to travel with the repo
  ([git_ignore.mdx В§5.3](./git_ignore.mdx), [git_backbone.mdx В§4.2.1](./git_backbone.mdx)). LFB verifies on
  scan that a broad ignore rule has not swallowed those committed artifacts and warns loudly if it has.

The Local-Storage manifest (and its optional sync-repo mirror) is written by code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/manifest.service.ts` вЂ” `writeManifest` (path from `manifestPath`, resolved via `tracking-root.service.ts` вЂ” `resolveTrackingRoot()` / `resolveStateSyncRepo()`).

### 2.1 The mirror subtree is keyed by the REMOTE вЂ” `repos/<repoUid>/` (LOCKED, supersedes `<repoKey>` above)

**Superseded:** the box above (and every `<syncRepo>/repos/<repoKey>/` in this spec) named the **mirror**
subtree by the path-derived `repoKey`. The mirror is now keyed by the repo's **normalized git remote**.
The rule is owned by [storage_company.mdx В§8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked);
this section binds it to the manifest's travel path and does not restate the rationale.

```
ALWAYS  в†’  ~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml   (Local Storage вЂ” PATH-keyed, unchanged)
ALSO    в†’  <syncRepo>/repos/<repoUid>/manifest.yaml                (the travelling MIRROR вЂ” REMOTE-keyed)
```

* **`repoUid` = short hash of `host/owner/name`** вЂ” lowercased, with scheme, credentials, any `.git`
  suffix and any trailing slash stripped, so `git@github.com:ACT3ai/charlie-kirk.git` and
  `https://github.com/ACT3ai/charlie-kirk` fold to the same `github.com/act3ai/charlie-kirk`.
* **Why the key had to change.** `repoKey` is `shortHash(resolve(repoPath))` вЂ” a **machine-local** value.
  The same repo is `/Users/bryan/BGit/вЂ¦` on one computer and `/Users/bryanstarbuck/BGit/вЂ¦` on the other, so
  the two machines wrote and looked for **different directories in the same sync repo**: the manifest
  travelled perfectly and was never found. A machine-local value must never key machine-shared data.
* **No remote в‡’ no mirror, and the product says so.** A repo with no git remote has no shared identity;
  LFB tells the user in the repo's settings ("no remote вЂ” tracking stays on this computer") instead of
  writing into a subtree no other computer will ever read.
* **Local Storage is unchanged.** `repos/<repoKey>/` stays path-keyed вЂ” it is machine-local state and
  *should* be. The same repo cloned twice on one machine keeps **two** Local-Storage state dirs (correct вЂ”
  two working trees) and converges on **one** mirror subtree (correct вЂ” one repo).
* **The union-merge attributes still cover it.** The `.gitattributes` the git backbone installs matches on
  the **filename** (`manifest.yaml`, `decisions.yaml`), so it applies at any depth вЂ” rekeying the subtree
  needs no attribute change ([git_backbone.mdx В§4.2](./git_backbone.mdx#42-append-mostly-shared-lists-merge-by-union)).

Deriving the remote-keyed mirror subtree is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` вЂ” `normalizeRemoteKey()` / `repoUidFor()` (re-exported from the leaf `repo-identity.ts`) consumed by `resolveStateSyncRepo()` (which joins `repos/<repoUid>/`); the remote-folding helper already shipping is `normalizeRemoteKey()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts`.

### 2.2 The owning SDL is the sync repo, and the mirror defaults ON (LOCKED, supersedes "default OFF")

**Superseded:** wherever this spec says the sync-repo travel is **default OFF** and an **opt-in**, the
default is now **ON** whenever an owning storage with a **git-backed SDL** resolves. The rule is owned by
[storage_company.mdx В§8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked):

* The owning company/Personal storage's SDL **is** the sync repo вЂ” auto-adopted from its own git root,
  the same rule `getGitBackboneRemote()` already applies to the Git backbone, so the sync repo and the
  backbone stop disagreeing about what the storage's repo *is*. An explicitly configured `sync_repo` still
  wins when present.
* **Ownership resolves from the repo's remote org first**, path containment only as a fallback.
* The per-repo toggle ([repo_settings.mdx В§2](./repo_settings.mdx)) still exists and still wins вЂ” it is now
  an **opt-out**. Only when **no** owning storage with a git-backed SDL resolves (or the user opted out) is
  the manifest Local-Storage-only.

## 3. Reconciliation with the placement policy (supersedes the old committed manifest)

The earlier design (and [storage_local.mdx В§9.2](./storage_local.mdx#92-repo-manifests-travel-via-git-the-elegant-part))
described a bare committed `<repo>/.lfbridge/manifest.yaml` вЂ” path в†’ CID + pin state, carried by the
**working** repo's git. **That is retired.** The mapping to the new model is exact:

| Old (committed `<repo>/.lfbridge/manifest.yaml`) | New (`repos/<repoKey>/manifest.yaml`, Local Storage + sync repo) |
| --- | --- |
| the manifest `files:` rows (`path`/`cid`/`size`/`sha256`/`modified_at`/`pinned_by`) | the **same rows**, unchanged in shape |
| committed into the **working** repo, carried by `git pull` on the working repo | written to **Local Storage always**; travels via the **sync repo** (union-merge), never the working repo |
| re-stamped on every pin pass в†’ **merge conflicts in the working repo** | re-stamped freely in Local Storage (no git); in the purpose-built sync repo the churn is expected (В§4) |

In short: **the manifest is Category B tracking state**, so it moved out of the working repo entirely. The
per-repo **sync-repo toggle** ([repo_settings.mdx В§2](./repo_settings.mdx)) вЂ” on a storage that has a sync
repo configured вЂ” is what turns on the travel; any pre-existing bare `<repo>/.lfbridge/manifest.yaml` is
migrated into Local Storage on the next pin pass and dropped from the working tree.

Resolving the Local-Storage manifest path and the optional sync-repo mirror on each pin pass is wired in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` вЂ” `pinRepoFolder` (writes `repos/<repoKey>/manifest.yaml` via `manifest.service.ts`, mirrored to the sync repo when `resolveStateSyncRepo()` is non-null).

## 4. Transport is the sync repo's git вЂ” the asynchronous change is a `git pull` there

On the whole-computer backbones, the "asynchronous change from another computer" is a cloud-folder sync.
**Here it is a `git pull` / merge on the company/Personal sync repo** (driven by the
[git backbone engine](./git_backbone.mdx#6-the-backbone-cycle--where-git-runs-inside-the-pass-locked-order)),
**not** on the working repo. When the pin pass runs its git cycle on the sync repo, git delivers whatever
`<syncRepo>/repos/<repoKey>/manifest.yaml` the peer committed and pushed. That incoming file is exactly the
kind of asynchronous, other-computer-authored input the hub trust model governs вЂ” it is a set of **claims,
never ground truth** (hub [В§5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule)).

* The backbone is enabled per-repo by the **sync-repo toggle** ([repo_settings.mdx В§2](./repo_settings.mdx))
  on a storage whose owning company/Personal storage has a **sync repo** configured. Default **OFF** вЂ” absent
  that, the manifest is Local-Storage-only (real and honored locally, simply not yet shared).
* After the sync repo's pull, LFB reads the delivered manifest, reconciles it against this computer's
  filesystem + IPFS pinset (hub В§5), folds it into the Local-Storage manifest, and fetches any CID it lacks
  over IPFS вЂ” the same fetch flow as before, now driven from Local Storage rather than a working-repo commit.
* **The per-scan re-stamp never dirties the working repo.** `refreshCounts()` / manifest writes route
  through `resolveTrackingRoot()` to Local Storage ([artifact_placement_policy.mdx В§6](./artifact_placement_policy.mdx)),
  so a repo the user never asked LFB to touch stays pristine.

Reading the manifest a sync-repo pull delivered and folding it into Local Storage is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/manifest.service.ts` вЂ” `readManifest` / `reconcileManifest` (consumed after `GitBackbone.pull()` on the sync repo).

### 4.1 Folding it in is a PER-ENTRY MERGE, on EVERY pull (LOCKED)

The bullet above says "folds it into the Local-Storage manifest." This section says **how**, and **when** вЂ”
because "fold" *was once* implemented as a last-writer **copy** with **no caller at all**. Both halves are
fixed: the fold **is** a per-entry union merge, and the pin passes call it. The rule is owned by
[storage_company.mdx В§8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked) and is the
hub's В§5 trust model ([sync_list.mdx В§5.7](./sync_list.mdx#57-the-sync-repo-mirror--keyed-by-remote-merged-on-every-pull-locked))
applied to this backbone:

* **Union by `path`** вЂ” an entry present on either side survives.
* **Union `pinned_by`** вЂ” the two computers' pin claims add together. A copy erases the peer's claim, and
  with it the entire *"a computer of yours has these bytes"* signal that the pull-down flow is built on.
* **Newer `modified_at` wins a conflicting `cid` for the same path**; ties keep the local value. A genuine
  content fork is still surfaced per hub [В§5.3](./sync_list.mdx#53-conflicts-resolve-by-revision-then-content-then-time).
* **Absence is never a delete** вЂ” restating В§5.3 below, and now binding on the merge itself.
* **This computer's own pin claim is re-derived from its real pinset** (hub В§5.1), never adopted from what
  arrived: a peer's manifest cannot make this machine believe it holds bytes it does not.
* **It runs on every sync-repo backbone pull**, immediately after fetch + merge вЂ” not on demand. A
  reconcile that only happens when someone clicks something is a reconcile that does not happen.

Merging a pulled sync-repo subtree into Local Storage per entry is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` вЂ” `reconcileFromSyncRepo()` (a per-entry union merge over `mergeManifests()`; it formerly copied the tree wholesale and had no callers вЂ” both halves are fixed), called from the repo pin pass in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` вЂ” `pinRepoFolder()`, and from the SDL git cycle (`syncStorageTextInner()` / `pinStorageUnitInner()`) via `reconcileMirroredRepos()` after `GitBackbone.pull()`.

## 5. Git-specific hazards the reader MUST handle

Git mangles files in git's own ways. Because the manifest now travels through the **sync repo's** git,
these hazards apply to the sync-repo copy. Beyond the generic backbone hazards in hub
[В§5.5](./sync_list.mdx#55-backbone-delivered-hazards-the-reader-must-expect), the reader must expect:

### 5.1 Merge conflicts in the YAML (LOCKED вЂ” refuse, quarantine, reconcile)

A `git pull` on the sync repo can leave the manifest with literal conflict markers:

```
large_files:
  repos:
    largefilebridge:
      files:
<<<<<<< HEAD
        - path: large/trailer_v3.mp4
          cid: bafybeigdвЂ¦              # this computer's line
=======
        - path: large/trailer_v3.mp4
          cid: bafybeih2вЂ¦              # the peer's line
>>>>>>> origin/main
```

The `merge=union` `.gitattributes` the git backbone installs on the sync repo
([git_backbone.mdx В§4.2](./git_backbone.mdx)) makes these rare, but the reader **MUST still detect
`<<<<<<<` / `=======` / `>>>>>>>` markers, REFUSE to load the conflicted file, and quarantine it** (never
parse a half-merged file as if it were valid YAML вЂ” it is not, and even if it parsed it would be a lie). It
then **reconciles the two sides per hub [В§5.3](./sync_list.mdx#53-conflicts-resolve-by-revision-then-content-then-time)**:
higher same-writer `revision` wins; different writers with different `cid`/`sha256` is a genuine content
fork вЂ” keep both, mark the entry `conflicted`, surface it in the web app, **never auto-delete either side's
bytes**. The previous good Local-Storage copy is retained until reconciliation resolves the markers.

### 5.2 A pull that is behind this computer's real state

A peer may have committed a `revision` **older than what this computer actually holds** (they pushed before
scanning; they were offline; the branch lagged). The pull will bring that older manifest, but it does
**not** override reality: LFB **verifies local claims** (hub
[В§5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)) вЂ” a `pinned_by` naming this computer is
trusted only if `ipfs pin ls` actually holds the CID; this computer's own filesystem is the source of truth
for what exists here. An older incoming revision never demotes a newer local truth.

### 5.3 Absence after a merge is NOT a delete

If a merged-in manifest **omits** an entry this computer currently has, that is **not** a delete signal (hub
[В§5.2](./sync_list.mdx#52-absence-is-not-deletion-default)). A shallow clone, a peer that had not scanned
that file yet, or a merge that dropped a side can all produce spurious absences. Deletes must be
**explicit** (`state: removed` + `removed_at`/`removed_by`). A merge that loses an entry never destroys the
bytes.

### 5.4 Detached / old / shallow checkouts

A detached-HEAD checkout, a stale branch, or a shallow clone of the sync repo can present a manifest that is
arbitrarily old. Treat it like any other untrusted input: reconcile by `origin.revision` + content, never
assume the checked-out file is current just because git handed it over.

## 6. Merge-friendliness вЂ” deterministic serialization (LOCKED)

Because git merges the manifest **line by line** in the sync repo, LFB writes the YAML **deterministically**
so that independent edits on two computers merge cleanly most of the time and **only genuinely-conflicting
entries collide.** The canonical serialization is a rule, not a preference (it applies to both the
Local-Storage copy and the sync-repo mirror, which stay byte-identical):

* **File entries are sorted by `path`** вЂ” a stable, total order, so two machines that add different
  files touch different regions of the file and merge without conflict.
* **Stable key ordering** within every mapping (e.g. `path`, `cid`, `size`, `sha256`, `modified_at`,
  `pinned_by` always in that order) вЂ” no key churn to create spurious diffs.
* **One entry per line-group**, consistently indented, so a single changed file is a single contiguous
  hunk git can merge in isolation.
* **Consistent formatting** вЂ” fixed indentation, no trailing whitespace, a trailing newline, stable
  quoting вЂ” so re-serializing an unchanged manifest produces a byte-identical file (no noise commits).

The payoff: two computers editing *different* files never conflict; only two computers editing the
*same* file's bytes produce a real content fork, which is exactly the case hub В§5.3 wants surfaced.

## 7. Publishing вЂ” regenerate into Local Storage, mirror to the sync repo

On scan or pin pass (hub [В§6](./sync_list.mdx#6-publishing--writing-our-list-out-to-the-backbones)), for a
repo whose owning storage has a sync repo configured and the per-repo toggle on, LFB regenerates the
manifest:

* **Write Local Storage first** вЂ” the authoritative copy at `repos/<repoKey>/manifest.yaml`.
* **Bump `origin.revision`**, stamp `origin.written_at`, set `origin.computer_id`/`label`/`owner` to
  **this** machine.
* Write **only what this computer can vouch for** вЂ” a `pinned_by` entry for this computer only for
  CIDs it actually pins right now (hub В§5.1); peer claims are copied forward from the last reconciled
  truth, never invented.
* Serialize **deterministically** (В§6) and write **atomically** (temp-then-rename,
  [storage_local.mdx В§15](./storage_local.mdx#15-the-store-layer-how-code-touches-disk)) so a concurrent
  reader or a git operation never sees a half-written file.
* **Mirror to the sync repo**, then let the git backbone commit + push it
  ([git_backbone.mdx В§6](./git_backbone.mdx)). The per-scan re-stamp *does* produce commits **there** вЂ” in
  the purpose-built repo, which the auto-commit/auto-push loop already carries. **The working repo's git is
  never touched.**

The atomic temp-then-rename write plus the sync-repo mirror are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/store/yaml-store.ts` вЂ” `writeYaml`, and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` вЂ” `mirrorToSyncRepo(repoRoot)` (invoked after the Local-Storage write; the git commit/push is `GitBackbone.commitAndPush()`).

The project `CLAUDE.md` forbids auto-branching/pushing a **working** repo; committing to the **sync repo** is
the granted, purpose-built exception ([git_backbone.mdx В§1](./git_backbone.mdx)). LFB commits only the
`repos/<repoKey>/` subtree on the sync repo's current branch, never the big files, and never touches the
working repo.

## 8. Authority вЂ” the per-repo manifest wins for that repo's files

Per hub [В§5.4](./sync_list.mdx#54-when-the-same-repo-arrives-on-two-backbones): a repo can appear both
in a *computer*-scope list (Dropbox/Drive) **and** in its own repo-scope manifest (the sync repo). **The
per-repo manifest is AUTHORITATIVE for that repo's `files`**, because it is the Category-B tracking state
LFB owns for that repo вЂ” resolved from Local Storage and reconciled through the sync repo, so it cannot
drift from the repo's real state the way a separately-carried computer-list mirror can. The computer-list
copy of that repo is a convenience mirror only. Settings still resolve on the normal inherit chain (per-repo
override в†’ computer `settings` в†’ global `config.yaml`).

---

## 9. Privacy & security

* **Only the user's own repos.** This backbone uses only the user's own **sync repo** (a dedicated
  `*_large_files_bridge` tracking repo the user configured) and the user's own computers вЂ” no third party's
  repo, no public relay ([git_backbone.mdx В§7](./git_backbone.mdx)). The big-file bytes move over IPFS
  between the user's own machines.
* **Only-our-content still holds.** The manifest's `settings.ipfs` keeps the
  [ipfs.mdx В§6](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) defaults вЂ”
  `reprovide_strategy: pinned`, no public/recursive gateway. Publishing a manifest never turns this
  machine into a gateway/relay.
* **Allow-listed owners only.** LFB never adopts a manifest whose `origin.owner` is not an
  allow-listed user for this install (hub В§5.5). A shared repo pins only for the emails in its
  `access.participants` ([repo_settings.mdx В§2.5](./repo_settings.mdx#25-sharing--some-repos-need-users-some-dont)).

## 11. Acceptance criteria

1. The per-repo manifest is **Category B** and lives in **Local Storage** at
   **`~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml`** always; the big files stay **git-ignored** in
   the working repo and move over IPFS вЂ” **git carries the list, IPFS carries the bytes, and the large files
   are never committed**.
2. The manifest is **never committed into the working repo**. The working repo's committed `.lfbridge/`
   holds only **Category-A** content (transcripts / AI descriptions); `.gitignore` **must NOT ignore
   `.lfbridge/`** so that content travels, and LFB warns loudly if a broad rule would swallow it.
3. The manifest is **`scope: repo`** with **exactly one entry under `repos:`**, **no `directories:`
   block**, and paths **relative to the repo working tree** (hub В§2, В§4).
4. This **supersedes** the earlier committed `<repo>/.lfbridge/manifest.yaml`: the `files:` rows are
   unchanged in shape, but the file moved to Local Storage (+ the sync repo); any pre-existing bare
   committed manifest is **migrated into Local Storage** and dropped from the working tree on the next pin
   pass.
5. Travel is enabled per-repo by the **sync-repo toggle** ([repo_settings.mdx В§2](./repo_settings.mdx)) on a
   storage that has a **sync repo** configured; the transport is a **`git pull`/merge on the sync repo**
   ([git_backbone.mdx В§6](./git_backbone.mdx)), and an incoming manifest is **reconciled, never applied**
   (hub В§5). Default **OFF** в†’ Local-Storage-only.
6. The reader **detects git merge-conflict markers (`<<<<<<<`/`=======`/`>>>>>>>`), REFUSES to load
   the file, quarantines it, and reconciles the two sides per hub В§5.3** вЂ” content forks kept +
   surfaced, never auto-deleted; the previous good Local-Storage copy is retained meanwhile.
7. A pull bringing a revision **behind** this computer's real state does not override it (local claims
   verified against the filesystem + IPFS pinset, hub В§5.1); an entry **absent** after a merge is
   **NOT a delete** (hub В§5.2); detached/old/shallow checkouts are treated as untrusted.
8. LFB writes the manifest **deterministically** вЂ” file entries **sorted by `path`**, stable key ordering,
   one entry per line-group, byte-stable formatting вЂ” so independent edits on two computers merge
   cleanly and only genuine content forks collide (В§6); the Local-Storage copy and the sync-repo mirror stay
   byte-identical.
9. On scan or pin pass, LFB **regenerates** the manifest into Local Storage (bump `origin.revision`, stamp
   `written_at`, write only self-verifiable `pinned_by`, deterministic + atomic write) and **mirrors it to
   the sync repo**, letting the git backbone commit/push **there** вЂ” it **never** touches the **working**
   repo's git and never creates/switches branches.
10. When a repo appears on both a computer list and its own per-repo manifest, the **per-repo manifest is
    authoritative for that repo's `files`** (hub В§5.4); it honors the IPFS only-our-content defaults, uses
    only the user's own sync repo (no third party), and adopts a manifest only from an **allow-listed owner**.
11. The **mirror subtree is keyed by the repo's normalized git REMOTE** вЂ” `<syncRepo>/repos/<repoUid>/`,
    `host/owner/name` with scheme/creds/`.git`/trailing slash stripped вЂ” **never** by the path-derived
    `<repoKey>`, so both computers read and write the same directory; a repo with **no remote does not
    mirror and says so**; Local Storage stays `repos/<repoKey>/` (В§2.1, superseding AC 1/AC 5's `<repoKey>`
    mirror wording).
12. The owning company/Personal storage's **git-backed SDL is auto-adopted as the sync repo** and the
    per-repo mirror **defaults ON** вЂ” the toggle is an **opt-out**, an explicit `sync_repo` still wins, and
    ownership resolves from the **remote org** before any path-containment fallback (В§2.2, superseding
    AC 5's "default OFF").
13. A pulled mirror is folded in by **per-entry MERGE on every sync-repo pull** вЂ” union by `path`, union
    `pinned_by`, newer `modified_at` wins a CID conflict, absence **never** a delete, own pin claim
    re-derived from the real pinset вЂ” **never** a last-writer copy and never on-demand-only (В§4.1).
