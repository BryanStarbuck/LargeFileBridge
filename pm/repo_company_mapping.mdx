# LargeFileBridge — Repo → Company / Personal mapping

Every git repo LFB discovers on this computer ([repos.mdx](./repos.mdx)) belongs to **one owner** — either a
**company** or the user's own **Personal** storage. This spec defines how that owner is **derived** — offline,
from the repo's git remote — how a derived company becomes a real [Company storage](./storage_company.mdx) with a
friendly name, how the user **overrides** the guess, and how the mapping makes repos **groupable under their
owner** in the left bar ([left_bar.mdx](./left_bar.mdx)) and the Repos list.

The one principle that governs everything here: **default is Personal.** A repo is only a company's when LFB can
show a reason — a forge org in the remote that is not one of *your* accounts. Everything else falls to Personal,
the existing catch-all ([storage_personal.mdx §9](./storage_personal.mdx#9-personal-is-the-default-owner-of-loose-files--and-where-their-artifacts-land)).

Read the shared model first; this spec does not restate it:

* [storages.mdx](./storages.mdx) — the storage family, the `storage.yaml` descriptor (§3), the `.lfbridge/`
  tracking area (§4), and **the Storages left-bar tab and its derived children** (§2).
* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — the storage-**kind** rule that
  decides where a root's LFB files land: a **working repo** (kind `repo`) keeps a hidden `.lfbridge/`; the
  **company storage** this mapping auto-creates is an **SDL** — a dedicated LFB file repo with **no
  `.lfbridge/` at all**, its `storage.yaml` and `devices/` sitting at the **root**. Relevant here because
  the mapping's product is a company storage.
* [storage_company.mdx](./storage_company.mdx) — the company storage this mapping **auto-creates**; owns the
  friendly display-name rename and the company sync repo.
* [storage_personal.mdx](./storage_personal.mdx) — the default owner; owns the **"my personal accounts"** list.
* [repos.mdx](./repos.mdx) / [one_repo.mdx](./one_repo.mdx) — the Repos list (row → `RepoRow`) and per-repo screen.
* [repo_settings.mdx](./repo_settings.mdx) — the per-repo settings page where the manual **Reassign owner** control lives.
* [repo_owner_propagation.mdx](./repo_owner_propagation.mdx) — the **travel + consent** layer: how a
  company assignment is committed into the company sync repo as an **assertion** and applied on other members'
  computers only through an explicit **review page**. This spec owns the *local* override; that one owns how it
  *travels*.
* [devices.mdx](./devices.mdx) — the device graft; unaffected by ownership (owner is a *grouping*, not a path).

## 1. What an "owner" is (LOCKED)

An **owner** is the answer to *"whose repo is this?"* — a **company** or **Personal**. It is **grouping metadata**,
not a location: assigning an owner **never moves a byte**, never re-roots the repo, and never touches the device
graft. It changes only **where the repo appears** (which group in the left bar / Repos list) and **which sync
repo** its tracking state may additionally mirror to ([§7](#7-how-ownership-drives-grouping-and-artifact-placement),
and [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). The tracking state itself lives in **Local
Storage always** regardless of owner — the owner only decides the additive travel mirror.

Two owner kinds:

* **Personal** — the repo belongs to the user themselves. There is exactly **one** Personal owner
  ([storage_personal.mdx](./storage_personal.mdx)); it is the **default** and the catch-all.
* **Company** — the repo belongs to a company, identified by a derived or assigned **company storage**
  ([storage_company.mdx](./storage_company.mdx)). A user in three companies can have repos under any of the three.

### 1.1 The repo `owner` field (LOCKED shape)

Every `RepoRow` / `RepoDetail` / `RepoSettings` gains an **`owner`** block so the UI can group and label without
re-deriving on every render:

```ts
// shared/src/types.ts — the owner of a repo (repo_company_mapping.mdx §1.1).
export interface RepoOwner {
  kind: "personal" | "company";
  companyId?: string;        // when kind==="company": the company STORAGE id it maps to (storages.mdx §2)
  displayName: string;       // the label shown everywhere: the company's friendly name, or "Personal"
  source: "auto" | "manual"; // "auto" = derived by the heuristic (§3); "manual" = a user override (§5)
  // Parsed provenance from the remote (§2) — retained so the UI can EXPLAIN the guess and re-derive on rescan:
  host: string | null;       // e.g. "github.com" · null when there is no forge remote
  ownerSlug: string | null;   // e.g. "ACT3ai" — the org/owner segment from the remote · null when none
}
```

* **`displayName`** is what the left bar and Repos grouping show. For a company it is the company storage's
  friendly name ([storage_company.mdx](./storage_company.mdx)); for Personal it is literally **"Personal"**.
* **`source`** records whether the value is the heuristic's guess (`auto`) or the user's decision (`manual`).
  A `manual` owner is **sticky** — a rescan never overwrites it (§5.2).
* **`host` / `ownerSlug`** are the parse result kept for the explain-string (*"grouped under ACT3ai because the
  remote is github.com/ACT3ai/…"*) and to let a rescan **re-derive** an `auto` owner when the remote changes.

The `owner` block is composed onto each repo row/detail in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `computeRepoRow()` / `computeRepoDetail()` (call `resolveRepoOwner(cfg)` after reading the unit config).

## 2. Parsing the remote — host · owner · repo (LOCKED)

Ownership derivation starts by **parsing the git remote origin URL** captured at repo registration
([units.service.ts](./repos.mdx) — `readGitRemote(dir)`, stored on the repo unit config as `repo.remote`). The parser
handles the common forge URL shapes:

| Remote form | Example | host | owner | repo |
| --- | --- | --- | --- | --- |
| **SCP-like SSH** | `git@github.com:ACT3ai/jfksocial_server.git` | `github.com` | `ACT3ai` | `jfksocial_server` |
| **HTTPS** | `https://github.com/ACT3ai/jfksocial_server.git` | `github.com` | `ACT3ai` | `jfksocial_server` |
| **SSH URL** | `ssh://git@gitlab.com/acme/tools.git` | `gitlab.com` | `acme` | `tools` |
| **git://** | `git://codeberg.org/user/thing.git` | `codeberg.org` | `user` | `thing` |
| **sr.ht** | `https://git.sr.ht/~sircmpwn/aerc` | `git.sr.ht` | `~sircmpwn` | `aerc` |

The parser is host-agnostic on shape but only treats a **known forge** host as forge-derivable for the heuristic
(§3): `github.com`, `gitlab.com`, `bitbucket.org`, `codeberg.org`, `git.sr.ht`. A `.git` suffix is stripped; a
trailing slash is trimmed; the URL's `user@` / credentials portion is discarded; the **first** path segment after
the host is the owner, the **second** is the repo.

```ts
// git.service.ts — parse a remote into its parts (repo_company_mapping.mdx §2). NO network — string work only.
export interface RemoteOwnerParse {
  host: string | null;   // "github.com" · null when unparseable / no remote
  owner: string | null;  // "ACT3ai" · null when the URL has no owner segment
  repo: string | null;   // "jfksocial_server" (".git" stripped)
  knownForge: boolean;   // host ∈ {github.com, gitlab.com, bitbucket.org, codeberg.org, git.sr.ht}
}
export function parseRemoteOwner(remoteUrl: string | null): RemoteOwnerParse { /* … */ }
```

Parsing host/owner/repo out of the remote URL (all URL shapes, no networking) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `parseRemoteOwner(remoteUrl)`, built beside the existing `classifyRemote()` / `classifyRemoteVisibility()` (which share the same forge-host regex).

### 2.1 The org level IS the company — a company storage is 1:1 with an organization (LOCKED)

The `owner` segment this parser extracts is not merely an *input to a guess*; it **is the company**. The
authority is [storage_company.mdx §10](./storage_company.mdx#10-a-company-is-a-forge-organization-locked);
what it means **here** is that the mapping this spec defines is a **grouping**, not an inference:

* **A company storage is one-to-one with a forge organization.** The org is a level in every repo's remote
  URL — `https://github.com/`**`ACT3ai`**`/charlie-kirk.git` — so **the repos already declare which company
  they belong to.** Large File Bridge does not ask the user to draw that map; it reads it off the disk.
* **Grouping needs no new scan and no network.** LFB already records each discovered repo's `origin`, so
  grouping by org is a fold over data it has: parse (§2) → `host / owner / repo`, group by `owner`. It
  **never** calls a forge API to ask who owns what
  ([external_tools_policy.mdx](./external_tools_policy.mdx),
  [storage_company.mdx §10.1](./storage_company.mdx#101-finding-the-organizations)) — and §3.1's honest
  limitation is precisely the price of that, deliberately paid.
* **A repo whose remote does not parse belongs to no org**, and therefore to **Personal** (§3 cases 1/4/5).
  It is not a company candidate at all.
* **Multi-company works without configuration.** Three orgs across the user's repos mean **three
  companies**, each with its own storage, each knowing exactly which repos are its own — and each repo
  resolving to **exactly one** of them
  ([storage_company.mdx §10.4](./storage_company.mdx#104-what-the-linkage-buys-locked)). That last point is
  a **confidentiality boundary**, not just tidiness: one company's tracking text is never pushed into
  another company's repo, and the org in the URL is what enforces it.

Grouping the discovered repos by their remote's org (offline, off `.git/config`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-discovery.service.ts` — `discoverOrganizations()` (and its pure core `buildCandidates()`), over the parse from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `parseRemoteOwner()`.

## 3. The heuristic — which parse becomes a company (LOCKED)

Given the parse (§2) and the user's **"my personal accounts"** list ([§4](#4-the-my-personal-accounts-list-locked),
owned by [storage_personal.mdx](./storage_personal.mdx)), LFB derives the `auto` owner:

1. **No remote, or an unparseable / local-path remote** (`knownForge === false` and no owner) → **Personal.**
2. **A known forge, owner segment present, owner IS one of the user's personal accounts** → **Personal.**
   (Your own `github.com/BryanStarbuck/…` repos stay yours.)
3. **A known forge, owner segment present, owner is NOT a personal account** → **Company**, named after the
   owner slug. LFB **derives a company** for that slug (§6) and maps the repo to it.
4. **A known forge host but no clear owner segment** (rare — e.g. a bare host) → **Personal** (nothing to name a
   company after).
5. **An unknown / self-hosted host** (e.g. `git.acme-corp.internal`) → **Personal by default** — we cannot assume
   a random host is a company forge, and the URL alone does not tell us. The user can **reassign** to a company
   (§5) if it is in fact a work server.

The heuristic is deliberately **conservative toward Personal**: a false "Personal" is harmless (the repo just
sits under your own files); a false "Company" spawns a stray company in the left bar. Case 3 is the only one that
creates a company, and only from a **known forge** with a **non-personal** owner.

Deriving the `auto` owner from the parse + personal-accounts list is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `deriveOwnerForRemote(remote)` (calls `parseRemoteOwner`, checks the personal-accounts list, and on case 3 calls `ensureCompanyForOwner(slug)` — §6).

### 3.1 The honest limitation (LOCKED)

A plain **`github.com/<owner>/<repo>`** cannot be perfectly distinguished as **organization vs. personal user**
from the URL alone — GitHub uses the same `github.com/<name>/…` shape for both an org and a person. LFB **does not
resolve this over the network** (the charter is local-first / no-phone-home). So:

* We **default per the heuristic** (§3): an owner that is not on your personal-accounts list is *assumed* a
  company, because that is the more useful grouping for a work machine, and it is trivially correctable.
* We **let the user correct it** — either by **reassigning** the repo (§5) or by **adding the owner to the
  personal-accounts list** (§4), after which every repo under that owner re-derives to Personal on the next scan.
* We **never guess silently and hide it**: the derived owner's `source: "auto"` and its `ownerSlug` are surfaced,
  and the left-bar group carries a subtle *(auto-detected)* affordance so the user knows it is a guess, not a fact.

This is a documented, accepted trade-off — **not** a defect. Do not "fix" it by adding a forge API call.

### 3.2 Which orgs are actually YOURS — the authored-commit membership test (LOCKED)

§3 case 3 says a known-forge org that is not a personal account **derives** a company. That is the right
rule for **mapping an existing repo**, and the wrong rule for **deciding which companies exist**. The
authority is [storage_company.mdx §10.2](./storage_company.mdx#102-which-organizations-are-actually-yours-locked);
this section is where the mapping spec absorbs it.

> **Why this is not theoretical.** A real developer machine is full of clones of **other people's**
> projects. Measured on the machine this rule was written against: **32 distinct orgs**, of which **28 were
> clones** — `KDE`, `OpenShot`, `xai-org`, `twitter`, `mltframework`, `NatronGitHub`, … A naive
> one-company-per-org rule would have created **31 unwanted storages** and 31 Storages entries for
> organizations the user has no membership in. §3's heuristic is conservative toward Personal for a *single
> repo*; across a whole disk it is not conservative enough.

The test is **local, cheap, and decisive**:

> **An org becomes a company only when the user's own git identity has authored at least one commit in one
> of that org's repos on this computer.**

* **Cloning is passive; committing is membership.** `git log --author=<your email>` answers it from the
  local object store — no network, no credentials, no forge API (§3.1's rule holds here too). On the
  measured machine it cut **32 orgs to 4**, and all 4 were correct.
* **Your own account is Personal, not a company.** An org matching the **personal-accounts list** (§4)
  resolves to **Personal** and is **never proposed** as a company — that is exactly what keeps
  `github.com/<you>/<your-repo>` out of the company list, and it is the same list §3 case 2 already
  consults.
* **Match the identity case-insensitively (LOCKED).** `git log --author=` is a **case-sensitive regex**,
  and a configured `user.email` reads back lowercased while commits record whatever casing the user typed —
  so a case-sensitive match answers "no" for the user's **own** repos. **A membership test that fails
  closed on a capital letter is worse than no test**, because its answer looks considered. Same rule as
  §4's case-insensitive `owner` match.
* **It is a PROPOSAL filter, never a lock.** The user can add any org as a company by hand and can dismiss
  a proposed one. A heuristic decides what to **offer**; the user decides what **exists** — the same
  discipline as §5 ("the user always wins").
* **Not-yet-committed is not never.** An org that fails the test today passes the first time the user
  authors a commit in one of its repos. Nothing has to be re-scanned by hand.
* **Say the number.** What the filter excluded is **reported**, with a way to see it — *"28 organizations
  you've only cloned from were ignored"*. A silent filter and a bug look identical
  ([warnings.mdx](./warnings.mdx)); surfaced in
  [storages.mdx §2.2](./storages.mdx#22-organizations-found-in-your-repos--proposed-company-storages-locked).
* **This does not change §3.** A repo whose org **is** a company still maps to it by §3 case 3. §3.2 only
  governs which orgs get **promoted to a company storage** in the first place; an org that fails the test
  simply has no company for its repos to map to, so they fall to **Personal** — the default (§1), and a
  correctable one.

Applying the authored-commit membership test and the personal-account exclusion when proposing companies is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-discovery.service.ts` — `discoverOrganizations()` (groups by org, then filters by the authored-commit test and `personal_accounts`), with the pure, testable core in `buildCandidates(repos, opts)`; the personal-account match itself is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `isPersonalAccount()`.

## 4. The "my personal accounts" list (LOCKED)

To keep the user's **own** forge repos out of the company buckets, the user maintains a small list of **their own
account handles** across forges. Any repo whose parsed owner matches an entry derives to **Personal** (§3 case 2),
even on a known forge. This list is owned by [storage_personal.mdx](./storage_personal.mdx) (it is a property of
*you*) but consumed here.

```yaml
# app config (storage_local.mdx) — the user's own forge accounts.
personal_accounts:
  - host: github.com      # match host + owner …
    owner: BryanStarbuck
  - host: gitlab.com
    owner: bstarbuck
  - owner: bryan-personal  # … or match owner on ANY host (host omitted = wildcard)
```

* **Match is case-insensitive** on `owner`; `host` is optional (omit it to treat the handle as personal on every
  forge). An sr.ht `~handle` matches with or without the leading `~`.
* **Editing the list re-derives.** Adding an owner marks every repo currently mapped **auto → company** under that
  owner for re-derivation on the next scan; removing one lets those repos fall back to a company again. **Manual**
  (`source: "manual"`) mappings are untouched (§5.2).
* **Seeding.** On first run LFB may **suggest** the owner of the repo whose remote matches the machine's git
  `user.email` domain, but never adds it silently — the list starts empty and is populated by the user (charter:
  surface and offer, never act unasked).

Persisting and matching the personal-accounts list is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig().personalAccounts` (read) / `setPersonalAccounts()` (write), matched by `isPersonalAccount(host, owner)` in `storage.service.ts`.

## 5. Manual reassignment — the user always wins (LOCKED)

The heuristic is a starting point; the **user's decision overrides it**. From the repo's
[settings page](./repo_settings.mdx#2-the-settings-on-this-page) (and the repo `⋮` menu) the user can **Reassign
owner**:

* **Set to Personal** — force this repo under Personal regardless of its remote.
* **Set to a Company** — pick an existing company, or type a new company name (which creates one, §6). Force this
  repo under that company.
* **Reset to auto** — discard the override and let the heuristic decide again (§3).

### 5.1 The Reassign control

A single **"Owner"** row on the repo settings page shows the current owner as a labeled control:

```
Owner   ( ) Personal    (•) Company:  [ ACT3ai ▾ ]   (auto-detected)      [ Reset to auto ]
                                        └─ picker: existing companies + "New company…"
```

* The two radios are **Personal** / **Company**; choosing **Company** enables the picker of existing companies
  plus a **New company…** entry (opens a name field → creates the company storage, §6).
* The **(auto-detected)** tag shows while `source === "auto"`; picking anything flips `source` to `"manual"` and
  hides the tag. **Reset to auto** returns `source` to `"auto"` and re-runs §3.
* A one-line **explain string** under the control states the basis: *"Auto-detected as ACT3ai from the remote
  github.com/ACT3ai/jfksocial_server"* or *"Manually assigned to ACT3 AI."*

### 5.2 Persistence — the override lives in the repo's `config.yaml` (LOCKED)

The **local grouping override** is a **per-computer** preference (like the bookmark, [repos.mdx §8.3](./repos.mdx#83-persistence--it-lives-in-the-repos-configyaml-locked)),
so it persists in that repo's pin-unit **`config.yaml`** — **not** in the shared manifest — and is **never
overwritten by a teammate**: only *this* member changes *this* computer's grouping. Only the **override** is
stored; an `auto` owner is *derived*, never written, so it always re-reflects the current remote.

**But the company assignment is also a team fact that travels — via consent, not silently.** Assigning a repo
to a company is more than a local grouping tweak: it changes **where that member's tracking state flows** (a
company-owned repo with the sync toggle on mirrors into the **shared** company sync repo). So the *decision*
must reach the other members — yet applying it to their machines unasked would be a security surprise. The
resolution, owned in full by [repo_owner_propagation.mdx](./repo_owner_propagation.mdx):

* The **local override here** stays exactly as above — machine-local, never auto-overwritten.
* In **addition**, when the company has a **sync repo** configured, LFB commits a small
  **ownership assertion** (*"this repo, by its git remote, belongs to company Y"*) into that sync repo's
  `owner_map.yaml`. That assertion **travels** to every member.
* On another member's computer the assertion becomes a **pending mapping** that is applied to their local
  `owner_override` **only after they consent** through the review page
  ([repo_owner_propagation.mdx §4](./repo_owner_propagation.mdx)). No teammate's grouping ever changes silently.

So "the override is per-computer" and "company ownership propagates" are both true: the *override* is local; the
*assertion* travels; the *application* on another machine is gated by that member's explicit choice.

```yaml
# pin/r/<repo>/config.yaml  (excerpt)
repo: { name: jfksocial_server, path: /Users/…/AI_Coding, remote: "git@github.com:ACT3ai/jfksocial_server.git" }
bookmarked: false
owner_override:                 # ABSENT unless the user reassigned (source becomes "manual" only when present)
  kind: company                 # personal | company
  company_id: "a1b2c3d4e5f6"    # when kind: company — the company storage id (storages.mdx §2)
```

* **Absent block = auto.** With no `owner_override`, `resolveRepoOwner()` returns the §3 derivation (`source:"auto"`).
* **Present block = manual.** `resolveRepoOwner()` returns the override (`source:"manual"`) and the scan never
  clobbers it — re-registering or rescanning the repo preserves `owner_override` exactly like `bookmarked`
  ([repos.mdx §8.3](./repos.mdx#83-persistence--it-lives-in-the-repos-configyaml-locked)).
* **API:** `POST /api/repos/:repoId/owner` with `{ kind, companyId? } | { reset: true }` writes/clears the
  override and returns the updated `RepoRow`. Idempotent; unknown `repoId` → 404; a `company` kind with an unknown
  `companyId` → 400.

Resolving the effective owner (override else derivation) and persisting the override is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `resolveRepoOwner(cfg)` (override-or-derive) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `reposRouter` → `POST /api/repos/:repoId/owner`. When the chosen owner is a **company with a sync repo**, the same endpoint also **asserts** the ownership into the company sync repo (and tombstones it on un-assign) via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/owner-propagation.service.ts` — `assertCompanyOwnership()` / `withdrawCompanyOwnership()` ([repo_owner_propagation.mdx §2](./repo_owner_propagation.mdx)).

## 6. Auto-creating the company — and its friendly name

When the heuristic (§3 case 3) or a manual "New company…" (§5) needs a company that does not exist yet, LFB
**creates a company storage** so it appears as its own child under **Storages** ([storages.mdx §2](./storages.mdx#2-the-storages-left-bar-tab)):

* The new company storage is seeded with `type: company` and `company_name` = the **owner slug** (e.g. `ACT3ai`),
  so it is discoverable immediately. Its **id** is the stable company id `resolveRepoOwner` stamps on the repo.
* Because org slugs are **cryptic** (`ACT3ai`, `acme-eng-2`, `~sircmpwn`), the company storage carries a
  user-settable **friendly display name** — set from the company's settings — which the product then shows
  **everywhere** (left bar, Repos grouping, the repo's Owner control). The rename flow and where the display name
  is stored are owned by [storage_company.mdx §6](./storage_company.mdx#6-the-friendly-display-name).
* Until the user renames it, the display name **falls back to the slug** — never blank, never the raw URL.

An auto-derived company from an org (case 3) is a **lightweight** entry: it exists to group repos and to be
renamed. It becomes a *full* company storage (mapped source directories, a sync repo, an allow-list) only if
the user configures those in [storage_company.mdx](./storage_company.mdx) — a repo-group company and a
media-pile company are the same storage kind at different levels of configuration.

**But "auto-create" never means "silently make a directory."** Creating a company storage makes a **real
directory** on the user's disk and runs **`git init`** in it, so the sweep that turns *discovered orgs* into
*company storages* is **proposed and one-click confirmed**, never done behind the user's back
([storage_company.mdx §10.3](./storage_company.mdx#103-proposing-and-creating-the-company-storages-locked),
surfaced in [storages.mdx §2.2](./storages.mdx#22-organizations-found-in-your-repos--proposed-company-storages-locked)).
Three rules follow, and none of them changes the mapping above:

* **Adopt before you create.** An org already **claimed** by an existing company storage — an explicit
  `company.owner_slugs` entry, or a normalized name match (§6.1) — is **adopted**, never duplicated. This
  is how `act3_large_files_bridge` serves the org `ACT3ai` instead of a second `act3ai_large_files_bridge`
  appearing beside it.
* **A dismissed org stays dismissed**, remembered per org so the same proposal does not return on the next
  scan; it is undoable from the company's settings.
* **The parent directory is a setting**, defaulting to wherever the user's existing `*_large_files_bridge`
  storages already live — never hardcoded to one person's layout
  ([storage_settings.mdx §4d](./storage_settings.mdx#4d-new-company-storages--the-proposal-surface-and-the-parent-directory-locked)).

Ensuring (creating-if-absent) the company storage for a derived/assigned owner slug is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureCompanyForOwner(slug)` (returns the company id; creates the `storage.yaml` via `ensureStorage(root, "company", …)` when missing), surfaced by `listStoragesPage()` as a company row.

### 6.1 Binding the slug to a real company storage row (LOCKED)

Deriving the string `ACT3ai` is not the same thing as knowing **which storage** that is. The binding is what
makes *"remote org `ACT3ai` → this SDL"* a **fact the code can act on** rather than a label, and everything
downstream — the sync-repo target (§7.3), the Owned-repos list
([storage_settings.mdx §4c](./storage_settings.mdx#4c-owned-repos--the-repos-mapped-to-this-storage)), the
ownership assertion ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx)) — needs it.

* **The owner block carries a real `companyId`.** A derived company owner (§1.1) must resolve its
  **`companyId`** to the id of an existing **company storage row**, not leave it null. The `ownerSlug` stays
  as the derivation's evidence; the `companyId` is the join.
* **Matching, in order.** (1) A **manual** `owner_override` naming a company id matches **exactly**. (2) An
  auto-derived slug matches a company storage row by its **explicit org binding** when the row records one.
  (3) Failing that, by a **normalized comparison** — lowercased, non-alphanumerics stripped — of the slug
  against the row's `company_name`, its row `name`, and its root-derived slug. (4) No match → **create** the
  company row (§6) and bind it.
* **The normalized name match is a best-effort bridge, not the design.** The **explicit** binding wins
  wherever it is present; a match won by the fuzzy rules is **written back** as an explicit binding so the
  heuristic runs once and the result becomes something the user can see, edit, and share.
* **Renaming does not break the binding.** The friendly display name is user-editable
  ([storage_company.mdx §6](./storage_company.mdx#6-the-friendly-display-name)); the binding is to the org
  **slug**, so renaming `ACT3ai` to *"ACT3 AI"* leaves every repo still bound to the same storage row.
* **Known gap, narrowed (stated so it is not mistaken for done).** The resolver itself **exists**:
  `ensureCompanyForOwner(slug)` implements all three rules above, including the write-back. What is **still
  missing is the join at derivation time** — `deriveOwnerForRemote()` returns **`companyId: null`**, so the
  owner block that the rest of the product reads is not yet attached to a company storage row, and every
  consumer that needs the row has to call `ensureCompanyForOwner()` itself. Filling that `companyId` is the
  remaining precondition for §7.3's ownership lookup to resolve without a second call. Same narrowing as
  [storage_company.mdx §8.7](./storage_company.mdx#87-four-smaller-defects-this-use-case-also-closes).

Resolving a slug to its company storage row (and recording the binding) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureCompanyForOwner()` (explicit `owner_slugs` claim → normalized name match → lone-company adoption, with write-back via `recordOwnerSlug()`).

Deriving the owner (the step that still leaves `companyId` unjoined) and the best-effort match the settings page uses are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `deriveOwnerForRemote(remote, personalAccounts)` (returns `companyId: null` — the remaining gap) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `ownerMatchesCompany(owner, row)` (exact `companyId` for a manual override, normalized slug/name comparison for an auto owner), consumed by `getOwnedRepos(storageId)` in the same file.

## 7. How ownership drives grouping and artifact placement

### 7.1 Grouping in the left bar and Repos list (LOCKED)

The mapping's main payoff: **repos become groupable under their owner.**

* In the **Storages** disclosure ([left_bar.mdx §3](./left_bar.mdx#3-the-app-bar-main-navigation)), each company
  child and the Personal child can **expand to show the repos owned by it** — the flat **Repos** child (the long
  list) stays, but repos also nest under their owner ([left_bar.mdx §3.2](./left_bar.mdx), defined by this spec).
* In the **Repos list** ([repos.mdx](./repos.mdx)), an optional **Group by owner** view clusters rows under an
  owner header (Personal, then each company by display name), each header showing its repo count. This reuses the
  table's grouping; the default flat sort is unchanged.
* A repo with an `auto` company owner renders its group header with a subtle **(auto-detected)** hint (§3.1); a
  `manual` owner shows no hint.

### 7.2 Artifact placement follows the owner

Ownership decides **which sync repo** a repo's LFB tracking state may travel to
([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
[repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)):

* A repo's tracking state (`repo_storage.yaml`, `files.yaml`, per-file sidecars, `history/<device>.txt`,
  `decisions.yaml`, `manifest.yaml`) lives in **Local Storage always** at
  `~/T/_large_files_bridge/repos/<repoKey>/` — machine-local, never in the working repo.
* If the repo's **owner** is a **company** that has configured a **sync repo**
  ([storage_company.mdx §7](./storage_company.mdx#7-the-company-sync-repo)) **and** the repo's per-repo sync
  toggle is on ([repo_settings.mdx §2.9](./repo_settings.mdx)), that state is **additionally mirrored** to
  `<syncRepo>/repos/<repoKey>/` — in addition to Local Storage, not instead of it — so a teammate on the same
  company sees it. A **Personal**-owned repo can likewise mirror to Personal's sync repo
  ([storage_personal.mdx §14](./storage_personal.mdx#14-personals-optional-sync-repo)).
* Ownership only decides **which** company-vs-personal sync repo is the mirror target; the additive mirror
  itself is ~~**default OFF**~~ and per-repo. **SUPERSEDED** — the mirror now defaults **ON** per
  [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)
  and §7.3 below; the per-repo toggle remains, as an **opt-out**. Transcripts / descriptions are a **separate
  data category** that obey the per-repo placement radios ([repo_settings.mdx](./repo_settings.mdx),
  [placement_radios.mdx](./placement_radios.mdx)).

### 7.3 Ownership decides the sync-repo target, remote first (LOCKED)

The authority is
[storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked).
What that rule adds **here** is that this spec's derivation is no longer only a *grouping* input — it is the
input that decides **where a repo's tracking text travels**. The two systems, the git-remote→company mapping
and the sync-repo→storage mapping, were disconnected; ownership is the joint.

* **The remote org resolves the owning storage; path containment is only the fallback.** A repo's owner is the
  storage its **remote org** binds to (§3, §6.1). Only when there is no forge org to bind — no remote, a
  local-path or self-hosted remote, or an owner on the personal-accounts list (§3 cases 1/2/4/5) — does
  physical containment inside a storage's directory decide.
* **Why the order matters, concretely.** `~/BGit/Bryan_git/charlie-kirk` has remote
  `github.com/ACT3ai/charlie-kirk` but does **not** live inside the `ACT3ai` company storage's directory. A
  containment-first test resolves it to **Personal** — the wrong storage, and therefore the wrong sync repo,
  and therefore a manifest that travels to nobody who needs it. A repo's **remote** is the thing its
  teammates and its other computers agree on; its **path** is a fact about one disk.
* **The owning storage's SDL is the target.** Having resolved the owner, the mirror root is that storage's
  sync repo — the explicitly configured one when present, else the storage's **own git root, auto-adopted**
  (§8.4.2). So the answer is normally *"yes, there is one"*, not *"none configured"*.
* **This changes no bytes and no grouping.** §7.1's left-bar/Repos grouping is unchanged; ownership was
  always the group key. What is new is that the same key now also names a **destination**, and that
  destination carries **small text only** ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).
* **Reassignment re-targets it.** A manual reassignment (§5) moves the mirror target with the ownership —
  that is exactly what
  [storage_settings.mdx §4c.2](./storage_settings.mdx#4c2-reassigning-records-the-travelling-assertion--and-re-targets-tracking-locked)
  already describes; §7.3 only makes the *auto* path resolve the same way the manual path already does.

Resolving the owning storage from the repo's remote before any containment test, and returning its sync repo, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveOwnerDedicatedRepo(repoRoot)` (today matches only by mapped dirs or the Personal catch-all — the containment-first behavior this section corrects), fed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `parseRemoteOwner()` / `deriveOwnerForRemote()` and consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)`.

## 8. Non-default & edge cases (LOCKED)

| Case | Behavior |
| --- | --- |
| **No remote at all** | `owner.kind: personal`, `host/ownerSlug: null`. A purely local repo is yours (§3 case 1). |
| **Local-path remote** (`/Users/…/mirror.git`, `../bare.git`) | `knownForge:false`, no owner → **Personal** (§3 case 1). |
| **SSH vs HTTPS of the same repo** | Both parse to the **same** `host`+`owner` (§2), so they derive the **same** owner — switching remote protocol never re-buckets a repo. |
| **Ambiguous `github.com/<owner>/<repo>`** | Cannot tell org vs. user from the URL (§3.1). Defaults to **company** unless `<owner>` is on the personal-accounts list; user corrects via reassign or the list. |
| **Owner IS a personal account** | **Personal** (§3 case 2) even on a known forge. |
| **Unknown / self-hosted host** | **Personal by default** (§3 case 5); reassign to a company if it's a work server. |
| **Renamed org** (remote owner slug changed) | On rescan, an **auto** owner **re-derives** to the new slug and (if new) creates/attaches a new company; a **manual** owner is untouched. If the old auto-company now owns no repos it becomes an empty company (§9). The user can rename the new company to the same friendly name, or reassign. |
| **Repo moved between owners** (transferred to another org) | Same as renamed org — the remote's owner segment changes, so an auto owner follows it on rescan. |
| **Multiple remotes** (`origin` + `upstream` + a fork) | Ownership derives from **`origin` only** (the remote `readGitRemote` returns). Other remotes are ignored for grouping. If the user works off a non-origin remote, they reassign. |
| **Monorepo with submodules from many owners** | The **top-level repo's `origin`** owns the whole tree; submodule remotes do **not** create per-submodule owners (a repo has exactly one owner). Submodule big files group under the parent repo's owner. |
| **Two forges, same owner name** (`github.com/acme` and `gitlab.com/acme`) | By default both derive to a company named `acme` and, absent a display-name change, **merge into one** company group (name-keyed). If they are different companies, the user renames one or reassigns; a future `host` qualifier on the company key can split them (out of scope here). |
| **Owner slug with odd characters** (`~sircmpwn`, `dept.team`) | Kept verbatim as the slug; the friendly display name is where the user makes it readable (§6). |

## 9. Error & empty cases

* **Empty company group.** A company whose repos were all reassigned/removed shows **0 repos**. If it was
  **auto-created** (§6) and never renamed or otherwise configured, LFB may **garbage-collect** it on scan (remove
  the bare, repo-less, unnamed company storage) — a user-renamed or otherwise-configured company is **kept** even
  at 0 repos. Never delete a company the user has touched.
* **Unparseable remote.** `parseRemoteOwner` returning all-nulls falls to **Personal** (§3 case 1) — never an
  error state; the repo simply groups under Personal.
* **Reassign to an unknown company id** → `POST /api/repos/:repoId/owner` returns **400**; the UI keeps the prior
  owner and toasts.
* **Company creation fails** (e.g. cannot write `storage.yaml`) → the reassign is rejected, the repo keeps its
  prior owner, and the failure is surfaced (not silently swallowed); the derivation retries next scan.
* **Personal-accounts list malformed** → treated as empty (log a warning), so nothing is miscategorized *toward*
  Personal by a bad entry; the user's repos may show as companies until fixed, which is visible and correctable.
* **Stale display name.** Renaming a company updates the label everywhere on next read; a repo row cached with the
  old `displayName` refreshes on rescan (the label is derived, not frozen on the repo).

## 10. Acceptance criteria

1. Every repo carries an **`owner`** block (`kind: personal | company`, `companyId?`, `displayName`,
   `source: auto | manual`, `host`, `ownerSlug`) on `RepoRow` / `RepoDetail` / `RepoSettings` (§1.1), composed in
   `computeRepoRow` / `computeRepoDetail`.
2. **`parseRemoteOwner(remoteUrl)`** parses `host` + `owner` + `repo` (and `knownForge`) from the SSH, HTTPS,
   `ssh://`, `git://`, and sr.ht URL shapes across github.com / gitlab.com / bitbucket.org / codeberg.org /
   git.sr.ht, with **no networking** (§2).
3. The heuristic (§3): no/unknown/local remote → **Personal**; a **known forge** owner that **is** a personal
   account → **Personal**; a known-forge owner that is **not** a personal account → **Company** named after the
   owner slug (auto-creating the company, §6); default leans **Personal**.
4. The **"my personal accounts"** list (§4) keeps the user's own forge repos under **Personal**; editing it
   re-derives `auto` owners on the next scan and never disturbs `manual` ones.
5. **Deriving a company auto-creates a company storage** (`ensureCompanyForOwner`) so it appears as its **own
   child** under Storages ([storages.mdx §2](./storages.mdx#2-the-storages-left-bar-tab)); its cryptic slug is
   overridable to a **friendly display name** shown everywhere (§6, [storage_company.mdx](./storage_company.mdx)).
6. The user can **reassign** any repo to Personal or a Company (existing or new), or **reset to auto**, from repo
   settings / the `⋮` menu (§5); the choice persists as **`owner_override`** in the repo's **`config.yaml`**
   (`source: "manual"`, sticky across rescans, **never overwritten by a teammate**), and an absent block means
   auto-derive.
6b. The **local override is per-computer**, but a company assignment is **also a travelling assertion**: when the
   company has a **sync repo**, LFB commits the ownership into `owner_map.yaml` there, and on other members'
   computers it is applied **only through the explicit review-and-consent flow**
   ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx)) — never silently. §5.2's "per-computer override"
   and propagation are reconciled, not contradictory.
7. The **limitation is documented honestly** (§3.1): plain `github.com/<owner>/<repo>` cannot be perfectly
   org-vs-personal from the URL; LFB **defaults per the heuristic, surfaces the guess as (auto-detected), and lets
   the user correct it** — and **never** calls a forge network API to resolve it.
8. Ownership makes repos **groupable under their owner** (§7.1) in the left-bar Storages disclosure and the Repos
   list, and decides **which company-vs-personal sync repo** a repo's tracking state additionally mirrors to when
   the per-repo sync toggle is on — the state lives in Local Storage always regardless (§7.2,
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).
8b. The derived owner is **bound to a real company storage row** (§6.1): its **`companyId`** names an existing
   company storage — matched by the manual override's exact id, else the row's explicit **org binding**, else a
   **normalized** slug/name comparison, else by **creating** the row (§6) — and the binding is to the **org
   slug**, so renaming the company's friendly display name never breaks it. The **known gap** is stated, not
   hidden — and it is narrower than it was: `ensureCompanyForOwner(slug)` **exists** and resolves the row, but
   `deriveOwnerForRemote()` still returns `companyId: null`, so the owner block the product reads is not yet
   joined to a company storage row.
8c. **Ownership decides the sync-repo target, resolved REMOTE-ORG FIRST** (§7.3, authority
   [storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)):
   the owning storage is the one the repo's **remote org** binds to, with **path containment only as the
   fallback** for repos with no forge org — so `~/BGit/Bryan_git/charlie-kirk` (remote
   `github.com/ACT3ai/charlie-kirk`) resolves to the **ACT3ai** company, not Personal, despite living outside
   that company's directory. The target is that storage's configured sync repo, else its **own git root
   auto-adopted**, and the additive mirror **defaults ON** (the per-repo toggle in
   [repo_settings.mdx §2.9.1](./repo_settings.mdx#291-the-default-is-on--this-toggle-is-an-opt-out-locked) is
   an **opt-out**) — superseding AC 8's and §7.2's "default OFF". No bytes move; grouping (§7.1) is unchanged.
3b. **A company storage is 1:1 with a forge ORGANIZATION** (§2.1, authority
   [storage_company.mdx §10](./storage_company.mdx#10-a-company-is-a-forge-organization-locked)): the org is a
   level in every repo's remote URL, so repos **already declare** their company. Grouping by org is an
   **offline fold** over remotes LFB has already recorded — **no forge API call, ever** — and each repo
   resolves to **exactly one** company, which is a **confidentiality boundary** (one company's tracking text
   never enters another's repo), not merely tidiness.
3c. **An org qualifies as a company only by the authored-commit membership test** (§3.2, authority
   [storage_company.mdx §10.2](./storage_company.mdx#102-which-organizations-are-actually-yours-locked)): the
   user's own git identity must have authored **at least one commit** in one of that org's repos on this
   computer. Measured evidence: **32 orgs → 4**, because **28** were clones of other people's projects
   (`KDE`, `OpenShot`, `xai-org`, `twitter`, …) that a naive one-per-org rule would have turned into **31
   unwanted storages**. An org in **`personal_accounts`** resolves to **Personal** and is **never proposed**.
   The test is a **proposal filter, never a lock** — the user may add any org by hand or dismiss a proposed
   one — an org that fails today **passes** the first time the user commits to one of its repos, and **what
   was skipped is counted and visible** (a silent filter and a bug look identical).
5b. **Creating a company storage is PROPOSED, never silent** (§6): it makes a real directory and runs
   `git init`, so it takes a one-click confirmation. An org already **claimed** by an existing company
   storage is **ADOPTED, never duplicated** (`ACT3ai` adopts `act3_large_files_bridge`), a **dismissed** org
   stays dismissed, and the **parent directory is a setting**, not a hardcoded path
   ([storage_company.mdx §10.3](./storage_company.mdx#103-proposing-and-creating-the-company-storages-locked)).
9. All **non-default / edge cases** (§8) resolve as tabled — SSH↔HTTPS stable, origin-only, one owner per repo
   (submodules included), renamed/moved org re-derives auto owners — and the **error cases** (§9) fail safe toward
   Personal, never delete a user-touched company, and surface (never swallow) creation failures.
