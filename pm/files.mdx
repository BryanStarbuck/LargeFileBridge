# LargeFileBridge вҖ” View one file

The **View one file** page is the single-entity page for **one** file. A file shows up in many places
in LFB вҖ” a [repo's files table](./one_repo.mdx), the [File System](./directories.mdx) column browser, the
[IPFS](./ipfs.mdx) pinned table вҖ” and from any of them the **View file** action (or clicking the
file's name) lands here at `/file?path=<abs>`. The page *is* that file, so it carries the entity
**`вӢ®` "more" menu** ([menus.mdx В§4](./menus.mdx)).

Siblings this builds on and does not repeat:

* [Menus](./menus.mdx) вҖ” the LOCKED **File action catalog** (В§5.3), the `вӢ®` more-menu (В§4), right-click
  (В§3.1), and the two sticky flags **Never IPFS** / **Do not compress** (В§6.6). This page renders that
  catalog; it never invents a different one.
* [One repo](./one_repo.mdx) вҖ” the **decision model** (Pin / Ignore / Undecided, В§1) and the
  **transfer-status pill** (В§4.6) shown here for the file's pin state.
* [File System](./directories.mdx) вҖ” the **code badges** (В§3) painted at the top of this page.
* [Directories](./directories.mdx) вҖ” the sibling **View one directory** page (same shape, dir catalog).
* [Media viewer](./media_viewer.mdx) вҖ” the **viewer-first** siblings **View one image** (`/image`) and
  **View one video** (`/video`). When a file is a viewable medium, rows/cells/menus open the viewer page
  instead of this one; this page stays the properties-first fallback for **every other kind** and is
  always reachable from a viewer via **View properties**. This page also links **out** to the viewer for
  a medium (В§7).

## 0. This page vs. the viewers

`/file` is **properties-first** вҖ” identity, badges, flags, and the Pin/Compression/IPFS/File fact cards.
It is the destination for files that are **not** a playable/viewable medium (docs, archives, source, etc.).

When the file **is** a medium, the primary experience is the [media viewer](./media_viewer.mdx):

* image вҶ’ `/image?path=`, video вҶ’ `/video?path=`.

Navigation (row click, cell click, the menu's Open action) routes a medium **straight to its viewer**. This
`/file` page still exists for a medium too вҖ” reachable via the viewer's **View properties** link вҖ” and when
it renders a medium it offers an **Open image/video viewer** primary (В§7) so the two are never a dead end.

## 2. The data payload (`EntityView`)

The backend returns one typed record the page renders directly (`@lfb/shared`, `GET /api/entity?path=`):

```ts
interface FileFlags {
  neverIpfs: boolean;   // sticky: never add to IPFS / pin (menus.mdx В§6.6)
  noCompress: boolean;  // sticky: suppress the "should compress" offer/badge
}

interface EntityView {
  kind: "file" | "dir";
  name: string;
  path: string;              // absolute, server-normalized
  exists: boolean;
  sizeBytes: number | null;  // file bytes (dir: rolled-up or null)
  createdAt: string | null;
  modifiedAt: string | null;
  badges: FsBadge[];         // same vocabulary as directories.mdx В§3, order rightmost-first
  flags: FileFlags;

  // Repo / pin context вҖ” populated when the entity is inside a registered repo.
  repo: { repoId: string; name: string; relPath: string } | null;
  decision: Decision | null;     // sync | ignore | undecided (null when not in a repo)
  transfer: TransferStatus | null;
  cid: string | null;
  peers: string[];

  // Compression heuristic (directories.mdx В§3.3) вҖ” drives the Compression card.
  compressible: "video" | "image" | null;   // eligible media kind, else null
  compressState: "should" | "done" | null;  // C / c / none
}
```

`decision`, `transfer`, `cid`, and `peers` are meaningful only when `repo` is non-null (a file must be
a tracked large file in a registered repo to carry a decision). A loose file on disk still gets its
identity, badges, compression heuristic, and both flags.

### 2.1 A file row is not always a local file вҖ” `presence` (LOCKED)

The row model this page is the single-entity view of carries an optional **`presence`**:

```ts
// packages/shared/src/types.ts вҖ” on FileRow
presence?: "local" | "remote-only";   // ABSENT вҮ’ "local"
```

* **Absent means `"local"`, so every existing row, reader, and page is unchanged.** This is additive by
  construction; nothing needs a migration and no caller has to learn a new field to keep working.
* **A `remote-only` file is one another of the user's computers has pinned and this one does not have.** Its
  name, `path`, `sizeBytes`, `cid`, and `peers` come from the **reconciled manifest** вҖ” **not** from a
  `statSync`, because there is no local file to stat вҖ” plus the peer **device** that holds it. The rule, and
  the two-computer use case it exists for, are LOCKED in
  [storage_company.mdx В§8.5](./storage_company.mdx#85-remote-only-rows--a-file-in-the-list-that-is-not-on-this-disk-locked);
  its rendering in the repo file list is [one_repo.mdx В§4.10](./one_repo.mdx#410-remote-only-rows--a-file-in-this-list-that-is-not-on-this-computer-locked).
* **On this page it is a HEALTHY state, and must never be shown as the not-found error.** В§5's *"This file
  is no longer at that path"* card means **something went wrong**. A remote-only file means **nothing went
  wrong** вҖ” the bytes are safe on the user's other computer and simply have not been pulled here yet. So
  when `exists` is false **because** the file is remote-only, the page renders the **red "not on this
  computer yet"** state instead: *"On {device} вҖ” not on this computer yet."*, with the identity, size, CID,
  and peers it does know, and a single **Pull it down** primary. Confusing the two would turn the product's
  normal second-computer experience into an alarm.
* **Its only action is pull it down.** **CompressвҖҰ / Transcribe / AI-describe / OCR / Git ignore** (В§7.1,
  В§4's Compression card) are **disabled and say why** вҖ” *"the bytes aren't on this computer yet"* вҖ” because
  queueing analysis on absent bytes queues work that cannot run. The two **sticky flags** (В§4) and the
  **pin decision** are still writable: they are intent, and intent does not need the bytes. After the pull,
  the next scan makes it an ordinary local file and the page renders normally.

Carrying `presence` on the row model, and composing a remote-only entry from the reconciled manifest rather than a local stat, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` вҖ” `FileRow.presence`, set by
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` вҖ” `composeFileRows()`;
this page's not-found-vs-remote-only branch belongs in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` вҖ” `buildEntityView()` and
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` вҖ” `ViewOneFilePage`.

Assembling this `EntityView` payload (identity, badges, repo/pin context, compression heuristic, flags) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` вҖ” `buildEntityView()`, served by `entity.router.ts` вҶ’ `GET /api/entity`.

## 3. Primary vs. more (LOCKED split)

The page follows [menus.mdx В§4](./menus.mdx)'s "one primary, rest under `вӢ®`" rule:

* **Primary button** = the single most useful next action for this file's state:
  * `neverIpfs` off **and** decision вү  sync вҶ’ **+ Add to IPFS** (sets decision `sync`).
  * decision = `sync` вҶ’ **Remove from IPFS** is *not* the primary; **Pin now** is (queue this file).
  * `neverIpfs` on вҶ’ no IPFS primary; the primary becomes **CompressвҖҰ** if the file is compressible and
    `noCompress` is off, else there is no primary button (only `вӢ®`).
* Everything from the **File catalog** ([menus.mdx В§5.3](./menus.mdx)) is inside `вӢ®`, including the
  primary again (the menu is a complete catalog), with destructive **Stop pinning (untrack/unpin)**
  separated last.

Choosing which primary button to show for the file's flags + decision is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` вҖ” `ViewOneFilePage` вҶ’ `pickPrimary()`.

Queueing the file for the **Pin now** primary (POST `/repos/:id/pin`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts` вҖ” `api.pinNow()`.

## 4. The two sticky-flag switches

The badge/flag strip shows both flags from [menus.mdx В§6.6](./menus.mdx) as **labeled switches**, not
buried in the menu вҖ” this is the page where a user manages a file's standing policy:

* **Never IPFS** вҖ” on вҮ’ the file is never pinned; **Add to IPFS** disappears from the primary and menu,
  the `P` badge cannot appear, and a `sync` decision is demoted to `ignore`. Toggling writes immediately.
* **Do not compress** вҖ” on вҮ’ the Compression card drops its **CompressвҖҰ** offer and the `C` badge is
  suppressed; the card still reports what the file *is* (e.g. "Video В· 8.2 GB"), just without the nag.

The `вӢ®` menu's toggle items and these switches are **the same state** вҖ” flipping one updates the other.

Writing the two sticky flags вҖ” including the Never IPFS side-effect that demotes `sync` decisions under the path вҖ” is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` вҖ” `setEntityFlags()`.

## 7. Open-viewer affordance (media only)

When this page renders a **medium** (`compressible` is `"image"` or `"video"`), it surfaces a visible
**Open image viewer** / **Open video viewer** primary (in addition to whatever IPFS primary В§3 would show,
placed first) that navigates to `/image` / `/video` for the same path. This is the one link from the
properties-first page to its viewer-first sibling; the reverse link is the viewer's **View properties**.
For non-media files there is no such affordance.

Rendering the media-only Open image/video viewer affordance is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` вҖ” `OpenViewer()`.

## 5. Empty & error states

* **File not found / moved** (`exists: false`): a card вҖ” "This file is no longer at that path." with
  **Copy path** and a link back to the [File System](./directories.mdx) at its parent directory.
* **Not in a repo:** the Pin/IPFS cards render a muted "Not tracked вҖ” this file isn't inside a
  registered repo" with an **Open containing folder** link; badges, compression, and flags still show.
* **Loading:** skeleton header + cards, no layout shift.
* **IPFS node unreachable:** the IPFS card shows the node is down; Add-to-IPFS/Pin-now disable with a
  tooltip, but decision/flag writes still save (parity with [one_repo.mdx В§5](./one_repo.mdx)).

## 6. Acceptance criteria

1. `/file?path=<abs>` renders **one file's** identity (name title, absolute path muted), its **code
   badges** ([directories.mdx В§3](./directories.mdx)), and stacked fact cards (Pin, Compression, IPFS, File).
2. The page carries a **top-right `вӢ®` "more" menu** rendering the **File catalog**
   ([menus.mdx В§5.3](./menus.mdx)) verbatim (the same as the row kebab), destructive item last;
   right-clicking anywhere on the page's entity opens the same menu ([menus.mdx В§3.1](./menus.mdx)).
3. The **primary button** follows В§3 (Add to IPFS / Pin now / CompressвҖҰ / none) based on the file's
   flags and decision; the primary action also appears inside `вӢ®`.
4. Both **sticky flags** (**Never IPFS**, **Do not compress**) show as labeled switches (В§4), write
   immediately, stay consistent with the menu toggles, and honor [menus.mdx В§6.6](./menus.mdx) (Never IPFS
   forbids pinning + hides Add-to-IPFS + suppresses `P`; Do not compress suppresses `C`/the offer; neither
   deletes or alters bytes).
5. **Decision / transfer / CID / peers** render only when the file is inside a registered repo; a loose
   file shows identity + badges + compression + flags with a "Not tracked" note (В§5).
6. Not-found / node-unreachable / loading states render per В§5; nothing on this page ever **deletes** or
   **alters** the local file.
7. **A remote-only file is a healthy state, not the not-found error (В§2.1).** The row model carries an
   optional **`presence: "local" | "remote-only"`** whose **absence means `"local"`**, so every existing
   reader is unchanged ([storage_company.mdx В§8.5](./storage_company.mdx#85-remote-only-rows--a-file-in-the-list-that-is-not-on-this-disk-locked)).
   When the file is **remote-only**, this page shows the **red *"On {device} вҖ” not on this computer yet"***
   state вҖ” with the identity, size, CID, and peers taken from the **reconciled manifest**, never a
   `statSync` вҖ” and **not** В§5's *"no longer at that path"* card. Its **only** action is **Pull it down**;
   Compress / Transcribe / AI-describe / OCR / Git-ignore are **disabled and say why**, while the pin
   decision and both sticky flags stay writable. After the pull it becomes an ordinary local file.
