# LargeFileBridge — View one file

The **View one file** page is the single-entity page for **one** file. A file shows up in many places
in LFB — a [repo's files table](./one_repo.mdx), the [File System](./directories.mdx) column browser, the
[IPFS](./ipfs.mdx) pinned table — and from any of them the **View file** action (or clicking the
file's name) lands here at `/file?path=<abs>`. The page *is* that file, so it carries the entity
**`⋮` "more" menu** ([menus.mdx §4](./menus.mdx)).

Siblings this builds on and does not repeat:

* [Menus](./menus.mdx) — the LOCKED **File action catalog** (§5.3), the `⋮` more-menu (§4), right-click
  (§3.1), and the two sticky flags **Never IPFS** / **Do not compress** (§6.6). This page renders that
  catalog; it never invents a different one.
* [One repo](./one_repo.mdx) — the **decision model** (Pin / Ignore / Undecided, §1) and the
  **transfer-status pill** (§4.6) shown here for the file's pin state.
* [File System](./directories.mdx) — the **code badges** (§3) painted at the top of this page.
* [Directories](./directories.mdx) — the sibling **View one directory** page (same shape, dir catalog).
* [Media viewer](./media_viewer.mdx) — the **viewer-first** siblings **View one image** (`/image`) and
  **View one video** (`/video`). When a file is a viewable medium, rows/cells/menus open the viewer page
  instead of this one; this page stays the properties-first fallback for **every other kind** and is
  always reachable from a viewer via **View properties**. This page also links **out** to the viewer for
  a medium (§7).

---

## 0. This page vs. the viewers

`/file` is **properties-first** — identity, badges, flags, and the Pin/Compression/IPFS/File fact cards.
It is the destination for files that are **not** a playable/viewable medium (docs, archives, source, etc.).

When the file **is** a medium, the primary experience is the [media viewer](./media_viewer.mdx):

* image → `/image?path=`, video → `/video?path=`.

Navigation (row click, cell click, the menu's Open action) routes a medium **straight to its viewer**. This
`/file` page still exists for a medium too — reachable via the viewer's **View properties** link — and when
it renders a medium it offers an **Open image/video viewer** primary (§7) so the two are never a dead end.

## 1. Layout

One content column: a **header** (identity + primary button + `⋮` more menu), a **badge/flag strip**,
and stacked **fact cards** (Pin, Compression, IPFS, Fingerprint).

```
┌────────────┬───────────────────────────────────────────────────────────────────────────┐
│  LEFT BAR  │  ‹ back      final_4k.mov                          ┌─────────────┐  ┌───┐   │
│            │  ~/BGit/…/Our_Movies/render/final_4k.mov           │ + Add to IPFS│  │ ⋮ │   │ ← primary + more
│            │                                                    └─────────────┘  └───┘   │
│            ├───────────────────────────────────────────────────────────────────────────┤
│            │   i  C  P  r      Never IPFS ◯off     Do not compress ◯off                  │ ← badges + flag switches
│            ├───────────────────────────────────────────────────────────────────────────┤
│            │  Pin                                                                         │
│            │    Decision  [Pin ▾]      Status ● Pinned     Peers 3     CID bafy…  ⧉       │
│            │  Compression                                                                 │
│            │    Video · looks uncompressed · 8.2 GB      [ Compress… ]                    │
│            │  IPFS                                                                        │
│            │    Pinned here · CID bafy… · reachable by 3 of your computers                │
│            │  File                                                                        │
│            │    8.2 GB · created 2026-05-01 · modified 2h ago · inside repo Our_Movies    │
└────────────┴───────────────────────────────────────────────────────────────────────────┘
```

* **Header.** A back affordance, the **file name** as the title, the absolute **path** muted beneath.
* **Primary button.** The single most-common action is a visible button (see §3); everything else is
  under `⋮`.
* **`⋮` more menu.** Opens the **File catalog** ([menus.mdx §5.3](./menus.mdx)) verbatim — the same
  list the file's row kebab shows. Actions that make no sense at this altitude (e.g. **View file** —
  you are already on it) are omitted.

## 2. The data payload (`EntityView`)

The backend returns one typed record the page renders directly (`@lfb/shared`, `GET /api/entity?path=`):

```ts
interface FileFlags {
  neverIpfs: boolean;   // sticky: never add to IPFS / pin (menus.mdx §6.6)
  noCompress: boolean;  // sticky: suppress the "should compress" offer/badge
}

interface EntityView {
  kind: "file" | "dir";
  name: string;
  path: string;              // absolute, server-normalized
  exists: boolean;
  sizeBytes: number | null;  // file bytes (dir: rolled-up or null)
  createdAt: string | null;
  modifiedAt: string | null;
  badges: FsBadge[];         // same vocabulary as directories.mdx §3, order rightmost-first
  flags: FileFlags;

  // Repo / pin context — populated when the entity is inside a registered repo.
  repo: { repoId: string; name: string; relPath: string } | null;
  decision: Decision | null;     // sync | ignore | undecided (null when not in a repo)
  transfer: TransferStatus | null;
  cid: string | null;
  peers: string[];

  // Compression heuristic (directories.mdx §3.3) — drives the Compression card.
  compressible: "video" | "image" | null;   // eligible media kind, else null
  compressState: "should" | "done" | null;  // C / c / none
}
```

`decision`, `transfer`, `cid`, and `peers` are meaningful only when `repo` is non-null (a file must be
a tracked large file in a registered repo to carry a decision). A loose file on disk still gets its
identity, badges, compression heuristic, and both flags.

Assembling this `EntityView` payload (identity, badges, repo/pin context, compression heuristic, flags) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` — `buildEntityView()`, served by `entity.router.ts` → `GET /api/entity`.

## 3. Primary vs. more (LOCKED split)

The page follows [menus.mdx §4](./menus.mdx)'s "one primary, rest under `⋮`" rule:

* **Primary button** = the single most useful next action for this file's state:
  * `neverIpfs` off **and** decision ≠ sync → **+ Add to IPFS** (sets decision `sync`).
  * decision = `sync` → **Remove from IPFS** is *not* the primary; **Pin now** is (queue this file).
  * `neverIpfs` on → no IPFS primary; the primary becomes **Compress…** if the file is compressible and
    `noCompress` is off, else there is no primary button (only `⋮`).
* Everything from the **File catalog** ([menus.mdx §5.3](./menus.mdx)) is inside `⋮`, including the
  primary again (the menu is a complete catalog), with destructive **Stop pinning (untrack/unpin)**
  separated last.

Choosing which primary button to show for the file's flags + decision is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` — `ViewOneFilePage` → `pickPrimary()`.

Queueing the file for the **Pin now** primary (POST `/repos/:id/pin`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/api/client.ts` — `api.pinNow()`.

## 4. The two sticky-flag switches

The badge/flag strip shows both flags from [menus.mdx §6.6](./menus.mdx) as **labeled switches**, not
buried in the menu — this is the page where a user manages a file's standing policy:

* **Never IPFS** — on ⇒ the file is never pinned; **Add to IPFS** disappears from the primary and menu,
  the `P` badge cannot appear, and a `sync` decision is demoted to `ignore`. Toggling writes immediately.
* **Do not compress** — on ⇒ the Compression card drops its **Compress…** offer and the `C` badge is
  suppressed; the card still reports what the file *is* (e.g. "Video · 8.2 GB"), just without the nag.

The `⋮` menu's toggle items and these switches are **the same state** — flipping one updates the other.

Writing the two sticky flags — including the Never IPFS side-effect that demotes `sync` decisions under the path — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` — `setEntityFlags()`.

## 7. Open-viewer affordance (media only)

When this page renders a **medium** (`compressible` is `"image"` or `"video"`), it surfaces a visible
**Open image viewer** / **Open video viewer** primary (in addition to whatever IPFS primary §3 would show,
placed first) that navigates to `/image` / `/video` for the same path. This is the one link from the
properties-first page to its viewer-first sibling; the reverse link is the viewer's **View properties**.
For non-media files there is no such affordance.

Rendering the media-only Open image/video viewer affordance is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` — `OpenViewer()`.

### 7.1 Page actions — media shortcuts (proposed)

Today, when `/file` renders a **medium**, the analysis actions live only inside the viewer
([media_viewer.mdx §6](./media_viewer.mdx)) or the `⋮` more menu; this properties-first page offers only
**Open …viewer** (§7). A customer looking at a media file's properties should be able to kick the
analysis/compress work **from here** too, without first opening the viewer:

| Action | Shown for | What it does | Status |
| --- | --- | --- | --- |
| **Transcribe** | audio + video (`compressible === "video"` or an audio file) | Runs Whisper locally ([Transcribe.mdx](./Transcribe.mdx)) for this file, same engine as the viewer's Transcription panel. | **proposed** |
| **AI-describe** | image + video | Runs a vision provider ([ai_description.mdx](./ai_description.mdx)); if no key resolves, opens the credentials-missing popup ([ai_credentials.mdx](./ai_credentials.mdx)). | **proposed** |
| **Compress** | image + video, when the file "should" compress and **Do not compress** is off | The explicit-click, confirm-gated compress offer (charter) — mirrors the Compression card's button but promoted next to the primary. | **proposed** (the Compress offer already exists on the Compression card) |

These render as buttons in the header **beside the primary** (§3) / **Open …viewer** (§7), or folded
into the `⋮` File catalog — they are page-level actions on this single-entity page, distinct from the
per-row `⋮` kebab elsewhere ([menus.mdx](./menus.mdx)). This page is not a file-list page, so it does
**not** carry the shared file action-links row ([page_actions.mdx §0](./page_actions.mdx)); these are its own
header buttons. When built they belong in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/entity/ViewOneFilePage.tsx` —
`ViewOneFilePage`, reusing the same launchers the viewer uses (`runDescribeFile` in `lib/describe.ts`,
the transcribe hook, and the compress offer).

## 5. Empty & error states

* **File not found / moved** (`exists: false`): a card — "This file is no longer at that path." with
  **Copy path** and a link back to the [File System](./directories.mdx) at its parent directory.
* **Not in a repo:** the Pin/IPFS cards render a muted "Not tracked — this file isn't inside a
  registered repo" with an **Open containing folder** link; badges, compression, and flags still show.
* **Loading:** skeleton header + cards, no layout shift.
* **IPFS node unreachable:** the IPFS card shows the node is down; Add-to-IPFS/Pin-now disable with a
  tooltip, but decision/flag writes still save (parity with [one_repo.mdx §5](./one_repo.mdx)).

## 6. Acceptance criteria

1. `/file?path=<abs>` renders **one file's** identity (name title, absolute path muted), its **code
   badges** ([directories.mdx §3](./directories.mdx)), and stacked fact cards (Pin, Compression, IPFS, File).
2. The page carries a **top-right `⋮` "more" menu** rendering the **File catalog**
   ([menus.mdx §5.3](./menus.mdx)) verbatim (the same as the row kebab), destructive item last;
   right-clicking anywhere on the page's entity opens the same menu ([menus.mdx §3.1](./menus.mdx)).
3. The **primary button** follows §3 (Add to IPFS / Pin now / Compress… / none) based on the file's
   flags and decision; the primary action also appears inside `⋮`.
4. Both **sticky flags** (**Never IPFS**, **Do not compress**) show as labeled switches (§4), write
   immediately, stay consistent with the menu toggles, and honor [menus.mdx §6.6](./menus.mdx) (Never IPFS
   forbids pinning + hides Add-to-IPFS + suppresses `P`; Do not compress suppresses `C`/the offer; neither
   deletes or alters bytes).
5. **Decision / transfer / CID / peers** render only when the file is inside a registered repo; a loose
   file shows identity + badges + compression + flags with a "Not tracked" note (§5).
6. Not-found / node-unreachable / loading states render per §5; nothing on this page ever **deletes** or
   **alters** the local file.
