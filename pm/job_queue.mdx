# LargeFileBridge — The Background Job Queue

A page action can add **many** items at once — "Create Transcriptions for this whole repo" might be 300
media files. Running those inline in the HTTP request would be wrong: the request would hang for an hour,
and closing the tab would lose the work. So LFBridge has a **server-side background queue**. A page action
**enqueues and returns**; a bounded pool of workers **drains the queue in the background**.

This spec owns the queue mechanism. It is deliberately small and reuses what already exists:

* [page_actions.mdx](./page_actions.mdx) — the **producer**. Its enqueue endpoints (§5 there) resolve the
  eligible set (scope + skip-already-done) and call `enqueue()`; they never wait for the work.
* [webapp.mdx §11–§14](./webapp.mdx) — the **progress dock**, the in-process **job registry**
  (`progress.registry.ts`), the operation **taxonomy** (kinds `transcribe` / `describe`), and the bounded
  `mapLimit` concurrency model. The queue is the missing piece **between** "a page action fires" and "the
  registry shows a running card": the registry shows what's **running now**; the queue holds what's
  **waiting to run**.
* [Transcribe.mdx §1/§5.1](./Transcribe.mdx) / [ai_description.mdx](./ai_description.mdx) — the **per-file
  runners** (`transcribeOne` / `describeOne`) the queue calls. Each already wraps its run in a
  `track("transcribe"/"describe", …)` registry job, so a queued item that starts **automatically gets a
  live dock card** — the queue does not need to touch the dock itself.

---

## 1. Why a queue and not just `mapLimit`

The browser-fan-out runner (`useProgress().run`, [webapp.mdx §12A](./webapp.mdx)) already does bounded
parallelism — but it lives **in the browser tab** and keeps one open request per item. That is right for a
handful of clicks; it is wrong for a 300-file batch, because:

* the batch must **survive navigation and tab close** — the user clicks "Create Transcriptions" and walks
  away;
* the server, not the browser, owns "how many are still to do";
* Whisper/ffmpeg are **heavy and local** — we want a **small, server-enforced** concurrency, not one
  request per file all racing.

So producing **page** actions (page_actions.mdx) go through **this server queue**; interactive **per-file**
and **small selected-set** clicks may still use the browser runner. The queue is the durable-within-the-
process path for "many".

## 2. The task shape & the queue

The queue is an in-process **FIFO** of small task records:

```ts
export type JobOp = "transcribe" | "describe" | "compress";

export interface QueueTask {
  op: JobOp;             // which per-file runner handles it
  path: string;          // the absolute media file path
  overwrite: boolean;    // re-do even if the output exists (default false)
  provider?: string;     // describe only — the chosen vision provider (or "auto")
  compress?: {
    deleteOriginal: "hard" | "trash";  // per-run originals disposition
    mediaKind: "image" | "video";      // which media-aware compress budget this task draws (§3)
  };
  batchId?: string;      // groups the per-file tasks of one bulk run (a "Compress inside" batch)
}
```

The `compress` task carries its **media kind** (image vs video) because the two draw **different**
concurrency budgets (§3) — image tools are single-threaded (fan wide), ffmpeg is multi-threaded (fan
narrow). The producer stamps it from the file's extension when it plans the batch
([compress_inside.mdx §5](./compress_inside.mdx)).

* **`enqueue(tasks: QueueTask[]): { queued: number; deduped: number }`** appends the tasks and **kicks the
  drain loop**, then returns immediately. `queued` is how many were actually added (after dedup); `deduped`
  is how many were dropped because an identical `op + path` was already **pending or running**.
* **Dedup (LOCKED).** The same file is never queued twice for the same op at the same time. Clicking
  **Create Transcriptions** twice in a row does not double-queue; the second call reports everything as
  `deduped`. (Skip-already-**done** is the producer's job, [page_actions.mdx §1.2](./page_actions.mdx);
  dedup here is about the same file being **in flight** twice.)
* **FIFO.** Items drain roughly in the order enqueued; ordering is best-effort, not a guarantee callers may
  rely on.

The FIFO queue, `enqueue()`, and the in-flight dedup set are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `enqueue()` / the module-level `pending[]` + `inflight` Set.

## 3. The workers — the CORE BUDGET, media-aware (LOCKED)

A pool of **worker loops** drains the queue. Each loop repeatedly takes the next task whose **admission
bucket** has a **free slot**, runs it, and loops until the queue empties, then parks (no busy-wait).
Concurrency is **not** a fixed tiny constant — it is **derived from the machine's core count** so a many-core
box actually gets used. The policy is the canonical **Core Budget** ([parallelization.mdx](./parallelization.mdx)):
the **mass-compute** budget is `round(cores × max_core_fraction)`, `max_core_fraction` **default 0.9** — *use
up to ~90% of cores*.

The `compress` op is **media-aware** because the two media kinds have opposite tool profiles
([parallelization.mdx §2](./parallelization.mdx)), so it splits into **two admission buckets**:

| Bucket | Concurrency (on a `budget`-core machine) | Per-job threads | Why |
| --- | --- | --- | --- |
| `compress:image` | **wide** — `budget` (≈ 90% of cores) | **1** | oxipng/cwebp/mozjpeg/magick are ~single-threaded per file — one job per core saturates the box. |
| `compress:video` | **narrow** — `floor(budget / videoThreads)` (`videoThreads` default 4) | **`videoThreads`** | ffmpeg is internally multi-threaded; a few thread-capped jobs fill the budget without cores² thrash. |
| `transcribe` | **narrow** — `floor(budget / whisperThreads)` (min 1) | whisper-internal | Whisper is CPU/GPU-heavy and multi-threaded; a couple saturate the machine. |
| `describe` | fixed **≈ 4–6** (network, not core-bound) | — | The heavy wait is a vision-API round-trip, not local CPU — overlap I/O, don't burn cores. |

* **Anti-oversubscription (LOCKED, [parallelization.mdx §2](./parallelization.mdx)).** For every bucket
  `concurrentJobs × threadsPerJob ≈ budget`, **never `cores²`**. Image jobs fan wide at 1 thread each; video
  jobs fan narrow and the queue **passes a `-threads` cap into ffmpeg** for each batched job so N videos do
  not each grab every core. A **one-off** single-file Compress (the file/viewer menu) is **not** a batch and
  passes **no** cap — that lone file may use the whole machine.
* **Budgets are read live** from the shared helper at **admission time**, so changing
  `performance.max_core_fraction` in Settings ([parallelization.mdx §4](./parallelization.mdx)) takes effect
  on the **next** task with no restart. They are computed in **one** place, not hardcoded per op.
* **A task runs by calling its per-file runner** — `transcribeOne(path, overwrite)`,
  `describeOne(path, { overwrite, provider })`, or (for `compress`) `compressFile(path, { deleteOriginal,
  threads })` wrapped in `track("compress", …)`. Those runners **already** register a
  `track("transcribe"/"describe"/"compress", …)` job in the progress registry ([webapp.mdx §12](./webapp.mdx)),
  so the moment a queued item **starts**, a **live dock card** appears (determinate % for transcribe/compress,
  spinner for describe) — including in a tab that did not start the batch. The queue writes **no** dock code
  of its own. The `compress` task's heavy transcode is **async** (`spawn`, [compress_inside.mdx §5](./compress_inside.mdx))
  so it **never blocks the event loop** while the batch drains.
* **Per-item isolation.** A task that throws is caught, logged to `error.err`
  ([ai_description.mdx §8](./ai_description.mdx)), and the worker moves on — one bad file never stalls the
  queue. Each runner already returns a truthful status rather than throwing for expected outcomes.

The media-aware, core-budgeted worker loops (take next runnable task by admission bucket, dispatch to the runner, catch-and-continue) are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `pump()` / `runTask()` / `capFor()` + `bucketOf()` (the live per-bucket caps drawn from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts` — `coreBudget()`; buckets `transcribe` / `describe` / `compress:image` / `compress:video`); the dispatch maps `op` → `transcribeOne` / `describeOne` / `compressFile` (the last wrapped in `track("compress", …)`, passed the per-job `threads` cap, and on return `recordBatchResult()` against its `batchId` — §4a).

## 4. What the user sees — running cards + a queued count

The dock already shows **running** work (source B poll of `GET /api/progress`, [webapp.mdx §12](./webapp.mdx)).
The queue adds one thing: the **pending backlog count**, so the user knows work is still waiting, not just
the few cards currently running.

* **Running items** → ordinary dock cards, unchanged — they come from the registry the runners populate.
* **Pending backlog** → a small **`queued` integer** folded into the same `GET /api/progress` payload:

```
GET /api/progress → { jobs: Job[], queued: number }   // queued = tasks waiting (not yet started)
```

The dock shows a subtle footer like **"+ 24 queued"** under the live cards when `queued > 0`, and it clears
to nothing as the backlog drains. No card is fabricated for a not-yet-started item ([webapp.mdx §13](./webapp.mdx):
"a queued item simply has no card yet") — `queued` is just a number.

The `queued` count is exposed by the queue and folded into the progress payload in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `queueDepth()`; read by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/progress/progress.router.ts` — `GET /` (adds `queued` to `ProgressListResult`). The dock footer renders it in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ProgressDock.tsx`.

## 4a. Batches — grouping one bulk run's tasks

A **batch** groups the per-file tasks of **one bulk run** — today, a **"Compress inside"** run
([compress_inside.mdx](./compress_inside.mdx)) — so a surface can show that run's **progress bar** and, when
it finishes, its **final error list**. Each enqueued `compress` task carries the run's `batchId` (§2).

* **`createBatch(label, total): batchId`** — opens a batch when the bulk producer plans the set; records its
  label (e.g. *Compress inside ~/Movies/Trip_2026*), the total file count, and `startedAt`.
* **`recordBatchResult(batchId, path, result)`** — called by `runTask()` as each `compress` task returns. A
  `compressed` / `skipped` result counts as **OK** (advances `done`); a `blocked` / `failed` result is
  appended to the batch's **error list** (`errors[]` as `{ path, reason }`) and counts as `failed`. A task
  that *throws* is caught (§3) and still recorded, so the batch always finishes.
* **`listBatches()`** — the active + recently-finished batches, folded into `GET /api/progress` as `batches`
  ([processing.mdx §5](./processing.mdx)). **Finished batches linger for a 30-min retention window** (then
  are pruned) so the user can still read the error list after the run ended.

The batch registry is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `createBatch()` / `recordBatchResult()` / `listBatches()` (the 30-min prune of finished batches). The producer that opens a batch is `compression.service.ts` — `enqueueCompressInside()` ([compress_inside.mdx §5](./compress_inside.mdx)); the surfaces that render batch progress + the error list are the [Processing page](./processing.mdx) (processing.mdx §4).

## 5. Persistence & crash posture (LOCKED)

The queue is **in-memory only**, exactly like the job registry ([webapp.mdx §12](./webapp.mdx)):

* The process **is** the single API server; a task that outlives a restart was **not** "in flight" anymore,
  so re-running it on boot would be wrong (it might duplicate work the user never re-requested).
* On restart the queue is simply **empty**. Nothing was lost that mattered: any output already written to
  disk stays; the user can re-invoke the page action, and **skip-already-done** ([page_actions.mdx §1.2](./page_actions.mdx))
  means only the genuinely-unfinished files re-queue. That idempotency is what makes no-persistence safe.
* There is **no** cross-process or on-disk queue file, and the queue never networks (it only calls the
  same local runners the per-file paths call — transcribe is fully local; describe's one deliberate upload
  is the runner's concern, [ai_description.mdx §7](./ai_description.mdx), not the queue's).

## 6. Acceptance criteria

1. `enqueue(tasks)` **appends** the tasks, **kicks** the drain, and **returns immediately** with
   `{ queued, deduped }`; the HTTP handler that called it also returns immediately (no waiting for the
   work).
2. **Dedup:** the same `op + path` already **pending or running** is not queued again; a double-click
   reports the second batch as `deduped`.
3. A pool drains the queue with **core-budgeted, media-aware** caps ([parallelization.mdx](./parallelization.mdx)):
   the mass-compute budget is `round(cores × max_core_fraction)` (default **0.9**), `compress:image` fans
   **wide** (`budget`, 1 thread/job), `compress:video` fans **narrow** (`floor(budget / videoThreads)`, threads
   capped), transcribe narrow, describe network-fixed. `concurrentJobs × threadsPerJob ≈ budget`, never
   `cores²`; a one-off single-file compress passes no cap. Extra tasks wait and start as slots free.
4. Each task runs by calling **`transcribeOne` / `describeOne` / `compressFile`** (the last wrapped in
   `track("compress", …)`, its heavy transcode **async** so it never blocks the event loop), so a started
   item shows a **live dock card** via the existing registry — including in a tab that did not start it; the
   queue adds no dock code.
5. A task that fails is **caught, logged, and skipped** — one bad file never stalls the queue.
6. `GET /api/progress` returns a **`queued`** count of not-yet-started tasks; the dock shows a
   **"+ N queued"** footer that drains to nothing. No fabricated cards for pending items.
7. The queue is **in-memory only** (no persistence); on restart it is empty, and re-invoking a page action
   re-queues only the still-unfinished files (skip-already-done makes this safe).
8. **Batches (§4a):** a bulk **"Compress inside"** run opens a batch (`createBatch`) whose per-file
   `compress` tasks carry its `batchId`; `recordBatchResult` counts `compressed`/`skipped` as OK and records
   `blocked`/`failed` in the batch's `errors[]`; `listBatches()` (30-min retention for finished batches) is
   folded into `GET /api/progress` for the [Processing page](./processing.mdx) progress bar + error list.
