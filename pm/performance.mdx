# Performance

Chrome reports **severe** performance problems in the web app — jank, long tasks,
and outright tab hangs. This spec is the single source of truth for **why** and
**how we fix it**.

It has three stages, matching the work order:

1. **Diagnosis** — the full catalogued list of performance problems (below, §P-01…).
2. **Architecture fixes** — the design-level fix for each problem (the `Fix:` line
   under each entry).
3. **Remediation** — the concrete code changes, tracked in the **Status** column
   and summarized in §"Remediation log".

Severity legend: 🔴 critical (causes hangs/long tasks) · 🟠 high (visible jank) ·
🟡 medium (wasteful, scales badly) · ⚪ low (hygiene).

---

## The root-cause themes

Almost every problem below is an instance of one of four themes. Fix the theme and
many symptoms disappear at once.

- **T1 — We render whole datasets to the DOM.** Tables materialize every row of a
  page (up to an **"All" = `Number.MAX_SAFE_INTEGER`** option, over a `5000`-row
  backend cap) as real DOM nodes, each with nested buttons, badges, a kebab menu,
  and per-cell `Date` parsing. Thousands of `<tr>` × ~10 cells is tens of thousands
  of nodes → giant layout/paint/GC, and the tab freezes.
- **T2 — Every keystroke/click re-renders everything.** Search, filter, and each
  checkbox toggle live in state that the whole table body depends on, with no
  memoized rows and no debounce, so one keypress re-runs filtering over the whole
  dataset **and** re-renders every visible row.
- **T3 — The backend blocks the Node event loop.** The flat large-file walk runs a
  tight synchronous `readdirSync`/`statSync` loop (up to `300_000` iterations)
  *inside the HTTP request*, starving every other request — polls, page loads, and
  the scan-status endpoint all hang while it runs.
- **T4 — We over-fetch and over-invalidate.** A global 5s poll runs on every page
  forever; single actions invalidate broad query keys that trigger full-filesystem
  re-walks; the sidebar subscribes to the entire IPFS payload to read one field.
- **T5 — We budget cores and ignore bytes.** Every admission decision asks "is a core
  free?" and none asks "does this fit in the heap?" The `describe` bucket admits **24**
  jobs by count on the theory that a network-bound job has no hardware bottleneck —
  true for CPU, **false for memory**: each in-flight upload pins its full payload
  (Buffer + base64 + JSON body) for the whole round-trip. 24 × ~66–90MB ≈ **1.6–2.2GB**,
  which is what reached **4.1GB** and aborted V8 on 2026-07-15. Concurrency has **two**
  budgets; we built one. Full treatment: **[memory.mdx](./memory.mdx)**.
- **T6 — Work fails silently, and we cannot see it after the fact.** `describeOne` and
  `transcribeOne` logged only **terminal outcomes** — no START line — so a job that died
  mid-flight left **no trace it had ever begun**. `adapters.ts`'s only log statement was a
  retry warning, so a provider call that *succeeded* was invisible. The progress registry
  **deletes** each record on completion, so at post-mortem the in-flight map is empty. The
  queue was **in-memory only**, so a crash vaporized a 1,440-file backlog in silence, and
  an empty queue rendered identically for "finished" and "annihilated". A performance
  problem you cannot observe is not a performance problem, it is a mystery. Full
  treatment: **[transactions_log.mdx](./transactions_log.mdx)**,
  **[crash_recovery.mdx](./crash_recovery.mdx)**, **[debugging.mdx](./debugging.mdx)**.

### The 2026-07-15 incident — the event that added T5 and T6

The user queued **1,440 AI descriptions** and walked away. At **22:13 PDT** the backend
died: `FATAL ERROR: Ineffective mark-compacts near heap limit`, V8 heap at **4.1GB**. The
in-memory queue vaporized the backlog. `tsx watch` restarts on **file change only, never on
crash**, so the backend stayed dead **~6 hours** while Vite on `:2222` kept serving pages —
**frontend up ≠ app up**. ~146–149 descriptions completed, all before 19:50; zero after. The
only evidence was in `launcher.log`, because a V8 abort **cannot run our exit handlers**.

Three of the four themes were already catalogued and fixed *for the UI*; the same diseases
were alive in the **background pipeline**, where nobody was watching. That asymmetry is the
lesson, and it is why the catalog now extends past the browser tab.

### Second pass (P-15…P-21) — the theme leaked past the tables we first hardened

The first remediation (P-01…P-14) hardened the two *TanStack* tables and the
`/fs/flat` walk. But the **same four themes recur on surfaces the first pass never
catalogued** — most importantly the **File System column browser**, which is the
*default landing view* of the File System page (it opens on the home directory).
It renders every directory entry as DOM (T1) off a backend endpoint that is
synchronous and uncapped (T3). That endpoint does *more* per-entry filesystem work
than the flat walk (a recursive "contains a repo below" probe, an IPFS-publish
probe, and a `hasChildren` `readdirSync`), so on a large home directory it is a
prime "Page Unresponsive" cause. The rule going forward: **every list that renders
rows must be windowed, and every filesystem-walking endpoint must be async +
capped + cooperatively yielding — no exceptions, on any page.**

## Diagnosis + architecture fixes

### 🔴 P-01 — Tables render every row of the page as DOM (no virtualization) · T1
- **Where:** `components/table/DataTable.tsx` (`table.getRowModel().rows.map(...)`),
  `pages/fs/FullPathsPage.tsx` (`pageRows.map(...)`).
- **Symptom:** With the default page size of **500** — and worse, the **"All"**
  option (`Number.MAX_SAFE_INTEGER`) over a backend cap of **5000** files — the
  browser builds 500–5000 `<tr>`, each holding 2 nav buttons, a checkbox,
  `formatBytes`, `relativeTime` + `absoluteTime` (Date parsing), a `<Badges>` chip
  group, and an `<EntityKebab>` (a button with handlers). That is 10k–50k DOM nodes
  and React elements built in one synchronous commit → multi-second hang.
- **Fix (architecture):** **Row windowing.** Render only the rows visible in the
  scroll viewport plus a small overscan, using a fixed row height and top/bottom
  spacer rows (`<tr>` with a single height-only `<td>`), inside a scroll container.
  No new dependency required — a small `useWindowedRows(count, rowHeight)` hook keyed
  off the scroll container's `scrollTop`. Also **delete the `Number.MAX_SAFE_INTEGER`
  "All" page-size footgun**; cap the page-size menu at sane values.
- **Height (webapp full-page-height rule).** The windowed body's scroll container
  is sized `flex-1 min-h-0` so it **fills the page down to the bottom of the
  viewport** (no fixed `max-h` leaving dead white space when the window is tall),
  matching P-14 and the [charter Tables rule](../CLAUDE.md). Windowing keys off this
  same container's live `clientHeight`, so a taller window simply renders a few more
  rows — never all 500/5000. The one exception is a table with content **beneath**
  it ([repos.mdx §3.3.1](./repos.mdx)), which stays a **bounded** height; windowing
  works identically in either case.

Row virtualization/windowing for large tables is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/useWindowedRows.ts` — `useWindowedRows()` hook

### 🔴 P-02 — Row selection copies the whole Set and re-renders the whole table · T2
- **Where:** `FullPathsPage.tsx` (`selected: Set<string>`, per-checkbox
  `new Set(selected)`), `DataTable.tsx` (same pattern in the `__select` column),
  `OneRepoPage.tsx`, `IpfsPage.tsx`.
- **Symptom:** Every checkbox toggle allocates a new `Set` (O(n)) **and** replaces
  the `selected` reference the entire body reads, so toggling one row re-renders all
  500 rows. "Select all" builds a Set of up to 5000 paths in one click.
- **Fix (architecture):** Extract a **`React.memo` row component** that receives a
  boolean `selected` and a **stable** `onToggle(id)` callback (from a `useCallback`
  that reads/writes the Set via the functional updater). Only the toggled row
  re-renders. Selection stays a `Set` for O(1) membership, but rows no longer close
  over the whole Set.

### 🔴 P-03 — `DataTable` rebuilds its entire column + row model every render · T2
- **Where:** `DataTable.tsx` — `tanColumns` `useMemo(..., [columns, selection, data,
  getRowId])`.
- **Symptom:** Callers pass `selection={{ selected, onChange, bulk }}` and
  `columns=[...]` as **fresh object/array literals every render**, so the memo key
  changes every render; `tanColumns` is rebuilt, which makes TanStack recompute the
  core/filtered/sorted/paginated row models from scratch every render — on every
  keystroke. The header checkbox also runs `data.map(getRowId)` inline each render.
- **Fix (architecture):** Split selection into **primitive/stable deps**: memoize
  the select column on `[getRowId]` only and read live selection through a ref or a
  memoized cell that subscribes to a boolean. Depend the column memo on `columns`
  and `getRowId` (stable), not on the `selection` object. Compute "all on page
  selected" from `pageIds` once, not per header render.

### 🔴 P-04 — The flat large-file endpoint blocks the event loop · T3
- **Where:** `backend/modules/fs/fs.service.ts` → `listFilesFlat()` (sync
  `while (stack.length)` with `readdirSync`/`statSync`, budget `300_000`), served by
  the `/fs/flat` route.
- **Symptom:** Unlike `scanner.service.ts` (which `await`s `setImmediate` every 400
  entries), the flat walk is fully synchronous inside the request handler. On a big
  home directory it pins the single Node thread for seconds — **all** concurrent
  requests (scan-status poll, other pages, auth) stall behind it. Reads as "the
  whole app hangs."
- **Fix (architecture):** Make `listFilesFlat` **async and cooperatively yielding**
  (`await setImmediate` every N entries), exactly like `walkUnit`. The route handler
  `await`s it. Keeps the caps; frees the loop between chunks.

The buffered async flat large-file listing that backs `GET /api/fs/flat` is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listFilesFlat()`

### 🟠 P-05 — Client does full-dataset filter/sort/paginate on the main thread · T1/T2
- **Where:** `FullPathsPage.tsx` `filtered = useMemo(files.filter(...))` +
  TanStack sort/paginate; `DataTable.tsx` `getFilteredRowModel`/`getSortedRowModel`.
- **Symptom:** The server returns up to 5000 rows; every filter/search re-scans the
  full array on the main thread on each keystroke. Combined with P-01/P-02 the work
  compounds.
- **Fix (architecture):** (a) **Debounce** the search/filter inputs (~200ms) so we
  filter once per pause, not per keypress. (b) Keep filtering client-side (dataset is
  capped at 5000) but ensure it runs on **debounced** values and feeds a **windowed**
  body. (c) Longer term, push filtering to the server with a `limit`/`query` param so
  the payload shrinks — noted as a follow-up, not required for the hang fix.

The debounced search/filter input hook is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/useDebounced.ts` — `useDebounced()`

### 🟠 P-06 — Time formatting allocates `Date` objects per cell per render · T2
- **Where:** `lib/format.ts` `relativeTime`/`absoluteTime` (called in `changed`,
  `lastPin`, `seen`, etc. columns, plus `absoluteTime` in `title=` attributes).
- **Symptom:** 500 rows × 2 time columns × (`new Date` + `Date.now` +
  `toLocaleString`) on every render. `toLocaleString` is comparatively expensive.
- **Fix (architecture):** With windowing (P-01) only ~30 rows format at a time, which
  largely resolves this. Additionally, compute `absoluteTime` **lazily** (only when
  the tooltip is needed) rather than eagerly into every `title` — or accept it once
  windowing caps the row count. Memoize the row so re-renders don't re-format.

### 🟠 P-07 — Global 5s poll runs on every page forever, hitting a blocking endpoint · T4
- **Where:** `components/ScanProgressBar.tsx` (mounted in `AppShell`, `refetchInterval`
  1s running / **5s always otherwise**), `pages/repos/ReposPage.tsx` (a **second**
  independent `scanStatus` poll), `pages/scans/ScansPage.tsx` (`refetchInterval: 10_000`).
- **Symptom:** Even when idle, the app polls `/api/repos/scan-status` every 5s on
  every page, in every tab, including background tabs — and that endpoint shares the
  thread the blocking walk (P-04) can occupy. ReposPage doubles it.
- **Fix (architecture):** (a) `refetchIntervalInBackground: false` so hidden tabs
  don't poll. (b) Idle interval up to **15s**, fast **1s** only while `running`.
  (c) ReposPage reads the **shared** `["scanStatus"]` cache (same key) without adding
  a second competing interval — keep one poller (the always-mounted `ScanProgressBar`)
  as the source of truth; ReposPage observes without its own `refetchInterval`, or
  they share and we accept one timer. (d) Keep the endpoint itself cheap (it already
  returns module state) and non-blocking (P-04).

### 🟠 P-08 — Single actions invalidate broad keys → full-filesystem re-walks · T4
- **Where:** `FullPathsPage.tsx` pin mutation `invalidateQueries(["fsFlat"])` +
  `["fs"]` + `["repos"]`; `EntityMenu.tsx` `invalidate()` hits `["fs"]` (every open
  column) + `["repo"]`.
- **Symptom:** Pinning one file invalidates `["fsFlat"]`, forcing a **re-walk of the
  entire directory tree** (the P-04 endpoint) just to flip one badge. Every flag
  toggle refetches every File-System column.
- **Fix (architecture):** **Optimistically patch the cache.** After a decision/flag
  change, `setQueryData` to flip the affected row's badges in the existing
  `FlatFileListing`/listings instead of invalidating the expensive walk. Fall back to
  a single targeted refetch of the specific entity, not the whole tree.

### 🟡 P-09 — Sidebar subscribes to the entire IPFS payload for one field · T4
- **Where:** `components/layout/Sidebar.tsx` — `useQuery(["ipfs"])` then reads only
  `ipfsData.repos`.
- **Symptom:** The `["ipfs"]` payload includes **all pins** (potentially large). The
  sidebar re-renders whenever any part of that object changes, though it only needs
  `repos`.
- **Fix (architecture):** Use React Query's **`select`** to subscribe to
  `data.repos` only, so the sidebar re-renders only when the repo list changes.

### 🟡 P-10 — No route-level code splitting; whole app in one bundle · T1(load)
- **Where:** `router.tsx` eagerly imports every page (all tables, `MediaViewer`, all
  icon usage).
- **Symptom:** First paint downloads/parses every page's code, including the heavy
  media viewer and every table, before the first route renders.
- **Fix (architecture):** `React.lazy` each route `component` + a `Suspense`
  fallback, so each page's JS loads on navigation. Vite splits them into async chunks.

### 🟡 P-11 — `autoResetPageIndex` + unstable inputs reset/recompute tables · T2
- **Where:** `DataTable.tsx` / `FullPathsPage.tsx` (`autoResetPageIndex: true`).
- **Symptom:** Any data identity change (including from over-invalidation, P-08)
  resets to page 1 and recomputes the row model.
- **Fix (architecture):** Keep `autoReset` where it's semantically needed, but remove
  the churn source (P-08). Stable data identity (optimistic patch, not refetch) means
  fewer resets.

### 🟡 P-12 — `Badges` allocates and re-renders per row on every table render · T1/T2
- **Where:** `components/fs/Badges.tsx` — `badges.slice().reverse().map(...)` with no
  memo; rendered once per row.
- **Symptom:** Every table render rebuilds every badge group (array copy + reverse +
  map) for every visible row.
- **Fix (architecture):** `React.memo(Badges)` (props are a small array; memo on
  identity is fine since the backend returns a fresh array only when data changes).
  Windowing (P-01) caps how many render at once.

### 🟡 P-13 — MediaViewer loads full-resolution originals into the DOM · T1(media)
- **Where:** `pages/entity/MediaViewer.tsx` — `<img src={grant.url}>` at natural
  size; zoom uses `max-w-none`.
- **Symptom:** A 50–100 MB image/PNG is fetched and **decoded on the main thread** at
  full resolution, causing a long decode/paint hitch; video is fine (`preload=
  "metadata"` + Range).
- **Fix (architecture):** Add `loading="lazy"` + `decoding="async"` to the `<img>`
  so decode moves off the critical path; longer term, request a backend-generated
  **downscaled preview** for very large images and only load the original on explicit
  "view full resolution." (Preview endpoint is a follow-up; the async-decode attrs
  ship now.)

### ⚪ P-14 — Redundant `Home`/icon imports & dead nodes · hygiene
- **Where:** `DataTable.tsx` renders a hidden `<Check className="hidden" />` (dead
  node); misc.
- **Fix:** Remove dead render nodes. Minor.

## Second-pass diagnosis + architecture fixes (P-15…P-21)

### 🔴 P-15 — The File System column browser renders every directory entry as DOM · T1
- **Where:** `pages/fs/FileSystemPage.tsx` — `FsColumn` does
  `q.data?.entries.map((e) => <FsRow …>)` with **no windowing and no cap**. Each
  `FsRow` mounts a `<Badges>` chip group, an `<EntityKebab>` (a button with its own
  `useState` + a portal-capable menu), a `useNavigate()` hook, an icon, and a size
  string.
- **Symptom:** This is the **default landing view** of the File System page (it opens
  on the OS home directory). A directory with a few thousand children builds a few
  thousand rows — each with a kebab button and badge group — in one synchronous
  commit. Tens of thousands of DOM nodes → the exact multi-second hang P-01 fixed for
  the *tables*, but on the page users hit first. Worse, several columns can be open at
  once (Miller columns), each un-windowed.
- **Fix (architecture):** **Window the column body** with the existing
  `useWindowedRows(count, rowHeight, scrollRef)` hook — the column is already an
  `overflow-y-auto` scroll container; give it a bounded height (`flex-1 min-h-0`),
  a fixed row height, and top/bottom spacer rows, so an N-entry directory costs ~30
  `FsRow` not N. Surface the backend's new `truncated` flag (P-16) as a "narrowed —
  showing first N" note. Also **`React.memo(FsRow)`** (P-20) so opening a child column
  or selecting a file re-renders only the affected rows, not the whole column.

### 🔴 P-16 — The column-browser backend `listDirectory` blocks the event loop, uncapped · T3
- **Where:** `backend/modules/fs/fs.service.ts` → `listDirectory()` (synchronous
  `readdirSync` + per-file `statSync` + per-dir `dirHasChildren()` `readdirSync` +
  `computeBadges()` which itself may run `containsRepoBelow()` — a bounded but
  **recursive** `readdirSync` probe — plus `dirPublishesIpfs()` and
  `isGitWorkingTree()`), served synchronously by `GET /api/fs`.
- **Symptom:** This is P-04's disease on the endpoint the first pass never touched —
  and it does **more** per-entry work than the flat walk. Unlike `listFilesFlat`
  (capped at 5000 + `await setImmediate` every 400 entries) it has **no entry cap and
  never yields**, so a large directory pins the single Node thread for seconds inside
  the request handler. Every concurrent request (scan-status poll, auth, other
  columns) stalls behind it → "the whole app hangs." And the unbounded `entries[]`
  payload is what P-15 then renders.
- **Fix (architecture):** Make `listDirectory` **async and cooperatively yielding**
  (`await setImmediate` every ~200 entries), exactly like `listFilesFlat`/`walkUnit`.
  **Cap the entry count** (`FS_ENTRY_CAP`, e.g. 5000) and add `truncated: boolean` to
  `FsListing` (surfaced by P-15), so a pathological directory can't produce an
  unbounded payload or unbounded work. Replace `dirHasChildren`'s full `readdirSync`
  with a **single-entry `opendirSync` probe** (read one dirent, close) so the
  disclosure check is O(1) syscalls, not O(children). Router handler `await`s it.

The async, entry-capped one-directory listing with the O(1) `hasChildren` probe is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/fs.service.ts` — `listDirectory()` (and `dirHasChildren()`)

### 🟠 P-17 — Entity flag/decision changes invalidate `["fs"]`/`["repo"]` → re-walk every open column · T4
- **Where:** `components/menu/EntityMenu.tsx` `invalidate()`
  (`invalidateQueries(["fs"])` + `["repo"]`), and `pages/entity/MediaViewer.tsx`
  `decide.onSuccess` (same two invalidations). Extends P-08, which only patched
  `FullPathsPage`'s pin path.
- **Symptom:** Toggling one sticky flag or decision from the ⋮ menu invalidates the
  `["fs", …]` key, which refetches **every open File System column** — each a full
  `listDirectory` walk (now the P-16 endpoint) — plus every `["repo", …]` view, just
  to flip one badge. The mutation already returns the fresh `EntityView`.
- **Fix (architecture):** **Optimistically patch the cache.** The mutation response
  carries the entity's new badges; use `qc.setQueriesData({ queryKey: ["fs"] }, …)`
  (and `["fsFlat"]`) to splice the updated badges onto the one matching row across all
  cached listings, instead of invalidating the expensive walks. `setQueryData(["entity", path])`
  already updates the open entity view. The scheduled scan remains the source of truth
  and reconciles later. No re-walk on a single-badge change.

### 🟡 P-18 — Windowing writes React state on every scroll event (no rAF coalescing) · T2
- **Where:** `components/table/useWindowedRows.ts` — the `scroll` listener calls
  `setScrollTop(el.scrollTop)` on **every** scroll event.
- **Symptom:** During a fast scroll the browser fires many scroll events per frame;
  each `setState` re-renders the whole table component and recomputes the visible
  slice. It's bounded (~30 rows) so it doesn't hang, but it wastes main-thread time
  and can drop frames on long lists.
- **Fix (architecture):** **Coalesce scroll updates with `requestAnimationFrame`** —
  at most one window recompute per frame — and **skip the state update when the derived
  `start` index hasn't changed** (most scroll deltas stay within the same row), so we
  only re-render when the visible window actually moves.

The rAF-coalesced scroll handler that skips re-renders when the window start is unchanged is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/table/useWindowedRows.ts` — `useWindowedRows()` scroll listener

### 🟡 P-19 — Two hand-rolled windowed tables duplicate logic and drift · architecture
- **Where:** `components/table/DataTable.tsx` and `pages/fs/FullPathsPage.tsx` each
  implement their own windowed `<tbody>` (spacer rows, `ROW_H`, `slice(win.start,
  win.end)`), and after P-15 the column browser is a *third* copy of the pattern.
- **Symptom:** Not a runtime cost, but a **regression surface**: a windowing fix
  applied to one table silently misses the others (this is exactly how the column
  browser was left un-windowed in the first place).
- **Fix (architecture):** Extract a shared windowed-body primitive (a
  `useWindowedRows` consumer that renders spacer rows given a row renderer), or a
  `<WindowedRows>` component, so all three surfaces share one implementation.
  **Follow-up, not blocking** — the three copies are individually correct once
  P-15/P-18 land; unify them next to prevent future drift.

### ⚪ P-21 — MediaViewer `<img>` still eager-loads (missing `loading="lazy"`) · T1(media)
- **Where:** `pages/entity/MediaViewer.tsx` `ViewerSurface` — the `<img>` has
  `decoding="async"` (P-13) but **not** `loading="lazy"`, which P-13 also specified.
- **Symptom:** Minor — the viewer image is the focal element, so lazy-loading buys
  little here, but the item is incomplete.
- **Fix:** Add `loading="lazy"` to complete P-13. (The real large-image win is the
  backend downscaled-preview endpoint, still a follow-up.)

### 🟠 P-23 — The pin pass pinned units one-at-a-time (sequential, ignoring 20–30 cores) · throughput
- **Where:** `code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll()` was a
  `for (const folder of listRepoFolders()) { await pinRepoFolder(folder) }` serial loop.
- **Symptom:** A pin pass is a **full pass** over every known unit
  ([pin_process.mdx §2](./pin_process.mdx#2-every-pin-pass-is-a-full-pass-locked)); pinning
  100+ repos strictly one-after-another leaves a many-core machine almost idle and makes a manual **Pin
  now** background pass drag on far longer than it should.
- **Fix (done):** Fan the pass out on two levels, both capped at `cores − 2`: a `runPool()` across
  **units** and one shared `ipfsLimiter` across **all per-file** add/pin/fetch work (independent IPFS
  calls now run under `Promise.all`, not a `for…await` loop). The single limiter keeps total in-flight
  ops bounded no matter how units × files multiply; the store layer's per-file mutex + atomic writes
  ([storage_local.mdx §15](./storage_local.mdx#15-the-store-layer-how-code-touches-disk)) make concurrent pins
  safe. The pass now also covers the **computer unit** (`pin/computer/`), sharing the same generic
  `runUnitPin` core as repos. Priority + parallelism design is LOCKED in
  [pin_process.mdx §3/§4](./pin_process.mdx#4-parallelism--many-cores-bounded-concurrency-locked-design).
- **Follow-up:** Publish the computer unit's manifest over **IPNS** (storage_local.mdx §9.3) — the transport
  is not yet built, so its local manifest + pins are written but not yet advertised to peers.

The bounded-concurrency pin pass is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll` + `runPool` + `PIN_CONCURRENCY` (units) and `ipfsLimiter` + `runUnitPin` (per-file) + `pinComputerUnit`

**Now unified with P-26.** P-23 (pin pass → the **responsive** budget, `cores − 2`) and P-26 below (the
job queue → the **mass-compute** budget, `round(cores × 0.9)`) no longer each roll their own core count:
both draw from the **same** shared helper, and the two named budgets are defined in one place —
[parallelization.mdx §1](./parallelization.mdx).

### 🟠 P-26 — The background job queue drained at a fixed tiny cap (2/3/2), ignoring many cores · throughput
- **Where:** `code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `pump()` drained each op

### 🔴 P-27 — AI description's compress-to-fit ran synchronously (`spawnSync`), freezing the server under a mass run · T3
- **Where:** `code/packages/backend/src/modules/describe/fit-media.ts` ran **every** ffmpeg / ffprobe /

- **Fix (done):** `fit-media.ts` now runs all heavy child processes through an **async** runner
  (`runAsync` = `child_process.spawn` with a hard-kill timeout, capturing stdout for the ffprobe JSON) — the
  exact mirror of the compression engine's `runAsync` ([P-26](#-p-26--the-background-job-queue-drained-at-a-fixed-tiny-cap-232-ignoring-many-cores--throughput)
  / compression.mdx). `fitMediaUnderLimit` / `fitVideoUnderLimit` / `fitImageUnderLimit` / `probeVideo` /
  `encodeCrf` / `encodeTwoPass` are **all `async`** and awaited by `describeOne()`; temp names use
  `crypto.randomUUID()` (no child process); and `readBase64Capped()` reads via **`fs.promises.readFile`**.
  Only the near-instant `which` tool-detection probe stays sync (a few ms — the same choice compression's
  `onPath()` makes). With this, 24 describe jobs run in parallel while `GET /api/progress` keeps answering
  and the Processing page loads instantly — the invariant is LOCKED in
  [ai_description.mdx §3.3.1](./ai_description.mdx) and [processing.mdx §4.4](./processing.mdx).

The async compress-to-fit + async base64 read are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/fit-media.ts` — `runAsync()` + the `async` fit/probe/encode functions; `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/describe/adapters.ts` — `readBase64Capped()` (`await fs.promises.readFile`), awaited in `gemini.describe()` / `chatVisionAdapter.describe()`.

- **What this fix did *not* do.** The `fs.promises.readFile` above fixed **T3** — the read stopped
  blocking the event loop — and did **nothing** for **T5**: it still materializes the entire
  up-to-18MB buffer in the heap. [memory.mdx P-29](./memory.mdx) catalogues that same call as copy 1
  of the 3–4× residency that OOMed us on 2026-07-15. Both entries are right, and P-27 is **superseded
  by streaming, not reverted** — the async form stays; the follow-up (multipart / Gemini's resumable
  Files API, memory.mdx P-29) removes the buffer, not the `await`.

**The general lesson, and the reason it is stated here once:** *"make it async"* and *"make it small"*
are **different fixes for different problems** — async moves work off the event loop (T3), small stops
the heap filling (T5). Neither implies the other. We did the first, assumed we had done the second, and
wrote **LOCKED** next to it. Before locking any hot-path fix, ask **both** questions separately.

### 🔴 P-28 — The describe bucket admits by count, never by bytes (the OOM) · T5
- **Where:** `code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` —
  `DESCRIBE_CONCURRENCY` (24, env `LFB_DESCRIBE_CONCURRENCY`), returned **unclamped** by
  `capFor()`. It is the **only** bucket not derived from `coreBudget()`.
- **Symptom:** 24 in-flight uploads × ~66–90MB of pinned payload ≈ 1.6–2.2GB of live heap
  against a ~4GB default ceiling → `Ineffective mark-compacts`, the process aborts, the
  backlog is lost. The code comment justifying the wide fan-out ("network-bound, no hardware
  bottleneck") is true for cores and **false for memory**.
- **Fix (done):** Admission is now the **conjunction of both budgets** — a job must find a
  free slot **and** fit its `estimatedBytes()` reservation inside a live **memory budget**
  (`heap_size_limit × performance.max_memory_fraction`, default 0.5). Reserve at admission,
  release in a `finally`. A file bigger than the whole budget **runs alone** rather than
  deadlocking (memory.mdx §2.4 LOCKED), and per-bucket **head-of-line blocking** stops a
  stream of small files from starving a large one. Effective concurrency becomes
  `min(count cap, what fits in bytes)` — the count cap stops being the safety mechanism and
  becomes what it always should have been: an upper bound.

## Part III — The architecture transformation (streaming background filesystem index)

Everything above (P-01…P-21) is **client-side triage plus "make the walk yield."** It cut the
worst freezes, yet Chrome still reports **"Page Unresponsive"** on a large home directory. That is
the tell that the fixes so far treat *symptoms* — the **request/response shape is still wrong at the
root**, and no amount of windowing or `setImmediate` yielding can fix a wrong shape. This Part is the
**one architectural change** that does, and it is exactly what the work order calls for: *"load data
from the file system as a background service … stream data asynchronously … React tables read it in
asynchronously without blocking."*

### P-22 — The remaining root cause: walk-per-request → one giant blob (the model, not the loop) · 🔴 T3/T1/T4

Two endpoints back the two heaviest pages, and both share the same broken model:

- **Where:** `code/packages/backend/src/modules/fs/fs.service.ts` — `listFilesFlat()` (backs
  `GET /api/fs/flat`, the **Full Paths** page `code/packages/frontend/src/pages/fs/FullPathsPage.tsx`)
  and `listDirectory()` (backs `GET /api/fs`, the **File System** column browser
  `code/packages/frontend/src/pages/fs/FileSystemPage.tsx`).
- **Why yielding + windowing weren't enough:**
  1. **The whole subtree is walked *inside the HTTP request*, every time.** `await setImmediate`
     every 400 entries keeps the *event loop* breathing, but it does **not** reduce the total work,
     and it does **not** let the browser render anything until the walk **finishes**. On a big root,
     **time-to-first-row == time-to-last-row** — seconds of blank skeleton, then one enormous commit.
  2. **`computeBadges()` (`code/packages/backend/src/modules/fs/badges.ts`) runs heavy per-entry
     filesystem probes.** Per directory row: `isGitWorkingTree()`, `containsRepoBelow()` (a *recursive*
     `readdirSync` probe, depth 5, budget 4000), `dirPublishesIpfs()` (`readdirSync` + `existsSync`),
     and `dirHasChildren()`. Per markdown/manifest file: `fs.readFileSync` up to **512 KB** + regex.
     For an N-entry directory this is O(N × several syscalls), all on the critical path before a
     single byte reaches the client.
  3. **The response is one blob of up to `FLAT_FILE_CAP = 5000` rows** (`FsEntry`, ~8 fields + a
     badges array each). The browser must `JSON.parse` all of it in one synchronous task, hold it all
     in memory, and TanStack must build core/filtered/sorted/paginated **row models over all 5000** —
     then re-run filter/sort over all 5000 on **every keystroke** (windowing only caps *DOM nodes*,
     not the JS working set).
  4. **Nothing is persisted or cached.** Every navigation, every `["fsFlat"]`/`["fs"]` invalidation
     (P-08/P-17 patched *some* of these) re-walks from scratch.
  5. **Two walk systems that never share.** The **background scanner**
     (`code/packages/backend/src/modules/scanner/scanner.service.ts` +
     `scan-job.ts`) *already* walks the tree, detached and single-flight, and **persists** results to
     `status.yaml`. The FS pages ignore it and re-walk live in the request. We built the right pattern
     once (`scan-job.ts`) and then bypassed it for the two pages that hang.
- **Fix (architecture): a Filesystem Index Service that walks in the background, caches, and
  streams.** See the target design below. `listFilesFlat`/`listDirectory` stop being
  request-synchronous producers of a blob and become **thin readers over a streaming index**.

The background streaming filesystem walk that replaces the walk-per-request blob is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fsindex/fsindex.service.ts` — `walkFilesFlatStreaming()`

### The target architecture

```
                    ┌─────────────────────── backend (Node) ────────────────────────┐
  browser           │                                                                │
 ┌────────┐  GET /api/fs/flat/stream   ┌───────────────────────────────────────────┐ │
 │ Full   │ ───────────────────────▶   │  Filesystem Index Service                  │ │
 │ Paths  │   (NDJSON, chunked)        │  code/packages/backend/src/modules/fsindex │ │
 │ table  │ ◀───────────────────────   │                                            │ │
 └────────┘  batch,batch,…,done        │  • in-memory index: Map<key, CachedListing>│ │
     ▲                                 │  • mtime-keyed, stale-while-revalidate      │ │
     │ useSyncExternalStore            │  • single-flight per (root,hidden)          │ │
     │ (throttled rAF flush)           │  • emits batches as the walk discovers rows │ │
 ┌────────┐                            │        │                                    │ │
 │ store  │  ← accumulates batches     │        ▼  (CPU-bound walk + badges)         │ │
 │ (ref)  │                            │   ┌───────────────────────┐                 │ │
 └────────┘                            │   │  walker (worker thread)│  never janks    │ │
                                       │   │  reuses walkUnit/badges│  the HTTP loop  │ │
 ┌──────────────┐   filter/sort in     │   └───────────────────────┘                 │ │
 │ filter.worker│ ◀ a Web Worker,      │                    ▲                        │ │
 │  (frontend)  │   off the main thread │        shares HARD_SKIP / badge logic       │ │
 └──────────────┘                      │        with scanner.service (one source)    │ │
                                       └───────────────────────────────────────────┘ │
                    └────────────────────────────────────────────────────────────────┘
```

Five properties, each mapping to an aspect of "performance in all ways":

1. **Background, off the request.** The walk runs in a detached job (the `scan-job.ts` pattern) and,
   for the CPU-bound portion, a **worker thread** — so the HTTP event loop that serves `/health`,
   `/auth/me`, the scan-status poll, and the *stream itself* is never starved. (This finally makes
   P-04/P-16's "yield every N" unnecessary as the *only* defense.)
2. **Streaming, not blob.** Rows are delivered in **batches over a chunked NDJSON response** as the
   walk finds them. **Time-to-first-row drops to tens of milliseconds**; the table fills
   progressively; the browser never faces one 5000-row parse+commit.
3. **Cached + stale-while-revalidate.** Listings are memoized in-process keyed by
   `(absPath, hidden, dirMtime)`. Re-opening a column or re-visiting Full Paths serves **instantly
   from the index**; a background refresh streams only if the directory's mtime moved.
4. **Two-phase rows (cheap now, expensive later).** A batch carries the **cheap metadata**
   (`name/size/mtime/kind/hasChildren`) that's available from `readdir`+`stat` immediately; the
   **expensive badges** are computed in a second pass and streamed as **badge-patch events** keyed by
   path, so rows appear at once and badges "fill in." Filters that depend on badges simply activate as
   the badge data arrives.
5. **One source of truth.** The indexer reuses `HARD_SKIP`
   (`code/packages/backend/src/shared/scan-filters.ts`) and the badge logic
   (`code/packages/backend/src/modules/fs/badges.ts`) — and, longer term, reads the scanner's
   persisted `status.yaml` so the FS pages and the Repos/Scans pages agree and never double-walk.

### Aspect 1 — Backend: the index service and the streaming walk

- **New module `code/packages/backend/src/modules/fsindex/fsindex.service.ts`.**
  - `walkFilesFlatStreaming(root, hidden, onBatch, signal): Promise<FlatWalkSummary>` — the existing

### Aspect 6a — Event-driven freshness is NOT polling (LOCKED)

"No background polling" says how a page must **not** stay fresh; it never said how a page **does** stay
fresh. The gap showed up as a real defect: a One-Repo page open while a git-backbone pull reconciled a
peer's manifest kept rendering the old table, because `["repo", repoId]` had no reason to refetch. The
user's only recovery was ⌘R.

The wrong fix is obvious and cheap: put an interval back. **That is P-07 returning under a new name** —
a timer that asks "did anything change?" on every open tab forever and is wrong nearly every time it asks.

The right fix is to **invert the question**. The backend keeps an in-process **revision bus**: every
server-side write that changes what a page shows bumps a monotonic counter for a **topic**
(`repo:<folder>`, `repos`, `storages`). The browser holds **one** authenticated NDJSON stream —
`GET /api/events/stream`, `hello` + `bump {topic, revision}` + heartbeat — and a bump invalidates only
the react-query keys that read that topic.

| | Polling (P-07) | Event stream |
| --- | --- | --- |
| Work when nothing changes | N requests/minute × every open tab | **zero** |
| Requests per subscribed key | one timer each | one shared connection for all |
| Latency to a real change | up to one interval | ~immediate |
| Wrong-answer rate | N−1 of N asks | none — it only speaks when something moved |

So this **strengthens** the locked rule rather than carving an exception out of it:

- **`refetchIntervalInBackground: false` stands, and no `refetchInterval` is added anywhere.** An idle
  page now does **strictly less** work than it does today, not more.
- **A bump is a doorbell, not a delivery truck** — it carries `{topic, revision}` and never rows, so
  there is still exactly one code path that produces a row (the page's own read, P-08/P-17's optimistic
  patching unaffected).
- **Invalidate narrowly** — a `repo:<folder>` bump touches that repo's key, never `["repos"]` or
  `["fs"]`. Broad invalidation is T4 itself.
- **NDJSON over an authed `fetch`, not `EventSource`** — for the exact reason given in Aspect 2:
  `EventSource` cannot set the `Authorization` header, so SSE would bypass the allow-list gate. This
  reuses the **same** reader the Full Paths stream already uses — one streaming transport, used twice.
- **Degrade to today, never to a timer.** The stream reconnects with backoff; if it cannot be
  established the page falls back to *navigation/manual reload* — correctness kept, latency lost. A
  fallback poll would run precisely on the machines where the stream is flakiest, which is how P-07
  would quietly ship again.

The product rule this serves is LOCKED in
[storage_company.mdx §8.9](./storage_company.mdx#89-the-page-must-learn-not-be-reloaded-locked), with the
per-page behavior in [one_repo.mdx §4.11](./one_repo.mdx).

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/events/events.router.ts` — the `GET /api/events/stream` handler.

### Aspect 6b — Sitewide coverage: every page paints instantly and stays live (LOCKED)

Aspect 6a built the mechanism for one page (One Repo) and one family of topics (`repo:*`, `repos`,
`storages`). This aspect makes the two behaviors it enables the **sitewide rule** — product direction:
*every* UI page loads asynchronously from the server, and *every* open page updates itself when the
server's state changes (the canonical case: a git-backbone sync of the company or personal storage repo
lands new files while a page sits open).

**Rule 1 — Shell-first paint (never block first render on server data).** A page renders its shell —
header, back link, title, controls — **immediately**, from local code alone; the content region shows a
skeleton until its query lands. A page must never early-return a bare "Loading…" (or `null`) for the
whole page: a server that takes minutes to answer must cost the user *content latency*, never *page
latency*. Auth/security gating in `main.tsx` is the one allowed whole-tree gate; **data** may gate only
the region that shows it. Tables keep their existing loading-table/skeleton pattern; detail/settings
pages use the shared skeleton.

### Aspect 7 — Scaling / server mode

- **Cursor/limit on the stream** — the backend already caps at 5000; expose a `?limit`/`?after`
  cursor so a hosted deployment can page truly huge trees without ever buffering them.
- **Abort propagation** — a disconnected client stops the walk (above), so a hosted server can't
  accumulate orphaned walks under many tabs.
- **Shared index across requests** — the in-memory index is per-process today; in server mode it
  becomes the natural seam for a shared cache (Redis/SQLite) so N users browsing the same tree walk it
  **once**.
- **One walk, many readers** — unify with the scanner's persisted `status.yaml` so a scheduled scan
  *warms* the FS-page index instead of the FS page triggering a second, redundant walk.
