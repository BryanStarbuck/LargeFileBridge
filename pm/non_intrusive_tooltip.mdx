# LargeFileBridge — Non-intrusive Tooltips (the left-bar hover-info panel)

A floating tooltip has one flaw: it **covers the thing you are pointing at**, and often the rows around it.
LFBridge's lists are dense with **colored letter chips** (the code badges — [directories.mdx §3](./directories.mdx))
and **tinted folder glyphs** (the interesting-directory coloring — [file_system.mdx §1–§3](./file_system.mdx)),
and a user shouldn't have to memorize any of them. So instead of a pop-up bubble, we give those
explanations a **fixed home that never occludes anything**: a **hover-info panel** pinned to the **bottom
of the left bar**, just **above the account slot**.

The user's words: *"an area where we effectively have what we call non-intrusive tooltips … right above
where the login email address is … When we hover over certain things that are important, we display what
that is or what that means over there."*

Siblings this builds on and does not repeat:

* [left_bar.mdx](./left_bar.mdx) — the bar itself; it **reserves the white space** for this panel and
  **renders the panel** above the account slot (§4 there). This file owns what goes *in* the panel.
* [directories.mdx §3.5.1](./directories.mdx) — the per-badge name/description metadata (`BADGE_META`). Those
  same names/lines are what a hovered badge shows **here** instead of in a floating bubble.
* [file_system.mdx §1–§3](./file_system.mdx) — the interesting-folder color key (brown/blue/outline). A
  hovered tinted glyph explains its color here.
* [git_ignore.mdx](./git_ignore.mdx) — the new **`I` git-ignored** badge, one of the codes explained here.
* [Task tabs](./task_tabs.mdx) / [one_repo.mdx](./one_repo.mdx) — the per-repo screen's **metric tiles**,
  **File cell**, **icon control columns** and **Compress status icon** publish into **this same panel**
  (§6 here). They used to publish into a region **docked** beside the metrics strip; that region is
  **deleted** — it reserved ≥25% of the row width and forced the metric tiles to wrap.

## 1. Where the panel lives (LOCKED)

* The panel is a **fixed region at the very bottom of the left bar**, **directly above the account slot**
  (the avatar + signed-in name + Google email lines, [left_bar.mdx §4](./left_bar.mdx)). It is **snapped to
  the bottom**, between the scrolling nav list and the account slot.
* It occupies a **reserved block of always-white space** whose height is about **two nav-item rows** — the
  user's rule: *"think of the height as two left-bar entries … always that much white space under the
  left-bar entries, and make sure that that's white."*
* **At rest the panel is blank white.** Reserving the space (rather than growing on demand) means the
  account slot **never jumps** when a hover starts or ends — the layout is stable; only the panel's
  *contents* appear and disappear.
* The panel **never scrolls and never grows.** It shows **only what fits** in the reserved height; anything
  beyond is dropped (§3). It must never push the account slot down or overlap it.

The reserved white block + the panel mount sit between the nav list and the account slot in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/layout/Sidebar.tsx` — `Sidebar` (a fixed-height `<div>` inserted between the closing `</nav>` and the account-slot `div`), rendering `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoPanel.tsx` — `HoverInfoPanel`.

## 2. What can populate it — the hover sources (LOCKED)

Hovering (or keyboard-focusing) any of these **important** things publishes an explanation into the panel;
moving off clears it (a short debounce prevents flicker between adjacent chips):

* **A code badge chip** — R/r · S · C/c · i · **I** ([directories.mdx §3](./directories.mdx)). The panel
  shows that badge's **code-key block** (§3.1).
* **An interesting-folder glyph** — the brown/blue/outline tinted `<Folder>` / `<Cloud>`
  ([file_system.mdx §2](./file_system.mdx)). The panel shows the matching **color key** (big-file / video /
  image) as a code-key block.
* **A file or directory row** — hovering the row (not a specific chip) shows the **detail block** for that
  entity (§3.2) **followed by** a code-key block for **each** badge the entity carries. This is the "hover a
  directory or item, and if there's more information we might want to display it" case.
* **A View-One-Repo metric tile** — the metric tiles are deliberately terse (label + number only), so the
  tile's prose explanation lands here as a **text block** (§3.2.1) rather than crowding the strip
  ([one_repo.mdx §3.2](./one_repo.mdx), [task_tabs.mdx](./task_tabs.mdx)).
* **A View-One-Repo file row's File cell** — publishes a one-paragraph **summary of that file** as a text
  block.
* **The five icon control columns — Pin · Ignore · Transcribe · AI description · OCR** — hovering (or
  keyboard-focusing) **either the column's icon header OR any cell icon** publishes a plain-English
  explanation of that task here (§6).
* **The Compress status icon** — the Compress tab's own three-state content icon publishes the same
  file summary as the File cell.

> **This replaces the floating tooltip for these surfaces (LOCKED).** The old portal tooltip
> ([directories.mdx §3.5.1](./directories.mdx)) covered content; for badges and folder glyphs the
> explanation now lands in this panel instead. A native `title` / `aria-label` is still set on the chip for
> **accessibility** (screen readers, and surfaces with no left bar), but the visible, non-intrusive
> explanation is the panel. Tooltips elsewhere in the app that are **not** about code keys (e.g. a truncated
> path's full text) are unaffected.

Each hoverable element registers itself as a source via one shared hook, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoContext.tsx` — `useHoverInfoSource(payload)` (returns `{ onMouseEnter, onMouseLeave, onFocus, onBlur }` that set/clear the app-level hover state), consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/fs/Badges.tsx` — `Badges` (per chip) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/fs/FileSystemPage.tsx` — `FsRow` (the row + the folder glyph).

The One-repo sources use the **imperative** form of the same publish path — they are not one element but
dozens of tiles, cells and icons — in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/HoverInfoRegion.tsx` — `setHoverInfo(text)` / `setHoverInfo(null)` (a tiny module-level store) and `HoverInfoBridge` (renders nothing; mounted once by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/MetricsStrip.tsx` — `MetricsStrip`) which forwards the store into the provider via `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/useHoverInfoPublisher.ts` — `useHoverInfoPublisher()`.

## 3. The content model — info blocks, only what fits (LOCKED)

The panel renders an ordered list of **info blocks**, grouped into two labeled **sections**. It fills the
reserved height **top-to-bottom by priority** and **drops** any block that would overflow — it never scrolls
and never grows.

### 3.1 Code-key block (a rounded-rectangle letter + its meaning)

One block per code, so an entity with several codes **stacks one block per code**, one over the other
(the user: *"it might have more than one, so they'll stack one over the other to show each one of those
and explain what those are"*). Each block is a single line:

* the **exact chip** shown in the list — the rounded-rectangle colored square with its white letter
  (`R`/`P`/`C`/`i`/`I`, or a folder-color swatch), rendered from the **same** `BADGE_META` / folder-color
  tokens so it can never drift;
* the **name** — a 1–3 word phrase (bold): *Repo root*, *Pinned*, *Compress*, *Compressed*, *IPFS
  artifact*, **Git-ignored**, *Big files here*, *Video here* …;
* **one short line** of meaning when the name alone isn't enough.

These blocks are the **"what the codes mean"** section.

### 3.2 Detail block (up to three lines about the entity)

When the hovered thing is a **file or directory** (a row, not a bare chip), the panel also shows **one
detail block** with **at most three lines** of extra information — the user's rule: *"we might want to use
up to three lines there to give more information about that, but not more than three lines. This is outside
of those color keys."* Typical lines: the **name**, the **size** (and, for a directory, a short rollup like
"12 videos • 3 GB"), and the **modified date** or the **absolute path** (middle-truncated). This is the
**"about this file/directory"** section, and it renders **above** the code keys for a row hover.

### 3.2.1 Text block — one wrapped paragraph of free prose (LOCKED)

The third block kind. Where a **detail block** is a small set of short facts (each line **individually
truncated**), a **text block** is **one paragraph of free prose that WRAPS** across the panel's width. It
exists because the One-repo sources (§2 — metric tiles, File cell, the five icon control columns, the
Compress status icon) publish a **sentence**, not a list of fields, and truncating a sentence to one line
would destroy it.

* **Wrapped, not truncated.** The prose flows onto as many lines as it needs.
* **Line-clamped at 5 lines**, *on top of* the panel's fit-or-drop pass (§3.3). The clamp is what keeps a
  long sentence from ever reaching for more room; the fit-or-drop pass is the outer guarantee. Together
  they mean a text block **cannot grow the panel and cannot move the account slot** — the locked promise of
  §1 is unchanged by this new kind.
* It is published as a **single-block payload** (`{ blocks: [{ kind: "text", text }] }`), so it never
  competes with code-key or detail blocks for the reserved height.

The three block kinds and the text block's wrapped, clamped rendering are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoContext.tsx` — `HoverInfoBlock` (the `"code" | "detail" | "text"` union) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoPanel.tsx` — `Block` (the `kind === "text"` branch, 5-line clamp).

### 3.3 Only what fits (LOCKED)

* The reserved height (**~two nav rows**, `PANEL_H = 88`) bounds the panel. Blocks are laid out **in priority order** and the
  first block that would exceed the height — and every block after it — is **not rendered**. There is **no
  scrollbar** and the panel **does not expand**.
* **Priority order** (so the most useful survive truncation):
  1. On a **badge/glyph hover**: that badge's code-key block first, then any sibling codes on the same
     entity.
  2. On a **row hover**: the **detail block** first, then the entity's code-key blocks (most-significant
     code first — the badge order R → I, [directories.mdx §3.5](./directories.mdx)).
  3. On a **One-repo source hover** (metric tile, File cell, icon column, Compress icon): a **single text
     block** (§3.2.1) — nothing else competes, and the 5-line clamp bounds it before fit-or-drop needs to.
* Because the panel treats each block as an independent unit and shows **only what fits**, a busy entity
  with many codes may show just some of them — that is intended; the rest are simply not shown rather than
  scrolled or crammed.

The block list, the two sections, the priority order, and the fit-or-drop layout (measure the reserved height, render blocks until the next would overflow, never scroll/grow) are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoPanel.tsx` — `HoverInfoPanel` (builds the ordered `blocks[]` from the active `HoverInfo` payload; the fit-or-drop pass), with the payload shape in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoContext.tsx` — `HoverInfo` / `HoverInfoBlock` (`{ kind: "code", chip, name, line } | { kind: "detail", title, lines } | { kind: "text", text }`).

## 4. The app-level hover state (LOCKED shape)

A single app-level context holds **the one currently-hovered payload** (or none). Sources set it on enter,
clear it on leave; the panel reads it. One writer at a time — the most recent hover wins — so moving from a
chip to a row cleanly swaps the content.

* The provider wraps the app once (near the sidebar, so the panel and every list share it).
* `useHoverInfoSource(payload)` gives an element the enter/leave/focus/blur handlers; passing a **stable**
  payload avoids needless re-renders.
* A brief **clear debounce** (≈120 ms) keeps the panel from flickering blank while the pointer crosses the
  1-px gap between two adjacent chips.
* **There are deliberately TWO contexts — do not "simplify" them into one (LOCKED).** The main context's
  value **changes on every payload**, so an *imperative* publisher that consumed it would re-render every
  time it published — and re-publish — forever. The provider therefore also exposes a **setter-only
  context with a STABLE identity**: **publishers read the setter context; readers read the main one.**

The provider, the single-slot hover state, the setter-only context, and the clear debounce are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoContext.tsx` — `HoverInfoProvider` / `useHoverInfo()` (read) / `useHoverInfoSource()` (declarative write) / `HoverInfoSetterContext` (the stable setter-only context), mounted at the app root in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/main.tsx`, with the imperative reader in `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/useHoverInfoPublisher.ts` — `useHoverInfoPublisher()`.

## 5. Acceptance criteria

1. **Placement (§1):** a fixed hover-info panel sits at the **bottom of the left bar, directly above the
   account slot**, occupying a **reserved block of white space ~two nav-item rows tall** that is **blank
   white at rest**; the account slot never jumps when a hover begins or ends.
2. **Never intrusive (§1/§2):** the panel **never scrolls, never grows, never pushes or overlaps** the
   account slot; explanations that used to float over content (badge / folder-glyph tooltips) now appear
   **here** instead of a bubble, while a `title`/`aria-label` remains for accessibility.
3. **Hover sources (§2):** hovering (or focusing) a **code badge**, an **interesting-folder glyph**, a
   **file/directory row**, a **View-One-Repo metric tile**, a **File cell**, any of the **five icon control
   columns** (header icon or cell icon) or the **Compress status icon** populates the panel; leaving clears
   it (debounced against flicker).
4. **Code-key blocks (§3.1):** each shows the **exact rounded-rectangle letter chip** from the list plus its
   **name** and one short line; an entity with several codes **stacks one block per code**.
5. **Detail block (§3.2):** hovering a file/directory row shows **up to three lines** of extra info about it
   (name, size/rollup, modified/path), **outside** the color keys, above the code keys.
6. **Only what fits (§3.3):** blocks render in priority order and any that would overflow the reserved
   height are **dropped** (no scrollbar, no growth); a busy entity may show only some of its codes.
7. **Single source of truth:** the chips and names come from the **same** `BADGE_META` / folder-color tokens
   the list uses ([directories.mdx §3.5.1](./directories.mdx)), so the panel and the list can never drift.
8. **Text block (§3.2.1):** the One-repo sources publish **one wrapped paragraph** (not per-line truncated),
   **line-clamped at 5 lines** on top of fit-or-drop, so it can never grow the panel or move the account slot.
9. **No docked region (§6):** there is **no** hover-info region docked beside the View-One-Repo metrics
   strip. The metrics strip gets the **full page width** and the metric tiles **never wrap** onto a second
   row.
10. **Never blank on the One-repo page (§6):** with nothing hovered, the panel shows the **active task tab's
    default hint**; hovering a source swaps it, leaving restores the hint, and navigating away clears it.
11. **Two contexts (§4):** publishers read the **stable setter-only** context and readers read the main one —
    an imperative publisher must never subscribe to the payload it writes.

## 6. The One-repo page publishes here — there is no docked region (LOCKED)

**Only the LOCATION changed; the content model did not.** The View-One-Repo screen
([one_repo.mdx](./one_repo.mdx)) used to render this tooltip content in a **hover-info region DOCKED to the
right of the metrics strip**. **That region is deleted.** It reserved **≥25% of the row width**, which
forced the metric tiles to **wrap onto a second row** — unacceptable: **the tiles must never wrap.** The
prose now goes to the **one** place the app already reserves for non-intrusive hover text: **this panel**,
at the bottom of the left bar, directly above the **JFK Social / ACT 3 Filmmaking** external links and the
account slot ([left_bar.mdx §4.1](./left_bar.mdx)).

The keep-and-move summary:

* **Kept — the content model.** A **terse tile, its prose explanation elsewhere.** The metric tiles stay
  label + number only; the sentence that explains them still lives somewhere fixed the eye learns to glance
  at. Same restraint — plain, short, never interaction-blocking.
* **Kept — the default state.** When **nothing** is hovered the panel shows a short **hint for the active
  task tab** (e.g. *"Transcribe: audio and video files with no transcript yet. Hover a file for details."*),
  so on the One-repo page the panel is **not blank**.
* **Moved — the destination.** From a docked region that **took page width** to the left bar's **already
  reserved** fixed block, which takes **none**.

**The sources feeding this panel from the One-repo page** are the **metric tiles**, the **File cell**, the
**five icon control columns** (**Pin · Ignore · Transcribe · AI description · OCR**) and the **Compress
status icon** — all listed in §2, all publishing a **text block** (§3.2.1).

**The publish API at the call sites is unchanged.** Every existing caller keeps working: `setHoverInfo(text)`
on mouseenter, `setHoverInfo(null)` on leave, held in a **tiny module-level store** — no context needed at
the call site.

**TWO INPUTS, ONE OUTPUT (LOCKED).** The store takes two separate inputs: **`setHoverInfo(text)`** — what is
hovered **right now** (`null` clears it) — and **`setHoverDefault(hint)` / `useHoverDefault(hint)`** — what
the panel shows when **nothing** is hovered. A page registers its at-rest hint with `useHoverDefault` for as
long as it is mounted; on the One-repo page `MetricsStrip` does this with the **active task tab's**
`defaultHint`, and it clears on unmount so stale text never lingers after navigating away.

**The bridge is mounted ONCE, GLOBALLY, in the app shell — deliberately NOT per page (LOCKED).** A single
**render-nothing** component — **`HoverInfoBridge`** — forwards the store (the hovered text, or at rest the
active page's default hint) into the app-wide provider the panel reads.

* **Why it moved.** It used to be mounted **inside `MetricsStrip`**, which meant **only the One-repo page
  ever had a consumer**. Every other table that uses the shared icon control columns (§6.1 — the IPFS pins
  table, the Storage detail table, …) published hover text into a store **nobody was listening to**, so
  hovering those icons **explained nothing**. One global mount makes the panel work on **every** page —
  which is why §6.1's teaching rule can be stated site-wide.

Implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/HoverInfoRegion.tsx` — `setHoverInfo()` (the module-level store), `setHoverDefault()` / `useHoverDefault()` (the at-rest hint) and `HoverInfoBridge` (renders nothing; forwards the store + `defaultHint` into the provider), mounted **once** at `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/app/AppShell.tsx` — `AppShell`, using `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/useHoverInfoPublisher.ts` — `useHoverInfoPublisher()` and rendered by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/hoverinfo/HoverInfoPanel.tsx` — `HoverInfoPanel` / `Block`. The One-repo per-tab at-rest hint is registered by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/MetricsStrip.tsx` — `MetricsStrip` calling `useHoverDefault(defaultHint)`.
