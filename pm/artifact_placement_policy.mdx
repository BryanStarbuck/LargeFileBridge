# LargeFileBridge — Artifact & Tracking-State Placement Policy

This spec answers "where does LFB write?" with **two orthogonal rules**, applied in order:

1. **§0 — the storage-KIND rule.** Is `.lfbridge/` used **at all**? Only in a **working git repo**. A
   **dedicated LFB file repo** (Personal / Company / Community SDL) has **none** — its root *is* the
   `.lfbridge` area.
2. **§1+ — the data-CATEGORY rule.** Given a root, which of LFB's files may live in it at all? Content (A)
   may; noisy tracking state (B) never enters a **working** repo.

The two compose: §0 picks the **base directory** inside a root; §1 picks **which root**.

---

LargeFileBridge (LFB) writes two very different kinds of per-repo file, and the rule for **where each
goes** is a hard split by **data category** — not by any threshold or timing:

* **Category A — the user's derived CONTENT.** Transcripts (`<rel>.transcription`) and AI descriptions
  (`<rel>.ai_description`). Stable, content-derived, valuable; the user asked LFB to *produce* them. These
  live in the **working repo's committed `.lfbridge/`**, path-mirrored, and **travel with the repo** via
  its own git.
* **Category B — LFB's noisy tracking state.** `repo_storage.yaml` (which re-stamps `last_scan.at` and
  recomputes `counts:` on **every** scan), `files.yaml`, the per-file `<name>.yaml` sidecars,
  `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`, and compression records. This is LFB's private
  bookkeeping about **compression, big files, and git-ignore decisions**. It **NEVER** enters the working
  repo.

> **Why the split.** Category B re-stamps a per-scan timestamp and per-machine counts. When it was
> committed into the working repo's `.lfbridge/`, two computers scanning the same repo produced a fresh
> **merge conflict every scan** (`repo_storage.yaml` was the reported offender). The fix is categorical:
> the repo carries the *content* (A); LFB's *bookkeeping* (B) stays out of git entirely.

This spec builds on, and does not restate, its siblings:

* [storage_local.mdx §2.2b](./storage_local.mdx) — **Local Storage**, the state root, and the
  `repos/<repoKey>/` per-repo directory this policy writes Category B into.
* [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx) — the **Category-B files** this policy places
  (`repo_storage.yaml`, sidecars, history) and their locked shapes.
* [repo_settings.mdx §2](./repo_settings.mdx) — the per-repo **sync-repo toggle** that turns on the

* [storage_company.mdx §7](./storage_company.mdx) / [storage_personal.mdx §14](./storage_personal.mdx) —
  the **owning storage** that configures a **sync repo** ("a Company repo only for the LFB webapp to sync
  changes").
* [repo_owner_propagation.mdx](./repo_owner_propagation.mdx) — the **travel + consent** layer over the owner
  mapping: a repo→company assignment redirects tracking into the shared company sync repo, so the decision
  travels as an assertion and is applied on a teammate's machine only with their consent (§4).
* [placement_radios.mdx](./placement_radios.mdx) — the shared 3-way radio that places the **Category-A**

  Category-A artifacts**.

## 0. The storage-kind rule — WHEN `.lfbridge/` is used at all (LOCKED)

Orthogonal to the A/B category split below, and applying to **every** LFB-written file in **any** root, is a
second rule: **`.lfbridge/` exists only inside a working git repo.** A **dedicated LFB file repo** — the
Personal / Company / Community **SDL storage** — has **no `.lfbridge/` directory at all**.

| Storage kind | Example root | LFB's files land at | `.lfbridge/`? |
| --- | --- | --- | --- |
| **`repo`** — a user's working git repo (Git Storage) | `~/BGit/Bryan_git/charlie-kirk/` | `<root>/.lfbridge/<rel>` | **YES** |
| **`personal` / `company` / `community`** — a dedicated LFB file repo (SDL) | `~/BGit/Bryan_git/personal_large_files_bridge/` | `<root>/<rel>` | **NO — never** |

```
GIT STORAGE (a working repo — LFB is a guest, so it stays in one hidden corner)
  media:      ~/BGit/Bryan_git/charlie-kirk/videos/_QwGo0LyZ3I.mp4
  transcript: ~/BGit/Bryan_git/charlie-kirk/.lfbridge/videos/_QwGo0LyZ3I.mp4.transcription
                                             ^^^^^^^^^^

SDL FILE REPO (a repo that exists ONLY for LFB — the whole repo IS the .lfbridge area)
  media:      ~/_Mirror/Politics/Charlie_Kirk_Mi/Teachers_Lounge/Teacher_Lounge_Shooter.mp4
  transcript: ~/BGit/Bryan_git/personal_large_files_bridge/_Mirror/Politics/Charlie_Kirk_Mi/Teachers_Lounge/Teacher_Lounge_Shooter.mp4.transcription
                                                          ^ no .lfbridge/ segment — the mirror hangs off the root
```

**Why.** In a working repo LFB is a **guest**: the repo belongs to the user's own project, so everything LFB
writes is quarantined under one hidden `.lfbridge/` directory that is obviously not the user's content. An
SDL file repo is the **opposite** — it is a repo whose entire reason to exist is to hold LFB's files. Hiding
them under `.lfbridge/` inside a repo that contains nothing else buys no separation and only adds a
confusing, meaningless path segment to every file. **The SDL repo root *is* the `.lfbridge` area.**

### 0.1 What this means for the SDL repo's own metadata (LOCKED)

The rule is **total** — it is not limited to the path-mirrored artifacts. **Every** file LFB previously wrote
under an SDL's `.lfbridge/` now sits at the SDL **root**:

```
~/BGit/Bryan_git/personal_large_files_bridge/
├── storage.yaml                ← was .lfbridge/storage.yaml   (storages.mdx §3)
├── mapped_dirs.yaml            ← was .lfbridge/mapped_dirs.yaml
├── files.yaml                  ← was .lfbridge/files.yaml      (the fingerprint index)
├── manifest.yaml               ← was .lfbridge/manifest.yaml
├── bookmarks.yaml              ← was .lfbridge/bookmarks.yaml
├── devices/                    ← was .lfbridge/devices/        (storage_personal.mdx §4.1)
│   └── <device-id>.yaml
├── analysis/                   ← was .lfbridge/analysis/
│   └── <rel>/visuals_by_time.yaml
└── _Mirror/                    ← was .lfbridge/_Mirror/        (the path-mirrored artifacts)
    └── Politics/…/Teacher_Lounge_Shooter.mp4.transcription
```

### 0.2 The rule is keyed on storage KIND, never on a path or a name (LOCKED)

The decision is made by the **resolved storage type**, never by string-matching a root's name:

* `resolveStorageType(root)` returns `repo` | `personal` | `company` | `community`
  ([storage-type.service.ts](#code)). An explicit `type:` in the storage descriptor wins; else the
  `<name>_large_files_bridge` naming convention names an SDL; else the root is a plain working **`repo`**.
* **The default is `repo`** — i.e. an unclassified root **keeps** `.lfbridge/`. This is the conservative
  direction: mis-classifying a working repo as an SDL would scatter LFB's files across the user's project
  tree, which is unrecoverable-feeling; mis-classifying an SDL as a working repo merely leaves an extra path
  segment.
* An SDL root that **also** carries a `.git/` (every SDL does — it is a git repo) is still an SDL. The kind
  wins over the presence of `.git/`; the containing-root walk (§1.1's rule A) must classify by type **before**
  it falls back to "has `.git/` → working repo".

The single choke point that turns a root + storage kind into the base directory every LFB file hangs under is
in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `trackingBaseDir(root, type?)` (returns `<root>/.lfbridge` for kind `repo`, and `<root>` itself for an SDL) and its predicate `usesLfbridgeDir(type)`. **Every** LFB writer resolves its base through this function and none joins `LFBRIDGE_DIR` directly.

### 0.3 Non-default & error cases (LOCKED)

* **A mapped-dir key that collides with an SDL metadata name.** Because the SDL mirror now hangs off the root,
  a mapped dir keyed `devices`, `analysis`, `files.yaml`, `storage.yaml`, `manifest.yaml`, `bookmarks.yaml`, or
  `mapped_dirs.yaml` would shadow the SDL's own metadata. These names are **reserved** at the SDL root: the
  mapped-dir add/rename validator rejects a key that matches one ([storage_settings.mdx §4a.1](./storage_settings.mdx#4a1-reserved-mapped-dir-keys--the-validator-this-page-owns-locked)),
  with a message naming the reserved word. Existing mapped dirs are checked on load and surfaced as a storage
  warning rather than silently overwriting metadata.
* **A pre-existing `.lfbridge/` inside an SDL** (a repo written by an older LFB). It is **migrated** — the
  entries are `git mv`-ed to the root and the now-empty `.lfbridge/` removed — by the one-shot migration
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/migrate-sdl-lfbridge.ts` — `migrateSdlLfbridge()`. Until a given SDL is migrated, readers **fall back** to the old `.lfbridge/` path so no artifact is
  reported missing (and none is regenerated) mid-migration.
* **A name collision during migration** (a file already exists at the root destination, e.g. the SDL already
  had a top-level `_Mirror/` alongside `.lfbridge/_Mirror/`). The migration **merges** directories and, for a
  genuine file-vs-file conflict, keeps the **root** copy, leaves the `.lfbridge/` copy in place, and logs a
  WARN naming both paths — it never overwrites and never deletes the loser.
* **A working repo is later reclassified as an SDL** (the user sets `type:` or renames it to the convention).
  Its `.lfbridge/` is migrated to the root by the same one-shot migration on the next pass. The reverse
  (SDL → working repo) is not migrated automatically; LFB logs a WARN and keeps reading both layouts.

### 0.4 The SDL's own committed `manifest.yaml` sits at the SDL ROOT (LOCKED)

§0.1 lists `manifest.yaml` among the SDL metadata that moved to the root, but that file has a **second,
separate writer** — the *committed* manifest a storage's git backbone carries. That writer **used to
hardcode** the `.lfbridge/` segment; it now resolves through `trackingBaseDir()` like every other path.
Naming the rule explicitly here so the old hardcode reads as the **violation** it was rather than as an
alternative convention:

| Storage kind | The storage's committed `manifest.yaml` belongs at |
| --- | --- |
| **`repo`** — a working git repo | it does **not** have one. A working repo's manifest is **Category-B** tracking state in Local Storage (§2), and travels via the sync repo — never committed into the working tree ([storage_local.mdx §9.2](./storage_local.mdx#92-repo-manifests-travel-via-the-sync-repo-not-the-working-repo-locked)). |
| **`personal` / `company` / `community`** — an SDL | **`<root>/manifest.yaml`** — the root, exactly like `storage.yaml`, `files.yaml`, and `devices/`. **Never `<root>/.lfbridge/manifest.yaml`.** |

* **There is exactly one correct path per storage.** An SDL that has both `<root>/manifest.yaml` and
  `<root>/.lfbridge/manifest.yaml` is in the failure state [storage_company.mdx §8.7](./storage_company.mdx#87-four-smaller-defects-this-use-case-also-closes)
  reports from the field: **two divergent `files: []` manifests in the same company storage**, each read by a
  different caller, neither ever agreeing with the other. Two manifests in one storage is not a fallback —
  it is a **bug**.
* **The path must be resolved, not composed.** Every manifest path — committed or tracking — resolves
  through the §0.2 choke point `trackingBaseDir(root, type?)`. A writer that joins `.lfbridge` itself has
  bypassed the storage-kind rule and will be wrong for every SDL, because **kind, not path shape, decides**.
* **The read-fallback of §0.3 still applies during migration.** An un-migrated SDL that only has
  `.lfbridge/manifest.yaml` is still read from there, so no manifest is reported missing mid-migration —
  but nothing **new** is ever written to that path.

Resolving the committed manifest path for a storage through the storage-kind choke point (it formerly hardcoded `<root>/.lfbridge/manifest.yaml`) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/manifest.service.ts` — `committedManifestPath(repoPath)`, consumed by `readCommittedManifest()` / `writeCommittedManifest()` (and by the SDL pin pass in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinStorageUnitInner()`); the storage-kind base directory it must defer to is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `trackingBaseDir()`.

## 1. The two categories (LOCKED)

### 1.1 Category A — content that travels with the repo

The two **durable user artifacts**:

* a **transcript** — a `.transcription` file ([Transcribe.mdx §3](./Transcribe.mdx)); and
* an **AI description** — a `.ai_description` file ([ai_description.mdx §2](./ai_description.mdx))
  (the visuals-and-action-by-time YAML rides with the AI-description analysis family).

These are written INSIDE the working repo's committed `.lfbridge/`, path-mirrored, extension **appended**,
and **travel with the repo**. Their placement offers a per-repo choice — `lfbridge` (default), `beside`, or
the company `sync_repo` — via the [placement radios](./placement_radios.mdx). Writing them into the repo is
gated only by the machine-local **keep-`.lfbridge/`** consent ([storage_settings.mdx §3](./storage_settings.mdx),
default on).

#### 1.1.1 The committability guard — a gitignored artifact is a stranded artifact (LOCKED)

"Travels with the repo" is only true if the artifact is **committable**. Found live on charlie-kirk
(2026-07-20): the repo's `.gitignore` had grown `.lfbridge/`, `*.transcription`, and `*.ai_description`
rules (relics of the retired beside-media era), so every committer — the user's auto-commit, `git add -A` —
silently skipped the 59 transcripts + 158 AI descriptions the tower had produced. `git status` stayed clean
and the repo stayed 0-ahead, so nothing looked wrong, while the laptop kept reporting all of that work as
still to do. The rule: after every artifact write into a working repo's `.lfbridge/`, verify the path is not
gitignored; if it is, repair by **appending** re-include rules (`!.lfbridge/` + `!.lfbridge/**`) — additive
only, never deleting a user rule. `.lfbridge/` is LFB's own quarantine directory, so this repair is exempt
from the "never add a `.gitignore` entry automatically" nudging policy (which governs the user's media). A
rule outside the repo's own `.gitignore` (e.g. `core.excludesFile`) is never edited — it is reported as an
ERROR to the durable fault trail instead.

Verifies-and-repairs committability after every artifact write in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-committability.service.ts`, `ensureArtifactCommittable()` (hooked from `sync-trigger.service.ts` `noteArtifactWritten()`).

#### 1.1.2 Legacy-line healing, delivery, and convergence — committable is not delivered (LOCKED)

Committability (§1.1.1) makes the artifact *eligible* for git; three more steps make it *arrive* on the
user's other computers, and each was missing in the 2026-07-20 strand:

* **Healing** — the exact legacy nudge lines LFB's own deleted writers appended to working-repo
  `.gitignore`s (`.lfbridge/` in any anchored/bare form, `*.transcription`, `*.ai_description`) are
  REMOVED when found, not merely overridden. Scope is exact-trimmed-line match only; every user-authored
  rule is preserved byte-for-byte. This is the working-repo twin of the SDL backbone's
  `ensureSdlCommittable()` healing (git_backbone.mdx §4.2.1).
  Removes the poison lines in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/gitignore.service.ts`, `repairLegacyArtifactIgnores()` (constant `LEGACY_ARTIFACT_IGNORE_LINES`).
* **Delivery** — writing an artifact into `<repo>/.lfbridge/` schedules a **pathspec-scoped** commit and
  push of the quarantine dir only (`git add -- .lfbridge`; commit with explicit pathspec; never `add -A`;
  one fetch + `--ff-only` + re-push retry; never force). Relying on a foreign auto-committer to sweep the
  files is the stowaway defect (storage_personal.mdx §18.5.1) by another name.
  Delivers in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/repo-artifact-sync.service.ts`, `syncWorkingRepoArtifacts()` (scheduled by `sync-trigger.service.ts` `noteArtifactWritten()` under a `repo:<root>` target with the same debounce/backoff as SDL syncs).
* **Convergence** — the receiving computer pulls the repo itself: on Repos-list / One-repo page loads, a
  throttled (30 min) background `git fetch` + `merge --ff-only` runs for tracked repos where LFB
  artifacts are in play (`.lfbridge/` on disk or the durable-artifact latch). ff-only can never conflict;
  a diverged branch is logged and left to the user.
  Converges in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/repo-artifact-sync.service.ts`, `maybeConvergeWorkingRepo()` / `convergeWorkingRepoFromOrigin()` (fired from `repos.router.ts` GET `/` and GET `/:repoId`).

See also backbone_resilience.mdx §6.4 (the narrowed guest rule these three steps implement).

### 1.2 Category B — tracking state that never enters the repo

Everything LFB writes as a side effect of scanning and of the user's compression / pin / git-ignore

| File | Spec | Churn profile |

| `history/<device>.txt` | [repo_tracking_scheme.mdx §4](./repo_tracking_scheme.mdx#4-historydevicetxt--the-per-computer-audit-log-locked) | Low — append-only per-computer log. |

**None of Category B is ever placed in the working repo.** It goes to the two homes in §2.

## 2. Where Category B lives — Local Storage always, sync repo also (LOCKED)

For **every** Category-B file, the destination is resolved by two rules that both apply:

```
ALWAYS  →  ~/T/_large_files_bridge/repos/<repoKey>/          (Local Storage — the authoritative working copy)

ALSO    →  <syncRepo>/repos/<repoKey>/                        (when the owning company/Personal storage has a
           IFF the sync repo is configured AND the per-repo    sync repo configured and the per-repo toggle is
           "sync tracking state" toggle is ON (default OFF)    ON — the travelling MIRROR, written IN ADDITION)
```

* **Local Storage is written on every operation.** `repo_storage.yaml`, sidecars, and history are read
  and written here first; this is the source the app operates on. It is under the state root
  ([storage_local.mdx §1](./storage_local.mdx)), so it **never touches git** and **never merge-conflicts**.
* **The sync repo is an ADDITIVE mirror, not a replacement.** When enabled, the same files are also
  written/committed to `<syncRepo>/repos/<repoKey>/` so they travel to the user's other computers. Local
  Storage stays authoritative; a **sync pass** reconciles the two (§5). Turning the mirror off never loses
  the Local-Storage copy.
* **`repoKey` = `shortHash(resolve(repoPath))`** — the 12-hex `storageSid()` scheme. The same repo on the
  same machine always resolves to the same directory, across restarts and re-scans (collision guard §7.4).
* **There is no tier that writes Category B into the working repo.** This supersedes the earlier
  content-threshold three-tier model: crossing a "durable artifact" threshold no longer redirects tracking
  state into `<repoRoot>/.lfbridge/`. Category B is Local Storage (+ optional sync mirror), full stop.

Resolving the Local-Storage tracking root for a repo, and the optional company sync-repo mirror root, is
handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveTrackingRoot(repoRoot)` (returns the Local-Storage `repos/<repoKey>/` root) and `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null`), consumed by `repo-storage.service.ts`, `file-sidecar.service.ts`, and `history-log.service.ts`.

## 3. The state-root per-repo directory `repos/<repoKey>/` (LOCKED)

Category B's always-home is a per-repo directory under the state root:

```
~/T/_large_files_bridge/repos/<repoKey>/
├── repo_storage.yaml          # the repo-wide file (repo_tracking_scheme §2) — re-stamped freely here
├── files.yaml                 # the fast per-file fingerprint index
├── files/                     # per-file <name>.yaml sidecars, path-mirrored (repo_tracking_scheme §3)
│   └── videos/trees.mov.yaml
├── history/                   # per-computer audit logs (repo_tracking_scheme §4)
│   └── studio-tower.txt
├── decisions.yaml             # the two-axis decision ledger (decisions.mdx)
├── manifest.yaml              # large-file → IPFS CID + pin state
├── analysis/<rel>/compression.yaml   # compression records (compression.mdx §8)
└── origin.txt                 # one line: the absolute repo path this <repoKey> maps to (collision guard §7.4)
```

* **This directory never touches git.** Nothing here is ever committed, git-ignored, or visible in the
  user's working tree.
* **The layout is exactly the subtree that mirrors into the sync repo** (§5): the sync mirror is
  `<syncRepo>/repos/<repoKey>/` with the **same** shape.

  `shortHash` collision is **detected**, not silently merged (§7.4).
* **Lifecycle & cleanup** are defined in [storage_local.mdx §2.2b](./storage_local.mdx): created lazily on
  the first tracking write, removed when the repo is de-listed (retained on a temporarily-missing drive).

The `repoKey` is computed by `shortHash()` / `storageSid()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts`.

## 4. The company/Personal sync repo (the additive mirror) (LOCKED)

A user who wants their tracking state to **travel** configures a **sync repo** on the **owning** company or
Personal storage — *"a Company repo only for the Large File Bridge webapp to sync changes."* When one
exists and the per-repo toggle is on, Category B is **also** written to `<syncRepo>/repos/<repoKey>/`, in
addition to Local Storage.

* **It never surprises the user.** A sync repo is a repo whose entire reason to exist is to carry this
  noise between computers. Committing per-scan churn there is the intended, purpose-built behavior — unlike
  a product/working repo, where such a commit is unwelcome. This is exactly why Category B may live there
  but never in the working repo.
* **Where it is configured.** On the owning company/Personal storage's settings, as a dedicated backing
  repo ([storage_company.mdx §7](./storage_company.mdx), [storage_personal.mdx §14](./storage_personal.mdx),
  [storage_settings.mdx §4b](./storage_settings.mdx)); the **per-repo opt-in** toggle lives in
  [repo_settings.mdx §2](./repo_settings.mdx). Default **OFF**.
* **Per-`repoKey` isolation inside the sync repo.** Each owned repo gets its own
  `<syncRepo>/repos/<repoKey>/` subtree, so many repos' tracking state coexist without collision, and a
  teammate pulling the sync repo sees one subtree per repo.
* **Ownership resolution reuses the owner index.** Which company/Personal storage owns a given repo — and
  whether it has a sync repo configured — is resolved through the same owner index the artifact placer
  builds ([repo_company_mapping.mdx](./repo_company_mapping.mdx)), so the two never drift.
* **Ownership itself travels and is consent-gated.** WHICH sync repo a repo mirrors to is decided by its
  **owner**, and that owner assignment is not a purely local fact: assigning a repo to a company redirects
  its Category-B tracking into the **shared** company sync repo, so the decision is committed as an
  assertion into `<syncRepo>/owner_map.yaml` (keyed by the git remote) and **travels** to other members —
  applied on their computers only through an explicit review-and-consent page, never silently. That travel +
  consent layer is owned by [repo_owner_propagation.mdx](./repo_owner_propagation.mdx); this policy only
  supplies the reason it matters (ownership moves where tracking flows).

Resolving the owning storage's sync repo for a given repo (null when none is configured or the toggle is
off) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveStateSyncRepo(repoRoot)` (reads the per-repo toggle + the owning storage's backing locations from `storage-settings.service.ts`; returns the sync-repo `repos/<repoUid>/` root or null — see §4.1), consumed by the Category-B writers.

### 4.1 The mirror subtree is keyed by `repoUid` — the REMOTE, not the path (LOCKED)

**This supersedes the `<syncRepo>/repos/<repoKey>/` wording in §2, §3, §4, §5 and §7** wherever it names the
**mirror**. The authority is [storage_company.mdx §8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked);
this section only records what it means for placement.

```
ALWAYS  →  ~/T/_large_files_bridge/repos/<repoKey>/     repoKey = shortHash(resolve(repoPath))   ← MACHINE-LOCAL
ALSO    →  <syncRepo>/repos/<repoUid>/                  repoUid = shortHash(host/owner/name)     ← MACHINE-SHARED
```

* **Two keys, two jobs.** `repoKey` names **"this repo on this computer"**; `repoUid` names **"this repo,
  anywhere."** Local Storage is machine-local state and **stays path-keyed** — that is correct and does not
  change. Only the **mirror** subtree changes key.
* **Why it had to change.** A **machine-local value must never key machine-shared data.** The same repo is
  `/Users/bryan/BGit/…` on one computer and `/Users/bryanstarbuck/BGit/…` on the other, so under the old
  path-hash key the two machines wrote and looked for **different directories inside the same sync repo**.
  The tracking text could travel perfectly and still never be found.
* **Normalization (the exact key).** Lowercase; strip scheme, credentials, any `.git` suffix, and any
  trailing slash; fold `git@host:owner/name` and `https://host/owner/name` to one **`host/owner/name`**.
  `https://github.com/ACT3ai/charlie-kirk.git` → `github.com/act3ai/charlie-kirk`. `repoUid` is a short hash
  of that string.
* **No remote → no mirror, said out loud.** A repo with no git remote has no shared identity, so it
  **cannot** mirror. LFB does not invent one and does not silently write into a subtree no other computer
  will read: the repo's settings say *"no remote — tracking stays on this computer"*
  ([repo_settings.mdx §2](./repo_settings.mdx)). Local Storage still works exactly as before.
* **The same repo cloned twice on one machine converges in the mirror and stays split locally.** Two working
  trees at two paths get **two** `repos/<repoKey>/` directories (right — they are different working trees)
  and **one** `<syncRepo>/repos/<repoUid>/` subtree (right — they are the same repo). Where the two clones
  disagree, §5.1's merge rules apply: union, never clobber.
* **The subtree shape is unchanged** — `<syncRepo>/repos/<repoUid>/` mirrors the same layout as §3. Only the
  directory name is derived differently.

Deriving the remote-keyed shared identity and joining the mirror subtree is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `normalizeRemoteKey()` / `repoUidFor()` (re-exported from the leaf `repo-identity.ts`) (the remote-derived key, over the existing normalizer `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `normalizeRemoteKey()` / `parseRemoteOwner()`) and `resolveStateSyncRepo()` (which joins `repos/<repoUid>/`, returning null when the repo has no remote); `repoKeyFor()` / `repoStateDir()` in the same file stay path-keyed for Local Storage.

## 5. The sync pass — reconciling Local Storage ↔ the sync repo (LOCKED)

Because Local Storage is authoritative and the sync repo is an additive mirror, keeping them together is a
**reconcile**, not a hand-off:

* **Push.** After a Category-B write, the changed files under `repos/<repoKey>/` are mirrored into
  `<syncRepo>/repos/<repoKey>/` and committed + pushed by the storage's git backbone
  ([backbone_resilience.mdx](./backbone_resilience.mdx)). The per-scan `repo_storage.yaml` re-stamp does
  produce commits there — in the purpose-built repo, which the auto-commit/auto-push loop already carries.
* **Pull + merge.** On another computer, pulling the sync repo brings that machine's siblings'
  `repos/<repoKey>/` subtrees. The reconcile folds them into local state:
  * **`history/<device>.txt`** — self-owned per computer; no merge needed, each machine reads the others.
  * **per-file sidecars** — append-only `events:`; merged by union of events (dedup by `at`+`kind`+`on_device`).
  * **`decisions.yaml` / `manifest.yaml`** — union-merge exactly as [decisions.mdx](./decisions.mdx) /
    [git_backbone.mdx](./git_backbone.mdx) already specify.
  * **`repo_storage.yaml`** — the counts/last_scan are per-machine rollups; the local machine's is
    authoritative for itself, and a re-scan rebuilds it, so no cross-machine merge is attempted.
* **Logs are excluded.** `launcher.log`, `log.log`, and `error.err` (Local Storage, [CLAUDE.md](../CLAUDE.md))
  are **never** mirrored into the sync repo — they are machine-local logs, not tracking state (§8).

Mirroring the changed `repos/<repoKey>/` subtree into the sync repo and reconciling a pulled subtree back
into Local Storage is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo(repoRoot)` / `reconcileFromSyncRepo(repoRoot)` (invoked after Category-B writes and on the storage's git-backbone pull).

### 5.1 Receiving is a per-entry MERGE, never a wholesale copy (LOCKED)

The pull half of §5 is a **merge**, and the rules are locked in
[storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked). Restated
here only as the constraint on this policy's `manifest.yaml` reconcile:

* **Union by `path`** — an entry present on either side survives.
* **Absence is never a delete** — an entry missing from the incoming copy is untrusted, not a removal.
* **Union `pinned_by`** — the two computers' pin claims are added together, never replaced. A
  last-writer copy erases the peer's claim and with it the entire "a peer has this" signal.
* **Newer `modified_at` wins** a conflicting CID for the same path; ties keep the local value.
* **This computer's own claim is re-derived from its real pinset**, never adopted from what arrived
  ([storage_local.mdx §9.5](./storage_local.mdx#95-the-ipfs-pinset-is-the-source-of-truth--pinned_by-is-a-stale-cache)).
* **The reconcile runs on every backbone pull**, not on demand — a merge that only happens when someone
  clicks something is a merge that does not happen.
* **It must land in BOTH manifests.** A repo's CID/pin facts live in two files
  ([storage_local.mdx §9.2.1](./storage_local.mdx#921-a-repos-cidpin-facts-live-in-two-files-locked),
  [storage_company.mdx §8.6](./storage_company.mdx#86-two-manifests-must-not-disagree)); reconciling only
  one leaves the product counting a pull it cannot show, or showing a row it cannot pull.

## 6. Interaction with the scan-timestamp churn (LOCKED)

The single loudest source of churn is `refreshCounts()` re-stamping `repo_storage.counts` **and**
`last_scan.at` on **every** scan ([scan.mdx §2](./scan.mdx)). Under this policy:

1. **It writes to Local Storage, never to `<repoRoot>/.lfbridge/`.** `refreshCounts()` (and the
   `readRepoStorage()` / `writeRepoStorage()` it calls) route through `resolveTrackingRoot()` (§2), which
   always returns `~/T/_large_files_bridge/repos/<repoKey>/`. The per-scan re-stamp lands there and
   **never dirties any working repo** — a repo the user never asked LFB to do anything with stays pristine.
2. **In the sync repo (when enabled) the churn is expected.** There the re-stamp produces commits in the
   purpose-built repo, carried by the backbone auto-push loop — the trade the user opted into (§4).

Routing `refreshCounts()` / `writeRepoStorage()` through `resolveTrackingRoot()` (so the per-scan re-stamp
lands in Local Storage, never in a working repo) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` — `refreshCounts()` / `writeRepoStorage()` / `repoStoragePath()`, driven by the sole recurring caller `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()`.

## 7. Non-default & error cases (LOCKED)

### 7.1 Sync-repo path missing or unwritable
The configured sync-repo path does not exist, or is on an unmounted/read-only volume. LFB does **not** fail
the write and does **not** fall back to the working repo (that would reintroduce churn). Local Storage is
written as always; the **sync mirror is skipped**, logs a WARN to `error.err` with the storage id + path,
and surfaces a storage-settings warning ([repo_settings.mdx §2](./repo_settings.mdx)). When the path
returns, the next sync pass resumes and reconciles the backlog.

### 7.2 No sync repo configured / toggle off
Category B is **Local-Storage-only**. `decisions.yaml` and `manifest.yaml` are real and honored locally;
they are simply **not yet shared** — consistent with [decisions.mdx §15](./decisions.mdx) (decisions reach
the team only **with** a git backbone). Turning the toggle on later starts mirroring **future** writes and
the next sync pass pushes the accumulated state.

### 7.3 Two computers, one repo, no sync repo
Each computer keeps its own Local-Storage `repos/<repoKey>/`. They do not share tracking state (by design),
and neither dirties the working repo. If the user wants both to share it, that is exactly what the **sync
repo** is for (§4).

### 7.4 `repoKey` collision
`repoKey = shortHash(resolve(path))` is a 48-bit hash of the **absolute** path. A genuine collision between
two different absolute paths is astronomically unlikely; `repos/<repoKey>/origin.txt` (§3) records the path
the key was minted from. On a write, if `origin.txt` exists and does **not** match the current repo's
absolute path, LFB treats it as a collision, appends `-2`/`-3` to the directory name (the pin-unit
disambiguation, [storage_local.mdx §6.1](./storage_local.mdx)), and records the resolved name so re-scans
stay stable.

### 7.5 Cloud-mounted repo (Dropbox / Google Drive / network volume)
A working tree on a cloud-synced mount is exactly where surprise `.lfbridge/` churn would be most expensive
(every re-stamp triggers a cloud upload). This policy avoids it entirely: Category B is under the **local**
state root, off the cloud mount. Only Category-A content artifacts (which the user explicitly generated) may
sit in the repo's `.lfbridge/`.

### 7.6 Permission errors writing Local Storage
The state root is guaranteed writable-or-`/tmp`-fallback ([state-dir.ts](./storage_local.mdx#1-the-state-root)).
A tracking write **never** crashes a scan or a user action (the store layer's never-throw discipline,
[storage_local.mdx §15](./storage_local.mdx)).

### 7.7 Relocated `.lfbridge/`
The keep-`.lfbridge/` relocation ([storage_settings.mdx §3](./storage_settings.mdx)) governs only
**Category-A** artifacts (where transcripts/descriptions land in the repo). Category B ignores it — it is
never in the repo to relocate.

### 7.8 De-listing a repo
When the user removes a repo from LFB, its `repos/<repoKey>/` state-root directory is cleaned up
([storage_local.mdx §2.2b](./storage_local.mdx)); a repo that merely went **missing** (unmounted drive)
retains its directory so configuration is not lost on remount.

## 10. Acceptance criteria

0. **`.lfbridge/` exists only inside a working git repo** (§0). A dedicated LFB file repo (Personal /
   Company / Community SDL) has **no `.lfbridge/` directory at all**: its artifacts **and** its own metadata
   (`storage.yaml`, `mapped_dirs.yaml`, `files.yaml`, `manifest.yaml`, `bookmarks.yaml`, `devices/`,
   `analysis/`) sit at the SDL **root**. `charlie-kirk/videos/x.mp4` → `charlie-kirk/.lfbridge/videos/x.mp4.transcription`;
   `~/_Mirror/a/b.mp4` → `personal_large_files_bridge/_Mirror/a/b.mp4.transcription` (**no** `.lfbridge/`
   segment).
0a. The decision is keyed on **resolved storage kind** via the single choke point `trackingBaseDir(root)` /
   `usesLfbridgeDir(type)` — never on a path or a name; **no** writer joins `LFBRIDGE_DIR` directly. The
   default kind is **`repo`** (keeps `.lfbridge/`), and an SDL's own `.git/` never demotes it to a working
   repo (§0.2).
0b. A pre-existing SDL `.lfbridge/` is **migrated** to the root (`migrate-sdl-lfbridge.ts`), readers **fall
   back** to the old path until it is, the migration **merges** rather than overwrites (root copy wins a
   file-vs-file conflict, loser retained + WARN), and mapped-dir keys colliding with a reserved SDL metadata
   name are **rejected** (§0.3).
1. LFB splits per-repo files by **data category**: **Category A** (transcripts + AI descriptions) lives in
   the working repo's committed `.lfbridge/` and travels with the repo; **Category B** (`repo_storage.yaml`,
   `files.yaml`, per-file sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`, compression
   records) **never** enters a working repo.
2. Category B lives in **Local Storage always** at `~/T/_large_files_bridge/repos/<repoKey>/`
   (`repoKey = shortHash(resolve(repoPath))`), which never touches git and never merge-conflicts; the repo
   working tree is untouched (no `.lfbridge/` tracking files, no `.gitignore` edit, no commit).
3. When the owning **company/Personal storage has a sync repo configured** and the per-repo toggle is
   **on**, Category B is **additionally** mirrored to `<syncRepo>/repos/<repoKey>/` so it travels — in
   addition to Local Storage, not instead of it. Default **off**.
4. There is **no tier that writes Category B into the working repo** — the earlier content-threshold
   three-tier model is superseded; producing a transcript no longer redirects tracking state into the repo.
5. The per-scan `refreshCounts()` re-stamp of `last_scan.at` / `counts:` routes through the resolver, so it
   lands in Local Storage and **never dirties a working repo** (§6).
6. `decisions.yaml` and `manifest.yaml` are **Local-Storage-always** and travel via the **sync repo**
   (union-merge) when it is enabled; absent a sync repo they are honored locally but not yet shared (§7.2).
7. The sync pass mirrors the changed `repos/<repoKey>/` subtree into the sync repo and reconciles a pulled
   subtree back (append-only merge for sidecars/history, union-merge for decisions/manifest) (§5).
8. LFB's logs (`launcher.log` / `log.log` / `error.err`) live **only** under the state root and are
   **never** committed into a working repo nor mirrored into the sync repo (§8).
9. Every non-default/error case in §7 holds: unwritable sync path → Local-Storage-only + WARN (never the
   working repo); no sync repo → Local-Storage-only; `repoKey` collision detected via `origin.txt`; cloud
   mount, permission error, relocated `.lfbridge/`, and de-listing all behave as specified — and a tracking
   write **never** crashes a scan or user action.
10. New/updated code exists and is cited: `resolveRepoStateDir(repoKey)` (state-dir.ts),
    `resolveTrackingRoot()` / `resolveStateSyncRepo()` (tracking-root.service.ts), `mirrorToSyncRepo()` /
    `reconcileFromSyncRepo()` (tracking-sync.service.ts), and the routed `refreshCounts()` /
    `writeRepoStorage()` (repo-storage.service.ts).
11. The **mirror** subtree is `<syncRepo>/repos/<repoUid>/`, where `repoUid` is a short hash of the
    **normalized git remote** (`host/owner/name`, lowercased, scheme/creds/`.git`/trailing slash stripped) —
    **never** the path-derived `repoKey` (§4.1, [storage_company.mdx §8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked)).
    Local Storage stays at `repos/<repoKey>/`. Two computers with the same repo at **different absolute
    paths** resolve to the **same** mirror subtree; the same repo cloned twice on one machine keeps **two**
    Local-Storage directories and **one** mirror subtree.
12. A repo with **no git remote** does **not** mirror, and the product **says so** in the repo's settings —
    it never writes into a subtree no other computer can find (§4.1).
13. An **SDL's committed `manifest.yaml` lives at `<root>/manifest.yaml`**, resolved through
    `trackingBaseDir()` — **never** `<root>/.lfbridge/manifest.yaml` (§0.4). A storage containing **both**
    is a defect, not a supported layout; the §0.3 read-fallback may still read an un-migrated
    `.lfbridge/` copy but nothing new is written there.
14. Reconciling a pulled mirror is a **per-entry merge** — union by `path`, union `pinned_by`, newer
    `modified_at` wins, **absence is never a delete**, this computer's own claim re-derived from its real
    pinset — and it lands in **both** manifests a repo keeps (§5.1).
