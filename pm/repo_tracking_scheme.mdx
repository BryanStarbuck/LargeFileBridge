# LargeFileBridge — Repo Tracking Scheme

Large File Bridge keeps a small pile of **per-repo tracking state** — the repo-wide `repo_storage.yaml`,
the per-file `<name>.yaml` sidecars, the `history/<device>.txt` logs, the `files.yaml` index, the
`decisions.yaml` ledger, and the `manifest.yaml` CID list. **None of it lives in the working repo.**

The one thing that DOES travel with the working repo is the user's **derived content** — transcripts and
AI descriptions. Everything else is LFB's own bookkeeping about **compression, big files, and git-ignore
decisions**, and it stays out of the repo entirely.

**This spec owns the shapes of three of those tracking files** — `repo_storage.yaml`, the per-file
`<name>.yaml` sidecars, and the `history/<device>.txt` logs — and pins **where** they live.

This spec builds on the siblings and does not repeat them:

* [storage_repo.mdx](./storage_repo.mdx) — the repo storage overview and the `.lfbridge/` layout.
* [storage_local.mdx](./storage_local.mdx) — **Local Storage**, the machine-local state root
  `~/T/_large_files_bridge/` that hosts `repos/<repoKey>/` (§1), the home for all tracking state.
* [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) — the **data-category placement
  policy** (§1+) that governs *where* every file below lives (Local Storage always; company sync repo also,
  when enabled; the working repo never). Its [**§0**](./artifact_placement_policy.mdx) owns the orthogonal
  **storage-kind** rule: `.lfbridge/` exists **only** inside a working git repo — the kind this spec is
  about. A dedicated LFB file repo (an **SDL**: Personal / Company / Community) has **no `.lfbridge/` at
  all**; its root *is* the `.lfbridge` area. The two rules compose: §0 picks the **base directory** inside a
  root, §1+ picks **which root**.
* [decisions.mdx](./decisions.mdx) — the `decisions.yaml` ledger. The per-file sidecars and history logs
  in this spec are the human-readable, per-machine mirror of that ledger plus the automatic events it does
  not carry. The ledger, too, now lives in Local Storage / the sync repo — not the working repo.
* [devices.mdx](./devices.mdx) — the unique per-computer name used for `history/<device>.txt` and stamped
  on every event.
* [special_files.mdx](./special_files.mdx) — the classifier that decides a file is **special** and earns a
  sidecar.
* [compression.mdx](./compression.mdx) — the before/after hash capture whose two hashes land in a
  sidecar's `compress` event.
* [perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx) — the local, network-free content
  fingerprint stored per file.
* [use_case_two_computers.mdx](./use_case_two_computers.mdx) — the month-by-month tower↔laptop tour that
  exercises every write defined here.

## 1. Where tracking state lives — Local Storage, never the working repo (LOCKED)

The split is by **data category**, and it is absolute:

* **The working repo's committed `.lfbridge/` holds ONLY two content artifacts** — the path-mirrored
  **`<name.ext>.transcription`** and **`<name.ext>.ai_description`** files (extension **appended**,
  [Transcribe.mdx §3](./Transcribe.mdx), [ai_description.mdx §2](./ai_description.mdx)). These are stable,
  content-derived, and *meant* to travel with the repo, so they are committed and ride the repo's own git.
* **Everything else — all of LFB's compression / big-file / git-ignore tracking state — lives in
  [Local Storage](./storage_local.mdx)**, at:

```
~/T/_large_files_bridge/repos/<repoKey>/          ← Local Storage home for THIS repo's tracking state
├── repo_storage.yaml            ← ★ THIS SPEC §2 — the repo-WIDE settings-and-state file (root key: repo_storage)
├── files.yaml                   ← fast per-file fingerprint index (storages.mdx §4.1)
├── files/<path>.yaml            ← ★ THIS SPEC §3 — per-file <name>.yaml sidecars, path-mirrored
│   └── videos/trees.mov.yaml
├── history/<device>.txt         ← ★ THIS SPEC §4 — one plain-text audit log PER COMPUTER
│   ├── studio-tower.txt
│   └── bryan-macbook-pro.txt
├── decisions.yaml               ← the per-file Add-to-IPFS / git-ignore decision ledger (decisions.mdx)
├── manifest.yaml                ← large-file → IPFS CID + pin state
├── analysis/<rel>/compression.yaml   ← compression records (compression.mdx §8)
└── origin.txt                   ← one line: the absolute repo path this <repoKey> maps to (collision guard)
```

* **`repoKey` = `shortHash(resolve(repoPath))`** — the 12-hex `storageSid()` scheme. The same repo on the
  same machine always resolves to the same `repos/<repoKey>/` directory.
* **This directory never touches git.** It is under the Local Storage state root, so nothing in it is ever
  committed, git-ignored, or visible in the user's working tree. That is exactly why it can never produce a
  **merge conflict** — the failure that motivated this whole scheme.

### 1.1 The company sync repo — an ADDITIONAL travelling home (when enabled)

Local Storage does not travel between computers. When a user wants their tracking state to travel — the
same `repo_storage.yaml`, sidecars, history, decisions, and manifest visible on the tower and the laptop —
they configure a **sync repo** on the owning **company** or **Personal** storage (*"a Company repo only
for the Large File Bridge webapp to sync changes"*), and turn on the per-repo toggle
([repo_settings.mdx](./repo_settings.mdx), [storage_company.mdx §7](./storage_company.mdx),
[storage_personal.mdx §14](./storage_personal.mdx)).

When that is on, this repo's tracking state is **also** written to `<syncRepo>/repos/<repoKey>/` —
**in addition to** Local Storage, not instead of it. Local Storage stays the authoritative working copy on
each machine; the sync repo is the travel vehicle. A sync pass reconciles the two: the append-only sidecars
and history logs merge, and `decisions.yaml` / `manifest.yaml` union-merge ([decisions.mdx](./decisions.mdx),
[git_backbone.mdx](./git_backbone.mdx)). **Default OFF** — absent a configured sync repo and the toggle,
tracking state is Local-Storage-only.

### 1.2 Two keys, two jobs — `repoKey` (this computer) vs `repoUid` (anywhere) (LOCKED)

**This refines §1 and §1.1**: the Local-Storage directory and the sync-repo subtree are keyed **differently,
on purpose**. The authority is [storage_company.mdx §8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked);
this section records the two keys so nothing in this spec reads as if there were only one.

| Key | Derived from | Names | Used for |
| --- | --- | --- | --- |
| **`repoKey`** | `shortHash(resolve(repoPath))` — the **absolute path on this computer** | *"this repo **on this computer**"* | the Local-Storage home `~/T/_large_files_bridge/repos/<repoKey>/` (§1) |
| **`repoUid`** | `shortHash(normalized git remote)` — `host/owner/name` | *"this repo, **anywhere**"* | the travelling mirror `<syncRepo>/repos/<repoUid>/` (§1.1) |

* **A machine-local value must never key machine-shared data.** The same repo is
  `/Users/bryan/BGit/…` on the tower and `/Users/bryanstarbuck/BGit/…` on the laptop. Keying the **mirror**
  by the path hash made the two computers write and look for **different directories inside the same sync
  repo** — the tracking text travelled correctly and was never found. Every path in §1.1, §3, §4 and §5
  that shows the sync repo as `<syncRepo>/repos/<repoKey>/…` means **`<repoUid>`**.
* **Local Storage is unchanged.** `repos/<repoKey>/` stays path-keyed, including its `origin.txt` collision
  guard. It is machine-local state and *should* be keyed by this machine's path.
* **The remote is normalized before hashing.** Lowercase; strip scheme, credentials, any `.git` suffix and
  trailing slash; `git@host:owner/name` and `https://host/owner/name` fold to the same `host/owner/name`.
* **No remote → no `repoUid` → no mirror.** Such a repo keeps full Local-Storage tracking and simply does
  not travel; the repo's settings say so rather than LFB writing into an unfindable subtree.
* **The same repo cloned at two local paths** has **two** `repoKey`s (two Local-Storage directories — right,
  they are two working trees) and **one** `repoUid` (one mirror subtree — right, it is one repo). Where the
  two clones disagree in the mirror, the union-merge rules apply
  ([storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked)):
  union by path, union `pinned_by`, newer `modified_at` wins, absence is **never** a delete.

Deriving the two keys — the path-keyed Local-Storage directory and the remote-keyed mirror subtree — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `repoKeyFor(root)` / `repoStateDir(root)` (path-keyed, Local Storage) and `repoUidFor(root)` / `resolveStateSyncRepo(root)` (remote-keyed, the `<syncRepo>/repos/<repoUid>/` mirror, null when the repo has no remote), the remote normalized by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `normalizeRemoteKey()`.

The full data-category placement policy (Local Storage always; company sync repo also when enabled; the
working repo never) is authoritative in [artifact_placement_policy.mdx](./artifact_placement_policy.mdx).

**The keep-`.lfbridge/` consent still gates the two content artifacts** ([storage_settings.mdx §3](./storage_settings.mdx),
default on): it governs whether transcripts/AI-descriptions may be written into the working repo's
`.lfbridge/`. Tracking state is never affected by it — that always goes to Local Storage.

Seeding a repo's Local-Storage `repos/<repoKey>/` directory and its `repo_storage.yaml` (plus `history/`,
`files/`) when a repo is enlisted is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureRepoStorage(repoRoot)` (seeds under the Local-Storage tracking root resolved by `tracking-root.service.ts` — `resolveTrackingRoot()`); it no longer git-ignores anything, since nothing churny is written into the working repo.

## 2. `repo_storage.yaml` — the repo-wide settings-and-state file (LOCKED)

When LFB **enlists** a repo, the web app **automatically** creates `repos/<repoKey>/repo_storage.yaml` in
**Local Storage**. It is the **one file that stores settings and state across the entire repo** —
everything that is true of the repo as a whole rather than of one file.

### 2.1 The single level-one key rule

```yaml
# ~/T/_large_files_bridge/repos/<repoKey>/repo_storage.yaml
repo_storage:                       # ← the ONLY level-one key. Everything else is nested below it.
  schema_version: 1
  name: charlie-kirk                # defaults to the repo folder name; user-editable (repo_settings.mdx)
  enlisted:
    at: 2026-01-08T15:04:22Z        # UTC — when this repo was first enlisted with LFB
    by: bryan@act3ai.com            # the allow-listed Google email that enlisted it
    on_device: studio-tower         # the unique device name (devices.mdx) that enlisted it
  counts:                           # rolled-up special-file metrics for the repo (special_files.mdx)
    special: 20                     # files that matched the special-file test
    large: 20
    ipfs_pinned: 5                  # pinned (by us OR observed already-pinned outside us)
    videos: 19
    images: 0
    audio: 1
    compressible: 15               # video/image/audio that look uncompressed (compression.mdx baseline)
    transcribable: 20             # video + audio that could be transcribed (Transcribe.mdx)
    transcribed: 0
  policy:                           # repo-wide defaults (mirror repo_settings.mdx; per-file may override)
    recommend_ipfs_pin: true
    recommend_compress: true
    recommend_transcribe: false
  last_scan:
    at: 2026-01-08T15:06:10Z
    on_device: studio-tower
    headless: false                 # true when the background scan wrote this, not a user in the web app
```

* **`repo_storage:` is the only level-one property.** A parser reads `doc.repo_storage.*` and never has
  to guess. This is the same hard rule the per-file sidecar follows (§3).
* **Written automatically on enlist**, then updated by every scan and every user action — always to Local
  Storage. It is **not** hand-authored — the web app owns it.
* **`counts:` is compression / big-file / git-ignore data.** That is exactly why it does **not** go in the
  working repo: it re-stamps `last_scan.at` and recomputes `counts:` on **every** scan, on **every**
  machine, so committing it into the repo produced a fresh merge conflict every scan. In Local Storage that
  churn is free and private to the machine.
* **Settings vs. the machine-local `config.yaml`.** `repo_storage.yaml` carries the repo-wide facts (its
  LFB name, enlist provenance, rolled-up counts, recommendation policy, last-scan stamp). The per-computer
  **participation** knobs (this machine's `pinned` master toggle, threshold override, pin/fetch policy,
  sharing, the sync-repo toggle) stay in `~/T/_large_files_bridge/pin/r/<repo>/config.yaml`
  ([repo_settings.mdx §3](./repo_settings.mdx)). Both are Local Storage; they never duplicate a field (§5).

### 2.2 `counts:` is the repo's special-file rollup

The `counts:` block answers *"what special files are in here and what could we do with them?"* The
classifier that fills it is [special_files.mdx](./special_files.mdx). (Note: the metrics the
[Repos](./repos.mdx) table and the One-repo screen render are computed live from the pin-unit `status.yaml`
candidates + the decision ledger, not read back from this file — this rollup is a cached convenience, not
the UI's source of truth.)

Reading, seeding, and updating `repo_storage.yaml` (single `repo_storage` root key; deterministic YAML; written to the Local-Storage tracking root, optionally mirrored to the company sync repo) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` — `readRepoStorage(repoRoot)` / `writeRepoStorage(repoRoot, doc)` / `refreshCounts(repoRoot)` (each derives its directory from `tracking-root.service.ts` — `resolveTrackingRoot()`, which returns the Local-Storage `repos/<repoKey>/` root and never the working repo).

## 3. Per-file `<name>.yaml` sidecars — identity + the file's event history (LOCKED)

For every **special** file ([special_files.mdx](./special_files.mdx)) LFB writes a **sidecar YAML** whose
name is the media file's name **plus `.yaml`**, under a **mirror of the file's repo-relative path** inside
the Local-Storage tracking root's `files/` directory.

```
repo file:   ~/BGit/charlie-kirk/videos/trees.mov
sidecar:     ~/T/_large_files_bridge/repos/<repoKey>/files/videos/trees.mov.yaml
             (and, when the sync repo is enabled, mirrored to <syncRepo>/repos/<repoKey>/files/videos/trees.mov.yaml)
```

* **We mirror the file's repo-relative path** under `files/`, then create `<filename>.yaml` at the leaf.
  `videos/trees.mov` → `files/videos/trees.mov.yaml`. The mirror keeps two files with the same basename in
  different folders from colliding.
* A sidecar is created **the first time the file is seen as special** — including when a scan (even a
  **headless** background scan) discovers it was already IPFS-pinned or compressed **outside** LFB (§3.3).
* **Sidecars never go in the working repo.** They carry compress / pin / git-ignore events — exactly the
  category kept out of the repo.

### 3.1 The single level-one key rule + identity

```yaml
# ~/T/_large_files_bridge/repos/<repoKey>/files/videos/trees.mov.yaml
file:                               # ← the ONLY level-one key. Everything else is nested below it.
  path: videos/trees.mov            # repo-relative path (the stable key even when absent locally)
  name: trees.mov
  categories: [large, video, ipfs_pinned]   # from special_files.mdx (why this file is special)
  size: 734003200                   # bytes, as last seen
  created: 2026-01-02T10:12:00Z     # filesystem created (UTC)
  modified: 2026-01-02T10:12:00Z    # filesystem modified (UTC)
  hash: 9f2c8b…                     # exact content hash (files.yaml fingerprint)
  fingerprint:                      # perceptual/content fingerprint (perceptual_fingerprint.mdx)
    algo: pdq                       # e.g. pdq (image) / vpdq (video)
    value: a1b2c3…
  first_seen:
    at: 2026-01-08T15:06:10Z
    on_device: studio-tower
  events:                           # ← append-only history of everything that happened to this file (§3.2)
    - …
```

* **`file:` is the only level-one property**, matching §2.1's rule.
* **`path`** (repo-relative) is the stable identity even on a computer that does **not** have the bytes
  yet. `hash` is the exact fingerprint; `fingerprint` is the perceptual one that survives a
  compress/convert/resize.
* Identity (`size`, `created`, `modified`, `hash`) matches the per-file metadata the charter requires. For
  compressible media the sidecar carries **both** a compressed and an uncompressed variant hash — recorded
  in the `compress` event (§3.2) and mirrored into `files.yaml`'s layered `hashes:` block.

### 3.2 `events:` — the append-only per-file history (LOCKED)

The heart of the sidecar. Every meaningful thing that happens to the file appends **one event**; events
are **never edited in place**. Every event carries **`at` (UTC), `on_device` (the unique computer name),
and `by`** (the allow-listed email, or the sentinel `not-lfbridge` when something happened outside us —
§3.3).

```yaml
  events:
    # 1) First time we scanned it and found it ALREADY IPFS-pinned outside LFB (§3.3)
    - kind: observed
      at: 2026-01-08T15:06:10Z
      on_device: studio-tower
      by: not-lfbridge            # ← we did NOT do this; our scan merely OBSERVED it
      ipfs:
        pinned: true
        cid: bafybe…
      note: "found already pinned on this computer's IPFS node during scan"

    # 2) A user decision, made in the web app (mirrors decisions.mdx; both axes)
    - kind: decision
      at: 2026-01-08T15:20:44Z
      on_device: studio-tower
      by: bryan@act3ai.com
      asked: true
      ipfs: true                  # AXIS 1 — Add to IPFS (pin)
      gitignore: true             # AXIS 2 — Add to git ignore

    # 3) IPFS pin turned ON by us (state transition)
    - kind: ipfs_pin
      at: 2026-01-08T15:20:45Z
      on_device: studio-tower
      by: bryan@act3ai.com
      pinned: true
      cid: bafybe…

    # 4) Compression — pre/post hashes captured (compression.mdx §)
    - kind: compress
      at: 2026-04-03T09:14:02Z
      on_device: studio-tower
      by: bryan@act3ai.com
      before: { hash: 9f2c8b…, size: 734003200 }   # captured BEFORE we touched the bytes
      after:  { hash: 4d7e1a…, size: 118489088 }   # captured AFTER compression
      fingerprint_before: a1b2c3…
      fingerprint_after:  a1b2c4…
      codec: { from: prores, to: h264 }

    # 5) Transcription produced (Transcribe.mdx)
    - kind: transcribe
      at: 2026-03-11T08:02:00Z
      on_device: bryan-macbook-pro
      by: bryan@act3ai.com
      output: videos/trees.transcription
```

**Event `kind`s (extensible, but these are the load-bearing ones):**

| `kind` | When it is appended | Key fields |
| --- | --- | --- |
| `observed` | A scan sees a state that LFB did **not** create — already-pinned or already-compressed outside us (§3.3). | `by: not-lfbridge`, the observed `ipfs`/`compress` facts. |
| `decision` | The user answers the two-axis question ([decisions.mdx §1](./decisions.mdx)). | `asked`, `ipfs`, `gitignore`. |
| `ipfs_pin` | LFB turns pinning **on or off** for the file. | `pinned` (bool), `cid`. |
| `compress` | LFB compresses the file — pre-compression hash/size captured first, post after ([compression.mdx](./compression.mdx)). | `before{hash,size}`, `after{hash,size}`, `fingerprint_before/after`, `codec`. |
| `convert` | A format conversion (e.g. PNG→JPEG) with the same before/after capture. | as `compress`, plus `format{from,to}`. |
| `transcribe` | A transcript is produced for a video/audio file. | `output` sidecar path. |

* **A file may be pinned then unpinned then re-pinned** — each transition is its own `ipfs_pin` event.
* **Both-off decisions are recorded too** (a `decision` event with `ipfs: false, gitignore: false`) — the
  sticky "reviewed, leave it" that stops the warning coming back ([decisions.mdx §2](./decisions.mdx)).

Appending an event to a per-file sidecar (create-on-first-special, mirror the path, append-only, stamp at/on_device/by; written to the Local-Storage tracking root, optionally mirrored to the sync repo) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/file-sidecar.service.ts` — `sidecarPath(repoRoot, relPath)` / `appendFileEvent(repoRoot, relPath, event)` / `readSidecar(repoRoot, relPath)` (path resolved via `tracking-root.service.ts` — `resolveTrackingRoot()`).

### 3.3 Provenance — actions taken OUTSIDE LargeFileBridge (LOCKED)

A repo lives in the real world: the user pins or compresses files on the **command line**, outside our web
app. LFB must **notice and record** that without pretending it did it.

* When a scan (including a **headless** background scan) finds a file **already IPFS-pinned** on this
  computer's node, or **already looks compressed**, and LFB has **no prior record** of doing it, LFB
  appends an **`observed`** event with **`by: not-lfbridge`** and the facts it read (the CID it is pinned
  under, or the compressed size/hash).
* The **`on_device`** on that `observed` event answers **which computer first observed** the external
  state — the load-bearing fact when the same repo is on two machines.
* `not-lfbridge` is a **reserved sentinel** in the `by` field.

Detecting already-pinned / already-compressed state during a scan and recording it as an `observed`/`not-lfbridge` event (idempotent — recorded once) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `reconcileExternalState(repoRoot, file)` (queries the IPFS node's pinset + the compression baseline, appends via `file-sidecar.service.ts`).

## 4. `history/<device>.txt` — the per-computer audit log (LOCKED)

Beside the per-file sidecars in Local Storage is a **`history/`** directory holding **one plain-text log
per computer**, named by that computer's **unique device name** ([devices.mdx](./devices.mdx)):

```
~/T/_large_files_bridge/repos/<repoKey>/history/studio-tower.txt
~/T/_large_files_bridge/repos/<repoKey>/history/bryan-macbook-pro.txt
```

* **One file per computer, keyed by the unique device name.** The tower writes only `studio-tower.txt`;
  the laptop writes only `bryan-macbook-pro.txt`. This is the same **self-owned-file** trust model as the
  device registry ([devices.mdx](./devices.mdx)) — a computer appends only to *its own* history file, and
  reads the others as claims. Two computers therefore never contend on the same log file — and, because
  these logs never enter the working repo, they never merge-conflict there either.
* It is the **history log for this computer for this repo**.

### 4.1 What we log, and the format (LOCKED)

**Every time the user makes a decision — and every automatic action — we append a line (or an indented
block) with a UTC date-time stamp, then what happened.** When one action spans several files with
**different** outcomes, the entry indents a **per-file** breakdown.

```
# ~/T/_large_files_bridge/repos/<repoKey>/history/studio-tower.txt
# Large File Bridge — history log for computer "studio-tower" (device id 7c1f…-…-a90) · repo charlie-kirk
# All timestamps UTC. Append-only. One line per event; indented block when files differ.

2026-01-08T15:06:10Z  SCAN  headless=false  found 20 special files (20 large, 5 already-pinned outside LFB)
2026-01-08T15:20:44Z  DECISION  by=bryan@act3ai.com  recommended IPFS-pin for 6 files; user chose:
```

* **Schema of a line.** `` <UTC-timestamp>␠␠<VERB>␠␠<key=value …>␠␠<summary>``. The **`VERB`** is the
  event kind (`SCAN`, `DECISION`, `IPFS-PIN`, `IPFS-UNPIN`, `COMPRESS`, `CONVERT`, `TRANSCRIBE`, `PULL`,
  `OBSERVED`). When decisions differ per file, indented lines carry one `` <axis>=<value>␠␠<path>`` per file.
* **UTC always.** Every stamp is UTC ISO-8601.
* **This is a mirror, not the source of truth.** The **decision** source of truth is the `decisions.yaml`
  ledger ([decisions.mdx](./decisions.mdx), now also in Local Storage / the sync repo); the **per-file**
  structured record is the sidecar (§3). The history log is the **legible, per-machine paper trail**.

## 5. Reconciliation — who owns what (LOCKED)

Every file below now lives in **Local Storage** (`~/T/_large_files_bridge/repos/<repoKey>/`), and is
**additionally** mirrored to `<syncRepo>/repos/<repoKey>/` when the company/Personal sync repo is enabled.
**None of them ever live in the working repo.**

| File | Scope | Home | Source of truth for… |
| --- | --- | --- | --- |
| `storage.yaml` | repo | Local Storage (+ sync repo) | storage **identity** (name, `type: repo`, `repo_root`, clones). |
| **`repo_storage.yaml`** (§2) | repo | Local Storage (+ sync repo) | repo-**wide** settings + rolled-up special-file **counts** + enlist/scan provenance. |
| `files.yaml` | repo | Local Storage (+ sync repo) | the **fast** per-file fingerprint index. |
| **`files/<path>.yaml`** (§3) | file | Local Storage (+ sync repo) | the **rich per-file history** (events, provenance, variant hashes, perceptual fp). |
| **`history/<device>.txt`** (§4) | computer | Local Storage (+ sync repo) | the **human-readable audit trail** for this computer. |
| `manifest.yaml` | repo | Local Storage (+ sync repo) | large-file → **IPFS CID + pin state**. |
| `decisions.yaml` | repo | Local Storage (+ sync repo) | the **two-axis decision ledger**. |
| **transcripts / AI descriptions** | file | **the working repo's committed `.lfbridge/`** | the user's derived **content** — the only thing that travels with the repo. |

* **The working repo carries content, not bookkeeping.** Transcripts and AI descriptions are content
  *about the media in this repo*; users expect them to travel *with* that repo, so they are committed
  there. Compression state, big-file lists, pin manifests, and git-ignore decisions are LFB's private
  bookkeeping and stay in Local Storage (and, if the user opts in, the sync repo).
* **`decisions.yaml` / `manifest.yaml` still travel — via the sync repo, not the working repo.** They
  remain union-mergeable travellers ([decisions.mdx](./decisions.mdx), [git_backbone.mdx](./git_backbone.mdx));
  their travelling vehicle is now the company/Personal sync repo instead of the working repo. Absent a sync
  repo they are Local-Storage-only (real and honored locally, simply not yet shared).
* **`repo_storage.yaml` vs. `config.yaml`.** `repo_storage.yaml` is *about the repo*; the machine-local
  `pin/r/<repo>/config.yaml` ([repo_settings.mdx](./repo_settings.mdx)) is *about this computer's
  participation* (its `pinned` toggle, threshold override, pin/fetch policy, sharing, sync-repo toggle).
  They never duplicate a field.

## 6. Acceptance criteria

1. **No LFB tracking state is ever written into a working repo.** `repo_storage.yaml`, `files.yaml`, the
   per-file `files/<path>.yaml` sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`, and
   compression records all live in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/`
   (`repoKey = shortHash(resolve(repoPath))`), which never touches git and therefore never merge-conflicts.
2. **The working repo's committed `.lfbridge/` holds ONLY** the path-mirrored transcript
   (`<name.ext>.transcription`) and AI-description (`<name.ext>.ai_description`) artifacts — the user's
   derived content — which are committed and travel with the repo.
3. On enlist, LFB **automatically** creates `repos/<repoKey>/repo_storage.yaml` with a **single** level-one
   key `repo_storage:` (everything nested below), carrying name, enlist provenance, the rolled-up
   `counts:`, repo-wide `policy:`, and the `last_scan` stamp (including `headless`).
4. For **every special file** LFB writes a sidecar at `repos/<repoKey>/files/<mirrored path>/<name>.yaml`
   with a **single** level-one key `file:`; it carries path, name, categories, size, created, modified,
   exact hash, the perceptual fingerprint, `first_seen`, and an **append-only `events:` history**. Each
   event carries `at` (UTC), `on_device`, and `by` (email or **`not-lfbridge`**). Event kinds include
   `observed`, `decision`, `ipfs_pin`, `compress` (before/after hash + size + `fingerprint_before/after`),
   `convert`, and `transcribe`.
5. When a scan (including a **headless** one) finds a file **already IPFS-pinned or compressed outside
   LFB**, LFB appends an **`observed`** event with `by: not-lfbridge`, the observed CID/size, and the
   `on_device` that first observed it — recorded once, not re-appended every pass.
6. `repos/<repoKey>/history/<device>.txt` exists **one per computer**, named by the **unique device name**;
   each computer appends only to **its own** file; every entry is **UTC-stamped**; multi-file actions with
   differing outcomes indent a **per-file** breakdown.
7. When the repo's owning **company/Personal storage has a sync repo configured** and the per-repo toggle
   is **on**, all of the above is **additionally** mirrored to `<syncRepo>/repos/<repoKey>/` so it travels;
   Local Storage remains the authoritative working copy. Default **off** — tracking state is
   Local-Storage-only. The full policy is authoritative in
   [artifact_placement_policy.mdx](./artifact_placement_policy.mdx).
8. **Two keys, two jobs** (§1.2): Local Storage is `repos/<repoKey>/` keyed by `shortHash(resolve(repoPath))`
   (machine-local, unchanged); the **mirror** is `<syncRepo>/repos/<repoUid>/` keyed by a short hash of the
   **normalized git remote** (`host/owner/name`). Two computers holding the same repo at different absolute
   paths resolve to the **same** mirror subtree. A repo with **no remote** has no `repoUid`, does not mirror,
   and says so. The same repo cloned twice on one machine keeps **two** Local-Storage directories and **one**
   mirror subtree, reconciled by union-merge
   ([storage_company.mdx §8.4.1 / §8.4.3](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked)).
