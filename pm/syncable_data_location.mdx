# LargeFileBridge — The Syncable Data Location

Every storage needs a home for the small, portable **text** that describes it — the descriptor, the
fingerprint index, the IPFS CID list, the device registry, and the expensive-to-generate derived
artifacts (transcriptions, AI descriptions, compression records, bookmarks). That home is a
**Syncable Data Location (SDL)**. It is the piece that **travels between the user's computers** and,
for a shared storage, between **teammates** — so the knowledge and the derived work are produced
**once** and then reused everywhere.

This spec gives the SDL a name, defines what lives in it, and locks the one idea that makes the whole
product coherent: **the SDL is separate from the big files it describes.** The bytes stay where they
naturally live (git-ignored, moved over IPFS); the SDL carries only the small YAML that lets any of
your computers reconstruct *what should exist, where, and what we already know about it*.

Read the shared model first; this spec does not restate it:

* [storages.mdx](./storages.mdx) — the storage family, the `storage.yaml` descriptor (§3), the hidden
  **`.lfbridge/`** tracking area (§4), and the per-file **media-analysis** outputs (§6). **The SDL is
  the directory that contains that `.lfbridge/` plus the Sync List.**
* [sync_list.mdx](./sync_list.mdx) — the portable `LargeFilesBridge_SyncList.yaml` the SDL carries and
  the **trust/reconcile** rules a reader applies. The SDL is *where a Sync List lives*; sync_list is
  *what it says and how it is trusted*.
* [storage_local.mdx](./storage_local.mdx) — the **machine-local** state root
  (`~/T/_large_files_bridge/`). That is emphatically **not** an SDL: it is this computer's private
  bookkeeping and never syncs. The SDL is the opposite — it is the shared, travelling half.
* [devices.mdx](./devices.mdx) — the **device registry** and the **per-device graft** that live
  *inside* the SDL (`.lfbridge/devices/`).
* [storage_settings.mdx](./storage_settings.mdx) — where a computer chooses **where its SDL sits** and
  which extra backing locations mirror it.

## 1. Definition — what a Syncable Data Location IS (LOCKED)

> **Syncable Data Location (SDL).** The directory that holds a storage's hidden **`.lfbridge/`**
> tracking area **and** its portable **`LargeFilesBridge_SyncList.yaml`** — the small, human-readable
> text that travels between the user's computers (over Git, IPFS/IPNS, Google Drive, or Dropbox) so
> every machine agrees on what the storage contains and what is already known about each file. The SDL
> holds **metadata and derived artifacts, never the primary big-file bytes.**

Three consequences flow from that definition and are LOCKED across specs:

1. **The SDL is small text, and it is versioned/backbone-carried.** For most storages the SDL is its
   **own little Git repo** — for a company, `~/BGit/Bryan_git/act3_large_files_bridge/`; for the user's
   personal pile, `~/BGit/Bryan_git/personal_large_files_bridge/`; for a repo storage, the repo itself.
   Git carries the small files; the SDL is exactly the kind of thing Git is good at. (A storage without
   a dedicated repo still has an SDL — the `.lfbridge/` area — carried over a
   [bridge backbone](./sync_list.mdx#1-vocabulary--sync-list--backbone--scope-locked).)

2. **The SDL is not where the big files live.** The large files stay in their **natural directories**
   on disk — a movie in `~/Movies`, a render tree under a project, a photo library under `~/Pictures`.
   The SDL only **points at them** (by relative path + CID + fingerprint) and stores what we **derived**
   from them. This is what lets a company map **many** unrelated directory hierarchies into one storage
   (§3) without relocating a single byte.

3. **The SDL is per-storage, and a computer can have many.** One SDL per storage; a user in three
   companies plus their personal files plus every repo has many SDLs at once, each its own
   `.lfbridge/` + Sync List. Which of a computer's SDLs it keeps `.lfbridge/` for, and where, is a
   machine-local choice ([storage_settings.mdx §3](./storage_settings.mdx#3-the-hidden-lfbridge-tracking-directory--keep-it-and-where)).

The SDL vs. the machine-local state root is the boundary that never blurs:

| | **Syncable Data Location** (this spec) | **Local Storage** ([storage_local.mdx](./storage_local.mdx)) |
| --- | --- | --- |
| Root | the storage's own dir, e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` | `~/T/_large_files_bridge/` |
| Travels between computers? | **Yes** — that is its entire purpose | **No** — wipe it and a rescan rebuilds it |
| Holds | descriptor, fingerprints, CID list, devices, derived artifacts, Sync List | this machine's config, auth, `peers.yaml`, logs, local index |
| Shared with teammates? | Yes, for company/community storages | Never |

## 2. What lives in the SDL — the YAML inventory (LOCKED)

Everything in the SDL is **small text** (YAML), so it rides a backbone cheaply and a human can read it.
The layout, rooted at the SDL:

```
<SDL root>/                              ← e.g. ~/BGit/Bryan_git/act3_large_files_bridge/
├── LargeFilesBridge_SyncList.yaml       ← the portable list (sync_list.mdx) — travels via git here
├── .lfbridge/                           ← the hidden tracking area (storages.mdx §4), git-ignored
│   ├── storage.yaml                     ← descriptor: name + type + type block (storages.mdx §3)
│   ├── mapped_dirs.yaml                 ← §3 the source directory hierarchies this storage covers
│   ├── files.yaml                       ← per-file fingerprint index (storages.mdx §4.1)
│   ├── manifest.yaml                    ← large-file → IPFS CID + pin state (storage_local.mdx §9)
│   ├── bookmarks.yaml                   ← §4.4 the user's bookmarked files (travels with the storage)
│   ├── devices/                         ← the device registry (devices.mdx) — one file per computer
│   │   ├── bryan-macbook.yaml           ←   that device's id, nice name, schedule, and GRAFT (devices.mdx)
│   │   └── bryan-studio.yaml
│   ├── <mapped>/<rel-dir>/<name.ext>.transcription   ← transcript — path-mirrored, ext APPENDED (§4.2)
│   ├── <mapped>/<rel-dir>/<name.ext>.ai_description  ← AI description — path-mirrored, ext APPENDED (§4.2)
│   └── analysis/                        ← per-file derived artifacts, keyed by relpath (§4)
│       └── <relpath>/
│           ├── visuals_by_time.yaml     ←   time × visual × story timeline (video)
│           └── compression.yaml         ←   §4.3 original name/ext + size before/after
│                                            (transcript & description live under .lfbridge/ above, §4.2)
└── …big files live in the MAPPED directories (§3), git-ignored; their transcript/description travel
    under .lfbridge/ above (path-mirrored, ext appended), committed with the SDL…
```

| File | Purpose | Owner | Defined in |
| --- | --- | --- | --- |
| `LargeFilesBridge_SyncList.yaml` | The portable declaration of what should exist + each CID | reconciled | [sync_list.mdx](./sync_list.mdx) |
| `.lfbridge/storage.yaml` | Name, type, type block, clones | discovery + settings | [storages.mdx §3](./storages.mdx#3-storageyaml--the-per-storage-descriptor-locked-shape) |
| `.lfbridge/mapped_dirs.yaml` | The source directory hierarchies mapped into this storage | storage settings | **§3 here** |
| `.lfbridge/files.yaml` | Per-file fingerprints (hash, size, dates; compressed/uncompressed variants) | tracking | [storages.mdx §4.1](./storages.mdx) |
| `.lfbridge/manifest.yaml` | path → CID + `pinned_by` | pin pass | [storage_local.mdx §9](./storage_local.mdx) |
| `.lfbridge/bookmarks.yaml` | The user's starred files (§4.4) | user action | **§4.4 here** |
| `.lfbridge/devices/<device>.yaml` | Device id, nice name, schedule, graft | each device (self-owned) | [devices.mdx](./devices.mdx) |
| `.lfbridge/analysis/<relpath>/*.yaml` | Visuals-by-time + compression record (transcript & description live **directly under `.lfbridge/`**, path-mirrored, not here) | analysis passes | [storages.mdx §6](./storages.mdx) + §4 here |
| `.lfbridge/<rel-dir>/<name.ext>.transcription` · `.lfbridge/<rel-dir>/<name.ext>.ai_description` | Transcript & AI description — **inside `.lfbridge/`, path-mirrored, extension APPENDED** (full media filename kept), committed & travels with the SDL | transcribe / describe | [Transcribe.mdx §3](./Transcribe.mdx), [ai_description.mdx §2](./ai_description.mdx) |

**What is committed depends on the backbone (LOCKED).** This paragraph governs a **company/Personal
storage's own SDL** — a dedicated `_large_files_bridge` repo purpose-built to carry this text (the
"sync repo"), where committing Category-B metadata is the intended behavior. (A **working code repo** is
different: its committed `.lfbridge/` holds **only** Category-A content, and its Category-B tracking state
lives in Local Storage / the sync repo per [artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
never committed to that working repo.) For a **cloud backbone** (Google Drive / Dropbox mirrors the whole
SDL directory), only `LargeFilesBridge_SyncList.yaml` is **committed to Git**; the rest of `.lfbridge/` is
**git-ignored** so it never bloats a code repo's commits, yet it still travels because the SDL directory
itself is mirrored over the chosen backbone
([storage_settings.mdx §4](./storage_settings.mdx#4-backing-locations--dedicated-repo--google-drive--dropbox)).
But when **Git itself is the backbone** — a **dedicated** LFB repo whose entire purpose is to carry this
text (the Personal or a company `_large_files_bridge` repo,
[storage_personal.mdx §1](./storage_personal.mdx#1-anatomy-of-the-personal-storage),
[git_backbone.mdx](./git_backbone.mdx)) — the SDL's small YAML **is committed** (SyncList, descriptor, mapped
dirs, fingerprints, manifest, bookmarks, `devices/`, `analysis/`), because Git is the vehicle that makes
it travel; only the **big-file bytes** are git-ignored, and those never live in the SDL. The git-ignore
rule keeps *bytes* out of Git, not the metadata whose whole job is to ride Git to the user's other
computers.

Composing the SDL — writing the descriptor, the mapped-dirs list, the fingerprint index, the device registry, and the analysis/compression/bookmark artifacts under one storage root — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureStorage()` / `readDescriptor()` / `writeDescriptor()`, and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` — `indexStorageFiles()` (the `.lfbridge/` constant `LFBRIDGE_DIR`).

## 3. Mapped source directories — one storage, many hierarchies (LOCKED)

A company (or personal) storage does **not** require the user to move their big files into one folder.
Instead, in the storage's settings the user lists **the directories of their hard drive that belong to
this storage**, and **everything recursive under each is in scope**. This is the load-bearing model the
charter's company-storage use case describes: *"their company has eight different directory hierarchies
in my file system … they add those to that company's storage settings, and then LargeFileBridge scans
them for the big files."*

* **A mapped directory is a source root.** Its whole subtree (recursively) is part of the storage. LFB
  scans each for big files, tracks them, pins the ones the user opts into over IPFS, and stores their
  derived artifacts in the SDL — **without relocating them**.
* **A storage maps a LIST of them.** Seven mapped directories → one company storage covering all seven
  subtrees. They can be anywhere on disk and need not share a parent.
* **The mapped list is shared; the local paths are per-device.** The *logical* list of mapped
  directories lives in the SDL (`.lfbridge/mapped_dirs.yaml`, below) so every computer agrees on the
  set. But the **absolute local path** of each mapped directory differs per computer — that per-device
  resolution is the **graft**, owned by [devices.mdx](./devices.mdx). `mapped_dirs.yaml` records a
  stable **key** + a **canonical/label** for each; each device's `devices/<device>.yaml` maps that key
  to its own absolute path.

```yaml
# <SDL>/.lfbridge/mapped_dirs.yaml — the source hierarchies this storage covers (shared, travels)
schema_version: 1
mapped:
  - key: film_renders            # stable id, referenced by every device's graft (devices.mdx)
    label: "Film renders"        # human name shown in the UI
    canonical: ~/Movies/act3/renders   # the WRITER's path — advisory only; each device re-roots it
    recursive: true              # the whole subtree is in scope (always true today)
  - key: photo_library
    label: "Photo library"
    canonical: ~/Pictures/act3
  - key: model_weights
    label: "Model weights"
    canonical: ~/BGit/act3/WebApp/ai-service/weights
  # …up to N mapped directories — a company with seven hierarchies has seven entries…
```

* `canonical` is the path **on the computer that added the mapping** — it is **advisory only**, exactly
  like `identity.path` in [sync_list.mdx §4](./sync_list.mdx#4-what-travels-vs-what-stays-local). A
  reader **never** trusts it; it resolves the mapped directory to *this* machine's path through the
  device graft (devices.mdx). A mapped directory that has no graft on this computer is shown as
  **unmapped here** (known, but not present on this box) rather than erroring.
* Big files inside a mapped directory are tracked with **relative paths** (relative to the mapped
  directory root), so `film_renders/shot_012/final_4k.mp4` resolves under whatever absolute path this
  device grafts `film_renders` to.

Where this sits relative to the two existing sync units: a **repo** storage's mapped directory is the
repo working tree (one hierarchy, path-resolved by Git). A **company/personal** storage generalizes
that to a **list** of hierarchies, path-resolved by the device graft instead of by Git. Both feed the
same manifest/fingerprint machinery.

Reading and writing the mapped-directory list, and adding/removing a mapped source directory from a storage's settings, is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getMappedDirs()` / `setMappedDirs()` (new; the settings form adds rows, the scanner reads them to know which subtrees to walk for this storage).

## 4. Derived artifacts — generate once, reuse everywhere (LOCKED value)

The most valuable thing in the SDL, after the CID list, is the pile of **derived artifacts**: things
LFB (or an AI pass) computed *from* a big file that are **expensive to produce** and therefore must be
**produced once and reused** across every computer and every teammate. The SDL is exactly the place to
keep them, because it travels and it does not disturb the natural directory the source file lives in.

> **The economics rule (LOCKED).** A derived artifact is keyed to its source file's **fingerprint**
> ([storages.mdx §4.1](./storages.mdx#41-per-file-tracking-metadata-charter)). If the fingerprint is
> unchanged, the artifact is **never regenerated** — not on this computer, not on another computer that
> pulls the SDL, not for a teammate on a shared company storage. Transcription and vision passes cost
> real money and time; paying for them more than once for the same bytes is a defect.

The artifact kinds:

### 4.1 Transcript · Description · Visuals-by-time
These three are defined in [storages.mdx §6](./storages.mdx#6-per-file-media-analysis-transcription--description--visuals-by-time)
and not restated here — speech-to-text; a prose summary (video) / contents (image); and a time × visual ×
story timeline (video). **Transcript and description live inside the owning storage's `.lfbridge/`,
path-mirrored, with the media's full filename and the extension APPENDED** —
[Transcribe.mdx §3](./Transcribe.mdx), [ai_description.mdx §2](./ai_description.mdx); **visuals-by-time**
stays as a YAML **under `.lfbridge/analysis/<relpath>/`**. All of them travel with the storage so a second
computer, or a teammate, gets them for free.

### 4.2 Path-mirrored under `.lfbridge/`, extension APPENDED, for transcript & description (LOCKED)
The transcript and AI description are written **inside the owning storage/repo's `.lfbridge/` tracking
directory**, in a **path-mirrored hierarchy**, under the media's **full filename (original extension kept)**
with the artifact extension **appended** as a second extension:
`<mapped>/TheFunnyVideo.mp4` → `.lfbridge/<mapped>/TheFunnyVideo.mp4.transcription` /
`.lfbridge/<mapped>/TheFunnyVideo.mp4.ai_description`. This is the product owner's explicit choice — no
`.transcribe/` and **no** `.lfbridge/analysis/` directory for these two; the artifact path is computed
deterministically from the media path (`join(root, ".lfbridge", relDir, name.ext + EXT)`). Because the
extension is **appended** rather than replacing the original, `TheFunnyVideo.mp4` and `TheFunnyVideo.mov`
**no longer collide**. It **travels**: `.lfbridge/` is committed and carried by git in every placement
case, so no `*.transcription` / `*.ai_description` git-ignore line is injected anymore (that nudge is
**retired** — the small text is committed; the multi-GB media stays git-ignored and rides IPFS).
(Visuals-by-time and the compression record remain in the SDL under `.lfbridge/analysis/` — §4.1.)

**Which SDL a loose file's artifact belongs to.** A media file usually lives out under a **mapped source
directory** (§3), not inside the storage's own root. Its artifact is still routed to the storage that
**owns** it: the most specific company/personal storage whose mapped directory (resolved through the
device **graft**, [devices.mdx §4](./devices.mdx)) contains the file, or — for anything under `~` that no
mapped dir claims — the **Personal** storage as the default owner. When that owning storage's **dedicated
Git repo** is enabled (storage_settings.mdx §4.1), the artifact lands **inside that repo**, mirroring the
file's hierarchy and **not** git-ignored. This ownership→placement rule is authoritative in
[Transcribe.mdx §3.4](./Transcribe.mdx) and is what keeps transcripts/descriptions from scattering beside
the media.

### 4.3 Compression records — original identity + before/after
When LFB compresses a file (only ever on explicit user request — charter), it records **what the file
was before** so the change is legible on every computer and reversible in intent. The compression record
is **Category-B tracking state**, so its always-home is **Local Storage** at
`~/T/_large_files_bridge/repos/<repoKey>/analysis/<rel>/compression.yaml` — never a working repo's
`.lfbridge/` — and it travels via the owning storage's **sync repo** (mirrored to
`<syncRepo>/repos/<repoKey>/analysis/<rel>/compression.yaml`) when enabled
([artifact_placement_policy.mdx](./artifact_placement_policy.mdx), [compression.mdx §8](./compression.mdx)):

```yaml
# ~/T/_large_files_bridge/repos/<repoKey>/analysis/film_renders/shot_012/final.mp4/compression.yaml
#   (+ mirrored to <syncRepo>/repos/<repoKey>/analysis/... when the sync repo is enabled)
source: film_renders/shot_012/final.mp4    # the CURRENT (compressed) path, relative to its mapped dir
original:
  name: final.mov                          # the ORIGINAL filename + extension before compression
  extension: mov
  size: 734003200                          # bytes BEFORE
compressed:
  codec: h264                              # the codec we compressed to (charter: X-safe, never AV1)
  size: 118489088                          # bytes AFTER
  ratio: 0.161                             # after / before
  at: 2026-07-05T10:09:00Z
```

* A second computer reading the SDL knows this file was compressed, from what, and by how much —
  without re-deriving anything. The **`text files … which files/paths have been compressed and what
  their original file name and extension were before … the file size before and after"** the charter's
  company use case asks for **are** these compression records.
* This complements the per-file `files.yaml` compressed/uncompressed **variant hashes**
  ([storages.mdx §4.1](./storages.mdx)); the fingerprints prove identity, the compression record
  narrates the human story (old name, old size, new size).

### 4.4 Bookmarks travel with the storage
A user can **bookmark** (star) any tracked file. Bookmarks are a property of the *storage*, not of the
computer, so they live in the SDL and **come across in the YAML** to every machine that carries it:

```yaml
# <SDL>/.lfbridge/bookmarks.yaml
schema_version: 1
bookmarked:
  - film_renders/shot_012/final.mp4
  - photo_library/hero_8k.png
```

This is the source of truth behind the **Bookmarked-first** default sort in every table
([tables.mdx §4](./tables.mdx), [storage_local.mdx §4](./storage_local.mdx) `tables.views`). A file
bookmarked on the laptop shows bookmarked on the studio machine after the next pin pass.

Caching every derived artifact against the source file's fingerprint (skip regeneration when the fingerprint is unchanged), and reading/writing the compression record and the bookmarks list, is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/analysis.service.ts` — `analyzeFile()` (fingerprint-keyed skip), with `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` — `readStorageIndex()` supplying the fingerprint, and a new `bookmarks` accessor in `storage.service.ts`.

## 5. End-to-end walkthrough — four computers, one company storage

The charter's canonical use case, told as an SDL story. A user with **four computers** and a company
with **seven mapped directories**:

1. **Computer 1 maps the seven directories.** In the ACT3 company storage's settings the user adds
   their seven directory hierarchies ([storage_settings.mdx](./storage_settings.mdx) + §3 here). LFB
   writes `mapped_dirs.yaml` into the SDL (`act3_large_files_bridge/`) and this computer's graft into
   `devices/computer-1.yaml` (devices.mdx). A scan of the seven subtrees finds **30 big files**.
2. **The user pins some to IPFS.** They click to add the 30 big files to IPFS. LFB pins them locally,
   records each CID in `manifest.yaml`, and marks this computer in `pinned_by`. (Some the user may
   instead have `ipfs pin add`-ed by hand — both land in the same pinset and are reconciled;
   [storage_local.mdx §9.5](./storage_local.mdx#95-the-ipfs-pinset-is-the-source-of-truth--pinned_by-is-a-stale-cache).)
3. **They transcribe a third.** For ~10 of the videos they click **Transcribe**; LFB writes a
   `.lfbridge/<rel-dir>/<name.ext>.transcription` artifact — path-mirrored under the dedicated repo's
   `.lfbridge/`, full filename with `.transcription` **appended** (§4.1, §4.2).
4. **They get AI video descriptions.** For the same or other videos they click to generate descriptions
   / visuals-by-time; the description lands as a `.lfbridge/<rel-dir>/<name.ext>.ai_description` artifact
   (extension appended), and visuals-by-time as `visuals_by_time.yaml` under `.lfbridge/analysis/`. Because
   a **dedicated Git repo** is configured for this company (`act3_large_files_bridge`), all of this lives in
   that repo's committed `.lfbridge/` and is carried.
5. **Compression records + transcriptions are stored in matching paths.** Any file the user compresses
   gets its `compression.yaml` (§4.3); every transcription/description sits under a `.lfbridge/` path that
   **mirrors the file's location within its mapped directory** — so the SDL's `.lfbridge/` tree reads like a
   shadow of the mapped hierarchy.
6. **Everything is committed to the SDL and pushed.** The small text (`SyncList`, descriptor, mapped
   dirs, fingerprints, manifest, devices, analysis, bookmarks) is committed to the
   `act3_large_files_bridge` repo and pushed. The big bytes never enter Git — they move over IPFS.
7. **Computer 2 just adds the Git repo.** On a second machine the user clones `act3_large_files_bridge`
   and points LFB at it (or LFB discovers it). The SDL **is carried in over the git backbone**. Now this computer knows,
   from the YAML alone:
   * **which files exist and their CIDs** — so it can fetch + `ipfs pin` down exactly the ones marked,
     to the **correct local locations** via its own device graft (devices.mdx);
   * **which files were pinned** (via the app or by hand) and by which computers (`pinned_by`);
   * **which were compressed** (and from what, to what) — no re-compression;
   * **which have transcriptions and AI descriptions** — no regeneration, no repeat cost (§4);
   * **which are bookmarked** — starred here too (§4.4).
   Computers 3 and 4 do the same. The pin pass makes **all** of this happen — pin the right
   files to the right places, and inherit every derived artifact — from the SDL's small YAML.

The pin pass that reconciles the incoming SDL against this computer's reality — fetching/pinning the marked CIDs to their grafted local paths and inheriting the derived artifacts without regenerating them — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll()` (per-unit `runUnitPin`), applying the trust rules of [sync_list.mdx §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule).

## 6. Two homes for a repo's tracking state — reconciled with the mapped-directory model (LOCKED)

Sections §1–§5 describe a **company/Personal** storage's SDL: its own root (`_large_files_bridge/`), its
`.lfbridge/`, and the **mapped source directories** whose big files it tracks in place. This section
reconciles that with a distinct question the storages family raised: **where does an individual Git
*repo*'s own Category-B tracking state live** — the `repo_storage.yaml`, the `files.yaml` fingerprint
index, the per-file sidecars, `decisions.yaml`, and `manifest.yaml` — given that a working repo must
**not** carry LFB's churny state in its committed history?

A repo's Category-B tracking state has **two homes** (additive), and a working repo is never one of them.
The umbrella and placement policy are authoritative elsewhere
([storages.mdx §8.1](./storages.mdx#81-the-churn-problem-and-the-two-homes-for-a-repos-tracking-state-locked),
[artifact_placement_policy.mdx](./artifact_placement_policy.mdx)); this spec locks how each relates to the
SDL model:

| Home | Path | Is it an SDL? | Travels? | When |
| --- | --- | --- | --- | --- |
| **Local Storage** (machine-local) | `~/T/_large_files_bridge/repos/<repoKey>/` | **No** — the non-syncing state root ([storage_local.mdx](./storage_local.mdx), the §1 table's right column) | **No** — the authoritative per-machine working copy | **Always** — every scan/action reads & writes here first |
| **The owning storage's sync repo** | `<syncRepo>/repos/<repoKey>/` | **Yes** — a *separate* SDL that carries **other repos'** state | **Yes**, via that repo's git backbone | **Also**, when the repo's **owner** (company/Personal) has **configured a sync repo** and the per-repo toggle is on ([storage_settings.mdx §4b](./storage_settings.mdx#4b-the-state-sync-repo--configuring-where-owned-repos-tracking-state-travels)) — default off |

The **only** thing that ever lives committed in the working repo is the repo storage's **Category-A**
derived content — its transcripts / AI descriptions in `<repoRoot>/.lfbridge/` (a repo storage **is** its
own SDL for *that* content, §1 consequence 1). Two clarifications this locks against the rest of the spec:

* **Local Storage `repos/<repoKey>/` is NOT an SDL.** It is exactly the machine-local, non-syncing
  bookkeeping the §1 table's right column names — it never travels by itself; the **sync repo** is what
  carries it between computers. It is the **always** home for a repo's Category-B tracking state; the
  earlier content-threshold three-tier model is **retired**.
* **The sync repo IS an SDL — but a shared, cross-repo one.** Unlike a mapped-directory SDL (which carries
  **its own** storage's metadata, §2) or a repo's own `.lfbridge/` (which carries **that repo's**
  Category-A content), a sync repo mirrors the Category-B tracking state of **many other repos**, each
  namespaced under `repos/<repoKey>/`. It is a first-class SDL by §1's definition — small text that travels
  over a git backbone — just one whose contents are keyed by `repoKey` rather than by mapped-directory `key`.

```
# the sync repo is an SDL whose payload is other repos' Category-B tracking state, keyed by <repoKey>
<syncRepo>/                                  ← e.g. ~/BGit/Bryan_git/act3_state_sync/  (an SDL, §1)
├── .lfbridge/storage.yaml                    ← its own descriptor (type: company or personal)
└── repos/
    ├── act3-webapp/                           ← <repoKey> namespace — NOT this SDL's mapped dirs
    │   ├── repo_storage.yaml                  ←   the owned repo's repo-wide facts (repo_tracking_scheme.mdx §2)
    │   ├── files.yaml                         ←   the owned repo's fingerprint index (storages.mdx §4.1)
    │   ├── decisions.yaml · manifest.yaml     ←   union-merged travellers (decisions.mdx)
    │   └── <rel-dir>/<name.ext>.transcription ←   Category-A artifacts routed here by the repo's placement radio (§7)
    └── act3-ai-service/
        └── repo_storage.yaml
```

Which homes a given repo's tracking state uses — the always Local-Storage root and the optional additive sync-repo mirror — is resolved in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveTrackingRoot(repoRoot)` (returns the Local-Storage `repos/<repoKey>/` root, drawn from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/state-dir.ts` — `resolveRepoStateDir()`) and `resolveStateSyncRepo(repoRoot)` (reads the owner's configured sync repo via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getStateSyncRepo()`; returns `<syncRepo>/repos/<repoKey>/` or `null`); the mirror is performed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo()` / `reconcileFromSyncRepo()`.

## 7. Owner mapping routes tracking state to the owning storage's SDL

The sync-repo home (§6) begs a question: **whose** sync repo? The answer is the repo's **owner** — the
company or Personal storage every repo maps to. That mapping is owned end-to-end by
[repo_company_mapping.mdx](./repo_company_mapping.mdx) (default **Personal**; a forge **ORG** in the git
remote → auto **company**; renamable + reassignable) and is **not** redefined here. Its consequence for
the syncable-data-location model **is**:

* **The owner is which SDL a repo's tracking state travels through.** A **Personal-owned** repo mirrors its
  Category-B tracking state (when a sync repo is configured and the toggle is on) to the
  **[Personal storage](./storage_personal.mdx)**'s sync repo; a **company-owned** repo mirrors to that
  **[company storage](./storage_company.mdx)**'s sync repo, under the company allow-list.
* **This is the same owner rule §4.2 already uses for loose artifacts.** [§4.2](#42-path-mirrored-under-lfbridge-extension-appended-for-transcript--description-locked)
  routes a **loose media file's** transcript/description to the storage that **owns** it (most-specific
  mapped directory, else Personal). The owner mapping generalizes that to a **whole repo's** tracking
  state: the repo's owner storage is the SDL its state belongs to. The two rules are the same idea applied
  at file-granularity vs. repo-granularity.
* **Placement radios pick which class travels via the sync repo.** A repo's transcripts / AI
  descriptions (Category A) follow the per-repo **placement radio** ([repo_settings.mdx](./repo_settings.mdx),
  [placement_radios.mdx](./placement_radios.mdx)): `.lfbridge/`, **beside the file**, or **the sync repo**.
  When set to the sync repo, those artifacts travel with the **owner's** SDL
  (`<syncRepo>/repos/<repoKey>/…`, extension appended) rather than with the working repo — the same
  path-mirroring rule as §4.2, just rooted at the sync repo.

Where each class of a repo's data lives and travels, once the owner and threshold are known:

| Data class | Home | Travels via |
| --- | --- | --- |
| **Big-file bytes** | in place, under the repo working tree (git-ignored) | **IPFS** (never an SDL) |
| **Category-B tracking state** (`repo_storage.yaml`, `files.yaml`, sidecars, history, `decisions.yaml`, `manifest.yaml`, compression records) | **Local Storage** `~/T/_large_files_bridge/repos/<repoKey>/` **always** (never the working repo) | **does not travel by itself**; mirrored to `<syncRepo>/repos/<repoKey>/` and carried by the **owner storage's** git backbone when a sync repo is enabled |
| **Transcripts / AI descriptions** (Category A) | per the **placement radio** — the working repo's `.lfbridge/`, beside the file, or `<syncRepo>/repos/<repoKey>/` | with whichever of those homes was chosen (the working repo's git, or the owner's sync-repo git) |
| **A company/Personal storage's own metadata** (descriptor, mapped dirs, fingerprints of mapped files, bookmarks) | the storage's **own** SDL (`_large_files_bridge/.lfbridge/`, §2) | the storage's git backbone |

Resolving the owner of a repo (to pick its sync repo) reuses the mapping the storages model already computes from the remote's forge org, defaulting to Personal, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `classifyByConvention()` / `buildRow()` (sets `companyName`); the authoritative repo→owner mapping it consumes is owned by [repo_company_mapping.mdx](./repo_company_mapping.mdx).

## 8. Acceptance criteria

1. **SDL is a named, first-class concept:** the directory holding a storage's `.lfbridge/` + Sync List,
   which **travels** between the user's computers (and teammates for shared storages). It holds
   **metadata + derived artifacts, never the primary big-file bytes** (§1). It is distinct from the
   machine-local, non-syncing state root ([storage_local.mdx](./storage_local.mdx)).
2. A storage's big files stay in their **natural directories**; a company/personal storage maps a
   **list** of source directory hierarchies (`mapped_dirs.yaml`), each **recursive**, resolved to
   per-computer absolute paths by the device **graft** ([devices.mdx](./devices.mdx)) — never by
   trusting the writer's `canonical` path (§3).
3. A **company/Personal storage's own SDL** holds the LOCKED YAML inventory of §2: SyncList, descriptor,
   mapped dirs, fingerprint index, manifest, devices, analysis, bookmarks — all small text; under a cloud
   backbone only the SyncList is committed to Git and the rest of `.lfbridge/` is git-ignored yet travels
   via the mirror, while under a dedicated git backbone the whole inventory is committed (§2). A **working
   code repo** commits only its Category-A content; its Category-B tracking state stays in Local Storage /
   the sync repo (§6, [artifact_placement_policy.mdx](./artifact_placement_policy.mdx)).
4. **Derived artifacts are generated once and reused everywhere**, keyed to the source fingerprint and
   never regenerated when it is unchanged — transcript, description, visuals-by-time (§4.1), the
   **compression record** carrying the original name/extension + size before/after (§4.3), and
   **bookmarks** that travel with the storage (§4.4).
5. A second computer that merely **adds the SDL's Git repo** learns from the YAML alone which files to
   fetch/pin and where (via its graft), which were pinned/compressed/transcribed/described, and which
   are bookmarked — with **no** re-pinning to the wrong place and **no** regeneration of paid artifacts
   (§5).
6. A **repo**'s Category-B tracking state lives in **two homes** (§6), never the working repo: the
   machine-local **state root** `~/T/_large_files_bridge/repos/<repoKey>/` **always** (**not** an SDL, the
   authoritative per-machine copy that does not travel by itself), and — **also**, when the owner
   configured a **sync repo** and the per-repo toggle is on — the owning storage's **sync repo**
   `<syncRepo>/repos/<repoKey>/` (a **shared, cross-repo SDL** keyed by `repoKey`, the travelling mirror).
   The earlier content-threshold three-tier model is retired; the placement policy is cross-linked, not
   redefined ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). Only the repo's
   **Category-A** derived content (transcripts / AI descriptions) is ever committed in the working repo's
   own `.lfbridge/`.
7. The sync repo **is** an SDL by §1's definition (small text over a git backbone) but a distinct one: it
   mirrors **other repos'** Category-B tracking state under `repos/<repoKey>/`, not its own mapped-directory
   metadata; the state root's `repos/<repoKey>/` is **not** an SDL (it is the machine-local, non-syncing
   root of the §1 table's right column, and the **always** home for that tracking state).
8. Every repo's **owner** (company or Personal, default Personal; forge ORG → company —
   [repo_company_mapping.mdx](./repo_company_mapping.mdx)) is **which** SDL its tracking state mirrors
   through (§7) — the same owner rule §4.2 already uses to route a loose file's transcript/description,
   applied at repo granularity. **Placement radios** ([repo_settings.mdx](./repo_settings.mdx),
   [placement_radios.mdx](./placement_radios.mdx)) pick whether a repo's transcripts / AI descriptions
   travel via the sync repo, `.lfbridge/`, or beside the file.
