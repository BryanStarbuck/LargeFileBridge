# LargeFileBridge ‚Äî View one file (image / video / audio viewer)

`LargeFileBridge` shows a file in many tables and browsers. When that file is a **playable/viewable
medium**, clicking its row/cell/menu opens a **viewer-first** page ‚Äî the **View one file** experience ‚Äî
instead of the properties page:

* an **image** file ‚Üí **`/image?path=<abs>`** (the image viewer; large-scale, zoom-to-fit),
* a **video** file ‚Üí **`/video?path=<abs>`** (the video viewer; click-to-play, full controls),
* an **audio** file (MP3/WAV/FLAC/‚Ä¶) ‚Üí **`/audio?path=<abs>`** (the audio player; click-to-play).

All three are the same page with a different centerpiece widget, so they share one layout (`MediaViewer`).
The generic, properties-first **[View one file](./files.mdx)** page (`/file`) remains the fallback for
everything that is not a viewable medium (and is always reachable from a viewer via **View properties**).

**How you get here.** Every large-file table routes a row click to this page by the file's kind
([one_repo.mdx ¬ß4.7](./one_repo.mdx), [full_paths.mdx](./full_paths.mdx)), the File System browser routes
a cell click the same way ([directories.mdx](./directories.mdx)), and the entity menus offer **View
image / View video / View audio** ([menus.mdx ¬ß5.3](./menus.mdx#53-file)). The single shared rule that
picks the route from a filename is `viewerRouteForName` in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/media.ts`.

Siblings this builds on and does not repeat:

  fact cards, and the top-right `‚ãÆ` more menu. The viewer pages reuse that same `EntityView` payload and

## 1. Why stream (no `file://`)

The media lives on the user's **local disk**, but the web app is served over `http://localhost` (and,
later, over a real server). A page loaded over http **cannot** load `file:///‚Ä¶` sub-resources ‚Äî browsers
block it ‚Äî so a bare `<img src="file://‚Ä¶">` / `<video src="file://‚Ä¶">` will not play. Even where it might,
it would leak absolute-path structure and bypass our allow-list.

So we **stream the bytes through the backend** on the same origin as the app:

* An **authenticated grant** endpoint mints a short-lived, HMAC-signed URL for one absolute path.
* The `<img>` / `<video>` element loads that URL from **`/api/media/raw`**, which the web dev server
  proxies to the backend (same origin in production). The raw endpoint honors **HTTP `Range`** so video
  seeks/streams instead of downloading whole.

This keeps the charter's promises: **no anonymous access** (every grant is minted for an allow-listed
session), and **no public gateway/relay** ‚Äî this is ordinary same-origin file serving to the signed-in
user's own browser, unrelated to IPFS gateway behavior (charter: IPFS no-relay policy).

## 2. The backend contract (`/api/media`)

Three endpoints in one module. Two are allow-list-gated like every data route; the third is gated by the
signed token so a plain media element (which cannot send a `Bearer` header) can still load.

```
GET /api/media/grant?path=<abs>     (allow-listed)  ‚Üí { url: "/api/media/raw?path=<abs>&e=<expMs>&t=<hmac>" }
GET /api/media/probe?path=<abs>     (allow-listed)  ‚Üí MediaProbe
GET /api/media/raw?path=&e=&t=       (token-gated)   ‚Üí the bytes, with Range support
```

**Grant.** `t = HMAC-SHA256( `path`\n`expMs`, MEDIA_SECRET )`, hex. `MEDIA_SECRET` is a random 32-byte
key minted once per backend process (tokens naturally invalidate on restart ‚Äî acceptable for a viewer).
Default expiry ‚âà 6h so a long viewing session with re-buffers keeps working.

Minting and verifying the short-lived HMAC-signed grant is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.service.ts` ‚Äî `mintGrant` / `verifyGrant` functions

**Raw.** Re-derives + constant-time-compares the HMAC, checks `expMs` is in the future, re-stats the path
(must be an existing **regular file** ‚Äî never a directory), sets `Content-Type` from a small extension‚ÜíMIME
map, `Accept-Ranges: bytes`, `Content-Disposition: inline`, and:

* No `Range` ‚Üí `200` with `Content-Length`, streams the whole file.
* `Range: bytes=a-b` ‚Üí `206 Partial Content` with `Content-Range`, `Content-Length` of the slice, streams
  just that slice (`fs.createReadStream(abs, { start, end })`).
* Unsatisfiable range ‚Üí `416`.

Verifying the token, parsing the HTTP `Range`, and streaming the byte slice (200 / 206 / 416) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.router.ts` ‚Äî the `GET /raw` route (with `parseRange` from `media.service.ts`)

**Probe** (`MediaProbe`, `@lfb/shared`). A **best-effort, local-only, no-shell** sniff (charter forbids
shell/ffprobe). It reads a bounded head (and, for videos, a bounded tail) and reports what it can:

```ts
interface MediaProbe {
  kind: "image" | "video" | "other"; // from extension family
  container: string | null;          // e.g. "MP4", "QuickTime", "Matroska", "WebM", "PNG", "JPEG"
  codec: string | null;              // e.g. "H.264", "HEVC", "AV1", "VP9", "ProRes", "PNG (lossless)"
  width: number | null;              // pixels ‚Äî parsed for images; videos when cheaply available
  height: number | null;
  compressState: "should" | "done" | null; // mirrors the C/c heuristic (badges.ts)
}
```

* **Images** ‚Äî dimensions parsed from the header (PNG IHDR, JPEG SOFn, GIF logical screen, WebP VP8X/VP8).
  `codec` is the format family; `compressState` mirrors `compressInfo` (PNG/BMP/TIFF/GIF ‚Üí `should`,
  JPEG/WebP/HEIC/AVIF ‚Üí `done`).
* **Video** ‚Äî `container` from extension; `codec` sniffed by scanning the bounded buffer for the sample
  entry fourcc (ISO-BMFF `avc1`/`hvc1`/`hev1`/`av01`/`vp09`/`ap4h`‚Ä¶; Matroska/WebM `V_MPEG4/ISO/AVC`,
  `V_MPEGH/ISO/HEVC`, `V_AV1`, `V_VP9`‚Ä¶). When the moov/tracks lie outside the bounded window, `codec` is
  `null` (honest unknown). Duration is **not** computed (needs a full parse) and is omitted.

Any field we cannot determine is `null`; the UI degrades to "‚Äî". The probe never blocks the viewer ‚Äî the
media loads from the grant URL regardless of what probe returns.

The local, no-shell container/codec/dimensions sniff (bounded header + tail read) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.service.ts` ‚Äî `probeMedia` function

## 3. Layout (shared `MediaViewer`) ‚Äî the vertical stack

The media-viewer routes (`/image`, `/video`, `/audio`) run **full-width** ‚Äî the app shell drops its
normal `max-w-6xl` centered column for these three routes so the content uses the **whole** width next
to the left bar, with no side gutters. This gives the action bar (¬ß4) the entire width to lay out
buttons before folding any into **More**. Widening only the viewer routes is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/app/AppShell.tsx` ‚Äî `AppShell` (`max-w-none` for `/image` ¬∑ `/video` ¬∑ `/audio`, `max-w-6xl` otherwise)

One column, **six bands stacked top ‚Üí bottom**. The order is deliberate: the small identifying text
is at the top, the **media is the star in the middle**, and the analysis panels sit at the bottom. The
bands, top to bottom:

1. **Full path** ‚Äî the file's absolute path in small, muted, monospaced text. This is the very top line.
2. **File name** ‚Äî the title of the page, in **large** text (it is the biggest text on the page).
3. **Action bar** ‚Äî the **full-width overflow action bar** (¬ß4): as many action buttons as FIT
   (IPFS primary ¬∑ Compress ¬∑ Open: in browser ¬∑ on Mac ¬∑ Open folder: in browser ¬∑ on Mac ¬∑
   Transcribe ¬∑ Copy path ¬∑ View properties ¬∑ Move), with anything that doesn't fit ‚Äî plus the sticky
   flag/danger extras ‚Äî folded into a trailing **More** menu. The top-right `‚ãÆ` still carries the
   complete File catalog.
4. **Property grid** ‚Äî **~4‚Äì5 columns wide √ó 2‚Äì3 rows tall** of `Label : value` pairs (¬ß3.1). It is
   kept short (few rows) on purpose so it sits neatly above a 16:9 medium without crowding it out.
5. **Media surface** ‚Äî the **star**: a **bounded stage** (`~58vh`, min `340px`). The image/video/audio

   bounded (not fill-to-viewport) precisely because band 6 sits beneath it ‚Äî the "content underneath"

6. **Analysis panels** ‚Äî the **AI description ¬∑ Transcription** panels shown **side by side** (¬ß6): AI

## 4. The action bar (band 3) ‚Äî a full-width, priority-ordered overflow row

Band 3 is a **full-width overflow action bar** ‚Äî the same operations a user reaches by right-clicking
the file or opening its `‚ãÆ` kebab, promoted to visible buttons. **Think of them as the file's
context-menu items laid out as buttons.** Because the viewer runs full-width (¬ß3), the bar has the whole
column to work with; it shows **as many buttons as fit** and folds the rest into a trailing **More**
menu. The set is customized by media type (image vs. audio vs. video), and the top-right `‚ãÆ` still
carries the complete catalog.

### 4.1 Overflow & priority (how the bar decides what to show)

Every action carries a **priority**. When the row is too narrow for all of them, the **lowest-priority**
items overflow into **More** first, so the most important actions stay visible the longest. A hidden
mirror row measures each button's natural width off-screen and a `ResizeObserver` tracks the available
width, so the fit is recomputed whenever the window (or the button set) changes ‚Äî no layout thrash. The
priority order, highest ‚Üí lowest: **IPFS primary** (100) ¬∑ **Compress** (90) ¬∑ **Open** (85) ¬∑
**Open folder** (75) ¬∑ **Transcribe** (60) ¬∑ **Copy path** (40) ¬∑ **View properties** (30) ¬∑ **Move**
(20). The trailing **More** menu always holds the overflow **plus** the sticky **extras** that are never
buttons: **Never publish via IPFS** (flag toggle), **Do not compress** (flag toggle), **Copy CID**
(when a CID exists), and **Delete‚Ä¶** (danger). "Open raw" is no longer its own button ‚Äî it is the
**Open: in browser** half of the Open compound (¬ß4.2).

### 4.5 Move & Delete ‚Äî the backend contract (explicit, guarded, recoverable)

Move and Delete are the only actions here that change the file on disk, so they are **explicit-click,
confirm-gated**, and **guarded** ‚Äî a direct honoring of the charter ("never alter a file unless the user
explicitly asks"; "never destroy bytes silently").

```
POST /api/entity/move    { path: <abs>, dest: <abs> }   ‚Üí { moved: true, path: <newAbs> }
POST /api/entity/delete  { path: <abs> }                ‚Üí { trashed: true, restorePath: <abs> }
```

* **Move** validates that `path` is an existing regular file and `dest` is absolute, its parent directory
  exists, and it does not overwrite an existing file; then `fs.renameSync`. The viewer re-points to the
  new path on success.
* **Delete** does **not** `unlink`. It moves the file into LFBridge's **trash** under the state directory
  (`<stateDir>/trash/<timestamp>__<name>`) so it is **recoverable**, and returns the restore path. The UI
  confirms first ("Move to LFBridge trash? ‚Äî recoverable").

Moving (guarded rename) and deleting (recoverable move-to-trash, never `unlink`) a single file are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/entity/entity.service.ts` ‚Äî `moveEntity` / `deleteEntity` functions (routed by `entity.router.ts` `POST /move` ¬∑ `POST /delete`)

## 5. Empty & error states

* **Not found / moved** (`exists:false`): the shared "no longer at that path" card (files.mdx ¬ß5) ‚Äî no
  viewer surface.
* **Wrong route for the kind:** the viewer routes by the file's **actual kind** (from its name, via
  `mediaKindForName`), so `/video` opened on an image redirects to `/image`, an audio file to `/audio`,
  and a non-media file to `/file` ‚Äî and likewise for every other mismatch. A correct link never lands on
  the wrong viewer. The redirect **replaces** history (does not push) so the wrong-kind URL never lingers
  to re-bounce on Back. (Note: audio is not a "compressible" kind on `EntityView`, so the redirect keys off
  the **name-based** kind, not `compressible`, or an audio file would wrongly fall through to `/file`.)
* **Decode/playback failure ‚Üí automatic backend stream (no user conversion).** When the `<video>`
  element can't decode the raw bytes (an unsupported codec ‚Äî HEVC/ProRes/MPEG-4-ASP, an "H.264" that is
  really 10-bit/4:2:2, an MP4 with AC-3 audio or `moov` at the tail, a `.mkv`/`.avi` container), the
  viewer does **not** dead-end. It silently re-points the same `<video>` to **`/api/media/stream`**, which
  the backend transcodes/remuxes into a browser-safe **fragmented MP4 (H.264 ¬∑ yuv420p ¬∑ AAC)** on the fly
  ‚Äî so it just plays. A small **"Compatibility mode ‚Äî streaming a converted copy (seeking may be
  limited)"** badge marks this. The full codec matrix, the "why H.264 still fails" causes, and the
  remux-vs-transcode decision tree live in **[codecs.mdx](./codecs.mdx)** (¬ß2, ¬ß6). Only if the backend
  stream **also** fails does the inline notice appear ‚Äî "This browser can't play this file's codec
  (`<codec>`)" ‚Äî with **Open raw / hand off to the OS** (the grant URL, which the OS can hand to a native
  app ‚Äî see **Open: on {label}**, ¬ß4.2) and, for video, **Convert to H.264 (MP4)‚Ä¶**: a bottom-of-card
  **offer** to re-encode the file to the universal-safe target that plays everywhere and uploads cleanly
  to TikTok/X/YouTube/Instagram ([codecs.mdx ¬ß5/¬ß9](./codecs.mdx)). Per the charter, conversion is always
  an explicit, confirm-gated offer ‚Äî bytes are never altered on their own.
* **Loading:** skeleton header + action row + a shimmering viewer block; the grant + probe fetch in
  parallel; the media element mounts as soon as the grant URL resolves (probe fills the codec chip later).
* **IPFS node unreachable:** the IPFS button disables with a tooltip (parity with one_repo.mdx ¬ß5); the
  viewer, probe, flags, and property links all still work.

## 7. Acceptance criteria

1. `/image?path=<abs>`, `/video?path=<abs>`, and `/audio?path=<abs>` each render the **shared MediaViewer**
   as the **six-band vertical stack** (¬ß3), top ‚Üí bottom: **full path** ¬∑ **file name (large title)** ¬∑
   **full-width overflow action bar + `‚ãÆ`** ¬∑ **property grid (~4‚Äì5 cols √ó 2‚Äì3 rows)** ¬∑ **bounded media
   stage** ¬∑ **analysis tabs**. The viewer routes run **full-width** (no side gutters); the media
   **auto-sizes to fit** the stage while **keeping aspect ratio**; the page **scrolls** to reveal the
   analysis tabs when needed.
2. The medium **plays/loads off the local disk** via the backend stream ‚Äî never a `file://` URL. Video and
   audio support **seeking** because `/api/media/raw` answers HTTP **`Range`** (`206` + `Content-Range`);
   images load whole. The stream is reachable only through a **short-lived signed grant** minted for an
   allow-listed session; the raw endpoint refuses expired/forged tokens and any non-file path.
3. The **action bar** (band 3) is a **full-width overflow row** (¬ß4): it shows the highest-priority buttons
   that fit and folds the rest ‚Äî plus the sticky extras (**Never publish via IPFS**, **Do not compress**,
   **Copy CID**, **Delete‚Ä¶**) ‚Äî into a trailing **More** menu; the lowest-priority items overflow first.
   **Open** and **Open folder** are compound "in browser ¬∑ on {label}" controls, the **on {label}** half
   present only on localhost ([os_open.mdx](./os_open.mdx)). Image + video additionally show **Compress**;
   audio shows neither Compress nor AI description. The top-right `‚ãÆ` menu renders the full File catalog
   ([menus.mdx ¬ß5.3](./menus.mdx)) verbatim.
4. The **property grid** (band 4) shows the type-appropriate facts: Size ¬∑ Modified ¬∑ Created ¬∑ Repo ¬∑
   Peers ¬∑ Decision ¬∑ Badges for all; **Dimensions + Codec + Compress** for image/video; **Duration** for
   audio. Unknown cells show `‚Äî`; the grid never exceeds ~3 rows.
5. `MediaProbe` is computed **locally with no shell** (charter): image dimensions + format from the header;
   video container from extension and codec by a bounded fourcc/EBML sniff; audio container/codec from the
   extension family; unknown fields are `null` and render as "‚Äî" without breaking the viewer.
6. **Move** renames the file (guarded: absolute dest, existing parent, no overwrite) and the viewer
   re-points to the new path. **Delete** moves the file to LFBridge's **trash** under the state dir
   (recoverable) ‚Äî it **never `unlink`s** ‚Äî after a confirm. Compress stays an explicit-click offer.
7. Error/empty states render per ¬ß5 (not-found card, kind-based wrong-route redirect incl. audio,
   decode-failure notice with Open-raw + Compress offer, loading skeleton, node-unreachable). No bytes are
   ever altered or destroyed without an explicit click + confirm.
8. The **analysis panels** (band 6) show **AI description** and **Transcription** **side by side** ‚Äî AI
   description on the left half, Transcription on the right half, **both visible at once** (no tabs/radio).
   A **video** shows both; an **image** shows only AI description (full width); an **audio** file shows
   only Transcription (full width); on a narrow window they stack. Each half carries a ‚úì when its content
   exists on disk and offers a **Generate** button when empty ‚Äî **Generate description** (a vision
   provider) or **Generate transcript** (local Whisper) ‚Äî each disabled with a clear hint when its
   dependency (an API key / whisper) is absent. See [ai_description.mdx](./ai_description.mdx) and
   [Transcribe.mdx](./Transcribe.mdx).
9. **Open on {Mac/PC/Linux}** is present only on localhost and hands the file/folder to the host OS
   default handler; in server mode (or for a non-loopback caller) the "on {label}" halves are hidden and
   only "in browser" remains ([os_open.mdx](./os_open.mdx)).
