# LargeFileBridge — View one file (image / video / audio viewer)

`LargeFileBridge` shows a file in many tables and browsers. When that file is a **playable/viewable
medium**, clicking its row/cell/menu opens a **viewer-first** page — the **View one file** experience —
instead of the properties page:

* an **image** file → **`/image?path=<abs>`** (the image viewer; large-scale, zoom-to-fit),
* a **video** file → **`/video?path=<abs>`** (the video viewer; click-to-play, full controls),
* an **audio** file (MP3/WAV/FLAC/…) → **`/audio?path=<abs>`** (the audio player; click-to-play).

All three are the same page with a different centerpiece widget, so they share one layout (`MediaViewer`).
The generic, properties-first **[View one file](./files.mdx)** page (`/file`) remains the fallback for
everything that is not a viewable medium (and is always reachable from a viewer via **View properties**).

**How you get here.** Every large-file table routes a row click to this page by the file's kind
([one_repo.mdx §4.7](./one_repo.mdx), [full_paths.mdx](./full_paths.mdx)), the File System browser routes
a cell click the same way ([directories.mdx](./directories.mdx)), and the entity menus offer **View
image / View video / View audio** ([menus.mdx §5.3](./menus.mdx#53-file)). The single shared rule that
picks the route from a filename is `viewerRouteForName` in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/media.ts`.

Siblings this builds on and does not repeat:

* [View one file](./files.mdx) — the entity identity, the two sticky flags, the sync/compression/IPFS
  fact cards, and the top-right `⋮` more menu. The viewer pages reuse that same `EntityView` payload and
  the same `⋮` menu; they only re-front it around a big media surface.
* [Menus](./menus.mdx) — the LOCKED **File action catalog** (§5.3) and the `⋮` more-menu (§4). The
  action-button row here is a *promotion* of a few catalog items to visible chips; it never invents new
  actions.
* [One repo](./one_repo.mdx) — the **decision model** (§1) behind Add/Remove IPFS and the repo chip.
* [Directory](./directories.mdx) — the **code badges** (§3) reused in the property strip.

## 1. Why stream (no `file://`)

The media lives on the user's **local disk**, but the web app is served over `http://localhost` (and,
later, over a real server). A page loaded over http **cannot** load `file:///…` sub-resources — browsers
block it — so a bare `<img src="file://…">` / `<video src="file://…">` will not play. Even where it might,
it would leak absolute-path structure and bypass our allow-list.

So we **stream the bytes through the backend** on the same origin as the app:

* An **authenticated grant** endpoint mints a short-lived, HMAC-signed URL for one absolute path.
* The `<img>` / `<video>` element loads that URL from **`/api/media/raw`**, which the web dev server
  proxies to the backend (same origin in production). The raw endpoint honors **HTTP `Range`** so video
  seeks/streams instead of downloading whole.

This keeps the charter's promises: **no anonymous access** (every grant is minted for an allow-listed
session), and **no public gateway/relay** — this is ordinary same-origin file serving to the signed-in
user's own browser, unrelated to IPFS gateway behavior (charter: IPFS no-relay policy).

## 2. The backend contract (`/api/media`)

Three endpoints in one module. Two are allow-list-gated like every data route; the third is gated by the
signed token so a plain media element (which cannot send a `Bearer` header) can still load.

```
GET /api/media/grant?path=<abs>     (allow-listed)  → { url: "/api/media/raw?path=<abs>&e=<expMs>&t=<hmac>" }
GET /api/media/probe?path=<abs>     (allow-listed)  → MediaProbe
GET /api/media/raw?path=&e=&t=       (token-gated)   → the bytes, with Range support
```

**Grant.** `t = HMAC-SHA256( `path`\n`expMs`, MEDIA_SECRET )`, hex. `MEDIA_SECRET` is a random 32-byte
key minted once per backend process (tokens naturally invalidate on restart — acceptable for a viewer).
Default expiry ≈ 6h so a long viewing session with re-buffers keeps working.

Minting and verifying the short-lived HMAC-signed grant is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.service.ts` — `mintGrant` / `verifyGrant` functions

**Raw.** Re-derives + constant-time-compares the HMAC, checks `expMs` is in the future, re-stats the path
(must be an existing **regular file** — never a directory), sets `Content-Type` from a small extension→MIME
map, `Accept-Ranges: bytes`, `Content-Disposition: inline`, and:

* No `Range` → `200` with `Content-Length`, streams the whole file.
* `Range: bytes=a-b` → `206 Partial Content` with `Content-Range`, `Content-Length` of the slice, streams
  just that slice (`fs.createReadStream(abs, { start, end })`).
* Unsatisfiable range → `416`.

Verifying the token, parsing the HTTP `Range`, and streaming the byte slice (200 / 206 / 416) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.router.ts` — the `GET /raw` route (with `parseRange` from `media.service.ts`)

**Probe** (`MediaProbe`, `@lfb/shared`). A **best-effort, local-only, no-shell** sniff (charter forbids
shell/ffprobe). It reads a bounded head (and, for videos, a bounded tail) and reports what it can:

```ts
interface MediaProbe {
  kind: "image" | "video" | "other"; // from extension family
  container: string | null;          // e.g. "MP4", "QuickTime", "Matroska", "WebM", "PNG", "JPEG"
  codec: string | null;              // e.g. "H.264", "HEVC", "AV1", "VP9", "ProRes", "PNG (lossless)"
  width: number | null;              // pixels — parsed for images; videos when cheaply available
  height: number | null;
  compressState: "should" | "done" | null; // mirrors the C/c heuristic (badges.ts)
}
```

* **Images** — dimensions parsed from the header (PNG IHDR, JPEG SOFn, GIF logical screen, WebP VP8X/VP8).
  `codec` is the format family; `compressState` mirrors `compressInfo` (PNG/BMP/TIFF/GIF → `should`,
  JPEG/WebP/HEIC/AVIF → `done`).
* **Video** — `container` from extension; `codec` sniffed by scanning the bounded buffer for the sample
  entry fourcc (ISO-BMFF `avc1`/`hvc1`/`hev1`/`av01`/`vp09`/`ap4h`…; Matroska/WebM `V_MPEG4/ISO/AVC`,
  `V_MPEGH/ISO/HEVC`, `V_AV1`, `V_VP9`…). When the moov/tracks lie outside the bounded window, `codec` is
  `null` (honest unknown). Duration is **not** computed (needs a full parse) and is omitted.

Any field we cannot determine is `null`; the UI degrades to "—". The probe never blocks the viewer — the
media loads from the grant URL regardless of what probe returns.

The local, no-shell container/codec/dimensions sniff (bounded header + tail read) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/media/media.service.ts` — `probeMedia` function

## 3. Layout (shared `MediaViewer`) — the vertical stack

The media-viewer routes (`/image`, `/video`, `/audio`) run **full-width** — the app shell drops its
normal `max-w-6xl` centered column for these three routes so the content uses the **whole** width next
to the left bar, with no side gutters. This gives the action bar (§4) the entire width to lay out
buttons before folding any into **More**. Widening only the viewer routes is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/app/AppShell.tsx` — `AppShell` (`max-w-none` for `/image` · `/video` · `/audio`, `max-w-6xl` otherwise)

One column, **six bands stacked top → bottom**. The order is deliberate: the small identifying text
is at the top, the **media is the star in the middle**, and the analysis tabs sit at the bottom. The
bands, top to bottom:

1. **Full path** — the file's absolute path in small, muted, monospaced text. This is the very top line.
2. **File name** — the title of the page, in **large** text (it is the biggest text on the page).
3. **Action bar** — the **full-width overflow action bar** (§4): as many action buttons as FIT
   (IPFS primary · Compress · Open: in browser · on Mac · Open folder: in browser · on Mac ·
   Transcribe · Copy path · View properties · Move), with anything that doesn't fit — plus the sticky
   flag/danger extras — folded into a trailing **More** menu. The top-right `⋮` still carries the
   complete File catalog.
4. **Property grid** — **~4–5 columns wide × 2–3 rows tall** of `Label : value` pairs (§3.1). It is
   kept short (few rows) on purpose so it sits neatly above a 16:9 medium without crowding it out.
5. **Media surface** — the **star**: a **bounded stage** (`~58vh`, min `340px`). The image/video/audio
   **auto-sizes to fit** the stage and **zooms up to fill** while **keeping its aspect ratio** (a 16:9
   medium letterboxes into the band). It never upscales past 100% for an image by default. Its height is
   bounded (not fill-to-viewport) precisely because band 6 sits beneath it — the "content underneath"
   exception in the charter's table rule applies here to the media stage.
6. **Analysis tabs** — the **Transcription · AI description** tabs (§6): each carries a ✓ when its
   content exists, defaults to whichever is available (preferring transcription), and offers a
   **Generate** button when the active tab has nothing yet. The page scrolls to reveal this band.

```
┌────────────┬───────────────────────────────────────────────────────────────────────────┐
│  LEFT BAR  │  ~/BGit/…/Our_Movies/render/final_4k.mov                            ┌───┐   │ ← 1 full path
│            │  final_4k.mov                                                       │ ⋮ │   │ ← 2 NAME (title)
│            │                                                                     └───┘   │
│            │ [+ Add to IPFS][⚡Compress] Open: in browser · on Mac  Open folder: … [More▾]│ ← 3 overflow action bar (full width)
│            ├───────────────────────────────────────────────────────────────────────────┤
│            │  Size    8.2 GB     Dimensions 3840×2160   Codec  H.264 · MP4   Repo Our_M… │ ← 4 property grid
│            │  Modified 2h ago    Created  May 1         Compress looks uncompressed  …    │   (~4–5 cols × 2–3 rows)
│            ├───────────────────────────────────────────────────────────────────────────┤
│            │                     ┌───────────────────────────────┐                       │
│            │                     │        ▶  video / image        │  ← MEDIA (bounded stage)
│            │                     └───────────────────────────────┘                       │
│            ├───────────────────────────────────────────────────────────────────────────┤
│            │  ✓ Transcription   ○ AI description                                          │ ← 6 analysis tabs
│            │  ────────────────                                                            │
│            │  [ transcript / description text, or a Generate button ]                    │
└────────────┴───────────────────────────────────────────────────────────────────────────┘
```

* **1 Full path.** The absolute path, muted + monospaced, click-to-copy. Truncates with a tooltip on
  overflow. The one line that lets a user confirm exactly which file on disk they are looking at.
* **2 File name.** The basename as the page **title**, large. A small back affordance sits just above it.
* **3 Action bar.** The full-width **overflow** action set (§4): the buttons that fit, then a trailing
  **More** menu holding the overflow plus the sticky flag/danger extras. The `⋮` more menu (top-right)
  still carries the **complete** File catalog ([menus.mdx §4](./menus.mdx)) verbatim — the bar is a
  promotion of the most-used items, never the only way to reach an action.
* **4 Property grid.** See §3.1.
* **5 Media surface.** Dark/neutral backdrop, the medium centered and scaled **to fit** the bounded
  stage while preserving aspect ratio (letterboxed). It fills the stage in a tall window and shrinks in a
  short one — no dead space; band 6 sits below it and the page scrolls to reach it.
  * **Image viewer:** `<img>` from the grant URL, object-fit contain. Click toggles **fit ↔ 100%**
    (pan by drag when zoomed). A checkerboard shows through real transparency.
  * **Video viewer:** `<video controls>` from the grant URL (native controls: play/scrub/volume/fullscreen).
    `preload="metadata"`; not autoplaying with sound. Click/`▶` plays.
  * **Audio player:** a large centered card with the `<audio controls>` transport (play/scrub/volume) and
    the track name; the grant streams with Range so scrubbing seeks. No visual frame to fit — the band
    holds the transport centered (§3.2).

### 3.1 The property grid (band 4)

A compact grid of **`Label : value`** pairs, **~4–5 columns across** and **2–3 rows tall**, so it reads
at a glance and leaves the media its ~70%. Cells are drawn from the `EntityView` + `MediaProbe`; a cell
whose value is unknown shows `—` (or is omitted) rather than stretching the grid. Typical cells:

| | | | | |
| --- | --- | --- | --- | --- |
| **Size** 8.2 GB | **Dimensions** 3840×2160 | **Codec** H.264 · MP4 | **Repo** Our_Movies | **Peers** 3 |
| **Modified** 2h ago | **Created** May 1 | **Compress** looks uncompressed | **Decision** Sync | **Badges** i C r |

Audio swaps **Dimensions** for **Duration** and drops the image/video-only cells (§4). The grid never
grows past ~3 rows — extra facts live in the `⋮` menu or the properties page, so the media keeps its space.

### 3.2 Height behavior

The page is a **flex column** that grows with its content. Bands 1–4 take their natural height; the
**media surface is a bounded stage** (`h-[58vh] min-h-[340px]`) so band 6 (the analysis tabs) always has
a place beneath it. The media element is `max-h-full max-w-full object-contain`, so it always fits inside
the stage regardless of window size. Because band 6 lives below the media, the page **scrolls** (the app
shell's `<main>` owns the scroll) when the combined height exceeds the viewport — this is the charter's
"content underneath" exception to the fill-to-viewport rule. Earlier this viewer filled ~70% of the
viewport with the media alone; adding the always-present analysis tabs made a bounded stage the correct
default.

## 4. The action bar (band 3) — a full-width, priority-ordered overflow row

Band 3 is a **full-width overflow action bar** — the same operations a user reaches by right-clicking
the file or opening its `⋮` kebab, promoted to visible buttons. **Think of them as the file's
context-menu items laid out as buttons.** Because the viewer runs full-width (§3), the bar has the whole
column to work with; it shows **as many buttons as fit** and folds the rest into a trailing **More**
menu. The set is customized by media type (image vs. audio vs. video), and the top-right `⋮` still
carries the complete catalog.

### 4.1 Overflow & priority (how the bar decides what to show)

Every action carries a **priority**. When the row is too narrow for all of them, the **lowest-priority**
items overflow into **More** first, so the most important actions stay visible the longest. A hidden
mirror row measures each button's natural width off-screen and a `ResizeObserver` tracks the available
width, so the fit is recomputed whenever the window (or the button set) changes — no layout thrash. The
priority order, highest → lowest: **IPFS primary** (100) · **Compress** (90) · **Open** (85) ·
**Open folder** (75) · **Transcribe** (60) · **Copy path** (40) · **View properties** (30) · **Move**
(20). The trailing **More** menu always holds the overflow **plus** the sticky **extras** that are never
buttons: **Never publish via IPFS** (flag toggle), **Do not compress** (flag toggle), **Copy CID**
(when a CID exists), and **Delete…** (danger). "Open raw" is no longer its own button — it is the
**Open: in browser** half of the Open compound (§4.2).

The width-measuring overflow bar (fit the highest-priority buttons, fold the rest into **More**) is
handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/menu/OverflowActionBar.tsx` — `OverflowActionBar` (with the `fit()` greedy solver + `MoreButton`)

### 4.2 The shared actions (all three types)

* **⚡ Compress** — the compress **offer** (explicit-click, confirm — charter §6.1; never automatic).
  Shown when the file "should" compress and **Do not compress** is off. Same state as the `⋮` item and the
  files.mdx switch — flipping one updates the others.
* **+ Add to IPFS / Remove from IPFS** — the IPFS primary, identical rule to [files.mdx §3](./files.mdx):
  in a repo with `neverIpfs` off and decision ≠ sync → **Add to IPFS** (Pin; sets `sync`); decision =
  sync → **Remove from IPFS** (Unpin; sets `ignore`); not in a repo or `neverIpfs` on → absent.
* **Open:** a **compound** control — the word "Open:" is not clickable; it is followed by up to two
  clickable sub-actions: **in browser** (opens the raw bytes via the signed grant URL in a new tab — the
  former "Open raw") · **on {label}** (hands the file to the host OS default handler — Preview/QuickTime/
  the native app). The **on {label}** half reads **on Mac / on PC / on Linux** by platform and is present
  only when OS hand-off is possible here (localhost). See [os_open.mdx](./os_open.mdx).
* **Open folder:** the same **compound** — **in browser** navigates the in-app **File System** tab
  (`/fs?path=<parent>`) · **on {label}** opens **Finder / Explorer** at the containing directory
  ([os_open.mdx](./os_open.mdx)). The label is not clickable.
* **Transcribe / Re-transcribe** — audio + video only; runs Whisper locally
  ([Transcribe.mdx](./Transcribe.mdx)). Also reachable as the Transcription tab's Generate button (§6).
* **Copy path** · **View properties** (`/file`) · **↷ Move** (§4.5).
* **In the More menu:** **Never publish via IPFS** (the `neverIpfs` sticky flag — forbids the sync
  decision), **Do not compress** (the `noCompress` sticky flag), **Copy CID**, and **🗑 Delete…**
  (recoverable move-to-trash, confirm-gated — §4.5).

### 4.3 Type-specific actions (think like a PM)

The set is **customized by type** — the actions that only make sense for one medium appear only there:

| Action | Image | Audio | Video | Notes |
| --- | --- | --- | --- | --- |
| ⚡ Compress | ✓ | — | ✓ | Charter compresses **video** (primary) and **image** (secondary); audio is out of scope, so no Compress on the audio player. |
| Transcribe / Re-transcribe | — | ✓ | ✓ | Whisper, local; also the Transcription tab's Generate (§6). Not shown for images. |
| AI description (Generate) | ✓ | — | ✓ | Image + video only, in the AI-description tab (§6, [ai_description.mdx](./ai_description.mdx)). |
| Fit ↔ 100% / zoom | ✓ | — | — | Image-only viewer control (click the image). |
| Play / pause / scrub | — | ✓ | ✓ | Native `<video>` / `<audio>` transport — audio and video only. |
| Open (in browser · on Mac) · Open folder · Copy path · View properties · Move · Delete · IPFS | ✓ | ✓ | ✓ | The shared set (§4.2) — every type. |

### 4.4 Property-grid facts by type

The property grid (§3.1) also flexes by type: **image** shows Dimensions; **video** shows Dimensions +
Codec + Compress-state; **audio** shows **Duration** (read from the media element's metadata) instead of
Dimensions, and drops Codec/Compress. Repo · Peers · Decision · Size · Created · Modified · Badges are
shared across all three.

### 4.5 Move & Delete — the backend contract (explicit, guarded, recoverable)

Move and Delete are the only actions here that change the file on disk, so they are **explicit-click,
confirm-gated**, and **guarded** — a direct honoring of the charter ("never alter a file unless the user
explicitly asks"; "never destroy bytes silently").

```
POST /api/entity/move    { path: <abs>, dest: <abs> }   → { moved: true, path: <newAbs> }
POST /api/entity/delete  { path: <abs> }                → { trashed: true, restorePath: <abs> }
```

* **Move** validates that `path` is an existing regular file and `dest` is absolute, its parent directory
  exists, and it does not overwrite an existing file; then `fs.renameSync`. The viewer re-points to the
  new path on success.
* **Delete** does **not** `unlink`. It moves the file into LFBridge's **trash** under the state directory
  (`<stateDir>/trash/<timestamp>__<name>`) so it is **recoverable**, and returns the restore path. The UI
  confirms first ("Move to LFBridge trash? — recoverable").

Moving (guarded rename) and deleting (recoverable move-to-trash, never `unlink`) a single file are handled in code:

## 5. Empty & error states

* **Not found / moved** (`exists:false`): the shared "no longer at that path" card (files.mdx §5) — no
  viewer surface.
* **Wrong route for the kind:** the viewer routes by the file's **actual kind** (from its name, via
  `mediaKindForName`), so `/video` opened on an image redirects to `/image`, an audio file to `/audio`,
  and a non-media file to `/file` — and likewise for every other mismatch. A correct link never lands on
  the wrong viewer. The redirect **replaces** history (does not push) so the wrong-kind URL never lingers
  to re-bounce on Back. (Note: audio is not a "compressible" kind on `EntityView`, so the redirect keys off
  the **name-based** kind, not `compressible`, or an audio file would wrongly fall through to `/file`.)
* **Decode/playback failure:** the browser element failed to render the bytes (unsupported codec —
  e.g. HEVC/ProRes in a browser that can't decode it). Show an inline notice over the viewer: "This
  browser can't play this file's codec (`<codec>`)." with **Open raw / hand off to the OS** (the grant URL,
  which the OS can then hand to a native app — see also **Open: on {label}**, §4.2) and, for video that
  looks uncompressed, the **Compress…** offer as the suggested fix — bytes are never altered without the
  user asking.
* **Loading:** skeleton header + action row + a shimmering viewer block; the grant + probe fetch in
  parallel; the media element mounts as soon as the grant URL resolves (probe fills the codec chip later).
* **IPFS node unreachable:** the IPFS button disables with a tooltip (parity with one_repo.mdx §5); the
  viewer, probe, flags, and property links all still work.

## 6. The analysis tabs (band 6) — Transcription · AI description

Beneath the media stage sits a **two-tab analysis panel**. It gives every media file a place to hold —
and generate — its **transcript** and its **AI description**, the two things that make a large media file
searchable and understandable without opening it.

* **Which tabs show.** **Transcription** appears for **video + audio** (media that can have speech);
  **AI description** appears for **video + image** (visual media). So a **video** shows **both** tabs, an
  **audio** file shows only **Transcription**, and an **image** shows only **AI description**.
* **The ✓ availability marker.** Each tab carries a check when its content already exists on disk (a
  transcript at `<storageRoot>/.transcribe/<rel>.txt`; a description at
  `<storageRoot>/.lfbridge/analysis/<rel>/description.yaml`). An empty tab shows an outlined placeholder.
* **Default tab.** The panel opens on the tab that has content, **preferring Transcription** when both
  are present; if neither has content it opens on the first tab. The user's click overrides this.
* **Content vs. Generate.** When the active tab has content, the panel shows it (transcript in monospace;
  description as prose) with a small **Regenerate** link. When it is empty, it shows a **Generate** button:
  * **Transcription → Generate transcript** runs Whisper **locally** ([Transcribe.mdx](./Transcribe.mdx)).
    Disabled with an install hint when `whisper` isn't present.
  * **AI description → Generate description** runs a vision provider ([ai_description.mdx](./ai_description.mdx)).
    Disabled with a "add an API key" hint when no provider is configured; the copy notes that **video is
    uploaded to Gemini** (the only provider that describes video) and an image to the configured provider.

## 7. Acceptance criteria

1. `/image?path=<abs>`, `/video?path=<abs>`, and `/audio?path=<abs>` each render the **shared MediaViewer**
   as the **six-band vertical stack** (§3), top → bottom: **full path** · **file name (large title)** ·
   **full-width overflow action bar + `⋮`** · **property grid (~4–5 cols × 2–3 rows)** · **bounded media
   stage** · **analysis tabs**. The viewer routes run **full-width** (no side gutters); the media
   **auto-sizes to fit** the stage while **keeping aspect ratio**; the page **scrolls** to reveal the
   analysis tabs when needed.
2. The medium **plays/loads off the local disk** via the backend stream — never a `file://` URL. Video and
   audio support **seeking** because `/api/media/raw` answers HTTP **`Range`** (`206` + `Content-Range`);
   images load whole. The stream is reachable only through a **short-lived signed grant** minted for an
   allow-listed session; the raw endpoint refuses expired/forged tokens and any non-file path.
3. The **action bar** (band 3) is a **full-width overflow row** (§4): it shows the highest-priority buttons
   that fit and folds the rest — plus the sticky extras (**Never publish via IPFS**, **Do not compress**,
   **Copy CID**, **Delete…**) — into a trailing **More** menu; the lowest-priority items overflow first.
   **Open** and **Open folder** are compound "in browser · on {label}" controls, the **on {label}** half
   present only on localhost ([os_open.mdx](./os_open.mdx)). Image + video additionally show **Compress**;
   audio shows neither Compress nor AI description. The top-right `⋮` menu renders the full File catalog
   ([menus.mdx §5.3](./menus.mdx)) verbatim.
4. The **property grid** (band 4) shows the type-appropriate facts: Size · Modified · Created · Repo ·
   Peers · Decision · Badges for all; **Dimensions + Codec + Compress** for image/video; **Duration** for
   audio. Unknown cells show `—`; the grid never exceeds ~3 rows.
5. `MediaProbe` is computed **locally with no shell** (charter): image dimensions + format from the header;
   video container from extension and codec by a bounded fourcc/EBML sniff; audio container/codec from the
   extension family; unknown fields are `null` and render as "—" without breaking the viewer.
6. **Move** renames the file (guarded: absolute dest, existing parent, no overwrite) and the viewer
   re-points to the new path. **Delete** moves the file to LFBridge's **trash** under the state dir
   (recoverable) — it **never `unlink`s** — after a confirm. Compress stays an explicit-click offer.
7. Error/empty states render per §5 (not-found card, kind-based wrong-route redirect incl. audio,
   decode-failure notice with Open-raw + Compress offer, loading skeleton, node-unreachable). No bytes are
   ever altered or destroyed without an explicit click + confirm.
8. The **analysis tabs** (band 6) show **Transcription** for audio/video and **AI description** for
   image/video (a video shows both), each with a ✓ when its content exists on disk. The panel defaults to
   the tab with content (preferring Transcription); an empty tab offers a **Generate** button —
   **Generate transcript** (local Whisper) or **Generate description** (a vision provider), each disabled
   with a clear hint when its dependency (whisper / an API key) is absent. See [Transcribe.mdx](./Transcribe.mdx)
   and [ai_description.mdx](./ai_description.mdx).
9. **Open on {Mac/PC/Linux}** is present only on localhost and hands the file/folder to the host OS
   default handler; in server mode (or for a non-loopback caller) the "on {label}" halves are hidden and
   only "in browser" remains ([os_open.mdx](./os_open.mdx)).
