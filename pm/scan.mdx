# LargeFileBridge — The Scanning Service (Big-File Discovery)

The **scan** is the periodic pass that answers one question across all of a user's computers:
**"Have any big files — or the repos that hold them — changed since last time?"** It discovers new
large files, notices files that have grown past the "big" threshold, spots moves/renames/deletes, and
registers repos that have appeared or vanished. It is the eyes of LargeFileBridge (LFB); the
[background pin](./storage_local.mdx#13-the-background-pin-process--the-transparency-contract) is the
hands that move the bytes.

Two scheduled jobs, two cadences, on purpose:

| Job (scheduleTask) | Cadence | What it does | Where spec'd |
| --- | --- | --- | --- |
| **Discovery scheduleTask** (this file) | **every 4 hours** | Deep, whole-filesystem **discovery** — stat every candidate file, find big files + filesystem/repo changes. Metadata only; no IPFS. | this file |
| **Pin scheduleTask** | every **15 minutes** | A **full pin pass** ([pin_process.mdx](./pin_process.mdx)): visit + reconcile **every** known unit; for units the user **opted into IPFS** (`pinned: true`) add/update manifests, pin, fetch missing CIDs over IPFS **— and read the local IPFS pinset (`ipfs pin ls`) to reconcile + import already-pinned files** (§2.1). Runs units concurrently (many cores). | [storage_local.mdx §13](./storage_local.mdx#13-the-background-pin-process--the-transparency-contract) · [pin_process.mdx](./pin_process.mdx) |

The discovery pass is expensive (it walks the tree and stats many files), so it is throttled to **every
four hours**. The 15-minute pass is cheap per unit and time-sensitive, so it runs over the small set the
discovery pass has already found. The 15-min pass may do a *light* rescan of a unit it is about to pin;
the **deep, full-filesystem discovery walk is the 4-hour job's alone.**

Both of those are **polling** jobs on a clock. There is also a **third, event-driven trigger** — the
**live filesystem watcher** (§2.2) — that is **not** a scheduleTask and has **no** cadence: it runs
**only while the web app process is up** and fires **the moment the OS tells it a file was added or
deleted**, so a big file dropped into a repo lights up in the UI in seconds rather than after the next
4-hour walk. The watcher never moves bytes and never opens files; when it sees a **qualifying** add/delete
(§2.2) it simply **kicks the same discovery worker** (coalesced) so all the existing tracking/output
machinery (§6) runs. It is a *latency* optimization on top of the 4-hour scan, **never a replacement** —
the scheduled walk still catches everything that changed **while the app was off** (when no watcher was
listening).

## 1. What the scan does — and deliberately does NOT do

The scan is a **metadata-only** pass. For every candidate path it reads only what the filesystem
already knows without opening the file:

* **name / path**
* **size in bytes** (from `stat`)
* **mtime** (last-modified time, from `stat`)
* file vs. directory vs. symlink

It **never opens or reads file contents.** It does **not** hash files, does **not** compute IPFS
CIDs, and does **not** touch IPFS at all. Hashing and CID computation are the pin's job, and only for
files the scan has already flagged as big and that live in a `pinned: true` unit. Keeping the scan to
pure `stat` is what makes it cheap enough to run over the whole filesystem every four hours.

What the scan produces is a **candidate list** per pin unit: the files that are large-file-bridge
payload per the **payload rule** (§4.1) — git-ignored media at *any* size, plus anything at/above the
big threshold — with a diff against the previous scan (added / grown / shrunk / moved / deleted),
written to that unit's `status.yaml` (§6). It changes **no** file contents and moves **no** bytes.

The metadata-only discovery pass that stats candidates and never opens files runs in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()` → `walkUnit()` (uses `fs.statSync` only)

## 2. Cadence — the two scheduleTasks (4-hour discovery + 15-minute pinset)

* The **discovery scheduleTask** (this file) runs **once every four hours** (`interval_hours: 4`, i.e.
  14400 seconds).
* It also runs **on demand** from the web app — a **"Rescan"** action on the Repos page and on any
  repo's page triggers the same worker immediately, out of band from the schedule.
* Like the 15-minute scheduleTask, it has the same **installed vs. on/off** transparency contract
  (§7): the plist/timer/task can be installed but disabled, and the user toggles it from the web app.
* Cadence is a setting, not a constant: `config.yaml → scan_process.interval_hours` (§5). Four hours
  is the default; the UI may expose it later. Changing it re-writes the schedule (§4).

### 2.1 The 15-minute scheduleTask reads the IPFS pinset (import already-pinned)

The **15-minute scheduleTask** ([storage_local.mdx §13](./storage_local.mdx#13-the-background-pin-process--the-transparency-contract))
does the byte work — but it now also **reads the local IPFS pinset (`ipfs pin ls`)** every run and
**reconciles it against the manifests**, so *"what files are being IPFS-pinned locally"* is always
current without waiting on the 4-hour walk. Charter: *"Part of that scan is to look at what files are
being IPFS-pinned locally … make that happen at the 15-minute [interval]."*

* **Reconcile pins ↔ manifests.** Every pinned CID **in** a manifest → confirm this computer's
  `pinned_by` claim (or, if the manifest claimed it but the pin is gone, drop the claim + queue a
  re-pin — [sync_list §5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)). Every pinned
  CID **not** in any manifest → an **Untracked (pinned)** import candidate.
* **Import already-pinned.** *"If we get to a computer and the files are already IPFS-pinned, import
  them."* Untracked pins are surfaced on the [IPFS page](./ipfs.mdx#4-import--the-load-bearing-feature)
  and adopted into tracking; import is **metadata-only** (record the manifest entry, mark the file
  **Add to IPFS (pin)** — the `sync` decision value) — the bytes are already pinned, so nothing is
  fetched, re-added, or re-pinned. An imported file is present-and-pinned, i.e. it **counts as Pinned**
  ([repos.mdx §3.2](./repos.mdx#32-columns)).
* **Still metadata-only.** Reading the pinset is a `pin ls` + `stat`; it opens no file and moves no
  bytes, exactly like the discovery pass (§1). It is the one place the *scan* touches IPFS at all — and
  only to *list* pins, never to serve or announce anything beyond the
  [only-our-content](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) defaults.
* **Needs the `ipfs` CLI.** The pinset read requires Kubo; if it is missing the preflight (§4.4)
  records a `needs_tool` gap and the app routes the user to [install it](./tools.mdx).

The 15-minute pass that reads the local pinset and reconciles/refreshes each `pinned_by` claim against the manifests runs in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinRepoFolder()` (reads `ipfs.listPins()`, refreshes via `setPinClaim()`)

### 2.2 The live filesystem watcher — event-driven, while the web app runs (LOCKED)

Every operating system offers a way to be **called back** when the filesystem changes — files added,
removed, or renamed under a directory — instead of forcing us to re-walk the tree to find out. LFBridge
uses that. **While the web app is running**, it registers a **recursive filesystem watcher** over the
**`scanner.roots`** so we learn about relevant changes **the instant they happen**, and update our
tracking and the File System tree without waiting up to four hours for the next discovery pass (§2).

> This watcher lives **with the web-app process**, not with the OS scheduler. It is **not** a
> scheduleTask — it has no plist/cron/timer and no cadence. It starts when the backend boots and stops
> when the backend exits. If the app is not running, there is no watcher; that gap is exactly what the
> 4-hour scheduled discovery pass (§2) exists to backfill.

**Which OS API.** One cross-platform Node/TS watcher, the same everywhere; only the kernel notification
source underneath differs per OS (Node's `fs.watch({ recursive: true })` binds to each natively):

| OS | Native change-notification source |
| --- | --- |
| **macOS** (primary) | **FSEvents** |
| **Linux** | **inotify** |
| **Windows** | **ReadDirectoryChangesW** |

**We react to ADD and DELETE — never to "modified" (LOCKED).** The user's contract is explicit: *"when
files are added or deleted … we don't care about modified."* So the watcher acts **only** on a file
**appearing** (added) or **disappearing** (deleted/renamed-away) under a root. A pure **content edit**
of an existing file (`write`/`change`) is **ignored** — a file being re-saved does not change whether it
is our payload, and reacting to every save would be needless churn. (A **rename** surfaces as a
delete-at-old-path + add-at-new-path pair, which the coalesced rescan then re-classifies as a **move**
via the size+mtime heuristic in §6 — so pins are kept, not re-added.)

**What we check for each added/deleted path.** The watcher answers the same two questions the charter
poses, using only cheap metadata — it **never opens the file** (the §1 no-content contract holds here
too):

1. **Is it at or over the big threshold?** For an **added** path we `stat` it once and compare against
   the effective big threshold (§4.2; the watcher uses the global default as a cheap gate and lets the
   rescan apply exact per-unit overrides). For a **deleted** path there is nothing left to `stat`, so
   size cannot be read — the extension test below carries the delete case.
2. **Is it a video, image, or audio file?** By extension via `isMediaFile()` (the shared
   `scan-filters.ts` list, §4.1) — the same media test the scan uses.

A change is **qualifying** when the path is a **media file (video/image/audio)** *or* an **added file at
or over the big threshold**. Non-media, non-big noise (a `.ts` source file, a `.log`) is dropped so the
watcher stays quiet on ordinary editing. Paths under any **`HARD_SKIP`** directory (§4.3) or matching
`scanner.ignore_globs` are ignored before either test — we never wake on `node_modules`/`build`/`.git`
churn.

**Debounce + coalesce.** Editors and tools emit **bursts** (a save writes a temp file, renames it,
deletes the temp). The watcher collects raw events into a pending set and **debounces** (default
~1.5 s of quiet, `watcher.debounce_ms`, §5) before deciding. If, after a burst settles, **at least one**
qualifying add/delete remains, the watcher **kicks the discovery worker once** — `startScan("manual")`,
the same single-flight, coalesced, detached background job the Rescan button drives (§10). Because
`startScan` is single-flight and coalescing, a storm of drops results in **at most one** in-flight walk
plus **one** queued follow-up — never a walk per file.

**How the tracking and the tree update.** The kicked discovery pass does exactly what §6 already
specifies: it re-diffs the affected unit(s) and rewrites each `status.yaml` — the **candidate list**,
the **big-file counts**, and the added/deleted **change set** (the "importance flags" the user means).
Because the File System page derives its **interesting-directory folder coloring** and counts from that
same tracking ([file_system.mdx §1](./file_system.mdx), [directories.mdx](./directories.mdx)), and
because the FS browser's per-directory listing cache is already **mtime-guarded** and self-invalidates
when a directory's contents change ([performance.mdx](./performance.mdx) / `fsindex.service.ts`
`getListingCached`), **adding or deleting a big/media file makes the tree and its coloring refresh on
their own** — no manual Rescan. That is the whole point: *"if [a file] was added then our file system
tree will update."*

**Security posture is unchanged.** The watcher subscribes **only** to our own `scanner.roots`, reads
**only** filesystem event metadata + one `stat`, opens no file contents, and touches **no** network and
**no** IPFS. It is fully inside the [only-our-content](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule)
posture.

Registering the recursive OS watcher over the scanner roots, filtering to qualifying add/delete events, debouncing bursts, and kicking the coalesced discovery worker happen in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/watcher/watcher.service.ts` — `startWatcher()` → `onFsEvent()` / `flushPending()` (calls `isMediaFile()` from `shared/scan-filters.ts`, then `startScan("manual")`)

Starting the watcher when the backend boots and stopping it on shutdown is wired in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/main.ts` — `main()` (calls `startWatcher()` after `app.listen`; `stopWatcher()` on `SIGINT`/`SIGTERM`)

### 2.3 Scan staleness → the repo header's "Scan now" primary (LOCKED)

A repo's metric panels ([task_tabs.mdx §2](./task_tabs.mdx)) are only as trustworthy as its **last scan**.
When a repo has **never been scanned**, or its **`lastScanAt` is older than 2 weeks**, we can't trust the
counts, so re-scanning takes **precedence over every recommendation**: the One-repo header primary
([one_repo.mdx §3.1](./one_repo.mdx#31-header)) becomes **"Scan now"** instead of "View recommendation".
Clicking it runs the same on-demand discovery worker (§2). When the scan lands, `lastScanAt` refreshes and
the button flips back to "View recommendation".

* The **2-week window** is the staleness threshold; the 4-hour scheduled pass (§2) and the live watcher
  (§2.2) normally keep a running install far fresher, so "Scan now" appears mainly on a repo that hasn't
  been opened/scanned in a long time (or was just registered). When the watcher knows the tree changed,
  that too triggers a scan, after which the button returns to a recommendation.
* `lastScanAt` is carried per-repo on `RepoDetail` (sourced from `UnitStatus.last_scan_at`), and the
  staleness decision is a pure client check so no extra fetch is needed.

The last-scan timestamp on the repo detail, the 2-week staleness gate, and the header wiring live in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `computeRepoDetail()` (sets `lastScanAt: status.last_scan_at`);
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/metricWarningDefs.tsx` — `scanIsStale()` (the 2-week gate);
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` (`scanDue` drives the "Scan now" button).

## 3. Cross-platform — runs on Mac, Windows, and Linux

LFB runs on the **Mac filesystem** primarily, but also on **Windows and Linux**. The scan **worker
is one cross-platform Node/TS program**; only the *scheduler that launches it* differs per OS. The
worker code path is identical everywhere — only path normalization and the schedule installer branch
on platform.

| OS | Scheduler | Unit | Primary? |
| --- | --- | --- | --- |
| **macOS** | **launchd** | a `.plist` LaunchAgent (`StartInterval` = 14400) | **Yes — primary target** |
| **Linux** | cron **or** systemd timer | a crontab line / `.timer` + `.service` | secondary |
| **Windows** | Task Scheduler | a scheduled task (4-hour trigger) | secondary |

The **Mac launchd plist is the primary, first-class implementation** ("the plist for Mac … the Mac
equivalent specific one"). Windows and Linux schedulers are supported and follow the same
installed / on-off model, implemented per-OS behind one installer interface.

## 4. Roots, discovery, and the mask

The scan walks the **`scanner.roots`**
([storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)),
honoring `scanner.ignore_globs` and `follow_symlinks`. In one pass it does three things, in order:

1. **Repo discovery.** Any directory containing a `.git` folder is registered/reconciled as a repo
   pin unit (`pin/r/<repo_name>/`). New repos are created (default `pinned: false` until the user
   opts in); already-known repos keep their settings; repos gone from disk are marked `missing` in
   their `status.yaml` — never silently deleted
   ([storage_local.mdx §7](./storage_local.mdx#7-the-scanner--discovering-repos-and-the-mask)).
2. **Build the mask.** The set of registered repo working-tree paths becomes the **mask** for the
   computer unit, so every file is attributed to exactly **one** unit — a repo unit or the computer
   unit, never both.
3. **Flag candidates.** Walk each unit's scope, `stat` every file, and record every file that is a
   large-file-bridge payload (§4.1). Diff against the previous scan to classify each change.

Repo discovery, registering each `.git` working tree, and building the repo mask before the per-unit walk happen in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()` (calls `findGitRepos()` then `registerRepo()`)

The scan **never descends** into `scanner.ignore_globs` (e.g. `**/node_modules/**`, `**/.git/**`,
`**/.Trash/**`), a unit's own `exclude_globs`, or the **hard-skip directory set** (§4.3) — this is
both correctness (don't bridge build junk) and the main performance lever.

> **The hard-skip set is one source of truth.** The directories the scan refuses to descend into are
> defined once in `code/.../shared/scan-filters.ts` (`HARD_SKIP`) and imported by **both** the scan
> worker and the File System browser's badge pass, so the two can never drift (the invariant
> previously written as a comment in `badges.ts`). See §4.3.

### 4.1 What counts as a candidate — the payload rule (LOCKED)

A file is a **candidate** (a large-file-bridge payload the scan records) when **any** of these hold:

1. **Git-ignored media — regardless of size.** The file matches the repo's `.gitignore` **and** is a
   **video / audio / image** (`isMediaFile`, defined in `shared/scan-filters.ts`). A git-ignored
   `.mp4`/`.mp3`/`.png` is the *whole reason LFBridge exists* — the user deliberately kept it out of
   git to bridge it over IPFS (README charter). **The big-file size threshold does NOT gate these.**
2. **Any file at or above the effective big threshold** (§4.2) — the classic "big file" case. For a
   repo with a followed `.gitignore`, this still requires the file to be git-ignored (a big file that
   is *checked in* is not our **payload** — though it may still be surfaced as a *nudge* by rule 4,
   which never makes it payload); the **computer unit** (no gitignore) uses the pure size gate.
3. **A forced include-glob match** (`large_files.include_globs`) — an explicit user opt-in.
4. **A checked-in file at or above the *checked-in* threshold** (§4.2) — the file is **not** git-ignored
   and is ≥ **50 MB** (default). This is the charter's *"big files that aren't a good idea to check in"*
   category: we surface it and offer the one-click ⊘ **git-ignore**. Repo units only — the computer unit
   has no working tree, so "checked in" is meaningless there.
5. **Analysis media — video / audio / image, regardless of size.** The file is a **video / audio / image**
   (`isMediaFile`) at **any** size, even a small, checked-in, not-git-ignored one — a 2 MB screenshot, a
   200 KB icon, a 30-second clip. It is admitted **only** so the analysis tabs can act on it: **OCR** a
   screenshot, **describe** an image, **transcribe** a short clip ([task_tabs.mdx §4.6/§4.7](./task_tabs.mdx),
   [ocr.mdx §11.1.1](./ocr.mdx)). This is the rule that lets *"open the OCR tab and OCR a file smaller than
   the large-file size"* work — without it, a small JPG is never a row and no UI filter could reveal it.

> **Rule 5 rows are `analysisOnly` — surfaced-but-small, and NEVER payload.** A rule-5 row exists so a small
> image/video/audio can be OCR'd / described / transcribed, nothing more. It is:
> * **flagged `analysisOnly`** on the candidate (and carried onto the `FileRow`) — the flag the promoted
>   **"Large files only"** rail toggle reads to hide these rows by default ([tables.mdx §2.9](./tables.mdx)).
>   A rule-5 row is *the* thing that toggle hides; every other candidate stays visible when it is on.
> * **`nudgeOnly` too** (it is not payload), so it inherits rule 4's auto-decide exclusion below verbatim —
>   surfacing a thumbnail for OCR must never auto-pin it to IPFS.
> * A file that is **already** payload (git-ignored media, rule 1) or a **checked-in nudge** (rule 4) is
>   **NOT** `analysisOnly` even though it is also media — those stay visible under "Large files only" (the
>   5–100 MB git-ignored clip case above must never be re-hidden). `analysisOnly` = *"admitted purely by
>   rule 5, by nothing higher."*

> **Rule 4 rows are a NUDGE, not payload.** Admitting a checked-in file as a *candidate* does not make it
> ours to bridge — it stays `undecided` with no CID and is never pinned. It is listed so the user can *see*
> it and decide. The user's ⊘ click is what turns it into a git-ignored file, and only then does rule 1/2
> make it payload. This is why rule 4 does not contradict "a big file that is checked in is not our
> payload" — it is not payload; it is a warning about git hygiene.
>
> **Rule 4 rows are EXCLUDED from the default-decision policy (LOCKED — a privacy gate).** The auto-decide
> policy (decisions.mdx §9) runs over each scan's `added` set. Rule-4 rows are filtered out of that set
> **before** the policy sees them, because a repo on `mode: auto, ipfs: true` would otherwise **auto-pin a
> checked-in file to IPFS that the user never chose to bridge** — publishing it. Auto-writing the
> `.gitignore` entry is separately forbidden by the charter (*"Never add a `.gitignore` entry
> automatically for anyone"*). Both axes stay **Undecided** until the user clicks. Every row admitted by
> rule 4 alone is flagged `nudgeOnly` in-memory by the walk (never persisted to `status.yaml`) purely to
> enforce this. **A future change that admits more non-git-ignored candidates MUST extend this filter** —
> widening the candidate set silently widens what auto-decide can publish. **Rule 5 (analysis media) is
> exactly such a widening and it obeys this rule:** rule-5 rows are `nudgeOnly` too, so they are already
> filtered out of `added` before the auto-decide policy runs — a small checked-in JPG surfaced for OCR is
> never auto-pinned. `analysisOnly` (persisted, for the UI toggle) is a **separate** flag from `nudgeOnly`
> (in-memory, for the auto-decide gate); rule-5 rows carry both.

The candidate predicate — git-ignored media at any size (rule 1), else the size gate (rule 2), plus forced
includes (rule 3), the checked-in nudge (rule 4), and analysis media at any size (rule 5) — is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `walkUnit()` (`isCandidate` / `analysisOnly`, using `isMediaFile()` from `shared/scan-filters.ts`)

> #### Why rule 4 exists — the bug it fixes (root-cause note, do not remove)
>
> Rules 1–3 made **every** row a git-ignored one, so the `bigNotIgnored` metric — *"Big not git-ignored"*
> (task_tabs.mdx §2.6) and the charter's category-3 rollup — was **structurally dead code**: it counts
> `!f.gitignore && size >= threshold`, but a row could only exist if git *did* ignore it, so the count could
> never rise above **0**. The charter promises we *"point out files that are big but not git-ignored, and
> give an easy one-click option to git-ignore them"*; the predicate made that unreachable. Found via a real
> file: `Bryan_git/Personal` held a **56 MB checked-in `Security Cameras.pptx`** that the one-repo search box
> could not find, because the scan never made it a row. After the fix that repo reports **13** candidates
> (was 12) and `bigNotIgnored: 1` — verified by rescanning against an isolated `LFB_STATE_DIR`.

> #### Why this rule exists — the bug it fixes (root-cause note, do not remove)
>
> The scan originally required a repo file to be **big AND git-ignored** (`size ≥ threshold &&
> gitIgnored`). Real repos broke this: in `charlie-kirk` the git-ignored `videos/` + `IPFS/videos/`
> folders hold **46** media files (24 `.mp4` + 22 `.mp3`) but only **3** exceed 100 MB — so the scan
> reported **3 of 46** and the repo looked almost empty. `Intel_Murder_Docus` reported **2 of 27**
> for the same reason (`static/videos/*` is 16 `.mp4` + 11 `.mp3`, mostly 5–100 MB). The 100 MB
> default (GitHub's hard limit) is the right nudge for *checked-in* files, but it must **never hide
> git-ignored media the user has already chosen to bridge**. After the fix the counts are the full
> **46** and **27** — verified by rescanning both repos and diffing `status.yaml` against a
> filesystem census. The `stat`-only, no-content contract (§1) is unchanged; only the *predicate* for
> "is this a candidate" changed.
>
> **Non-media git-ignored files are deliberately NOT bypassed** — a git-ignored `.env`, `.log`, or
> `.DS_Store` must still clear the size threshold, so junk and secrets aren't swept into the bridge.

### 4.2 The effective big threshold

Whether a file is "big" is decided **per unit** by the threshold resolution in
[settings.mdx](./settings.mdx):

```
effective_threshold(file) =
  the repo's override        if that repo's "override big threshold" checkbox is ON
  else the global setting    (default 100 MB)
```

The scan reads the resolved threshold for each unit and flags files whose **size ≥ threshold**.
The scan does **not** define the threshold or its default — it only applies it (§9 Reconciliation).
The threshold gates case 2 of §4.1 only; git-ignored **media** (case 1) ignores it.

Resolving the per-unit effective threshold (repo override else the global default) happens in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `resolveThreshold()`

#### The checked-in threshold — a SECOND, lower gate (LOCKED)

Rule 4 (the checked-in nudge) uses its **own** threshold, **not** the big threshold:

```
checked_in_threshold = min(global big_file.checked_in_threshold_bytes,  // default 50 MB
                           effective_threshold(unit))                   // never exceed the payload gate
```

* **Default 50 MB — GitHub's *warning* line**, deliberately **lower** than the 100 MB payload threshold
  (GitHub's *hard* limit). The point is to warn **before** GitHub blocks the push, so a 56 MB checked-in
  `.pptx` is caught while it is still fixable.
* **It is clamped to the unit's effective threshold.** A per-repo `big_file_override` that drops the big
  threshold below 50 MB would otherwise nudge on files the same scan already treats as payload; the `min()`
  makes the nudge gate always ≤ the payload gate.
* **The `bigNotIgnored` metric MUST count at this same threshold**, never at the 100 MB payload threshold —
  otherwise a row appears in the table that the metric offering to fix it does not count.

The checked-in gate and its clamp live in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `walkUnit()` (`checkedInBig`) and the `checkedInThreshold: Math.min(...)` at the repo call site;
the matching metric count in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `computeTaskMetrics()` / `checkedInThresholdBytes()`

### 4.3 The hard-skip directory set (never descend)

Some directories are never walked at all — regenerated build output and dependency/VCS/OS junk. They
are listed once in `shared/scan-filters.ts` as `HARD_SKIP` and shared with the FS browser:

```
.git  node_modules  .Trash  .cache  Caches            ← vcs / deps / os junk
build  dist  out  .docusaurus  .next  .nuxt  .svelte-kit  .turbo  coverage   ← build outputs
```

**Why build outputs are skipped (second root-cause note):** a Docusaurus/Next build **copies** the
source media into its output dir, so before this list the scan reported `Intel_Murder_Docus`'s hero
video **twice** — once as `static/videos/x.mp4` and once as the generated `build/videos/x.mp4`.
Build output is ephemeral and regenerated; it is **never** a pin candidate. Skipping these dirs
removes the phantom duplicates (Intel dropped from 27+build-dupes to a clean **27**) and is also the
main performance lever on large trees.

The single-source-of-truth hard-skip directory set, shared with the FS-browser badge pass, lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/scan-filters.ts` — `HARD_SKIP`

### 4.4 Preflight — required tools before a scan (brew-install)

Charter: the scan *"can and will use brew-install tools … Before it does a scan, see if any are missing
and show that page to ask the user to install them."* So **before** a scan/scheduleTask runs, LFB runs
the **tools preflight** ([tools.mdx §3](./tools.mdx#3-the-preflight--check-before-you-scan-locked)):

* It probes the tools the run needs on `PATH` — **`ipfs` (Kubo)** is **required** (the pinset read §2.1
  and all IPFS work depend on it); **`ffmpeg`/`ffprobe`** and image tooling are **optional** (they
  sharpen the compress `C`/`c` judgment but never block discovery).
* **Interactive run** (a user clicked Rescan): a missing **required** tool routes to the
  [install-tools page](./tools.mdx) — detect Homebrew, list each tool with status + install command, and
  a one-click **Install** (`brew install ipfs ffmpeg`) — with the pending scan **remembered** to resume
  after install. Nothing installs without an explicit click (charter consent).
* **Scheduled run** (the `launchd` scheduleTask, unattended): a missing **required** tool is recorded as
  a **`needs_tool`** status the web app surfaces (banner + notification), and that run is **skipped** —
  the scheduleTask never installs and never error-loops. The user installs the tool next time they open
  the app; the schedule then resumes.

## 5. Config — `scan_process` in the app config

The scan's schedule/state is a hierarchical block in the app-level, computer-wide `config.yaml`
([storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)),
parallel to `pin_process`:

```yaml
# ~/T/_large_files_bridge/config.yaml  (excerpt — the scanning service)
scan_process:
  installed: false            # is the scan scheduler (launchd/cron/task) installed on this machine?
  enabled: false              # is it currently ON? (user toggles from the web app)

  label: com.largefilebridge.scan   # launchd label (Mac) / task name / timer name

  last_run_ok: true           # machine-written: did the last scan finish without a fatal error?

# The live filesystem watcher (§2.2) — event-driven, runs with the web-app process (no scheduler).
watcher:
  enabled: true               # subscribe to OS file-change events over scanner.roots while the app runs
  debounce_ms: 1500           # quiet-period before a settled burst kicks a coalesced discovery rescan
```

The existing `scanner:` block (roots / ignore_globs / follow_symlinks) stays where it is in
`config.yaml` — `scan_process` adds only the **scheduling + on/off** facts, exactly mirroring how
`pin_process` is structured. The `watcher:` block is separate because the watcher is **not** a
scheduled job (§2.2): it has an `enabled` switch and a debounce window, but **no** `installed`/plist —
it exists whenever the backend is up.

## 6. Output — what the scan writes

The scan writes **only** operational metadata; it never writes a manifest or moves bytes. Per pin
unit it updates that unit's `status.yaml`
([storage_local.mdx §6.3 / §8.2](./storage_local.mdx#5-the-two-kinds-of-pin-unit--the-core-model)):

```yaml
# pin/r/<repo_name>/status.yaml  (excerpt — written by the scan)
schema_version: 1
last_scan_at: 2026-07-02T12:00:03.114Z
scan_source: scheduled            # scheduled | manual (Rescan button)
effective_threshold_bytes: 104857600   # 100 MB — what "big" meant for THIS unit this run
big_file_count: 7                 # files at/above the threshold right now
big_file_bytes: 5629499534        # total bytes of those big files
changes_since_last_scan:
  added:   [large/trailer_v4.mp4]        # newly appeared big files
  grew:    [assets/render_out.exr]       # crossed the threshold since last scan
  shrank:  []                            # dropped below the threshold
  moved:   [{ from: old/hero.png, to: assets/hero.png }]  # same size+mtime, new path
  deleted: [large/trailer_v2.mp4]        # gone from disk
repo_state: present               # present | missing (drive unmounted / repo deleted)
```

* **Move/rename detection** uses size + mtime heuristics only (no content hashing) — a file that
  vanished at path A and appeared at path B with the same size and mtime is reported as a **move**,
  not a delete+add, so the pin pass can keep the existing pin instead of re-adding bytes.
* The **computer unit** (`pin/computer/status.yaml`) gets the same shape for everything outside
  repos.
* The scan **stamps `last_scan_at`** and the change set; the **pin pass** later reads these to know what
  work to do. The scan itself pins nothing and fetches nothing.

The web app reads these `status.yaml` files to show, per repo and on the Scans page: last scan time,
big-file counts, and "N new big files found" badges — all without the scan ever having opened a file.

Diffing against the previous scan and writing the added/grew/deleted change set to each unit's `status.yaml` happens in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `diffStatus()` (persisted via `writeStatus()`)

> **The per-scan repo-wide counts re-stamp lands in Local Storage, never the working repo (LOCKED).** Beyond
> each pin unit's `status.yaml`, the scan also re-stamps the repo-wide `repo_storage.yaml` (`counts:` +
> `last_scan.at`) — **Category B** tracking state. It is written to **Local Storage** at
> `~/T/_large_files_bridge/repos/<repoKey>/repo_storage.yaml` **always** (and, when the owning
> company/Personal storage has a **sync repo** configured and the per-repo toggle is on, additionally
> mirrored to `<syncRepo>/repos/<repoKey>/`) — **never** committed into the working repo, so the per-scan
> re-stamp can never merge-conflict. This is the exact churn the placement policy keeps out of git.

The per-scan `repo_storage.yaml` counts/`last_scan` re-stamp routing to the Local-Storage tracking root (never a working repo) is defined authoritatively in [artifact_placement_policy.mdx §6](./artifact_placement_policy.mdx) and implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` — `refreshCounts()` / `writeRepoStorage()` (routed through `tracking-root.service.ts` — `resolveTrackingRoot()`), driven by `scanner.service.ts` — `scanAll()`.

## 8. Acceptance criteria

The scan is correct when **all** hold:

1. The scan runs **every 4 hours** by default (`scan_process.interval_hours: 4`), on its own schedule
   **separate** from the every-15-minute pin, and also runs on demand from a **Rescan** action.
2. The scan is **metadata-only**: it `stat`s files for name/size/mtime and **never opens, reads, or
   hashes file contents**, and **never touches IPFS**.
3. The scan is **cross-platform**, scheduled by a **launchd plist on Mac** (primary), and by cron /
   systemd timer on Linux and Task Scheduler on Windows — one shared Node/TS worker, per-OS installer.
4. In one pass the scan **discovers/reconciles repos**, builds the **repo mask**, and records every
   **candidate** per the payload rule (§4.1), attributing each file to exactly one pin unit.
4a. **Git-ignored media (video/audio/image) is a candidate regardless of size** — the size threshold
    must **never** hide the 5–100 MB git-ignored videos/images a user has chosen to bridge. Regression
    guard: rescanning `charlie-kirk` yields **46** candidates (not 3) and `Intel_Murder_Docus` yields
    **27** (not 2). Non-media git-ignored files (`.env`, `.log`, `.DS_Store`) are **not** bypassed.
4b. **Build-output directories are never walked** (`build`, `dist`, `.docusaurus`, `.next`, … — the
    shared `HARD_SKIP`, §4.3), so a build's *copy* of the source media is not counted as a second
    candidate. `Intel_Murder_Docus` shows **no** `build/…` paths in its candidate list.
5. The effective threshold is resolved **per unit** — the repo's override when its checkbox is on,
   else the **global default (100 MB)** — read from [settings.mdx](./settings.mdx); the scan applies
   it but does not define it. It gates only the non-media cases of §4.1 (case 2 and case 3).
6. The scan writes **only** operational metadata to each unit's `status.yaml` (last scan time,
   big-file counts, and an added/grew/shrank/moved/deleted change set); it writes **no** manifest and
   moves **no** bytes.
7. Move/rename is detected by size+mtime (no hashing) and reported as a **move**, not delete+add.
8. `config.yaml → scan_process` records **installed** and **enabled** as two independent facts, both
   displayed in the web app, with install/uninstall + on/off toggle + Rescan available from the UI.
9. Ignored paths (`scanner.ignore_globs`, per-unit `exclude_globs`) are never descended into.
10. The scheduled jobs are **scheduleTasks** in code/config/labels and **"scans"** in the UI; the rename
    covers the **scheduled process only** — the per-file **Sync** decision and the **Sync List** keep
    their names (Naming, §2).
11. The **15-minute scheduleTask reads the IPFS pinset** (`ipfs pin ls`) each run and **reconciles it
    against the manifests** — confirming/dropping `pinned_by` claims and surfacing **pinned-but-untracked**
    files as **import** candidates ([IPFS §4](./ipfs.mdx#4-import--the-load-bearing-feature)); import is
    **metadata-only** and an imported file **counts as Pinned** (§2.1). The read is metadata-only and
    honors the only-our-content defaults.
12. A scan runs the **tools preflight** first (§4.4): a missing **required** tool (`ipfs`) routes an
    interactive run to the [install-tools page](./tools.mdx) (remembering the pending scan) and makes an
    unattended scheduleTask record a **`needs_tool`** gap and **skip** that run — never installing on its
    own, never error-looping.
13. An on-demand scan is a **detached, server-side background job** (§10). The **Rescan** route returns
    immediately; the walk is **never tied to the HTTP request** that started it, so closing the tab,
    navigating away, or a request timeout **never cancels** a scan. Its **live progress** is readable
    from `GET /api/repos/scan-status` on **any** page, so the progress indicator **re-attaches** after
    the user navigates away and back.
14. The scan is **single-flight and non-blocking**: only one discovery walk runs at a time (a second
    Rescan is **coalesced** into one queued follow-up pass, never a concurrent racer on the same
    `status.yaml`), and the walk **cooperatively yields** so it does **not freeze** the web app —
    `scan-status` polls and every other page stay responsive while a scan runs.
15. A **live filesystem watcher** (§2.2) runs **while the web app is up**, subscribing to the OS's
    native change notifications (FSEvents / inotify / ReadDirectoryChangesW) over `scanner.roots`. It
    reacts to files being **added or deleted — never to a content "modified"** — and only when the path
    is **qualifying** (a video/image/audio file by `isMediaFile`, or an added file **at/over the big
    threshold**); non-media, non-big churn and any `HARD_SKIP`/`ignore_globs` path are dropped. It is
    **metadata-only** (one `stat` on an add; no content read), touches **no** IPFS and **no** network,
    **debounces** bursts (`watcher.debounce_ms`), and on a settled qualifying change **kicks the same
    single-flight, coalesced discovery worker** (`startScan`, §10) — so the `status.yaml` tracking, the
    interesting-directory coloring, and the File System tree refresh within seconds **without** waiting
    for the 4-hour pass. The watcher is `watcher.enabled` (§5), has **no** scheduler/plist, and never
    replaces the scheduled walk (which backfills changes that happened while the app was off).
16. **Parallel unit walks** (§11): the scan walks **independent repo units in parallel**, bounded by the
    **responsive budget** (`cores − 2`) via `mapLimit()`, and fans **root discovery** out the same way —
    leaving 2 cores for the HTTP loop + the IPFS node ([parallelization.mdx §1/§3](./parallelization.mdx)).
    Each walk **still cooperatively yields** (§10.3) so the app never freezes; each unit's `status.yaml`
    write is **atomic** so concurrent walks never collide; the computer-unit walk and the per-unit
    diff/write are unchanged; and the whole pass stays **single-flight** (two passes never overlap).

## 10. On-demand scan execution — the detached background job (LOCKED)

A **Rescan** (Repos page or a repo page, §2) and the scheduled launchd run (§3.1) both drive the **same
worker**, and both run it as a **server-side background job** owned by the backend process — **never**
inside, or tied to, the HTTP request that triggered it. This section is the authoritative contract for
*how* an on-demand scan runs, and it exists to fix a real, observed bug.

> #### The bug this fixes (root-cause note, do not remove)
>
> The Rescan route originally did `await scanAll()` **inside the request handler** and only responded
> when the whole walk finished, and the walk itself was **fully synchronous** (`readdirSync`/`statSync`
> in a tight loop). Two failures fell out of that:
>
> 1. **Progress was request/component state, not server state.** The only "progress" was the React
>    mutation's `isPending` on the Repos page. Navigating away unmounted it, so the spinner vanished and
>    the scan **looked stopped**; navigating back showed nothing, because the client had no way to ask
>    the server *"is a scan running, and how far along?"*
> 2. **The synchronous walk froze the whole app.** Because the walk never yielded, it **starved the Node
>    event loop for the entire scan** — `scan-status` polls, page loads, every request hung until the
>    walk finished. A frozen app reads as *"the scan (and everything) stopped."*
>
> The fix makes the scan a **detached, observable, cooperatively-yielding** background job.

### 10.1 Detached from the request (never cancel on navigation)

* The Rescan route calls `startScan(source)` and **returns immediately** (`{ started, job }`); it does
  **not** `await` the walk. The walk runs in the backend process, **outliving** the request.
* Therefore **nothing the browser does cancels a scan** — closing the tab, route-changing away from the
  Repos page, or an aborted/timed-out request. (This also fixes the scheduled path: the launchd
  `run-worker` fetch has a 60 s timeout; a full-filesystem walk can exceed it, so the internal run route
  **also** starts the detached job and returns immediately instead of blocking and being aborted.)
* The job runner **stamps `scan_process.last_run_at` / `last_run_ok` on completion** (§5) — for **both**
  manual and scheduled runs — reflecting the real finish, not the moment the trigger route returned.

The detached background job — kicked off by a route that returns immediately and never awaits the walk — is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scan-job.ts` — `startScan()` → `runJob()` (Rescan route: `reposRouter.post("/rescan")` in `repos.router.ts`)

### 10.2 Observable — live progress lives on the server

* The backend holds one **scan job record** (`ScanJob`): `status` (`idle | running | done | error`),
  `source`, `startedAt` / `finishedAt`, `phase` (`discovering → repos → computer → done`),
  `reposTotal` / `reposDone`, `currentUnit`, `candidatesFound`, and `error` / `ok`.
* `GET /api/repos/scan-status` returns it. The web app mounts **one global progress bar** (in the app
  shell, present on **every** page) that **polls** this endpoint — fast (~1 s) while running, slow
  (~5 s) when idle so a scan started **elsewhere** (a scheduled run, or Rescan on another tab/computer)
  still lights it up. Because the state is on the server and the bar is always mounted, **navigating
  away and back simply re-attaches** to the same live job — the progress on the bottom returns.
* The Repos page's **Rescan button reflects the *server* scan status**, not local mutation state — it
  shows *"Scanning…"* and is disabled whenever a scan is running, even one it did not start.

The single always-mounted progress bar that polls the server job and re-attaches after navigation is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ScanProgressBar.tsx` — `ScanProgressBar` (polls `GET /api/repos/scan-status`, served by `getScanJob()`)

### 10.3 Single-flight, coalesced, non-blocking

* **One at a time.** `startScan` is single-flight: if a scan is already running it does **not** start a
  second (which would race the first on the same `status.yaml` files). It instead sets a **rerun
  flag** and returns `{ started: false }`; when the current pass finishes, **exactly one** follow-up
  pass runs. This coalesces double-clicks **and** guarantees a repo **added mid-scan** is still covered.
* **Cooperative yielding.** The walk (`walkUnit`, the computer-unit walk, and repo discovery
  `findGitRepos`) **yields to the event loop** every few hundred entries (`await setImmediate`). This
  keeps the HTTP server responsive throughout a scan — verified: `scan-status` polls return in
  **milliseconds** while a 90-repo / thousands-of-candidates walk is in flight, with `reposDone` and
  `candidatesFound` advancing between polls. The `stat`-only, no-content contract (§1) is unchanged;
  yielding changes only *when* the synchronous stats run, never *what* they read.

Single-flight coalescing and the cooperative event-loop yielding that keeps the app responsive during a walk are in code:

## 11. Parallel unit walks — many units at once (LOCKED)

A scan is dozens of **independent** walks: each repo unit's `stat`-only walk touches only its own
subtree and writes only its own `status.yaml`, and root discovery over the scanner roots is likewise
independent per root. Walking them **one after another** — the current `for (const folder of
repoFolders)` loop in `scanAll()` — leaves a many-core machine idle while a single
`readdirSync`/`statSync` walk plods through one repo at a time. Big-file discovery is
**embarrassingly parallel**, so it fans **wide**.

* **Repo units fan out under the responsive budget.** The repo-units loop becomes a bounded
  `mapLimit()` sized to the **responsive budget** (`cores − 2`), so up to that many repo walks run at
  once while **2 cores stay free for the HTTP loop + the IPFS node**. This is the right budget — a scan
  runs **alongside** interactive use, not as user-kicked-off mass compute — exactly as
  [parallelization.mdx §1/§3](./parallelization.mdx) assigns the "Scan — repo units" row to the
  **responsive**, **wide** shape. Root discovery (`findGitRepos` over each root) fans out the same way.
  Being `stat`/IO-bound, these walks want to run **wide** over independent units, not narrow with capped
  threads — that anti-oversubscription rule ([parallelization.mdx §2](./parallelization.mdx)) is for the
  internally-multi-threaded compress/transcode jobs, not for stat-only walks.
* **The §10.3 cooperative yield is preserved, not replaced.** Each walk still `await`s `yieldToLoop()`
  every `YIELD_EVERY` entries, so even with many walks in flight the event loop keeps servicing
  `scan-status` polls and page loads — the app never freezes. Fan-out and yielding are complementary:
  the fan-out uses more cores, the yield keeps each walk cooperative on the core it lands on.
* **The computer-unit walk and the per-unit diff/write are unchanged.** The computer unit is still
  walked **after** the repo units (its mask is the set of discovered repos), and each unit's
  `status.yaml` write stays **atomic** (temp-then-rename through the store layer), so concurrent unit
  walks **never collide** — every unit owns a distinct status file and touches only its own subtree.
* **Single-flight is unchanged (§10.3).** Parallelism is *within* one pass; two whole passes still never
  overlap — a second Rescan coalesces into one queued follow-up, never a concurrent racer on the same
  `status.yaml` files.

The repo-units loop running as a `mapLimit()` at the responsive budget while each walk still yields to the event loop is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` — `scanAll()` (the `for (const folder of repoFolders)` loop becomes `mapLimit(repoFolders, responsiveBudget(), …)`; `walkUnit()` / `findGitRepos()` keep their `await yieldToLoop()`), drawing `responsiveBudget()` + `mapLimit()` from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts`.
