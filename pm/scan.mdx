# LargeFileBridge ‚Äî The Scanning Service (Big-File Discovery)

The **scan** is the periodic pass that answers one question across all of a user's computers:
**"Have any big files ‚Äî or the repos that hold them ‚Äî changed since last time?"** It discovers new
large files, notices files that have grown past the "big" threshold, spots moves/renames/deletes, and
registers repos that have appeared or vanished. It is the eyes of LargeFileBridge (LFB); the
[background sync](./storage.mdx#13-the-background-sync-process--the-transparency-contract) is the
hands that move the bytes.

Two scheduled jobs, two cadences, on purpose:

| Job (scheduleTask) | Cadence | What it does | Where spec'd |
| --- | --- | --- | --- |
| **Discovery scheduleTask** (this file) | **every 4 hours** | Deep, whole-filesystem **discovery** ‚Äî stat every candidate file, find big files + filesystem/repo changes. Metadata only; no IPFS. | this file |
| **Sync scheduleTask** | every **15 minutes** | A **full sync pass** ([sync_process.mdx](./sync_process.mdx)): visit + reconcile **every** known unit; for units the user **opted into IPFS** (`synced: true`) add/update manifests, pin, fetch missing CIDs over IPFS **‚Äî and read the local IPFS pinset (`ipfs pin ls`) to reconcile + import already-pinned files** (¬ß2.1). Runs units concurrently (many cores). | [storage.mdx ¬ß13](./storage.mdx#13-the-background-sync-process--the-transparency-contract) ¬∑ [sync_process.mdx](./sync_process.mdx) |

The discovery pass is expensive (it walks the tree and stats many files), so it is throttled to **every
four hours**. The 15-minute pass is cheap per unit and time-sensitive, so it runs over the small set the
discovery pass has already found. The 15-min pass may do a *light* rescan of a unit it is about to sync;
the **deep, full-filesystem discovery walk is the 4-hour job's alone.**

### Naming ‚Äî "scheduleTask" internally, "scan" in the UI (LOCKED)

The scheduled background jobs are called **scheduleTasks** in code, config, and scheduler labels ‚Äî **not
"sync"** as the generic term. In the **UI** the user sees what they do as **"scans"** (the left-bar
[**Scans**](./left_bar.mdx) page, formerly *Sync*, is their transparency+control surface). Two names,
one concept: a *scheduleTask* is the mechanism; a *scan* is what the user watches it do.

* This rename covers the **scheduled process only**. The per-file **Sync** *decision*
  ([one_repo.mdx ¬ß1](./one_repo.mdx#1-the-decision-model-locked)) and the portable **Sync List**
  ([sync_list.mdx](./sync_list.mdx)) are **different concepts and keep their names.**
* The config blocks stay where they are ‚Äî `scan_process` (this discovery task, ¬ß5) and `sync_process`
  ([storage.mdx ¬ß13](./storage.mdx#13-the-background-sync-process--the-transparency-contract), the
  15-min task) ‚Äî but both are understood as **scheduleTasks**, both surfaced as **scans** in the UI, and
  both gated by the **tools preflight** (¬ß4.4).

> This file is the scanning-service spec. It refines the scanner described in
> [storage.mdx ¬ß7](./storage.mdx#7-the-scanner--discovering-repos-and-the-mask) into a scheduled,
> cross-platform service with its own cadence, and it is the authoritative source for the **scan
> interval**. The **"big" threshold** it applies is owned by [settings.mdx](./settings.mdx)
> (global default **100 MB**, per-repo overridable). See "Reconciliation" (¬ß9).

Tech stack (repo `CLAUDE.md`): **TypeScript on Node** ‚Äî the scan worker is a Node/TS entry point, run
by the OS scheduler. Never raw shell/`find`/`du`.

## 1. What the scan does ‚Äî and deliberately does NOT do

The scan is a **metadata-only** pass. For every candidate path it reads only what the filesystem
already knows without opening the file:

* **name / path**
* **size in bytes** (from `stat`)
* **mtime** (last-modified time, from `stat`)
* file vs. directory vs. symlink

It **never opens or reads file contents.** It does **not** hash files, does **not** compute IPFS
CIDs, and does **not** touch IPFS at all. Hashing and CID computation are the sync's job, and only for
files the scan has already flagged as big and that live in a `synced: true` unit. Keeping the scan to
pure `stat` is what makes it cheap enough to run over the whole filesystem every four hours.

What the scan produces is a **candidate list** per sync unit: the files that are large-file-bridge
payload per the **payload rule** (¬ß4.1) ‚Äî git-ignored media at *any* size, plus anything at/above the
big threshold ‚Äî with a diff against the previous scan (added / grown / shrunk / moved / deleted),
written to that unit's `status.yaml` (¬ß6). It changes **no** file contents and moves **no** bytes.

The metadata-only discovery pass that stats candidates and never opens files runs in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `scanAll()` ‚Üí `walkUnit()` (uses `fs.statSync` only)

## 2. Cadence ‚Äî the two scheduleTasks (4-hour discovery + 15-minute pinset)

* The **discovery scheduleTask** (this file) runs **once every four hours** (`interval_hours: 4`, i.e.
  14400 seconds).
* It also runs **on demand** from the web app ‚Äî a **"Rescan"** action on the Repos page and on any
  repo's page triggers the same worker immediately, out of band from the schedule.
* Like the 15-minute scheduleTask, it has the same **installed vs. on/off** transparency contract
  (¬ß7): the plist/timer/task can be installed but disabled, and the user toggles it from the web app.
* Cadence is a setting, not a constant: `config.yaml ‚Üí scan_process.interval_hours` (¬ß5). Four hours
  is the default; the UI may expose it later. Changing it re-writes the schedule (¬ß4).

### 2.1 The 15-minute scheduleTask reads the IPFS pinset (import already-pinned)

The **15-minute scheduleTask** ([storage.mdx ¬ß13](./storage.mdx#13-the-background-sync-process--the-transparency-contract))
does the byte work ‚Äî but it now also **reads the local IPFS pinset (`ipfs pin ls`)** every run and
**reconciles it against the manifests**, so *"what files are being IPFS-pinned locally"* is always
current without waiting on the 4-hour walk. Charter: *"Part of that scan is to look at what files are
being IPFS-pinned locally ‚Ä¶ make that happen at the 15-minute [interval]."*

* **Reconcile pins ‚Üî manifests.** Every pinned CID **in** a manifest ‚Üí confirm this computer's
  `pinned_by` claim (or, if the manifest claimed it but the pin is gone, drop the claim + queue a
  re-pin ‚Äî [sync_list ¬ß5.1](./sync_list.mdx#51-local-claims-are-verified-not-believed)). Every pinned
  CID **not** in any manifest ‚Üí an **Untracked (pinned)** import candidate.
* **Import already-pinned.** *"If we get to a computer and the files are already IPFS-pinned, import
  them."* Untracked pins are surfaced on the [IPFS page](./ipfs.mdx#4-import--the-load-bearing-feature)
  and adopted into tracking; import is **metadata-only** (record the manifest entry, mark the file
  **Sync**) ‚Äî the bytes are already pinned, so nothing is fetched, re-added, or re-pinned. An imported
  file is present-and-pinned, i.e. it **counts as Synced** ([repos.mdx ¬ß3.2](./repos.mdx#32-columns)).
* **Still metadata-only.** Reading the pinset is a `pin ls` + `stat`; it opens no file and moves no
  bytes, exactly like the discovery pass (¬ß1). It is the one place the *scan* touches IPFS at all ‚Äî and
  only to *list* pins, never to serve or announce anything beyond the
  [only-our-content](../knowledge/ipfs.mdx#6-only-our-content-the-non-negotiable-rule) defaults.
* **Needs the `ipfs` CLI.** The pinset read requires Kubo; if it is missing the preflight (¬ß4.4)
  records a `needs_tool` gap and the app routes the user to [install it](./tools.mdx).

The 15-minute pass that reads the local pinset and reconciles/refreshes each `pinned_by` claim against the manifests runs in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` ‚Äî `syncRepoFolder()` (reads `ipfs.listPins()`, refreshes via `setPinClaim()`)

## 3. Cross-platform ‚Äî runs on Mac, Windows, and Linux

LFB runs on the **Mac filesystem** primarily, but also on **Windows and Linux**. The scan **worker
is one cross-platform Node/TS program**; only the *scheduler that launches it* differs per OS. The
worker code path is identical everywhere ‚Äî only path normalization and the schedule installer branch
on platform.

| OS | Scheduler | Unit | Primary? |
| --- | --- | --- | --- |
| **macOS** | **launchd** | a `.plist` LaunchAgent (`StartInterval` = 14400) | **Yes ‚Äî primary target** |
| **Linux** | cron **or** systemd timer | a crontab line / `.timer` + `.service` | secondary |
| **Windows** | Task Scheduler | a scheduled task (4-hour trigger) | secondary |

The **Mac launchd plist is the primary, first-class implementation** ("the plist for Mac ‚Ä¶ the Mac
equivalent specific one"). Windows and Linux schedulers are supported and follow the same
installed / on-off model, implemented per-OS behind one installer interface.

### 3.1 The Mac launchd plist

A LaunchAgent, distinct from the sync job's plist (`com.largefilebridge.sync`,
[storage.mdx ¬ß3](./storage.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)):

```xml
<!-- ~/Library/LaunchAgents/com.largefilebridge.scan.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>                <string>com.largefilebridge.scan</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/local/bin/node</string>
    <string>/path/to/LargeFileBridge/code/.../scan-worker.js</string>
  </array>
  <key>StartInterval</key>        <integer>14400</integer>   <!-- every 4 hours -->
  <key>RunAtLoad</key>            <false/>                    <!-- honor the on/off toggle -->
  <key>StandardOutPath</key>      <string>~/T/_large_files_bridge/log.log</string>
  <key>StandardErrorPath</key>    <string>~/T/_large_files_bridge/error.err</string>
</dict>
</plist>
```

* **Installed** = the plist exists at `~/Library/LaunchAgents/com.largefilebridge.scan.plist`.
* **On** = the job is `launchctl bootstrap`/`enable`d; **off** = `bootout`/`disable`d. Installed and
  on/off are two independent facts (¬ß7), mirrored in `config.yaml ‚Üí scan_process`.
* stdout/stderr flow into the same two log files as the rest of LFB
  ([storage.mdx ¬ß12](./storage.mdx#12-logging)).

The Mac launchd LaunchAgent plist is rendered and installed/enabled/disabled via `launchctl` in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/os/launchd.ts` ‚Äî `launchdInstaller` ‚Üí `install()` (renders via `renderPlist()`)

## 4. Roots, discovery, and the mask

The scan walks the **`scanner.roots`**
([storage.mdx ¬ß3](./storage.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)),
honoring `scanner.ignore_globs` and `follow_symlinks`. In one pass it does three things, in order:

1. **Repo discovery.** Any directory containing a `.git` folder is registered/reconciled as a repo
   sync unit (`sync/r/<repo_name>/`). New repos are created (default `synced: false` until the user
   opts in); already-known repos keep their settings; repos gone from disk are marked `missing` in
   their `status.yaml` ‚Äî never silently deleted
   ([storage.mdx ¬ß7](./storage.mdx#7-the-scanner--discovering-repos-and-the-mask)).
2. **Build the mask.** The set of registered repo working-tree paths becomes the **mask** for the
   computer unit, so every file is attributed to exactly **one** unit ‚Äî a repo unit or the computer
   unit, never both.
3. **Flag candidates.** Walk each unit's scope, `stat` every file, and record every file that is a
   large-file-bridge payload (¬ß4.1). Diff against the previous scan to classify each change.

Repo discovery, registering each `.git` working tree, and building the repo mask before the per-unit walk happen in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `scanAll()` (calls `findGitRepos()` then `registerRepo()`)

The scan **never descends** into `scanner.ignore_globs` (e.g. `**/node_modules/**`, `**/.git/**`,
`**/.Trash/**`), a unit's own `exclude_globs`, or the **hard-skip directory set** (¬ß4.3) ‚Äî this is
both correctness (don't bridge build junk) and the main performance lever.

> **The hard-skip set is one source of truth.** The directories the scan refuses to descend into are
> defined once in `code/.../shared/scan-filters.ts` (`HARD_SKIP`) and imported by **both** the scan
> worker and the File System browser's badge pass, so the two can never drift (the invariant
> previously written as a comment in `badges.ts`). See ¬ß4.3.

### 4.1 What counts as a candidate ‚Äî the payload rule (LOCKED)

A file is a **candidate** (a large-file-bridge payload the scan records) when **any** of these hold:

1. **Git-ignored media ‚Äî regardless of size.** The file matches the repo's `.gitignore` **and** is a
   **video / audio / image** (`isMediaFile`, defined in `shared/scan-filters.ts`). A git-ignored
   `.mp4`/`.mp3`/`.png` is the *whole reason LFBridge exists* ‚Äî the user deliberately kept it out of
   git to bridge it over IPFS (README charter). **The big-file size threshold does NOT gate these.**
2. **Any file at or above the effective big threshold** (¬ß4.2) ‚Äî the classic "big file" case. For a
   repo with a followed `.gitignore`, this still requires the file to be git-ignored (a big file that
   is *checked in* is not our payload); the **computer unit** (no gitignore) uses the pure size gate.
3. **A forced include-glob match** (`large_files.include_globs`) ‚Äî an explicit user opt-in.

The candidate predicate ‚Äî git-ignored media at any size, else the size gate, plus forced includes ‚Äî is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `walkUnit()` (`isCandidate`, using `isMediaFile()` from `shared/scan-filters.ts`)

> #### Why this rule exists ‚Äî the bug it fixes (root-cause note, do not remove)
>
> The scan originally required a repo file to be **big AND git-ignored** (`size ‚â• threshold &&
> gitIgnored`). Real repos broke this: in `charlie-kirk` the git-ignored `videos/` + `IPFS/videos/`
> folders hold **46** media files (24 `.mp4` + 22 `.mp3`) but only **3** exceed 100 MB ‚Äî so the scan
> reported **3 of 46** and the repo looked almost empty. `Intel_Murder_Docus` reported **2 of 27**
> for the same reason (`static/videos/*` is 16 `.mp4` + 11 `.mp3`, mostly 5‚Äì100 MB). The 100 MB
> default (GitHub's hard limit) is the right nudge for *checked-in* files, but it must **never hide
> git-ignored media the user has already chosen to bridge**. After the fix the counts are the full
> **46** and **27** ‚Äî verified by rescanning both repos and diffing `status.yaml` against a
> filesystem census. The `stat`-only, no-content contract (¬ß1) is unchanged; only the *predicate* for
> "is this a candidate" changed.
>
> **Non-media git-ignored files are deliberately NOT bypassed** ‚Äî a git-ignored `.env`, `.log`, or
> `.DS_Store` must still clear the size threshold, so junk and secrets aren't swept into the bridge.

### 4.2 The effective big threshold

Whether a file is "big" is decided **per unit** by the threshold resolution in
[settings.mdx](./settings.mdx):

```
effective_threshold(file) =
  the repo's override        if that repo's "override big threshold" checkbox is ON
  else the global setting    (default 100 MB)
```

The scan reads the resolved threshold for each unit and flags files whose **size ‚â• threshold**.
The scan does **not** define the threshold or its default ‚Äî it only applies it (¬ß9 Reconciliation).
The threshold gates cases 2 and 3 of ¬ß4.1 only; git-ignored **media** (case 1) ignores it.

Resolving the per-unit effective threshold (repo override else the global default) happens in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `resolveThreshold()`

### 4.3 The hard-skip directory set (never descend)

Some directories are never walked at all ‚Äî regenerated build output and dependency/VCS/OS junk. They
are listed once in `shared/scan-filters.ts` as `HARD_SKIP` and shared with the FS browser:

```
.git  node_modules  .Trash  .cache  Caches            ‚Üê vcs / deps / os junk
build  dist  out  .docusaurus  .next  .nuxt  .svelte-kit  .turbo  coverage   ‚Üê build outputs
```

**Why build outputs are skipped (second root-cause note):** a Docusaurus/Next build **copies** the
source media into its output dir, so before this list the scan reported `Intel_Murder_Docus`'s hero
video **twice** ‚Äî once as `static/videos/x.mp4` and once as the generated `build/videos/x.mp4`.
Build output is ephemeral and regenerated; it is **never** a sync candidate. Skipping these dirs
removes the phantom duplicates (Intel dropped from 27+build-dupes to a clean **27**) and is also the
main performance lever on large trees.

The single-source-of-truth hard-skip directory set, shared with the FS-browser badge pass, lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/scan-filters.ts` ‚Äî `HARD_SKIP`

### 4.4 Preflight ‚Äî required tools before a scan (brew-install)

Charter: the scan *"can and will use brew-install tools ‚Ä¶ Before it does a scan, see if any are missing
and show that page to ask the user to install them."* So **before** a scan/scheduleTask runs, LFB runs
the **tools preflight** ([tools.mdx ¬ß3](./tools.mdx#3-the-preflight--check-before-you-scan-locked)):

* It probes the tools the run needs on `PATH` ‚Äî **`ipfs` (Kubo)** is **required** (the pinset read ¬ß2.1
  and all IPFS work depend on it); **`ffmpeg`/`ffprobe`** and image tooling are **optional** (they
  sharpen the compress `C`/`c` judgment but never block discovery).
* **Interactive run** (a user clicked Rescan): a missing **required** tool routes to the
  [install-tools page](./tools.mdx) ‚Äî detect Homebrew, list each tool with status + install command, and
  a one-click **Install** (`brew install ipfs ffmpeg`) ‚Äî with the pending scan **remembered** to resume
  after install. Nothing installs without an explicit click (charter consent).
* **Scheduled run** (the `launchd` scheduleTask, unattended): a missing **required** tool is recorded as
  a **`needs_tool`** status the web app surfaces (banner + notification), and that run is **skipped** ‚Äî
  the scheduleTask never installs and never error-loops. The user installs the tool next time they open
  the app; the schedule then resumes.

## 5. Config ‚Äî `scan_process` in the app config

The scan's schedule/state is a hierarchical block in the app-level, computer-wide `config.yaml`
([storage.mdx ¬ß3](./storage.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)),
parallel to `sync_process`:

```yaml
# ~/T/_large_files_bridge/config.yaml  (excerpt ‚Äî the scanning service)
scan_process:
  installed: false            # is the scan scheduler (launchd/cron/task) installed on this machine?
  enabled: false              # is it currently ON? (user toggles from the web app)
  interval_hours: 4           # cadence ‚Äî the scan runs every 4 hours
  label: com.largefilebridge.scan   # launchd label (Mac) / task name / timer name
  last_run_at: 2026-07-02T12:00:00Z # machine-written: when the scan last completed
  last_run_ok: true           # machine-written: did the last scan finish without a fatal error?
```

The existing `scanner:` block (roots / ignore_globs / follow_symlinks) stays where it is in
`config.yaml` ‚Äî `scan_process` adds only the **scheduling + on/off** facts, exactly mirroring how
`sync_process` is structured.

## 6. Output ‚Äî what the scan writes

The scan writes **only** operational metadata; it never writes a manifest or moves bytes. Per sync
unit it updates that unit's `status.yaml`
([storage.mdx ¬ß6.3 / ¬ß8.2](./storage.mdx#5-the-two-kinds-of-sync-unit--the-core-model)):

```yaml
# sync/r/<repo_name>/status.yaml  (excerpt ‚Äî written by the scan)
schema_version: 1
last_scan_at: 2026-07-02T12:00:03.114Z
scan_source: scheduled            # scheduled | manual (Rescan button)
effective_threshold_bytes: 104857600   # 100 MB ‚Äî what "big" meant for THIS unit this run
big_file_count: 7                 # files at/above the threshold right now
big_file_bytes: 5629499534        # total bytes of those big files
changes_since_last_scan:
  added:   [large/trailer_v4.mp4]        # newly appeared big files
  grew:    [assets/render_out.exr]       # crossed the threshold since last scan
  shrank:  []                            # dropped below the threshold
  moved:   [{ from: old/hero.png, to: assets/hero.png }]  # same size+mtime, new path
  deleted: [large/trailer_v2.mp4]        # gone from disk
repo_state: present               # present | missing (drive unmounted / repo deleted)
```

* **Move/rename detection** uses size + mtime heuristics only (no content hashing) ‚Äî a file that
  vanished at path A and appeared at path B with the same size and mtime is reported as a **move**,
  not a delete+add, so the sync can keep the existing pin instead of re-adding bytes.
* The **computer unit** (`sync/computer/status.yaml`) gets the same shape for everything outside
  repos.
* The scan **stamps `last_scan_at`** and the change set; the **sync** later reads these to know what
  work to do. The scan itself pins nothing and fetches nothing.

The web app reads these `status.yaml` files to show, per repo and on the Sync page: last scan time,
big-file counts, and "N new big files found" badges ‚Äî all without the scan ever having opened a file.

Diffing against the previous scan and writing the added/grew/deleted change set to each unit's `status.yaml` happens in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `diffStatus()` (persisted via `writeStatus()`)

## 7. The transparency contract (installed vs. on/off)

Identical in spirit to the sync process
([storage.mdx ¬ß13](./storage.mdx#13-the-background-sync-process--the-transparency-contract)) ‚Äî the
web app is transparent about the scan and lets the user control it:

* **Installed** ‚Äî is the scan scheduler present on this machine? (`scan_process.installed`, the plist
  on Mac / cron entry or timer on Linux / task on Windows.) It must be installed for the scan to
  exist at all.
* **On / off** ‚Äî is it currently enabled? (`scan_process.enabled`.) Separate from installed: the user
  can install it and still leave it off.
* The web app **displays** both facts (reconciled with the real launchd/cron/task state) and lets the
  user **install/uninstall** and **toggle on/off** from the UI, plus **Rescan now** on demand.

This surfaces on the **Sync** page alongside the 15-min sync's own installed/on-off state, so the two
scheduled jobs are visible side by side.

Reading each worker's installed/enabled state reconciled against real launchd state, and the install/uninstall/enable/disable controls, live in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/schedule.service.ts` ‚Äî `workerState()` / `control()`

## 8. Acceptance criteria

The scan is correct when **all** hold:

1. The scan runs **every 4 hours** by default (`scan_process.interval_hours: 4`), on its own schedule
   **separate** from the every-15-minute sync, and also runs on demand from a **Rescan** action.
2. The scan is **metadata-only**: it `stat`s files for name/size/mtime and **never opens, reads, or
   hashes file contents**, and **never touches IPFS**.
3. The scan is **cross-platform**, scheduled by a **launchd plist on Mac** (primary), and by cron /
   systemd timer on Linux and Task Scheduler on Windows ‚Äî one shared Node/TS worker, per-OS installer.
4. In one pass the scan **discovers/reconciles repos**, builds the **repo mask**, and records every
   **candidate** per the payload rule (¬ß4.1), attributing each file to exactly one sync unit.
4a. **Git-ignored media (video/audio/image) is a candidate regardless of size** ‚Äî the size threshold
    must **never** hide the 5‚Äì100 MB git-ignored videos/images a user has chosen to bridge. Regression
    guard: rescanning `charlie-kirk` yields **46** candidates (not 3) and `Intel_Murder_Docus` yields
    **27** (not 2). Non-media git-ignored files (`.env`, `.log`, `.DS_Store`) are **not** bypassed.
4b. **Build-output directories are never walked** (`build`, `dist`, `.docusaurus`, `.next`, ‚Ä¶ ‚Äî the
    shared `HARD_SKIP`, ¬ß4.3), so a build's *copy* of the source media is not counted as a second
    candidate. `Intel_Murder_Docus` shows **no** `build/‚Ä¶` paths in its candidate list.
5. The effective threshold is resolved **per unit** ‚Äî the repo's override when its checkbox is on,
   else the **global default (100 MB)** ‚Äî read from [settings.mdx](./settings.mdx); the scan applies
   it but does not define it. It gates only the non-media cases of ¬ß4.1 (case 2 and case 3).
6. The scan writes **only** operational metadata to each unit's `status.yaml` (last scan time,
   big-file counts, and an added/grew/shrank/moved/deleted change set); it writes **no** manifest and
   moves **no** bytes.
7. Move/rename is detected by size+mtime (no hashing) and reported as a **move**, not delete+add.
8. `config.yaml ‚Üí scan_process` records **installed** and **enabled** as two independent facts, both
   displayed in the web app, with install/uninstall + on/off toggle + Rescan available from the UI.
9. Ignored paths (`scanner.ignore_globs`, per-unit `exclude_globs`) are never descended into.
10. The scheduled jobs are **scheduleTasks** in code/config/labels and **"scans"** in the UI; the rename
    covers the **scheduled process only** ‚Äî the per-file **Sync** decision and the **Sync List** keep
    their names (Naming, ¬ß2).
11. The **15-minute scheduleTask reads the IPFS pinset** (`ipfs pin ls`) each run and **reconciles it
    against the manifests** ‚Äî confirming/dropping `pinned_by` claims and surfacing **pinned-but-untracked**
    files as **import** candidates ([IPFS ¬ß4](./ipfs.mdx#4-import--the-load-bearing-feature)); import is
    **metadata-only** and an imported file **counts as Synced** (¬ß2.1). The read is metadata-only and
    honors the only-our-content defaults.
12. A scan runs the **tools preflight** first (¬ß4.4): a missing **required** tool (`ipfs`) routes an
    interactive run to the [install-tools page](./tools.mdx) (remembering the pending scan) and makes an
    unattended scheduleTask record a **`needs_tool`** gap and **skip** that run ‚Äî never installing on its
    own, never error-looping.
13. An on-demand scan is a **detached, server-side background job** (¬ß10). The **Rescan** route returns
    immediately; the walk is **never tied to the HTTP request** that started it, so closing the tab,
    navigating away, or a request timeout **never cancels** a scan. Its **live progress** is readable
    from `GET /api/repos/scan-status` on **any** page, so the progress indicator **re-attaches** after
    the user navigates away and back.
14. The scan is **single-flight and non-blocking**: only one discovery walk runs at a time (a second
    Rescan is **coalesced** into one queued follow-up pass, never a concurrent racer on the same
    `status.yaml`), and the walk **cooperatively yields** so it does **not freeze** the web app ‚Äî
    `scan-status` polls and every other page stay responsive while a scan runs.

## 10. On-demand scan execution ‚Äî the detached background job (LOCKED)

A **Rescan** (Repos page or a repo page, ¬ß2) and the scheduled launchd run (¬ß3.1) both drive the **same
worker**, and both run it as a **server-side background job** owned by the backend process ‚Äî **never**
inside, or tied to, the HTTP request that triggered it. This section is the authoritative contract for
*how* an on-demand scan runs, and it exists to fix a real, observed bug.

> #### The bug this fixes (root-cause note, do not remove)
>
> The Rescan route originally did `await scanAll()` **inside the request handler** and only responded
> when the whole walk finished, and the walk itself was **fully synchronous** (`readdirSync`/`statSync`
> in a tight loop). Two failures fell out of that:
>
> 1. **Progress was request/component state, not server state.** The only "progress" was the React
>    mutation's `isPending` on the Repos page. Navigating away unmounted it, so the spinner vanished and
>    the scan **looked stopped**; navigating back showed nothing, because the client had no way to ask
>    the server *"is a scan running, and how far along?"*
> 2. **The synchronous walk froze the whole app.** Because the walk never yielded, it **starved the Node
>    event loop for the entire scan** ‚Äî `scan-status` polls, page loads, every request hung until the
>    walk finished. A frozen app reads as *"the scan (and everything) stopped."*
>
> The fix makes the scan a **detached, observable, cooperatively-yielding** background job.

### 10.1 Detached from the request (never cancel on navigation)

* The Rescan route calls `startScan(source)` and **returns immediately** (`{ started, job }`); it does
  **not** `await` the walk. The walk runs in the backend process, **outliving** the request.
* Therefore **nothing the browser does cancels a scan** ‚Äî closing the tab, route-changing away from the
  Repos page, or an aborted/timed-out request. (This also fixes the scheduled path: the launchd
  `run-worker` fetch has a 60 s timeout; a full-filesystem walk can exceed it, so the internal run route
  **also** starts the detached job and returns immediately instead of blocking and being aborted.)
* The job runner **stamps `scan_process.last_run_at` / `last_run_ok` on completion** (¬ß5) ‚Äî for **both**
  manual and scheduled runs ‚Äî reflecting the real finish, not the moment the trigger route returned.

The detached background job ‚Äî kicked off by a route that returns immediately and never awaits the walk ‚Äî is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scan-job.ts` ‚Äî `startScan()` ‚Üí `runJob()` (Rescan route: `reposRouter.post("/rescan")` in `repos.router.ts`)

### 10.2 Observable ‚Äî live progress lives on the server

* The backend holds one **scan job record** (`ScanJob`): `status` (`idle | running | done | error`),
  `source`, `startedAt` / `finishedAt`, `phase` (`discovering ‚Üí repos ‚Üí computer ‚Üí done`),
  `reposTotal` / `reposDone`, `currentUnit`, `candidatesFound`, and `error` / `ok`.
* `GET /api/repos/scan-status` returns it. The web app mounts **one global progress bar** (in the app
  shell, present on **every** page) that **polls** this endpoint ‚Äî fast (~1 s) while running, slow
  (~5 s) when idle so a scan started **elsewhere** (a scheduled run, or Rescan on another tab/computer)
  still lights it up. Because the state is on the server and the bar is always mounted, **navigating
  away and back simply re-attaches** to the same live job ‚Äî the progress on the bottom returns.
* The Repos page's **Rescan button reflects the *server* scan status**, not local mutation state ‚Äî it
  shows *"Scanning‚Ä¶"* and is disabled whenever a scan is running, even one it did not start.

The single always-mounted progress bar that polls the server job and re-attaches after navigation is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/components/ScanProgressBar.tsx` ‚Äî `ScanProgressBar` (polls `GET /api/repos/scan-status`, served by `getScanJob()`)

### 10.3 Single-flight, coalesced, non-blocking

* **One at a time.** `startScan` is single-flight: if a scan is already running it does **not** start a
  second (which would race the first on the same `status.yaml` files). It instead sets a **rerun
  flag** and returns `{ started: false }`; when the current pass finishes, **exactly one** follow-up
  pass runs. This coalesces double-clicks **and** guarantees a repo **added mid-scan** is still covered.
* **Cooperative yielding.** The walk (`walkUnit`, the computer-unit walk, and repo discovery
  `findGitRepos`) **yields to the event loop** every few hundred entries (`await setImmediate`). This
  keeps the HTTP server responsive throughout a scan ‚Äî verified: `scan-status` polls return in
  **milliseconds** while a 90-repo / thousands-of-candidates walk is in flight, with `reposDone` and
  `candidatesFound` advancing between polls. The `stat`-only, no-content contract (¬ß1) is unchanged;
  yielding changes only *when* the synchronous stats run, never *what* they read.

Single-flight coalescing and the cooperative event-loop yielding that keeps the app responsive during a walk are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/scanner/scanner.service.ts` ‚Äî `walkUnit()` / `findGitRepos()` (`await yieldToLoop()` every `YIELD_EVERY`; single-flight `rerunQueued` in `scan-job.ts`)
