# LargeFileBridge — The Git Backbone Engine

A storage's small, travelling text — its descriptor, its device registry, its mapped-directory list, its
IPFS manifest, and its expensive-to-generate derived artifacts — has to get **between the user's
computers**. For most storages the vehicle for that text is a **Git repository**: the
[Syncable Data Location](./syncable_data_location.mdx) *is* a little Git repo, and Git is exactly the
tool for versioning small text and reconciling edits made on two machines at once.

This spec is the **authoritative** description of how LFB drives that Git repo. When the user turns the
**Git backbone** on for a storage ([storage_settings.mdx §4.1](./storage_settings.mdx#41-dedicated-git-repo-preferred-when-available)),
they are **granting LFB permission** to run the real Git commands on that repo — **`git fetch`,
`git merge` (with automatic merge), `git add` / `git commit`, and `git push`** — on **every pin pass**.
Nothing else in the product moves the SDL's YAML between computers; this engine does.

Read the shared model first; this spec does not restate it:

* [storage_settings.mdx §4.1](./storage_settings.mdx#41-dedicated-git-repo-preferred-when-available) —
  the per-storage setting that **turns this backbone on** and holds the remote (a local path or a URL).
* [storage_personal.mdx](./storage_personal.mdx) — the **canonical use case**: one user, two computers,
  one Personal Git repo, the same repo configured on both. That spec drives every requirement here.
* [syncable_data_location.mdx](./syncable_data_location.mdx) — **what** travels (the SDL's YAML
  inventory, §2) and the **trust/reconcile** rule this engine obeys when two devices edit at once.
* [devices.mdx](./devices.mdx) — the **self-owned** `devices/<device>.yaml` files that are the main
  thing this engine commits and pushes; self-ownership is what makes auto-merge safe (§4).
* [pin_process.mdx](./pin_process.mdx) — the **full pass** this pull/merge/push cycle runs inside,
  once per storage that has a Git backbone.

## 1. What LFB is permitted to do — the granted Git operations (LOCKED)

Turning on the Git backbone for a storage is an **explicit, user-confirmed grant**. Until the user
enables it and supplies a remote, LFB runs **no** Git command against that storage beyond the
already-existing `git init` of a brand-new dedicated repo. Once enabled, LFB has permission to run — and
**only** — this closed set of operations against **that one configured repo**:

| Operation | Command | When | Why |
| --- | --- | --- | --- |
| **Fetch** | `git fetch <remote>` | start of every pass | pull down what the user's *other* computers pushed |
| **Merge** | `git merge --no-edit` (ort strategy) | after fetch, if the remote advanced | fold the other computers' SDL edits into this working copy, **automatically** |
| **Stage** | `git add <this device's own files>` | after local changes | queue only **this** device's self-owned files (§4) |
| **Commit** | `git commit --no-edit` | if anything is staged | record this device's SDL edits |
| **Push** | `git push <remote> <branch>` | after commit | send this device's edits to the user's other computers |

Two hard boundaries the charter imposes (§7):

* **Only the user's own configured repos.** LFB fetches/merges/pushes **exactly** the repo the user
  pointed a storage's Git backbone at — never a repo it merely *discovered*, never a third party's repo,
  never a public one. This is the Git analogue of the charter's *no-public-relay* posture: we move
  **our own** content between **our own** machines, nothing else.
* **Never destructive without cause.** LFB never runs `push --force`, never `reset --hard` a user's
  work, never `clean` their tree. The only writes it makes are the additive commit of its own files and
  a fast-forwardable push; anything it cannot reconcile it **surfaces**, it does not overwrite (§3.3).

The granted operation set, gated behind the per-storage "Git backbone enabled" flag, is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone` class (`fetch()`, `merge()`, `commitSelf()`, `push()`); the enable gate is read from the per-storage settings by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `readStorageSettings()` (`backing.dedicated_repo`).

## 2. The library — `simple-git` over the system git binary (LOCKED)

LFB does **not** re-implement Git. It shells out to the **system `git` binary** through
[`simple-git`](https://www.npmjs.com/package/simple-git) — a thin, pure-JavaScript, promise-based
wrapper with **no native bindings**. This is a locked decision; the alternatives were evaluated and
rejected:

| Library | Verdict | Reason |
| --- | --- | --- |
| **`simple-git`** ✅ | **chosen** | Wraps the installed `git` CLI, so **real Git does the merge** — the modern **ort** recursive strategy with rename detection, exactly what "handle automatic merge" requires. ~8M weekly downloads, actively maintained, pure JS (no build step), and it reuses the same `git` LFB already calls for `git init`. |
| `isomorphic-git` | rejected | A pure-JS re-implementation. Its merge is **limited** — fast-forward and simple 3-way only, weak on renames, and it **throws** on cases real git would resolve. We must not hand the user's history to a partial merge engine. |
| `nodegit` | rejected | Native **libgit2** bindings — painful, fragile installs across macOS/Linux and Node versions. Heavy for what we need, and still not the canonical git merge. |

Because the merge is performed by the **real `git`** the user already has, LFB inherits git's full,
battle-tested conflict resolution, its `.gitattributes` merge drivers (§4.2), and its credential
handling (§5) for free. The requirement — *"find the right library … and let it handle all the Git
commands, including automatic merge"* — is met by delegating to git itself and orchestrating it with
`simple-git`.

The `simple-git` dependency (added to the backend workspace) and the single wrapper that all Git operations flow through are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `openRepo(workingDir)` returns a configured `SimpleGit` instance (system-git, non-interactive env). Declared in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/package.json` — `dependencies.simple-git`.

## 3. The two remote shapes — a local path vs. an HTTP(S) URL (LOCKED)

When the user turns the Git backbone on for a storage they supply a **remote**. LFB accepts **two
shapes**, and the difference is the whole reason this section exists — one is already a working tree on
this computer, the other is not.

### 3.1 A local file path — a checked-out working tree on this box

The common case in the [Personal use case](./storage_personal.mdx): the user has **already cloned/created**
the Git repo on this computer (e.g. `~/BGit/Bryan_git/personal_large_files_bridge/`) and points the
storage's Git backbone at **that local path**.

* LFB treats that directory **as the working copy** — it reads and writes the SDL YAML there directly,
  and runs `fetch` / `merge` / `commit` / `push` **in place** against whatever `origin` the user's clone
  already has.
* This is the preferred, simplest shape: the user manages the clone; LFB just keeps it pinned. Because
  the repo is a real checkout, the user can also inspect it with their own Git tools.

### 3.2 An HTTP(S) URL — a Git server not mapped to the local filesystem

The user may instead give a **URL** — `https://github.com/BryanStarbuck/personal_large_files_bridge.git`
or any Git server — **without** a corresponding local checkout. *"Sometimes people will add an HTTP URL,
and we have to go talk to that Git server without it being mapped to the local file system."* LFB then
owns the working copy:

* On first use LFB **clones the URL into a machine-local working-copy cache** under the state root —
  `~/T/_large_files_bridge/pin/s/<storage_id>/git/` — a hidden checkout LFB manages, **not** a place
  the user browses. (This cache is machine-local scaffolding, exactly like the rest of
  [storage_local.mdx](./storage_local.mdx); wipe it and the next pass re-clones.)
* Every pass thereafter runs `fetch` / `merge` / `commit` / `push` inside that cached checkout, against
  the configured URL as `origin`.
* The SDL YAML that other parts of LFB read is resolved to **that cache path** for a URL-backed storage,
  vs. the user's own checkout for a local-path storage. Everything downstream (devices, manifest,
  analysis) is indifferent to which shape produced the working copy.

### 3.3 Both shapes converge on one working copy

After resolution, both shapes yield a single **working directory** the rest of the engine operates on: a
user-managed checkout (local path) or an LFB-managed cache clone (URL). §4–§6 are written against that
working directory and do not care which shape it came from.

Resolving the configured remote to a working directory — using the local path in place, or clone-on-first-use / open-thereafter for a URL — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `resolveWorkingCopy(storageId, remote)` (returns `{ dir, origin }`; clones a URL remote into `pin/s/<id>/git/` via `resolveStateDir()`, opens a local path directly). The remote itself (kind + value) is persisted per-storage by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `writeStorageSettings()`.

## 4. Automatic merge — why conflicts are rare, and how git resolves them (LOCKED)

The load-bearing requirement: *"deal with any merge."* Two of the user's computers can edit the same SDL
repo before either has pulled the other's push. LFB must **fetch, merge automatically, and push** without
the user ever hand-resolving a conflict in the normal case. Three things make that true.

### 4.1 Self-owned files rarely collide

The SDL is deliberately structured so that concurrent edits land in **different files** (the trust rule
of [sync_list.mdx §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule),
[devices.mdx §2.1](./devices.mdx#21-a-device-file-is-self-owned-trust)):

* A computer **writes only its own** `devices/<device>.yaml`. Two computers editing the registry at the
  same time touch **two different files** — git merges them with **no conflict at all** (a clean union of
  two added/modified paths).
* `pinned_by` claims, schedules, and grafts are all **per-device**, so the same rule holds for them.

This is why *"we're making a unique device … then it won't conflict with other devices"* is true at the
Git level, not just conceptually: self-ownership maps directly onto per-file, non-overlapping commits.

### 4.2 Append-mostly shared lists merge by union

A few files are genuinely **shared** — the per-repo **`manifest.yaml`** (large-file → CID + pin state) and
the per-file **`decisions.yaml`** ledger. These are **Category-B** tracking state
([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)): they live in **Local Storage** always and
travel through this backbone as the additive **sync-repo mirror** at `<syncRepo>/repos/<repoKey>/` — **never**
committed into a working code repo. For them, LFB installs a **`.gitattributes`** in the sync repo that marks
them `merge=union`, so git **keeps both sides' lines** instead of conflicting when two computers — or two
**teammates** — append different entries. LFB then folds the merged file on the next read (the manifest is a
set keyed by path/CID; the decision ledger is an **append log folded to
latest-`decided_at`-per-`(sid,path)`** — [decisions.mdx §5](./decisions.mdx)), so a union followed by the
fold is always correct and a **ten-person team never resolves a decision conflict by hand**.

```gitattributes
# <syncRepo root>/.gitattributes  (written by LFB when it enables the Git backbone)
repos/*/manifest.yaml   merge=union
repos/*/decisions.yaml  merge=union
```

**The union is the transport, not the reconciliation.** Git's `merge=union` keeps both computers' lines so
the *file* never conflicts; what LFB then believes is decided by the **per-entry merge** that reads the
result — union by `path`, union `pinned_by`, newer `modified_at` wins a CID conflict, absence never a
delete, this computer's own pin claim re-derived from its real pinset
([storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked),
[sync_list.mdx §5.7](./sync_list.mdx#57-the-sync-repo-mirror--keyed-by-remote-merged-on-every-pull-locked)).
A union-merged file that is then **copied** wholesale into Local Storage throws that away — which is exactly
what happened. The fold belongs in step 4 of the cycle, immediately after the merge (§6).

**The union patterns are matched by filename, so the mirror's rekey is free.** The subtree a repo's tracking
mirrors to is now keyed by the repo's **normalized git remote** — `repos/<repoUid>/`, not the machine-local
`repos/<repoKey>/` ([storage_company.mdx §8.4.1](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked)).
The attribute lines LFB writes are bare filenames (`manifest.yaml`, `decisions.yaml`), which git applies at
**any depth**, so the rekey needs **no** `.gitattributes` change — but a future pattern that anchors a path
segment would silently stop union-merging the mirror, and that is a defect to hunt for.

The union-merge patterns (bare filenames, matched at any depth, so both `repos/<repoKey>/` and `repos/<repoUid>/` are covered) are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `UNION_MERGE_PATHS` written by `GitBackbone.ensureMergeAttributes()`.

### 4.2.1 The sync repo's committed tracking payload must never be git-ignored (LOCKED, learned from a defect)

This backbone drives a **sync repo** — a dedicated `*_large_files_bridge` company/Personal tracking repo
whose *whole reason to exist* is to carry LFB's **Category-B** tracking state
([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). Because a sync repo **is an SDL** — a
dedicated LFB file repo — it has **no `.lfbridge/` directory at all**; its root *is* the `.lfbridge` area
([artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)). So its committed payload sits at the
**repo root**: the `repos/<repoKey>/` subtrees (manifest, decisions, sidecars, history, `repo_storage.yaml`)
plus the SDL's own **root metadata** — the **device registry** (`<sdl>/devices/<device>.yaml`,
[devices.mdx](./devices.mdx)), `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`, `manifest.yaml`,
`bookmarks.yaml`, and `analysis/`. That payload must be committable — the invariant below exists because it
was **violated in production** and it broke the entire two-computer story:

> **The defect (provenance — do not lose this).** The generic storages rule *"keep `.lfbridge/`
> git-ignored"* ([storages.mdx §4](./storages.mdx#4-where-a-storages-state-lives--the-kind-split-then-category-a-vs-category-b-locked)) was
> applied to **every** git-backed storage root, so LFB wrote a bare `.lfbridge/` line into the **sync
> repo's** `.gitignore`. That silently made `git add -A` stage **nothing** under `.lfbridge/`: each computer
> wrote its `devices/<self>.yaml` to disk, committed only `.gitattributes`, and **pushed no device file at
> all**. Two computers each pushed nothing about themselves, so **neither ever saw the other** — the symptom
> "I'm not seeing my other computer."
>
> **What changed, and what did not.** Under [§0](./artifact_placement_policy.mdx) the sync repo no longer has
> a `.lfbridge/` for that line to hit — but **the hazard did not go away, it moved**: an ignore rule that
> swallows the SDL's **root metadata** or its **`devices/` directory** reproduces the exact same silent
> failure. The lesson is unchanged; only the path it applies to is.

The rule this locks in: **for a Git backbone, the sync repo's tracking text is the payload that must
travel** ([storage_personal.mdx §1](./storage_personal.mdx#1-anatomy-of-the-personal-storage)) — only the
big-file **bytes** are ignored, and those never live in the sync repo. So **before every commit** the
backbone **strips any ignore line that would swallow the SDL's committed tracking text** — its root metadata
(`devices/`, `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`, `manifest.yaml`, `bookmarks.yaml`,
`analysis/`), its `repos/<repoKey>/` subtrees, and a **legacy bare `.lfbridge/` line** (still stripped: a
not-yet-migrated SDL may hold content there until `migrateSdlLfbridge()` runs, §0.3). This heals a sync repo
that a prior build (or the user) wrongly ignored. It is a **design smell to hunt for**: any code path that
git-ignores the very tracking text a Git backbone exists to carry.

> **A plain working code repo is the opposite case — and it never git-ignores `.lfbridge/`.** A `repo`-type
> code repo is the one kind that **does** keep a `.lfbridge/` (§0: LFB is a guest there, so it quarantines
> its files in one hidden corner). It is **never** a Git backbone for tracking state: its Category-B state
> goes to Local Storage (+ the sync repo), **never** committed into the code repo. Its committed
> `.lfbridge/` holds only the **Category-A** content — transcripts + AI descriptions — that is *meant* to
> travel with the repo ([artifact_placement_policy.mdx §1.1](./artifact_placement_policy.mdx)), so it must
> never be git-ignored either. Both kinds keep their tracking text **committed**; they simply put it in
> different places (code repo → `.lfbridge/`; SDL → the root) and carry different payloads.

Ensuring the tracking text is committable on every backbone commit (strip any ignore over the SDL's root metadata / `devices/` / `repos/`, plus a legacy bare `.lfbridge/` line) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone.ensureSdlCommittable()` (called from `commitAndPush()` alongside `ensureMergeAttributes()`). The complementary fix — LFB no longer **adds** an ignore over its own tracking text for **any** storage: a company/Personal **sync repo** must commit its root payload, and a `repo`-type **code repo** commits its Category-A `.lfbridge/` content while its Category-B tracking state is written to Local Storage instead of the repo — is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `reconcileLfbridgeIgnore(root, type)` (now `unignoreLfbridge` for every type; Category-B never lands in a working repo), called from `ensureStorage()` and `ensureBackingLocations()`. Which base directory a root's tracking text hangs off — and therefore what must stay committable — is resolved by storage **kind** in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `trackingBaseDir(root, type?)` / `usesLfbridgeDir(type)`.

> **The user-facing Git Ignore action honors this too (LOCKED).** The **Git Ignore dialog**
> ([git_ignore.mdx](./git_ignore.mdx)) — the deliberate, user-driven way to git-ignore big files so they
> pin over IPFS instead of committing — writes **anchored, repo-root-relative** lines and **refuses to emit
> any line that would ignore LFB's own committed tracking text**: a `.lfbridge/` path in a **working repo**,
> or the **SDL root metadata** (`devices/`, `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`,
> `manifest.yaml`, `bookmarks.yaml`, `analysis/`, `repos/`) in a **sync repo**. That is defense-in-depth
> against re-introducing this defect from the UI. It only ever **adds** big-file ignore lines; LFB's
> committed text it never touches ([git_ignore.mdx §5.3–§5.4](./git_ignore.mdx)).

### 4.3 The strategy, and the fallback when git truly cannot resolve

* LFB merges with git's default **ort** recursive strategy (`git merge --no-edit`), which resolves the
  overwhelming majority of real-world cases (including renames) on its own.
* If — despite §4.1/§4.2 — git reports a **genuine conflict** it cannot auto-resolve, LFB does **not**
  guess and does **not** clobber. It **aborts the merge** (`git merge --abort`, leaving the working copy
  clean), records the conflicted paths, and **surfaces them in the UI** as a backbone problem on that
  storage, so the user can resolve it with their own Git tools. Reconciliation of *content* claims (a
  stale `pinned_by`, a superseded revision) still follows the SDL trust rules
  ([sync_list.mdx §5.3](./sync_list.mdx#53-conflict-resolution)); this fallback is only for the rare
  textual conflict git itself declines.

Performing the automatic merge (ort strategy, `--no-edit`), writing the `merge=union` `.gitattributes` when the backbone is enabled, and the abort-and-surface fallback on an unresolvable conflict, are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `merge()` (calls `git.merge(['--no-edit'])`, catches a conflict → `git.merge(['--abort'])` and returns a `GitConflict` the pass records) and `ensureMergeAttributes(dir)` (writes `.gitattributes`). The surfaced problem is written into the storage's `status.yaml` by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinStorageUnit(id)`.

## 5. Credentials for a URL remote — the OS git credential helper, never a prompt (LOCKED)

A local-path remote needs no credentials — it is a local checkout whose `origin` the user already
authenticated. A **URL** remote (§3.2) needs to authenticate to the Git server. LFB **never** invents
its own password prompt and **never** stores a plaintext password in its own config:

* LFB runs git **non-interactively** (`GIT_TERMINAL_PROMPT=0`) and lets git use the **OS credential
  helper** (macOS Keychain via `osxkeychain`) or an **already-configured token** in the URL/helper, the
  same way the user's own `git push` authenticates. If the user can `git push` that URL from a terminal,
  LFB can.
* If authentication **fails** (no cached credential, expired token), LFB **surfaces a clear "connect /
  re-authenticate this Git remote" problem** on the storage rather than hanging or retrying blindly —
  and simply keeps pinning over IPFS in the meantime (the Git backbone is an *additional* path, never
  the only one).
* This honors the charter's security posture: no secrets in LFB's own files, no phone-home, and the
  user stays in control of what LFB can reach.

Running git with a non-interactive, credential-helper-backed environment and turning an auth failure into a surfaced storage problem (not a hang) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `openRepo()` sets `GIT_TERMINAL_PROMPT=0` and inherits the OS credential helper; `push()` / `fetch()` classify an auth error into a `GitAuthProblem` surfaced by `pinStorageUnit(id)`.

## 6. The backbone cycle — where Git runs inside the pass (LOCKED order)

Every storage that has the Git backbone enabled runs **one Git cycle per pin pass**
([pin_process.mdx §2](./pin_process.mdx#2-every-pin-pass-is-a-full-pass-locked)), in this fixed
order, wrapped around the storage's normal reconcile + IPFS byte work:

1. **Resolve the working copy** (§3) — local path in place, or clone/open the URL cache.
2. **`git fetch`** — bring down what the user's other computers pushed.
3. **`git merge` (auto)** — fold their SDL edits into this working copy (§4); on an unresolvable
   conflict, abort + surface and continue with IPFS-only for this pass.
4. **Reconcile against the merged SDL** — now that the incoming devices/manifest/analysis are merged in,
   run the storage's normal reconcile ([pin_process.mdx §5](./pin_process.mdx#5-what-a-pass-does-per-unit--and-what-the-opt-in-gates)):
   fetch/pin the marked CIDs to their **grafted** local paths ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)),
   inherit derived artifacts, write this device's own `devices/<device>.yaml`, update `manifest.yaml`.
   **This is also where a pulled `repos/<repoUid>/` mirror is folded into Local Storage by per-entry merge**
   ([§4.2](#42-append-mostly-shared-lists-merge-by-union),
   [storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked)) — on
   **every** pull, not on demand.
5. **`git add` (self-owned only) → `git commit`** — stage **only this device's** files (§4.1) plus any
   shared list this device appended to, and commit if anything changed. Nothing to commit → no empty
   commit.
6. **`git push`** — send this device's commit up. On a **non-fast-forward reject** (another computer
   pushed between our fetch and our push), LFB **re-fetches, re-merges (§4), and re-pushes once** — the
   standard pull-then-push retry — rather than force-pushing.

The cycle is **idempotent** and safe to run every 15 minutes: with nothing changed on either side it
fetches, finds nothing to merge, stages nothing, and pushes nothing.

**The device-registration worker reuses this exact cycle on its own 10-minute cadence.** The dedicated
every-10-minute device write-back ([devices.mdx §11](./devices.mdx#11-the-device-registration-write-back--every-10-minutes-locked))
runs a **narrowed** version of steps 1–3 + 5–6 — resolve → **fetch → auto-merge** → write **only** this
device's own `devices/<self>.yaml` → commit → push — for every Git-backed storage, **whether or not** the
storage is opted into IPFS pinning. Its step 2 fetch+merge runs **every 10 minutes even when this computer
has nothing to change**, so device edits made on another computer are pulled down. Because it and the
15-minute pass share one process, both take a **per-storage lock** so they never run git on the same
working copy at once (`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts`
— `ensureDeviceRegistered(id)` under `withStorageGitLock`).

Driving this ordered cycle for each Git-backed storage inside the full pass — fetch/merge before reconcile, commit/push after, with the non-fast-forward retry — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinStorageUnit(id)` calls `GitBackbone.pull()` (fetch+merge) before `runUnitPin(...)` and `GitBackbone.commitAndPush()` after, both from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` (`pull()`, `commitAndPush()` with the fetch-merge-push retry). Only storages whose `backing.dedicated_repo.enabled` is true run the cycle; the rest pin over IPFS alone.

### 6.1 Always push when the branch is ahead — deliver stranded / foreign commits (LOCKED, learned from a defect)

Step 6's push must fire whenever the local branch is **ahead of the remote — for ANY reason** — not only
when *this* pass produced its own commit. This is a locked correctness rule because its absence stranded a
commit in production (the [2026-07-07 incident](./backbone_resilience.mdx#8-the-incident-that-motivated-this-spec-2026-07-07)):

> **The defect.** `commitAndPush()` staged the tree and, **if nothing new was staged, returned before it
> ever called `git push`.** So a branch that was *already* ahead of the remote — because something OTHER
> than LFB had committed into the repo (a machine-wide `git add -A && git commit` auto-commit), or a
> prior push had failed and left the commit local — was pushed **only if this pass also happened to make
> its own commit.** A stranded commit sat unpushed while the other computer advanced the remote; the two
> diverged and never reconverged.

The rule: after the (possibly empty) stage-and-commit, LFB computes the **ahead-count**
`origin/<branch>..HEAD`; if it is **greater than zero it pushes**, regardless of whether this pass
committed anything itself — delivering LFB's own commit, a foreign committer's commit, a merge commit, or
a commit left behind by an earlier failed push. The non-fast-forward retry (step 6) still applies. LFB
must **not** rely on the device write-back's `updated_at` churn to force a push (it usually does, which is
what *accidentally* masked the bug); a pass that legitimately writes nothing must still deliver a branch
that is ahead. This is the Git-layer half of the broader resilience contract in
[backbone_resilience.mdx §6](./backbone_resilience.mdx#6-always-push-when-the-branch-is-ahead-locked-learned-from-a-defect)
(and its §4 foreign-committer coexistence rule).

Attempting the push whenever the branch is ahead of the remote — not only after a fresh local commit — via an ahead-count check is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone.commitAndPush()` (pushes when `result.committed` OR `aheadCount(branch) > 0`) and its helper `GitBackbone.aheadCount(branch)` (`git rev-list --count origin/<branch>..HEAD`, treating a missing upstream ref as "push to establish it").

### 6.2 A mirror write nudges its SDL's backbone — debounced (LOCKED)

Steps 5–6 fire on a **timer**. A tracking mirror written *between* ticks therefore sits in the sync repo's
working tree, uncommitted, until the next 15-minute pass or 10-minute device write-back happens to run —
so a decision the user just made can take ten minutes to leave the machine. **The write is the trigger:**
after LFB mirrors a repo's tracking state into `<syncRepo>/repos/<repoUid>/`, it **announces that write to
the SDL it wrote into**, which schedules a **debounced** commit + push of that storage — the same mechanism
durable artifacts already use. Rule owned by
[storage_company.mdx §8.7](./storage_company.mdx#87-four-smaller-defects-this-use-case-also-closes):

* **Debounced, never per-write.** A pin pass mirrors many files; each announcement coalesces into **one**
  commit + push per storage, so the SDL does not get a commit per file.
* **The nudge is additive, not a replacement.** The timer cycle (§6) still runs, and a missed nudge is
  covered by the next tick — this only removes the *latency*, never the guarantee.
* **The debounced push is the ordinary §6 cycle** — fetch → auto-merge → commit → push, with the
  always-push-when-ahead rule (§6.1) and the non-fast-forward retry. It is **not** a second, weaker path.
* **It announces the destination it wrote into — a path inside the SDL.** `noteArtifactWritten()` resolves
  the owning storage by **root prefix**, so announcing the *working repo's* root would find nothing
  (`charlie-kirk` is not inside `act3_large_files_bridge`) and the nudge would silently no-op. The mirror
  step therefore announces **`dst`**, the mirror destination under `<syncRepo>/repos/<repoUid>/`, which it
  already knows — and the prefix lookup lands on the right SDL. This trap is **avoided**, not removed:
  prefix resolution stays fragile precisely because its failure mode is a silent no-op, so never hand
  `noteArtifactWritten()` a path outside the storage you mean.

Announcing a mirror write to its SDL so the backbone commits + pushes it on a debounce is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/sync-trigger.service.ts` — `noteArtifactWritten()` (resolves the storage by root prefix via `storageIdForPath()`, which is why it is handed the SDL-side destination), invoked from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo()` (passes `dst`, not the repo root).

## 7. Compliance boundary — the charter's no-relay posture, in Git terms (LOCKED)

The charter forbids LFB from becoming a gateway/relay for **other people's** content. The Git backbone
has the same shape of rule and the same default:

* LFB drives Git **only** on the repos the user **explicitly configured** as a storage's backbone. It
  never fetches, hosts, or serves a repo it merely discovered on disk, and it never proxies a third
  party's repo. The set of repos LFB will `push` to is exactly the set of storage backbones the user
  turned on — no wider.
* **Always look for out-of-compliance cases and flag/fix them**: any code path that would run a Git
  command against a repo the user did **not** configure as a backbone is a defect, exactly like a code
  path that would turn this machine into a public IPFS gateway. The only acceptable Git targets are the
  user's own configured storage backbones.
* **A `*_large_files_bridge` company/personal tracking repo counts as user-configured.** When such a
  storage's **own root is a git working tree**, LFB **auto-adopts that root** as its backbone
  ([devices.mdx §11.2.1](./devices.mdx#1121-a-company--personal-tracking-repo-auto-adopts-its-own-git-root--commit--push-first-write-and-every-update-locked-learned-from-a-defect))
  — this is **inside** the boundary, not a widening of it: a purpose-built tracking repo the user made
  for LFB is a repo the user *configured*, and LFB only ever pushes **our own** device registry into it.
  A plain **code** repo (storage type `repo`) is **never** adopted — its Category-B tracking state routes to
  **Local Storage** (+ the sync repo), never the code repo, so no git cycle ever runs against it. Its
  committed `.lfbridge/` holds only Category-A content and is **not** git-ignored
  ([git_ignore.mdx §5.3](./git_ignore.mdx)); it is simply excluded from every tracking-state git cycle.

Enforcing that Git operations only ever target a user-configured storage backbone (never a merely-discovered code repo) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `getGitBackboneRemote()` returns a backbone **only** for an explicit dedicated-repo backing OR a company/personal storage whose own `<root>/.git` exists (a `repo`-type code repo returns null); `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone` is constructed only from that resolved remote, and there is no entry point that runs `push`/`fetch` on an arbitrary path.

## 8. Acceptance criteria

1. Turning on a storage's **Git backbone** ([storage_settings.mdx §4.1](./storage_settings.mdx#41-dedicated-git-repo-preferred-when-available))
   grants LFB permission to run **`git fetch`, `git merge` (automatic), `git add`/`commit`, and
   `git push`** against **that one repo** on **every pin pass** — and no Git command runs against a
   storage whose backbone is off (§1).
2. The Git library is **`simple-git`** driving the **system `git` binary**, so the **merge is done by
   real git** (ort strategy); `isomorphic-git` and `nodegit` are explicitly rejected (§2).
3. LFB accepts **two remote shapes**: a **local file path** to a checked-out working tree (used in
   place), and an **HTTP(S) URL** with **no** local checkout (cloned into a machine-local
   `pin/s/<id>/git/` cache and managed there). Both converge on one working copy the rest of the engine
   operates on (§3).
4. Concurrent edits from two of the user's computers **merge automatically**: self-owned per-device
   files never collide, append-mostly shared lists merge by **`merge=union`**, and only a **genuine**
   git conflict is surfaced — via **`merge --abort` + a UI problem**, never a clobber or a force-push
   (§4).
5. A **URL** remote authenticates through the **OS git credential helper** with `GIT_TERMINAL_PROMPT=0`;
   an auth failure is **surfaced** as a "re-authenticate" problem while IPFS pinning continues — LFB never
   prompts for or stores a plaintext password (§5).
6. The per-pass **cycle order is fixed**: resolve working copy → fetch → auto-merge → reconcile/pin to
   grafted paths → stage self-owned + commit → push, with a **fetch-merge-push retry** on a
   non-fast-forward reject; the cycle is idempotent (§6).
7. LFB drives Git **only** on user-configured storage backbones — never a merely-discovered or
   third-party repo, never a force-push — the Git analogue of the charter's no-public-relay rule (§1, §7).
8. **`merge=union` is the transport, not the reconciliation.** The union keeps both sides' lines so the
   file never conflicts; what LFB believes afterward comes from the **per-entry merge** of the pulled
   `repos/<repoUid>/` mirror in **step 4** of the cycle — union by `path`, union `pinned_by`, newer
   `modified_at` wins a CID conflict, absence never a delete — never a wholesale copy (§4.2, §6,
   [storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked)). The
   attribute patterns are **filename-matched**, so the mirror's remote-keyed rekey needs no
   `.gitattributes` change.
9. **A mirror write nudges its SDL's backbone on a debounce** rather than waiting for the next timer tick;
   the nudge coalesces to one commit + push per storage, runs the ordinary §6 cycle, and resolves the SDL
   by **ownership** (a working-repo path prefix resolves to nothing) — §6.2.
