# LargeFileBridge ‚Äî Storages (the umbrella)

A **storage** is a place LargeFileBridge (LFB) **tracks and syncs large files**. This is the umbrella
spec: it defines the **family** of storages, the shared **`storage.yaml`** schema, the hidden
**`.lfbridge/`** tracking layout, the per-file **media-analysis** outputs, and the **Storages** left-bar
tab. Each kind then has its own detail spec.

A person is normally involved with **more than one storage at once** ‚Äî several **communities**, one or
more **companies**, and their own **personal** files ‚Äî plus every **repo** on the machine, all sitting on
top of this computer's **local** settings. The whole point of the Storages tab is to make that
multiplicity legible in one place.

The five kinds:

| Storage | Type | Root directory | Detail spec |
| --- | --- | --- | --- |
| **Local** | `local` | `~/T/_large_files_bridge/` (settings/config; the DB replacement ‚Äî **no** `storage.yaml`) | [storage_local.mdx](./storage_local.mdx) |
| **Repo** | `repo` | any Git repo root, e.g. `~/BGit/xyz/` ‚Üí tracking in `~/BGit/xyz/.lfbridge/` | [storage_repo.mdx](./storage_repo.mdx) |
| **Personal** | `personal` | `~/BGit/Bryan_git/personal_large_files_bridge/` | [storage_personal.mdx](./storage_personal.mdx) |
| **Company** | `company` | one per company, e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` | [storage_company.mdx](./storage_company.mdx) |
| **Community** | `community` | one per community the user helps / downloads / shares | [storage_community.mdx](./storage_community.mdx) |

## 1. Local vs. directory-based storages (LOCKED)

Two categories, and they play different roles:

* **Local Storage** (`local`) is **this computer's own settings, configuration, and bookkeeping** ‚Äî the
  **database replacement** ([storage_local.mdx](./storage_local.mdx)). It does not itself sync between
  computers; wipe it and a rescan rebuilds it.
* **Directory-based storages** (`repo` ¬∑ `personal` ¬∑ `company` ¬∑ `community`) are **real directory
  hierarchies that hold the shared large files**. They:
  1. carry a **`storage.yaml`** at their root (¬ß3) declaring name + type + type-specific properties,
  2. carry a hidden **`.lfbridge/`** tracking area at their root (¬ß4) ‚Äî the per-file fingerprint index and
     the per-file media-analysis outputs,
  3. **sync between the user's computers over IPFS** (git for the small files + manifest; IPFS for the
     bytes ‚Äî charter), and
  4. **often also have a clone in Google Drive and/or Dropbox** (¬ß5) ‚Äî the same hierarchy mirrored to a
     cloud drive as a second, human-reachable copy.

The one boundary that never blurs: **Local holds settings; the directory-based storages hold files.**

## 2. The Storages left-bar tab

The left bar gains a **Storages** entry (source of truth `pm/left_bar.yaml`, human companion
[left_bar.mdx](./left_bar.mdx)). It is a grouping tab whose **children** are the storages this user
belongs to ‚Äî so the multiplicity is visible at a glance:

* **Personal** ‚Üí the user's [Personal storage](./storage_personal.mdx).
* **One child per Company** ‚Üí each [Company storage](./storage_company.mdx) the user is in (label = the
  company name).
* **Repos** ‚Üí **does not** list every repo here; clicking it routes to the dedicated **[Repos](./repos.mdx)
  tab**, which owns the long list. (The Storages tab is the *map of storage kinds*, not the repo list.)
* **Communities** ‚Üí the [Community storages](./storage_community.mdx) the user has opted into ‚Äî the ones
  they chose to **download** for themselves or **share** to help keep secure. (The separate top-level

The Storages children are **derived** ‚Äî Personal from the personal storage's presence, Companies +
Communities by scanning each directory-based storage's `storage.yaml` `type`, and Repos as the fixed link
to the Repos tab.

## 3. `storage.yaml` ‚Äî the per-storage descriptor (LOCKED shape)

Every **directory-based** storage has a **`storage.yaml`** at the **root of its directory**. It declares
the storage's identity and its type-specific properties. Local Storage has none (it is not a directory
storage).

```yaml
# <storage root>/storage.yaml
name: "ACT3 Company Files"      # human name shown in the Storages tab
type: company                   # local | repo | personal | company | community
created: 2026-05-01T10:12:00Z
# ‚îÄ‚îÄ Type-specific block (exactly one, matching `type`) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
company:                        # present when type: company
  company_name: "ACT3 AI"
  # ‚Ä¶company-scoped properties (division, drive folder id, allow-list, ‚Ä¶)
community:                      # present when type: community
  id: "holon-citizen"           # which community (stable key; matches communities.mdx)
  role: support                 # download | share | support ‚Äî why the user carries it
  # ‚Ä¶community-scoped child properties (publisher pubkey, budget mode, ‚Ä¶)
personal: {}                    # present when type: personal (usually empty / user prefs)
repo:                           # present when type: repo
  repo_root: "~/BGit/xyz"       # the Git repo this storage tracks
# ‚îÄ‚îÄ Optional cloud mirrors (¬ß5) ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ
clones:
  google_drive: "My Drive/LFB/ACT3"     # relative path inside the connected Drive, or null
  dropbox: "Apps/LFB/ACT3"              # relative path inside Dropbox, or null
```

* **`name`** and **`type`** are required on every storage. **Exactly one** type-specific block is present,
  matching `type`.
* **Company** carries **`company_name`** plus company-scoped child properties.
* **Community** carries **`community`** naming **which** community (its stable id, matching
  [communities.mdx](./communities.mdx)) plus community-scoped child properties (e.g. the publisher and the
  Block/Recommended/Full backup budget mode).
* **`clones`** records where, if anywhere, this same hierarchy is mirrored on Google Drive / Dropbox (¬ß5).

Reading and writing `storage.yaml` (atomic, YAML, per-storage) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` ‚Äî `readDescriptor` / `writeDescriptor` (+ `ensureStorage`).

## 4. `.lfbridge/` ‚Äî the hidden per-storage tracking area

Alongside `storage.yaml`, each directory-based storage keeps a hidden **`.lfbridge/`** folder at its root
(for a **repo**, that is `~/BGit/xyz/.lfbridge/`). It holds everything LFB knows about the files in *this*
storage ‚Äî kept **out of Git** (add `.lfbridge/` to the repo's `.gitignore`) so tracking never bloats
commits, while the small committed manifest still travels with the code.

```
<storage root>/
‚îú‚îÄ‚îÄ storage.yaml                 # ¬ß3 descriptor
‚îú‚îÄ‚îÄ .lfbridge/
‚îÇ   ‚îú‚îÄ‚îÄ files.yaml               # per-file fingerprint index (¬ß4.1)
‚îÇ   ‚îú‚îÄ‚îÄ manifest.yaml            # large-file ‚Üí IPFS CID + pin state (storage_local.mdx ¬ß9)
‚îÇ   ‚îî‚îÄ‚îÄ analysis/                # per-file media-analysis outputs (¬ß6)
‚îÇ       ‚îî‚îÄ‚îÄ <relpath>/
‚îÇ           ‚îú‚îÄ‚îÄ visuals_by_time.yaml
‚îÇ           ‚îî‚îÄ‚îÄ compression.yaml
‚îú‚îÄ‚îÄ <relpath-without-ext>.transcription    # transcript SIDECAR beside the media (¬ß6)
‚îú‚îÄ‚îÄ <relpath-without-ext>.ai_description   # AI description SIDECAR beside the media (¬ß6)
‚îî‚îÄ‚îÄ ‚Ä¶the storage's actual files (each with its two sidecars right beside it)‚Ä¶
```

### 4.1 Per-file tracking metadata (charter)

`.lfbridge/files.yaml` records, per file, enough to tell whether it changed since last seen ‚Äî a **hash**
(fingerprint), **size**, **created**, and **modified**. Compressible media (video/image) carry **two**
hashes, **compressed** and **uncompressed**, nested under the file entry (charter ‚Äî see the CLAUDE.md
"Per-file tracking metadata" example). This is the same shape whatever the storage type:

```yaml
files:
  render/final_4k.mp4:
    size: 734003200
    created: 2026-05-01T10:12:00Z
    modified: 2026-06-18T14:33:00Z
    hashes:
      uncompressed: { hash: <fingerprint>, size: 734003200 }
      compressed:   { hash: <fingerprint>, size: 118489088 }
    analysis: [transcript, description, visuals_by_time]   # which ¬ß6 outputs exist
```

Writing the per-file fingerprint index (hash + size + dates; compressed/uncompressed variant hashes for media) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` ‚Äî `indexStorageFiles`.

## 5. Cloud clones (Google Drive / Dropbox)

A directory-based storage's **IPFS copy is the primary sync path** between the user's computers, but a
storage will **sometimes also have a clone in Google Drive and/or Dropbox** ‚Äî the same directory
hierarchy mirrored into a cloud drive as a second, human-reachable, non-IPFS copy. The `clones` block in
`storage.yaml` (¬ß3) records the relative path of that mirror in each connected drive (or `null`).

This reuses the existing Drive/Dropbox integration specs ‚Äî [google_drive.mdx](./google_drive.mdx) and
[dropbox.mdx](./dropbox.mdx) ‚Äî applied per storage: a storage may be pinned over IPFS, mirrored to Drive,
mirrored to Dropbox, or any combination. The clones are **mirrors, not the source of truth**; the
`.lfbridge/` index + IPFS manifest remain authoritative.

## 6. Per-file media analysis (transcription ¬∑ description ¬∑ visuals-by-time)

For a file in a storage ‚Äî **especially a video** ‚Äî LFB can run analysis passes. Their outputs are
**generated artifacts** that travel with the storage, so any of the user's computers (or a teammate on a
company/community storage) gets them without re-running. Each pass is an explicit, opt-in operation (never
automatic on bytes ‚Äî charter ¬ß6.1 spirit). **Transcript and description are written as sidecars beside the
media** ([Transcribe.mdx ¬ß3](./Transcribe.mdx), [ai_description.mdx ¬ß2](./ai_description.mdx) are
authoritative ‚Äî extension replaced, **no** `.lfbridge/analysis/` directory); **visuals-by-time and the
compression record** remain **YAML under `.lfbridge/analysis/<relpath>/`**.

The passes:

* **Transcript** (**`<rel-without-ext>.transcription`** sidecar beside the media) ‚Äî **speech-to-text** of
  what people are *saying* in a video (or in an audio file). This is the "what was said" track.
  Authoritative: [Transcribe.mdx ¬ß3](./Transcribe.mdx).
* **Description** (**`<rel-without-ext>.ai_description`** sidecar beside the media) ‚Äî a prose
  **summary/description** of the file: for a **video**, what it is about; for an **image**, **what is in the
  image**. The "what is it" track. Authoritative: [ai_description.mdx ¬ß2](./ai_description.mdx).
* **Visuals & action by time** (`visuals_by_time.yaml`, under `.lfbridge/analysis/<relpath>/`) ‚Äî primarily
  for **video**: a timeline of
  **time ranges** with, for each, **what is visually going on** and **what is going on story-wise** ‚Äî so
  the material can be re-used or re-created (think cinematography beats/shots). The "what is happening,
  when" track.

```yaml
# .lfbridge/analysis/render/final_4k.mp4/visuals_by_time.yaml
source: render/final_4k.mp4
segments:
  - start: "00:00:00"
    end:   "00:00:06"
    visual: "Wide desert dawn; single figure walks camera-left to -right."
    story:  "Establishing the journey; isolation."
  - start: "00:00:06"
    end:   "00:00:14"
    visual: "Cut to close-up, dust on face; hard side light."
    story:  "Cost of the journey registers on the character."
```

* **Images** get **Description** (what's in the image); Transcript and Visuals-by-time are video/audio
  concepts and are skipped for a still image.
* The results are cached against the file's fingerprint (¬ß4.1) ‚Äî re-running is skipped when the hash is
  unchanged.

Running the analysis passes and writing the per-file YAML outputs into `.lfbridge/analysis/` is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/analysis.service.ts` ‚Äî `analyzeFile` (dispatches transcript / description / visuals-by-time by media kind).

## 8. Acceptance criteria

1. There are **five storage kinds** ‚Äî `local`, `repo`, `personal`, `company`, `community` ‚Äî each with a
   detail spec; **Local** is settings/config (the DB replacement, no `storage.yaml`), the other four are
   directory hierarchies that hold the shared files.
2. Every **directory-based** storage has a **`storage.yaml`** at its root declaring **`name`** + **`type`**
   and **exactly one** type-specific block; **company** carries `company_name`, **community** carries which
   `community` (+ child props), each with type-scoped properties.
3. Every directory-based storage has a hidden **`.lfbridge/`** at its root (for a repo, `~/BGit/xyz/.lfbridge/`)
   holding the per-file fingerprint index (hash + size + dates; compressed/uncompressed variant hashes for
   media), the IPFS manifest, and the `analysis/` outputs ‚Äî all git-ignored.
4. A user can belong to **many** storages at once (multiple communities + companies + personal + every
   repo). The **Storages** left-bar tab shows them: **Personal**, one child **per company**, a **Repos**
   child that routes to the Repos tab (not a long list), and the opted-in **Communities**.
5. Storages sync between the user's computers over **IPFS** and **may also** clone to **Google Drive** and
   **Dropbox** (recorded in `storage.yaml` `clones`); the clones are mirrors, not the source of truth.
6. **Media analysis** can produce, per file under `.lfbridge/analysis/`: a **transcript** (speech +
   time), a **description** (video summary / image contents), and **visuals-and-action-by-time** (time
   ranges √ó visual √ó story) ‚Äî video gets all three, images get description only; results cache against the
   file fingerprint.
