# LargeFileBridge — Storages (the umbrella)

A **storage** is a place LargeFileBridge (LFB) **tracks and pins large files**. This is the umbrella
spec: it defines the **family** of storages, the shared **`storage.yaml`** schema, the **storage-kind
split** that decides where LFB's files hang under a root (§4), the per-file **media-analysis** outputs,
and the **Storages** left-bar tab. Each kind then has its own detail spec.

A person is normally involved with **more than one storage at once** — several **communities**, one or
more **companies**, and their own **personal** files — plus every **repo** on the machine, all sitting on
top of this computer's **local** settings. The whole point of the Storages tab is to make that
multiplicity legible in one place.

The five kinds:

| Storage | Type | Root directory | Detail spec |
| --- | --- | --- | --- |
| **Local** | `local` | `~/T/_large_files_bridge/` (settings/config; the DB replacement — **no** `storage.yaml`) | [storage_local.mdx](./storage_local.mdx) |
| **Repo** | `repo` | any Git repo root, e.g. `~/BGit/xyz/` → LFB's files in `~/BGit/xyz/.lfbridge/` (**the only kind with a `.lfbridge/`** — §4) | [storage_repo.mdx](./storage_repo.mdx) |
| **Personal** | `personal` | `~/BGit/Bryan_git/personal_large_files_bridge/` (an **SDL** — **no** `.lfbridge/`; LFB's files hang off the root) | [storage_personal.mdx](./storage_personal.mdx) |
| **Company** | `company` | one per company, e.g. `~/BGit/Bryan_git/act3_large_files_bridge/` (an **SDL** — **no** `.lfbridge/`) | [storage_company.mdx](./storage_company.mdx) |
| **Community** | `community` | one per community the user helps / downloads / shares (an **SDL** — **no** `.lfbridge/`) | [storage_community.mdx](./storage_community.mdx) |

Siblings this builds on and does not repeat:

* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — **the authoritative storage-KIND
  rule**: `.lfbridge/` exists **only** inside a working git repo; an **SDL** (personal / company /
  community) has **none** — its root *is* the `.lfbridge` area (§4), including the **reserved names** at
  that root (§0.3).
* [syncable_data_location.mdx](./syncable_data_location.mdx) — names the directory-based storage's root as
  a **Syncable Data Location** (the small travelling text: the tracking area + Sync List), and defines the
  **mapped source directories** model (a company/personal storage maps a *list* of hierarchies; the big
  files stay in place) and the **derived-artifact reuse** economics (generate once, reuse everywhere).
* [devices.mdx](./devices.mdx) — the **device registry** (`devices/`) and the per-computer **graft** that
  resolves mapped directories to local paths; both live inside the storage's **tracking area** — which for
  a working `repo` is its `.lfbridge/`, and for an **SDL** is the root itself (§4).
* [storage_local.mdx](./storage_local.mdx) — the state root, app/user config, auth session, `peers.yaml`,
  logs, and the local index. **Local Storage is settings + this machine's bookkeeping; it is not one of
  the directory-based storages** and has no `storage.yaml`.
* [Communities](./communities.mdx) — the *subscribe/consume/support* model for other people's public
  files. A **Community storage** (here) is the on-disk hierarchy a community subscription materializes to.
* [Repos](./repos.mdx) / [One repo](./one_repo.mdx) — the repos list and per-repo screen. A **repo is one
  kind of storage**; the Storages tab links to the Repos tab rather than re-listing every repo.
* [README](../README.md) / [CLAUDE.md](../CLAUDE.md) — the charter: git for small files, IPFS for big
  ones; per-file tracking metadata (hashes, size, dates, compressed/uncompressed variants); only-our-content.

## 1. Local vs. directory-based storages (LOCKED)

Two categories, and they play different roles:

* **Local Storage** (`local`) is **this computer's own settings, configuration, and bookkeeping** — the
  **database replacement** ([storage_local.mdx](./storage_local.mdx)). It does not itself travel between
  computers; wipe it and a rescan rebuilds it.
* **Directory-based storages** (`repo` · `personal` · `company` · `community`) are **real directory
  hierarchies that hold the shared large files**. They:
  1. carry a **`storage.yaml`** at their root (§3) declaring name + type + type-specific properties,
  2. carry a **tracking area** whose location **depends on the storage kind** (§4): a working **`repo`**
     keeps a hidden **`.lfbridge/`** at its root, holding the **Category-A derived content** (transcript /
     AI-description / visuals-by-time); an **SDL** (`personal` · `company` · `community`) has **no
     `.lfbridge/` at all** — its root *is* the tracking area. Either way, their **Category-B tracking
     state** (the per-file fingerprint index, the IPFS manifest, and the rest) never enters a **working
     repo** (§4),
  3. **pin their large files between the user's computers over IPFS** (git for the small files +
     manifest; IPFS for the bytes — charter), and
  4. **often also have a clone in Google Drive and/or Dropbox** (§5) — the same hierarchy mirrored to a
     cloud drive as a second, human-reachable copy.

The one boundary that never blurs: **Local holds settings; the directory-based storages hold files.**

## 2. The Storages left-bar tab

The left bar gains a **Storages** entry (source of truth `pm/left_bar.yaml`, human companion
[left_bar.mdx](./left_bar.mdx)). It is a grouping tab whose **children** are the storages this user
belongs to — so the multiplicity is visible at a glance:

* **Personal** → the user's [Personal storage](./storage_personal.mdx).
* **One child per Company** → each [Company storage](./storage_company.mdx) the user is in (label = the
  company name).
* **Repos** → **does not** list every repo here; clicking it routes to the dedicated **[Repos](./repos.mdx)
  tab**, which owns the long list. (The Storages tab is the *map of storage kinds*, not the repo list.)
* **Communities** → the [Community storages](./storage_community.mdx) the user has opted into — the ones
  they chose to **download** for themselves or **share** to help keep secure. (The separate top-level
  **[Communities](./communities.mdx)** tab remains the place to *discover and subscribe*; the Storages
  child is the *on-disk result*.)

The Storages children are **derived** — Personal from the personal storage's presence, Companies +
Communities by scanning each directory-based storage's `storage.yaml` `type`, and Repos as the fixed link
to the Repos tab.

Each **row** in the Storages table carries, in its trailing action cell, an **Open / Initialize** control,
then a **⚙ gear**, then the row **`⋮` kebab**. The gear sits **just left of the kebab** and opens that
storage's **per-storage settings** ([storage_settings.mdx §1](./storage_settings.mdx#1-how-you-get-here--gear-kebab-right-click-more-menu)) —
the one-click shortcut to the same place the kebab's **Storage settings…** item reaches, for Company,
Personal, Repo, or Community storages alike.

### 2.2 Organizations found in your repos — proposed company storages (LOCKED)

Company storages are not only *discovered on disk* — they are also **proposed from the user's own repos**.
A company storage is **1:1 with a forge organization**, and the org is a level in every repo's git remote
URL, so the repos already say which companies exist
([storage_company.mdx §10](./storage_company.mdx#10-a-company-is-a-forge-organization-locked),
[repo_company_mapping.mdx §2.1](./repo_company_mapping.mdx#21-the-org-level-is-the-company--a-company-storage-is-11-with-an-organization-locked)).
The Storages page is where that proposal surfaces, because *"which companies do I have?"* is a
**cross-storage** question — no single storage's settings page can answer it.

**A proposal panel sits above the Storages table.** It appears only when there is something to say: at
least one qualifying, unclaimed, undismissed org, **or** a non-zero skipped count.

* **One row per qualifying org**, carrying the **org name**, its **repo count**, the **directory that would
  be created**, and a **checkbox**. A single **Create company storages** button makes all the checked ones
  at once.
* **Nothing touches the disk until that click.** Creation makes a real directory and runs `git init`, which
  is exactly what the charter says Large File Bridge surfaces and offers rather than doing on its own.
  Unchecking a row before creating means **no directory is ever made** for it
  ([storage_company.mdx §10.3](./storage_company.mdx#103-proposing-and-creating-the-company-storages-locked)).
* **A per-row dismiss** removes an org from the proposal permanently — remembered per org, so it does not
  return on the next scan, and undoable from company settings.
* **Already-claimed orgs are shown as ADOPTED, not proposed.** An org an existing company storage claims
  (via `company.owner_slugs`, or a normalized name match) is adopted and the binding is written back —
  never duplicated into a second storage beside the first.
* **The skipped-clones count is stated, never implied.** Orgs that failed the **authored-commit membership
  test** ([repo_company_mapping.mdx §3.2](./repo_company_mapping.mdx#32-which-orgs-are-actually-yours--the-authored-commit-membership-test-locked))
  are reported as a count with a way to see them: *"28 organizations you've only cloned from were ignored —
  you have not committed to any of them on this computer."* Expanding lists each org with its repo count.
  **A silent filter and a bug look identical** ([warnings.mdx](./warnings.mdx)); on the machine this was
  measured against the filter removed **28 of 32** orgs, so it is far too consequential to leave unsaid.
* **Created storages join the table by the ordinary rules.** A newly created company appears as a normal
  `type: company` row (§2, §2.1) — the proposal panel is a *creation* affordance, not a second listing.
  Each one is born with **no git remote**, so it immediately carries the *"no remote — nothing is reaching
  your other computers"* fault until the user supplies one
  ([storage_settings.mdx §4.1.1](./storage_settings.mdx#411-no-remote-yet-is-a-fault-this-page-shows-and-fixes-locked)).
* **The parent directory is a setting**, shown in the panel with whether it was **configured** or
  **derived** from where existing `*_large_files_bridge` storages live
  ([storage_settings.mdx §4d](./storage_settings.mdx#4d-new-company-storages--the-proposal-surface-and-the-parent-directory-locked)).

Discovering the organizations, applying the membership test, and creating the checked ones are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/company-discovery.service.ts` — `discoverOrganizations()` (proposals + adopted + `skipped` / `skippedCount` + `parentDir` / `parentDirIsConfigured`), `createCompanyStorages(orgs)`, and `setOrganizationDismissed(org)`, exposed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.router.ts` — `GET /api/storages/organizations` / `POST /api/storages/companies` and rendered by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/storages/StoragesPage.tsx` — `OrganizationProposals`.

## 3. `storage.yaml` — the per-storage descriptor (LOCKED shape)

Every **directory-based** storage has a **`storage.yaml`** at the **root of its directory**. It declares
the storage's identity and its type-specific properties. Local Storage has none (it is not a directory
storage).

```yaml
# <storage root>/storage.yaml
name: "ACT3 Company Files"      # human name shown in the Storages tab
type: company                   # local | repo | personal | company | community
created: 2026-05-01T10:12:00Z
# ── Type-specific block (exactly one, matching `type`) ────────────────────────
company:                        # present when type: company
  company_name: "ACT3 AI"
  # …company-scoped properties (division, drive folder id, allow-list, …)
community:                      # present when type: community
  id: "holon-citizen"           # which community (stable key; matches communities.mdx)
  role: support                 # download | share | support — why the user carries it
  # …community-scoped child properties (publisher pubkey, budget mode, …)
personal: {}                    # present when type: personal (usually empty / user prefs)
repo:                           # present when type: repo
  repo_root: "~/BGit/xyz"       # the Git repo this storage tracks
# ── Optional cloud mirrors (§5) ───────────────────────────────────────────────
clones:
  google_drive: "My Drive/LFB/ACT3"     # relative path inside the connected Drive, or null
  dropbox: "Apps/LFB/ACT3"              # relative path inside Dropbox, or null
```

* **`name`** and **`type`** are required on every storage. **Exactly one** type-specific block is present,
  matching `type`.
* **Company** carries **`company_name`** plus company-scoped child properties.
* **Community** carries **`community`** naming **which** community (its stable id, matching
  [communities.mdx](./communities.mdx)) plus community-scoped child properties (e.g. the publisher and the
  Block/Recommended/Full backup budget mode).
* **`clones`** records where, if anywhere, this same hierarchy is mirrored on Google Drive / Dropbox (§5).

Reading and writing `storage.yaml` (atomic, YAML, per-storage) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `readDescriptor` / `writeDescriptor` (+ `ensureStorage`).

## 4. Where a storage's state lives — the kind split, then Category A vs. Category B (LOCKED)

Alongside `storage.yaml`, each directory-based storage keeps a **tracking area**. **Two** rules decide
what that means, applied in order — both authoritative in
[artifact_placement_policy.mdx](./artifact_placement_policy.mdx) (§0 and §1+ respectively), with the file
shapes in [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx):

**First, the storage-KIND split — `.lfbridge/` exists only inside a working git repo.** It is a
**working-repo-only concept**. A **dedicated LFB file repo** — the
`personal` / `company` / `community` **SDL storage** — has **no `.lfbridge/` directory at all**: **the SDL
root *is* the `.lfbridge` area**, so everything hangs off `<root>` directly. The rule is owned by
[artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) and only summarized here:

| Storage kind | Example root | LFB's files land at | `.lfbridge/`? |
| --- | --- | --- | --- |
| **`repo`** — a user's working git repo (Git Storage) | `~/BGit/Bryan_git/charlie-kirk/` | `<root>/.lfbridge/<rel>` | **YES** |
| **`personal` / `company` / `community`** — a dedicated LFB file repo (SDL) | `~/BGit/Bryan_git/personal_large_files_bridge/` | `<root>/<rel>` | **NO — never** |

**Why.** In a working repo LFB is a **guest**, so it quarantines its files under one hidden directory that
is obviously not the user's content. An SDL is the opposite — a repo whose entire reason to exist is to
hold LFB's files, so a `.lfbridge/` inside it buys no separation and only adds a meaningless path segment.
The split is keyed on the **resolved storage kind**, never on a path or a name, and the default is `repo`
(§0.2 there); the choke point is
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` —
`trackingBaseDir(root, type?)` / `usesLfbridgeDir(type)`.

**Reserved names at the SDL root.** Because the SDL mirror hangs off the root, a **mapped dir**
([syncable_data_location.mdx](./syncable_data_location.mdx)) keyed `devices`, `analysis`, `files.yaml`,
`storage.yaml`, `manifest.yaml`, `bookmarks.yaml`, or `mapped_dirs.yaml` would **shadow the SDL's own
metadata**. These names are therefore **reserved** at an SDL root — the rule is owned by
[artifact_placement_policy.mdx §0.3](./artifact_placement_policy.mdx) and the add/rename validator by
[storage_settings.mdx §4a.1](./storage_settings.mdx#4a1-reserved-mapped-dir-keys--the-validator-this-page-owns-locked); neither is restated here.

**Then, the DATA-CATEGORY split.** Given the base directory the kind split picked, what may live in it is
a hard split by **data category**:

* **Category A — derived content (in the committed tracking area the kind split just picked).** Only two artifact kinds: the
  path-mirrored **`<name.ext>.transcription`** and **`<name.ext>.ai_description`** files
  ([Transcribe.mdx §3](./Transcribe.mdx), [ai_description.mdx §2](./ai_description.mdx)), plus the
  **visuals-and-action-by-time** YAML that rides with the AI-description family (§6). These are stable,
  content-derived, cheap to commit, and *should* travel so every computer and teammate reuses them — so the
  working repo's `.lfbridge/` is **committed** and carries them. **Only the multi-GB media files themselves
  are git-ignored** (they ride IPFS); these two content artifacts are **not**.
* **Category B — LFB's noisy tracking state (NEVER in a working repo).** The per-file fingerprint index
  (`files.yaml`), the IPFS manifest (`manifest.yaml`), the repo-wide `repo_storage.yaml`, the per-file
  sidecars, the per-computer history logs, the decision ledger, and the compression records. This
  bookkeeping re-stamps a per-scan timestamp + counts on every scan, so committing it into a working repo
  produced a fresh **merge conflict every scan**. It therefore lives in [Local Storage](./storage_local.mdx)
  at **`~/T/_large_files_bridge/repos/<repoKey>/`** ALWAYS (`repoKey = shortHash(resolve(repoPath))`), which
  never touches git, and is **additionally** mirrored to a configured company/Personal **sync repo** (§8)
  when the per-repo toggle is on. (This supersedes any earlier "`.lfbridge/` is COMMITTED and holds the
  manifest/index/`repo_storage.yaml`" wording — that churny state now stays out of git entirely.)

```
KIND `repo` — a WORKING git repo (LFB is a guest → one hidden corner)
<repo root>/
├── storage.yaml                 # §3 descriptor
├── .lfbridge/                   # COMMITTED — holds ONLY Category-A derived content
│   ├── <rel-dir>/<name.ext>.transcription    # transcript — path-mirrored, ext APPENDED (§6)
│   ├── <rel-dir>/<name.ext>.ai_description   # AI description — path-mirrored, ext APPENDED (§6)
│   └── analysis/                # visuals-by-time — the AI-description family (§6)
│       └── <relpath>/
│           └── visuals_by_time.yaml
└── …the repo's actual files (media untouched; transcript/description live under .lfbridge/ above)…

KIND `personal` / `company` / `community` — an SDL (the root IS the .lfbridge area → NO .lfbridge/)
<SDL root>/                      # e.g. ~/BGit/Bryan_git/personal_large_files_bridge/
├── storage.yaml                 # §3 descriptor — at the ROOT
├── mapped_dirs.yaml             # the mapped source directories (syncable_data_location.mdx)
├── files.yaml                   # per-file fingerprint index — at the ROOT (§4.1)
├── manifest.yaml                # large-file → IPFS CID + pin state
├── bookmarks.yaml
├── devices/<device-id>.yaml     # the device registry (devices.mdx)
├── analysis/<relpath>/visuals_by_time.yaml   # (§6)
└── <rel-dir>/<name.ext>.transcription        # the path-mirrored artifacts hang off the ROOT (§6)
    #   ↑ NO .lfbridge/ segment anywhere. The names above are RESERVED at this root
    #     (artifact_placement_policy.mdx §0.3) — a mapped dir may not shadow them.

~/T/_large_files_bridge/repos/<repoKey>/   # LOCAL STORAGE — Category-B tracking state, NEVER in the repo
├── repo_storage.yaml            # repo-wide settings + special-file counts (repo_tracking_scheme.mdx §2)
├── files.yaml                   # per-file fingerprint index (§4.1)
├── files/<path>.yaml            # per-file sidecars (repo_tracking_scheme.mdx §3)
├── history/<device>.txt         # per-computer audit logs (repo_tracking_scheme.mdx §4)
├── decisions.yaml               # the two-axis decision ledger (decisions.mdx)
├── manifest.yaml                # large-file → IPFS CID + pin state (storage_local.mdx §9)
└── analysis/<rel>/compression.yaml   # compression records (compression.mdx §8)
    #   ↑ additionally mirrored to <syncRepo>/repos/<repoKey>/ when the sync repo is enabled (§8)
```

### 4.1 Per-file tracking metadata (charter)

The per-file fingerprint index **`files.yaml`** is Category B, and **where it lives depends on the storage
kind**: for a **working `repo`** it lives in Local Storage at
`~/T/_large_files_bridge/repos/<repoKey>/files.yaml`, **never in the working tree** (so a repo with no
transcripts keeps no `.lfbridge/` at all); for a dedicated **SDL storage** (personal / company / community)
it is committed to that storage's own **`<root>/files.yaml`** — at the **root**, with **no `.lfbridge/`
segment**, because an SDL has none (§4) — so the index **travels** with the purpose-built LFB repo to
teammates and the user's other computers (storage_personal.mdx §1, storage_company.mdx §1). The kind is
resolved leaf-safe in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` —
`resolveStorageType()` / `tracksIndexInLocalStorage()`, consumed by `tracking.service.ts` `filesYamlPath()`.
Whichever the location, `files.yaml` records, per file, enough
to tell whether it changed since last seen: a **hash** (fingerprint), **size**, **created**, and
**modified**. Compressible media (video/image) carry **two** hashes, **compressed** and **uncompressed**,
nested under the file entry (charter — see the CLAUDE.md "Per-file tracking metadata" example). This is the
same shape whatever the storage type:

```yaml
files:
  render/final_4k.mp4:
    size: 734003200
    created: 2026-05-01T10:12:00Z
    modified: 2026-06-18T14:33:00Z
    hashes:
      uncompressed: { hash: <fingerprint>, size: 734003200 }
      compressed:   { hash: <fingerprint>, size: 118489088 }
    analysis: [transcript, description, visuals_by_time]   # which §6 outputs exist
```

Writing the per-file fingerprint index (hash + size + dates; compressed/uncompressed variant hashes for media) is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` — `indexStorageFiles`.

**Building the index parallelizes the per-file fingerprinting.** A storage can hold thousands of large
files, and each file's fingerprint is a **head+tail byte read + sha256** — independent, I/O-ish work. So
`indexStorageFiles` fans the per-file `fingerprint()` calls **wide** across files through `mapLimit()`,
bounded by the **responsive budget** (`cores − 2`): many files are hashed at once while **2 cores stay
free for the HTTP loop + the IPFS node**, so indexing a large storage finishes quickly without pinning
the app. This is exactly the **responsive**, **wide-over-files** shape
[parallelization.mdx §3](./parallelization.mdx) assigns to the "Storage fingerprint / index" row —
fingerprinting is stat/IO + hashing, so it fans wide over independent files (not the capped-thread
fan-out reserved for the internally-multi-threaded compress/transcode jobs). **Indexing multiple
storages parallelizes across storages too** — each storage's index is independent and writes only its
own Local-Storage `repos/<repoKey>/files.yaml`, so a re-index that spans several storages runs them
concurrently under the same budget rather than one storage after another.

Parallelizing the per-file fingerprint (head+tail read + sha256) across files under the responsive budget, and parallelizing across storages when several are indexed, is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking.service.ts` — `indexStorageFiles` (the walk collects the large-file paths, then `mapLimit(paths, responsiveBudget(), fingerprint)`), drawing `responsiveBudget()` + `mapLimit()` from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts`.

## 6. Per-file media analysis (transcription · description · visuals-by-time)

For a file in a storage — **especially a video** — LFB can run analysis passes. Their outputs are
**generated artifacts** that travel with the storage, so any of the user's computers (or a teammate on a
company/community storage) gets them without re-running. Each pass is an explicit, opt-in operation (never
automatic on bytes — charter §6.1 spirit). **Transcript and description are written INSIDE the storage's
tracking area, path-mirrored, with the media's full filename and the extension APPENDED** — that area is
the working repo's **`.lfbridge/`** for kind `repo`, and the **root itself** for an SDL, which has no
`.lfbridge/` (§4)
([Transcribe.mdx §3](./Transcribe.mdx), [ai_description.mdx §2](./ai_description.mdx) are
authoritative — **no** extension replacement, **no** `analysis/` directory for these two);
**visuals-by-time** stays as **YAML under `<tracking area>/analysis/<relpath>/`** (Category-A, part of the
AI-description family). The **compression record** is **Category-B tracking state**, so it lives in Local
Storage at `~/T/_large_files_bridge/repos/<repoKey>/analysis/<relpath>/compression.yaml` (+ the sync-repo
mirror when enabled) — **never** in the working repo's `.lfbridge/` (§4, [compression.mdx §8](./compression.mdx)).

The passes:

* **Transcript** (**`<tracking area>/<rel-dir>/<name.ext>.transcription`** — path-mirrored, extension
  appended; `<repo>/.lfbridge/…` for a working repo, `<SDL root>/…` for an SDL)
  — **speech-to-text** of what people are *saying* in a video (or in an audio file). This is the "what was
  said" track. Authoritative: [Transcribe.mdx §3](./Transcribe.mdx).
* **Description** (**`<tracking area>/<rel-dir>/<name.ext>.ai_description`** — path-mirrored, extension
  appended)
  — a prose **summary/description** of the file: for a **video**, what it is about; for an **image**,
  **what is in the image**. The "what is it" track. Authoritative: [ai_description.mdx §2](./ai_description.mdx).
* **Visuals & action by time** (`visuals_by_time.yaml`, under `<tracking area>/analysis/<relpath>/`) — primarily
  for **video**: a timeline of
  **time ranges** with, for each, **what is visually going on** and **what is going on story-wise** — so
  the material can be re-used or re-created (think cinematography beats/shots). The "what is happening,
  when" track.

```yaml
# <tracking area>/analysis/render/final_4k.mp4/visuals_by_time.yaml
#   working repo: <repo>/.lfbridge/analysis/…   ·   SDL: <SDL root>/analysis/…
source: render/final_4k.mp4
segments:
  - start: "00:00:00"
    end:   "00:00:06"
    visual: "Wide desert dawn; single figure walks camera-left to -right."
    story:  "Establishing the journey; isolation."
  - start: "00:00:06"
    end:   "00:00:14"
    visual: "Cut to close-up, dust on face; hard side light."
    story:  "Cost of the journey registers on the character."
```

* **Images** get **Description** (what's in the image); Transcript and Visuals-by-time are video/audio
  concepts and are skipped for a still image.
* The results are cached against the file's fingerprint (§4.1) — re-running is skipped when the hash is
  unchanged.

Running the analysis passes and writing the per-file outputs — transcript/description as path-mirrored
`<name.ext>.transcription` / `<name.ext>.ai_description` files directly under the storage's committed
tracking area, visuals-by-time as YAML under that area's `analysis/`, and the Category-B compression record
under Local Storage `repos/<repoKey>/analysis/` (never the repo) — is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/analysis.service.ts` — `analyzeFile` (dispatches transcript / description / visuals-by-time by media kind), resolving the tracking area through `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `trackingBaseDir()` (never joining `LFBRIDGE_DIR` directly).

## 8. The sync repo — a storage's additive mirror for tracking state

A **company** or **Personal** storage may configure a **sync repo**: a dedicated Git repo, purpose-built
to **carry LFB's per-repo Category-B tracking state** across the user's computers — *"a Company repo only
for the Large File Bridge webapp to sync changes."* It is an **additive mirror** of the always-present
Local-Storage copy, not a replacement, and not a place that state moves *into* instead of the working repo
(Category B is never in the working repo regardless). This is the storages-family glue that makes tracking
state **travel** without polluting any working repo. The ideas it rests on are owned by sibling specs and
only cross-linked here:

* [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) — the authoritative **data-category
  placement policy**: Category B lives in Local Storage always, and in the sync repo also when enabled;
  it never enters a working repo (the earlier content-threshold three-tier model is retired).
* [repo_company_mapping.mdx](./repo_company_mapping.mdx) — the authoritative **owner mapping** (every repo
  maps to a company or Personal owner). The owner is **which** storage a repo's sync repo comes from (§9).
* [repo_owner_propagation.mdx](./repo_owner_propagation.mdx) — how a repo→company **ownership decision
  travels** to teammates (an assertion committed into the sync repo's `owner_map.yaml`) and is applied on
  another member's computer **only with that member's consent** (§9).
* [placement_radios.mdx](./placement_radios.mdx) / [repo_settings.mdx](./repo_settings.mdx) — the per-repo
  **placement radios** that let a repo's transcripts / AI descriptions go to `.lfbridge/`, **beside the
  file**, or **into the sync repo** (§8.2), and the per-repo **sync-repo toggle** (default off).

### 8.1 The churn problem, and the two homes for a repo's tracking state (LOCKED)

A working code repo must **not** carry LFB's churny tracking files — `repo_storage.yaml`, `files.yaml`,
and the per-file sidecars re-write on **every scan** and would pollute that repo's Git history with noise
unrelated to its actual source (`repo_storage.yaml` was the reported merge-conflict offender). So they
never go there. But that state still has to **travel** between the user's computers. The resolution is
categorical, by **data category** (artifact_placement_policy.mdx), not by any content threshold: Category-B
tracking state lives in **two homes** (additive), and a working repo is never one of them:

| Home | Path | Travels? | When |
| --- | --- | --- | --- |
| **Local Storage** (machine-local) | `~/T/_large_files_bridge/repos/<repoKey>/` | **No** — the authoritative per-machine working copy | **Always** — every scan/action reads & writes here first |
| **The owning storage's sync repo** | `<syncRepo>/repos/<repoKey>/` | **Yes**, via that repo's git backbone | **Also**, when the owning company/Personal storage has **configured a sync repo** AND the per-repo toggle is on (default off) — the travelling mirror, written IN ADDITION to Local Storage |

The sync repo is the **additive** second home: it gives a repo's tracking state a **travelling** copy while
Local Storage stays the authoritative working copy and the working repo stays pristine. One sync repo
gathers the tracking state of **every repo its owning storage owns**, each under `<syncRepo>/repos/<repoKey>/`,
keyed by the same stable **`repoKey`** Local Storage uses. Only the **Category-A** content artifacts
(transcripts / AI descriptions) ever live in the working repo's own committed `.lfbridge/` (§4).

```
~/BGit/Bryan_git/act3_state_sync/          ← ACT3 company storage's SYNC repo (a dedicated git repo)
├── storage.yaml                            ← type: company (this repo can itself be the company SDL, or a sibling)
└── repos/
    ├── act3-webapp/                         ← one <repoKey> dir per repo this storage OWNS (§9)
    │   ├── repo_storage.yaml                ←   the repo-wide facts (repo_tracking_scheme.mdx §2)
    │   ├── files.yaml                       ←   the per-file fingerprint index (§4.1)
    │   └── <rel-dir>/<name.ext>.transcription   ←   artifacts routed here by the placement radio (§8.2)
    └── act3-ai-service/
        └── repo_storage.yaml
```

Resolving the always-present Local-Storage tracking root and the optional additive sync-repo mirror root for a repo is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveTrackingRoot(repoRoot)` (returns the Local-Storage `repos/<repoKey>/` root, drawn from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/state-dir.ts` — `resolveRepoStateDir()`) and `resolveStateSyncRepo(repoRoot)` (returns `<syncRepo>/repos/<repoKey>/` or `null`); the mirror is performed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `mirrorToSyncRepo()` / `reconcileFromSyncRepo()`.

### 8.2 Placement radios can target the sync repo

A repo's transcripts and AI descriptions (Category A) are placed by the per-repo **placement radios**
([repo_settings.mdx](./repo_settings.mdx), [placement_radios.mdx](./placement_radios.mdx)): they may go
**in `.lfbridge/`**, **beside the media file**, or **into the sync repo**. When the radio is set to the
sync repo, an artifact for `<repoRoot>/<rel-dir>/<name.ext>` is written path-mirrored to
`<syncRepo>/repos/<repoKey>/<rel-dir>/<name.ext>.transcription` (extension **appended**, matching §6) and
travels with that sync repo rather than with the working repo. This is the storages-family reading of the
placement policy; the radio semantics themselves are authoritative in placement_radios.mdx.

## 9. Company / Personal ownership in the storages model

Every repo **maps to a company or Personal owner** — default **Personal**, auto-**company** when the git
remote carries a forge **ORG**, renamable + reassignable by the user. That mapping is owned end-to-end by
[repo_company_mapping.mdx](./repo_company_mapping.mdx) and **not** redefined here. What the **storages
model** adds is the consequence: the owner is **which** directory-based storage a repo belongs to, and
therefore **whose** sync repo (§8) its tracking state mirrors to.

* **Personal-owned repos** mirror their tracking state (when a sync repo is configured and the toggle is
  on) to the **[Personal storage](./storage_personal.mdx)**'s sync repo; unowned/unmapped repos default to
  Personal.
* **Company-owned repos** mirror to that **[company storage](./storage_company.mdx)**'s sync repo — so all
  of a company's repos share one travelling tracking-state repo under the company allow-list.
* The owner shows in the storages UI: a company storage's detail page can surface **the repos it owns**
  — an **owner-scoped** view of the [Repos](./repos.mdx) list — and each `StorageRow` already carries
  `companyName` for the label. "Filtered view" here means **scoping by navigation**: the storage you are
  on *is* the predicate, so it is not a rail facet the user unchecks
  ([tables.mdx §2.2](./tables.mdx#22-the-facet-rail--filtering-without-opening-a-dropdown-locked)) —
  the same distinction the IPFS page draws for its `?repo=` child filter
  ([ipfs.mdx §5.1.1](./ipfs.mdx)). Whatever facets that repo list carries still apply **within** the
  scope, all-checked by default.
* **A company assignment travels to teammates, consent-gated.** Assigning a repo to a company is not only a
  local grouping choice — because it changes **whose** sync repo the repo's tracking mirrors into (from
  Personal to the *shared* company repo), the decision is committed as an **assertion** into that company's
  sync repo (`<syncRepo>/owner_map.yaml`, keyed by the git remote) and **travels** to other members. It is
  never applied on another member's machine silently: the receiving member reviews and consents through a
  review page before any of their repos are re-mapped. This travel + consent layer is owned by
  [repo_owner_propagation.mdx](./repo_owner_propagation.mdx); this spec only notes that the owner→sync-repo
  routing above is what makes the decision a team fact worth propagating.

The owner→storage resolution the storages model consumes (already setting `companyName` per row from the remote's forge org, defaulting to Personal) — which the sync-repo resolver `resolveStateSyncRepo()` reuses to find whose sync repo a repo mirrors to — lives in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `classifyByConvention()` and `buildRow()` (sets `companyName`); the repo→owner mapping it reads is owned by [repo_company_mapping.mdx](./repo_company_mapping.mdx).

## 11. Acceptance criteria

1. There are **five storage kinds** — `local`, `repo`, `personal`, `company`, `community` — each with a
   detail spec; **Local** is settings/config (the DB replacement, no `storage.yaml`), the other four are
   directory hierarchies that hold the shared files.
2. Every **directory-based** storage has a **`storage.yaml`** at its root declaring **`name`** + **`type`**
   and **exactly one** type-specific block; **company** carries `company_name`, **community** carries which
   `community` (+ child props), each with type-scoped properties.
3. **`.lfbridge/` is a working-repo-only concept (§4, LOCKED — owned by
   [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)).** Only a **`repo`** storage has a
   hidden **`.lfbridge/`** at its root (`~/BGit/xyz/.lfbridge/`); a **`personal` / `company` / `community`
   SDL has none at all** — its **root *is* the `.lfbridge` area**, so `storage.yaml`, `mapped_dirs.yaml`,
   `files.yaml`, `manifest.yaml`, `bookmarks.yaml`, `devices/`, `analysis/`, and the path-mirrored
   artifacts all hang off `<root>` directly (which makes those names **reserved** at an SDL root — §0.3
   there, validator in [storage_settings.mdx §4a.1](./storage_settings.mdx#4a1-reserved-mapped-dir-keys--the-validator-this-page-owns-locked)). The split is keyed on the
   **resolved storage kind**, never a path or name (default `repo`). That tracking area is
   **committed and travels with the storage** but holds **only Category-A derived content** — the
   path-mirrored transcript / AI-description artifacts plus the visuals-by-time YAML of the AI-description
   family (§4); only the multi-GB media files themselves are git-ignored. **All Category-B tracking state**
   — the per-file fingerprint index (hash + size + dates; compressed/uncompressed variant hashes for
   media), the IPFS manifest, `repo_storage.yaml`, sidecars, history, decisions, and compression records —
   lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/`, **never** in a working repo,
   and is additionally mirrored to a configured sync repo (§8, §11.8).
4. A user can belong to **many** storages at once (multiple communities + companies + personal + every
   repo). The **Storages** left-bar tab shows them: **Personal**, one child **per company**, a **Repos**
   child that routes to the Repos tab (not a long list), and the opted-in **Communities**.
5. Storages are pinned between the user's computers over **IPFS** and **may also** clone to **Google Drive** and
   **Dropbox** (recorded in `storage.yaml` `clones`); the clones are mirrors, not the source of truth.
6. **Media analysis** can produce, per file: a **transcript** and a **description** written **directly
   under the storage's tracking area, path-mirrored, with the media's full filename and the extension
   APPENDED** (`<tracking area>/<rel-dir>/<name.ext>.transcription` · `.ai_description` — Transcribe.mdx §3 /
   ai_description.mdx §2), plus **visuals-and-action-by-time** (time ranges × visual × story) as YAML under
   `<tracking area>/analysis/` — where `<tracking area>` is `<repo>/.lfbridge/` for a working repo and the
   **`<SDL root>` itself** for personal / company / community (§4, AC #3) — video gets all three, images get
   description only; results cache against the file fingerprint.
7. **Building the fingerprint index parallelizes** the per-file fingerprint (head+tail read + sha256)
   **across files** under the **responsive budget** (`cores − 2`) via `mapLimit()`, and **across
   storages** when a re-index spans several — so a large storage indexes quickly without pinning the app,
   leaving 2 cores for the HTTP loop + IPFS (§4.1, [parallelization.mdx §3](./parallelization.mdx)).
8. A **company** or **Personal** storage may configure an optional **sync repo** (§8) — a dedicated Git
   repo that **mirrors owned repos'** noisy per-repo tracking state under `<syncRepo>/repos/<repoKey>/`.
   A repo's Category-B tracking state lives in **two homes** (§8.1), never in a working repo: **Local
   Storage** `~/T/_large_files_bridge/repos/<repoKey>/` **always** (the authoritative per-machine copy),
   and the owning storage's **sync repo** **also**, when configured and the per-repo toggle is on (the
   additive travelling mirror, default off). The earlier content-threshold three-tier model is retired;
   the placement policy is authoritative in [artifact_placement_policy.mdx](./artifact_placement_policy.mdx).
9. Every repo **maps to a company or Personal owner** (default Personal; forge ORG → company) per
   [repo_company_mapping.mdx](./repo_company_mapping.mdx); the storages model consumes that only to decide
   **whose** sync repo a repo mirrors to (§9) — Personal-owned → the Personal storage's, company-owned
   → that company storage's. Placement radios ([repo_settings.mdx](./repo_settings.mdx)) may target the
   sync repo for transcripts / AI descriptions (§8.2). Because a company assignment redirects the repo's
   tracking into the **shared** company sync repo, the decision **travels** to teammates as an assertion in
   `<syncRepo>/owner_map.yaml` and is applied on another member's computer **only with that member's consent**
   ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx), §9).
9a. **Both tables filter by value checkboxes on the facet rail, never a `<select>`**
    ([tables.mdx §2](./tables.mdx#2-filtering--the-value-visibility-checkbox-model-locked)): the
    **Storages** table's **Type** facet is ☑ Personal ☑ Company ☑ Community, all checked (§2.1), and the
    **storage-detail files** table's **Kind** facet is ☑ Video ☑ Image ☑ Audio, all checked (§7.0). The
    Kind facet **declares its vocabulary** and its **accessor reads the media kind its cell renders** —
    closing both §2.8 offences (a dead `any`-only dropdown, and an accessor/cell mismatch against
    `compressible`); a file with no media kind holds no value on the axis and is not a facet value.
9b. **The storage-detail files table LEADS with the three analysis icon columns** — **Transcribe · AI
    Description · OCR**, in that order, right of the row-select checkbox (§7.2). These are the **applicable
    subset** of the product-wide five-icon vocabulary
    ([tables.mdx §4c](./tables.mdx#4c-the-icon-control-columns--pin--ignore--transcribe--ai-description--ocr-locked-site-wide)):
    a `StorageFileRow` has **no per-file pin/git-ignore decision**, so **Pin and Ignore are omitted** here
    (the old Decision / Decided-by / Decided / Status columns never applied either). **Transcribe / AI
    Description / OCR status is DERIVED FROM each file's `analysis[]` outputs plus the filename kind** for
    applicability (Transcribe → audio/video, AI Description → image/video, OCR → image/video/PDF). The three
    render through the shared three-state `StatusActionIcon`, with an icon-only header and hover filling the
    left-bar hover-info panel, each in its unique done color (§7.2).
4a. **The Storages page proposes company storages from the user's own repos** (§2.2): a company storage is
    **1:1 with a forge organization**, read offline from each repo's remote, so a panel above the table lists
    every **qualifying** org with its **repo count** and the **directory that would be created**, each with a
    checkbox and a single **Create company storages** button. **Nothing touches the disk until that click** —
    creation makes a real directory and runs `git init`. A per-row **dismiss** is remembered; an org already
    **claimed** by an existing storage is shown as **adopted**, never duplicated; and each newly created
    company immediately carries the **"no remote yet"** fault
    ([storage_settings.mdx §4.1.1](./storage_settings.mdx#411-no-remote-yet-is-a-fault-this-page-shows-and-fixes-locked)).
    The **parent directory is a setting**, displayed as configured-or-derived.
4b. **The skipped-clones count is stated, never implied** (§2.2): orgs that fail the **authored-commit
    membership test** ([repo_company_mapping.mdx §3.2](./repo_company_mapping.mdx#32-which-orgs-are-actually-yours--the-authored-commit-membership-test-locked))
    are reported as a **count** with a way to see the list — *"28 organizations you've only cloned from were
    ignored"* — because **a silent filter and a bug look identical**. On the measured machine the filter
    removed **28 of 32** orgs, so the number is load-bearing, not decoration.
10. The Storages-page payload gains a per-storage **`stateSyncRepo`** descriptor (§10) — `configured`,
    `root`, `ownedRepoCount`, `remote` — non-null only for **company** and **Personal** storages, `null`
    for `local` and `community`; it extends `StorageRow` / `StoragesPageData` and is populated by
    `listStoragesPage()`.
