# LargeFileBridge — Compression

Compression is a **core value** ([charter](../CLAUDE.md#compression-a-core-value)): many large files people
pin are **uncompressed when they don't need to be**. LFBridge finds those and, **only when the user
asks**, compresses them — **images** and **video** (audio is out of scope for now, planned later).

This spec is the **superset**: it defines the compression **engine** (the quality-controllable brew tools
we drive, the medium-quality / prefer-lossless policy, the two hard invariants — resolution-preserving +
alpha-safety — and the codec preference settings), **and** the user-facing **Compress action + every
surface** that launches it (§4), the **data-safety guarantee** (§5), and the **"Compress inside" pop-over
dialog + per-file-safe background batch** (§12). It covers the two headline transforms the user cares about
— **PNG → JPEG** and **video re-encode to a smaller codec** — the **"may we change the file extension?"**
setting, and the **hard data-safety guarantee** (a compress must never leave a broken file behind).

## 1. Principles (LOCKED)

1. **Never automatic.** LFBridge never compresses on its own. The user clicks **Compress** on a file, or
   selects several / a whole table filter and clicks compress. Charter §6.1.
2. **Medium quality — the MIDDLE of the range — prefer lossless.** We target the **middle** of the
   quality↔size range (default `medium`), for **both images and video** — not maximum-aggression that
   throws away visible quality, and not maximum quality that barely shrinks. Where a **lossless**
   re-encoding applies (e.g. recompressing a PNG, or a lossless WebP), we use it: the file won't shrink as
   much, but **no quality is lost**. Tools are always invoked with an **explicit numeric quality
   parameter** — we **only** use tools that let us **control the quality level across the full range** (so
   it never silently maxes out compression). This is the **HARD REQUIREMENT** spelled out in **§2.1**.
3. **Never change the pixels' size.** The output MUST keep the **same aspect ratio** and the **same pixel
   resolution** as the input — for **both images and video**. **We never downscale.** This is verified
   after encoding (§5) — a size change aborts the operation.
4. **Alpha safety first.** Before compressing we run the **alpha-channel safety check** (§6). If real
   transparency would be lost, we refuse (or steer to an alpha-preserving codec).
5. **Recoverable.** Compression replaces the file via a **temp file**, and the original is moved to the
   **LFBridge trash** (recoverable) — never hard-deleted.
6. **Audio is out of scope** for now (planned later). The engine and settings carry an audio slot, but it
   is disabled by default.

## 2. The brew tools (quality-controllable)

We only use tools where the **quality level is an input we control**. Detected on PATH at runtime; a
missing tool degrades the relevant operation with a clear "install X" message (never a silent fail).

| Media | Tool (brew) | Role | Quality control |
| --- | --- | --- | --- |
| **Video** | `ffmpeg` | transcode video | `-crf <n>` (lower = higher quality); medium ≈ CRF 20–23 for H.264 |
| **Image** | `ImageMagick` (`magick`) | convert / re-encode, read dimensions + alpha | `-quality <n>` (≈ 85 medium); `identify` for `%w %h` + `%[opaque]` |
| **Image (PNG)** | `oxipng` | **lossless** PNG recompression (same pixels) | lossless levels; never changes pixels |
| **Image (WebP)** | `cwebp` | WebP encode (lossy `-q` or `-lossless`) | `-q <n>` / `-lossless` |
| **Image (JPEG)** | `mozjpeg` (`cjpeg`) / `jpegoptim` | better/lossless JPEG (re)encode | `-quality <n>` / `--max=<n>` |
| **Probe** | `ffprobe` | video dimensions + `pix_fmt` (alpha detection) | n/a |

`brew install ffmpeg imagemagick oxipng webp mozjpeg jpegoptim` provides the full set. The engine picks
per file from what's installed.

Detecting installed tools (which of ffmpeg/ffprobe/magick/oxipng/cwebp/cjpeg/jpegoptim are on PATH) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `detectTools`.

## 2.1 Quality control — the MIDDLE setting (HARD REQUIREMENT, LOCKED)

This is the **single most important compression rule** and it applies **equally to IMAGE and VIDEO**:

1. **Same pixel resolution.** Every compressed output keeps the **same aspect ratio and the same pixel
   resolution** as the input — for **both images and video**. We **never downscale** (verified after
   encoding, §5). Compression buys smaller files by spending bits more efficiently, **not** by throwing
   away pixels.
2. **A modern algorithm with a controllable quality↔size range.** For each media type we use a **modern,
   high-efficiency** codec/encoder that exposes a **continuous quality knob** — one end = **highest
   quality / larger file**, the other end = **maximum compression / visibly degraded**. We do **not** use
   any tool that only has a single "make it as small as possible" mode.
3. **We deliberately pick the MIDDLE of that range.** The default quality is **`medium`** (config default,
   never the extreme). At medium we keep **nearly all** the perceptible quality and still get a **much
   smaller** file — we give up only a **minuscule** amount of quality in exchange for a large size win.
   `high` and `low` (and `lossless`) exist for the user who wants to slide toward either end, but the
   product default sits in the middle.

### 2.1.1 The RISK this requirement guards against (why the tool choice matters)

The failure mode we are explicitly protecting against: **a compression tool that has no middle — it only
races to the tiniest possible file and sacrifices a lot of quality**, with no dial to trade a little size
back for a lot of quality. Such a tool is **disqualified**. Every tool we drive is chosen **precisely
because it takes an explicit numeric quality parameter** that spans the full range, so the engine can pin
it to the middle. The tools are **always** invoked **with that parameter set** — the quality is **never
left to the tool's default and never maxed out silently**.

### 2.1.2 The exact knob, range, and MIDDLE value per media type

| Media | Tool / encoder | The range knob | Scale (best-quality ↔ smallest) | MIDDLE (`medium`, the default) |
| --- | --- | --- | --- | --- |
| **Video** | `ffmpeg` — `libx264` (H.264), `libx265` (HEVC), `libaom-av1` (AV1) | **`-crf <n>`** (Constant Rate Factor) | `0` = lossless … `51` = worst; **lower = higher quality/bigger** | **CRF 21** for H.264/AV1, **CRF 24** for HEVC |
| **Image (lossy)** | ImageMagick `-quality <n>` / `cwebp -q <n>` (JPEG / WebP / JPEG 2000) | **`-quality` / `-q`** | `0` = smallest/worst … `100` = best/biggest | **quality 85** |
| **Image (lossless)** | `oxipng` (PNG kept as PNG) | optimization **level** only | n/a — pixels never change | lossless (no quality given up) |

Mapping of the four quality steps to concrete tool values (so `medium` is always the middle, not an
extreme):

* **Video CRF** — `lossless → 0`, `high → base−3`, **`medium → base`**, `low → base+5`, where `base = 24`
  for HEVC and `base = 21` for H.264 / AV1.
* **Image quality** — `lossless → 100`, `high → 92`, **`medium → 85`**, `low → 70`.

**The mechanism that guarantees the middle** is two things together: (a) the config default
`quality: "medium"` for both `images` and `video`, and (b) the two pure mapping functions that translate
`medium` into the concrete mid-range number. If either the default or a mapping ever drifted to an
extreme, the middle guarantee would break — so both are LOCKED here.

Choosing the concrete quality value for the chosen tool is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`videoCrf()` (CRF for H.264/HEVC/AV1) and `jpegQuality()` (0–100 for JPEG/WebP/JPEG 2000). The knob is
then passed to the actual command in `imageCommand()` (`-quality` / `cwebp -q`) and in `compressFile()`'s
ffmpeg build (`-crf`), both of which **omit any `-vf scale` / `-s` / resize** so resolution is preserved.
The `medium` default itself lives in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — the `compression.images.quality`
and `compression.video.quality` defaults.

## 3. What we do per file (the plan)

The engine builds a **plan** for a file from its kind + the user's codec preferences (§7), then runs it.

### 3.1 Images

* **PNG / BMP / TIFF / GIF (lossless source):**
  * If the user's quality is **lossless**, or the file stays PNG → **`oxipng`** lossless recompression
    (same pixels, smaller file).
  * Otherwise **convert to a compressible format** — normally **PNG → JPEG** at medium quality. If the
    user prefers/allows it and it is genuinely better, **JPEG 2000** (`.jp2`) may be used instead; a user
    can **deny JPEG 2000** in settings and it is then never chosen.
  * **Alpha guard:** JPEG has **no alpha**. If a PNG's alpha is actually used (§6), we do **not** convert
    to JPEG — we either keep PNG (lossless recompress) or, when the user allows it, target **lossless
    WebP** (which preserves alpha).
* **JPEG / WebP (already lossy):** re-encode at medium quality with `mozjpeg`/`cwebp` only when it yields a
  real gain; otherwise report **"already compressed — no gain"** and change nothing.
* **HEIC / HEIF / AVIF (Apple photos & efficient formats): a COMPATIBILITY conversion — normally → JPEG.**
  These are byte-efficient (HEVC/AV1-coded) but **poorly supported** by the sites people post to, so the
  default is to **convert to JPEG** for universal compatibility (governed by the `convert_types` setting
  and the per-extension checkbox — [images.mdx §2](./images.mdx#2-the-two-settings-settings--compression)).
  Two things make this different from a normal image compress:
  * **The container trap — extract the PRIMARY still (LOCKED).** A HEIC is a container that can hold a
    primary image, thumbnails, depth/auxiliary images, and (for Live Photos) a motion clip. We decode
    **only the primary still** (the `pitm` item) — never a thumbnail, an aux/depth image, or a frame from
    the motion video / its lead-in frames. Full rule + tooling in
    [images.mdx §4.1](./images.mdx#41-the-heic-container-trap--extract-the-primary-still-not-a-previewmotion-frame-locked).
  * **Exempt from the size-gain guard (§5).** JPEG is *less* efficient than HEVC, so HEIC → JPEG usually
    produces a **larger** file. Because it is a **compatibility conversion**, not a size reduction, it is
    **not blocked** by the "output must be smaller" rule (like the H.264 compatibility convert, §8.1) — the
    resolution-preserving and alpha-safety invariants still apply. The alpha guard steers a HEIC with
    **used** transparency to lossless WebP instead of JPEG.

The full **file-type conversion policy** (the two settings, the per-extension scope, and the HEIC
handling) is specified in its own part spec: **[images.mdx](./images.mdx)**. This engine honors it.

### 3.2 Video

* **`ffmpeg`** transcode to the user's preferred codec (default **H.264**, social-media-safe — §7) at a
  **medium CRF**, **preserving resolution** (no `-vf scale`, no `-s`). Audio stream copied through.
* **Alpha guard:** if the source video carries a **used** alpha channel (§6.2) and the target codec does
  **not** preserve alpha, we refuse — or, if the user's preferences include an alpha-preserving codec, we
  steer to it.

Building the per-file plan (source kind → tool + target codec + quality, honoring allow/deny prefs, the
`convert_types` / `skip_exts` conversion settings, and the HEIC-family → JPEG compatibility conversion with
its `formatConvert` size-gain exemption) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `pickImageTarget()` (images) and `pickVideoTarget()` (video), assembled by `checkFile()`. HEIC/HEIF/AVIF are treated as convertible (not "already done") in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` — `compressInfo()` / `compressBadge()`.

## 5. The data-safety guarantee — a compress must NEVER leave a broken file (LOCKED)

This is the **most important correctness rule** of the whole feature. Compressing a file means
**replacing** it, and a replace that ships a **broken** file is **data loss**. The engine is
**transactional and fail-closed**: it works on a **temp file**, and it only disposes of the original
**after** the new file has **passed every check**. A file that fails any check keeps its original
**untouched**.

The checks, in the **exact order** they run inside `compressFile()`
(`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts`):

1. **Tool exit + non-empty.** The transcode must exit `0` and produce a file with **bytes > 0**. Else
   fail, delete temp, keep original.
   * `runAsync()` result check + `safeSize()` (bytes > 0) in code — inside `compressFile()`.
2. **Output integrity — the output must actually DECODE (LOCKED — the 2026-07 fix).** We re-probe the
   output's pixel dimensions with the **same** probe that read the input. If the input's dimensions were
   readable but the **output's are not**, the encoder produced a **corrupt/truncated** file → **refuse,
   keep the original**. This is **independent of the `preserve_resolution` setting** — a user turning that
   off must not disable corruption detection.
   * The fail-closed integrity gate runs in code — the `if (inDims && !outDims)` block inside
     `compressFile()` (added 2026-07-14).
3. **Resolution unchanged.** Output width × height must equal input; a size change **aborts** — we never
   ship a downscaled file. (Verified after encoding via `videoInfo()` / `imageDims()`, i.e. `ffprobe` /
   `magick identify`; gated on the `preserve_resolution` setting.)
4. **Real size gain.** The output must be **smaller** than the input, or we keep the original and report
   **"no gain."** **One deliberate exception:** a **compatibility conversion** — HEIC/HEIF/AVIF → JPEG
   (§3.1) or the forced H.264 convert (§8.1) — carries a **`formatConvert`** flag and is **exempt from
   this one check** (never from checks 2 or 3), because its purpose is universal playback/compatibility,
   not shrinkage, and it may legitimately grow the file.
5. **Recoverable replace.** Only now do we dispose of the original (to **trash** by default — recoverable,
   §7) and move the temp into place (with the new extension if the format changed).
   * The disposition + rename run in code — the disposition block at the end of `compressFile()`;
     `trashOriginal()`.

The resolution-preserving and alpha-safety invariants are **never** waived.

### 5.1 The bug we fixed — a big file replaced by a ~134-byte broken JPEG

**Symptom (reported 2026-07):** JPEG files that were several KB were found reduced to **~134 bytes of
broken garbage** after a compress run; they had to be restored from a bigger prior version.

**Root cause — a missing output-integrity gate.** Before the fix, the only output validation between the
transcode and the irreversible replace was **`safeSize()` = "bytes > 0."** A **134-byte truncated/corrupt
JPEG passes that check** (134 > 0). The one deeper guard — the **resolution check** — was written as
`if (preserveResolution && inDims && outDims && dims differ)`, so when the output was **too broken to
probe** (`magick identify` exits non-zero → `imageDims()` returns `null`), that guard was **silently
skipped**. The corrupt 134-byte file is **smaller** than the original, so the **"no gain"** check *passed*
too — and the engine went on to **replace the original with garbage**. Verified reproduction: a 134-byte
truncated JPEG returns non-zero from `magick identify` (so `imageDims → null`) yet passes `safeSize`.

**The fix — fail closed on unreadable output.** We added **check 2 above**: if the input's dimensions were
readable but the **output's are not**, the output is corrupt → **block and keep the original**. A 134-byte
file can no longer decode to `640×480`, so it is now caught and the original is preserved. The fix is in
code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`compressFile()`'s output-integrity gate (the `if (inDims && !outDims)` block, added 2026-07-14).

**Two facts that keep this honest:**

* The default originals disposition is **trash** (`replace_original_to_trash: true`, §7) — which is *why
  the restore was possible*. The bulk "Compress inside" flow can opt into **hard delete** (§12), which
  would have made the same corruption **unrecoverable** — so the integrity gate matters most there.
* The specific example file (`austin_food_truck_tacos.jpg` in the `jfk/code` repo) currently shows
  **129-byte blobs in git** — but those are **Git LFS pointer files** (that repo's `.gitattributes`
  LFS-tracks `*.jpg/*.png/…`, per its "Renormalize test feed images into Git LFS" commit), **not**
  compressor output. That is a separate git-LFS smudge artifact. The **latent compressor defect above is
  real and independent**, fits the reported symptom exactly, and is now closed regardless of that repo's
  LFS state.

### 5.2 Residual / follow-up

If ImageMagick is **absent** but `oxipng` is present (a PNG-only lossless path), `imageDims()` can't read
input dims, so the integrity gate can't compare — the pre-existing tool-gap behavior. In practice `magick`
is installed (it's needed for almost every image plan and for the alpha check), so the gate is active on
the real path. A future hardening could add a format-agnostic decode probe for the oxipng-without-magick
case.

## 6. The alpha-channel safety check (LOCKED)

Before **every** compression we check whether real **transparency** would be lost.

### 6.1 Images (PNG etc.)

* Having an alpha *channel* is not the same as *using* it. We check whether the alpha is **actually
  used** — i.e. any pixel is non-opaque. With ImageMagick: `magick identify -format '%[opaque]'` returns
  `True` when the image is fully opaque (alpha unused) and `False` when transparency is present.
* If alpha **is** used and the target format has **no alpha** (JPEG / JPEG 2000), we **do not** convert —
  we keep PNG (lossless recompress) or target an alpha-preserving format (lossless WebP) when allowed, and
  **warn the user** that a JPEG conversion would lose transparency.
* If alpha is **not** used, converting PNG → JPEG is safe.
* If we **cannot determine** alpha usage (tool missing), we treat it as **possibly used** and refuse the
  lossy-to-no-alpha conversion — fail safe.

### 6.2 Video (which formats carry alpha)

Most delivery codecs have **no** alpha; a few do. We check the source's pixel format (`ffprobe` `pix_fmt`
— an alpha format looks like `yuva420p`, `rgba`, `bgra`, `argb`…) and the target codec's capability:

* **Alpha-capable (may carry used transparency):** **ProRes 4444 / 4444 XQ** (in MOV), **QuickTime
  Animation (RLE)**, **QuickTime PNG**, **VP8 / VP9 with alpha** (in WebM), **FFV1**, **GoPro CineForm**,
  uncompressed RGBA. (Verified 2026; a web check can refresh this list.)
* **No alpha (typical delivery, social-media-safe):** **H.264 / AVC**, **H.265 / HEVC** (main profiles),
  **MPEG-4**, most **MP4**. **AV1** alpha support is niche/inconsistent — we treat it as no-alpha for
  safety.
* **Rule:** if the source `pix_fmt` indicates a **used** alpha channel and the target codec is in the
  *no-alpha* set, **refuse** — unless the user's preferences select an alpha-preserving target (e.g. VP9
  alpha in WebM), in which case we use that.

Running the alpha safety check (PNG `%[opaque]`; video `pix_fmt` vs. target codec alpha capability) before any compression is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `checkFile()`, using `imageAlphaUsed()` (image `%[opaque]`) and `pixFmtHasAlpha()` (video `pix_fmt`).

## 7. Codec preference settings (per media type)

[Settings](./settings.mdx) carries a **Compression** section letting the user say, **per media type**
(images / video / **audio** — audio disabled for now), which codecs they **want** and which they **never
want**. The reason to deny a technically-best codec is often **social-media compatibility**: the newest,
smallest codec may not be supported by the sites the user posts to.

Settings, per media type:

* **Enabled** — may this media type be compressed at all.
* **Quality** — `low | medium | high | lossless` (default **medium**; prefer lossless where it applies).
* **Prefer** — ordered list of target codecs; the first one that is **allowed** and **available** wins.
* **Deny** — codecs the user never wants chosen (e.g. **JPEG 2000** for images, **AV1** for video).
* **Convert file types** (`convert_types`, default **ON**) — may a compress **change the format** to a
  better compressible / more-compatible target (HEIC → JPEG, PNG → JPEG)? Off = format-preserving only.
  Specified in [images.mdx §2.1](./images.mdx#21-convert-file-types-when-compressing--master-toggle-default-on).
* **Compress all file types** (`skip_exts`, default **empty = all in**) — a per-extension **opt-OUT** list;
  every recognized extension is in scope unless the user unchecks it. Specified in
  [images.mdx §2.2](./images.mdx#22-compress-all-file-types--scope-default-on-with-a-per-extension-opt-out-list).

Good defaults (chosen for **social-media compatibility**):

| Media | Quality | Prefer (in order) | Deny (default) |
| --- | --- | --- | --- |
| **Images** | medium | `jpeg`, `webp` (lossless WebP for alpha) | `jpeg2000` |
| **Video** | medium | `h264`, `hevc` | `av1` |
| **Audio** (disabled) | medium | `aac` | — |

Global toggles: **preserve resolution** (LOCKED on — §5) and **replace-original-to-trash** (recoverable —

## 8. Running a tool (temp file → replace)

brew tools transcode to a **new file**, so the flow per file is:

1. Build the plan (§3) and run the **alpha check** (§6) — abort with a clear reason if unsafe.
2. **Capture the BEFORE state first — always, for every file, before we touch a byte** (§8.0): the
   original's **exact content hash + size**, and for **video/image** the **perceptual content
   fingerprint** ([perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx)).
3. Transcode to a **temp file** (under the state dir's temp area).
4. **Verify** output resolution == input and output < input (§5) — abort (delete temp) otherwise.
5. Move the **original** to the **LFBridge trash** (recoverable), then move the **temp** into the final
   path. When the format changes (PNG → JPEG), the final path takes the new extension; when it stays the
   same format, the path is unchanged.
6. **Capture the AFTER state** (exact hash + size, and the post perceptual fingerprint), then record the
   result in **three** places. **All three are Category-B tracking state** — LFB's compression bookkeeping,
   which **never enters the working repo**. They are written to **Local Storage** at
   `~/T/_large_files_bridge/repos/<repoKey>/…` **always**, and — when the owning company/Personal storage
   has a **sync repo** configured and the per-repo toggle is on — additionally mirrored to
   `<syncRepo>/repos/<repoKey>/…` so they travel ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
   the data-category split):
   * the file's tracking entry in `files.yaml` — the **compressed** and **uncompressed** hashes + sizes
     ([charter per-file tracking](../CLAUDE.md); [repo_tracking_scheme.mdx §3.1](./repo_tracking_scheme.mdx));
   * a **compression record** in Local Storage —
     `~/T/_large_files_bridge/repos/<repoKey>/analysis/<relpath>/compression.yaml` — capturing the
     **original filename + extension** and the **size before / after**
     ([artifact_placement_policy.mdx §3](./artifact_placement_policy.mdx)). It lives in Local Storage
     always; when the sync repo is enabled it also travels there, so every other computer that carries the
     storage learns the file was compressed, from what, and by how much, and never re-compresses it — but
     it is **never** written into the working repo.
   * a **`compress` event** appended to the file's per-file sidecar under
     `~/T/_large_files_bridge/repos/<repoKey>/files/<relpath>.yaml`
     ([repo_tracking_scheme.mdx §3.2](./repo_tracking_scheme.mdx)) — `before{hash,size}` /
     `after{hash,size}` / `fingerprint_before` / `fingerprint_after` / `codec` / `on_device` / `by` — the
     human-legible per-file history of exactly what this compression did.

### 8.0 Capture before, then after — the pre/post hash + perceptual fingerprint (LOCKED)

**Before we compress a file — and this is true for ALL files — we store the hash of it first.** For
**video and image** files we additionally compute the **perceptual content fingerprint**
([perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx)) — the PhotoDNA-style, **local-only,
never-phones-home** hash that recognizes the same content across resolution / compression / codec /
format changes. **After** the compress we capture the post-compression hash + size and the post
fingerprint.

* **Why the pre-capture must happen first.** Once step 5 replaces the bytes, the original is only in the
  trash — the authoritative "what it was" must be recorded *before* the transcode, never reconstructed
  after.
* **The fingerprint pair is the proof of preservation.** `fingerprint_before` and `fingerprint_after`
  should match **within the perceptual threshold** ([perceptual_fingerprint.mdx §4](./perceptual_fingerprint.mdx)) —
  that is how LFB verifies the compression **kept the same content** rather than corrupting it. The exact
  hashes prove byte-change; the fingerprints prove content-sameness.
* This is the same before/after pair the [two-computer tour's April step](./use_case_two_computers.mdx#5-april--studio-tower--the-files-arrive-via-git-compress-half)
  walks. It applies identically to a **format conversion** (§8.1, PNG→JPEG / HEIC→JPEG) via a `convert`
  event.

Capturing the pre-compression hash/size/fingerprint before the transcode and the post values after, and appending the sidecar `compress`/`convert` event, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile` (calls `perceptual.service.ts` `fingerprintImage`/`fingerprintVideo` before + after, and `file-sidecar.service.ts` `appendFileEvent`).

Running the tool to a temp file, verifying, and doing the recoverable replace is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` — `compressFile` (single) / `compressBatch` (many); the compression record is written via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/analysis.service.ts` — `writeCompressionRecord()` (new), which resolves its `analysis/<relpath>/compression.yaml` destination through `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `resolveTrackingRoot()` (the Local-Storage `repos/<repoKey>/` root, never the working repo) and mirrors it to the sync repo (when enabled) via `tracking-sync.service.ts` — `mirrorToSyncRepo()`.

### 8.2 Per-run originals disposition (bulk Compress inside)

global `compression.replace_original_to_trash` default (§7). The bulk **Compress Inside** dialog (§12)
deliberately differs: it surfaces disposition as an **explicit per-run choice** and **defaults it to HARD
DELETE** — a reconciled contradiction (the user asked for exactly that for a bulk run; the one-off default
is untouched, §12.5).

The engine honors a per-call `deleteOriginal: "hard" | "trash"` override (`CompressFileOpts`) that **wins
over the global default** — `"hard"` → `fs.unlinkSync`, `"trash"` → `trashOriginal`; when omitted it falls

part-way keeps its original either way (§5). And because a bulk run may be hundreds of files, the heavy
transcode now runs via an **async** `spawn` so a background batch **never blocks the event loop** (quick
probes stay synchronous). Both are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`compressFile()` (`const disposition = o.deleteOriginal ?? (settings.replaceOriginalToTrash ? "trash" : "hard")`)
and `runAsync()` (the `child_process.spawn` runner for the heavy tool).

### 8.3 Parallelism & per-job thread-capping (LOCKED)

Compression is the app's heaviest bulk work, so it draws the canonical **Core Budget**
([parallelization.mdx](./parallelization.mdx)) — but the engine itself stays deliberately simple: it
compresses **one file at a time** and does **not** decide how parallel to be. The fan-out lives one level
up, in the queue that drives it ([job_queue.mdx §3](./job_queue.mdx)). What the engine owns is the **per-job
thread cap** — the knob that lets the queue pack many jobs into the budget without oversubscribing the box.

* **A one-off compress uses all cores (no cap).** When `compressFile` is called for a **single** file — the
  media viewer's **Compress** button (§4), the **Convert to H.264** offer ([codecs.mdx §9](./codecs.mdx)) —
  it is **not** a batch. It passes **no** thread cap, so ffmpeg/oxipng/cwebp use their **own all-core
  default** and that one interactive file finishes as fast as the machine allows. This is the §2 corollary of
  the parallelization policy ([parallelization.mdx §2](./parallelization.mdx)).
* **A batch caps each job so the queue can fan WIDE/NARROW.** When the background queue drains a **"Compress
  inside"** run over **many** files (§12), each `compress` task is
  **thread-capped** so `concurrentJobs × threadsPerJob ≈ budget`, never `cores²`
  ([parallelization.mdx §2](./parallelization.mdx), LOCKED). The two media kinds fan **opposite** directions
  because their tools have opposite threading profiles:
  * **Image jobs run 1-thread — fan WIDE.** oxipng/cwebp/mozjpeg/ImageMagick are ~single-threaded per file,
    so the queue runs **`budget` of them at once**, one thread each — one job per core saturates the box.
  * **Video jobs get a small cap — fan NARROW.** ffmpeg is internally multi-threaded, so the queue runs
    **`floor(budget / videoThreads)`** jobs and passes each a **`videoThreads`** cap (default 4). A few
    capped jobs fill the budget; without the cap, N videos would each grab every core and thrash.

The mechanism is a single optional field the queue sets and the engine plumbs into whichever tool it runs:

* **`CompressFileOpts.threads`** — the per-job cap. **Omitted** for a one-off (tool uses its all-core
  default); **set** by the queue for a batched job. It maps per tool: **ffmpeg `-threads <n>`**, **oxipng
  `--threads <n>`**, **cwebp** multithread **on/off** (`-mt` when `n > 1`, single-threaded otherwise), and
  ImageMagick **`-limit thread <n>`**. Because the engine only forwards the number, the "how parallel"
  decision stays in **one** place (the queue's live Core Budget), never hardcoded here.

Accepting and threading the per-job cap into each tool is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`CompressFileOpts.threads` (the optional cap), applied in `compressFile()`'s ffmpeg build (`-threads`) and in
`imageCommand()` (oxipng `--threads`, `cwebp` `-mt`, `magick -limit thread`); omitted → the tool's all-core
default. The queue that supplies the cap per media bucket is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` — `capFor()` / `bucketOf()` drawing `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts` — `coreBudget()`.

### 8.4 The "already-compressed" marker — durable, in-file, mesh-wide skip (LOCKED)

**The problem this solves.** A user points **Compress inside** at a huge, deep tree (e.g. **18,000 files**)
and the run may take **several passes** — a reboot mid-run, a machine that only got halfway, or simply
re-running it later. On the **second** pass, and on **every other computer in the mesh**, files we already
compressed must be **skipped instantly**, not re-encoded. Today they are **not**: the only real "already
compressed" guard is the **post-transcode size check** (§3.1/§8) — the engine runs a **full ffmpeg
transcode** and only *then* discovers "no gain". That is exactly the wasted work the user is seeing, and it
repeats on **each** machine because there is no durable, travelling "we did this" signal the walk can read
**before** deciding to compress.

We fix it with an **in-file marker** the engine reads **before** any transcode.

**The IN-FILE marker (authoritative, travels with the bytes). LOCKED — IMPLEMENTED.**
When the engine compresses a file, it stamps a small tag **into the output's own container metadata** — the
standard **`comment`** field, which nothing else uses — recording that **LargeFileBridge already compressed
this**. This is the layer the user asked for by name: *"find a property in an MP4 file or other video files
or JPG files that are unused and we are allowed to use."* Its whole virtue is that it **rides with the
file's bytes**: when the compressed file is pinned to another computer over IPFS
([ipfs.mdx](../knowledge/ipfs.mdx)) the marker arrives **inside** it, so **every** machine — regardless of
storage type — reads the same "already compressed" truth from the file itself. It does **not** depend on any
tracking-state travel, which is why it works even when a repo has **no sync repo** configured — the
Category-B compression record ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)) is
Local-Storage-only and does not travel in that case, but the in-file marker rides with the bytes regardless.

* **No new tool — written inline by the encoder we already run (LOCKED).** Because every compress already
  re-encodes the file, the marker is set **in that same pass**, for free: **video** via
  **ffmpeg `-metadata comment=…`** (into the mp4 `moov/udta`), **images** via **ImageMagick
  `-set comment …`** (into the JPEG `COM` / PNG `tEXt`). No `exiftool`, no extra remux/rewrite pass. (An
  optional `exiftool` could later broaden coverage to formats that skip the ffmpeg/ImageMagick path — e.g.
  a WebP produced by `cwebp`, or a PNG by `oxipng --strip`, both of which drop the comment — but it is **not
  required** and is not a dependency.)
* **The payload.** `comment = "LFBcompressed;v1;<codec>"`:
  * `LFBcompressed;` — the **presence prefix** (the whole skip test is "does the comment start with this").
  * `v1` — the **marker version**, so a future re-tuned engine can invalidate old marks by bumping it
    (a `v1` mark is treated as **absent** once the engine reads `v2`, so those files re-sweep).
  * `<codec>` — what it was compressed to (e.g. `h264`, `jpeg`), for the read-out (§9).
* **The read (skip decision) — enforced in the engine, per file.** At the top of a compress, the engine
  reads the file's `comment` (**ffprobe** for video, **`magick identify -format %c`** for images — the same
  fast probes used for dimensions); if it starts with `LFBcompressed;` the file is **skipped immediately**,
  status `skipped`, reason **"already compressed (marker)"**, with **no transcode and no size probe**. This
  runs in **`compressFile`**, so **both** a one-off compress and **every file of a bulk batch** honor it.
* **Why per-file at compress time, not an up-front walk scan (LOCKED).** Reading a container tag on 18,000
  files up front would itself be slow, so the marker is checked when each file **reaches the front of the
  queue** — the fast probe *replaces* the full transcode for that file. A re-run of an already-compressed
  tree therefore becomes **"probe-and-skip per file"** (tens of ms each) instead of **"re-encode per file"**
  (seconds each), and a peer that already holds the compressed files skips them the same way — which is what
  the user wants across passes **and** across the mesh.

**The extension heuristic still runs first.** The cheapest test is unchanged: `compressInfo` (§8 /
[scan.mdx §4](./scan.mdx)) drops a non-media or already-lossy **extension** with zero I/O, so a re-encoded
`.jpg`/`.webp` is skipped by extension **before** the marker is even read. The marker is what catches the
cases the extension can't — a **video** (an `.mp4` always *looks* compressible by extension) and a **PNG
kept as PNG** — on re-runs and on peers.

**The Local-Storage compression record (fast local cache) — future optimization.** The record already
written today, `~/T/_large_files_bridge/repos/<repoKey>/analysis/<relpath>/compression.yaml`
(`writeCompressionRecord`, §8), is still **written but not yet read**. Wiring a `readCompressionRecord()`
into `walkCompressible()` (keyed by relpath +
[fingerprint](./storage_local.mdx#93-repo-manifests-travel-via-git-the-elegant-part)) would let a bulk run
**drop known-compressed files from the batch entirely** — skipping even the per-file probe on a same-machine
re-run — but it is a **speed optimization on top of** the authoritative in-file marker, not a correctness
requirement. Tracked as a follow-up; the marker above already makes re-runs and peers correct.

Detecting the marker before compressing, and stamping it during the encode, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`isAlreadyCompressed()` / `readMarker()` (ffprobe `format_tags=comment` for video, `magick identify %c` for
images), consulted at the top of `compressFile()` so a marked file returns `skipped` **before** any
transcode (covering one-off **and** batched runs); the marker itself is written by `markerPayload()` folded
into the **ffmpeg** args (`-metadata comment=…`) and the **ImageMagick** args (`-set comment …`) inside
`imageCommand()`. No new tool is probed in `detectTools()`.

## 10. Acceptance criteria

1. Compression runs **only** on explicit user action — one file, several checked files, or **every file in
   the current table filter** — never automatically.
2. It uses **quality-controllable brew tools** (`ffmpeg`, ImageMagick, `oxipng`, `cwebp`, `mozjpeg`) at a
   **medium** quality, **preferring lossless** where it applies; a missing tool degrades with an
   "install X" message, not a silent fail.
2a. **The MIDDLE-quality requirement (§2.1) holds for both images and video:** the tool exposes a full
    quality↔size range (video `-crf 0…51`; image `-quality 0…100`), and the engine pins it to the middle
    by default — **CRF 21** (H.264/AV1) / **CRF 24** (HEVC) for video, **quality 85** for lossy images —
    at the **same pixel resolution** as the input. A tool with no controllable middle is disqualified.
3. The output **always keeps the same aspect ratio and pixel resolution** (verified after encoding; a size
   change aborts). We **never downscale**.
4. Before compressing, the **alpha-channel safety check** runs: a **used** PNG alpha blocks JPEG/JPEG-2000
   conversion (keep PNG / lossless WebP); a **used** video alpha blocks a no-alpha target codec — unless
   the chosen codec preserves alpha. Undeterminable alpha fails safe (refuse).
5. **PNG → JPEG** by default; **JPEG 2000** only when better **and** not denied; the user can **deny**
   specific codecs **per media type** (images / video / audio) in Settings, with **social-media-safe
   defaults** (deny `jpeg2000` / `av1`). **Audio is not compressed** (disabled) for now.
5a. **File-type conversion (images.mdx):** two per-media settings both **default ON** — `convert_types`
    (may change format on compress) and `skip_exts` (per-extension opt-OUT; empty = all in). **HEIC/HEIF/
    AVIF → JPEG** is a **compatibility conversion**: it decodes the **primary still** only (never a
    thumbnail / depth-aux / Live-Photo motion frame — [images.mdx §4.1](./images.mdx)), keeps the **same
    pixel resolution**, and is **exempt from the size-gain guard** (§5). With `convert_types` off,
    compression is format-preserving.
6. Tools run via a **temp file**; the original is moved to the **LFBridge trash** (recoverable). **Before
   the transcode, for every file, the BEFORE hash + size (and, for video/image, the perceptual content
   fingerprint) are captured first** (§8.0); the AFTER values are captured after. Tracking records the
   compressed + uncompressed hashes/sizes, **a compression record**
   (`~/T/_large_files_bridge/repos/<repoKey>/analysis/<relpath>/compression.yaml`, original name/extension
   + size before/after), **and a per-file sidecar `compress` event**
   ([repo_tracking_scheme.mdx §3.2](./repo_tracking_scheme.mdx)) carrying `before`/`after` hash+size and
   `fingerprint_before`/`fingerprint_after` (which match within the perceptual threshold as proof the
   content was preserved). **All three are Category-B tracking state** — written to **Local Storage**
   always, additionally mirrored to the company/Personal **sync repo** when enabled, and **never** into
   the working repo ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)). The authoritative
   **in-file marker** (§8.4, implemented) is what makes other computers never re-compress; reading these
   records as a local cache to prune the batch up-front is a tracked follow-up.
7. The **View-one-file** page shows the file's **codec** and **compressed/uncompressed** state.
8. **Parallelism (§8.3, LOCKED):** the engine compresses **one file at a time** and takes an optional
   **`CompressFileOpts.threads`** cap. A **one-off** compress **omits** it (the tool uses its all-core
   default); a **batched** job driven by the queue is **capped** so `concurrentJobs × threadsPerJob ≈ budget`,
   never `cores²` — **image** jobs run **1-thread** (fan wide, `budget` at once), **video** jobs get a small
   cap (fan narrow). The cap maps to **ffmpeg `-threads`**, **oxipng `--threads`**, **cwebp** MT on/off, and
   **`magick -limit thread`** ([parallelization.mdx](./parallelization.mdx), [job_queue.mdx §3](./job_queue.mdx)).
9. **Already-compressed skip (§8.4, LOCKED — IMPLEMENTED):** a file we already compressed is **skipped
   before any transcode**, on **re-runs** and on **every other computer in the mesh**, via an **in-file
   marker** written into the output's own **`comment`** container tag (`"LFBcompressed;v1;<codec>"`). It is
   written **inline by the encoder we already run** — **ffmpeg `-metadata comment=…`** (video) / **ImageMagick
   `-set comment …`** (images), **no new tool** — and read back with **ffprobe** / **`magick identify %c`**
   at the top of `compressFile()`, so a marked file returns `skipped — "already compressed (marker)"` for
   **both** one-off and bulk runs. Because the marker **rides in the file's bytes over IPFS**, a peer that
   already holds the compressed file skips it too. The cheap **extension heuristic** still runs first, and
   the version tag (`v1`) lets a re-tuned engine invalidate old marks. Verified end-to-end: compress →
   111,360→28,253 bytes; a second pass on the output → `skipped (marker)`, no re-encode. (Reading the
   Local-Storage compression record to prune the batch up-front is a tracked speed follow-up, not required

10. **Never leaves a broken file (§5, the data-safety guarantee, LOCKED):** before the irreversible

    (c) keep the exact resolution, and (d) be smaller (except a `formatConvert` compatibility conversion).
    The original goes to **recoverable trash** by default. Regression guarded: a ~134-byte truncated JPEG
    (passes `safeSize`, fails to probe) is now caught and the original preserved.

### 12.4 The per-file transactional safety rule (LOCKED — the load-bearing invariant)

This is the heart of the bulk flow and it is **LOCKED**. The batch treats **each file independently** and
it **never deletes an original that did not successfully compress**:

* **Per file, in order:** compress the file to a **temp** output, **verify** it (§5: decode, same
  resolution, real size gain), and **only then** dispose of the original per the Originals radio
  (hard-delete or trash) and move the temp into place.
* **A file that errors part-way is left completely untouched.** If a file's transcode fails, or its output
  fails verification, or the alpha check blocks it, **its original is not deleted** — neither hard-deleted
  nor trashed — and the temp is discarded. The batch **records the error and continues to the next file**.
  One bad file never halts the run and never endangers a different file's original.
* **Half-done is safe.** If the run compresses the first ten files fine (and, with Hard delete chosen,
  those ten originals are gone) and the eleventh errors, the eleventh's original **remains on disk**; files
  twelve onward still process. Deletion is strictly downstream of that one file's verified success.
* **Final error list.** When the whole run finishes, the files that errored are **shown to the user as a
  list** (with each file's reason). A clean run shows a success summary and no list.

This safety is structural: disposal lives at the very end of the single-file path in the engine, reached
only after verify + gain (§5). It is enforced in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`compressFile()` (the disposition block runs only after the integrity/resolution/gain checks; on any
earlier failure it returns a `blocked`/`failed` result **without** touching the original). The per-op
override the radio drives is the `deleteOriginal` field of `CompressFileOpts` (`"hard"` → `fs.unlinkSync`,
`"trash"` → `trashOriginal`). The batch's per-file isolation (catch a task's throw, record it against the
batch, move on) is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` —
`runTask()` (the `compress` branch: `track(...)` around `compressFile`, then `recordBatchResult()`;
`compressed`/`skipped` count as OK, `blocked`/`failed` are recorded as errors; a thrown task is still
counted so the batch always finishes) and `recordBatchResult()` (appends `{ path, reason }` to the batch's
`errors[]`).

### 12.5 The default-hard-delete contradiction, reconciled (LOCKED)

§1.5 / §8 and the charter say a compressed file's original always goes to the **recoverable LFBridge
trash** — never hard-deleted — and that stays the default for a **one-off** Compress (the file/viewer menu,
[menus.mdx §5.3](./menus.mdx)). **This bulk dialog deliberately differs:** it makes disposition an
**explicit per-run choice** and **defaults it to Hard delete**, because the user asked for exactly that.
The reconciliation:

* The **global** `compression.replace_original_to_trash` setting still governs one-off compresses (trash by
  default).
* This **dialog's radio** is a **per-run override** passed to the engine (`deleteOriginal`), so a bulk run
  does what the user picked **for that run** regardless of the global default.
* The dialog's **default is Hard delete**; the user can switch to the recoverable trash for any run.

The per-run override winning over the global default is in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` —
`compressFile()` (`const disposition = o.deleteOriginal ?? (settings.replaceOriginalToTrash ? "trash" : "hard")`).

### 12.6 Continue → background batch, return immediately (LOCKED)

**Continue** does **not** run the work in the request. It calls the enqueue endpoint, which plans the set
and hands it to the [background queue](./job_queue.mdx), then **returns immediately** so the user can keep
navigating other tabs while the batch drains ([processing.mdx](./processing.mdx)).

* **The walk (server-side).** The endpoint walks the directory for the **selected kinds**, honoring the
  **recursive** flag and skipping the scan's `HARD_SKIP` / hidden / tracking dirs. It includes only files
  whose **extension heuristic** (`compressInfo`) says they should compress, so already-lossy extensions are
  dropped with zero I/O. The stronger, cross-machine skip — the authoritative **in-file marker** (§8.4) —
  is enforced **per file when it reaches the front of the queue**, inside `compressFile()`: a file that
  already carries our `comment` marker (`LFBcompressed;…`) returns `skipped ("already compressed (marker)")`
  **before any transcode**. That is why a **re-run** (a second pass after a reboot) — and a run on
  **another computer in the mesh** that already has the compressed files — skips them **instantly instead
  of re-encoding**: the marker rides inside the file's bytes over IPFS. As it goes, the walk **stamps each
  eligible file with its media kind** — `image` or `video` — because the two kinds draw **different**
  concurrency budgets downstream (§8.3, [job_queue.mdx §3](./job_queue.mdx)).
* **A batch is created** grouping those files, and each file is enqueued as a `compress` task carrying the
  chosen `deleteOriginal` mode, the stamped **`mediaKind`** (image vs video), and the `batchId`.
* **The plan returns immediately** — `{ batchId, considered, eligible, queued, images, videos }` — and a
  single toast states what will happen (§12.7). The queue drains the tasks against the **media-aware
  mass-compute core budget** (§8.3, [parallelization.mdx](./parallelization.mdx)): **image** tasks fan
  **wide** — ≈90% of cores, one thread each — while **video** tasks fan **narrower** — `floor(budget /
  videoThreads)` jobs, each ffmpeg `-threads`-capped — so `concurrentJobs × threadsPerJob ≈ budget`, never
  `cores²`. A folder of 300 files therefore **pins the machine near 90%** until it's done, instead of
  crawling two-at-a-time. The heavy transcode runs via an **async** `spawn` (`runAsync()`) so it **never
  blocks the event loop**.

> **Parallelism does not weaken the safety rule (§12.4, LOCKED).** Draining many files at once changes
> **how many** files are in flight, never **how** each one is handled: every file is still processed
> **independently** — temp → verify → dispose — so the transactional guarantee of §12.4 is **unchanged**.

`POST /api/compress/inside` with body `{ root, images, videos, recursive, deleteOriginal }` returns the
