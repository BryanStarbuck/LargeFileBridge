# LargeFileBridge ‚Äî Compression

Compression is a **core value** ([charter](../CLAUDE.md#compression-a-core-value)): many large files people
sync are **uncompressed when they don't need to be**. LFBridge finds those and, **only when the user
asks**, compresses them ‚Äî **images** and **video** (audio is out of scope for now, planned later).

This spec defines the compression **engine**: the quality-controllable brew tools we drive, the
medium-quality / prefer-lossless policy, the two hard invariants (resolution-preserving + alpha-safety),
the codec preference settings, and how the user triggers it on one / many / a whole filter.

Siblings this builds on and does not repeat:

* [charter ‚Äî Compression](../CLAUDE.md) ‚Äî the product rules: video first / image second; never alter a
  file unless the user asks; the learned uncompressed-vs-compressed size baselines; the PNG alpha-check
  requirement; the two-hash (compressed/uncompressed) per-file tracking.
* [Menus](./menus.mdx) ‚Äî the **Compress‚Ä¶** and **Convert to compressible‚Ä¶** catalog items (explicit-click
  offers with a confirm, ¬ß6.1). This spec is what those items run.
* [Compress inside](./compress_inside.mdx) ‚Äî the pop-over dialog + **per-file-safe background batch** that
  drives this engine over **many** files in a directory (the directory `‚ãÆ` and page-level compress links).
  It surfaces originals disposition as a per-run choice (¬ß8.2) and drives `compressFile` file-by-file.
* [View one file / media viewer](./media_viewer.mdx) ‚Äî where a single file's **codec + compressed state**
  are shown and the **Compress** button lives (¬ß4).
* [One repo](./one_repo.mdx) ‚Äî the files table where the user **checks several** files or **compresses the
  whole current filter** (¬ß4.4 bulk actions).
* [Settings](./settings.mdx) ‚Äî where the **codec allow/deny per media type** preferences live.

## 1. Principles (LOCKED)

1. **Never automatic.** LFBridge never compresses on its own. The user clicks **Compress** on a file, or
   selects several / a whole table filter and clicks compress. Charter ¬ß6.1.
2. **Medium quality ‚Äî the MIDDLE of the range ‚Äî prefer lossless.** We target the **middle** of the
   quality‚Üîsize range (default `medium`), for **both images and video** ‚Äî not maximum-aggression that
   throws away visible quality, and not maximum quality that barely shrinks. Where a **lossless**
   re-encoding applies (e.g. recompressing a PNG, or a lossless WebP), we use it: the file won't shrink as
   much, but **no quality is lost**. Tools are always invoked with an **explicit numeric quality
   parameter** ‚Äî we **only** use tools that let us **control the quality level across the full range** (so
   it never silently maxes out compression). This is the **HARD REQUIREMENT** spelled out in **¬ß2.1**.
3. **Never change the pixels' size.** The output MUST keep the **same aspect ratio** and the **same pixel
   resolution** as the input ‚Äî for **both images and video**. **We never downscale.** This is verified
   after encoding (¬ß5) ‚Äî a size change aborts the operation.
4. **Alpha safety first.** Before compressing we run the **alpha-channel safety check** (¬ß6). If real
   transparency would be lost, we refuse (or steer to an alpha-preserving codec).
5. **Recoverable.** Compression replaces the file via a **temp file**, and the original is moved to the
   **LFBridge trash** (recoverable) ‚Äî never hard-deleted.
6. **Audio is out of scope** for now (planned later). The engine and settings carry an audio slot, but it
   is disabled by default.

## 2. The brew tools (quality-controllable)

We only use tools where the **quality level is an input we control**. Detected on PATH at runtime; a
missing tool degrades the relevant operation with a clear "install X" message (never a silent fail).

| Media | Tool (brew) | Role | Quality control |
| --- | --- | --- | --- |
| **Video** | `ffmpeg` | transcode video | `-crf <n>` (lower = higher quality); medium ‚âà CRF 20‚Äì23 for H.264 |
| **Image** | `ImageMagick` (`magick`) | convert / re-encode, read dimensions + alpha | `-quality <n>` (‚âà 85 medium); `identify` for `%w %h` + `%[opaque]` |
| **Image (PNG)** | `oxipng` | **lossless** PNG recompression (same pixels) | lossless levels; never changes pixels |
| **Image (WebP)** | `cwebp` | WebP encode (lossy `-q` or `-lossless`) | `-q <n>` / `-lossless` |
| **Image (JPEG)** | `mozjpeg` (`cjpeg`) / `jpegoptim` | better/lossless JPEG (re)encode | `-quality <n>` / `--max=<n>` |
| **Probe** | `ffprobe` | video dimensions + `pix_fmt` (alpha detection) | n/a |

`brew install ffmpeg imagemagick oxipng webp mozjpeg jpegoptim` provides the full set. The engine picks
per file from what's installed.

Detecting installed tools (which of ffmpeg/ffprobe/magick/oxipng/cwebp/cjpeg/jpegoptim are on PATH) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `detectTools`.

## 2.1 Quality control ‚Äî the MIDDLE setting (HARD REQUIREMENT, LOCKED)

This is the **single most important compression rule** and it applies **equally to IMAGE and VIDEO**:

1. **Same pixel resolution.** Every compressed output keeps the **same aspect ratio and the same pixel
   resolution** as the input ‚Äî for **both images and video**. We **never downscale** (verified after
   encoding, ¬ß5). Compression buys smaller files by spending bits more efficiently, **not** by throwing
   away pixels.
2. **A modern algorithm with a controllable quality‚Üîsize range.** For each media type we use a **modern,
   high-efficiency** codec/encoder that exposes a **continuous quality knob** ‚Äî one end = **highest
   quality / larger file**, the other end = **maximum compression / visibly degraded**. We do **not** use
   any tool that only has a single "make it as small as possible" mode.
3. **We deliberately pick the MIDDLE of that range.** The default quality is **`medium`** (config default,
   never the extreme). At medium we keep **nearly all** the perceptible quality and still get a **much
   smaller** file ‚Äî we give up only a **minuscule** amount of quality in exchange for a large size win.
   `high` and `low` (and `lossless`) exist for the user who wants to slide toward either end, but the
   product default sits in the middle.

### 2.1.1 The RISK this requirement guards against (why the tool choice matters)

The failure mode we are explicitly protecting against: **a compression tool that has no middle ‚Äî it only
races to the tiniest possible file and sacrifices a lot of quality**, with no dial to trade a little size
back for a lot of quality. Such a tool is **disqualified**. Every tool we drive is chosen **precisely
because it takes an explicit numeric quality parameter** that spans the full range, so the engine can pin
it to the middle. The tools are **always** invoked **with that parameter set** ‚Äî the quality is **never
left to the tool's default and never maxed out silently**.

### 2.1.2 The exact knob, range, and MIDDLE value per media type

| Media | Tool / encoder | The range knob | Scale (best-quality ‚Üî smallest) | MIDDLE (`medium`, the default) |
| --- | --- | --- | --- | --- |
| **Video** | `ffmpeg` ‚Äî `libx264` (H.264), `libx265` (HEVC), `libaom-av1` (AV1) | **`-crf <n>`** (Constant Rate Factor) | `0` = lossless ‚Ä¶ `51` = worst; **lower = higher quality/bigger** | **CRF 21** for H.264/AV1, **CRF 24** for HEVC |
| **Image (lossy)** | ImageMagick `-quality <n>` / `cwebp -q <n>` (JPEG / WebP / JPEG 2000) | **`-quality` / `-q`** | `0` = smallest/worst ‚Ä¶ `100` = best/biggest | **quality 85** |
| **Image (lossless)** | `oxipng` (PNG kept as PNG) | optimization **level** only | n/a ‚Äî pixels never change | lossless (no quality given up) |

Mapping of the four quality steps to concrete tool values (so `medium` is always the middle, not an
extreme):

* **Video CRF** ‚Äî `lossless ‚Üí 0`, `high ‚Üí base‚àí3`, **`medium ‚Üí base`**, `low ‚Üí base+5`, where `base = 24`
  for HEVC and `base = 21` for H.264 / AV1.
* **Image quality** ‚Äî `lossless ‚Üí 100`, `high ‚Üí 92`, **`medium ‚Üí 85`**, `low ‚Üí 70`.

**The mechanism that guarantees the middle** is two things together: (a) the config default
`quality: "medium"` for both `images` and `video`, and (b) the two pure mapping functions that translate
`medium` into the concrete mid-range number. If either the default or a mapping ever drifted to an
extreme, the middle guarantee would break ‚Äî so both are LOCKED here.

Choosing the concrete quality value for the chosen tool is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî
`videoCrf()` (CRF for H.264/HEVC/AV1) and `jpegQuality()` (0‚Äì100 for JPEG/WebP/JPEG 2000). The knob is
then passed to the actual command in `imageCommand()` (`-quality` / `cwebp -q`) and in `compressFile()`'s
ffmpeg build (`-crf`), both of which **omit any `-vf scale` / `-s` / resize** so resolution is preserved.
The `medium` default itself lives in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` ‚Äî the `compression.images.quality`
and `compression.video.quality` defaults.

## 3. What we do per file (the plan)

The engine builds a **plan** for a file from its kind + the user's codec preferences (¬ß7), then runs it.

### 3.1 Images

* **PNG / BMP / TIFF / GIF (lossless source):**
  * If the user's quality is **lossless**, or the file stays PNG ‚Üí **`oxipng`** lossless recompression
    (same pixels, smaller file).
  * Otherwise **convert to a compressible format** ‚Äî normally **PNG ‚Üí JPEG** at medium quality. If the
    user prefers/allows it and it is genuinely better, **JPEG 2000** (`.jp2`) may be used instead; a user
    can **deny JPEG 2000** in settings and it is then never chosen.
  * **Alpha guard:** JPEG has **no alpha**. If a PNG's alpha is actually used (¬ß6), we do **not** convert
    to JPEG ‚Äî we either keep PNG (lossless recompress) or, when the user allows it, target **lossless
    WebP** (which preserves alpha).
* **JPEG / WebP (already lossy):** re-encode at medium quality with `mozjpeg`/`cwebp` only when it yields a
  real gain; otherwise report **"already compressed ‚Äî no gain"** and change nothing.
* **HEIC / HEIF / AVIF (Apple photos & efficient formats): a COMPATIBILITY conversion ‚Äî normally ‚Üí JPEG.**
  These are byte-efficient (HEVC/AV1-coded) but **poorly supported** by the sites people post to, so the
  default is to **convert to JPEG** for universal compatibility (governed by the `convert_types` setting
  and the per-extension checkbox ‚Äî [images.mdx ¬ß2](./images.mdx#2-the-two-settings-settings--compression)).
  Two things make this different from a normal image compress:
  * **The container trap ‚Äî extract the PRIMARY still (LOCKED).** A HEIC is a container that can hold a
    primary image, thumbnails, depth/auxiliary images, and (for Live Photos) a motion clip. We decode
    **only the primary still** (the `pitm` item) ‚Äî never a thumbnail, an aux/depth image, or a frame from
    the motion video / its lead-in frames. Full rule + tooling in
    [images.mdx ¬ß4.1](./images.mdx#41-the-heic-container-trap--extract-the-primary-still-not-a-previewmotion-frame-locked).
  * **Exempt from the size-gain guard (¬ß5).** JPEG is *less* efficient than HEVC, so HEIC ‚Üí JPEG usually
    produces a **larger** file. Because it is a **compatibility conversion**, not a size reduction, it is
    **not blocked** by the "output must be smaller" rule (like the H.264 compatibility convert, ¬ß8.1) ‚Äî the
    resolution-preserving and alpha-safety invariants still apply. The alpha guard steers a HEIC with
    **used** transparency to lossless WebP instead of JPEG.

The full **file-type conversion policy** (the two settings, the per-extension scope, and the HEIC
handling) is specified in its own part spec: **[images.mdx](./images.mdx)**. This engine honors it.

### 3.2 Video

* **`ffmpeg`** transcode to the user's preferred codec (default **H.264**, social-media-safe ‚Äî ¬ß7) at a
  **medium CRF**, **preserving resolution** (no `-vf scale`, no `-s`). Audio stream copied through.
* **Alpha guard:** if the source video carries a **used** alpha channel (¬ß6.2) and the target codec does
  **not** preserve alpha, we refuse ‚Äî or, if the user's preferences include an alpha-preserving codec, we
  steer to it.

Building the per-file plan (source kind ‚Üí tool + target codec + quality, honoring allow/deny prefs, the
`convert_types` / `skip_exts` conversion settings, and the HEIC-family ‚Üí JPEG compatibility conversion with
its `formatConvert` size-gain exemption) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `pickImageTarget()` (images) and `pickVideoTarget()` (video), assembled by `checkFile()`. HEIC/HEIF/AVIF are treated as convertible (not "already done") in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/fs/badges.ts` ‚Äî `compressInfo()` / `compressBadge()`.

## 5. The resolution-preserving verification (LOCKED)

After encoding to the temp file and **before** replacing the original, the engine **re-measures the
output's width √ó height** and compares to the input:

* **Equal** ‚Üí proceed (move original to trash, move temp into place).
* **Different** ‚Üí **abort**: delete the temp, leave the original untouched, and report
  "compression changed the resolution ‚Äî refused". We never ship a downscaled file.

We also require the output to be **smaller** than the input; if it isn't, we keep the original and report
**"no gain"**. **One deliberate exception:** a **compatibility conversion** ‚Äî HEIC/HEIF/AVIF ‚Üí JPEG (¬ß3.1)
or the forced H.264 convert (¬ß8.1) ‚Äî is **exempt from the size-gain check**, because its purpose is
universal playback/compatibility, not shrinkage, and it may legitimately grow the file. The
resolution-preserving and alpha-safety invariants are **never** waived. The exemption is carried by a
`formatConvert` flag on the plan.

Verifying the output keeps the same width √ó height (and is smaller) before any replacement is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî the resolution + size-gain checks inside `compressFile()` (dimensions re-measured via `videoInfo()` / `imageDims()`, i.e. `ffprobe` / `magick identify`), gated on the `preserve_resolution` setting.

## 6. The alpha-channel safety check (LOCKED)

Before **every** compression we check whether real **transparency** would be lost.

### 6.1 Images (PNG etc.)

* Having an alpha *channel* is not the same as *using* it. We check whether the alpha is **actually
  used** ‚Äî i.e. any pixel is non-opaque. With ImageMagick: `magick identify -format '%[opaque]'` returns
  `True` when the image is fully opaque (alpha unused) and `False` when transparency is present.
* If alpha **is** used and the target format has **no alpha** (JPEG / JPEG 2000), we **do not** convert ‚Äî
  we keep PNG (lossless recompress) or target an alpha-preserving format (lossless WebP) when allowed, and
  **warn the user** that a JPEG conversion would lose transparency.
* If alpha is **not** used, converting PNG ‚Üí JPEG is safe.
* If we **cannot determine** alpha usage (tool missing), we treat it as **possibly used** and refuse the
  lossy-to-no-alpha conversion ‚Äî fail safe.

### 6.2 Video (which formats carry alpha)

Most delivery codecs have **no** alpha; a few do. We check the source's pixel format (`ffprobe` `pix_fmt`
‚Äî an alpha format looks like `yuva420p`, `rgba`, `bgra`, `argb`‚Ä¶) and the target codec's capability:

* **Alpha-capable (may carry used transparency):** **ProRes 4444 / 4444 XQ** (in MOV), **QuickTime
  Animation (RLE)**, **QuickTime PNG**, **VP8 / VP9 with alpha** (in WebM), **FFV1**, **GoPro CineForm**,
  uncompressed RGBA. (Verified 2026; a web check can refresh this list.)
* **No alpha (typical delivery, social-media-safe):** **H.264 / AVC**, **H.265 / HEVC** (main profiles),
  **MPEG-4**, most **MP4**. **AV1** alpha support is niche/inconsistent ‚Äî we treat it as no-alpha for
  safety.
* **Rule:** if the source `pix_fmt` indicates a **used** alpha channel and the target codec is in the
  *no-alpha* set, **refuse** ‚Äî unless the user's preferences select an alpha-preserving target (e.g. VP9
  alpha in WebM), in which case we use that.

Running the alpha safety check (PNG `%[opaque]`; video `pix_fmt` vs. target codec alpha capability) before any compression is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `checkFile()`, using `imageAlphaUsed()` (image `%[opaque]`) and `pixFmtHasAlpha()` (video `pix_fmt`).

## 7. Codec preference settings (per media type)

[Settings](./settings.mdx) carries a **Compression** section letting the user say, **per media type**
(images / video / **audio** ‚Äî audio disabled for now), which codecs they **want** and which they **never
want**. The reason to deny a technically-best codec is often **social-media compatibility**: the newest,
smallest codec may not be supported by the sites the user posts to.

Settings, per media type:

* **Enabled** ‚Äî may this media type be compressed at all.
* **Quality** ‚Äî `low | medium | high | lossless` (default **medium**; prefer lossless where it applies).
* **Prefer** ‚Äî ordered list of target codecs; the first one that is **allowed** and **available** wins.
* **Deny** ‚Äî codecs the user never wants chosen (e.g. **JPEG 2000** for images, **AV1** for video).
* **Convert file types** (`convert_types`, default **ON**) ‚Äî may a compress **change the format** to a
  better compressible / more-compatible target (HEIC ‚Üí JPEG, PNG ‚Üí JPEG)? Off = format-preserving only.
  Specified in [images.mdx ¬ß2.1](./images.mdx#21-convert-file-types-when-compressing--master-toggle-default-on).
* **Compress all file types** (`skip_exts`, default **empty = all in**) ‚Äî a per-extension **opt-OUT** list;
  every recognized extension is in scope unless the user unchecks it. Specified in
  [images.mdx ¬ß2.2](./images.mdx#22-compress-all-file-types--scope-default-on-with-a-per-extension-opt-out-list).

Good defaults (chosen for **social-media compatibility**):

| Media | Quality | Prefer (in order) | Deny (default) |
| --- | --- | --- | --- |
| **Images** | medium | `jpeg`, `webp` (lossless WebP for alpha) | `jpeg2000` |
| **Video** | medium | `h264`, `hevc` | `av1` |
| **Audio** (disabled) | medium | `aac` | ‚Äî |

Global toggles: **preserve resolution** (LOCKED on ‚Äî ¬ß5) and **replace-original-to-trash** (recoverable ‚Äî
on).

Reading/writing the per-media codec allow/deny + quality preferences is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.router.ts` ‚Äî `GET/PATCH /api/compress/settings` (persisted in the app `config.yaml` `compression` block).

## 8. Running a tool (temp file ‚Üí replace)

brew tools transcode to a **new file**, so the flow per file is:

1. Build the plan (¬ß3) and run the **alpha check** (¬ß6) ‚Äî abort with a clear reason if unsafe.
2. Transcode to a **temp file** (under the state dir's temp area).
3. **Verify** output resolution == input and output < input (¬ß5) ‚Äî abort (delete temp) otherwise.
4. Move the **original** to the **LFBridge trash** (recoverable), then move the **temp** into the final
   path. When the format changes (PNG ‚Üí JPEG), the final path takes the new extension; when it stays the
   same format, the path is unchanged.
5. Record the result in **two** places:
   * the file's tracking entry ‚Äî the **compressed** and **uncompressed** hashes + sizes
     ([charter per-file tracking](../CLAUDE.md); [storages.mdx ¬ß4.1](./storages.mdx));
   * a **compression record** in the storage's Syncable Data Location ‚Äî
     `.lfbridge/analysis/<relpath>/compression.yaml` ‚Äî capturing the **original filename + extension**
     and the **size before / after**
     ([syncable_data_location.mdx ¬ß4.3](./syncable_data_location.mdx#43-compression-records--original-identity--beforeafter)).
     Because this record lives in the SDL, it **travels**: every other computer that carries the storage
     learns the file was compressed, from what, and by how much, and never re-compresses it.

Running the tool to a temp file, verifying, and doing the recoverable replace is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `compressFile` (single) / `compressBatch` (many); the travelling compression record is written via `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/analysis.service.ts` ‚Äî `writeCompressionRecord()` (new).

### 8.1 Forced-codec convert (browser/upload compatibility)

The same engine also powers the media viewer's **Convert to H.264 (MP4)‚Ä¶** offer ‚Äî the fix for a video
whose codec the browser can't decode ([codecs.mdx ¬ß9](./codecs.mdx)). It calls `compressFile` with an
optional **forced video codec**, which overrides the user's preferred-codec list and pins the output to
**H.264** ‚Äî so the result plays in every browser AND uploads cleanly to TikTok/X/YouTube/Instagram (the
universal-safe target, [codecs.mdx ¬ß5](./codecs.mdx)). It is otherwise an ordinary compress run
(explicit-click, confirm-gated, recoverable replace). Honoring the forced codec is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî `compressFile(input, forceVideoCodec)` / `pickVideoTarget` (the `force` parameter), routed by `compression.router.ts` `POST /file` (optional `videoCodec`).

### 8.2 Per-run originals disposition (bulk Compress inside)

A **one-off** compress (¬ß1.5 / ¬ß8) always moves the original to the **recoverable LFBridge trash** ‚Äî the
global `compression.replace_original_to_trash` default (¬ß7). The bulk **Compress Inside** dialog
([compress_inside.mdx](./compress_inside.mdx)) deliberately differs: it surfaces disposition as an
**explicit per-run choice** and **defaults it to HARD DELETE** ‚Äî a reconciled contradiction (the user asked
for exactly that for a bulk run; the one-off default is untouched, [compress_inside.mdx ¬ß4.1](./compress_inside.mdx)).

The engine honors a per-call `deleteOriginal: "hard" | "trash"` override (`CompressFileOpts`) that **wins
over the global default** ‚Äî `"hard"` ‚Üí `fs.unlinkSync`, `"trash"` ‚Üí `trashOriginal`; when omitted it falls
back to the global setting. Disposal still runs **only after** verify + size-gain, so a file that errors
part-way keeps its original either way (¬ß5). And because a bulk run may be hundreds of files, the heavy
transcode now runs via an **async** `spawn` so a background batch **never blocks the event loop** (quick
probes stay synchronous). Both are in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî
`compressFile()` (`const disposition = o.deleteOriginal ?? (settings.replaceOriginalToTrash ? "trash" : "hard")`)
and `runAsync()` (the `child_process.spawn` runner for the heavy tool).

### 8.3 Parallelism & per-job thread-capping (LOCKED)

Compression is the app's heaviest bulk work, so it draws the canonical **Core Budget**
([parallelization.mdx](./parallelization.mdx)) ‚Äî but the engine itself stays deliberately simple: it
compresses **one file at a time** and does **not** decide how parallel to be. The fan-out lives one level
up, in the queue that drives it ([job_queue.mdx ¬ß3](./job_queue.mdx)). What the engine owns is the **per-job
thread cap** ‚Äî the knob that lets the queue pack many jobs into the budget without oversubscribing the box.

* **A one-off compress uses all cores (no cap).** When `compressFile` is called for a **single** file ‚Äî the
  media viewer's **Compress** button (¬ß4), the **Convert to H.264** offer ([codecs.mdx ¬ß9](./codecs.mdx)) ‚Äî
  it is **not** a batch. It passes **no** thread cap, so ffmpeg/oxipng/cwebp use their **own all-core
  default** and that one interactive file finishes as fast as the machine allows. This is the ¬ß2 corollary of
  the parallelization policy ([parallelization.mdx ¬ß2](./parallelization.mdx)).
* **A batch caps each job so the queue can fan WIDE/NARROW.** When the background queue drains a **"Compress
  inside"** run over **many** files ([compress_inside.mdx](./compress_inside.mdx)), each `compress` task is
  **thread-capped** so `concurrentJobs √ó threadsPerJob ‚âà budget`, never `cores¬≤`
  ([parallelization.mdx ¬ß2](./parallelization.mdx), LOCKED). The two media kinds fan **opposite** directions
  because their tools have opposite threading profiles:
  * **Image jobs run 1-thread ‚Äî fan WIDE.** oxipng/cwebp/mozjpeg/ImageMagick are ~single-threaded per file,
    so the queue runs **`budget` of them at once**, one thread each ‚Äî one job per core saturates the box.
  * **Video jobs get a small cap ‚Äî fan NARROW.** ffmpeg is internally multi-threaded, so the queue runs
    **`floor(budget / videoThreads)`** jobs and passes each a **`videoThreads`** cap (default 4). A few
    capped jobs fill the budget; without the cap, N videos would each grab every core and thrash.

The mechanism is a single optional field the queue sets and the engine plumbs into whichever tool it runs:

* **`CompressFileOpts.threads`** ‚Äî the per-job cap. **Omitted** for a one-off (tool uses its all-core
  default); **set** by the queue for a batched job. It maps per tool: **ffmpeg `-threads <n>`**, **oxipng
  `--threads <n>`**, **cwebp** multithread **on/off** (`-mt` when `n > 1`, single-threaded otherwise), and
  ImageMagick **`-limit thread <n>`**. Because the engine only forwards the number, the "how parallel"
  decision stays in **one** place (the queue's live Core Budget), never hardcoded here.

Accepting and threading the per-job cap into each tool is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî
`CompressFileOpts.threads` (the optional cap), applied in `compressFile()`'s ffmpeg build (`-threads`) and in
`imageCommand()` (oxipng `--threads`, `cwebp` `-mt`, `magick -limit thread`); omitted ‚Üí the tool's all-core
default. The queue that supplies the cap per media bucket is `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/jobqueue/jobqueue.service.ts` ‚Äî `capFor()` / `bucketOf()` drawing `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts` ‚Äî `coreBudget()`.

### 8.4 The "already-compressed" marker ‚Äî durable, in-file, mesh-wide skip (LOCKED)

**The problem this solves.** A user points **Compress inside** at a huge, deep tree (e.g. **18,000 files**)
and the run may take **several passes** ‚Äî a reboot mid-run, a machine that only got halfway, or simply
re-running it later. On the **second** pass, and on **every other computer in the mesh**, files we already
compressed must be **skipped instantly**, not re-encoded. Today they are **not**: the only real "already
compressed" guard is the **post-transcode size check** (¬ß3.1/¬ß8) ‚Äî the engine runs a **full ffmpeg
transcode** and only *then* discovers "no gain". That is exactly the wasted work the user is seeing, and it
repeats on **each** machine because there is no durable, travelling "we did this" signal the walk can read
**before** deciding to compress.

We fix it with an **in-file marker** the engine reads **before** any transcode.

**The IN-FILE marker (authoritative, travels with the bytes). LOCKED ‚Äî IMPLEMENTED.**
When the engine compresses a file, it stamps a small tag **into the output's own container metadata** ‚Äî the
standard **`comment`** field, which nothing else uses ‚Äî recording that **LargeFileBridge already compressed
this**. This is the layer the user asked for by name: *"find a property in an MP4 file or other video files
or JPG files that are unused and we are allowed to use."* Its whole virtue is that it **rides with the
file's bytes**: when the compressed file syncs to another computer over IPFS
([ipfs.mdx](../knowledge/ipfs.mdx)) the marker arrives **inside** it, so **every** machine ‚Äî regardless of
storage type ‚Äî reads the same "already compressed" truth from the file itself. It does **not** depend on any
sidecar syncing, which is why it works even for `repo` storages whose `.lfbridge/` is git-ignored and never
travels ([storage_local.mdx](./storage_local.mdx), `repo` storages).

* **No new tool ‚Äî written inline by the encoder we already run (LOCKED).** Because every compress already
  re-encodes the file, the marker is set **in that same pass**, for free: **video** via
  **ffmpeg `-metadata comment=‚Ä¶`** (into the mp4 `moov/udta`), **images** via **ImageMagick
  `-set comment ‚Ä¶`** (into the JPEG `COM` / PNG `tEXt`). No `exiftool`, no extra remux/rewrite pass. (An
  optional `exiftool` could later broaden coverage to formats that skip the ffmpeg/ImageMagick path ‚Äî e.g.
  a WebP produced by `cwebp`, or a PNG by `oxipng --strip`, both of which drop the comment ‚Äî but it is **not
  required** and is not a dependency.)
* **The payload.** `comment = "LFBcompressed;v1;<codec>"`:
  * `LFBcompressed;` ‚Äî the **presence prefix** (the whole skip test is "does the comment start with this").
  * `v1` ‚Äî the **marker version**, so a future re-tuned engine can invalidate old marks by bumping it
    (a `v1` mark is treated as **absent** once the engine reads `v2`, so those files re-sweep).
  * `<codec>` ‚Äî what it was compressed to (e.g. `h264`, `jpeg`), for the read-out (¬ß9).
* **The read (skip decision) ‚Äî enforced in the engine, per file.** At the top of a compress, the engine
  reads the file's `comment` (**ffprobe** for video, **`magick identify -format %c`** for images ‚Äî the same
  fast probes used for dimensions); if it starts with `LFBcompressed;` the file is **skipped immediately**,
  status `skipped`, reason **"already compressed (marker)"**, with **no transcode and no size probe**. This
  runs in **`compressFile`**, so **both** a one-off compress and **every file of a bulk batch** honor it.
* **Why per-file at compress time, not an up-front walk scan (LOCKED).** Reading a container tag on 18,000
  files up front would itself be slow, so the marker is checked when each file **reaches the front of the
  queue** ‚Äî the fast probe *replaces* the full transcode for that file. A re-run of an already-compressed
  tree therefore becomes **"probe-and-skip per file"** (tens of ms each) instead of **"re-encode per file"**
  (seconds each), and a peer that already holds the compressed files skips them the same way ‚Äî which is what
  the user wants across passes **and** across the mesh.

**The extension heuristic still runs first.** The cheapest test is unchanged: `compressInfo` (¬ß8 /
[scan.mdx ¬ß4](./scan.mdx)) drops a non-media or already-lossy **extension** with zero I/O, so a re-encoded
`.jpg`/`.webp` is skipped by extension **before** the marker is even read. The marker is what catches the
cases the extension can't ‚Äî a **video** (an `.mp4` always *looks* compressible by extension) and a **PNG
kept as PNG** ‚Äî on re-runs and on peers.

**The travelling `.lfbridge` record (fast local cache) ‚Äî future optimization.** The record already written
today, `.lfbridge/analysis/<relpath>/compression.yaml` (`writeCompressionRecord`, ¬ß8), is still **written
but not yet read**. Wiring a `readCompressionRecord()` into `walkCompressible()` (keyed by relpath +
[fingerprint](./storage_local.mdx#93-repo-manifests-travel-via-git-the-elegant-part)) would let a bulk run
**drop known-compressed files from the batch entirely** ‚Äî skipping even the per-file probe on a same-machine
re-run ‚Äî but it is a **speed optimization on top of** the authoritative in-file marker, not a correctness
requirement. Tracked as a follow-up; the marker above already makes re-runs and peers correct.

Detecting the marker before compressing, and stamping it during the encode, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/compress/compression.service.ts` ‚Äî
`isAlreadyCompressed()` / `readMarker()` (ffprobe `format_tags=comment` for video, `magick identify %c` for
images), consulted at the top of `compressFile()` so a marked file returns `skipped` **before** any
transcode (covering one-off **and** batched runs); the marker itself is written by `markerPayload()` folded
into the **ffmpeg** args (`-metadata comment=‚Ä¶`) and the **ImageMagick** args (`-set comment ‚Ä¶`) inside
`imageCommand()`. No new tool is probed in `detectTools()`.

## 10. Acceptance criteria

1. Compression runs **only** on explicit user action ‚Äî one file, several checked files, or **every file in
   the current table filter** ‚Äî never automatically.
2. It uses **quality-controllable brew tools** (`ffmpeg`, ImageMagick, `oxipng`, `cwebp`, `mozjpeg`) at a
   **medium** quality, **preferring lossless** where it applies; a missing tool degrades with an
   "install X" message, not a silent fail.
2a. **The MIDDLE-quality requirement (¬ß2.1) holds for both images and video:** the tool exposes a full
    quality‚Üîsize range (video `-crf 0‚Ä¶51`; image `-quality 0‚Ä¶100`), and the engine pins it to the middle
    by default ‚Äî **CRF 21** (H.264/AV1) / **CRF 24** (HEVC) for video, **quality 85** for lossy images ‚Äî
    at the **same pixel resolution** as the input. A tool with no controllable middle is disqualified.
3. The output **always keeps the same aspect ratio and pixel resolution** (verified after encoding; a size
   change aborts). We **never downscale**.
4. Before compressing, the **alpha-channel safety check** runs: a **used** PNG alpha blocks JPEG/JPEG-2000
   conversion (keep PNG / lossless WebP); a **used** video alpha blocks a no-alpha target codec ‚Äî unless
   the chosen codec preserves alpha. Undeterminable alpha fails safe (refuse).
5. **PNG ‚Üí JPEG** by default; **JPEG 2000** only when better **and** not denied; the user can **deny**
   specific codecs **per media type** (images / video / audio) in Settings, with **social-media-safe
   defaults** (deny `jpeg2000` / `av1`). **Audio is not compressed** (disabled) for now.
5a. **File-type conversion (images.mdx):** two per-media settings both **default ON** ‚Äî `convert_types`
    (may change format on compress) and `skip_exts` (per-extension opt-OUT; empty = all in). **HEIC/HEIF/
    AVIF ‚Üí JPEG** is a **compatibility conversion**: it decodes the **primary still** only (never a
    thumbnail / depth-aux / Live-Photo motion frame ‚Äî [images.mdx ¬ß4.1](./images.mdx)), keeps the **same
    pixel resolution**, and is **exempt from the size-gain guard** (¬ß5). With `convert_types` off,
    compression is format-preserving.
6. Tools run via a **temp file**; the original is moved to the **LFBridge trash** (recoverable); tracking
   records the compressed + uncompressed hashes/sizes, **and** a travelling **compression record**
   (`.lfbridge/analysis/<relpath>/compression.yaml`) captures the original name/extension + size
   before/after. The authoritative **in-file marker** (¬ß8.4, implemented) is what makes other computers never
   re-compress; reading this record as a local cache to prune the batch up-front is a tracked follow-up
   ([syncable_data_location.mdx ¬ß4.3](./syncable_data_location.mdx)).
7. The **View-one-file** page shows the file's **codec** and **compressed/uncompressed** state.
8. **Parallelism (¬ß8.3, LOCKED):** the engine compresses **one file at a time** and takes an optional
   **`CompressFileOpts.threads`** cap. A **one-off** compress **omits** it (the tool uses its all-core
   default); a **batched** job driven by the queue is **capped** so `concurrentJobs √ó threadsPerJob ‚âà budget`,
   never `cores¬≤` ‚Äî **image** jobs run **1-thread** (fan wide, `budget` at once), **video** jobs get a small
   cap (fan narrow). The cap maps to **ffmpeg `-threads`**, **oxipng `--threads`**, **cwebp** MT on/off, and
   **`magick -limit thread`** ([parallelization.mdx](./parallelization.mdx), [job_queue.mdx ¬ß3](./job_queue.mdx)).
9. **Already-compressed skip (¬ß8.4, LOCKED ‚Äî IMPLEMENTED):** a file we already compressed is **skipped
   before any transcode**, on **re-runs** and on **every other computer in the mesh**, via an **in-file
   marker** written into the output's own **`comment`** container tag (`"LFBcompressed;v1;<codec>"`). It is
   written **inline by the encoder we already run** ‚Äî **ffmpeg `-metadata comment=‚Ä¶`** (video) / **ImageMagick
   `-set comment ‚Ä¶`** (images), **no new tool** ‚Äî and read back with **ffprobe** / **`magick identify %c`**
   at the top of `compressFile()`, so a marked file returns `skipped ‚Äî "already compressed (marker)"` for
   **both** one-off and bulk runs. Because the marker **rides in the file's bytes over IPFS**, a peer that
   already holds the compressed file skips it too. The cheap **extension heuristic** still runs first, and
   the version tag (`v1`) lets a re-tuned engine invalidate old marks. Verified end-to-end: compress ‚Üí
   111,360‚Üí28,253 bytes; a second pass on the output ‚Üí `skipped (marker)`, no re-encode. (Reading the local
   `.lfbridge` record to prune the batch up-front is a tracked speed follow-up, not required for this.)
