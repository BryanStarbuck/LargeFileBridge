# LargeFileBridge — Web Sessions & Staleness-Triggered Auto-Sync

A user might not open LargeFileBridge (LFB) on a given computer for days. When they come back, the large
files on that machine can be **stale** — a laptop that has been closed since Tuesday has whatever its
peers pinned on Tuesday, not what changed since. This spec exists to fix exactly that: **the moment a
user starts a new web session on a computer that has not synchronized in more than 48 hours, LFB
automatically kicks off a synchronization** — quietly, in the background, so the user is brought
up-to-date without ever having to think about it or wait for it.

To decide "a user is starting a new session," we need a small, robust notion of a **web session**. This
spec defines that notion, how we persist the last five of them, and the freshness rule they drive.

> **The one rule this feature is built on.** Starting a new web session on a computer that is **> 48
> hours** since its last sync **automatically triggers a synchronization**, and that trigger is
> **non-blocking** — it is fired and forgotten on the server so the web request returns immediately and
> the UI never blocks on IPFS. Everything else here (the 4-hour idle window, the retroactive end time,
> the five-session history) exists to make that trigger fire correctly and exactly once per real return.

## 1. Vocabulary — web session vs. auth session (LOCKED)

These two are **different things** and must never be conflated.

| Term | Definition |
| --- | --- |
| **Web session** | A window of **activity** on one computer, measured from **page renders**. It **starts** on the first render after a quiet period and is considered **discontinued** once **4 hours** pass with **no render**. It is the thing this spec tracks. It has no cookie, no token, no security role — it is purely a usage window used to decide freshness. |
| **Auth session** | The **OpenAuthFederated** login ([storage.mdx §10](./storage.mdx#10-auth-session--signing-secret-runtime-not-settings)): the signed cookie + rotating Bearer that keep the user **signed in** for up to 10 months across restarts. It is bounded by a 10-month ceiling, not by 4 hours. A single auth session spans **many** web sessions. |
| **Render / activity ping** | A signal from the frontend that a page was **rendered** (app opened or route navigated). Each ping is one unit of "the user did something that drew a page." Rolls into the open web session. |
| **Idle window** | The **4-hour** span of no activity after which an open web session is considered discontinued. |
| **Stale threshold** | The **48-hour** span since the last successful sync beyond which **starting a new web session** auto-triggers a synchronization. |
| **Last synchronized** | The most recent of: the app-config `sync_process.last_run_at` (the scheduled worker's last run) **and** every repo unit's `status.yaml` `last_sync_at`. A manual "Sync now" writes the latter; the launchd worker writes both. |

The "Last synchronized" value (newest of the scheduled worker's run and every repo's `status.yaml`) is computed in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.service.ts` — `lastSyncAt()`

The web session exists **only** to answer one question well: *"Is the user starting fresh on a machine
that might be stale?"* Keep it that small.

## 2. The session lifecycle

### 2.1 Renders roll into one open session

The frontend emits an **activity ping** whenever it **renders a page** — once when the app is opened,
and again on every **route navigation** ([router.tsx](../code/packages/frontend/src/router.tsx)
route change). The backend folds every ping into a single **open** session (the newest entry whose
`ended_at` is `null`):

The browser render/navigation ping (throttled, first render always reported) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/lib/useSessionPing.ts` — `useSessionPing()` hook

* If an **open** session exists and the gap since its `last_activity_at` is **≤ 4 hours**, the ping
  **extends** it: set `last_activity_at = now`. Same session — the user is still going.
* If there is **no** open session (or the open one is already older than the idle window), the ping
  **starts a new** session and runs the §3 staleness check.

The frontend throttles pings (it need not report every keystroke — at most once per ~60 s), because the
windows we care about are measured in hours. The **first render after opening the app is always
reported**, so a fresh return is detected immediately.

Folding each ping into the open session (extend, or close-and-start-new) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.service.ts` — `recordActivity()`

The activity ping endpoint the frontend posts to is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.router.ts` — `POST /api/sessions/activity`

### 2.2 Discontinuing a session — the 4-hour idle timer, back-dated end

A session is **discontinued** once **4 hours** elapse with **no render**. Two things make this precise:

1. **A timer fires at the 4-hour mark.** Each ping (re)arms a 4-hour idle timer. If it fires with no
   intervening activity, the open session is closed. Example: last render at **3:00 p.m.**; the timer
   is armed for **7:00 p.m.**; at 7:00 p.m. nothing more has happened, so the session is closed.
2. **The end time is the LAST render, not the timer.** The session that ended in the example above is
   recorded as `ended_at = 3:00 p.m.` — the time of the last thing the user actually did — **not**
   7:00 p.m., the time the timer happened to fire. The 4-hour timer is only *how we notice*; the
   *recorded* end is always the last activity.

The close is also computed **lazily and idempotently**: if the server was down when the timer would
have fired, the **next** ping observes that the open session's `last_activity_at` is more than 4 hours
old, closes it (still back-dated to that `last_activity_at`), and starts a new session. So the timer is
a best-effort convenience; the authoritative rule is "an open session whose last activity is > 4 h ago
is discontinued, ended at its last activity." Once discontinued, **no further activity is ever added to
that session** — a later ping always begins a new one.

The 4-hour idle timer and the back-dated (last-render) session close are handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.service.ts` — `armIdleTimer()` / `closeIdleSession()`

## 3. The freshness rule — auto-sync on a stale return (the point of all this)

**When a new web session STARTS** (§2.1), and only then, LFB checks:

```
now − lastSynchronized > 48 hours   ?   →   kick off a synchronization
```

* `lastSynchronized` is the "Last synchronized" value from §1 (max of `sync_process.last_run_at` and
  every repo's `status.yaml.last_sync_at`). If the machine has **never** synced, treat it as
  infinitely stale → trigger.
* If stale, LFB **kicks off a synchronization** — the same whole-computer `syncAll()` the launchd
  worker runs ([storage.mdx §13](./storage.mdx), [sync_list.mdx](./sync_list.mdx)).
* The check runs **only on session START**, not on every ping — so a user actively working (many pings
  inside one session) triggers **at most one** auto-sync for that return, at the moment they arrive.

The >48h staleness check that runs only on a new session start is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.service.ts` — `recordActivity()` (via `isStale()`)

### 3.1 Non-blocking — never stall the UI

The auto-sync is **fire-and-forget**:

* The server starts `syncAll()` **without awaiting it** and returns the activity-ping response
  **immediately**. The browser gets its answer in milliseconds; the sync runs on the server afterward.
* A guard prevents **overlapping** auto-syncs: if one is already in flight (from a near-simultaneous
  ping, or still running), a second start is a no-op.
* The scheduled launchd worker and a manual "Sync now" are unaffected — this is an *additional*,
  opportunistic trigger, not a replacement. It obeys the same per-repo `synced` gating and the same
  only-our-content IPFS rules ([knowledge/ipfs.mdx §6](../knowledge/ipfs.mdx)) that every sync does.

The non-blocking, single-guarded auto-sync (fire-and-forget, never awaited by the request) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sessions/sessions.service.ts` — `triggerAutoSync()` (fires `syncAll()`)

This is the whole payoff: **a user who opens LFB on a machine that has been dark for two days is
silently brought current, with no click and no wait.**

## 4. Storage — the five-session history in `users/<email>/config.yaml`

The session history is **per-user, personal state**, so it lives in that user's config
([storage.mdx §4](./storage.mdx#4-per-logged-in-user-storage--the-per-user-requirement)), under a
top-level `sessions:` key. We keep the **last five** sessions, **newest first**; a sixth start drops the
oldest.

```yaml
# ~/T/_large_files_bridge/users/bryan@act3ai.com/config.yaml  (excerpt)
sessions:                                   # last 5 web sessions, newest first (§2)
  - started_at:       2026-07-05T15:00:00.000Z
    last_activity_at: 2026-07-05T15:42:10.000Z
    ended_at:         null                  # OPEN — inside the 4-hour idle window
  - started_at:       2026-07-04T09:03:00.000Z
    last_activity_at: 2026-07-04T11:20:00.000Z
    ended_at:         2026-07-04T11:20:00.000Z   # closed; end == last render, NOT the timer time
  - started_at:       2026-07-01T18:15:00.000Z
    last_activity_at: 2026-07-01T18:59:00.000Z
    ended_at:         2026-07-01T18:59:00.000Z
```

Per-entry fields:

| Field | Meaning |
| --- | --- |
| `started_at` | ISO timestamp of the first render that began this session. |
| `last_activity_at` | ISO timestamp of the most recent render folded into this session. |
| `ended_at` | ISO timestamp the session was discontinued — **always equal to `last_activity_at`** — or `null` while the session is still open. |

At most **one** entry may have `ended_at: null` (the current open session), and it is always index `0`.
Writes go through the store's per-file mutex + atomic write ([storage.mdx §15](./storage.mdx)); a
missing `sessions:` key defaults to an empty list (fresh user, no history yet).

## 6. Invariants (must always hold)

1. **A web session is not an auth session.** Never gate access on it, never derive identity from it.
2. **At most one open session** per user; it is always index 0; `ended_at === null` only there.
3. **`ended_at` is the last render**, never the timer's fire time.
4. **The staleness check runs only on session START**, so one return → at most one auto-sync.
5. **Auto-sync is fire-and-forget** — the activity request never awaits it; the UI never blocks.
6. **At most one auto-sync in flight**; a second trigger while one runs is a no-op.
7. **Exactly five** sessions are retained, newest first; the sixth start evicts the oldest.
8. **All timestamps are ISO-8601 UTC**, matching every other timestamp LFB writes.
