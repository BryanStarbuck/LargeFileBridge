# LargeFileBridge — Subsets

The **Subsets** page answers: *which of my videos are really just a piece of another, longer
video?* The canonical case: a **10-minute video** and a **3-minute video** where the 3-minute one
was **cut out of the middle** of the 10-minute one. The 3-minute file is a **subset**; the
10-minute file is its **superset**. The user reviews these pairs side-by-side — previews, where
the clip sits inside the parent, what differs — and then decides what to do with them (typically:
delete the clip and keep the superset, or keep both on purpose). **LFB never deletes on its own**
(charter: detect, surface, offer — never act).

**Videos only.** Unlike [Duplicates](./duplicates.mdx), images are out of scope here — "a segment
somewhere inside a longer timeline" is a video concept. This page narrows the hub's scope
([videos.mdx §2](./videos.mdx)) to video files exclusively.

This page **mirrors [duplicates.mdx](./duplicates.mdx) section-for-section** — same layout, same
table rules, same right-column mechanics, same pop-up shape, same progress model, same CSV-in-state-root
pattern — and states only its deltas, plus the parts that are genuinely its own: the
**segment-containment fingerprint library** (§7, web-researched) and the **subset calc engine +
`subsets.csv`** (§8–§9). Exact SHA-256 matching plays no role here — a subset is by definition
**not** byte-identical to its superset; everything rides on fingerprints.

> **HONEST STATUS — BUILT (2026-07-22).** The page, pop-up, engine, and CSV exist; code pointers
> name the **actual** homes. Implementation deltas from the planned pointers: the router is the one
> shared `videos.router.ts`, the pop-up is the shared `StartScanModal.tsx`, and the engine lives in

## 2. Vocabulary (LOCKED)

| Term | Definition |
| --- | --- |
| **Superset** | The longer video that **contains** the other's content — exactly **one per group**. |
| **Subset** | A video whose content is a **contiguous segment** of the superset's timeline (any position — start, middle, end). ≥ 1 per group. |
| **Subset group** | One superset plus every detected subset of it, identified by a `subset_group` id in `subsets.csv` (§9). |
| **Member** | Any file in a group (the superset or a subset) — one CSV row, one table row, one preview block. |
| **Containment range** | Where the subset's content sits inside the superset: `{start}–{end}` in superset time, derived from the signature match offsets (§8). |
| **Common / differing attribute** | Same meaning and same print rules as [duplicates.mdx §2/§4](./duplicates.mdx). Duration **always differs** in a subset group by definition. |

A video can appear in more than one group (a clip cut from a clip); groups are kept flat — each
`subset_group` has one superset, and a middle-length video may be the superset of one group and a
subset in another.

## 7. The fingerprint library — web-researched evaluation & choice (LOCKED)

Detecting "video A is a segment of video B" needs a fingerprint that matches **subsequences with
offsets**, not just whole files — a harder problem than duplicate matching, and one the duplicate
engine's same-content thresholds don't answer. The same **four hard requirements** as the
duplicate libraries apply ([duplicates.mdx §7.1](./duplicates.mdx)): **open-source and
embeddable** in our TypeScript/Node app; a **storable fingerprint block** we can reference from
the CSV; a **compare/lookup function that takes previously stored fingerprints back in** (never
re-reading the media to compare); and **no phone-home** — no network to do the work or to report
anything.

**Per media type:** the choice below is **for videos, to detect subsets** — the only choice this
page needs. **Images: not applicable** — this page is videos-only (§ intro); there is no image
subset detection, and image *duplicate* matching is [duplicates.mdx §7.4](./duplicates.mdx)'s
territory.

### 7.1 Options compared

| Option | Containment (clip-inside-longer-video)? | Offsets reported? | Robust to codec/size/resolution change? | Local, no network | Integration cost |
| --- | --- | --- | --- | --- | --- |
| **MPEG-7 Video Signature — ffmpeg `signature` filter** (ISO/IEC 15938-3) | ✅ **designed for exactly this** — partial-match lookup of one sequence inside another | ✅ matching frame ranges / where the match starts | ✅ (standardized fine + coarse signatures, resistant to lossy re-encode) | ✅ (a filter inside the **ffmpeg we already bundle**) | **Zero new dependencies** — `-filter_complex signature=detectmode=full` on two inputs; export per-video signatures as binary/XML |
| **vPDQ** (Meta ThreatExchange) | ✅ matches subsequences/clips within videos (per Meta: TMK optimizes for identical videos, **vPDQ can match clips**) | ⚠️ derivable — a contiguous run of matched per-frame PDQ hashes gives the offset | ✅ (inherits PDQ transform-robustness) | ✅ | Low — the duplicate engine **already computes per-frame PDQ** ([perceptual_fingerprint.mdx §3](./perceptual_fingerprint.mdx)); containment = longest-common-run over Hamming-matched frames; linear O(n·Fc·Fq) comparison |
| **TMK+PDQF** (Meta) | ⛔ whole-video, fixed-length signature; optimizes for **identical same-length** videos | ⛔ | ✅ | ✅ | **Disqualified** — cannot see a clip inside a longer video |
| **Whole-video hashes** (videohash et al.) | ⛔ collapse the whole timeline into one hash | ⛔ | ⚠️ | ✅ | **Disqualified** — same reason |

Sources: [FFmpeg signature filter source (MPEG-7 video signature)](https://www.ffmpeg.org/doxygen/3.4/vf__signature_8c_source.html) ·
[American Archive — Adventures in Perceptual Hashing (ffmpeg signature locating a matched segment ~80 s into a longer video)](https://blog.americanarchive.org/2017/04/20/adventures-in-perceptual-hashing/) ·
[Meta ThreatExchange README (TMK = identical videos; vPDQ = clips/subsequences)](https://github.com/facebook/ThreatExchange/blob/main/README.md) ·
[vPDQ](https://github.com/facebook/ThreatExchange/tree/main/vpdq) ·
[Tech Coalition — PHVSpec benchmark of video perceptual hash systems](https://technologycoalition.org/wp-content/uploads/Tech-Coalition-Video-Hash-Benchmark-Paper.pdf).

### 7.2 The choice (LOCKED)

* **Primary: MPEG-7 Video Signature via the bundled ffmpeg `signature` filter.** It is the ISO
  standard built precisely for "is this sequence contained in that one, and where" — fine
  per-frame signatures plus coarse segment signatures, with a two-input detect mode that reports
  the matching ranges. It costs **zero new dependencies** (the same bundled ffmpeg the
  fingerprint and compression paths already shell out to), is fully offline, and its per-video
  signature exports to a compact binary file we cache per video.
* **Cross-check: vPDQ contiguous-frame-run containment.** We already hold per-frame PDQ hashes
  for every scanned video (the duplicate engine, [duplicates.mdx §7–§8](./duplicates.mdx)). A
  candidate pair from the MPEG-7 pass is confirmed when the subset's frame sequence Hamming-matches
  a **contiguous run** inside the superset's frame sequence — and the run's position independently
  corroborates the reported offset. Two independent algorithms agreeing is what lets the UI state
  a containment range with confidence.
* **TMK+PDQF and whole-video hashes are disqualified** — they only answer "are these the same
  whole video," which is the Duplicates page's question, not this one.
* **Consequence for storage:** the fingerprint value this page records is **different from the
  duplicates one** — a **per-video MPEG-7 signature file reference** (plus the reused `.vpdq`
  frame list), not a single PDQ hash (§9).

### 7.3 How to use the algorithm

* **Signature generation (per video, once):** run the bundled ffmpeg with the `signature` filter
  over the file and export the signature to a binary file —
  `ffmpeg -i <video> -vf signature=format=binary:filename=<out>.mpeg7sig -f null -` — producing
  fine per-frame signatures plus coarse signatures over ~90-frame segments. Cache the file (§7.4);
  never recompute unless the video's `sha256` changed.
* **Containment lookup (per candidate pair):** run the filter in its two-input detect mode over
  the **stored** signatures (`signature=detectmode=full` semantics, wrapped by
  `findContainment()`), which reports whether the shorter sequence appears in the longer one and
  the **matching frame ranges** — converted to `start_offset_s`/`end_offset_s` in superset time.
* **Cross-check:** align the subset's stored `.vpdq` frame lines against the superset's, looking
  for a **contiguous Hamming-matched run**; the run's position must corroborate the MPEG-7 offset.

### 7.4 Where the fingerprint values are stored

* **`~/T/_large_files_bridge/videos/signatures/<sha256>.mpeg7sig`** — the cached per-video MPEG-7
  signature (binary; keyed by content hash, so a re-encode gets a fresh signature).
* The **`fingerprint` column of `subsets.csv`** carries the **relative reference** to that
  signature file (§9) — the signature block itself is binary, so the reference (a path string) is
  what keeps the CSV text-clean; the `.vpdq` cross-check lists are plain text lines and live in
  the per-file sidecar area ([perceptual_fingerprint.mdx §5](./perceptual_fingerprint.mdx)).
* Category-B computed state: state root only, never a working repo.

### 7.5 How comparisons run later — stored values back into the library (LOCKED)

Re-grouping and future re-scans compare **stored artifacts only** — two cached `.mpeg7sig` files
into `findContainment()`, two stored `.vpdq` frame lists into the run matcher. The media bytes
are re-read only when a fingerprint is missing or its `sha256` invalidated — never to compare.

### 7.5a CLI reality — a recorded deviation from §7.5 (2026-07-22)

ffmpeg's `signature` filter **cannot load two stored `.mpeg7sig` files for comparison** — its
two-input detect mode recomputes signatures from the two **video inputs**, so the MPEG-7 pair
lookup re-decodes media. Only the **vPDQ cross-check** truly compares stored values (and it runs
**first**, as a cheap pruner, so the expensive detect only runs on plausible pairs — the confirmed
set is the same intersection §8 requires either way). The cached `.mpeg7sig` files still satisfy
the §7.4 storage contract (a durable, referenceable fingerprint artifact per video) and would plug
straight into a future signature-file comparator.
Recorded in code: `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/mpeg7-signature.service.ts` — the "CLI REALITY NOTE" header comment.

### 7.6 When to run it

Only inside the dedicated subset scan (§8): signature generation in phase 2 (missing/stale files
only), containment lookups in phase 4. Never on page load, never on a request path, never in the
15-minute/4-hour scheduleTasks.

The library boundary is one module so the algorithm can be swapped without touching the engine, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/mpeg7-signature.service.ts` — `computeSignature(path, sha256)` (ffmpeg `signature` export, sha256-keyed cache) / `findContainment(subsetPath, supersetPath, subsetDurationS)` (two-input detect, §7.5a; log parsed by the unit-tested `parseSignatureDetectLog()`) / `mpeg7Available()` (filter probe — absent filter degrades to vPDQ-only confirmations, `match_basis: vpdq`).

## 8. The calc engine — the dedicated subset scan (LOCKED)

Its **own dedicated scan** — separate from the duplicate scan, its own batch kind, its own run
stamp — run on demand from the pop-up (§5). Pipeline:

1. **Candidate set.** All **video files** LFB knows on this computer (every scanned unit —
   [scan.mdx §4](./scan.mdx)). Videos only; no images.
2. **Signature phase.** For each candidate without a cached signature (or whose content hash
   changed since it was computed): run the ffmpeg `signature` filter and cache the binary
   signature file in Local Storage (§9). One item per video in the Processing batch.
3. **Pair pruning.** Only ordered pairs `(shorter, longer)` where
   `shorter.duration ≤ 0.9 × longer.duration` are compared — equal-length same-content pairs are
   the **Duplicates** page's territory and are excluded here; members of an existing duplicate
   group are collapsed to one representative before pairing so a duplicate never masquerades as a
   subset.
4. **Match phase.** For each surviving pair, run the MPEG-7 containment lookup
   (`findContainment`); a hit yields the containment range in superset time plus a confidence
   score. **Cross-check** each hit with the vPDQ contiguous-run test (§7.2); hits failing the
   cross-check are dropped (precision over recall — a wrong "this is inside that" claim invites a
   wrong delete).
5. **Group assembly.** Confirmed hits fold into groups: one superset + all its confirmed subsets
   under one `subset_group` id.
6. **Write** `subsets.csv` atomically (temp + rename), stamp `subset_run.yaml` for the §5
   staleness clock, bump the live-refresh topic.

**Signal only.** The engine records relations; it never deletes, replaces, or alters a file. All
computation is local — ffmpeg and the PDQ matcher, no sockets (charter;
[perceptual_fingerprint.mdx §6](./perceptual_fingerprint.mdx)).

The engine is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/videos/subset-scan.service.ts` — `runSubsetScan()` (phases 1–6) and `matchPair()` (vPDQ run first as the cheap pruner, MPEG-7 detect confirms + provides offsets, corroboration overlap check — §7.5a), on `mpeg7-signature.service.ts` (§7) and `vpdq.service.ts` — `longestSharedRun()` (cross-check).

## 9. The file format — `subsets.csv` (LOCKED)

Durable output in **Local Storage** (state root — never a working repo), beside the duplicates
outputs:

```
~/T/_large_files_bridge/videos/subsets.csv
~/T/_large_files_bridge/videos/subset_run.yaml     # last-run stamp + counts (drives §5 staleness)
~/T/_large_files_bridge/videos/signatures/<sha256>.mpeg7sig   # cached per-video MPEG-7 signatures (§7.4, §8 phase 2)
```

**One row per member file**; rows sharing a `subset_group` value form one group. Exactly one row
per group carries `role: superset`. Columns:

| Column | Type | Meaning |
| --- | --- | --- |
| `subset_group` | string | **Unique id of the group** (short hash). Shared by the superset and all its subsets. |
| `full_path` | string | Absolute path of this member. |
| `role` | enum | `superset` \| `subset` — exactly one superset per group. |
| `sha256` | string | Exact content hash of this member (identity/change detection only — never the match basis here). |
| `fingerprint` | string | **This page's fingerprint is a reference, not a single hash**: the cached MPEG-7 signature file (`signatures/<sha256>.mpeg7sig`, §7.4), with the `.vpdq` frame-list sidecar reused for the cross-check ([perceptual_fingerprint.mdx §5](./perceptual_fingerprint.mdx)). |
| `match_basis` | enum | `mpeg7` (primary) \| `vpdq` (cross-check-only confirmations, if the primary ever yields none). |
| `start_offset_s` / `end_offset_s` | float | **Subset rows only:** the containment range in **superset** time — feeds the §4 "Contained at {start}–{end}" line. Empty on the superset row. |
