# LargeFileBridge — Repo Storage

A **repo storage** is the most common kind: **any Git repository on the machine** that LFBridge tracks
large files for. It is the **canonical directory-based storage** — the other three (personal / company /
community) follow the same shape.

Read the shared model first: the storage family, the `storage.yaml` schema, the tracking layout,
cloud clones, and media analysis are all defined once in **[storages.mdx](./storages.mdx)**. This
spec covers only what is **specific to a repo storage**; it does not restate the shared parts.

> **A repo storage is the only kind with a `.lfbridge/` (LOCKED).** `.lfbridge/` is a **working-repo-only**
> concept: here LFB is a **guest** in the user's own project, so everything it writes is quarantined under
> one hidden directory that is obviously not the user's content. The **dedicated LFB file repos**
> (personal / company / community — the **SDLs**) have **no `.lfbridge/` at all**; their tracking text and
> artifacts hang off the storage **root**, because a repo that exists *only* to hold LFB's files gains
> nothing from hiding them. The rule is owned by
> [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — it is keyed on the resolved
> storage **kind**, never on a path or a name, and kind `repo` is the **default**.

## 1. Anatomy of a repo storage

```
~/BGit/xyz/                 ← the repo root = the storage root ("XYZ")
├── .git/                   ← Git (small files + the Category-A content travel here)
├── .gitignore             ← lists the large files only (they pin via IPFS, not Git); .lfbridge/ is COMMITTED
├── .lfbridge/              ← COMMITTED per-storage content area — holds ONLY Category-A derived content
│   ├── <rel-dir>/<name.ext>.transcription   ← transcript — path-mirrored, ext APPENDED (storages.mdx §6)
│   ├── <rel-dir>/<name.ext>.ai_description  ← AI description — path-mirrored, ext APPENDED (storages.mdx §6)
│   └── analysis/<relpath>/visuals_by_time.yaml   ← visuals-by-time — the AI-description family (storages.mdx §6)
└── …the repo's tracked + git-ignored files…

~/T/_large_files_bridge/repos/<repoKey>/   ← LOCAL STORAGE — Category-B tracking state, NEVER in the repo
│                                             (repoKey = shortHash(resolve(repoPath)); + sync-repo mirror when enabled)
├── storage.yaml           ← descriptor: name + type: repo + repo_root (storages.mdx §3) — §2
├── repo_storage.yaml      ← repo-WIDE settings + special-file counts (single key: repo_storage) — repo_tracking_scheme.mdx §2
├── files.yaml             ← per-file fingerprint index (storages.mdx §4.1)
├── files/<path>.yaml      ← per-file <name>.yaml sidecars: identity + append-only event history — repo_tracking_scheme.mdx §3
├── history/<device>.txt   ← one UTC-timestamped audit log PER COMPUTER — repo_tracking_scheme.mdx §4
├── manifest.yaml          ← large-file → IPFS CID + pin state (storage_local.mdx §9) — Local-Storage-always; travels via sync repo
├── decisions.yaml         ← the team-SHARED per-file decision ledger (decisions.mdx) — Local-Storage-always; travels via sync repo
└── analysis/<rel>/compression.yaml   ← compression records (compression.mdx §8)
```

* The **storage root is the repo root** — there is no separate directory; a repo *is* the storage.
* The working repo's **`~/BGit/xyz/.lfbridge/`** is **committed and travels with the repo**, but holds
  **only Category-A derived content** (transcripts / AI descriptions / visuals-by-time) — the two content
  artifacts the user asked LFB to produce. Only the multi-GB media files themselves are git-ignored (they
  ride IPFS); `.lfbridge/` is **not**. This is the working repo's whole LFB footprint.
* **All Category-B tracking state** — the churny `repo_storage.yaml` / `files.yaml` / sidecars / history,
  and the union-mergeable `manifest.yaml` / `decisions.yaml` / compression records — lives in **Local
  Storage** at `~/T/_large_files_bridge/repos/<repoKey>/` **always** (never the working repo, so it can
  never merge-conflict), and is **additionally** mirrored to the owning company/Personal **sync repo**
  when the per-repo toggle is on ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx),
  [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx)).
* `storage.yaml` here is minimal — see §2.

## 2. `storage.yaml` for a repo

The repo's `storage.yaml` descriptor is **Category-B tracking state** (stable, reconstructable identity),
so it lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/storage.yaml` — **never** in
the working repo — alongside the rest of the tracking state (§1), and is mirrored to the sync repo when
enabled:

```yaml
# ~/T/_large_files_bridge/repos/<repoKey>/storage.yaml
name: "XYZ"                 # defaults to the repo folder name; user-editable
type: repo
created: 2026-05-01T10:12:00Z
repo:
  repo_root: "~/BGit/xyz"   # the Git repo this storage tracks
clones:
  google_drive: null        # or "My Drive/LFB/XYZ" once mirrored (storages.mdx §5)
  dropbox: null
```

* **`name`** defaults to the repo folder name (`xyz` → `XYZ`) and is user-editable in
  [Repo settings](./repo_settings.mdx).
* **`type: repo`** with a **`repo`** block naming the `repo_root`. A repo storage never carries a
  `company`/`community`/`personal` block.

Seeding the repo's Local-Storage `repos/<repoKey>/` tracking directory and its `storage.yaml` /
`repo_storage.yaml` when a repo is registered — and creating the working repo's committed `.lfbridge/`
for its Category-A content (it **no longer** adds `.lfbridge/` to the repo `.gitignore`, since nothing
churny is written there) — is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureRepoStorage(repoRoot)` (resolving the tracking root via `tracking-root.service.ts` — `resolveTrackingRoot()`).

## 3. What "tracking a repo" means here

A repo storage is the on-disk backing for everything the [One repo](./one_repo.mdx) screen shows:

* **Discovery** — the scanner finds every Git repo under the configured roots
  ([storage_local.mdx §7](./storage_local.mdx), [scan.mdx](./scan.mdx)); each becomes a repo storage.
* **Per-file decision** — the **two-axis** decision (Add to IPFS *and/or* Add to git ignore) per large
  file, recorded in the **team-shared** `decisions.yaml` ledger with who/when/SID
  ([decisions.mdx](./decisions.mdx)) — Local-Storage-always at `repos/<repoKey>/decisions.yaml`, never the
  working repo; its IPFS axis projects onto the frozen `sync`/`ignore`/`undecided` enum
  ([one_repo.mdx §1](./one_repo.mdx#1-the-decision-model-locked)). Because the ledger is union-merged and
  travels via the **sync repo** when enabled, the whole team shares one set of decisions and no teammate is
  re-asked about a file another already triaged. The machine-local `pin/r/<repo>/config.yaml → decisions:`
  map is now a reconciled cache of that ledger's IPFS axis, not the source of truth.
* **Per-file fingerprints** — the Local-Storage `repos/<repoKey>/files.yaml` records hash + size + created +
  modified, and the compressed/uncompressed variant hashes for media ([storages.mdx §4.1](./storages.mdx)),
  so LFB knows what changed since last pin.
* **IPFS bytes** — files marked **Add to IPFS (pin)** (the `sync` decision value) are pinned and moved between the user's computers over IPFS
  ([storage_local.mdx §9](./storage_local.mdx)); the small `manifest.yaml` lives in Local Storage and travels
  via the sync repo (not the working repo, [storage_local.mdx §9.2](./storage_local.mdx)) so another computer
  knows what to fetch.

The repos list and per-repo UI over this storage are rendered in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/repos/OneRepoPage.tsx` — `OneRepoPage` (served by `repos.router.ts`).

## 4. Media analysis + cloud clones (per repo)

* A repo storage can produce the [storages.mdx §6](./storages.mdx) **media-analysis** outputs for its
  video/image files: the **transcript** (`.lfbridge/<rel-dir>/<name.ext>.transcription`) and
  **description** (`.lfbridge/<rel-dir>/<name.ext>.ai_description`) live **inside the repo's `.lfbridge/`,
  path-mirrored, with the media's full filename and the extension APPENDED**, while
  **visuals-and-action-by-time** (`visuals_by_time.yaml`, time × visual × story) stays under
  `~/BGit/xyz/.lfbridge/analysis/<relpath>/`. These are **Category-A** content artifacts that ride along
  with the storage — because `.lfbridge/` is **committed and travels with the repo**, the
  transcript/description are committed and synced by git (no `*.transcription` / `*.ai_description`
  git-ignore nudge — retired), while the multi-GB media stays git-ignored and rides IPFS.
* The **compression record** is **Category-B tracking state**, so — unlike the content artifacts above —
  it does **not** go in the working repo's `.lfbridge/analysis/`. It lives in Local Storage at
  `~/T/_large_files_bridge/repos/<repoKey>/analysis/<relpath>/compression.yaml` (+ the sync-repo mirror
  when enabled), per [artifact_placement_policy.mdx](./artifact_placement_policy.mdx) /
  [compression.mdx §8](./compression.mdx).
* A repo storage **may also clone** to Google Drive / Dropbox ([storages.mdx §5](./storages.mdx)) — the
  same repo hierarchy mirrored to a cloud drive as a second, human-reachable copy — recorded in the
  `clones` block.

## 6. TO DO Batch for this storage (LOCKED)

A repo storage participates in the product-wide **TO DO Batch** system
([to_do_batches.mdx](./to_do_batches.mdx)). A **TO DO Batch is a per-storage bundle of recommended file
actions** — there is exactly **one** batch per repo, persisted as hierarchical YAML at
**`~/T/_large_files_bridge/_do_batches/repo_<repo-slug>_2_do.yaml`** (the `<repo-slug>` derived from the
repo root). Like every batch it is **local, disposable, and uncommitted** — it lives only in the
machine-local state root, is **never** written into the repo's `.lfbridge/`, and is **never** team-shared.
(Contrast the durable `decisions.yaml` ledger of §1 — Local-Storage-always and sync-repo-travelling: the
batch *reads* that ledger; it is not part of it.)

* **This is the richest batch — all five categories apply.** A repo is the **only** storage where
  **`git_ignore`** ("big files that aren't a good idea to check in") is meaningful, so a repo batch can
  carry **`compress_video`**, **`compress_image`**, **`git_ignore`**, **`pin`**, and **`pull_down`** — the
  full set — plus the transcription discovery below.
* **Produced from the repo's classified files + the folded decisions ledger.** The calc engine
  ([to_do_batch_calc_engine.mdx](./to_do_batch_calc_engine.mdx)) builds the batch from the repo's
  **classified special files** ([special_files.mdx](./special_files.mdx)) and then **folds in the
  team-shared decisions ledger** ([decisions.mdx](./decisions.mdx)): it **recommends only** — never writes a
  decision, never moves a byte — and **skips any file/axis the team already decided**, so a teammate's
  earlier triage is never re-asked.
* **Recalculate-and-replace each scan.** The batch is **recomputed from scratch and replaced** on every
  scan pass; it holds no durable state of its own.
* **Same model, two surfaces (see [to_do.mdx §8](./to_do.mdx#8)).** This same batch is what the per-repo
  screen surfaces as its **yellow warning banner** ([warnings.mdx](./warnings.mdx)) *and* what the To-Do
  page renders as a **slug**. The repo page and the To-Do page are two views of one underlying batch, not
  two computations. Opening the slug's chevron shows the recommendations as two-axis **decision toggles**
  pre-set to the recommended state ([decision_toggles.mdx](./decision_toggles.mdx)), which the user confirms
  and **Applies**.
* **Transcription discovery for the repo.** The on-demand **"Show what could be transcribed"** link runs
  the **What Can Be Transcribed Calc Engine** ([transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx)),
  which walks this repo for **video/audio that has no `.transcription` artifact under `.lfbridge/`** and emits a per-repo
  **transcribe batch** → a **transcribe slug** (a ~80%-width, default-checked popup: *"Transcribe N
  files"* → background jobs on [processing.mdx](./processing.mdx)).

Producing the repo batch during scan recalc, and naming/replacing its YAML, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batch.engine.ts` —
`TodoBatchCalcEngine.recalcForStorage(storage)` writing via
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batches.store.ts` —
`batchFileName(storage)` (→ `repo_<repo-slug>_2_do.yaml`); the transcribe discovery walk is
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/transcribe-scan.engine.ts`.

## 7. Merge conflicts (why tracking state never lives in the working repo)

**The single most common failure this storage is designed to prevent is a Git merge conflict on our own
tracking files.** Because a repo storage lives inside a real Git repo the user is *also* committing and
pushing across their computers, anything churny LFB writes into the working tree gets swept into the user's
commits — and then two computers editing it produce a merge conflict *in the user's repo*, on a file the
user never touched and cannot meaningfully resolve.

The canonical break — and the exact command trail that triggered this section:

```
~/BGit/act3/act3_Marketing_NextGen> git fetch
~/BGit/act3/act3_Marketing_NextGen> git merge
CONFLICT (modify/delete): .lfbridge/repo_storage.yaml deleted in HEAD and modified in
  refs/remotes/origin/main.  Version refs/remotes/origin/main of .lfbridge/repo_storage.yaml
  left in tree.
Automatic merge failed; fix conflicts and then commit the result.
```

* This is a **modify/delete** conflict, the transitional signature of the placement redesign. Under the
  **old** scheme `repo_storage.yaml` was committed inside the working repo's `.lfbridge/`. This computer
  (**HEAD**) had already migrated to the new scheme and **deleted** the in-repo copy; the other computer
  (**origin/main**) was still on the old scheme and **modified** and pushed it. Git can't reconcile
  "deleted here / edited there," so it stops and leaves the stale remote copy in the tree.
* It is **not** a real content disagreement. `repo_storage.yaml` is machine-generated **Category-B tracking
  state** — repo-wide settings + special-file counts that Local Storage regenerates on the next scan. There
  is nothing for the user to hand-merge; **neither committed side is worth keeping.**

### 7.1 Prevention — Category-B tracking state is never committed

The **root fix** is the placement policy of §1 / [artifact_placement_policy.mdx](./artifact_placement_policy.mdx):
**every churny tracking file lives in Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/`, **never
in the working repo**, so it is **outside the user's Git history entirely and can never merge-conflict.**
This covers `repo_storage.yaml`, `storage.yaml`, `files.yaml`, the per-file sidecars, and the per-computer
history. Registering a repo seeds that Local-Storage directory and **no longer** writes `repo_storage.yaml`
into the repo or adds `.lfbridge/` to the repo `.gitignore` (nothing churny is written there anymore).

* Enforcing "Local Storage always, working repo never" for `repo_storage.yaml` (path resolved so it lands
  under `resolveTrackingRoot()`, never a working repo) is done in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/repo-storage.service.ts` —
  `repoStoragePath(repoRoot)` / `writeRepoStorage(repoRoot, doc)` (Local-Storage-always, not gated on the
  keep-`.lfbridge/` consent).

### 7.2 The two shared ledgers that DO travel via git — folded, never conflicted

A few files are *meant* to be shared across computers through the sync repo (the SyncList, the IPFS
`manifest.yaml`, and the team-shared `decisions.yaml` ledger). These are **append/union-shaped**, so instead
of conflicting, their two sides are **unioned** — every computer's lines are kept. LFB writes a
`.gitattributes` marking those paths `merge=union` **before** each merge, so git folds them conflict-free.

* Writing the union-merge `.gitattributes` ahead of every fetch/merge is done in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` —
  `GitBackbone.ensureMergeAttributes()` (constant `UNION_MERGE_PATHS`), called at the top of
  `GitBackbone.pull()` ([git_backbone.mdx §4.2](./git_backbone.mdx)).

### 7.3 Cure — auto-resolving the legacy `repo_storage.yaml` conflict

Repos that already committed the old in-repo `repo_storage.yaml` will still hit the modify/delete conflict
above **once**, during the first merge after either computer migrates. LFB's backbone detects this narrow
case and heals it deterministically: when **every** conflicted path is a `repo_storage.yaml` (any depth —
the legacy `.lfbridge/repo_storage.yaml` and the `repos/<repoKey>/repo_storage.yaml` sync-repo mirror
alike), it **drops the file from the merge** (`git rm -f`) rather than keeping either stale side, commits
the resolution, and lets Local Storage / the next mirror pass regenerate it. **If any *other* path is
conflicted, LFB does the safe thing — aborts the merge and surfaces the conflict for the user's own Git
tools; it never clobbers real content** ([git_backbone.mdx §4.3](./git_backbone.mdx)).

* Detecting the all-`repo_storage.yaml` conflict, dropping the regeneratable file, and committing (else
  abort-and-surface) is done in code:
  `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` —
  `GitBackbone.pull()` guarded by `isRepoStorageYamlPath(...)`, resolving via
  `GitBackbone.autoResolveRepoStorageConflicts(conflicts)`.

### 7.4 The standing rule

**Never write a churny, machine-generated file into a working repo the user commits.** New per-repo state is
Category-B by default and belongs in Local Storage; the only things that ride the user's git are the
Category-A content artifacts (§1, §4) and the three union-merged shared ledgers of §7.2. Any new file that
would land in the working tree and change on most scans is a latent merge conflict — treat it as
out-of-compliance and move it to Local Storage.

## 8. Acceptance criteria

1. Any Git repo LFB tracks is a **repo storage** whose **root is the repo root** (`~/BGit/xyz/`); its
   **committed** **`~/BGit/xyz/.lfbridge/`** holds **only Category-A derived content** (transcripts / AI
   descriptions / visuals-by-time) and travels with the repo, while **all Category-B tracking state** lives
   in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/` (never the working repo, so it can
   never merge-conflict), additionally mirrored to the owning storage's **sync repo** when the per-repo
   toggle is on (default off) — per [artifact_placement_policy.mdx](./artifact_placement_policy.mdx).
2. The repo's `storage.yaml` (Category B, in Local Storage `repos/<repoKey>/`) has **`name`** (default =
   repo folder name), **`type: repo`**, and a **`repo`** block with `repo_root`; no other type block.
3. The Local-Storage `repos/<repoKey>/` holds the per-file fingerprint index (hash + size + dates;
   compressed/uncompressed variant hashes for media), `repo_storage.yaml`, sidecars, history, the IPFS
   manifest, `decisions.yaml`, and the compression records; the working repo's `.lfbridge/` holds only the
   Category-A content — matching [storages.mdx §4](./storages.mdx) / [repo_tracking_scheme.mdx](./repo_tracking_scheme.mdx).
4. Files marked **Add to IPFS (pin)** (the `sync` decision value) move between the user's computers over
   **IPFS**; the small `manifest.yaml` lives in Local Storage and travels via the owning storage's **sync
   repo** (not the working repo's git, §1/§3); the repo may also mirror to **Google Drive / Dropbox** per
   its `clones` block.
5. The repo storage is surfaced through the **Repos** tab (long list + per-repo screen) and the **Repos**
   child of the Storages tab; it is **not** individually listed inside the Storages tab.
6. Each repo owns exactly **one TO DO Batch**, persisted local/disposable/uncommitted at
   `~/T/_large_files_bridge/_do_batches/repo_<repo-slug>_2_do.yaml`, recalculated-and-replaced each scan
   from the repo's classified files folded with the team-shared decisions ledger; it is the **richest
   batch** — **all five** categories apply (**`compress_video`, `compress_image`, `git_ignore`, `pin`,
   `pull_down`**) plus the on-demand transcribe batch; the **same** batch is the repo's yellow warning
   banner and the To-Do page slug (§6).
