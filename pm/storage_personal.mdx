# LargeFileBridge — Personal Storage

**Personal storage** is the user's own private pile of big files — videos, images, archives — that they
keep pinned across **their own** computers. There is exactly **one** personal storage per user. It is also the
product's **canonical Git-backbone use case**: the simplest, most common shape of "I have two computers
and I want the same files, and the same *knowledge about* those files, on both."

This spec tells that story end to end — a user with **two computers**, one **Personal Git repo**,
configured on **each** machine — and locks how Git carries the small text while IPFS carries the bytes,
how each computer **registers itself as a unique device**, and how the two machines **discover each
other** and stay in step.

Read the shared model first; this spec does not restate it:

* [storages.mdx](./storages.mdx) — the storage family: `storage.yaml` (§3), the tracking-area layout
  (§4), cloud clones (§5), media analysis (§6).
* [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — **the owner of the storage-kind
  rule**, which this spec follows and does not restate: **`.lfbridge/` is a working-repo-only concept.**
  A **dedicated LFB file repo** — and the Personal SDL *is* one — has **no `.lfbridge/` directory at
  all**; **the Personal repo root *is* the `.lfbridge` area**, so every path below hangs off the root.
* [storage_settings.mdx](./storage_settings.mdx) — the per-storage settings page where the user **turns
  the Git backbone on** and supplies the remote (§4.1 there).
* [git_backbone.mdx](./git_backbone.mdx) — the **Git engine** this spec drives: the granted operations, the
  `simple-git` library, the local-path-vs-URL remote shapes, and the automatic-merge design.

* [syncable_data_location.mdx](./syncable_data_location.mdx) — the **SDL** (the travelling text) that the
  Personal Git repo *is*.
* [Communities](./communities.mdx) — *other people's* files; personal storage is the opposite: **your
  own** private files, not shared with anyone else.
* [repo_company_mapping.mdx](./repo_company_mapping.mdx) — the owner mapping (Personal is the default

## 1. Anatomy of the personal storage

The Personal storage's Syncable Data Location is, conventionally, a Git repo the user checks out on each
computer:

```
~/BGit/Bryan_git/personal_large_files_bridge/     ← the personal storage root (its SDL; a Git repo)
│                                                  ← NO .lfbridge/ — this ROOT *is* the tracking area (§1.1)
├── .git/                                          ← the user's clone (local-path remote) or LFB's cache clone (URL remote)
├── .gitattributes                                 ← merge=union for the append-mostly lists (git_backbone.mdx §4.2)
├── LargeFilesBridge_SyncList.yaml                 ← the portable list (committed, travels via Git)
├── storage.yaml        ← name + type: personal (storages.mdx §3)
├── mapped_dirs.yaml    ← the home-tree hierarchies this Personal storage covers (SDL §3)
├── files.yaml          ← per-file fingerprint index (storages.mdx §4.1)
├── manifest.yaml       ← large-file → IPFS CID + pin state
├── bookmarks.yaml      ← the user's starred files
├── devices/            ← the DEVICE REGISTRY — one file per computer (§4)
│   ├── bryan-macbook-pro.yaml
│   └── bryan-mac-studio.yaml
├── <rel-dir>/<name.ext>.transcription   ← transcript — path-mirrored, ext APPENDED (storages.mdx §6)
├── <rel-dir>/<name.ext>.ai_description  ← AI description — path-mirrored, ext APPENDED (storages.mdx §6)
└── analysis/           ← visuals-by-time + compression record only (storages.mdx §6)

…the user's personal big files stay in their natural home-tree directories (git-ignored, moved over IPFS)…
```

* **Root:** conventionally `~/BGit/Bryan_git/personal_large_files_bridge/` (the `{personal_storage_root}`).
  It is the storage's SDL — its own little Git repo ([syncable_data_location.mdx §1](./syncable_data_location.mdx#1-definition--what-a-syncable-data-location-is-locked)).
* **The big files are not here.** Like a company storage, Personal maps a **list** of the user's home-tree
  directories ([syncable_data_location.mdx §3](./syncable_data_location.mdx#3-mapped-source-directories--one-storage-many-hierarchies-locked));
  the bytes stay in `~/Movies`, `~/Pictures`, `~/_Mirror/…`, git-ignored, pinned over IPFS. Only the small
  YAML lives in this repo — and that is exactly what Git is good at carrying.
* **Committed vs. git-ignored, for a Git backbone (LOCKED, §6).** When the backbone is a **Git repo**, the
  SDL's small text — `LargeFilesBridge_SyncList.yaml`, `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`,
  `manifest.yaml`, `bookmarks.yaml`, `devices/`, and `analysis/` — **is committed**, because Git is the
  vehicle that makes it travel. Only the **big-file bytes** are git-ignored (and they never live in this
  repo anyway). This narrows the generic *"keep the tracking area git-ignored"* rule
  ([storages.mdx §4](./storages.mdx#4-where-a-storages-state-lives--the-kind-split-then-category-a-vs-category-b-locked)) to its true intent:
  keep the *bytes* out of Git, not the metadata whose entire purpose is to ride Git to your other
  computers.

  LFB **enforces** this: for a personal (or company/community) SDL repo it never adds an ignore for the
  SDL's own text, and before every backbone commit it **strips** a stray one so the device registry is
  actually committed and pushed. Because the SDL has **no `.lfbridge/`** (§1.1), the hazard is now an
  ignore on the SDL's **root metadata** — `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`,
  `manifest.yaml`, `bookmarks.yaml`, `analysis/`, and above all **`devices/`**. A bare **`devices/`** line
  (or a broad root ignore such as `*.yaml` or `/*` with no re-include) is the new **silent killer**: it
  blocks every device file, so the two computers never see each other and the failure looks like "the
  backbone just doesn't work" — the same defect that was hit in production, whose hazard **moved** from a
  `.lfbridge/` line to the root metadata rather than going away
  ([git_backbone.mdx §4.2.1](./git_backbone.mdx#421-the-sync-repos-committed-tracking-payload-must-never-be-git-ignored-locked-learned-from-a-defect)).
  Handled in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts`
  — `reconcileLfbridgeIgnore(root, type)` (for an SDL it reconciles the **root** metadata + `devices/`, not
  a `.lfbridge/` line) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts`
  — `GitBackbone.ensureSdlCommittable()`.

### 1.1 The Personal repo root *is* the `.lfbridge` area — there is no `.lfbridge/` here (LOCKED)

`.lfbridge/` is a **working-repo-only** concept. In a **working git repo** LFB is a **guest**, so it
quarantines everything it writes under one hidden `.lfbridge/` that is obviously not the user's content.
The Personal storage's SDL is the **opposite**: a repo whose entire reason to exist is to hold LFB's files.
Hiding them under `.lfbridge/` inside a repo that contains nothing else buys no separation and only adds a
meaningless path segment — so **the Personal repo has no `.lfbridge/` directory at all, and its root *is*
the `.lfbridge` area**. Everything in the tree above — `storage.yaml`, `mapped_dirs.yaml`, `files.yaml`,
`manifest.yaml`, `bookmarks.yaml`, `devices/`, `analysis/`, and the path-mirrored artifact tree — sits at
the **root**:

```
CORRECT  ~/BGit/Bryan_git/personal_large_files_bridge/_Mirror/Politics/…/FINAL_CIA_Drones.mp4.transcription
WRONG    ~/BGit/Bryan_git/personal_large_files_bridge/.lfbridge/_Mirror/Politics/…/FINAL_CIA_Drones.mp4.transcription
```

The rule is **owned by** [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx) — with the
SDL's own metadata in §0.1, the kind-not-name keying in §0.2, and the reserved root names (a mapped-dir key
named `devices`, `analysis`, `files.yaml`, … would shadow the SDL's metadata) plus the migration of a
pre-existing `.lfbridge/` in §0.3. This spec **follows** it and does not restate it. Two consequences worth
naming here: the decision is keyed on the resolved **storage kind** (`personal`), never on the repo's path
or name; and Personal's own `.git/` — every SDL has one — **never** demotes it to a working repo.

## 2. `storage.yaml` for personal

```yaml
# ~/BGit/Bryan_git/personal_large_files_bridge/storage.yaml   ← at the ROOT — an SDL has no .lfbridge/ (§1.1)
name: "My Personal Files"
type: personal
created: 2026-05-01T10:12:00Z
personal: {}                # usually empty / a place for personal preferences
clones:
  google_drive: null        # e.g. "My Drive/LFB/Personal" once mirrored (storages.mdx §5)
  dropbox: null             # e.g. "Apps/LFB/Personal"
```

* **`type: personal`** with a (usually empty) **`personal`** block. Its content is the user's own — no
  `company`/`community` identity.
* Because it is **private to one user**, it never has a shared allow-list the way a company/community
  storage might — it is pinned only across **this user's** computers (its own device registry, §4).

## 3. Use case — two computers, one Personal repo (the driving story)

This is the exact scenario the product is built to serve, told from the user's chair:

> *I have two computers. Each runs the Large File Bridge web app. I created a Git repo for my personal
> files and checked it out on both machines. On each computer I opened **Personal storage settings**,
> turned **Git on**, and pointed it at where I checked the repo out. Now I want both computers to hold my
> big files and to know everything about them — and I want them to know about **each other**.*

Everything below makes that true:

* **Turning Git on grants the git backbone permission (§5).** It authorizes LFB to run `git fetch`, `git merge`,
  `git commit`, and `git push` on that repo every pass ([git_backbone.mdx §1](./git_backbone.mdx#1-what-lfb-is-permitted-to-do--the-granted-git-operations-locked)).
* **The remote is one of two shapes (§5).** A **local file path** to the checkout the user already made,
  or an **HTTP(S) URL** to a Git server with no local checkout.
* **Each computer becomes a unique device (§4).** It writes its own `devices/<device-id>.yaml` — id +
  hardware fingerprint — into the repo, so it never collides with the other computer's file.
* **The computers discover each other (§4).** Each reads the *other* devices' files from the repo. That
  is how "computer B" appears on "computer A" in the **Devices** left-bar tab, and vice-versa.
* **They stay in step (§6).** Every pass: fetch what the other pushed, auto-merge it, do the byte
  work over IPFS, commit this computer's own changes, and push them back.

The rest of this spec is the mechanism behind each bullet.

## 4. Each computer registers itself as a unique device

When a Personal storage exists on a computer, that computer **writes itself into the repo** as one
device file, and **reads the others** to learn about the user's other machines. The full device model —
identity, fingerprint, disambiguation, the graft, the Devices/Peers table — is
[devices.mdx](./devices.mdx); this section is the Personal-storage view of it.

### 4.1 A unique id per device, under `devices/` at the repo root

At the **root** of the Personal repo — an SDL has no `.lfbridge/` (§1.1) — LFB keeps a **`devices/`**
directory, and each computer creates **one file keyed by its own unique id**:

```
~/BGit/Bryan_git/personal_large_files_bridge/devices/
├── bryan-macbook-pro.yaml    ← written & OWNED by the laptop (its id inside)
└── bryan-mac-studio.yaml     ← written & OWNED by the studio tower
```

* **The id is a stable, minted UUID** ([devices.mdx §1](./devices.mdx#1-device-identity--a-minted-id--a-nice-name-locked)),
  distinct from the IPFS PeerID, minted once on first run and stored both in this machine's local
  `~/T` config **and** copied into this device file. Because it is unique per computer, *"it won't
  conflict with other devices"* — two computers writing the registry at the same time touch **two
  different files**, and Git merges them with no conflict ([git_backbone.mdx §4.1](./git_backbone.mdx#41-self-owned-files-rarely-collide)).
* **Self-owned (trust).** A computer writes **only its own** device file and treats every other device
  file as a **claim** ([devices.mdx §2.1](./devices.mdx#21-a-device-file-is-self-owned-trust)) — it never
  edits another computer's file.

### 4.2 The device file carries this computer's specifics

Each device file records as much as it can to make the physical machine **recognizable and unique** —
first from OS APIs, and, in the worst case, from a **composed** fingerprint of whatever hardware facts
are available ([devices.mdx §7](./devices.mdx#7-the-hardware-fingerprint--identify-the-physical-machine-no-networking)):

* **From the OS directly (always available, no network):** platform, architecture, hostname, **logged-in
  username** + **home directory** (which user this machine belongs to), CPU core count, RAM, and total
  disk size.
* **From macOS tooling (best-effort):** the **model identifier** (`sysctl -n hw.model` → `Mac14,7`), the
  **model name** + **chip** (`system_profiler`), and the **display count**.
* **From a built-in model table:** the **marketing name**, **year**, and **built-in screen size** resolved
  from the model identifier.
* **If the OS exposes a stable machine id, we use it;** otherwise the id is our minted UUID and the
  **composed** set above (model + year + disk size + screen size + RAM + chip + username) is what keeps
  this machine distinct from every other — *"put together the pieces that make this computer unique … as
  many things like that as possible."*

The whole fingerprint is collected **locally, with no networking** (the charter's no-phone-home rule),
then committed into the device file so **other** computers can see and disambiguate this one.

### 4.3 Discovering the other computers

Because every computer's device file is **in the repo**, a fetch+merge brings the **full set** onto this
machine. LFB then reads the whole `devices/` directory:

* The file whose `id` matches **this** machine's local `config.yaml → computer.id` is **"this computer"**;
  every other file is one of the user's **other** computers ([devices.mdx §9](./devices.mdx#9-auto-seed--the-registry-is-never-empty-locked)).
* Those other devices — the second computer, a future third, a family member's machine — appear as rows
  in the **Devices** left-bar tab ([devices.mdx §6](./devices.mdx#6-the-devices--peers-page-the-table)),
  where the user can open one and see its full hardware details. *"That way, in the Devices left-bar tab,
  they appear there in the UI, and we can open them up and get details on them."*

Writing this computer's own `devices/<id>.yaml` into the Personal repo (self-owned) and reading the whole registry to discover the other computers is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/devices.service.ts` — `writeSelfDevice(storageRoot)` / `readDevices(storageRoot)`, with the fingerprint from `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/hardware.service.ts` — `collectHardware()`; surfaced in the Devices tab by `~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/devices/DevicesPage.tsx` — `DevicesPage`.

## 5. Turning the Git backbone on — local path vs. HTTP(S) URL

The user turns Git on for Personal from its **[storage settings](./storage_settings.mdx#41-dedicated-git-repo-preferred-when-available)**
page (the dedicated-Git-repo backing location). Doing so is the **explicit grant** that lets LFB run Git
on the repo ([git_backbone.mdx §1](./git_backbone.mdx#1-what-lfb-is-permitted-to-do--the-granted-git-operations-locked)).
The user supplies the repo as **one of two remote shapes** ([git_backbone.mdx §3](./git_backbone.mdx#3-the-two-remote-shapes--a-local-path-vs-an-https-url-locked)):

* **A local file path** — the checkout already on this computer, e.g.
  `~/BGit/Bryan_git/personal_large_files_bridge/`. This is what the user in the story did on each machine
  (*"added that local file path where I have checked in"*). LFB uses that directory **in place** as the
  working copy and runs Git against whatever `origin` the user's clone already has.
* **An HTTP(S) URL** — e.g. `https://github.com/BryanStarbuck/personal_large_files_bridge.git`, with
  **no** local checkout (*"sometimes people will add an HTTP URL, and we have to talk to that Git server
  without it being mapped to the local file system"*). LFB **clones** it into a machine-local
  working-copy cache (`~/T/_large_files_bridge/pin/s/personal/git/`) and manages Git there. Credentials
  come from the **OS git credential helper**, never an LFB password prompt
  ([git_backbone.mdx §5](./git_backbone.mdx#5-credentials-for-a-url-remote--the-os-git-credential-helper-never-a-prompt-locked)).

Both shapes converge on one working copy; everything downstream is indifferent to which one the user
picked.

Persisting the Personal storage's Git remote (its kind — local path or URL — and value) from the settings page, and reading it back to resolve the working copy, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-settings.service.ts` — `writeStorageSettings()` / `readStorageSettings()` (the `backing.dedicated_repo` block), consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `resolveWorkingCopy(storageId, remote)`.

## 6. First-time usage — configuring both computers, step by step

The canonical Personal walkthrough, matching the story in §3. It shows exactly what happens the **first**
time on each machine and how the two converge.

**Computer 1 (the laptop):**

1. **Open Personal storage settings; turn Git on.** The user enables the dedicated-Git-repo backing
   location and enters the **local path** to their checkout (§5).
2. **LFB seeds the SDL.** On the next pass it writes, **at the repo root** (§1.1), `storage.yaml`
   (`type: personal`), the `mapped_dirs.yaml` for the home-tree directories the user added, the
   `.gitattributes` (`merge=union`, [git_backbone.mdx §4.2](./git_backbone.mdx#42-append-mostly-shared-lists-merge-by-union)),
   and — crucially — **this laptop's own** `devices/bryan-macbook-pro.yaml` with its id + fingerprint (§4).
3. **The user pins some big files to IPFS.** LFB records each CID in `manifest.yaml` and marks this
   laptop in `pinned_by`.
4. **LFB commits and pushes.** The small SDL text is committed to the Personal repo and pushed to the
   remote ([git_backbone.mdx §6](./git_backbone.mdx)).
   The bytes never enter Git — they move over IPFS.

**Computer 2 (the studio tower):**

5. **Open Personal storage settings; turn Git on.** The user enters this machine's path to the **same**
   repo (a local checkout) — or a **URL** if they'd rather LFB manage the clone (§5).
6. **LFB fetches + auto-merges.** The pass pulls down the laptop's push. Now this computer knows, from
   the merged YAML alone: which big files exist and their **CIDs**, which were **pinned/compressed/
   transcribed/described**, which are **bookmarked**, and — the point of this spec — that a **second
   device** (the laptop) exists.
7. **This computer registers itself.** It writes its **own** `devices/bryan-mac-studio.yaml` (a *different*
   file — no conflict, §4.1), pulls the marked CIDs down over IPFS to its **grafted** local paths
   ([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)), then
   **commits and pushes** its own device file back.
8. **Both computers now see both devices.** On the next pass the laptop fetches the tower's device file;
   each machine's **Devices** tab shows two rows, one tagged *This computer*. The user can open the other
   row and see its full hardware details (§4.3).

Seeding the Personal SDL on first configuration (descriptor + mapped dirs + `.gitattributes` + self device) and driving the first fetch/merge/commit/push is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts` — `ensureStorage()` (writes the descriptor + self device on create/index) and `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinStorageUnit(id)` (the Git cycle around the reconcile).

## 7. The backbone loop — keeping the two computers in step

Every pin pass ([pin_process.mdx](./pin_process.mdx), the 15-minute background process +
Pin-now), for the Personal storage with its Git backbone on, LFB runs the fixed git backbone cycle
([git_backbone.mdx §6](./git_backbone.mdx)) so that
*"as changes are happening, we're updating the app files and pulling down any changes, dealing with any
merge, then checking in changes and getting them synced with the server side"*:

1. **Resolve the working copy** — the local checkout, or the URL cache clone (§5).
2. **`git fetch`** — bring down what the **other** computer pushed since last time.
3. **`git merge` (automatic)** — fold the other computer's device files, manifest, and analysis into this
   working copy. Self-owned device files don't collide; the shared lists union-merge; only a genuine
   conflict is surfaced (never clobbered) ([git_backbone.mdx §4](./git_backbone.mdx#4-automatic-merge--why-conflicts-are-rare-and-how-git-resolves-them-locked)).
4. **Reconcile + IPFS byte work** — pull/pin the marked CIDs to this computer's grafted paths, inherit
   derived artifacts without regenerating them, refresh this computer's own device file, update the
   manifest.
5. **`git add` (self-owned only) → `git commit`** — stage **only this computer's** files plus any shared
   list it appended to; commit if anything changed (no empty commits).
6. **`git push`** — send this computer's commit up. On a non-fast-forward reject (the other computer
   pushed in between), LFB **re-fetches, re-merges, and re-pushes once** — never a force-push.

The loop is **idempotent**: with nothing changed on either side, a pass fetches, finds nothing to merge,
commits nothing, and pushes nothing. Run every 15 minutes on both machines, the two computers converge —
the same files, and the same knowledge about them, on both.

**Your device info is written back even before you opt into IPFS pinning.** A separate,
**every-10-minute** device-registration check ([devices.mdx §11](./devices.mdx#11-the-device-registration-write-back--every-10-minutes-locked))
guarantees this computer's `devices/<self>.yaml` is **written and pushed** to the Personal repo on its own
cadence — **git pull + merge first, then commit + push**, always pulling even with nothing to change — and
it runs **whether or not** the Personal storage is opted into IPFS pinning (`pinned`). So the moment you
turn Git on for Personal, your device appears in the repo (and on your other computer's Devices tab)
within ten minutes, regardless of the byte-pin opt-in. It is a distinct worker with its own
installed / on-off switch on the Scans page, so *"I turn that on"* is a single, visible toggle.

Driving the per-pass Git cycle for the Personal storage (fetch+merge before the reconcile, commit+push after, with the non-fast-forward retry) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinStorageUnit(id)`, calling `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone.pull()` then `GitBackbone.commitAndPush()`.

## 12. TO DO Batch for this storage (LOCKED)

Personal storage participates in the product-wide **TO DO Batch** system
([to_do_batches.mdx](./to_do_batches.mdx)). A **TO DO Batch is a per-storage bundle of recommended file
actions** — there is exactly **one** batch for the Personal storage, persisted as hierarchical YAML at
**`~/T/_large_files_bridge/_do_batches/personal_2_do.yaml`**. Like every batch it is **local, disposable,
and uncommitted** — it lives only in the machine-local state root, is **never** written into the Personal
SDL (nor into any working repo's `.lfbridge/`), and is **never** team-shared (it is not a decision; the folded decisions ledger stays the source of truth).

* **Which categories apply to Personal.** Because loose personal files live **outside any repo**, the
  **`git_ignore`** category ("big-but-not-git-ignored") is **only meaningful under a repo**, so it **never**
  appears in a Personal batch. Personal batches are therefore typically **`compress_video`**,
  **`compress_image`**, **`pin`**, and **`pull_down`** — plus the transcription discovery below.
* **Recalculate-and-replace each scan.** The batch is **recomputed from scratch and replaced** on every
  scan pass by the calc engine ([to_do_batch_calc_engine.mdx](./to_do_batch_calc_engine.mdx)); it
  **recommends only** — it never writes a decision or moves a byte, and it **skips any file/axis the team
  already decided** so we never re-ask a folded question.
* **The cross-computer `pull_down` story (§3).** This is where the two-computers use case (§3) becomes a
  concrete recommendation. Last month the **tower** pinned a batch of new images; this month, on the
  **laptop**, those same images have no local copy — they exist on the tower over IPFS but are **absent
  here** — so they land in the laptop's Personal batch as **`pull_down`** recommendations ("pull it down").
  The batch is the actionable form of *"my other computer has files this one is missing."*
* **It surfaces on the To Do page as a slug.** Any batch **with work** renders on the To Do page
  ([to_do.mdx](./to_do.mdx)) as a **slug**; opening the slug's chevron shows the recommendations as
  two-axis **decision toggles** pre-set to the recommended state ([decision_toggles.mdx](./decision_toggles.mdx)),
  which the user confirms and **Applies**.
* **Transcription discovery for Personal.** The on-demand **"Show what could be transcribed"** link runs
  the **What Can Be Transcribed Calc Engine** ([transcribe_calc_engine.mdx](./transcribe_calc_engine.mdx)),
  which walks the Personal storage's mapped home-tree directories for **video/audio that has no
  `.transcription` artifact in the Personal SDL's root-hung mirror** (§1.1) and emits a per-storage **transcribe batch** → a **transcribe slug** (a
  ~80%-width, default-checked popup: *"Transcribe N files"* → background jobs on
  [processing.mdx](./processing.mdx)).

Producing the Personal batch during scan recalc, and naming/replacing its YAML, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batch.engine.ts` —
`TodoBatchCalcEngine.recalcForStorage(storage)` writing via
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/todo/todo-batches.store.ts` —
`batchFileName(storage)` (→ `personal_2_do.yaml`); the transcribe discovery walk is

## 13. Repo ownership — Personal is the default, and your personal-accounts list

Beyond being the default owner of **loose home-tree files** (§9), Personal is also the **default owner of repos**
([repo_company_mapping.mdx](./repo_company_mapping.mdx)). Every git repo LFB knows about maps to an owner — a
company or Personal — and **Personal is the catch-all**:

* **A repo with no forge org falls to Personal.** No remote, a local-path remote, an unknown/self-hosted host, or a
  known forge whose owner is **you** → the repo groups under **Personal**
  ([repo_company_mapping.mdx §3](./repo_company_mapping.mdx#3-the-heuristic--which-parse-becomes-a-company-locked)).
  Only a forge **org** that is **not yours** derives a company.
* **The "my personal accounts" list (LOCKED) — owned here.** So your **own** forge repos never miscategorize as a
  company, you maintain a small list of your account handles across forges. Any repo whose remote owner matches an
  entry derives to **Personal** even on a known forge.

  ```yaml
  # app config (storage_local.mdx) — YOUR own forge accounts, consumed by repo_company_mapping.mdx §4.
  personal_accounts:
    - { host: github.com, owner: BryanStarbuck }   # host + owner …
    - { host: gitlab.com, owner: bstarbuck }
    - { owner: bryan-personal }                     # … or owner on ANY host (host omitted = wildcard)
  ```

  * Match is **case-insensitive** on `owner`; `host` is **optional** (omit = personal on every forge). An sr.ht
    `~handle` matches with or without the leading `~`.
  * **Editing the list re-derives.** Adding an owner re-maps its repos **company → Personal** on the next scan;
    removing one lets them fall back to a company. **Manual** owner overrides
    ([repo_company_mapping.mdx §5](./repo_company_mapping.mdx#5-manual-reassignment--the-user-always-wins-locked)) are untouched.
  * The list **starts empty** and is populated by the user — LFB may *suggest* your handle (from the machine's git
    `user.email` domain) but never adds it silently (charter: surface and offer).
* **You can always reassign.** A repo the heuristic put under a company can be forced to Personal from repo
  settings / the `⋮` menu ([repo_company_mapping.mdx §5](./repo_company_mapping.mdx#5-manual-reassignment--the-user-always-wins-locked)),
  persisted as `owner_override` in the repo's `config.yaml`.

Persisting and matching the personal-accounts list is to be implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig().personalAccounts` / `setPersonalAccounts()`, matched by `isPersonalAccount(host, owner)` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage.service.ts`.

## 15. Acceptance criteria

1. A user has **exactly one** personal storage, whose SDL is conventionally the Git repo
   `~/BGit/Bryan_git/personal_large_files_bridge/`. Being a **dedicated LFB file repo**, it has **no
   `.lfbridge/` directory at all** — **its root *is* the tracking area** (§1.1,
   [artifact_placement_policy.mdx §0](./artifact_placement_policy.mdx)); `.lfbridge/` is a
   working-repo-only concept.
2. Its **root** `storage.yaml` — `~/BGit/Bryan_git/personal_large_files_bridge/storage.yaml`, **not**
   `.lfbridge/storage.yaml` — has **`name`** + **`type: personal`** and a (usually empty) `personal`
   block; no `company`/`community`/`repo` block.
3. **The Git backbone is turned on per-computer from Personal storage settings**, and doing so **grants
   LFB permission** to run `git fetch` / `merge` (automatic) / `commit` / `push` on that repo every
   backbone pass ([git_backbone.mdx](./git_backbone.mdx)). The remote may be a **local file path** (a checkout used in
   place) or an **HTTP(S) URL** (cloned into a machine-local cache) (§5).
4. **When the backbone is a Git repo, the SDL's root text is committed** (SyncList, storage.yaml,
   mapped_dirs, files, manifest, bookmarks, `devices/`, `analysis/`); only the **big-file bytes** are
   git-ignored, and those never live in the repo (§1). Bytes move over IPFS; text moves over Git. LFB
   never ignores that root text and **strips** a stray ignore before each backbone commit — a bare
   **`devices/`** line or a broad root ignore is the silent killer that makes the two computers invisible
   to each other (§1).
5. **Each computer registers itself as a unique device**: it writes a self-owned
   `devices/<device-id>.yaml` **at the Personal repo root** keyed by its **stable minted id**, carrying a **hardware
   fingerprint** (OS machine id when available, else composed from model + year + disk + screen + RAM +
   chip + OS user), collected **locally with no networking**. Two computers write **different** files, so
   the registry **never conflicts** on merge (§4).
6. **The computers discover each other**: after a fetch+merge, each reads the whole `devices/` directory,
   identifies **"this computer"** by matching its local `config.yaml → computer.id`, and shows every
   **other** computer as a row in the **Devices** left-bar tab, openable for full hardware details (§4.3).
7. **First-time usage converges** (§6): computer 1 turns Git on, seeds the SDL + its device file, pins
   files, commits & pushes; computer 2 turns Git on, fetches & auto-merges, pins the marked CIDs to its
   **grafted** paths, writes its **own** device file, and commits & pushes — after which both machines see
   both devices and hold the same files + the same derived knowledge.
8. **The backbone loop keeps them in step** (§7): every pass resolves the working copy → fetch →
   auto-merge (self-owned files don't collide; shared lists union-merge; genuine conflicts surfaced, never
   clobbered) → reconcile/pin over IPFS → stage self-owned + commit → push, with a **fetch-merge-push
   retry** on a non-fast-forward reject; the loop is idempotent.
9. Personal remains the **default owner** of loose home-tree media; with its dedicated repo ON, derived
   artifacts are **committed under that repo** mirroring the file's `~`-relative path (§9), and it appears
   as the **Personal** child of the Storages tab (§11).
10. Personal owns exactly **one TO DO Batch**, persisted local/disposable/uncommitted at
    `~/T/_large_files_bridge/_do_batches/personal_2_do.yaml`, recalculated-and-replaced each scan; its
    applicable categories are **`compress_video`, `compress_image`, `pin`, and `pull_down`** — **not
    `git_ignore`** (loose files sit outside any repo) — plus the on-demand transcribe batch; batches with
    work surface as slugs on the To Do page (§12).
11. Personal is the **default owner of repos** too (§13): a repo with no forge org — no/local/unknown remote, or a
    known-forge owner that is **on your personal-accounts list** — groups under **Personal**
    ([repo_company_mapping.mdx §3](./repo_company_mapping.mdx#3-the-heuristic--which-parse-becomes-a-company-locked)).
    The **"my personal accounts"** list is owned here, stored in app config (`personal_accounts`; `host` optional =
    wildcard), re-derives `auto` owners on edit, leaves `manual` overrides alone, and starts empty (§13).
12. Personal **can also have a sync repo** — the §5 Personal Git backbone **is** it (§14): when configured and the
    repo's per-repo toggle is on, a **Personal-owned** repo's LFB tracking state is **additionally mirrored** to
    **`<syncRepo>/repos/<repoKey>/`** — in addition to Local Storage `~/T/_large_files_bridge/repos/<repoKey>/`,
    never instead of it ([artifact_placement_policy.mdx](./artifact_placement_policy.mdx)) — and carried to the
    user's other computers over the same backbone; ~~default is Local-Storage-only~~ (**superseded by AC 12b**),
    and **ownership** decides.

## 16. The engine model — a network of parties, one site of convergence

### 16.1 Which repos this part governs (LOCKED)

Part II applies **only** to a **dedicated LFB file repo** — a repo whose entire reason to exist is to hold
LFB's own text:

* **The Personal SDL** — one per user, turned on in Personal storage settings (§5), conventionally
  `~/BGit/Bryan_git/personal_large_files_bridge/`.
* **A company sync repo** — one per company the user added ([storage_company.mdx §7](./storage_company.mdx#7-the-company-sync-repo)),
  e.g. `~/BGit/Bryan_git/act3_large_files_bridge/`.

**Why the distinction is the whole license.** In a **working repo** LFB is a **guest**: someone else's
code lives there, the user commits to it by hand, and LFB must never run `git commit`/`push` on their
branch. In a **dedicated file repo** LFB is effectively the **only author** — *"since it's only going to
be used by Large File Bridge, we're allowed to fully synchronize it and use it as we want."* That is what
makes an **automatic, headless, no-user-interface** merge policy (§23) legitimate here and **nowhere
else**. A working repo NEVER gets the Part II engine — it is governed by
[git_backbone.mdx](./git_backbone.mdx) and [git_ignore.mdx](./git_ignore.mdx).

**The kind decides, not the path** — the same rule as §1.1, keyed on the resolved storage kind via
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` —
`usesLfbridgeDir(kind)` / `trackingBaseDir(root, kind)`.

#### 16.1.1 Both repos are governed autonomously — the company case is not a lesser case (LOCKED)

§16.1 already puts a **company sync repo** in Part II's scope. This subsection makes explicit what that
means, because the company repo is where the rules bite hardest and it is easy to read Part II as a
Personal-only document.

* **The autonomy rules apply to BOTH, identically.** Everything
  [storage_company.mdx §11](./storage_company.mdx#11-the-company-tracking-repo-governs-itself-locked) locks —
  fetch / merge / commit / push **without ever waiting on the user**, per-file conflict resolution, the
  no-origin fault, one lock per storage, the three-attempt push — is a rule about a **dedicated LFB file
  repo**, and the Personal SDL and a company sync repo are the **same kind of object**. Neither gets a
  weaker engine.
* **The licence is the same licence.** §16.1's argument is that in a dedicated file repo LFB is effectively
  the only author, so a headless merge policy (§23) is legitimate. That argument does not mention Personal;
  it turns entirely on the **kind**. A company sync repo the user's team created *for Large File Bridge* is
  exactly as much LFB's to drive as the Personal one.
* **The company case is the harder one, not the softer one.** A Personal SDL has one author across a
  handful of the user's own machines. A company sync repo has **several members** pushing, and it carries
  the whole `repos/<repoUid>/` mirror rather than a device file — so conflicts and push races that are
  theoretical for Personal are routine there. That is *why* the ladder and the retry budget exist, not a
  reason to exempt it.
* **The confidentiality boundary is a company-only addition.** One thing is genuinely extra for companies:
  each org's tracking never enters another org's repo
  ([storage_company.mdx §10.4](./storage_company.mdx#104-what-the-linkage-buys-locked)). Personal has no
  equivalent because there is only one Personal.
* **`owner_map.yaml` remains the single consent-gated exception** (§22.8) — the assertion converges
  automatically like every other append-log, but its **effect** never applies on another member's machine
  without a human. Autonomy is about *git*, never about silently re-mapping someone's repos.

Keying the engine to a dedicated LFB file repo — Personal SDL or company sync repo alike — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/storage-type.service.ts` — `usesLfbridgeDir(kind)` / `trackingBaseDir(root, kind)`, with the per-storage serialization in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git-lock.ts` — `withStorageGitLock()` and the per-file conflict resolution in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `resolutionFor()` / `GitBackbone.resolveConflicts()`.

### 16.3 The v2 mental model (LOCKED)

Model the Personal/company repo as a **network of multiple parties checking in changes**, with **exactly
one site of convergence**: the local working copy, during a pass. Not a database, not a lock server — a
**convergent replica** whose merge function must be **total** (defined for every input) and **headless**
(§23).

Three invariants, in priority order. When they conflict, the **lower number wins**:

1. **Never lose data.** No pass may destroy a byte another party produced. Conflict → **keep both** (§23);
   the last resort is quarantine, never `--force`, never a silent overwrite.
2. **Always converge.** Every pass must end with the working copy either **fully pushed** or **explicitly
   reported as blocked** (§25). "Quietly dirty forever" — the live state that motivated this spec — is a
   **defect**, not a resting state.
3. **Converge fast.** Target: a change is on the server within **10 minutes**; the exposed window in which
   a conflict can even form is **≤ 60 seconds** (§18).

The engine is **not** "commit whatever is dirty when a timer happens to fire" (v1(a)+(b)). It is a
**four-stage transaction** (§17) over an explicit **outbox** (§19).

## 17. The four stages of a synchronization pass (LOCKED, NEW)

Every pass over a dedicated file repo runs these four stages **in this order**. The stage boundaries are
the contract; the work inside them may change.

```
   ┌── STAGE 1 ── PULL ─────────────────────────────────────────────┐
   │  fetch + merge from the server. MUST NOT conflict (§17.1).     │
   └────────────────────────────────────────────────────────────────┘
   ┌── STAGE 2 ── WRITE + RECONCILE ────────────────────────────────┐
   │  drain the outbox into the tree; reconcile by schema (§22).    │
   └────────────────────────────────────────────────────────────────┘
   ┌── STAGE 3 ── RESOLVE ──────────────────────────────────────────┐
   │  headless resolution of anything left (§23). No UI. Ever.      │
   └────────────────────────────────────────────────────────────────┘
   ┌── STAGE 4 ── PUSH ─────────────────────────────────────────────┐
   │  commit + push. On NFF: re-enter STAGE 1. Bounded retries.     │
   └────────────────────────────────────────────────────────────────┘
```

### 17.1 Stage 1 — PULL: a clean-tree precondition (CHANGED)

*"We pull down from the server side, and there should be no merge conflict at this point."*

That is a **precondition to enforce**, not a hope. It holds **if and only if the working tree is clean
when the merge runs** — a merge into a clean tree cannot conflict on files the local side has not touched.
v1 cannot promise this: `pull()` merges into whatever the artifact writers have strewn about
(v1(b)), because writers write **straight into the working copy**.

**v2 rule (NEW).** Stage 1 requires **`git status --porcelain` to be empty** before `git merge` runs. It
is empty because **producers never write into the working copy** — they write to the **outbox** (§19), and
the outbox drains in **Stage 2**, *after* the merge. This is the structural reason §19 exists.

* **Clean tree** → `git fetch origin <branch>` then `git merge --no-edit origin/<branch>`
  (git.service.ts:389-394, BUILT). A fast-forward or a clean ort merge. **Conflict here is a bug**, and is
  reported as one (§25).
* **Dirty tree** → this is v1 residue or an outside actor (the user's own editor; the "Bryan 26 Tower"
  auto-committer, [backbone_resilience.mdx](./backbone_resilience.mdx)). **Do not abort the pass.** Take the
  dirty paths into the outbox as **`observed`** entries (§19.3), restore the tree to `HEAD`, merge clean,
  then let Stage 2 re-apply them through the normal reconciliation. Never `git stash` (it strands data in a
  ref no one reads); never `git checkout --` a file whose content is not already captured in the outbox.
* **`.gitattributes` first.** `ensureMergeAttributes()` must run **before** the merge, or the union drivers
  don't apply to the merge that needs them (git.service.ts:378-382, BUILT — and correct).

### 17.2 Stage 2 — WRITE + RECONCILE (NEW)

*"We have data to synchronize into it, and we write that data in there."*

Now — and **only** now, with the server's state merged in and visible — the outbox drains:

1. **Refresh self-owned state**: `writeSelfDevice(root)` (devices.service.ts:115). Self-owned files can be
   written blind — no other party writes them (§22.1).
2. **Drain the outbox in commit order** (§19.4). For each entry, apply the **reconciliation rule for its
   data class** (§21) using the **merged** tree as the base. This is where "what's the newest change" is
   decided (§20), and it is decided **in our code, against merged content** — not by git, and not by
   whichever timer fired last.
3. **Byte work does NOT happen here (CHANGED — this is the fix for v1(d)).** IPFS add/pin/fetch is
   **evicted from between pull and push**. See §18.2.

### 17.3 Stage 3 — RESOLVE (NEW)

Anything Stage 2 could not fold cleanly is resolved **here**, **headlessly**, by §23's ladder. Stage 3 is
allowed to be a no-op — and on a healthy single-writer machine it always is.

### 17.4 Stage 4 — PUSH (CHANGED)

1. `git add -A` (git.service.ts:491, BUILT — keep it; it is why artifacts travel at all).
2. **Commit with a descriptive message (CHANGED — fixes v1(c)).** The message must state what the commit
   actually carries, composed from the staged set:

   ```
   LFB: 47 AI descriptions, 3 transcripts, device state  (bryan-mac-pro)
   LFB: device state  (bryan-mac-pro)
   LFB: manifest +12 pins, 2 decisions  (bryan-mac-pro)
   ```

   Format: `LFB: <comma-separated counted categories>  (<device nice name>)`. Categories are derived from
   the staged paths by the same classifier that assigns data classes in §21. `LFB: backbone device state`
   remains legal **only** when device state is genuinely all that changed.
3. **Push, and on failure re-enter Stage 1** (git.service.ts:511-527, BUILT — but CHANGED in three ways):
   * **Log the original error** — v1 discards `e` (v1(g)). It must be logged at WARN with its
     classification before the retry.
   * **Retry up to 3 times with backoff** (0s, 2s, 8s), not once. A busy repo with three parties can lose
     two races legitimately.
   * **Never `--force`, never `--force-with-lease`.** Non-negotiable (§23.0).
4. **Push when ahead even with nothing committed** (git.service.ts:506-509, BUILT — correct and important:
   it is what heals a previous pass that committed but failed to push).

#### 17.4.3 The retry rule is now BUILT — and it is the same rule for company repos (LOCKED)

Item 3's three changes above were written as **CHANGED** against v1. They are now **BUILT**, and the engine
spec states them: [git_backbone.mdx §6.3](./git_backbone.mdx#63-the-push-retry-budget--three-attempts-with-backoff-locked).

* **Three attempts, `0s / ~2s / ~8s` with ±40% jitter** (the jitter keeps two of the user's computers from
  retrying in lockstep — [git_backbone.mdx §6.3](./git_backbone.mdx#63-the-push-retry-budget--three-attempts-with-backoff-locked)).
  Exactly the budget asked for here. Each attempt **re-pulls first**, so
  every retry pushes on top of the newest remote state; if that re-pull leaves something genuinely
  unresolvable, the cycle **stops and surfaces** rather than pushing blindly.
* **The original error is logged**, with its classification and its attempt number — closing v1(g), where
  `e` was discarded and an auth failure was indistinguishable in the log from a routine non-fast-forward
  race. A later success **clears** the earlier attempt's problem.
* **Never `--force`, at any attempt.**
* **Why three, stated once:** a single retry loses to any peer that pushes twice while we are working. On
  the **Personal** SDL that is rare; on a **company sync repo** with several members it is an ordinary
  Tuesday ([storage_company.mdx §11.4](./storage_company.mdx#114-push-retries)). §16.1.1 is why the same
  budget serves both.
* **Exhausting the budget is not loss.** The commit stays local, the fault is reported, and item 4's
  push-when-ahead sends it on the next pass — and it is now also **recorded, re-armed on a jittered
  backoff, and surfaced to the user** until a push actually lands
  ([git_backbone.mdx §6.4](./git_backbone.mdx#64-giving-up-this-cycle-must-never-mean-giving-up-locked)). A repo with **no `origin`** at all is a different, louder case
  ([git_backbone.mdx §4.3.2](./git_backbone.mdx#432-a-repo-with-no-origin-is-a-loud-standing-fault-locked),
  [backbone_resilience.mdx §6.2](./backbone_resilience.mdx#62-a-repo-with-no-origin--committing-forever-reaching-nobody-locked)) —
  it is a **standing fault**, not a retry.

The three-attempt push budget, its per-attempt re-pull, and its per-attempt error logging are handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/git/git.service.ts` — `GitBackbone.commitAndPush()` (the `BACKOFF_MS = [0, 2000, 8000]` loop, each iteration calling `GitBackbone.pull()`), with the honest commit message of item 2 in the same file's `composeCommitMessage()`.

### 18.2 The window — v1 vs. v2 (CHANGED)

The **conflict window** is the interval during which a *remote* change can land that our pass has not
seen: from the **end of Stage 1's merge** to the **completion of Stage 4's push**. Anything a peer pushes
inside that window costs us a non-fast-forward (cheap, retried) or a real conflict (expensive, §23).

**v1, `pinStorageUnit` — the window contains the byte work (v1(d)):**

```
   pull ──┤                                                          ├── push
          └────────── runUnitPin: IPFS add/pin/cat, UNBOUNDED ───────┘
          pin.service.ts:412                            pin.service.ts:462
          window = seconds … HOURS  (first sync of a 500 GB storage)
```

**v2 — byte work is evicted from the window (NEW, the core timing fix):**

```
   pull ─┤ drain outbox · reconcile · commit · push ├─   ~1–5 s typical, ≤ 60 s BUDGET
         └───────────────────────────────────────────┘
                        …then, OUTSIDE the transaction, the IPFS byte work runs…
```

**The rule (LOCKED):** *No unbounded operation may sit between Stage 1 and Stage 4.* Concretely:

* **IPFS add/pin/fetch moves out of `pinStorageUnitInner`'s git window.** The pass does git-only, then
  hands byte work to the job queue ([job_queue.mdx](./job_queue.mdx)) as ordinary background jobs. Byte
  results (a new CID, a `pinned_by` entry) are **outbox writes** (§19) picked up by the **next** pass —
  which is what the outbox is *for*. This decouples "our text is on the server" from "our bytes finished
  transferring", and those two have no reason to be coupled.
* **Anything in the window is bounded by the WINDOW BUDGET: 60 seconds.** Exceeding it is a logged WARN
  with the stage breakdown (§25). A pass that cannot finish inside the budget is a pass whose scope is
  wrong.

### 18.3 Window-of-conflict estimation (the math the design rests on)

With `N` parties each syncing on interval `T`, each holding the window open for `W`, the probability that
a given pass overlaps another party's window is approximately:

```
   P(collision per pass)  ≈  (N − 1) × W / T
```

| Case | N | T | W | P(collision) | Verdict |
|---|---|---|---|---|---|
| **v2, two computers** | 2 | 600 s | 5 s | **≈ 0.8 %** | Negligible. NFF retry absorbs it. |
| **v2, budget worst case** | 2 | 600 s | 60 s | **≈ 10 %** | Acceptable — NFF retry, no conflict. |
| **v2, five-person company** | 5 | 600 s | 5 s | **≈ 3.3 %** | Fine. |
| **v1, steady state** | 2 | 600 s | ~2 s | ≈ 0.3 % | Fine — **by luck**, no byte work pending. |
| **v1, first sync / big pull** | 2 | 900 s | **3600 s+** | **> 100 % — collision is CERTAIN** | The window exceeds the interval. Every pass collides; the retry re-pulls and re-collides. |

**That last row is the point.** When `W > T` the engine cannot converge — passes overlap continuously and
the storage is permanently behind. v1 reaches that state on any first sync. **v2's `W ≤ 60 s` budget with
`T = 600 s` keeps `W/T ≤ 10 %` structurally**, and it is the byte-work eviction (§18.2) that buys it.

**Corollary — a collision is not a conflict.** Overlapping windows cost a non-fast-forward, which Stage
4's retry absorbs silently. A *conflict* additionally requires **both parties to have edited the same
file** in that window. Since almost every file is self-owned or append-only (§21), the real conflict rate
is far below even these numbers — **except** for the shared, rewrite-in-place, non-union files, which is
exactly why §21 classifies them and §22 gives each one a rule.

### 18.4 Event-driven sync — closing the latency gap (NEW, fixes v1(b))

Timers alone gave us 1,933 artifacts riding on the device worker's coattails, and a 100-minute gap when
that worker died. v2 adds an **event trigger**:

* **On outbox commit** (§19.2), schedule a pass with a **20-second debounce** and a **120-second maximum
  delay**. A describe batch writing 300 descriptions produces **one** pass ~20 s after the last write, not
  300 passes and not a 10-minute wait.
* **The 10-minute timer stays** as the floor — it catches whatever the event path missed and is the only
  thing that pulls *in* remote changes when this machine is producing nothing.
* **The watchdog stays** and is promoted from backstop to **first-class**: per v1(e) it is already what
  keeps the product working.
* **Batch completion enqueues a pass.** The `jobqueue.service.ts:362-365` completion hook — which today
  only logs — commits the batch's outbox entries and schedules the pass. This is the single wire that was
  missing.

### 18.5 The stowaway defect — there is NO purposeful commit path (LOCKED, the root cause)

This section is the direct answer to *"files are left modified locally but never checked in
automatically — why?"* It is the most important finding in Part II.

#### 18.5.1 What actually happens today

**No code in LFB has "commit the user's finished work" as its purpose.** Not one function. The artifact
writers (`describe.service.ts:400`, `Transcribe.ts:217`, `ocr.service.ts:192`) write a file and return.
Nothing observes them, nothing counts them, nothing schedules anything.

Artifacts reach the server **only** as **stowaways** on a commit made for an unrelated reason: the
**device-registration worker** wakes every 10 minutes to update one small YAML — `devices/<self>.yaml` —
and on its way past runs the pathspec-less `git add -A` (`git.service.ts:491`), which happens to sweep up
whatever else is lying in the tree.

**Caught live, and this is the whole spec in one diff.** `git show --stat HEAD` after the 15:40:45 pass:

```
 12 × _Mirror/…/*.ai_description        | 2385 ++++++++++++++++++++++
 devices/bryan-mac-pro.yaml             |    2 +-      ← the ONLY file the pass INTENDED to commit
 13 files changed, 2386 insertions(+), 1 deletion(-)
```

Commit message: **`LFB: backbone device state`**. Two thousand three hundred and eighty-five lines of the
user's AI descriptions, committed under the title *"device state"*, by a worker that does not know they
exist, because a wildcard caught them.

**That is the defect.** Not the latency it produces — the **architecture**. The observed 10–30 minute
delay, the uneven gaps (14:22 → 14:52 = **30 min**, 15:09 → 15:30 = **21 min** on a *10-minute* worker),
and the "modified but never checked in" symptom are all downstream of one fact: **the commit is a side
effect, so its timing is governed by something with no relationship to the work.**

#### 18.5.2 Why this is fragile, not merely slow

A stowaway's survival depends on its carrier. **Every one of these severs the ride, and the artifacts then
sit uncommitted forever with no error anywhere:**

| # | Cause | Evidence | Diagnostic today |
|---|---|---|---|
| **F1** | `getGitBackboneRemote(id)` returns `null` → `gitBackbone = null`, `result.ran = false`, `writeSelfDevice` still runs, **nothing is ever committed** | `pin.service.ts:497-499` | **NONE — completely silent.** No log, no warning. `ran:false` is indistinguishable from "not attempted". |
| **F2** | Storage drops out of `safeStorageIds()` → the worker never visits it | `pin.service.ts:554-561` — `catch { return [] }` | **NONE — `[]` is a silent success.** |
| **F3** | Discovery loses the root. `discoverRoots` **hardcodes** `~/BGit/Bryan_git/personal_large_files_bridge` and types it by `basename` (`classifyByConvention`, `storage.service.ts:240`) — **and the live SDL has no `storage.yaml`** (v1(i)), so the *name* is the only identity. **Rename the directory → the storage ceases to exist.** | `storage.service.ts:248-271` | **NONE.** |
| **F4** | `resolveWorkingCopy` / `hasOrigin` returns null (no `.git`, clone failed) | `git.service.ts:260-273` | INFO only — invisible in practice. |

fallback at `storage-settings.service.ts:281-284` (personal/company whose root happens to contain `.git`).
Neither is a decision anyone made about *this* user's data. **This is the definition of working by luck.**

#### 18.5.3 The v2 rule (LOCKED)

> **Producing a durable artifact is what schedules its own synchronization. The commit is the
> WORK'S event, never another worker's side effect.**

Concretely — and these are requirements, not preferences:

1. **The write IS the trigger.** Every durable-artifact writer — describe, transcribe, OCR, compress,
   analysis, manifest, decisions — commits an **outbox entry** (§19.2) as the final, atomic step of its
   own operation. `markDurableArtifact(root)` is the natural seam: it already fires exactly where "this
   file is real and must persist" is known. **A writer that does not commit an outbox entry has not
   finished.**
2. **The outbox entry schedules the pass** (§18.4): 20 s debounce, 120 s max delay. A 300-file describe
   batch → **one** pass ~20 s after the last write. Not 300 passes. Not a 10-minute wait. Not a
   30-minute one.
3. **Batch completion forces a pass**, bypassing the debounce (`jobqueue.service.ts:362-365`).
4. **The device worker is decoupled from artifact sync.** It goes back to doing its own small job. It may
   still `git add -A` — that is a fine safety net — but **no artifact's delivery may depend on it.** If
   the device worker were deleted tomorrow, artifacts must still reach the server on time. **That
   sentence is the acceptance test for this whole section.**
5. **The pin worker's absence must be visible.** `pin_process: installed:false` (v1(b2)) is today an
   invisible, unremarked hole in the product's stated §7 behavior. Any worker that is `installed:false`
   while its storages have pending work **raises a warning** (§25.2).
6. **Every F1–F6 silent-null becomes loud.** A storage that is discovered but has **no resolvable
   backbone** is not a quiet no-op — it is a **warning** naming the storage and the reason
   (§25.2). `catch { return [] }` and `catch { return null }` on a **discovery** path are forbidden: an
   empty list must mean "there are none", never "something broke". **The engine may not confuse
   'nothing to do' with 'I failed to find out'.**
7. **Nothing is left dirty at the end of a pass.** A pass that ends with uncommitted SDL-root files it
   *could* have committed is a **failed pass** and says so (§25.2). "Modified locally, not checked in" is
   the state this engine exists to prevent, and it must be **impossible to reach silently**.

#### 18.5.4 Lock coalescing — the queue must not thrash (CHANGED)

`withStorageGitLock` (`pin.service.ts:371-383`) is an **unbounded FIFO, not a mutex with drop
semantics.** Under a long hold, every 10-minute tick **appends another closure**; nothing coalesces them.
When the lock frees they **all run**, back-to-back, each paying a full fetch + push. The watchdog then
sees the worker as overdue and calls `pushDeviceBackbone()` **in-process every 5 minutes**, appending yet
more waiters to the same starved chain — its single-flight (`ticking`) prevents overlapping *ticks*, not
accumulating *lock waiters*. `passInFlight` does **not** help: it is scoped to `pinAll` only
(`pin.service.ts:580-584`); **the device path has zero in-flight collapsing.** And `run-worker.mjs`
aborting its fetch at 60 s does **not** stop the work — Express does not propagate client disconnect, so
`internal.router.ts:31-35` keeps running and keeps enqueueing.

**v2:** the lock is **coalescing**. At most **one** pass may run and **one** may be queued per storage; a
request arriving when one is already queued **collapses into it** and returns the same promise. A pass is
a *reconciliation to current state*, not a work item — **running it twice is never more correct than
running it once.** With §18.2's window budget the hold is seconds, so the queue rarely forms at all.

## 19. The outbox — the queue-up directory (LOCKED, NEW)

*"When changes happen, we might well queue them up, so we might end up having a queue-up directory."*

Yes — and it is what makes §17.1's clean-tree precondition true. **This does not exist in v1** (verified:
no staging area, no pending-sync queue, no debounce on the git path; the "staging area" is git's own index
computed fresh at push time from whatever is dirty).

### 19.1 Where it lives

```
~/T/_large_files_bridge/outbox/<storage-id>/
├── 000000001737054012341-a3f2.entry.yaml
├── 000000001737054012389-b7c1.entry.yaml
└── payload/
    └── b7c1/_Mirror/Politics/.../Foo.mp4.ai_description
```

**Machine-local state root, never inside the SDL repo** — the same rule as the TO DO batches (§12). The
outbox is **this machine's private intent**; it must never be committed, never travel, and never appear in
a peer's tree. Its filename is `<zero-padded epoch-ms><short-random>.entry.yaml`, so a lexical sort **is**
commit order (§19.4) and two writers in the same millisecond cannot collide.

### 19.2 The producer contract (LOCKED)

**Producers never write into the SDL working copy.** A producer — the describe service, the transcriber,
OCR, the compressor, the decisions ledger, the manifest — writes its output as an **outbox entry**:

1. Write the payload to `outbox/<storage-id>/payload/<id>/<rel-path>` (a real file, on the same volume, so
   the drain is a rename).
2. Write the `.entry.yaml` **atomically last** (write-temp + `rename`). The entry's existence **is** the
   commit record; a crash mid-write leaves an orphan payload and no entry, which is harmless and swept
   (§19.5).
3. Signal the debounced pass (§18.4).

This inverts v1, where `describe.service.ts:400` writes straight into the SDL and hopes a timer notices.

### 19.3 Entry schema

```yaml
# outbox/personal/000000001737054012341-a3f2.entry.yaml
id: a3f2
seq: 1737054012341            # epoch-ms; the ordering key (§19.4)
storage_id: personal
device_id: fdc6e91d           # WHO produced it — becomes the tiebreak (§24.3)
class: artifact               # the DATA CLASS — selects the reconciliation rule (§21)
op: upsert                    # upsert | delete | append | observed
path: _Mirror/Politics/Charlie_Kirk_Mi/CIA/FINAL_CIA_Drones.mp4.ai_description
payload: payload/a3f2/_Mirror/Politics/Charlie_Kirk_Mi/CIA/FINAL_CIA_Drones.mp4.ai_description
produced_at: 2026-07-16T22:20:12.341Z    # LOGICAL time of production (§20.2)
content_hash: sha256:9f2c…    # dedupe + idempotence (§19.4)
source_hash: sha256:1a4b…     # the MEDIA file's hash this was derived from (§22.4)
attempts: 0
```

* **`op: observed`** is the Stage-1 escape hatch (§17.1): "this appeared in the working copy and I did not
  put it there." It carries the salvaged content so the tree can be reset to `HEAD` without data loss.
* **`produced_at` is the reconciliation clock (§20.2)** — the moment the work *happened*, not the moment
  it syncs. This is the whole reason the outbox can safely sit for minutes.

### 19.4 Draining (Stage 2)

* **In `seq` order, ascending.** Later work wins over earlier work from the **same** device
  deterministically, without consulting a clock.
* **Idempotent.** If the merged tree already holds `content_hash`, the entry is a **no-op** and is
  retired. This is what makes crash-and-retry free: a pass that dies after writing but before deleting
  simply re-drains and finds nothing to do.
* **Fail-closed per entry.** A failing entry increments `attempts`, stays in the outbox, and **does not
  block** the others. At `attempts >= 5` it moves to `outbox/<storage-id>/failed/` and raises a warning
  ([warnings.mdx](./warnings.mdx)). **Never silently dropped.**
* **Retire on success**, after the payload is in the tree — not after the push. The commit is git's job; a
  drained entry that fails to push is recovered by push-when-ahead (§17.4.4).

### 19.5 Durability (LOCKED — learned from a real defect)

The job queue is **in-memory with no persistence**, and a 4.1 GB V8 OOM once vaporized a 1,440-file
describe backlog silently. **The outbox must not repeat that mistake.** It is **on-disk, atomic, and
crash-survivable by construction**: entries are files, the rename is the commit, and startup **re-drains
whatever is there**. Orphan payloads with no entry (crash between step 1 and 2) are swept on startup after
a 1-hour grace. An outbox that is non-empty at boot is **normal**, not an error.

## 20. Schema requirements for reconcilable data (LOCKED)

*"Think of when you do data synchronization with timestamps. You can find out what's the newest."*

Only if the schema **carries** one. Per the v1 audit, most do not.

### 20.1 The mandatory reconciliation header (NEW)

**Every shared YAML the engine writes into a dedicated file repo must carry:**

```yaml
schema: 2                                  # schema version — REQUIRED for forward compat (§20.4)
updated_at: 2026-07-16T22:20:12.341Z       # UTC, ISO-8601, millisecond precision, ALWAYS 'Z'
updated_by: fdc6e91d                       # the device id that wrote it — the deterministic tiebreak
```

Absent that header, "who is newest" is **unanswerable** and the only honest resolution is keep-both (§23).
This header is what makes §22's matrix decidable.

**Self-owned** files (§22.1) need it too — not to arbitrate (nobody contends) but so a peer can render
"last seen 4 minutes ago" and detect a stale device.

### 20.2 Logical time, not wall time (LOCKED)

`updated_at` is the **`produced_at` of the outbox entry that produced the value** (§19.3) — *when the work
happened*, not when it drained. A description generated at 22:20 and synced at 22:31 carries **22:20**.
Otherwise a machine that syncs late would fraudulently "win" against fresher work from a machine that
synced promptly, and **the slowest machine would dominate the network** — precisely backwards.

### 20.3 Clocks are not trusted (LOCKED)

Two computers **will** disagree, by seconds normally and by hours after a BIOS reset, a bad VM restore, or
a timezone-mangled write. **Timestamps are a HINT, never an authority.** Therefore:

* **Never** "newest timestamp wins" on data we cannot regenerate. Time only breaks ties **after** the
  content-based rules of §22 have run.
* **Reject the impossible.** An `updated_at` more than **24 h in the future** relative to local time is
  **not trusted**; it sorts as its *received* time and raises a clock-skew warning (§25). This defeats the
  "one machine's clock is set to 2038 and wins forever" failure mode.
* **Prefer content over time.** Equal `content_hash` → not a conflict at all, regardless of timestamps.
  This resolves the overwhelming majority of apparent conflicts before time is ever consulted.

### 20.4 Schema version and forward compatibility (NEW)

Two computers **will** run different LFB versions — one updates, the other is a tower that has been on for
a month. So:

* **`schema:` is required on every shared file.** A reader seeing `schema:` **greater** than it understands
  must **not** rewrite the file. It reads what it can, and treats the file as **read-only** until upgraded,
  raising a warning. **A newer peer's data is never downgraded by an older peer** — an older writer
  round-tripping a newer file would silently drop the fields it does not know, which is data loss wearing a
  merge's clothes.
* **Unknown keys are PRESERVED on rewrite.** Any writer that must rewrite a file it partially understands
  round-trips unknown keys verbatim. This is the single cheapest forward-compat guarantee available and it
  is mandatory.
* `schema: 1` is assumed when the key is absent (every v1 file on disk today).

### 21.2 The timestamp gap, stated plainly

`writeYaml` stamps `updated_at` on every write it performs:

```ts
// yaml-store.ts:63
export function writeYaml<T extends Record<string, unknown>>(file: string, value: T): void {
  ensureDir(path.dirname(file));
  const stamped = { ...value, updated_at: new Date().toISOString() };
```

**That covers only #1, #3, #5, #15.** Everything git-tracked deliberately **bypasses** it, for a good
reason — `manifest.service.ts:1-7` explains that a volatile `updated_at` would make an unchanged list
re-serialize to different bytes, producing spurious commits and conflicts on every pass.

**That reason is sound and v2 keeps it (§22.3): a volatile timestamp on a union-merged append log is
actively harmful.** But it was applied too broadly. The correct split is:

* **Class B (append-log)** — **no** file-level `updated_at`. Per-**entry** timestamps only. A stable file
  serialization is what lets `merge=union` work. (#6, #7, #17, #19)
* **Class D (shared-mutable)** — **must** carry the §20.1 header. Stamped only when content **actually
  changed** (compare `content_hash` before writing; unchanged → don't write, don't stamp). This gets
  reconcilability **without** reintroducing churn. (#2, #3, #5, #8, #21)
* **Class C (derived)** — the **source** hash matters, the write time does not (§22.4).

## 22. Reconciliation — every permutation (LOCKED)

The complete matrix. `L` = our local value, `R` = the peer's (merged-in) value, `B` = the merge base.

### 22.0 The universal pre-checks (run first, in order, for every class)

| # | Condition | Outcome |
|---|---|---|
| 0.1 | `hash(L) == hash(R)` | **Not a conflict.** Identical content. Keep either. Never consult time. Resolves most apparent conflicts. |
| 0.2 | `L == B` (we didn't touch it) | **Take R.** Fast-forward. |
| 0.3 | `R == B` (they didn't touch it) | **Keep L.** |
| 0.4 | `schema(R) > schema(ours)` | **Take R untouched, do not rewrite.** Warn. (§20.4) |
| 0.5 | `schema(L) > schema(R)` | **Keep L.** The peer is stale; it will catch up. |
| 0.6 | Both changed, both understood | → the class rule below. |

### 22.1 Class A — self-owned (`devices/<self>.yaml`, `history/<device>.txt`)

| Permutation | Resolution |
|---|---|
| Only we wrote it | Keep ours. **Contention is impossible by construction.** |
| A peer edited **our** file | **They violated the contract.** Ours wins — we are the sole authority on our own hardware. Overwrite, log a WARN. |
| Two devices minted the **same** id | **Pathological** (a cloned disk). Detect via a differing hardware fingerprint under one id; the device that notices **re-mints** its id, writes the new file, and retires the old (§22.7). |
| Their file, our tree | **Never touch it.** Read-only to us. |

**No timestamps needed.** This is why the device registry has "never conflicted" — and why so much rides
on it (v1(a)): the registry is the only thing v1 reliably syncs.

### 22.2 Class B — append-log (`decisions.yaml`, `owner_map.yaml`, sidecar `events[]`, `.gitattributes`)

| Permutation | Resolution |
|---|---|
| Both appended different entries | **Union.** Both survive. `merge=union` handles it in git; our fold handles semantics. |
| Both appended the **same** entry | Union yields a duplicate → **the fold dedupes** by entry identity. |
| Same **key**, different **values**, different times | Both entries survive in the log; **the FOLD picks the winner** — latest `decided_at`, ties broken by `device_id` (§24.3). **The log is never lossy; only the projection chooses.** This is the property that makes Class B safe. |
| One party **deleted** a line | Union **resurrects** it — a known `merge=union` behavior. **Therefore: never delete from an append log.** Append a **tombstone** entry; the fold honors the newest. |
| Log grows unboundedly | Compaction is **allowed only** when this device is the sole party with unpushed work and only for entries older than **90 days** superseded by a later entry. **Compaction is a rewrite (Class D) and must be pushed as its own commit**, never bundled. |

#### 22.2.1 The sidecar mirror bug (v1 defect, must fix)

`mirrorToSyncRepo` (`tracking-sync.service.ts:68`) uses `copyFileSync` — whole-file last-writer-wins — and
the code concedes it (`:83`): *"Append-only sidecars / history and union-mergeable decisions/manifest
tolerate a last-writer copy for the common case; a richer per-file merge is a later refinement."*

**It does not tolerate it.** A sidecar's `events[]` is Class B; a whole-file copy **silently drops a
peer's events** — real data loss, in the one direction (invariant 1, §16.3) v2 forbids. **v2: the mirror
must union `events[]` by entry identity, not copy the file.** This is the "later refinement", now due.

### 22.3 Class B special — `manifest.yaml` (the IPFS pin state)

The manifest is an append log whose entries mutate (`pinned_by` grows as computers pin). Its "no volatile
timestamp" rule (`manifest.service.ts:1-7`) is **correct** and stays.

| Permutation | Resolution |
|---|---|
| Two devices pin the **same** file → same CID | **Not a conflict.** Content-addressing means agreement is automatic. `pinned_by` **unions**. |
| Two devices pin the same file → **different** CIDs | The file genuinely differs (one compressed it). **Both CIDs survive** as separate entries; §22.4's derived rules decide which is current. **Never drop a CID** — a dropped CID is an unpinnable, unrecoverable file. |
| One **unpins** while another **pins** | **Pin wins.** Asymmetric on purpose: a spurious pin costs disk, a spurious unpin costs **the file**. |
| `pinned_by` diverges | **Union.** It is a set of facts, each asserted by one device about itself. |
| One removes an entry | **Only its own `pinned_by` membership.** No device may remove another's, or the entry itself. |

### 22.4 Class C — derived artifacts (transcripts, descriptions, OCR, analysis, `files.yaml`)

The most common class (1,933 files and counting) and, happily, the **easiest**: derived data is
**regenerable**, so "losing" a conflict costs CPU, not data.

**The keying rule (LOCKED):** a derived artifact is identified by `(source_path, source_hash, generator,
generator_version)`. Two artifacts with the same key are **interchangeable regardless of content** — a
transcript re-run is nondeterministic in wording but equivalent in meaning.

| Permutation | Resolution |
|---|---|
| Same key, different bytes (both re-ran the same source) | **Keep either — deterministically the same one on every machine.** Lowest `device_id` lexically wins (§24.3). **Never both**, or the tree ping-pongs forever. Not a conflict; a duplicate. |
| Different `source_hash` (the media changed) | **Newest `source_hash` wins** — matched against the media file **as it exists now**. If neither matches the current media, both are **stale**: keep the newer, mark the artifact for **regeneration**. |
| Different `generator_version` (`qwen3-asr` vs. `whisper-base`) | **Higher-fidelity generator wins**, by the engine ranking in [transcribe_engine.mdx](./transcribe_engine.mdx) — *not* by timestamp. A tower's qwen3 transcript must not lose to a laptop's later whisper fallback. **This permutation is why "newest wins" is wrong as a global rule.** |
| One party has it, the other doesn't | **Take it.** The entire value proposition: *generate once, reuse everywhere.* |
| One party **deleted** it | Deletion of a derived artifact is a **local** act (disk pressure), **never a network assertion**. **The surviving copy wins and is restored.** To truly remove one, use a Class B decision, not `rm`. |
| Both empty | **Empty is success** for OCR (a text-free image). Byte-identical → §22.0.1. |

#### 22.4.2 `files.yaml` — the sharpest hazard (v1 defect, must fix)

Per §21: shared, rewritten wholesale every scan, **no `updated_at`, no device id, no fold rule, and
absent from `UNION_MERGE_PATHS`** (which lists only SyncList/manifest/decisions/owner_map,
`git.service.ts:30-38`). **Two devices indexing the same SDL produce a straight textual conflict with
nothing to reconcile on** — and its per-entry `modified`/`created` are the **media's filesystem mtimes**,
not write times, so they are useless for LWW. It survives today only because it is **never written** in
the live repo (v1(i)) — a defect masking a defect.

**v2:** `files.yaml` is **Class C — a per-device index of what THIS device sees**, not a shared truth.
Either (**preferred**) split it per device — `files/<device-id>.yaml`, promoting it to **Class A** where
contention is impossible — or keep one file and merge **per-entry, keyed by path**, taking the entry whose
`source_hash` matches local reality and unioning the rest. **Never** whole-file LWW.

#### 22.4.3 `repo_storage.yaml` — stop the churn at the source

`refreshCounts` (`repo-storage.service.ts:213`) re-stamps `last_scan.at` on **every** scan and the mirror
carries it up — a **guaranteed** conflict on a file whose content did not meaningfully change. v1 concedes
this with `autoResolveRepoStorageConflicts` (`git.service.ts:450`), which `git rm -f`'s the file. That
works, but it treats a symptom.

**v2:** `last_scan.at` is **per-device state and does not belong in a shared file** — it moves to
`last_scan: { <device-id>: { at, counts } }` (Class B, union by device key). The remaining content is
stable, so the file stops conflicting. **Keep the `git rm -f` escape hatch** — it is correct, cheap, and
right (the file is Class C, regenerable from Local Storage) — but it should stop firing.

### 22.5 Class D — shared-mutable (`storage.yaml`, `mapped_dirs.yaml`, `bookmarks.yaml`, `decisions_policy.yaml`)

The genuinely hard class: multiple devices rewrite the same keys, and the content is **not** regenerable.
Small, rare, and human-set — which is exactly why losing a write here is unacceptable.

#### 22.5.1 Field-level, not file-level (LOCKED)

**Never resolve a Class D file as a blob.** Merge **field by field**; two devices editing *different*
fields of the same file is **not a conflict** and must never be treated as one.

| Permutation | Resolution |
|---|---|
| Different fields changed | **Merge both.** No conflict. The common case. |
| **Same** field, same value | Not a conflict (§22.0.1). |
| **Same** field, different values, different `updated_at` | **Later `updated_at` wins**, subject to §20.3's skew rejection. The loser's value is **recorded in the file's `superseded[]` log** — never vanished (§23.3). |
| Same field, `updated_at` within **2 s** (a genuine race) | **Deterministic tiebreak by `device_id`** (§24.3), *not* by clock. Sub-2-second clock deltas are noise. |
| Same field, one side's `updated_at` **> 24 h in the future** | **Untrusted.** The other side wins; clock-skew warning (§20.3, §25). |
| Neither has `updated_at` (a v1 file, e.g. `storage.yaml`) | **Cannot arbitrate.** Fall to **keep-both** (§23.3) and stamp both going forward. **This is why §20.1 is mandatory.** |
| One side **deleted** a field | Deletion is a **tombstone** (`field: null` + `updated_at`), never an absence. **Absence means "I don't know", not "delete it"** — an older peer that doesn't know a field must not delete it (§20.4). |

#### 22.5.2 Set-valued fields are unions, not scalars

`mapped_dirs.yaml` and `bookmarks.yaml` hold **sets**. A set is Class B in Class D clothing:

* **Add + add** → **union**. Two computers mapping two different directories both win. Treating this as
  LWW would silently unmap a directory — a top-tier data-loss bug.
* **Add + remove** → **add wins** (asymmetric like §22.3's pin/unpin: a spurious mapped dir costs a scan,
  a spurious unmap costs *visibility of the files*).
* **Real removal** → a **tombstone** entry with `removed_at`, honored by the fold, garbage-collected after
  90 days.

### 22.6 Class E — local-only (must never enter the repo)

TO DO batches, logs, the outbox, `.lfbridge/` **inside an SDL**, `.transcribe/`, `/tmp` scratch.

| Permutation | Resolution |
|---|---|
| A Class E path appears in the SDL | **Not a merge — a bug.** Remove it locally, log it, do not commit it. |
| It is already **committed** (the live state, v1(i)) | **Remove in a dedicated commit**, `LFB: retire local-only path <p>`. This is the **only** sanctioned deletion the engine performs unprompted, and it is legitimate **only** because the path can never have been anyone's data. |
| A peer commits one | Same. Removal is safe: it was never shared state. |

**The live `.lfbridge/` migration warning must stop repeating.** v1 logs the same
`[migrate] conflict: … already exists — keeping it and LEAVING … in place` on **every pass**, forever
(v1(i)). Warning without resolving is not caution; it is noise that trains people to ignore warnings. **v2:
resolve it once** — the root file wins (§1.1), the `.lfbridge/` copy is retired via the rule above — or
**escalate it once** to a real warning ([warnings.mdx](./warnings.mdx)) and stay quiet thereafter.

### 22.7 Device identity permutations (`devices/`)

The rename permutation below is a **hazard to close, not a live defect**. An earlier draft opened this
section by asserting the live repo had "two files for one computer" from a rename — **it does not**: the two
files are two real machines with different minted ids, both advancing (v1(i)). The rule stands on its own
merits regardless, because **the id is the identity and the filename is only a convenience** — which is
precisely why the false reading was possible, and why a peer must never infer identity from a filename.

| Permutation | Resolution |
|---|---|
| Device **renamed** (`nice_name` changed) | The **id** is the identity, the filename is a **convenience**. On rename, write the new file **and `git rm` the old one in the same commit**. Peers see a rename, not a birth. |
| An orphan file with **our** id, different filename | **We own it — retire it.** Only the device itself may do this (Class A). |
| An orphan whose device hasn't reported in **90 days** | **Do not delete.** Mark `stale: true`, surface it in the Devices tab ([devices.mdx](./devices.mdx)), let the **user** decide. A tower off for a season is not a dead tower. |
| Same **hardware fingerprint**, two ids | A restored/cloned disk. Surface as a **possible duplicate**; never auto-merge — the ids may be legitimately distinct machines from one image. |
| Device file for an id **we have never seen** | **Normal.** That is discovery working (§4.3). |

### 22.8 `owner_map.yaml` — the consent-gated exception (company repos)

Class B, union-merged, **but the fold does not auto-apply**. An ownership assertion travels; it applies on
another member's machine **only via review-and-consent** ([repo_owner_propagation.mdx](./repo_owner_propagation.mdx),
§14.1). **This is the one place where "the engine converges automatically" stops at the security boundary**
— convergence of the *assertion* is automatic; convergence of the *effect* requires a human. Merging it is
never a decision to obey it.

## 23. Headless conflict resolution — no user interface, ever (LOCKED)

*"How do you resolve conflicts without a user interface?"*

**Requirement.** These repos are LFB's own (§16.1). There is no branch a human is nursing, no PR, no
reviewer. A pass runs from launchd at 03:00 with nobody logged in. **A resolution path that requires a
human is not a resolution path** — it is the storage silently stopping, which is the exact failure the
watchdog exists to paper over. v1 has **exactly one** headless resolution (`repo_storage.yaml`, v1(h));
every other conflict runs `git merge --abort` (`git.service.ts:424`) and waits for a terminal that may
never open.

### 23.0 The two absolute prohibitions

1. **Never `git push --force` / `--force-with-lease`.** Not on retry, not on "obviously stale", not ever.
   v1 honors this (verified: no force push anywhere) — **keep it that way.**
2. **Never resolve by discarding.** No `git checkout --theirs/--ours` on data we did not author, no
   `reset --hard` on unpushed work, no `stash` (it strands data in a ref nothing reads). The **one**
   exception is Class E (§22.6), where the path can never have held anyone's data.

### 23.1 The resolution ladder

Applied in order. **Every rung is deterministic** — two computers running it on the same inputs reach the
same answer without talking, which is what makes convergence possible at all.

| Rung | Test | Action |
|---|---|---|
| **1** | Identical content (§22.0.1) | **Not a conflict.** Most cases die here. |
| **2** | One side unchanged from base (§22.0.2-3) | **Take the changed side.** |
| **3** | Class B (append-log) | **Union**, then fold. Structurally cannot conflict. |
| **4** | Class A (self-owned) | **Owner wins.** No arbitration. |
| **5** | Class C (derived) | **Key comparison** (§22.4) → keep one, or **regenerate** and be certain. Costs CPU, never data. |
| **6** | Class D, different fields | **Field-level merge** (§22.5.1). |
| **7** | Class D, same field, trustworthy timestamps | **Later `produced_at` wins**; loser → `superseded[]`. |
| **8** | Class D, same field, untrustworthy/absent timestamps | **Deterministic tiebreak** by `device_id` (§24.3); loser → `superseded[]`. |
| **9** | Nothing above applies | **KEEP BOTH — quarantine** (§23.3). |

**Rung 9 is the floor, and it is why the ladder is total.** There is no rung 10 and no "ask the user"
branch. Every conflict has a defined, terminating answer.

### 23.2 Regeneration beats arbitration (Class C)

For derived data, the **cheapest correct** answer is often "stop arguing and recompute." If two transcripts
conflict and the ladder can't cleanly pick, **regenerate from the source media** and write a fresh
artifact with a current `source_hash`. Deterministic, requires no peer agreement, costs CPU — and CPU is
the thing we have. This is available **only** for Class C, which is precisely why the classification in
§21 is load-bearing.

### 23.3 Quarantine — the floor (NEW)

When rung 9 is reached, **both versions survive**:

```
<path>                          ← the ladder's best guess (deterministic: lowest device_id, §24.3)
<path>.conflict-<device-id>-<ts>   ← the other version, committed alongside
```

* **Both are committed and pushed.** The conflict travels; every machine sees the same two files and the
  same winner. **Convergence is preserved even though the semantic question is unanswered** — this is the
  key move that makes headless operation safe.
* **A warning is raised** ([warnings.mdx](./warnings.mdx)) with both versions and a one-click resolution.
  The **user is informed, never blocked** — the difference between "no UI" and "no transparency".
* **The engine never opens a `.conflict-*` file again.** It is inert data awaiting a human.
* **`git merge --abort` is no longer a resting state.** v1 aborts and stops (`git.service.ts:424`); v2
  aborts only to take control, then resolves via the ladder and **always** ends Stage 4 pushed or
  quarantined.

### 23.4 Why this is safe here and nowhere else

Every rung's authority — union-merge, deterministic overwrite, regeneration, quarantine commits — rests on
**LFB being the sole author** (§16.1). In a **working repo** the same rungs would be vandalism: rung 4
would overwrite a colleague's work, rung 9 would litter their tree. **The classification of the repo is the
license for the algorithm.** A bug that misclassifies a working repo as a dedicated one is a **Sev-1**, not
a papercut.

## 24. Priority — who wins (LOCKED)

*"Who has priority to win?"*

### 24.1 The priority ladder (in order, first applicable wins)

1. **The truth on disk beats every claim about it.** If the media file's hash contradicts a YAML, **the
   filesystem wins** and the YAML is stale. No timestamp overrides physical reality.
2. **The owner wins for self-owned data** (§22.1). Your computer is the sole authority on its own hardware.
3. **The user's explicit decision beats any engine inference.** A folded decision
   ([decisions.mdx](./decisions.mdx)) beats a recommendation, always, on every machine.
4. **Safety-asymmetric operations beat their inverse.** Pin beats unpin; add beats remove; keep beats
   delete (§22.3, §22.5.2). Always resolve toward **more data retained**.
5. **Higher fidelity beats newer** for derived data (§22.4): a qwen3 transcript beats a later whisper one.
6. **Later logical time beats earlier** (§20.2) — *only* among trustworthy timestamps (§20.3), *only* for
   Class D, and *only* after 1–5.
7. **Deterministic tiebreak** (§24.3) — never a coin flip, never wall-clock arrival order.

**No device has standing.** There is no primary, no server-of-record, no "the tower wins." Every party is
equal; the **data class** decides, not the machine. A laptop that syncs once a week is exactly as
authoritative about its own facts as the tower that syncs every ten minutes.

### 24.2 Explicitly rejected priority rules

* ❌ **"Newest wins"** globally — breaks §24.5 (a stale machine's late sync would beat fresh work) and
  §22.4's fidelity rule.
* ❌ **"The pushing device wins"** — makes the *fastest network* authoritative. Absurd, and it is
  effectively v1's accidental behavior under `git add -A` + LWW mirrors.
* ❌ **"The biggest/most complete file wins"** — a truncated 5 GB write would beat a correct 2 KB YAML.
* ❌ **"Ask the user"** — not available (§23).

### 24.3 The deterministic tiebreak (LOCKED)

When the ladder reaches rung 7, compare **`device_id` lexically; the LOWEST wins.**

Arbitrary — deliberately. Its virtues are the only ones that matter here: **total** (any two ids compare),
**stable** (same answer forever, so no ping-pong), and **agreed without communication** (both machines
compute the same winner in isolation). The loser is **never discarded** — it lands in `superseded[]`
(§22.5.1) or quarantine (§23.3).

### 24.4 The loser is always recoverable

The engine may **choose**; it may not **destroy**. Every rung that picks a winner **records the loser** —
`superseded[]`, a `.conflict-*` file, or an append-log entry that survives the fold. **A user must always
be able to answer "what did the other computer say?"** This is invariant 1 (§16.3) made operational.

## 25. Observability and failure modes (LOCKED)

Per v1, this engine's failures are **silent** — a 100-minute stall, a forever-repeating migration warning,
a discarded push error, and six SDL files that were never written at all.
**Silence is the defect.**

### 25.1 Every pass emits a structured record

```yaml
# ~/T/_large_files_bridge/sync/<storage-id>/last_pass.yaml
started: 2026-07-16T22:20:12.341Z
stages: { pull_ms: 412, drain_ms: 88, resolve_ms: 0, push_ms: 640 }
window_ms: 1140                 # merge → push. THE metric (§18.2). Budget: 60000.
outbox: { drained: 47, failed: 0, remaining: 0 }
merged: { fetched: true, took_remote: 3, conflicts: 0, quarantined: 0 }
committed: "LFB: 47 AI descriptions, device state  (bryan-mac-pro)"
pushed: true
problem: null
```

### 25.2 The conditions that must raise a warning

| Condition | Why it matters |
|---|---|
| `window_ms > 60000` | The §18.2 budget is blown; collision probability is climbing (§18.3). |
| Outbox non-empty for **> 30 min** | Convergence has stopped. **This is the reported symptom** (v1(a)) — v2 must *say so* instead of quietly deferring. |
| Any entry in `outbox/*/failed/` | Data is stuck and will never sync unaided. |
| A pass **stalled > 3 × interval** | The v1(e) 100-minute gap. The watchdog already detects it; it must also **warn the user**, not just self-heal quietly. |
| Any `.conflict-*` file exists | A human owes a decision (§23.3). |
| Clock skew **> 24 h** | Reconciliation is compromised (§20.3). |
| A Class E path is committed | The live `.lfbridge/` + `.transcribe/` defect (v1(i), §22.6). |
| A device file orphaned by rename | §22.7. A HAZARD, not a live defect — the two device files in the live repo are two real computers (v1(i)). |
| Push failed **after all retries** | Ordinary (offline) — but must be **visible**, not swallowed (v1(g)). |
| **Uncommitted files at the SDL root** older than 30 min | The direct, checkable form of *"files are not getting checked in."* |

### 25.3 The cross-process lock (CHANGED — fixes v1(f))

`withStorageGitLock` (`pin.service.ts:371-383`) is in-process, and
`owner-propagation.service.ts:59-64` runs a **second, uncoordinated** chain over the same working copy.

**v2:** one **file-based** lock per working copy — `<state>/sync/<storage-id>/git.lock` holding pid +
started-at, with a **10-minute stale takeover**. Every git-touching path acquires it: the pin cycle, the
device worker, owner propagation, and the CLI. A stale lock from a crashed process must never wedge the
storage forever (invariant 2, §16.3).

## 26. Acceptance criteria — the v2 synchronization engine

14. **Scope.** Part II governs **only** dedicated LFB file repos — the Personal SDL and company sync repos
    (§16.1), keyed on the resolved **storage kind**, never on a path or name. A **working repo** never runs
    this engine; misclassifying one is a **Sev-1** (§23.4).
15. **Four stages, in order** (§17): **PULL** (into a **clean tree** — a conflict here is a bug) →
    **WRITE+RECONCILE** (drain the outbox against the merged tree) → **RESOLVE** (headless) → **PUSH**
    (commit + push; on non-fast-forward, re-enter Stage 1; **≤ 3 retries** with backoff; **never
    `--force`**).
16. **Producers never write into the working copy** (§19.2). They write **outbox** entries to
    `~/T/_large_files_bridge/outbox/<storage-id>/` — machine-local, **never committed**, atomic
    (write-payload → rename-entry), **crash-survivable and re-drained at boot** (§19.5), drained in `seq`
    order, idempotent by `content_hash`, **failing entries quarantined at 5 attempts and never silently
    dropped** (§19.4).
17. **The conflict window excludes byte work** (§18.2). No unbounded operation may sit between Stage 1 and
    Stage 4; IPFS add/pin/fetch moves **out** of the git window into background jobs, with results
    returning via the outbox. **Window budget: 60 s**; exceeding it warns. This repeals the v1 shape where
    `pull` (pin.service.ts:412) and `commitAndPush` (pin.service.ts:462) straddle an unbounded
    `runUnitPin` (v1(d)) — the case where `W > T` and convergence becomes **impossible** (§18.3).
18. **Sync is event-driven with a timer floor** (§18.4): an outbox commit schedules a pass with a **20 s
    debounce / 120 s max delay**; **batch completion enqueues a pass** (the missing wire — today
    `jobqueue.service.ts:362-365` only logs, v1(b)); the **10-minute** device timer and the **watchdog**
    remain as the floor. A change reaches the server within **10 minutes**.
19. **Every shared file carries the reconciliation header** (§20.1): `schema`, `updated_at` (UTC ISO-8601
    ms, `Z`), `updated_by` (device id). `updated_at` is **logical time** — the `produced_at` of the work,
    not the sync time (§20.2), so a late-syncing machine can never dominate. **Class B append-logs are
    exempt** at file level and stamp **per entry** (§21.2) — a volatile file-level stamp would break
    `merge=union` and re-churn (`manifest.service.ts:1-7`, correct and retained).
20. **Clocks are a hint, never an authority** (§20.3): content-equality is checked **before** time; an
    `updated_at` **> 24 h** in the future is untrusted and warns; **never** "newest wins" on data that
    cannot be regenerated.
21. **Schema versioning is enforced** (§20.4): a reader seeing a **higher** `schema` **must not rewrite**
    the file and treats it read-only; **unknown keys are preserved verbatim** on any rewrite. **An older
    peer may never downgrade a newer peer's data.**
22. **Every kind of synced data is classified** (§21) as **A** self-owned / **B** append-log / **C**
    derived / **D** shared-mutable / **E** local-only, and the class selects the reconciliation rule.
    Named v1 defects that **must** be fixed: **`files.yaml`** (shared, timestamp-less, non-union, whole-file
    rewrite → per-device or per-entry merge, §22.4.2); **`repo_storage.yaml`** churn (`last_scan.at`
