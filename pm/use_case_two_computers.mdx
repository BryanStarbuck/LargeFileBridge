# LargeFileBridge — Two-Computer Tour (Mac tower ↔ Mac laptop)

This is a **chronological scenario tour**. Where [use_cases.mdx](./use_cases.mdx) answers *"which screen
answers which question"* with UC-N flows, this document answers *"what actually happens, in order, as one
repo lives on two of my computers over five months"* — and in doing so it exercises **every** on-disk
write and **every** decision surface a repo storage has. It is the reference the team walks to confirm
the whole system holds together.

**The authority for the cross-computer mechanics is
[storage_company.mdx §8](./storage_company.mdx#8-the-load-bearing-use-case--one-file-two-computers-locked).**
That section LOCKS *how* one file reaches a second computer — the six-link chain, the remote-derived
mirror key, the auto-adopted SDL sync repo with the mirror **default ON**, merge-not-copy on receive, and
the new `FileRow.presence: "remote-only"`. This tour **agrees with it and does not restate it**: §9 below
walks that exact journey step by step with the real machines, and every rule it leans on is cited back to
§8 rather than re-argued here. Where this tour's earlier months describe the mechanics in older wording
(the mirror subtree key, the per-repo toggle's default), §8 **supersedes** it — flagged inline.

Two computers, one repo, five months:

* **`studio-tower`** — the user's **day-to-day Mac tower** (a desktop, not a laptop). Their main machine.
* **`bryan-macbook-pro`** — their **Mac laptop**, enlisted in the same repo a month later.

(The unique per-computer names come from [devices.mdx §1/§8](./devices.mdx) — a stable minted device id
plus a fingerprint-seeded nice name. Those nice names are the filenames of the per-computer history logs
and the `on_device` stamp on every event.)

This spec builds on the siblings and does not repeat them:

## 0. The setup — a repo storage, and what counts as "special"

**A storage is a section of the file system LFB tracks.** The simplest kind is a **repo storage**: the
**root directory where a repo is cloned locally, plus the directory hierarchy underneath it that stays
within that repo** ([storage_repo.mdx](./storage_repo.mdx)). If a **sub-repo** is cloned inside — a rare
corner case — that cloned sub-hierarchy is carved out as its own storage. (The rest of the file system —
everything *outside* repos — is the **personal** file system, handled later; this tour is only about
**repo storage**.)

Inside a repo, LFB only busies itself with **special files** ([special_files.mdx](./special_files.mdx)).
A file is special when it is **large** (payload rule), **already IPFS-pinned** (by us or observed pinned
outside us), or a **video / image / audio** file. For media it also asks two questions:

* **Compressible?** — does the video/image/audio look **uncompressed** vs. the learned baseline?
* **Transcribable?** — a **video has an audio track** so it can be transcribed; **audio** can be, too
  (optionally). Both may or may not end up transcribed.

The repo in this tour is the **Charlie Kirk repo** — a folder full of video clips.

**Where tracking state lives — and how it travels.** Under the
[data-category placement policy](./artifact_placement_policy.mdx), none of LFB's tracking state
(`repo_storage.yaml`, the per-file sidecars, `history/<device>.txt`, `decisions.yaml`, `manifest.yaml`) ever
lives in the working repo — it lives in **Local Storage** at `~/T/_large_files_bridge/repos/<repoKey>/`
(`repoKey = shortHash(resolve(repoPath))`), machine-local and never touching git. Because Local Storage does
**not** travel between computers, the two machines in this tour share tracking state **only** through a
**sync repo**: the user has configured one on the owning **Personal** storage (*"a Company repo only for the
Large File Bridge webapp to sync changes"*) and turned on the Charlie Kirk repo's per-repo **sync toggle**
([repo_settings.mdx §2](./repo_settings.mdx), default OFF). With that on, each machine **also** mirrors its
`repos/<repoKey>/` subtree to `<syncRepo>/repos/<repoKey>/`, so the tower's manifest, decisions, and sidecars
reach the laptop. (Absent the sync repo, each computer would simply keep its own Local-Storage copy and the
February pull-down reconcile below would have nothing to reconcile against.)

**Superseded in two details** by
[storage_company.mdx §8.4.1/§8.4.2](./storage_company.mdx#841-a-repos-shared-identity-is-its-remote-never-its-path-locked),
which is the authority:

* **The mirror subtree is `<syncRepo>/repos/<repoUid>/`, not `repos/<repoKey>/`** — `repoUid` is derived
  from the repo's **normalized git remote**, because `repoKey` is a hash of an **absolute path** and the
  two computers therefore wrote and looked in **different directories**. Local Storage stays path-keyed
  (`~/T/_large_files_bridge/repos/<repoKey>/`) on purpose; only the **mirror** changes key.
* **The per-repo mirror is ON by default**, and the owning company storage's git-backed SDL is
  **auto-adopted** as the sync repo — the user does not have to find and flip a setting to make the
  product's core promise work. The per-repo toggle survives as an **opt-out**.

## 2. JANUARY — `studio-tower` — enlist the Charlie Kirk repo

**What the user does.** On their Mac tower they clone the Charlie Kirk repo and add it to LargeFileBridge
(**+ Add repo** — [repos.mdx §6](./repos.mdx)). The repo has **20 video files** now; it will grow to
**100**. Separately, **before or outside LFB, the user pinned 5 of the 20 by hand on the command line**
(`ipfs add` / `ipfs pin add`).

**What LFB does automatically.**

* **Creates the Local-Storage tracking area.** The web app **automatically** creates
  `~/T/_large_files_bridge/repos/<repoKey>/` (`repoKey = shortHash(resolve(repoPath))`) and **automatically**
  writes **`repo_storage.yaml`** there — the repo-wide settings-and-state file whose **only** level-one key is
  `repo_storage:`, everything else nested below ([repo_tracking_scheme.mdx §2](./repo_tracking_scheme.mdx)).
  **Nothing is written into the repo's working tree** — no `.lfbridge/`, no `.gitignore` edit. Enlist
  provenance (`at` / `by` / `on_device: studio-tower`) is stamped.
* **Creates the history directory** and this computer's log in Local Storage, **named by the unique computer
  name**: `~/T/_large_files_bridge/repos/<repoKey>/history/studio-tower.txt` — the **history log for this
  computer for this repo** ([repo_tracking_scheme.mdx §4](./repo_tracking_scheme.mdx)).
* **Mirrors to the sync repo.** Because the repo's sync toggle is on (§0), each of these Local-Storage writes
  is **additionally** mirrored to `<syncRepo>/repos/<repoKey>/`, so the laptop will see them in February.
* **Classifies the files.** All 20 are `video` (and `large` per the media-at-any-size payload rule); 19
  are compressible, all 20 transcribable; **5 are `ipfs_pinned`** — but LFB did **not** pin them.
* **Notices the outside pins (headless-safe).** When LFB scans — including in **headless** background
  mode — it reads this computer's IPFS pinset and finds **5 files already pinned that LFB has no record
  of pinning**. For each it writes a **sidecar** (in Local Storage) and appends an **`observed`** event with
  **`by: not-lfbridge`**, the CID it is pinned under, and **`on_device: studio-tower`** — recording *the first
  time we scanned and found our IPFS had pinned them, the time/date, and which computer observed it*
  ([repo_tracking_scheme.mdx §3.3](./repo_tracking_scheme.mdx)).

  ```
  repo file:   ~/BGit/charlie-kirk/videos/trees.mov      (pinned on the CLI, outside LFB)
  sidecar:     ~/T/_large_files_bridge/repos/<repoKey>/files/videos/trees.mov.yaml
               (mirrored to <syncRepo>/repos/<repoKey>/files/videos/trees.mov.yaml)
  ```

  Each of the 5 sidecars carries the file **name, the repo-relative path, the exact hash, the file size,
  a created/modified stamp**, and the `observed`/`not-lfbridge` first-seen event — **all captured on the
  tower**. (Those identity fields are exactly what lets the *laptop*, in February, show the files before
  it has the bytes — §3.)

**The decision prompt.** LFB surfaces the triage: it **recommends IPFS-pinning 6 files**. The user
**chooses yes on 3 and no on 3**. Each answer is:

* written to the **team-shared** `decisions.yaml` ledger in Local Storage
  (`~/T/_large_files_bridge/repos/<repoKey>/decisions.yaml`, mirrored to the sync repo — the source of
  truth: `asked: true`, the two axes, `decided_by`, `decided_at`, `sid` — [decisions.mdx §3](./decisions.mdx));
* mirrored as a **`decision` event** in each file's Local-Storage sidecar; and
* appended to the **history log** with a **per-file breakdown** because the outcomes differ:

  ```
  # ~/T/_large_files_bridge/repos/<repoKey>/history/studio-tower.txt
  2026-01-08T15:20:44Z  DECISION  by=bryan@act3ai.com  recommended IPFS-pin for 6 files; user chose:
                          pin=yes   videos/a.mov
                          pin=yes   videos/b.mov
                          pin=yes   videos/c.mov
                          pin=no    videos/d.mov
                          pin=no    videos/e.mov
                          pin=no    videos/f.mov
  ```

  *(Every time a user makes decisions we log it, UTC-stamped, and when different files got different
  answers we indent the per-file decision so it is unambiguous and parsable later.)*

* LFB then **pins the 3 yes files**, appending an `ipfs_pin` event (with the resulting CID) to each and an
  `IPFS-PIN` line to the history log. A file can later be turned **off then on again** — each transition
  is its own event, so the sidecar keeps the full on/off/on history.

**What we double-check (January):**
* `repo_storage.yaml` has **exactly one** level-one key (`repo_storage:`) and correct `counts:` (20
  special, 5 `ipfs_pinned`, 19 compressible, 20 transcribable).
* The 5 CLI-pinned files each have a sidecar whose first event is `observed` / `by: not-lfbridge` /
  `on_device: studio-tower` — LFB is **not** claiming it pinned them.
* The history log's DECISION entry shows the 3-yes / 3-no split per file.

## 3. FEBRUARY — `bryan-macbook-pro` — enlist the same repo on the laptop

**What the user does.** On the Mac laptop they clone the **same** Charlie Kirk repo — very likely to a
**different location** on disk (the path *up to* the repo differs) — install LargeFileBridge, enlist the
repo, and open the web app. The **different local path is fine**: the laptop derives its own
`repoKey = shortHash(resolve(repoPath))` for its Local-Storage tracking, and the **graft**
([devices.mdx §4](./devices.mdx)) maps the repo's shared hierarchy onto *this* computer's absolute path.

**What LFB does automatically — the sync-repo reconcile.** The git clone brought only the repo's own content
(source, and any committed transcripts in `.lfbridge/`) — **no** `manifest.yaml`, `decisions.yaml`, or
sidecars, because those are Category-B tracking state that never rides the working repo. Instead LFB pulls
the **sync repo** and **reconciles** its `repos/<repoKey>/` subtree into the laptop's Local Storage
([artifact_placement_policy.mdx §5](./artifact_placement_policy.mdx)) — that is how the tower's manifest,
decisions, and sidecars arrive. LFB then checks the manifest's CIDs against **the local repo and the running
IPFS node** and finds the pinned files are **absent locally and not yet pinned here** — the concrete
reconcile report:

```csv
exists_status,ipfs_pinned,file_path
missing,false,~/BGit/charlie-kirk/videos/2074257725697970542.mp4
missing,false,~/BGit/charlie-kirk/videos/2075191642366054733.mp4
missing,false,~/BGit/charlie-kirk/videos/2075290171105288315.mp4
missing,false,~/BGit/charlie-kirk/videos/2075531573907071094.mp4
missing,false,~/BGit/charlie-kirk/videos/2074257725697970542.mp3
```
*(All absent from the working tree; none of the manifest CIDs pinned on this laptop's node — `ipfs pin ls`
returns "not pinned" for each. The `.mp3` has no manifest CID, so it is shown for completeness but is not
a pull candidate.)*

**Reconciling is a MERGE, not a copy** — union by `path`, **union `pinned_by`**, newer `modified_at` wins a
CID conflict, absence is **never** a delete, and this computer's own pin claim is re-derived from its real
pinset. That rule is LOCKED in
[storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked); a
last-writer copy would erase the tower's pin claim on the laptop and with it the entire "a peer has this"
signal this month depends on. The reconcile runs on **every** backbone pull of the SDL, never on demand.

**The warning.** This becomes a **warning with a button** — *"A computer of yours pinned files this
computer doesn't have yet — pull them down."* ([warnings.mdx §10.8](./warnings.mdx), the pull-them-down
warning). Its blue arrow opens the **guided popup**.

**The popup — a checklist even though the bytes aren't here.** The right pane is the **subjects list**: a
**checkbox per file** the user can toggle to choose which to pull. LFB can **name and describe the files
even though they are not on this computer**, because the tower's per-file sidecars and manifest — reconciled
in from the **sync repo** — carry the **file name, the directory path it belongs under, the hash, the file
size, and which computer added them**. So the laptop shows, per row: **file size · target directory ·
"added by studio-tower."**

The user **starts with all 5 checked, turns 1 off, and pulls 4.**

**The two checkboxes + the dynamic action button.** The popup's options pane has **two checkboxes**, both
checked by default: **Add to IPFS (pin)** and **Compress**. The bottom-right blue button **spells out the
checked axes** and updates live ([warnings.mdx §4.4](./warnings.mdx), the composed-label rule):

```
☑ Add to IPFS (pin)     ☑ Compress
                                          [ Compress and Continue: IPFS Add › ]

# user unchecks Compress →
☑ Add to IPFS (pin)     ☐ Compress
                                          [ Continue: IPFS Add › ]
```

The label is **composed from what is checked**, thoughtfully spelled out, and — as always — carries the
live checked **count** of the subjects list; unchecking a subject or an axis rewrites the button text
immediately.

**Apply → the bytes arrive.** On Apply, LFB pins the 4 checked files locally. Because the manifest/sidecar
carry the **CID (the SID)**, pinning **fetches the bytes down to the laptop over IPFS** — the files now
exist locally and are pinned. Each gets an `ipfs_pin` (and a `pull`) event `on_device: bryan-macbook-pro`
in the laptop's Local-Storage sidecars, and the laptop's own
`~/T/_large_files_bridge/repos/<repoKey>/history/bryan-macbook-pro.txt` records the pull with the per-file
breakdown (4 pulled, 1 left). These writes mirror back out to the sync repo so the tower sees them too.

**What we double-check (February):**
* The tower's tracking (manifest / decisions / sidecars) reached the laptop **via the sync repo**, not the
  working repo — the working repo carried none of it.
* The pull-down warning renders a **per-file checklist** with **size / directory / added-by** even though
  the files are **absent locally**.
* The button reads **"Compress and Continue: IPFS Add ›"** with both boxes checked and **"Continue: IPFS
  Add ›"** with Compress unchecked, and the label tracks the checked count.
* After Apply, the 4 chosen files are present and pinned on the laptop; the 5th is not; the laptop appends
  only to **its own** history file in Local Storage.

## 4. MARCH — `bryan-macbook-pro` — new videos land, "these can be compressed"

**What the user does.** On the laptop, **20 new video files** are downloaded into the repo and get
**checked into Git** (they are new and not yet git-ignored). At first the user **does nothing**.

**What LFB does automatically.** A scan (the laptop may run the scanning phase) classifies the 20 as
`video` + `compressible`. Because they look uncompressed, LFB raises the **"N videos here can be
compressed"** offer ([warnings.mdx §10.5](./warnings.mdx)) — amber, an **offer, never automatic**. The
scan may record `compressible: true` in each file's sidecar, **but the user has not been asked** — so
there is **no decision event yet**; the files stay Undecided.

**The user walks the UI to decide.** When ready, the user walks LargeFileBridge's UI to decide on the 20.
Each decision is stamped with **which computer** (`on_device: bryan-macbook-pro`) and **where the videos
came from** (provenance recorded in the sidecar). All of it lands in the **per-file sidecar properties** in
Local Storage (`~/T/_large_files_bridge/repos/<repoKey>/files/…`, mirrored to the sync repo) — plus the
shared ledger and the history log there.

**What we double-check (March):**
* The 20 new files are classified `compressible`; the offer appears; **nothing is compressed without a
  click**.
* A scan-written `compressible: true` does **not** count as a decision — the files are Undecided until the
  user answers.
* Each decision records `on_device` and the file's provenance.

## 5. APRIL — `studio-tower` — the files arrive via Git; compress half

**What the user does.** On the tower, the 20 files **arrive via Git**. In the LargeFileBridge web app the
user sees the **"can be compressed"** warning, opens it, and **checks half of them to compress**, then
hits the button. LFB **compresses that half**.

**What LFB does automatically — capture before, then after.** **Before compressing — and this is true for
every file — LFB stores the file's hash and size *first*.** For **video and image** files it also computes
the **perceptual content fingerprint** ([perceptual_fingerprint.mdx](./perceptual_fingerprint.mdx)) — the
PhotoDNA-style, **local-only, never-phones-home** hash that recognizes the same content across
resolution/compression/codec/format changes. After compression it stores the **post-compression hash and
size** (and the post fingerprint, which should match the pre within threshold — proof the content
survived). All of it appends a **`compress` event** to each sidecar:

```yaml
- kind: compress
  at: 2026-04-03T09:14:02Z
  on_device: studio-tower
  by: bryan@act3ai.com
  before: { hash: 9f2c8b…, size: 734003200 }   # captured BEFORE we touched the bytes
  after:  { hash: 4d7e1a…, size: 118489088 }   # captured AFTER
  fingerprint_before: a1b2c3…
  fingerprint_after:  a1b2c4…                  # matches within threshold → same content
  codec: { from: prores, to: h264 }
```

**The other half is a decision too.** By choosing **which half to compress**, the user is also deciding
**"no, do not compress the other half."** That declined half gets a **both-relevant `decision` event
recorded** ([decisions.mdx §1](./decisions.mdx)) — a **sticky "reviewed — leave it"** — so the compress
offer stops nagging for those files. This recorded "no" is what makes **May** correct.

**What we double-check (April):**
* Every compressed file has a `compress` event with **both** a `before` and an `after` `{hash,size}`, and
  media files carry `fingerprint_before` / `fingerprint_after`.
* The declined half has a **recorded decision** (not the *absence* of one) so the offer is sticky-resolved.
* The perceptual fingerprint step made **no network call** ([perceptual_fingerprint.mdx §6](./perceptual_fingerprint.mdx)).

## 6. MAY — `bryan-macbook-pro` — we do NOT re-ask about decided files

**What the user does.** Back on the laptop, more videos **may** have arrived separately from the April
batch.

**What LFB does — and deliberately does not.** For the **April files**, LFB **does not re-raise the
compress warning**:

* The half that was **compressed** is already compressed (the `compress` event and the in-file
  `LFBcompressed` marker say so — [compression.mdx §8.4](./compression.mdx)).
* The half the user **declined** on the tower carries a **recorded "no" decision** — the sidecar and the
  shared ledger both show **which computer, what time/date, and that the user chose no**. Because a
  decision **exists**, the file is **Decided**, and **LFB never re-asks** ([decisions.mdx §2](./decisions.mdx)).
* **It does not matter whether the recorded answer was yes or no** — a recorded decision is sticky and
  shared, so the laptop does not re-prompt about these files.

If **other, genuinely new** video files are present (never decided anywhere), LFB **may** surface a fresh
notification about *those* — but it will **not** re-litigate the files already decided in April. This is
the payoff of the team-shared, per-file decision memory: **ask once, on any computer; never re-ask on
another.**

**What we double-check (May):**
* No compress warning re-appears for the April files (compressed or declined) on the laptop.
* A recorded **no** is as sticky as a recorded **yes** — neither is re-asked.
* Only truly-undecided new files can raise a fresh notification.

## 7. The cross-cutting invariants this tour proves

1. **Tracking state lives in Local Storage on both computers, never the working repo**, and travels between
   them **only** through the sync repo; each computer owns **its own** `history/<device>.txt` and stamps
   `on_device` on every event. The working repo carries only content (transcripts / AI descriptions).
2. **Provenance is honest.** Anything done **outside** LFB (CLI pins, external compression) is recorded as
   `observed` / `by: not-lfbridge`, with the computer that observed it — LFB never claims another actor's
   work.
3. **Identity travels without the bytes.** File name, path, hash, size, and CID in the sync-repo manifest
   (mirrored in sidecars) let a computer **show and choose** files it does not physically have yet.
4. **Decisions are sticky and shared.** Ask once; a recorded yes **or** no stops the warning on **every**
   computer — the fix for "I did this before but the warning came back."
5. **Compression is reversible-by-record.** Before/after hashes + the perceptual fingerprint mean LFB can
   always prove what a file was, that the content survived, and that nothing was compressed twice.
6. **The action button always tells the truth** — its label is **composed from the checked axes** and the
   live subject count, so the user reads exactly what Apply will do.

6. Across all months the six invariants in §7 hold, and every write matches

**The concrete journey (§9), per

## 9. The load-bearing journey, end to end — one file, two computers (LOCKED)

Months §2–§6 above tour the *whole* repo-storage surface. **This section is the single narrowest path
through it** — one file, two named computers, nine steps — and it is the run that decides whether the
product works at all. Its rules are **owned by**
[storage_company.mdx §8](./storage_company.mdx#8-the-load-bearing-use-case--one-file-two-computers-locked);
this section is the *walk*, that section is the *law*. Where they could disagree, §8 wins.

### 9.1 The cast (the real machines)

| | |
| --- | --- |
| **The file** | `videos/2079054276320440416.mp4` — 5.3 MB, **git-ignored** (`.gitignore` → `videos/`) |
| **Working repo** | `~/BGit/Bryan_git/charlie-kirk` — remote `https://github.com/ACT3ai/charlie-kirk.git` |
| **Company storage (SDL)** | `~/BGit/Bryan_git/act3_large_files_bridge` — remote `github.com/ACT3ai/act3_large_files_bridge` |
| **Computer A — "the Tower"** | device **`bryan-mac-pro`** — has the file, pinned it, CID `bafybeihf5o…k7tmi` |
| **Computer B — "the MacBook"** | device **`bryanstarbuck-macbook-pro`** — has the repo cloned, **not** the file |

Both devices already appear in `<act3_large_files_bridge>/devices/`
([devices.mdx §10.1](./devices.mdx#101-the-device-registry-is-the-proof-of-life-of-the-backbone-locked)),
which is exactly why this journey is a **cargo** problem and not a **transport** problem: the road between
the two computers has been open and in daily use the whole time.

### 9.2 Step 1 — the Tower pins the file and records it

The user decides to sync the file ([decisions.mdx](./decisions.mdx)); LFB pins it and writes the
`path → cid + pinned_by: bryan-mac-pro` entry into the repo's Local-Storage tracking manifest at
`~/T/_large_files_bridge/repos/<repoKey>/manifest.yaml`. This link has always worked. Nothing here is new
— it is where the cargo starts.

### 9.3 Step 2 — the manifest mirrors into the company SDL, keyed by the REMOTE

The repo's owner resolves from its **remote org** (`ACT3ai`) to the **ACT3 company storage**, whose
git-backed SDL is **auto-adopted** as the sync repo, and whose per-repo mirror is **ON by default**
([storage_company.mdx §8.4.2](./storage_company.mdx#842-the-company-sdl-is-the-sync-repo-and-the-mirror-is-on-by-default-locked)).
The mirror lands at:

```
~/BGit/Bryan_git/act3_large_files_bridge/repos/<repoUid>/manifest.yaml
        repoUid = shortHash("github.com/act3ai/charlie-kirk")   ← the NORMALIZED REMOTE
```

Path containment would have resolved `charlie-kirk` to **Personal** — the wrong storage — because the
working repo does not live inside the SDL. And a path-hash key would have written a directory the MacBook
would never look in. **A machine-local value must never key machine-shared data.**

Deriving the shared repo identity and the mirror subtree is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-root.service.ts` — `normalizeRemoteKey()` / `repoUidFor()` (re-exported from the leaf `repo-identity.ts`) / `resolveStateSyncRepo()`.

Resolving the owning storage from the remote org and auto-adopting its SDL is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/artifact-placement.service.ts` — `resolveOwnerDedicatedRepo()`, consumed by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `ensureSyncRepoMarker()` / `mirrorToSyncRepo()`.

### 9.4 Step 3 — the SDL commits and pushes (and does it now, not in ten minutes)

The SDL's git backbone stages pathspec-lessly, so **anything written into the SDL ships**. What the mirror
must add is a **trigger**: a mirror write announces itself to the SDL it wrote into (debounced), so a
decision the user just made leaves the Tower in seconds rather than on the next device-worker tick
([storage_company.mdx §8.7](./storage_company.mdx#87-four-smaller-defects-this-use-case-also-closes)).

Nudging the owning SDL's backbone after a tracking write is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/sync-trigger.service.ts` — `noteArtifactWritten()` (which must resolve an SDL for a path inside a **working repo**, not no-op).

### 9.5 Step 4 — the MacBook fetches, merges, and reconciles per entry

On the MacBook, the SDL's ordinary backbone cycle (`fetch` → automatic `merge`) brings
`repos/<repoUid>/manifest.yaml` down. LFB then **reconciles it into Local Storage as a per-entry merge**
([storage_company.mdx §8.4.3](./storage_company.mdx#843-receiving-is-a-merge-never-a-copy-locked)): union
by `path`, **union `pinned_by`** (so the Tower's claim survives alongside this machine's), newer
`modified_at` wins a CID conflict, **absence is never a delete**, and this computer's own claim is
re-derived from its **real pinset** — a peer's manifest can never make this machine believe it holds bytes
it does not. It runs on **every** backbone pull, not when someone clicks something.

The merge must land in **both** per-repo manifests the UI reads, or the MacBook counts a `Pull down` it
cannot show a row for ([storage_company.mdx §8.6](./storage_company.mdx#86-two-manifests-must-not-disagree)).

Merging a pulled sync-repo subtree into Local Storage is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/storage/tracking-sync.service.ts` — `reconcileFromSyncRepo()` (a per-entry union merge over `mergeManifests()`), called from the repo pin pass in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinRepoFolder()`, and from the SDL git cycle (`syncStorageTextInner()` / `pinStorageUnitInner()`) via `reconcileMirroredRepos()`.

### 9.6 Step 5 — the red remote-only row appears in View One Repo

The user opens [View One Repo](./one_repo.mdx) for `charlie-kirk` on the MacBook. The file is **a row in
the list even though no local file exists** — composed from the reconciled manifest entry rather than a
disk `stat`, carrying `presence: "remote-only"`, its name, size, and CID, and `addedByDevice:
bryan-mac-pro`. It renders **red**, and its tooltip reads:

```
On bryan-mac-pro — not on this computer yet.
```

The peer's **device label** in that sentence comes from the travelling device registry
([devices.mdx §6.9](./devices.mdx#69-a-peer-device-label-names-a-remote-only-row-locked)) — which is why
the row can name *which* of your computers has the bytes instead of saying "somewhere else".

This is a **healthy** state, not an alarm: red here means *"available, not here yet"*, never *"lost"*.
Its **only** action is **pull it down** — compress / transcribe / describe / OCR / git-ignore are disabled
and say why, because there are no local bytes to work on
([storage_company.mdx §8.5](./storage_company.mdx#85-remote-only-rows--a-file-in-the-list-that-is-not-on-this-disk-locked)).

Composing remote-only rows from the reconciled manifest is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `composeFileRows()`, with the field in `~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/types.ts` — `FileRow.presence`.

### 9.7 Step 6 — it is counted, and it is offered

Showing the row is not enough; the user must be **told there is something to do** without hunting:

* **`Pull down` metric tile** — always counts it. That tile's whole job is *"a computer of yours has bytes
  you don't."*
* **`Add to IPFS` metric** — counts it while **this** computer has not decided it; the row is a
  recommendation to bring the file here and pin it.
* **The [To Do](./to_do.mdx) tab** — it appears as a `pull_down` recommendation in this company's batch,
  the same batch-popup surface as UC-20 in [use_cases.mdx §9](./use_cases.mdx).
* **No space, compression, git-ignore, or analysis metric counts it** — there are no local bytes to
  reclaim, shrink, ignore, or analyze.

Reading the peers-have-it list and pulling it are handled in code (the **same** manifest for both, per §8.6):
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `missingPinnedFromPeers()` and `pullMissing()`.

### 9.8 Step 7 — the user pulls, and the row becomes ordinary

The user clicks pull (from the row, the tile, or the To-Do batch). LFB pins the CID locally, which
**fetches the bytes down over IPFS** to the grafted local path
([devices.mdx §4](./devices.mdx#4-the-graft--machine-independent-hierarchy--local-paths-locked)) — the
MacBook's own `~/BGit/Bryan_git/charlie-kirk/videos/`, not the Tower's path. A `pull` + `ipfs_pin` event is
appended `on_device: bryanstarbuck-macbook-pro`, the MacBook's own history log records it, and those writes
mirror **back** into the SDL so the Tower learns the file is now on two computers.

The next scan finds a real file on disk, so the row loses `presence: "remote-only"` and becomes an
**ordinary local row**. Nothing re-prompts: the decision is recorded, sticky, and shared (§6).

### 9.9 What this journey proves that the five-month tour does not

The Jan–May tour proves the **surfaces** are right. This journey proves the **chain** is connected:
identity (a remote-derived key), ownership (a remote-derived owner), default (a mirror that is on),
transport (a commit that is triggered), receipt (a merge, with callers), and display (a row that can exist
without a file). Five of those six links existed while the product still delivered nothing —
*"a working road and a working destination, and never put the cargo on the truck"*
([storage_company.mdx §8.3](./storage_company.mdx#83-the-chain-and-where-each-link-was-broken)).
