description: The authoritative spec for what happens when LargeFileBridge pins — the "pin pass," the every-15-minute pass that adds files to IPFS and pins them. A pin pass is ALWAYS a full pass over every unit LFB knows on this computer (every git repo it has discovered, plus the computer unit), never one repo in isolation. However it is triggered — the 15-minute scheduleTask, the 48-hour session catch-up, or a manual "Pin now" on a single repo — it runs the full pass; a repo-scoped trigger only sets PRIORITY (that repo pins first, then the pass continues over the rest). Reconciles with storage_local.mdx (on-disk units + §13 background process), scan.mdx (the scheduleTasks + pinset reconcile), one_repo.mdx (the Pin-now button), and sync_list.mdx (the trust/reconcile rules).

# LargeFileBridge — The Pin Pass (add to IPFS)

This is the **primary, authoritative** spec for the act of pinning — adding files to IPFS and pinning
them. Three neighbors describe *pieces* of it and defer here for the process itself:

* [storage_local.mdx](./storage_local.mdx) — where pin state lives on disk (per-unit `config.yaml` +
  `manifest.yaml`) and the **background pin process** transparency contract (§13). The *mechanics of
  a pass* are here; storage owns the *state a pass reads and writes*.
* [scan.mdx](./scan.mdx) — the two **scheduleTasks** (4-hour discovery + 15-minute pinset) that
  *trigger* work. A **scan** discovers; a **pin pass** (this file) moves bytes.
* [one_repo.mdx](./one_repo.mdx) — the **Pin now** button on a repo page. The button is one *trigger*
  of the pass defined here.
* [sync_list.mdx](./sync_list.mdx) — the **trust & reconciliation** rules (§5 there) a pass applies
  when it reads any incoming list. The pass never *trusts*; it *reconciles*.

The pin pass runs LFB's own TypeScript-on-Node worker (never raw shell/`curl`), identically on a
laptop and on a server; only the resolved state root changes.

---

## 1. LFB knows every repo — there is no "LFB-managed" gate (LOCKED)

**Retire the idea of an "LFB-managed" repo.** A repo does **not** have to be added to, adopted by, or
turned on in LFB before LFB knows about it. LFB **discovers and understands every git repository on
the filesystem automatically** ([storage_local.mdx §7](./storage_local.mdx#7-the-scanner--discovering-repos-and-the-mask),
[scan.mdx](./scan.mdx)). Knowing a repo, tracking its large files, computing its counts, showing it in
the UI, and **reconciling it during a pin pass** all require **zero** user action.

Two distinct concepts — keep them separate everywhere:

| Concept | Who triggers it | What it means |
| --- | --- | --- |
| **Known / understood** | **Automatic** — the scanner | LFB has discovered this repo (or the computer unit), tracks its big files, reflects it in the UI, and **visits it every pin pass**. Every repo on disk is known. There is no opt-in to be known. |
| **IPFS-enabled (opted in to pin)** | **Explicit user action** — a confirmed choice | LFB may **add bytes to IPFS, pin them, and fetch them from peers** for this unit. This is the **only** thing that is opt-in. It is `pinned: true` in the unit's `config.yaml` (storage_local.mdx §6.2). |

So the per-unit **`pinned`** flag does **not** mean "managed vs. unmanaged." It means **"the user has
opted this unit into IPFS pinning/adding."** A `pinned: false` repo is **not** invisible or ignored —
it is fully known and fully visited by the pass; LFB simply does not add/pin/fetch its bytes until the
user opts in. This matches the charter: LFB never turns a computer into a pinner of content without an
explicit, user-confirmed action, and it never quietly mutates a repo.

> **Why this matters (charter alignment).** IPFS pinning is the one action with an outward, hard-to-
> reverse footprint (it makes this machine store and serve content). The charter requires it be
> **user-confirmed**. *Understanding* a repo has no such footprint, so it needs no gate. Conflating
> the two produced the "managed" mental model, which made a fresh repo look like it "wasn't in LFB"
> when in fact LFB knew it perfectly well and was merely not pinning it yet.

Discovering and registering every git working tree with no user action (identity + `pinned` seeded off, i.e. pinning-not-yet-opted-in) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/units.service.ts` — `registerRepo`

Wording rule for all specs and UI copy: prefer **"every repo on this computer"** / **"known repo"** /
**"opted in to IPFS pinning"** over **"managed repo."** "Managed" survives only where it plainly means
"the row in our list," never as an opt-in gate.

## 2. Every pin pass is a FULL PASS (LOCKED)

A **pin pass** is one traversal of **every unit LFB knows on this computer** — every discovered repo
(`pin/r/<repo_name>/`) **plus** the computer unit (`pin/computer/`,
[storage_local.mdx §8](./storage_local.mdx#8-synccomputer--the-whole-computer-sync-unit)). One pass = the whole
machine, always.

**No trigger reduces the scope to a single unit.** The three triggers differ only in *when* and *what
runs first*, never in *what gets covered*:

| Trigger | Source | Scope | Order |
| --- | --- | --- | --- |
| **Scheduled** — 15-min scheduleTask | launchd/cron ([storage_local.mdx §13](./storage_local.mdx#13-the-background-sync-process--the-transparency-contract)) | Full pass | Natural order |
| **Session catch-up** — > 48h stale | a returning web session ([sessions.mdx §3](./sessions.mdx)) | Full pass | Natural order |
| **Manual — "Pin now" on a repo** | the repo page / row kebab / bulk ([one_repo.mdx §3.1](./one_repo.mdx#31-header)) | **Full pass** | **That repo first (priority §3)**, then the rest |

The load-bearing rule the button must honor: **a manual Pin now on one repo prioritizes that repo but
still runs the full pass.** It does the user's repo first (so their result is immediate) and then keeps
going across every other known unit. It must **never** pin one repo and stop.

**A fourth, implicit trigger: the in-process watchdog backstop.** The three triggers above are all
*external* to the running app — a launchd fire, a returning session, a button. If the **launchd trigger
itself dies** (a stale trampoline path, an unloaded job), none of them fire and the flow stops silently.
So the always-on web app **watches its own workers**: any installed + enabled worker that has not run
within twice its interval is **run in-process** and its OS job is **repaired**, so a dead OS trigger can
never silently halt the pass while the app is up. This backstop — and the full taxonomy of ways the flow
can stall and how each self-heals or is surfaced — is specified in
[backbone_resilience.mdx](./backbone_resilience.mdx) (§3 the watchdog, §7 the overdue signal), implemented in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/schedule/watchdog.service.ts` —
`startWatchdog()`.

The full pass over **every repo plus the computer unit**, and a manual Pin now that pins the clicked repo first (as the priority unit) then runs the pass over the remaining units in the background, are implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll({ priorityDone })` (repos fan out via `runPool`, then `pinComputerUnit()`; both repo and computer share one generic `runUnitPin` core), invoked after the priority repo by `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `POST /api/repos/:repoId/pin`

## 3. Prioritization — order, never scope

When a pass is triggered from a specific repo, that repo's unit is the **priority unit**:

* It is pinned **first and awaited**, so the UI can return that repo's fresh detail immediately and the
  button feels instant.
* The rest of the pass then runs, in the **background**, so the response is never blocked by a long
  fan-out over 100+ repos.
* Priority changes **order only**. The remaining units are still all covered.

A paths-scoped manual pin (bulk **Pin now (selected)**, [one_repo.mdx §4.4](./one_repo.mdx#44-bulk-actions))
prioritizes this repo restricted to the selected files, then the full pass follows for the rest.

**Overlap guard.** Passes are idempotent, but overlapping passes waste work, so a single in-flight
guard collapses a second pass that starts while one is running (rapid Pin-now clicks, or a
scheduleTask that fires mid-pass). The **priority unit is always pinned by its own trigger** and is
never dropped by the guard — only the shared *remainder* fan-out is de-duplicated.

Answering with the priority repo's detail immediately, then running the remainder pass in the background under the in-flight guard, is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `pinAll` (`passInFlight` guard) + `repos.router.ts` `POST /:repoId/pin` (`void pinAll({ priorityDone })`)

## 4. Parallelism — many cores, bounded concurrency (LOCKED design)

Modern machines have **20–30 cores**; pinning units one-at-a-time wastes almost all of them. A pin
pass is **concurrent by default**:

* **One global budget for all heavy work.** Every add/pin/fetch is drawn through a **single limiter**
  sized **`cores − 2`**, so total in-flight IPFS operations stay bounded **no matter how units × files
  multiply**. Two cores are left for the web app and the IPFS node so the UI stays responsive during a
  big pass. Concurrent state writes are safe because every write goes through the store layer's
  **per-file mutex + atomic temp-then-rename**
  ([storage_local.mdx §15](./storage_local.mdx#15-the-store-layer-how-code-touches-disk)).
* **Across units.** Units pin concurrently.
* **Within a unit.** A unit's independent per-file operations (add + pin new/changed files, fetch
  missing CIDs) **fan out in parallel** through the same global limiter — not a strict file-by-file
  loop. Ordering within a unit does not matter; each file owns a distinct manifest-row key, so the
  shared in-memory manifest/pinset maps are mutated without collision.
* **The priority unit** starts immediately; the remainder fan out behind it.

Sequential pinning is a **defect to fix**, not an acceptable baseline. Any place that still loops
`for … await` over units or files across a pass should be flagged and converted to the pool/limiter.

**Canonical budget (unified — one definition).** The `cores − 2` figure this pass uses is now the canonical
**RESPONSIVE budget** defined in [parallelization.mdx §1](./parallelization.mdx) and exposed as
`responsiveBudget()` in `~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts`.
The pin pass is its **reference consumer** — the work that runs *alongside* interactive use, so it leaves 2
cores for the web app + IPFS node rather than running hot at ~90% like the mass-compute budget the job queue
draws ([performance.mdx P-26](./performance.mdx)). The inline `PIN_CONCURRENCY = cores − 2` in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` should be
**expressed via `responsiveBudget()`** so there is exactly **one** definition of the number, not a second
copy that can drift. The two-level fan-out design above is unchanged — this only unifies *where the count
comes from*.

The bounded fan-out — a `cores − 2` pool across units and one shared `cores − 2` limiter that caps ALL per-file add/pin/fetch work — is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/pin/pin.service.ts` — `runPool` + `PIN_CONCURRENCY` (units) and `ipfsLimiter` + `runUnitPin` (per-file `Promise.all` through the limiter)

## 5. What a pass does per unit — and what the opt-in gates

For **every** known unit the pass visits, it always **reconciles** (no opt-in required):

1. **Rescan light** — pick up new/changed/deleted big files the last scan recorded ([scan.mdx](./scan.mdx)).
2. **Reconcile the pinset** — read the local IPFS pinset (`ipfs pin ls`) and refresh the unit's
   `pinned_by` claims against ground truth: confirm real pins, **drop** stale self-claims for pins that
   were lost, and surface pinned-but-untracked CIDs as **import** candidates
   ([storage_local.mdx §9.5](./storage_local.mdx#95-the-ipfs-pinset-is-the-source-of-truth--pinned_by-is-a-stale-cache),
   [scan.mdx §2.1](./scan.mdx#21-the-15-minute-scheduletask-reads-the-ipfs-pinset-import-already-pinned)).
3. **Reconcile any incoming Sync List** — never trusted, always reconciled per
   [sync_list.mdx §5](./sync_list.mdx#5-trust--reconciliation-the-load-bearing-rule).
4. **Write `status.yaml`** — last sync time, counts, errors — so the UI reflects the visit.

Only the **byte work** is gated by the `synced` opt-in (§1):

* If the unit is **opted in** (`synced: true`): add new/changed `sync`-decided files to IPFS, pin
  locally, fetch missing manifest CIDs from peers, and (if `publish_manifest`) write the committed
  in-repo manifest so git carries the list.
* If the unit is **not opted in** (`synced: false`): the pass still visits and reconciles it (steps
  1–4) but adds/pins/fetches **nothing** — until the user opts in.

**Device write-back is explicitly NOT gated by `synced`.** Writing **this computer's own**
`devices/<self>.yaml` into a Git-backed storage — and running that storage's Git cycle to carry it — is
the "understand", not the "pin", side of §1: it puts your own identity text into your own configured
repo, with **no** outward footprint. So a Git-backed storage runs **pull → write self device → commit →
push every pass regardless of `synced`**; only the byte reconcile above waits on the opt-in. This is the
guarantee the [every-10-minute device-registration worker](./devices.mdx#11-the-device-registration-write-back--every-10-minutes-locked)
is built around, and it is applied to this Transfer pass too — implemented in
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` —
`syncStorageUnit(id)` (Git cycle + `writeSelfDevice` outside the `synced` gate; only `runUnitSync` inside it).

**A manual Sync now is itself the opt-in.** Clicking Sync now on a not-yet-opted-in repo is an explicit
user action, so it flips that repo to `synced: true` and does the byte work, then the every-15-min
scheduleTask keeps it fresh. (This is the exact bug §6 records: the manual path used to be gated by the
same `synced:false` guard as the background scheduler and silently did nothing.)

The manual-vs-background gating — the background scheduler respects `synced:false`, a manual Sync now moves bytes and flips the unit's opt-in on — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/sync/sync.service.ts` — `syncRepoFolder(folder, onlyPaths, { manual })`

## 6. Never a silent no-op — report truthfully (LOCKED, learned from a defect)

This invariant exists because it was violated in production. The defect and the rule:

> **The defect.** `syncRepoFolder()` bailed at the top with `if (!cfg.synced) return`, and that guard
> fired for the **manual** Sync now too — not just the background scheduler. A fresh repo defaults to
> `synced: false` ([storage_local.mdx §6.2](./storage_local.mdx#62-syncrrepo_nameconfigyaml)), so clicking **Sync
> now** returned instantly, pinned nothing, wrote no manifest — **yet the UI still toasted "Sync
> complete."** A user could mark files and click Sync forever and nothing would ever pin. The status
> popped up and vanished because the backend genuinely did nothing.

The rules this locks in:

* **A user-initiated sync must do real work.** The manual path is the repo's *primary* action; it may
  never be short-circuited by the background-only `synced` gate (§5).
* **Report what actually happened, not a fixed string.** A sync result carries **counts** — added,
  pinned, fetched, skipped, failed — and the UI reflects them. If a run was genuinely a no-op (nothing
  eligible), say so explicitly ("Nothing to sync — no files marked Sync"), never a blanket "complete."
* **Success ≠ "the call returned."** A green toast requires the work to have occurred (or an honest
  "nothing to do"), and a failure must surface, not be swallowed.

This is a **design smell to hunt for everywhere**: any action whose success UI is decoupled from
whether it did anything. Flag and fix such silent no-ops on sight.

## 7. Acceptance criteria

1. **Every repo is known automatically** with no user action; the pass **visits and reconciles every
   known unit** (all repos + the computer unit) regardless of opt-in (§1, §2, §5). No spec or UI copy
   frames a repo as needing to be "added"/"managed" before LFB knows it.
2. The per-unit **`synced` flag means "opted in to IPFS pinning,"** not "managed." A `synced: false`
   unit is still fully known, visited, and reconciled; only byte work (add/pin/fetch) is withheld (§1,
   §5).
3. **A synchronization is always a full pass.** No trigger — scheduled, session catch-up, or manual
   Sync now — reduces scope to one unit (§2).
4. A **manual Sync now on one repo prioritizes that repo (synced first, awaited) and then runs the full
   pass over the remaining units** in the background; it never syncs one repo and stops (§2, §3).
5. The pass runs with **bounded concurrency** (default `cores − 2`) across units, and fans out per-file
   work within a unit; sequential `for…await` over a whole pass is treated as a defect (§4).
6. A **manual Sync now is the explicit opt-in** for its repo (flips `synced: true` and moves bytes),
   while the **background scheduler still respects `synced: false`** (§5).
7. **No silent no-ops.** A user-initiated sync does real work or reports an honest "nothing to sync,"
   with truthful counts; a green success toast is never shown for a run that did nothing, and failures
   surface (§6).
8. Overlapping passes are **collapsed by a single in-flight guard**; the priority unit is always synced
   by its trigger and never dropped by the guard (§3).
