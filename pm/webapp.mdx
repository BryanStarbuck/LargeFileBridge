# LargeFileBridge — Web App Entry & Sign-In (credentials setup)

This spec covers the **first thing anyone sees**: the web app's **entry gate**. Before a session
belongs to an authenticated, allow-listed user, the app renders the **Sign-In page** — and when the
server has **no Google OAuth client credentials yet**, that page shows a **"sign-in is not configured
yet" remediation panel** telling the operator exactly how to install them.

The remediation panel is the **focus of this file**. Per the charter, LFB uses **OpenAuthFederated**
with **Google** sign-in and an **allow-list** — there is **no anonymous / no-login path**. So if the
OAuth client isn't configured, nobody can get in, and the page's whole job is to make fixing that
**unambiguous**: the **exact file path**, the **exact JSON to paste**, and the **exact redirect URI**
to register.

Grounded in the real code:

* Component: `code/packages/frontend/src/pages/sign-in/SignInPage.tsx`
* Auth gate that mounts it: `code/packages/frontend/src/main.tsx`
* Config signal: `GET /api/health/auth-config` → `{ oauthConfigured, devAuth }`
  (`code/packages/backend/src/modules/health/health.router.ts` → `authConfig()`)
* Where creds are read: `code/packages/backend/src/config/credentials-file.ts`

Siblings: [storage.mdx §10](./storage.mdx#10-auth-session--signing-secret-runtime-not-settings) (the
signing secret + why client creds are **not** in the state store) ·
[storage.mdx §4](./storage.mdx#4-per-logged-in-user-storage--the-per-user-requirement) (the
allow-list that decides who, once signed in, is actually let in) ·
[settings.mdx §4](./settings.mdx#4-the-rest-of-the-global-settings-surface) (allow-list editing,
admin-only) · [code_plan.mdx](./code_plan.mdx) (ports: web app **:2222**, API **:8787**).

## 1. When this page renders — the auth gate

The gate lives in `main.tsx`. It calls `GET /api/auth/me` and shows the Sign-In page unless the
caller is **both** authenticated **and** allow-listed:

```
me = GET /api/auth/me
if (!me.authenticated || !me.allowListed)  →  render <SignInPage/>
else                                        →  render the app (RouterProvider)
```

So the Sign-In page is the pre-app gate. It has **no left bar** and **no nav** — it precedes
[left_bar.mdx](./left_bar.mdx) entirely. It is not a routed page; it is what `<Root>` renders in place
of the router when the gate is closed.

The auth gate that renders `<SignInPage/>` unless the caller is authenticated and allow-listed is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/main.tsx` — `Root()` component

The Sign-In page then asks a **second** question — *is the server even able to sign anyone in?* — via
`GET /api/health/auth-config`, which returns:

| Field | Meaning |
| --- | --- |
| `oauthConfigured` | `true` when the backend has **both** a Google client id **and** secret (from env or the credentials file). Drives the "Sign in" button vs. the remediation panel. |
| `devAuth` | `true` when no creds are set **and** `LFB_DEV_AUTH !== "false"` — the localhost offline bypass (§4). |

The `GET /api/health/auth-config` signal that drives the button vs. the panel is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/health/health.router.ts` — `GET /auth-config` (delegates to `authConfig()`)

## 3. The remediation panel — "Sign-in is not configured yet"

This is the UiPageWeRender the task calls for. When `oauthConfigured` is `false`, the card body
replaces the sign-in button with a setup panel. It must contain **all four** of the following, in
plain, copy-pasteable form.

### 3.2 The four required elements

1. **The full install path — shown literally.** `~/.credentials/large_files_bridge.json`. This is the
   default read location in `credentials-file.ts` (`path.join(os.homedir(), ".credentials",
   "large_files_bridge.json")`). It must be rendered as **selectable, monospaced text with a
   copy button** — never buried in prose. State plainly that **the file does not exist yet** (that is
   the normal first-run condition). If the operator overrode the path with **`LFB_CREDENTIALS_FILE`**,
   the panel should show **that** path instead (see §6).

2. **The JSON layout — shown literally.** The exact shape the backend parses (§4). Rendered in a
   copy-able code block with placeholder values so the operator can paste and fill in.

3. **The exact redirect URI — shown literally.** `http://localhost:8787/api/v1/oauth_callback`
   (the `GOOGLE_REDIRECT_URI` default in `.env.example`, built from the **API** port **8787** — *not*
   the web app port 2222). This must be registered **verbatim** on the Google Cloud OAuth client's
   "Authorized redirect URIs," or Google rejects the round-trip. Copy button here too.

4. **The restart nudge.** Credentials are read at boot (`loadGoogleCreds()` runs when the auth
   Frontend API is built), so the operator must **restart the backend** and reload. Say so explicitly.

The panel should also surface the **allowed sign-in domain(s)** as context — default **`act3ai.com`**
(`AUTH_ALLOWED_DOMAINS`) — so the operator knows *which* Google account will be accepted even once the
client is wired up. Being on an allowed domain still isn't sufficient by itself: the email must also
be on the **allow-list** ([storage.mdx §4](./storage.mdx#4-per-logged-in-user-storage--the-per-user-requirement)).

## 4. The credentials JSON — the layout the backend actually parses

`credentials-file.ts` accepts **two shapes** and reads `clientId` / `clientSecret` from whichever is
present. The panel shows the **nested** (preferred) form; document both here so the spec matches code.

**Preferred (namespaced) form — what the UI prints:**

```json
{
  "large_files_bridge": {
    "google": {
      "clientId": "1234567890-abcdef.apps.googleusercontent.com",
      "clientSecret": "GOCSPX-xxxxxxxxxxxxxxxxxxxx"
    }
  }
}
```

**Also accepted (top-level) form:**

```json
{
  "google": {
    "clientId": "1234567890-abcdef.apps.googleusercontent.com",
    "clientSecret": "GOCSPX-xxxxxxxxxxxxxxxxxxxx"
  }
}
```

Resolution in code: `json.large_files_bridge?.google ?? json.google ?? {}`. The namespaced form is
preferred because **one `~/.credentials/*.json` file can hold keys for several sibling apps** — namespacing
under `large_files_bridge` keeps LFB's block from colliding with theirs.

Reading both accepted shapes (with invisible-whitespace repair) is handled in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/config/credentials-file.ts` — `loadGoogleCreds()`

**This is a secret file.** It holds the OAuth **client secret** and therefore lives **out of the repo**
and **out of the state store** — never checked in, never inside `~/T/_large_files_bridge/`
([storage.mdx §10](./storage.mdx#10-auth-session--signing-secret-runtime-not-settings)). The panel may
remind the operator to keep it private (e.g. `chmod 600`).

## 5. Precedence — where the client id/secret come from

The backend resolves credentials with **env-wins-then-file** (`loadGoogleCreds()`):

```
clientId / clientSecret =
  1. process.env.GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET      (if BOTH set → use, stop)
  2. else  <credsFile>.large_files_bridge.google.{clientId,clientSecret}
  3. else  <credsFile>.google.{clientId,clientSecret}
  4. else  "" / ""  →  oauthConfigured = false  →  render §3 panel
credsFile = process.env.LFB_CREDENTIALS_FILE
            || ~/.credentials/large_files_bridge.json
```

So the panel's two documented paths — **the JSON file** and **the two env vars** — are exactly the two
inputs the code honors, in that priority. `oauthConfigured` is simply `Boolean(clientId && clientSecret)`
after this resolution; a half-filled config (id but no secret) still reads as **not configured** and
still shows the §3 panel.

## 7. Acceptance criteria

1. The entry gate (`main.tsx`) renders `<SignInPage/>` whenever the caller is not **both**
   authenticated **and** allow-listed; otherwise it renders the app.
2. `SignInPage` reads `GET /api/health/auth-config` and shows **state A** (Sign in with Google →
   `/api/v1/sign_in/sso`) when `oauthConfigured` is true, and the **§3 remediation panel** when it is
   false.
3. The remediation panel shows the **full credentials path** —
   `~/.credentials/large_files_bridge.json` (or the resolved `LFB_CREDENTIALS_FILE`) — as selectable,
   copy-able monospaced text, and states that the file does not exist yet.
4. The panel prints the **exact JSON layout** to paste (`large_files_bridge.google.{clientId,
   clientSecret}`), copy-able, matching what `credentials-file.ts` parses.
5. The panel shows the **exact redirect URI** to register on the Google Cloud OAuth client —
   `http://localhost:8787/api/v1/oauth_callback` (API port 8787, **not** web port 2222) — copy-able.
6. The panel tells the operator to **restart the backend** and reload, and documents the **env-var
   alternative** (`GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`, which win over the file) and the
   **localhost `LFB_DEV_AUTH=true`** bypass.
7. All app-identity strings say **Large File Bridge** and use LFB's path/key/port/domain — **no**
   leftover sister-app values (§6).
8. Displayed path / redirect URI / JSON key / allowed domain reflect the backend's **resolved** values
   (env or file overrides included), not hard-coded UI defaults.

# Part II — Progress & asynchronous feedback (the app shell)

Everything above is the **pre-app gate**. Once a user is signed in and allow-listed, `main.tsx` renders
the app (`<RouterProvider>`), and the shell gains a second job: **tell the user when work is happening
in the background**. LFBridge is a **long-job** app — a scan walks thousands of files, an IPFS
add/pin/publish moves hundreds of megabytes, a video compress runs `ffmpeg` for minutes, a
content-fingerprint pass hashes whole files. None of that is instant, and today the UI gives **no live
signal** while it runs (the current endpoints simply **block** — `POST /api/repos/rescan` awaits
`scanAll()`, `POST /api/repos/:repoId/sync` awaits `syncRepoFolder()` — and return only when finished;
`repos.router.ts:43`, `:90`). Part II fixes that.

The blocking handlers Part II reshapes into job-registering endpoints are today in code:

`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/repos/repos.router.ts` — `POST /rescan` and `POST /:repoId/sync`

We follow the pattern proven in the sister app **EmailDeliveryHero** ([pm/progress_ui.mdx](../../EmailDeliveryHero/pm/progress_ui.mdx)):
put the "something is happening" feedback **at the bottom**, **bottom-left**, **clear of the left bar**,
as a **stack of small cards that drain independently** as each job finishes. LFBridge adapts it in two
ways that its architecture demands: the cards name **many kinds of operation** (not just one "scan"),
and the dock reflects **server-side background-worker jobs** too — not only ones this browser started —
because the sync/scan workers run under **`launchd`** on their own schedule (charter: sync every 15
min; [scan.mdx §2](./scan.mdx): scan every 4 h).

There are **two** surfaces, and they must not be confused:

1. **Toasts** — the transient `sonner` event notifications that already fire on mutations ("… updated",
   error messages). Part II **re-positions and enlarges** them; it does **not** change *when* they fire.
2. **The progress dock** — a **new, dedicated** surface. While ≥1 long job runs, it shows one short card
   per in-flight job with a spinning icon and **"Running <op> <target>"**; cards **stack** and each
   **removes itself the instant its job finishes**.

Both live in the **bottom-left**, offset past the **256px** left bar ([left_bar.mdx](./left_bar.mdx)),
the dock sitting **above** the toast stack so the two never collide. Ports for reference:
web **:2222**, API **:8787** ([code_plan.mdx](./code_plan.mdx)).

## 8. Goals (what "on the bottom, doing work asynchronously" means here)

* When something happens (a decision saved, a worker toggled), a **toast pops in from the bottom-left**
  — not the top-right it uses today.
* Toasts appear **about twice as big** as the default and anchored **bottom-left**, but **never
  overlapping the left navigation bar**.
* Any **long-running job** shows its progress in that same bottom-left region, **below/clear of the left
  bar**, as a **short, somewhat-wide card**.
* Jobs run **in parallel**, bounded by a small concurrency limiter. Syncing four repos, or compressing
  three videos, shows **all of them at once**, and each card **leaves as its own job finishes** — so a
  slow job lingers after a fast one is gone.
* Each card reads **"Running <op> <target>"** (e.g. *Syncing movie.mp4*, *Compressing trailer.mov*,
  *Scanning act3ai-repo*) with an **animated circular-arrows icon** signalling live work.
* Where the backend can report a **fraction** (bytes added, `ffmpeg` percent, files walked), the card
  also shows a **thin determinate bar** and a small detail line (e.g. `412 MB / 734 MB`). Where it
  can't, the spinner alone carries "still working."
* **When a job finishes, its card disappears.** The stack drains unevenly.
* The dock reflects **server-side jobs too** — a scan or sync kicked off by `launchd` (or from another
  browser tab) appears here, not just jobs this tab started.

`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/main.tsx` — `<Toaster position="bottom-right" richColors />` in the root render

## 10. The progress dock (new surface)

### 10.1 Where it lives

A **fixed-position** region in the **bottom-left** of the viewport, offset the same **272px** to clear
the 256px left bar, sitting **above** the toast stack so the two never collide (dock `bottom: 88px`,
toasts `bottom: 16px`; both anchored left). It renders **only while ≥1 job is active** and is otherwise
**absent from the DOM** (no empty box).

```
┌────────────┬──────────────────────────────────────────────────────────┐
│  LEFT BAR  │  content area                                             │
│  (256px)   │                                                           │
│  Repos     │                                                           │
│  File Sys  │                                                           │
│  Peers     │                                                           │
│  Sync      │   ┌─────────────────────────────────────────┐  ← dock,   │
│  Settings  │   │ ⟳  Syncing  movie.mp4    412 / 734 MB    │    stacked │
│            │   │ ▇▇▇▇▇▇▇▇▇▇▇▇▇░░░░░░░░░                    │    one per │
│            │   ├─────────────────────────────────────────┤    job     │
│  account   │   │ ⟳  Compressing  trailer.mov   47%        │            │
│  slot      │   │ ▇▇▇▇▇▇▇▇▇░░░░░░░░░░░                      │            │
│            │   ├─────────────────────────────────────────┤            │
│            │   │ ⟳  Scanning  act3ai-repo   1,240 files   │            │
│            │   └─────────────────────────────────────────┘            │
│            │   ── toast stack rises below the dock ──                  │
└────────────┴──────────────────────────────────────────────────────────┘
   ▲ dock left edge = 272px (never over the 256px bar)
```

### 10.2 The card

One card per **active job**:

* **Icon** — lucide-react **`RefreshCw`** (two arrows forming a circle) with `animate-spin`, sized to the
  text (~`h-4 w-4`). It circles continuously = "progress is happening." (A per-kind glyph may lead the
  label — see §11 — but the spinning arrows are the constant.)
* **Label** — muted **verb** ("Syncing", "Compressing", "Scanning", "Pinning", "Hashing", "Ignoring") +
  the **target** (repo name or file basename) in medium weight, single line, truncated if long.
* **Progress detail (optional).** When the job reports a fraction, a **thin determinate bar** under the
  label plus a small right-aligned detail (`412 / 734 MB`, `47%`, `1,240 files`). When it doesn't, **no
  bar** — the spinner alone signals indeterminate work.
* **Shape** — **short** (label line + optional bar, `py-2`) and **somewhat wide**
  (`min-w-[260px]`, `max-w-[360px]`). Near-white card, subtle border (`--lfb-border`) and shadow,
  rounded. Each card is an isolated element so it can leave independently.
* **Stacking** — a vertical `flex-col` with a small gap; newest on top.

### 10.3 Lifecycle

1. A job **starts** → a card is added immediately. For a **browser-initiated** job the add is
   **optimistic** (the user sees it the moment they click **Sync now** / **Compress** / **Rescan**,
   before the round-trip resolves). For a **server-initiated** job it appears on the **next progress
   poll** (§12).
2. The card stays while the job is in flight, updating its bar/detail from progress reports.
3. The job **finishes** (success **or** error) → **its card is removed at once**. Success also
   invalidates the relevant queries (the repo/file grids recolor, the uncompressed/ignored counts
   update); failure raises an **error toast** (§9). A brief fade-out is allowed but must not delay
   removal past ~200 ms.
4. When the **last** card leaves, the dock unmounts. A single batch **"…complete"** success toast fires
   once a user-initiated batch settles (not one per job — the cards already gave per-job feedback);
   each **failed** job still gets its own error toast.

### 10.4 Accessibility

* The dock is an `aria-live="polite"` region so screen readers announce "Syncing movie.mp4" and its
  removal. Determinate cards expose `role="progressbar"` with `aria-valuenow`.
* The spinner is decorative (`aria-hidden`); the **text** carries the meaning — motion/color is never
  the only signal.

## 11. What shows up in the dock — the LFBridge operation taxonomy

Unlike the sister app (one "scan" verb), LFBridge runs **several kinds** of long job. The dock renders
one small, shared card type; the **verb** and any progress detail come from the job's `kind`:

| `kind` | Card verb | Target | Progress detail (when available) | Where it starts |
| --- | --- | --- | --- | --- |
| `scan` | **Scanning** | repo / root name | files walked (`1,240 files`) | `POST /api/repos/rescan` (`scanAll`); the scan worker |
| `sync` | **Syncing** | file or repo name | bytes added (`412 / 734 MB`) | `POST /api/repos/:repoId/sync`; the sync worker |
| `pin` | **Pinning** | file / CID | indeterminate | inside a sync job |
| `publish` | **Publishing** | backbone / IPNS | indeterminate | inside a sync job |
| `compress` | **Compressing** | file basename | `ffmpeg` percent (`47%`) | a user-invoked compress action (charter: video 1st, image 2nd) |
| `hash` | **Hashing** | file basename | bytes hashed | tracking a file (per-file fingerprint / compressed+uncompressed hashes) |
| `fingerprint` | **Fingerprinting** | file basename | indeterminate | perceptual/content-fingerprint pass (local-only, no network) |
| `ignore` | **Ignoring** | file basename | (fast) | one-click git-ignore of a big, non-ignored file |
| `import` | **Importing** | file / CID | (fast) | adopting an already-pinned CID into tracking ([ipfs.mdx §4](./ipfs.mdx#4-import--the-load-bearing-feature)) — metadata-only, no bytes moved |

Two policy reminders the dock inherits (charter): **never** compress/convert/ignore a file **unless the
user explicitly asked** — a `compress`/`ignore` card only ever exists because the user clicked it; and
the `fingerprint`/`hash` work is **strictly local** (it must never appear to "phone home," and it
doesn't — no network job kind exists).

## 12. Where progress comes from — two sources, one dock

The dock's active set is the **union** of two streams:

**A. Browser-initiated jobs (optimistic).** The pages route their long actions through **one** shared
runner (`useProgress().run(...)`) instead of the old blocking `mutate`. On click it adds a card, fires
the request, and removes the card when the request resolves — exactly the EmailDeliveryHero fan-out
model. This is what makes **Sync all** / **Compress selected** show every item at once and drain
unevenly.

**B. Server-initiated jobs (polled).** Because the **scan** and **sync** workers run under `launchd` on
their own schedule — and because a job may have been started in **another browser tab** — the dock must
show work **this tab did not start**. A new backend **job registry** tracks every in-flight job
(browser- or worker-initiated) with `{ id, kind, target, startedAt, progress? }`, exposed at:

```
GET /api/progress   →  { ok: true, data: { jobs: Job[] } }
Job = { id, kind, target, startedAt, done?, total?, unit? }   // done/total present iff determinate
```

The frontend polls it (TanStack Query `refetchInterval`) at a **gentle cadence while idle** and a
**faster one while the dock is non-empty**, and **merges** those jobs into the same dock by `id`. A
server job's card leaves when it stops appearing in the poll. Optimistic (A) and polled (B) cards are
**de-duplicated by job id**, so a job this tab started (and already shows optimistically) is not
doubled when the same job comes back on the poll.

> This keeps the "background process transparency" promise of the app whole: the **Sync page** already
> shows whether each worker is **installed** and **on/off** ([scan.mdx §7](./scan.mdx), SyncPage); the
> dock now shows, live, **when a worker is actually running right now**.

## 13. Concurrency model — jobs run in parallel, bounded

* **Bounded fan-out.** Both the browser runner and the backend workers cap parallelism with a small
  limiter (default **4** concurrent). Extra items **queue** and start as slots free; a card is added
  only when the item **actually starts**, so the dock reflects **true in-flight** work (a queued item
  simply has no card yet).
* **A tiny shared `mapLimit`.** A dependency-free `mapLimit(items, limit, fn)` (a `p-limit`-style
  runner, a few lines) lives in **both** packages — `frontend/src/lib/concurrency.ts` and
  `backend/src/shared/concurrency.ts` — the single "task view" the browser fan-out and the server
  runner share. No worker threads / `piscina`: scan, IPFS, hashing and even `ffmpeg` supervision are
  **I/O- or process-bound**, so bounded async concurrency on the event loop is the right, lightweight
  tool.
* **Endpoints become non-blocking + reportable.** The blocking handlers (`/rescan`, `/:repoId/sync`)
  are re-shaped to **register a job**, return promptly with the job id, and **report progress** into
  the registry as they run — so `GET /api/progress` can surface them and the card can show a bar.
* **Safe persistence under parallelism.** Parallel jobs that write the same on-disk state (a repo's
  `status.yaml` / `manifest.yaml`, [storage.mdx](./storage.mdx)) **serialize their write step** through
  an in-service async lock (a promise chain) so no concurrent job clobbers another's result — invisible
  to callers; only the disk write is serialized.

## 15. Acceptance criteria (Part II)

1. Toasts render **bottom-left**, ~**2× larger**, and the container's left edge is **≥ 272px** so a
   toast never overlaps the 256px left bar — on **every** route. (Replaces `position="bottom-right"` in
   `main.tsx:60`.)
2. Existing mutations still fire their same toasts ("… updated", errors) — only position/size change.
3. Starting **N** long jobs (e.g. **Sync all** across N repos, or compress N files) adds **N cards** (up
   to the concurrency cap at a time), each **"Running <op> <target>"** with a **spinning `RefreshCw`**;
   cards **stack**, sit clear of the left bar, are short and somewhat wide.
4. Jobs run **concurrently (bounded)** — a fast job's card disappears while a slow job's card remains;
   the dock **drains card-by-card** and **unmounts** when the last finishes.
5. Jobs that report a fraction show a **determinate bar + detail** (`412 / 734 MB`, `47%`,
   `1,240 files`); jobs that can't show the **spinner only**.
6. A job started **server-side** (a `launchd` scan/sync run) or in **another tab** appears in the dock
   via `GET /api/progress`; optimistic and polled cards are **de-duplicated by id** (no doubles).
7. The blocking handlers (`/rescan`, `/:repoId/sync`) **register jobs and report progress**; concurrent
   jobs **never lose a result** to a competing state write (serialized persist).
8. The concurrency limiter is a **small dependency-free `mapLimit`** shared in both packages; **no**
   worker-thread / `piscina` dependency is added.
9. `compress` and `ignore` cards exist **only** because the **user explicitly invoked** them; no
   `fingerprint`/`hash` job performs any **network** call (charter: local-only, never phones home).
10. On a batch settling, a single **"…complete"** toast fires; each failed job raises its own error
    toast; successful jobs **invalidate queries** so grids/counts refresh with no full reload.
