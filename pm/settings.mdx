# LargeFileBridge — Settings (Global)

The **Settings** page (`/settings`, the Settings nav item —
[left_bar.mdx §3](./left_bar.mdx#3-the-app-bar-main-navigation)) holds LargeFileBridge's (LFB)
**computer-wide, global** configuration. Its most important control, and the focus of this spec, is
the **big-file threshold**: the single number that decides what LFB even considers a "big file."

Everything on this page writes to the app-level, computer-wide `config.yaml`
([storage_local.mdx §3](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical)). These
are global defaults for the whole install; a repo may override a subset of them on its own page
([repo_settings.mdx](./repo_settings.mdx)).

Siblings: [repo_settings.mdx](./repo_settings.mdx) (per-repo overrides, reached via the gear icon) ·
[scan.mdx](./scan.mdx) (the scan that *applies* the threshold) · [storage_local.mdx](./storage_local.mdx) (where
these values persist).

---

## 1. The big-file threshold — the headline setting

**What counts as "big."** A file is **big** when its size is **at or above the threshold**. Only big
files are candidates for bridging over IPFS; everything smaller is left to Git as usual. This one
number is the dividing line the whole product turns on.

* **Default: 100 MB.** A fresh install treats any file **≥ 100 MB** as big. (This mirrors GitHub's own
  100 MB hard push limit — see `README.md` — so the default catches exactly the files Git hosting
  rejects.)
* It is a **global setting**: one value for the whole computer, inherited by every repo unless that
  repo overrides it (§3).
* The **[scan](./scan.mdx)** reads this value every run and flags every file at or above it; the scan
  applies the threshold but does not define it — **this page is the authoritative source.**

### 1.1 The control — a value field with a unit dropdown

The threshold renders as **one number input with a unit dropdown on its right**:

```
 Big-file threshold
 ┌───────────────┐  ┌──────┐
 │      100      │  │ MB ▾ │      Files at or above this size are bridged over IPFS.
 └───────────────┘  └──────┘
      value            unit
```

* **Value** — the numeric field. Default `100`.
* **Unit dropdown (on the right)** — toggles between **MB**, **GB**, and **TB**. Default **MB**.
  Changing the unit **does not silently rescale the number** — it changes the interpretation of the
  value the user typed (100 MB → switch to GB → 100 GB). The UI shows the resolved byte size as helper
  text (`= 104,857,600 bytes`) so the meaning is never ambiguous.
* The pair (value + unit) is the human-facing form; LFB **stores the resolved byte count** in
  `config.yaml` so all downstream code compares plain bytes.

The value + unit threshold control (and the resolved-bytes helper text) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/settings/SettingsPage.tsx` — `SettingsPage`

### 1.2 Units — MB / GB / TB

| Unit | Multiplier (binary, matching `storage_local.mdx` byte fields) | Example value → bytes |
| --- | --- | --- |
| **MB** | 1 MB = 1,048,576 bytes (MiB) | 100 MB → 104,857,600 |
| **GB** | 1 GB = 1,073,741,824 bytes (GiB) | 2 GB → 2,147,483,648 |
| **TB** | 1 TB = 1,099,511,627,776 bytes (TiB) | 1 TB → 1,099,511,627,776 |

The dropdown offers exactly these three units, in this order. (Binary multipliers keep the stored
byte values consistent with the `size:`/`min_bytes:` fields already used throughout
[storage_local.mdx](./storage_local.mdx).)

### 1.3 Where it persists

```yaml
# ~/T/_large_files_bridge/config.yaml  (excerpt — the global big-file threshold)
big_file:
  threshold_bytes: 104857600   # resolved bytes — the SOURCE OF TRUTH downstream code reads
  # The value+unit the user typed, kept so the UI redisplays their form exactly:
  threshold_display:
    value: 100
    unit: MB                    # MB | GB | TB
```

This `big_file` block is the authoritative home of the threshold. It **supersedes** the older
`pin_process.large_file_min_bytes` / per-repo `large_files.min_bytes` defaults in
[storage_local.mdx §3/§6](./storage_local.mdx#3-app-level-configyaml--computer-wide-settings-hierarchical), which
should be re-pointed at this value (see [scan.mdx §9](./scan.mdx#9-reconciliation-with-existing-specs)).

Saving the threshold — resolving value+unit to `threshold_bytes` and persisting the `big_file` block — is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/settings/settings.router.ts` — `settingsRouter` `PATCH /` (uses `toBytes()` into `updateAppConfig()`)

Reading and writing the computer-wide `config.yaml` (the `big_file` block and every other global default) is handled in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/modules/store-model/config.service.ts` — `getAppConfig()` / `updateAppConfig()`

## 2. Threshold resolution — global default, per-repo override

The threshold is **global with per-repo override**. Resolution for any given repo:

```
effective_threshold(repo) =
  repo.big_file_override.value (in its unit)   if repo's override checkbox is ON
  else the GLOBAL big_file.threshold_bytes      (default 100 MB)
```

* By default **every repo inherits the global value** — there is nothing to set per repo.
* A repo opts out only by **turning on its override checkbox** and typing its own value+unit. That
  affects **only that repo**.
* The [scan](./scan.mdx) computes `effective_threshold` per unit each run and records it in that
  unit's `status.yaml` (`effective_threshold_bytes`) so the UI can always show what "big" meant for a
  given repo on a given scan.

## 3. The per-repo override — checkbox OFF by default

The override lives on the **repo's own settings page** ([repo_settings.mdx](./repo_settings.mdx)),
but the *rule* is defined here because it is a property of this global setting:

* Each repo has a **checkbox: "Override the global big-file threshold for this repo."**
* **Default: OFF** for every repo. OFF means **inherit the global value** — the repo shows the global
  threshold, greyed out, as read-only context.
* When a user **turns the checkbox ON**, the repo reveals its own value field + unit dropdown
  (same MB/GB/TB control as §1.1), pre-filled with the current global value, which they then change.
* Turning it back **OFF** discards the override and the repo reverts to inheriting the global value.

This is deliberately a two-step opt-in: the common case is "one threshold for everything," and a user
only goes out of their way to override when a specific repo needs it (e.g. a repo full of 40 MB images
where they want a lower cutoff, or a video repo where they want a higher one).

The per-repo override persists in that repo's own config
([storage_local.mdx §6.2](./storage_local.mdx#62-pinrrepo_nameconfigyaml)):

```yaml
# pin/r/<repo_name>/config.yaml  (excerpt — the per-repo threshold override)
big_file_override:
  enabled: false                # the checkbox — OFF = inherit the global threshold
  value: 100                    # only meaningful when enabled: true
  unit: MB                      # MB | GB | TB
```

### 4.1 Compression (codec allow/deny + file-type conversion, per media type)

The **Compression** section lets the user choose, **per media type** (images / video / audio), which
codecs they **want** and which they **never want** — the deny list exists mostly for **social-media
compatibility** (the newest, smallest codec may not be accepted by the sites they post to). Per media
type: **Enabled**, **Quality** (`low | medium | high | lossless`, default **medium**), an ordered
**Prefer** list, and a **Deny** list. Defaults: images prefer `jpeg`/`webp` and deny `jpeg2000`; video
prefers `h264`/`hevc` and denies `av1`; **audio is disabled** (planned later). The **preserve-resolution**
invariant is locked on. Full behavior is in [compression.mdx](./compression.mdx).

#### 4.1.1 File-type conversion — two controls, per media type, BOTH DEFAULT ON

Beyond the codec allow/deny, the Compression section carries the **file-type conversion** controls
specified in full in [images.mdx §2](./images.mdx#2-the-two-settings-settings--compression). They are
**per media type** (images **and** video) and **both default ON**:

* **"Convert file types when compressing" (`convert_types`, default ON).** When on, a compress may
  **change the format** to a better compressible / more-compatible target — **HEIC → JPEG**, PNG → JPEG,
  etc. When **off**, compression is **format-preserving** (re-encode in place / lossless recompress; the
  extension never changes).
* **"Compress all file types" — a per-extension checkbox list (`skip_exts`, default empty = all in).**
  Every recognized **image** and **video** extension is **checked (in scope) by default**. Unchecking one
  leaves that extension **out of compression entirely**. The list is an **opt-OUT** (extensions to skip),
  so "all in" stays the default even as new extensions are added.

The flagship row is Apple's **`.heic`**: it is checked and shows **`→ JPEG`**, **ON by default** — because
HEIC is poorly supported by the sites people post to while JPEG is universal. HEIC needs special decode
care (extract the **primary still**, never a thumbnail / depth-aux image / Live-Photo motion frame) and
all conversions keep the **same pixel resolution** — both are specified in
[images.mdx §4](./images.mdx#4-apple-heic--jpeg-the-flagship-conversion--locked).

```
 Images                                        Videos
 ┌──────────────────────────────┐             ┌──────────────────────────────┐
 │ ☑ Convert file types         │             │ ☑ Convert file types         │
 │ Compress these types:        │             │ Compress these types:        │
 │  [✓] .heic  → JPEG           │             │  [✓] .mov  → H.264 (MP4)     │
 │  [✓] .png   → JPEG           │             │  [✓] .avi  → H.264 (MP4)     │
 │  [✓] .avif  → JPEG           │             │  [✓] .mkv  → H.264 (MP4)     │
 │  [✓] .jpg   → re-encode      │             │  [✓] .mp4  → re-encode       │
 │  …                           │             │  …                           │
 └──────────────────────────────┘             └──────────────────────────────┘
```

These two fields persist on the existing `compression.images` / `compression.video` blocks as
`convert_types` (bool, default `true`) and `skip_exts` (string[], default `[]`).

### 4.3 Parallelism — the CPU-core budget

Large File Bridge's heaviest work is **bulk, embarrassingly-parallel file work** — compress every video
and image inside a directory, transcribe or fingerprint a whole repo, batch-transcode. On a many-core
machine that work should chew through the box in parallel rather than crawl one file at a time. The
**Parallelism** section exposes the **single global knob** that governs how hard it may push
([parallelization.mdx §4](./parallelization.mdx)).

The control is **one percent slider/field**, labeled **"Use up to N% of CPU cores for background
compression & processing,"** defaulting to **90%**. Beneath it, the UI shows the **live resolved core
count** so the abstract percentage is never ambiguous:

```
 Parallelism
 ┌──────────────────────────────────────────┐
 │  Use up to  [ ▓▓▓▓▓▓▓▓▓░ ]  90 %          │   ≈ 14 of 16 cores
 └──────────────────────────────────────────┘
   Background compression & processing may use up to this share of your CPU.
```

* **What it tunes — the MASS-COMPUTE budget only.** The percentage backs
  `performance.max_core_fraction` ∈ `(0, 1]`, and the mass-compute budget is
  `round(cores × max_core_fraction)` — the "use up to ~90% of cores" budget for **pure batch CPU work
  the user explicitly kicked off** (compression, fingerprinting, batch transcode). Turning the slider
  down keeps more headroom on a laptop; turning it up pushes a workstation harder.
* **What it does NOT tune — the RESPONSIVE budget.** Work that runs **alongside** interactive use — the
  15-min **pin pass** and the discovery **scan/index walks** — draws the **responsive budget**
  (`cores − 2`, leaving 2 cores for the web app + IPFS). That floor is a **fixed safety margin and is not
  user-tunable**; the slider never touches it.
* **Live, no restart.** The resolved core count shown beside the slider is exactly what the next bulk
  operation will use — the budget helper reads the setting **live**, so a change takes effect on the
  **next** run with no restart.

The percentage field lives in the app config schema, in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/shared/src/schemas.ts` — `AppConfigSchema.performance`
(`max_core_fraction`, default `0.9`).

The slider surface (and the live "≈ N of M cores" readout) is implemented in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/frontend/src/pages/settings/SettingsPage.tsx` —
`SettingsPage` Parallelism section.

The resolved mass-compute budget is computed live from this setting in code:
`~/BGit/Bryan_git/LargeFileBridge/code/packages/backend/src/shared/concurrency.ts` — `coreBudget()`
(reads `performance.max_core_fraction`); the fixed responsive floor is `responsiveBudget()` (`cores − 2`).
